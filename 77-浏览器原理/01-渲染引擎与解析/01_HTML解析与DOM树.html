HTML解析与DOM树

核心概念：浏览器将HTML文档解析为一棵DOM树（Document Object Model Tree），这是页面渲染的基础数据结构。解析过程包括词法分析（Tokenization）、语法分析（Tree Construction）和推测解析（Speculative Parsing）三个关键阶段。

一、整体解析流程

HTML字节流 → 字符流 → Tokenizer(词法分析) → Token序列 → Tree Construction(语法分析) → DOM树 ↑ ↓ 网络层接收数据 与CSSOM合并 → 渲染树

1.1 字节流到字符流

浏览器从网络层接收HTML字节流后，根据 <meta charset> 或 HTTP响应头中的 Content-Type 确定编码方式，将字节流转换为字符流。

  • 编码检测优先级：BOM > HTTP Content-Type > meta charset > 自动检测（chardet）
  • UTF-8 BOM：如果字节流以 EF BB BF 开头，强制使用UTF-8
  • 编码转换错误会导致乱码，这在中文网页中尤为常见

1.2 解析模型

HTML解析采用的不是传统的上下文无关文法（Context-Free Grammar），而是由HTML规范定义的状态机解析模型。这是因为HTML具有容错性要求——浏览器必须能够解析格式不规范的HTML。

特性XML解析HTML解析
文法类型上下文无关文法状态机 + 规范定义的规则
容错性严格，遇错即停高度容错，自动修复
解析算法SAX / DOM增量式解析（Incremental）
脚本处理不涉及可能阻塞解析

二、词法分析（Tokenization）

2.1 Token类型

Tokenizer是一个有限状态机（Finite State Machine），逐字符扫描输入流，将字符序列转换为Token序列。HTML5规范定义了以下Token类型：

Token类型示例说明
DOCTYPE<!DOCTYPE html>文档类型声明
Start Tag<div class="box">开始标签，含属性
End Tag</div>结束标签
Character文本内容连续字符数据
Comment<!-- 注释 -->HTML注释
End-of-File-输入结束标记

2.2 状态机核心状态

Tokenizer包含约80个状态，以下是核心状态转换：

// 简化的Tokenizer状态机伪代码
enum State {
    Data,               // 默认状态
    TagOpen,            // 遇到 '<'
    TagName,            // 读取标签名
    BeforeAttributeName,// 标签名后的空白
    AttributeName,      // 读取属性名
    BeforeAttributeValue,// 遇到 '='
    AttributeValueDoubleQuoted, // 双引号属性值
    AttributeValueSingleQuoted, // 单引号属性值
    AttributeValueUnquoted,     // 无引号属性值
    SelfClosingStartTag, // 遇到 '/'
    EndTagOpen,         // '</' 后
    ScriptData,         // <script> 内容
    RAWTEXT,            // <style> / <textarea> 内容
};

// 核心循环
function tokenize(input) {
    let state = State.Data;
    let currentToken = null;
    let tokens = [];

    for (let char of input) {
        switch (state) {
            case State.Data:
                if (char === '<') {
                    state = State.TagOpen;
                } else {
                    emitCharacterToken(char);
                }
                break;
            case State.TagOpen:
                if (char === '/') {
                    state = State.EndTagOpen;
                } else if (isAsciiAlpha(char)) {
                    currentToken = { type: 'StartTag', name: char, attributes: [] };
                    state = State.TagName;
                }
                break;
            case State.TagName:
                if (isWhitespace(char)) {
                    state = State.BeforeAttributeName;
                } else if (char === '>') {
                    emitToken(currentToken);
                    state = State.Data;
                } else {
                    currentToken.name += char;
                }
                break;
            // ... 更多状态处理
        }
    }
}

2.3 属性解析

属性解析的关键规则：

  • 属性值引号：双引号、单引号、无引号三种形式均合法
  • 属性去重：同名属性只保留第一个（除了 class 等特殊属性会合并）
  • 布尔属性：如 disabled、checked，存在即为true
  • 属性名大小写不敏感（HTML模式），属性值大小写敏感
<!-- 以下三种写法都合法 -->
<input type="text" value="hello">
<input type='text' value='hello'>
<input type=text value=hello>

<!-- 同名属性去重，只保留第一个 -->
<div id="a" id="b">  →  id="a"

<!-- class属性会合并（特殊规则）-->
<div class="a" class="b">  →  class="a b"

三、语法分析（Tree Construction）

3.1 DOM树构建过程

Tree Construction阶段接收Token序列，构建DOM树。这是一个增量式过程——每个Token到达时立即处理，不需要等待整个HTML解析完毕。

// 简化的Tree Construction伪代码
function treeConstruction(tokens) {
    let openElements = [];  // 开放元素栈
    let current = document;

    for (let token of tokens) {
        switch (token.type) {
            case 'StartTag':
                let element = createElement(token);
                current.appendChild(element);
                if (!token.selfClosing) {
                    openElements.push(current);
                    current = element;
                }
                break;
            case 'EndTag':
                // 从开放元素栈中找到匹配的开始标签
                let matchIndex = findMatchingTag(openElements, token.name);
                if (matchIndex !== -1) {
                    // 关闭中间的未闭合标签（容错处理）
                    while (openElements.length > matchIndex) {
                        current = openElements.pop();
                    }
                    current = openElements.pop() || document;
                }
                break;
            case 'Character':
                current.appendChild(createTextNode(token.value));
                break;
        }
    }
}

3.2 插入模式（Insertion Modes）

HTML解析器使用"插入模式"来决定如何处理Token。不同的上下文（如表格内部、模板元素内）有不同的插入模式：

插入模式触发条件行为
initial解析开始处理DOCTYPE，跳过空白
before htmlDOCTYPE之后创建 <html> 元素
before headhtml元素内创建隐式 <head>
in headhead元素内处理meta/link/script/style等
after headhead关闭后创建隐式 <body>
in bodybody元素内处理大部分HTML内容
in tabletable元素内特殊处理tr/td等
in selectselect元素内限制允许的子元素
textscript/style内原始字符数据

3.3 容错机制

HTML规范定义了详细的容错规则，浏览器遇到格式错误时会自动修复。常见的容错场景：

<!-- 1. 缺少结束标签 -->
<p>段落1
<p>段落2
<!-- 浏览器自动补全：<p>段落1</p><p>段落2</p> -->

<!-- 2. 错误的嵌套 -->
<p><div>块级元素嵌套在行内元素内</div></p>
<!-- 浏览器修正为：<p></p><div>...</div><p></p> -->

<!-- 3. 表格中的非表格元素 -->
<table>
  <div>这不是表格内容</div>
</table>
<!-- 浏览器将div移到table之前 -->

<!-- 4. 重复的html/body标签 -->
<html><body>内容</body><html>
<!-- 忽略第二个html标签 -->

<!-- 5. 未关闭的格式化元素 -->
<b>粗体<i>斜体
<!-- 浏览器自动补全结束标签 -->
注意：虽然浏览器会容错，但不规范的HTML可能导致意外的DOM结构，影响CSS选择器匹配和JavaScript DOM操作。始终使用合法的HTML。

五、CSSOM（CSS Object Model）

5.1 CSSOM的构建

与DOM类似，浏览器也会将CSS解析为一棵CSSOM树。CSS解析同样使用状态机，但相比HTML解析更简单——CSS语法是上下文无关文法。

CSS字节流 → 字符流 → Tokenizer → CSSOM树 ↓ Style Rules（选择器 + 声明块） ↓ computed Style → 渲染树
  • 选择器解析：从右向左匹配（高效匹配策略）
  • 层叠计算：根据 specificity、importance、source order 确定最终样式
  • 继承：某些属性（如 color、font-size）从父元素继承

5.2 CSSOM Tree结构

/* CSS 样式表 */
body { font-size: 16px; color: #333; }
h1 { font-size: 32px; color: #1a73e8; }
.class { background: #f0f0f0; }
CSSStyleSheet ├── CSSRule: body │ ├── style.fontSize = "16px" │ └── style.color = "#333" ├── CSSRule: h1 │ ├── style.fontSize = "32px" │ └── style.color = "#1a73e8" └── CSSRule: .class └── style.background = "#f0f0f0"

5.3 DOM + CSSOM → 渲染树

DOM树和CSSOM树合并形成渲染树（Render Tree）：

  • 渲染树只包含可见节点（display: none 的元素不包含在内）
  • visibility: hidden 的元素仍占据空间，包含在渲染树中
  • 每个节点都计算了最终的computed style
  • 伪元素（::before、::after）会作为子节点添加到渲染树
关键点：CSS解析是阻塞渲染的。在CSSOM构建完成之前，浏览器不会开始渲染页面。因此应尽量减少关键CSS的大小，非关键CSS使用异步加载。

六、推测解析（Speculative Parsing / Preload Scanner）

6.1 什么是推测解析

当主线程的HTML解析器遇到阻塞资源（如 <script> 标签）而暂停时，浏览器会启动一个预加载扫描器（Preload Scanner），在后台提前发现并下载后续的关键资源。

主线程：HTML解析 → 遇到 <script> → 暂停解析，等待JS执行 后台： 预加载扫描器 → 扫描后续HTML → 发现资源 → 提前下载 ├── <link rel="stylesheet"> ├── <script src> ├── <img src> └── <link rel="preload">

6.2 推测解析的作用

  • 并行下载：在主线程阻塞时仍能发现并下载资源
  • 资源排序：根据资源类型和优先级安排下载顺序
  • 不执行：预加载扫描器只负责下载，不修改DOM或执行脚本
实践建议：
  • 将CSS放在 <head> 中，让预加载扫描器尽早发现
  • 使用 <link rel="preload"> 提示浏览器提前下载关键资源
  • 避免在 <body> 开头放置阻塞脚本，否则预加载扫描器无法提前工作

七、script标签对解析的影响

7.1 默认行为：阻塞解析

默认情况下，<script> 标签会阻塞HTML解析。浏览器暂停解析，下载（如果是外部脚本）并执行JavaScript代码，然后继续解析。

<!-- 解析到此处会暂停，等待下载和执行完毕 -->
<script src="app.js"></script>
<!-- 脚本执行完毕后继续解析 -->
<p>这段内容在脚本执行后才被解析</p>

7.2 async与defer属性

属性下载阻塞执行阻塞执行顺序DOMContentLoaded
无属性阻塞阻塞按顺序脚本执行完后触发
async不阻塞阻塞下载完立即执行（不确定顺序）可能在之前或之后
defer不阻塞不阻塞按文档顺序之前执行
type="module"不阻塞不阻塞默认defer行为之前执行
<!-- async：下载不阻塞，下载完立即执行，顺序不确定 -->
<script async src="analytics.js"></script>
<script async src="app.js"></script>

<!-- defer：下载不阻塞，HTML解析完按顺序执行 -->
<script defer src="vendor.js"></script>
<script defer src="app.js"></script>

<!-- ES Module 默认具有defer行为 -->
<script type="module" src="app.mjs"></script>
最佳实践：
  • 无依赖的独立脚本（如统计分析）使用 async
  • 有依赖关系的脚本使用 defer
  • 现代项目优先使用 type="module"
  • 将脚本放在 </body> 前的做法已过时，defer 更优

7.3 动态脚本

// 动态创建的脚本默认是async行为
const script = document.createElement('script');
script.src = 'app.js';
document.head.appendChild(script);
// 默认async=true，可通过设置script.async = false改为按顺序执行

八、常见面试题

Q1: DOMContentLoaded和load事件的区别？

  • DOMContentLoaded：HTML解析完毕，DOM树构建完成时触发。不等待样式表、图片、子框架加载
  • load：整个页面（包括所有依赖资源：样式表、图片等）加载完毕后触发
document.addEventListener('DOMContentLoaded', () => {
    console.log('DOM ready, 可以操作DOM了');
});

window.addEventListener('load', () => {
    console.log('所有资源加载完毕');
});

Q2: document.write为什么不好？

  • 在页面加载过程中调用会覆盖整个文档
  • 在页面加载后调用会重新打开文档流，覆盖所有内容
  • 阻塞HTML解析
  • 无法与CSP（Content Security Policy）兼容
  • 唯一合理场景：异步加载polyfill时配合 document.write

Q3: innerHTML vs textContent vs innerText？

属性读取写入性能
innerHTML返回HTML字符串解析HTML并重建DOM较慢（触发解析）
textContent返回所有文本（含隐藏元素）设置纯文本快
innerText返回可见文本设置纯文本最慢（触发回流）
总结：HTML解析是一个高度优化的增量过程，涉及Tokenization（词法分析）、Tree Construction（语法分析）和推测解析三个核心阶段。理解这些原理对于优化页面加载性能（如脚本加载策略、CSS优化）至关重要。
