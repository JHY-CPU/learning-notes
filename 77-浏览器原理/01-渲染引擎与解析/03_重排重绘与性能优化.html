重排重绘与性能优化

核心概念：重排（Reflow）和重绘（Repaint）是浏览器渲染过程中的两个关键操作。理解什么会触发它们，以及如何最小化它们的影响，是前端性能优化的核心技能。本文档详细讨论触发条件、布局抖动问题、will-change、content-visibility等优化手段。

一、重排（Reflow / Layout）

1.1 什么是重排

重排（也叫回流）是指浏览器重新计算元素的几何属性（位置和大小），并重新构建部分或全部布局树。重排是渲染管线中最昂贵的操作之一。

触发重排 → 重新计算受影响元素的几何属性 → 可能影响子元素、兄弟元素、祖先元素 → 重建布局树的受影响部分 → 触发重绘 → 触发合成

1.2 触发重排的操作

类别具体操作影响范围
几何属性变更width, height, margin, padding, border, font-size, line-height元素自身及其后代
位置变更top, left, bottom, right（对 positioned 元素）元素自身及可能影响其他元素
显示/隐藏display: none ↔ display: block元素自身及后续元素
DOM操作添加/删除/移动DOM节点受影响的子树
内容变化文本内容变化导致尺寸变化、图片加载完成元素自身及祖先
布局属性变更position, float, clear, overflow元素自身及周边元素
读取布局属性offsetTop, offsetLeft, offsetWidth, offsetHeight, scrollTop, scrollLeft, scrollWidth, scrollHeight, getComputedStyle(), getBoundingClientRect()强制同步布局
窗口大小变更窗口resize整个页面
字体加载Web字体加载完成使用该字体的所有元素

1.3 重排的影响范围

  • 自身：元素几何属性变化会影响自身
  • 后代：父元素尺寸变化影响所有后代元素
  • 祖先：子元素变化可能导致祖先元素尺寸变化（如 height: auto 的容器）
  • 兄弟：浮动元素、flex/grid布局中一个元素变化影响兄弟元素
  • 后续元素：文档流中前面元素尺寸变化影响后面所有元素的位置
display:none vs visibility:hidden： display: none 触发重排（元素从渲染树中移除），visibility: hidden 只触发重绘（元素仍在渲染树中，只是不可见）。

二、重绘（Repaint）

2.1 什么是重绘

重绘是指当元素的外观属性（不影响布局的属性）发生变化时，浏览器重新绘制受影响的区域。重绘的成本低于重排，但仍需消耗CPU资源。

2.2 只触发重绘（不触发重排）的属性

属性说明
color文字颜色
background-color, background-image背景色/背景图
visibility可见性（不同于display:none）
box-shadow盒子阴影
text-shadow文字阴影
border-radius圆角
outline, outline-color轮廓
border-color边框颜色（注意border-width会触发重排）

2.3 既不触发重排也不触发重绘的属性（只触发Composite）

属性说明
transform2D/3D变换（translate, rotate, scale）
opacity透明度
filter滤镜效果（需独立合成层）
will-change提示浏览器提前优化
性能黄金法则：做动画时，优先使用 transform 和 opacity，它们只触发GPU合成，性能最好。

三、布局抖动（Layout Thrashing）

3.1 什么是布局抖动

布局抖动是指在同一个JavaScript任务中，交替进行写操作（修改DOM/样式）和读操作（读取布局属性），导致浏览器被迫进行多次同步布局计算。

❌ 布局抖动（每读一次就强制同步布局一次）： for循环 { 写操作（修改style）→ 标记布局为dirty 读操作（offsetHeight）→ 强制同步布局（flush）→ 获取值 写操作（修改style）→ 标记布局为dirty 读操作（offsetHeight）→ 强制同步布局（flush）→ 获取值 } → 触发N次重排，性能极差

3.2 强制同步布局（Forced Synchronous Layout）

当JavaScript在修改样式后立即读取布局属性时，浏览器必须先完成所有挂起的布局计算，才能返回正确的值。这称为强制同步布局。

// ❌ 强制同步布局 - 性能灾难
function badExample() {
    const boxes = document.querySelectorAll('.box');
    boxes.forEach(box => {
        // 写操作：标记布局为dirty
        box.style.width = (box.offsetWidth * 2) + 'px';
        // 读操作：强制浏览器立即执行布局，才能获取正确的offsetWidth
        // 布局执行完毕后，下一个循环的写操作又标记为dirty
        // 循环往复...
    });
}

// ✅ 正确做法 - 先读后写
function goodExample() {
    const boxes = document.querySelectorAll('.box');

    // 第一轮：只读
    const widths = [];
    boxes.forEach(box => {
        widths.push(box.offsetWidth);  // 浏览器可以延迟布局
    });

    // 第二轮：只写
    boxes.forEach((box, i) => {
        box.style.width = (widths[i] * 2) + 'px';  // 只标记dirty，不执行布局
    });
    // 循环结束后，浏览器统一执行一次布局
}

3.4 使用FastDOM等库

// FastDOM思路：自动分离读写操作
import fastdom from 'fastdom';

fastdom.measure(() => {
    const height = element.offsetHeight;  // 读操作
    fastdom.mutate(() => {
        element.style.height = (height * 2) + 'px';  // 写操作
    });
});

// 现代替代方案：requestAnimationFrame
function updateLayout() {
    // 所有读操作
    const heights = Array.from(boxes).map(el => el.offsetHeight);

    requestAnimationFrame(() => {
        // 所有写操作
        boxes.forEach((el, i) => {
            el.style.height = (heights[i] * 2) + 'px';
        });
    });
}

四、批量DOM操作优化

4.1 DocumentFragment

// ❌ 逐个添加，每个都触发一次回流
for (let i = 0; i < 1000; i++) {
    const li = document.createElement('li');
    li.textContent = `Item ${i}`;
    ul.appendChild(li);  // 每次都可能触发回流
}

// ✅ 使用DocumentFragment，只触发一次回流
const fragment = document.createDocumentFragment();
for (let i = 0; i < 1000; i++) {
    const li = document.createElement('li');
    li.textContent = `Item ${i}`;
    fragment.appendChild(li);
}
ul.appendChild(fragment);  // 只触发一次回流

4.2 离线DOM操作

// 方法1：display:none → 操作 → display:block
const el = document.getElementById('list');
el.style.display = 'none';  // 触发1次重排
// ... 大量DOM修改操作 ...
el.style.display = 'block'; // 触发1次重排

// 方法2：将元素移出DOM → 操作 → 放回DOM
const el = document.getElementById('list');
const parent = el.parentNode;
parent.removeChild(el);      // 触发1次重排
// ... 大量DOM修改操作 ...
parent.appendChild(el);      // 触发1次重排

// 方法3：克隆 → 操作 → 替换
const el = document.getElementById('list');
const clone = el.cloneNode(true);
// ... 在克隆节点上做大量修改 ...
el.parentNode.replaceChild(clone, el);  // 触发1次重排

4.3 CSS类切换代替逐属性修改

// ❌ 每个style修改都可能触发重排
element.style.width = '100px';
element.style.height = '100px';
element.style.margin = '10px';
element.style.padding = '5px';

// ✅ 一次类切换，只触发1次重排
element.classList.add('resized');

// .resized {
//     width: 100px;
//     height: 100px;
//     margin: 10px;
//     padding: 5px;
// }

五、will-change属性

5.1 什么是will-change

will-change 是一个CSS属性，用于提前告知浏览器某个元素将要发生什么变化，让浏览器可以提前进行优化（如创建独立的合成层）。

/* 语法 */
will-change: auto;          /* 默认 */
will-change: scroll-position;  /* 将改变滚动位置 */
will-change: contents;         /* 将改变内容 */
will-change: transform;        /* 将改变transform */
will-change: opacity;          /* 将改变opacity */
will-change: transform, opacity;  /* 多个属性 */

5.2 will-change的工作原理

  1. 提升到独立合成层：对于 will-change: transform，浏览器提前将元素提升到独立的GPU层
  2. 创建新的层叠上下文：元素变为新的层叠上下文根
  3. 提前分配GPU资源：为该元素预留显存等资源
没有will-change： 用户交互 → 浏览器检测到变化 → 创建合成层 → 开始GPU加速（有延迟） 有will-change： 页面加载 → 浏览器看到will-change → 预创建合成层 → 变化发生时直接GPU加速（无延迟）

5.3 will-change的最佳实践

不要滥用will-change！每个 will-change 都会创建一个独立的合成层，消耗额外的内存（在低端设备上可能达数MB）。以下是一些规则： 
/* ❌ 错误：对所有元素使用 */
* { will-change: transform; }

/* ❌ 错误：在元素上永久保留 */
.element { will-change: transform; }

/* ✅ 正确：只在动画开始前添加，动画结束后移除 */
.element {
    transition: transform 0.3s;
}

.element.animate {
    will-change: transform;  /* 动画前添加 */
    transform: translateX(100px);
}

/* ✅ 正确：用JavaScript动态管理 */
function startAnimation(el) {
    el.style.willChange = 'transform';
    el.classList.add('animating');
}

function endAnimation(el) {
    el.classList.remove('animating');
    el.style.willChange = 'auto';
}

el.addEventListener('animationend', () => endAnimation(el));

/* ✅ 正确：对hover动画，用:hover触发 */
.button:hover {
    will-change: transform;
    transform: scale(1.05);
}

六、content-visibility深入

6.1 content-visibility: auto

这是 content-visibility 最有用的值，它会自动跳过屏幕外元素的渲染工作：

/* 长列表优化 */
.virtual-list-item {
    content-visibility: auto;
    contain-intrinsic-size: 0 50px; /* 每项预估高度50px */
}

/* 长文章优化 */
.article-body {
    content-visibility: auto;
    contain-intrinsic-size: auto 1000px;
}
没有content-visibility： 浏览器渲染全部元素（包括屏幕外的） → 初始渲染慢，内存占用高 有content-visibility: auto： 浏览器只渲染视口内的元素 → 初始渲染快，内存占用低 用户滚动 → 滚入视口的元素被渲染 用户滚出 → 滚出视口的元素释放渲染状态

6.2 contain-intrinsic-size

当 content-visibility 跳过渲染时，元素没有大小信息。contain-intrinsic-size 用于指定占位大小，防止布局跳动：

/* 固定占位大小 */
.item {
    content-visibility: auto;
    contain-intrinsic-size: 200px;  /* 高度200px */
}

/* 宽度和高度 */
.item {
    content-visibility: auto;
    contain-intrinsic-size: 300px 200px;  /* 宽300px，高200px */
}

/* auto关键字：记住上次渲染时的实际大小 */
.item {
    content-visibility: auto;
    contain-intrinsic-size: auto;  /* 记住上次渲染的大小 */
    contain-intrinsic-size: auto 200px;  /* 宽度自动，高度200px */
}

6.3 content-visibility的注意事项

  • 锚点跳转：<a href="#section"> 跳转到被跳过渲染的元素时，浏览器会自动渲染该元素
  • Find-in-Page：浏览器搜索功能会渲染被跳过的元素以搜索文本
  • 焦点管理：被跳过渲染的元素内的可聚焦元素仍可获得焦点
  • Intersection Observer：可能在元素实际可见之前就触发回调
  • 不适用于：需要精确控制首屏内容的场景（如LCP优化）

七、动画性能优化

7.1 CSS动画 vs JavaScript动画

特性CSS动画（transition/animation）JavaScript动画（requestAnimationFrame）
性能可被提升到合成线程在主线程执行
灵活性有限（预定义的关键帧）完全灵活
控制通过animation-play-state完全控制暂停/继续/反向
适用场景简单的状态过渡复杂的交互式动画
主线程阻塞可能不阻塞（合成器动画）会阻塞

7.2 合成器线程动画（Compositor Thread Animation）

如果CSS动画只涉及 transform 和 opacity，并且元素在独立的合成层上，动画可以完全在合成器线程上运行，不依赖主线程：

/* 合成器线程动画 - 不受主线程JS阻塞影响 */
@keyframes slide {
    from { transform: translateX(0); }
    to   { transform: translateX(100vw); }
}

.animated {
    animation: slide 2s linear infinite;
    will-change: transform; /* 确保在独立合成层 */
}

/* 即使主线程被大量JS阻塞，这个动画仍然流畅 */
性能对比：
  • 在主线程上的动画（如 left、top）：每帧需要 Layout + Paint + Composite
  • 合成器线程动画（transform、opacity）：每帧只需 Composite
  • 在主线程繁忙时，合成器线程动画仍保持流畅

7.3 使用requestAnimationFrame

// requestAnimationFrame: 在下一次浏览器重绘前执行回调
// 约 60fps（每秒60帧），即每 16.67ms 执行一次

let position = 0;
function animate() {
    position += 2;
    element.style.transform = `translateX(${position}px)`;

    if (position < 300) {
        requestAnimationFrame(animate);  // 递归调用
    }
}
requestAnimationFrame(animate);

// ❌ 不要用setTimeout/setInterval做动画
// - 不与浏览器刷新率同步，可能丢帧或重复帧
// - 即使页面不可见也会执行，浪费资源
// - 无法自动暂停

// ✅ rAF的优势：
// - 与浏览器刷新率同步（通常60Hz）
// - 页面不可见时自动暂停，节省资源
// - 批量执行：多个rAF回调在同一帧内执行

7.4 使用requestIdleCallback

// requestIdleCallback: 在浏览器空闲时执行低优先级任务
// 不阻塞关键渲染和用户交互

requestIdleCallback((deadline) => {
    // deadline.timeRemaining(): 返回当前帧剩余的空闲时间（ms）
    // deadline.didTimeout: 是否已超时

八、性能优化检查清单

8.1 减少重排

  • 使用 transform 代替 top/left 做位移动画
  • 使用 opacity 代替 visibility 做显隐动画
  • 批量修改DOM（DocumentFragment、离线DOM、一次性innerHTML）
  • 使用CSS类切换代替逐属性修改style
  • 将动画元素设置为 position: absolute/fixed，脱离文档流
  • 避免在循环中读取布局属性（先读后写）
  • 使用 contain 限制布局影响范围

8.2 减少重绘

  • 减少透明度变化的面积
  • 使用 will-change 提示浏览器提前创建合成层
  • 使用 content-visibility: auto 跳过屏幕外元素
  • 避免不必要的 box-shadow 和渐变背景（它们的绘制成本较高）

8.3 利用DevTools诊断

  • Performance面板：查看Layout、Paint、Composite各阶段的耗时
  • Layers面板：检查图层分解是否合理
  • Rendering面板：开启 "Paint flashing" 查看重绘区域
  • Rendering面板：开启 "Layout Shift Regions" 检测CLS问题
  • Performance Monitor：实时监控CPU、JS堆大小、DOM节点数
总结：重排重绘优化的核心思路是：(1) 优先使用 transform/opacity 做动画，利用GPU合成；(2) 避免布局抖动，分离读写操作；(3) 使用 will-change 预创建合成层；(4) 使用 content-visibility 减少不必要的渲染；(5) 批量操作DOM，减少重排次数。
