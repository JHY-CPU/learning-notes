浏览器网络栈

核心概念：浏览器网络栈是从用户输入URL到页面内容加载完成的完整网络处理过程，包括DNS解析、TCP连接、TLS握手、HTTP请求/响应等关键环节。现代浏览器通过DNS预解析、连接复用、HTTP/2多路复用、资源优先级等技术大幅优化网络性能。

一、整体网络请求流程

1.1 从URL到页面的完整流程

URL输入 ↓ 1. URL解析（URL Parser） ↓ 2. DNS解析（DNS Lookup）→ IP地址 ↓ 3. TCP连接（TCP Handshake）→ 三次握手 ↓ 4. TLS握手（TLS Handshake）→ 加密通道（HTTPS） ↓ 5. HTTP请求/响应（Request/Response） ↓ 6. 内容解码（gzip/brotli/deflate） ↓ 7. 资源解析（HTML/CSS/JS解析） ↓ 页面渲染

1.2 浏览器网络栈架构

┌─────────────────────────────────────────────┐ │ Browser Process │ │ ┌─────────────────────────────────────────┐ │ │ │ Network Service (独立进程) │ │ │ │ ┌───────────────────────────────────┐ │ │ │ │ │ Chrome Multi-Layer Cache │ │ │ │ │ │ HTTP Cache / Service Worker Cache │ │ │ │ │ └───────────────────────────────────┘ │ │ │ │ ┌───────────────────────────────────┐ │ │ │ │ │ HTTP/1.1 / HTTP/2 / HTTP/3 (QUIC) │ │ │ │ │ └───────────────────────────────────┘ │ │ │ │ ┌───────────────────────────────────┐ │ │ │ │ │ TLS 1.2 / TLS 1.3 │ │ │ │ │ └───────────────────────────────────┘ │ │ │ │ ┌───────────────────────────────────┐ │ │ │ │ │ TCP / QUIC (UDP) │ │ │ │ │ └───────────────────────────────────┘ │ │ │ └─────────────────────────────────────────┘ │ └─────────────────────────────────────────────┘

二、DNS解析（DNS Lookup）

2.1 DNS解析过程

将域名转换为IP地址的过程：

  1. 浏览器缓存：浏览器首先检查自身的DNS缓存
  2. 操作系统缓存：检查OS的DNS缓存（ipconfig /displaydns）
  3. hosts文件：检查 /etc/hosts 或 C:\Windows\System32\drivers\etc\hosts
  4. 本地DNS服务器：向ISP的DNS服务器发起查询
  5. 递归查询：本地DNS服务器向根域名服务器 → 顶级域名服务器 → 权威域名服务器递归查询
浏览器 → 浏览器缓存 → OS缓存 → hosts文件 → 本地DNS服务器 ↓（缓存未命中） 本地DNS → 根DNS(.) → 顶级DNS(com) → 权威DNS(example.com) → 返回IP ←────────────────────────────────────────────────────┘

2.2 DNS预解析（DNS Prefetch）

DNS预解析允许浏览器在用户点击链接之前提前解析域名，减少DNS查询延迟：

<!-- 对页面中所有链接启用DNS预解析 -->
<meta http-equiv="x-dns-prefetch-control" content="on">

<!-- 预解析特定域名 -->
<link rel="dns-prefetch" href="//api.example.com">
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="dns-prefetch" href="//fonts.googleapis.com">

<!-- DNS预解析 + 预连接 -->
<link rel="preconnect" href="https://api.example.com">
<!-- preconnect = DNS + TCP + TLS，比dns-prefetch更激进 -->

<!-- JavaScript方式触发DNS预解析 -->
<script>
// 创建不可见的link元素
const link = document.createElement('link');
link.rel = 'dns-prefetch';
link.href = '//api.example.com';
document.head.appendChild(link);
</script>
DNS Prefetch vs Preconnect：
  • dns-prefetch：只预解析DNS，不建立连接。开销极小
  • preconnect：预解析DNS + 建立TCP连接 + TLS握手。开销较大但收益更高
  • 对于确定会使用的域名，使用 preconnect
  • 对于可能使用的域名，使用 dns-prefetch

三、TCP连接

3.1 三次握手（Three-Way Handshake）

Client Server │ │ │ ──── SYN (seq=x) ──────────────────→ │ 1. 客户端发送SYN │ │ │ ←── SYN+ACK (seq=y, ack=x+1) ───── │ 2. 服务器回复SYN+ACK │ │ │ ──── ACK (ack=y+1) ────────────────→ │ 3. 客户端发送ACK │ │ │ ←── 连接建立，可以传输数据 ───────────→ │
  • SYN：Synchronize Sequence Numbers，同步序列号
  • ACK：Acknowledgment，确认
  • RTT（Round Trip Time）：一次往返时间，三次握手需要1.5个RTT
  • TCP Fast Open（TFO）可以在握手的同时传输数据，减少1个RTT

3.2 四次挥手（Four-Way Termination）

Client Server │ │ │ ──── FIN ──────────────────────────→ │ 1. 客户端发送FIN │ │ │ ←── ACK ─────────────────────────── │ 2. 服务器确认 │ │ │ ←── FIN ─────────────────────────── │ 3. 服务器发送FIN │ │ │ ──── ACK ──────────────────────────→ │ 4. 客户端确认 │ │ │ 连接关闭 │

3.3 TCP连接复用

  • HTTP/1.1 Keep-Alive：一个TCP连接可以发送多个HTTP请求，避免反复建立连接
  • HTTP/1.1 管道化（Pipelining）：可以同时发送多个请求，但响应必须按顺序返回（队头阻塞）
  • HTTP/2 多路复用（Multiplexing）：一个TCP连接上可以并行传输多个请求/响应，互不阻塞
HTTP版本连接数传输方式队头阻塞
HTTP/1.0每请求一个连接串行有
HTTP/1.1 + Keep-Alive6连接/域名串行复用有
HTTP/1.1 + Pipelining6连接/域名管道化有（响应级别）
HTTP/21连接多路复用有（TCP级别）
HTTP/3 (QUIC)1连接多路复用无

四、TLS握手

4.1 TLS 1.2 握手（2-RTT）

Client Server │ │ │ ──── ClientHello ─────────────────────→ │ │ 支持的TLS版本、加密套件列表、 │ │ 客户端随机数(Client Random) │ │ │ │ ←── ServerHello ────────────────────── │ │ 选定的TLS版本、加密套件、 │ │ 服务器随机数(Server Random)、 │ │ 服务器证书、ServerKeyExchange │ │ │ │ ──── ClientKeyExchange ─────────────→ │ │ 客户端密钥交换参数 │ │ ──── ChangeCipherSpec ──────────────→ │ │ ──── Finished ───────────────────────→ │ │ │ │ ←── ChangeCipherSpec ──────────────── │ │ ←── Finished ──────────────────────── │ │ │ │ 加密通信开始 │

4.2 TLS 1.3 握手（1-RTT）

Client Server │ │ │ ──── ClientHello ─────────────────────→ │ │ 支持的TLS版本、加密套件、 │ │ 客户端密钥共享参数(Key Share)、 │ │ 早期数据(0-RTT, 可选) │ │ │ │ ←── ServerHello + EncryptedExtensions + │ │ Certificate + CertificateVerify + │ │ Finished ──────────────────────── │ │ │ │ 加密通信开始（仅1-RTT） │
  • TLS 1.3 减少了握手往返次数（2-RTT → 1-RTT）
  • TLS 1.3 支持 0-RTT 恢复（使用预共享密钥PSK），但有重放攻击风险
  • TLS 1.3 只支持前向安全（Forward Secrecy）的加密套件
  • TLS 1.3 移除了不安全的加密算法（RC4、DES、3DES、MD5、SHA-1等）

五、HTTP/2多路复用

5.1 HTTP/2核心特性

特性说明收益
多路复用一个TCP连接上并行传输多个流（Stream）消除HTTP级别的队头阻塞
头部压缩（HPACK）使用静态+动态表压缩HTTP头部减少传输数据量
服务器推送服务器可以主动推送资源减少请求往返
流优先级客户端可以指定流的依赖关系和权重优先加载关键资源
二进制分帧将HTTP消息分为二进制帧传输更高效的解析和传输

5.2 HTTP/2帧结构

HTTP/1.1: 文本协议，请求/响应以明文传输 HTTP/2: 二进制协议，分为帧（Frame）传输 帧结构： ┌──────────────────────────────────────┐ │ Length (24 bits) │ Type (8 bits) │ ├───────────────────┼──────────────────┤ │ Flags (8 bits) │ R (1 bit) │ ├──────────────────────────────────────┤ │ Stream Identifier (31 bits) │ ├──────────────────────────────────────┤ │ Frame Payload (variable length) │ └──────────────────────────────────────┘ 帧类型： - DATA：请求/响应体数据 - HEADERS：请求/响应头部 - PRIORITY：流优先级 - RST_STREAM：重置流 - SETTINGS：连接配置 - PUSH_PROMISE：服务器推送 - PING：连接保活 - GOAWAY：关闭连接

5.3 HTTP/2的局限性 - TCP层队头阻塞

虽然HTTP/2解决了HTTP层面的队头阻塞，但TCP层面的队头阻塞仍然存在：

HTTP/2多路复用： Stream 1: [帧1] [帧2] [帧3] Stream 2: [帧A] [帧B] [帧C] Stream 3: [帧X] [帧Y] [帧Z] → 交错传输，各自独立 但是！TCP要求数据按序到达： 如果帧2丢失 → TCP等待帧2重传 → 帧3、帧A、帧B、帧C...全部阻塞！ → 这就是TCP层的队头阻塞 HTTP/3 (QUIC) 的解决方案： 每个Stream独立的可靠传输 Stream 2的帧丢失不影响Stream 1和3

资源类型优先级说明

6.2 Fetch Priority API

<!-- 设置图片加载优先级 -->
<!-- 首屏关键图片：高优先级 -->
<img src="hero.webp" fetchpriority="high" alt="Hero">

<!-- 非关键图片：低优先级 -->
<img src="below-fold.webp" fetchpriority="low" loading="lazy" alt="...">

<!-- CSS预加载：高优先级 -->
<link rel="preload" href="critical.css" as="style" fetchpriority="high">

<!-- 非关键脚本：低优先级 -->
<link rel="preload" href="analytics.js" as="script" fetchpriority="low">

<!-- JavaScript中使用 -->
<script>
fetch('/api/data', { priority: 'high' });
fetch('/api/analytics', { priority: 'low' });

// 也可以通过 import() 控制模块加载优先级
import(/* webpackFetchPriority: "high */ './critical.js');
</script>

七、HTTP/3与QUIC

7.1 HTTP/3核心改进

  • 基于UDP：使用QUIC协议代替TCP，解决TCP层队头阻塞
  • 0-RTT连接建立：QUIC内建TLS 1.3，首次连接1-RTT，恢复连接0-RTT
  • 连接迁移：使用Connection ID代替IP+Port，网络切换（如WiFi到4G）时不断连
  • 独立的流：每个流独立可靠传输，一个流的丢包不影响其他流
HTTP/2 over TCP: ┌─────────────────────────────┐ │ HTTP/2 Streams (多路复用) │ ├─────────────────────────────┤ │ TLS 1.2/1.3 (加密) │ ├─────────────────────────────┤ │ TCP (可靠传输) │ ← 队头阻塞在这里发生 ├─────────────────────────────┤ │ IP │ └─────────────────────────────┘ HTTP/3 over QUIC: ┌─────────────────────────────┐ │ HTTP/3 Streams │ ├─────────────────────────────┤ │ QUIC (可靠传输 + 加密 + 流控制)│ ← 每个流独立可靠 ├─────────────────────────────┤ │ UDP (不可靠传输) │ ├─────────────────────────────┤ │ IP │ └─────────────────────────────┘

八、常见面试题

Q1: 从输入URL到页面展示发生了什么？

  1. URL解析 → 判断是搜索还是URL
  2. DNS解析 → 获取IP地址（浏览器缓存 → OS缓存 → DNS服务器递归查询）
  3. TCP三次握手 → 建立连接（1.5个RTT）
  4. TLS握手 → 建立加密通道（1-2个RTT，TLS 1.3可0-1RTT）
  5. 发送HTTP请求 → 服务器处理并返回响应
  6. 浏览器解析HTML → 构建DOM树
  7. 解析CSS → 构建CSSOM树
  8. 执行JavaScript → 可能修改DOM/CSSOM
  9. 构建渲染树 → 布局 → 绘制 → 合成
  10. TCP四次挥手 → 关闭连接

Q2: HTTP/1.1的队头阻塞问题是什么？

HTTP/1.1 虽然支持Keep-Alive复用TCP连接，但在一个连接上请求必须串行处理——上一个请求的响应完整返回后，才能处理下一个请求。浏览器通过开启多个TCP连接（每个域名6个）来缓解此问题，但连接数仍然有限。

HTTP/2通过多路复用解决了HTTP层的队头阻塞，但TCP层的队头阻塞仍然存在。HTTP/3使用QUIC协议彻底解决了此问题。

总结：浏览器网络栈的优化目标是最小化请求延迟。关键技术包括：DNS预解析减少DNS查询时间、预连接减少TCP+TLS握手时间、HTTP/2多路复用减少连接数和队头阻塞、Fetch Priority API优化资源加载顺序、Preload/Prefetch提前下载关键资源。
