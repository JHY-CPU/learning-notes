# 状态机图（State Machine Diagram）

## 一、概述

状态机图（State Machine Diagram，又称状态图 State Diagram）描述了一个对象在其**生命周期**内经历的**状态序列**，以及引起状态转换的**事件**和**条件**。

状态机图回答的核心问题：
- 对象有哪些**状态**？
- 什么**事件**导致状态转换？
- 转换时发生什么**动作**？
- 对象的**生命周期**是怎样的？

> 适用场景：具有复杂生命周期的对象，如订单、用户会话、设备控制器等

## 二、基本元素

### 2.1 状态（State）

**定义**：对象在某一时刻所处的状况。

**表示**：圆角矩形

```
┌──────────────────┐
│     StateName    │  ← 状态名
└──────────────────┘
```

**完整状态的内部结构**：

```
┌──────────────────────────────┐
│        StateName             │  ← 状态名
├──────────────────────────────┤
│ entry / 动作                  │  ← 进入动作
│ do / 活动                    │  ← 内部活动
│ exit / 动作                  │  ← 退出动作
├──────────────────────────────┤
│ 事件 / 动作                   │  ← 内部转换
│ 事件 [条件] / 动作            │  ← 带条件的内部转换
└──────────────────────────────┘
```

**状态内部的动作类型**：

| 动作类型 | 格式 | 说明 | 触发时机 |
|---------|------|------|---------|
| **entry**（进入） | `entry / action` | 进入状态时执行 | 进入状态时自动执行 |
| **exit**（退出） | `exit / action` | 退出状态时执行 | 退出状态时自动执行 |
| **do**（活动） | `do / activity` | 状态内的持续活动 | 状态保持期间执行 |
| **内部事件** | `event / action` | 不引起状态变化的事件 | 事件触发时执行 |

**示例**：
```
┌──────────────────────────────┐
│         Processing           │
├──────────────────────────────┤
│ entry / startTimer           │
│ do / monitorProgress         │
│ exit / stopTimer             │
│ timeout / notifyUser         │
└──────────────────────────────┘
```

### 2.2 特殊状态

#### 初始状态（Initial State）

**表示**：实心圆 ●

```
  ● ───→ [状态A]
```
- 每个状态机图**有且只有一个**初始状态
- 表示对象生命周期的**起点**

#### 终止状态（Final State）

**表示**：牛眼 ⊙（同心圆）

```
  [状态Z] ───→ ⊙
```
- 可以有**多个**终止状态
- 表示对象生命周期的**终点**

#### 历史状态（History State）

**表示**：带圈的 H

| 类型 | 符号 | 说明 |
|------|------|------|
| **浅历史** | H | 记住上次离开复合状态时的直接子状态 |
| **深历史** | H* | 记住上次离开复合状态时的深层子状态 |

```
        ┌──→ [子状态A]
   → H ─┤
        └──→ [子状态B]
```

作用：返回复合状态时，恢复到上次离开时的状态，而不是从头开始。

### 2.3 转换（Transition）

**定义**：从一个状态到另一个状态的改变。

**表示**：带箭头的实线

```
  [状态A] ──────→ [状态B]
```

**转换的完整格式**：

```
触发事件 [监护条件] / 动作列表 {发送目标}
```

| 组成部分 | 说明 | 示例 |
|---------|------|------|
| **触发事件** | 引起转换的事件 | `click`、`timeout`、`receive(msg)` |
| **监护条件** | 转换必须满足的布尔条件 | `[x > 0]`、`[approved]` |
| **动作** | 转换过程中执行的操作 | `/ log()`、`/ sendEmail()` |
| **发送目标** | 向其他对象发送的消息 | `{send ready}` |

**示例**：

```
         submit [valid] / save, notify
[Draft] ──────────────────────────────→ [Submitted]
   │
   │    cancel / deleteDraft
   └──────────────────────────────────→ ⊙
```

解读：当对象处于 Draft 状态时，发生 submit 事件且条件 [valid] 为真，执行 save 和 notify 动作，然后转换到 Submitted 状态。

### 2.4 转换的类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **外部转换** | 引起状态变化的转换 | A → B |
| **内部转换** | 不引起状态变化，在状态内部处理 | `timeout / log` |
| **自转换** | 从状态到自身的转换 | A → A（会触发 entry/exit） |
| **完成转换** | 无触发事件，状态内活动完成后自动转换 | 自动流转 |

**内部转换 vs 自转换**：
```
内部转换（不触发 entry/exit）：
┌──────────────┐
│   Active     │
│ timeout / log│  ← 保持在 Active，不触发 exit/entry
└──────────────┘

自转换（触发 entry/exit）：
  [Active] ──timeout──→ [Active]
  （会触发 Active 的 exit 动作，再触发 entry 动作）
```

### 2.5 监护条件（Guard Condition）

用方括号表示的布尔表达式，只有条件为真时转换才发生：

```
         [x > 0]
[S1] ─────────────→ [S2]

         [x <= 0]
[S1] ─────────────→ [S3]
```

- 多个转换可以有相同的触发事件，但监护条件必须**互斥**
- 监护条件在事件发生时求值

## 三、复合状态（Composite State）

### 3.1 概述

复合状态是包含**子状态**的状态，用于组织和简化复杂的状态图。

```
┌──────────────────────────────────────┐
│           Active                     │  ← 复合状态
│                                      │
│   ┌──────┐      ┌──────┐            │
│   │SubA  │─────→│SubB  │            │
│   └──────┘      └──────┘            │
│                                      │
└──────────────────────────────────────┘
```

### 3.2 顺序复合状态

子状态之间有**明确的转换关系**。

```
┌──────────────────────────────────────────┐
│              Active                      │
│                                          │
│   ●──→ [Running] ──pause──→ [Paused]     │
│              ↑                  │        │
│              └──resume──────────┘        │
│                                          │
└──────────────────────────────────────────┘
```

### 3.3 并发复合状态（正交状态）

子状态**同时存在**，用虚线分隔。

```
┌──────────────────────────────────────────┐
│              Active                      │
│   ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │
│   Region 1:                              │
│   ●──→ [Running] ──pause──→ [Paused]    │
│                                          │
│   ════════════════════════════════════   │
│                                          │
│   Region 2:                              │
│   ●──→ [Connected] ──disconnect→         │
│          [Disconnected]                  │
│                                          │
└──────────────────────────────────────────┘
```

- 正交状态表示对象**同时处于多个子状态机中**
- 每个区域（Region）是独立的子状态机
- 同一时刻，对象在每个区域中处于一个子状态

### 3.4 子状态机引用

```
┌────────────────────────┐
│     Active             │
│                        │
│   ┌─── ref ──────┐    │
│   │  详细状态图   │    │  ← 引用另一个状态机
│   │  （附录A）    │    │
│   └──────────────┘    │
└────────────────────────┘
```

## 五、事件的类型

| 事件类型 | 说明 | 示例 |
|---------|------|------|
| **信号事件** | 外部发送的信号 | `receive(request)` |
| **调用事件** | 方法被调用 | `submit()` |
| **时间事件** | 时间条件满足 | `after(5s)`、`when(12:00)` |
| **改变事件** | 某个条件变为真 | `when(temperature > 100)` |
| **自身事件** | 对象自身发送的事件 | `self(timeout)` |

### 时间事件的表示

```
相对时间：
[StateA] ──── after(5s) ────→ [StateB]

绝对时间：
[StateA] ──── when(2024-12-31) ────→ [StateB]
```

## 六、状态图 vs 活动图

| 特性 | 状态图 | 活动图 |
|------|--------|--------|
| **关注点** | 对象的状态变化 | 操作的执行流程 |
| **核心概念** | 状态、事件、转换 | 动作、控制流 |
| **视角** | 单个对象 | 整个过程/系统 |
| **适用场景** | 生命周期建模 | 业务流程、算法流程 |
| **触发方式** | 事件驱动 | 完成驱动 |

## 七、考试重点

1. **基本概念**：状态、转换、事件、动作
2. **状态内部结构**：entry、exit、do 动作的含义
3. **转换格式**：事件 [条件] / 动作
4. **特殊状态**：初始状态、终止状态、历史状态
5. **复合状态**：顺序复合、正交（并发）复合
6. **监护条件**：条件对转换的约束
7. **常见场景**：能根据描述绘制状态图
