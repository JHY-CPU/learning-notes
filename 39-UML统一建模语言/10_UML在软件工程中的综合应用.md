# UML 在软件工程中的综合应用

## 一、UML 建模的完整流程

### 1.1 软件开发中的 UML 应用阶段

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  需求阶段          分析阶段          设计阶段         实现阶段     │
│  ─────────       ─────────        ─────────       ─────────    │
│                                                                 │
│  用例图 ──────→  分析类图 ──────→  设计类图 ──────→ 组件图       │
│      │              │                │               │         │
│      │           活动图            时序图           部署图        │
│      │              │                │               │         │
│  用例规约         状态图            状态图          包图          │
│                                                                 │
│  ↑              ↑                 ↑              ↑              │
│ 用户需求        领域模型           架构设计        物理实现        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 各阶段 UML 图的使用

| 阶段 | 主要 UML 图 | 目的 |
|------|------------|------|
| **需求分析** | 用例图、活动图 | 理解系统做什么 |
| **领域分析** | 领域类图、对象图 | 理解业务领域 |
| **系统设计** | 设计类图、时序图、状态图 | 设计系统结构和行为 |
| **详细设计** | 详细类图、时序图 | 指导编码实现 |
| **架构设计** | 组件图、部署图、包图 | 设计系统架构 |
| **测试** | 用例图（测试用例参考） | 验证系统正确性 |

#### 用例规约示例

```
用例名称：借阅图书
用例编号：UC-003
参与者：读者（Reader）
前置条件：
  1. 读者已登录系统
  2. 读者账户状态正常（无逾期未还）
  3. 读者当前借阅数未超过上限
后置条件：
  1. 图书状态变为"已借出"
  2. 借阅记录创建成功
  3. 应还日期已计算并保存

主成功场景：
  1. 读者搜索并选择要借阅的图书
  2. 系统显示图书详细信息和可借状态
  3. 读者确认借阅
  4. 系统检查图书是否可借
  5. 系统检查读者借阅资格
  6. 系统计算应还日期（借阅日期 + 30天）
  7. 系统创建借阅记录
  8. 系统更新图书状态为"已借出"
  9. 系统显示借阅成功信息

扩展流程：
  4a. 图书不可借（已被借出或预约）：
      4a1. 系统提示图书不可借
      4a2. 系统提供预约选项
      4a3. 读者可选择预约或取消
  5a. 读者有逾期未还图书：
      5a1. 系统提示存在逾期图书
      5a2. 系统显示逾期图书列表
      5a3. 必须先归还逾期图书才能继续借阅
  5b. 读者已达到借阅上限：
      5b1. 系统提示已达借阅上限
      5b2. 系统建议先归还部分图书

特殊需求：
  - 借阅期限默认30天
  - 每位读者最多同时借阅5本
  - 续借最多2次，每次续借延长15天
```

## 三、UML 图之间的关系和转换

### 3.1 图之间的关联

```
用例图 ──→ 活动图      (每个用例可以用活动图描述流程)
   │
   ↓
时序图               (用例的实现可以用时序图描述交互)
   │
   ↓
类图                 (时序图中的对象对应类图中的类)
   │
   ↓
状态图               (类图中状态复杂的类用状态图描述)
   │
   ↓
组件图               (类组织成组件)
   │
   ↓
部署图               (组件部署到物理节点)
```

### 3.2 常见的图转换

| 起点 | 终点 | 转换规则 |
|------|------|---------|
| 用例图 → 活动图 | 每个用例可以展开为一个活动图 |
| 用例图 → 时序图 | 每个用例场景对应一个时序图 |
| 时序图 → 类图 | 时序图中的对象 → 类；消息 → 方法 |
| 类图 → 对象图 | 类图是模板，对象图是实例 |
| 类图 → 状态图 | 具有复杂状态的类需要状态图 |
| 类图 → 组件图 | 类组织成组件，接口对应接口 |
| 组件图 → 部署图 | 组件部署到节点上 |

## 四、UML 建模最佳实践

### 4.1 选择合适的图

```
你想表达什么？          →  使用什么图
──────────────────────────────────────
系统有哪些功能？        →  用例图
业务流程是什么？        →  活动图
对象结构是什么？        →  类图
对象之间如何交互？      →  时序图
对象的生命周期？        →  状态图
系统有哪些模块？        →  组件图
系统如何部署？          →  部署图
代码如何组织？          →  包图
某一时刻的数据？        →  对象图
```

### 4.2 建模的粒度

| 阶段 | 粒度 | 特点 |
|------|------|------|
| 概念建模 | 粗粒度 | 只有类名，没有属性和方法 |
| 分析建模 | 中粒度 | 有关键属性，方法不完整 |
| 设计建模 | 细粒度 | 完整的属性、方法、类型 |

### 4.3 常见误区

1. **过度建模**：画所有可能的图，实际上 3-5 个图就足够
2. **过细粒度**：把每个 getter/setter 都画出来
3. **忽视更新**：模型和代码不同步
4. **追求完美**：UML 是沟通工具，不是艺术作品
5. **遗漏约束**：只画结构，不画业务规则

### 4.4 推荐的建模顺序

```
1. 用例图      → 了解系统边界和功能
2. 活动图      → 描述核心业务流程
3. 领域类图    → 理解业务概念
4. 时序图      → 设计核心用例的实现
5. 设计类图    → 完善类的结构
6. 状态图      → 描述复杂对象的状态
7. 组件图      → 设计系统架构
8. 部署图      → 规划物理部署
```
