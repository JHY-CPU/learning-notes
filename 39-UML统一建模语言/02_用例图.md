# 用例图（Use Case Diagram）

## 一、概述

用例图是 UML 中**最接近用户视角**的图，用于描述系统的**功能需求**——即系统"做什么"，而不是"怎么做"。

用例图回答的核心问题：
- 谁使用系统？（参与者）
- 他们使用系统做什么？（用例）
- 系统的边界在哪里？（系统边界框）

## 二、基本元素

### 2.1 参与者（Actor）

**定义**：与系统交互的外部实体（人、其他系统、硬件设备）。

**表示**：简笔画小人符号

```
        O
       /|\
       / \
```

**命名规范**：
- 使用名词或名词短语
- 通常是单数：`Customer`、`Administrator`、`Bank System`
- 位于系统外部

**分类**：
| 类型 | 说明 | 示例 |
|------|------|------|
| 主要参与者 | 主动发起用例的参与者 | 顾客、管理员 |
| 次要参与者 | 被动响应系统请求的参与者 | 支付网关、邮件服务器 |
| 主要参与者 | 直接使用系统的用户 | 前台操作员 |
| 次要参与者 | 为系统提供服务的外部系统 | 第三方认证服务 |

**参与者的继承**：
```
        User（抽象参与者）
       /        \
  Admin      Customer
```
- 子参与者继承父参与者的所有用例
- 子参与者可以有自己额外的用例

### 2.2 用例（Use Case）

**定义**：系统提供的一个功能单元，描述参与者与系统之间的一组交互序列。

**表示**：椭圆形，内部写用例名

**命名规范**：
- 使用**动词 + 宾语**的格式（动宾结构）
- 例如：`登录系统`、`查询订单`、`生成报表`、`Manage Account`
- 表达用户的目标，而非具体操作步骤

**用例 vs 功能**：
| 用例 | 功能 |
|------|------|
| 面向用户目标 | 面向系统功能分解 |
| 包含交互过程 | 只是一个操作 |
| 跨越多个功能模块 | 通常是单一模块 |
| `下订单` | `验证库存`、`计算价格`、`保存订单` |

### 2.3 系统边界（System Boundary）

**定义**：界定系统范围的矩形框。

**表示**：矩形框，系统名称写在顶部或左上角

```
┌─────────────────────────────────┐
│        在线购物系统              │
│                                 │
│  (浏览商品)  (加入购物车)        │
│                                 │
│  (下单)      (支付)             │
│                                 │
└─────────────────────────────────┘
```

> 系统边界外是参与者，边界内是用例

### 2.4 关系

#### （1）关联关系（Association）

**符号**：实线，连接参与者和用例

**含义**：参与者参与了某个用例

```
Customer ———— (下订单)
```

**方向性**：
- 无箭头表示双向通信（默认）
- 有箭头表示信息/控制流的方向

#### （2）包含关系（Include）

**符号**：虚线箭头 + `<<include>>`

**含义**：**基用例**总是会执行**被包含用例**的行为。被包含用例是基用例的必要组成部分。

```
(下订单) ---<<include>>---> (验证身份)
```

**特点**：
- 必然发生，无条件执行
- 用于提取公共行为，避免重复
- 箭头方向：从**基用例**指向**被包含用例**
- 被包含用例通常不单独使用

**典型场景**：
```
(取款) ---<<include>>---> (验证密码)
(转账) ---<<include>>---> (验证密码)
(查询余额) ---<<include>>---> (验证密码)
```

#### （3）扩展关系（Extend）

**符号**：虚线箭头 + `<<extend>>`

**含义**：**扩展用例**在满足特定条件时，可以**扩展**基用例的行为。扩展是可选的。

```
(下订单) <---<<extend>>--- (使用优惠券)
```

**特点**：
- 有条件地发生，不是每次都会执行
- 用于描述可选行为
- 箭头方向：从**扩展用例**指向**基用例**
- 可以指定**扩展点**（Extension Point）

**扩展点的表示**：
```
(下订单)
 ─────────────
 扩展点：选择支付方式
 ─────────────
```

**典型场景**：
```
(结账) <---<<extend>>--- (使用积分抵扣)
(登录) <---<<extend>>--- (忘记密码重置)
```

#### （4）泛化关系（Generalization）

**符号**：实线 + 空心三角箭头

**两种泛化**：

**a) 用例之间的泛化**：
```
(支付) ◁—— (在线支付)
             (货到付款)
```
- 子用例继承父用例的所有行为
- 子用例可以添加或覆盖父用例的行为

**b) 参与者之间的泛化**：
```
   User ◁—— Admin
          ◁—— Guest
```
- 子参与者继承父参与者关联的所有用例

## 四、用例描述（用例规约）

用例图只是用例的**可视化摘要**，每个用例还需要详细的**用例规约**（Use Case Specification）来描述具体的交互流程。

### 4.1 用例规约模板

```
用例名称：下订单
用例编号：UC-001
参与者：顾客
前置条件：用户已登录，购物车中有商品
后置条件：订单创建成功，库存已更新

主成功场景（基本流程）：
  1. 用户点击"去结算"
  2. 系统显示收货地址选择页面
  3. 用户选择/输入收货地址
  4. 系统显示商品清单和总价
  5. 用户选择支付方式
  6. 用户确认下单
  7. 系统验证库存
  8. 系统创建订单
  9. 系统跳转到支付页面
  10. 用户完成支付

扩展流程（替代流程）：
  7a. 库存不足：
      7a1. 系统提示"库存不足"
      7a2. 用户可以选择减少数量或移除商品
      7a3. 回到步骤 1
  10a. 支付失败：
      10a1. 系统提示支付失败原因
      10a2. 用户可重新支付或取消订单

特殊需求：
  - 支持支付宝、微信、银行卡支付
  - 订单金额需保留两位小数
```

### 4.2 基本流程 vs 扩展流程

| | 基本流程 | 扩展流程 |
|---|---------|---------|
| 定义 | 一切顺利时的标准流程 | 出现异常或特殊情况的处理 |
| 条件 | 无错误、无异常 | 错误、异常、可选操作 |
| 数量 | 只有一条 | 可以有多条 |
| 别名 | 主场景、快乐路径 | 备选场景、异常场景 |

## 五、Include vs Extend 详解

### 5.1 核心区别

| 特性 | Include | Extend |
|------|---------|--------|
| **发生性** | 必然发生 | 有条件发生 |
| **方向** | 基用例 → 被包含用例 | 扩展用例 → 基用例 |
| **目的** | 提取公共行为 | 添加可选行为 |
| **驱动方** | 基用例主动调用 | 扩展用例主动插入 |
| **类比** | 函数调用 | if 语句块 |
| **典型场景** | 验证身份、日志记录 | 使用优惠券、异常处理 |

### 5.2 代码类比

```
Include 类比：
void placeOrder() {
    verifyIdentity();  // 必然执行
    // ...
}

Extend 类比：
void placeOrder() {
    // ...
    if (userHasCoupon) {
        applyCoupon();  // 有条件执行
    }
    // ...
}
```

### 5.3 判断技巧

问自己：
- **这个行为每次都发生吗？**
  - 是 → Include
  - 否 → Extend

- **是谁发起的？**
  - 基用例发起 → Include
  - 扩展条件发起 → Extend

## 六、常见错误

### 6.1 用例粒度不当

| 错误类型 | 错误示例 | 正确做法 |
|---------|---------|---------|
| 太细 | `点击登录按钮`、`输入密码` | 合并为 `登录系统` |
| 太粗 | `使用系统` | 拆分为 `登录`、`查询`、`下单` 等 |
| 包含实现细节 | `调用数据库查询用户` | `验证用户身份` |

### 6.2 关系使用错误

- 把必然行为写成 Extend：如 `下订单` ——`<<extend>>`——> `验证库存`
- 把可选行为写成 Include：如 `登录` ——`<<include>>`——> `记住密码`

### 6.3 参与者位置错误

- 参与者不应在系统边界内部
- 用例不应在系统边界外部

## 七、考试重点

1. **Include 和 Extend 的区分**：必考，注意箭头方向
2. **用例规约**：知道基本格式和内容
3. **识别参与者**：能从需求描述中提取出所有参与者
4. **识别用例**：能从需求描述中提取出用例并建立关系
5. **用例图的绘制**：按照规范绘制完整的用例图
