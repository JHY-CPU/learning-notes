# 活动图（Activity Diagram）

## 一、概述

活动图描述系统的**业务流程**或**操作执行流程**，强调**动作的执行顺序**和**控制流**。

活动图类似于流程图，但增加了**并发**、**泳道**、**对象流**等高级特性。

活动图回答的核心问题：
- 业务流程的执行顺序是什么？
- 哪些操作可以**并行**执行？
- 不同的**角色/部门**负责哪些操作？
- 操作过程中涉及哪些**数据**？

## 二、基本元素

### 2.1 动作节点（Action Node）

**动作（Action）**：原子级别的执行单元，不可进一步分解。

```
┌────────────────────┐
│   Process Order    │  ← 圆角矩形
└────────────────────┘
```

**活动（Activity）**：可以分解为多个子动作的复合操作。

### 2.2 控制节点（Control Node）

| 节点 | 符号 | 说明 |
|------|------|------|
| **初始节点** | 实心圆 ● | 活动的起点 |
| **活动终止节点** | 牛眼 ⊙ | 整个活动的终点 |
| **流终止节点** | X 叉号 | 当前控制流的终止 |
| **决策/合并节点** | 菱形 ◇ | 条件分支/合并 |
| **分叉/汇合节点** | 粗横线 ━ | 并行控制 |

#### 决策节点（Decision）

```
                ┌──── [条件A] ───→ [动作X]
                │
   [动作] ──→  ◇
                │
                └──── [条件B] ───→ [动作Y]
```

- 一个入边，多个出边
- 出边上标注**监护条件**（方括号内）
- 条件必须**互斥且完备**（至少一个为真）

#### 合并节点（Merge）

```
   [动作X] ──→
                ╲
                 ◇ ──→ [动作Z]
                ╱
   [动作Y] ──→
```

- 多个入边，一个出边
- 将多个分支合并为一个控制流

#### 分叉节点（Fork）

```
   [动作] ──→  ━━━━━━━ ──→ [动作A]
                │
                └────────→ [动作B]
                │
                └────────→ [动作C]
```

- 一个入边，多个出边
- 将控制流**分成多个并行**的流

#### 汇合节点（Join）

```
   [动作A] ──→  ╲
                  ━━━━━━━ ──→ [动作]
   [动作B] ──→  ╱
                  │
   [动作C] ──→  ╱
```

- 多个入边，一个出边
- 等待**所有**并行流到达后才继续
- **汇合后继续并行**：汇合后可以立即再分叉

### 2.3 边（Edge）

#### 控制流

```
[动作A] ──────────────→ [动作B]
```

**箭头表示执行方向**

#### 监护条件

```
[动作A] ──── [condition] ────→ [动作B]
```

#### 对象流（Object Flow）

活动图中可以显示**数据的传递**：

```
          ┌───────────┐
          │  :Order   │  ← 对象节点
          │ status=OK │
          └─────┬─────┘
                │  （对象流）
                ▼
┌────────────────────┐
│   Update Inventory │
└────────────────────┘
```

### 2.4 泳道（Swimlane）

泳道将活动图按**角色**、**部门**或**子系统**划分，清晰地展示谁负责什么。

#### 水平泳道

```
┌──────────────┬──────────────┬──────────────┐
│   客户        │   销售部      │   仓库        │
│              │              │              │
│  ●           │              │              │
│  │           │              │              │
│  ▼           │              │              │
│ 浏览商品      │              │              │
│  │           │              │              │
│  ▼           │              │              │
│ 下单 ─────────→ 处理订单      │              │
│              │    │         │              │
│              │    ▼         │              │
│              │  审核 ────────→ 检查库存      │
│              │              │    │         │
│              │              │    ▼         │
│              │              │  出库         │
│              │    │         │    │         │
│              │    ▼         │    │         │
│              │  发货通知 ←───┘    │         │
│  │           │              │              │
│  ▼           │              │              │
│ 确认收货      │              │              │
│  │           │              │              │
│  ▼           │              │              │
│ ⊙           │              │              │
└──────────────┴──────────────┴──────────────┘
```

#### 垂直泳道

```
│ 客户 │  →  │ 销售部 │  →  │ 仓库  │
│      │     │        │     │       │
│  ●   │     │        │     │       │
│  │   │     │        │     │       │
│  ▼   │     │        │     │       │
│ 下单 ─┼────→│ 处理   │     │       │
│      │     │   │    │     │       │
│      │     │   ▼    │     │       │
│      │     │ 审核 ──┼────→│ 出库  │
│      │     │        │     │       │
```

### 2.5 引用活动（Activity Invocation）

```
┌──────────────────┐
│ ┌─── ref ──────┐ │
│ │  验证用户身份  │ │  ← 引用另一个活动
│ │  （UC-001）   │ │
│ └──────────────┘ │
└──────────────────┘
```

## 四、高级特性

### 4.1 信号发送与接收

#### 发送信号

```
┌────────────────────┐
│   Send Order       │  ← 信号发送（凸五边形/信号图标）
│   Confirmation     │
└────────────────────┘
```

#### 接收信号

```
┌────────────────────┐
│   Receive Payment  │  ← 信号接收（凹五边形/信号图标）
│   Notification     │
└────────────────────┘
```

#### 时间事件

```
┌────────────────────┐
│   when(12:00)      │  ← 时间事件触发
│   发送日报          │
└────────────────────┘
```

### 4.2 异常处理

```
┌──────────────────────────────────────┐
│                                      │
│  ┌──────────────┐                    │
│  │ 处理文件读取  │                    │
│  └──────┬───────┘                    │
│         │                            │
│   ⚡ 异常处理区域                     │
│         │                            │
│    ⚡────┼─── IOException ──→ 处理错误│
│         │                            │
│         ▼                            │
│  ┌──────────────┐                    │
│  │ 解析文件内容  │                    │
│  └──────────────┘                    │
│                                      │
└──────────────────────────────────────┘
```

### 4.3 可中断区域（Interruptible Activity Region）

用虚线矩形围起来的区域，可以被中断事件**打断**：

```
┌─── 可中断区域 ───────────────────┐
│                                  │
│  [动作A] ──→ [动作B] ──→ [动作C] │
│                                  │
│  中断事件 ──────────────────→ ⊙  │
│                                  │
└──────────────────────────────────┘
```

如果中断事件发生，区域内的所有活动停止，直接跳到终止。

### 4.4 针（Pin）

针是动作节点上的**小矩形**，表示动作的输入和输出：

```
        输入针    输出针
          │        │
          ▼        ▼
       ┌──────────────┐
       │   计算总价    │
       └──────────────┘
```

```
  ┌─────────┐    ┌──────────────────┐    ┌─────────┐
  │ order   │───→│   计算总价        │───→│ total   │
  └─────────┘    └──────────────────┘    └─────────┘
```

## 五、活动图 vs 其他图

| 图 | 关注点 | 典型场景 |
|----|--------|---------|
| **活动图** | 动作和控制流 | 业务流程、算法流程 |
| **状态图** | 对象的状态变化 | 对象生命周期 |
| **时序图** | 对象间的消息交互 | API 调用流程 |
| **用例图** | 系统功能需求 | 需求分析 |

### 活动图 vs 流程图

| 特性 | 流程图 | 活动图 |
|------|--------|--------|
| 并发 | 不支持 | 支持分叉/汇合 |
| 泳道 | 不支持 | 支持角色划分 |
| 对象流 | 不支持 | 支持数据流 |
| 异常处理 | 不支持 | 支持异常区域 |
| 面向 | 过程 | 面向对象 |
| 标准 | 无统一标准 | UML 标准 |

## 六、考试重点

1. **基本元素**：动作、决策/合并、分叉/汇合、初始/终止
2. **泳道**：用泳道划分角色和职责
3. **并发**：分叉和汇合的画法和含义
4. **对象流**：数据在活动之间的传递
5. **与流程图的区别**：活动图支持并发和泳道
6. **与状态图的区别**：活动图关注操作流程，状态图关注对象状态
