# 对象图与包图

## 一、对象图（Object Diagram）

### 1.1 概述

对象图是类图的**实例化版本**，描述系统在**某一特定时刻**的对象及其关系的快照。

- **类图**：静态蓝图（描述所有可能的状态）
- **对象图**：具体快照（描述某一时刻的实际状态）

> 类比：类图是建筑图纸，对象图是建筑在某个时刻的照片

### 1.2 对象的表示

对象用矩形表示，与类图类似但有区别：

```
┌────────────────────────────┐
│  对象名: 类名               │  ← 对象名有下划线
├────────────────────────────┤
│ 属性名 = 具体值             │  ← 必须是具体值，不能是类型
│                            │
├────────────────────────────┤
│                            │  ← 通常省略方法（因为和类一样）
└────────────────────────────┘
```

**三种表示形式**：

```
形式1（完整）：                形式2（匿名）：           形式3（省略类名）：
┌──────────────────┐     ┌──────────────────┐    ┌──────────────────┐
│ zhangSan: Student│     │ : Student        │    │ zhangSan         │
├──────────────────┤     ├──────────────────┤    ├──────────────────┤
│ name = "张三"    │     │ name = "张三"    │    │ name = "张三"    │
│ age = 20         │     │ age = 20         │    │ age = 20         │
│ gpa = 3.8        │     │ gpa = 3.8        │    │ gpa = 3.8        │
└──────────────────┘     └──────────────────┘    └──────────────────┘
```

- 对象名和类名都要带**下划线**
- 类名前加冒号 `:`
- 匿名对象只写 `:类名`
- 可以只写对象名不写类名

**场景**：2024年春季学期选课

│ s1: Student      │          │ c1: Course       │

│ id = "2024001"   │          │ code = "CS301"   │

        │        │ e1: Enrollment     ││

                 │ semester = "2024S" │
                 └────────────────────┘

┌──────────────────┐          ┌──────────────────┐

### 1.4 对象图与类图的对比

| 特性 | 类图 | 对象图 |
|------|------|--------|
| 描述 | 类的定义（模板） | 对象的实例（具体值） |
| 时间性 | 无时间概念 | 某一时刻的快照 |
| 对象名 | 无 | 有下划线 |
| 属性 | 类型 + 可选默认值 | 具体值 |
| 多重性 | 明确标注 | 实际数量 |
| 用途 | 描述系统结构 | 验证类图的正确性 |

### 1.5 对象图的用途

1. **验证类图设计**：通过实例化来验证类图的完整性和正确性
2. **展示具体场景**：用真实数据说明系统的运行状态
3. **调试和分析**：帮助理解复杂的对象关系
4. **数据建模**：展示数据库中的数据快照

### 1.6 对象图的局限

- 只能展示**一个时间点**的状态
- 无法展示系统的**动态行为**
- 对于复杂系统，对象图可能变得非常庞大
- 通常用于**小范围**的场景展示

## 二、包图（Package Diagram）

### 2.1 概述

包图用于组织和管理模型元素，将相关的类、用例、组件等分组到**包**（Package）中。

包图的作用：
- **组织模型**：将大型系统分解为可管理的部分
- **控制可见性**：管理包之间和包内的访问权限
- **管理依赖**：展示包之间的依赖关系

> 类比：包就像文件系统中的文件夹，用于组织和分类

### 2.2 包的表示

```
┌─────────────────────────┐
│ « subsystem »           │  ← 构造型（可选）
│ ┌───┐                   │
│ │ A │  PackageName      │  ← 标签（包名）
│ └───┘                   │
│                         │
│  (包含的元素)            │  ← 可选展开内容
│  ClassName1             │
│  ClassName2             │
│                         │
└─────────────────────────┘
```

**两种表示形式**：

```
形式1（简化，不显示内容）：     形式2（展开，显示内容）：
┌──────────────────┐      ┌──────────────────┐
│ ┌──┐             │      │ Domain           │
│ │  │ Domain      │      ├──────────────────┤
│ └──┘             │      │ Student          │
│                  │      │ Course           │
└──────────────────┘      │ Teacher          │
                          └──────────────────┘
```

### 2.3 包内元素的可见性

| 标记 | 含义 | 说明 |
|------|------|------|
| `+` | 公有（Public） | 对所有包可见 |
| `-` | 私有（Private） | 仅本包内可见 |
| `~` | 包内（Package） | 仅本包和嵌套包可见 |

```
┌────────────────────────┐
│ package Domain         │
├────────────────────────┤
│ + Student              │  ← 其他包可访问
│ - InternalHelper       │  ← 仅本包可见
│ ~ Repository           │  ← 包内可见
└────────────────────────┘
```

### 2.4 包之间的关系

#### （1）依赖关系

**符号**：虚线箭头 `---->`

**含义**：一个包使用了另一个包中的元素

```
┌──────────┐          ┌──────────┐
│   UI     │ -------> │  Domain  │
└──────────┘  依赖     └──────────┘
```

**依赖的类型**：

| 构造型 | 含义 | 说明 |
|--------|------|------|
| `«import»` | 导入 | 被导入包的公有元素可在导入包中直接使用 |
| `«access»` | 访问 | 可以访问但必须使用限定名 |
| `«merge»` | 合并 | 将一个包的内容合并到另一个包 |
| `«trace»` | 追踪 | 跨模型的依赖关系 |
| `«use»` | 使用 | 一般性的使用关系（默认） |

```
┌──────────┐  «import»   ┌──────────┐
│   UI     │ ──────────→ │  Domain  │
└──────────┘             └──────────┘
     │
     │  «import»
     ↓
┌──────────┐
│  Utils   │
└──────────┘
```

#### （2）嵌套关系

**符号**：大包包含小包

```
┌──────────────────────────────┐
│ package System               │
│                              │
│  ┌──────────┐ ┌──────────┐  │
│  │ Frontend │ │ Backend  │  │
│  └──────────┘ └──────────┘  │
│                              │
└──────────────────────────────┘
```

#### （3）合并/泛化

**符号**：虚线 + `«merge»` 或 实线 + 空心三角（泛化）

```
BasePackage ◁—— SpecializedPackage
```

### 2.5 包图的常见组织模式

#### 分层架构

```
┌──────────────────┐
│ Presentation     │  ← 表示层
├──────────────────┤
│ «import»         │
└───────┬──────────┘
        │ 依赖
        ↓
┌──────────────────┐
│ Business Logic   │  ← 业务逻辑层
├──────────────────┤
│ «import»         │
└───────┬──────────┘
        │ 依赖
        ↓
┌──────────────────┐
│ Data Access      │  ← 数据访问层
└──────────────────┘
```

> **依赖原则**：上层依赖下层，下层不依赖上层

#### MVC 架构

```
┌──────────────────────────────────────────┐
│ package Web Application                  │
│                                          │
│  ┌──────────────┐                        │
│  │    Model     │ ◁──┐                   │
│  └──────┬───────┘    │                   │
│         │ 观察       │                   │
│         ↓            │ 更新              │
│  ┌──────────────┐    │                   │
│  │    View      │ ───┘                   │
│  └──────┬───────┘                        │
│         │ 控制                            │
│         ↓                                │
│  ┌──────────────┐                        │
│  │  Controller  │                        │
│  └──────────────┘                        │
└──────────────────────────────────────────┘
```

### 2.6 包的设计原则

1. **高内聚**：同一个包内的元素应该紧密相关
2. **低耦合**：包之间的依赖应该尽量少
3. **稳定依赖原则**：依赖应该指向更稳定的方向
4. **无环依赖**：包之间的依赖不应形成环

```
好的设计（无环）：          坏的设计（有环）：
A → B → C                 A → B
                          ↑   ↓
                          D ← C
```
