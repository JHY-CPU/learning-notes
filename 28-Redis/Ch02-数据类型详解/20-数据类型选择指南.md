# 数据类型选择指南

## 一、概念说明

选择正确的数据类型对Redis的性能和内存使用至关重要。本指南帮助根据实际场景选择最合适的数据类型。

## 二、按场景选择数据类型

### 简单键值存储

```bash
# 场景：缓存配置、Session、Token
# 选择：String
SET session:token "abc123" EX 3600
SET config:max_users 10000
# 优势：简单高效，支持过期时间
```

### 对象存储

```bash
# 场景：用户信息、商品信息
# 选择：Hash
HSET user:1001 name "张三" age 25 email "test@example.com"
# 优势：可单独修改字段，节省内存（ziplist）
# 对比：String需要整体序列化
```

### 列表/队列

```bash
# 场景：消息队列、最新列表、时间线
# 选择：List
LPUSH queue:orders "order1"
BRPOP queue:orders 0
# 优势：支持阻塞操作，天然队列
```

### 去重/集合

```bash
# 场景：标签、共同好友、去重统计
# 选择：Set
SADD user:1001:tags "Java" "Redis"
SINTER user:1001:tags user:1002:tags
# 优势：集合运算，去重
```

### 排行榜/有序数据

```bash
# 场景：排行榜、延迟队列、时间序列
# 选择：ZSet
ZADD leaderboard 100 "player1"
ZREVRANGE leaderboard 0 9 WITHSCORES
# 优势：自动排序，范围查询高效
```

### 基数统计

```bash
# 场景：UV统计、独立IP统计
# 选择：HyperLogLog
PFADD visitors "user1" "user2" "user3"
PFCOUNT visitors
# 优势：极省内存（12KB），有微小误差
```

### 地理位置

```bash
# 场景：附近的人/店铺、距离计算
# 选择：Geo
GEOADD stores 116.40 39.90 "store1"
GEORADIUS stores 116.40 39.90 1 km
# 优势：地理位置计算内置支持
```

### 消息队列

```bash
# 场景：可靠消息队列、事件流
# 选择：Stream
XADD events * type "login" user "1001"
XREADGROUP GROUP mygroup consumer1 COUNT 10 STREAMS events >
# 优势：消费者组、ACK确认、消息回溯
```

## 三、数据类型对比表

| 需求 | 推荐类型 | 原因 |
|------|----------|------|
| 简单缓存 | String | 最简单高效 |
| 对象存储 | Hash | 独立字段操作 |
| 队列 | List/Stream | 阻塞消费支持 |
| 去重集合 | Set | 自动去重 |
| 排行榜 | ZSet | 自动排序 |
| UV统计 | HyperLogLog | 极省内存 |
| 地理位置 | Geo | 内置地理计算 |
| 可靠消息 | Stream | ACK机制 |

## 四、选择原则

```bash
# 1. 最小化数据类型
# 能用String解决就不用Hash
# 能用Set解决就不用ZSet

# 2. 考虑操作频率
# 频繁修改的字段用Hash（避免整体序列化）
# 频繁排序的用ZSet

# 3. 考虑内存效率
# 小数据量优先使用ziplist编码
# 大规模统计考虑HyperLogLog

# 4. 考虑功能需求
# 需要集合运算 → Set
# 需要范围查询 → ZSet
# 需要阻塞消费 → List/Stream
```

## 五、注意事项与常见陷阱

1. **不要强制用复杂类型**：简单场景用String最高效
2. **编码自动优化**：Redis根据数据量自动选择编码
3. **大Key问题**：任何类型都应控制单个Key的大小
4. **过期时间**：只有String/Hash/Set/ZSet支持Key级别过期
5. **事务需求**：考虑使用MULTI/EXEC或Lua脚本
