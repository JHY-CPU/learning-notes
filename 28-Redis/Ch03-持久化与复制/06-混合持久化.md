# 混合持久化

## 一、概念说明

混合持久化是Redis 4.0引入的特性，将RDB和AOF结合使用。在AOF重写时，使用RDB格式记录全量数据，后续增量数据使用AOF格式。这种方式兼顾了恢复速度和数据安全性。

## 二、具体用法

### 开启混合持久化

```bash
# redis.conf 配置

# 必须先开启AOF
appendonly yes

# 开启混合持久化
aof-use-rdb-preamble yes

# AOF文件格式
# 文件开头：RDB格式的全量数据
# 文件结尾：AOF格式的增量数据
```

### 文件格式示例

```bash
# 混合AOF文件结构
# ┌─────────────────────────────────────┐
# │         RDB格式的全量数据            │
# │   （二进制数据，包含所有当前数据）    │
# ├─────────────────────────────────────┤
# │         AOF格式的增量数据            │
# │   （自上次RDB快照后的写命令）         │
# └─────────────────────────────────────┘

# 查看混合AOF文件
head -c 100 /var/lib/redis/appendonly.aof
# 开头是REDIS标记（RDB格式）
```

## 三、优势分析

```bash
# 对比纯AOF
# 1. 恢复速度更快（RDB部分直接加载）
# 2. 文件大小更小（RDB格式更紧凑）
# 3. 数据安全性相同（增量AOF保证）

# 对比纯RDB
# 1. 数据安全性更高（增量AOF）
# 2. 可接受的数据丢失更少

# 实际效果
# 恢复时间：纯AOF > 混合 > 纯RDB
# 数据安全：纯AOF = 混合 > 纯RDB
# 文件大小：纯AOF > 混合 > 纯RDB
```

## 四、配置建议

```bash
# 生产环境推荐配置
appendonly yes
aof-use-rdb-preamble yes
appendfsync everysec
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# 为什么不使用always？
# everysec是安全性和性能的最佳平衡
# always的性能开销太大
```

## 五、注意事项与常见陷阱

1. **Redis 4.0+才支持**：旧版本不支持混合持久化
2. **需要开启AOF**：混合持久化依赖AOF
3. **向后兼容**：旧版本Redis无法加载混合格式的AOF文件
4. **重写触发**：混合格式只在AOF重写时生成
5. **不改变RDB行为**：RDB快照仍然按规则独立执行
6. **推荐开启**：生产环境建议开启混合持久化
