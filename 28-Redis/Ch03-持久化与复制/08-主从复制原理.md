# 主从复制原理

## 一、概念说明

Redis主从复制包括全量同步和增量同步两种方式。全量同步用于首次连接或断线后无法增量同步的情况，增量同步用于从节点短暂断线后的数据恢复。

## 二、全量同步过程

```
1. 从节点发送PSYNC命令
   │
2. 主节点判断是否可以增量同步
   │
   ├── 可以增量同步 → 进入增量同步
   │
   └── 不能 → 全量同步
         │
3. 主节点执行BGSAVE
         │
4. 主节点发送RDB文件
         │
5. 从节点接收并加载RDB
         │
6. 主节点发送缓冲区中的写命令
         │
7. 同步完成
```

### 关键参数

```bash
# 复制积压缓冲区（repl_backlog）
# 主节点维护的环形缓冲区
# 默认1MB
repl-backlog-size 1mb

# 缓冲区空闲超时
repl-backlog-ttl 3600
# 从节点断开1小时后释放缓冲区
```

## 三、增量同步过程

```
1. 从节点断线重连
   │
2. 发送PSYNC <runid> <offset>
   │
3. 主节点检查runid是否匹配
   │
4. 检查offset是否在复制积压缓冲区中
   │
   ├── 在 → 发送offset后的增量数据
   │
   └── 不在 → 全量同步
```

### 运行ID与偏移量

```bash
# 查看运行ID
INFO server | grep run_id
# 输出: run_id:abc123...

# 查看复制偏移量
INFO replication
# 主节点: master_repl_offset:1000
# 从节点: slave_repl_offset:1000

# 偏移量一致表示数据同步
```

## 四、异步复制机制

```bash
# 主节点写入流程
# 1. 执行写命令
# 2. 写入复制积压缓冲区
# 3. 返回客户端成功
# 4. 异步发送到从节点

# 复制延迟
# 查看从节点延迟
INFO replication
# slave0:lag=0  # 0表示无延迟
# lag > 0 表示有延迟
```

## 五、注意事项与常见陷阱

1. **异步复制延迟**：从节点数据可能不是最新的
2. **runid变化**：重启后runid会变，导致全量同步
3. **积压缓冲区大小**：设置太小会导致频繁全量同步
4. **大Key同步**：大Key传输可能导致同步超时
5. **网络带宽**：全量同步消耗大量网络带宽
6. **磁盘IO**：BGSAVE期间有磁盘IO压力
