# 缓存更新策略

## 一、概念说明

缓存更新策略决定何时以及如何更新缓存数据。不同策略有不同的数据一致性保证和性能特征。

## 二、常见策略

### 先更新DB再删缓存（推荐）

```bash
# 流程
1. 更新数据库
2. 删除缓存

# 代码
UPDATE users SET name='新名字' WHERE id=1001
DEL user:1001

# 优点：简单可靠
# 缺点：短暂不一致
```

### 先删缓存再更新DB

```bash
# 流程
1. 删除缓存
2. 更新数据库

# 问题：并发时可能不一致
# 线程A删除缓存
# 线程B读取，缓存未命中，查DB，写入旧数据
# 线程A更新DB
# 缓存中是旧数据

# 解决：延迟双删
1. 删除缓存
2. 更新DB
3. sleep(500)
4. 再次删除缓存
```

### 更新DB同时更新缓存

```bash
# 流程
1. 更新数据库
2. 更新缓存

# 问题：并发更新顺序不确定
# 线程A更新DB（name=A）
# 线程B更新DB（name=B）
# 线程B更新缓存（name=B）
# 线程A更新缓存（name=A）
# 结果：DB是B，缓存是A（不一致）
```

## 三、异步更新策略

```bash
# 使用消息队列保证最终一致
1. 更新DB
2. 发送MQ消息
3. 消费者更新/删除缓存

# 使用Canal监听binlog
1. 更新DB
2. Canal监听到变更
3. 发送到MQ
4. 更新缓存
```

## 四、策略对比

| 策略 | 一致性 | 性能 | 复杂度 |
|------|--------|------|--------|
| 先DB后删缓存 | 最终一致 | 高 | 低 |
| 先删缓存后DB | 可能不一致 | 高 | 中 |
| 同时更新 | 可能不一致 | 中 | 低 |
| MQ异步 | 最终一致 | 高 | 高 |

## 五、注意事项

1. **推荐方案**：先更新DB，再删除缓存
2. **并发问题**：考虑多线程竞争
3. **重试机制**：删除缓存失败时重试
4. **监控不一致**：定期检查缓存与DB一致性
