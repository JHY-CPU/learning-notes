# 缓存更新策略

## 一、概念说明

缓存更新策略决定何时以及如何更新缓存数据。不同策略有不同的数据一致性保证和性能特征。

## 二、常见策略

### 先更新DB再删缓存（推荐）

```bash
# 流程
1. 更新数据库
2. 删除缓存

# 代码
UPDATE users SET name='新名字' WHERE id=1001
DEL user:1001

# 优点：简单可靠
# 缺点：短暂不一致
```

### 先删缓存再更新DB

```bash
# 流程
1. 删除缓存
2. 更新数据库

# 问题：并发时可能不一致
# 线程A删除缓存
# 线程B读取，缓存未命中，查DB，写入旧数据
# 线程A更新DB
# 缓存中是旧数据

# 解决：延迟双删
1. 删除缓存
2. 更新DB
3. sleep(500)
4. 再次删除缓存
```

### 更新DB同时更新缓存

```bash
# 流程
1. 更新数据库
2. 更新缓存

# 问题：并发更新顺序不确定
# 线程A更新DB（name=A）
# 线程B更新DB（name=B）
# 线程B更新缓存（name=B）
# 线程A更新缓存（name=A）
# 结果：DB是B，缓存是A（不一致）
```

## 三、异步更新策略

```bash
# 使用消息队列保证最终一致
1. 更新DB
2. 发送MQ消息
3. 消费者更新/删除缓存

# 使用Canal监听binlog
1. 更新DB
2. Canal监听到变更
3. 发送到MQ
4. 更新缓存
```

## 四、策略对比

| 策略 | 一致性 | 性能 | 复杂度 |
|------|--------|------|--------|
| 先DB后删缓存 | 最终一致 | 高 | 低 |
| 先删缓存后DB | 可能不一致 | 高 | 中 |
| 同时更新 | 可能不一致 | 中 | 低 |
| MQ异步 | 最终一致 | 高 | 高 |

## 五、注意事项

1. **推荐方案**：先更新DB，再删除缓存
2. **并发问题**：考虑多线程竞争
3. **重试机制**：删除缓存失败时重试
4. **监控不一致**：定期检查缓存与DB一致性

## 六、各策略详细对比

```bash
# 策略1：先更新DB，再删缓存（推荐）
# 步骤：
# 1. UPDATE users SET name='新' WHERE id=1
# 2. DEL user:1

# 优点：
# - 实现简单
# - 并发安全（短暂不一致窗口很小）
# - 删除操作幂等

# 缺点：
# - 存在短暂不一致窗口
# - 缓存未命中时查询压力大

# 不一致窗口分析：
# 线程A: UPDATE DB
# 线程B: GET cache (miss)
# 线程B: GET DB (旧值)
# 线程A: DEL cache
# 线程B: SET cache (旧值) ← 不一致！
# 但这个窗口很短，缓存TTL到期后自动修复
```

```bash
# 策略2：先删缓存，再更新DB
# 问题分析：
# 线程A: DEL cache
# 线程B: GET cache (miss)
# 线程B: GET DB (旧值)
# 线程B: SET cache (旧值)
# 线程A: UPDATE DB (新值)
# 缓存中是旧值！不一致持续到下次删除

# 延迟双删修复：
# 1. DEL cache
# 2. UPDATE DB
# 3. sleep(500ms)
# 4. DEL cache (再次删除)

# 延迟时间选择：
# - 需要大于一次读操作的时间
# - 通常500ms-1000ms
# - 根据业务调整
```

```bash
# 策略3：同时更新DB和缓存
# 问题：
# - 并发更新顺序不确定
# - 可能导致缓存是旧值

# 示例：
# 线程A: UPDATE DB (name=A)
# 线程B: UPDATE DB (name=B)
# 线程B: SET cache (name=B)
# 线程A: SET cache (name=A)
# 结果：DB是B，缓存是A

# 结论：不推荐此方案
```

## 七、异步更新方案详解

```python
import redis
import json
import time
from threading import Thread
from queue import Queue

r = redis.Redis()

class AsyncCacheUpdater:
    def __init__(self):
        self.update_queue = Queue()
        self.worker = Thread(target=self._process_updates, daemon=True)
        self.worker.start()
    
    def update_db_and_cache(self, key, db_update_func):
        """更新DB并异步更新缓存"""
        # 1. 更新数据库
        db_update_func()
        
        # 2. 放入队列
        self.update_queue.put(key)
    
    def _process_updates(self):
        """异步处理缓存更新"""
        while True:
            key = self.update_queue.get()
            try:
                # 删除缓存
                r.delete(key)
                print(f"异步删除缓存: {key}")
            except Exception as e:
                print(f"删除失败: {e}")
                # 可以重试或放入死信队列
            finally:
                self.update_queue.task_done()

# 使用
updater = AsyncCacheUpdater()

def update_user(user_id, name):
    def db_update():
        # 模拟数据库更新
        print(f"DB更新: user:{user_id} -> {name}")
    
    updater.update_db_and_cache(f"user:{user_id}", db_update)
```

## 八、Canal binlog方案

```bash
# 架构
# MySQL → Canal Server → MQ → Consumer → Redis

# 优势
# 1. 完全解耦，应用无感知
# 2. 可靠性高，不遗漏变更
# 3. 支持多表、多库同步
# 4. 可以批量处理

# 配置示例
# canal.properties
canal.instance.master.address=127.0.0.1:3306
canal.instance.dbUsername=canal
canal.instance.dbPassword=canal

# Consumer伪代码
def consume_binlog(event):
    if event.table == 'users':
        user_id = event.data['id']
        if event.type == 'UPDATE':
            r.delete(f"user:{user_id}")
        elif event.type == 'DELETE':
            r.delete(f"user:{user_id}")
```
