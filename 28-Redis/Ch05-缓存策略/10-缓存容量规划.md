# 缓存容量规划

## 一、概念说明

缓存容量规划是估算Redis所需的内存空间，选择合适的实例规格和淘汰策略。

## 二、内存估算

```bash
# 单个Key内存估算
# String类型
SET user:1001 '{"name":"张三","age":25}'
# Key: user:1001 ≈ 11 bytes
# Value: JSON ≈ 35 bytes
# Redis对象头 ≈ 40 bytes
# 总计 ≈ 86 bytes

# 100万用户估算
100万 * 86 bytes ≈ 86 MB

# Hash类型更省内存
HSET user:1001 name "张三" age 25
# ziplist编码更紧凑
```

### 详细估算公式

```bash
# String
内存 = Key长度 + Value长度 + 对象头(约40bytes)

# Hash (ziplist)
内存 ≈ 字段数 * (字段名 + 值 + 开销)

# List (quicklist)
内存 ≈ 元素数 * (元素值 + 指针开销)

# Set/ZSet
内存 ≈ 元素数 * (元素值 + 额外开销)
```

## 三、容量规划

```bash
# 预估数据量
用户数: 100万
每个用户缓存: 1 KB
总内存需求: 1 GB

# 考虑开销
实际内存 ≈ 预估 * 1.5（预留50%余量）
实际内存 ≈ 1.5 GB

# 选择实例
# 推荐实例内存 = 实际内存 * 2
# 给持久化、复制等留出空间
```

## 四、淘汰策略配置

```bash
# 根据业务需求选择
# 缓存场景：allkeys-lru
maxmemory 4gb
maxmemory-policy allkeys-lru

# 混合场景：volatile-lru
maxmemory 4gb
maxmemory-policy volatile-lru

# 监控内存使用
INFO memory
# used_memory_human:2.50G
# maxmemory_human:4.00G
```

## 五、注意事项

1. **预留空间**：实际内存需求的2倍
2. **考虑过期Key**：过期Key也占用内存
3. **碎片率**：mem_fragmentation_ratio > 1.5需要关注
4. **扩容计划**：提前规划扩容方案
5. **监控告警**：内存使用超过80%告警

## 七、数据类型内存计算

```python
def estimate_memory(data_type, count, key_size, value_size):
    """估算内存使用"""
    # Redis对象头：约40 bytes
    object_overhead = 40
    
    if data_type == 'string':
        # Key + Value + 对象头
        return (key_size + value_size + object_overhead) * count
    
    elif data_type == 'hash':
        # ziplist编码（小Hash）
        if count <= 128 and value_size <= 64:
            # ziplist更紧凑
            return (key_size + value_size * 2 + 10) * count
        else:
            # hashtable编码
            return (key_size + value_size * 2 + object_overhead + 50) * count
    
    elif data_type == 'list':
        # quicklist编码
        return (value_size + 20) * count  # 每个节点20字节开销
    
    elif data_type == 'set':
        if all_int_values and count <= 512:
            # intset编码
            return (8 + object_overhead) * count
        else:
            return (value_size + object_overhead + 50) * count
    
    elif data_type == 'zset':
        if count <= 128 and value_size <= 64:
            # ziplist编码
            return (value_size + 8 + 10) * count
        else:
            # skiplist编码
            return (value_size + 8 + object_overhead + 50) * count

# 示例计算
# 100万用户，Hash类型，每个用户5个字段
memory = estimate_memory('hash', 1000000, 15, 20)
print(f"预估内存: {memory / 1024 / 1024:.0f} MB")
```

## 八、容量规划实战

```bash
# 步骤1：数据量统计
# 统计各业务的Key数量和大小
INFO keyspace
# db0:keys=5000000,expires=3000000,avg_ttl=3600000

# 步骤2：内存使用分析
INFO memory
# used_memory:2147483648 (2GB)
# maxmemory:4294967296 (4GB)
# 使用率: 50%

# 步骤3：增长预测
# 假设每月增长10%
# 6个月后: 2GB * 1.1^6 ≈ 3.5GB
# 12个月后: 2GB * 1.1^12 ≈ 6.3GB

# 步骤4：实例规格选择
# 当前使用: 2GB
# 6个月后: 3.5GB
# 12个月后: 6.3GB
# 考虑50%余量: 6.3GB * 1.5 ≈ 10GB
# 建议实例: 16GB

# 步骤5：淘汰策略配置
maxmemory 12gb
maxmemory-policy allkeys-lru
# 预留4GB给系统和复制积压
```

## 九、扩容方案

```bash
# 方案1：垂直扩容
# 增加单节点内存
# 4GB → 8GB → 16GB → 32GB
# 优点：简单
# 缺点：有上限

# 方案2：水平扩容（集群）
# 增加节点数量
# 3节点 → 6节点 → 12节点
# 优点：无上限
# 缺点：需要数据迁移

# 方案3：数据分层
# 热数据：Redis（内存）
# 温数据：SSD存储
# 冷数据：对象存储

# 方案4：数据压缩
# 使用msgpack代替JSON
# 使用Snappy/LZ4压缩大Value
# 可节省30-50%内存
```
