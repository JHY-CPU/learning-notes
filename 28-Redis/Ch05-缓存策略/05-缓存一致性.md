# 缓存一致性

## 一、概念说明

缓存一致性是指缓存数据与数据库数据保持一致。由于缓存和数据库是两个独立的系统，保持一致性是一个挑战。

## 二、常见策略

### Cache-Aside（旁路缓存）

```bash
# 读操作
1. 读缓存
2. 命中则返回
3. 未命中则读DB，写缓存

# 写操作
1. 更新DB
2. 删除缓存（推荐）

# 为什么不更新缓存？
# 并发写可能导致脏数据
# 删除更简单可靠
```

### Write-Through（写穿透）

```bash
# 写操作同时更新缓存和DB
1. 更新缓存
2. 更新DB
3. 两个都成功才返回

# 优点：缓存始终最新
# 缺点：写入延迟增加
```

### Write-Behind（写回）

```bash
# 写操作只更新缓存
1. 更新缓存
2. 异步更新DB

# 优点：写入性能高
# 缺点：可能丢失数据
```

## 三、延迟双删

```bash
# 解决并发读写不一致
1. 删除缓存
2. 更新DB
3. 延迟一段时间后再次删除缓存

# 伪代码
redis.del("user:1001")
db.update(user)
sleep(500)  # 等待读请求完成
redis.del("user:1001")  # 再次删除
```

## 四、基于消息队列

```bash
# 更新DB后发送消息
1. 更新DB
2. 发送MQ消息
3. 消费者删除缓存

# 保证最终一致性
```

## 五、注意事项

1. **最终一致性**：大多数场景接受最终一致
2. **删除优于更新**：写操作优先删除缓存
3. **并发控制**：考虑并发读写的顺序
4. **异常处理**：缓存操作失败的重试机制
5. **监控不一致**：定期对比缓存和DB数据

## 六、延迟双删详解

```python
import time
import redis
import threading

r = redis.Redis()

def update_with_double_delete(key, db_update_func, delay=0.5):
    """延迟双删方案"""
    # 1. 第一次删除缓存
    r.delete(key)
    
    # 2. 更新数据库
    db_update_func()
    
    # 3. 延迟后第二次删除
    def delayed_delete():
        time.sleep(delay)
        r.delete(key)
        print(f"延迟删除完成: {key}")
    
    # 异步执行延迟删除
    threading.Thread(target=delayed_delete, daemon=True).start()

# 使用示例
def update_user_name(user_id, new_name):
    def db_update():
        # 模拟数据库更新
        print(f"更新数据库: user:{user_id} -> {new_name}")
    
    update_with_double_delete(
        f"user:{user_id}",
        db_update,
        delay=0.5
    )
```

## 八、一致性监控与修复

```python
def check_consistency(key, db_query_func):
    """检查缓存与数据库一致性"""
    # 获取缓存数据
    cached = r.get(key)
    if not cached:
        return "缓存不存在"
    
    cached_data = json.loads(cached)
    
    # 获取数据库数据
    db_data = db_query_func()
    
    if not db_data:
        return "数据库不存在，缓存应删除"
    
    if cached_data != db_data:
        return f"不一致: 缓存={cached_data}, 数据库={db_data}"
    
    return "一致"

def fix_inconsistency(key, db_query_func):
    """修复不一致"""
    db_data = db_query_func()
    if db_data:
        r.setex(key, 3600, json.dumps(db_data))
        print(f"已修复: {key}")
    else:
        r.delete(key)
        print(f"已删除: {key}")

# 定期检查
def periodic_check(keys, db_query_func):
    for key in keys:
        result = check_consistency(key, db_query_func)
        if result != "一致":
            print(f"发现不一致: {key} - {result}")
            fix_inconsistency(key, db_query_func)
```

## 九、一致性方案选择

```bash
# 强一致性（几乎不可能）
# 分布式事务：2PC/TCC
# 性能代价高，复杂度高

# 最终一致性（推荐）
# 方案1：先更新DB再删缓存（简单可靠）
# 方案2：延迟双删（处理并发读）
# 方案3：Canal监听binlog（异步可靠）
# 方案4：消息队列（可靠投递）

# 选择建议
# 简单场景 → 先更新DB再删缓存
# 并发读写多 → 延迟双删
# 数据量大 → Canal方案
# 需要可靠 → 消息队列方案
```
