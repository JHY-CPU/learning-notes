# 缓存击穿

## 一、概念说明

缓存击穿是指热点Key过期的瞬间，大量并发请求同时打到数据库，导致数据库压力骤增。

## 二、问题分析

```bash
# 热点Key过期
GET hot:product:1001
# Key刚过期，缓存nil
# 1000个并发请求同时查数据库
# 数据库瞬间压力增大

# 对比缓存穿透：查询不存在的数据
# 缓存击穿：数据存在但缓存过期
```

## 三、解决方案

### 方案一：互斥锁

```bash
# 使用Redis分布式锁
SET lock:product:1001 "uuid" NX EX 10
# 获取锁成功，查询数据库并写缓存
# 获取锁失败，等待或返回默认值

# 伪代码
if redis.get("product:1001"):
    return redis.get("product:1001")
if redis.set("lock:product:1001", uuid, NX=True, EX=10):
    data = db.query("product:1001")
    redis.set("product:1001", data, EX=3600)
    return data
else:
    sleep(0.1)
    retry()
```

### 方案二：逻辑过期

```bash
# 不设置物理过期时间
# 在Value中存储逻辑过期时间

SET product:1001 '{"data":"商品信息","expire":1704067200}'

# 读取时检查逻辑过期
value = redis.get("product:1001")
if value.expire < now():
    # 异步更新缓存
    async_update_cache()
return value.data
```

### 方案三：永不过期+异步更新

```bash
# Key永不过期
SET product:1001 data

# 后台任务定期更新
# 每隔一段时间更新热点数据
```

## 四、方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| 互斥锁 | 强一致 | 有延迟 |
| 逻辑过期 | 高性能 | 数据可能不一致 |
| 永不过期 | 简单 | 占用内存 |

## 五、注意事项

1. **锁超时时间**：避免死锁
2. **重试机制**：获取锁失败时的处理
3. **热点识别**：识别并保护热点Key
4. **降级策略**：极端情况下的降级
