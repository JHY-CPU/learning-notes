# 慢查询日志

## 一、概念说明

Redis慢查询日志记录执行时间超过阈值的命令，帮助发现和优化性能问题。类似于MySQL的慢查询日志。

## 二、具体用法

### 配置慢查询

```bash
# redis.conf 配置

# 慢查询阈值（微秒）
slowlog-log-slower-than 10000
# 超过10000微秒（10毫秒）的命令记录

# 慢查询日志最大条数
slowlog-max-len 128
# 最多记录128条

# 禁用慢查询日志
slowlog-log-slower-than 0

# 记录所有命令
slowlog-log-slower-than -1
```

### 查看慢查询日志

```bash
# 获取慢查询日志
SLOWLOG GET 10
# 输出:
# 1) 1) (integer) 10     # 日志ID
#    2) (integer) 1704067200  # 时间戳
#    3) (integer) 15000  # 执行时间（微秒）
#    4) 1) "KEYS"        # 命令
#       2) "*"
#    5) "192.168.1.1:6379"  # 客户端地址
#    6) ""              # 客户端名称

# 获取慢查询数量
SLOWLOG LEN
# 输出: (integer) 5

# 清空慢查询日志
SLOWLOG RESET
```

## 三、分析慢查询

```bash
# 常见慢查询原因
# 1. KEYS * 命令
# 2. 大集合操作（SMEMBERS大Set）
# 3. 大Hash的HGETALL
# 4. SORT命令无LIMIT
# 5. Lua脚本执行时间过长

# 优化建议
# 使用SCAN替代KEYS
# 使用SSCAN/HSCAN/ZSCAN
# 分批处理大数据
```

## 四、注意事项

1. **阈值设置**：生产环境建议10000微秒（10毫秒）
2. **日志大小**：slowlog-max-len不宜过大
3. **内存中存储**：慢查询日志存在内存中
4. **定期检查**：定期分析慢查询日志
5. **不记录阻塞命令**：BLPOP等阻塞命令不记录

## 五、Python分析慢查询

```python
import redis
from datetime import datetime

r = redis.Redis()

# 获取慢查询日志
logs = r.slowlog_get(100)

for entry in logs:
    print(f"ID: {entry['id']}")
    print(f"时间: {datetime.fromtimestamp(entry['start_time'])}")
    print(f"耗时: {entry['duration']} 微秒")
    print(f"命令: {' '.join(entry['command'])}")
    print(f"客户端: {entry.get('client_address', 'N/A')}")
    print("---")

# 自动分析慢查询
def analyze_slow_queries():
    logs = r.slowlog_get(1000)
    command_count = {}
    
    for entry in logs:
        cmd = entry['command'][0] if entry['command'] else 'UNKNOWN'
        command_count[cmd] = command_count.get(cmd, 0) + 1
    
    # 按频率排序
    sorted_cmds = sorted(command_count.items(), key=lambda x: x[1], reverse=True)
    
    print("慢查询命令统计:")
    for cmd, count in sorted_cmds[:10]:
        print(f"  {cmd}: {count} 次")
    
    # 平均耗时
    total_duration = sum(e['duration'] for e in logs)
    avg_duration = total_duration / len(logs) if logs else 0
    print(f"\n平均耗时: {avg_duration:.0f} 微秒")
```

## 六、生产环境调优

```bash
# 1. 合理设置阈值
# 开发环境：1000微秒（1ms）
# 测试环境：5000微秒（5ms）
# 生产环境：10000微秒（10ms）
# 高性能要求：50000微秒（50ms）

# 2. 定期分析
# 使用脚本定期收集慢查询
# 发送告警通知

# 3. 配合大Key分析
redis-cli --bigkeys
# 大Key操作通常是慢查询的根源

# 4. 配合内存分析
MEMORY USAGE key_name
# 内存大的Key操作更慢

# 5. 使用LATENCY命令
LATENCY LATEST
LATENCY HISTORY command_name
LATENCY DOCTOR
```

## 七、慢查询与大Key的关系

```bash
# 大Key操作必然是慢查询
# KEYS * 遍历所有Key
# SMEMBERS 大Set（50000+元素）
# HGETALL 大Hash（10000+字段）
# LRANGE 0 -1 大List（10000+元素）

# 优化方案
# 1. 使用SCAN/HSCAN/SSCAN/ZSCAN替代
# 2. 分页获取数据
# 3. 拆分大Key为多个小Key
# 4. 使用UNLINK异步删除大Key
```
