# 高级特性最佳实践

## 一、概念说明

Redis高级特性（事务、Lua脚本、Pipeline、ACL等）在正确使用时可以显著提升应用的性能和安全性。本节总结各特性的最佳实践。

## 二、事务最佳实践

```bash
# 1. 使用WATCH保证数据一致性
WATCH balance
MULTI
DECRBY balance 100
INCRBY target 100
EXEC

# 2. 事务中避免大Key操作
# 3. 事务失败后重试机制
# 4. 复杂逻辑使用Lua脚本替代
```

## 三、Lua脚本最佳实践

```bash
# 1. 使用EVALSHA减少网络传输
SCRIPT LOAD "return redis.call('GET',KEYS[1])"
EVALSHA <sha1> 1 mykey

# 2. 脚本保持简短
# 3. 避免在脚本中循环大量数据
# 4. 集群环境下key必须在同一slot
# 5. 设置合理的lua-time-limit
lua-time-limit 5000
```

## 五、ACL最佳实践

```bash
# 1. 最小权限原则
ACL SETUSER app on >pass ~app:* +@read +@write

# 2. 禁用危险命令
ACL SETUSER app -@dangerous

# 3. 定期审计权限
ACL LIST

# 4. 使用独立用户
# 不同应用使用不同的ACL用户
```

## 八、注意事项

1. **特性叠加**：组合使用时注意性能影响
2. **版本升级**：新特性可能有兼容性变化
3. **客户端支持**：确保客户端库支持相应特性
4. **集群限制**：部分特性在集群模式下有限制
5. **文档更新**：关注Redis官方文档更新

## 九、特性选择决策树

```bash
# 需要原子性操作？
# ├── 简单操作 → MULTI/EXEC 事务
# └── 复杂逻辑 → Lua脚本 / Redis Functions

# 需要减少网络开销？
# ├── 批量操作 → Pipeline
# └── 批量+原子性 → Pipeline + 事务

# 需要权限控制？
# ├── Redis 6.0+ → ACL
# └── 旧版本 → requirepass + rename-command

# 需要消息通信？
# ├── 实时通知（允许丢失）→ Pub/Sub
# ├── 可靠消息 → Stream
# └── 任务队列 → Stream + 消费者组

# 需要扩展功能？
# ├── 全文搜索 → RediSearch
# ├── JSON存储 → RedisJSON
# ├── 时序数据 → RedisTimeSeries
# ├── 去重/过滤 → RedisBloom
# └── 图计算 → RedisGraph

# 需要客户端缓存？
# ├── Redis 6.0+ → 客户端缓存（RESP3）
# └── 旧版本 → 本地缓存 + TTL
```

## 十一、常见陷阱与避坑

```bash
# 陷阱1：事务中不支持回滚
# 事务中的命令执行失败不会回滚已执行的命令
# 解决：使用Lua脚本实现复杂逻辑

# 陷阱2：Pub/Sub消息丢失
# 订阅者离线时的消息会丢失
# 解决：使用Stream替代

# 陷阱3：Pipeline没有原子性
# Pipeline只是减少RTT，不保证原子性
# 解决：Pipeline + MULTI/EXEC

# 陷阱4：Lua脚本阻塞
# 长时间Lua脚本会阻塞整个Redis
# 解决：保持脚本简短，设置超时

# 陷阱5：ACL默认用户未禁用
# 默认用户有所有权限
# 解决：禁用默认用户或限制权限
```
