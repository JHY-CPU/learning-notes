# 客户端缓存

## 一、概念说明

Redis客户端缓存（Client-side Caching）允许客户端在本地缓存数据，当服务端数据变化时通知客户端失效缓存。基于Redis 6.0的RESP3协议和TRACKING功能。

## 二、具体用法

### 服务端配置

```bash
# 开启RESP3协议
HELLO 3

# 开启客户端追踪
CLIENT TRACKING ON
# 开启后，服务端会追踪客户端GET的键
# 当键被修改时通知客户端

# 设置追踪前缀
CLIENT TRACKING ON PREFIX user: REDIRECT client-id

# 设置缓存TTL
CLIENT TRACKING ON OPTIN
# 手动选择要缓存的命令
```

### 客户端使用

```python
# Python示例
import redis

r = redis.Redis(protocol=3)  # RESP3

# 开启追踪
r.client_tracking_on()

# 缓存GET结果
value = r.get("user:1001")
# 客户端本地缓存此值

# 当其他客户端修改时
r.set("user:1001", "newvalue")
# 服务端通知客户端失效缓存
```

### 缓存模式

```bash
# OPTIN模式 - 手动缓存
CLIENT TRACKING ON OPTIN
CLIENT CACHING YES  # 下一个命令结果被缓存
GET user:1001

# OPTOUT模式 - 默认缓存
CLIENT TRACKING ON OPTOUT
CLIENT CACHING NO   # 下一个命令结果不缓存
GET user:1001
```

## 三、工作原理

```
客户端A GET user:1001 → 服务端 → 返回值 → 客户端A本地缓存
        │
客户端B SET user:1001 → 服务端 → 通知客户端A失效缓存
        │
客户端A GET user:1001 → 本地缓存已失效 → 从服务端获取
```

## 四、注意事项

1. **RESP3协议**：需要使用RESP3协议
2. **无效化通知**：数据变化时收到INVALIDATE消息
3. **内存限制**：客户端需要管理本地缓存大小
4. **网络开销**：追踪有额外的网络开销
5. **集群模式**：需要连接到正确的节点

## 五、客户端缓存实现示例

### Python实现

```python
import redis
from threading import Lock

class RedisClientCache:
    def __init__(self):
        self.r = redis.Redis(protocol=3)
        self.local_cache = {}
        self.lock = Lock()
        
        # 开启追踪
        self.r.client_tracking_on(bcast=True)
    
    def get(self, key):
        # 检查本地缓存
        with self.lock:
            if key in self.local_cache:
                return self.local_cache[key]
        
        # 从Redis获取
        value = self.r.get(key)
        
        # 写入本地缓存
        if value:
            with self.lock:
                self.local_cache[key] = value
        
        return value
    
    def invalidate(self, key):
        """缓存失效回调"""
        with self.lock:
            if key in self.local_cache:
                del self.local_cache[key]
                print(f"本地缓存已失效: {key}")
```

### Java实现

```java
public class RedisClientCache {
    private final RedisClient client;
    private final Map<String, String> localCache = new ConcurrentHashMap<>();
    
    public RedisClientCache(String uri) {
        this.client = RedisClient.create(uri);
        StatefulRedisConnection<String, String> conn = client.connect();
        
        // 开启追踪
        conn.sync().clientTracking(true);
    }
    
    public String get(String key) {
        // 检查本地缓存
        if (localCache.containsKey(key)) {
            return localCache.get(key);
        }
        
        // 从Redis获取
        String value = client.connect().sync().get(key);
        if (value != null) {
            localCache.put(key, value);
        }
        return value;
    }
}
```

## 六、客户端缓存策略选择

```bash
# 默认模式
CLIENT TRACKING ON
# 所有GET结果自动缓存
# 数据变化时自动失效

# OPTIN模式（推荐）
CLIENT TRACKING ON OPTIN
# 需要手动开启缓存
CLIENT CACHING YES
GET user:1001  # 这个结果会被缓存
GET order:2001 # 这个结果不会被缓存

# OPTOUT模式
CLIENT TRACKING ON OPTOUT
# 默认缓存所有GET结果
CLIENT CACHING NO
GET temp:key   # 这个结果不会被缓存

# 前缀模式
CLIENT TRACKING ON PREFIX user: REDIRECT client-id
# 只追踪user:前缀的Key
```

## 七、性能影响评估

```bash
# 优点
# 1. 减少Redis访问次数
# 2. 降低网络延迟
# 3. 减轻Redis服务器压力

# 缺点
# 1. 增加客户端内存占用
# 2. 追踪信息占用Redis内存
# 3. 网络断开时需要重建缓存
# 4. 缓存一致性有短暂延迟

# 适用场景
# 1. 读多写少的场景
# 2. 对延迟要求极高的场景
# 3. 同一Key频繁访问
# 4. 客户端内存充足
```
