分布式文件系统

一、Google File System（GFS）

GFS是Google在2003年发表的分布式文件系统论文中提出的设计，是大数据存储系统的奠基性工作。

1.1 设计目标与假设

  • 组件故障是常态而非异常（大量廉价服务器）
  • 文件以大块（Chunk，64MB）存储，减少元数据开销
  • 主要负载是大文件的追加写（Append），而非随机写
  • 优化顺序读和追加写的带宽，而非延迟

1.2 架构

┌─────────────┐
│ Master (单点) │ ← 元数据：文件名→Chunk映射、Chunk位置
└──────┬──────┘
        │ 元数据操作
┌──────┼──────┐
▼     ▼     ▼
┌────┐ ┌────┐ ┌────┐
│Chunk│ │Chunk│ │Chunk│ ← 数据存储在ChunkServer
│Srv 1│ │Srv 2│ │Srv 3│
└────┘ └────┘ └────┘

1.3 核心组件

组件职责特点
Master存储元数据、管理Chunk租约、垃圾回收、Chunk迁移单点（实际有Shadow备份），全部元数据存内存
ChunkServer存储Chunk数据（Linux本地文件系统）多个，定期向Master汇报心跳和Chunk状态
Client向Master获取元数据，直接与ChunkServer交互读写数据以库形式链接到应用

1.4 读写流程

读流程：
1. Client → Master：文件名+偏移量 → Master返回Chunk句柄+副本位置
2. Client → ChunkServer：直接读取数据（后续缓存元数据）

写流程（追加）：
1. Client → Master：获取Primary副本和副本位置
2. Client → 所有副本：推送数据到各副本的缓存
3. Client → Primary：发送写请求
4. Primary → 所有Secondary：确定写入顺序，下发写命令
5. 所有副本完成 → Primary → Client：返回成功

1.5 Chunk租约（Lease）

  • Master为每个Chunk指定一个Primary副本，授予60秒租约
  • Primary负责确定所有副本的写入顺序（串行化）
  • 租约到期后可由其他副本接任

二、HDFS（Hadoop Distributed File System）

HDFS是GFS的开源实现，是Hadoop生态系统的核心存储组件。

2.1 架构对比GFS

特性GFSHDFS
主节点MasterNameNode
数据节点ChunkServerDataNode
块大小64MB128MB（2.x默认）
副本数33（默认）
写模型追加+租约流水线写（Pipeline）
元数据单Master内存NameNode内存+EditLog

2.2 NameNode高可用

  • NameNode HA：Active/Standby双NameNode，通过ZKFC（ZK Failover Controller）自动故障切换
  • 共享编辑日志：使用QJM（Quorum Journal Manager）在JournalNode集群中同步EditLog
  • Secondary NameNode：定期合并FsImage和EditLog（非热备，非HA组件）

2.3 副本放置策略

默认3副本放置：
副本1：与客户端同一节点（同机架）
副本2：不同机架的节点
副本3：与副本2同机架的不同节点

目标：平衡可靠性和带宽开销

三、复制与一致性

3.1 一致性模型

一致性级别含义GFS/HDFS策略
强一致性所有副本同时更新不采用（代价高）
顺序一致性所有操作按全局顺序执行Primary串行化写入
最终一致性副本最终趋于一致通过租约和流水线保证

3.2 GFS的一致性保证

  • 文件命名空间的修改是原子的（由Master锁保证）
  • 文件区域在成功写入后是"已定义"（Defined）的
  • 写入失败可能导致"不一致"（Inconsistent）的区域
  • 读者可能会读到过期数据（取决于缓存的Chunk位置信息）

3.3 纠删码（Erasure Coding）

纠删码是副本策略的替代方案，在保证相同可靠性的同时大幅减少存储开销。

RS(k, m) 纠删码：
- k个数据块 + m个校验块
- 可容忍任意m个块的故障
- 存储开销：(k+m)/k，相比3副本的3倍大幅降低
- 例：RS(6,3) 存储开销1.5倍，容忍3块故障
纠删码 vs 副本：
副本：恢复快（直接读副本），带宽开销小，存储开销大
纠删码：恢复慢（需要解码），CPU开销大，存储开销小
实践中通常对热数据用副本，冷数据用纠删码（如HDFS EC、Ceph）

四、知识要点总结

  1. GFS架构：单Master + 多ChunkServer，大块（64MB），3副本
  2. GFS写模型：流水线写 + 租约机制确定Primary
  3. HDFS：GFS开源实现，NameNode/DataNode架构
  4. NameNode HA：Active/Standby + QJM
  5. 副本放置策略：跨机架放置，平衡可靠性和带宽
  6. 纠删码：RS(k,m)，存储开销低，适合冷数据
