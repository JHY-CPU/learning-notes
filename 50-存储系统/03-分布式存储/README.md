# 03-分布式存储

## 1. 分布式存储的挑战

### 1.1 CAP 定理

分布式系统中，以下三个属性最多只能同时满足两个：

- **C（Consistency，一致性）：** 所有节点在同一时间看到相同的数据
- **A（Availability，可用性）：** 每个请求都能在有限时间内获得响应
- **P（Partition Tolerance，分区容忍性）：** 网络分区发生时系统仍能运行

```
        C（一致性）
       /          \
      /    只能    \
     /   选两个    \
    A ──────────── P
 （可用性）    （分区容忍性）
```

### 1.2 CAP 的实际选择

| 选择 | 系统示例 | 说明 |
|------|---------|------|
| CP | ZooKeeper、HBase、MongoDB | 牺牲部分可用性，保证强一致 |
| AP | Cassandra、DynamoDB、CouchDB | 牺牲强一致，保证高可用 |
| CA | 传统 RDBMS（单机） | 无分区容忍性，不适用于分布式 |

### 1.3 BASE 理论

AP 系统通常遵循 BASE：
- **B**asically **A**vailable：基本可用
- **S**oft State：软状态，允许中间状态
- **E**ventually Consistent：最终一致性

### 1.4 一致性模型

| 模型 | 说明 |
|------|------|
| 强一致性 | 写后立即读到新值 |
| 线性一致性 | 强一致性 + 实时性保证 |
| 因果一致性 | 有因果关系的操作保持顺序 |
| 最终一致性 | 经过一段时间后所有副本一致 |
| 会话一致性 | 同一会话内保证一致 |

## 2. 一致性哈希算法

### 2.1 传统哈希的问题

普通哈希 `hash(key) % N` 在节点数 N 变化时，大量 key 需要重新映射。

### 2.2 一致性哈希原理

```
           Node A (0°)
          ╱
    ┌────●──────────┐
    │   ╱  哈希环    │
    │  ╱            │
    ● Node C (240°) │  ← 沿顺时针方向找到最近的节点
    │  ╲            │
    │   ╲           │
    └────●──────────┘
          ╲
           Node B (120°)
```

- 将哈希空间组织为一个环（0 ~ 2^32-1）
- 节点和 key 都映射到环上
- key 顺时针找到第一个节点作为其归属
- 增删节点仅影响相邻 key 范围

### 2.3 虚拟节点

**问题：** 节点数少时分布不均匀。

**解决方案：** 每个物理节点映射多个虚拟节点到环上。

- 物理节点 A → 虚拟节点 A1, A2, ..., Ak
- 提高数据分布均匀性
- 节点失效时负载分散更均匀
- 典型虚拟节点数：100~200

### 2.4 一致性哈希的改进

- **Jump Consistent Hash (Google)：** O(log n) 时间，零内存消耗
- **Maglev Consistent Hash：** Google LB 使用，快速查找表
- **Ketama：** 经典虚拟节点实现，广泛用于 Memcached

## 3. 数据复制策略

### 3.1 同步复制 vs 异步复制

| 特性 | 同步复制 | 异步复制 |
|------|---------|---------|
| 数据一致性 | 强一致 | 最终一致 |
| 写延迟 | 高（等待所有副本确认） | 低 |
| 可用性 | 受最慢副本影响 | 高 |
| 适用场景 | 金融、关键数据 | 高性能、高可用 |

### 3.2 半同步复制

- 写入时等待部分副本（而非全部）确认
- 如 3 副本中等待 2 个确认（Quorum）
- 平衡一致性和性能

### 3.3 Quorum 机制

设 N = 副本总数，W = 写成功所需确认数，R = 读所需副本数：

- **强一致条件：** W + R > N
- **典型配置：**
  - N=3, W=2, R=2（平衡型）
  - N=3, W=1, R=3（写快读慢）
  - N=3, W=3, R=1（写慢读快）

### 3.4 复制拓扑

| 拓扑 | 说明 |
|------|------|
| 主从（Master-Slave） | 写主读从，简单但主节点是瓶颈 |
| 主主（Master-Master） | 多节点可写，需解决冲突 |
| 多主（Multi-Master） | 跨数据中心场景 |
| 无主（Leaderless） | 所有节点等价（如 Dynamo、Cassandra） |

## 4. 纠删码（Erasure Coding）

### 4.1 基本概念

纠删码将数据编码为 n 个片段，只需其中任意 k 个即可恢复原始数据（k < n）。

- **编码率：** k/n（原始数据占比）
- 对比复制：3 副本容量利用率 33%，纠删码(8+4)利用率 67%

### 4.2 Reed-Solomon 编码

最常见的纠删码算法：

```
原始数据: D1, D2, ..., Dk
校验数据: P1, P2, ..., Pm（其中 m = n - k）

P = D × G（生成矩阵）

恢复：丢失 ≤ m 个片段后可通过矩阵求解恢复
```

**常见配置：**

| 配置 | 原始数据 | 校验数据 | 容错 | 容量利用率 |
|------|---------|---------|------|-----------|
| 4+2 | 4 | 2 | 2 盘 | 67% |
| 8+3 | 8 | 3 | 3 盘 | 73% |
| 10+4 | 10 | 4 | 4 盘 | 71% |

### 4.3 纠删码 vs 复制

| 特性 | 复制 | 纠删码 |
|------|------|--------|
| 容量开销 | 高（3x） | 低（1.2~1.5x） |
| 读性能 | 快（任一副本） | 需解码 |
| 写性能 | 快 | 需编码计算 |
| 修复开销 | 复制整份数据 | 需读多个片段+解码 |
| 适用 | 热数据、元数据 | 冷数据、大对象 |

## 5. GFS / HDFS 架构

### 5.1 GFS（Google File System）

Google 的分布式文件系统，面向大文件、追加写、高吞吐。

**架构：**

```
┌──────────────┐
│   Master     │ ← 存储元数据（命名空间、Chunk 映射、位置信息）
│  (单点)      │
└──────┬───────┘
       │
  ┌────┼────┐
  ▼    ▼    ▼
┌───┐┌───┐┌───┐
│C1 ││C2 ││C3 │  ← Chunk Server（存储 64MB 数据块）
└───┘└───┘└───┘
```

**关键设计：**
- 文件被切分为固定大小的 Chunk（默认 64MB）
- 每个 Chunk 默认 3 副本
- Master 管理元数据，Chunk Server 管理数据
- 应用场景：大数据批处理（MapReduce）

### 5.2 HDFS（Hadoop Distributed File System）

GFS 的开源实现，Hadoop 生态的核心存储层。

**与 GFS 的主要区别：**
- NameNode（主节点）支持 HA（高可用）
- 支持联邦（Federation），多个 NameNode 管理不同命名空间
- Secondary NameNode 辅助 checkpoint
- 默认副本数 3，机架感知放置策略

**机架感知副本放置（3 副本）：**
- 副本 1：客户端所在节点
- 副本 2：不同机架的节点
- 副本 3：与副本 2 同机架的不同节点

### 5.3 HDFS 读写流程

**写入流程：**
1. 客户端向 NameNode 请求写入
2. NameNode 返回可写入的 DataNode 列表
3. 客户端建立 Pipeline，数据以 Packet 为单位串行写入
4. DataNode 之间级联复制
5. 客户端收到所有 ACK 后确认写入成功

**读取流程：**
1. 客户端向 NameNode 请求读取
2. NameNode 返回 Block 位置列表（按网络拓扑排序）
3. 客户端就近读取 DataNode
4. 校验和验证数据完整性

## 6. 对象存储（S3 兼容接口）

### 6.1 对象存储概念

对象存储以对象（Object）为单位管理数据，扁平命名空间。

**对比：**

| 特性 | 文件系统 | 块存储 | 对象存储 |
|------|---------|--------|---------|
| 访问方式 | 文件路径 | LBA | REST API / Object ID |
| 元数据 | 有限（inode） | 无 | 丰富、可自定义 |
| 扩展性 | 中等 | 低 | 极高 |
| 一致性 | 强 | 强 | 最终一致（通常） |
| 典型应用 | 通用文件 | 数据库、VM | 图片、视频、备份 |

### 6.2 S3 接口

Amazon S3（Simple Storage Service）已成为对象存储的事实标准。

**核心概念：**
- **Bucket：** 容器，全局唯一命名
- **Object：** 数据对象 = Key（名称）+ Value（数据）+ Metadata
- **扁平命名：** Key 看似有路径（`a/b/c.txt`），实际是扁平字符串

**常用 API：**
```
PUT    /{bucket}/{key}     → 上传对象
GET    /{bucket}/{key}     → 下载对象
DELETE /{bucket}/{key}     → 删除对象
LIST   /{bucket}?prefix=   → 列出对象
```

### 6.3 S3 兼容实现

| 实现 | 说明 |
|------|------|
| MinIO | 高性能、云原生、Go 实现 |
| Ceph RGW | Ceph 的对象存储网关 |
| Swift | OpenStack 对象存储 |
| SeaweedFS | 简单高性能分布式文件系统 |
| Garage | Rust 实现的分布式对象存储 |

### 6.4 对象存储的存储后端

对象存储系统底层通常使用：
- 本地文件系统（如 XFS、ext4）
- 分布式块存储
- 专用元数据存储（如 RocksDB）

## 7. 分布式块存储（Ceph RBD）

### 7.1 Ceph 概述

Ceph 是统一的分布式存储系统，提供块、对象、文件三种接口。

**核心组件：**

```
┌─────────────────────────────────┐
│            客户端                │
│   ┌──────┬──────┬──────────┐   │
│   │ RBD  │ RGW  │ CephFS   │   │
│   │ 块   │ 对象  │ 文件     │   │
│   └──┬───┴──┬───┴────┬─────┘   │
│      │      │        │         │
│  ┌───┴──────┴────────┴────┐    │
│  │        RADOS            │    │ ← 可靠的自主分布式对象存储
│  │  ┌─────┐  ┌─────────┐  │    │
│  │  │ OSD │  │ Monitor │  │    │
│  │  └─────┘  └─────────┘  │    │
│  └────────────────────────┘    │
└─────────────────────────────────┘
```

### 7.2 CRUSH 算法

Ceph 的数据分布算法：
- **伪随机：** 无需中心元数据表，客户端可自行计算数据位置
- **拓扑感知：** 考虑机架、数据中心等物理拓扑
- **权重控制：** 按 OSD 容量分配权重
- 输入：Pool ID + Object ID + CRUSH Map
- 输出：OSD 列表（主、从）

### 7.3 Ceph RBD（RADOS Block Device）

- 将 RADOS 对象存储暴露为块设备
- 可通过内核模块或 librbd 访问
- 支持精简配置（Thin Provisioning）
- 支持快照和克隆（COW）
- 广泛用于 OpenStack/Kubernetes 虚拟机磁盘

### 7.4 Pool 和 PG

- **Pool：** 逻辑分区，管理副本数、纠删码配置
- **PG（Placement Group）：** Pool 内的管理单元，一组对象映射到同一 PG
- 对象 → PG → OSD，两层映射

## 8. 分布式文件系统

### 8.1 CephFS

Ceph 的 POSIX 兼容分布式文件系统：
- 基于 RADOS 存储数据
- MDS（Metadata Server）管理元数据
- 支持多 MDS 活跃（提升元数据性能）
- 强一致性（通过 MDS 锁机制）

### 8.2 GlusterFS

Red Hat 开源的分布式文件系统：
- 无中心元数据节点（纯对等架构）
- 通过弹性哈希算法定位文件
- 卷类型：
  - **Distributed：** 数据分散在各节点
  - **Replicated：** 数据复制
  - **Striped：** 条带化
  - **Distributed-Replicated：** 分布式+复制

### 8.3 其他分布式文件系统

| 系统 | 特点 |
|------|------|
| Lustre | 高性能计算（HPC）领域主流 |
| MooseFS | 简单易用，元数据服务器单点 |
| MinIO | S3 兼容，云原生设计 |
| JuiceFS | 云原生，元数据与数据分离 |

### 8.4 文件系统对比

| 特性 | HDFS | CephFS | GlusterFS |
|------|------|--------|-----------|
| 元数据 | 中心化 NameNode | MDS（可多活） | 无中心 |
| API | Hadoop API | POSIX | POSIX |
| 适合场景 | 大数据分析 | 通用存储 | 通用存储 |
| 小文件 | 差 | 好 | 中等 |
| 扩展性 | 受 NameNode 限制 | 好 | 好 |

## 9. 存储虚拟化

### 9.1 概念

存储虚拟化将物理存储资源抽象为逻辑存储池，屏蔽底层硬件差异。

```
┌─────────────────────────┐
│    虚拟存储层（抽象）     │
│  ┌─────┬─────┬──────┐  │
│  │ LUN │ FS  │ Object│  │  ← 统一接口
│  └──┬──┴──┬──┴───┬───┘  │
│     │     │      │      │
│  ┌──┴─────┴──────┴───┐  │
│  │ 虚拟化引擎          │  │
│  └──┬─────┬──────┬───┘  │
│     │     │      │      │
└─────┼─────┼──────┼──────┘
  ┌───┴┐ ┌──┴──┐ ┌┴─────┐
  │SAN │ │NAS  │ │本地盘  │
  └────┘ └─────┘ └──────┘
```

### 9.2 虚拟化层次

| 层次 | 方式 | 示例 |
|------|------|------|
| 块级虚拟化 | LUN 虚拟化、SAN 虚拟化 | vSAN、Storage Spaces |
| 文件级虚拟化 | 统一命名空间 | DFS、Global Namespace |
| 对象级虚拟化 | 统一对象接口 | S3 网关 |

### 9.3 软件定义存储（SDS）

- 将存储控制面与数据面分离
- 控制面用软件管理，运行在标准服务器上
- 支持异构硬件
- 典型产品：VMware vSAN、Ceph、MinIO

### 9.4 云存储与存储虚拟化

- **块存储：** AWS EBS、Azure Managed Disks
- **文件存储：** AWS EFS、Azure Files
- **对象存储：** AWS S3、Azure Blob、Google Cloud Storage
- 云平台天然实现了存储虚拟化和按需分配
