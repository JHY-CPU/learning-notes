# 事件响应流程


## 事件响应流程


SRE事件响应On-Call


标准化的事件响应流程是降低MTTR（平均恢复时间）的关键。


## 事件严重等级定义


```
严重等级分级：

┌──────┬──────────┬──────────────────────────────────────────┐
│ 等级  │ 名称       │ 定义                                      │
├──────┼──────────┼──────────────────────────────────────────┤
│ P1   │ 严重(Critical)│ 核心功能完全不可用，影响全部用户            │
│      │          │ 例：支付系统故障、全站宕机                   │
│      │          │ 响应：5分钟内响应，15分钟内上线War Room      │
│      │          │ 升级：立即通知VP/CTO                        │
├──────┼──────────┼──────────────────────────────────────────┤
│ P2   │ 高(High)     │ 核心功能严重降级，影响大部分用户             │
│      │          │ 例：搜索不可用、下单成功率<90%               │
│      │          │ 响应：15分钟内响应                          │
│      │          │ 升级：通知技术负责人                         │
├──────┼──────────┼──────────────────────────────────────────┤
│ P3   │ 中(Medium)   │ 非核心功能异常或核心功能轻微降级             │
│      │          │ 例：评论加载慢、推荐不准确                   │
│      │          │ 响应：1小时内响应                           │
│      │          │ 升级：团队内部处理                           │
├──────┼──────────┼──────────────────────────────────────────┤
│ P4   │ 低(Low)      │ 微小问题，不影响用户体验                    │
│      │          │ 例：UI显示异常、非关键日志报错               │
│      │          │ 响应：工作时间内处理                         │
│      │          │ 升级：正常工作流                             │
└──────┴──────────┴──────────────────────────────────────────┘

等级判定标准：
- 影响范围：用户数/收入/核心功能
- 影响程度：完全不可用/部分降级/体验下降
- 业务影响：收入损失/合规风险/品牌影响

模糊情况处理：
- 宁高勿低：不确定时定高一级
- 随时调整：发现影响扩大可升级
- 事后复盘：校准等级判定标准
```


## 事件响应流程


```
六步事件响应流程：

1. 检测 (Detect)
   触发源：告警系统 / 用户反馈 / 监控发现
   - 自动告警：PagerDuty / 飞书告警 / Opsgenie
   - 人工发现：客服工单、群内反馈

2. 响应 (Respond)
   On-Call人员第一响应：
   - 确认收到告警（Acknowledge）
   - 初步判断影响范围和严重等级
   - 升级（如需要）：拉群、呼叫其他on-call
   - 创建事件工单，开始记录时间线

3. 诊断 (Diagnose)
   快速定位问题：
   - 检查最近变更（部署、配置修改）
   - 查看监控面板和日志
   - 检查依赖服务状态
   - 缩小影响范围（隔离、降级、限流）

4. 修复 (Fix)
   恢复服务优先：
   - 回滚最近变更（最快方案）
   - 扩容 / 重启
   - 切换降级方案
   - 热修复（如有把握）

5. 验证 (Verify)
   确认问题已解决：
   - 监控指标恢复正常
   - 用户反馈确认
   - 功能测试验证

6. 复盘 (Postmortem)
   事后分析和改进（见下节）

War Room（作战室）：
- 物理/虚拟空间，P1/P2事件需开设
- 明确角色分工：
  ┌──────────────┬──────────────────────────────┐
  │ 角色           │ 职责                          │
  ├──────────────┼──────────────────────────────┤
  │ 事件指挥官(IC) │ 协调资源、做决策、控制节奏     │
  │ 通讯负责人(CL) │ 对外沟通、更新状态页           │
  │ 操作员(Ops)    │ 执行诊断和修复操作             │
  │ 领域专家(SME)  │ 提供技术知识支持               │
  │ 记录员(Scribe) │ 记录时间线和操作日志           │
  └──────────────┴──────────────────────────────┘

沟通模板（状态更新）：
【事件通知 - P1】
时间：2024-01-15 14:30
影响：支付系统不可用
状态：排查中
负责人：@张三
下次更新：15分钟后
```


## Postmortem（事后复盘）


```
Postmortem模板：

# 事件复盘报告

## 基本信息
- 事件ID：INC-2024-0115-001
- 严重等级：P1
- 持续时间：45分钟（14:30 - 15:15）
- 影响范围：支付服务，约12000笔订单受影响
- 复盘日期：2024-01-16
- 参与人员：张三(IC)、李四(Ops)、王五(SME)

## 时间线
| 时间 | 事件 | 操作人 |
|------|------|--------|
| 14:25 | 部署v2.3.1到生产环境 | CI/CD |
| 14:30 | 支付成功率告警触发 | 系统 |
| 14:32 | On-Call确认告警 | 张三 |
| 14:35 | 拉起War Room | 张三 |
| 14:40 | 确认是v2.3.1引起 | 李四 |
| 14:45 | 开始回滚 | 李四 |
| 15:00 | 回滚完成，验证通过 | 李四 |
| 15:15 | 确认服务完全恢复 | 张三 |

## 根因分析 (5 Whys)
1. 为什么支付失败？→ 新版本中数据库连接池配置错误
2. 为什么配置错误？→ 配置值在测试环境和生产环境不同
3. 为什么环境不同？→ 测试环境使用的是默认配置
4. 为什么用默认配置？→ 没有环境一致性检查机制
5. 为什么没有检查？→ 缺少配置漂移检测流程

根因：配置管理流程缺陷，测试环境未覆盖生产配置差异

## 影响评估
- 用户影响：~12000笔支付失败
- 收入损失：约50万元
- SLA影响：月度可用性从99.95%降至99.91%

## 改进项（Action Items）
| 序号 | 改进措施 | 负责人 | 截止日期 | 优先级 |
|------|---------|--------|---------|--------|
| 1 | 建立配置中心，统一管理环境配置 | 李四 | 01-30 | P0 |
| 2 | 添加金丝雀发布，逐步放量 | 王五 | 02-15 | P0 |
| 3 | 部署前配置diff检查 | 张三 | 02-01 | P1 |
| 4 | 支付服务添加配置热更新 | 赵六 | 02-28 | P1 |
| 5 | 增加集成测试覆盖生产配置 | 钱七 | 03-15 | P2 |

## 经验教训
做得好的：
- 告警在5分钟内触发
- War Room快速建立
- 回滚决策果断

需要改进的：
- 配置管理缺乏统一管控
- 金丝雀发布覆盖不足
- 支付服务缺少自动回滚机制

Postmortem原则：
- 无责文化 (Blameless)：关注系统改进而非个人过失
- 每个P1/P2事件必须有Postmortem
- Action Items必须有owner和deadline
- 定期回顾Action Items完成情况
```


> **Note:** 事件响应的核心是"恢复优先，根因其次"。P1事件要求5分钟内响应，War Room中需明确角色分工。Postmortem必须是无责文化(Blameless)，关注系统改进而非个人失误。每个Action Items必须有明确的owner和deadline。
