# 容量规划 - 可观测性与监控笔记


## 容量规划


## 一、容量规划概述


容量规划是预测未来资源需求并确保系统在预期负载下正常运行的过程。目标是在成本和可靠性之间找到平衡点。


- **过度配置：**
   浪费成本，资源闲置
- **配置不足：**
   服务降级，用户体验受损
- **目标：**
   以最低成本满足性能 SLA


## 二、容量建模


### 1. 自顶向下建模


```
业务指标 → 技术指标 → 资源需求

示例：电商平台
业务增长预测：下季度 DAU 从 100 万 → 150 万（增长 50%）
  │
  ├── 每用户日均请求：10 次
  ├── 峰值倍率：3x（集中在晚间）
  └── 计算：
      日请求量 = 150 万 × 10 = 1500 万/天
      QPS 均值 = 1500 万 / 86400 ≈ 174 QPS
      QPS 峰值 = 174 × 3 ≈ 522 QPS
      单机处理能力 = 200 QPS（已知）
      需要机器数 = 522 / 200 = 3 台
      考虑冗余（N+1）= 4 台
```


### 2. 自底向上建模


```
资源指标 → 容量上限

当前资源使用率监控数据：
- CPU 峰值使用率：65%
- 内存峰值使用率：70%
- 磁盘 IOPS 使用率：40%
- 网络带宽使用率：30%

瓶颈资源 = CPU（65% 最接近上限）
剩余容量 = (100% - 65%) / 65% = 53.8% 的增长空间
当前 QPS = 300
理论最大 QPS = 300 / 0.65 ≈ 461 QPS
```


### 3. 性能建模公式


```
利特尔法则 (Little's Law):
L = λ × W
L = 系统中平均请求数
λ = 平均到达速率（QPS）
W = 平均响应时间

示例：
QPS = 1000, 平均响应时间 = 50ms
系统中并发请求 = 1000 × 0.05 = 50 个

排队论 (M/M/c 模型):
用于估算在给定到达率和服务率下，需要多少服务器
到达率 λ = 1000 req/s
单机服务率 μ = 200 req/s
最少服务器数 = λ/μ = 5 台（100% 利用率）
实际需要 > 5 台以保持合理的利用率和队列长度
```


## 三、压测验证容量


### 1. 压测类型


| 类型 | 目的 | 方法 | 适用阶段 |
| --- | --- | --- | --- |
| 负载测试 | 验证预期负载下性能 | 逐步增加到目标 QPS | 常规验证 |
| 压力测试 | 找到系统极限 | 持续增加直到崩溃 | 容量评估 |
| 浸泡测试 | 发现内存泄漏等问题 | 中等负载长时间运行 | 稳定性验证 |
| 尖峰测试 | 验证突发流量处理 | 瞬间冲击式加压 | 大促前 |


### 2. 压测工具


```
# wrk - 轻量级 HTTP 压测

# -t12: 12 线程, -c400: 400 并发连接, -d30s: 持续 30 秒

# k6 - 现代压测工具（JavaScript 脚本）
k6 run --vus 100 --duration 30s script.js

# Locust - Python 分布式压测
locust -f locustfile.py --headless -u 1000 -r 100 -t 5m

# 生成详细 HTML 报告

# JMeter - 企业级压测平台
# GUI 配置 + CLI 执行
```


### 3. 全链路压测


```
全链路压测 vs 单接口压测：
┌────────┐   ┌────────┐   ┌────────┐   ┌────────┐
│ 网关   │ → │ 服务A  │ → │ 服务B  │ → │ 数据库  │
└────────┘   └────────┘   └────────┘   └────────┘
     ↑ 单接口：只压测服务A的单个接口
     ↑ 全链路：按真实流量比例，同时压测整条链路

全链路压测关键：
1. 流量录制回放：录制线上流量，在影子环境回放
2. 影子数据：压测数据打标，不影响真实业务数据
3. 链路标记：Header 中传递压测标记，各服务透传
4. 实时监控：压测过程中实时观察各节点指标
```


## 四、余量规划（Headroom）


### 1. 余量计算


```
余量 = (容量上限 - 当前使用) / 当前使用 × 100%

目标余量（Google SRE 推荐）：
- 峰值负载后至少保留 50% 余量
- 意味着系统应在 50% 利用率下运行
- 剩余 50% 应对：突发流量 + 机器故障 + 滚动部署

示例：
当前峰值 QPS = 500
目标余量 = 50%
规划容量 = 500 / (1 - 0.5) = 1000 QPS
需要机器 = 1000 / 单机能力
```


### 2. N+1 冗余


```
// 集群容量规划
假设需要 4 台服务器满足流量需求
实际部署 4 + 1 = 5 台（N+1 冗余）
任意 1 台故障 → 剩余 4 台仍可服务

// Google 推荐的 N+2
// 任意 2 台同时故障仍可服务
// 适用关键服务
```


## 五、Auto-Scaling 策略


### 1. 扩缩容触发条件


```
# 指标驱动（推荐）
scale_up:
  when:
    - CPU > 70% for 5min
    - memory > 80% for 5min
    - queue_depth > 1000 for 3min
    - response_time_p99 > 2000ms for 5min
  action:
    - add_instances(count=2)
    - cooldown(10min)

scale_down:
  when:
    - CPU < 30% for 15min
    - memory < 40% for 15min
  action:
    - remove_instances(count=1)
    - cooldown(15min)
```


### 2. 预测性扩容（Predictive Scaling）


```
// 基于历史数据预测未来流量，提前扩容

// 示例：每天 20:00 流量高峰
// 预测性策略：19:30 开始扩容，22:00 开始缩容

AWS Auto Scaling 预测性扩容：
- 分析过去 14 天的流量模式
- 预测未来 48 小时的容量需求
- 提前扩容避免冷启动延迟
```


### 3. Kubernetes HPA 配置


```
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
    - type: Pods           # 自定义指标
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "1000"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Percent
          value: 50          # 每次最多扩容 50%
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 25          # 每次最多缩容 25%
          periodSeconds: 120
```


## 六、容量规划流程


```
1. 收集基线数据
   ├── 历史流量趋势（季度/月度增长）
   ├── 资源使用率峰值
   └── 性能指标（RT、错误率）

2. 需求预测
   ├── 业务增长预测（新用户、活动）
   ├── 季节性因素（节假日、大促）
   └── 技术变更（新功能上线）

3. 容量建模
   ├── 瓶颈资源识别
   ├── 扩展因子计算
   └── 成本估算

4. 压测验证
   ├── 全链路压测
   ├── 故障注入测试
   └── 结果分析

5. 制定计划
   ├── 扩容时间表
   ├── 预算审批
   └── 自动化策略

6. 持续监控
   ├── 用量跟踪仪表盘
   ├── 告警阈值更新
   └── 季度回顾调整
```


> **Note:** **面试要点：**
> 容量规划核心是自顶向下（业务→技术→资源）和自底向上（资源→容量）两种建模方式。Google 推荐峰值后保留 50% 余量，N+1 或 N+2 冗余。Auto-scaling 要考虑冷却时间和缩容速度（缩容应比扩容慢）。全链路压测是验证容量的金标准。利特尔法则 L = lambda * W 是基本工具。

总结：容量规划目标是在成本和可靠性间平衡；建模方式分自顶向下和自底向上；利特尔法则是基础计算工具；余量规划推荐峰值后保留50%余量+N+1冗余；压测分负载/压力/浸泡/尖峰四种类型，全链路压测是金标准；Auto-scaling需考虑指标选择、冷却时间、预测性扩容；Kubernetes HPA支持多指标和行为配置
