# 02-SRE实践

## SRE（Site Reliability Engineering）概述

### Google SRE 理念

SRE 由 Google 提出，核心定义：

> "SRE is what happens when you ask a software engineer to design an operations team."

核心原则：
- **用工程方法解决运维问题**：自动化替代人工操作（Toil），用代码管理基础设施。
- **Error Budget**：允许一定程度的故障，将"可靠性"转化为可量化的指标。
- **50% 时间做工程**：SRE 团队至少一半时间用于开发工具和自动化，而非灭火。
- **减少 Toil**：重复性、手动、可自动化的工作应被消除。
- **事后复盘（Blameless Postmortem）**：关注系统改进而非追究个人责任。

### SRE vs DevOps

| 维度 | DevOps | SRE |
|------|--------|-----|
| 目标 | 打破 Dev 和 Ops 的壁垒 | 用工程手段保障可靠性 |
| 角色 | 文化和实践，无特定角色 | 有明确的 SRE 团队和岗位 |
| 关注点 | CI/CD、自动化、协作 | 可靠性、容量、应急响应 |
| 关系 | SRE 是 DevOps 的一种具体实践形式 | |

## SLI/SLO/SLA

### SLI 定义

SLI（Service Level Indicator）是衡量服务健康程度的具体指标。

常见 SLI：
- **可用性**：`成功请求数 / 总请求数 × 100%`，例如 99.95%。
- **延迟**：P99 请求延迟 < 200ms。
- **吞吐量**：系统能处理的 QPS 上限。
- **错误率**：`错误请求数 / 总请求数`，例如 < 0.1%。
- **数据持久性**：数据丢失的概率（如 99.999999999%）。

SLI 应直接反映用户体验，而非系统内部指标（如 CPU 利用率）。

### SLO 设定方法

SLO（Service Level Objective）是 SLI 的目标值。

设定原则：
- 基于用户期望而非技术能力设定。
- 从现有水平出发，逐步提升（不要一开始就承诺 99.999%）。
- 保留缓冲区：SLO 应高于 SLA（商业承诺），留出安全余量。
- SLO 应有明确的计算窗口（如滚动 30 天、按月计算）。

示例：
- 可用性 SLO：99.95%（30 天窗口内最多 21.9 分钟不可用）。
- 延迟 SLO：P99 < 300ms（99% 的请求在 300ms 内完成）。

### Error Budget（错误预算）

Error Budget = 1 - SLO。例如 SLO 为 99.9%，则 Error Budget 为 0.1%。

核心理念：
- **Error Budget 是团队的"容错额度"**，在预算耗尽前可以冒险发布新功能。
- **预算耗尽后冻结发布**，团队专注于可靠性改进，直到预算恢复。
- 将"稳定性 vs 速度"的矛盾转化为可量化的决策依据。

计算示例：
- SLO = 99.95%，30 天窗口，总请求数 1,000,000。
- Error Budget = 1,000,000 × 0.0005 = 500 个错误请求。
- 如果本月已消耗 480 个错误，预算剩余 4%，需要收紧发布节奏。

### SLA 的商业含义

SLA（Service Level Agreement）是服务提供商与客户之间的商业合约。

- 违反 SLA 可能导致赔偿、退款、合同终止。
- SLA 通常比 SLO 更宽松（SLO 内部目标 > SLA 商业承诺）。
- 例如：内部 SLO 99.99%，对外 SLA 承诺 99.9%。

## 告警管理

### 告警策略

- **基于阈值**：指标超过固定值触发告警（如 CPU > 90%）。简单直接，适合已知问题。
- **基于异常检测**：使用统计方法或 ML 检测异常模式（如偏离历史基线 3 个标准差）。适合未知故障模式。

### 告警分级

| 级别 | 含义 | 响应时间 | 示例 |
|------|------|----------|------|
| **P1** | 核心功能不可用，影响全部用户 | 5 分钟内响应 | 全站不可用、数据丢失 |
| **P2** | 核心功能严重降级，影响部分用户 | 30 分钟内响应 | 支付失败率升高、延迟飙升 |
| **P3** | 非核心功能异常，影响少量用户 | 4 小时内响应 | 报表生成超时、非关键接口错误 |
| **P4** | 需关注但不紧急 | 下个工作日处理 | 存储空间预警、证书即将过期 |

### 告警收敛与抑制

- **分组（Grouping）**：同一来源的多条告警合并为一条通知，避免告警风暴。
- **抑制（Inhibition）**：高级别告警抑制低级别告警。例如全站不可用时，抑制单个服务的超时告警。
- **静默（Silence）**：计划内维护期间临时屏蔽告警。
- **去重（Deduplication）**：相同告警在恢复前只通知一次。

### 值班与升级机制

- **一级值班**：收到告警后的第一响应人，负责初步排查和止血。
- **二级升级**：一级无法解决时升级到领域专家（如数据库 DBA、网络工程师）。
- **三级升级**：二级无法解决时升级到架构师或管理层。
- 轮值周期：通常每周轮换，避免个人过度疲劳。
- 使用 PagerDuty、Opsgenie 等工具管理值班排期和升级链路。

### 告警疲劳与治理

告警疲劳是指过多、重复、低质量的告警导致值班人员麻木，错过真正重要的告警。

治理措施：
- 定期审查告警规则，删除无用/重复告警。
- 每条告警必须有明确的 Runbook（处理手册）。
- 跟踪告警的 Action Rate（告警后采取行动的比例），低于 50% 的告警应重新评估。
- 逐步引入告警质量评分，淘汰"狼来了"式告警。

## 可视化

### Grafana 仪表板设计

- **分层展示**：顶部放全局概览（RED 指标），下方按服务/组件展开详情。
- **关联跳转**：从 Metrics 仪表板跳转到对应 Logs 或 Traces。
- **变量（Variables）**：使用下拉变量选择环境、服务、实例，一个仪表板适用于所有场景。
- **自动刷新**：关键仪表板设置 10-30 秒自动刷新。

### 黄金指标仪表板

Google SRE 提出的四个黄金信号：

| 指标 | 含义 | PromQL 示例 |
|------|------|-------------|
| **Latency** | 请求耗时 | `histogram_quantile(0.99, rate(request_duration_seconds_bucket[5m]))` |
| **Traffic** | 请求量 | `sum(rate(requests_total[5m]))` |
| **Errors** | 错误率 | `sum(rate(requests_total{status=~"5.."}[5m])) / sum(rate(requests_total[5m]))` |
| **Saturation** | 资源饱和度 | `node_load1 / count(node_cpu_seconds_total{mode="idle"})` |

### 业务指标仪表板

- 核心业务指标：订单量、支付成功率、新用户注册数、DAU/MAU。
- 业务指标与技术指标关联：当支付成功率下降时，能快速关联到对应的微服务延迟。
- 告警到仪表板的联动：告警触发时自动跳转到相关仪表板的时间段。

## On-Call 机制

### 轮值制度

- 通常采用每周轮换制，每位工程师轮值一周。
- Primary + Backup 双人值班，Primary 为第一响应人。
- 假期和特殊时期需提前安排代班。
- 使用专业工具（PagerDuty、Opsgenie）管理排期和通知。

### 事故响应流程

1. **收到告警**：值班工程师在规定时间内确认（P1: 5 分钟，P2: 30 分钟）。
2. **初步评估**：判断影响范围和严重程度，确定事故等级。
3. **创建事故工单**：记录时间线、影响、初步分析。
4. **拉起响应团队**：P1/P2 事故拉起 War Room（Slack 频道或会议室）。
5. **止血优先**：回滚、扩容、降级等手段先恢复服务，再排查根因。
6. **持续沟通**：向利益相关方（客户、管理层）同步进展。
7. **恢复确认**：确认服务恢复正常，关闭事故。

### 事后复盘（Postmortem）

- **Blameless（无责）**：关注系统改进而非追究个人责任。
- 内容包括：时间线、影响、根因分析（5 Whys）、修复措施、预防措施。
- 每条改进措施需有 Owner 和 Deadline。
- 复盘文档共享给全团队，作为学习材料。
- 定期回顾改进措施的执行情况。

## 混沌工程

### 原则与实践

混沌工程由 Netflix 提出，核心思想：**通过主动注入故障来发现系统的薄弱点**。

五大原则（Principles of Chaos Engineering）：
1. 建立稳态假说（Steady State Hypothesis）。
2. 引入真实世界的事件（如服务器宕机、网络延迟、磁盘填满）。
3. 在生产环境中进行实验。
4. 最小化爆炸半径（从最小影响范围开始）。
5. 自动化实验，持续运行。

### 故障注入工具

| 工具 | 特点 |
|------|------|
| **Chaos Monkey** | Netflix 出品，随机终止生产环境中的实例 |
| **LitmusChaos** | CNCF 项目，Kubernetes 原生，丰富的 ChaosHub 实验库 |
| **ChaosBlade** | 阿里开源，支持 JVM、容器、K8s、OS 多层次故障注入 |
| **Gremlin** | 商业化混沌工程平台，UI 友好，企业级安全控制 |

### 混沌实验设计

1. **定义稳态**：确定系统正常运行时的关键指标（如 P99 延迟 < 200ms，错误率 < 0.1%）。
2. **设计实验**：选择要注入的故障类型（杀进程、增加延迟、丢包、CPU 打满）。
3. **控制爆炸半径**：从非生产环境开始，逐步扩展到生产环境的小范围。
4. **执行实验**：注入故障，观察系统行为和指标变化。
5. **验证假设**：系统是否按预期降级/恢复？是否有意外的级联故障？
6. **改进系统**：根据实验结果修复薄弱点。
7. **持续实验**：将混沌实验集成到 CI/CD 管道中。

## SLO 驱动的发布

SLO 驱动的发布是将 Error Budget 与发布策略结合的实践：

- **绿灯阶段**：Error Budget 充足（> 50%），可以正常发布新功能。
- **黄灯阶段**：Error Budget 消耗过半（20%-50%），收紧发布节奏，增加 Code Review 和测试覆盖。
- **红灯阶段**：Error Budget 耗尽（< 20%），冻结所有非关键发布，全队投入可靠性改进。

实现方式：
- 在 CI/CD 管道中加入 SLO 检查 Gate。
- Canary 发布 + 自动回滚：新版本先灰度 1% 流量，观察 SLO 指标，异常自动回滚。
- Feature Flag：新功能通过开关控制，随时可以关闭有问题的功能而不回滚整个版本。
- 渐进式发布（Progressive Delivery）：Canary -> 10% -> 50% -> 全量，每个阶段检查 SLO 指标。
