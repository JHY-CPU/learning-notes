# SLO与错误预算


## SLO与错误预算


SRESLO混沌工程


SRE(站点可靠性工程)通过SLO和错误预算平衡可靠性和迭代速度。


## SLI / SLO / SLA


```
SLI (Service Level Indicator) 服务水平指标：
- 可用性 = 成功请求数 / 总请求数
- 延迟 = p99响应时间 < 200ms
- 吞吐量 = TPS
- 错误率 = 5xx / 总请求
- 正确性 = 返回正确结果的比例

SLO (Service Level Objective) 服务水平目标：
- 可用性 > 99.9%（每月停机43.2分钟）
- p99延迟 < 200ms
- 错误率 < 0.1%

SLA (Service Level Agreement) 服务水平协议：
- 与客户签订的正式合同
- 违反SLA的经济赔偿
- 通常SLO > SLA（留有余量）

示例：
┌─────────────────────────────────────────────┐
│  SLI：API可用性 = 成功响应/总请求             │
│  SLO：30天窗口内可用性 ≥ 99.9%               │
│  SLA：低于99.9%退还10%月费                   │
│                                             │
│  可用性目标：                                 │
│  99%   → 年停机3.65天                        │
│  99.9% → 年停机8.76小时                      │
│  99.95%→ 年停机4.38小时                      │
│  99.99%→ 年停机52.6分钟                      │
└─────────────────────────────────────────────┘
```


## 错误预算 (Error Budget)


```
错误预算 = 1 - SLO = 允许的不可靠程度

示例：SLO = 99.9%可用性
错误预算 = 0.1% × 30天 × 24小时 × 60分钟 = 43.2分钟/月

错误预算的使用：
┌─────────────────────────────────────────────┐
│  预算充足（>50%剩余）                         │
│  → 团队可以自由发布新功能                     │
│  → 可以进行有风险的变更                       │
│                                             │
│  预算紧张（<50%剩余）                         │
│  → 放慢发布节奏                               │
│  → 增加变更审查                               │
│                                             │
│  预算耗尽（=0）                               │
│  → 冻结发布（只允许紧急修复）                 │
│  → 全力投入可靠性改进                         │
│  → 直到预算恢复                               │
└─────────────────────────────────────────────┘

错误预算的优势：
- 量化了"多可靠才算够可靠"
- 给团队一个客观的发布信号
- 平衡了创新速度和稳定性
- 避免过度工程（99.999% vs 99.9%的成本差异）
```


## 事件管理与混沌工程


```
事件响应流程：
1. 检测：告警触发
2. 响应：On-call工程师接手
3. 评估：影响范围和严重程度
4. 缓解：临时措施恢复服务
5. 根因分析：找到根本原因
6. 修复：永久性修复
7. 复盘：Blameless Postmortem

严重等级 (Severity)：
- SEV1：服务完全不可用，所有用户受影响
- SEV2：核心功能严重降级，大部分用户受影响
- SEV3：非核心功能异常，部分用户受影响
- SEV4：轻微异常，极少数用户受影响

无责复盘 (Blameless Postmortem)：
- 聚焦系统改进而非个人责任
- 记录：时间线、影响、根因、行动项
- 分析：什么导致了事件？如何防止再发生？
- 行动项：明确负责人和截止日期

混沌工程 (Chaos Engineering)：
主动注入故障，验证系统韧性
- Netflix Chaos Monkey：随机杀进程
- Litmus/Chaos Mesh：K8s混沌工程
- 实验：假设系统能承受X故障 → 验证
- 从生产环境的小规模实验开始
```


> **Note:** 错误预算将可靠性从"越可靠越好"转变为"足够可靠即可"，解放了团队的创新速度。
>         无责复盘文化是SRE团队的核心价值观。
