# 链路追踪进阶


## 链路追踪进阶


可观测性分布式追踪OpenTelemetry


链路追踪是理解分布式系统请求流转的核心能力，进阶话题涉及上下文传播、采样策略等。


## Span上下文传播


```
W3C Trace Context 标准：
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
             ──┬───────────────────────────────────┬──────────────────┬──┬─
               │                                   │                  │  │
             版本                                Trace ID          Span ID 采样标志
             (00)                              (32位hex)          (16位hex) (01=采样)

tracestate: vendorname=value  # 供应商特定状态

上下文传播方式：

1. 跨进程传播（HTTP Headers）：
GET /api/orders HTTP/1.1
traceparent: 00-4bf92f...-00f067...-01
tracestate: congo=t61rcWkgMzE

2. 跨线程传播：
// Go: 使用context.Context
ctx, span := tracer.Start(parentCtx, "operation")
defer span.End()

// Java: 使用ThreadLocal + Scope
try (Scope scope = span.makeCurrent()) {
  // 当前线程可访问当前span
}

3. 消息队列传播（放入消息头）：
// Kafka消息头
Headers: traceparent=00-4bf92f...-00f067...-01

Baggage（行李）机制：
- 在trace中传递业务数据（用户ID、租户ID等）
- 跨服务边界传递，但不存储在trace后端

// Go: 设置和读取Baggage
member, _ := baggage.NewMember("user.id", "12345")
bag, _ := baggage.New(member)
ctx = baggage.ContextWithBaggage(ctx, bag)

// 下游服务读取
bag := baggage.FromContext(ctx)
userID := bag.Member("user.id").Value()

Baggage注意事项：
- 不要传递敏感数据（明文传输）
- 控制Baggage大小（每条都在每个Span中重复）
- 适用于：租户ID、用户ID、A/B测试标记
```


## 采样策略


```
为什么需要采样：
- 生产环境流量巨大，全量收集成本高
- 存储和计算资源有限
- 大多数请求是正常的，不需要详细追踪

采样类型：

┌──────────────┬──────────────────────────────────────────┐
│ 类型          │ 说明                                      │
├──────────────┼──────────────────────────────────────────┤
│ Head采样      │ 在Trace开始时决定是否采样                  │
│              │ 决策点：入口服务                           │
│              │ 优点：简单，开销小                         │
│              │ 缺点：不知道trace是否有错误                │
├──────────────┼──────────────────────────────────────────┤
│ Tail采样      │ 在Trace完成后决定是否保留                  │
│              │ 决策点：收集器端                           │
│              │ 优点：可保留有价值的trace                  │
│              │ 缺点：需要缓冲，复杂度高                    │
└──────────────┴──────────────────────────────────────────┘

Head采样策略：
// 固定比例采样：10%的请求
Sampler: TraceIDRatioBased(0.1)

// 速率限制采样：每秒100条
Sampler: RateLimiting(100)

// 基于规则的采样
Rules:
  - path: /health    → drop (不采样)
  - path: /api/pay   → always_sample
  - status: error    → always_sample
  - default          → 0.1

Tail采样（OpenTelemetry Collector）：
processors:
  tail_sampling:
    decision_wait: 10s  # 等待10秒再做决策
    policies:
      # 错误的trace全部保留
      - name: error-policy
        type: status_code
        status_code: {status_codes: [ERROR]}
      # 慢请求保留
      - name: latency-policy
        type: latency
        latency: {threshold_ms: 1000}
      # 特定服务保留
      - name: service-policy
        type: string_attribute
        string_attribute:
          key: service.name
          values: [payment-service]
      # 其他10%采样
      - name: probabilistic
        type: probabilistic
        probabilistic: {sampling_percentage: 10}

自适应采样：
- 根据系统负载动态调整采样率
- 高负载时降低采样率
- 检测到异常时提高采样率
- 基于错误率/延迟自动调整

采样决策传播：
- 一旦Head决定采样，下游必须跟随
- 避免trace不完整
- 通过traceparent的sampled标志传播
```


## OpenTelemetry实战


```
OpenTelemetry架构：
┌────────────────────────────────────────────────────┐
│  应用层 (SDK + 自动/手动埋点)                       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐            │
│  │ Traces  │  │ Metrics │  │  Logs   │            │
│  └────┬────┘  └────┬────┘  └────┬────┘            │
│       └────────────┼────────────┘                  │
│                    ▼                               │
│          OTel SDK (数据采集)                       │
│                    │                               │
│          OTLP Exporter (导出协议)                  │
└────────────┬───────┬───────┬───────────────────────┘
             │       │       │
    ┌────────▼───────▼───────▼────────┐
    │  OTel Collector (数据处理管道)    │
    │  Receiver → Processor → Exporter│
    └────────────┬────────────────────┘
                 │
    ┌────────────▼────────────────────┐
    │  后端存储 (Jaeger/Tempo/Grafana) │
    └─────────────────────────────────┘

Go自动埋点：
import (
  "go.opentelemetry.io/otel"
  "go.opentelemetry.io/otel/exporters/otlp/otlptrace/otlptracegrpc"
  "go.opentelemetry.io/otel/sdk/trace"
)

func initTracer() func() {
  exporter, _ := otlptracegrpc.New(context.Background(),
    otlptracegrpc.WithEndpoint("otel-collector:4317"),
    otlptracegrpc.WithInsecure(),
  )

  tp := trace.NewTracerProvider(
    trace.WithBatcher(exporter),
    trace.WithResource(resource.NewWithAttributes(
      semconv.SchemaURL,
      semconv.ServiceNameKey.String("order-service"),
    )),
  )

  otel.SetTracerProvider(tp)
  return func() { tp.Shutdown(context.Background()) }
}

Collector配置 (config.yaml)：
receivers:
  otlp:
    protocols:
      grpc: { endpoint: "0.0.0.0:4317" }
      http: { endpoint: "0.0.0.0:4318" }

processors:
  batch: { timeout: 5s, send_batch_size: 1024 }
  memory_limiter: { limit_mib: 512, check_interval: 1s }

exporters:
  otlp/jaeger:
    endpoint: jaeger:4317
    tls: { insecure: true }
  prometheus:
    endpoint: "0.0.0.0:8889"

service:
  pipelines:
    traces: { receivers: [otlp], processors: [batch], exporters: [otlp/jaeger] }
    metrics: { receivers: [otlp], processors: [batch], exporters: [prometheus] }
```


> **Note:** Head采样简单但可能丢失有价值的trace，Tail采样更智能但需要更多资源。W3C Trace Context是跨服务传播的标准。Baggage用于传递业务上下文，但注意不要传递敏感数据。OpenTelemetry统一了Traces/Metrics/Logs三大信号的采集。
