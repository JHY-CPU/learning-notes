# 火焰图与性能剖析 - 性能工程笔记


## 火焰图与性能剖析


## 一、性能剖析基础


### 1. 剖析方法分类


| 方法 | 原理 | 开销 | 精度 | 适用场景 |
| --- | --- | --- | --- | --- |
| 采样（Sampling） | 定时中断采集调用栈 | 低（1-5%） | 中 | 生产环境 |
| 插桩（Instrumentation） | 函数入口/出口插入探针 | 高（10-30%） | 高 | 开发/测试环境 |
| 事件追踪（Tracing） | 记录特定事件（系统调用等） | 中 | 高 | 问题定位 |
| 硬件计数器（PMC） | CPU 硬件计数缓存命中/分支预测 | 极低 | 高 | 底层优化 |


## 二、火焰图（Flame Graph）


### 1. 什么是火焰图


```
火焰图是一种可视化调用栈的图表：
- X轴：字母序排列的函数（宽度 = 采样比例）
- Y轴：调用栈深度（底部是入口，顶部是叶子函数）
- 颜色：无特殊含义（通常按调用栈哈希着色）

       ┌────────────────────────────────────┐
       │         parseRequest()             │  ← 顶层
       ├────────────┬───────────┬───────────┤
       │  validate()│ decode()  │  route()  │
       ├────────────┤           ├───────────┤
       │ checkAuth()│           │ handler() │
       ├────────────┤           ├──────┬────┤
       │ queryDB()  │           │send()│log │
       └────────────┴───────────┴──────┴────┘
 ↑ 深度（栈顶）                         深度（栈底） ↓

宽度越大的函数 → 占用CPU时间越多 → 优化优先级越高
```


### 2. 火焰图类型


- **CPU 火焰图：**
   分析 CPU 时间消耗热点
- **Off-CPU 火焰图：**
   分析阻塞等待（IO、锁、调度）
- **内存火焰图：**
   分析内存分配热点
- **差分火焰图（Diff）：**
   对比两个时间点的火焰图，定位变化
- **热力火焰图：**
   用颜色深浅表示热度


### 3. 使用 perf 生成火焰图（Linux）


```
# 1. 采集 CPU 采样数据（30秒，99Hz）
perf record -F 99 -p $(pgrep java) -g -- sleep 30

# 2. 生成折叠调用栈
perf script | stackcollapse-perf.pl > out.perf-folded

# 3. 生成火焰图 SVG
flamegraph.pl out.perf-folded > flamegraph.svg
```


### 4. 使用 async-profiler（Java）


```
# CPU 火焰图
./profiler.sh -d 30 -f cpu-flamegraph.html $(pgrep java)

# 内存分配火焰图
./profiler.sh -d 30 -e alloc -f alloc-flamegraph.html $(pgrep java)

# 锁争用火焰图
./profiler.sh -d 30 -e lock -f lock-flamegraph.html $(pgrep java)

# 壁钟时间火焰图（包括阻塞等待）
./profiler.sh -d 30 -e wall -f wall-flamegraph.html $(pgrep java)
```


### 5. 使用 eBPF/BCC（Linux 内核级追踪）


```
# CPU 火焰图
perf record -F 99 -a -g -- sleep 30
# 或使用 BCC tools
/usr/share/bcc/tools/profile -f 30 -p 1234 > out.stacks

# Off-CPU 火焰图（分析阻塞原因）
/usr/share/bcc/tools/offcputime -df 30 -p 1234 > out.offcpu

# eBPF 的优势：安全的内核级追踪，极低开销，可追踪任意内核函数
```


## 三、火焰图分析技巧


### 1. 读图方法


```
自底向上阅读（从栈底到栈顶）：
- 底部是程序入口（main / thread run）
- 顶部是实际消耗 CPU 的函数（叶子节点）
- 每一层是上一层的调用者

关键观察点：
- 顶部平坦的大块 → 这是热点函数，重点优化
- 顶部有多个小块 → 工作分散，难以优化
- 火焰图很宽很矮 → 大量平级调用
- 火焰图很窄很高 → 深度递归或深调用链
```


### 2. 逆火图（Icicle Graph）


火焰图的上下翻转版本，根在上、叶在下。有些人觉得更直观，因为"上面是入口"。


### 3. 常见模式识别


- **宽平台型：**
   底层库函数占用大量时间（如序列化、正则匹配）
- **锯齿型：**
   GC 相关函数频繁出现（Java 火焰图常见）
- **对称双峰：**
   生产者-消费者模式，等待和处理各占一半
- **倒三角：**
   递归或循环调用


## 四、其他剖析工具


### 1. 系统级工具


```
# Linux perf
perf top -p $(pgrep java)          # 实时热点
perf stat -p $(pgrep java) sleep 5 # 硬件计数器统计
perf record -g -p $(pgrep java)    # 记录调用栈

# DTrace（Solaris/macOS）
dtrace -n 'pid$target:::entry { @[probefunc] = count(); }' -p 1234

# strace/ltrace
strace -cp $(pgrep java)           # 系统调用统计
strace -e trace=network -p 1234    # 网络相关系统调用
```


### 2. Java 专用工具


```
# JFR (Java Flight Recorder) — 生产级低开销剖析
jcmd $(pgrep java) JFR.start duration=60s filename=recording.jfr
jcmd $(pgrep java) JFR.dump filename=recording.jfr

# JMC (Java Mission Control) 分析 JFR 文件
# 或使用命令行：
jfr print --events jdk.CPULoad,jdk.JavaMonitorEnter recording.jfr

# arthas（阿里开源在线诊断）
# dashboard   → 实时面板
# profiler    → 生成火焰图
# thread -b   → 检测死锁
# trace       → 方法调用追踪
```


### 3. Go 语言工具


```
# 内置 pprof
import _ "net/http/pprof"

# CPU 剖析（30秒）
go tool pprof http://localhost:6060/debug/pprof/profile?seconds=30

# 内存剖析
go tool pprof http://localhost:6060/debug/pprof/heap

# 阻塞剖析
go tool pprof http://localhost:6060/debug/pprof/block
```


## 五、性能剖析工作流


```
1. 基线测量    → 记录当前性能指标（RT、吞吐量、CPU 使用率）
2. 选择工具    → 根据问题类型选择剖析工具
3. 采集数据    → 生产环境用采样（低开销），测试环境用插桩
4. 可视化      → 生成火焰图/调用图
5. 识别热点    → 找到占比最大的函数
6. 优化        → 修改代码/配置
7. 验证        → 重新测量，对比基线
8. 回归测试    → 确保优化没有破坏功能和其他指标
```


> **Note:** **面试要点：**
> 火焰图通过采样调用栈可视化 CPU 热点，X轴是函数名（宽度=采样比例），Y轴是调用栈深度。perf + FlameGraph.pl 是 Linux 标准方案，async-profiler 是 Java 首选（支持 CPU/内存/锁剖析）。eBPF 是新一代内核追踪技术，开销极低。JFR 是 Java 生产环境的标准剖析工具。

总结：性能剖析方法分采样（低开销）、插桩（高精度）、事件追踪和硬件计数器；火焰图是可视化调用栈的利器，通过宽度显示CPU消耗热点；Linux用perf+FlameGraph.pl生成，Java用async-profiler，Go用内置pprof；eBPF是新一代低开销内核追踪技术；分析流程：基线→采集→可视化→识别→优化→验证
