数据库性能优化

性能工程数据库索引

数据库往往是后端性能瓶颈的根源，掌握查询优化和索引调优至关重要。

执行计划分析

EXPLAIN 详解（MySQL）： EXPLAIN SELECT * FROM orders WHERE user_id = 100 AND status = 'PAID'; 关键字段解读： ┌──────────┬──────────────────────────────────────────┐ │ 字段 │ 说明 │ ├──────────┼──────────────────────────────────────────┤ │ type │ 访问类型：system > const > eq_ref > ref │ │ │ > range > index > ALL │ │ │ ALL(全表扫描)是最差情况 │ │ key │ 实际使用的索引 │ │ rows │ 预估扫描行数 │ │ Extra │ 额外信息 │ │ filtered │ 经过条件过滤后的比例 │ └──────────┴──────────────────────────────────────────┘ 常见问题和解决： - type=ALL：缺少索引，需添加合适索引 - key=NULL：未使用索引，检查索引列和查询条件 - rows过大：索引选择性差，考虑组合索引 - Using filesort：排序未用索引，添加排序列索引 - Using temporary：使用临时表，优化GROUP BY - Using index：覆盖索引，理想情况 EXPLAIN分析技巧： 1. 从下往上读（嵌套循环先看最内层） 2. 关注type从ALL到ref的改进空间 3. key_len反映索引使用程度 4. rows × filtered ≈ 实际扫描行数 慢查询分析： # 开启慢查询日志 SET GLOBAL slow_query_log = 'ON'; SET GLOBAL long_query_time = 1; -- 超过1秒记录 # 分析慢查询日志 mysqldumpslow -s t -t 10 /var/log/mysql/slow.log # 使用pt-query-digest深度分析 pt-query-digest slow.log > report.txt

索引调优

索引类型与选择： ┌──────────────┬──────────────────────────────────────────┐ │ 索引类型 │ 特点和适用场景 │ ├──────────────┼──────────────────────────────────────────┤ │ B+Tree │ 最常用，支持范围查询和排序 │ │ Hash │ 精确匹配快，不支持范围查询 │ │ 全文索引 │ 文本搜索，InnoDB 5.6+支持 │ │ 空间索引 │ GIS数据，R-Tree结构 │ │ 前缀索引 │ 长字符串列，只索引前N个字符 │ └──────────────┴──────────────────────────────────────────┘ 组合索引设计原则： 1. 最左前缀原则：WHERE a=? AND b=? 可用(a,b)索引 2. 区分度优先：区分度高的列放前面 3. 覆盖索引：索引包含查询所需的所有列 -- 区分度分析 SELECT COUNT(DISTINCT status) / COUNT(*) AS selectivity_status, COUNT(DISTINCT user_id) / COUNT(*) AS selectivity_user_id FROM orders; -- user_id区分度更高，应放在前面 索引优化实例： -- 问题查询：全表扫描 SELECT * FROM orders WHERE user_id = ? AND status = 'PAID' ORDER BY created_at DESC; -- 优化1：添加组合索引 CREATE INDEX idx_user_status_created ON orders(user_id, status, created_at DESC); -- 优化2：覆盖索引（减少回表） SELECT user_id, status, created_at, amount FROM orders FORCE INDEX(idx_user_status_created) WHERE user_id = ? AND status = 'PAID'; 索引维护： -- 查找未使用的索引 SELECT * FROM sys.schema_unused_indexes; -- 查找冗余索引 SELECT * FROM sys.schema_redundant_indexes; -- 查找索引碎片 SELECT table_name, index_name, stat_value * @@innodb_page_size / 1024 / 1024 AS size_mb, stat_description FROM mysql.innodb_index_stats WHERE stat_name = 'size' AND index_name != 'PRIMARY' ORDER BY stat_value DESC; 什么时候不建索引： - 表数据量很小（<几千行） - 列的区分度很低（如布尔类型单独索引） - 频繁更新的列（写入开销） - WHERE条件中不会出现的列

连接池与读写分离

连接池配置原则（HikariCP为例）： 连接池大小 = CPU核心数 × 2 + 磁盘数 # 4核CPU，1块SSD spring.datasource.hikari.maximum-pool-size=10 spring.datasource.hikari.minimum-idle=5 spring.datasource.hikari.connection-timeout=30000 spring.datasource.hikari.idle-timeout=600000 spring.datasource.hikari.max-lifetime=1800000 常见错误：连接池过大 - 连接数过多 → 上下文切换开销 - 内存消耗增加 - 数据库连接数耗尽 - 实际上降低了吞吐量 读写分离架构： ┌─→ Read Replica 1 App → ProxySQL/Mycat ─┤ ├─→ Read Replica 2 └─→ Master (读写) 实现方式： 1. 应用层分离：@Transactional(readOnly=true) → 从库 2. 中间件分离：ProxySQL, ShardingSphere 3. ORM框架：Spring + DynamicDataSource 读写分离注意事项： - 主从延迟：写后立即读可能读不到 - 解决方案： a. 写后强制读主库 b. 判断延迟（SHOW SLAVE STATUS） c. 业务容忍短暂不一致 数据库分区策略： -- 范围分区（按时间） CREATE TABLE orders ( id BIGINT, user_id BIGINT, created_at DATETIME, amount DECIMAL(10,2) ) PARTITION BY RANGE (YEAR(created_at)) ( PARTITION p2023 VALUES LESS THAN (2024), PARTITION p2024 VALUES LESS THAN (2025), PARTITION p2025 VALUES LESS THAN (2026) ); -- 哈希分区（均匀分布） CREATE TABLE logs ( id BIGINT, user_id BIGINT ) PARTITION BY HASH(user_id) PARTITIONS 16; 分区优势与限制： + 快速删除旧分区（ALTER TABLE DROP PARTITION） + 查询优化器可跳过不相关分区 + 分区可以存储在不同磁盘 - 不支持外键 - 分区键必须在主键/唯一索引中 - 单表分区数有限制（MySQL 8.0最多8192个）

查询优化实战技巧

常见慢查询模式及优化： 1. SELECT * 的危害 ✗ SELECT * FROM users WHERE id = 100; ✓ SELECT id, name, email FROM users WHERE id = 100; 原因：避免读取不需要的列，支持覆盖索引 2. 大偏移量分页 ✗ SELECT * FROM orders ORDER BY id LIMIT 1000000, 20; ✓ SELECT * FROM orders WHERE id > 1000000 ORDER BY id LIMIT 20; 或者使用延迟关联： ✓ SELECT o.* FROM orders o INNER JOIN (SELECT id FROM orders ORDER BY id LIMIT 1000000, 20) t ON o.id = t.id; 3. IN列表过长 ✗ WHERE id IN (1,2,3,...,10000) ✓ 使用临时表 JOIN： CREATE TEMPORARY TABLE tmp_ids (id BIGINT PRIMARY KEY); INSERT INTO tmp_ids VALUES (1),(2),...; SELECT o.* FROM orders o JOIN tmp_ids t ON o.id = t.id; 4. 隐式类型转换 ✗ WHERE phone = 13800138000 -- phone是VARCHAR ✓ WHERE phone = '13800138000' 原因：类型不匹配导致索引失效 5. OR条件 ✗ WHERE a = 1 OR b = 2 -- 两列各有索引 ✓ 使用UNION ALL： SELECT * FROM t WHERE a = 1 UNION ALL SELECT * FROM t WHERE b = 2 AND a != 1; 6. 模糊查询前缀通配符 ✗ WHERE name LIKE '%张%' ✓ WHERE name LIKE '张%'（可走索引） 或使用全文索引 7. NOT IN / NOT EXISTS ✗ WHERE id NOT IN (SELECT user_id FROM blacklist) ✓ LEFT JOIN + IS NULL： SELECT u.* FROM users u LEFT JOIN blacklist b ON u.id = b.user_id WHERE b.user_id IS NULL;
数据库优化遵循"先索引、后架构、再硬件"的优先级。EXPLAIN是最重要的诊断工具，关注type、key、rows三个字段。连接池大小不是越大越好，推荐公式：核心数×2+磁盘数。
