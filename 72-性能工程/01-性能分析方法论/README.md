# 01-性能分析方法论

## 性能指标

### 吞吐量（Throughput）

吞吐量衡量系统在单位时间内处理请求的能力。

- **QPS（Queries Per Second）**：每秒查询数，常用于读密集型场景。例如搜索引擎每秒处理的搜索请求数。
- **TPS（Transactions Per Second）**：每秒事务数，强调业务事务的完整性。例如电商系统每秒完成的下单事务数。
- **RPS（Requests Per Second）**：每秒请求数，HTTP 层面的常用指标。

吞吐量与资源消耗直接相关，是容量规划的核心依据。

### 延迟（Latency）

延迟衡量请求从发出到收到响应的时间。

- **平均延迟**：所有请求延迟的算术平均值，容易被极端值拉偏，不适合长尾分布。
- **P50（中位数）**：50% 的请求在此时间内完成，代表"典型用户体验"。
- **P95**：95% 的请求在此时间内完成，用于衡量大部分用户的体验。
- **P99**：99% 的请求在此时间内完成，用于衡量极端情况下的体验。
- **最大延迟**：最慢请求的延迟，通常不稳定但影响最差体验。

**直方图（Histogram）** 比平均值更适合描述延迟分布，能揭示长尾效应。

### 并发数（Concurrency）

- 同一时刻系统正在处理的请求数量。
- Little 定律：`L = λW`，平均并发数 = 到达速率 × 平均处理时间。
- 并发数过高会导致排队延迟急剧上升，系统进入饱和状态。

### 错误率（Error Rate）

- 错误请求数占总请求数的比例。
- 常见错误类型：5xx 服务器错误、超时、连接拒绝、熔断降级。
- 错误率是可用性的直接体现，通常要求在 99.9% 以上。

## USE 方法

USE 方法由 Brendan Gregg 提出，适用于硬件与基础设施资源的性能分析。

| 维度 | 含义 | 示例 |
|------|------|------|
| **Utilization（利用率）** | 资源繁忙的时间比例 | CPU 利用率 80%、磁盘 I/O 利用率 90% |
| **Saturation（饱和度）** | 资源上排队等待的工作量 | CPU 运行队列长度、磁盘等待队列深度 |
| **Errors（错误数）** | 资源发生的错误事件计数 | 网络丢包数、磁盘 I/O 错误数 |

**分析流程**：对每个资源（CPU、内存、磁盘、网络、NIC）检查 U/S/E 三个维度，定位瓶颈资源。

## RED 方法

RED 方法由 Tom Wilkie 提出，面向微服务/面向请求的系统。

| 维度 | 含义 |
|------|------|
| **Rate（速率）** | 每秒请求数（QPS/RPS） |
| **Errors（错误）** | 每秒失败请求数 |
| **Duration（耗时）** | 请求处理时间（延迟分布） |

RED 方法关注的是"服务对外表现"，USE 方法关注的是"底层资源状态"，两者互补。

## 性能分析流程

### 1. 定义性能目标

- 基于业务需求定义 SLO：P99 延迟 < 200ms，可用性 > 99.95%。
- 明确场景：峰值流量、长时间运行稳定性、突发流量弹性。
- 目标需可量化、可测试、有明确的通过/失败标准。

### 2. 基准测试设计

- **基线测试**：在已知配置下建立性能基线。
- **对比测试**：修改参数后与基线对比，衡量优化效果。
- **压力测试**：逐步加压直到系统瓶颈或崩溃点，确定容量上限。

### 3. 瓶颈识别

- 自顶向下分析：从应用层开始，逐步深入到中间件、OS、硬件。
- 使用 USE 方法检查资源利用率。
- 使用火焰图识别 CPU 热点函数。
- 检查日志与 APM 追踪中的慢请求。

### 4. 优化与验证

- 每次只修改一个变量，验证优化效果。
- 优化后重新运行基准测试，对比前后数据。
- 持续监控，防止引入新的性能退化。

### 压测工具

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| **wrk** | 高性能、C 语言、Lua 脚本扩展 | HTTP 基准测试、高并发场景 |
| **Apache Bench (ab)** | 简单易用、单线程 | 快速验证、简单场景 |
| **JMeter** | 图形化界面、支持多协议 | 复杂场景、团队协作 |
| **Locust** | Python 编写、分布式、代码定义场景 | 灵活场景编排、API 压测 |
| **k6** | JavaScript 脚本、CI/CD 友好 | 开发者驱动的性能测试 |

### 基准测试设计原则

- **梯度加压**：从低并发逐步增加到高并发（如 10→50→100→200→500），观察性能拐点。
- **持续时间**：短时间测试（< 2 分钟）可能遗漏预热效应和 GC 影响，建议至少持续 5-10 分钟。
- **预热（Warm-up）**：JIT 编译、缓存填充需要时间，正式采集数据前应有 1-3 分钟预热期。
- **环境一致性**：确保测试环境与基准环境一致（CPU、内存、网络、数据量）。
- **数据隔离**：避免测试间数据相互干扰，每次测试使用独立的数据集。

## 火焰图（Flame Graph）

火焰图由 Brendan Gregg 发明，是一种可视化 CPU/资源使用热点的图表。

### CPU 火焰图

- 展示 CPU 上运行的函数调用栈及其采样频率。
- X 轴按字母排序，Y 轴表示调用栈深度。
- 越宽的条表示该函数在 CPU 上花费的时间越多。
- 查找"平顶"（plateaus）来定位 CPU 密集型热点函数。

### 内存火焰图

- 展示分配内存的调用栈及其分配量。
- 用于识别内存泄漏和高内存分配路径。
- 可区分分配的字节数和分配的对象数。

### off-CPU 火焰图

- 展示线程不在 CPU 上运行时的等待原因（I/O 等待、锁等待、睡眠）。
- 用于诊断高延迟但 CPU 利用率不高的场景。
- 揭示阻塞式 I/O、锁竞争等问题。

### 生成工具

- **perf**：Linux 内核性能分析工具，`perf record` 采集数据，配合 FlameGraph 脚本生成 SVG。
- **async-profiler**：JVM 专用分析器，支持 CPU、内存、锁分析，低开销。
- **FlameGraph**：Brendan Gregg 的开源工具集，将 `perf` 数据转换为交互式 SVG 火焰图。

## 性能建模

### 排队论基础（M/M/1 队列）

- **M/M/1 模型**：单服务台、泊松到达、指数服务时间。
- 到达率为 λ，服务率为 μ，利用率 ρ = λ/μ。
- 平均队列长度：`L = ρ / (1 - ρ)`。
- 平均等待时间：`W = 1 / (μ - λ)`。
- 当 ρ 接近 1 时，等待时间急剧增长——这就是为什么不能将系统利用率推到 100%。

### Little 定律

> `L = λW`

- L：系统中的平均客户数（并发数）。
- λ：平均到达速率（吞吐量）。
- W：系统中的平均停留时间（延迟）。
- Little 定律是排队论中最基础的关系式，适用于任何稳定系统。

### 容量规划

- 基于历史数据预测未来流量增长。
- 使用排队论模型估算在目标延迟下的最大吞吐量。
- 留出安全余量（通常为峰值的 1.5-2 倍）应对突发流量。
- 容量规划应结合业务增长预测和季节性波动。
