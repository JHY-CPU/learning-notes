# 容量评估与压测


## 容量评估与压测


性能工程容量规划压测


容量评估回答"系统能承受多大压力"这个问题，是架构设计和运维决策的基础。


## 负载模型与Little定律


```
Little定律 (Little's Law)：
L = λ × W

- L：系统中的平均请求数（并发数）
- λ：平均到达率（吞吐量，req/s）
- W：请求在系统中的平均停留时间（响应时间）

应用场景：
已知：平均响应时间 W = 200ms，目标QPS = 1000
求：需要多少并发连接？
L = 1000 × 0.2 = 200 并发连接

已知：系统支持100并发，平均处理时间 = 50ms
求：最大吞吐量？
λ = L / W = 100 / 0.05 = 2000 req/s

负载模型组成要素：
┌──────────────┬──────────────────────────────┐
│ 要素          │ 说明                          │
├──────────────┼──────────────────────────────┤
│ 用户行为模型  │ 操作类型、比例、间隔            │
│ 流量模型      │ 日均/峰值QPS、流量曲线          │
│ 数据模型      │ 数据量、分布、增长趋势          │
│ 资源模型      │ CPU/内存/IO/网络的使用模式      │
└──────────────┴──────────────────────────────┘

容量推算公式：
峰值QPS = 日均PV × 峰值因子 / 86400
所需实例数 = 峰值QPS / 单实例处理能力 × 冗余系数

示例：
- 日均PV = 1000万
- 峰值因子 = 2.5（峰值是平均的2.5倍）
- 单实例QPS = 500
- 冗余系数 = 1.5（留50%余量）
所需实例 = (10,000,000 × 2.5 / 86,400) / 500 × 1.5 ≈ 87台
```


## 压测策略与方法


```
压测类型：

1. 负载测试 (Load Testing)
   - 逐步增加负载到预期水平
   - 验证系统在预期负载下的表现
   - 目标：确认SLA可达

2. 压力测试 (Stress Testing)
   - 超过预期负载，找到系统极限
   - 目标：找到性能拐点和瓶颈

3. 浸泡测试 (Soak/Endurance Testing)
   - 长时间（数小时到数天）维持中高负载
   - 发现内存泄漏、连接耗尽等问题
   - 目标：验证稳定性

4. 尖峰测试 (Spike Testing)
   - 短时间内突发大量请求
   - 模拟秒杀、热点事件
   - 目标：验证弹性伸缩能力

压测执行流程：
1. 确定目标（SLA指标和容量需求）
2. 编写压测脚本（模拟真实用户行为）
3. 准备环境（独立或影子流量）
4. 准备监控（系统指标+业务指标）
5. 预热（warm-up请求）
6. 执行压测（逐步加压）
7. 收集数据（响应时间、错误率、资源使用）
8. 分析结果（找到瓶颈）
9. 优化并重复

压测脚本示例 (k6)：
import http from 'k6/http';
import { check, sleep } from 'k6';

export let options = {
  stages: [
    { duration: '2m', target: 100 },   // 爬升
    { duration: '5m', target: 100 },   // 持续
    { duration: '2m', target: 200 },   // 加压
    { duration: '5m', target: 200 },   // 持续
    { duration: '2m', target: 0 },     // 收尾
  ],
  thresholds: {
    http_req_duration: ['p(95)<500'],  // p95 < 500ms
    http_req_failed: ['rate<0.01'],     // 错误率 < 1%
  },
};

export default function () {
  let res = http.get('https://api.example.com/products');
  check(res, {
    'status is 200': (r) => r.status === 200,
  });
  sleep(1);
}
```


## 容量报告与决策


```
容量报告模板：

一、执行摘要
- 测试目标和范围
- 关键发现（一句话总结）
- 资源需求预测

二、测试环境
- 硬件配置（CPU/内存/磁盘/网络）
- 软件版本和配置
- 与生产环境的差异

三、测试结果
┌────────┬────────┬────────┬────────┬────────┐
│ 并发数  │ QPS    │ p50    │ p99    │ 错误率  │
├────────┼────────┼────────┼────────┼────────┤
│ 50     │ 480    │ 85ms   │ 150ms  │ 0%     │
│ 100    │ 920    │ 95ms   │ 200ms  │ 0%     │
│ 200    │ 1650   │ 120ms  │ 380ms  │ 0.2%   │
│ 300    │ 1800   │ 180ms  │ 850ms  │ 2.1%   │
│ 400    │ 1750   │ 250ms  │ 2100ms │ 8.5%   │
└────────┴────────┴────────┴────────┴────────┘
分析：系统在200并发时达到最优效率，300并发时开始出现性能下降。

四、瓶颈分析
- CPU密集型服务在300并发时CPU > 85%
- 数据库连接池在250并发时耗尽
- 网络带宽未成为瓶颈

五、容量预测
- 当前配置支持峰值 1650 QPS
- 预计QPS增长到2000需要：扩容2台应用服务器 + 连接池翻倍
- 预计QPS增长到5000需要：架构升级（读写分离+缓存层）

六、建议
1. 短期：调整连接池配置（20→50），预计提升至2000 QPS
2. 中期：添加Redis缓存层，预计提升至3500 QPS
3. 长期：数据库读写分离 + 分库分表
```


> **Note:** Little定律(L=λW)是容量推算的数学基础。压测不只是"打满看崩"，而是逐步找到性能拐点。容量报告要包含明确的行动建议和资源预测，而非只是数据罗列。
