常见优化策略

性能工程缓存优化

系统性的性能优化策略，从缓存到异步处理。

缓存策略

多级缓存架构： ┌─────────────────────────────────────────────┐ │ 客户端缓存 → CDN → 网关缓存 → 应用缓存 → DB │ │ (浏览器) (边缘) (Nginx) (Redis) │ └─────────────────────────────────────────────┘ 缓存模式： ┌─────────────────────────────────────────────┐ │ Cache-Aside (旁路缓存) │ │ 读：先查缓存，未命中查DB并回写 │ │ 写：先更新DB，再删除缓存 │ │ 最常用，应用层控制 │ │ │ │ Write-Through (写穿透) │ │ 写：同时写缓存和DB │ │ 读：只读缓存 │ │ 缓存层负责同步 │ │ │ │ Write-Behind (写回) │ │ 写：只写缓存，异步写DB │ │ 性能最好但有数据丢失风险 │ │ │ │ Read-Through (读穿透) │ │ 缓存未命中时由缓存层从DB加载 │ └─────────────────────────────────────────────┘ 缓存问题与解决： - 缓存穿透（查不存在的数据）：布隆过滤器/空值缓存 - 缓存击穿（热点key过期）：互斥锁/永不过期+异步更新 - 缓存雪崩（大量key同时过期）：随机过期时间 - 数据一致性：延迟双删/订阅binlog

数据库优化

SQL优化： - 避免 SELECT *，只查需要的列 - 合理使用索引（覆盖索引、联合索引） - 避免在WHERE中对字段使用函数 - 大分页优化：延迟关联/书签 - 批量操作代替循环单条操作 索引优化： - 最左前缀原则 - 覆盖索引：索引包含查询所需的所有列 - 避免索引失效：函数、类型转换、OR、LIKE左模糊 - EXPLAIN分析执行计划 连接池： - 避免频繁创建/销毁连接 - 合理设置池大小（CPU核心数 × 2 + 磁盘数） - HikariCP (Java), pgbouncer (PostgreSQL) - 空闲连接回收 分库分表： - 垂直拆分：按业务模块 - 水平拆分：按用户ID/时间 - 读写分离：主库写，从库读

异步与并发

异步处理： ┌─────────────────────────────────────────────┐ │ 同步：请求 → 处理 → 响应（阻塞等待） │ │ 异步：请求 → 接收 → 后台处理 → 回调/Webhook │ │ │ │ 适用场景： │ │ - 耗时操作：发送邮件/短信/推送 │ │ - 批量处理：数据导入/报表生成 │ │ - 事件驱动：消息队列解耦 │ └─────────────────────────────────────────────┘ 并发优化： - 线程池：避免线程创建开销 - 连接池：复用数据库/HTTP连接 - 对象池：复用重量级对象 - 协程(Goroutine/asyncio)：轻量级并发 CDN优化： - 静态资源CDN分发 - 浏览器缓存 + CDN缓存 + 源站 - 缓存时间策略：不变资源长期缓存(Hash文件名) - HTTP/2 多路复用 - 资源压缩：gzip/brotli 代码级优化： - 减少内存分配（对象池、预分配） - 避免不必要的序列化/反序列化 - 批量操作代替循环单条 - 算法优化（O(n²) → O(n log n)） - 并行化计算密集型任务
优化优先级：缓存 > 数据库优化 > 异步化 > 代码优化。 80%的性能问题通过缓存和数据库优化就能解决。
