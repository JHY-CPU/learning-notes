JVM 性能调优

一、JVM 内存模型

JVM 内存结构（JDK 17+）：
┌─────────────────────────────────────────────┐
│                  堆 (Heap)                    │
│  ┌───────────────┐  ┌──────────────────────┐ │
│  │  Young Gen    │  │    Old Gen           │ │
│  │ ┌────┬────┬──┐│  │                      │ │
│  │ │Eden│S0  │S1││  │                      │ │
│  │ └────┴────┴──┘│  │                      │ │
│  └───────────────┘  └──────────────────────┘ │
├─────────────────────────────────────────────┤
│  元空间 (Metaspace)   ← 类元数据（不在堆中）   │
│  线程栈 (Thread Stack) ← 每线程独立           │
│  代码缓存 (Code Cache) ← JIT 编译后的代码     │
│  直接内存 (Direct Memory) ← NIO Buffer      │
└─────────────────────────────────────────────┘

二、GC 算法与调优

1. GC 算法对比

GC目标延迟适用堆大小特点
Serial>100ms<512MB单线程，简单
Parallel>50ms<8GB多线程吞吐优先
G1<200ms4GB-64GB区域化分代，平衡延迟和吞吐
ZGC<1ms任意大小低延迟，染色指针
Shenandoah<10ms任意大小并发压缩

2. G1 调优参数

# G1 核心参数
-Xms4g -Xmx4g                    # 堆大小（建议相等，避免动态调整）
-XX:+UseG1GC                      # 启用 G1
-XX:MaxGCPauseMillis=200          # 目标最大停顿时间（默认200ms）
-XX:G1HeapRegionSize=16m          # Region 大小（1-32MB，2的幂）
-XX:InitiatingHeapOccupancyPercent=45 # 触发并发标记的堆占用阈值
-XX:G1ReservePercent=10           # 预留空间防 to-space exhausted
-XX:ConcGCThreads=4               # 并发 GC 线程数
-XX:ParallelGCThreads=8           # STW 阶段线程数

# G1 调优思路：
# 1. 如果 GC 频繁 → 增大堆或降低 IHOP
# 2. 如果停顿时间长 → 降低 MaxGCPauseMillis（但会增加 GC 频率）
# 3. 如果 Humongous 对象多 → 调大 RegionSize
# 4. 如果 Mixed GC 后老年代仍满 → 降低 IHOP 触发更早的并发标记

3. ZGC 调优参数

# ZGC 参数（JDK 17+）
-Xms8g -Xmx8g
-XX:+UseZGC                       # 启用 ZGC
-XX:+ZGenerational                # JDK 21+ 分代 ZGC（大幅提升吞吐）
-Xlog:gc*:file=gc.log:time,uptime # GC 日志

# ZGC 特点：
# - 染色指针：GC 元数据存在指针中而非对象头
# - 读屏障：并发转移/重定位时自动修正引用
# - 停顿时间与堆大小无关，通常 < 1ms
# - 代价：更高的内存开销和 CPU 开销

三、堆大小调优

1. 堆大小计算

堆大小公式：
Xmx = 总物理内存 - (Metaspace + 线程栈 × 线程数 + 直接内存 + GC本身开销)

示例：8GB 物理内存的服务器
- 系统预留: 1GB
- 线程栈: 1MB × 500线程 = 500MB
- Metaspace: 256MB
- 直接内存: 512MB
- GC 开销: ~5%
→ Xmx ≈ 8 - 1 - 0.5 - 0.25 - 0.5 = 5.75GB，取 5g 留余量

2. 新生代与老年代比例

# G1 不需要手动设置比例（自动管理）
# Parallel GC 旧版调优：
-XX:NewRatio=2          # 老年代:新生代 = 2:1
-XX:NewSize=1g          # 新生代初始大小
-XX:MaxNewSize=2g       # 新生代最大大小

# 新生代大小影响：
# - 新生代大 → Minor GC 频率低，但每次时间长
# - 新生代小 → Minor GC 频率高，但每次时间短
# - 对象晋升需要 Survivor 空间足够（避免直接晋升老年代）

四、JIT 编译优化

1. JIT 分层编译

Level 0: 解释执行
Level 1: C1 编译（无 profiling）
Level 2: C1 编译（基本 profiling）
Level 3: C1 编译（完全 profiling）
Level 4: C2 编译（激进优化）

热点阈值：
-XX:CompileThreshold=10000   # 方法调用 10000 次后编译
-XX:Tier3InvocationThreshold=200
-XX:Tier4InvocationThreshold=5000

2. 常见 JIT 优化

  • 方法内联（Inlining）：将小方法体直接嵌入调用处，消除方法调用开销
  • 逃逸分析：对象不逃出方法时，栈上分配或标量替换
  • 锁消除：逃逸分析后发现锁对象不逃出方法，消除锁
  • 循环展开：减少循环判断次数
  • 死代码消除：移除永远不会执行的代码

五、GC 日志分析

1. 开启 GC 日志

# JDK 11+ 统一日志
-Xlog:gc*:file=/var/log/gc.log:time,uptime,level,tags:filecount=5,filesize=100m

# JDK 8
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-XX:+PrintGCTimeStamps
-XX:+PrintHeapAtGC
-Xloggc:/var/log/gc.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=5
-XX:GCLogFileSize=100m

2. 关键 GC 指标

监控指标：
- GC 频率：Young GC < 5/min, Full GC ≈ 0
- GC 停顿：Young GC < 50ms, Full GC < 1s
- 堆使用率：Old Gen 70% 以下健康
- 晋升速率：对象从 Young → Old 的速度
- 分配速率：新对象创建速度

工具：
- jstat -gcutil $(pgrep java) 1000  → 实时 GC 统计
- GCViewer / gceasy.io → GC 日志可视化分析
- JFR → 低开销全面剖析

六、线程池调优

// 线程池参数
new ThreadPoolExecutor(
    corePoolSize,    // 核心线程数
    maxPoolSize,     // 最大线程数
    keepAliveTime,   // 空闲线程存活时间
    TimeUnit.SECONDS,
    new LinkedBlockingQueue(queueCapacity),  // 队列容量
    new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略
);

// CPU 密集型：线程数 = CPU核数 + 1
int cpuThreads = Runtime.getRuntime().availableProcessors() + 1;

// IO 密集型：线程数 = CPU核数 × (1 + IO等待时间/CPU计算时间)
// 简化：2 × CPU核数 或更高
int ioThreads = Runtime.getRuntime().availableProcessors() * 2;

// 拒绝策略选择：
// AbortPolicy     → 抛异常（默认，适合不能丢的请求）
// CallerRunsPolicy → 调用者线程执行（适合能接受延迟的场景）
// DiscardPolicy   → 静默丢弃（适合可丢失的监控数据）
// DiscardOldestPolicy → 丢弃最老任务（适合只关心最新数据）
