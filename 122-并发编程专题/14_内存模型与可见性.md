# 内存模型与可见性

## Java Memory Model（JMM）

JMM 定义了 Java 程序中变量的访问规则，规定了线程如何以及何时可以看到其他线程修改过的共享变量。

### 核心概念

```
线程 1                      线程 2
  │                           │
  ▼                           ▼
工作内存 1                  工作内存 2
(本地缓存/寄存器)           (本地缓存/寄存器)
  │                           │
  │  ┌─────────────────────┐  │
  └──│     主内存           │──┘
     │ (共享变量 x, y, z)  │
     └─────────────────────┘
```

JMM 中的操作：
- **load**: 从主内存读取到工作内存
- **store**: 从工作内存写回主内存
- **lock/unlock**: 对监视器加锁/解锁
- **read/write**: 底层传输操作

### volatile 关键字

```java
public class VolatileDemo {
    // volatile 保证可见性和有序性，不保证原子性
    private volatile boolean running = true;
    private volatile int counter = 0;

    public void stop() {
        running = false; // 立即对其他线程可见
    }

    public void worker() {
        while (running) {
            // 每次循环都会重新读取 running 的值
            // 非 volatile 可能因编译器优化而缓存旧值
        }
        System.out.println("工作线程停止");
    }

    // volatile 不保证原子性的问题
    public void unsafeIncrement() {
        counter++; // 不是线程安全的！读-改-写三个步骤
    }

    // 解决方案：使用 AtomicInteger
    private final java.util.concurrent.atomic.AtomicInteger atomicCounter =
        new java.util.concurrent.atomic.AtomicInteger(0);

    public void safeIncrement() {
        atomicCounter.incrementAndGet();
    }
}
```

### Double-Checked Locking

```java
public class Singleton {
    // 必须用 volatile 修饰，防止指令重排
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                 // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {         // 第二次检查（有锁）
                    // 无 volatile 时可能发生的问题：
                    // 1. 分配内存
                    // 2. instance 指向内存（此时 instance != null）
                    // 3. 执行构造函数
                    // 如果线程在步骤 2 后、步骤 3 前访问，得到未初始化的对象
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

### happens-before 规则总结

```
规则                    说明                         可见性保证
────────────────────────────────────────────────────────────────
程序顺序规则            同一线程内的语句按序执行        前面的写对后面的读可见
监视器锁规则            unlock happens-before lock     释放锁前的修改对获取锁后的操作可见
volatile 规则           写 happens-before 读          volatile 写对后续读可见
线程启动规则            start() happens-before 线程体  启动前的操作对线程体可见
线程终止规则            线程体 happens-before join()   线程的操作对 join 后的操作可见
中断规则                interrupt() happens-before 检测中断
终结器规则              构造函数 happens-before finalize
传递性                  A happens-before B, B happens-before C → A happens-before C
```

## C++ 内存模型（std::memory_order）

C++11 引入了细粒度的内存序控制，比 Java 的 volatile 更灵活。

### 六种内存序

```cpp
#include <atomic>
#include <thread>
#include <iostream>
#include <cassert>

/*
  memory_order_relaxed: 最宽松，只保证原子性，不保证顺序
  memory_order_acquire: 获取语义，后续读写不会被重排到此操作之前
  memory_order_release: 释放语义，之前读写不会被重排到此操作之后
  memory_order_acq_rel: 同时具有 acquire 和 release 语义
  memory_order_seq_cst: 顺序一致性（默认），最强保证
  memory_order_consume: 数据依赖（目前编译器通常退化为 acquire）
*/

std::atomic<bool> ready{false};
int data = 0;

void producer() {
    data = 42;                                          // (1)
    ready.store(true, std::memory_order_release);       // (2) release
    // 保证 (1) 在 (2) 之前完成
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {    // (3) acquire
        std::this_thread::yield();
    }
    // 保证 (3) 之后才能看到 (1) 的结果
    assert(data == 42); // 安全
    std::cout << "data = " << data << std::endl;
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join();
    t2.join();
    return 0;
}
```

### Release-Acquire 语义

```cpp
#include <atomic>
#include <thread>
#include <iostream>
#include <vector>

std::atomic<int> counter{0};
std::atomic<int> ready_count{0};
const int NUM_THREADS = 4;

void worker(int id, int& result) {
    result = id * 10; // 每个线程计算自己的结果
    ready_count.fetch_add(1, std::memory_order_release);
}

int main() {
    std::vector<int> results(NUM_THREADS);
    std::vector<std::thread> threads;

    for (int i = 0; i < NUM_THREADS; i++) {
        threads.emplace_back(worker, i, std::ref(results[i]));
    }

    // 等待所有线程完成
    while (ready_count.load(std::memory_order_acquire) < NUM_THREADS) {
        std::this_thread::yield();
    }

    for (int i = 0; i < NUM_THREADS; i++) {
        threads[i].join();
        std::cout << "线程 " << i << " 结果: " << results[i] << std::endl;
    }

    return 0;
}
```

### Relaxed 语义

```cpp
#include <atomic>
#include <thread>
#include <iostream>

std::atomic<int> x{0}, y{0};
std::atomic<int> r1{0}, r2{0};

void thread1() {
    x.store(1, std::memory_order_relaxed);  // 可能与其他操作重排
    r1.store(y.load(std::memory_order_relaxed), std::memory_order_relaxed);
}

void thread2() {
    y.store(1, std::memory_order_relaxed);
    r2.store(x.load(std::memory_order_relaxed), std::memory_order_relaxed);
}

int main() {
    // 在 relaxed 语义下，r1 == 0 && r2 == 0 是可能的
    // 因为 store 和 load 之间没有顺序约束
    for (int i = 0; i < 100000; i++) {
        x.store(0); y.store(0); r1.store(0); r2.store(0);

        std::thread t1(thread1);
        std::thread t2(thread2);
        t1.join(); t2.join();

        if (r1.load() == 0 && r2.load() == 0) {
            std::cout << "发现 relaxed 重排: r1=0, r2=0" << std::endl;
        }
    }
    return 0;
}
```

### Seq-Cst（顺序一致性）

```cpp
#include <atomic>
#include <thread>
#include <iostream>

std::atomic<bool> x_flag{false}, y_flag{false};
std::atomic<int> z{0};

void write_x() {
    x_flag.store(true, std::memory_order_seq_cst); // 顺序一致性
}

void write_y() {
    y_flag.store(true, std::memory_order_seq_cst);
}

void read_x_then_y() {
    while (!x_flag.load(std::memory_order_seq_cst));
    if (y_flag.load(std::memory_order_seq_cst)) {
        ++z;
    }
}

void read_y_then_x() {
    while (!y_flag.load(std::memory_order_seq_cst));
    if (x_flag.load(std::memory_order_seq_cst)) {
        ++z;
    }
}

int main() {
    // seq_cst 保证全局顺序，z 不可能为 0
    // 因为如果线程看到 x=true 但 y=false，那么其他线程必定看到 y=true 后 x 也必须为 true
    for (int i = 0; i < 100000; i++) {
        x_flag = false; y_flag = false; z = 0;

        std::thread t1(write_x);
        std::thread t2(write_y);
        std::thread t3(read_x_then_y);
        std::thread t4(read_y_then_x);
        t1.join(); t2.join(); t3.join(); t4.join();

        if (z.load() == 0) {
            std::cout << "seq_cst 违反（理论上不可能）" << std::endl;
        }
    }
    std::cout << "z 的最终值: " << z.load() << std::endl; // 至少为 1
    return 0;
}
```

## Rust 内存模型

Rust 的内存模型基于所有权和借用规则，在编译期消除数据竞争。`Ordering` 参数与 C++ 类似。

```rust
use std::sync::atomic::{AtomicBool, AtomicUsize, Ordering};
use std::thread;

fn main() {
    // Release-Acquire 配对
    let data = AtomicUsize::new(0);
    let ready = AtomicBool::new(false);

    let producer = thread::spawn(move || {
        data.store(42, Ordering::Relaxed); // 普通写入
        ready.store(true, Ordering::Release); // release: 之前的写入对 acquire 可见
    });

    let consumer = thread::spawn(move || {
        while !ready.load(Ordering::Acquire) {
            thread::yield_now();
        }
        // acquire 保证能看到 release 之前的写入
        let val = data.load(Ordering::Relaxed);
        println!("数据: {}", val); // 一定是 42
    });

    producer.join().unwrap();
    consumer.join().unwrap();

    // SeqCst: 最强保证
    let x = AtomicBool::new(false);
    let y = AtomicBool::new(false);
    let z = AtomicUsize::new(0);

    let handles: Vec<_> = (0..4).map(|i| {
        let x = &x;
        let y = &y;
        let z = &z;
        thread::spawn(move || {
            match i {
                0 => x.store(true, Ordering::SeqCst),
                1 => y.store(true, Ordering::SeqCst),
                2 => {
                    while !x.load(Ordering::SeqCst) {}
                    if y.load(Ordering::SeqCst) { z.fetch_add(1, Ordering::SeqCst); }
                }
                3 => {
                    while !y.load(Ordering::SeqCst) {}
                    if x.load(Ordering::SeqCst) { z.fetch_add(1, Ordering::SeqCst); }
                }
                _ => {}
            }
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
}
```

## volatile 关键字

### Java volatile

```java
public class VolatilePattern {
    // 使用 volatile 发布不可变对象
    private volatile ImmutableConfig config = new ImmutableConfig("default", 8080);

    public void reloadConfig() {
        // 一个线程写入
        config = new ImmutableConfig("production", 443);
    }

    public ImmutableConfig getConfig() {
        // 其他线程读取，保证看到最新值
        return config;
    }

    // volatile 用于状态标志
    private volatile boolean shutdownRequested = false;

    public void shutdown() {
        shutdownRequested = true;
    }

    public void run() {
        while (!shutdownRequested) {
            // 执行工作
        }
    }
}

class ImmutableConfig {
    final String env;
    final int port;
    ImmutableConfig(String env, int port) {
        this.env = env;
        this.port = port;
    }
}
```

### C++ volatile vs atomic

```cpp
#include <iostream>
#include <atomic>
#include <thread>
#include <chrono>

// C++ 中的 volatile 不保证线程安全！
// volatile 只是告诉编译器不要优化这个变量的访问
// 不提供原子性和内存序保证

// 错误：用 volatile 做线程同步
volatile bool flag = false;  // 不安全
int shared_data = 0;

void unsafe_writer() {
    shared_data = 42;
    flag = true;  // 可能被重排，shared_data 的写入可能不可见
}

// 正确：用 atomic
std::atomic<bool> safe_flag{false};
int safe_data = 0;

void safe_writer() {
    safe_data = 42;
    safe_flag.store(true, std::memory_order_release); // 安全
}

int main() {
    // volatile 的正确用途：
    // 1. 内存映射 IO（MMIO）
    // 2. 信号处理中的全局变量
    // 3. 与硬件交互
    // 4. setjmp/longjmp 场景

    // 线程间同步应使用 atomic
    safe_writer();
    while (!safe_flag.load(std::memory_order_acquire)) {
        std::this_thread::yield();
    }
    std::cout << "safe_data = " << safe_data << std::endl;

    return 0;
}
```

## 内存屏障（Memory Fence）

内存屏障强制处理器按照指定的顺序执行操作，防止指令重排。

```cpp
#include <atomic>
#include <thread>
#include <iostream>

std::atomic<bool> x(false), y(false);
std::atomic<int> z(0);

void thread1() {
    x.store(true, std::memory_order_relaxed);
    std::atomic_thread_fence(std::memory_order_release); // 释放屏障
    // 屏障前的写入不会被重排到屏障后
    y.store(true, std::memory_order_relaxed);
}

void thread2() {
    while (!y.load(std::memory_order_relaxed));
    std::atomic_thread_fence(std::memory_order_acquire); // 获取屏障
    // 屏障后的读取不会被重排到屏障前
    if (x.load(std::memory_order_relaxed)) {
        ++z;
    }
}

int main() {
    for (int i = 0; i < 10000; i++) {
        x = false; y = false; z = 0;
        std::thread t1(thread1);
        std::thread t2(thread2);
        t1.join(); t2.join();
        if (z.load() == 0) {
            std::cout << "发现重排（有屏障时理论上不可能）" << std::endl;
        }
    }
    std::cout << "z = " << z.load() << std::endl;
    return 0;
}
```

## Go 内存模型

Go 的内存模型规定了在什么条件下一个 goroutine 中的读取一定能观察到另一个 goroutine 中的写入。

```go
package main

import (
    "fmt"
    "sync"
    "sync/atomic"
)

func main() {
    // Go 内存模型：happens-before 关系
    // 1. 程序顺序：同 goroutine 内按序执行
    // 2. channel: 发送 happens-before 接收
    // 3. mutex: Unlock happens-before Lock
    // 4. Once: Do happens-before 返回
    // 5. goroutine 启动: go happens-before goroutine 执行
    // 6. goroutine 退出: 无保证（需要 sync/atomic 或 channel）

    // 正确方式 1: 使用 channel
    var data int
    ch := make(chan struct{})
    go func() {
        data = 42
        ch <- struct{}{} // 发送 happens-before 接收
    }()
    <-ch
    fmt.Println(data) // 保证是 42

    // 正确方式 2: 使用 mutex
    var mu sync.Mutex
    var value int
    mu.Lock()
    value = 100
    mu.Unlock()

    go func() {
        mu.Lock()
        fmt.Println(value) // 保证看到 100
        mu.Unlock()
    }()

    // 正确方式 3: 使用 atomic
    var counter int64
    atomic.StoreInt64(&counter, 42)
    val := atomic.LoadInt64(&counter)
    fmt.Println("atomic:", val)
}
```

| 特性 | Java (JMM) | C++ | Rust | Go |
