# 并发编程基础概念

## 并发与并行的本质区别

并发（Concurrency）是指程序的多个部分在逻辑上交替执行，而并行（Parallelism）是指多个部分在物理上同时执行。并发关注的是程序结构的设计，而并行关注的是执行的效率。

```
并发 (Concurrency)                并行 (Parallelism)

CPU1: [A][B][A][C][B]              CPU1: [A][A][A]
CPU2: [空闲]                        CPU2: [B][B][B]
                                    CPU3: [C][C][C]
```

并发不一定需要多核，但并行必须依赖多核硬件支持。

### 并发模型的三大流派

并发编程在历史上形成了三大主流模型，它们在底层原理、性能特征和适用场景上有本质区别：

```
┌─────────────────────────────────────────────────────────────┐
│                    并发模型三大流派                           │
├──────────────┬──────────────────┬────────────────────────────┤
│   共享内存    │    消息传递       │     CSP (通信顺序进程)      │
│  (Shared     │   (Message       │   (Communicating           │
│   Memory)    │    Passing)      │    Sequential Processes)   │
├──────────────┼──────────────────┼────────────────────────────┤
│ Java/C++     │ Erlang/Akka     │ Go (goroutine+channel)     │
│ Rust (Arc+   │ Rust (mpsc)     │ Occam                      │
│   Mutex)     │                  │                            │
├──────────────┼──────────────────┼────────────────────────────┤
│ 需要锁同步   │ 不可变消息        │ channel 作为一等公民        │
│ 性能高但复杂 │ 容错性好          │ "不要通过共享内存来通信，    │
│              │                  │  而要通过通信来共享内存"     │
└──────────────┴──────────────────┴────────────────────────────┘
```

### 操作系统层面的并发实现

在操作系统层面，并发的实现经历了从多进程到多线程再到协程的演进：

```
┌──────────────────────────────────────────────────────────────┐
│                    并发实现层次                                │
├────────────┬──────────────────┬───────────────────────────────┤
│   层次     │     机制          │        特征                   │
├────────────┼──────────────────┼───────────────────────────────┤
│ 硬件层     │ 超线程/SMT       │ 一个物理核心模拟两个逻辑核心   │
│ 操作系统层 │ 进程/线程        │ 内核调度，上下文切换开销大     │
│ 运行时层   │ 协程/goroutine   │ 用户态调度，切换开销极小       │
│ 语言层     │ async/await     │ 状态机编译，零运行时开销       │
└────────────┴──────────────────┴───────────────────────────────┘
```

**上下文切换的代价对比**：

```
操作类型                  典型耗时          说明
──────────────────────────────────────────────────────
函数调用                  ~1-5 ns         寄存器保存/恢复
协程切换 (goroutine)      ~100-300 ns     用户态栈切换
线程切换 (pthread)        ~1-10 μs        内核态切换 + TLB 刷新
进程切换 (fork/exec)      ~10-100 μs      完整地址空间切换
```

关键差异：goroutine 切换比线程切换快约 **20-50 倍**，因为 goroutine 切换发生在用户态，无需陷入内核、无需刷新 TLB（Translation Lookaside Buffer）、无需保存/恢复完整的 CPU 寄存器状态。

## 竞态条件（Race Condition）

竞态条件是指程序的输出依赖于多个线程的执行顺序，当调度顺序不同时，程序可能产生不同的结果。这是并发编程中最常见的错误之一。

### 经典示例：银行转账

```java
// 错误示范：存在竞态条件
public class BankAccount {
    private int balance;

    public BankAccount(int balance) {
        this.balance = balance;
    }

    // 两个线程同时调用此方法可能导致余额不一致
    public void transfer(BankAccount target, int amount) {
        if (this.balance >= amount) {
            this.balance -= amount;
            target.balance += amount;
        }
    }
}
```

当两个线程同时从账户 A 向账户 B 转账时，读取到的余额可能都是旧值，导致透支。

**竞态条件的底层分析**：

```
时间轴    线程 1                      线程 2             内存
──────────────────────────────────────────────────────────────
T1       读 balance = 100                                    A.balance = 100
T2                                读 balance = 100           A.balance = 100
T3       检查: 100 >= 50 ✓
T4                                检查: 100 >= 50 ✓
T5       写 balance = 50                                    A.balance = 50
T6                                写 balance = 50           A.balance = 50 (错误!)
──────────────────────────────────────────────────────────────
结果: A.balance = 50 (应为 0), 两次转账被"吞掉"了一次
```

问题根源：`balance -= amount` 不是原子操作。在 CPU 层面，它至少包含三条指令：
1. `LOAD balance` -> 寄存器
2. `SUB amount` -> 寄存器
3. `STORE 寄存器 -> balance`

线程调度可能发生在任何一条指令之间。

### 解决方案：细粒度锁排序

```java
// 使用 synchronized 消除竞态条件，通过锁排序避免死锁
public void transfer(BankAccount target, int amount) {
    // 按照对象哈希值排序锁，避免死锁
    BankAccount first = this;
    BankAccount second = target;
    if (System.identityHashCode(first) > System.identityHashCode(second)) {
        first = target;
        second = this;
    }

    synchronized (first) {
        synchronized (second) {
            if (this.balance >= amount) {
                this.balance -= amount;
                target.balance += amount;
            }
        }
    }
}
```

### 解决方案：原子操作（CAS）

```java
import java.util.concurrent.atomic.AtomicInteger;

public class LockFreeBankAccount {
    private final AtomicInteger balance;

    public LockFreeBankAccount(int balance) {
        this.balance = new AtomicInteger(balance);
    }

    // 使用 CAS (Compare-And-Swap) 实现无锁转账
    public boolean transfer(LockFreeBankAccount target, int amount) {
        while (true) {
            int current = this.balance.get();
            if (current < amount) return false;

            // CAS: 如果当前值仍是 current，则更新为 current - amount
            if (this.balance.compareAndSet(current, current - amount)) {
                target.balance.addAndGet(amount);
                return true;
            }
            // CAS 失败说明其他线程修改了 balance，自动重试
        }
    }
}
```

**CAS 的底层原理**：

CAS 是 CPU 提供的原子指令（x86 上是 `CMPXCHG`），它在一条指令内完成"比较+交换"两个操作：

```
CAS(addr, expected, new_value):
  原子地执行:
    if *addr == expected:
        *addr = new_value
        return true
    else:
        return false
```

CAS 的问题：
- **ABA 问题**：值从 A 变 B 再变回 A，CAS 无法察觉。解决方案：使用版本号（`AtomicStampedReference`）。
- **自旋开销**：高竞争下 CAS 自旋浪费 CPU。解决方案：超过阈值退化为锁。
- **只能保证单个变量的原子性**：多变量原子操作需要锁或事务内存。

## 死锁（Deadlock）

死锁是指两个或多个线程互相持有对方需要的锁，导致所有线程都无法继续执行。

### 死锁产生的四个必要条件

1. **互斥条件**：资源一次只能被一个线程持有
2. **持有并等待**：线程持有一个锁的同时等待另一个锁
3. **不可剥夺**：锁只能由持有它的线程释放
4. **循环等待**：存在一个线程链，每个线程都在等待下一个线程持有的锁

```
线程 A                    线程 B
  │                         │
  ├─ 持有锁 X               ├─ 持有锁 Y
  │                         │
  ├─ 等待锁 Y ◄────────────┼─ 等待锁 X
  │                         │
  ▼ 死锁                    ▼ 死锁
```

### 死锁示例

```cpp
#include <mutex>
#include <thread>
#include <iostream>

std::mutex mutex_a, mutex_b;

void thread_1() {
    std::lock_guard<std::mutex> lock_a(mutex_a);
    std::this_thread::sleep_for(std::chrono::milliseconds(10));
    std::lock_guard<std::mutex> lock_b(mutex_b); // 可能死锁
    std::cout << "Thread 1 acquired both locks\n";
}

void thread_2() {
    std::lock_guard<std::mutex> lock_b(mutex_b);
    std::this_thread::sleep_for(std::chrono::milliseconds(10));
    std::lock_guard<std::mutex> lock_a(mutex_a); // 可能死锁
    std::cout << "Thread 2 acquired both locks\n";
}
```

### 死锁的检测与预防

**检测方法：资源分配图**

```
构建有向图:
  节点: 线程 T1, T2... 和 锁 L1, L2...
  边: T1 -> L1 (T1 请求 L1)
      L1 -> T2 (T2 持有 L1)

如果图中存在环路，则存在死锁风险。
```

**预防策略对比**：

```
策略                    方法                                   代价
──────────────────────────────────────────────────────────────────────
锁排序                  所有线程按相同顺序获取锁                需要全局规划
一次性申请              一次性获取所有需要的锁                  可能降低并发度
超时放弃                tryLock 超时后释放已持有锁              需要处理重试逻辑
死锁检测                周期性检查资源分配图是否有环            检测本身有开销
```

### C++ 中的安全锁获取

```cpp
#include <mutex>
#include <thread>

std::mutex m1, m2;

// 方法1: std::lock 同时获取多个锁，内部自动避免死锁
void safe_lock_both() {
    std::lock(m1, m2); // 同时锁定两个 mutex
    std::lock_guard<std::mutex> lg1(m1, std::adopt_lock);
    std::lock_guard<std::mutex> lg2(m2, std::adopt_lock);
    // 安全地使用两个锁...
}

// 方法2: 锁排序（通过地址比较）
void ordered_lock() {
    auto first = &m1;
    auto second = &m2;
    if (first > second) std::swap(first, second);

    std::lock_guard<std::mutex> lg1(*first);
    std::lock_guard<std::mutex> lg2(*second);
    // 安全地使用两个锁...
}
```

### 死锁在真实项目中的案例

**MySQL InnoDB 死锁**：事务 T1 锁定行 A 后尝试锁定行 B，事务 T2 锁定行 B 后尝试锁定行 A。InnoDB 使用 wait-for graph 检测死锁，并自动回滚代价最小的事务。

```sql
-- 事务 1
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
-- 此时持有 id=1 的行锁
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
-- 等待 id=2 的行锁

-- 事务 2（同时执行）
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;
-- 此时持有 id=2 的行锁
UPDATE accounts SET balance = balance + 50 WHERE id = 1;
-- 等待 id=1 的行锁 → 死锁!
```

## 活锁（Livelock）

活锁是指线程不断重复执行相同的操作，但始终无法取得进展。与死锁不同的是，活锁中的线程是活跃的，但它们在互相"让路"中浪费资源。

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

type Resource struct {
    name  string
    mutex sync.Mutex
}

// 活锁示例：两个人互相让路
func livelockDemo() {
    hallway := make(chan bool, 1)
    var wg sync.WaitGroup
    wg.Add(2)

    go func() {
        defer wg.Done()
        for i := 0; i < 5; i++ {
            select {
            case hallway <- true:
                fmt.Println("Person A walks through")
                <-hallway
                return
            default:
                fmt.Println("Person A steps aside")
                time.Sleep(100 * time.Millisecond)
            }
        }
    }()

    go func() {
        defer wg.Done()
        for i := 0; i < 5; i++ {
            select {
            case hallway <- true:
                fmt.Println("Person B walks through")
                <-hallway
                return
            default:
                fmt.Println("Person B steps aside")
                time.Sleep(100 * time.Millisecond)
            }
        }
    }()

    wg.Wait()
}
```

### 活锁的解决：随机退避

```go
import (
    "math/rand"
    "time"
)

func tryWithBackoff(fn func() bool, maxRetries int) bool {
    for i := 0; i < maxRetries; i++ {
        if fn() {
            return true
        }
        // 随机退避，打破对称性
        backoff := time.Duration(rand.Intn(1<<uint(i))) * time.Millisecond
        time.Sleep(backoff)
    }
    return false
}
```

随机退避的核心思想：通过引入随机性打破线程行为的对称性。如果所有线程都使用相同的退避策略（比如固定等待 100ms），它们可能再次同时尝试，导致持续活锁。随机延迟使得各线程的重试时刻错开。

## 饥饿（Starvation）

饥饿是指某个线程长时间无法获取所需的资源，导致无法推进。常见原因包括：

- 高优先级线程持续占用 CPU
- 锁被其他线程频繁获取
- 不公平的调度策略

```java
// 饥饿示例：不公平锁导致某些线程得不到执行
public class StarvationDemo {
    private final ReentrantLock lock = new ReentrantLock(false); // 非公平锁

    public void greedyTask() {
        while (true) {
            lock.lock();
            try {
                // 持续占用锁，其他线程可能饥饿
                Thread.sleep(1);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            } finally {
                lock.unlock();
            }
        }
    }

    // 解决方案：使用公平锁
    private final ReentrantLock fairLock = new ReentrantLock(true); // 公平锁
}
```

### 公平锁 vs 非公平锁的性能分析

```
锁类型     吞吐量    饥饿风险    上下文切换次数
──────────────────────────────────────────────
非公平锁   高        有          少
公平锁     低        无          多
```

为什么非公平锁吞吐量更高？因为新来的线程可能在等待队列中的线程被唤醒之前就获取到锁（"插队"），减少了上下文切换。但代价是等待队列中的线程可能长时间得不到执行。

**Java 中的公平性控制**：

```java
// ReentrantLock: 可选公平/非公平
ReentrantLock fair = new ReentrantLock(true);   // 公平
ReentrantLock unfair = new ReentrantLock(false); // 非公平（默认）

// synchronized: 非公平，但 JVM 有自适应策略
// 在 Java 6+ 中，synchronized 经历 偏向锁 -> 轻量级锁 -> 重量级锁 的升级
```

## 线程安全（Thread Safety）

线程安全是指代码在多线程环境下能够正确执行，不产生意外结果。实现线程安全的方式包括：

### 1. 不可变对象

不可变对象天然是线程安全的，因为其状态在创建后不会改变。

```java
public final class ImmutablePoint {
    private final int x;
    private final int y;

    public ImmutablePoint(int x, int y) {
        this.x = x;
        this.y = y;
    }

    // 没有 setter 方法，字段都是 final
    public int getX() { return x; }
    public int getY() { return y; }
}
```

**不可变性的 JVM 保证**：

`final` 关键字在 JVM 层面提供了特殊的内存语义：在构造函数中对 `final` 字段的写入，对其他线程可见，无需额外同步。这是因为 JVM 在构造函数末尾插入了 `StoreStore` 内存屏障。

### 2. 同步机制

使用互斥锁、读写锁等同步机制保护共享状态。

```rust
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Final count: {}", *counter.lock().unwrap());
}
```

### 3. Thread-Local 存储

每个线程拥有自己的数据副本，避免共享。

```python
import threading

local_data = threading.local()

def worker():
    local_data.value = threading.current_thread().name
    print(f"Thread {local_data.value} has its own copy")

threads = [threading.Thread(target=worker) for _ in range(3)]
for t in threads:
    t.start()
for t in threads:
    t.join()
```

### 4. 无锁数据结构

无锁（Lock-Free）数据结构使用原子操作（CAS）代替锁，提供更好的并发性能和无死锁保证。

```java
import java.util.concurrent.atomic.AtomicReference;

// 无锁栈（Treiber Stack）
public class LockFreeStack<T> {
    private final AtomicReference<Node<T>> top = new AtomicReference<>(null);

    private static class Node<T> {
        final T value;
        Node<T> next;
        Node(T value, Node<T> next) {
            this.value = value;
            this.next = next;
        }
    }

    public void push(T value) {
        Node<T> newNode;
        while (true) {
            Node<T> currentTop = top.get();
            newNode = new Node<>(value, currentTop);
            if (top.compareAndSet(currentTop, newNode)) {
                return; // CAS 成功
            }
            // CAS 失败，重试
        }
    }

    public T pop() {
        while (true) {
            Node<T> currentTop = top.get();
            if (currentTop == null) return null;
            Node<T> newTop = currentTop.next;
            if (top.compareAndSet(currentTop, newTop)) {
                return currentTop.value;
            }
        }
    }
}
```

### 线程安全级别分类

Herb Sutter 将并发数据结构分为四个安全级别：

```
级别          定义                                        实现方式
──────────────────────────────────────────────────────────────────────
非线程安全    完全不支持并发访问                            无同步
条件安全      多线程可以同时读，但写需要独占                 读写锁
线程安全      多线程同时操作不同实例安全，                   互斥锁
              同一实例需要外部同步
完全线程安全  多线程同时操作同一实例安全                     内置同步/无锁
```

## Happens-Before 关系

Happens-before 是内存模型中定义操作可见性和顺序性的核心概念。如果操作 A happens-before 操作 B，则 A 的结果对 B 可见。

### Java 中的 happens-before 规则

```
规则                          说明
─────────────────────────────────────────────────
程序顺序规则                 同一线程中，前面的操作 happens-before 后面的操作
监视器锁规则                 解锁操作 happens-before 后续的加锁操作
volatile 变量规则           写 volatile happens-before 读 volatile
线程启动规则                 Thread.start() happens-before 该线程的任何操作
线程终止规则                 线程的所有操作 happens-before 其他线程检测到该线程终止
传递性                      A happens-before B, B happens-before C → A happens-before C
```

### 代码示例

```java
public class HappensBeforeDemo {
    private int value = 0;
    private volatile boolean ready = false;

    // 线程 1
    public void writer() {
        value = 42;          // (1)
        ready = true;        // (2) volatile 写
    }

    // 线程 2
    public void reader() {
        if (ready) {         // (3) volatile 读
            // 根据 volatile 规则，(1) happens-before (3)
            // 所以这里一定能看到 value == 42
            System.out.println(value);
        }
    }
}
```

### 内存屏障的底层实现

Happens-before 在硬件层面由内存屏障（Memory Barrier/Fence）实现：

```
屏障类型        x86 指令        ARM 指令        作用
──────────────────────────────────────────────────────────────
StoreStore      (隐含)         DMB ISHST      确保之前的写操作先于之后的写操作完成
LoadLoad        (隐含)         DMB ISHLD      确保之前的读操作先于之后的读操作完成
StoreLoad       MFENCE         DMB ISH        确保之前的写操作先于之后的读操作完成
Full            MFENCE         DMB ISH        完全屏障
```

x86 是 TSO（Total Store Order）内存模型，天然保证了 StoreStore、LoadLoad、LoadStore 的顺序性，只需要 StoreLoad 屏障（通过 `MFENCE` 或带 `lock` 前缀的指令）。ARM 等弱内存模型架构需要显式插入更多屏障。

### volatile 的硬件语义

```
volatile 写 = 普通写 + StoreStore屏障 + StoreLoad屏障
volatile 读 = LoadLoad屏障 + LoadLoad屏障 + 普通读

编译器层面: 禁止将 volatile 变量缓存到寄存器，禁止对 volatile 操作重排序
CPU 层面: 插入内存屏障，使修改对其他核心可见
```

## 内存可见性问题

多线程环境下，一个线程对共享变量的修改可能对其他线程不可见，因为：

1. **CPU 缓存**：每个核心有自己的缓存，修改可能只存在于本地缓存
2. **编译器优化**：编译器可能将变量缓存到寄存器中
3. **指令重排**：处理器可能改变指令执行顺序

```
线程 1 (Core 1)          线程 2 (Core 2)
    │                        │
    ▼                        ▼
L1 缓存: x=1             L1 缓存: x=0
    │                        │
    │  写 x=2                │  读 x → 0 (看不到线程1的修改)
    ▼                        ▼
内存: x=0 (未同步)        内存: x=0
```

### 缓存一致性协议：MESI

现代 CPU 通过 MESI 协议解决缓存一致性问题：

```
状态          含义                                  其他核心的缓存副本
────────────────────────────────────────────────────────────────────
Modified      已修改，与内存不一致                  无
Exclusive     独占，与内存一致                      无
Shared        共享，与内存一致                      可能有
Invalid       无效，数据已过期                      可能有

写操作流程 (Core 0 想写变量 X):
1. Core 0 向总线发送 Invalidate 请求
2. 其他核心将各自的 X 缓存行标记为 Invalid
3. Core 0 将 X 标记为 Modified，执行写入
4. 其他核心再次读 X 时，从 Core 0 的缓存获取最新值
```

MESI 保证了一致性，但不保证即时可见性——写入可能延迟刷新到内存。这就是为什么需要 `volatile` 或内存屏障来保证及时可见。

### Store Buffer 与失效队列

```
CPU Core 0                    CPU Core 1
    │                             │
    ▼                             ▼
 Store Buffer                Invalidate Queue
 (写操作暂存)                (失效请求暂存)
    │                             │
    ▼                             ▼
 L1 Cache ──── MESI 协议 ──── L1 Cache
    │                             │
    ▼                             ▼
 共享内存 (RAM)
```

为了进一步优化性能，CPU 引入了 Store Buffer 和 Invalidate Queue：
- **Store Buffer**：写操作先写入 Store Buffer，稍后才刷入缓存。这允许 CPU 继续执行而不必等待写操作完成。
- **Invalidate Queue**：收到失效请求后立即确认，稍后再处理。

这就是为什么即使有 MESI 协议，仍需要内存屏障——屏障确保 Store Buffer 被排空。

## 指令重排序

编译器和处理器为了优化性能，可能改变指令的执行顺序。在单线程中这是安全的（as-if-serial 语义），但在多线程中可能导致意外行为。

### 经典示例：DCL (Double-Checked Locking)

```java
// 错误的 DCL（Java 1.4 及之前）
public class Singleton {
    private static Singleton instance; // 没有 volatile!

    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {            // 第二次检查（有锁）
                    instance = new Singleton();    // 问题所在!
                }
            }
        }
        return instance;
    }
}
```

`instance = new Singleton()` 被分解为三步：
1. 分配内存
2. 调用构造函数初始化
3. 将引用赋给 `instance`

但步骤 2 和 3 可能被重排序为：1 → 3 → 2。此时另一个线程可能在第一次检查时看到非 null 的 `instance`，但其指向的对象尚未初始化完成。

**正确做法：加 `volatile`**

```java
private static volatile Singleton instance; // 禁止重排序
```

### x86 与 ARM 的重排序差异

```
重排序类型       x86 允许?     ARM 允许?     说明
────────────────────────────────────────────────────────
Load-Load       禁止          允许          x86 保证
Load-Store      禁止          允许          x86 保证
Store-Store     禁止          允许          x86 保证
Store-Load      允许          允许          x86 也允许!
```

x86 只允许 Store-Load 重排序（即写后读可以重排序），而 ARM 等弱内存模型允许所有四种重排序。这也是为什么在 x86 上测试通过的并发代码，可能在 ARM 上失败。

```
语言      内存模型          默认保证                显式同步关键字
────────────────────────────────────────────────────────────────────
```

### C++11 内存顺序详解

```cpp
#include <atomic>
#include <thread>

std::atomic<bool> ready{false};
int data = 0;

void producer() {
    data = 42;                                          // (1)
    ready.store(true, std::memory_order_release);       // (2) release
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {}   // (3) acquire
    assert(data == 42);                                 // (4) 一定成立
}

// memory_order_relaxed: 只保证原子性，不保证顺序
// memory_order_acquire: 读操作，之后的读写不能重排到此之前
// memory_order_release: 写操作，之前的读写不能重排到此之后
// memory_order_seq_cst: 完全顺序一致性（默认），最安全但最慢
```

## 超线程（Hyper-Threading）与并发

超线程（Intel HT / AMD SMT）是硬件层面的并发技术：

```
传统双核:
  Core 0: [线程 A 完整执行]
  Core 1: [线程 B 完整执行]

超线程双核（4逻辑核心）:
  Core 0: [线程 A ──── 等待内存 ──── 线程 A ────]
           [线程 B ──── 线程 B ──── 等待内存 ────]
  Core 1: [线程 C ──── 等待内存 ──── 线程 C ────]
           [线程 D ──── 线程 D ──── 等待内存 ────]
```

原理：当一个逻辑核心在等待内存数据时，另一个逻辑核心可以利用执行单元进行计算。超线程通常提升 **15-30%** 吞吐量，而非翻倍。对于计算密集型负载，超线程的提升很小，因为两个逻辑核心共享执行单元。

## 完整工程级示例：并发安全的缓存系统

```java
import java.util.concurrent.*;
import java.util.concurrent.locks.*;

// 支持过期的并发安全缓存
public class ConcurrentCache<K, V> {
    private final ConcurrentHashMap<K, CacheEntry<V>> map = new ConcurrentHashMap<>();
    private final ScheduledExecutorService cleanupExecutor =
        Executors.newSingleThreadScheduledExecutor();

    private static class CacheEntry<V> {
        final V value;
        final long expireAt;

        CacheEntry(V value, long ttlMs) {
            this.value = value;
            this.expireAt = System.currentTimeMillis() + ttlMs;
        }

        boolean isExpired() {
            return System.currentTimeMillis() > expireAt;
        }
    }

    public ConcurrentCache() {
        // 定期清理过期条目
        cleanupExecutor.scheduleAtFixedRate(this::evictExpired,
            1, 1, TimeUnit.MINUTES);
    }

    public void put(K key, V value, long ttlMs) {
        map.put(key, new CacheEntry<>(value, ttlMs));
    }

    public V get(K key) {
        CacheEntry<V> entry = map.get(key);
        if (entry == null || entry.isExpired()) {
            if (entry != null) map.remove(key);
            return null;
        }
        return entry.value;
    }

    // 原子性地获取或计算（防止缓存击穿）
    public V getOrCompute(K key, long ttlMs, Callable<V> compute) throws Exception {
        CacheEntry<V> entry = map.get(key);
        if (entry != null && !entry.isExpired()) {
            return entry.value;
        }

        // 使用 putIfAbsent 保证只有一个线程执行计算
        V value = compute.call();
        map.put(key, new CacheEntry<>(value, ttlMs));
        return value;
    }

    private void evictExpired() {
        map.entrySet().removeIf(e -> e.getValue().isExpired());
    }

    public void shutdown() {
        cleanupExecutor.shutdown();
    }
}
```

## 总结

| 概念 | 核心问题 | 解决思路 |
|------|---------|---------|
| 竞态条件 | 操作顺序不确定 | 同步机制、原子操作、CAS |
| 死锁 | 互相等待 | 锁排序、超时、一次性申请、死锁检测 |
| 活锁 | 互相让路 | 随机退避 |
| 饥饿 | 资源分配不均 | 公平调度、优先级老化 |
| 可见性 | 缓存不一致 | volatile、内存屏障、MESI 协议 |
| 重排序 | 指令执行顺序改变 | 内存屏障、happens-before |
| 线程安全 | 多线程并发正确性 | 不可变对象、锁、原子操作、TLS |
