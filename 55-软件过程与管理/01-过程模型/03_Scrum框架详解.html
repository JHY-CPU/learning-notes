Scrum框架详解

一、Scrum概述

Scrum是一个轻量级的敏捷开发框架，通过迭代和增量的方式交付产品。它定义了角色、事件和工件三个核心组成部分。

1.1 Scrum三大支柱

支柱含义实践体现
透明性 (Transparency)过程对所有参与者可见Sprint看板、每日站会、信息辐射器
检视 (Inspection)频繁检查进展和工件Sprint评审、Sprint回顾
适应 (Adaptation)发现问题时及时调整每日站会调整计划、回顾改进

二、Scrum角色

2.1 三种角色

角色职责常见误区
产品负责人 (Product Owner)最大化产品价值；管理产品待办列表；定义验收标准；代表利益相关方不是"需求传话筒"，需有决策权
Scrum Master确保Scrum被正确理解和服务；移除障碍；促进团队自组织；保护团队免受干扰不是项目经理，不分配任务
开发团队 (Developers)自组织地交付产品增量；跨职能（具备所有必要技能）；共同对增量负责不只是程序员，包含所有需要的角色

2.2 团队规模

  • 开发团队：3-9人（不含PO和SM）
  • 太小（<3人）：缺乏协作和技能互补
  • 太大（>9人）：沟通开销增加，协调复杂
  • 多个Scrum团队共享同一PO和产品待办列表

三、Scrum事件（Ceremonies）

3.1 Sprint

Sprint是Scrum的核心，是一个固定时间的迭代周期（通常1-4周）。在Sprint期间，不允许变更Sprint目标，质量和目标范围不降低。

3.2 五大事件

事件时间盒参与者目的产出
Sprint计划会Sprint长度的2h/周整个Scrum团队确定Sprint目标和待办项Sprint待办列表 + Sprint目标
每日站会15分钟开发团队（SM主持）同步进展，识别障碍当日计划调整
Sprint评审会Sprint长度的1h/周Scrum团队+利益相关方演示增量，收集反馈更新的产品待办列表
Sprint回顾会Sprint长度的1.5h/周Scrum团队检视过程，持续改进改进措施清单
Sprint本身固定（1-4周）整个Scrum团队交付可用的产品增量完成的增量

3.3 每日站会格式

每日站会三个问题（传统格式）：
1. 昨天完成了什么？
2. 今天计划做什么？
3. 有什么阻碍？

改进格式（面向目标）：
每个人依次回答：为了达成Sprint目标，我接下来要做什么？
重点讨论阻碍和需要协作的事项

3.4 Sprint评审 vs 回顾

维度Sprint评审会Sprint回顾会
关注点产品（做了什么）过程（怎么做的）
参与者含利益相关方仅Scrum团队
氛围展示和反馈反思和改进
产出调整产品待办列表过程改进措施

四、Scrum工件 (Artifacts)

4.1 三大工件

工件描述承诺
产品待办列表 (Product Backlog)产品所有待实现功能的有序列表产品目标 (Product Goal)
Sprint待办列表 (Sprint Backlog)Sprint计划完成的待办项 + 实现计划Sprint目标 (Sprint Goal)
增量 (Increment)Sprint结束时可用的产品增量完成的定义 (Definition of Done)

4.2 产品待办列表精炼

  • DEEP原则：Detailled appropriately（详略得当）、Emergent（涌现的）、Estimated（估算过的）、Prioritized（排优先级的）
  • 用户故事格式：作为[角色]，我想要[功能]，以便[价值]
  • INVEST标准：Independent、Negotiable、Valuable、Estimable、Small、Testable

4.3 完成的定义 (Definition of Done)

Definition of Done 示例：
- [x] 代码编写完成并通过代码审查
- [x] 单元测试通过（覆盖率≥80%）
- [x] 集成测试通过
- [x] 无P1/P2级别Bug
- [x] API文档已更新
- [x] 通过产品负责人的验收
- [x] 可部署到生产环境

DoD vs 验收标准：
DoD是所有待办项的通用标准；验收标准是单个待办项的具体条件

五、常见反模式与改进建议

5.1 常见问题

反模式表现改进建议
Scrum-but"我们用Scrum，但是不做回顾会"先完整执行再调整
PO缺失PO不在场或无决策权确保PO全职参与
Sprint目标模糊待办项列表无明确目标每个Sprint定义清晰目标
站会变汇报成员向SM汇报而非团队同步面向团队，聚焦障碍
Sprint中插入需求频繁打断Sprint计划保护Sprint，紧急需求走S外流程
估算变成承诺故事点被当作deadline估算用于规划，非绩效考核
Scrum的精髓：Scrum不是一系列流程的集合，而是一个帮助团队发现工作方式的框架。关键不在于执行了哪些仪式，而在于是否真正实现了透明、检视和适应的良性循环。
