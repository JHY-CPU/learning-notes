# 01-过程模型

## 1. 瀑布模型（Waterfall）

瀑布模型是最早也是最经典的软件开发过程模型，由Winston Royce于1970年提出。

### 阶段划分

```
需求分析 → 系统设计 → 编码实现 → 测试 → 部署 → 维护
```

每个阶段完成后才进入下一阶段，如同瀑布般线性流动。

### 特点

- **文档驱动**：每个阶段产出详细文档
- **阶段性强**：严格的阶段评审和里程碑
- **计划性强**：适合需求明确、变更少的项目
- **易于管理**：进度可预测，便于成本估算

### 适用场景

- 需求明确且稳定的项目（如政府项目、嵌入式系统）
- 合同制开发，需要明确交付物
- 安全关键系统（航天、医疗）

### 局限性

- 需求变更代价极高
- 客户在后期才能看到产品
- 飝险后置，问题发现晚

## 2. V模型

V模型是瀑布模型的扩展，强调验证和确认活动与开发阶段的对应关系。

### 结构

```
需求分析  ←——————————→  验收测试
  ↓                        ↑
系统设计  ←——————————→  系统测试
  ↓                        ↑
详细设计  ←——————————→  集成测试
  ↓                        ↑
  编码  ←——————————→  单元测试
```

### 核心思想

- 左侧是开发活动（从上到下细化）
- 右侧是测试活动（从下到上验证）
- 每个开发阶段都有对应的测试阶段

### 优势

- 强调测试的重要性
- 测试计划可提前编写
- 质量保证贯穿整个开发过程

## 3. 增量模型

增量模型将系统划分为多个增量，每个增量交付一部分功能。

### 过程

```
增量1：核心功能 → 设计 → 开发 → 测试 → 交付
增量2：扩展功能 → 设计 → 开发 → 测试 → 交付
增量3：增强功能 → 设计 → 开发 → 测试 → 交付
```

### 特点

- 早期交付核心功能
- 降低一次性交付的风险
- 每个增量可独立验证
- 允许需求逐步明确

### 与迭代的区别

- **增量**：每次增加新功能
- **迭代**：每次完善已有功能
- 实践中两者常结合使用

## 4. 螺旋模型（Spiral）

螺旋模型由Barry Boehm提出，将瀑布模型与原型模型结合，以风险管理为核心。

### 四个象限（每次迭代）

```
1. 确定目标和约束
       ↓
2. 识别和分析风险
       ↓
3. 开发和验证原型/产品
       ↓
4. 计划下一轮迭代
```

### 核心特点

- **风险驱动**：每次迭代重点解决最紧迫的风险
- **原型化**：通过原型降低不确定性
- **灵活的迭代周期**：根据风险大小决定迭代范围
- **客户参与**：每次迭代都有客户评审

### 适用场景

- 高风险、大规模项目
- 需求不确定的创新项目
- 需要频繁验证假设的项目

## 5. 原型模型

原型模型通过快速构建原型来澄清需求、验证设计。

### 原型类型

| 类型 | 目的 | 处理方式 |
|------|------|---------|
| 抛弃式原型 | 快速探索需求 | 验证后丢弃 |
| 演进式原型 | 逐步完善产品 | 演进为最终产品 |
| 增量式原型 | 逐个功能构建 | 合并为最终产品 |

### 过程

```
需求初步分析 → 快速设计 → 构建原型 → 客户评估 → 修改需求 → 正式开发
```

### 优势

- 降低需求不确定性
- 早期获得用户反馈
- 减少后期变更

### 风险

- 客户可能将原型误认为最终产品
- 过度投入原型开发
- 技术债务可能累积

## 6. 敏捷开发宣言与原则

2001年，17位软件开发专家发布了《敏捷软件开发宣言》。

### 四大价值观

```
个体和互动    高于  流程和工具
可工作的软件  高于  详尽的文档
客户合作      高于  合同谈判
响应变化      高于  遵循计划
```

> 右侧项并非没有价值，但我们更重视左侧项。

### 十二项原则（精简版）

1. 尽早和持续交付有价值的软件
2. 欢迎需求变更，即使在开发后期
3. 频繁交付可工作的软件（周为单位）
4. 业务人员和开发人员密切合作
5. 围绕有积极性的个人构建项目
6. 面对面交谈是最有效的沟通方式
7. 可工作的软件是首要的进度度量
8. 可持续的开发节奏
9. 持续关注技术卓越和良好设计
10. 简洁——最大化未完成工作量的艺术
11. 自组织团队产生最好的架构
12. 团队定期反思并调整行为

## 7. Scrum框架

Scrum是最流行的敏捷框架，由Ken Schwaber和Jeff Sutherland提出。

### 三个角色

| 角色 | 职责 |
|------|------|
| **产品负责人（PO）** | 管理产品待办列表，定义优先级，代表客户利益 |
| **Scrum Master** | 移除障碍，确保Scrum流程执行，促进团队自组织 |
| **开发团队** | 跨职能、自组织的团队，负责交付增量 |

### 五个事件

1. **Sprint**：固定长度的迭代周期（通常2-4周）
2. **Sprint计划会议**：确定Sprint目标和待办事项
3. **每日站会**：15分钟同步进度、计划和障碍
4. **Sprint评审**：展示增量，收集反馈
5. **Sprint回顾**：反思过程，识别改进项

### 三个工件

| 工件 | 说明 |
|------|------|
| **产品待办列表（Product Backlog）** | 按优先级排序的需求列表 |
| **Sprint待办列表（Sprint Backlog）** | 当前Sprint选择的需求和计划 |
| **增量（Increment）** | Sprint结束时交付的可工作产品 |

## 8. 看板方法（Kanban）

看板源自丰田生产系统，是一种渐进式改进的方法。

### 核心原则

1. **从现有流程开始**：不改变当前角色和流程
2. **同意渐进式变革**：小步改进而非大刀阔斧
3. **尊重当前的角色和职责**
4. **鼓励各层级的领导力**

### 六大实践

1. **可视化工作流**：使用看板板展示工作流程
2. **限制在制品（WIP）**：每列设置最大进行中工作数
3. **管理工作流动**：识别和消除瓶颈
4. **明确流程策略**：定义"完成"的标准
5. **建立反馈环**：定期评审和改进
6. **协作改进**：实验性演进

### 看板板示例

```
| 待办 | 开发中(WIP:3) | 测试中(WIP:2) | 已完成 |
|------|---------------|---------------|--------|
| T-5  | T-2           | T-1           | T-8    |
| T-6  | T-3           | T-4           | T-9    |
| T-7  |               |               |        |
```

### 与Scrum的区别

| 维度 | Scrum | 看板 |
|------|-------|------|
| 迭代 | 固定Sprint | 持续流动 |
| 角色 | 定义明确 | 不强制 |
| 变更 | Sprint内不变更 | 随时可加入 |
| 度量 | 速度（Velocity） | 前置时间、周期时间 |

## 9. 极限编程（XP）实践

极限编程（eXtreme Programming）由Kent Beck提出，强调技术实践。

### 核心实践

| 实践 | 说明 |
|------|------|
| **结对编程** | 两人共用一台电脑编程 |
| **测试驱动开发（TDD）** | 先写测试，再写代码 |
| **持续集成** | 频繁集成代码，每次集成自动化测试 |
| **简单设计** | 只实现当前需求，避免过度设计 |
| **重构** | 持续改进代码结构 |
| **集体代码所有权** | 任何人均可修改任何代码 |
| **编码规范** | 统一编码风格 |
| **小型发布** | 频繁发布小版本 |
| **系统隐喻** | 用简单的故事描述系统工作方式 |
| **可持续节奏** | 每周40小时，不加班 |

### XP的价值观

- **沟通**：团队内和与客户的持续沟通
- **简单**：选择最简单的可行方案
- **反馈**：通过测试和客户反馈获取信息
- **勇气**：敢于面对和解决问题
- **尊重**：团队成员相互尊重

## 10. DevOps与持续交付

DevOps是开发（Development）和运维（Operations）的结合，旨在缩短交付周期。

### 核心原则

- **文化**：打破开发与运维的壁垒
- **自动化**：构建、测试、部署全流程自动化
- **精益**：减少浪费，快速反馈
- **度量**：用数据驱动改进
- **分享**：经验和知识在团队间流动

### 持续交付流水线

```
代码提交 → 自动构建 → 单元测试 → 集成测试 → 部署到预发布 → 验收测试 → 部署到生产
```

### 关键实践

| 实践 | 工具示例 |
|------|---------|
| 持续集成（CI） | Jenkins、GitLab CI、GitHub Actions |
| 持续交付/部署（CD） | ArgoCD、Spinnaker、Flux |
| 基础设施即代码（IaC） | Terraform、Ansible、Pulumi |
| 容器化 | Docker、Kubernetes |
| 监控与可观测性 | Prometheus、Grafana、ELK |

### DevOps成熟度

```
Level 0：手动构建和部署
Level 1：自动化构建和测试
Level 2：自动化部署到部分环境
Level 3：全自动化CI/CD流水线
Level 4：自助服务平台，GitOps
```

## 11. 精益软件开发

精益软件开发源自丰田生产系统（TPS），由Mary和Tom Poppendieck引入软件领域。

### 七大原则

1. **消除浪费**：消除不增值的活动
2. **增强学习**：通过短周期迭代学习
3. **尽量延迟决策**：在最后责任时刻做决策
4. **尽快交付**：缩短交付周期
5. **赋能团队**：信任和尊重团队
6. **内建质量**：质量不是事后检验
7. **全局优化**：优化整体而非局部

### 七种浪费（软件开发）

| 浪费类型 | 说明 |
|---------|------|
| 部分完成的工作 | 未集成的代码、未测试的功能 |
| 多余的功能 | 从未使用的功能 |
| 重复学习 | 知识未共享 |
| 交接 | 信息在传递中丢失 |
| 延迟 | 等待审批、等待环境 |
| 任务切换 | 多任务并行导致效率下降 |
| 缺陷 | Bug修复的返工成本 |

## 12. 各模型的适用场景对比

| 模型 | 需求确定性 | 团队规模 | 风险级别 | 交付频率 | 适用场景 |
|------|-----------|---------|---------|---------|---------|
| 瀑布模型 | 高 | 中-大 | 低 | 低 | 合同制、嵌入式、安全关键 |
| V模型 | 高 | 中-大 | 中 | 低 | 航空航天、医疗设备 |
| 螺旋模型 | 低 | 中-大 | 高 | 中 | 高风险、创新型项目 |
| Scrum | 中-低 | 小-中 | 中 | 高 | 产品开发、互联网 |
| 看板 | 任意 | 小-中 | 低-中 | 高 | 运维、持续支持 |
| XP | 低 | 小 | 中 | 高 | 技术密集、质量要求高 |
| DevOps | 任意 | 任意 | 低 | 极高 | SaaS、持续交付 |
| 精益 | 低 | 小-中 | 中 | 高 | 创业公司、MVP开发 |

### 选型建议

- **需求稳定、合同制**：瀑布/V模型
- **需求不确定、快速迭代**：Scrum + XP实践
- **持续运维和改进**：看板
- **大规模产品开发**：SAFe（大规模敏捷框架）
- **高风险项目**：螺旋模型
- **初创产品**：精益 + Scrum
