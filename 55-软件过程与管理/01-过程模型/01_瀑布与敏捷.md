# 瀑布与敏捷过程模型


## 瀑布与敏捷过程模型对比

一、软件过程模型概述

## 一、软件过程模型概述


软件过程模型定义了软件开发活动的组织方式和执行顺序，为项目管理提供框架。

二、瀑布模型

## 二、瀑布模型（Waterfall Model）


瀑布模型是最经典的软件开发模型，由Winston Royce于1970年提出，采用线性顺序的开发流程。


### 2.1 阶段划分


1. **需求分析：**
   收集和分析用户需求，生成需求规格说明
2. **系统设计：**
   架构设计和详细设计
3. **编码实现：**
   将设计转化为代码
4. **测试验证：**
   发现和修复缺陷
5. **部署运行：**
   系统上线运行
6. **维护：**
   修复问题和适应变化


### 2.2 特点

**优点：**
过程清晰、文档驱动、易于管理、适合需求稳定的项目


**缺点：**
需求变更困难、反馈滞后、风险后置、不适合需求不确定的项目

### 2.3 V模型（V-Model）


V模型是瀑布模型的变体，强调开发阶段与测试阶段的对应关系。


| 开发阶段（左） | 测试阶段（右） |
| --- | --- |
| 需求分析 | 验收测试 |
| 系统设计 | 系统测试 |
| 概要设计 | 集成测试 |
| 详细设计 | 单元测试 |
| 编码 |

三、Scrum

## 三、Scrum框架


Scrum是最流行的敏捷框架之一，通过迭代增量的方式交付软件。


### 3.1 角色


| 角色 | 职责 |
| --- | --- |
| 产品负责人（Product Owner） | 定义产品待办列表，确定优先级，代表客户利益 |
| Scrum Master | 移除障碍，确保Scrum流程执行，辅导团队 |
| 开发团队（Dev Team） | 自组织、跨职能的开发团队（3-9人） |


### 3.2 核心工件


- **产品待办列表（Product Backlog）：**
   按优先级排序的需求列表
- **冲刺待办列表（Sprint Backlog）：**
   当前迭代要完成的任务
- **产品增量（Increment）：**
   每个迭代交付的可工作软件


### 3.3 事件/仪式


| 事件 | 时长 | 目的 |
| --- | --- | --- |
| 冲刺（Sprint） | 1-4周 | 固定时间盒，交付增量 |
| 冲刺计划会 | 2-4小时/周 | 确定冲刺目标和任务 |
| 每日站会 | 15分钟 | 同步进度，识别障碍 |
| 冲刺评审会 | 1-2小时 | 展示增量，收集反馈 |
| 冲刺回顾会 | 1-1.5小时 | 反思改进过程 |


> **Note:** **每日站会三个问题：**
> 昨天做了什么？今天计划做什么？有什么障碍？

四、Kanban

## 四、看板方法（Kanban）


看板源自丰田生产方式，通过可视化工作流和限制在制品数量来优化交付效率。


### 4.1 核心原则


1. **可视化工作流：**
   用看板展示所有工作项及其状态
2. **限制在制品数量（WIP Limit）：**
   限制每个阶段的并行任务数
3. **管理流动：**
   关注工作项的流动效率
4. **明确流程策略：**
   定义工作项的流转规则
5. **建立反馈循环：**
   定期评审和改进
6. **协作改进：**
   团队共同优化流程


### 4.2 Scrum vs Kanban


| 维度 | Scrum | Kanban |
| --- | --- | --- |
| 节奏 | 固定迭代（Sprint） | 持续流动 |
| 角色 | PO、SM、Dev Team | 无固定角色 |
| 变更 | 迭代内不变更 | 随时可以变更 |
| 度量 | 速度（Velocity） | 交付周期、前置时间 |
| 会议 | 固定的仪式 | 可选的会议 |

五、极限编程

## 五、极限编程（XP）


极限编程强调通过工程实践来保证软件质量和响应变化。


### 5.1 核心实践


- **结对编程：**
   两人共用一台电脑编程
- **测试驱动开发（TDD）：**
   先写测试，再写代码
- **持续集成：**
   频繁集成代码，快速发现冲突
- **重构：**
   持续改善代码结构
- **简单设计：**
   只实现当前需要的功能
- **小版本发布：**
   频繁交付可用的软件
- **编码标准：**
   团队统一的编码规范
- **集体代码所有权：**
   任何人可以修改任何代码

六、模型对比总结

## 六、过程模型综合对比


| 模型 | 适用场景 | 优势 | 劣势 |
| --- | --- | --- | --- |
| 瀑布 | 需求明确、变更少 | 简单、可预测 | 不灵活 |
| V模型 | 质量要求高、安全关键 | 测试前置规划 | 同瀑布 |
| Scrum | 需求变化快、创新型项目 | 灵活、快速反馈 | 需要成熟团队 |
| Kanban | 运维、持续交付 | 灵活、可视化 | 缺乏时间盒约束 |
| XP | 高质量要求、技术驱动 | 质量保证好 | 实践要求高 |
