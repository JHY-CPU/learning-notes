# 03-软件度量

## 1. 度量的目的与原则

软件度量通过量化的方式评估软件产品、过程和资源的属性。

### 度量的目的

- **理解**：客观了解当前状态
- **预测**：基于数据预测项目进度和质量
- **改进**：识别瓶颈，驱动持续改进
- **控制**：设定目标并监控偏差

### 度量原则（GQM方法）

GQM（Goal-Question-Metric）是定义度量的标准方法：

1. **Goal（目标）**：明确度量的业务目标
2. **Question（问题）**：为达成目标需要回答的问题
3. **Metric（度量）**：回答问题所需的数据

### 好的度量标准

- **可量化**：有明确的计算公式
- **可采集**：数据可通过工具或流程获取
- **可行动**：度量结果能指导行动
- **可比较**：支持趋势分析和基准对比

## 2. 代码度量

### LOC（Lines of Code）

代码行数，最简单的规模度量。

- **SLOC**（Source Lines of Code）：有效源代码行数
- **ELOC**（Executable Lines）：可执行语句行数
- **CLOC**（Comment Lines）：注释行数

**局限性**：不同语言、不同风格的代码行数不可直接比较。

### 圈复杂度（Cyclomatic Complexity）

衡量代码中独立执行路径的数量。

```
V(G) = E - N + 2P
```

- E：边的数量
- N：节点的数量
- P：连通分量数量（通常为1）

简化计算：`V(G) = 判断语句数 + 1`

| 圈复杂度 | 风险级别 | 建议 |
|---------|---------|------|
| 1-10 | 低 | 代码简单，易测试 |
| 11-20 | 中 | 建议重构 |
| 21-50 | 高 | 必须重构 |
| >50 | 极高 | 不可测试，需拆分 |

### 代码重复率

重复代码占总代码的比例。

- **精确重复**：完全相同的代码片段
- **结构重复**：结构相似但变量名/值不同
- **工具**：SonarQube、PMD、Simian

**目标**：通常控制在 5% 以下。

### 其他代码度量

| 度量 | 说明 | 目标 |
|------|------|------|
| 类耦合度（CBO） | 类之间的耦合程度 | < 14 |
| 继承深度（DIT） | 类在继承树中的深度 | < 6 |
| 方法数（WMC） | 类中方法的加权复杂度 | < 20 |
| 包的内聚性 | 包内类之间的关联度 | 越高越好 |

## 3. 功能点分析（FPA）

功能点分析从用户视角度量软件功能规模，与技术实现无关。

### 五类功能组件

| 类型 | 说明 | 示例 |
|------|------|------|
| **外部输入（EI）** | 数据进入系统 | 用户注册、订单提交 |
| **外部输出（EO）** | 经处理的数据输出 | 报表生成、统计导出 |
| **外部查询（EQ）** | 简单数据检索 | 商品搜索、订单查询 |
| **内部逻辑文件（ILF）** | 系统维护的数据 | 用户表、订单表 |
| **外部接口文件（EIF）** | 外部系统维护的数据 | 第三方API数据 |

### 计算步骤

1. **识别功能组件**：确定每类组件的数量
2. **确定复杂度**：每个组件分为低、中、高三档
3. **计算未调整功能点（UFP）**
4. **计算调整因子（VAF）**
5. **计算调整功能点（AFP）**

```
AFP = UFP × (0.65 + 0.01 × VAF)
```

### 优势

- 与编程语言无关
- 适合早期估算
- 支持跨项目比较
- 可用于合同定价

## 4. 缺陷密度

缺陷密度衡量单位规模内的缺陷数量。

```
缺陷密度 = 缺陷数 / 代码规模（千行或功能点）
```

### 行业基准

| 类型 | 缺陷密度（个/千行代码） |
|------|----------------------|
| 优秀 | < 0.5 |
| 良好 | 0.5 - 1.0 |
| 一般 | 1.0 - 3.0 |
| 较差 | > 3.0 |

### 缺陷分类

- 按严重级别：致命、严重、一般、轻微
- 按引入阶段：需求缺陷、设计缺陷、编码缺陷
- 按发现阶段：单元测试、集成测试、系统测试、生产环境

### 缺陷移除效率（DRE）

```
DRE = 发现的缺陷数 / (发现的缺陷数 + 交付后发现的缺陷数) × 100%
```

**目标**：DRE > 95%

## 5. 测试覆盖率度量

### 代码覆盖率类型

| 类型 | 覆盖内容 | 强度 |
|------|---------|------|
| **行覆盖率** | 被执行的代码行数 | 最弱 |
| **分支覆盖率** | 被执行的判断分支 | 中等 |
| **条件覆盖率** | 每个布尔子表达式 | 较强 |
| **路径覆盖率** | 所有执行路径 | 最强 |

### 与缺陷发现率的关系

```
覆盖率    缺陷发现边际收益
0-60%    高（大部分简单缺陷）
60-80%   中等
80-90%   较低
>90%     很低（成本高收益低）
```

### 实践建议

- **单元测试**：分支覆盖率 > 80%
- **关键模块**：分支覆盖率 > 90%
- **整体项目**：综合覆盖率 > 70%
- 不要为了覆盖率而写无意义的测试

## 6. Halstead度量

Halstead度量基于程序中运算符和操作数的数量来评估程序复杂度。

### 基本度量

- **n1**：不同运算符的数量
- **n2**：不同操作数的数量
- **N1**：运算符总数
- **N2**：操作数总数

### 派生度量

| 度量 | 公式 | 说明 |
|------|------|------|
| 程序词汇量 | n = n1 + n2 | 程序中不同符号的数量 |
| 程序长度 | N = N1 + N2 | 程序中符号总数 |
| 预期长度 | N' = n1×log₂(n1) + n2×log₂(n2) | 理论程序长度 |
| 体积 | V = N × log₂(n) | 程序的信息量 |
| 难度 | D = (n1/2) × (N2/n2) | 理解程序的难度 |
| 工作量 | E = D × V | 编写程序的工作量 |
| 时间 | T = E / 18 | 预计编程时间（秒） |

### 应用

- 估算代码审查时间
- 评估模块复杂度
- 与其他度量交叉验证

## 7. 技术债务量化

技术债务是为快速交付而做出的权宜之计在未来需要偿还的代价。

### 技术债务比率

```
技术债务比率 = 修复技术债务的成本 / 开发总成本 × 100%
```

### SonarQube的债务模型

| 严重级别 | 修复时间 |
|---------|---------|
| Blocker | 4小时 |
| Critical | 8小时 |
| Major | 4小时 |
| Minor | 30分钟 |
| Info | 10分钟 |

### 技术债务四象限

```
            知道
           ┌──────────┬──────────┐
    故意   │ 有意识的   │ 有意识的   │
           │ 债务       │ 债务      │
    ───────┼──────────┼──────────┤
    无意   │ 无意识的   │ 无意识的   │
           │ 债务       │ 债务      │
           └──────────┴──────────┘
              不知道
```

### 管理策略

- **记录**：将技术债务纳入Backlog
- **预算**：每个Sprint预留20%时间偿还债务
- **优先级**：优先偿还高风险、高频率使用的代码
- **预防**：代码评审、自动化测试、编码标准

## 8. DORA指标

DORA（DevOps Research and Assessment）定义了四个核心效能指标。

### 四个关键指标

| 指标 | 说明 | 精英水平 | 高水平 | 中等 | 低水平 |
|------|------|---------|--------|------|--------|
| **部署频率** | 多久部署一次到生产 | 按需/每天多次 | 每天-每周 | 每月 | 每月-半年 |
| **变更前置时间** | 从提交到部署的时间 | < 1小时 | 1天-1周 | 1周-1月 | 1月-半年 |
| **变更失败率** | 导致生产故障的变更比例 | 0-15% | 16-30% | 16-30% | > 30% |
| **恢复时间** | 生产故障后恢复的时间 | < 1小时 | < 1天 | 1天-1周 | > 6个月 |

### 团队效能分类

```
精英团队：所有四项指标均为精英/高水平
高效团队：大部分指标为高水平
中等团队：指标处于中等水平
低效团队：大部分指标为低水平
```

### DORA指标的意义

- 与组织绩效正相关
- 高效能团队的软件交付速度和稳定性均优
- 不存在速度与稳定性的权衡

## 9. CMMI能力成熟度等级

CMMI（Capability Maturity Model Integration）评估组织的过程成熟度。

### 五个等级

| 等级 | 名称 | 特征 |
|------|------|------|
| **1** | 初始级 | 过程不可预测，依赖个人能力 |
| **2** | 已管理级 | 项目层面有基本的计划和跟踪 |
| **3** | 已定义级 | 组织层面有标准过程和裁剪指南 |
| **4** | 量化管理级 | 过程和产品质量可量化度量和控制 |
| **5** | 优化级 | 持续改进，基于量化数据驱动创新 |

### 各级关键实践

**Level 2 - 已管理级**：
- 需求管理
- 项目计划与监控
- 配置管理
- 质量保证
- 度量与分析

**Level 3 - 已定义级**：
- 组织过程焦点
- 组织过程定义
- 集成项目管理
- 风险管理
- 决策分析

**Level 4 - 量化管理级**：
- 组织过程绩效
- 量化项目管理
- 统计过程控制

**Level 5 - 优化级**：
- 组织绩效管理
- 原因分析与解决

### 实践建议

- 大多数企业目标为Level 3
- 过度追求高等级可能增加官僚成本
- CMMI是过程改进框架，不是认证标准
