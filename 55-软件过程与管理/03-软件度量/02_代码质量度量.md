# 代码质量度量


## 代码质量度量

一、SonarQube质量模型

## 一、SonarQube质量模型


SonarQube是最广泛使用的代码质量管理平台，其质量模型由五个维度构成，提供可量化的质量指标。


### 1.1 五大质量维度


| 维度 | 度量指标 | 目标 | 说明 |
| --- | --- | --- | --- |
| 可靠性 (Reliability) | Bug数量和等级 | 0 Bugs | 可能导致运行时错误的代码问题 |
| 安全性 (Security) | 漏洞数量和等级 | 0 Vulnerabilities | 可能被攻击者利用的安全弱点 |
| 可维护性 (Maintainability) | 代码异味(Code Smells) | 技术债务比 <5% | 影响维护效率的代码问题 |
| 覆盖率 (Coverage) | 测试覆盖率 | ≥80% | 被单元测试覆盖的代码比例 |
| 重复度 (Duplications) | 重复代码比例 | ≤3-5% | 重复出现的代码块比例 |


### 1.2 质量门 (Quality Gate)


> **Note:** **SonarQube Quality Gate 标准配置：**
>
>
>
>
> - 新代码覆盖率 ≥ 80%
>
>
> - 新代码重复度 ≤ 3%
>
>
> - 新代码无Blocker/Critical级别Bug
>
>
> - 新代码无Blocker/Critical级别漏洞
>
>
> - 新代码维护性评级 = A
>
>
> - 技术债务比 ≤ 5%
>
>
>
>
> **"修复泄漏"原则：**
> Quality Gate聚焦于新代码，而非历史遗留问题。确保新的代码质量不退化，历史问题逐步修复。

二、技术债务

## 二、技术债务 (Technical Debt)


技术债务是软件开发中为追求短期速度而做出的权宜之计，需要在未来"偿还"（即重构或修复）。


### 2.1 技术债务分类


| 类型 | 成因 | 修复成本 | 示例 |
| --- | --- | --- | --- |
| 鲁莽/刻意 | "我们没时间做好设计" | 高 | 硬编码配置、跳过测试 |
| 谨慎/刻意 | "先发布，后面再重构" | 中 | 快速原型未重构 |
| 鲁莽/无意 | "什么是分层架构？" | 高 | 缺乏设计模式知识 |
| 谨慎/无意 | "现在才知道应该怎么做" | 低 | 经验积累后的认知迭代 |


### 2.2 技术债务比率 (TDR)


技术债务比率 = 修复所有问题所需时间 / 开发整个项目所需时间


- TDR < 5%：健康
- TDR 5-10%：需关注
- TDR 10-20%：需制定还债计划
- TDR > 20%：严重，需优先处理

三、代码异味

## 三、代码异味 (Code Smells)


代码异味是代码中可能表明更深层问题的表面特征。它们不直接导致Bug，但降低可维护性。


### 3.1 常见代码异味


| 异味类型 | 描述 | 影响 | 修复方法 |
| --- | --- | --- | --- |
| 过长方法 | 方法超过40-60行 | 难以理解和测试 | 提取方法（Extract Method） |
| 过大的类 | 类承担过多职责 | 违反单一职责原则 | 提取类（Extract Class） |
| 重复代码 | 相似代码在多处出现 | 修改需要同步多处 | 提取公共方法/基类 |
| 过长参数列表 | 方法参数超过3-4个 | 调用困难，易出错 | 引入参数对象 |
| 魔术数字 | 未解释的数字常量 | 含义不明 | 提取为命名常量 |
| 深度嵌套 | if/for嵌套超过3层 | 认知复杂度高 | 提前返回、策略模式 |
| 过大的继承层次 | 继承层级超过3-4层 | 脆弱基类问题 | 组合替代继承 |
| 临时字段 | 类中有仅特定情况使用的字段 | 类职责不清晰 | 提取到专用类 |

四、认知复杂度

## 四、认知复杂度 (Cognitive Complexity)


认知复杂度是SonarSource提出的新度量标准，相比圈复杂度(Cyclomatic Complexity)更贴合人类阅读代码的实际困难。


### 4.1 认知复杂度 vs 圈复杂度


| 维度 | 圈复杂度 | 认知复杂度 |
| --- | --- | --- |
| 计算方式 | 线性独立路径数 | 人类理解难度评分 |
| 逻辑运算符 | 每个+1 | 连续同一运算符不累加 |
| 嵌套 | 不额外惩罚 | 每层嵌套+1增量 |
| 递归 | 不特殊处理 | 额外加分 |
| 方法调用 | 不计入 | 线性结构不加分 |
| 适用场景 | 测试路径覆盖 | 可维护性和可读性 |


### 4.2 建议阈值


- 单个函数：认知复杂度 ≤ 15
- 单个类：平均认知复杂度 ≤ 25
- 复杂度 0-5：简单，易理解
- 复杂度 6-10：中等，可接受
- 复杂度 11-15：较复杂，考虑重构
- 复杂度 >15：复杂，必须重构

五、可维护性指数

## 五、可维护性指数 (Maintainability Index)


可维护性指数是一个综合评分（0-100），综合考虑了圈复杂度、代码行数和Halstead量度。


### 5.1 计算公式


> **Note:** **Visual Studio的可维护性指数公式：**
>
>
>
>
> MI = MAX(0, (171 - 5.2 × ln(HV) - 0.23 × CC - 16.2 × ln(LOC)) × 100 / 171)
>
>
>
>
> 其中：
>
>
> - HV = Halstead Volume（代码量）
>
>
> - CC = Cyclomatic Complexity（圈复杂度）
>
>
> - LOC = Lines of Code（代码行数）
>
>
>
>
> 评分标准：
>
>
> - 20-100：良好（绿色）
>
>
> - 10-19：中等（黄色）
>
>
> - 0-9：差（红色），需要重构

## 六、代码质量度量实践

### 6.2 度量指标看板


| 指标 | 数据源 | 关注频率 | 行动阈值 |
| --- | --- | --- | --- |
| 技术债务比 | SonarQube | 每Sprint | > 5%需制定还债计划 |
| 测试覆盖率 | SonarQube / JaCoCo | 每次提交 | < 80%阻断合并 |
| 代码重复度 | SonarQube | 每Sprint | > 5%需重构 |
| 认知复杂度 | SonarQube | 代码审查时 | > 15需重构 |
| 代码审查参与率 | Git平台 | 每月 | < 80%需关注 |
| Bug逃逸率 | Jira/Bugzilla | 每Sprint | > 10%需改进测试 |

**度量的陷阱：**
度量指标是手段而非目的。"一旦指标成为目标，它就不再是好指标"（Goodhart定律）。关注趋势而非绝对值，关注改进方向而非分数排名。避免将度量指标与个人绩效挂钩。
