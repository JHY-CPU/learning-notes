# 02-项目管理

## 1. 项目管理铁三角（范围、时间、成本）

项目管理铁三角描述了项目的三个核心约束，三者相互制约。

```
          范围（Scope）
           /        \
          /          \
        时间 ———————— 成本
       (Time)       (Cost)
```

### 核心关系

- 增加范围 → 需要更多时间或成本
- 缩短时间 → 需要增加成本或缩减范围
- 减少成本 → 需要缩减范围或延长时间

### 质量位于中心

质量受三者共同影响。在三者约束下追求最优质量，是项目管理的核心挑战。

### 实践中的取舍

- **固定价格合同**：成本固定，范围和时间需要灵活
- **敏捷项目**：时间固定（Sprint），范围灵活（可调整Backlog优先级）
- **紧急项目**：时间固定，可能增加成本（加人）或缩减范围

## 2. WBS（工作分解结构）

WBS（Work Breakdown Structure）将项目可交付成果逐层分解为可管理的工作包。

### 分解原则

- **100%原则**：WBS必须包含项目的所有工作
- **互斥原则**：工作包之间不重叠
- **可管理性**：每个工作包可以被估算、分配和跟踪
- **8/80规则**：工作包的工时在8-80小时之间

### 示例结构

```
项目
├── 1. 需求工程
│   ├── 1.1 需求调研（20h）
│   ├── 1.2 需求分析（30h）
│   └── 1.3 需求评审（10h）
├── 2. 系统设计
│   ├── 2.1 架构设计（25h）
│   ├── 2.2 详细设计（40h）
│   └── 2.3 设计评审（8h）
├── 3. 编码实现
│   ├── 3.1 前端开发（80h）
│   ├── 3.2 后端开发（100h）
│   └── 3.3 数据库开发（30h）
├── 4. 测试
│   ├── 4.1 测试计划（10h）
│   ├── 4.2 测试执行（60h）
│   └── 4.3 缺陷修复（40h）
└── 5. 部署上线
    ├── 5.1 环境准备（15h）
    ├── 5.2 部署实施（10h）
    └── 5.3 上线验证（8h）
```

### WBS词典

每个工作包应有对应的WBS词典条目，包含：
- 工作描述
- 负责人
- 工期估算
- 交付物
- 前置条件
- 验收标准

## 3. 甘特图与关键路径法（CPM）

### 甘特图

甘特图以条形图形式展示项目任务的时间安排。

```
任务          1月    2月    3月    4月
需求分析      ████
系统设计           ████
前端开发                ██████
后端开发                 ████████
测试                            ██████
部署                                 ██
```

**用途**：展示任务进度、里程碑、依赖关系

### 关键路径法（CPM）

关键路径是项目中耗时最长的任务序列，决定了项目的最短工期。

### 计算步骤

1. 列出所有任务及其工期
2. 确定任务之间的依赖关系
3. 计算每个任务的最早开始/完成时间（正推）
4. 计算每个任务的最晚开始/完成时间（逆推）
5. 总浮动时间为零的任务组成关键路径

### 示例

```
A(3天) → B(5天) → D(4天)
A(3天) → C(2天) → D(4天)

路径1：A → B → D = 3 + 5 + 4 = 12天（关键路径）
路径2：A → C → D = 3 + 2 + 4 = 9天

浮动时间：C有3天浮动
```

### 关键路径的意义

- 关键路径上的任何延迟都会延迟整个项目
- 管理重点应放在关键路径任务上
- 非关键任务可利用浮动时间调配资源

## 4. PERT图与三点估算

### PERT图

PERT（Program Evaluation and Review Technique）使用有向图表示任务依赖关系。

### 三点估算

考虑不确定性，对每个任务进行三种估算：

- **O（乐观估算）**：最顺利情况下的工期
- **M（最可能估算）**：正常情况下的工期
- **P（悲观估算）**：最不利情况下的工期

### 计算公式

```
期望工期 = (O + 4M + P) / 6
标准差   = (P - O) / 6
方差     = ((P - O) / 6)²
```

### 示例

| 任务 | 乐观O | 最可能M | 悲观P | 期望值 | 标准差 |
|------|-------|---------|-------|--------|--------|
| A | 2 | 4 | 6 | 4.0 | 0.67 |
| B | 3 | 5 | 12 | 5.83 | 1.50 |
| C | 1 | 2 | 3 | 2.0 | 0.33 |

### 完工概率

- 项目在期望工期完成的概率为 **50%**
- 项目在 期望+1标准差 内完成的概率为 **84%**
- 项目在 期望+2标准差 内完成的概率为 **97.7%**

## 5. 风险管理

风险管理贯穿项目全生命周期，包含四个环节。

### 5.1 风险识别

常用方法：
- **头脑风暴**：团队集思广益
- **检查清单**：基于历史项目的风险清单
- **SWOT分析**：分析优势、劣势、机会、威胁
- **假设分析**：检验项目假设的有效性

### 5.2 风险评估

使用 **概率-影响矩阵** 评估风险：

```
影响    低(1)    中(2)    高(3)
高(3)   中       高       极高
中(2)   低       中       高
低(1)   低       低       中
  ↑
概率
```

风险值 = 概率 × 影响

### 5.3 风险应对策略

| 策略 | 适用场景 | 示例 |
|------|---------|------|
| **规避** | 消除风险源 | 改用成熟技术替代新技术 |
| **转移** | 将风险转移给第三方 | 购买保险、外包给专业团队 |
| **减轻** | 降低概率或影响 | 增加备份方案、培训团队 |
| **接受** | 风险影响小或成本过高 | 制定应急计划 |

### 5.4 风险监控

- 定期审查风险登记册
- 跟踪已识别风险的状态
- 识别新风险
- 评估应对措施的有效性
- 更新风险优先级

## 6. 团队管理与沟通

### 团队发展阶段（Tuckman模型）

```
形成期(Storming) → 震荡期(Storming) → 规范期(Norming) → 执行期(Performing) → 解散期(Adjourning)
```

### 沟通方式选择

| 沟通方式 | 适用场景 | 效率 |
|---------|---------|------|
| 面对面 | 复杂问题、冲突解决 | 最高 |
| 视频会议 | 远程团队、方案讨论 | 高 |
| 电话 | 紧急事务 | 中 |
| 即时消息 | 快速确认、简单沟通 | 中 |
| 邮件 | 正式通知、留档 | 低 |

### 沟通渠道数量

```
沟通渠道数 = n(n-1) / 2
```

n为团队人数。5人团队有10条沟通渠道，10人团队有45条。

### 冲突管理

| 策略 | 适用场景 |
|------|---------|
| 协作（双赢） | 问题重要，需要最佳方案 |
| 妥协 | 双方势均力敌，需要折中 |
| 强制 | 紧急情况，需要快速决策 |
| 回避 | 问题不重要或需要冷却 |
| 适应 | 关系比问题更重要 |

## 7. 干系人管理

### 干系人识别

- 客户/用户
- 项目发起人
- 项目团队成员
- 职能部门经理
- 供应商/合作伙伴
- 监管机构

### 权力-利益矩阵

```
权力
高 │ 重点管理    │ 密切合作
   │ (发起人)    │ (关键客户)
───┼─────────────┼─────────────
   │ 保持满意    │ 保持知情
低 │ (监管机构)  │ (普通用户)
   └─────────────┴─────────────
         低            高
              利益
```

### 干系人参与策略

| 象限 | 策略 |
|------|------|
| 高权力高利益 | 重点管理，频繁沟通 |
| 高权力低利益 | 保持满意，定期汇报 |
| 低权力高利益 | 保持知情，收集反馈 |
| 低权力低利益 | 最少精力，必要时通知 |

## 8. 变更管理

### 变更控制流程

```
变更请求 → 影响分析 → 变更审批 → 实施变更 → 验证变更 → 更新文档
```

### 变更控制委员会（CCB）

- 评估变更的影响（范围、成本、进度）
- 审批或拒绝变更请求
- 确定变更的优先级

### 变更管理原则

- 所有变更必须书面记录
- 评估变更对三重约束的影响
- 避免范围蔓延（Scope Creep）
- 变更经批准后方可实施
- 维护变更日志

## 9. 敏捷估算

### 故事点（Story Points）

故事点衡量用户故事的相对复杂度，而非绝对工时。

常用标尺：**斐波那契数列** 1, 2, 3, 5, 8, 13, 21

### 计划扑克（Planning Poker）

1. 每人拿到一组斐波那契牌
2. PO讲解用户故事
3. 每人独立出牌
4. 差异大时讨论，重复直到收敛

### 速度（Velocity）

```
速度 = 一个Sprint完成的故事点总和
```

用于预测未来Sprint能完成多少工作。

### 速率图

```
Sprint:  1    2    3    4    5    6
速度:   21   25   28   30   32   30
              ↑
        趋于稳定的速度 = ~29点/Sprint
```

### 容量规划

```
可用容量 = 团队人数 × Sprint天数 × 每天有效工时 × 专注度
可承诺的故事点 = 可用容量 × 平均速度 / 基准工时
```

## 10. 燃尽图与燃起图

### 燃尽图（Burndown Chart）

展示剩余工作量随时间的减少趋势。

```
剩余故事点
50│╲
  │  ╲
40│    ╲
  │      ╲  实际线
30│        ╲
  │    --------  理想线
20│              ╲
  │                ╲
10│                  ╲
  │
 0└──────────────────────→ 时间
   S1  S2  S3  S4  S5  S6
```

**用途**：跟踪Sprint进度，预测能否按时完成

### 燃起图（Burnup Chart）

展示已完成工作量随时间的增长趋势，以及总范围的变化。

```
总故事点
80│- - - - - - - - - ╱  ← 总范围（可能变化）
  │               ╱
60│            ╱
  │         ╱
40│      ╱
  │   ╱  ╱  ← 已完成
20│╱
  │
 0└──────────────────────→ 时间
```

**优势**：能直观展示范围变更的影响

### 两者的对比

| 维度 | 燃尽图 | 燃起图 |
|------|--------|--------|
| 展示方式 | 剩余工作减少 | 已完成工作增加 |
| 范围变更 | 难以体现 | 直观体现 |
| 适用层级 | Sprint级 | 发布级 |
| 心理影响 | 聚焦"还剩多少" | 聚焦"完成了多少" |
