敏捷实践详解

用户故事、故事点、速率、Sprint 规划、回顾技术、看板指标

1. 用户故事 (User Story)

格式：As a [角色], I want [功能], So that [价值]。用户故事不是需求规格，是对话的起点。 
As a  在线购物用户
I want  将商品加入购物车
So that  我可以一次性结算多件商品

INVEST 原则

字母含义说明
IIndependent独立的，不强依赖其他故事
NNegotiable可协商的，不是固定合同
VValuable有价值的，对用户或业务有意义
EEstimable可估算的，团队能评估工作量
SSmall小的，一个 Sprint 内可完成
TTestable可测试的，有明确的验收标准

验收标准 (Acceptance Criteria)

用户故事：将商品加入购物车
验收标准（Given-When-Then 格式）：
  Given  我在商品详情页
  When   我点击"加入购物车"按钮
  Then   商品出现在购物车中
    And  购物车图标显示数量+1
    And  显示"已加入购物车"提示

  Given  商品库存为 0
  When   我点击"加入购物车"
  Then   按钮变为禁用状态
    And  显示"该商品已售罄"

2. 故事点与速率

2.1 故事点 (Story Points)

什么是故事点

  • 相对估算单位，不是时间
  • 基于复杂度、工作量、不确定性
  • 常用斐波那契数列：1, 2, 3, 5, 8, 13, 21
  • 一个"1"的故事是团队能轻松完成的最小任务

估算方法

  • 计划扑克 (Planning Poker)：团队每人出牌讨论
  • T 恤尺码：S/M/L/XL 快速估算
  • 相对估算：与已知故事对比
  • 宽带 Delphi：匿名估算 + 讨论

2.2 团队速率 (Velocity)

// 速率 = 每个 Sprint 完成的故事点总和
Sprint 1: 完成 24 点
Sprint 2: 完成 28 点
Sprint 3: 完成 26 点
Sprint 4: 完成 30 点
Sprint 5: 完成 27 点

平均速率 = (24+28+26+30+27) / 5 = 27 点/Sprint

// 预测：剩余 135 点，还需多少 Sprint？
135 / 27 ≈ 5 个 Sprint
注意：速率是团队指标而非个人指标。不应用速率来比较不同团队。速率需要 3-5 个 Sprint 才趋于稳定。

3. Sprint 规划 (Sprint Planning)

Product Backlog→ Sprint 规划会→ Sprint Backlog→ 每日站会→ Sprint 评审→ Sprint 回顾

Sprint 规划会议（2-4 小时）

  • Part 1 — 做什么：从 Backlog 选择 Sprint 目标
  • Part 2 — 怎么做：拆分任务、估算工作量
  • Product Owner 解释优先级和价值
  • 团队根据速率确定承诺范围
  • 输出：Sprint Goal + Sprint Backlog

每日站会 (Daily Standup)

  • 每天 15 分钟，固定时间地点
  • 每人回答三个问题：
  • 1. 昨天完成了什么？
  • 2. 今天计划做什么？
  • 3. 有什么障碍？
  • 只暴露问题，不解决问题

4. 回顾技术 (Retrospective Techniques)

帆船回顾 (Sailboat)

  • 风 = 帮助我们前进的因素
  • 锚 = 阻碍我们的因素
  • 礁石 = 潜在风险
  • 岛屿 = 我们的目标

4L 回顾

  • Liked — 喜欢的
  • Learned — 学到的
  • Lacked — 缺少的
  • Longed for — 期望的

开始/停止/继续

  • Start — 应该开始做什么
  • Stop — 应该停止做什么
  • Continue — 应该继续做什么

5 Why 分析

  • 针对具体问题连续追问"为什么"
  • 追溯到根本原因（通常 5 层）
  • 示例：上线失败 → 配置错误 → 无检查 → 无模板 → 无规范
  • 行动：建立配置模板和检查清单
回顾会关键：心理安全是基础（不追责），输出必须有具体行动项（Action Items），下次回顾检查行动完成情况。

5. 看板指标 (Kanban Metrics)

在制品限制 (WIP Limit)

  • 限制每个阶段同时进行的任务数
  • 暴露瓶颈、减少上下文切换
  • WIP = Work In Progress
  • 经验法则：团队人数 × 1.5

周期时间 (Cycle Time)

  • 从开始工作到完成的时间
  • 衡量流程效率的核心指标
  • 目标：缩短并稳定周期时间
  • 通过累积流图可视化

前置时间 (Lead Time)

  • 从需求提出到交付的总时间
  • Lead Time = 等待时间 + Cycle Time
  • 客户视角的交付效率
  • 比 Cycle Time 更全面

吞吐量 (Throughput)

  • 单位时间内完成的工作项数量
  • 衡量团队产出能力
  • 与 Cycle Time 关联（利特尔法则）
  • Lead Time = WIP / Throughput

利特尔法则 (Little's Law)

// L = λ × W
// 平均在制品数 = 吞吐量 × 平均周期时间

// 示例：
平均在制品数 (WIP):     12 个任务
平均周期时间 (CT):      6 天
吞吐量 (Throughput):    12 / 6 = 2 个任务/天

// 如果要缩短周期时间到 4 天：
需要 WIP 降到: 2 × 4 = 8 个任务
// 减少同时进行的任务数 → 缩短每个任务的完成时间
