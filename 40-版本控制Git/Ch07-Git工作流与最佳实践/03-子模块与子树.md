# 子模块与子树

## 一、Git 子模块（Submodule）

子模块允许在一个 Git 仓库中嵌入另一个 Git 仓库，保持两个仓库独立管理。

### 1. 添加子模块

```bash
# 添加子模块
git submodule add https://github.com/lib/library.git libs/library

# 添加并指定分支
git submodule add -b main https://github.com/lib/library.git libs/library

# 添加后会创建 .gitmodules 文件
cat .gitmodules
# [submodule "libs/library"]
#     path = libs/library
#     url = https://github.com/lib/library.git
#     branch = main
```

### 2. 克隆含子模块的仓库

```bash
# 方式1：克隆后初始化子模块
git clone https://github.com/user/project.git
cd project
git submodule update --init --recursive

# 方式2：一步完成（推荐）
git clone --recurse-submodules https://github.com/user/project.git

# 方式3：已克隆的仓库初始化子模块
git submodule init
git submodule update
```

### 3. 更新子模块

```bash
# 更新子模块到远程最新版本
git submodule update --remote

# 更新指定子模块
git submodule update --remote libs/library

# 更新后需要提交父仓库
git add .gitmodules libs/library
git commit -m "chore: update submodule to latest"
```

### 4. 子模块操作

```bash
# 查看子模块状态
git submodule status

# 查看子模块详细信息
git submodule foreach 'git log --oneline -3'

# 在所有子模块中执行命令
git submodule foreach 'git pull origin main'

# 查看子模块的远程 URL
git submodule foreach 'git remote -v'
```

### 5. 删除子模块

```bash
# 1. 删除子模块配置
git submodule deinit -f libs/library

# 2. 删除子模块目录
rm -rf libs/library

# 3. 删除 .git/modules 中的缓存
rm -rf .git/modules/libs/library

# 4. 删除 .gitmodules 中的配置
git rm -f libs/library

# 5. 提交
git commit -m "chore: remove submodule libs/library"
```

### 6. 子模块的问题与挑战

```
子模块的缺点：
✗ 操作复杂，学习成本高
✗ 容易忘记更新子模块
✗ 切换分支时子模块状态可能不一致
✗ CI/CD 需要特殊处理
✗ 团队成员容易遇到子模块同步问题
```

## 二、Git 子树（Subtree）

子树是另一种在仓库中包含外部项目的方式，将外部仓库的内容直接合并到项目中。

### 1. 添加子树

```bash
# 添加子树（首次）
git subtree add --prefix=libs/library https://github.com/lib/library.git main --squash

# 参数说明：
# --prefix  子树在本地的路径
# main      要引入的分支
# --squash  压缩外部仓库的历史
```

### 2. 更新子树

```bash
# 拉取子树更新
git subtree pull --prefix=libs/library https://github.com/lib/library.git main --squash

# 不压缩历史
git subtree pull --prefix=libs/library https://github.com/lib/library.git main
```

### 3. 推送子树修改

```bash
# 将本地对子树的修改推送到原仓库
git subtree push --prefix=libs/library https://github.com/lib/library.git main
```

### 4. 添加远程仓库别名

```bash
# 为子树仓库添加远程别名（方便操作）
git remote add library-remote https://github.com/lib/library.git

# 使用别名
git subtree add --prefix=libs/library library-remote main --squash
git subtree pull --prefix=libs/library library-remote main --squash
git subtree push --prefix=libs/library library-remote main
```

## 三、子模块 vs 子树对比

| 特性 | 子模块 | 子树 |
|------|--------|------|
| 仓库独立性 | 独立仓库 | 代码合并到主仓库 |
| 历史记录 | 独立历史 | 可选保留或压缩 |
| 克隆操作 | 需要额外初始化 | 直接可用 |
| 更新方式 | `submodule update` | `subtree pull` |
| 推送修改 | 直接在子模块中推送 | `subtree push` |
| 学习曲线 | 较高 | 较低 |
| 适用场景 | 大型库/框架引用 | 共享代码/组件 |

### 选择建议

```
使用子模块：
✓ 需要严格版本控制
✓ 引用的项目有自己的发布周期
✓ 需要跟踪外部项目的完整历史
✓ 引用的项目体积很大

使用子树：
✓ 不想维护额外的配置
✓ 团队成员不熟悉子模块
✓ 共享的代码/组件需要统一管理
✓ 引用的项目体积较小
```

## 四、npm 包管理替代方案

对于前端项目，通常用 npm/yarn 代替子模块：

```bash
# 使用 npm 包代替子模块
npm install @company/shared-utils

# 或使用私有包
npm install @company/internal-lib --registry=https://npm.company.com

# 或使用 git URL
npm install git+https://github.com/company/lib.git
npm install git+ssh://git@github.com/company/lib.git#v1.0.0

# 或使用 workspace monorepo
# package.json
{
  "workspaces": [
    "packages/*"
  ]
}
```

## 五、Monorepo 方案

### 使用 npm/yarn Workspaces

```json
// 根目录 package.json
{
  "private": true,
  "workspaces": [
    "packages/*"
  ]
}
```

```
项目结构：
/
├── package.json
├── packages/
│   ├── shared/        ← 共享库
│   ├── frontend/      ← 前端应用
│   └── backend/       ← 后端服务
```

```bash
# 安装所有 workspace 依赖
npm install

# 在特定 workspace 执行命令
npm run build --workspace=packages/shared
npm run test --workspace=packages/frontend

# workspace 间互相引用
# packages/frontend/package.json
{
  "dependencies": {
    "@company/shared": "*"
  }
}
```

### 使用 Turborepo

```bash
# 初始化
npx create-turbo@latest my-monorepo

# turbo.json
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "test": {
      "dependsOn": ["build"]
    }
  }
}

# 构建所有 packages
npx turbo run build
```

## 六、子模块常见问题

### 问题1：子模块显示修改但没有实际改动

```bash
# 原因：子模块指向的提交与 .gitmodules 中记录的不同
git submodule status
# -abc1234 libs/library  （- 前缀表示有差异）

# 解决：更新子模块
git submodule update --init

# 或提交子模块引用更新
git add libs/library
git commit -m "chore: update submodule reference"
```

### 问题2：切换分支后子模块混乱

```bash
# 切换分支后，子模块可能处于分离 HEAD 或错误版本
git submodule deinit -f --all
git submodule update --init --recursive
```

### 问题3：子模块认证失败

```bash
# 配置 SSH 方式
git config submodule.libs/library.url git@github.com:lib/library.git

# 或使用 credential helper
git config credential.helper store
```
