# 协作开发规范

## 一、提交信息规范

### Conventional Commits 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type 类型说明

| Type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(auth): add OAuth2 login` |
| `fix` | Bug 修复 | `fix(api): handle timeout error` |
| `docs` | 文档变更 | `docs: update README` |
| `style` | 格式调整（不影响逻辑） | `style: fix indentation` |
| `refactor` | 重构（非新功能、非修复） | `refactor: extract helper` |
| `perf` | 性能优化 | `perf: cache API responses` |
| `test` | 测试相关 | `test: add unit tests for auth` |
| `chore` | 构建/工具变更 | `chore: update dependencies` |
| `ci` | CI 配置 | `ci: add GitHub Actions` |
| `revert` | 回退提交 | `revert: undo feat X` |

### Commit 规范示例

```bash
# 简单提交
git commit -m "feat: add user registration"
git commit -m "fix: resolve login timeout"

# 带范围
git commit -m "feat(auth): add JWT authentication"
git commit -m "fix(ui): correct header alignment"

# 带正文和脚注
git commit -m "fix: prevent race condition in order processing

The previous implementation did not acquire a lock before
accessing shared state, causing intermittent data corruption.

Closes #123
Reviewed-by: @bob"

# 破坏性变更（用 BREAKING CHANGE 或 !）
git commit -m "feat!: change API response format"
git commit -m "feat(api): change response format

BREAKING CHANGE: API responses now use camelCase instead of snake_case"
```

### 使用 commitlint 自动检查

```bash
# 安装
npm install -D @commitlint/cli @commitlint/config-conventional

# 配置 .commitlintrc.js
echo "module.exports = { extends: ['@commitlint/config-conventional'] };" > .commitlintrc.js

# 配置 husky hook
npx husky add .husky/commit-msg 'npx --no -- commitlint --edit ${1}'
```

## 二、分支命名规范

### 格式

```
<type>/<description>

类型：
├── feature/  或 feat/   新功能
├── bugfix/   或 fix/    Bug 修复
├── hotfix/              紧急修复
├── release/             发布分支
├── docs/                文档更新
├── refactor/            重构
├── test/                测试
├── chore/               杂务
└── ci/                  CI 相关
```

### 示例

```bash
# 好的分支名
feature/user-authentication
feature/add-dark-mode
bugfix/fix-login-redirect
hotfix/security-patch
release/v2.1.0
docs/update-api-reference

# 不好的分支名
my-branch
test
feature1
dev
tmp
```

## 三、代码审查规范

### 审查清单

```
功能正确性：
□ 代码逻辑是否正确
□ 是否处理了边界情况
□ 是否有安全隐患
□ 是否考虑了性能

代码质量：
□ 是否遵循项目代码风格
□ 变量/函数命名是否清晰
□ 是否有不必要的重复代码
□ 是否有未使用的代码

测试：
□ 是否添加了必要的测试
□ 测试是否覆盖了主要场景
□ 测试是否清晰可维护

文档：
□ 是否更新了相关文档
□ 是否更新了 CHANGELOG
□ 复杂逻辑是否有注释说明
```

### 审查沟通规范

```
审查者：
✓ "建议：这里可以使用 Map 来优化查找性能"
✓ "疑问：这个 null 检查是否必要？"
✓ "问题：如果 userId 为空会怎样？"
✓ "表扬：这个抽象做得很好"

避免：
✗ "这代码写得不好"
✗ "你怎么能这样写"
✗ "这不是我想要的"
```

## 四、版本发布规范

### CHANGELOG 格式

```markdown
# Changelog

## [1.2.0] - 2025-01-15

### Added
- User authentication with JWT
- Password reset functionality

### Changed
- Improved error messages in API responses
- Updated login page design

### Fixed
- Fixed memory leak in event listener
- Fixed incorrect date formatting

### Removed
- Deprecated legacy API endpoints

## [1.1.0] - 2025-01-01
...
```

### 发布流程

```bash
# 1. 确保 main 是最新的
git switch main
git pull origin main

# 2. 创建发布分支
git switch -c release/v1.2.0

# 3. 更新版本号
npm version 1.2.0 --no-git-tag-version

# 4. 更新 CHANGELOG
# ... 编辑 CHANGELOG.md ...

# 5. 提交并合并
git add .
git commit -m "chore(release): bump version to 1.2.0"
git switch main
git merge --no-ff release/v1.2.0

# 6. 打标签
git tag -a v1.2.0 -m "Release 1.2.0"

# 7. 推送
git push origin main --tags

# 8. 清理
git branch -d release/v1.2.0
```

## 五、代码风格与工具

### EditorConfig

```ini
# .editorconfig
root = true

[*]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true

[*.md]
trim_trailing_whitespace = false

[Makefile]
indent_style = tab
```

### Pre-commit 检查

```bash
# 使用 husky 设置 pre-commit hook
npm install -D husky lint-staged

# 初始化
npx husky init

# .husky/pre-commit
npx lint-staged

# .lintstagedrc.js
module.exports = {
  '*.{js,ts,tsx}': ['eslint --fix', 'prettier --write'],
  '*.{json,md,yml}': ['prettier --write'],
};
```

## 六、协作工具配置

### GitHub CLI 常用命令

```bash
# 登录
gh auth login

# 创建仓库
gh repo create my-project --public

# 克隆仓库
gh repo clone user/repo

# 创建 PR
gh pr create --title "feat: add feature" --body "Description"

# 查看 PR
gh pr list
gh pr view 123
gh pr diff 123

# 合并 PR
gh pr merge 123 --squash

# 创建 Issue
gh issue create --title "Bug: description" --body "Details"

# 查看 CI 状态
gh run list
gh run watch
```

### GitHub Actions 示例

```yaml
# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run lint
      - run: npm test
      - run: npm run build
```

## 七、团队协作最佳实践

```
每日流程：
1. git pull --rebase origin main（获取最新代码）
2. 创建功能分支
3. 开发并频繁提交（小步提交）
4. 推送并创建 PR
5. 请求 Code Review
6. 根据反馈修改
7. 合并后清理分支

团队约定：
✓ 统一的代码风格（ESLint/Prettier）
✓ 统一的提交信息规范（Conventional Commits）
✓ PR 必须经过审查
✓ CI 必须通过才能合并
✓ 功能分支保持短生命周期
✓ 本地不要积累太多未推送的提交
✓ 使用 --force-with-lease 而非 --force
```
