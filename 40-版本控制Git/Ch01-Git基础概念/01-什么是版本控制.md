# 什么是版本控制

## 一、版本控制的概念

版本控制（Version Control System，VCS）是一种记录一个或若干文件内容变化，以便将来查阅特定版本修订情况的系统。它能够：

- **追踪文件变化**：记录每一次修改的内容、时间和作者
- **回退历史版本**：随时将文件恢复到任意历史状态
- **多人协作**：多人同时编辑同一项目而不互相覆盖
- **分支实验**：创建独立分支进行功能开发，不影响主线

## 二、版本控制的发展历程

### 1. 本地版本控制系统

最简单的版本管理方式，将文件的各个版本保存在本地数据库中。

```
项目文件/
├── v1.0/
├── v1.1/
├── v2.0/
└── 最新版/
```

**缺点**：无法多人协作，容易覆盖他人修改。

代表工具：RCS（Revision Control System）

### 2. 集中式版本控制系统（CVCS）

使用中央服务器保存所有版本，开发者通过网络连接到服务器获取最新版本或提交更新。

```
        ┌──────────┐
        │ 中央服务器 │
        │ (所有版本) │
        └────┬─────┘
       ┌─────┼─────┐
       ▼     ▼     ▼
    开发者A 开发者B 开发者C
```

**优点**：
- 每个人都能看到其他人的修改
- 管理权限易于控制

**缺点**：
- 中央服务器单点故障，宕机后无法工作
- 网络依赖严重

代表工具：SVN（Subversion）、CVS

### 3. 分布式版本控制系统（DVCS）

每个开发者的本地仓库都是完整的版本库，包含项目的完整历史记录。

```
    开发者A              开发者B              开发者C
  ┌─────────┐        ┌─────────┐        ┌─────────┐
  │ 完整仓库 │ ◄────► │ 完整仓库 │ ◄────► │ 完整仓库 │
  │ 历史+分支 │   同步  │ 历史+分支 │   同步  │ 历史+分支 │
  └─────────┘        └─────────┘        └─────────┘
       ▲                  ▲                  ▲
       └──────────────────┼──────────────────┘
                    远程仓库 (GitHub等)
```

**优点**：
- 不依赖中央服务器即可工作
- 每次克隆都是完整备份，数据安全性极高
- 支持离线工作
- 分支管理极其高效

**缺点**：
- 学习曲线较陡
- 大型二进制文件仓库体积膨胀

代表工具：Git、Mercurial

## 三、版本控制的核心概念

### 1. 仓库（Repository）

仓库是版本控制系统中存储项目所有文件和历史记录的地方。

- **本地仓库**：存储在开发者本机
- **远程仓库**：存储在服务器上（如 GitHub、GitLab）

### 2. 提交（Commit）

提交是将文件的当前状态保存到仓库中的操作，每次提交都有唯一的标识符。

```
Commit 3 (abc123f)  ← 当前 HEAD
  │ 添加了用户登录功能
  │
Commit 2 (def456a)
  │ 实现了数据库连接
  │
Commit 1 (789g0hi)  ← 初始提交
    项目初始化
```

### 3. 分支（Branch）

分支是从主线分离出来的独立开发线，用于并行开发不同功能。

```
main:    A ── B ── C ── D
                  \
feature:           E ── F
```

### 4. 合并（Merge）

将一个分支的修改合并到另一个分支的操作。

```
main:    A ── B ── C ──────── G (合并提交)
                  \          /
feature:           E ── F ──┘
```

### 5. 工作区、暂存区、仓库区

Git 的三个工作区域：

```
┌─────────────┐    git add    ┌─────────────┐   git commit   ┌─────────────┐
│  工作区      │ ──────────►  │  暂存区      │ ────────────►  │  本地仓库    │
│ (Working Dir)│              │ (Staging)    │                │ (.git)      │
└─────────────┘              └─────────────┘                └─────────────┘
     你编辑的文件                 准备提交的快照                  已提交的历史
```

## 四、为什么选择 Git

### Git vs SVN 对比

| 特性 | Git | SVN |
|------|-----|-----|
| 架构 | 分布式 | 集中式 |
| 速度 | 极快（本地操作） | 较慢（依赖网络） |
| 分支 | 轻量级，秒级创建 | 较重，目录拷贝 |
| 离线工作 | 完全支持 | 不支持 |
| 学习曲线 | 较陡 | 较平缓 |
| 数据完整性 | SHA-1 校验 | CRC 校验 |
| 适用场景 | 开源项目、敏捷开发 | 企业内部、大文件管理 |

### Git 的核心优势

1. **速度快**：几乎所有操作都在本地完成，无需网络
2. **数据安全**：使用 SHA-1 哈希保证数据完整性
3. **强大的分支**：分支创建和切换几乎是瞬间完成
4. **社区生态**：GitHub、GitLab 等平台提供了完整的协作生态
5. **行业标准**：绝大多数开源项目和企业项目都使用 Git

## 五、Git 的简要历史

- **2005年**：Linus Torvalds 为 Linux 内核开发创建了 Git
- **起因**：BitKeeper（此前使用的商业 VCS）不再免费提供给 Linux 社区
- **设计目标**：速度、简单设计、对非线性开发的强力支持、完全分布式
- **名字由来**：Linus 自嘲为"愚蠢的信息追踪器"（stupid content tracker），git 在英国俚语中也有"讨厌的人"之意
