# 撤销与回退操作

## 一、操作对比总览

| 操作 | 适用场景 | 是否改变历史 | 是否保留修改 |
|------|---------|------------|------------|
| `git restore` | 丢弃工作区修改 | 否 | 否 |
| `git restore --staged` | 取消暂存 | 否 | 是 |
| `git commit --amend` | 修改上次提交 | 是（仅限本地） | 是 |
| `git reset` | 回退提交/取消暂存 | 是 | 可选 |
| `git revert` | 撤销已推送的提交 | 否（新增提交） | 否 |
| `git reflog` | 恢复任何操作 | - | - |

## 二、git restore —— 恢复文件

### 1. 丢弃工作区修改

```bash
# 恢复单个文件（丢弃未暂存的修改）
git restore filename.txt

# 恢复所有文件
git restore .

# 恢复指定目录
git restore src/

# 从指定版本恢复文件
git restore --source=abc1234 filename.txt
git restore --source=HEAD~2 filename.txt
```

### 2. 取消暂存

```bash
# 将文件从暂存区移回工作区
git restore --staged filename.txt

# 取消暂存所有文件
git restore --staged .
```

### 3. 恢复已删除的文件

```bash
# 从最后一次提交恢复文件
git restore filename.txt

# 从指定提交恢复
git restore --source=abc1234 filename.txt

# 恢复整个目录
git restore --source=HEAD -- src/
```

## 三、git reset —— 重置提交

### 1. 三种模式对比

```bash
# --soft：只移动 HEAD，保留暂存区和工作区
git reset --soft HEAD~1

# --mixed（默认）：移动 HEAD，重置暂存区，保留工作区
git reset HEAD~1
git reset --mixed HEAD~1

# --hard：移动 HEAD，重置暂存区和工作区（危险！）
git reset --hard HEAD~1
```

```
场景：最近3次提交 A → B → C，HEAD 在 C

执行前：
HEAD → C (暂存区和工作区都是 C 的内容)

--soft HEAD~1：
HEAD → B
暂存区: C 的内容（已暂存的修改）
工作区: C 的内容
用途：合并多个提交

--mixed HEAD~1（默认）：
HEAD → B
暂存区: B 的内容
工作区: C 的内容（未暂存的修改）
用途：重新组织提交

--hard HEAD~1：
HEAD → B
暂存区: B 的内容
工作区: B 的内容
用途：彻底丢弃 C 的所有修改
```

### 2. 常用操作

```bash
# 撤销最近一次提交（保留修改）
git reset HEAD~1

# 撤销最近一次提交（丢弃修改，危险！）
git reset --hard HEAD~1

# 撤销最近 3 次提交
git reset --soft HEAD~3

# 回退到指定提交
git reset --hard abc1234

# 只重置暂存区，不影响工作区
git reset HEAD filename.txt
git reset HEAD .

# 重置到远程分支状态
git reset --hard origin/main
```

### 3. reset 的安全操作

```bash
# 在执行危险的 reset 前，先保存当前状态
git branch backup-branch

# 或者使用 reflog 作为安全网
git reset --hard HEAD~3
# 后悔了？
git reflog
# abc1234 HEAD@{0}: reset: moving to HEAD~3
# def5678 HEAD@{1}: commit: ...
git reset --hard HEAD@{1}  # 恢复到 reset 前的状态
```

## 四、git revert —— 撤销提交

### 1. 撤销单个提交

```bash
# 创建一个新提交来撤销指定提交的修改
git revert abc1234

# 撤销最近一次提交
git revert HEAD

# 撤销但不自动提交（手动调整后提交）
git revert --no-commit abc1234
git revert -n abc1234
```

### 2. 撤销多个提交

```bash
# 撤销最近 3 个提交
git revert HEAD~2..HEAD
# 或
git revert HEAD~3..HEAD

# 撤销一系列提交（按从新到旧的顺序）
git revert --no-commit HEAD~3..HEAD
git commit -m "revert: undo last 3 commits"

# 撤销合并提交
git revert -m 1 <merge-commit-hash>
# -m 1 表示保留主分支（parent 1）的内容
```

### 3. revert vs reset 对比

```
场景：main 分支有 A → B → C，需要撤销 B

git reset --hard B（将 main 移到 B）：
main: A → B        C 变成悬空提交
      （丢失 C 的历史）

git revert B（创建新提交 D 来撤销 B）：
main: A → B → C → D
      D 的内容等于 B 修改之前的状态
      （保留所有历史）
```

```
使用 reset：
✓ 还没有推送到远程
✓ 需要彻底删除某些提交
✓ 本地开发阶段

使用 revert：
✓ 已经推送到远程
✓ 公共分支（多人协作）
✓ 需要保留完整的提交历史
✓ 需要撤销一个合并提交
```

## 五、git checkout —— 旧版恢复

```bash
# 恢复文件（Git 2.23+ 推荐用 git restore）
git checkout -- filename.txt

# 从指定版本恢复文件
git checkout abc1234 -- filename.txt

# 恢复所有文件到上次提交状态
git checkout -- .

# 查看旧版本的文件（分离 HEAD）
git checkout abc1234
```

## 六、恢复丢失的数据

### 1. 使用 reflog 恢复

```bash
# 查看所有操作记录
git reflog
# abc1234 HEAD@{0}: reset: moving to HEAD~1
# def5678 HEAD@{1}: commit: feat: important work
# 789i0j1 HEAD@{2}: checkout: moving from feature to main

# 恢复到 reset 前的状态
git reset --hard HEAD@{1}
# 或创建新分支
git branch recover def5678
```

### 2. 恢复删除的分支

```bash
# 分支被删除后
git branch -D feature

# 通过 reflog 找到最后的提交
git reflog | grep feature
# 或
git reflog --all

# 基于找到的提交创建分支
git branch feature abc1234
```

### 3. 恢复删除的储藏

```bash
# 查找丢失的储藏
git fsck --unreachable | grep commit

# 验证找到的提交
git show <commit-hash>

# 恢复为储藏
git stash apply <commit-hash>
# 或创建分支
git branch recovered <commit-hash>
```

### 4. 恢复删除的标签

```bash
# 通过 reflog 找到标签指向的提交
git reflog | grep tag
# 或查看对象
git fsck --unreachable | grep tag

# 重新创建标签
git tag v1.0.0 <commit-hash>
```

## 七、回退操作决策树

```
需要撤销什么？
│
├── 工作区的修改（未 add）
│   └── git restore filename.txt
│
├── 暂存区的修改（已 add 未 commit）
│   └── git restore --staged filename.txt
│
├── 最近一次提交（未 push）
│   ├── 保留修改 → git reset HEAD~1
│   └── 丢弃修改 → git reset --hard HEAD~1
│
├── 最近一次提交信息（未 push）
│   └── git commit --amend -m "新信息"
│
├── 已推送的提交
│   └── git revert <commit-hash>
│
├── 找回丢失的数据
│   └── git reflog → git reset --hard HEAD@{n}
│
└── 彻底重置到远程状态
    └── git fetch origin && git reset --hard origin/main
```
