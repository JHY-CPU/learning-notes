# 分支策略与工作流

## 一、常见的分支策略

### 1. 主干开发（Trunk-based Development）

所有开发者直接在 `main` 分支上工作，通过短生命周期的功能分支或直接提交。

```
main: ──A──B──C──D──E──F──G──H──
         │     │        │
feat1:   └─b───┘        │
feat2:                  └─c──┘

特点：
- 分支生命周期极短（通常 < 1-2 天）
- 频繁合并到 main
- 需要完善的 CI/CD
- 适合持续部署
```

### 2. GitHub Flow

简单的分支模型，适合持续部署的团队：

```
main:    ──A──────C────────E──────
              \      \        \
feature1:      └──B──┘        │
feature2:             └──D───┘

流程：
1. 从 main 创建功能分支
2. 在功能分支上提交
3. 发起 Pull Request
4. 代码审查后合并到 main
5. 立即部署 main
```

### 3. Git Flow

经典的分支模型，适合有明确发布周期的项目：

```
main:        ──A───────────G─────────────L──
                 \         / \           /
hotfix:            └──F───┘   │         /
                              │        /
release:                      └──H──I─┘
                             /         \
develop:   ──B──C──D───────J───K───────────M──
               /           /
feature1:    └─E─┘        /
feature2:               └─N─O─┘

分支说明：
├── main         生产环境代码，只接受合并
├── develop      开发主线，功能分支的合并目标
├── feature/*    功能开发分支
├── release/*    预发布分支
└── hotfix/*     生产紧急修复分支
```

### 4. GitLab Flow

在 GitHub Flow 基础上增加了环境分支：

```
main ──────► staging ──────► production
  │              │                │
  └──feature──►  │                │
                 └──bugfix──────► │
```

## 二、Git Flow 详解

### 分支类型

| 分支 | 命名 | 来源 | 合并到 | 说明 |
|------|------|------|--------|------|
| main | `main` | - | - | 生产代码 |
| develop | `develop` | main | main | 开发主线 |
| feature | `feature/*` | develop | develop | 功能开发 |
| release | `release/*` | develop | main + develop | 发布准备 |
| hotfix | `hotfix/*` | main | main + develop | 紧急修复 |

### Git Flow 命令示例

```bash
# 初始化（创建 develop 分支）
git checkout -b develop main

# 开始新功能
git checkout -b feature/user-auth develop
# ... 开发 ...
git checkout develop
git merge --no-ff feature/user-auth
git branch -d feature/user-auth

# 准备发布
git checkout -b release/1.2.0 develop
# ... 修复小问题 ...
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0
git checkout develop
git merge --no-ff release/1.2.0
git branch -d release/1.2.0

# 紧急修复
git checkout -b hotfix/security-fix main
# ... 修复 ...
git checkout main
git merge --no-ff hotfix/security-fix
git tag -a v1.2.1
git checkout develop
git merge --no-ff hotfix/security-fix
git branch -d hotfix/security-fix
```

### 使用 git-flow 工具

```bash
# 安装 git-flow
# macOS
brew install git-flow-avh

# 初始化 git-flow（交互式）
git flow init

# 开始功能开发
git flow feature start user-auth

# 完成功能开发（合并到 develop）
git flow feature finish user-auth

# 发布功能分支（推送并创建 PR）
git flow feature publish user-auth

# 开始发布
git flow release start 1.2.0

# 完成发布（合并到 main 和 develop，打标签）
git flow release finish 1.2.0

# 开始热修复
git flow hotfix start security-fix

# 完成热修复
git flow hotfix finish security-fix
```

## 三、GitHub Flow 详解

### 工作流程

```bash
# 1. 从 main 创建功能分支
git checkout main
git pull origin main
git checkout -b feature/add-search

# 2. 提交代码
git add .
git commit -m "feat: add search functionality"
git push origin feature/add-search

# 3. 在 GitHub 上创建 Pull Request
gh pr create --title "Add search functionality" --body "Description..."

# 4. 代码审查后合并
gh pr merge --squash

# 5. 删除远程分支
git push origin --delete feature/add-search

# 6. 更新本地
git checkout main
git pull origin main
git branch -d feature/add-search
```

### Pull Request 最佳实践

```
PR 标题：简短描述改动
PR 描述：
  ## 改动说明
  - 做了什么

  ## 测试方式
  - 如何验证

  ## 相关 Issue
  Closes #123

审查要求：
  - 至少一个 reviewer 批准
  - CI 检查通过
  - 无合并冲突
```

## 四、保护分支

### GitHub 设置保护规则

```yaml
# .github/branch-protection.yml（概念展示，实际在网页设置）
branches:
  main:
    protection:
      required_pull_request_reviews:
        required_approving_review_count: 1
        dismiss_stale_reviews: true
      required_status_checks:
        strict: true
        contexts: ["ci/build", "ci/test"]
      enforce_admins: false
      restrictions: null
```

### 常见保护规则

| 规则 | 说明 |
|------|------|
| 要求 PR 审查 | 必须经过至少 N 人审查批准 |
| 要求 CI 通过 | 所有状态检查必须通过 |
| 要求签名提交 | 所有提交必须 GPG 签名 |
| 禁止强制推送 | 不允许 `git push --force` |
| 禁止直接推送 | 只能通过 PR 合并 |
| 要求线性历史 | 只允许 rebase/merge commit |
| 限制推送权限 | 只有特定用户/团队可推送 |

## 五、分支管理实用技巧

### 1. 清理已合并的远程分支

```bash
# 获取最新远程信息并清理已删除的远程分支
git fetch --prune

# 查看哪些远程分支已被删除
git remote prune origin --dry-run

# 清理本地对已删除远程分支的引用
git remote prune origin
```

### 2. 批量管理分支

```bash
# 列出所有已合并到 main 的分支
git branch --merged main

# 批量删除已合并的分支（保留 main 和 develop）
git branch --merged main \
  | grep -v "^\*\|main\|develop" \
  | xargs -n 1 git branch -d

# 列出超过30天未更新的分支
git for-each-ref --sort=-committerdate refs/heads/ \
  --format='%(refname:short) %(committerdate:relative)' \
  | awk '{if(NR>30) print $1}' \
  | xargs -n 1 git branch -d
```

### 3. 比较分支

```bash
# 查看 feature 相对于 main 的所有提交
git log main..feature --oneline

# 查看 main 相对于 feature 的所有提交
git log feature..main --oneline

# 查看两个分支的共同祖先
git merge-base main feature

# 查看分支差异的文件列表
git diff --name-status main..feature
```
