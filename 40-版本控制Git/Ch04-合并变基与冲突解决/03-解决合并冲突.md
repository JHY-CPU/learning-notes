# 解决合并冲突

## 一、什么是合并冲突

当两个分支修改了同一文件的同一区域时，Git 无法自动判断使用哪个版本，就会产生合并冲突。

### 冲突产生场景

```
场景1：同一文件同一行被修改
main:    修改第10行为 "Hello World"
feature: 修改第10行为 "Hi Git"

场景2：一个分支删除，另一个分支修改
main:    删除 file.txt
feature: 修改 file.txt

场景3：两个分支添加了不同内容到同一位置
main:    在第5行后插入 "Line A"
feature: 在第5行后插入 "Line B"
```

## 二、冲突标记格式

### 1. 标准冲突标记

```text
<<<<<<< HEAD
这是当前分支的内容（main）
=======
这是被合并分支的内容（feature）
>>>>>>> feature-login
```

各部分含义：

| 标记 | 含义 |
|------|------|
| `<<<<<<< HEAD` | 冲突开始，以下是当前分支的内容 |
| `=======` | 分隔符 |
| `>>>>>>> branch` | 冲突结束，以上是被合并分支的内容 |

### 2. 完整冲突示例

```javascript
function greet(name) {
<<<<<<< HEAD
    return `Hello, ${name}! Welcome back.`;
=======
    return `Hi ${name}, nice to meet you!`;
>>>>>>> feature-new-greeting
}

// 解决后：
function greet(name) {
    return `Hello, ${name}! Welcome to our platform.`;
}
```

### 3. 复杂冲突示例

```text
<<<<<<< HEAD
function calculate(a, b) {
    const sum = a + b;
    const product = a * b;
    return { sum, product };
}
=======
function calculate(x, y) {
    return {
        addition: x + y,
        multiplication: x * y,
        division: x / y
    };
}
>>>>>>> feature-extended-calc
```

## 三、解决冲突的步骤

### 基本流程

```bash
# 1. 尝试合并，发现冲突
git merge feature
# CONFLICT (content): Merge conflict in src/utils.js
# Automatic merge failed; fix conflicts and then commit the result.

# 2. 查看冲突文件状态
git status
# Both modified: src/utils.js

# 3. 编辑冲突文件，解决冲突
code src/utils.js  # 用编辑器打开

# 4. 标记冲突已解决
git add src/utils.js

# 5. 完成合并
git commit
# 或
git commit -m "merge: resolve conflicts in utils.js"
```

### 使用 VS Code 解决冲突

VS Code 内置了强大的冲突解决工具：

```
┌──────────────────────────────────────────┐
│ <<<<<<< HEAD                             │
│ [当前更改] [接受当前更改]                    │
│ 这是当前分支的内容                          │
│ =======                                  │
│ [传入更改] [接受传入更改]                    │
│ 这是被合并分支的内容                         │
│ >>>>>>> feature                          │
│ [接受两个更改] [比较更改]                    │
└──────────────────────────────────────────┘
```

## 四、解决冲突的工具

### 1. 使用图形化合并工具

```bash
# 配置 VS Code 为合并工具
git config --global merge.tool vscode
git config --global mergetool.vscode.cmd 'code --wait --merge $REMOTE $LOCAL $BASE $MERGED'

# 启动合并工具
git mergetool

# 配置 Beyond Compare
git config --global merge.tool bc
git config --global mergetool.bc.path "'C:/Program Files/Beyond Compare 4/bcomp.exe'"

# 配置 IntelliJ IDEA
git config --global merge.tool intellij
git config --global mergetool.intellij.cmd 'idea merge "$LOCAL" "$REMOTE" "$BASE" "$MERGED"'
```

### 2. 使用 vimdiff

```bash
# 启动 vimdiff
git mergetool --tool=vimdiff

# vimdiff 界面布局：
# ┌─────────────┬─────────────┐
# │   LOCAL     │   REMOTE    │
# │ (当前分支)   │ (被合并分支)  │
# ├─────────────┴─────────────┤
# │           BASE            │
# │         (共同祖先)         │
# ├───────────────────────────┤
# │          MERGED           │
# │        (合并结果)          │
# └───────────────────────────┘

# vimdiff 快捷键：
# ]c  - 跳到下一个冲突
# [c  - 跳到上一个冲突
# do  - diff obtain（获取其他窗口的修改）
# dp  - diff put（将当前修改放到其他窗口）
# :wq - 保存并退出
```

### 3. 使用命令行方式

```bash
# 查看冲突的文件
git diff

# 查看冲突文件的内容
cat conflicted-file.txt

# 使用 sed 手动处理（高级）
# 接受当前分支版本
git checkout --ours conflicted-file.txt

# 接受被合并分支版本
git checkout --theirs conflicted-file.txt

# 然后手动编辑需要的部分
```

## 五、ours 和 theirs

### 含义

```
合并（merge）时：
ours  = 当前分支（HEAD）
theirs = 被合并的分支

变基（rebase）时（含义相反！）：
ours  = 被变基到的分支（原 main）
theirs = 正在变基的分支（原 feature）
```

### 选择版本

```bash
# 接受当前分支的版本（丢弃被合并分支的修改）
git checkout --ours conflicted-file.txt

# 接受被合并分支的版本（丢弃当前分支的修改）
git checkout --theirs conflicted-file.txt

# 批量处理所有冲突文件
# 接受当前分支的所有版本
git checkout --ours .

# 接受被合并分支的所有版本
git checkout --theirs .
```

### 变基时的 ours/theirs 反转

```bash
# 注意：变基时 ours 和 theirs 含义与合并时相反！
git rebase main feature
# 发生冲突后：
git checkout --ours file.txt   # 这里的 ours 是 main（变基目标）
git checkout --theirs file.txt # 这里的 theirs 是 feature（正在变基的）
```

## 六、避免冲突的策略

### 1. 保持分支短生命周期

```bash
# 频繁同步主分支
git switch feature
git fetch origin
git rebase origin/main

# 功能完成后尽快合并
# 分支存在时间越长，冲突可能性越大
```

### 2. 使用 .gitattributes 定义合并策略

```gitattributes
# 对某些文件始终使用特定版本
# 始终使用当前分支版本
config.json merge=ours

# 使用 union 合并（保留两边的修改）
CHANGELOG.md merge=union
```

### 3. 团队约定

```
减少冲突的团队实践：
✓ 不同开发者修改不同文件/目录
✓ 使用模块化架构，减少代码重叠
✓ 频繁集成，小步提交
✓ 功能分支保持短生命周期
✓ 使用代码格式化工具（统一格式）
✓ 定期 rebase/merge 主分支
```

## 七、特殊冲突处理

### 1. 二进制文件冲突

```bash
# 二进制文件冲突无法自动解决
# 查看冲突
git status

# 选择一个版本
git checkout --ours image.png    # 使用当前分支版本
git checkout --theirs image.png  # 使用被合并分支版本

# 或者手动替换文件后
git add image.png
git commit
```

### 2. 子模块冲突

```bash
# 子模块引用冲突
git checkout --ours path/to/submodule
git checkout --theirs path/to/submodule

# 或者手动更新子模块
cd path/to/submodule
git checkout desired-commit
cd ../..
git add path/to/submodule
git commit
```

### 3. 中止合并

```bash
# 如果冲突太复杂，可以中止合并
git merge --abort
# 或
git rebase --abort

# 恢复到合并/变基前的状态
```

## 八、冲突解决后验证

```bash
# 检查是否还有未解决的冲突
git diff --check
# 如果有冲突标记残留会报错

# 查看合并状态
git status

# 运行测试确认功能正常
npm test

# 确认无误后提交
git commit -m "merge: resolve conflicts"
```
