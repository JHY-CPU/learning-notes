# 故障排查

## 一、常见问题

```yaml
N+1 查询:
  症状: 数据库查询过多
  排查: 查看 SQL 日志
  解决: DataLoader 批量加载

查询超时:
  症状: 请求响应慢
  排查: 链路追踪定位慢字段
  解决: 优化查询 + 缓存

内存泄漏:
  症状: 内存持续增长
  排查: heap dump 分析
  解决: 检查 DataLoader 和缓存

连接池耗尽:
  症状: 数据库连接超时
  排查: 连接池监控
  解决: 增大连接池 / 优化查询
```

## 四、常见错误码与排查

```yaml
错误码排查:
  GRAPHQL_PARSE_FAILED:
    原因: 查询语法错误
    排查: 检查查询语句语法
    示例: 缺少花括号、字段名拼写错误

  GRAPHQL_VALIDATION_FAILED:
    原因: 查询不符合 Schema
    排查: 检查字段名、参数类型
    示例: 使用了不存在的字段

  BAD_USER_INPUT:
    原因: 变量类型或值错误
    排查: 检查变量类型和值
    示例: ID 类型传了数字

  UNAUTHENTICATED:
    原因: 未认证
    排查: 检查 Token 是否过期
    方案: 刷新 Token 或重新登录

  FORBIDDEN:
    原因: 无权访问
    排查: 检查用户角色和权限
    方案: 确认操作权限

  INTERNAL_SERVER_ERROR:
    原因: 服务端异常
    排查: 查看服务端日志和堆栈
    方案: 检查数据库连接、外部服务
```

## 五、链路追踪集成

```typescript
// OpenTelemetry 集成
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { GraphQLInstrumentation } from '@opentelemetry/instrumentation-graphql';

const sdk = new NodeSDK({
  traceExporter: new OTLPTraceExporter({
    url: 'http://jaeger:4318/v1/traces',
  }),
  instrumentations: [
    new GraphQLInstrumentation({
      // 记录解析器耗时
      depth: 5,
      mergeItems: true,
    }),
  ],
});

sdk.start();

// 追踪输出:
// trace: abc123
//   span: GraphQL Query: GetUser (120ms)
//     span: Query.user (30ms)
//     span: User.orders (85ms)  ← 慢！
//       span: DataLoader.ordersByUserId (20ms)
//         span: SQL: SELECT * FROM orders (15ms)
```

## 六、生产排查流程

```yaml
排查步骤:
  1. 确认问题:
     - 错误率上升？延迟上升？
     - 影响范围：全部查询 or 特定查询？

  2. 定位问题:
     - APM 看慢查询 Top 10
     - 链路追踪看字段级耗时
     - 日志搜索错误堆栈

  3. 常见原因:
     - N+1: DataLoader 是否生效？
     - 缓存穿透: 热点数据缓存失效？
     - 连接池: 是否有等待？
     - 慢 SQL: 是否缺少索引？

  4. 修复验证:
     - 修复后压测
     - 监控指标恢复
     - 写入故障复盘文档
```

## 七、注意事项

1. **日志要结构化，便于查询**
2. **APM 集成到开发和生产环境**
3. **慢查询要持续监控和优化**
4. **故障要有回滚方案**
5. **定期做性能基准测试**
6. **错误码标准化**，便于客户端统一处理
7. **链路追踪定位字段级瓶颈**
