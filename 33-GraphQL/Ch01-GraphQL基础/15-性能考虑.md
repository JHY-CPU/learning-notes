# 性能考虑

## 一、N+1 问题

```graphql
# 查询: 1 次查用户, N 次查订单
query {
  users {         # SELECT * FROM users → 1 次
    name
    orders {      # SELECT * FROM orders WHERE user_id = ? → N 次
      id
      total
    }
  }
}
```

## 二、DataLoader 解决方案

```java
// DataLoader 批量加载
@Component
public class OrderDataLoader {

    private final DataLoader<String, List<Order>> ordersByUserIdLoader;

    public OrderDataLoader(OrderRepository orderRepository) {
        this.ordersByUserIdLoader = DataLoaderFactory.newDataLoader(
            userIds -> CompletableFuture.supplyAsync(() -> {
                // 一次查询: SELECT * FROM orders WHERE user_id IN (...)
                Map<String, List<Order>> ordersMap = orderRepository
                    .findByUserIdIn(userIds)
                    .stream()
                    .collect(Collectors.groupingBy(Order::getUserId));

                return userIds.stream()
                    .map(id -> ordersMap.getOrDefault(id, List.of()))
                    .collect(Collectors.toList());
            }),
            DataLoaderOptions.newOptions().setBatchingEnabled(true)
        );
    }

    public DataLoader<String, List<Order>> getOrdersLoader() {
        return ordersByUserIdLoader;
    }
}

// Resolver 中使用
@SchemaMapping(typeName = "User")
public CompletableFuture<List<Order>> orders(User user,
        OrderDataLoader loader) {
    return loader.getOrdersLoader().load(user.getId());
}
```

## 三、查询复杂度限制

```graphql
# 查询复杂度计算
query {
  users {           # 复杂度: 10 (返回多个)
    name            # 复杂度: 1
    orders {        # 复杂度: 10 * 5 = 50 (嵌套)
      id            # 复杂度: 1
      items {       # 复杂度: 50 * 10 = 500 (深度嵌套)
        product {   # 复杂度: 500 * 1 = 500
          name      # 复杂度: 1
        }
      }
    }
  }
}
# 总复杂度: ~1000+ → 应该被拒绝
```

```yaml
# Apollo Server 复杂度限制
const server = new ApolloServer({
  validationRules: [
    depthLimit(10),
    createComplexityLimitRule(1000, {
      scalarCost: 1,
      objectCost: 10,
      listFactor: 10,
    })
  ]
});
```

## 四、缓存策略

```yaml
缓存方案:
  响应缓存:
    - Apollo Server Response Cache
    - 按查询哈希缓存

  持久化查询:
    - APQ (Automatic Persisted Queries)
    - 查询 ID 映射

  字段级缓存:
    - DataLoader 缓存
    - Redis 缓存热点数据

  CDN 缓存:
    - GET 请求 + 查询参数
    - 适合公共数据
```

## 五、注意事项

1. **DataLoader 是解决 N+1 的标配**
2. **查询深度和复杂度必须限制**
3. **批量加载在同一个事件循环内合并**
4. **缓存要区分公共和私有数据**
5. **监控慢查询和高频查询**
