# 高级特性最佳实践

## 一、特性选型

```yaml
场景与推荐:
  微服务拆分:
    - 推荐: Apollo Federation
    - 替代: Schema Stitching (不推荐)

  大字段延迟加载:
    - 推荐: @defer
    - 替代: 分开查询

  长列表流式加载:
    - 推荐: @stream
    - 替代: 分页

  高频重复查询:
    - 推荐: APQ + CDN
    - 替代: 客户端缓存

  实时数据推送:
    - 推荐: Subscription
    - 替代: 轮询 / SSE

  自定义扩展:
    - 推荐: 自定义指令
    - 替代: 中间件
```

## 二、实践清单

```yaml
Federation:
  □ 子图边界清晰
  □ @key 定义合理
  □ Gateway 高可用
  □ Schema 编译检查

实时:
  □ WebSocket 鉴权
  □ Redis PubSub 跨实例
  □ 断线重连处理
  □ 消息频率控制

性能:
  □ APQ 启用
  □ 查询深度限制
  □ 复杂度限制
  □ DataLoader 配置
```

## 三、Apollo Federation 实践

```yaml
Federation 最佳实践:
  子图划分:
    - 按业务领域划分 (用户、订单、商品)
    - 每个子图独立部署
    - 子图只定义自己拥有的字段

  实体设计:
    - @key 选择稳定字段 (ID)
    - 扩展实体只添加关联字段
    - 避免跨子图的复杂查询

  Gateway:
    - 使用 Apollo Router (高性能)
    - 配置超时和重试
    - Schema 变更后自动更新
```

```graphql
# Federation 2 完整示例
# 子图 1: 用户服务
type User @key(fields: "id") {
  id: ID!
  name: String!
  email: String!
}

# 子图 2: 订单服务
type Order @key(fields: "id") {
  id: ID!
  total: Float!
  status: OrderStatus!
}

type User @key(fields: "id") {
  id: ID! @external
  orders: [Order!]!
}

# 子图 3: 商品服务
type Product @key(fields: "id") {
  id: ID!
  name: String!
  price: Float!
}

type OrderItem @key(fields: "id") {
  id: ID!
  product: Product!
  quantity: Int!
}
```

## 五、注意事项

1. **按需引入高级特性**，不要过度设计
2. **Federation 是微服务 GraphQL 的标准**
3. **实验性特性谨慎用于生产**
4. **监控各特性的性能影响**
5. **保持技术栈简洁**
6. **Apollo Router 替代 Gateway 获得更好性能**
7. **自定义指令要标准化管理**
