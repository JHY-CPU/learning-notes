# 非空与列表

## 一、非空标记

```graphql
type User {
  id: ID!               # 非空 - 不能返回 null
  name: String!          # 非空
  email: String!         # 非空
  bio: String            # 可空 - 可以返回 null
  phone: String          # 可空
}

# 非空在输入类型中
input CreateUserInput {
  name: String!          # 必填
  email: String!         # 必填
  bio: String            # 可选
}

# 非空在返回值中
type Mutation {
  createUser(input: CreateUserInput!): User!   # 输入非空，返回非空
  deleteUser(id: ID!): Boolean                  # 返回可空
}
```

## 二、列表的非空层次

```graphql
type User {
  # 列表和元素的不同非空组合:
  friends: [User]           # [User]  → 可 null, 元素可 null
  tags: [String!]           # [String!] → 可 null, 元素非空
  orders: [Order!]!         # [Order!]! → 非空, 元素非空
  emails: [String]!         # [String]! → 非空, 元素可 null
}
```

```
对应 JSON:
friends: null              ✓
friends: [null, {...}]     ✓
friends: [{...}, {...}]    ✓

tags: null                 ✓
tags: ["a", "b"]           ✓
tags: ["a", null]          ✗ 错误

orders: null               ✗ 错误
orders: []                 ✓
orders: [{...}]            ✓
```

## 三、设计建议

```yaml
非空使用建议:
  总是非空:
    - id
    - createdAt
    - 核心业务字段

  通常可空:
    - 可选信息 (bio, avatar)
    - 计算成本高的字段
    - 需要权限才能查看的字段

  列表设计:
    - 空列表用 [X!]! 而非 null
    - 元素通常非空 [X!]
    - 可空元素意味着列表中有空洞
```

## 四、注意事项

1. **非空标记影响客户端错误处理**
2. **不要过度使用非空**
3. **空列表比 null 更好**
4. **输入非空 = 必填参数**
5. **列表可空意味着客户端要处理 null 和 [] 两种情况**
