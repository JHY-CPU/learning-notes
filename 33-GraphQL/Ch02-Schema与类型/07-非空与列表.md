# 非空与列表

## 一、非空标记

```graphql
type User {
  id: ID!               # 非空 - 不能返回 null
  name: String!          # 非空
  email: String!         # 非空
  bio: String            # 可空 - 可以返回 null
  phone: String          # 可空
}

# 非空在输入类型中
input CreateUserInput {
  name: String!          # 必填
  email: String!         # 必填
  bio: String            # 可选
}

# 非空在返回值中
type Mutation {
  createUser(input: CreateUserInput!): User!   # 输入非空，返回非空
  deleteUser(id: ID!): Boolean                  # 返回可空
}
```

## 二、列表的非空层次

```graphql
type User {
  # 列表和元素的不同非空组合:
  friends: [User]           # [User]  → 可 null, 元素可 null
  tags: [String!]           # [String!] → 可 null, 元素非空
  orders: [Order!]!         # [Order!]! → 非空, 元素非空
  emails: [String]!         # [String]! → 非空, 元素可 null
}
```

```
对应 JSON:
friends: null              ✓
friends: [null, {...}]     ✓
friends: [{...}, {...}]    ✓

tags: null                 ✓
tags: ["a", "b"]           ✓
tags: ["a", null]          ✗ 错误

orders: null               ✗ 错误
orders: []                 ✓
orders: [{...}]            ✓
```

## 三、设计建议

```yaml
非空使用建议:
  总是非空:
    - id
    - createdAt
    - 核心业务字段

  通常可空:
    - 可选信息 (bio, avatar)
    - 计算成本高的字段
    - 需要权限才能查看的字段

  列表设计:
    - 空列表用 [X!]! 而非 null
    - 元素通常非空 [X!]
    - 可空元素意味着列表中有空洞
```

## 四、非空与 Resolver 的关系

```graphql
# Schema 声明非空
type User {
  id: ID!
  name: String!
  email: String!
}

# 如果 Resolver 返回 null 给非空字段
# GraphQL 会将错误向上冒泡到最近的可空父字段

# 示例：email Resolver 返回 null
query {
  user(id: "1") {
    name
    email           # ← 这里 Resolver 返回 null
  }
}

# 由于 email 是 String! 非空
# 错误会冒泡到 user 字段（假设 user 也是可空的）
# 最终结果：
{
  "data": {
    "user": null    # ← 整个 user 变成 null
  },
  "errors": [
    {
      "message": "Cannot return null for non-nullable field User.email",
      "path": ["user", "email"]
    }
  ]
}
```

## 五、非空设计决策

```yaml
应该标记非空:
  - 主键 ID: 永远存在
  - 创建时间: 自动生成
  - 核心业务字段: 必须有值
  - 状态字段: 一定有初始状态

应该允许可空:
  - 可选个人信息: bio, avatar, phone
  - 需要权限才能查看的字段
  - 计算成本高的关联数据
  - 可能不存在的关联

列表非空建议:
  - [X!]! - 最常见：列表非空，元素非空
  - [X!]  - 允许 null 但元素不为空
  - [X]!  - 列表存在但可能有 null 元素
  - [X]   - 最宽松，不推荐
```

## 六、类型缩放 (Type Narrowing)

```graphql
# 类型缩放 - 放宽是安全的，收紧是破坏性的

# ✓ 安全：放宽类型
# String! → String（可空化）
type User {
  name: String      # 之前是 String!
}

# ✓ 安全：放宽列表
# [Order!]! → [Order]!
type User {
  orders: [Order]!   # 元素可空
}

# ✗ 破坏性：收紧类型
# String → String!
type User {
  name: String!      # 之前是 String，客户端没处理 null
}

# ✗ 破坏性：收紧列表
# [Order] → [Order!]!
type User {
  orders: [Order!]!   # 之前客户端可能收到 null 列表和 null 元素
}
```

## 七、注意事项

1. **非空标记影响客户端错误处理**
2. **不要过度使用非空**
3. **空列表比 null 更好**
4. **输入非空 = 必填参数**
5. **列表可空意味着客户端要处理 null 和 [] 两种情况**
6. **非空字段的 Resolver 不能返回 null**，否则整个查询可能失败
7. **新字段默认可空**，除非明确知道永远有值
