# 查询与变更最佳实践

## 一、实践总结

```yaml
查询 (Query):
  设计:
    - 命名查询，方便追踪
    - 变量传参，提高复用
    - 片片复用，减少重复

  性能:
    - DataLoader 解决 N+1
    - 深度/复杂度限制
    - 分页查询控制数据量
    - 缓存策略优化

变更 (Mutation):
  设计:
    - Input 类型包装参数
    - 返回更新后完整对象
    - Payload 模式返回错误
    - 支持部分更新

  安全:
    - 输入验证
    - 权限检查
    - 操作审计日志
    - 幂等性保证

订阅 (Subscription):
  设计:
    - 按条件过滤事件
    - WebSocket 连接鉴权
    - 断线重连处理
    - PubSub 跨实例同步
```

## 二、常见反模式

```yaml
反模式:
  查询:
    - 匿名查询
    - 硬编码参数
    - 过深嵌套 (无限制)
    - N+1 查询
    - SELECT *

  变更:
    - 返回 Boolean
    - 无输入验证
    - 无错误处理
    - 大批量无分页
```

## 三、检查清单

| 检查项 | 说明 | 状态 |
|--------|------|------|
| 命名查询 | 便于监控和调试 | 必须 |
| DataLoader | 解决 N+1 问题 | 必须 |
| 查询限制 | 深度 + 复杂度 | 必须 |
| Input 类型 | 包装 Mutation 参数 | 必须 |
| 返回完整对象 | Mutation 后便于缓存 | 建议 |
| 错误 Payload | 规范错误返回 | 建议 |
| 审计日志 | 敏感操作记录 | 必须 |

## 四、缓存策略

```yaml
Apollo Client 缓存策略:
  cache-first (默认):
    - 先查缓存，有则返回
    - 适合: 大多数场景

  cache-and-network:
    - 同时返回缓存和网络数据
    - 适合: 需要实时性的列表

  network-only:
    - 每次都走网络
    - 适合: 关键操作

  no-cache:
    - 不使用缓存
    - 适合: 一次性查询

  cache-only:
    - 只查缓存
    - 适合: 离线场景
```

```javascript
// 缓存更新策略
const [createTodo] = useMutation(CREATE_TODO, {
  update(cache, { data: { createTodo } }) {
    // 方式 1: 手动更新缓存
    cache.modify({
      fields: {
        todos(existingTodos = []) {
          const newTodoRef = cache.writeFragment({
            data: createTodo,
            fragment: TODO_FRAGMENT,
          });
          return [...existingTodos, newTodoRef];
        },
      },
    });

    // 方式 2: 重新读取查询
    // cache.evict({ fieldName: 'todos' });
  },
});
```

## 五、测试策略

```javascript
// 单元测试 Resolver
import { createTestClient } from 'apollo-server-testing';

describe('User Resolvers', () => {
  it('should create a user', async () => {
    const { mutate } = createTestClient(server);

    const res = await mutate({
      mutation: CREATE_USER,
      variables: {
        input: { name: '张三', email: 'test@test.com', password: '123456' },
      },
    });

    expect(res.data.createUser.name).toBe('张三');
    expect(res.data.createUser.id).toBeDefined();
  });

  it('should return error for duplicate email', async () => {
    const { mutate } = createTestClient(server);

    const res = await mutate({
      mutation: CREATE_USER,
      variables: {
        input: { name: '李四', email: 'existing@test.com', password: '123456' },
      },
    });

    expect(res.errors).toBeDefined();
    expect(res.errors[0].extensions.code).toBe('DUPLICATE_EMAIL');
  });
});
```

## 六、注意事项

1. **查询和变更的规范要团队统一**
2. **Codegen 保证前后端类型一致**
3. **性能优化要数据驱动**
4. **安全策略要前置到 Schema 设计**
5. **持续监控和优化查询性能**
6. **Resolver 和业务逻辑分层测试**
7. **缓存策略要根据业务场景配置**
