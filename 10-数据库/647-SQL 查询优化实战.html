案例 1: 索引优化

// ========== 案例: 电商订单查询 ========== -- 问题: 查询用户某个月的订单很慢 SELECT * FROM orders WHERE user_id = 123 AND created_at >= '2024-01-01' AND created_at < '2024-02-01' ORDER BY created_at DESC; -- EXPLAIN 显示: type=ALL (全表扫描!) -- 解决方案: 创建复合索引 CREATE INDEX idx_orders_user_date ON orders(user_id, created_at); -- 再次 EXPLAIN: type=ref, Extra=Using index condition -- 行数从 100 万降到 50! -- ========== 案例 2: 覆盖索引 ========== -- 问题: 查询经常慢 SELECT id, user_id, amount, created_at FROM orders WHERE user_id = 123 ORDER BY created_at DESC; -- 如果查询只涉及这几列,创建覆盖索引: CREATE INDEX idx_orders_covering ON orders(user_id, created_at, amount); -- Extra: Using index (完全不需要回表!) -- ========== 案例 3: 索引选择性 ========== -- 低选择性索引 (效果差): -- gender 列只有'男','女' → 选择性低 CREATE INDEX idx_users_gender ON users(gender); -- 几乎没用! -- 高选择性索引 (效果好): CREATE INDEX idx_users_email ON users(email); -- 选择性近 100% -- 联合索引: 高选择性列放前面 -- ❌: INDEX(gender, email) -- gender 选择性低 -- ✅: INDEX(email, gender) -- email 选择性高

案例 2: JOIN 优化

// ========== JOIN 优化核心 ========== -- 1. JOIN 列必须建索引! -- 2. 小表驱动大表! -- 3. 只返回需要的列! -- ========== 案例: 用户订单统计 ========== -- 原始查询: SELECT u.name, COUNT(o.id) AS order_count, SUM(o.amount) AS total FROM users u LEFT JOIN orders o ON u.id = o.user_id GROUP BY u.id, u.name; -- EXPLAIN: -- orders: type=ALL, Extra=Using join buffer ← 没索引! -- 优化 1: 在 o.user_id 加索引 CREATE INDEX idx_orders_user_id ON orders(user_id); -- 再次 EXPLAIN: type=ref, 不再 Using join buffer -- 优化 2: 先聚合再 JOIN SELECT u.name, COALESCE(os.order_count, 0) AS order_count, COALESCE(os.total, 0) AS total FROM users u LEFT JOIN ( SELECT user_id, COUNT(*) AS order_count, SUM(amount) AS total FROM orders GROUP BY user_id ) os ON u.id = os.user_id; -- ========== 案例: 多表 JOIN ========== -- 性能关键: 确保 ON 条件上的外键都有索引 -- 用户 → 订单 → 订单明细 → 商品 SELECT u.name, o.id AS order_id, p.name AS product, oi.quantity FROM users u JOIN orders o ON u.id = o.user_id -- o.user_id 有索引? JOIN order_items oi ON o.id = oi.order_id -- oi.order_id 有索引? JOIN products p ON oi.product_id = p.id -- p.id (PK 有索引) -- 所有 JOIN 列都应建索引!

案例 3: 分页优化

// ========== 大分页优化 ========== -- 问题: 偏移量越大越慢 SELECT * FROM orders ORDER BY id LIMIT 100000, 20; -- 扫描 100020 行,丢弃前 100000 行! -- 优化 1: 延迟关联 (先查 ID,再 JOIN) SELECT o.* FROM orders o JOIN (SELECT id FROM orders ORDER BY id LIMIT 100000, 20) tmp ON o.id = tmp.id; -- 优化 2: 游标分页 (基于上一页最后 ID) -- 适用于: 滚动加载,不跳页 SELECT * FROM orders WHERE id > 100000 -- 上一页最后 ID ORDER BY id LIMIT 20; -- 优化 3: 复合索引 -- 如果 ORDER BY user_id, created_at 是覆盖索引 -- 延迟关联效果更明显 -- ========== COUNT 优化 ========== -- 问题: COUNT(*) 遍历整个表 SELECT COUNT(*) FROM orders; -- 1000 万行 → 慢! -- 优化 1: 使用近似值 -- SHOW TABLE STATUS LIKE 'orders'; -- PostgreSQL: ANALYZE orders; SELECT reltuples FROM pg_class; -- 优化 2: 使用二级索引 (比主键小) SELECT COUNT(id) FROM orders; -- 如果 id 是主键,用主键索引 -- 实际上 InnoDB 会自动选最小的索引 -- 优化 3: 条件 COUNT -- COUNT(status = 'completed') 用条件索引 CREATE INDEX idx_orders_completed ON orders(status) WHERE status = 'completed'; -- PG -- 优化 4: 缓存计数 -- 用 Redis 维护计数器 -- INSERT → INCR, DELETE → DECR -- ========== OR 优化 ========== -- ❌ OR 可能导致索引失效: SELECT * FROM users WHERE city = 'Beijing' OR age = 25; -- ✅ 用 UNION: SELECT * FROM users WHERE city = 'Beijing' UNION ALL SELECT * FROM users WHERE age = 25 AND city <> 'Beijing'; -- ✅ 或改用 IN: SELECT * FROM users WHERE city IN ('Beijing', 'Shanghai'); -- 走索引

优化综合案例

// ========== 综合案例: 电商仪表板 ========== -- 需求: 查询今年每个月的销售额,以及同比去年增长 -- ❌ 原始 (慢): SELECT MONTH(created_at) AS month, SUM(amount) AS revenue_this_year, (SELECT SUM(amount) FROM orders WHERE YEAR(created_at) = YEAR(CURDATE()) - 1 AND MONTH(created_at) = MONTH(o.created_at) ) AS revenue_last_year FROM orders o WHERE YEAR(created_at) = YEAR(CURDATE()) GROUP BY MONTH(created_at) ORDER BY month; -- 问题: 关联子查询每行执行一次! WHERE YEAR(col) 不走索引! -- ✅ 优化: SELECT this.month, this.revenue AS revenue_this_year, COALESCE(last.revenue, 0) AS revenue_last_year, ROUND((this.revenue - COALESCE(last.revenue, 0)) / COALESCE(last.revenue, 1) * 100, 1) AS growth_pct FROM ( SELECT MONTH(created_at) AS month, SUM(amount) AS revenue FROM orders WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01' GROUP BY MONTH(created_at) ) this LEFT JOIN ( SELECT MONTH(created_at) AS month, SUM(amount) AS revenue FROM orders WHERE created_at >= '2023-01-01' AND created_at < '2024-01-01' GROUP BY MONTH(created_at) ) last ON this.month = last.month ORDER BY this.month; -- 优化点: -- 1. 范围查询代替 YEAR() 函数 (可以走索引) -- 2. 提前聚合,避免关联子查询 -- 3. 只需要扫描两遍数据 (不是 12 遍)
💡 优化要点: 慢查询日志找问题; 索引最左前缀; 覆盖索引避免回表; JOIN 列必须索引; 大分页用延迟关联; 范围查询代替函数; 子查询尽量改 JOIN; OR 改 UNION; 监控索引使用情况。
