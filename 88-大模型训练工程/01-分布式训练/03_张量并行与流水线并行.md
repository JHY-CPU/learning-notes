## 1. 并行策略概述


当模型参数量超过单 GPU 显存容量时，需要将模型本身拆分到多个 GPU 上。主要有两种策略：张量并行（Tensor Parallelism）和流水线并行（Pipeline Parallelism）。


| 并行方式 | 切分维度 | 通信频率 | 通信量 | GPU 利用率 | 典型框架 |
| --- | --- | --- | --- | --- | --- |
| 数据并行 | 数据 | 每 step | 低 | 高 | DDP, FSDP |
| 张量并行 | 层内张量 | 每层 | 高 | 高 | Megatron-LM |
| 流水线并行 | 层间 | 每 micro-batch | 中 | 中(气泡) | GPipe, PipeDream |
| 序列并行 | 序列长度 | 每层 | 中 | 高 | Ring Attention |


> **Note:** **核心原则：**
> 张量并行需要高带宽互连（NVLink），适合节点内；流水线并行对带宽要求较低，适合跨节点。数据并行是最外层的并行，可以与任何模型并行策略组合。


## 2. 张量并行 — 列切分 (Column Parallelism)


张量并行的核心思想是将模型中的大矩阵乘法拆分到多个 GPU 上并行执行。以 Megatron-LM 为例，前馈网络（FFN）中的线性层可以沿列维度切分。


### 2.1 FFN 第一个线性层 (Column Split)


对于 FFN 的第一层 Linear(in_features, 4*in_features)，沿输出维度（列）切分：


```
Column Parallelism — 线性层 Y = XW, W ∈ R^{d × 4d}

输入 X (d × N) 在所有 GPU 上相同
        │
        ▼
┌───────────────────────────────────────────────────┐
│                                                   │
│  GPU 0              GPU 1              GPU 2      │
│  W_0 ∈ R^{d×d}     W_1 ∈ R^{d×d}     W_2 ∈ R^{d×d}│
│       │                  │                  │     │
│       ▼                  ▼                  ▼     │
│  Y_0 = X·W_0       Y_1 = X·W_1       Y_2 = X·W_2│
│                                                   │
│  每个 GPU 计算输出的一部分, 无需通信                │
│  (因为 X 相同, W 切分, 结果自然切分)               │
└───────────────────────────────────────────────────┘

结果: Y = concat(Y_0, Y_1, Y_2) ∈ R^{N × 4d}
       (仅在需要完整输出时才 All-Gather)
```


```
python
import torch
import torch.nn as nn
from megatron.core.tensor_parallel import (
    ColumnParallelLinear, RowParallelLinear
)

# Column Parallel Linear — FFN 第一层
# 权重矩阵按列切分到 tp_size 个 GPU 上
fc1 = ColumnParallelLinear(
    input_size=4096,          # d
    output_size=16384,        # 4d
    bias=True,
    gather_output=False,      # 不收集完整输出 (与下一层串联)
    init_method=init_method,
    async_tensor_model_parallel_allreduce=True # 异步通信
)
# 每个 GPU 上的权重大小: 4096 × 4096 (总参数量不变)
```


## 3. 张量并行 — 行切分 (Row Parallelism)


FFN 的第二层采用行切分，与第一层的列切分配合，只需在两层之间做一次通信（Reduce-Scatter 或 All-Reduce）。


### 3.1 FFN 第二个线性层 (Row Split)


```
Row Parallelism — 线性层 Y = XW, W ∈ R^{4d × d}

输入 X 已经是切分的: X_0, X_1, X_2 (每个 GPU 一份)
        │
        ▼
┌───────────────────────────────────────────────────┐
│                                                   │
│  GPU 0              GPU 1              GPU 2      │
│  X_0 ∈ R^{N×d}     X_1 ∈ R^{N×d}     X_2 ∈ R^{N×d}│
│  W_0 ∈ R^{d×d}     W_1 ∈ R^{d×d}     W_2 ∈ R^{d×d}│
│       │                  │                  │     │
│       ▼                  ▼                  ▼     │
│  Y_0 = X_0·W_0     Y_1 = X_1·W_1     Y_2 = X_2·W_2│
│       │                  │                  │     │
│       └──────────────────┼──────────────────┘     │
│                          ▼                        │
│                  All-Reduce (求和)                  │
│                          ▼                        │
│                    Y = Σ Y_i                        │
│                  (每个 GPU 得到相同结果)             │
└───────────────────────────────────────────────────┘
```


```
python
# Row Parallel Linear — FFN 第二层
fc2 = RowParallelLinear(
    input_size=16384,         # 4d
    output_size=4096,         # d
    bias=True,
    input_is_parallel=True,   # 输入已经是切分的
    init_method=init_method,
    skip_bias_add=False
)
```


### 3.2 FFN 张量并行完整流程


```
FFN 张量并行完整流程:

LayerNorm(X)  ──── 全 GPU 持有相同输入 ────►
    │
    ▼
ColumnParallelLinear (W1: 列切分)
    │
    │  每个 GPU 输出一部分: [Y_0 | Y_1 | Y_2]
    ▼
激活函数 (SiLU/GELU)
    │
    │  各 GPU 独立计算, 无需通信
    ▼
RowParallelLinear (W2: 行切分)
    │
    │  需要 All-Reduce 求和
    ▼
All-Reduce + 残差连接
    │
    ▼
输出 (每个 GPU 持有相同结果)
```


> **Note:** **通信优化：**
> Megatron-LM 通过将 All-Reduce 与反向传播的梯度计算重叠（通过异步通信），减少了通信对训练速度的影响。在张量并行组内，通信量相对于数据并行要大得多，因此张量并行组应放在高带宽互连（如 NVLink）的 GPU 之间。


## 4. 注意力层的张量并行


多头注意力（MHA）天然适合张量并行——每个 GPU 负责一部分注意力头的计算。


### 4.1 注意力头切分


```
多头注意力张量并行 (tp_size=2, num_heads=32):

GPU 0: heads [0..15]     GPU 1: heads [16..31]

输入 X (相同)
  │                         │
  ▼                         ▼
Q_0,K_0,V_0 = X·W_q0,..   Q_1,K_1,V_1 = X·W_q1,..
  │                         │
  ▼                         ▼
Attention(Q_0,K_0,V_0)    Attention(Q_1,K_1,V_1)
  │                         │
  │    Output_0             │    Output_1
  └────────┬────────────────┘
           ▼
     All-Reduce 或保持切分 (取决于下一层)
           │
           ▼
    W_o: RowParallelLinear (行切分, 与 FFN 同理)
```


### 4.2 KV Cache 与张量并行


在推理时，每个 GPU 只需要维护自己负责的那部分 KV Cache，显存占用降低为原来的 1/tp_size。但在训练时，需要注意 KV Cache 的同步。


## 5. 流水线并行基础


流水线并行（Pipeline Parallelism）将模型的不同层分配到不同的 GPU 上，数据像流水线一样依次通过各阶段。


### 5.1 朴素流水线的气泡问题


```
朴素流水线 (4 stages, 4 micro-batches):

时间 →
GPU 0: [F0]──[F1]──[F2]──[F3]────────[B3]──[B2]──[B1]──[B0]
GPU 1:      [F0]──[F1]──[F2]──[F3]──[B3]──[B2]──[B1]──[B0]
GPU 2:           [F0]──[F1]──[F2]──[F3]──[B3]──[B2]──[B1]──[B0]
GPU 3:                [F0]──[F1]──[F2]──[F3]──[B3]──[B2]──[B1]──[B0]

气泡 (bubble) = GPU 空闲时间
气泡率 ≈ (pp_size - 1) / (pp_size - 1 + 2 × num_micro_batches)
```


> **Warning:** **气泡问题：**
> 朴素流水线在填充和排空阶段有大量 GPU 空闲。气泡率与 micro-batch 数量成反比——micro-batch 越多，气泡率越低。但 micro-batch 过小会降低计算效率。


### 5.2 GPipe 策略


GPipe 将一个 mini-batch 拆成多个 micro-batch，前向阶段连续执行所有 micro-batch，然后再统一执行反向阶段。这样可以减少气泡。


```
GPipe (4 stages, 8 micro-batches):

GPU 0: [F0][F1][F2][F3][F4][F5][F6][F7]──[B7][B6][B5][B4][B3][B2][B1][B0]
GPU 1:  [F0][F1][F2][F3][F4][F5][F6][F7]──[B7][B6][B5][B4][B3][B2][B1][B0]
GPU 2:   [F0][F1][F2][F3][F4][F5][F6][F7]──[B7][B6][B5][B4][B3][B2][B1][B0]
GPU 3:    [F0][F1][F2][F3][F4][F5][F6][F7]──[B7][B6][B5][B4][B3][B2][B1][B0]

气泡率 = (pp-1) / (pp-1+2N) ≈ (pp-1)/2N (当 N 很大时)
```


GPipe 需要保存所有 micro-batch 的激活值直到反向传播完成，内存占用较高。


## 6. 1F1B 调度策略


1F1B（One Forward One Backward）是 Megatron-LM 采用的流水线调度策略，通过交替执行前向和反向来降低内存峰值。


### 6.1 调度流程


```
1F1B 调度 (4 stages, 8 micro-batches):

GPU 0: [F0][F1][F2][F3][F4][F5][F6][F7]
              [B0][F7'] [B1][B2][B3][B4][B5][B6][B7]
GPU 1:  [F0][F1][F2][F3][F4][F5][F6]
             [B0][F7] [B1][B2][B3][B4][B5][B6][B7]
GPU 2:   [F0][F1][F2][F3][F4][F5]
            [B0][F6] [B1][B2][B3][B4][B5][B6][B7]
GPU 3:    [F0][F1][F2][F3][F4]
           [B0][F5] [B1][B2][B3][B4][B5][B6][B7]

关键: 稳态时, 每次只做 1 个前向 + 1 个反向
      → 内存峰值 = 仅需保存 pp_size 个 micro-batch 的激活值
```


### 6.2 内存优势


- **GPipe 内存峰值：**
   需要保存所有 micro-batch 的激活值 (num_micro_batches × activation_size)
- **1F1B 内存峰值：**
   只需保存 pp_size 个 micro-batch 的激活值
- 1F1B 在 micro-batch 数量较多时，内存占用远低于 GPipe


### 6.3 虚拟流水线并行 (Virtual Pipeline Parallelism)


Megatron-LM 进一步提出虚拟流水线并行，将模型切分为更多的小块（virtual stages），交错调度以减少气泡：


```
python
# 虚拟流水线并行配置
# num_layers=32, pipeline_model_parallel_size=4
# virtual_pipeline_model_parallel_size=2
# 每个 GPU 持有 2 个虚拟阶段, 每阶段 4 层
# 训练参数 (Megatron-LM)
python pretrain_gpt.py \
    --tensor-model-parallel-size 2 \
    --pipeline-model-parallel-size 4 \
    --num-layers 32 \
    --hidden-size 4096 \
    --num-attention-heads 32 \
    --micro-batch-size 1 \
    --global-batch-size 256 \
    --num-virtual-pipeline-pipeline-parallel-stages 2
```


> **Tip:** **虚拟流水线效果：**
> 虚拟阶段将每个物理 GPU 上的层分成多个虚拟阶段，增加了流水线的填充深度，从而降低气泡率。在 Megatron-LM 论文中，虚拟流水线可以将气泡率从 ~30% 降低到 ~10%。


## 7. 3D 并行组合


3D 并行是数据并行（DP）、张量并行（TP）和流水线并行（PP）的组合，是训练超大模型的标准策略。


### 7.1 组合方式


```
3D 并行拓扑 (假设 64 GPU, 8 GPU/node, 8 nodes):

                ┌──────────── Node 0 (8 GPU) ────────────┐
                │                                          │
                │  TP=2: ┌─────┬─────┐                    │
                │        │GPU 0│GPU 1│ ← 高带宽 NVLink    │
                │        └─────┴─────┘                    │
                │  PP=4: ┌─────┬─────┬─────┬─────┐        │
TP (节点内)      │        │Stage0│Stage1│Stage2│Stage3│       │ DP (最外层)
  ←──────→      │        └─────┴─────┴─────┴─────┘        │  ←──────→
                └──────────────────────────────────────────┘
                ┌──────────── Node 1 (8 GPU) ────────────┐
                │          ... (相同结构, 数据并行)          │
                └──────────────────────────────────────────┘

配置: TP=2, PP=4, DP=8 (2×4×8 = 64 GPU)
```


### 7.2 组合原则


1. **张量并行放在节点内：**
   TP 需要每层通信，对带宽要求最高，应放在 NVLink 互联的 GPU 之间
2. **流水线并行跨节点：**
   PP 只在 micro-batch 边界通信，对带宽要求较低，可以跨 InfiniBand
3. **数据并行放在最外层：**
   DP 通信频率最低（每 step 一次 All-Reduce），可以跨节点


$$
总 GPU 数 = tp_size × pp_size × dp_size
$$


### 7.3 配置示例


| 模型规模 | 总 GPU | TP | PP | DP | 说明 |
| --- | --- | --- | --- | --- | --- |
| 7B | 8 | 1 | 1 | 8 | 只需数据并行 |
| 13B | 16 | 2 | 1 | 8 | TP 跨 2 卡 |
| 70B | 64 | 2 | 4 | 8 | 标准 3D 并行 |
| 175B | 256 | 4 | 8 | 8 | 大规模 3D 并行 |
| 530B | 1024 | 8 | 16 | 8 | 超大规模 |

## 8. 最佳实践与选型


### 8.1 选型决策树


```
并行策略选型:

模型参数量估算 (16Ψ 显存)
│
├─ 16Ψ < 单卡显存
│   └─ DDP 数据并行即可
│
├─ 16Ψ > 单卡, 但 16Ψ/N < 单卡显存 (N = 可用 GPU 数)
│   ├─ 节点内 NVLink 充足 → FSDP (ZeRO-3)
│   └─ 跨节点带宽有限   → ZeRO-2 + DP
│
├─ 模型单层权重 > 单卡显存
│   └─ 必须使用张量并行
│       └─ TP ≤ 节点内 GPU 数 (保证 NVLink)
│
└─ 超大模型 (>100B)
    └─ 3D 并行: TP(节点内) + PP(跨节点) + DP(最外层)
```


### 8.2 关键调优参数


- **micro-batch size：**
   越大越好（减少气泡），但受显存限制
- **TP size：**
   不超过节点内 GPU 数，通常为 2 或 4
- **PP size：**
   尽量大以减少气泡率，但受通信延迟限制
- **Activation recompute：**
   PP 场景下强烈推荐开启


> **Tip:** **经验法则：**
>
> 1. TP size 越小越好（通信开销大），尽量只在节点内
> 2. PP size 越大越好（气泡小），但受网络延迟影响
> 3. DP size 是最灵活的维度
> 4. 通信重叠是关键——让计算掩盖通信延迟
