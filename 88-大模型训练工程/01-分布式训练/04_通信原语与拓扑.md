## 1. 集合通信原语


集合通信（Collective Communication）是分布式训练中多 GPU 之间协调数据传输的基本操作。理解这些原语是优化分布式训练通信开销的基础。


### 1.1 基本通信原语


| 原语 | 功能 | 输入 | 输出 | 典型用途 |
| --- | --- | --- | --- | --- |
| Broadcast | 一对多广播 | 1 个 GPU 的数据 | 所有 GPU 获得相同数据 | 参数初始化同步 |
| Reduce | 多对一归约 | 所有 GPU 的数据 | 1 个 GPU 获得归约结果 | 损失收集 |
| All-Reduce | 多对多归约 | 所有 GPU 的数据 | 所有 GPU 获得归约结果 | 梯度同步 (DDP) |
| Scatter | 一对多分散 | 1 个 GPU 的大块数据 | 各 GPU 获得数据切片 | 数据分发 |
| Gather | 多对一收集 | 各 GPU 的数据切片 | 1 个 GPU 获得完整数据 | 结果收集 |
| All-Gather | 多对多收集 | 各 GPU 的数据切片 | 所有 GPU 获得完整数据 | ZeRO-3 参数收集 |
| Reduce-Scatter | 归约+分散 | 所有 GPU 的完整数据 | 各 GPU 获得归约结果的切片 | ZeRO-2 梯度分片 |
| All-to-All | 全交换 | 各 GPU 的数据 | 各 GPU 获得重排后的数据 | 专家并行路由 |


### 1.2 原语之间的等价关系


$$
All-Reduce = Reduce-Scatter + All-Gather
$$


这个等价关系是 Ring All-Reduce 算法的基础。通过将 All-Reduce 分解为两步，可以实现与 GPU 数量无关的通信量。


## 2. All-Reduce 算法


All-Reduce 是分布式训练中最核心的通信操作，用于同步各 GPU 上的梯度。


### 2.1 Ring All-Reduce


```
Ring All-Reduce (4 GPU, 每 GPU 数据量 K/4):

Phase 1: Scatter-Reduce (N-1 = 3 步)
─────────────────────────────────────────
Step 1:
  GPU 0 → GPU 1: a    (发送 K/4)
  GPU 1 → GPU 2: b    (发送 K/4)
  GPU 2 → GPU 3: c    (发送 K/4)
  GPU 3 → GPU 0: d    (发送 K/4)
  各 GPU 接收并累加: GPU0={a+d}, GPU1={b+a}, GPU2={c+b}, GPU3={d+c}

Step 2:
  GPU 0 → GPU 1: a+d  (发送 K/4)
  GPU 1 → GPU 2: b+a  (发送 K/4)
  GPU 2 → GPU 3: c+b  (发送 K/4)
  GPU 3 → GPU 0: d+c  (发送 K/4)

Step 3:
  ... 继续累加, 最终每个 GPU 持有完整 Σ 的一个切片

Phase 2: All-Gather (N-1 = 3 步)
─────────────────────────────────────────
每个 GPU 将自己的切片传递给邻居, 最终所有 GPU 获得完整结果

总通信量 = 2(N-1) × K/N ≈ 2K (与 N 渐近无关!)
带宽受限时间 = 2K(N-1)/(N×BW) ≈ 2K/BW
```


### 2.2 Tree All-Reduce


Tree All-Reduce 使用二叉树结构，先自底向上归约（Reduce），再自顶向下广播（Broadcast）：


```
Tree All-Reduce (8 GPU):

Phase 1: Reduce (log₂N = 3 层)
         ┌─────┐
         │  Σ  │ ← 最终归约结果
         └──┬──┘
      ┌─────┴─────┐
   ┌──┴──┐     ┌──┴──┐
   │Σ_0-3│     │Σ_4-7│
   └──┬──┘     └──┬──┘
  ┌───┴───┐   ┌───┴───┐
┌─┴─┐  ┌─┴─┐ ┌─┴─┐  ┌─┴─┐
│0,1│  │2,3│ │4,5│  │6,7│

Phase 2: Broadcast (log₂N = 3 层)
         ┌─────┐
         │  Σ  │ ──► 广播到所有 GPU
         └──┬──┘
      ┌─────┴─────┐
   ┌──┴──┐     ┌──┴──┐
   └──┬──┘     └──┬──┘
  ┌───┴───┐   ┌───┴───┐
┌─┴─┐  ┌─┴─┐ ┌─┴─┐  ┌─┴─┐
│0,1│  │2,3│ │4,5│  │6,7│

通信量 = 2 × (N-1)/N × K ≈ 2K
延迟 = 2 × log₂(N) × α  (α = 单次传输延迟)
```


### 2.3 Ring vs Tree 对比


| 算法 | 通信量 | 延迟 | 适用场景 |
| --- | --- | --- | --- |
| Ring | 2K(N-1)/N ≈ 2K | 2(N-1) × α | 大数据量, 高带宽 |
| Tree | 2K(N-1)/N ≈ 2K | 2log₂(N) × α | 小数据量, 低延迟 |
| Recursive Halving | 2K | 2log₂(N) × α | 中等数据量 |


## 3. All-Gather 与 Reduce-Scatter


### 3.1 Ring All-Gather


```
Ring All-Gather (4 GPU):

初始: 每个 GPU 持有数据的一个切片
GPU 0: [A]  GPU 1: [B]  GPU 2: [C]  GPU 3: [D]

Step 1: 每个 GPU 发送自己的切片给下一个 GPU
GPU 0: [A, D]  GPU 1: [B, A]  GPU 2: [C, B]  GPU 3: [D, C]

Step 2: 继续传递
GPU 0: [A, D, C]  GPU 1: [B, A, D]  GPU 2: [C, B, A]  GPU 3: [D, C, B]

Step 3: 最终所有 GPU 拥有完整数据
GPU 0: [A, B, C, D]  GPU 1: [A, B, C, D]  GPU 2: [A, B, C, D]  GPU 3: [A, B, C, D]

总通信量 = (N-1) × K/N × N = K(N-1) ≈ K
```


### 3.2 Ring Reduce-Scatter


```
Ring Reduce-Scatter (4 GPU):

初始: 每个 GPU 持有完整数据 [A, B, C, D]

Step 1: 发送对应切片, 接收并累加
GPU 0 发送 D 给 GPU 1, 接收 A 并累加 → [A+B, B, C, D]
GPU 1 发送 A 给 GPU 2, 接收 B 并累加
...

Step 2-3: 继续轮转
最终每个 GPU 只保留 Σ 的一个切片
GPU 0: Σ_A (所有 GPU 的 A 切片之和)
GPU 1: Σ_B
GPU 2: Σ_C
GPU 3: Σ_D

通信量 = (N-1) × K/N × N ≈ K
```


### 3.3 在分布式训练中的应用


- **All-Reduce：**
   DDP 中同步梯度 — 每个 GPU 都需要完整的平均梯度
- **Reduce-Scatter：**
   ZeRO Stage 2 中分片梯度 — 每个 GPU 只保留自己负责的那部分
- **All-Gather：**
   ZeRO Stage 3 中收集参数 — 前向/反向时临时收集完整参数
- **All-to-All：**
   MoE（混合专家）中路由 token 到不同专家


## 4. NCCL 通信库


NCCL（NVIDIA Collective Communications Library）是 NVIDIA 提供的高性能 GPU 通信库，是大模型训练中事实上的标准。


### 4.1 NCCL 核心特性


- **硬件感知：**
   自动检测 NVLink、PCIe、InfiniBand 等互联拓扑
- **算法选择：**
   根据数据量和 GPU 数量自动选择最优算法（Ring/Tree）
- **多流并行：**
   支持在多个 CUDA Stream 上并发执行通信
- **拓扑优化：**
   根据硬件拓扑构建最优通信路径


### 4.2 NCCL 环境变量


```
bash

export NCCL_IB_GID_INDEX=3 # IB GID 索引
# 调试与日志

export NCCL_DEBUG_SUBSYS=ALL           # 子系统: INIT/GRAPH/TUNING/ALL
export NCCL_DEBUG_FILE=nccl_log_%h.log # 日志文件路径
# 性能调优
export NCCL_NTHREADS=256 # 每个 GPU 的通信线程数
export NCCL_MIN_NCHANNELS=4 # 最小通信通道数
export NCCL_MAX_NCHANNELS=16 # 最大通信通道数
export NCCL_BUFFSIZE=4194304 # 缓冲区大小 (4MB)
# 跨节点通信

# 拓扑相关

export NCCL_IGNORE_CPU_AFFINITY=1 # 忽略 CPU 亲和性
```


### 4.3 PyTorch 中使用 NCCL


```
python
import torch.distributed as dist

# 初始化进程组 (使用 NCCL 后端)
dist.init_process_group(
    backend="nccl",        # GPU 通信必须用 nccl
    init_method="env://",   # 从环境变量读取配置
    world_size=8,
    rank=local_rank
)

# 手动执行集合通信
tensor = torch.randn(1000).cuda()
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)   # All-Reduce
dist.all_gather(tensor_list, tensor)              # All-Gather
dist.reduce_scatter(output, input_list, op=dist.ReduceOp.SUM)  # Reduce-Scatter
# P2P 通信
if rank == 0:
    dist.send(tensor, dst=1)
elif rank == 1:
    dist.recv(tensor, src=0)
```


## 5. 硬件拓扑


理解 GPU 之间的硬件连接拓扑对于优化通信至关重要。不同互联技术的带宽差异巨大。


### 5.1 互联技术对比


| 互联技术 | 单向带宽 | 双向带宽 | 延迟 | 范围 |
| --- | --- | --- | --- | --- |
| NVLink 4.0 | 900 GB/s (H100) | 1800 GB/s | ~1 us | 节点内 GPU |
| NVLink 3.0 | 600 GB/s (A100) | 1200 GB/s | ~1 us | 节点内 GPU |
| NVSwitch | 900 GB/s per GPU | 全互联 | ~1 us | 节点内全互联 |
| PCIe 5.0 x16 | 64 GB/s | 128 GB/s | ~2 us | 节点内 |
| InfiniBand NDR | 400 Gb/s (50 GB/s) | 800 Gb/s | ~1-5 us | 跨节点 |
| InfiniBand HDR | 200 Gb/s (25 GB/s) | 400 Gb/s | ~1-5 us | 跨节点 |
| RoCE v2 | 100-400 Gb/s | 200-800 Gb/s | ~2-10 us | 跨节点 |


### 5.2 典型拓扑结构


```
DGX A100 8卡 拓扑 (NVSwitch 全互联):

          ┌────────── NVSwitch ──────────┐
          │                              │
    ┌─────┼─────┬─────┬─────┬─────┬─────┼─────┐
    │     │     │     │     │     │     │     │
   GPU0  GPU1  GPU2  GPU3  GPU4  GPU5  GPU6  GPU7
    └─────┴─────┴─────┴─────┴─────┴─────┴─────┘
          │          PCIe Gen4           │
          └──────── CPU + NIC ───────────┘
                     │
              InfiniBand HDR (跨节点)

任意两 GPU 间 NVLink 带宽: 600 GB/s (A100)
跨节点 InfiniBand 带宽: 25 GB/s per NIC

→ 张量并行必须放在节点内 (NVLink)
→ 流水线并行可以跨节点 (InfiniBand)
```


> **Note:** **带宽比：**
> NVLink (600 GB/s) vs InfiniBand (25 GB/s) = 24:1。这就是为什么张量并行（需要每层通信）必须放在节点内，而流水线并行（每 micro-batch 通信一次）可以跨节点。


## 6. 通信计算重叠


通信计算重叠（Communication-Computation Overlap）是最大化 GPU 利用率的关键技术。通过让通信和计算在不同 CUDA Stream 上并行执行，可以将通信延迟隐藏在计算时间之后。


### 6.1 CUDA Stream 并行


```
标准执行 (无重叠):
时间 →
Stream 0 (计算): [前向 Layer1] [前向 Layer2] ... [反向 Layer2] [反向 Layer1]
Stream 1 (通信):                                     [All-Reduce]
                 ──────────────────────────────────────────────────►
                                                    ↑
                                              通信阻塞计算

重叠执行:
时间 →
Stream 0 (计算): [前向 Layer1] [前向 Layer2] ... [反向 Layer3] [反向 Layer2] [反向 Layer1]
Stream 1 (通信):        [All-Reduce Bucket1] [All-Reduce Bucket2] [All-Reduce Bucket3]
                 ──────────────────────────────────────────────────────────────────►
                         ↑                   ↑                   ↑
                    通信被计算掩盖       通信被计算掩盖       通信被计算掩盖
```


### 6.2 DDP 中的重叠实现


DDP 通过以下机制实现通信计算重叠：


1. **反向桶排序：**
   按参数使用顺序的逆序组织桶，确保梯度先计算完的先通信
2. **异步 All-Reduce：**
   使用非阻塞的
   `ncclAllReduce`
   ，通信在独立 CUDA Stream 上执行
3. **梯度就绪钩子：**
   每个桶的梯度就绪时立即触发通信，无需等待所有梯度


### 6.3 张量并行中的重叠


```
python
# Megatron-LM 中的异步 All-Reduce
# 在反向传播中, 梯度计算与 All-Reduce 重叠
class ColumnParallelLinear(nn.Module):
    def __init__(self, ...):
        self.async_tensor_model_parallel_allreduce = True
# 注册反向钩子
if self.async_tensor_model_parallel_allreduce:
            self.grad_accum_hook = register_allreduce_hook(self.weight)

    def forward(self, x):
        # 前向: Column Parallel, 无需通信
        output = torch.matmul(x, self.weight)
        return output
        # 反向时, 钩子自动触发异步 All-Reduce
```


### 6.4 ZeRO 中的预取重叠


- **FSDP forward_prefetch：**
   在当前层前向计算时，预取下一层的参数（All-Gather）
- **FSDP backward_prefetch：**
   在当前层反向计算时，预取上一层的参数
- **目的：**
   将 All-Gather 的延迟隐藏在计算中


> **Tip:** **重叠效率评估：**
> 设计算时间为 T_comp，通信时间为 T_comm，重叠效率 = max(0, 1 - T_comm / T_comp)。如果 T_comm 完全被 T_comp 覆盖（T_comm ≤ T_comp），则通信开销为零。


## 7. 通信性能分析


### 7.1 理论带宽估算


$$
通信时间 = 数据量 / 有效带宽 + 延迟开销
$$


以 All-Reduce 梯度同步为例，假设模型参数量为 P（FP16 = 2P bytes）：


- **All-Reduce 数据量：**
   2P bytes（Ring 算法的 2K 理论值）
- **节点内 NVLink：**
   2P / 600GB/s — 例如 7B 模型：14GB / 600GB/s ≈ 23ms
- **跨节点 InfiniBand：**
   2P / 25GB/s — 例如 7B 模型：14GB / 25GB/s ≈ 560ms

### 7.3 常见性能瓶颈


| 瓶颈类型 | 症状 | 排查方法 | 解决方案 |
| --- | --- | --- | --- |
| 带宽不足 | 通信时间占比高 | Profiler 查看 ncclKernel | 增大桶, 减少通信次数 |
| 通信气泡 | GPU 空闲等待 | Timeline 查看 gap | 增加重叠, 调整桶顺序 |
| PCIe 瓶颈 | 节点内通信慢 | 检查 NVLink 拓扑 | 确保 NVLink 连通 |
| 网络抖动 | 偶发慢通信 | NCCL_DEBUG=INFO | 增加 NCCL_IB_TIMEOUT |
| CPU 瓶颈 | Launch 延迟 | Profiler CPU 时间 | 设置 OMP_NUM_THREADS |


> **Warning:** **NCCL 调试技巧：**
> 设置
> `NCCL_DEBUG=INFO`
> 可以查看 NCCL 选择的算法和拓扑信息。如果看到
> `via=PCIe`
> 而不是
> `via=NVL`
> ，说明 GPU 没有通过 NVLink 连接，性能会大幅下降。
