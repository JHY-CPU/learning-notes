## 1. 显存占用分析


大模型训练的显存占用主要由四部分组成：模型参数、梯度、优化器状态和激活值。理解各部分的占比是优化显存的前提。


### 1.1 显存组成


```
训练显存组成 (以 7B 模型, BF16 + Adam 为例):

┌─────────────────────────────────────────────────────┐
│                  GPU 显存 (80GB)                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌──────────────────────────────────────────┐       │
│  │ 参数 (Parameters)          14 GB (17.5%) │       │
│  │ BF16: 7B × 2 bytes                     │       │
│  └──────────────────────────────────────────┘       │
│  ┌──────────────────────────────────────────┐       │
│  │ 梯度 (Gradients)           14 GB (17.5%) │       │
│  │ BF16: 7B × 2 bytes                     │       │
│  └──────────────────────────────────────────┘       │
│  ┌──────────────────────────────────────────┐       │
│  │ 优化器状态 (Optimizer)      84 GB (105%) │       │
│  │ FP32 Master: 7B × 4 = 28 GB             │       │
│  │ Adam m:       7B × 4 = 28 GB             │       │
│  │ Adam v:       7B × 4 = 28 GB             │       │
│  └──────────────────────────────────────────┘       │
│  ┌──────────────────────────────────────────┐       │
│  │ 激活值 (Activations)   变化大 (10-50 GB) │       │
│  │ 取决于 batch_size, seq_len, hidden_size  │       │
│  └──────────────────────────────────────────┘       │
│                                                     │
│  无优化总计: 112 GB + 激活 >> 80 GB → OOM!          │
└─────────────────────────────────────────────────────┘
```


### 1.2 各部分的优化方向


| 组件 | 占比 | 优化方法 | 节省效果 |
| --- | --- | --- | --- |
| 参数 | ~17% | 混合精度 / ZeRO-3 / Offload | 50% ~ 90% |
| 梯度 | ~17% | 混合精度 / ZeRO-2 | 50% |
| 优化器 | ~50% | ZeRO-1 / 8-bit Adam / Offload | 75% ~ 100% |
| 激活值 | ~16% | Activation Checkpointing | 60% ~ 80% |


## 2. Activation Checkpointing


Activation Checkpointing（也称 Gradient Checkpointing）通过在反向传播时重新计算前向激活值来减少显存占用，是除 ZeRO 之外最有效的显存优化技术之一。


### 2.1 原理


```
标准训练 vs Checkpoint 训练:

标准训练 (保存所有激活值):
前向: [L1] → [L2] → [L3] → [L4] → Loss
       ↓A1    ↓A2    ↓A3    ↓A4
       保存   保存   保存   保存   ← 显存: O(L)

反向: [L1] ← [L2] ← [L3] ← [L4]
       ↑A1    ↑A2    ↑A3    ↑A4
       直接用  直接用  直接用  直接用

─────────────────────────────────────────
Checkpoint 训练 (仅保存部分激活值):
前向: [L1] → [L2] → [L3] → [L4] → Loss
       ↓A1          ↓A3
       保存checkpoint  保存checkpoint  ← 显存: O(L/k)

反向: [L1] ← [L2] ← [L3] ← [L4]
       ↑A1    ↑A2    ↑A3    ↑A4
       直接用  重新计算  直接用  重新计算
              (从A1前向)
                      (从A3前向)

额外计算: ~33% (每层平均重新前向一次)
显存节省: 与 checkpoint 粒度成反比
```


### 2.2 内存-计算权衡


$$
标准: 内存 = O(L), 计算 = 1× 前向 + 1× 反向
                Checkpoint: 内存 = O(L/k), 计算 = 1× 前向 + (1 + 1/k)× 反向
$$


其中 L 为层数，k 为 checkpoint 间隔。k = 1（每层 checkpoint）时内存最省但计算增加最多；k = L（不 checkpoint）时内存最大但计算最优。


> **Note:** **最优 checkpoint 数：**
> 对于 L 层的网络，最优 checkpoint 数为 O(√L)，此时内存为 O(L/√L) = O(√L)，计算开销增加约 33%。


## 3. Gradient Checkpointing 实现


### 3.1 PyTorch 原生实现


```
python
import torch
from torch.utils.checkpoint import checkpoint, checkpoint_sequential

# 方法一: 对单个模块使用 checkpoint
class CheckpointedBlock(nn.Module):
    def __init__(self, block):
        super().__init__()
        self.block = block

    def forward(self, x):
        # checkpoint 会在反向时重新计算 forward
return checkpoint(self.block, x, use_reentrant=False)

# 方法二: 对 Sequential 的子模块使用
model = nn.Sequential(*[Block(i) for i in range(24)])
output = checkpoint_sequential(model, segments=4, input=x)

# 方法三: HuggingFace Transformers 中使用
model.gradient_checkpointing_enable()
```


### 3.2 use_reentrant 参数


| 参数值 | 机制 | 兼容性 | 推荐 |
| --- | --- | --- | --- |
| use_reentrant=True | 旧版实现, 非重入式 | 需 requires_grad=True | 不推荐 |
| use_reentrant=False | 新版实现, 包装式 | 兼容性更好 | **推荐** |


### 3.3 Transformer 模型中的应用


```
python
# 为每个 Transformer 层应用 checkpoint
import functools
from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    checkpoint_wrapper, apply_activation_checkpointing, CheckpointImpl
)

# 方法一: 自动应用到所有 DecoderLayer
apply_activation_checkpointing(
    model,
    checkpoint_wrapper_fn=functools.partial(
        checkpoint_wrapper,
        checkpoint_impl=CheckpointImpl.NO_REENTRANT
    ),
    check_fn=lambda m: isinstance(m, LlamaDecoderLayer)
)

# 方法二: HuggingFace Transformers
from transformers import AutoModelForCausalLM
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b")
model.gradient_checkpointing_enable(
    gradient_checkpointing_kwargs={"use_reentrant": False}
)
```


> **Warning:** **注意事项：**
>
> - Checkpoint 不兼容
>    `torch.autograd.grad()`
>    的某些用法
> - 使用 checkpoint 时，反向传播中的随机数必须与前向一致（设置相同随机种子）
> - Checkpoint 会增加约 33% 的训练时间


## 4. 显存卸载 (Offloading)


显存卸载将暂时不用的数据从 GPU 显存转移到 CPU 内存甚至磁盘，在需要时再加载回来。这是显存不足时的终极手段。


### 4.1 卸载类型


| 卸载目标 | 数据类型 | 传输量 | 速度影响 |
| --- | --- | --- | --- |
| CPU Offload | 参数、梯度、优化器 | 大 | 显著变慢 (2-10x) |
| NVMe Offload | 优化器状态 | 非常大 | 更慢 (10-50x) |
| 激活值卸载 | 中间激活值 | 中等 | 中等变慢 (1.5-3x) |


### 4.2 激活值卸载


```
python
# 激活值卸载到 CPU
from torch.utils.checkpoint import checkpoint

# 方法: 使用 torch.utils.checkpoint 的 CPU 卸载
def checkpoint_with_cpu_offload(module, *args):
    def forward_with_offload(*inputs):
        # 前向: 在 GPU 计算
return module(*inputs)

    def backward_hook(ctx, *grad_outputs):
        # 反向: 将保存的激活值从 CPU 加载回 GPU
pass
return checkpoint(forward_with_offload, *args, use_reentrant=False)

# DeepSpeed Activation Checkpointing + CPU Offload
# 在 ds_config.json 中配置:
# "activation_checkpointing": {
#   "partition_activations": true,
#   "cpu_checkpointing": true,
#   "contiguous_memory_optimization": true
# }
```


## 5. ZeRO-Offload 详解


ZeRO-Offload 是 DeepSpeed 提出的技术，将优化器状态和梯度计算卸载到 CPU，让 GPU 只负责前向和部分反向计算。


### 5.1 数据流


```
ZeRO-Offload 数据流:

GPU                              CPU
─────────────────────     ─────────────────────
│ 模型参数 (FP16)  │     │                   │
│ 激活值          │     │ 优化器状态 (FP32)  │
│                 │     │ Master Weight      │
│ 前向计算        │     │ Adam m, v         │
│       ↓         │     │                   │
│ 反向计算        │     │                   │
│       ↓         │     │                   │
│ 梯度 (FP16)     │────►│ 梯度累加          │
│                 │传输 │       ↓           │
│                 │◄────│ 参数更新          │
│ 参数更新后      │传输 │ (CPU上的Adam更新)  │
│                 │     │                   │
─────────────────────     ─────────────────────

通信瓶颈: GPU↔CPU 通过 PCIe (32 GB/s)
实际加速: 通过重叠通信与计算部分缓解
```


### 5.3 适用场景


> **Tip:** - **单卡超大模型：**
>    单卡无法放下模型 + 优化器时使用
> - **推理：**
>    不需要优化器时，仅参数卸载开销较小
> - **非延迟敏感：**
>    训练速度不是首要目标时


## 6. 显存估算公式


在训练前准确估算显存需求，可以帮助选择合适的并行策略和显存优化方案。


### 6.1 参数和梯度显存


$$
参数显存 = Ψ × bytes_per_param
                梯度显存 = Ψ × bytes_per_param
                其中: Ψ = 总参数量
$$


### 6.2 优化器状态显存


$$
Adam: 优化器显存 = Ψ × 12 bytes
                     (FP32 Master: 4 + m: 4 + v: 4)

                SGD+Momentum: 优化器显存 = Ψ × 4 bytes
                             (FP32 momentum)

                8-bit Adam (bitsandbytes): 优化器显存 = Ψ × 6 bytes
                                          (FP32 Master: 4 + 量化m: 1 + 量化v: 1)
$$


### 6.3 激活值显存（Transformer）


$$
单层激活 ≈ batch × seq_len × hidden × (10~20) bytes
                总激活 ≈ L × batch × seq_len × hidden × factor
$$


其中 factor 取决于注意力头数、中间维度等。精确估算需要考虑：


- 注意力矩阵: batch × heads × seq_len × seq_len × 2 bytes
- FFN 中间激活: batch × seq_len × intermediate × 2 bytes
- LayerNorm 输出: batch × seq_len × hidden × 2 bytes


### 6.4 完整估算示例


```
python
def estimate_memory(
    num_params,           # 模型参数量 (如 7B)
    batch_size,           # 全局 batch size
    seq_length,           # 序列长度
    hidden_size,          # 隐藏层维度
    num_layers,           # Transformer 层数
    num_heads,            # 注意力头数
    intermediate_size,    # FFN 中间维度
    dtype="bf16",
    optimizer="adam"
):
    bytes_param = 2 if dtype in ["bf16", "fp16"] else 4
# 1. 参数 + 梯度
    param_mem = num_params * bytes_param * 2 # 参数 + 梯度
# 2. 优化器状态
if optimizer == "adam":
        opt_mem = num_params * 12 # FP32 master + m + v
elif optimizer == "8bit_adam":
        opt_mem = num_params * 6
# 3. 激活值 (粗略估算)
# 注意力矩阵
    attn_mem = batch_size * num_heads * seq_length * seq_length * 2 * num_layers
    # FFN 中间激活
    ffn_mem = batch_size * seq_length * intermediate_size * 2 * num_layers
    # 其他激活
    other_act = batch_size * seq_length * hidden_size * 2 * num_layers * 4
    act_mem = attn_mem + ffn_mem + other_act

    total_gb = (param_mem + opt_mem + act_mem) / (1024**3)

    print(f"参数+梯度: {param_mem/1024**3:.1f} GB")
    print(f"优化器:    {opt_mem/1024**3:.1f} GB")
    print(f"激活值:    {act_mem/1024**3:.1f} GB")
    print(f"总计:      {total_gb:.1f} GB")
    return total_gb
```


## 7. 实战调优策略


### 7.1 显存优化技术组合


| 技术 | 显存节省 | 速度影响 | 实现复杂度 |
| --- | --- | --- | --- |
| 混合精度 | 30-40% | 加速 1.5-2x | 低 |
| Gradient Checkpointing | 激活值 60-80% | 减速 33% | 低 |
| ZeRO Stage 1 | 优化器 75% | 几乎无影响 | 低 |
| ZeRO Stage 2 | 优化器+梯度 85% | 轻微减速 | 低 |
| ZeRO Stage 3 | 参数+梯度+优化器 ~95% | 减速 20-50% | 中 |
| CPU Offload | 接近 100% | 减速 2-10x | 中 |
| 梯度累积 | 等效增大 batch | 无影响 | 低 |


### 7.2 推荐组合方案


> **Tip:** **方案一 (推荐):**
> BF16 + Gradient Checkpointing + ZeRO-2
>
>
> 适用于大多数场景，显存节省显著且速度影响可控。
>
>
>
>
> **方案二 (显存紧张):**
> BF16 + Gradient Checkpointing + ZeRO-3
>
>
> 适用于模型较大、GPU 数量充足的场景。
>
>
>
>
> **方案三 (极端场景):**
> BF16 + Checkpointing + ZeRO-3 + CPU Offload
>
>
> 适用于 GPU 数量有限的超大模型训练，但速度会显著下降。
>
>
>
>
> **方案四 (推理优化):**
> BF16 + 8-bit 量化 + CPU Offload
>
>
> 适用于大模型推理，显存需求降到最低。
