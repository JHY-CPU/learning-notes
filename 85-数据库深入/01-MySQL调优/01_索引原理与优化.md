# 索引原理与优化 - MySQL调优

*深入理解 B+ 树索引结构、聚簇与二级索引、覆盖索引、联合索引最左前缀及索引下推等核心概念*

B+ 树 vs B 树 vs 哈希索引

| 特性 | B+ 树 | B 树 | 哈希索引 |
| --- | --- | --- | --- |
| 数据存储位置 | 仅叶子节点存数据 | 所有节点均存数据 | 桶中存储 |
| 范围查询 | 支持（叶子链表） | 效率低 | 不支持 |
| 排序能力 | 天然有序 | 天然有序 | 无序 |
| 磁盘 I/O | 更少（非叶节点小） | 较多 | 最少（O(1)） |
| 适用场景 | 范围/排序/等值 | 较少使用 | 仅等值查询 |

三层 B+ 树可存储的数据量估算

```
假设：
  - 主键类型：bigint（8 字节）
  - 指针大小：6 字节
  - 页大小：16KB = 16384 字节
  - 每行数据大小：约 1KB

根节点可容纳键值数 = 16384 / (8 + 6) ≈ 1170
第二层可容纳键值数 = 1170（每个第二层节点同样）
叶子节点可容纳行数 = 16384 / 1024 ≈ 16 行

三层 B+ 树总容量 = 1170 × 1170 × 16 ≈ 2190 万行
四层 B+ 树总容量 = 1170 × 1170 × 1170 × 16 ≈ 256 亿行
```

聚簇索引的选择规则

1. 如果有主键（PRIMARY KEY），则主键索引就是聚簇索引
2. 如果没有主键但有非空唯一索引（UNIQUE NOT NULL），则选择第一个这样的索引
3. 如果两者都没有，InnoDB 会隐式生成一个 6 字节的
   `_rowid`
   作为聚簇索引

覆盖索引判断方法

通过 EXPLAIN 查看 `Extra` 字段：`Using index` 表示使用了覆盖索引。


```
EXPLAIN SELECT id, name, age FROM users WHERE name = 'Bob';

-- 如果存在联合索引 idx_name_age(name, age)
-- Extra 列显示: Using index  ← 覆盖索引，无需回表
```

idx(a, b, c) 各种查询条件的索引命中情况

| 查询条件 | 命中索引 | 说明 |
| --- | --- | --- |
| WHERE a = 1 | 完全命中 | 命中索引第一列 a |
| WHERE a = 1 AND b = 2 | 完全命中 | 命中 a、b 两列 |
| WHERE a = 1 AND b = 2 AND c = 3 | 完全命中 | 命中全部三列 |
| WHERE b = 2 | 不命中 | 缺少最左列 a |
| WHERE b = 2 AND c = 3 | 不命中 | 缺少最左列 a |
| WHERE a = 1 AND c = 3 | 部分命中 | 命中 a，c 无法用索引 |
| WHERE a > 1 AND b = 2 | 部分命中 | a 范围查询后 b 无法用索引 |
| WHERE a = 1 AND b > 2 AND c = 3 | 部分命中 | 命中 a、b，c 因 b 范围而失效 |

索引失效的典型情况

| 场景 | 示例 | 原因 |
| --- | --- | --- |
| 对索引列使用函数 | `WHERE YEAR(create_time) = 2024` | 破坏索引有序性 |
| 隐式类型转换 | `WHERE varchar_col = 123` | 触发全表扫描 |
| LIKE 左模糊 | `WHERE name LIKE '%明'` | 无法使用 B+ 树有序性 |
| OR 条件不全有索引 | `WHERE a=1 OR b=2`（b 无索引） | 退化为全表扫描 |
| NOT IN / NOT EXISTS | `WHERE id NOT IN (...)` | 优化器可能选择全表扫描 |
| 索引列参与计算 | `WHERE id + 1 = 10` | 无法利用 B+ 树查找 |
