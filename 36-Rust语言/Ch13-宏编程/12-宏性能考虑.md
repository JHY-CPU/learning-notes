# 宏性能考虑

## 一、概念说明

宏在编译时展开，会影响编译时间但不影响运行时性能。合理使用宏可以生成更优化的代码。

```rust
// 宏展开增加编译时间
// 但运行时性能等同于手写代码
macro_rules! fast_add {
    ($a:expr, $b:expr) => { $a + $b };
}
```

## 二、具体用法

### 2.1 编译时 vs 运行时

```rust
// 编译时：宏展开
// 生成的代码在编译时优化

macro_rules! compute_at_compile {
    ($n:expr) => {{
        const RESULT: i32 = $n * $n;
        RESULT
    }};
}

let x = compute_at_compile!(5); // 编译时常量25
```

### 2.2 避免过度展开

```rust
// 过度使用宏增加编译时间
// 平衡代码复用和编译性能

// 好：简单的宏
macro_rules! create_vec {
    ($($x:expr),*) => { vec![$($x),*] };
}

// 坏：过度复杂的宏
// 考虑使用函数或 trait 代替
```

### 2.3 基准测试

```rust
// 测试宏生成代码的性能
use criterion::{criterion_group, criterion_main, Criterion};

fn benchmark_macro(c: &mut Criterion) {
    c.bench_function("macro_generated", |b| {
        b.iter(|| {
            // 测试宏生成的代码
            macro_generated_function()
        })
    });
}

criterion_group!(benches, benchmark_macro);
criterion_main!(benches);
```

### 2.5 避免宏膨胀

```rust
// 不好：为每个类型生成大量重复代码
macro_rules! bad_impl {
    ($($t:ty),+) => {
        $(
            impl MyTrait for $t {
                fn method1(&self) { /* 大量代码 */ }
                fn method2(&self) { /* 大量代码 */ }
                fn method3(&self) { /* 大量代码 */ }
                // ... 100 行代码
            }
        )+
    };
}

// 好：提取通用逻辑到函数
macro_rules! good_impl {
    ($($t:ty),+) => {
        $(
            impl MyTrait for $t {
                fn method1(&self) { common_method1(self) }
                fn method2(&self) { common_method2(self) }
                fn method3(&self) { common_method3(self) }
            }
        )+
    };
}

fn common_method1<T: MyTrait>(t: &T) { /* 共享逻辑 */ }
fn common_method2<T: MyTrait>(t: &T) { /* 共享逻辑 */ }
fn common_method3<T: MyTrait>(t: &T) { /* 共享逻辑 */ }
```

### 2.6 使用 const generics 替代宏

```rust
// 以前需要宏来支持不同大小的数组
macro_rules! impl_for_array {
    ($($n:expr),+) => {
        $(
            impl<T> MyTrait for [T; $n] {
                fn len(&self) -> usize { $n }
            }
        )+
    };
}

// 现在使用 const generics
impl<T, const N: usize> MyTrait for [T; N] {
    fn len(&self) -> usize {
        N
    }
}
```

## 四、编译优化选项

```toml
# Cargo.toml
[profile.dev]
# 开发模式：保持快速编译
opt-level = 0
incremental = true

[profile.release]
# 发布模式：优化运行时
opt-level = 3
lto = true
codegen-units = 1

[profile.dev.build-override]
# 优化编译时宏展开
opt-level = 2
```

## 五、注意事项与常见陷阱

1. **编译时间**：大量宏展开增加编译时间，使用增量编译缓解
2. **代码膨胀**：宏生成大量代码可能增加二进制大小，提取共享逻辑
3. **优化机会**：宏生成的代码可能有更好的优化机会（内联等）
4. **缓存影响**：宏变化可能导致更多代码重新编译，减少宏的依赖
5. **测量指导**：用编译时间数据指导宏的使用，定期检查编译性能
6. **const generics**：优先使用 const generics 替代宏处理数组大小
7. **feature flags**：使用 feature flags 控制哪些宏被展开
