# 宏性能考虑

## 一、概念说明

宏在编译时展开，会影响编译时间但不影响运行时性能。合理使用宏可以生成更优化的代码。

```rust
// 宏展开增加编译时间
// 但运行时性能等同于手写代码
macro_rules! fast_add {
    ($a:expr, $b:expr) => { $a + $b };
}
```

## 二、具体用法

### 2.1 编译时 vs 运行时

```rust
// 编译时：宏展开
// 生成的代码在编译时优化

macro_rules! compute_at_compile {
    ($n:expr) => {{
        const RESULT: i32 = $n * $n;
        RESULT
    }};
}

let x = compute_at_compile!(5); // 编译时常量25
```

### 2.2 避免过度展开

```rust
// 过度使用宏增加编译时间
// 平衡代码复用和编译性能

// 好：简单的宏
macro_rules! create_vec {
    ($($x:expr),*) => { vec![$($x),*] };
}

// 坏：过度复杂的宏
// 考虑使用函数或 trait 代替
```

### 2.3 基准测试

```rust
// 测试宏生成代码的性能
use criterion::{criterion_group, criterion_main, Criterion};

fn benchmark_macro(c: &mut Criterion) {
    c.bench_function("macro_generated", |b| {
        b.iter(|| {
            // 测试宏生成的代码
            macro_generated_function()
        })
    });
}

criterion_group!(benches, benchmark_macro);
criterion_main!(benches);
```

## 三、注意事项与常见陷阱

1. **编译时间**：大量宏展开增加编译时间
2. **代码膨胀**：宏生成大量代码可能增加二进制大小
3. **优化机会**：宏生成的代码可能有更好的优化机会
4. **缓存影响**：宏变化可能导致更多代码重新编译
5. **测量指导**：用编译时间数据指导宏的使用
