# 宏最佳实践

## 一、概念说明

总结宏编程的最佳实践，帮助编写高质量、可维护的宏代码。

```rust
// 好：简洁明了的宏
macro_rules! create_vec {
    ($($x:expr),*) => { vec![$($x),*] };
}

// 坏：过于复杂的宏
// 考虑使用函数或 trait 代替
```

## 二、具体用法

### 2.1 设计原则

```rust
// 1. 优先使用函数，复杂场景用宏
// 2. 保持宏简单，单一职责
// 3. 提供清晰的错误消息
// 4. 保持卫生性
// 5. 编写充分的测试
```

### 2.2 文档规范

```rust
/// 创建一个向量。
///
/// # 示例
///
/// ```
/// let v = create_vec![1, 2, 3];
/// assert_eq!(v, vec![1, 2, 3]);
/// ```
///
/// # 宏卫生性
///
/// 此宏是卫生的，不会污染外部作用域。
macro_rules! create_vec {
    ($($x:expr),*) => { vec![$($x),*] };
}
```

### 2.3 错误消息

```rust
macro_rules! validate {
    ($condition:expr, $msg:expr) => {
        if !$condition {
            panic!("验证失败: {}", $msg);
        }
    };
    ($condition:expr) => {
        validate!($condition, stringify!($condition));
    };
}
```

### 2.4 版本兼容

```rust
// 考虑 API 兼容性
// 使用 #[macro_export] 控制可见性
// 使用属性控制宏行为

#[macro_export]
macro_rules! my_macro {
    // 版本1
    ($x:expr) => { /* ... */ };
    // 版本2（带选项）
    ($x:expr, option = $opt:expr) => { /* ... */ };
}
```

### 2.5 宏的命名约定

```rust
// 1. 声明式宏：使用动词或功能名
macro_rules! create_vec { () => {} }
macro_rules! assert_eq { () => {} }

// 2. 派生宏：使用形容词或功能名
// #[derive(Debug, Clone, Serialize)]

// 3. 属性宏：使用动词或功能名
// #[instrument]
// #[tokio::main]

// 4. 函数宏：使用动词或功能名
// println!, format!, vec!

// 命名原则：
// - 清晰表达功能
// - 避免与标准库冲突
// - 保持一致的命名风格
```

### 2.6 宏的测试策略

```rust
#[cfg(test)]
mod macro_tests {
    // 测试正常情况
    #[test]
    fn test_normal_input() {
        let result = my_macro!(valid_input);
        assert_eq!(result, expected);
    }

    // 测试边界情况
    #[test]
    fn test_edge_cases() {
        let empty = my_macro!();
        let single = my_macro!(one);
        let multiple = my_macro!(one, two, three);
    }

    // 使用 trybuild 测试编译错误
    #[test]
    fn test_compile_errors() {
        let t = trybuild::TestCases::new();
        t.compile_fail("tests/fail/*.rs");
    }
}
```

### 2.7 宏的演进策略

```rust
// 版本化 API
#[macro_export]
macro_rules! my_macro_v1 {
    ($x:expr) => { /* v1 实现 */ };
}

#[macro_export]
macro_rules! my_macro_v2 {
    ($x:expr) => { /* v2 实现 */ };
    ($x:expr, option = $opt:expr) => { /* v2 新功能 */ };
}

// 向后兼容：保留旧版本
// 清晰文档：说明版本差异和迁移指南

// 弃用策略
#[deprecated(note = "请使用 my_macro_v2")]
#[macro_export]
macro_rules! my_macro_old {
    ($x:expr) => { my_macro_v2!($x) };
}
```

## 四、宏设计决策树

```
需要代码复用？
├── 否 → 使用函数/方法
├── 是
│   ├── 需要可变参数？
│   │   ├── 是 → 使用宏
│   │   └── 否 → 使用泛型
│   ├── 需要编译时代码生成？
│   │   ├── 简单模式匹配 → macro_rules!
│   │   └── 复杂 AST 处理 → 过程宏
│   └── 需要语法扩展？
│       ├── 自定义属性 → 属性宏
│       ├── 函数调用语法 → 函数宏
│       └── 自动实现 trait → 派生宏
```

## 五、注意事项与常见陷阱

1. **适度使用**：不是所有问题都需要宏解决，函数和泛型通常更简单
2. **性能平衡**：编译时 vs 运行时的权衡，宏增加编译时间但优化运行时
3. **维护成本**：宏代码更难维护，需要充分的文档和测试
4. **团队协作**：确保团队理解宏的用法，提供清晰的示例和文档
5. **持续改进**：根据使用反馈改进宏，收集用户意见
6. **向后兼容**：宏 API 的变更需要谨慎，避免破坏现有代码
7. **错误处理**：宏的错误消息应清晰有用，帮助用户快速定位问题
