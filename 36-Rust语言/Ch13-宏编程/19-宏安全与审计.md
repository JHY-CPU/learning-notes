# 宏安全与审计

## 一、概念说明

宏代码的安全审计需要特别注意，因为宏可以生成任意代码。

```rust
// 宏可能隐藏危险操作
macro_rules! dangerous {
    () => {
        unsafe { /* 危险代码 */ }
    };
}
```

### 2.2 常见安全问题

```rust
// 问题1：未检查的 unsafe
macro_rules! unchecked {
    ($ptr:expr) => {
        unsafe { *$ptr } // 可能解引用空指针
    };
}

// 解决方案：添加安全检查
macro_rules! safe_unwrap {
    ($ptr:expr) => {
        unsafe {
            if $ptr.is_null() {
                panic!("空指针");
            }
            *$ptr
        }
    };
}
```

### 2.4 常见安全漏洞模式

```rust
// 漏洞1: 注入攻击
// 不安全：直接将字符串插入代码
macro_rules! unsafe_sql {
    ($query:expr) => {
        format!("SELECT * FROM users WHERE name = '{}'", $query)
    };
}

// 安全：使用参数化查询
macro_rules! safe_sql {
    ($query:expr, $param:expr) => {
        ("SELECT * FROM users WHERE name = ?", $param)
    };
}

// 漏洞2: 格式化字符串攻击
macro_rules! unsafe_format {
    ($user_input:expr) => {
        // 如果 $user_input 包含 {} 或 {:?}，可能导致问题
        println!($user_input);
    };
}

// 安全：使用格式化占位符
macro_rules! safe_format {
    ($user_input:expr) => {
        println!("{}", $user_input);
    };
}
```

### 2.6 使用 unsafe 的安全模式

```rust
// 不安全的模式
macro_rules! bad_unsafe {
    ($ptr:expr) => {
        unsafe { *$ptr } // 可能解引用空指针或悬垂指针
    };
}

// 安全的模式
macro_rules! safe_unwrap {
    ($ptr:expr) => {{
        let ptr = $ptr;
        if ptr.is_null() {
            panic!("空指针解引用");
        }
        // 检查对齐
        if (ptr as usize) % std::mem::align_of::<_>() != 0 {
            panic!("指针未对齐");
        }
        unsafe { &*ptr }
    }};
}
```

## 四、安全工具矩阵

| 工具 | 检测类型 | 使用方式 |
|------|---------|---------|
| cargo audit | 依赖漏洞 | `cargo audit` |
| cargo deny | 许可证/安全策略 | `cargo deny check` |
| clippy | 代码质量 | `cargo clippy` |
| miri | 未定义行为 | `cargo +nightly miri test` |
| semver-checks | API 兼容性 | `cargo semver-checks` |
| unsafe-review | unsafe 代码审查 | `cargo +nightly unsafe-review` |

## 五、注意事项与常见陷阱

1. **代码审查**：宏代码需要严格审查，特别关注 unsafe 代码
2. **安全测试**：为宏编写安全测试，包括边界条件和恶意输入
3. **依赖管理**：定期更新依赖，使用 cargo audit 检查漏洞
4. **文档透明**：文档中说明宏的安全性，是否包含 unsafe
5. **最小权限**：宏只应使用必要的权限，遵循最小权限原则
6. **供应链安全**：验证依赖的来源和完整性
7. **安全更新流程**：建立安全漏洞响应和更新流程
