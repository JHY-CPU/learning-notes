# 集合性能对比

## 一、概念说明

选择合适的集合类型对程序性能至关重要。不同集合在插入、删除、查找、遍历等操作上有不同的时间复杂度和内存特性。

```rust
// 性能对比参考（平均情况）
// Vec:        随机访问O(1), 尾部插入O(1), 搜索O(n)
// VecDeque:   两端操作O(1), 随机访问O(n)
// LinkedList: 两端操作O(1), 随机访问O(n)
// HashMap:    插入/查找/删除 O(1)
// BTreeMap:   插入/查找/删除 O(log n)
```

## 二、具体用法

### 2.1 基准测试示例

```rust
use std::collections::{HashMap, BTreeMap, HashSet};
use std::time::Instant;

fn benchmark_vec_insert(n: usize) {
    let start = Instant::now();
    let mut v = Vec::with_capacity(n);
    for i in 0..n {
        v.push(i);
    }
    println!("Vec插入{}: {:?}", n, start.elapsed());
}

fn benchmark_hashmap_insert(n: usize) {
    let start = Instant::now();
    let mut map = HashMap::with_capacity(n);
    for i in 0..n {
        map.insert(i, i);
    }
    println!("HashMap插入{}: {:?}", n, start.elapsed());
}

fn benchmark_btreemap_insert(n: usize) {
    let start = Instant::now();
    let mut map = BTreeMap::new();
    for i in 0..n {
        map.insert(i, i);
    }
    println!("BTreeMap插入{}: {:?}", n, start.elapsed());
}
```

### 2.2 按场景选择集合

```rust
// 场景1: 需要快速随机访问 → Vec
fn process_scores(scores: &[f64]) -> f64 {
    scores.iter().sum::<f64>() / scores.len() as f64
}

// 场景2: 需要键值查找 → HashMap
fn lookup_user(id: u32, users: &HashMap<u32, String>) -> Option<&String> {
    users.get(&id)
}

// 场景3: 需要有序遍历 → BTreeMap
fn print_sorted(data: &BTreeMap<String, i32>) {
    for (key, val) in data {
        println!("{}: {}", key, val);
    }
}

// 场景4: 需要去重 → HashSet
fn unique_words(text: &str) -> HashSet<&str> {
    text.split_whitespace().collect()
}

// 场景5: 需要队列 → VecDeque
fn process_queue(queue: &mut VecDeque<String>) {
    while let Some(item) = queue.pop_front() {
        println!("处理: {}", item);
    }
}
```

### 2.3 缓存友好性分析

```rust
// Vec: 内存连续，缓存命中率最高
// LinkedList: 节点分散，缓存命中率最低
// HashMap: 哈希桶分散，中等缓存性能
// BTreeMap: 节点相对集中，缓存性能较好

fn cache_friendly_sum(data: &[i32]) -> i32 {
    data.iter().sum() // 顺序遍历，缓存友好
}
```

## 三、注意事项与常见陷阱

1. **选择原则**：先用 Vec，除非有明确需求；Vec 通常是默认选择
2. **容量预分配**：已知大小时使用 `with_capacity` 减少重分配
3. **小数据量**：小集合(<100) 时线性搜索可能比哈希更快
4. **字符串键**：短字符串 HashMap 性能好，长字符串考虑 BTreeMap
5. **内存碎片**：频繁插入删除大量数据可能导致内存碎片
