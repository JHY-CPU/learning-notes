# 内存模型与屏障

## 一、概念说明

Rust 的内存模型基于 C++11 的内存模型，提供多种内存序选项控制指令重排和缓存同步。

```rust
use std::sync::atomic::{Ordering, fence, AtomicBool, AtomicI32};

// 内存屏障保证操作顺序
fence(Ordering::Release);
```

## 二、具体用法

### 2.1 内存序详解

```rust
use std::sync::atomic::Ordering;

// Relaxed: 只保证原子性，不保证顺序
// 适用于计数器等简单场景

// Acquire: 读操作的获取语义
// 保证之后的操作不会重排到此之前

// Release: 写操作的释放语义
// 保证之前的操作不会重排到此之后

// AcqRel: 同时具有 Acquire 和 Release 语义

// SeqCst: 完全顺序一致性
// 所有线程看到相同的操作顺序
```

### 2.2 实际应用

```rust
use std::sync::atomic::{AtomicBool, AtomicI32, Ordering, fence};
use std::thread;

fn memory_order_example() {
    let data = AtomicI32::new(0);
    let ready = AtomicBool::new(false);

    // 生产者线程
    let producer = thread::spawn(move || {
        data.store(42, Ordering::Relaxed);
        fence(Ordering::Release); // 保证 data 写入在 ready 之前可见
        ready.store(true, Ordering::Relaxed);
    });

    // 消费者线程
    let consumer = thread::spawn(move || {
        while !ready.load(Ordering::Relaxed) {
            thread::yield_now();
        }
        fence(Ordering::Acquire); // 保证能看到 data 的写入
        data.load(Ordering::Relaxed)
    });

    producer.join().unwrap();
    let result = consumer.join().unwrap();
}
```

### 2.3 自旋锁实现示例

```rust
use std::sync::atomic::{AtomicBool, Ordering, fence};
use std::cell::UnsafeCell;

struct SpinLock<T> {
    locked: AtomicBool,
    data: UnsafeCell<T>,
}

impl<T> SpinLock<T> {
    fn new(data: T) -> Self {
        SpinLock {
            locked: AtomicBool::new(false),
            data: UnsafeCell::new(data),
        }
    }

    fn lock(&self) -> SpinLockGuard<T> {
        // 自旋等待获取锁
        while self.locked.compare_exchange_weak(
            false, true,
            Ordering::Acquire,
            Ordering::Relaxed,
        ).is_err() {
            // 忙等待（实际中应使用 std::hint::spin_loop()）
            std::hint::spin_loop();
        }
        SpinLockGuard { lock: self }
    }
}

struct SpinLockGuard<'a, T> {
    lock: &'a SpinLock<T>,
}

impl<T> std::ops::Deref for SpinLockGuard<'_, T> {
    type Target = T;
    fn deref(&self) -> &T {
        unsafe { &*self.lock.data.get() }
    }
}

impl<T> std::ops::DerefMut for SpinLockGuard<'_, T> {
    fn deref_mut(&mut self) -> &mut T {
        unsafe { &mut *self.lock.data.get() }
    }
}

impl<T> Drop for SpinLockGuard<'_, T> {
    fn drop(&mut self) {
        // Release 确保之前的修改对其他线程可见
        self.lock.locked.store(false, Ordering::Release);
    }
}

// SpinLock 需要 Sync 标记才能跨线程
unsafe impl<T: Send> Sync for SpinLock<T> {}
```

### 2.4 原子操作的实际应用

```rust
use std::sync::atomic::{AtomicU64, Ordering};
use std::thread;
use std::time::Instant;

fn atomic_timestamp() {
    // 高精度时间戳（原子操作）
    static LAST_UPDATE: AtomicU64 = AtomicU64::new(0);

    fn update_timestamp() -> u64 {
        let now = Instant::now().elapsed().as_nanos() as u64;
        LAST_UPDATE.fetch_max(now, Ordering::Release);
        now
    }

    let handles: Vec<_> = (0..4).map(|_| {
        thread::spawn(|| {
            for _ in 0..1000 {
                update_timestamp();
            }
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
    println!("最后更新: {}", LAST_UPDATE.load(Ordering::Acquire));
}
```

### 2.5 Double-Checked Locking

```rust
use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::{Mutex, OnceLock};

static INITIALIZED: AtomicBool = AtomicBool::new(false);
static DATA: OnceLock<String> = OnceLock::new();

fn get_or_init() -> &'static String {
    // 快速路径：已初始化则直接返回
    if INITIALIZED.load(Ordering::Acquire) {
        return DATA.get().unwrap();
    }

    // 慢速路径：初始化
    let data = DATA.get_or_init(|| {
        // 昂贵的初始化操作
        "初始化完成".to_string()
    });

    INITIALIZED.store(true, Ordering::Release);
    data
}
```

### 2.6 SeqCst vs 较弱顺序的权衡

```rust
use std::sync::atomic::{AtomicBool, AtomicUsize, Ordering};
use std::thread;

fn demonstrate_ordering_difference() {
    let a = AtomicBool::new(false);
    let b = AtomicBool::new(false);
    let x = AtomicUsize::new(0);
    let y = AtomicUsize::new(0);

    // SeqCst：所有线程看到相同的全局顺序
    // 可以保证：如果 r1 == 0 && r2 == 0，不会发生

    // Relaxed：只保证原子性，不保证顺序
    // 可能发生：r1 == 0 && r2 == 0（在某些架构上）

    // 在实际代码中：
    // - 计数器/统计 → Relaxed
    // - 发布-订阅 → Acquire/Release
    // - 全局标志 → SeqCst
}
```

## 四、内存序速查表

| 内存序 | 强度 | 典型用途 | 性能 |
|--------|------|---------|------|
| Relaxed | 最弱 | 计数器、统计 | 最快 |
| Acquire | 中等 | 读锁、状态检查 | 快 |
| Release | 中等 | 写锁、发布数据 | 快 |
| AcqRel | 较强 | 读-修改-写 | 中等 |
| SeqCst | 最强 | 全局同步 | 最慢 |

## 五、注意事项与常见陷阱

1. **默认选择**：不确定时使用 SeqCst，先保证正确性
2. **性能影响**：SeqCst 性能开销最大，能用较弱顺序就用
3. **正确性优先**：先保证正确，再优化性能，不要过早优化
4. **验证困难**：内存序错误难以测试和调试，使用 loom 或 formal verification
5. **学习曲线**：内存模型理解需要深入学习，建议阅读 "Rust Atomics and Locks"
6. **跨平台差异**：不同 CPU 架构对弱内存序的支持不同
7. **fence 使用**：fence 比直接在原子操作上指定内存序更灵活
