# 从并发到并行

## 一、概念说明

并发（Concurrency）是同时处理多个任务的能力，并行（Parallelism）是同时执行多个任务。Rust 同时支持这两种模式。

```rust
// 并发：async/await, tokio
// 并行：rayon, std::thread

// 区别：
// 并发关注任务调度
// 并行关注任务执行
```

## 二、具体用法

### 2.1 I/O 密集型（并发）

```rust
use tokio;

#[tokio::main]
async fn io_concurrent() {
    let mut handles = vec![];

    for i in 0..10 {
        handles.push(tokio::spawn(async move {
            // 模拟 I/O 操作
            tokio::time::sleep(std::time::Duration::from_millis(100)).await;
            format!("任务 {} 完成", i)
        }));
    }

    for handle in handles {
        println!("{}", handle.await.unwrap());
    }
}
```

### 2.2 CPU 密集型（并行）

```rust
use rayon::prelude::*;

fn cpu_parallel() {
    let data: Vec<i64> = (0..10_000_000).collect();

    let sum: i64 = data.par_iter()
        .map(|x| {
            // CPU 密集计算
            let mut result = *x;
            for _ in 0..100 {
                result = result.wrapping_mul(17).wrapping_add(31);
            }
            result
        })
        .sum();

    println!("结果: {}", sum);
}
```

### 2.3 混合模式

```rust
async fn mixed_workload() {
    // I/O 部分使用 async
    let data = fetch_data().await;

    // CPU 部分使用 rayon
    let processed: Vec<_> = data.par_iter()
        .map(|x| heavy_computation(x))
        .collect();

    // 结果写回
    save_results(&processed).await;
}
```

### 2.4 异步与并行混合模式

```rust
async fn hybrid_workload() -> Vec<String> {
    // 第一阶段：并行 I/O
    let urls = vec![
        "https://api.example.com/data1",
        "https://api.example.com/data2",
        "https://api.example.com/data3",
    ];

    let fetches: Vec<_> = urls.into_iter()
        .map(|url| {
            tokio::spawn(async move {
                reqwest::get(url).await.unwrap().text().await.unwrap()
            })
        })
        .collect();

    let mut raw_data = Vec::new();
    for fetch in fetches {
        raw_data.push(fetch.await.unwrap());
    }

    // 第二阶段：并行 CPU 处理
    let processed: Vec<String> = raw_data.par_iter()
        .map(|data| {
            // CPU 密集型处理
            heavy_processing(data)
        })
        .collect();

    // 第三阶段：并行 I/O 写回
    let writes: Vec<_> = processed.into_iter()
        .map(|result| {
            tokio::spawn(async move {
                save_to_db(&result).await;
                result
            })
        })
        .collect();

    let mut final_results = Vec::new();
    for write in writes {
        final_results.push(write.await.unwrap());
    }

    final_results
}

fn heavy_processing(data: &str) -> String {
    // 模拟 CPU 密集型计算
    data.chars().rev().collect()
}

async fn save_to_db(data: &str) {
    // 模拟数据库保存
    tokio::time::sleep(std::time::Duration::from_millis(10)).await;
}
```

### 2.5 异步运行时中的阻塞处理

```rust
// 不要在 async 中执行阻塞操作！
async fn bad_example() {
    // 错误：阻塞整个运行时
    // std::thread::sleep(std::time::Duration::from_secs(1));
}

async fn good_example() {
    // 正确：使用 spawn_blocking
    tokio::task::spawn_blocking(|| {
        std::thread::sleep(std::time::Duration::from_secs(1));
    }).await.unwrap();
}

// 或使用 rayon 处理 CPU 密集任务
async fn cpu_intensive_with_rayon(data: Vec<i64>) -> i64 {
    use rayon::prelude::*;

    tokio::task::spawn_blocking(move || {
        data.par_iter()
            .map(|&x| {
                let mut result = x;
                for _ in 0..1000 {
                    result = result.wrapping_mul(17).wrapping_add(31);
                }
                result
            })
            .sum()
    }).await.unwrap()
}
```

### 2.6 Channel 在并发/并行间的桥梁

```rust
use tokio::sync::mpsc;
use rayon::prelude::*;

async fn channel_bridge() {
    let (tx, mut rx) = mpsc::channel(100);

    // 异步任务：发送数据到 channel
    tokio::spawn(async move {
        for i in 0..1000 {
            tx.send(i).await.unwrap();
        }
    });

    // 收集数据
    let mut data = Vec::new();
    while let Some(item) = rx.recv().await {
        data.push(item);
    }

    // 并行处理收集到的数据
    let result: i64 = data.par_iter()
        .map(|&x| x as i64 * x as i64)
        .sum();

    println!("结果: {}", result);
}
```

### 2.7 性能对比与选择

```rust
fn choose_concurrency_model() -> &'static str {
    // 决策矩阵
    // 任务类型          推荐模型
    // ─────────────────────────
    // I/O 密集型        async/await (tokio)
    // CPU 密集型        rayon 数据并行
    // 混合型            async + spawn_blocking
    // 简单并行          std::thread
    // 流处理            async stream + rayon

    // 一般原则：
    // 1. I/O 密集 → async
    // 2. CPU 密集 → 并行
    // 3. 大规模数据 → rayon
    // 4. 实时响应 → 多线程
    "根据具体场景选择"
}
```

## 四、模型选择决策流程图

```
开始
├── 需要同时处理多个任务？
│   ├── 是 → 继续
│   └── 否 → 单线程即可
├── 主要是什么类型的等待？
│   ├── I/O（网络/文件）→ async/await
│   ├── CPU 计算 → rayon 或 std::thread
│   └── 混合 → async + spawn_blocking
├── 数据规模？
│   ├── 小数据（<1000）→ 单线程
│   ├── 中等 → rayon par_iter
│   └── 大规模 → 自定义线程池
└── 需要实时响应？
    ├── 是 → 专用线程 + channel
    └── 否 → 标准方案
```

## 五、注意事项与常见陷阱

1. **选择正确模式**：I/O 密集用 async，CPU 密集用并行，选错会严重影响性能
2. **避免阻塞**：不要在 async 上下文中执行阻塞操作，使用 spawn_blocking
3. **混合使用**：复杂系统可能同时需要并发和并行，合理分层设计
4. **资源限制**：控制并发度避免资源耗尽，使用信号量限制并发数
5. **性能监控**：持续监控并发/并行的实际效果，使用 metrics 和 tracing
6. **调试策略**：不同模型的调试方式不同，选择合适的调试工具
7. **迁移成本**：从一种模型迁移到另一种成本较高，初始设计时慎重选择
