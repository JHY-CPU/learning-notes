# 并发性能调优

## 一、概念说明

并发程序的性能受锁竞争、缓存一致性、线程切换等因素影响。合理调优可以显著提升性能。

## 二、具体用法

### 2.1 锁优化

```rust
// 锁分片
struct ShardedMap<T> {
    shards: Vec<std::sync::Mutex<std::collections::HashMap<String, T>>>,
}

impl<T> ShardedMap<T> {
    fn new(shards: usize) -> Self {
        ShardedMap {
            shards: (0..shards)
                .map(|_| std::sync::Mutex::new(std::collections::HashMap::new()))
                .collect(),
        }
    }

    fn get_shard(&self, key: &str) -> &std::sync::Mutex<std::collections::HashMap<String, T>> {
        use std::collections::hash_map::DefaultHasher;
        use std::hash::{Hash, Hasher};
        let mut hasher = DefaultHasher::new();
        key.hash(&mut hasher);
        let index = (hasher.finish() as usize) % self.shards.len();
        &self.shards[index]
    }
}
```

### 2.2 读写分离

```rust
use std::sync::RwLock;

struct Cache {
    data: RwLock<std::collections::HashMap<String, Vec<u8>>>,
}

impl Cache {
    // 读多写少场景
    fn get(&self, key: &str) -> Option<Vec<u8>> {
        self.data.read().unwrap().get(key).cloned()
    }

    fn set(&self, key: String, value: Vec<u8>) {
        self.data.write().unwrap().insert(key, value);
    }
}
```

### 2.3 无锁数据结构

```rust
use std::sync::atomic::{AtomicUsize, Ordering};

struct LockFreeStack<T> {
    head: AtomicUsize,
    _marker: std::marker::PhantomData<T>,
}

// 实际的无锁栈需要更复杂的 ABA 保护
// 这里展示基本思路

fn use_atomic_counters() {
    // AtomicUsize 比 Mutex<i32> 快 10-100 倍
    use std::sync::atomic::AtomicUsize;
    use std::thread;

    let counter = AtomicUsize::new(0);

    let handles: Vec<_> = (0..8).map(|_| {
        let c = &counter;
        thread::spawn(move || {
            for _ in 0..1_000_000 {
                c.fetch_add(1, Ordering::Relaxed);
            }
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
    println!("计数: {}", counter.load(Ordering::Relaxed));
}
```

### 2.4 Work Stealing 调度

```rust
// Rayon 使用 work-stealing 调度器
// 每个线程有自己的任务队列，空闲时从其他线程偷取任务

use rayon::prelude::*;

fn work_stealing_example() {
    let data: Vec<i64> = (0..10_000_000).collect();

    // Rayon 自动平衡工作负载
    let sum: i64 = data.par_iter()
        .map(|&x| {
            // 不均匀的计算量
            if x % 1000 == 0 {
                // 偶尔有较重的计算
                (0..1000).fold(x, |acc, _| acc.wrapping_mul(17))
            } else {
                x
            }
        })
        .sum();
}
```

### 2.5 缓存行优化

```rust
use std::sync::atomic::{AtomicUsize, Ordering};
use std::thread;

// 避免 false sharing：将频繁访问的变量放在不同缓存行
#[repr(align(64))] // 缓存行大小通常是 64 字节
struct CacheLine<T>(T);

fn avoid_false_sharing() {
    // 不好：相邻变量在同一缓存行
    // let counters = [AtomicUsize::new(0); 8];

    // 好：每个计数器独占一个缓存行
    let counters: Vec<CacheLine<AtomicUsize>> = (0..8)
        .map(|_| CacheLine(AtomicUsize::new(0)))
        .collect();

    let handles: Vec<_> = (0..8).map(|i| {
        let counter = unsafe {
            &*(&counters[i] as *const CacheLine<AtomicUsize>)
        };
        thread::spawn(move || {
            for _ in 0..1_000_000 {
                counter.0.fetch_add(1, Ordering::Relaxed);
            }
        })
    }).collect();

    for h in handles { h.join().unwrap(); }
}
```

### 2.6 批量处理减少锁竞争

```rust
use std::sync::{Arc, Mutex};

struct BatchProcessor {
    buffer: Arc<Mutex<Vec<String>>>,
    batch_size: usize,
}

impl BatchProcessor {
    fn new(batch_size: usize) -> Self {
        BatchProcessor {
            buffer: Arc::new(Mutex::new(Vec::with_capacity(batch_size))),
            batch_size,
        }
    }

    fn push(&self, item: String) {
        let mut buf = self.buffer.lock().unwrap();
        buf.push(item);

        if buf.len() >= self.batch_size {
            let batch: Vec<String> = buf.drain(..).collect();
            // 处理整个批次
            self.process_batch(batch);
        }
    }

    fn process_batch(&self, batch: Vec<String>) {
        // 批量处理，减少锁获取次数
        println!("处理批次: {} 项", batch.len());
    }
}
```

## 五、注意事项与常见陷阱

1. **先测量**：用 benchmark 数据指导优化，不要凭直觉
2. **瓶颈分析**：找到真正的瓶颈再优化，可能在 I/O 而非 CPU
3. **可读性**：过度优化可能降低代码可维护性，平衡性能与可读性
4. **Amdahl 定律**：串行部分限制并行加速比，优化串行路径
5. **硬件因素**：考虑 CPU 缓存、NUMA 等硬件特性，缓存行对齐
6. **过早优化**：先写正确代码，再优化热点
7. **平台差异**：不同平台的优化策略可能不同
