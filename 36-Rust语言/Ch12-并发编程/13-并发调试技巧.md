# 并发调试技巧

## 一、概念说明

并发 bug 难以复现和调试。Rust 的编译时检查防止了数据竞争，但逻辑错误仍需借助工具和技巧排查。

```rust
// 使用 RUST_BACKTRACE=1 运行
// 使用 thread sanitizer: -Z sanitizer=thread
// 使用 miri 检测未定义行为
```

## 二、具体用法

### 2.1 Thread Sanitizer

```bash
# 编译时启用 Thread Sanitizer
RUSTFLAGS="-Z sanitizer=thread" cargo run
# 可以检测数据竞争和死锁
```

### 2.4 Loom 并发验证

```rust
// Cargo.toml: loom = "0.7" (仅测试依赖)
#[cfg(test)]
mod loom_tests {
    use loom::sync::{Arc, Mutex};
    use loom::thread;

    #[test]
    fn test_concurrent_increment() {
        loom::model(|| {
            let data = Arc::new(Mutex::new(0));
            let handles: Vec<_> = (0..2).map(|_| {
                let data = Arc::clone(&data);
                thread::spawn(move || {
                    *data.lock().unwrap() += 1;
                })
            }).collect();

            for h in handles { h.join().unwrap(); }
            assert_eq!(*data.lock().unwrap(), 2);
        });
    }
}
```

### 2.5 Backtrace 分析

```rust
fn debug_with_backtrace() {
    // 设置环境变量
    // RUST_BACKTRACE=1 cargo run
    // RUST_BACKTRACE=full cargo run  // 完整 backtrace

    // 在代码中打印 backtrace
    let trace = std::backtrace::Backtrace::capture();
    eprintln!("Backtrace:\n{}", trace);

    // 在 panic hook 中记录
    std::panic::set_hook(Box::new(|info| {
        let backtrace = std::backtrace::Backtrace::capture();
        eprintln!("Panic: {}\nBacktrace:\n{}", info, backtrace);
    }));
}
```

### 2.6 交叉编译调试

```rust
// 使用 MIRI 检测未定义行为
// cargo +nightly miri test

// MIRI 可以检测：
// - 数据竞争
// - 未初始化内存访问
// - 悬垂指针
// - 内存泄漏
// - 未定义行为（在 unsafe 代码中）

fn miri_detectable_issue() {
    use std::sync::atomic::{AtomicUsize, Ordering};
    use std::thread;

    let a = AtomicUsize::new(0);

    // 不安全：使用非原子操作
    // let val = unsafe { std::ptr::read_volatile(a.as_ptr()) };

    // 安全：使用原子操作
    let val = a.load(Ordering::SeqCst);
}
```

### 2.7 压力测试策略

```rust
#[cfg(test)]
mod stress_tests {
    use std::sync::{Arc, Mutex};
    use std::thread;
    use std::time::Duration;

    #[test]
    fn stress_test_mutex() {
        let data = Arc::new(Mutex::new(0u64));
        let n_threads = 16;
        let n_iterations = 100_000;

        let handles: Vec<_> = (0..n_threads).map(|_| {
            let data = Arc::clone(&data);
            thread::spawn(move || {
                for _ in 0..n_iterations {
                    *data.lock().unwrap() += 1;
                }
            })
        }).collect();

        for h in handles { h.join().unwrap(); }

        assert_eq!(*data.lock().unwrap(), n_threads * n_iterations);
    }

    #[test]
    fn timeout_stress_test() {
        let handle = thread::spawn(|| {
            // 长时间运行的测试
            thread::sleep(Duration::from_secs(10));
        });

        // 使用超时避免测试无限挂起
        handle.join().unwrap_or_else(|_| {
            panic!("测试超时");
        });
    }
}
```

## 四、调试工具总结

| 工具 | 用途 | 运行方式 |
|------|------|---------|
| ThreadSanitizer | 数据竞争检测 | RUSTFLAGS="-Z sanitizer=thread" |
| MIRI | 未定义行为检测 | cargo +nightly miri test |
| Loom | 并发模型验证 | loom::model(|| { ... }) |
| Backtrace | 堆栈追踪 | RUST_BACKTRACE=1 |
| Tracing | 结构化日志 | tracing + tracing-subscriber |
| cargo-flamegraph | 火焰图 | cargo flamegraph |

## 五、注意事项与常见陷阱

1. **日志开销**：并发日志可能打乱执行顺序，影响调试准确性
2. **Heisenbug**：添加调试代码可能改变并发行为，导致问题消失
3. **时间敏感**：避免依赖特定时间顺序，使用确定性调度
4. **工具选择**：sanitizer、miri、loom 各有适用场景，组合使用效果更好
5. **最小复现**：尽量缩小复现范围，去除无关代码
6. **CI 集成**：在 CI 中运行 sanitizer 测试，及早发现问题
7. **环境隔离**：测试环境与生产环境配置一致
