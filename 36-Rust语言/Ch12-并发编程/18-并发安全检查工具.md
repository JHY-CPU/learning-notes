# 并发安全检查工具

## 一、概念说明

Rust 和生态系统提供了多种工具检测并发问题：Miri（未定义行为）、Thread Sanitizer（数据竞争）、loom（并发正确性）。

```bash
# Miri 检测未定义行为
cargo +nightly miri run

# Thread Sanitizer
RUSTFLAGS="-Z sanitizer=thread" cargo run
```

## 二、具体用法

### 2.1 Miri 使用

```bash
# 安装
rustup +nightly component add miri

# 运行测试
cargo +nightly miri test

# Miri 能检测：
# - 数据竞争
# - 未初始化内存访问
# - 内存泄漏
# - 悬垂指针
```

### 2.2 Clippy 并发 lint

```rust
// Clippy 会警告常见的并发错误
// 警告：在循环中持有锁
fn bad_pattern(data: &std::sync::Mutex<Vec<i32>>) {
    // Clippy 警告：锁的持有时间过长
    let mut guard = data.lock().unwrap();
    for _ in 0..10 {
        guard.push(1);
    }
}
```

### 2.3 测试策略

```rust
#[cfg(test)]
mod tests {
    use std::sync::{Arc, Mutex};
    use std::thread;

    #[test]
    fn test_no_data_race() {
        let data = Arc::new(Mutex::new(0));
        let handles: Vec<_> = (0..10).map(|_| {
            let d = Arc::clone(&data);
            thread::spawn(move || {
                for _ in 0..100 {
                    *d.lock().unwrap() += 1;
                }
            })
        }).collect();

        for h in handles { h.join().unwrap(); }
        assert_eq!(*data.lock().unwrap(), 1000);
    }
}
```

### 2.4 Loom 并发模型检查

```rust
// Cargo.toml: [dev-dependencies] loom = "0.7"

#[cfg(test)]
mod loom_tests {
    use loom::sync::{Arc, Mutex};
    use loom::thread;

    #[test]
    fn verify_mutex_correctness() {
        loom::model(|| {
            let data = Arc::new(Mutex::new(0));
            let mut handles = vec![];

            for _ in 0..2 {
                let data = Arc::clone(&data);
                handles.push(thread::spawn(move || {
                    *data.lock().unwrap() += 1;
                }));
            }

            for h in handles { h.join().unwrap(); }
            assert_eq!(*data.lock().unwrap(), 2);
        });
    }

    #[test]
    fn verify_channel_ordering() {
        use loom::sync::mpsc::channel;

        loom::model(|| {
            let (tx, rx) = channel();
            let tx2 = tx.clone();

            thread::spawn(move || {
                tx.send(1).unwrap();
            });

            thread::spawn(move || {
                tx2.send(2).unwrap();
            });

            let mut received = vec![];
            received.push(rx.recv().unwrap());
            received.push(rx.recv().unwrap());
            // loom 会测试所有可能的消息顺序
        });
    }
}
```

### 2.5 cargo-tarpaulin 代码覆盖率

```bash
# 安装
cargo install cargo-tarpaulin

# 生成覆盖率报告
cargo tarpaulin --out Html --output-dir coverage

# 只测试库代码
cargo tarpaulin --lib --out Lcov

# 排除某些文件
cargo tarpaulin --exclude-files "tests/*"
```

### 2.6 扩展 Clippy 检查

```toml
# Cargo.toml
[lints.clippy]
all = "warn"
mutex_integer = "warn"          # Mutex<i32> 应该用 Atomic
await_holding_lock = "warn"     # await 时持有锁
needless_pass_by_ref_mut = "warn"
significant_drop_tightening = "warn"  # 释放锁后仍使用锁内数据
```

### 2.7 运行时死锁检测

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::{Duration, Instant};

fn deadlock_detector_example() {
    let data = Arc::new(Mutex::new(0));
    let detector = Arc::new(Mutex::new(Vec::<Instant>::new()));

    // 模拟可能的死锁检测
    let data_clone = Arc::clone(&data);
    let detector_clone = Arc::clone(&detector);

    let handle = thread::spawn(move || {
        let start = Instant::now();
        match data_clone.try_lock() {
            Ok(guard) => {
                *detector_clone.lock().unwrap().push(start);
                drop(guard);
            }
            Err(_) => {
                if start.elapsed() > Duration::from_secs(1) {
                    eprintln!("警告：可能的死锁检测到");
                }
            }
        }
    });

    handle.join().unwrap();
}
```

## 四、检查工具对比

| 工具 | 检测类型 | 性能影响 | 使用难度 |
|------|---------|---------|---------|
| ThreadSanitizer | 数据竞争 | 高 | 中 |
| MIRI | 未定义行为 | 很高 | 低 |
| Loom | 并发正确性 | 中 | 高 |
| Clippy | 代码质量 | 无 | 低 |
| cargo-tarpaulin | 测试覆盖率 | 低 | 低 |
| deadlock_detector | 死锁 | 中 | 中 |

## 五、CI 集成最佳实践

```yaml
# .github/workflows/concurrency-checks.yml
name: Concurrency Checks

on: [push, pull_request]

jobs:
  clippy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: rustup component add clippy
      - run: cargo clippy -- -D warnings

  sanitizer:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: rustup toolchain install nightly
      - run: RUSTFLAGS="-Z sanitizer=thread" cargo +nightly test

  miri:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: rustup toolchain install nightly
      - run: rustup +nightly component add miri
      - run: cargo +nightly miri test

  coverage:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cargo install cargo-tarpaulin
      - run: cargo tarpaulin --out Lcov
```

## 六、注意事项与常见陷阱

1. **CI 集成**：在 CI 中运行 sanitizer 测试，及早发现问题
2. **性能影响**：sanitizer 会显著降低运行速度，仅在 CI 中使用
3. **工具组合**：结合使用多种工具效果更好，不同工具检测不同问题
4. **测试覆盖**：确保并发代码有足够测试覆盖，特别是边界情况
5. **持续监控**：生产环境也需要监控并发问题，使用 metrics 和 tracing
6. **误报处理**：某些工具可能产生误报，需要合理配置排除规则
7. **测试可重复性**：并发测试需要保证可重复性，使用确定性种子
