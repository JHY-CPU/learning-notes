# 死锁与避免

## 一、概念说明

死锁是并发编程中常见的问题，指两个或多个线程互相等待对方释放资源。Rust 的类型系统可以防止数据竞争，但不能完全防止死锁。

```rust
// 死锁场景示例
use std::sync::{Arc, Mutex};
use std::thread;

let mutex1 = Arc::new(Mutex::new(0));
let mutex2 = Arc::new(Mutex::new(0));

// 线程1: 先锁mutex1，再锁mutex2
// 线程2: 先锁mutex2，再锁mutex1
// 可能导致死锁
```

## 二、具体用法

### 2.1 锁顺序协议

```rust
// 避免死锁：统一锁的获取顺序
fn safe_lock(m1: &Mutex<i32>, m2: &Mutex<i32>) {
    // 按地址顺序获取锁
    let first = if std::ptr::eq(m1, m2) {
        return;
    } else if (m1 as *const _) < (m2 as *const _) {
        m1.lock().unwrap();
        m2.lock().unwrap()
    } else {
        m2.lock().unwrap();
        m1.lock().unwrap()
    };
}
```

### 2.2 try_lock 避免

```rust
use std::sync::Mutex;

fn try_both(m1: &Mutex<i32>, m2: &Mutex<i32>) -> bool {
    let lock1 = m1.try_lock();
    if lock1.is_err() {
        return false;
    }

    let lock2 = m2.try_lock();
    if lock2.is_err() {
        drop(lock1);
        return false;
    }

    // 成功获取两个锁
    true
}
```

### 2.3 带超时的锁获取

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

fn timeout_lock_example() {
    let data = Arc::new(Mutex::new(0));
    let data_clone = Arc::clone(&data);

    // 线程1 持有锁较长时间
    let h1 = thread::spawn(move || {
        let mut guard = data_clone.lock().unwrap();
        *guard = 42;
        thread::sleep(Duration::from_secs(5));
    });

    // 线程2 尝试带超时获取锁
    let h2 = thread::spawn(move || {
        // 使用 parking_lot 的超时锁
        // 或者手动实现超时逻辑
        let start = std::time::Instant::now();
        loop {
            match data.try_lock() {
                Ok(guard) => {
                    println!("获取锁成功: {}", *guard);
                    break;
                }
                Err(_) => {
                    if start.elapsed() > Duration::from_secs(2) {
                        println!("获取锁超时，放弃");
                        break;
                    }
                    thread::sleep(Duration::from_millis(10));
                }
            }
        }
    });

    h1.join().unwrap();
    h2.join().unwrap();
}
```

### 2.5 消息传递避免锁

```rust
use std::sync::mpsc;
use std::thread;

fn message_passing_no_lock() {
    let (tx, rx) = mpsc::channel();

    // 使用消息传递替代共享状态
    thread::spawn(move || {
        for i in 0..10 {
            tx.send(i * i).unwrap();
        }
    });

    for received in rx {
        println!("收到: {}", received);
    }
    // 完全没有锁，不可能死锁
}
```

### 2.6 活锁与饥饿

```rust
use std::sync::{Arc, Mutex};
use std::thread;
use std::time::Duration;

fn livelock_example() {
    // 活锁：线程不断重试但无法进展
    let data = Arc::new(Mutex::new(0));

    // 避免活锁：使用指数退避
    fn try_with_backoff(data: &Mutex<i32>, max_retries: usize) -> bool {
        for attempt in 0..max_retries {
            match data.try_lock() {
                Ok(mut guard) => {
                    *guard += 1;
                    return true;
                }
                Err(_) => {
                    // 指数退避
                    let delay = Duration::from_millis(1 << attempt);
                    thread::sleep(delay);
                }
            }
        }
        false
    }
}
```

## 四、死锁四要素分析

```
死锁四要素（Coffman 条件）：
1. 互斥：资源只能被一个线程持有
2. 持有等待：持有一个锁的同时等待另一个
3. 不可剥夺：锁不能被强制释放
4. 循环等待：线程间形成等待环

打破任意一个条件即可避免死锁：
- 打破条件2：一次性获取所有锁
- 打破条件3：使用 try_lock + 超时
- 打破条件4：统一锁获取顺序
```

## 五、注意事项与常见陷阱

1. **锁顺序**：所有线程以相同顺序获取锁，按指针地址排序
2. **超时机制**：使用 try_lock 或带超时的锁，避免无限等待
3. **锁粒度**：减少同时持有的锁数量，使用消息传递代替共享状态
4. **检测工具**：使用 thread sanitizer、loom、RUST_BACKTRACE 检测死锁
5. **设计原则**：减少锁的使用，优先使用消息传递和原子操作
6. **活锁风险**：重试逻辑不当可能导致活锁，使用指数退避
7. **饥饿问题**：不公平调度可能导致某些线程饥饿，考虑公平锁
