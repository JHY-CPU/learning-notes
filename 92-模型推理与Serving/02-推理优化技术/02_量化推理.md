# 量化推理 - 模型推理与Serving

*GPTQ/AWQ/GGUF三大量化方案的原理、实现与选型，INT4 Kernel性能分析与llama.cpp/GGUF生态*

量化收益一览

| 精度 | 每参数字节 | 70B模型显存 | 相对FP16加速 | 精度损失 |
| --- | --- | --- | --- | --- |
| FP16/BF16 | 2 bytes | ~140 GB | 1x (基准) | 无 |
| FP8 | 1 byte | ~70 GB | 1.5x-2x | 极小(<0.5%) |
| INT8 | 1 byte | ~70 GB | 1.5x-2x | 小(<1%) |
| INT4 | 0.5 byte | ~35 GB | 2x-3x | 可感知(1-3%) |
| INT4+精调 | 0.5 byte | ~35 GB | 2x-3x | 极小(<1%) |

GPTQ 算法流程

1. **逐层处理**
   : 按层对权重矩阵进行量化
2. **Hessian近似**
   : 使用少量校准数据（128-512条）计算每层的Hessian矩阵（二阶导数信息）
3. **贪心量化**
   : 按Hessian对角元素从小到大排序，优先量化"不敏感"的权重
4. **误差补偿**
   : 量化一个权重后，将误差通过Hessian信息传播到未量化的权重上
5. **分块处理**
   : 将权重矩阵分为128列一组的block，block内做精确误差补偿，block间做近似

AWQ 关键洞察

- **观察**
   : 并非所有权重同等重要。激活值大的通道对应的权重对模型输出影响更大（仅1%的"显著权重"贡献了大部分性能）
- **方法**
   : 识别显著权重通道，对其进行per-channel scaling保护（量化前乘以缩放因子s，反量化后除以s）
- **效果**
   : 在相同bit数下，AWQ比GPTQ精度更高；在相同精度下，AWQ可以使用更低的bit数

llama.cpp 生态系统

- **llama.cpp**
   : C/C++实现的LLM推理引擎，支持CPU + GPU混合推理
- **GGUF格式**
   : 统一的模型文件格式，包含权重、配置、tokenizer
- **量化方案**
   : Q2_K到Q8_0多种量化级别，满足不同精度/速度需求
- **ollama**
   : 基于llama.cpp的用户友好工具，一键运行模型
- **llama-cpp-python**
   : Python绑定，兼容OpenAI API
- **LM Studio**
   : 基于llama.cpp的GUI应用
- **LocalAI**
   : 基于llama.cpp的本地API服务

不同量化Kernel的推理性能 (Llama 2 7B, A100-80G)

| 精度 | 推理速度 (tok/s) | 显存 (GB) | 吞吐提升 |
| --- | --- | --- | --- |
| FP16 (基准) | ~120 | ~14 | 1x |
| INT8 (W8A16) | ~160 | ~8 | 1.33x |
| INT4 GPTQ (W4A16) | ~200 | ~5 | 1.67x |
| INT4 AWQ (W4A16) | ~220 | ~5 | 1.83x |
| FP8 (W8A8) | ~250 | ~8 | 2.08x |
| GPTQ + Marlin Kernel | ~280 | ~5 | 2.33x |


** Marlin是专门为INT4优化的高性能CUDA Kernel，比标准Kernel快40-50%*

三大量化方案对比

| 维度 | GPTQ | AWQ | GGUF |
| --- | --- | --- | --- |
| 核心原理 | Hessian误差补偿 | 激活感知缩放 | K-Quant分组量化 |
| 主要精度 | INT4/INT3/INT2 | INT4/INT3 | 2-8bit多种 |
| 硬件支持 | NVIDIA GPU | NVIDIA GPU | CPU + GPU混合 |
| 推理引擎 | vLLM/TGI/TRT-LLM | vLLM/TGI/TRT-LLM | llama.cpp/ollama |
| 消费级硬件 | 需要GPU | 需要GPU | CPU也可用 |
| 量化速度 | 慢 | 快 | 中等 |
| 推理速度 | 中 | 快 | 依赖硬件 |
| 社区生态 | 成熟 | 增长快 | 最大(HuggingFace) |
