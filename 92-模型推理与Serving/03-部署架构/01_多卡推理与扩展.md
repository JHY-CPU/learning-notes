# 多卡推理与扩展 - 模型推理与Serving

*Tensor Parallel、Pipeline Parallel分布式推理策略，多LoRA Serving、Prefix Caching与智能模型路由*

各层的TP切分方式

| 层类型 | 切分方式 | 通信操作 |
| --- | --- | --- |
| Embedding | 按词表维度切分，每GPU持有1/N词表 | AllGather |
| Q/K/V投影 | 按head维度切分，每GPU计算部分head | 无需通信（独立计算） |
| Attention输出 | 按head维度拼接后投影 | AllReduce |
| MLP上投影 | Column Parallel（按列切分） | AllGather |
| MLP下投影 | Row Parallel（按行切分） | AllReduce |
| LM Head | 按词表维度切分 | AllGather |

适用场景

- **System Prompt**
   : 所有请求共享相同的系统提示（通常500-2000 tokens）
- **Few-shot示例**
   : 多个请求使用相同的示例集
- **RAG文档**
   : 多个问题针对同一篇文档
- **长上下文场景**
   : 输入上下文越长，缓存收益越大


**收益**: Prefill阶段可节省50-90%的计算量（取决于前缀共享比例），首token延迟显著降低

分级模型路由

```
请求进入
    │
    ├── 分类器/规则判断请求复杂度
    │       │
    │       ├── 简单请求（翻译、格式化）→ 小模型 (7B/13B)
    │       │   └─ 延迟低，成本低
    │       │
    │       ├── 中等请求（问答、摘要）→ 中模型 (70B)
    │       │   └─ 平衡质量与成本
    │       │
    │       └── 复杂请求（推理、创作）→ 大模型 (405B/GPT-4)
    │           └─ 最高质量
    │
    └── 汇聚响应返回客户端
```


**收益**: 70%+的简单请求被路由到小模型，整体成本降低40-60%

## Tensor Parallel 详解

Tensor Parallel 将模型的单个层内的张量切分到多个 GPU 上。

```
MLP 层的 Tensor Parallel：
输入 x (完整在每个GPU上)
    │
    ├── GPU0: Column Parallel Linear (权重按列切分)
    │         计算 Y0 = X·W0
    │
    ├── GPU1: Column Parallel Linear
    │         计算 Y1 = X·W1
    │
    │ AllReduce 求和
    ▼
Y = Y0 + Y1 (结果在每个GPU上都有完整副本)
```

Attention 层的 Tensor Parallel：
- Q/K/V 投影矩阵按 head 维度切分到不同 GPU
- 每个 GPU 独立计算自己的 head
- Attention 输出通过 AllReduce 合并
- 通信量与注意力 head 数成正比

## Pipeline Parallel 详解

Pipeline Parallel 将模型的不同层切分到不同 GPU 上。

```
GPU0: Embedding + Layer 0-9
      │ micro-batch 1 通过
      ▼
GPU1: Layer 10-19
      │ micro-batch 1 通过
      ▼
GPU2: Layer 20-29 + LM Head
      │
      └── 同时 GPU0 处理 micro-batch 2
```

GPipe vs 1F1B 调度：

| 调度方式 | 原理 | 内存效率 | 延迟 |
|----------|------|---------|------|
| GPipe | 先全部前向，再全部反向 | 高（峰值内存大） | 高 |
| 1F1B | 前向反向交替执行 | 中（峰值内存小） | 低 |

## 多 LoRA Serving

在基础模型上同时服务多个 LoRA 适配器：

```python
# vLLM 多 LoRA 支持
from vllm import LLM, SamplingParams

llm = LLM(
    model="base-model",
    enable_lora=True,
    max_loras=4,  # 同时加载最多4个LoRA
    max_lora_rank=64,
)

# 使用不同 LoRA
output1 = llm.generate(
    prompts,
    sampling_params,
    lora_request=LoRARequest("medical-lora", 1, "/path/to/medical-lora"),
)
output2 = llm.generate(
    prompts,
    sampling_params,
    lora_request=LoRARequest("legal-lora", 2, "/path/to/legal-lora"),
)
```

LoRA 的优势：
- 共享基础模型权重，只加载小的适配器矩阵（通常 < 100MB）
- 支持快速切换不同领域的微调模型
- 内存效率高：N 个 LoRA 只需基础模型 + N × LoRA 权重

## Prefix Caching 详解

当多个请求共享相同的前缀（如 System Prompt、Few-shot 示例）时，缓存前缀的 KV 向量：

```
无 Prefix Caching：
请求1: [System Prompt + 问题1] → 每次都计算 System Prompt 的 KV
请求2: [System Prompt + 问题2] → 重复计算 System Prompt 的 KV

有 Prefix Caching：
System Prompt 的 KV Cache 缓存在 GPU 显存中
请求1: [System Prompt(命中缓存) + 问题1] → 只计算问题1
请求2: [System Prompt(命中缓存) + 问题2] → 只计算问题2
```

适用场景：
- **System Prompt**: 所有请求共享相同的系统提示（通常500-2000 tokens）
- **Few-shot示例**: 多个请求使用相同的示例集
- **RAG文档**: 多个问题针对同一篇文档
- **长上下文场景**: 输入上下文越长，缓存收益越大

## 多卡部署方案对比

| 方案 | 切分维度 | 通信开销 | 适用模型 | 最大GPU数 |
|------|---------|---------|---------|----------|
| Tensor Parallel | 层内张量 | 高（AllReduce） | 7B-70B | 8 (单节点) |
| Pipeline Parallel | 层间 | 低（发送激活） | 70B+ | 64+ |
| TP + PP 混合 | 层内+层间 | 中 | 70B-405B | 256+ |
| Expert Parallel | 专家维度 | 中 | MoE 模型 | 与专家数相关 |

## 实际部署示例

### 70B 模型 4 卡部署

```bash
# vLLM 部署 Qwen2-72B
python -m vllm.entrypoints.openai.api_server \
    --model Qwen/Qwen2-72B-Instruct \
    --tensor-parallel-size 4 \
    --gpu-memory-utilization 0.9 \
    --max-model-len 32768 \
    --enable-prefix-caching \
    --port 8000
```

### 405B 模型 8 卡部署

```bash
# 使用 TP=8 部署
python -m vllm.entrypoints.openai.api_server \
    --model meta-llama/Llama-3.1-405B \
    --tensor-parallel-size 8 \
    --dtype bfloat16 \
    --max-model-len 8192 \
    --enforce-eager \  # 大模型避免 CUDA Graphs OOM
    --port 8000
```
