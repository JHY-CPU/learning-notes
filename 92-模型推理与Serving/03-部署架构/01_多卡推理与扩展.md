# 多卡推理与扩展 - 模型推理与Serving

*Tensor Parallel、Pipeline Parallel分布式推理策略，多LoRA Serving、Prefix Caching与智能模型路由*

各层的TP切分方式

| 层类型 | 切分方式 | 通信操作 |
| --- | --- | --- |
| Embedding | 按词表维度切分，每GPU持有1/N词表 | AllGather |
| Q/K/V投影 | 按head维度切分，每GPU计算部分head | 无需通信（独立计算） |
| Attention输出 | 按head维度拼接后投影 | AllReduce |
| MLP上投影 | Column Parallel（按列切分） | AllGather |
| MLP下投影 | Row Parallel（按行切分） | AllReduce |
| LM Head | 按词表维度切分 | AllGather |

适用场景

- **System Prompt**
   : 所有请求共享相同的系统提示（通常500-2000 tokens）
- **Few-shot示例**
   : 多个请求使用相同的示例集
- **RAG文档**
   : 多个问题针对同一篇文档
- **长上下文场景**
   : 输入上下文越长，缓存收益越大


**收益**: Prefill阶段可节省50-90%的计算量（取决于前缀共享比例），首token延迟显著降低

分级模型路由

```
请求进入
    │
    ├── 分类器/规则判断请求复杂度
    │       │
    │       ├── 简单请求（翻译、格式化）→ 小模型 (7B/13B)
    │       │   └─ 延迟低，成本低
    │       │
    │       ├── 中等请求（问答、摘要）→ 中模型 (70B)
    │       │   └─ 平衡质量与成本
    │       │
    │       └── 复杂请求（推理、创作）→ 大模型 (405B/GPT-4)
    │           └─ 最高质量
    │
    └── 汇聚响应返回客户端
```


**收益**: 70%+的简单请求被路由到小模型，整体成本降低40-60%
