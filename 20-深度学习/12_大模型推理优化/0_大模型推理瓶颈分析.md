# 0_大模型推理瓶颈分析

## 1. 推理过程概述

大模型推理（Inference）是将训练好的模型应用于实际预测的过程。对于自回归语言模型，推理即**逐 token 生成**文本。

```
自回归生成过程：

输入: "今天天气"
  ↓
[Prompt Prefill] → 并行处理所有输入 token
  ↓
[Decode Step 1]  → 生成 "真"
[Decode Step 2]  → 生成 "好"
[Decode Step 3]  → 生成 "，"
[Decode Step 4]  → 生成 "适合"
...
  ↓
输出: "今天天气真好，适合出门散步"

Prefill 阶段: 并行计算，受计算瓶颈限制
Decode 阶段: 逐 token 串行，受内存带宽瓶颈限制
```

## 2. 计算瓶颈 vs 内存瓶颈

### 2.1 Roofline 模型

```
性能
  │
  │         ╭───────────── 计算上限 (FLOPS)
  │        ╱
  │       ╱
  │      ╱
  │     ╱    ← 瓶颈区域
  │    ╱
  │   ╱
  │──╱────────────────────── 内存带宽上限
  │ ╱
  │╱
  └────────────────────────→ 算术强度 (FLOPs/Byte)

低算术强度 → 内存带宽瓶颈 (Memory-bound)
高算术强度 → 计算瓶颈 (Compute-bound)
```

### 2.2 Prefill vs Decode 阶段对比

```python
# Prefill 阶段分析（处理输入 prompt）
# 假设输入 n 个 token，模型维度 d，层数 L
def prefill_compute(n, d, L):
    # 每层注意力: QKV 投影 + 注意力计算 + 输出投影
    # = O(n * d^2) + O(n^2 * d) + O(n * d^2)
    attn_flops = L * (4 * n * d**2 + 2 * n**2 * d)
    # FFN: 两个线性层
    ffn_flops = L * 8 * n * d**2
    return attn_flops + ffn_flops

# Decode 阶段分析（生成单个 token）
def decode_compute(d, L):
    # 注意力: 只有一个新 token，与 KV Cache 交互
    # = O(1 * d^2) + O(seq_len * d) + O(1 * d^2)
    attn_flops = L * (4 * d**2 + 2 * 1 * d)  # n=1
    ffn_flops = L * 8 * d**2
    return attn_flops + ffn_flops

# 对比: Prefill vs Decode
# Prefill: 计算密集，可并行 → Compute-bound
# Decode: 内存密集，串行 → Memory-bound
```

## 3. 关键瓶颈分析

### 3.1 内存带宽瓶颈

```
LLM 推理的内存访问模式：

Decode 步骤:
  1. 加载模型权重 W (GB 级别)
  2. 加载 KV Cache (随序列长度增长)
  3. 计算单个 token 的输出
  4. 写入新 token 到 KV Cache

实际利用率分析 (A100 80GB):
  - 内存带宽: 2 TB/s
  - FP16 模型 7B 参数: 14 GB
  - 理论最大 decode 速度: 2000 / 14 ≈ 143 tokens/s
  - 实际: 30-50 tokens/s (其他开销)

关键洞察: Decode 阶段几乎所有时间花在读取模型权重上，
计算只占极小比例。这就是为什么增大 batch size 能提升吞吐量。
```

### 3.2 KV Cache 内存占用

```python
def calculate_kv_cache_memory(
    batch_size: int,     # 批大小
    seq_length: int,     # 序列长度
    num_layers: int,     # 模型层数
    num_heads: int,      # 注意力头数
    head_dim: int,       # 每头维度
    dtype_bytes: int = 2  # FP16 = 2 bytes
) -> float:
    """计算 KV Cache 内存占用 (Bytes)"""
    # 每个 token 每层: K 和 V 两个向量
    per_token_per_layer = 2 * num_heads * head_dim * dtype_bytes
    total = batch_size * seq_length * num_layers * per_token_per_layer
    return total

# 以 LLaMA-7B 为例
memory = calculate_kv_cache_memory(
    batch_size=1,
    seq_length=2048,
    num_layers=32,
    num_heads=32,
    head_dim=128
)
print(f"KV Cache 内存: {memory / 1024**3:.2f} GB")
# ≈ 4 GB (单请求, 2048 长度)
```

### 3.3 延迟分解

```python
class LatencyProfiler:
    """推理延迟分解"""

    def profile(self, model, input_tokens) -> dict:
        timings = {}

        # Prefill 阶段
        start = time.time()
        _ = model.prefill(input_tokens)
        timings["prefill"] = time.time() - start

        # Decode 阶段
        decode_times = []
        for _ in range(100):  # 生成 100 个 token
            start = time.time()
            _ = model.decode_step()
            decode_times.append(time.time() - start)

        timings["decode_avg"] = sum(decode_times) / len(decode_times)
        timings["decode_p99"] = sorted(decode_times)[98]

        # 各层耗时
        timings["layer_breakdown"] = self.profile_layers(model)

        return timings
```

## 4. 延迟与吞吐量权衡

```
延迟 (Latency) vs 吞吐量 (Throughput):

              延迟
               ↑
     单请求    │  ○ 低延迟，低吞吐
     低并发    │
               │
               │      ○○○ 批处理
               │      高延迟，高吞吐
               │
               └──────────────────→ 吞吐量

TTFT (Time To First Token): 首 token 时间
  - 主要受 Prefill 影响
  - 用户体验直接相关

TPOT (Time Per Output Token): 每 token 时间
  - 主要受 Decode 影响
  - 决定整体生成速度

总延迟 = TTFT + TPOT × 生成 token 数
```

## 5. 推理优化方向总览

```
推理优化技术栈：

硬件层
├── GPU/TPU/NPU
├── 专用推理芯片 (Groq, Cerebras)
└── 边缘设备 (手机, 嵌入式)

模型层
├── 量化 (INT8/INT4)      → 减少内存占用和带宽需求
├── 蒸馏                   → 用小模型逼近大模型
├── 剪枝                   → 移除冗余参数
└── 架构优化 (GQA, MoE)   → 降低计算量

推理引擎层
├── KV Cache 管理         → 高效内存管理
├── 连续批处理             → 提升 GPU 利用率
├── 推测解码               → 减少串行步骤
├── 并行策略 (TP/PP)       → 分布式推理
└── 算子融合               → 减少 kernel launch

服务层
├── vLLM / TGI            → 高性能推理框架
├── 流式输出               → 改善用户体验
└── 负载均衡               → 资源调度
```

## 总结

大模型推理的核心瓶颈是**内存带宽**而非计算能力。Decode 阶段是典型的 Memory-bound 操作，优化方向围绕**减少内存访问量**（量化、KV Cache 优化）和**提升并行度**（连续批处理、推测解码）展开。理解这些瓶颈是学习后续优化技术的基础。
