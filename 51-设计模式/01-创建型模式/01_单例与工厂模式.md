# 单例与工厂模式


## 一、创建型模式概述


创建型模式关注对象的创建过程，将对象的创建与使用分离，提高系统的灵活性和可维护性。


| 模式 | 核心意图 | 关键点 |
| --- | --- | --- |
| 单例模式 | 确保一个类只有一个实例 | 全局访问点 |
| 工厂方法 | 定义创建对象的接口，让子类决定实例化哪个类 | 延迟实例化 |
| 抽象工厂 | 创建一系列相关对象 | 产品族 |


## 二、单例模式（Singleton）


### 2.1 意图


确保一个类只有一个实例，并提供一个全局访问点。


### 2.2 JavaScript实现


```
// 方法一：闭包实现
const Singleton = (function() {
    let instance;

    function createInstance() {
        return {
            data: '单例数据',
            getData() { return this.data; }
        };
    }

    return {
        getInstance() {
            if (!instance) {
                instance = createInstance();
            }
            return instance;
        }
    };
})();

// 使用
const a = Singleton.getInstance();
const b = Singleton.getInstance();
console.log(a === b); // true
```


```
// 方法二：ES6 Class实现
class Database {
    static #instance = null;

    constructor(url) {
        if (Database.#instance) {
            throw new Error('请使用 getInstance()');
        }
        this.url = url;
        this.connection = null;
    }

    static getInstance(url) {
        if (!Database.#instance) {
            Database.#instance = new Database(url);
        }
        return Database.#instance;
    }

    query(sql) {
        console.log(`执行: ${sql}`);
    }
}
```


### 2.3 适用场景


- 数据库连接池、线程池
- 日志记录器（Logger）
- 应用配置管理
- 缓存管理器
- 事件总线（Event Bus）


> **Important:** **注意事项：**
> 单例模式可能导致代码耦合度高、难以测试。在现代开发中，更推荐使用依赖注入（DI）容器来管理单例。


## 三、工厂方法模式（Factory Method）


### 3.1 意图


定义一个创建对象的接口，让子类决定实例化哪个类。工厂方法让类的实例化推迟到子类。


### 3.2 JavaScript实现


```
// 产品接口
class Button {
    render() { throw new Error('子类必须实现'); }
    onClick() { throw new Error('子类必须实现'); }
}

// 具体产品
class WindowsButton extends Button {
    render() { console.log('渲染Windows按钮'); }
    onClick() { console.log('Windows按钮点击'); }
}

class MacButton extends Button {
    render() { console.log('渲染Mac按钮'); }
    onClick() { console.log('Mac按钮点击'); }
}

// 创建者（工厂）
class Dialog {
    // 工厂方法
    createButton() { throw new Error('子类必须实现'); }

    render() {
        const button = this.createButton();
        button.render();
        button.onClick();
    }
}

// 具体创建者
class WindowsDialog extends Dialog {
    createButton() { return new WindowsButton(); }
}

class MacDialog extends Dialog {
    createButton() { return new MacButton(); }
}

// 使用
const os = 'windows';
const dialog = os === 'windows'
    ? new WindowsDialog()
    : new MacDialog();
dialog.render();
```


## 四、抽象工厂模式（Abstract Factory）


### 4.1 意图


提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们具体的类。


### 4.2 JavaScript实现


```
// 抽象工厂
class UIFactory {
    createButton() { throw new Error('子类必须实现'); }
    createInput() { throw new Error('子类必须实现'); }
    createCheckbox() { throw new Error('子类必须实现'); }
}

// 具体工厂 - Material风格
class MaterialFactory extends UIFactory {
    createButton() {
        return { style: 'material', type: 'button', render() {
            console.log('Material风格按钮（有阴影效果）');
        }};
    }
    createInput() {
        return { style: 'material', type: 'input', render() {
            console.log('Material风格输入框（下划线）');
        }};
    }
}

// 具体工厂 - Ant风格
class AntFactory extends UIFactory {
    createButton() {
        return { style: 'ant', type: 'button', render() {
            console.log('Ant Design风格按钮');
        }};
    }
    createInput() {
        return { style: 'ant', type: 'input', render() {
            console.log('Ant Design风格输入框');
        }};
    }
}

// 使用：切换整个UI风格只需换工厂
function buildUI(factory) {
    const button = factory.createButton();
    const input = factory.createInput();
    button.render();
    input.render();
}

buildUI(new MaterialFactory());
buildUI(new AntFactory());
```


### 4.3 工厂方法 vs 抽象工厂


| 对比 | 工厂方法 | 抽象工厂 |
| --- | --- | --- |
| 产品数量 | 一个产品 | 一族相关产品 |
| 扩展方式 | 添加新的创建者子类 | 添加新的工厂类 |
| 适用场景 | 创建单一产品，具体类型运行时决定 | 创建一系列产品，需要保证兼容性 |


## 五、知识要点总结


1. 单例模式：一个类一个实例，全局访问点
2. 工厂方法：延迟实例化到子类，创建单一产品
3. 抽象工厂：创建一族相关产品，保证产品兼容性
4. 三者都遵循开闭原则（对扩展开放，对修改关闭）
5. 现代JS/TS中，模块系统天然提供单例；依赖注入替代工厂
