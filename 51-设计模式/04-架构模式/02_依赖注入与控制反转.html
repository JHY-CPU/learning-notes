依赖注入与控制反转 (DI & IoC)

Dependency Injection & Inversion of Control — 解耦组件、提升可测试性与可维护性

1. 核心概念

控制反转 (IoC)

  • 传统流程：对象自行创建依赖 A → new B()
  • IoC 流程：由外部容器注入依赖，对象不再主动控制
  • 好莱坞原则："Don't call us, we'll call you"
  • 控制权从组件代码转移到框架/容器

依赖注入 (DI)

  • IoC 的一种具体实现方式
  • 将依赖从外部传入而非内部创建
  • 降低耦合度，依赖于抽象而非实现
  • 便于单元测试（可注入 Mock 对象）

IoC 容器

  • 管理对象生命周期的框架
  • 自动解析依赖图 (Dependency Graph)
  • 支持作用域：Singleton、Transient、Scoped
  • 示例：Spring、Unity、Autofac、Guice

2. 三种注入方式

2.1 构造函数注入 (Constructor Injection)

推荐方式，依赖在对象创建时即被满足，保证对象始终处于有效状态。

public class OrderService {
    private readonly IPaymentGateway _payment;
    private readonly IInventoryRepo _inventory;

    // 构造函数注入：依赖明确、不可变
    public OrderService(IPaymentGateway payment,
                        IInventoryRepo inventory) {
        _payment = payment ?? throw new ArgumentNullException();
        _inventory = inventory ?? throw new ArgumentNullException();
    }
}

2.2 属性注入 (Setter / Property Injection)

适用于可选依赖。缺点：对象可能在依赖未注入时被使用。

public class NotificationService {
    // 属性注入：可选依赖
    public ILogger Logger { get; set; } = new NullLogger();

    public void Notify(string message) {
        Logger?.Log("Sending: " + message);
    }
}

2.3 接口注入 (Interface Injection)

类实现特定接口，容器通过接口回调注入依赖。较少使用。

public interface ILoggerAware {
    void SetLogger(ILogger logger);
}
public class OrderProcessor : ILoggerAware {
    private ILogger _logger;
    public void SetLogger(ILogger logger) { _logger = logger; }
}

Spring DI 配置示例

// 接口定义
public interface MessageService {
    void send(String message);
}

// 实现类
@Component
public class EmailService implements MessageService {
    public void send(String message) {
        System.out.println("Email: " + message);
    }
}

// 注入使用
@Component
public class NotificationManager {
    private final MessageService service;

    @Autowired
    public NotificationManager(MessageService service) {
        this.service = service;
    }
}

.NET 内置 DI 示例

// 注册服务 (Program.cs)
builder.Services.AddScoped<IOrderRepository, OrderRepository>();
builder.Services.AddSingleton<ICacheService, RedisCacheService>();
builder.Services.AddTransient<IEmailSender, SmtpEmailSender>();

// 控制器中注入
public class OrderController : ControllerBase {
    private readonly IOrderRepository _repo;
    public OrderController(IOrderRepository repo) { _repo = repo; }
}

4. Service Locator vs 依赖注入

注意：Service Locator 被 Martin Fowler 称为"反模式"（在大多数场景下），因为它隐藏了类的真实依赖关系。

Service Locator（反模式）

// 服务定位器：类主动"拉取"依赖
public class OrderService {
    public void CreateOrder() {
        // 依赖被隐藏在方法内部，难以从构造函数看出
        var payment = ServiceLocator.Get<IPaymentGateway>();
        var inventory = ServiceLocator.Get<IInventory>();
        payment.Charge(amount);
    }
}

依赖注入（推荐）

// 依赖注入：依赖明确暴露在构造函数
public class OrderService {
    private readonly IPaymentGateway _payment;
    private readonly IInventory _inventory;

    public OrderService(IPaymentGateway p, IInventory i) {
        _payment = p; _inventory = i;
    }
}
维度Service Locator依赖注入
依赖可见性隐藏在方法内部构造函数明确声明
可测试性需配置全局 Locator直接传入 Mock
耦合程度耦合 Locator 本身仅耦合接口
适用场景遗留代码改造新项目首选

5. 高级话题

5.1 循环依赖处理

构造函数循环依赖

  • A 依赖 B，B 依赖 A
  • 直接导致容器初始化失败
  • 解决方案：重新设计，提取公共依赖到第三个类

Setter 循环依赖

  • Spring 默认支持（三级缓存）
  • 先创建实例，再注入属性
  • 不推荐依赖此特性，应优先重构

5.2 生命周期管理

// 三种常见作用域
Singleton    // 整个应用生命周期内唯一实例
Transient    // 每次请求创建新实例
Scoped       // 同一作用域内唯一（如同一个 HTTP 请求）

// 典型场景：
// - 数据库连接池 → Scoped
// - 配置服务 → Singleton
// - 工作单元对象 → Transient

5.3 最佳实践

原则总结： 1) 始终面向接口编程，注册为接口而非具体类； 2) 优先使用构造函数注入； 3) 避免在容器中注册过多服务（关注点分离）； 4) 注意生命周期的匹配（不要将 Scoped 注入到 Singleton 中）。

6. 常见面试题

Q: IoC 和 DI 有什么区别？

IoC 是设计原则（控制权反转），DI 是 IoC 的一种实现手段。IoC 还包括事件驱动、模板方法等实现方式。

Q: 为什么推荐构造函数注入？

① 依赖不可变（final 字段）；② 对象创建时所有依赖就位；③ 构造函数签名即文档；④ 便于发现循环依赖。

Q: DI 容器是必须的吗？

不是。手动构造依赖图（Pure DI / Poor Man's DI）在小型项目中完全可行。容器在大型项目中提供便利，但也引入了复杂度和运行时开销。
