🧩 进程与线程

408：进程状态、PCB、进程通信（共享存储/管道/消息）、线程实现与多线程模型为必背块。Process vs Thread、PCB、地址空间、上下文切换成本。

进程 Process

进程是资源分配的基本单位：拥有独立的虚拟地址空间、打开的文件描述符表、信号处理方式等。

// ========== 进程控制块 PCB（概念）========== // 进程 ID (PID)、父进程 PPID // CPU 上下文：寄存器、程序计数器 PC、栈指针 // 内存描述：页表基址、内存映射信息 // 调度信息：优先级、状态、运行时间统计 // 打开文件表、当前工作目录、umask 等 // ========== 进程状态（简化）========== // Running / Ready: 可运行或正在 CPU 上执行 // Blocked (Waiting): 等待 I/O 或锁，不可调度 // Zombie: 已 exit，父进程尚未 wait，仅占 PCB // Orphan: 父进程先退出，由 init 收养（依系统实现）

线程 Thread

线程是CPU 调度的基本单位（现代 OS）：同一进程内多线程共享地址空间与文件描述符，各自有独立栈与寄存器上下文。

// ========== 进程 vs 线程 ========== // 进程 线程 // 地址空间 独立 共享（同进程内） // 创建开销 大（fork + 映射） 较小（pthread_create） // 通信 IPC（管道、共享内存） 共享内存+同步原语 // 崩溃影响 通常隔离 可能拖垮整个进程 // ========== 内核线程 vs 用户线程 ========== // 内核可见的线程：由调度器直接调度（如 Linux 的 task 与 NPTL） // 纯用户态线程（历史模型）：内核不知晓，阻塞一个可能阻塞整组
📌 Linux 把可调度的执行单元抽象为 task_struct；“线程”常是共享 mm_struct 的一组 task。

上下文切换 Context Switch

保存当前运行实体的寄存器与内核栈，恢复另一个实体的状态。切换越频繁，缓存局部性越差，开销越大。

// ========== 典型成本来源 ========== // 保存/恢复通用寄存器、浮点/SIMD 状态（若使用） // 切换地址空间 → 刷新 TLB（或更新 ASID），代价高 // 缓存污染：新进程工作集不同 // ========== 与 I/O 密集 / CPU 密集 ========== // I/O 密集：经常阻塞 → 调度器切换线程保持 CPU 忙碌 // CPU 密集：过多线程反而增加切换开销，线程数 ≈ CPU 核数 常更合理
