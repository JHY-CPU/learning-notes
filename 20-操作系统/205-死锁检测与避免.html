死锁的四个必要条件

四者同时成立才可能死锁；打破任一即可预防。

// 1. 互斥 Mutual exclusion：资源不可共享占用 // 2. 占有且等待 Hold and wait：持有一部分资源同时等待其他资源 // 3. 不可抢占 No preemption：已分配资源不能被强制剥夺 // 4. 循环等待 Circular wait：存在进程资源的环形等待链 // ========== 预防示例 ========== // 一次性申请所有资源 → 破坏“占有且等待” // 资源有序编号、按序申请 → 破坏“循环等待”

检测与恢复

// ========== 资源分配图 ========== // 进程 → 资源：请求边 // 资源 → 进程：分配边 // 若图中有环且每类资源只有一个实例 → 死锁 // 多实例时需用类似“安全性算法”检测 // ========== 恢复 ========== // 终止进程（代价不同，可选择 youngest / 最小回滚成本） // 抢占资源（实现困难，需保存状态）

银行家算法（避免）

在每次资源请求时，试探性分配后判断系统是否仍处于安全状态：存在一种调度顺序使所有进程能完成。

// ========== 直觉 ========== // Available: 当前空闲资源向量 // Max: 各进程最大需求 // Allocation: 已分配 // Need = Max - Allocation // 若请求 ≤ Need 且 ≤ Available，则试分配 // 运行安全性检查：能否找到进程序列 p1..pn， // 使每个 pi 的 Need 可被当前 Available 满足，完成后释放 Allocation // ========== 局限 ========== // 需预知 Max；开销；通用 OS 很少完整实现（数据库等场景仍有类似思想）
📌 工程上更常见：锁顺序一致、超时、try-lock、死锁检测工具（如线程 dump）。
