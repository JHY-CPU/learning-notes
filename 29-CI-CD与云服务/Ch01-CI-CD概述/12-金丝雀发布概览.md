# 金丝雀发布概览

## 一、概念说明

金丝雀发布是渐进式发布策略，将一小部分流量导向新版本，观察无问题后逐步增加流量。

## 二、发布流程

```
1. 部署新版本（金丝雀）
2. 分配5%流量到新版本
3. 监控指标
4. 无问题 → 增加到20%
5. 继续监控
6. 逐步增加到100%
7. 完成发布
```

## 三、Nginx权重配置

```nginx
upstream backend {
    server 192.168.1.100:8080 weight=95;  # 旧版本95%
    server 192.168.1.101:8080 weight=5;   # 新版本5%
}
```

## 四、Kubernetes金丝雀

```yaml
# Deployment - 旧版本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-stable
spec:
  replicas: 9

---
# Deployment - 金丝雀
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app-canary
spec:
  replicas: 1
```

## 六、注意事项

1. **流量分配**：根据风险承受能力分配
2. **监控充分**：金丝雀期间重点监控
3. **快速回滚**：发现问题立即回滚
4. **会话粘性**：同一用户可能访问不同版本

## 七、Istio金丝雀发布

```yaml
# VirtualService - 流量分割
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: myapp
spec:
  hosts:
    - myapp.example.com
  http:
    - route:
        - destination:
            host: myapp-stable
          weight: 90
        - destination:
            host: myapp-canary
          weight: 10

---
# DestinationRule
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: myapp
spec:
  host: myapp
  subsets:
    - name: stable
      labels:
        version: v1
    - name: canary
      labels:
        version: v2
```

## 八、Flagger自动化金丝雀

```yaml
# Flagger Canary资源
apiVersion: flagger.app/v1beta1
kind: Canary
metadata:
  name: myapp
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  service:
    port: 80
  analysis:
    interval: 1m
    threshold: 5
    maxWeight: 50
    stepWeight: 10
    metrics:
      - name: request-success-rate
        thresholdRange:
          min: 99
        interval: 1m
      - name: request-duration
        thresholdRange:
          max: 500
        interval: 1m
    webhooks:
      - name: load-test
        url: http://flagger-loadtester.test/
        metadata:
          cmd: "hey -z 1m -q 10 -c 2 http://myapp-canary.test/"
```

## 九、金丝雀发布监控仪表板

```yaml
# Prometheus金丝雀监控规则
groups:
  - name: canary
    rules:
      - record: canary:error_rate
        expr: |
          sum(rate(http_requests_total{version="canary",status=~"5.."}[5m]))
          /
          sum(rate(http_requests_total{version="canary"}[5m]))

      - record: canary:latency_p99
        expr: |
          histogram_quantile(0.99,
            sum(rate(http_request_duration_seconds_bucket{version="canary"}[5m])) by (le)
          )

      - alert: CanaryHighErrorRate
        expr: canary:error_rate > 0.05
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "金丝雀版本错误率过高，触发回滚"
```

## 十、金丝雀发布决策流程

```
1. 部署金丝雀版本（5%流量）
   ├── 错误率 > 5%? → 自动回滚
   ├── P99延迟 > 500ms? → 自动回滚
   └── 一切正常 → 继续

2. 增加到 20% 流量
   ├── 监控5分钟
   └── 一切正常 → 继续

3. 增加到 50% 流量
   ├── 监控10分钟
   └── 一切正常 → 继续

4. 全量切换到 100%
   ├── 监控30分钟
   └── 完成发布

每一步都保留自动回滚能力
```
