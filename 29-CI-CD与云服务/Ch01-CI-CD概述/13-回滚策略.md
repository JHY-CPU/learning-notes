# 回滚策略

## 一、概念说明

回滚是在部署失败时恢复到上一个稳定版本的操作。快速可靠的回滚是CI/CD的重要组成部分。

## 二、回滚方式

```bash
# 1. 镜像回滚
# 使用上一个版本的Docker镜像
docker pull app:previous-version
docker stop current-container
docker run app:previous-version

# 2. Kubernetes回滚
kubectl rollout undo deployment/myapp
kubectl rollout undo deployment/myapp --to-revision=2

# 3. Git回滚
git revert HEAD
git push
```

## 三、数据库回滚

```bash
# 数据库迁移回滚
# Flyway
flyway undo

# Liquibase
liquibase rollback

# 自定义迁移脚本
# 每个up迁移对应down迁移
```

## 四、自动回滚

```yaml
# ArgoCD自动回滚
apiVersion: argoproj.io/v1alpha1
kind: Application
spec:
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```

## 五、注意事项

1. **回滚测试**：定期测试回滚流程
2. **数据兼容**：新版本数据兼容旧版本
3. **快速回滚**：回滚应在分钟级完成
4. **回滚记录**：记录回滚原因和时间

## 六、GitHub Actions自动回滚

```yaml
name: Deploy with Auto-Rollback

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy
        id: deploy
        run: |
          kubectl set image deployment/myapp app=$IMAGE:$GITHUB_SHA
          kubectl rollout status deployment/myapp --timeout=300s

      - name: Health Check
        id: health
        run: |
          for i in {1..10}; do
            if curl -f https://example.com/health; then
              echo "Health check passed"
              exit 0
            fi
            sleep 10
          done
          exit 1

      - name: Rollback on Failure
        if: failure() && steps.deploy.outcome == 'success'
        run: |
          echo "Health check failed, rolling back..."
          kubectl rollout undo deployment/myapp
          kubectl rollout status deployment/myapp
          echo "::error::Deployment rolled back due to health check failure"
```

## 七、数据库回滚详细策略

```sql
-- 使用迁移版本控制
-- 每个迁移文件包含 up 和 down

-- V1__create_users.up.sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

-- V1__create_users.down.sql
DROP TABLE IF EXISTS users;

-- V2__add_age_column.up.sql
ALTER TABLE users ADD COLUMN age INTEGER;

-- V2__add_age_column.down.sql
ALTER TABLE users DROP COLUMN age;
```

```bash
# Flyway迁移和回滚
flyway migrate                    # 执行迁移
flyway undo                       # 回滚最后一次迁移
flyway repair                     # 修复迁移历史

# Alembic (Python/SQLAlchemy)
alembic upgrade head              # 迁移到最新
alembic downgrade -1              # 回滚一步
alembic downgrade <revision>      # 回滚到指定版本
```

## 八、回滚检查清单

```
部署前检查:
  [ ] 数据库迁移包含down脚本
  [ ] 新版本API向后兼容
  [ ] 回滚脚本已测试
  [ ] 监控告警已配置
  [ ] 回滚审批人已指定

回滚执行:
  [ ] 确认问题原因
  [ ] 通知团队回滚决定
  [ ] 执行回滚（应用/数据库/配置）
  [ ] 验证回滚后系统正常
  [ ] 记录回滚事件
  [ ] 事后分析和改进

回滚时间线:
  0-5 min: 发现问题
  5-10 min: 决定回滚
  10-15 min: 执行回滚
  15-20 min: 验证恢复
  目标: 20分钟内完成回滚
```

## 九、Git版本回滚方式对比

```bash
# git revert - 创建新提交撤销变更（推荐用于已推送的提交）
git revert HEAD                  # 撤销最后一次提交
git revert HEAD~3..HEAD          # 撤销最近3次提交
git push

# git reset - 移动HEAD指针（仅用于本地未推送的提交）
git reset --soft HEAD~1          # 保留变更在暂存区
git reset --mixed HEAD~1         # 保留变更在工作区
git reset --hard HEAD~1          # 丢弃所有变更

# 版本回退 + 重新部署
git revert HEAD
git push
# CI/CD自动触发重新部署旧版本
```
