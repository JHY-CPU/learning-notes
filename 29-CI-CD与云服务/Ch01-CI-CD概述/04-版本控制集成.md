# 版本控制集成

## 一、概念说明

版本控制是CI/CD的基础。Git是最流行的版本控制系统，与CI/CD工具深度集成。

## 二、Git工作流

```bash
# Git Flow
main      ← 生产分支
develop   ← 开发分支
feature/* ← 功能分支
release/* ← 发布分支
hotfix/*  ← 热修复分支

# GitHub Flow（推荐）
main      ← 主分支
feature/* ← 功能分支，PR合并到main
```

## 三、CI/CD触发

```yaml
# GitHub Actions触发
on:
  push:
    branches: [main, develop]
    paths: ['src/**']
  pull_request:
    branches: [main]
  release:
    types: [published]
```

## 四、分支策略

```bash
# 保护main分支
# - 需要PR审查
# - 需要CI通过
# - 禁止force push
# - 需要签名提交
```

## 五、最佳实践

1. **分支保护**：main分支必须设置保护规则，禁止直接推送
2. **PR检查**：CI必须通过才能合并，配置required checks
3. **提交规范**：使用conventional commits（feat/fix/docs等前缀），便于自动生成CHANGELOG
4. **标签管理**：使用语义化版本号（v1.0.0），配合release分支自动打标签
5. **小步提交**：每个PR只做一件事，便于代码审查和回滚

## 六、常见陷阱

1. **忽略分支保护**：直接push到main导致未经审查的代码上线
2. **触发路径过宽**：没有配置paths过滤，文档修改也触发完整CI流程，浪费资源
3. **合并冲突**：长期不更新feature分支导致大量冲突，建议定期rebase
4. **钩子遗漏**：pre-commit钩子未配置，导致格式不一致的代码进入仓库

## 七、Git工作流详细对比

### Git Flow vs GitHub Flow vs GitLab Flow

```
Git Flow（适合发布周期固定的项目）:
main ──●──────────●──────────●── (生产)
       │          │          │
develop ──●──●──●──●──●──●──●── (开发)
           │     │     │
feature/* ─●     │     ●── (功能)
                 │
release/* ───────●── (预发布)

GitHub Flow（适合持续部署）:
main ──●──●──●──●──●──●── (始终可部署)
       │     │     │
feat/* ─●     │     ●── (PR合并)

GitLab Flow（环境分支）:
production ──●──●──●── (生产)
             │     │
staging ─────●─────●── (预生产)
             │     │
main ────────●─────●── (开发)
```

### 语义化版本号规范

```
格式: MAJOR.MINOR.PATCH

MAJOR: 不兼容的API变更
MINOR: 向后兼容的新功能
PATCH: 向后兼容的Bug修复

示例:
1.0.0 → 1.0.1 (Bug修复)
1.0.1 → 1.1.0 (新功能)
1.1.0 → 2.0.0 (破坏性变更)

预发布标签:
2.0.0-alpha.1 → 2.0.0-beta.1 → 2.0.0-rc.1 → 2.0.0
```

## 八、Conventional Commits规范

```
格式: <type>(<scope>): <description>

type类型:
  feat:     新功能
  fix:      Bug修复
  docs:     文档变更
  style:    代码格式（不影响功能）
  refactor: 重构
  perf:     性能优化
  test:     测试相关
  chore:    构建/工具变更

示例:
feat(auth): add JWT token refresh
fix(api): handle null response from user endpoint
docs(readme): update installation instructions
ci(github): add Node.js 20 to matrix build
```

### 提交钩子配置（husky + commitlint）

```json
// package.json
{
  "husky": {
    "hooks": {
      "commit-msg": "commitlint -E HUSKY_GIT_PARAMS",
      "pre-commit": "lint-staged"
    }
  },
  "lint-staged": {
    "*.{js,ts}": ["eslint --fix", "prettier --write"],
    "*.{json,md}": ["prettier --write"]
  }
}
```

```javascript
// commitlint.config.js
module.exports = {
  extends: ['@commitlint/config-conventional'],
  rules: {
    'type-enum': [2, 'always', [
      'feat', 'fix', 'docs', 'style', 'refactor',
      'perf', 'test', 'chore', 'ci', 'revert'
    ]],
    'subject-max-length': [2, 'always', 100]
  }
};
```
