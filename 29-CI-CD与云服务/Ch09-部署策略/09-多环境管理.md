# 多环境管理

## 一、概念说明

多环境管理是指对开发、测试、预发布、生产等不同环境的配置、部署和运维进行统一管理。核心目标是确保环境一致性和配置隔离。

| 环境 | 用途 | 特点 |
|------|------|------|
| Dev | 开发调试 | 频繁变更、单人使用 |
| Test/QA | 功能测试 | 独立数据、自动化测试 |
| Staging | 预发布 | 接近生产、压测 |
| Production | 生产 | 高可用、严格变更 |

## 二、具体用法

### 环境配置管理

```yaml
# 配置层次结构
config/
├── base/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
├── overlays/
│   ├── dev/
│   │   ├── kustomization.yaml
│   │   ├── replica-patch.yaml
│   │   └── configmap.yaml
│   ├── staging/
│   │   ├── kustomization.yaml
│   │   ├── replica-patch.yaml
│   │   └── configmap.yaml
│   └── production/
│       ├── kustomization.yaml
│       ├── replica-patch.yaml
│       ├── hpa.yaml
│       └── configmap.yaml
```

### Namespace隔离

```bash
# 创建环境Namespace
kubectl create namespace dev
kubectl create namespace staging
kubectl create namespace production

# 设置ResourceQuota
kubectl apply -f - <<EOF
apiVersion: v1
kind: ResourceQuota
metadata:
  name: dev-quota
  namespace: dev
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"
EOF

# 设置LimitRange
kubectl apply -f - <<EOF
apiVersion: v1
kind: LimitRange
metadata:
  name: dev-limits
  namespace: dev
spec:
  limits:
  - default:
      cpu: "500m"
      memory: "512Mi"
    defaultRequest:
      cpu: "100m"
      memory: "128Mi"
    type: Container
EOF
```

### 多环境Pipeline

```yaml
# GitHub Actions多环境部署
name: Deploy to Multiple Environments
on:
  push:
    branches: [main, develop]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build and Push
        run: |
          docker build -t myregistry/web-app:${{ github.sha }} .
          docker push myregistry/web-app:${{ github.sha }}

  deploy-dev:
    needs: build
    if: github.ref == 'refs/heads/develop'
    environment: development
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Dev
        run: |
          kubectl config use-context dev-cluster
          kustomize build overlays/dev | kubectl apply -f -

  deploy-staging:
    needs: build
    if: github.ref == 'refs/heads/main'
    environment: staging
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Staging
        run: |
          kubectl config use-context staging-cluster
          kustomize build overlays/staging | kubectl apply -f -

  deploy-prod:
    needs: deploy-staging
    environment: production
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to Production
        run: |
          kubectl config use-context prod-cluster
          kustomize build overlays/production | kubectl apply -f -
```

### 环境差异管理

```yaml
# overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: dev
bases:
  - ../../base
replicas:
  - name: web-app
    count: 1
patchesStrategicMerge:
  - deployment-patch.yaml
configMapGenerator:
  - name: app-config
    literals:
      - LOG_LEVEL=debug
      - DB_POOL_SIZE=5
      - ENABLE_DEBUG=true

# overlays/production/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
namespace: production
bases:
  - ../../base
replicas:
  - name: web-app
    count: 5
patchesStrategicMerge:
  - deployment-patch.yaml
  - hpa.yaml
configMapGenerator:
  - name: app-config
    literals:
      - LOG_LEVEL=warn
      - DB_POOL_SIZE=50
      - ENABLE_DEBUG=false
```

### Secrets管理

```bash
# 不同环境的Secrets
kubectl create secret generic db-secret \
    --namespace=dev \
    --from-literal=host=dev-db.internal \
    --from-literal=password=dev-pass

kubectl create secret generic db-secret \
    --namespace=production \
    --from-literal=host=prod-db.internal \
    --from-literal=password=strong-prod-pass

# 使用外部Secret管理（Sealed Secrets / External Secrets）
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-secret
  namespace: production
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: db-secret
  data:
  - secretKey: host
    remoteRef:
      key: production/db
      property: host
  - secretKey: password
    remoteRef:
      key: production/db
      property: password
```

## 三、注意事项与常见陷阱

1. **配置分离**：配置和代码分离，敏感信息用Secrets管理
2. **环境对等**：Staging尽量与Production保持一致
3. **资源隔离**：不同环境使用不同Namespace甚至不同集群
4. **数据隔离**：测试环境不要使用生产数据
5. **命名规范**：资源命名包含环境标识
6. **权限控制**：开发人员不要有生产环境的写权限
7. **配置漂移**：定期检查环境配置是否一致
