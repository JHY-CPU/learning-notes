# 部署最佳实践

## 一、概念说明

部署最佳实践涵盖从代码构建到生产发布的全流程规范，目标是实现安全、可靠、高效的软件交付。

| 阶段 | 关注点 | 核心实践 |
|------|--------|----------|
| 构建 | 可重复性 | 容器化、镜像扫描 |
| 测试 | 质量门禁 | 自动化测试、覆盖率 |
| 部署 | 安全性 | 渐进式发布、回滚 |
| 运维 | 可观测性 | 监控、告警、日志 |

## 二、具体用法

### 部署Checklist


### 发布流程自动化

```yaml
# 完整的CI/CD Pipeline
name: Production Release
on:
  release:
    types: [published]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: |
          npm ci
          npm run test:unit
          npm run test:integration
          npm run test:e2e

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Security scan
        run: |
          trivy fs --exit-code 1 .
          npm audit --audit-level=high

  build:
    needs: [test, security]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build and push
        run: |
          docker build -t myregistry/web-app:${{ github.event.release.tag_name }} .
          trivy image --exit-code 1 myregistry/web-app:${{ github.event.release.tag_name }}
          docker push myregistry/web-app:${{ github.event.release.tag_name }}

  deploy-staging:
    needs: build
    environment: staging
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        run: |
          kubectl set image deployment/web-app web=myregistry/web-app:${{ github.event.release.tag_name }} -n staging
          kubectl rollout status deployment/web-app -n staging --timeout=300s

  e2e-staging:
    needs: deploy-staging
    runs-on: ubuntu-latest
    steps:
      - name: Run E2E tests on staging
        run: npm run test:e2e:staging

  deploy-production:
    needs: e2e-staging
    environment: production
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to production (Canary)
        run: |
          kubectl set image deployment/web-app-canary web=myregistry/web-app:${{ github.event.release.tag_name }} -n production
          kubectl rollout status deployment/web-app-canary -n production
      - name: Monitor canary (10 min)
        run: |
          sleep 600
          # 验证指标
          ERROR_RATE=$(curl -s "http://prometheus:9090/api/v1/query?query=error_rate")
          if [ "$ERROR_RATE" > "0.05" ]; then
            echo "Error rate too high, rolling back"
            kubectl rollout undo deployment/web-app-canary -n production
            exit 1
          fi
      - name: Full rollout
        run: |
          kubectl set image deployment/web-app web=myregistry/web-app:${{ github.event.release.tag_name }} -n production
          kubectl rollout status deployment/web-app -n production
```

### 回滚策略

```yaml
# 自动回滚条件
Rollback_Triggers:
  错误率: ">5% 持续2分钟"
  P99延迟: ">3秒 持续5分钟"
  健康检查: "连续3次失败"
  业务指标: "订单量下降>20%"

# 回滚流程
Rollback_Process:
  1. 检测到异常指标
  2. 自动暂停金丝雀放量
  3. 回滚到上一稳定版本
  4. 通知相关人员
  5. 记录故障报告
  6. 复盘改进
```

### 发布日规范

```yaml
发布日流程:
  发布前:
    - 14:00 - 发布准备会议
    - 14:30 - 开始部署Staging
    - 15:00 - Staging验证通过
    - 15:30 - 开始生产金丝雀部署

  发布中:
    - 16:00 - 金丝雀10%验证
    - 16:30 - 金丝雀50%验证
    - 17:00 - 全量发布
    - 17:30 - 监控确认正常

  发布后:
    - 发布成功 → 通知团队
    - 发布异常 → 立即回滚
    - 次日 → 发布复盘
```

## 三、注意事项与常见陷阱

1. **发布窗口**：选择业务低峰期发布，避开重要活动
2. **变更冻结**：重要节假日前冻结变更
3. **一键回滚**：确保回滚操作简单快速
4. **多人确认**：生产变更需要至少两人确认
5. **文档同步**：部署变更同步更新文档
6. **复盘机制**：每次发布异常都要进行复盘
7. **持续改进**：定期优化部署流程和工具
