# 部署策略概述

## 一、概念说明

部署策略定义了如何将新版本的应用发布到生产环境。好的部署策略能平衡发布速度与系统稳定性，降低发布风险。

| 策略 | 风险 | 停机 | 回滚速度 | 资源成本 |
|------|------|------|----------|----------|
| 直接部署 | 高 | 有 | 慢 | 低 |
| 滚动更新 | 中 | 无 | 中 | 低 |
| 蓝绿部署 | 低 | 无 | 快 | 高 |
| 金丝雀 | 最低 | 无 | 快 | 中 |
| A/B测试 | 低 | 无 | 快 | 中 |

## 二、具体用法

### 策略选择流程

```yaml
选择指南:
  小型项目/开发环境:
    推荐: 直接部署/滚动更新
    理由: 简单高效，对停机不敏感

  中型项目/生产环境:
    推荐: 滚动更新
    理由: 平衡风险和成本

  大型项目/关键业务:
    推荐: 蓝绿部署/金丝雀
    理由: 最小化发布风险

  需要验证新功能:
    推荐: 金丝雀/A/B测试
    理由: 可控制流量比例逐步放量
```

### Kubernetes原生部署

```yaml
# Deployment支持的策略
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
spec:
  replicas: 4
  strategy:
    # 滚动更新（默认）
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多出的Pod数
      maxUnavailable: 1  # 最多不可用的Pod数
    # 或重建部署
    # type: Recreate
```

### 部署健康检查

```yaml
# 就绪门控
spec:
  minReadySeconds: 30  # Pod就绪后至少等30秒
  progressDeadlineSeconds: 600  # 部署超时时间
  revisionHistoryLimit: 5  # 保留的旧版本数
```

### 回滚机制

```bash
# 查看部署历史
kubectl rollout history deployment/web-app

# 回滚到上一版本
kubectl rollout undo deployment/web-app

# 回滚到指定版本
kubectl rollout undo deployment/web-app --to-revision=2

# 查看部署状态
kubectl rollout status deployment/web-app

# 暂停部署（用于多次更新后一起发布）
kubectl rollout pause deployment/web-app
# ... 执行多次更新 ...
kubectl rollout resume deployment/web-app
```

## 三、注意事项与常见陷阱

1. **环境一致性**：确保开发、测试、生产环境配置一致
2. **数据库迁移**：数据库变更要向后兼容，先于应用部署
3. **特性开关**：使用Feature Flag控制新功能的灰度发布
4. **监控就绪**：部署前确保监控和告警配置到位
5. **回滚计划**：每个部署都要有明确的回滚计划
6. **发布窗口**：选择业务低峰期进行部署
7. **团队沟通**：部署前通知相关团队，发布后确认无异常
