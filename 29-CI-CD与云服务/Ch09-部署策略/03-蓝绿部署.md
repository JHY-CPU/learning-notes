# 蓝绿部署

## 一、概念说明

蓝绿部署通过维护两套完全相同的环境（蓝/绿），实现零停机和即时回滚。新版本部署到非活跃环境，验证后切换流量。

| 阶段 | 蓝环境 | 绿环境 |
|------|--------|--------|
| 初始 | 生产流量（活跃） | 空闲 |
| 部署 | 生产流量 | 部署新版本 |
| 验证 | 生产流量 | 测试新版本 |
| 切换 | 空闲 | 生产流量（活跃） |

## 二、具体用法

### Kubernetes蓝绿部署

```yaml
# 蓝版本（当前生产）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app-blue
  labels:
    app: web-app
    version: blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
      version: blue
  template:
    metadata:
      labels:
        app: web-app
        version: blue
    spec:
      containers:
      - name: web
        image: myregistry/web-app:1.0
        ports:
        - containerPort: 8080

---
# 绿版本（新版本）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app-green
  labels:
    app: web-app
    version: green
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-app
      version: green
  template:
    metadata:
      labels:
        app: web-app
        version: green
    spec:
      containers:
      - name: web
        image: myregistry/web-app:2.0
        ports:
        - containerPort: 8080

---
# Service - 通过selector切换版本
apiVersion: v1
kind: Service
metadata:
  name: web-app-svc
spec:
  selector:
    app: web-app
    version: blue  # 切换时改为 green
  ports:
  - port: 80
    targetPort: 8080
  type: LoadBalancer
```

### 自动化切换脚本

```bash
#!/bin/bash
# blue-green-deploy.sh

APP_NAME="web-app"
NEW_VERSION="2.0"
CURRENT_VERSION=$(kubectl get svc ${APP_NAME}-svc -o jsonpath='{.spec.selector.version}')

# 确定目标颜色
if [ "$CURRENT_VERSION" = "blue" ]; then
    TARGET_COLOR="green"
    OLD_COLOR="blue"
else
    TARGET_COLOR="blue"
    OLD_COLOR="green"
fi

echo "当前版本: ${OLD_COLOR}, 部署目标: ${TARGET_COLOR}"

# 部署新版本
kubectl set image deployment/${APP_NAME}-${TARGET_COLOR} web=myregistry/${APP_NAME}:${NEW_VERSION}
kubectl rollout status deployment/${APP_NAME}-${TARGET_COLOR}

# 验证新版本健康
echo "等待新版本就绪..."
kubectl wait --for=condition=ready pod -l app=${APP_NAME},version=${TARGET_COLOR} --timeout=300s

# 切换流量
echo "切换流量到 ${TARGET_COLOR}..."
kubectl patch svc ${APP_NAME}-svc -p "{\"spec\":{\"selector\":{\"version\":\"${TARGET_COLOR}\"}}}"

# 等待验证
echo "新版本已上线，等待60秒验证..."
sleep 60

# 验证通过后缩容旧版本（保留以便快速回滚）
echo "缩容旧版本到0..."
kubectl scale deployment/${APP_NAME}-${OLD_COLOR} --replicas=0

echo "部署完成！当前版本: ${TARGET_COLOR}"
echo "如需回滚: kubectl patch svc ${APP_NAME}-svc -p '{\"spec\":{\"selector\":{\"version\":\"${OLD_COLOR}\"}}}'"
```

### Docker Compose蓝绿部署

```yaml
# docker-compose.yml
version: '3.8'
services:
  app-blue:
    image: myapp:1.0
    networks:
      - app-network

  app-green:
    image: myapp:2.0
    networks:
      - app-network

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
    depends_on:
      - app-blue
      - app-green
```

```nginx
# nginx.conf
upstream backend {
    # 切换时修改这一行: blue -> green
    server app-blue:8080;
}

server {
    listen 80;
    location / {
        proxy_pass http://backend;
    }
}
```

## 三、注意事项与常见陷阱

1. **资源成本**：需要双倍的计算资源
2. **数据库兼容**：新旧版本可能同时处理请求，确保数据兼容
3. **会话状态**：有状态应用需要考虑Session迁移
4. **验证充分**：切换前必须在绿环境充分测试
5. **快速回滚**：切换后保留蓝环境以便回滚
6. **缓存预热**：新环境的缓存需要预热
7. **监控双环境**：切换后密切监控新环境指标
