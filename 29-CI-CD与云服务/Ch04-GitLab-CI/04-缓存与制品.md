# 缓存与制品

## 一、缓存配置

```yaml
# 全局缓存
cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/
    - .cache/

# Job缓存
build:
  cache:
    key: build-cache
    paths:
      - node_modules/
```

## 二、制品配置

```yaml
build:
  artifacts:
    paths:
      - dist/
    expire_in: 1 week
    reports:
      junit: test-results.xml

test:
  dependencies:
    - build
  script:
    - ls dist/
```

## 三、缓存vs制品

```bash
# 缓存（cache）
# 在Job之间共享文件
# 加速依赖下载
# 可能被覆盖

# 制品（artifacts）
# 在Stage之间传递文件
# 保存构建产物
# 可下载查看
```

## 四、最佳实践

1. **缓存key设计**：使用`${CI_COMMIT_REF_SLUG}`按分支隔离缓存，避免不同分支缓存互相污染
2. **缓存策略选择**：依赖下载用`pull-push`（只拉不推），共享中间结果用`push-pull`（只推不拉）
3. **制品精简**：只保留必要产物，设置合理的expire_in，避免存储费用飙升
4. **缓存锁定**：关键依赖缓存设置`unprotect: true`防止被意外覆盖

## 五、常见陷阱

1. **缓存未命中**：缓存key设置不当（如使用commit SHA）导致每次都未命中，等于没有缓存
2. **缓存污染**：全局缓存key导致不同项目的node_modules互相污染
3. **制品过大**：未限制制品大小导致存储费用异常，应在settings中配置制品大小上限
4. **缓存竞争**：多个Pipeline并行运行时，后完成的Pipeline可能覆盖前者的缓存，使用`when: on_success`控制

## 六、缓存策略详解

```yaml
# 默认缓存策略: pull-push (拉取并推送)
# 适用于: 每次构建都更新的缓存

# push-pull (推送并拉取)
# 适用于: 不同Job共享缓存
build:
  cache:
    key: build
    paths:
      - dist/
    policy: push  # 只推送

test:
  needs: [build]
  cache:
    key: build
    paths:
      - dist/
    policy: pull  # 只拉取

# pull-push (拉取并推送) - 默认
# 适用于: 同一Job内更新缓存

# pull (仅拉取)
# 适用于: 不需要更新缓存的Job
lint:
  cache:
    key: deps
    paths:
      - node_modules/
    policy: pull
```

## 七、制品报告类型

```yaml
test:
  script:
    - npm test -- --reporter=junit --output-file=results.xml
    - npm run test:coverage
  artifacts:
    paths:
      - coverage/
    reports:
      junit: results.xml                    # 测试报告
      coverage_report:
        coverage_format: cobertura          # 覆盖率报告
        path: coverage/cobertura.xml
      codequality: gl-code-quality.json     # 代码质量
      sast: gl-sast-report.json             # SAST报告
      dast: gl-dast-report.json             # DAST报告
      dependency_scanning: gl-dependency-scanning-report.json
      container_scanning: gl-container-scanning-report.json
      secret_detection: gl-secret-detection-report.json
    expire_in: 30 days
  coverage: '/Lines\s*:\s*(\d+\.?\d*)%/'
```

## 八、制品依赖传递

```yaml
# 使用needs指定制品依赖
build:
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  needs:
    - job: build
      artifacts: true
  script:
    - ls dist/
    - npm test

deploy:
  stage: deploy
  needs:
    - job: build
      artifacts: true
    - job: test
      artifacts: false  # 不需要测试制品
  script:
    - cp -r dist/ /deploy/path/
```

## 九、S3/GCS制品存储

```yaml
# GitLab SaaS使用内置存储
# 自托管GitLab可以配置外部存储

# 使用脚本上传到S3
upload-s3:
  stage: deploy
  script:
    - aws s3 sync dist/ s3://my-bucket/releases/${CI_COMMIT_SHA}/
  environment:
    name: production
```

```toml
# /etc/gitlab/gitlab.rb - 外部制品存储
gitlab_rails['artifacts_object_store_enabled'] = true
gitlab_rails['artifacts_object_store_remote_directory'] = "gitlab-artifacts"
gitlab_rails['artifacts_object_store_connection'] = {
  'provider' => 'AWS',
  'region' => 'us-east-1',
  'aws_access_key_id' => 'YOUR_KEY',
  'aws_secret_access_key' => 'YOUR_SECRET'
}
```
