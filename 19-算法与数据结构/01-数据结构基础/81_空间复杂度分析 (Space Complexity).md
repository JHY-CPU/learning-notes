# Space Complexity

### 空间复杂度分析

空间复杂度衡量算法运行过程中所需的额外存储空间（不含输入数据本身）。不同数据结构在存储相同数量元素时的空间开销差异显著。

### 关键特性

- **指针开销**：链表、树每个节点额外存储指针（64 位系统中每个指针 8 字节）
- **桶开销**：哈希表需要预分配桶数组，负载因子低时浪费空间
- **稀疏存储**：稀疏数组、压缩结构可以大幅节省空间
- **时间-空间权衡**：预计算（如前缀和）用空间换时间

### 空间复杂度对比

| 数据结构 | 空间 | 每元素额外开销 |
|---------|------|--------------|
| 数组 | O(n) | 几乎无 |
| 链表 | O(n) | 每节点 1 指针 |
| 双向链表 | O(n) | 每节点 2 指针 |
| 哈希表 | O(n) | 桶数组 + 链表指针 |
| BST | O(n) | 每节点 2 指针 |
| 布隆过滤器 | O(m) | 固定 m 位，与 n 无关 |

### 适用场景 vs 替代方案

- **内存紧张**：用数组而非链表，用位图而非集合
- **时间优先**：用额外空间做缓存或预计算
- **大数据**：外存数据结构（B 树、LSM 树）
- **近似存储**：布隆过滤器用少量空间近似集合

### 常见陷阱

- 只看渐进复杂度忽略常数：哈希表的 O(n) 实际比数组的 O(n) 大很多
- 递归调用的隐式栈空间未计入
- JavaScript 垃圾回收延迟导致峰值内存远超实际需要

```
// 空间复杂度对比
// 数组: O(n) - 连续存储
// 链表: O(n) - 每个节点额外存储指针
// 哈希表: O(n) - 存储键值对+桶
// 二叉树: O(n) - 每个节点存储左右指针
// 图(邻接矩阵): O(V²)
// 图(邻接表): O(V+E)
// 并查集: O(n)
// 布隆过滤器: O(k) - 固定空间，与n无关
function estimateMemory(n, type) {
  const sizes = { array: n * 8, linkedList: n * 24, hashTable: n * 40, tree: n * 32, graphMatrix: n * n * 8, graphList: n * 24 };
  return sizes[type] || '未知';
}
console.log(`100万元素数组约 ${estimateMemory(1000000, 'array') / 1024 / 1024} MB`);
```


### 实际应用

- **嵌入式开发**：内存严格受限，必须精确计算每个字节
- **大规模数据处理**：选择内存高效的结构处理海量数据
- **前端优化**：减少 DOM 节点和 JavaScript 对象的内存占用
- **数据库设计**：根据内存预算选择合适的索引结构
