# 平衡树对比 (Balanced BST Comparison)

## 1. 概述

本节对比常见的自平衡二叉搜索树：AVL树、红黑树、Treap、Splay树、替罪羊树，帮助在不同场景下选择合适的数据结构。

## 2. 综合对比表

| 数据结构 | 查找 | 插入 | 删除 | 旋转/重构 | 是否需要额外信息 |
|----------|------|------|------|----------|----------------|
| AVL树 | O(log n) | O(log n) | O(log n) | 插入1次，删除O(log n)次 | height |
| 红黑树 | O(log n) | O(log n) | O(log n) | 插入2次，删除3次 | color (1bit) |
| Treap | O(log n) | O(log n) | O(log n) | 期望O(1) | priority |
| Splay树 | O(log n) 均摊 | O(log n) 均摊 | O(log n) 均摊 | splay操作 | 无 |
| 替罪羊树 | O(log n) | O(log n) 均摊 | O(log n) 均摊 | 全树重建 | subtree size |

## 3. 平衡程度对比

```
平衡严格程度（从严格到宽松）：

AVL树 > 红黑树 > 替罪羊树 > Treap > Splay树
  |        |         |         |       |
高度差<=1  路径差<=2  α-平衡   随机平衡  无保证
```

### 3.1 树高对比

| 数据结构 | 最坏树高 | 平均树高 |
|----------|---------|---------|
| AVL树 | 1.44 log n | ~log n |
| 红黑树 | 2 log n | ~1.1 log n |
| Treap | O(n) 期望log n | ~1.39 log n |
| Splay树 | O(n) | ~log n 均摊 |
| 替罪羊树 | log n / log(1/alpha) | ~log n |

## 4. AVL树 vs 红黑树

### 4.1 核心区别

| 维度 | AVL树 | 红黑树 |
|------|-------|--------|
| 平衡条件 | 严格（高度差<=1） | 宽松（路径差<=2） |
| 查找 | 更快（树更矮） | 略慢 |
| 插入旋转 | 最多1次 | 最多2次 |
| 删除旋转 | 最多O(log n)次 | 最多3次 |
| 额外空间 | height (int) | color (1 bit) |
| 适用 | 查找密集型 | 插入删除密集型 |

### 4.2 使用场景选择

- **选AVL树**：查找操作远多于修改，需要稳定的最坏性能
- **选红黑树**：插入删除操作较多，需要综合性能平衡

## 5. Treap (随机平衡)

### 5.1 核心思想

Treap = Tree + Heap。每个节点有两个属性：
- key：满足BST性质
- priority：满足堆性质（通常是随机值）

随机优先级保证了树的期望高度为 O(log n)。

### 5.2 优缺点

| 优点 | 缺点 |
|------|------|
| 实现简单 | 随机性依赖 |
| 期望性能好 | 最坏情况O(n)（概率极小） |
| 无需旋转维护 | 不适合实时系统 |
| 支持分裂合并 | 额外存储priority |

## 6. Splay树（均摊平衡）

### 6.1 核心思想

每次访问节点后，通过splay操作将其旋转到根。不保证最坏单次复杂度，但保证均摊 O(log n)。

### 6.2 优缺点

| 优点 | 缺点 |
|------|------|
| 无需存储平衡信息 | 单次操作可能O(n) |
| 自适应（频繁访问的更快） | 不适合随机访问 |
| 实现相对简单 | 常数较大 |
| 支持区间操作 | |

### 6.3 时间复杂度

| 操作 | 最坏 | 均摊 |
|------|------|------|
| 查找 | O(n) | O(log n) |
| 插入 | O(n) | O(log n) |
| 删除 | O(n) | O(log n) |

## 7. 替罪羊树（重建平衡）

### 7.1 核心思想

不使用旋转，而是在某个子树"不平衡"时，将整棵子树重建为完全二叉树。参数 alpha 控制平衡的严格程度。

### 7.2 优缺点

| 优点 | 缺点 |
|------|------|
| 实现简单（只需重建） | 重建时有O(n)的暂停 |
| 不需要旋转 | 均摊而非最坏 |
| alpha参数可调 | 不适合实时性要求高的场景 |

## 8. 选择指南

### 8.1 按场景选择

| 场景 | 推荐 | 原因 |
|------|------|------|
| 通用有序容器 | 红黑树 | 综合性能好 |
| 查找密集 | AVL树 | 树更矮，查找更快 |
| 竞赛算法 | Treap/Splay | 实现灵活 |
| 区间操作 | Splay树 | 天然支持split/merge |
| 简单实现 | 替罪羊树 | 只需重建 |
| 工程实践 | 红黑树 | 成熟稳定 |

### 8.2 语言标准库使用

| 语言 | 底层结构 |
|------|---------|
| C++ STL | 红黑树 |
| Java TreeMap/TreeSet | 红黑树 |
| Python sortedcontainers | 不固定 |

## 9. 性能实测参考

在 n=10^6 的随机数据下的大致耗时（相对值）：

| 操作 | AVL树 | 红黑树 | Treap | Splay树 |
|------|-------|--------|-------|---------|
| 随机插入 | 1.0x | 0.9x | 1.1x | 1.2x |
| 随机查找 | 0.9x | 1.0x | 1.1x | 1.3x |
| 随机删除 | 1.0x | 0.9x | 1.1x | 1.2x |
| 顺序插入 | 1.0x | 0.9x | 1.0x | 2.0x |
| 局部访问 | 1.0x | 1.0x | 1.0x | 0.5x |

## 10. 总结

没有"最好"的平衡树，只有"最适合"的：
- **追求稳定性能**：AVL树
- **追求综合性能**：红黑树
- **追求实现简单**：Treap或替罪羊树
- **追求自适应性**：Splay树
- **工程实践**：红黑树（成熟、有标准库支持）
