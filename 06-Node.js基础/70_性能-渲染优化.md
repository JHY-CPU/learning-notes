## 渲染优化 API


```
// ========== 渲染流水线 ==========
// JavaScript → Style → Layout → Paint → Composite
//           重排↑        重绘↑   合成↑

// ========== 触发重排 (Layout) ==========
// 读取/修改几何属性: width/height/top/left/margin/padding
// 读取布局信息: offsetHeight/clientHeight/getBoundingClientRect

// ========== 触发重绘 (Paint) ==========
// color, background, visibility, outline

// ========== 仅触发合成 (Composite) ==========
// transform, opacity
// ✅ 推荐动画属性

// ========== GPU 加速 ==========
will-change: transform;  // 提示浏览器优化
transform: translateZ(0); // 强制 GPU 层
```

## 渲染流水线

浏览器渲染过程：JS执行 → 样式计算(Style) → 布局(Layout) → 绘制(Paint) → 合成(Composite)。越靠后的阶段开销越小。

## 各阶段触发条件

- **重排（Layout）**：修改宽高、位置、字体大小等几何属性；读取 `offsetHeight`、`getBoundingClientRect` 等也会强制同步重排
- **重绘（Paint）**：修改颜色、背景、阴影等不影响布局的属性
- **合成（Composite）**：仅 `transform` 和 `opacity`，最高效，可被GPU加速

## 优化策略

1. **用 transform 替代 top/left**：避免触发Layout
2. **批量DOM操作**：使用 `DocumentFragment` 或离线DOM（`display:none` 操作后再显示）
3. **避免强制同步布局**：不要在修改DOM后立即读取布局属性
4. **使用 `will-change`**：提前告知浏览器将要变化的属性，创建合成层
5. **`contain` 属性**：限制重排/重绘范围到元素内部
6. **长任务拆分**：用 `requestAnimationFrame` 或 `requestIdleCallback` 分帧处理

## GPU 加速

`transform: translateZ(0)` 或 `will-change: transform` 可将元素提升为独立的合成层，由GPU处理。但每层都消耗内存，不宜滥用。
