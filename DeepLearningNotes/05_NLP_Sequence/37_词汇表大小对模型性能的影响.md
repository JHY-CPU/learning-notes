# 37_词汇表大小对模型性能的影响

## 核心概念
- **词汇表大小 (Vocabulary Size)**：Token 词汇的总数量，直接影响模型的嵌入层参数量、计算效率和语言覆盖能力。
- **嵌入层参数量**：词嵌入矩阵的大小为 $V \times d$，其中 $V$ 是词汇表大小，$d$ 是嵌入维度。在 BERT-base（$V=30K$, $d=768$）中，嵌入层占约 23M 参数（总参数 110M 的 21%）。
- **词汇表大小的权衡**：$V$ 越大，每个 token 的信息密度越高（一个词不需要拆成多个子词），但嵌入层参数增加，计算量（softmax 输出层）也增大；$V$ 越小，子词拆分越细，序列变长，计算效率降低。
- **序列长度的影响**：词汇表越小，平均 token 序列越长（因为子词拆分更细）。例如同一条句子，字符级 tokenizer 的序列长度是 BPE 的 2-3 倍，导致 Transformer 的 $O(n^2)$ 计算代价显著增加。
- **OOV 覆盖率**：词汇表越大，未登录词越少。但理论上 BPE 等子词方法已经解决了 OOV 问题（通过字符回退），所以词汇表大小对 OOV 的影响变弱。
- **浮点运算量的分配**：输出层 $V \times d$ 的矩阵乘法在每一步都发生。当 $V$ 很大（如 250K）时，输出层的计算可能占总 FLOPs 的 30% 以上。
- **多语言词汇表的挑战**：多语言模型中，词汇表需要覆盖多种语言，通常需要更大的词汇表（如 mBERT 的 110K、XLM-R 的 250K）。

## 数学推导
嵌入层参数量：$P_{\text{embed}} = V \times d$

输出层参数量（未共享权重）：$P_{\text{out}} = d \times V$

总参数量简化模型：
$$
P_{\text{total}} = \underbrace{V \times d}_{\text{embed}} + \underbrace{L \times (4d^2 + O(d))}_{\text{Transformer}} + \underbrace{d \times V}_{\text{output}}
$$

其中 $L$ 是层数。

**序列长度与计算量的关系**：假设平均 token 数为 $n$，词汇表 $V$ 下句子被拆分成的 token 数 $n \propto 1/\text{avg\_token\_length}$。注意力复杂度为 $O(n^2 d)$，所以 $V$ 减半导致 token 序列长度近似翻倍，注意力计算量翻 4 倍。

**最优词汇表大小的经验法则**：
$$
V_{\text{opt}} \approx \sqrt{\frac{2 \cdot n_{\text{params}}}{d}}
$$

但实际中更多由实验和经验确定（常见 32K-128K）。

## 直观理解
- **词汇表大小像词典的厚度**：薄词典（小词汇表）每个词需要拆解成字母或偏旁来解释（子词拆分多），查词慢（序列长）。厚词典（大词汇表）包含更多完整的词，但携带起来更重（参数多）。
- **中文的特殊性**：中文的"字"和"词"界限模糊。"机器学习"是一个词（大词汇表时可以作为一个 token），也可能被拆成"机器"和"学习"（中等词汇表）或"机"、"器"、"学"、"习"（小词汇表）。大词汇表对中文更友好。
- **极端情况分析**：如果词汇表只有 256（字节级），每个句子会变得非常长（每个汉字拆成 3 个字节），注意力计算量爆炸。如果词汇表有 500K，嵌入层参数为 $500K \times 4096 \approx 2B$，过于庞大。32K-128K 是实践中证明的"甜区"。

## 代码示例
```python
import torch
import math

# 分析不同词汇表大小的影响
def analyze_vocab_size(vocab_size, embed_dim=768, seq_len=128):
    embed_params = vocab_size * embed_dim
    output_params = embed_dim * vocab_size
    # 注意力参数
    n_layers = 12
    n_heads = 12
    d_k = embed_dim // n_heads
    attn_params = n_layers * 4 * embed_dim * embed_dim  # Q,K,V,O
    ffn_params = n_layers * 2 * embed_dim * (4 * embed_dim)  # FFN
    transformer_params = attn_params + ffn_params

    total_params = embed_params + transformer_params + output_params
    return {
        "vocab_size": vocab_size,
        "embed_params": embed_params / 1e6,
        "output_params": output_params / 1e6,
        "transformer_params": transformer_params / 1e6,
        "total_params": total_params / 1e6,
        "attn_cost_MACs": seq_len ** 2 * d_k * n_heads * n_layers / 1e6,
    }

vocab_sizes = [8000, 16000, 32000, 64000, 128000, 256000]
print(f"{'V':>8} {'嵌入(M)':>10} {'输出(M)':>10} {'Transformer(M)':>16} {'总计(M)':>10} {'注意力MACs(M)':>14}")
for vs in vocab_sizes:
    info = analyze_vocab_size(vs)
    print(f"{vs:>8} {info['embed_params']:>10.1f} {info['output_params']:>10.1f} "
          f"{info['transformer_params']:>16.1f} {info['total_params']:>10.1f} "
          f"{info['attn_cost_MACs']:>14.1f}")

# 实际使用不同 tokenizer 查看序列长度差异
from transformers import AutoTokenizer

text = "Transformers have revolutionized natural language processing."

names_and_toks = [
    ("GPT-2 (BPE, V=50K)", "gpt2"),
    ("BERT (WordPiece, V=30K)", "bert-base-uncased"),
    ("XLM-R (Unigram, V=250K)", "xlm-roberta-base"),
]

for name, tokenizer_name in names_and_toks:
    tok = AutoTokenizer.from_pretrained(tokenizer_name)
    tokens = tok.encode(text)
    print(f"{name}: {len(tokens)} tokens")
```

## 深度学习关联
- **词嵌入与模型容量**：词汇表大小是模型架构设计的基础决策之一。它影响嵌入层参数量，进而影响总体模型容量。更大的词汇表需要更多数据来充分学习。
- **多语言模型的特例**：XLM-R 使用 250K 词汇表以覆盖 100 种语言，但其嵌入层参数量（$250K \times 1024 \approx 256M$）已占模型总参数的 20% 以上。为解决此问题，ALBERT 提出因式分解嵌入。
- **训练-推理效率平衡**：小的词汇表使序列变长，注意力计算随序列长度平方增长。在生产部署中，需要在词汇表大小（影响嵌入层大小）和序列长度（影响注意力计算）之间做权衡。这一权衡影响了 T5、PaLM 等模型的词汇表设计。
