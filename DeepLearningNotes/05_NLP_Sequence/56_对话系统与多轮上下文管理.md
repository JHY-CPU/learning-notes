# 56_对话系统与多轮上下文管理

## 核心概念
- **对话系统 (Dialogue System)**：能与人进行自然语言对话的计算机系统。分为任务型（Task-oriented，如客服、订票）和开放域/闲聊型（Chat-oriented，如 ChatGPT、闲聊机器人）。
- **多轮对话管理 (Multi-turn Dialogue Management)**：跟踪和管理多轮对话中的上下文信息，确保系统理解当前状态并做出恰当响应。核心问题是"如何记住并利用历史信息"。
- **对话状态追踪 (Dialogue State Tracking, DST)**：在任务型对话中，持续追踪用户的目标和系统状态（如订票任务的日期、目的地、人数等槽位值）。
- **对话历史表示**：将历史对话（多轮交互）编码为模型可处理的表示形式。常见做法是将历史对话拼接为长文本（如 "[轮1: 你好] [轮2: 请问需要什么帮助？]..."）。
- **上下文窗口限制**：LLM 的上下文窗口有限（如 GPT-3.5 的 4K、GPT-4 的 32K），长对话需要策略性地截断或压缩历史。
- **滑动窗口式对话管理**：保留最近 N 轮对话作为上下文，抛弃最早的轮次。简单但丢失了早期上下文。
- **摘要式记忆 (Summary Memory)**：使用 LLM 定期对老对话进行摘要压缩，将压缩后的摘要作为长期记忆保留。
- **检索式记忆 (Retrieval Memory)**：将对话历史分块并向量化存储，当新轮次到来时检索与当前 query 最相关的历史片段。类似 RAG 的思路。

## 数学推导
**多轮对话的条件生成**：
$$
P(y_t | x_t, H_{<t}) = \prod_{i=1}^{|y_t|} P(y_{t,i} | y_{t,<i}, x_t, H_{<t})
$$

其中 $x_t$ 是当前轮的用户输入，$H_{<t} = \{(u_1, s_1), (u_2, s_2), \ldots, (u_{t-1}, s_{t-1})\}$ 是前 $t-1$ 轮的对话历史。

**滑动窗口上下文**：
$$
H_{\text{window}} = \{(u_{t-k}, s_{t-k}), \ldots, (u_{t-1}, s_{t-1})\}
$$

其中 $k$ 是窗口大小。

**检索式上下文**：
$$
H_{\text{retrieved}} = \text{top-k}_{q=x_t}(H)
$$

通过计算当前 query 与历史记录的语义相似度，选取最相关的 k 条历史。

## 直观理解
- **多轮对话管理像"短期记忆"**：你和朋友聊天时，会记得刚才说过的话题。多轮对话管理就是让 AI 也拥有这种"我记得你刚才说了什么"的能力。如果没有它，每轮对话都是"失忆后重新开始"。
- **上下文窗口限制像"只能记住最近 N 句话"**：人的短期记忆容量有限（通常 7±2 个信息块），LLM 的上下文窗口也有限。当对话很长时，只能记住最近几轮——但有时关键的"线索"在早期对话中。
- **摘要式记忆像"写日记"**：对话持续一小时（50 轮对话），不能都记住，所以每隔 10 轮就把前面的内容写一段"日记摘要"保存起来。这样虽然记不清每个细节，但整体脉络还在。
- **检索式记忆像"随时查聊天记录"**：不主动记住所有内容，但需要时可以在聊天记录中搜索。当用户说"还记得我上个月说的那个计划吗？"，模型去历史记录中搜索"计划"相关的对话。

## 代码示例
```python
from transformers import AutoTokenizer, AutoModelForCausalLM
import torch

# 模拟多轮对话管理

class DialogueManager:
    """简单的多轮对话管理器"""
    def __init__(self, max_turns=5):
        self.max_turns = max_turns
        self.history = []

    def add_turn(self, user, assistant):
        self.history.append(("user", user))
        self.history.append(("assistant", assistant))
        # 滑动窗口：保留最近 max_turns 轮
        if len(self.history) > self.max_turns * 2:
            self.history = self.history[-(self.max_turns * 2):]

    def get_context(self):
        """格式化历史为字符串"""
        lines = []
        for role, text in self.history:
            prefix = "用户" if role == "user" else "助手"
            lines.append(f"{prefix}: {text}")
        return "\n".join(lines)

    def format_prompt(self, user_input):
        context = self.get_context()
        return f"以下是一个多轮对话：\n\n{context}\n用户: {user_input}\n助手:"

# 模拟对话
manager = DialogueManager(max_turns=3)

# 不需要实际 LLM 调用，展示上下文管理
conversation = [
    "你好！",
    "你好！我是智能助手，有什么可以帮助你的？",
    "我想订一张去北京的机票。",
    "好的，请问您想什么时候出发？",
    "下周一出发。",
    "下周一，也就是 5月7日，请问您需要几点的航班？",
    "上午的航班有吗？",
]

for i in range(0, len(conversation), 2):
    user_msg = conversation[i]
    assistant_msg = conversation[i+1] if i+1 < len(conversation) else ""
    manager.add_turn(user_msg, assistant_msg)
    print(f"\n--- 第 {i//2 + 1} 轮 ---")
    print(f"用户: {user_msg}")
    print(f"助手: {assistant_msg}")
    print(f"当前上下文大小: {len(manager.history)} 条消息")
    print(f"上下文 (最近 {manager.max_turns} 轮):")
    print(manager.get_context())
```

## 深度学习关联
- **ChatGPT 的对话革命**：ChatGPT 展示了基于 GPT-3.5/4 的强大多轮对话能力，通过 RLHF + 指令微调实现了流畅的人类对话。其核心能力是"记住上下文 + 维持一致性"。
- **KV cache 在对话中的作用**：在每轮对话中，系统不需要重新编码之前的对话。通过 KV cache，之前轮次的计算结果被缓存，新轮次只需计算新输入。PagedAttention 等优化显著提升了多轮对话的推理效率。
- **超长上下文的挑战**：对话系统对上下文长度的需求是无限的（理论上可以聊几天几夜）。FlashAttention、YaRN（位置编码扩展）等技术推动着上下文窗口从 4K 向 100K、1M 甚至更高发展。
