# 数据仓库概述

## 什么是数据仓库

数据仓库（Data Warehouse）是一个面向主题的、集成的、相对稳定的、反映历史变化的数据集合，用于支持管理决策。其核心目标是将分散在各个业务系统中的数据整合起来，为分析和决策提供统一的数据视图。

### 数据仓库的四大特征

| 特征 | 英文 | 说明 |
|------|------|------|
| 面向主题 | Subject-Oriented | 按业务主题组织数据，而非按业务系统 |
| 集成 | Integrated | 统一数据编码、命名规范，消除不一致性 |
| 相对稳定 | Non-Volatile | 数据加载后一般不修改和删除 |
| 反映历史 | Time-Variant | 保留历史数据，支持趋势分析 |

## OLTP vs OLAP

### OLTP（联机事务处理）

OLTP系统是面向业务操作的数据库系统，支撑日常业务交易。

**特点**：
- 面向业务操作，支撑高频短事务
- 数据高度规范化（3NF），减少冗余
- 支持高并发的增删改查
- 数据量相对较小，单表百万级
- 代表：MySQL、PostgreSQL、Oracle

### OLAP（联机分析处理）

OLAP系统面向数据分析，支持复杂查询和多维分析。

**特点**：
- 面向分析决策，支撑复杂聚合查询
- 数据反规范化（星型/雪花模型），冗余换取性能
- 以读为主，批量写入
- 数据量大，单表可达数十亿行
- 代表：Hive、ClickHouse、Snowflake、StarRocks

### 对比总结

| 维度 | OLTP | OLAP |
|------|------|------|
| 目的 | 支撑日常业务操作 | 支撑分析决策 |
| 用户 | 业务人员、应用系统 | 数据分析师、管理层 |
| 读写比 | 读写均衡 | 以读为主 |
| 查询复杂度 | 简单CRUD | 复杂聚合、多表关联 |
| 数据模型 | 3NF规范化 | 星型/雪花模型 |
| 数据量 | GB~TB级 | TB~PB级 |
| 响应时间 | 毫秒级 | 秒~分钟级 |

## Inmon vs Kimball 方法论

### Inmon方法论（自顶向下）

Bill Inmon被称为"数据仓库之父"，主张自顶向下的方法论。

**核心思想**：
1. 先建立企业级数据仓库（EDW），采用3NF规范化模型
2. 在EDW基础上构建数据集市（Data Mart）
3. 强调数据的一致性和集成性

```text
         +------------------+
         |    数据集市(DM)   |  ← 面向部门/主题
         +--------+---------+
                  |
         +--------+---------+
         |  企业数据仓库(EDW) |  ← 3NF规范化，全企业数据集成
         +--------+---------+
                  |
    +------+------+------+
    |      |      |      |
  +---+  +---+  +---+  +---+
  |OLTP| |CRM| |ERP| |日志|  ← 各业务系统
  +---+  +---+  +---+  +---+
```

**优点**：数据一致性好，适合大型企业
**缺点**：建设周期长，前期投入大

### Kimball方法论（自底向上）

Ralph Kimball主张自底向上的维度建模方法。

**核心思想**：
1. 以业务过程为中心，通过维度建模构建数据集市
2. 使用一致性维度（Conformed Dimensions）将各数据集市关联
3. 所有数据集市的集合构成数据仓库总线架构

```text
+--------+   +--------+   +--------+
| 交易DM  |   | 流量DM  |   | 用户DM  |
| (事实表) |   | (事实表) |   | (事实表) |
+---+----+   +---+----+   +---+----+
    |            |            |
    +------+-----+------+-----+
           |            |
     +-----+-----+  +--+--+
     |一致性维度   |  |一致性维度 |
     |(时间/用户)  |  |(产品)    |
     +-----------+  +--------+
```

**优点**：迭代交付，见效快，业务驱动
**缺点**：可能产生数据不一致

### 方法论对比

| 维度 | Inmon（自顶向下） | Kimball（自底向上） |
|------|------------------|-------------------|
| 起点 | 企业级数据模型 | 业务过程 |
| 数据模型 | 3NF规范化 | 维度模型（星型/雪花） |
| 交付周期 | 长（年） | 短（周/月） |
| 风险 | 前期投入大 | 数据不一致风险 |
| 适用场景 | 大型企业、强治理 | 敏捷团队、快速迭代 |

## 数据仓库架构分层

现代数据仓库通常采用分层架构，将数据处理过程划分为多个层次。

### 典型分层架构

```text
+--------------------------------------------------+
|  ADS (Application Data Service) - 应用数据层      |
|  面向具体应用的报表、API数据                        |
+--------------------------------------------------+
|  DWS (Data Warehouse Summary) - 数据汇总层        |
|  按主题轻度汇总，形成宽表                          |
+--------------------------------------------------+
|  DWD (Data Warehouse Detail) - 数据明细层          |
|  清洗、规范化后的明细数据                           |
+--------------------------------------------------+
|  DIM (Dimension) - 公共维度层                     |
|  共享维度表：用户、商品、时间等                     |
+--------------------------------------------------+
|  ODS (Operational Data Store) - 操作数据层         |
|  原始数据接入层，保持源系统数据原貌                 |
+--------------------------------------------------+
```

### 各层详细说明

#### ODS层（操作数据层）

```sql
-- ODS层示例：保持源系统数据原貌
CREATE EXTERNAL TABLE ods_order (
    id              BIGINT,
    user_id         BIGINT,
    product_id      BIGINT,
    quantity        INT,
    amount          DECIMAL(10,2),
    status          INT,
    create_time     TIMESTAMP,
    update_time     TIMESTAMP
)
PARTITIONED BY (dt STRING)
STORED AS ORC
LOCATION '/data/ods/order/';
```

**ODS层特点**：
- 数据与源系统保持一致，不做业务逻辑处理
- 通常使用外部表，防止误删原始数据
- 按日期分区存储，支持增量和全量抽取
- 数据保留周期根据业务需求确定（通常90天~无限期）

#### DWD层（数据明细层）

```sql
-- DWD层示例：清洗、规范化后的明细数据
CREATE TABLE dwd_order_detail (
    order_id        BIGINT,
    user_id         BIGINT,
    product_id      BIGINT,
    quantity        INT,
    unit_price      DECIMAL(10,2),
    total_amount    DECIMAL(12,2),
    order_status    STRING,       -- 状态码转为可读文本
    order_time      TIMESTAMP,
    etl_time        TIMESTAMP
)
PARTITIONED BY (dt STRING)
STORED AS ORC;

-- ETL转换：从ODS到DWD
INSERT OVERWRITE TABLE dwd_order_detail PARTITION (dt = '2024-01-15')
SELECT
    id AS order_id,
    user_id,
    product_id,
    quantity,
    amount / quantity AS unit_price,
    amount AS total_amount,
    CASE status
        WHEN 0 THEN '待付款'
        WHEN 1 THEN '已付款'
        WHEN 2 THEN '已发货'
        WHEN 3 THEN '已完成'
        WHEN 4 THEN '已取消'
        ELSE '未知'
    END AS order_status,
    create_time AS order_time,
    CURRENT_TIMESTAMP() AS etl_time
FROM ods_order
WHERE dt = '2024-01-15';
```

**DWD层特点**：
- 数据清洗：去重、空值处理、异常值过滤
- 数据规范化：统一编码、字段命名标准化
- 维度退化：将常用维度字段冗余到事实表
- 保留最细粒度的明细数据

#### DWS层（数据汇总层）

```sql
-- DWS层示例：按主题汇总
CREATE TABLE dws_user_trade_daily (
    user_id             BIGINT,
    order_count         INT,
    total_amount        DECIMAL(14,2),
    avg_order_amount    DECIMAL(12,2),
    product_count       INT,
    first_order_time    TIMESTAMP,
    last_order_time     TIMESTAMP
)
PARTITIONED BY (dt STRING)
STORED AS ORC;
```

**DWS层特点**：
- 按主题域进行轻度汇总
- 数据粒度比DWD层粗（如按用户+天汇总）
- 形成宽表，减少下游查询的Join操作
- 支持多时间粒度（日/周/月）

#### ADS层（应用数据层）

```sql
-- ADS层示例：直接面向报表应用
CREATE TABLE ads_daily_sales_report (
    report_date         DATE,
    total_orders        BIGINT,
    total_amount        DECIMAL(16,2),
    avg_order_amount    DECIMAL(12,2),
    paying_users        BIGINT,
    new_users           BIGINT
);
```

**ADS层特点**：
- 面向具体应用场景（报表、大屏、API）
- 数据粒度符合应用需求
- 通常数据量较小
- 可以跨主题域关联

### 分层数据流转示例

```sql
-- 完整的分层ETL流程示例

-- Step 1: ODS -> DWD（清洗与规范化）
INSERT OVERWRITE TABLE dwd_user_event PARTITION (dt = '${biz_date}')
SELECT
    event_id,
    user_id,
    CASE event_type
        WHEN 1 THEN 'click'
        WHEN 2 THEN 'view'
        WHEN 3 THEN 'purchase'
        ELSE 'other'
    END AS event_type,
    event_time,
    page_url,
    COALESCE(ref_url, '') AS ref_url,
    CURRENT_TIMESTAMP() AS etl_time
FROM ods_user_event
WHERE dt = '${biz_date}'
    AND event_id IS NOT NULL;

-- Step 2: DWD -> DWS（汇总聚合）
INSERT OVERWRITE TABLE dws_user_behavior_daily PARTITION (dt = '${biz_date}')
SELECT
    user_id,
    COUNT(CASE WHEN event_type = 'click' THEN 1 END) AS click_count,
    COUNT(CASE WHEN event_type = 'view' THEN 1 END) AS view_count,
    COUNT(CASE WHEN event_type = 'purchase' THEN 1 END) AS purchase_count,
    MIN(event_time) AS first_event_time,
    MAX(event_time) AS last_event_time
FROM dwd_user_event
WHERE dt = '${biz_date}'
GROUP BY user_id;

-- Step 3: DWS -> ADS（应用指标）
INSERT OVERWRITE TABLE ads_user_growth_daily
SELECT
    '${biz_date}' AS report_date,
    COUNT(DISTINCT user_id) AS dau,
    SUM(CASE WHEN first_event_time = last_event_time THEN 1 ELSE 0 END) AS single_visit_users,
    SUM(purchase_count) AS total_purchases
FROM dws_user_behavior_daily
WHERE dt = '${biz_date}';
```

## 数据仓库技术演进

| 阶段 | 时间 | 技术 | 特点 |
|------|------|------|------|
| 传统数仓 | 2000s | Oracle, Teradata, DB2 | 一体机，昂贵 |
| 大数据数仓 | 2010s | Hive, Spark, Hadoop | 开源，水平扩展 |
| 云原生数仓 | 2020s | Snowflake, BigQuery, Redshift | 弹性计算，按量付费 |
| Lakehouse | 2020s+ | Delta Lake, Iceberg, Hudi | 数仓+数据湖融合 |

## 总结

数据仓库是企业数据资产化的核心基础设施。从Inmon的自顶向下方法论到Kimball的维度建模，从传统的关系型数据库到现代的Lakehouse架构，数据仓库技术在不断演进。理解OLTP与OLAP的区别、掌握ODS/DWD/DWS/ADS的分层思想，是构建高质量数据平台的基础。
