# 维度建模理论

## 维度建模概述

维度建模（Dimensional Modeling）由Ralph Kimball提出，是一种针对数据分析场景优化的数据建模方法。其核心思想是将数据分为**事实表（Fact Table）**和**维度表（Dimension Table）**，以业务过程为中心组织数据。

### 事实表 vs 维度表

| 特性 | 事实表 (Fact Table) | 维度表 (Dimension Table) |
|------|---------------------|------------------------|
| 数据内容 | 度量/指标（可加数值） | 描述性属性（文本、分类） |
| 行数 | 大（数百万-数十亿） | 小（数千-数百万） |
| 变化频率 | 只增不改（Append） | 偶尔更新 |
| 典型列 | 外键 + 度量值 | 主键 + 描述属性 |
| 示例 | 订单金额、点击次数、温度 | 用户信息、产品属性、时间维度 |

## 星型模型（Star Schema）

星型模型是最常见的维度模型，以一个事实表为中心，周围关联多个维度表，形成星形结构。

```text
                    +---------------+
                    |  dim_user     |
                    | (用户维度)     |
                    +-------+-------+
                            |
+---------------+    +------+------+    +---------------+
| dim_product   |----| fct_order   |----| dim_date     |
| (产品维度)    |    | (订单事实)   |    | (时间维度)    |
+---------------+    +------+------+    +---------------+
                            |
                    +-------+-------+
                    | dim_channel   |
                    | (渠道维度)    |
                    +---------------+
```

### 建表示例

```sql
-- 事实表：订单事实表
CREATE TABLE fct_order (
    order_id        BIGINT,
    user_key        INT,          -- 外键 -> dim_user
    product_key     INT,          -- 外键 -> dim_product
    date_key        INT,          -- 外键 -> dim_date
    channel_key     INT,          -- 外键 -> dim_channel
    -- 度量（可加数值）
    quantity        INT,
    unit_price      DECIMAL(10,2),
    discount_amount DECIMAL(10,2),
    total_amount    DECIMAL(12,2),
    profit_amount   DECIMAL(12,2)
);

-- 维度表：用户维度
CREATE TABLE dim_user (
    user_key        INT PRIMARY KEY,
    user_id         BIGINT,
    user_name       STRING,
    gender          STRING,
    age_range       STRING,
    city            STRING,
    province        STRING,
    member_level    STRING,
    register_date   DATE
);

-- 维度表：时间维度
CREATE TABLE dim_date (
    date_key        INT PRIMARY KEY,    -- YYYYMMDD格式
    calendar_date   DATE,
    year            INT,
    quarter         INT,
    month           INT,
    week_of_year    INT,
    day_of_week     INT,
    day_name        STRING,
    is_weekend      BOOLEAN,
    is_holiday      BOOLEAN,
    holiday_name    STRING
);

-- 维度表：产品维度
CREATE TABLE dim_product (
    product_key     INT PRIMARY KEY,
    product_id      BIGINT,
    product_name    STRING,
    category        STRING,
    subcategory     STRING,
    brand           STRING,
    supplier        STRING,
    unit_cost       DECIMAL(10,2)
);
```

### 查询示例

```sql
-- 按城市、月份统计销售额和利润
SELECT
    u.city,
    d.year,
    d.month,
    SUM(f.total_amount) AS total_sales,
    SUM(f.profit_amount) AS total_profit,
    COUNT(DISTINCT f.order_id) AS order_count
FROM fct_order f
JOIN dim_user u ON f.user_key = u.user_key
JOIN dim_date d ON f.date_key = d.date_key
WHERE d.year = 2024
GROUP BY u.city, d.year, d.month
ORDER BY total_sales DESC;
```

## 雪花模型（Snowflake Schema）

雪花模型是星型模型的规范化形式，将维度表进一步拆分为子维度表，减少数据冗余但增加Join复杂度。

```text
              +-----------+
              | dim_user  |
              +-----+-----+
                    |
+-----------+  +---+---+  +-----------+
|dim_brand  |--|fct_   |--|dim_date   |
+-----+-----+ |order  |  +-----------+
      |       +---+---+
+-----+-----+     |
|dim_category|    |
+-----------+     |
           +------+------+
           | dim_product  |
           +--------------+
```

```sql
-- 雪花模型：产品维度规范化
CREATE TABLE dim_product (
    product_key     INT PRIMARY KEY,
    product_name    STRING,
    category_key    INT,          -- 外键 -> dim_category
    brand_key       INT           -- 外键 -> dim_brand
);

CREATE TABLE dim_category (
    category_key    INT PRIMARY KEY,
    category_name   STRING,
    department      STRING
);

CREATE TABLE dim_brand (
    brand_key       INT PRIMARY KEY,
    brand_name      STRING,
    country         STRING
);
```

### 星型 vs 雪花模型对比

| 维度 | 星型模型 | 雪花模型 |
|------|---------|---------|
| 规范化 | 反规范化（冗余） | 规范化（无冗余） |
| 查询性能 | 快（Join少） | 慢（Join多） |
| 存储空间 | 较大（冗余） | 较小（无冗余） |
| 维护成本 | 低（简单） | 高（复杂） |
| 适用场景 | 数据仓库、BI查询 | 数据湖、需要灵活扩展 |
| 推荐程度 | **推荐** | 特殊场景使用 |

## 事实表类型

### 1. 事务事实表（Transaction Fact Table）

记录业务事件的每一笔事务，粒度为单条事务。

```sql
-- 交易事实表：每一行代表一笔交易
CREATE TABLE fct_transaction (
    transaction_id  BIGINT,
    user_key        INT,
    product_key     INT,
    date_key        INT,
    transaction_time TIMESTAMP,
    amount          DECIMAL(12,2),
    quantity        INT,
    payment_method  STRING
);
```

### 2. 周期快照事实表（Periodic Snapshot Fact Table）

按固定周期对可加度量进行快照记录。

```sql
-- 用户账户余额日快照表
CREATE TABLE fct_user_balance_daily (
    user_key        INT,
    date_key        INT,
    balance_amount  DECIMAL(14,2),  -- 当日余额
    frozen_amount   DECIMAL(14,2),  -- 冻结金额
    available_amount DECIMAL(14,2)  -- 可用金额
);
```

### 3. 累计快照事实表（Accumulating Snapshot Fact Table）

记录具有明确生命周期的业务过程，每个里程碑对应一个时间戳列。

```sql
-- 订单生命周期事实表
CREATE TABLE fct_order_lifecycle (
    order_id            BIGINT,
    user_key            INT,
    product_key         INT,
    -- 里程碑时间戳
    order_date_key      INT,      -- 下单日期
    pay_date_key        INT,      -- 付款日期
    ship_date_key       INT,      -- 发货日期
    receive_date_key    INT,      -- 签收日期
    -- 时间间隔度量
    pay_lag_days        INT,      -- 下单到付款天数
    ship_lag_days       INT,      -- 付款到发货天数
    delivery_lag_days   INT,      -- 发货到签收天数
    -- 度量
    order_amount        DECIMAL(12,2),
    shipping_fee        DECIMAL(8,2)
);
```

### 4. 无事实事实表（Factless Fact Table）

记录没有度量的事件或条件，常用于覆盖率分析。

```sql
-- 学生选课事实表（无度量）
CREATE TABLE fct_student_enrollment (
    student_key     INT,
    course_key      INT,
    date_key        INT,
    semester        STRING
);

-- 查询：哪些学生没有选任何课？
SELECT s.student_name
FROM dim_student s
LEFT JOIN fct_student_enrollment e ON s.student_key = e.student_key
WHERE e.student_key IS NULL;
```

## 维度表设计

### 维度表关键属性

```sql
-- 维度表标准设计模板
CREATE TABLE dim_customer (
    -- 主键
    customer_key    INT PRIMARY KEY,   -- 代理键(Surrogate Key)
    -- 业务键
    customer_id     BIGINT,            -- 自然键(Natural Key)
    -- 描述属性
    customer_name   STRING,
    email           STRING,
    phone           STRING,
    gender          STRING,
    birth_date      DATE,
    age             INT,
    -- 地理属性
    country         STRING,
    province        STRING,
    city            STRING,
    district        STRING,
    -- 分类属性
    customer_tier   STRING,            -- 客户等级
    industry        STRING,            -- 行业
    -- 时间属性
    first_order_date DATE,
    register_date   DATE,
    -- 元数据
    effective_from  TIMESTAMP,         -- SCD有效开始时间
    effective_to    TIMESTAMP,         -- SCD有效结束时间
    is_current      BOOLEAN            -- 是否当前记录
);
```

### 代理键 vs 自然键

| 类型 | 说明 | 优点 | 示例 |
|------|------|------|------|
| 代理键(Surrogate Key) | 数据库生成的无业务含义的键 | 稳定、性能好、支持SCD | `customer_key INT` |
| 自然键(Natural Key) | 业务系统中的业务标识 | 可读性好 | `customer_id BIGINT` |

**最佳实践**：维度表使用代理键作为主键，自然键作为属性存储，这样可以更好地处理缓慢变化维度。

## 缓慢变化维度（SCD）

缓慢变化维度（Slowly Changing Dimension）处理维度属性随时间变化的问题。

### SCD Type 0：保留原始值

维度属性永不更新，保持最初的值。

```sql
-- 注册渠道一旦确定就不改变
-- 不需要特殊处理，直接覆盖即可
```

### SCD Type 1：覆盖更新

直接覆盖旧值，不保留历史。适用于不需要追踪历史的属性（如纠错）。

```sql
-- 直接UPDATE覆盖
UPDATE dim_user
SET email = 'new_email@test.com',
    phone = '13800138000'
WHERE user_id = 1001;

-- 使用dbt snapshot（默认行为）
-- snapshots/snap_users_scd1.sql
{% snapshot snap_users_scd1 %}
{{
    config(
        strategy='check',
        check_cols=['email', 'phone'],
        unique_key='user_id',
        updated_at='updated_at'
    )
}}
SELECT * FROM {{ source('raw', 'users') }}
{% endsnapshot %}
```

### SCD Type 2：新增记录

维度属性变化时插入新记录，保留完整历史。这是最常用的SCD类型。

```sql
-- SCD Type 2 实现
-- dim_user表结构
CREATE TABLE dim_user (
    user_key        INT,          -- 代理键
    user_id         BIGINT,       -- 自然键
    user_name       STRING,
    email           STRING,
    member_level    STRING,
    -- SCD Type 2 字段
    effective_from  TIMESTAMP,    -- 生效开始时间
    effective_to    TIMESTAMP,    -- 生效结束时间（9999-12-31表示当前有效）
    is_current      BOOLEAN       -- 是否当前记录
);

-- 示例数据
-- user_id=1001 在2024-01-01注册为普通会员
-- 在2024-06-01升级为VIP
INSERT INTO dim_user VALUES
(1, 1001, '张三', 'zhang@test.com', 'normal', '2024-01-01', '2024-06-01', false),
(2, 1001, '张三', 'zhang@test.com', 'vip',    '2024-06-01', '9999-12-31', true);

-- MERGE实现SCD Type 2
MERGE INTO dim_user AS target
USING ods_user_update AS source
ON target.user_id = source.user_id AND target.is_current = true

-- 数据有变化：关闭旧记录，插入新记录
WHEN MATCHED AND (
    target.email <> source.email OR
    target.member_level <> source.member_level
) THEN
    -- 先更新旧记录的结束时间
    UPDATE SET
        effective_to = source.update_time,
        is_current = false;

-- 然后插入新记录
INSERT INTO dim_user (user_key, user_id, user_name, email, member_level, effective_from, effective_to, is_current)
SELECT
    NEXT VALUE FOR user_key_seq,  -- 生成新代理键
    source.user_id,
    source.user_name,
    source.email,
    source.member_level,
    source.update_time,
    '9999-12-31',
    true
FROM ods_user_update source
JOIN dim_user target ON source.user_id = target.user_id
WHERE target.is_current = false
    AND target.effective_to = source.update_time;
```

### SCD Type 3：增加列

增加新列来存储上一个值，只保留有限的历史。

```sql
CREATE TABLE dim_product (
    product_key     INT PRIMARY KEY,
    product_id      BIGINT,
    product_name    STRING,
    -- SCD Type 3 字段
    current_price   DECIMAL(10,2),    -- 当前价格
    previous_price  DECIMAL(10,2),    -- 上一次价格
    price_change_date DATE            -- 价格变更日期
);

-- 更新时同时保存旧值
UPDATE dim_product
SET previous_price = current_price,
    current_price = 199.99,
    price_change_date = '2024-06-01'
WHERE product_id = 1001;
```

### SCD类型对比

| 类型 | 历史追踪 | 实现复杂度 | 存储开销 | 适用场景 |
|------|---------|-----------|---------|---------|
| Type 0 | 不追踪 | 无需处理 | 最小 | 日期类属性 |
| Type 1 | 不保留 | 简单 | 最小 | 纠错、不需要历史 |
| Type 2 | 完整历史 | 较复杂 | 较大 | 等级、地址等重要属性 |
| Type 3 | 有限历史 | 中等 | 中等 | 价格等需要对比的属性 |

## 维度建模四步法

Kimball推荐的维度建模四步法：

1. **选择业务过程（Select the Business Process）**
   - 确定要分析的业务活动：订单、点击、支付、退款等

2. **声明粒度（Declare the Grain）**
   - 确定事实表每一行代表什么：一笔订单、一个订单项、一次点击

3. **识别维度（Identify the Dimensions）**
   - 确定描述事实的角度：时间、用户、产品、地区、渠道

4. **识别事实（Identify the Facts）**
   - 确定需要度量的指标：金额、数量、次数

```sql
-- 四步法示例：电商订单分析

-- 步骤1：业务过程 = 订单交易
-- 步骤2：粒度 = 每笔订单的每个商品项
-- 步骤3：维度 = 时间、用户、产品、渠道、地区
-- 步骤4：事实 = 数量、单价、金额、利润

CREATE TABLE fct_order_item (
    -- 粒度：订单项级别
    order_item_id   BIGINT,
    order_id        BIGINT,
    -- 维度外键
    date_key        INT,
    user_key        INT,
    product_key     INT,
    channel_key     INT,
    -- 度量
    quantity        INT,
    unit_price      DECIMAL(10,2),
    discount_amount DECIMAL(10,2),
    total_amount    DECIMAL(12,2),
    cost_amount     DECIMAL(12,2),
    profit_amount   DECIMAL(12,2)
);
```

## 最佳实践

1. **一致性维度**：跨业务过程共享同一套维度表（Conformed Dimensions）
2. **代理键优先**：维度表使用代理键，避免业务键变化导致的问题
3. **粒度优先**：建模第一步确定粒度，粒度决定了事实表的设计
4. **星型优先**：除非有充分理由，否则选择星型模型而非雪花模型
5. **SCD按需选择**：关键属性用Type 2，不重要属性用Type 1
