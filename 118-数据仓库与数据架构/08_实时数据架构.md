# 实时数据架构

## 从批处理到实时处理

传统数据仓库采用批处理模式（T+1），数据延迟以天/小时计。随着业务对实时性的要求提升，实时数据架构应运而生。

| 特性 | 批处理 | 准实时 | 实时处理 |
|------|--------|--------|---------|
| 延迟 | 小时/天级 | 分钟级 | 秒/毫秒级 |
| 吞吐量 | 高 | 中高 | 中 |
| 复杂度 | 低 | 中 | 高 |
| 典型工具 | Hive, Spark SQL | Spark Streaming | Flink, Kafka Streams |
| 适用场景 | 日报/月报 | 实时大盘 | 风控/推荐/监控 |

## Lambda架构

Lambda架构由Nathan Marz提出，通过批处理层(Batch Layer)和速度层(Speed Layer)的组合，同时保证数据的完整性和实时性。

```text
                    +---------------------------+
                    |       数据源 (Kafka)        |
                    +---------------------------+
                           |            |
              +------------+            +------------+
              |                                      |
              v                                      v
    +-------------------+                +-------------------+
    |   批处理层          |                |   速度层            |
    |   (Batch Layer)    |                |   (Speed Layer)    |
    |                    |                |                    |
    |   Spark/Hive       |                |   Flink/Storm      |
    |   全量计算          |                |   增量计算          |
    |   高延迟/高准确     |                |   低延迟/近似       |
    +-------------------+                +-------------------+
              |                                      |
              v                                      v
    +-------------------+                +-------------------+
    |   批处理视图        |                |   实时视图          |
    |   (Batch View)     |                |   (Real-time View) |
    |   Hive/Delta Lake  |                |   Redis/HBase      |
    +-------------------+                +-------------------+
              |                                      |
              +------------------+-------------------+
                                 |
                                 v
                    +---------------------------+
                    |        服务层              |
                    |     (Serving Layer)        |
                    |  合并批处理和实时视图结果     |
                    +---------------------------+
                                 |
                                 v
                          +-----------+
                          |  应用/API  |
                          +-----------+
```

### Lambda架构实现示例

```python
# 批处理层：Spark全量计算
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder.appName("LambdaBatchLayer").getOrCreate()

# 全量读取历史数据
full_data = spark.read.format("delta").load("/data/lake/events")

# 全量聚合：用户行为统计
batch_view = full_data.groupBy("user_id").agg(
    F.count("*").alias("total_events"),
    F.countDistinct("event_type").alias("event_type_count"),
    F.min("event_time").alias("first_event"),
    F.max("event_time").alias("last_event"),
    F.sum(F.when(F.col("event_type") == "purchase", F.col("amount")).otherwise(0)).alias("total_purchase")
)

# 写入批处理视图（Hive/Delta）
batch_view.write.format("delta").mode("overwrite").saveAsTable("batch_user_metrics")
```

```python
# 速度层：Flink实时计算
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.table import StreamTableEnvironment, EnvironmentSettings

env = StreamExecutionEnvironment.get_execution_environment()
settings = EnvironmentSettings.in_streaming_mode()
t_env = StreamTableEnvironment.create(env, settings)

# Kafka Source
t_env.execute_sql("""
    CREATE TABLE event_stream (
        user_id STRING,
        event_type STRING,
        amount DECIMAL(10,2),
        event_time TIMESTAMP(3),
        WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
    ) WITH (
        'connector' = 'kafka',
        'topic' = 'user-events',
        'properties.bootstrap.servers' = 'kafka:9092',
        'scan.startup.mode' = 'latest-offset',
        'format' = 'json'
    )
""")

# 实时视图 Sink (Redis/HBase)
t_env.execute_sql("""
    CREATE TABLE realtime_user_metrics (
        user_id STRING,
        total_events BIGINT,
        total_purchase DECIMAL(12,2),
        last_event_time TIMESTAMP(3),
        PRIMARY KEY (user_id) NOT ENFORCED
    ) WITH (
        'connector' = 'jdbc',
        'url' = 'jdbc:mysql://mysql:3306/realtime',
        'table-name' = 'user_metrics',
        'username' = 'root',
        'password' = 'password'
    )
""")

# 实时聚合
t_env.execute_sql("""
    INSERT INTO realtime_user_metrics
    SELECT
        user_id,
        COUNT(*) AS total_events,
        SUM(CASE WHEN event_type = 'purchase' THEN amount ELSE 0 END) AS total_purchase,
        MAX(event_time) AS last_event_time
    FROM event_stream
    GROUP BY user_id
""")
```

### 服务层合并查询

```python
# 服务层：合并批处理视图和实时视图
import redis
from pyhive import hive

class ServingLayer:
    def __init__(self):
        self.redis_client = redis.Redis(host='redis', port=6379, db=0)
        self.hive_conn = hive.Connection(host='hive-server', port=10000)

    def get_user_metrics(self, user_id: str) -> dict:
        """合并批处理和实时视图"""

        # 1. 从Hive读取批处理视图（T-1的全量数据）
        cursor = self.hive_conn.cursor()
        cursor.execute(f"""
            SELECT total_events, total_purchase
            FROM batch_user_metrics
            WHERE user_id = '{user_id}'
        """)
        batch_result = cursor.fetchone()
        batch_total_events = batch_result[0] if batch_result else 0
        batch_total_purchase = batch_result[1] if batch_result else 0

        # 2. 从Redis读取实时增量数据（当天新增）
        realtime_key = f"realtime:metrics:{user_id}"
        realtime_data = self.redis_client.hgetall(realtime_key)
        realtime_events = int(realtime_data.get(b"total_events", 0))
        realtime_purchase = float(realtime_data.get(b"total_purchase", 0))

        # 3. 合并结果
        return {
            "user_id": user_id,
            "total_events": batch_total_events + realtime_events,
            "total_purchase": float(batch_total_purchase) + realtime_purchase
        }
```

## Kappa架构

Kappa架构由Jay Kreps提出，移除了批处理层，仅使用流处理引擎处理所有数据。其核心思想是：**一切皆流**。

```text
    +---------------------------+
    |       数据源 (Kafka)        |
    +---------------------------+
                 |
                 v
    +---------------------------+
    |      流处理引擎             |
    |    (Flink / Kafka Streams) |
    |                            |
    |  - 实时计算                 |
    |  - 历史重放 (Replay)        |
    +---------------------------+
                 |
                 v
    +---------------------------+
    |       服务层/存储           |
    |    (Elasticsearch/Redis/   |
    |     Delta Lake)            |
    +---------------------------+
                 |
                 v
           +-----------+
           |  应用/API  |
           +-----------+
```

### Kappa架构的核心优势

1. **单一技术栈**：只需维护一套流处理代码，降低运维复杂度
2. **历史重放能力**：通过Kafka保留历史数据，可从头消费重新计算
3. **代码一致性**：批处理和实时处理使用同一套代码逻辑

### Kafka Streams实现Kappa

```java
import org.apache.kafka.streams.*;
import org.apache.kafka.streams.kstream.*;
import java.time.Duration;
import java.util.Properties;

public class KappaUserMetrics {
    public static void main(String[] args) {
        Properties props = new Properties();
        props.put(StreamsConfig.APPLICATION_ID_CONFIG, "kappa-user-metrics");
        props.put(StreamsConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka:9092");
        props.put(StreamsConfig.DEFAULT_KEY_SERDE_CLASS_CONFIG,
                  org.apache.kafka.common.serialization.Serdes.String().getClass());
        props.put(StreamsConfig.DEFAULT_VALUE_SERDE_CLASS_CONFIG,
                  org.apache.kafka.common.serialization.Serdes.String().getClass());

        StreamsBuilder builder = new StreamsBuilder();

        // 1. 从Kafka读取事件流
        KStream<String, String> events = builder.stream("user-events",
            Consumed.with(Serdes.String(), Serdes.String()));

        // 2. 按用户分组，窗口聚合
        KTable<Windowed<String>, Long> userEventCounts = events
            .mapValues(value -> parseEvent(value))
            .groupBy((key, event) -> event.getUserId())
            .windowedBy(TimeWindows.of(Duration.ofMinutes(5)).grace(Duration.ofSeconds(30)))
            .count(Materialized.as("user-event-counts"));

        // 3. 输出到结果Topic
        userEventCounts.toStream()
            .map((windowedKey, count) -> KeyValue.pair(
                windowedKey.key(),
                String.format("{\"user_id\":\"%s\",\"window_start\":%d,\"count\":%d}",
                    windowedKey.key(), windowedKey.window().start(), count)
            ))
            .to("user-metrics-output");

        // 4. 构建并启动
        KafkaStreams streams = new KafkaStreams(builder.build(), props);
        streams.start();

        Runtime.getRuntime().addShutdownHook(new Thread(streams::close));
    }
}
```

### Kafka架构

```text

+--------+                     +------------------+
| App A  |---+                 | Consumer 1       |
+--------+   |   +---------+   |   (Partition 0)  |
             +-->| Broker 0|-->+------------------+
```

### Kafka Producer

```python
from confluent_kafka import Producer
import json
import time

# Producer配置
producer_config = {
    'bootstrap.servers': 'kafka-broker1:9092,kafka-broker2:9092',
    'acks': 'all',                  # 所有副本确认
    'retries': 3,                    # 重试次数
    'batch.size': 16384,             # 批次大小(字节)
    'linger.ms': 10,                 # 等待时间(毫秒)
    'compression.type': 'snappy',   # 压缩类型
    'enable.idempotence': True,      # 幂等生产者
}

producer = Producer(producer_config)

def delivery_callback(err, msg):
    """消息投递回调"""
    if err:
        print(f"消息发送失败: {err}")
    else:
        print(f"消息发送成功: topic={msg.topic()}, partition={msg.partition()}, offset={msg.offset()}")

# 发送消息
for i in range(100):
    event = {
        "event_id": f"evt_{i}",
        "user_id": f"user_{i % 10}",
        "event_type": "click",
        "event_time": int(time.time() * 1000),
        "page": f"/product/{i % 50}"
    }
    producer.produce(
        topic='user-events',
        key=event["user_id"],          # 按用户ID分区
        value=json.dumps(event),
        callback=delivery_callback
    )

producer.flush()  # 确保所有消息发送完成
```

### Kafka Consumer

```python
from confluent_kafka import Consumer, KafkaException
import json

consumer_config = {
    'bootstrap.servers': 'kafka-broker1:9092,kafka-broker2:9092',
    'group.id': 'analytics-consumer-group',
    'auto.offset.reset': 'earliest',          # 从最早的消息开始
    'enable.auto.commit': False,               # 手动提交偏移量
    'max.poll.interval.ms': 300000,            # 最大poll间隔
    'session.timeout.ms': 30000,               # 会话超时
    'max.partition.fetch.bytes': 1048576,      # 每分区最大拉取字节
}

consumer = Consumer(consumer_config)
consumer.subscribe(['user-events'])

try:
    while True:
        msg = consumer.poll(timeout=1.0)  # 拉取消息
        if msg is None:
            continue
        if msg.error():
            if msg.error().code() == KafkaException._PARTITION_EOF:
                continue
            else:
                print(f"Consumer error: {msg.error()}")
                break

        # 处理消息
        event = json.loads(msg.value().decode('utf-8'))
        process_event(event)

        # 每100条消息提交一次偏移量
        if msg.offset() % 100 == 0:
            consumer.commit(asynchronous=False)

except KeyboardInterrupt:
    pass
finally:
    consumer.close()
```

### Flink CDC (SQL方式)

```sql
-- 使用Flink CDC直接从MySQL读取变更流
CREATE TABLE mysql_orders (
    id BIGINT,
    user_id BIGINT,
    product_id BIGINT,
    amount DECIMAL(10, 2),
    status STRING,
    create_time TIMESTAMP(3),
    update_time TIMESTAMP(3),

) WITH (
    'connector' = 'mysql-cdc',
    'hostname' = 'mysql-host',
    'port' = '3306',
    'username' = 'cdc_user',
    'password' = 'password',
    'database-name' = 'ecommerce',
    'table-name' = 'orders',
    'server-time-zone' = 'Asia/Shanghai',
    'debezium.snapshot.mode' = 'initial'
);

-- 将CDC数据写入Kafka
CREATE TABLE kafka_orders (
    id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10, 2),
    status STRING,
    op_type STRING,        -- c:创建, u:更新, d:删除
    op_time TIMESTAMP(3),
    PRIMARY KEY (id) NOT ENFORCED
) WITH (
    'connector' = 'kafka',
    'topic' = 'cdc-orders',
    'properties.bootstrap.servers' = 'kafka:9092',
    'format' = 'json'
);

-- 实时同步
INSERT INTO kafka_orders
SELECT id, user_id, amount, status,
       CASE op_type
           WHEN 'CREATE' THEN 'c'
           WHEN 'UPDATE' THEN 'u'
           WHEN 'DELETE' THEN 'd'
       END AS op_type,
       update_time AS op_time
FROM mysql_orders;
```

## 架构选型对比

| 维度 | Lambda架构 | Kappa架构 |
|------|-----------|----------|
| 复杂度 | 高（维护两套代码） | 低（一套代码） |
| 准确性 | 高（批处理兜底） | 依赖流处理准确性 |
| 延迟 | 批处理高延迟+实时低延迟 | 统一低延迟 |
| 运维成本 | 高 | 低 |
| 历史修正 | 批处理自动修正 | 需要重放 |
| 适用场景 | 对准确性要求极高 | 追求简洁和实时 |

**选型建议**：
- 金融、账务等对数据准确性要求极高的场景，选择Lambda架构
- 推荐、监控、日志分析等可容忍少量误差的场景，选择Kappa架构
- 新项目建议从Kappa架构起步，必要时再增加批处理层
