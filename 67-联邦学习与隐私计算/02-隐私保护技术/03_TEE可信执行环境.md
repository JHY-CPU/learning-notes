# TEE可信执行环境


## TEE可信执行环境


TEEIntel SGXTrustZone远程认证


TEE通过硬件隔离创建安全执行区域，为联邦学习提供高性能的安全计算环境。


## TEE概述与原理


```
TEE (Trusted Execution Environment)：
在硬件层面创建隔离的安全执行区域，保护运行中的代码和数据

核心思想：
┌─────────────────────────────────────────────────────┐
│  传统安全：保护静态数据（加密存储）                   │
│  TEE安全：保护运行中的数据（加密计算）               │
│                                                     │
│  问题：数据在CPU中处理时是明文的                     │
│  恶意OS/Root用户/物理攻击者可能读取                  │
│  TEE将敏感计算隔离到硬件安全区中                     │
│  即使操作系统被攻破，安全区仍然安全                  │
└─────────────────────────────────────────────────────┘

安全保证模型：
┌─────────────────────────────────────────────────────┐
│  可信计算基 (TCB, Trusted Computing Base)：          │
│  系统中需要信任的部分                                │
│                                                     │
│  传统系统TCB：                                      │
│  - 操作系统内核                                      │
│  - Hypervisor                                       │
│  - 固件                                             │
│  - 硬件                                             │
│  → TCB非常大，攻击面广                              │
│                                                     │
│  TEE系统TCB：                                       │
│  - 安全区内的代码                                   │
│  - 硬件安全模块                                     │
│  - 极小的微码                                       │
│  → TCB极小，攻击面大幅缩小                          │
└─────────────────────────────────────────────────────┘

TEE安全特性：
1. 机密性 (Confidentiality)
   安全区外的任何实体无法读取安全区内的数据
   内存加密，CPU缓存隔离

2. 完整性 (Integrity)
   安全区内的代码和数据不能被外部修改
   任何篡改都会被检测到

3. 可信执行 (Trusted Execution)
   确保代码按照预期执行
   没有被恶意替换或修改

4. 远程认证 (Remote Attestation)
   远程验证安全区内运行的代码确实是指定代码
   建立远程信任链
```


## Intel SGX


```
SGX (Software Guard Extensions)：
Intel的TEE技术，在CPU中创建安全飞地(Enclave)

架构：
┌─────────────────────────────────────────────────────┐
│                    应用程序                          │
│  ┌────────────────────────────────────────────┐    │
│  │              不可信部分 (App)                │    │
│  │    负责普通逻辑，I/O，数据准备              │    │
│  └──────────────┬─────────────────────────────┘    │
│                 │ ECALL / OCALL                     │
│  ┌──────────────▼─────────────────────────────┐    │
│  │              可信部分 (Enclave)              │    │
│  │    处理敏感数据，执行安全计算                │    │
│  │    代码和数据在安全区内存中                  │    │
│  └────────────────────────────────────────────┘    │
│                      │                              │
│  ════════════ 硬件隔离边界 ════════════             │
│                      │                              │
│  ┌──────────────────▼──────────────────────┐       │
│  │  操作系统 / Hypervisor / 其他进程         │       │
│  │  无法访问Enclave的内存                    │       │
│  └──────────────────────────────────────────┘       │
│                      │                              │
│  ┌──────────────────▼──────────────────────┐       │
│  │  CPU (SGX硬件)                           │       │
│  │  内存加密引擎 (MEE)                      │       │
│  │  Enclave Page Cache (EPC)                │       │
│  └──────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘

关键组件：
- Enclave：安全执行区域，代码和数据隔离存放
- EPC (Enclave Page Cache)：安全区内存页，硬件加密保护
- MEE (Memory Encryption Engine)：内存加密引擎
- SEAL Key：密封密钥，绑定平台和Enclave身份

SGX编程接口：
┌─────────────────────────────────────────────────────┐
│  ECALL (Enclave Call)：从外部调用Enclave内函数      │
│  OCALL (Out Call)：从Enclave内调用外部函数          │
│                                                     │
│  编程模式：                                          │
│  1. 定义Enclave接口 (EDL文件)                       │
│  2. 实现Enclave内函数                               │
│  3. 编译生成Enclave签名模块                         │
│  4. App加载Enclave并调用ECALL                       │
│                                                     │
│  注意事项：                                          │
│  - Enclave代码需要特殊编译                          │
│  - EPC大小有限（128MB~256MB）                       │
│  - 频繁的ECALL/OCALL开销大                          │
│  - 不支持系统调用（需要通过OCALL）                  │
└─────────────────────────────────────────────────────┘

SGX版本演进：
┌──────────────┬──────────────┬──────────────────────┐
│ 版本          │ 年份         │ 主要改进              │
├──────────────┼──────────────┼──────────────────────┤
│ SGX 1        │ 2015         │ 初版，固定EPC大小     │
│ SGX 1.5      │ 2017         │ 动态EPC管理           │
│ SGX 2        │ 2021+        │ 动态Enclave管理       │
│              │              │ 页重映射，更大内存     │
│ TDX          │ 2023+        │ 虚拟机级别隔离        │
│              │              │ 更大的机密计算空间     │
└──────────────┴──────────────┴──────────────────────┘
```


## ARM TrustZone


```
TrustZone：ARM的TEE技术，基于CPU级别的安全隔离

架构：
┌─────────────────────────────────────────────────────┐
│  ┌──────────────────┬──────────────────┐            │
│  │  普通世界         │  安全世界          │            │
│  │  (Normal World)  │  (Secure World)   │            │
│  │                  │                  │            │
│  │  操作系统        │  Trusted OS       │            │
│  │  (Linux/Android) │  (OP-TEE/TEE OS)  │            │
│  │  普通应用        │  Trusted App (TA) │            │
│  │                  │                  │            │
│  │  普通内存        │  安全内存          │            │
│  │                  │                  │            │
│  └──────────────────┴──────────────────┘            │
│          ↑                    ↑                      │
│          │    Monitor Mode    │                      │
│          └────────────────────┘                      │
│          (世界切换，安全仲裁)                        │
│                      │                              │
│  ┌──────────────────▼──────────────────────┐       │
│  │         ARM CPU (TZASC, TZMA)            │       │
│  │    内存区域划分，外设安全控制              │       │
│  └──────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────┘

关键特性：
- 硬件级别的世界隔离（Normal/Secure）
- CPU、内存、外设都可以划分安全区域
- 通过Monitor Mode在两个世界间切换
- 中断可以标记为安全/非安全

与SGX的对比：
┌──────────────┬──────────────────┬──────────────────┐
│              │ Intel SGX        │ ARM TrustZone    │
├──────────────┼──────────────────┼──────────────────┤
│ 隔离粒度      │ 进程级(Enclave)  │ 系统级(世界)      │
│ 安全模型      │ 应用创建Enclave  │ 固定的安全世界    │
│ 内存大小      │ 有限(EPC 128MB)  │ 可配置           │
│ 信任基        │ Intel + Enclave  │ ARM + Trusted OS │
│ 主要应用      │ 服务器/桌面      │ 移动/IoT         │
│ 编程模型      │ SDK库函数        │ TA开发框架       │
│ 开源实现      │ OpenEnclave      │ OP-TEE          │
└──────────────┴──────────────────┴──────────────────┘

OP-TEE (Open Portable TEE)：
- 开源的TrustZone TEE OS
- 支持多种ARM平台
- GP TEE API标准兼容
- 可用于开发Trusted Application (TA)
```


## 远程认证 (Remote Attestation)


```
远程认证：验证远程TEE中运行的代码确实是预期代码

SGX远程认证流程：
┌─────────────────────────────────────────────────────┐
│  远程验证者                Intel         目标机器    │
│  (Verifier)            (IAS/AESM)       (SGX平台)   │
│      │                     │                │       │
│      │  1. 请求认证         │                │       │
│      │────────────────────────────────────→│       │
│      │                     │                │       │
│      │  2. 生成Quote        │                │       │
│      │    (Enclave报告      │                │       │
│      │     + 平台信息)      │                │       │
│      │                     │ 3. 验证Quote    │       │
│      │                     │←───────────────│       │
│      │                     │                │       │
│      │  4. 返回验证结果      │                │       │
│      │←────────────────────│                │       │
│      │                     │                │       │
│      │  5. 建立安全通道      │                │       │
│      │  (基于认证结果)       │                │       │
│      │══════════════════════════════════════│       │
└─────────────────────────────────────────────────────┘

Quote内容：
┌─────────────────────────────────────────────────────┐
│  - MRENCLAVE：Enclave代码的哈希值（身份标识）        │
│  - MRSIGNER：Enclave签名者的公钥哈希                 │
│  - ISVPRODID/ISVSVN：产品ID和安全版本号             │
│  - REPORTDATA：开发者自定义数据（64字节）            │
│  - 平台信息：TCB版本，安全属性等                     │
│                                                     │
│  验证内容：                                          │
│  1. Quote签名有效（Intel认证）                       │
│  2. MRENCLAVE匹配预期值                             │
│  3. 平台TCB状态安全                                 │
│  4. 自定义数据符合预期                               │
└─────────────────────────────────────────────────────┐

证明类型：
┌─────────────────────────────────────────────────────┐
│  EPID (Enhanced Privacy ID)：                       │
│  - Intel早期方案                                     │
│  - 基于群签名，匿名认证                             │
│  - 需要联系Intel IAS服务                            │
│                                                     │
│  DCAP (Data Center Attestation Primitives)：        │
│  - 自托管认证方案                                    │
│  - 使用Intel提供的缓存认证信息                       │
│  - 不需要每次联系Intel                               │
│  - 适合大规模部署                                    │
│                                                     │
│  本地认证 (Local Attestation)：                     │
│  - 同一平台上的Enclave间认证                        │
│  - 使用REPORT机制                                    │
│  - 不需要Intel参与                                   │
└─────────────────────────────────────────────────────┘

在FL中的应用：
1. 客户端认证：验证客户端TEE中运行正确的FL训练代码
2. 服务端认证：验证服务器TEE中运行正确的聚合代码
3. 模型保护：确保模型参数只在TEE中解密和计算
4. 审计追踪：TEE记录所有计算操作，可审计验证
```


## TEE在联邦学习中的应用


```
方案一：TEE安全聚合服务器
┌─────────────────────────────────────────────────────┐
│  传统方案：服务器直接看到各客户端梯度（不安全）       │
│  TEE方案：聚合逻辑在Enclave中执行                    │
│                                                     │
│  ┌─────────────────────────────────────────┐       │
│  │  SGX Enclave (安全聚合服务)               │       │
│  │  1. 接收加密梯度 Enc(Δwk)                │       │
│  │  2. 在Enclave内解密                      │       │
│  │  3. 执行聚合：w_new = Σ wk/n            │       │
│  │  4. 加密聚合结果                         │       │
│  │  5. 返回 Enc(w_new)                      │       │
│  └─────────────────────────────────────────┘       │
│           │                                         │
│           │  ECALL: 加密梯度进入Enclave              │
│           │  OCALL: 加密结果返回                     │
│           │                                         │
│  ┌─────────────────────────────────────────┐       │
│  │  不可信OS / 其他进程                     │       │
│  │  无法读取Enclave内的明文梯度             │       │
│  │  只看到加密数据进出                      │       │
│  └─────────────────────────────────────────┘       │
│                                                     │
│  优势：                                              │
│  - 计算性能远高于同态加密                           │
│  - 不需要复杂的密码学协议                           │
│  - 支持任意聚合逻辑                                 │
│                                                     │
│  劣势：                                              │
│  - 需要信任Intel/ARM等硬件厂商                      │
│  - EPC大小限制（SGX）                               │
│  - 可能存在侧信道攻击                               │
└─────────────────────────────────────────────────────┘

方案二：客户端TEE本地训练
┌─────────────────────────────────────────────────────┐
│  在客户端TEE中执行本地训练，保护训练过程              │
│                                                     │
│  ┌─────────────────────────────────────────┐       │
│  │  客户端Enclave                            │       │
│  │  1. 接收加密的本地数据                    │       │
│  │  2. 解密数据，在Enclave内训练             │       │
│  │  3. 返回加密的模型更新                    │       │
│  │  数据只在Enclave内明文存在                │       │
│  └─────────────────────────────────────────┘       │
│                                                     │
│  应用场景：                                          │
│  - 医疗数据：患者数据在TEE中加密存储                 │
│  - 金融数据：交易数据在TEE中处理                    │
│  - 个人隐私数据：手机上的敏感数据                   │
└─────────────────────────────────────────────────────┘

方案三：混合方案 (HE + TEE)
┌─────────────────────────────────────────────────────┐
│  结合同态加密和TEE的优势                             │
│                                                     │
│  设计：                                              │
│  1. 使用HE加密客户端梯度                            │
│  2. TEE内执行高效的密文聚合                         │
│     - TEE提供计算加速                               │
│     - HE提供数学隐私保证                            │
│  3. 即使TEE被攻破，HE仍保护数据                     │
│                                                     │
│  性能：                                              │
│  - 纯HE：100-1000x 减速                             │
│  - TEE + HE：10-50x 减速                            │
│  - 纯TEE：1.5-3x 减速                               │
└─────────────────────────────────────────────────────┘

TEE安全挑战：
┌─────────────────────────────────────────────────────┐
│  1. 侧信道攻击 (Side-channel Attack)                │
│     - 缓存时序攻击：通过缓存访问模式推断数据         │
│     - 功耗分析：通过功耗模式推断计算内容             │
│     - 电磁泄露：通过电磁信号推断数据                 │
│     防御：常数时间算法、缓存分区、噪声注入          │
│                                                     │
│  2. 故障注入 (Fault Injection)                      │
│     - 电压毛刺、时钟干扰                           │
│     - 激光照射芯片特定区域                          │
│     导致计算错误，可能泄露密钥                      │
│                                                     │
│  3. 微码漏洞                                        │
│     - SGAxe/CrossTalk等SGX漏洞                     │
│     - 微码更新可能导致Enclave被攻破                 │
│     Intel通过微码更新修补                           │
│                                                     │
│  4. EPC分页攻击                                     │
│     - EPC页面换出时可能被截获                       │
│     - 需要加密保护换出的页面                        │
│                                                     │
│  5. 供应链攻击                                      │
│     - 硬件制造过程中的植入                         │
│     - 难以检测和防御                               │
└─────────────────────────────────────────────────────┘
```


> **Note:** TEE提供了高性能的安全计算环境，比纯密码学方案快几个数量级。但需要信任硬件厂商，且存在侧信道风险。实践中常与密码学技术组合使用。
