# 横向联邦与纵向联邦


## 横向联邦与纵向联邦


联邦学习HFLVFL特征对齐


横向联邦和纵向联邦是联邦学习的两大范式，分别解决样本切分和特征切分场景下的协作学习问题。


## 横向联邦学习 (Horizontal FL)


```
核心思想：各参与方拥有相同特征空间、不同样本空间的数据

场景举例：
┌─────────────────────────────────────────────────────┐
│  同一地区的不同银行，各自持有不同客户的数据            │
│  特征空间相同：收入、信用评分、年龄、职业等           │
│  样本空间不同：银行A有客户1~1000                      │
│               银行B有客户1001~2000                   │
│               银行C有客户2001~3000                   │
└─────────────────────────────────────────────────────┘

数据分布示意：
  银行A          银行B          银行C
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 收入  信用 │  │ 收入  信用 │  │ 收入  信用 │
│ 5000  650 │  │ 8000  720 │  │ 3500  580 │
│ 7200  710 │  │ 6500  690 │  │ 9000  750 │
│ ...    ...│  │ ...    ...│  │ ...    ...│
│ 样本1-1000│  │ 样本1001  │  │ 样本2001  │
└──────────┘  └──────────┘  └──────────┘

训练流程 (FedAvg)：
1. 服务器初始化全局模型参数 w0
2. 每轮通信 t = 1, 2, ..., T:
   a. 服务器随机选择 K 个客户端参与训练
   b. 服务器下发当前全局模型 wt 到各选中客户端
   c. 各客户端 k 使用本地数据 Dk 训练 E 个epoch:
      wk = wk - η∇L(wk; Dk)
   d. 各客户端上传本地模型更新 Δwk = wk - wt
   e. 服务器聚合：wt+1 = wt + Σ(nk/n) × Δwk

数学表达：
w(t+1) = Σ(k=1..K) (nk / n) × wk(t)

其中 nk 是客户端k的样本数，n = Σnk 是总样本数

FedAvg的收敛条件：
- 数据为IID（独立同分布）时收敛性好
- Non-IID时需要更多通信轮次或特殊处理

优势：
- 实现简单，通信效率较高
- 各方只上传模型参数，不暴露原始数据
- 天然支持并行训练

局限：
- 强假设：各方特征空间完全相同
- Non-IID数据导致收敛困难
- 模型更新仍可能泄露隐私信息
```


## 纵向联邦学习 (Vertical FL)


```
核心思想：各参与方拥有相同样本空间、不同特征空间的数据

场景举例：
┌─────────────────────────────────────────────────────┐
│  银行和电商平台对同一批用户持有互补的数据              │
│  样本空间相同：都是同一批用户（需要实体对齐）          │
│  特征空间不同：银行：收入、存款、贷款、信用评分       │
│               电商：购买记录、浏览历史、评价数据       │
└─────────────────────────────────────────────────────┘

数据分布示意：
  银行              电商              联合
┌────────────┐  ┌────────────┐  ┌──────────────┐
│用户 收入 贷款│  │用户 购买 浏览│  │收入 贷款 购买 浏览│
│张三 8000 Y  │  │张三 156 230 │  │8000 Y 156 230│
│李四 5000 N  │  │李四  89 120 │  │5000 N  89 120│
│王五 12000 Y │  │王五 302 450 │  │12000 Y 302 450│
└────────────┘  └────────────┘  └──────────────┘

关键步骤一：安全实体对齐 (PSI)
┌─────────────────────────────────────────────────────┐
│  隐私集合求交 (Private Set Intersection)             │
│  目标：在不暴露非交集数据的情况下找到共同用户          │
│                                                     │
│  基于哈希的PSI：                                     │
│  1. 各方对用户ID计算哈希：Hi = Hash(IDi || salt)    │
│  2. 交换哈希值，找到匹配项                           │
│  3. 使用盲化技术防止哈希被反查                       │
│                                                     │
│  基于OT的PSI：                                       │
│  使用不经意传输协议，更安全但效率更低                 │
│                                                     │
│  性能：O(n log n) 通信，支持百万级数据               │
└─────────────────────────────────────────────────────┘

关键步骤二：安全样本对齐后的联合训练

训练流程（以SecureBoost为例）：
1. 由主动方(Arbitrator)协调训练
2. 被动方(Passive Party)持有特征，不持有标签
3. 主动方或协调方持有标签
4. 加密梯度/中间结果在各方间传递

同态加密聚合方式：
┌─────────────────────────────────────────────────────┐
│  主动方（持有标签Y）                                 │
│  1. 计算加密梯度：Enc(g)                            │
│  2. 将Enc(g)发送给被动方                            │
│                                                     │
│  被动方（持有特征X）                                 │
│  1. 在密文上计算：Enc(g × X)                        │
│  2. 返回Enc(结果)给主动方                           │
│                                                     │
│  主动方：                                           │
│  1. 解密得到梯度，更新模型                          │
│  2. 被动方始终看不到明文梯度                        │
└─────────────────────────────────────────────────────┘

纵向联邦的典型算法：
- SecureBoost：联邦梯度提升树
- SecureLR：联邦逻辑回归
- SecureNN：联邦神经网络
- VERTIGO：联邦矩阵分解
```


## 联邦迁移学习 (Federated Transfer Learning)


```
核心思想：各参与方的样本空间和特征空间都不同，通过迁移学习桥接

场景：
┌─────────────────────────────────────────────────────┐
│  中国的银行A（中国用户，特征：收入、房产）             │
│  和美国的电商B（美国用户，特征：购买、浏览）           │
│  样本不同，特征也不同                                 │
│  希望联合学习提升各自的推荐/风控模型                   │
└─────────────────────────────────────────────────────┘

与HFL/VFL的区别：
┌──────────────┬──────────┬──────────┬──────────────┐
│              │ 横向FL   │ 纵向FL   │ 联邦迁移学习  │
├──────────────┼──────────┼──────────┼──────────────┤
│ 样本空间      │ 不同     │ 相同     │ 不同         │
│ 特征空间      │ 相同     │ 不同     │ 不同         │
│ 实体对齐      │ 不需要   │ 需要PSI  │ 需要特征映射  │
│ 核心技术      │ 模型平均 │ 加密计算  │ 迁移学习     │
│ 数据重叠      │ 无重叠   │ 样本重叠  │ 可能少量重叠  │
│ 典型场景      │ 跨机构   │ 跨行业   │ 跨地域/跨域   │
└──────────────┴──────────┴──────────┴──────────────┘

FTL技术方案：
1. 特征表示迁移
   - 各方训练本地特征提取器
   - 通过迁移对齐不同域的特征分布
   - 使用MMD/对抗训练减小域差异

2. 联邦知识蒸馏
   - 各方用本地数据训练教师模型
   - 通过中间层输出作为知识进行迁移
   - 不需要直接共享数据或梯度

3. 联邦域适应
   - 利用GAN进行特征空间映射
   - 共享判别器的对抗信息
   - 在特征层面进行对齐

FTL训练框架：
┌─────────────────────────────────────────────────────┐
│  1. 双方各自训练本地特征提取器 fA, fB               │
│  2. 安全对齐少量公共样本（如有）                     │
│  3. 对齐后的特征拼接：h = concat(fA(xA), fB(xB))   │
│  4. 通过迁移损失对齐特征分布                        │
│  5. 共同训练下游任务模型                             │
│  6. 梯度/参数更新通过加密传递                        │
└─────────────────────────────────────────────────────┘

应用案例：
- 跨国金融风控：不同国家的银行联合建模
- 跨平台推荐：不同互联网平台用户画像互补
- 跨域医疗：不同病种的医院联合研究
```


> **Note:** 横向联邦是"数据行的切分"，纵向联邦是"数据列的切分"，联邦迁移学习则处理更一般的情况。实际中纵向联邦在金融交叉建模中应用最广泛。


## 纵向联邦中的安全机制


```
安全实体解析 (Secure Entity Resolution)：
┌─────────────────────────────────────────────────────┐
│  传统实体解析：直接比对ID/姓名，存在隐私泄露风险      │
│                                                     │
│  安全实体解析方案：                                  │
│  1. 基于密码学                                       │
│     - PSI-CA (PSI with Cardinality)：只求交集大小    │
│     - OPRF：伪随机函数上的不经意评估                  │
│     - 错误容忍PSI：处理ID拼写差异                    │
│                                                     │
│  2. 基于隐私记录链接                                  │
│     - 编码+加密：先对记录编码再比较                  │
│     - Bloom Filter编码：高效但有误判率               │
│     - MinHash近似匹配：处理近似重复                  │
│                                                     │
│  3. 混合方案                                          │
│     - 使用TEE加速PSI计算                            │
│     - 差分隐私保护匹配结果                           │
└─────────────────────────────────────────────────────┘

特征对齐与选择：
- 在安全对齐后，双方需要协商共同使用的特征
- 特征选择算法需在加密状态下运行
- 使用同态加密计算特征重要性

差分隐私增强：
- 在纵向联邦中添加DP噪声保护梯度
- 使用安全聚合隐藏各方贡献
- 通过审计机制检测隐私泄露风险

通信安全：
- TLS/SSL加密通信信道
- 数字签名验证消息完整性
- 身份认证防止中间人攻击
```


## HFL与VFL工程化对比


```
通信模式对比：
┌─────────────────────────────────────────────────────┐
│  横向联邦 (HFL)：星形拓扑                            │
│  Server ←→ ClientA                                  │
│    ↕                                                 │
│  ClientB ←→ ClientC                                 │
│  - 服务器为中心，客户端间不通信                       │
│  - 通信内容：模型参数/梯度                           │
│  - 通信量：O(K × |w|) 每轮，K=客户端数              │
│                                                     │
│  纵向联邦 (VFL)：环形或链式拓扑                      │
│  Arbitrator ←→ PartyA ←→ PartyB                     │
│  - 需要协调方（或主动方）                            │
│  - 通信内容：加密梯度/中间结果                       │
│  - 通信量：O(T × |g|) 每轮，T=训练步数             │
└─────────────────────────────────────────────────────┘

计算开销对比：
┌────────────────┬────────────┬────────────────────┐
│                │ HFL        │ VFL                │
├────────────────┼────────────┼────────────────────┤
│ 主要计算       │ 本地训练    │ 加密/解密运算       │
│ 计算瓶颈       │ 客户端算力  │ 加密操作的开销      │
│ 并行度         │ 高（客户端）│ 低（顺序依赖）      │
│ 加速手段       │ GPU加速    │ 批量加密、HE优化    │
│ 通信轮次       │ 数百~数千   │ 数千~数万          │
└────────────────┴────────────┴────────────────────┘

实际部署考量：
- HFL适合：同类机构合作（多家银行/医院）
- VFL适合：跨行业合作（银行+电商/保险+医疗）
- 混合方案：某些特征共享（VFL）、某些样本共享（HFL）
- 监管合规：数据不出域是核心要求
```


> **Note:** 选择HFL还是VFL取决于数据分布：同特征不同样本选HFL，同样本不同特征选VFL，都不同则考虑联邦迁移学习。
