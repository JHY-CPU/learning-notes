# 服务注册发现最佳实践

## 一、实践清单

```yaml
部署:
  □ 注册中心至少 3 节点
  □ 跨机房部署
  □ 定期备份配置

配置:
  □ 健康检查频率合理（10-30秒）
  □ 开启自我保护模式
  □ 配置命名空间隔离环境

运维:
  □ 监控注册中心状态
  □ 优雅上下线
  □ 客户端缓存开启
```

## 二、常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| 服务找不到 | 注册失败 | 检查网络和配置 |
| 调用超时 | 服务不可用 | 检查健康状态 |
| 数据不一致 | 同步延迟 | 调整同步频率 |
| 内存溢出 | 服务太多 | 分片或扩容 |

## 三、优雅上下线

优雅上下线是避免请求丢失的关键。上线时先注册服务，等待健康检查通过后再开放流量；下线时先从注册中心摘除实例，等待进行中的请求处理完毕（通常等待 5-10 秒），再关闭进程。Spring Boot 可通过 actuator 的 shutdown endpoint 配合 preStop 钩子实现。

## 四、优缺点

**优点：**
- 规范化实践减少线上故障
- 客户端缓存提高注册中心不可用时的可用性
- 命名空间隔离避免环境间互相干扰

**缺点：**
- 优雅上下线增加部署流程复杂度
- 客户端缓存可能导致短暂使用已下线实例
- 监控告警需要额外投入运维成本

```yaml
scrape_configs:
  - job_name: 'nacos'
    metrics_path: '/nacos/actuator/prometheus'
    static_configs:
      - targets: ['nacos1:8848', 'nacos2:8848', 'nacos3:8848']

# 关键告警指标
groups:
  - name: nacos-alerts
    rules:
      - alert: NacosClusterSplit
        expr: nacos_cluster_node_count < 3
        for: 1m
        labels:
          severity: critical
      - alert: NacosServiceCountAbnormal
        expr: abs(nacos_service_count - nacos_service_count offset 1h) > 50
        for: 5m
        labels:
          severity: warning
      - alert: NacosSyncDelay
        expr: nacos_sync_delay_seconds > 30
        for: 2m
        labels:
          severity: warning
```

## 六、容灾降级方案

```
注册中心不可用时的降级方案:
├── 客户端缓存降级
│   ├── Eureka 客户端默认缓存 30 秒的服务列表
│   ├── Nacos 客户端也有本地缓存
│   └── 注册中心挂掉后，缓存仍可提供服务发现
├── 本地文件降级
│   ├── 将关键服务地址写入本地文件
│   ├── 注册中心不可用时读取本地文件
│   └── 定期从注册中心更新本地文件
└── 静态配置降级
    ├── 在配置文件中保留核心服务地址
    └── 作为最后手段使用
```

```java
// 本地降级实现
@Component
public class ServiceDiscoveryFallback {
    private static final String FALLBACK_FILE = "/tmp/service-instances.json";

    @Value("${discovery.fallback.enabled:true}")
    private boolean fallbackEnabled;

    public List<ServiceInstance> getInstances(String serviceName) {
        try {
            return discoveryClient.getInstances(serviceName);
        } catch (Exception e) {
            if (fallbackEnabled) {
                return loadFromLocalCache(serviceName);
            }
            throw e;
        }
    }

    // 定期备份到本地
    @Scheduled(fixedRate = 30000)
    public void backupToLocal() {
        List<String> services = discoveryClient.getServices();
        Map<String, List<ServiceInstance>> backup = new HashMap<>();
        for (String service : services) {
            backup.put(service, discoveryClient.getInstances(service));
        }
        // 写入本地文件
        writeToFile(FALLBACK_FILE, backup);
    }
}
```

## 七、注意事项

1. **注册中心单点部署** - 宕机后整个服务发现不可用，必须至少 3 节点
2. **健康检查频率过高** - 给注册中心和服务带来额外压力，10-30 秒足够
3. **注册中心与服务混合部署** - 资源竞争影响稳定性，应该独立部署
4. **忘记清理已下线服务** - 注册中心数据膨胀，设置过期清理策略
5. **日志和监控不足** - 问题难以排查，关键指标必须有告警
6. **容灾降级方案必须有** - 注册中心不可用时客户端缓存 + 本地文件兜底
