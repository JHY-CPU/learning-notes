# 服务元数据

## 一、元数据类型

```yaml
# Nacos 服务元数据
spring:
  cloud:
    nacos:
      discovery:
        metadata:
          version: v1.0
          region: cn-east
          zone: az1
          tags: high-priority
          weight: 100
```

## 二、元数据用途

| 用途 | 说明 |
|------|------|
| 版本路由 | 灰度发布，按版本分流 |
| 区域路由 | 就近访问，减少延迟 |
| 权重控制 | 流量分配比例 |
| 标签路由 | 按标签筛选服务 |

## 三、灰度发布

```java
// 按版本路由
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id) {
    // v1 版本调用 user-service:v1
    // v2 版本调用 user-service:v2
    return userClient.getUser(id);
}
```

## 四、工作原理

服务元数据以键值对形式随服务实例一起注册到注册中心。消费者从注册中心获取服务实例列表时，会同时获取每个实例的元数据。客户端根据元数据进行二次过滤和路由决策。元数据变更后，注册中心会通过订阅机制推送给所有消费者，实现动态路由调整。Spring Cloud 的 metadata map 会自动映射到 ServiceInstance 的 getMetadata() 方法中。

## 五、优缺点

**优点：**
- 无需修改代码即可实现灵活的路由策略
- 支持灰度发布、多版本共存、就近访问等高级特性
- 与注册中心天然集成，零额外组件

**缺点：**
- 元数据容量有限，不适合存储大量业务数据
- 元数据变更传播有延迟，极端情况下短暂不一致
- 客户端路由逻辑复杂度随元数据维度增加而上升

## 六、最佳实践

1. 元数据使用简短的键值对，避免存储超过 1KB 的数据
2. 统一元数据命名规范，如 version、region、zone、tags
3. 版本路由配合 Ribbon/LoadBalancer 自定义规则使用
4. 区域路由实现同区域优先、跨区域兜底策略

## 七、常见陷阱

1. **元数据变更后消费者未及时刷新**，导致路由到旧版本实例
2. **元数据大小超过注册中心限制**，导致注册失败
3. **多个维度组合过滤导致无可用实例**，需设置兜底策略
4. **灰度发布只改元数据不改代码**，新人难以理解路由逻辑
