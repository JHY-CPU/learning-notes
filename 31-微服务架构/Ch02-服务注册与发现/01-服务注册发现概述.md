# 服务注册发现概述

## 一、为什么需要服务发现

```yaml
传统方式的问题:
  - 硬编码服务地址，扩展困难
  - 无法自动感知服务上下线
  - 负载均衡需要手动配置

服务发现解决:
  - 服务自动注册
  - 客户端自动发现
  - 健康检查自动剔除
```

## 二、核心组件

```
Service Registry (注册中心)
├── 服务注册: 服务启动时注册自身地址
├── 服务发现: 客户端查询可用服务列表
├── 健康检查: 定期检测服务是否存活
└── 服务剔除: 移除不健康的服务实例

服务A (Provider)  →  注册  →  Registry
服务B (Consumer)  →  发现  →  Registry →  调用  →  服务A
```

## 三、主流注册中心对比

| 组件 | 一致性 | AP/CP | 语言 | 特点 |
|------|--------|-------|------|------|
| Nacos | AP+CP | AP/CP | Java | 阿里开源，配置+注册 |
| Eureka | AP | AP | Java | Netflix，简单 |
| Consul | CP | CP | Go | HashiCorp，功能全 |
| ZooKeeper | CP | CP | Java | 强一致，老牌 |

## 四、Nacos 快速入门

```bash
# Docker 启动 Nacos
docker run -d --name nacos \
  -p 8848:8848 \
  -e MODE=standalone \
  nacos/nacos-server:v2.3.0
```

```java
// 服务注册
@SpringBootApplication
@EnableDiscoveryClient
public class UserServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(UserServiceApplication.class, args);
    }
}

// 服务发现
@FeignClient(name = "order-service")
public interface OrderClient {
    @GetMapping("/orders/{id}")
    Order getOrder(@PathVariable Long id);
}
```

## 五、注意事项

1. **注册中心是微服务的核心组件**，必须高可用
2. **Nacos 国内最流行**，支持 AP 和 CP 模式
3. **健康检查频率要合理**，太频繁增加负载
4. **客户端缓存服务列表**，注册中心挂了仍可工作
5. **生产环境注册中心至少 3 节点**
