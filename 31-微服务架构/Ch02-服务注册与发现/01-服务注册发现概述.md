# 服务注册发现概述

## 一、为什么需要服务发现

```yaml
传统方式的问题:
  - 硬编码服务地址，扩展困难
  - 无法自动感知服务上下线
  - 负载均衡需要手动配置

服务发现解决:
  - 服务自动注册
  - 客户端自动发现
  - 健康检查自动剔除
```

## 二、核心组件

```
Service Registry (注册中心)
├── 服务注册: 服务启动时注册自身地址
├── 服务发现: 客户端查询可用服务列表
├── 健康检查: 定期检测服务是否存活
└── 服务剔除: 移除不健康的服务实例

服务A (Provider)  →  注册  →  Registry
服务B (Consumer)  →  发现  →  Registry →  调用  →  服务A
```

## 三、主流注册中心对比

| 组件 | 一致性 | AP/CP | 语言 | 特点 |
|------|--------|-------|------|------|
| Nacos | AP+CP | AP/CP | Java | 阿里开源，配置+注册 |
| Eureka | AP | AP | Java | Netflix，简单 |
| Consul | CP | CP | Go | HashiCorp，功能全 |
| ZooKeeper | CP | CP | Java | 强一致，老牌 |

## 五、服务注册发现工作流程

```
服务启动流程:
1. 服务启动 → 初始化 Spring Context
2. 向注册中心发送注册请求（服务名 + IP + 端口 + 元数据）
3. 注册中心保存服务实例信息
4. 定期发送心跳续约（默认 30 秒）

消费者调用流程:
1. 消费者向注册中心查询服务列表
2. 注册中心返回可用实例列表
3. 消费者本地缓存服务列表
4. 负载均衡选择一个实例发起调用
5. 定期刷新本地缓存

故障处理流程:
1. 注册中心检测到实例心跳超时
2. 标记实例为不健康
3. 超过阈值后从注册表中移除
4. 通知消费者更新本地缓存
```

## 六、CAP 定理与注册中心

```
CAP 定理:
├── Consistency (一致性) - 所有节点看到相同数据
├── Availability (可用性) - 每个请求都能收到响应
└── Partition tolerance (分区容错) - 网络分区时系统继续工作

注册中心选择:
├── AP 模式 (Nacos/Eureka) - 优先可用性
│   └── 适合大多数业务场景，允许短暂数据不一致
├── CP 模式 (Consul/ZooKeeper) - 优先一致性
│   └── 适合金融等强一致性要求场景
└── Nacos 支持 AP/CP 切换 - 灵活选择
```

## 七、注意事项

1. **注册中心是微服务的核心组件** - 必须高可用，至少 3 节点
2. **Nacos 国内最流行** - 支持 AP 和 CP 模式，同时提供配置管理
3. **健康检查频率要合理** - 10-30 秒，太频繁增加负载
4. **客户端缓存服务列表** - 注册中心挂了仍可工作，提高可用性
5. **生产环境注册中心至少 3 节点** - 避免单点故障
6. **服务名命名要规范** - 统一小写，用连字符分隔（如 user-service）
