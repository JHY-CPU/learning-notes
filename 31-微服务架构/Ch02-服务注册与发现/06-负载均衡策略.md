# 负载均衡策略

## 一、负载均衡算法

| 算法 | 说明 | 适用场景 |
|------|------|---------|
| 轮询 | 依次分配 | 通用 |
| 加权轮询 | 按权重分配 | 机器性能不同 |
| 随机 | 随机选择 | 简单场景 |
| 最少连接 | 选择连接最少的 | 长连接场景 |
| 一致性哈希 | 相同请求到同一节点 | 缓存场景 |

```java
// Spring Cloud LoadBalancer
@Bean
@LoadBalanced
public RestTemplate restTemplate() {
    return new RestTemplate();
}

// 自定义负载均衡
@Bean
public ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
    Environment environment, ServiceInstanceListSupplier supplier) {
    return new RandomLoadBalancer(supplier, "user-service");
}
```

## 二、工作原理

负载均衡分为客户端和服务端两种模式。客户端负载均衡（如 Spring Cloud LoadBalancer）在调用方本地维护服务实例列表，通过算法直接选择目标实例发起请求，无需经过额外代理。服务端负载均衡（如 Nginx、F5）则在流量入口统一分发。客户端模式减少了一次网络跳转，延迟更低，但需要每个调用方自行实现。一致性哈希算法通过构建虚拟节点环，将请求参数哈希到环上，顺时针找到最近节点，保证相同请求总是路由到同一实例。

## 三、优缺点

**优点：**
- 客户端负载均衡减少中间环节，降低延迟
- 多种算法适配不同业务场景
- 可与服务发现联动，自动剔除不健康实例

**缺点：**
- 客户端实现复杂度高于服务端方案
- 权重调整需要重新加载配置，存在短暂不一致
- 一致性哈希实现复杂，虚拟节点数需调优

## 四、最佳实践

1. 通用场景使用轮询或加权轮询即可
2. 有状态服务（如缓存）使用一致性哈希
3. 权重支持动态调整，配合监控自动适配
4. 健康检查失败的实例立即从候选列表移除

## 五、常见陷阱

1. **权重为零的实例被忽略**，可能导致流量全部集中到少数节点
2. **雪崩效应**：某个实例变慢导致请求堆积，需配合超时和熔断
3. **本地缓存的服务列表过期**，需要设置合理的刷新间隔
4. **长连接场景下轮询效果差**，应使用最少连接算法
