# 注册中心高可用

## 一、核心概念

注册中心是微服务架构的"心脏"，一旦宕机，新服务无法注册，消费者无法获取最新服务列表。本节介绍集群部署方案、脑裂问题处理、容灾降级策略。

### 1.1 高可用目标

- **无单点故障**：至少 3 节点部署
- **数据一致性**：集群节点之间的注册数据保持一致
- **故障自动恢复**：节点故障后自动重新加入集群
- **客户端无感**：注册中心故障对业务透明

## 三、脑裂问题与处理

### 3.1 什么是脑裂

脑裂（Split Brain）指网络分区导致集群分裂为多个独立子集群，每个子集群都认为自己是"主"，产生数据不一致。

```
网络分区前:
┌─────────────────────────────────────┐
│         Nacos Cluster (3节点)        │
│  Node1(Leader) ── Node2 ── Node3    │
└─────────────────────────────────────┘

网络分区后（脑裂）:
┌──────────────────┐  ┌───────────────┐
│  分区A            │  │  分区B         │
│  Node1 (Leader)  │  │  Node2        │
│  Node3           │  │  Node3 (新选)  │
│  服务A 注册在此   │  │  服务B 注册在此 │
└──────────────────┘  └───────────────┘
```

### 3.2 脑裂防护机制

Raft 协议的 Majority 机制：3 节点集群需要至少 2 节点同意才能选出 Leader；5 节点集群需要至少 3 节点。当网络分区发生时，少数派分区自动降级为只读。

```yaml
nacos:
  core:
    protocol:
      raft:
        election_timeout_ms: 5000
        heartbeat_interval_ms: 1000
        snapshot_interval_secs: 300
        max_snapshot_count: 50000
```

### 3.3 脑裂检测与恢复

```java
@Component
public class SplitBrainDetector {

    @Scheduled(fixedRate = 10000)
    public void detectSplitBrain() {
        List<ClusterNode> allNodes = clusterService.getAllNodes();
        int reachableCount = 0;
        for (ClusterNode node : allNodes) {
            if (isReachable(node)) reachableCount++;
        }
        int majority = allNodes.size() / 2 + 1;
        if (reachableCount < majority) {
            clusterService.setReadOnly(true);
            alertService.send("SPLIT_BRAIN_DETECTED",
                "Cluster split brain detected");
        }
    }
}
```

## 四、容灾降级策略

### 4.1 客户端本地缓存

```java
@Component
public class LocalCacheServiceRegistry {

    private final Map<String, List<ServiceInstance>> localCache =
        new ConcurrentHashMap<>();
    private final AtomicBoolean registryAvailable =
        new AtomicBoolean(true);

    public List<ServiceInstance> getInstances(String serviceId) {
        try {
            List<ServiceInstance> instances =
                nacosClient.getInstances(serviceId);
            localCache.put(serviceId, instances);
            persistToDisk(serviceId, instances);
            registryAvailable.set(true);
            return instances;
        } catch (Exception e) {
            registryAvailable.set(false);
            return localCache.getOrDefault(
                serviceId, Collections.emptyList());
        }
    }
}
```

### 4.2 多级降级策略

```
Level 0: 正常模式   → 从注册中心实时获取
Level 1: 缓存降级   → 使用客户端本地缓存
Level 2: 静态降级   → 使用配置文件中的静态地址
Level 3: DNS 降级   → 通过 DNS 解析获取地址
```

```java
@Component
public class MultiLevelFallbackDiscovery {

    public List<ServiceInstance> getInstances(String serviceId) {
        // Level 0
        try {
            List<ServiceInstance> instances =
                registryClient.getInstances(serviceId);
            if (!instances.isEmpty()) return instances;
        } catch (Exception ignored) {}

        // Level 1
        List<ServiceInstance> cached = localCache.get(serviceId);
        if (cached != null && !cached.isEmpty()) return cached;

        // Level 2
        List<String> backupUrls = backupConfig.get(serviceId);
        if (backupUrls != null) {
            return backupUrls.stream()
                .map(url -> new DefaultServiceInstance(
                    serviceId, url, extractPort(url)))
                .collect(Collectors.toList());
        }

        throw new ServiceUnavailableException(serviceId);
    }
}
```

## 五、监控与告警

### 5.1 关键监控指标

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| Leader 状态 | 当前节点角色 | 非预期角色告警 |
| 健康检查失败数 | 每分钟失败次数 | > 10/min |
| 客户端请求量 | QPS 变化 | 突降 > 50% |
| 推送延迟 | 数据同步延迟 | > 1000ms |

### 5.2 告警规则

```yaml
groups:
  - name: nacos-alerts
    rules:
      - alert: NacosClusterNodeDown
        expr: up{job="nacos-cluster"} == 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Nacos 节点宕机"

      - alert: NacosClusterSplitBrain
        expr: count(leader_status{leader="true"}) > 1
        for: 30s
        labels:
          severity: critical
        annotations:
          summary: "检测到脑裂"
```

## 六、最佳实践

1. **至少 3 节点部署**，建议奇数个节点（3 或 5）
2. **部署在不同物理机/可用区**，避免同机房故障
3. **客户端务必开启本地缓存**，注册中心故障时仍可工作
4. **定期备份注册数据**（Nacos 使用 MySQL 持久化）
5. **监控注册中心集群健康状态**，配置多级告警
6. **跨机房/跨可用区部署**提高容灾能力
7. **设置合理的超时和重试参数**，避免客户端长时间阻塞
8. **灰度变更注册中心配置**，避免一次性变更导致全量故障
