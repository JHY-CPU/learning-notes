# 多注册中心与注册中心对比

## 一、核心概念

在微服务架构中，服务注册中心是核心基础设施。当企业面临技术栈迁移、多云部署或异构系统集成时，往往需要同时使用多个注册中心。本节对比主流注册中心（Eureka、Consul、ZooKeeper、Nacos），并介绍多注册中心协同方案。

### 1.1 注册中心核心职责

- **服务注册**：服务启动时将自身地址、端口、元数据注册到注册中心
- **服务发现**：消费者从注册中心获取服务提供者的地址列表
- **健康检查**：定期检测服务实例的健康状态，剔除不健康实例
- **配置管理**：部分注册中心（如 Nacos）同时提供配置管理能力

## 二、主流注册中心对比

### 2.1 对比总览

| 特性 | Eureka | Consul | ZooKeeper | Nacos |
|------|--------|--------|-----------|-------|
| 开发语言 | Java | Go | Java | Java |
| CAP 模型 | AP | CP | CP | AP/CP 可切换 |
| 一致性协议 | 自研（Peer-to-Peer） | Raft | ZAB | Raft/Distro |
| 健康检查 | 客户端心跳 | 多种（HTTP/TCP/gRPC/脚本） | 会话心跳 | 客户端心跳/服务端探测 |
| 配置管理 | 不支持 | 支持 KV 存储 | 支持（不推荐） | 原生支持 |
| 跨数据中心 | 不支持 | 支持 | 不支持 | 支持 |
| 社区活跃度 | 停止维护 | 活跃 | 稳定 | 非常活跃 |
| Spring Cloud 集成 | 原生 | 原生 | 需额外适配 | 原生 |

### 2.2 Eureka

Eureka 是 Netflix 开源的服务发现组件，采用 AP 模型，保证高可用性。

```java
// Eureka Server 配置
@SpringBootApplication
@EnableEurekaServer
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
```

```yaml
# application.yml
server:
  port: 8761
eureka:
  instance:
    hostname: eureka-server
  client:
    register-with-eureka: false
    fetch-registry: false
    service-url:
      defaultZone: http://eureka-server-peer:8761/eureka/
  server:
    enable-self-preservation: true
    eviction-interval-timer-in-ms: 60000
```

**优点**：简单易用、Spring Cloud 原生集成、AP 模型保证高可用
**缺点**：Netflix 已停止维护 2.x，1.x 功能相对简单

### 2.3 Consul

Consul 是 HashiCorp 开源的服务网格解决方案，采用 CP 模型（基于 Raft 协议）。

```yaml
# Spring Cloud Consul 配置
spring:
  cloud:
    consul:
      host: consul-server
      port: 8500
      discovery:
        service-name: ${spring.application.name}
        health-check-interval: 10s
        prefer-ip-address: true
        tags: version=1.0,env=prod
```

**优点**：健康检查机制丰富、原生支持多数据中心、内置 KV 存储和 DNS 接口
**缺点**：CP 模型在网络分区时可能短暂不可用

### 2.4 ZooKeeper

ZooKeeper 是 Apache 基金会的分布式协调服务，广泛用于分布式锁和配置管理。

```yaml
spring:
  cloud:
    zookeeper:
      connect-string: zk1:2181,zk2:2181,zk3:2181
      discovery:
        enabled: true
        root: /services
      connection-timeout: 10s
```

**优点**：成熟的分布式协调能力、强一致性保证
**缺点**：会话管理复杂、不适合大规模服务注册场景

### 2.5 Nacos

Nacos 是阿里巴巴开源的服务发现和配置管理平台，支持 AP/CP 模式切换。

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos1:8848,nacos2:8848,nacos3:8848
        namespace: prod-namespace-id
        group: DEFAULT_GROUP
        cluster-name: cn-east
        metadata:
          version: 1.0
          region: east-china
      config:
        server-addr: nacos1:8848,nacos2:8848,nacos3:8848
        file-extension: yaml
        shared-configs:
          - data-id: common.yaml
            group: DEFAULT_GROUP
            refresh: true
```

**优点**：同时支持服务发现和配置管理、AP/CP 兼顾、国产开源社区活跃
**缺点**：相对较新，大规模生产验证不如 Consul

## 三、多注册中心架构

### 3.1 双注册方案

在迁移或异构场景中，服务同时注册到两个注册中心：

```yaml
# 同时注册到 Nacos 和 Consul
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-server:8848
        enabled: true
    consul:
      host: consul-server
      port: 8500
      discovery:
        enabled: true
```

### 3.2 注册中心同步架构

```
                    ┌──────────────┐
                    │  同步服务     │
                    └──┬───────┬───┘
                       │       │
              ┌────────┘       └────────┐
              ▼                         ▼
     ┌────────────────┐       ┌────────────────┐
     │  Nacos Cluster │       │ Consul Cluster  │
     │                │◄─────►│                │
     │  服务A, 服务B  │  同步  │  服务A, 服务B  │
     └────────────────┘       └────────────────┘
              │                         │
              ▼                         ▼
     ┌────────────────┐       ┌────────────────┐
     │  消费者 Group1  │       │  消费者 Group2  │
     └────────────────┘       └────────────────┘
```

### 3.3 同步服务实现

```java
@Service
public class RegistrySyncService {

    @Autowired
    private NacosDiscoveryClient nacosClient;

    @Autowired
    private ConsulDiscoveryClient consulClient;

    /**
     * 定期同步服务实例
     */
    @Scheduled(fixedRate = 30000)
    public void syncServices() {
        List<String> nacosServices = nacosClient.getServices();
        for (String serviceId : nacosServices) {
            List<ServiceInstance> instances =
                nacosClient.getInstances(serviceId);
            syncToConsul(serviceId, instances);
        }
    }

    private void syncToConsul(String serviceId,
                               List<ServiceInstance> instances) {
        for (ServiceInstance instance : instances) {
            ConsulRegistration registration =
                buildRegistration(serviceId, instance);
            consulClient.register(registration);
        }
    }
}
```

## 四、双写方案保证数据一致性

### 4.1 写入流程

```java
@Service
public class DualWriteRegistryService {

    @Autowired
    private NacosServiceRegistry nacosRegistry;

    @Autowired
    private ConsulServiceRegistry consulRegistry;

    @Transactional
    public void register(ServiceRegistration registration) {
        // 先注册到主注册中心（Nacos）
        nacosRegistry.register(registration);
        try {
            // 再注册到副注册中心（Consul）
            consulRegistry.register(registration);
        } catch (Exception e) {
            log.warn("Consul registration failed, will retry", e);
            retryQueue.add(registration);
        }
    }

    public void deregister(ServiceRegistration registration) {
        nacosRegistry.deregister(registration);
        consulRegistry.deregister(registration);
    }
}
```

### 4.2 最终一致性保障

```java
@Component
public class RegistryConsistencyChecker {

    @Scheduled(fixedRate = 60000)
    public void checkConsistency() {
        Map<String, List<ServiceInstance>> nacosInstances =
            getAllInstances(nacosClient);
        Map<String, List<ServiceInstance>> consulInstances =
            getAllInstances(consulClient);

        for (String serviceId : nacosInstances.keySet()) {
            Set<String> nacosAddrs = toAddressSet(
                nacosInstances.get(serviceId));
            Set<String> consulAddrs = toAddressSet(
                consulInstances.getOrDefault(
                    serviceId, Collections.emptyList()));

            Sets.SetView<String> missing = Sets.difference(
                nacosAddrs, consulAddrs);
            if (!missing.isEmpty()) {
                log.warn("Inconsistency for {}: missing in consul: {}",
                    serviceId, missing);
                syncMissing(serviceId, missing);
            }
        }
    }
}
```

## 五、最佳实践

### 5.1 选型建议

| 场景 | 推荐方案 |
|------|---------|
| Spring Cloud 技术栈 | Nacos（功能全面） |
| 已有 HashiCorp 生态（Vault/Terraform） | Consul |
| 强一致性要求（分布式锁） | ZooKeeper + Nacos |
| 多云/混合云部署 | Consul（多数据中心支持好） |
| 国内企业/信创环境 | Nacos（社区支持好） |

### 5.2 迁移策略

```
阶段1: 双注册   → 新旧注册中心同时运行
阶段2: 流量切换  → 消费者逐步切换到新注册中心
阶段3: 验证期   → 确认新注册中心稳定运行
阶段4: 下线旧   → 移除旧注册中心
```

### 5.3 注意事项

1. **多注册中心增加系统复杂度**，只在必要时（如迁移、多云）使用
2. **选择一个作为主要注册中心**，其他作为备份或过渡
3. **客户端缓存服务列表**，注册中心故障时仍能工作
4. **定期检查数据一致性**，防止注册中心之间数据不一致
5. **迁移期间做好监控**，特别关注服务调用成功率
6. **使用命名空间隔离**不同环境的服务注册
