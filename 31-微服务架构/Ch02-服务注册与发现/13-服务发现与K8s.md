# 服务发现与 Kubernetes

## 一、K8s 内置服务发现

```yaml
# K8s Service
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service
  ports:
  - port: 80
    targetPort: 8080
  type: ClusterIP

# 通过 DNS 访问
# user-service.default.svc.cluster.local
```

## 二、K8s 与注册中心

```yaml
# K8s 中使用 Nacos
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-headless:8848
        namespace: ${NAMESPACE}
```

## 三、工作原理

Kubernetes 内置服务发现基于 Service 和 DNS 实现。创建 Service 对象后，kube-proxy 在每个节点上维护 iptables 或 IPVS 规则，将 ClusterIP 流量转发到后端 Pod。CoreDNS 为每个 Service 创建 DNS A 记录和 SRV 记录，Pod 内通过服务名即可访问。Nacos 在 K8s 中部署时，通常以 StatefulSet 运行，通过 Headless Service 保证稳定的网络标识，应用通过环境变量或 ConfigMap 获取 Nacos 地址。Istio Sidecar 模式在每个 Pod 注入 Envoy 代理，接管所有进出流量，实现更细粒度的流量管理。

## 四、优缺点

**优点：**
- K8s 原生服务发现零额外组件，运维成本低
- Nacos 提供更强的配置管理和服务治理能力
- Istio 提供统一的流量控制、安全和可观测性

**缺点：**
- K8s 原生服务发现仅限集群内部，跨集群需额外方案
- Nacos + K8s 双重服务发现增加架构复杂度
- Istio Sidecar 增加资源开销和调用延迟

## 五、最佳实践

1. 纯 K8s 环境优先使用原生 Service + DNS
2. 需要配置管理或非 K8s 环境互通时引入 Nacos
3. 大规模微服务考虑 Istio 服务网格统一管理
4. 使用 CRD 扩展 K8s 服务发现能力

## 六、常见陷阱

1. **K8s Service 的 ClusterIP 只在集群内可达**，外部访问需 NodePort 或 LoadBalancer
2. **Headless Service 不分配 ClusterIP**，DNS 返回所有 Pod IP
3. **Istio Sidecar 注入后端口协议需正确标注**，否则流量被拒绝
4. **Nacos 和 K8s 同时注册服务**，可能导致双注册冲突
