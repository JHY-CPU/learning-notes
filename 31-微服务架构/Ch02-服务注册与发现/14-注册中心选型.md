# 注册中心选型

## 一、核心概念

注册中心选型需要综合考虑 CAP 模型、性能、生态、运维成本等因素。本节从多个维度对比主流注册中心，给出不同场景下的选型建议。

### 1.1 CAP 理论回顾

```
        Consistency (一致性)
              /\
             /  \
            /    \
           / 选2个 \
          /________\
   Availability    Partition
  (可用性)        Tolerance(分区容错性)

CP 模型：保证一致性，牺牲部分可用性（Consul、ZooKeeper）
AP 模型：保证可用性，牺牲部分一致性（Eureka、Nacos AP 模式）
```

## 二、选型维度对比

### 2.1 功能对比矩阵

| 维度 | Nacos | Eureka | Consul | ZooKeeper |
|------|-------|--------|--------|-----------|
| CAP | AP/CP | AP | CP | CP |
| 服务发现 | 原生 | 原生 | 原生 | 需适配 |
| 配置管理 | 原生 | 无 | KV存储 | 有限支持 |
| 健康检查 | 心跳/探测 | 心跳 | 多种方式 | 会话 |
| 多数据中心 | 支持 | 不支持 | 原生支持 | 不支持 |
| 管理界面 | 完善 | 基础 | 完善 | 无 |
| 社区活跃度 | 非常活跃 | 停止维护 | 活跃 | 稳定 |

### 2.2 性能对比

```
服务注册 QPS（3节点集群）:
  Nacos:    ~15000
  Consul:   ~10000
  ZooKeeper: ~8000
  Eureka:   ~12000

集群收敛时间:
  Nacos:    10-30s
  Consul:   5-15s
  ZooKeeper: 5-10s
  Eureka:   30-120s
```

### 2.3 运维复杂度

| 维度 | Nacos | Eureka | Consul | ZooKeeper |
|------|-------|--------|--------|-----------|
| 部署难度 | 低 | 低 | 中 | 中 |
| 集群搭建 | 简单 | 简单 | 中等 | 复杂 |
| 监控支持 | 内置 | 基础 | 内置 | 需额外工具 |
| 数据备份 | MySQL | 无 | 内置快照 | 内置快照 |

## 三、场景化选型建议

### 3.1 决策流程

```
是否需要配置管理？
  ├─ 是 → Nacos
  └─ 否 ↓
是否有多数据中心需求？
  ├─ 是 → Consul
  └─ 否 ↓
是否需要强一致性？
  ├─ 是 → ZooKeeper 或 Consul
  └─ 否 ↓
技术栈是什么？
  ├─ Spring Cloud → Nacos
  ├─ K8s 原生 → Consul
  └─ 传统 Java → Nacos 或 ZooKeeper
```

### 3.2 典型场景推荐

#### Spring Cloud 微服务（推荐 Nacos）

```yaml
spring:
  cloud:
    nacos:
      discovery:
        server-addr: nacos-cluster:8848
        namespace: ${ENV:dev}
      config:
        server-addr: nacos-cluster:8848
        file-extension: yaml
        shared-configs:
          - data-id: common.yaml
            refresh: true
```

#### Kubernetes 微服务（推荐 Consul）

```yaml
apiVersion: consul.hashicorp.com/v1alpha1
kind: ServiceDefaults
metadata:
  name: user-service
spec:
  protocol: http
```

#### 金融级强一致（推荐 ZooKeeper + Nacos）

```
核心交易系统 → ZooKeeper (强一致性)
非核心业务   → Nacos (高可用优先)
```

## 四、选型注意事项

1. **避免频繁更换注册中心**，迁移成本很高
2. **充分评估运维团队的技术栈**
3. **考虑长期维护性**，Eureka 已停止维护
4. **性能不是唯一标准**，稳定性和生态更重要
5. **先在非核心业务验证**，再推广到核心业务
6. **预留迁移方案**，使用适配层隔离差异

## 五、总结

| 你的需求 | 推荐方案 |
|---------|---------|
| 国内企业、Spring Cloud | **Nacos** |
| HashiCorp 生态、多云 | **Consul** |
| 强一致性、分布式协调 | **ZooKeeper** |
| 已有 Eureka 遗留系统 | **继续使用，逐步迁移到 Nacos** |
| K8s Service Mesh | **Consul Connect** |
