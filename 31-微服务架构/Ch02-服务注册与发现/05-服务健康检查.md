# 服务健康检查

## 一、健康检查方式

```java
// Spring Boot Actuator 健康端点
management:
  endpoints:
    web:
      exposure:
        include: health,info
  endpoint:
    health:
      show-details: always

// 自定义健康检查
@Component
public class DatabaseHealthIndicator implements HealthIndicator {
    @Override
    public Health health() {
        try {
            jdbcTemplate.queryForObject("SELECT 1", Integer.class);
            return Health.up().withDetail("database", "MySQL").build();
        } catch (Exception e) {
            return Health.down(e).build();
        }
    }
}
```

## 二、检查类型

| 类型 | 说明 | 频率 |
|------|------|------|
| 心跳检查 | 定期发送心跳 | 30 秒 |
| HTTP 检查 | 调用健康端点 | 10 秒 |
| TCP 检查 | 检查端口可达 | 5 秒 |
| 脚本检查 | 执行自定义脚本 | 按需 |

## 三、K8s 健康检查探针

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
spec:
  template:
    spec:
      containers:
      - name: user-service
        image: user-service:1.0
        livenessProbe:          # 存活探针 - 容器是否需要重启
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 30    # 启动后 30 秒开始检查
          periodSeconds: 10          # 每 10 秒检查一次
          failureThreshold: 3        # 连续 3 次失败则重启
        readinessProbe:         # 就绪探针 - 是否可以接收流量
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5
          failureThreshold: 3
        startupProbe:           # 启动探针 - 应用启动是否完成
          httpGet:
            path: /actuator/health
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5
          failureThreshold: 30       # 最多等 30*5=150 秒
```

## 四、健康检查深度示例

```java
// 复合健康检查 - 同时检查所有关键依赖
@Component
public class CompositeHealthIndicator implements HealthIndicator {
    @Autowired private DataSource dataSource;
    @Autowired private RedisConnectionFactory redisFactory;
    @Autowired private DiscoveryClient discoveryClient;

    @Override
    public Health health() {
        Health.Builder builder = Health.up();

        // 检查数据库
        try {
            dataSource.getConnection().close();
            builder.withDetail("database", "UP");
        } catch (Exception e) {
            builder.down().withDetail("database", "DOWN: " + e.getMessage());
        }

        // 检查 Redis
        try {
            redisFactory.getConnection().ping();
            builder.withDetail("redis", "UP");
        } catch (Exception e) {
            builder.down().withDetail("redis", "DOWN: " + e.getMessage());
        }

        // 检查依赖服务
        List<String> services = discoveryClient.getServices();
        builder.withDetail("registered-services", services.size());

        return builder.build();
    }
}
```

## 五、健康检查策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 心跳检查 | 简单、开销小 | 不能反映真实健康 | 轻量服务 |
| HTTP 探针 | 能检查业务逻辑 | 有网络开销 | Web 服务 |
| TCP 探针 | 通用性好 | 只检查端口 | 非 HTTP 服务 |
| 脚本检查 | 灵活性高 | 安全风险 | 复杂检查逻辑 |

## 六、注意事项

1. **健康检查要检查所有关键依赖** - 数据库、Redis、MQ 等
2. **检查频率要合理** - 太频繁增加负载，建议 10-30 秒
3. **区分存活探针和就绪探针** - liveness 重启容器，readiness 摘流量
4. **K8s 中配置 liveness 和 readiness 探针** - startupProbe 解决慢启动问题
5. **检查失败后要有自动恢复机制** - 避免人工干预
6. **避免健康检查依赖外部服务** - 检查本身不应该导致雪崩
