# 服务健康检查优化

## 一、核心概念

服务健康检查是注册中心的核心功能，用于自动发现和剔除不健康实例。本节深入介绍心跳机制、主动探测、健康检查优化策略。

### 1.1 健康检查方式对比

| 方式 | 原理 | 适用场景 | 优缺点 |
|------|------|---------|--------|
| 客户端心跳 | 实例定期向注册中心发送心跳 | Eureka、Nacos | 轻量但可能误判 |
| 服务端探测 | 注册中心主动探测实例 | Consul、Nacos 2.0 | 准确但增加负担 |
| 混合模式 | 心跳 + 探测结合 | 生产环境推荐 | 兼顾性能和准确性 |

## 二、心跳机制详解

### 2.1 Eureka 心跳机制

```
服务实例                          Eureka Server
   │                                  │
   │──── 注册服务 ────────────────────►│
   │                                  │
   │──── 心跳 (30s) ─────────────────►│  更新 lastRenewalTimestamp
   │──── 心跳 (30s) ─────────────────►│
   │     ...                          │
   │     (心跳丢失 90s)                │

```

    public void run() {

        }
    }

### 2.3 心跳优化策略

```java
@Component
public class AdaptiveHeartbeat {
    private long interval = 5000;
    private final long MIN_INTERVAL = 2000;
    private final long MAX_INTERVAL = 30000;

    public void adjustInterval(boolean lastBeatSuccess) {
        if (lastBeatSuccess) {
            interval = Math.min(interval * 1.2, MAX_INTERVAL);
        } else {
            interval = Math.max(interval * 0.5, MIN_INTERVAL);
        }
    }

    public void retryWithBackoff(Runnable beatTask) {
        int retryCount = 0;
        long backoff = 1000;
        while (retryCount < 5) {
            try {
                beatTask.run();
                return;
            } catch (Exception e) {
                retryCount++;
                backoff = Math.min(backoff * 2, 16000);
                sleep(backoff);
            }
        }
    }
}
```

@RestController
public class HealthCheckController {

    @Autowired
    private DataSource dataSource;

    @Autowired
    private RedisConnectionFactory redisFactory;

    @GetMapping("/actuator/health")
    public HealthCheckResult check() {
        HealthCheckResult result = new HealthCheckResult();
        result.setDb(checkDatabase());
        result.setRedis(checkRedis());
        result.setDisk(checkDiskSpace());
        return result;
    }

    private boolean checkDatabase() {
        try (Connection conn = dataSource.getConnection()) {
            return conn.isValid(2);
        } catch (SQLException e) {
            return false;
        }
    }
}

## 四、健康检查优化策略

### 4.1 分层健康检查

```java
@Component
public class TieredHealthCheck {

    // L1: 轻量级检查（每 5s）- 端口可达性
    @Scheduled(fixedRate = 5000)
    public void l1LightweightCheck() {
        boolean alive = isPortReachable("localhost", 8080);
        healthStatus.setL1(alive);
    }

    // L2: 标准检查（每 30s）- 依赖服务连通性
    @Scheduled(fixedRate = 30000)
    public void l2StandardCheck() {
        boolean healthy = checkDatabase() && checkRedis();
        healthStatus.setL2(healthy);
    }

    // L3: 深度检查（每 5min）- 业务功能验证
    @Scheduled(fixedRate = 300000)
    public void l3DeepCheck() {
        boolean functional = runSmokeTest();
        healthStatus.setL3(functional);
    }

    public HealthStatus getOverallStatus() {
        if (!healthStatus.isL1()) return HealthStatus.DOWN;
        if (!healthStatus.isL2()) return HealthStatus.OUT_OF_SERVICE;
        if (!healthStatus.isL3()) return HealthStatus.WARNING;
        return HealthStatus.UP;
    }
}
```

### 4.2 并行探测优化

```java
@Component
public class ParallelHealthChecker {
    private final ExecutorService executor =
        Executors.newFixedThreadPool(4);

    public Map<String, Boolean> checkAll(
            List<Dependency> dependencies) {
        List<CompletableFuture<CheckResult>> futures =
            dependencies.stream()
                .map(dep -> CompletableFuture.supplyAsync(
                    () -> checkSingle(dep), executor)
                    .exceptionally(e -> new CheckResult(
                        dep.getName(), false, e.getMessage())))
                .collect(Collectors.toList());

        return CompletableFuture.allOf(
                futures.toArray(new CompletableFuture[0]))
            .thenApply(v -> futures.stream()
                .map(CompletableFuture::join)
                .collect(Collectors.toMap(
                    CheckResult::getName,
                    CheckResult::isHealthy)))
            .join();
    }
}
```

### 4.3 健康检查结果缓存

@Component
public class CachedHealthChecker {

## 五、自我保护机制

### 5.1 Eureka 自我保护

```
正常: 实际心跳数 > 0.85 * 期望心跳数 → 正常模式
异常: 实际心跳数 < 0.85 * 期望心跳数 → 自我保护模式
  ┌─────────────────────────────────────┐
  │  • 不再剔除任何服务实例              │
  │  • 保留所有注册信息（包括过期的）     │
  │  • 等待网络恢复后自动退出            │
  └─────────────────────────────────────┘
```

### 5.2 Nacos 保护机制

```yaml
spring:
  cloud:
    nacos:
      discovery:
        naming-push-empty-protection: true
        healthy-instance-threshold: 0.1
```

## 六、最佳实践

1. **生产环境建议混合使用心跳和主动探测**
2. **心跳间隔建议 5-15 秒**，过短增加网络负担
3. **超时时间一般为心跳间隔的 3 倍**
4. **开启自我保护机制**，防止网络抖动导致误摘除
5. **分层健康检查**，轻量级频率高，深度检查频率低
6. **检查端点要做鉴权**，防止恶意调用
7. **健康检查要做超时控制**，避免级联故障
8. **记录健康检查日志**，便于故障排查
