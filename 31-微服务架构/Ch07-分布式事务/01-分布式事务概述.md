# 分布式事务概述

## 一、分布式事务挑战

```yaml
问题:
  - 多个服务数据库独立
  - 跨服务操作无法保证原子性
  - 网络不可靠导致数据不一致

解决方案:
  - 强一致性: 2PC/3PC/TCC
  - 最终一致性: Saga/本地消息表
  - 柔性事务: 尽量避免分布式事务
```

## 二、方案对比

| 方案 | 一致性 | 性能 | 复杂度 | 适用场景 |
|------|--------|------|--------|---------|
| 2PC | 强一致 | 低 | 中 | 数据库层 |
| TCC | 强一致 | 中 | 高 | 资金类 |
| Saga | 最终一致 | 高 | 中 | 长事务 |
| 本地消息表 | 最终一致 | 高 | 低 | 通用 |
| 事务消息 | 最终一致 | 高 | 低 | MQ 场景 |

## 三、工作原理

分布式事务的核心挑战在于跨服务操作无法通过单一数据库事务保证原子性。强一致性方案（2PC、TCC）通过协调者统一控制所有参与者的提交或回滚，保证任意时刻数据一致，但性能开销大。最终一致性方案（Saga、本地消息表、事务消息）允许中间状态短暂不一致，通过异步补偿或消息投递最终达成一致。柔性事务从设计层面规避分布式事务，如将多步操作合并到同一服务内。

## 四、优缺点

**优点：**
- 解决跨服务数据一致性问题
- 多种方案可选，适配不同业务场景
- 最终一致性方案性能接近本地事务

**缺点：**
- 强一致性方案性能和可用性受限于 CAP 定理
- 补偿逻辑增加代码复杂度
- 故障场景多，测试覆盖困难

## 五、分布式事务核心问题

```
核心挑战:
1. 原子性 - 所有参与者要么全成功，要么全失败
2. 隔离性 - 并发事务互不干扰
3. 持久性 - 提交后的结果不丢失
4. 可用性 - 协调者/参与者故障时的处理

CAP 定理约束:
├── 强一致性 (2PC/TCC) → 牺牲可用性
├── 最终一致性 (Saga/消息表) → 牺牲一致性窗口
└── 无事务 (业务设计规避) → 最佳性能
```

```java
// 场景分析: 订单创建
// 涉及: 扣库存、扣余额、创建订单

// 强一致性方案 (TCC):
// Try:   冻结库存、冻结余额、预创建订单
// Confirm: 确认扣减、确认订单
// Cancel: 解冻库存、解冻余额、取消订单
// 复杂度: 高，性能: 中

// 最终一致性方案 (本地消息表):
// Step 1: 扣库存 + 写消息表 (本地事务)
// Step 2: 定时投递消息到 MQ
// Step 3: 扣余额服务消费消息
// Step 4: 创建订单服务消费消息
// 复杂度: 低，性能: 高

// 业务规避方案:
// 将库存、余额、订单合并到一个服务中
// 使用单一数据库事务
// 复杂度: 最低，性能: 最高
```

## 六、Seata 框架概览

```
Seata 支持的事务模式:
├── AT 模式 - 无侵入，自动生成反向 SQL
├── TCC 模式 - 手动编写 Try/Confirm/Cancel
├── Saga 模式 - 状态机驱动长事务
└── XA 模式 - 数据库原生 XA 协议

Seata 架构:
├── TC (Transaction Coordinator) - 事务协调者
├── TM (Transaction Manager) - 事务管理器
└── RM (Resource Manager) - 资源管理器
```

## 七、常见陷阱

1. **所有场景都追求强一致** - 性能和可用性严重下降，应按需选择
2. **补偿操作不幂等** - 重试导致数据重复扣减
3. **忽略悬挂和空回滚问题** - TCC 场景下数据错乱
4. **分布式事务超时未设置** - 资源长时间锁定
5. **一开始就引入分布式事务** - 应先尝试业务设计规避
