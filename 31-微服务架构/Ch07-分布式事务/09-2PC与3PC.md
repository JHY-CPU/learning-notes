# 2PC 与 3PC

## 一、2PC (两阶段提交)

```
阶段1: Prepare - 询问所有参与者是否可以提交
阶段2: Commit/Rollback - 根据投票结果提交或回滚
```

## 二、3PC (三阶段提交)

```
阶段1: CanCommit - 询问是否可以提交
阶段2: PreCommit - 预提交
阶段3: DoCommit - 正式提交
```

## 三、工作原理

2PC（两阶段提交）由协调者统一控制。Prepare 阶段：协调者向所有参与者发送 Prepare 请求，参与者执行本地事务但不提交，锁定资源并回复 Yes/No。Commit/Rollback 阶段：若所有参与者回复 Yes，协调者发送 Commit；否则发送 Rollback。3PC 在 2PC 基础上增加 CanCommit 阶段（预检查）和超时机制，减少阻塞时间。CanCommit 阶段仅询问不锁定资源，PreCommit 阶段预提交，DoCommit 阶段正式提交。3PC 的超时机制使参与者在协调者故障时能自主决定提交或回滚。

## 四、优缺点

**2PC 优点：** 强一致性保证，实现相对简单
**2PC 缺点：** 同步阻塞（Prepare 后所有参与者锁定资源等待）、协调者单点故障、数据不一致风险（Commit 阶段部分参与者网络不可达）

**3PC 优点：** 减少阻塞时间，引入超时机制降低单点故障影响
**3PC 缺点：** 增加一次网络往返，超时后自主提交可能导致不一致

## 五、Seata XA 模式

```java
// Seata XA 模式 - 数据库原生分布式事务
@Service
public class OrderService {
    @GlobalTransactional(name = "create-order", timeoutMills = 30000)
    public void createOrder(Order order) {
        // 所有数据库操作自动参与 XA 事务
        orderDao.insert(order);
        inventoryFeign.deduct(order.getProductId(), order.getQuantity());
        accountFeign.deduct(order.getUserId(), order.getAmount());
        // 全部成功则提交，任一失败则回滚
    }
}
```

## 六、2PC/3PC 与现代方案对比

| 维度 | 2PC/3PC | Seata AT | TCC | Saga |
|------|---------|----------|-----|------|
| 一致性 | 强一致 | 强一致 | 强一致 | 最终一致 |
| 性能 | 极低 | 中 | 中 | 高 |
| 可用性 | 低 | 中 | 中 | 高 |
| 复杂度 | 低(数据库层) | 低 | 高 | 中 |
| 适用场景 | 传统数据库 | 通用 | 资金类 | 长事务 |
| 现代推荐 | 不推荐 | 推荐 | 推荐 | 推荐 |

> 2PC/3PC 主要用于理解分布式事务原理，现代微服务推荐使用 Seata AT/TCC 或 Saga

## 七、常见陷阱

1. **2PC 中协调者故障** - 参与者一直阻塞在 Prepare 状态，需要超时机制
2. **网络分区时 Commit 只到达部分参与者** - 数据不一致
3. **3PC 超时后自主提交** - 但其他参与者可能回滚了
4. **XA 事务在高并发下性能极差** - 不适合互联网场景
5. **在新项目中使用原生 2PC** - 应使用 Seata AT 模式替代
