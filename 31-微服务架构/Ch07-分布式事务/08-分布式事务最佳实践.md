# 分布式事务最佳实践

## 一、实践清单

```yaml
原则:
  □ 尽量避免分布式事务
  □ 优先用最终一致性
  □ 补偿操作要幂等
  □ 超时机制要有
  □ 监控事务状态

方案:
  □ 通用场景: 本地消息表
  □ MQ 场景: 事务消息
  □ 资金场景: TCC
  □ 长事务: Saga
```

## 二、工作原理

分布式事务最佳实践的核心原则是"尽量避免"。从业务设计层面规避分布式事务是最优解：合并服务边界、使用领域事件驱动、引入最终一致性。无法避免时，根据业务场景选择最简方案。通用场景用本地消息表（最简单），MQ 场景用事务消息（最自然），资金场景用 TCC（最强一致），长事务用 Saga（最灵活）。所有补偿操作必须幂等，所有方案都需要监控事务状态。

## 三、优缺点

**优点：**
- 避免分布式事务从根源上消除问题
- 最终一致性方案对性能影响极小
- 补偿机制兜底保障数据最终一致

**缺点：**
- 业务设计需要提前规划，改造成本高
- 最终一致性有延迟窗口，不适合实时性要求极高的场景
- 补偿逻辑的开发和维护成本不可忽视

## 四、事务监控方案

```java
// Seata 事务监控
@RestController
public class TransactionMonitorController {
    @GetMapping("/api/monitor/transactions")
    public TransactionStats getStats() {
        // 查询 Seata 中的全局事务状态
        return TransactionStats.builder()
            .activeCount(seataManager.getActiveTransactionCount())
            .committedCount(seataManager.getCommittedCount())
            .rolledBackCount(seataManager.getRolledBackCount())
            .timeoutCount(seataManager.getTimeoutCount())
            .build();
    }
}

// 本地消息表监控
@Scheduled(fixedRate = 60000)
public void monitorPendingMessages() {
    long pendingCount = outboxMessageMapper.countByStatus("PENDING");
    if (pendingCount > 1000) {
        alertService.send("消息表积压告警", "待发送消息数: " + pendingCount);
    }

    long failedCount = outboxMessageMapper.countByStatus("FAILED");
    if (failedCount > 0) {
        alertService.send("消息发送失败告警", "失败消息数: " + failedCount);
    }
}
```

## 五、数据一致性校验

```java
// 定期对账 - 检查数据一致性
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点
public void reconciliation() {
    // 检查订单状态与库存是否一致
    List<Order> paidOrders = orderRepository.findByStatus(OrderStatus.PAID);
    for (Order order : paidOrders) {
        Inventory inventory = inventoryRepository.findByProductId(order.getProductId());
        if (inventory.getReserved() < order.getQuantity()) {
            log.error("数据不一致: 订单 {} 库存预留不足", order.getId());
            // 记录不一致数据，人工处理或自动修复
            reconciliationRepository.save(new ReconciliationRecord(order, inventory));
        }
    }

    // 检查支付状态与订单状态
    List<Payment> completedPayments = paymentRepository.findByStatus(PaymentStatus.COMPLETED);
    for (Payment payment : completedPayments) {
        Order order = orderRepository.findById(payment.getOrderId()).orElse(null);
        if (order == null || order.getStatus() != OrderStatus.PAID) {
            log.error("数据不一致: 支付 {} 订单状态异常", payment.getId());
        }
    }
}
```

## 六、常见陷阱

1. **一开始就引入分布式事务框架** - 过度设计，先尝试业务设计规避
2. **补偿操作不测试** - 上线后回滚失败
3. **无监控无告警** - 事务失败后数据不一致无人知晓
4. **忽略业务层面的规避手段** - 技术方案复杂化
5. **没有对账机制** - 数据不一致无法自动发现
6. **补偿超时未处理** - 长时间未完成的事务占用资源
