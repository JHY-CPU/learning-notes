# 分布式事务测试

## 一、测试策略

```java
@Test
public void testDistributedTransaction() {
    // 1. 正常流程测试
    orderService.createOrder(order);
    assertThat(order.getStatus()).isEqualTo("CREATED");

    // 2. 异常回滚测试
    try {
        orderService.createOrder(orderWithInsufficientStock);
    } catch (Exception e) {
        // 验证数据一致性
        assertThat(inventoryMapper.get(productId)).isEqualTo(originalStock);
    }

    // 3. 超时测试
    // 4. 并发测试
}
```

## 二、工作原理

分布式事务测试需要覆盖正常流程、异常回滚、超时、并发四个维度。正常流程验证事务提交后所有参与方数据一致。异常回滚验证某参与者失败后，已完成的参与者数据正确回滚。超时测试验证事务超时后资源正确释放。并发测试验证多线程同时执行事务时不出现数据错乱。混沌工程工具模拟网络分区、服务宕机等真实故障，验证事务在极端场景下的表现。

## 三、优缺点

**优点：**
- 充分测试提前发现数据不一致问题
- 压力测试确定事务框架的性能瓶颈
- 混沌测试验证故障恢复能力

**缺点：**
- 分布式事务测试场景多，测试用例编写成本高
- 混沌测试可能影响测试环境稳定性
- 部分场景难以模拟（如网络分区的精确控制）

## 四、最佳实践

1. 单元测试覆盖所有补偿/回滚逻辑
2. 集成测试使用 Testcontainers 模拟多数据库环境
3. 压力测试确定事务框架在目标 QPS 下的延迟和成功率
4. 定期进行混沌演练，验证故障恢复能力

## 五、Testcontainers 多数据库测试

```java
@SpringBootTest
@Testcontainers
public class DistributedTransactionIntegrationTest {
    @Container
    static MySQLContainer<?> orderDb = new MySQLContainer<>("mysql:8.0")
        .withDatabaseName("order_db");

    @Container
    static MySQLContainer<?> inventoryDb = new MySQLContainer<>("mysql:8.0")
        .withDatabaseName("inventory_db");

    @Test
    public void testSagaCompensation() {
        // 正向流程 - 库存不足时触发补偿
        OrderRequest request = new OrderRequest(1L, 9999);  // 超量库存

        assertThrows(InsufficientStockException.class, () -> {
            orderSaga.execute(request);
        });

        // 验证补偿 - 库存已恢复
        Inventory inventory = inventoryRepository.findByProductId(1L);
        assertThat(inventory.getAvailable()).isEqualTo(originalStock);

        // 验证补偿 - 余额已恢复
        Account account = accountRepository.findByUserId(1L);
        assertThat(account.getBalance()).isEqualTo(originalBalance);
    }

    @Test
    public void testIdempotency() {
        OrderRequest request = new OrderRequest(1L, 1);
        String idempotencyKey = UUID.randomUUID().toString();

        // 第一次执行
        orderService.createOrder(request, idempotencyKey);
        Order first = orderRepository.findByIdempotencyKey(idempotencyKey);

        // 幂等重试 - 不应创建新订单
        orderService.createOrder(request, idempotencyKey);
        long count = orderRepository.countByIdempotencyKey(idempotencyKey);

        assertThat(count).isEqualTo(1);
    }
}
```

## 六、测试覆盖率要求

```yaml
分布式事务测试覆盖:
  正常流程:      100%  所有事务模式的正常提交
  异常回滚:      100%  所有补偿/回滚逻辑
  超时场景:       80%  事务超时处理
  并发场景:       80%  多线程并发执行
  幂等性:        100%  重复请求处理
  网络分区:       50%  混沌工程测试
  协调者故障:     50%  混沌工程测试
```

## 七、常见陷阱

1. **只测试正常流程不测试异常流程** - 回滚逻辑有 Bug
2. **不测试幂等性** - 重试场景数据重复
3. **测试数据不隔离** - 测试之间互相干扰
4. **混沌测试无回滚方案** - 破坏测试环境
5. **不测试补偿操作的幂等性** - 补偿重试导致数据异常
6. **测试环境与生产环境差异大** - 超时配置、并发数不同
