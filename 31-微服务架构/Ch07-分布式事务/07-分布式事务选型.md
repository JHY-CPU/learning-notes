# 分布式事务选型

## 一、选型决策树

```
需要强一致性？
├── 是 → 资金类场景？ → 是 → TCC
│                  → 否 → Seata AT
└── 否 → 最终一致性
    ├── 有 MQ？ → 是 → 事务消息
    │          → 否 → 本地消息表
    └── 长事务？ → 是 → Saga
              → 否 → 本地消息表
```

## 二、工作原理

选型决策基于业务对一致性、性能、复杂度的权衡。资金类场景（转账、扣款）要求强一致性且不能容忍中间状态，选择 TCC。有 MQ 的场景优先选择事务消息，利用 MQ 原生支持简化实现。通用业务场景选择本地消息表，实现简单且不依赖外部框架。长事务（如订单履约、物流流转）选择 Saga，无锁高吞吐。Seata AT 模式适合快速接入但高并发场景有全局锁瓶颈。

## 三、优缺点

**选型的共同优点：** 根据业务场景选择最合适的方案，避免过度设计
**选型的共同缺点：** 没有银弹，每个方案都有适用边界

## 四、方案详细对比

| 维度 | Seata AT | Seata TCC | Saga | 本地消息表 | 事务消息 |
|------|----------|-----------|------|-----------|---------|
| 一致性 | 强一致 | 强一致 | 最终一致 | 最终一致 | 最终一致 |
| 性能 | 中 | 中 | 高 | 高 | 高 |
| 侵入性 | 无 | 高 | 中 | 低 | 低 |
| 依赖 | Seata Server | Seata Server | 状态机引擎 | 无 | MQ |
| 学习成本 | 低 | 高 | 中 | 低 | 低 |
| 适用场景 | 通用 | 资金类 | 长事务 | 通用 | MQ 场景 |
| 资源锁定 | 全局锁 | 冻结资源 | 无 | 无 | 无 |
| 回滚方式 | 自动生成反向 SQL | 手动 Cancel | 补偿操作 | 幂等消费 | 幂等消费 |

## 五、选型建议矩阵

```
业务类型          → 推荐方案
──────────────────────────────
电商下单（非支付）  → 本地消息表
电商支付           → Seata TCC
物流追踪           → Saga
库存同步           → 事务消息
积分兑换           → 本地消息表
跨行转账           → Seata TCC
订单履约（多步骤）  → Saga
数据同步           → 事务消息/Canal
```

## 六、常见陷阱

1. **所有场景都用同一种方案** - 如全部用 TCC 导致开发成本过高
2. **强一致场景用最终一致方案** - 导致资金差错
3. **最终一致场景用强一致方案** - 性能严重下降
4. **不考虑团队技术能力** - 选择了复杂方案但实现质量差
5. **忽略运维成本** - Seata Server 需要高可用部署
6. **不考虑迁移成本** - 从一种方案迁移到另一种成本很高
