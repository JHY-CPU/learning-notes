# 最大努力通知

## 一、方案说明

```java
// 支付回调场景
public void onPaymentSuccess(Payment payment) {
    // 1. 更新本地状态
    orderMapper.updateStatus(payment.getOrderId(), "PAID");

    // 2. 通知下游服务
    notifyService.notify(payment);

    // 3. 定时重试通知
    @Scheduled(fixedDelay = 5000)
    public void retryNotify() {
        List<Notification> pending = notificationMapper.selectPending();
        for (Notification n : pending) {
            try {
                httpClient.post(n.getUrl(), n.getBody());
                notificationMapper.updateStatus(n.getId(), "SUCCESS");
            } catch (Exception e) {
                notificationMapper.incrementRetry(n.getId());
            }
        }
    }
}
```

## 二、工作原理

最大努力通知适用于跨系统、跨公司的通知场景（如支付回调、银行通知）。核心思想是：发起方尽最大努力通知接收方，但不保证一定成功。通知失败后按递增间隔重试（如 1 分钟、5 分钟、30 分钟、2 小时），达到最大重试次数后停止自动重试。接收方需要主动查询对账，补偿未收到的通知。通知记录表记录每次通知的状态、重试次数和最后通知时间。

## 三、优缺点

**优点：**
- 实现简单，不依赖分布式事务框架
- 跨系统解耦，双方独立演进
- 重试机制确保高概率送达

**缺点：**
- 不保证 100% 送达，需要对账兜底
- 接收方必须幂等
- 通知延迟受重试间隔影响

## 四、最佳实践

1. 通知接口幂等，根据业务 ID 去重
2. 重试间隔递增（指数退避），避免频繁冲击接收方
3. 设置最大重试次数（如 10 次），超过后转人工处理
4. 日终对账，主动检查并补偿未送达的通知

## 五、常见陷阱

1. **通知接口不幂等**，重试导致重复处理
2. **无对账机制**，通知丢失后数据永久不一致
3. **重试间隔太短**，对接收方造成压力
4. **通知记录不清理**，表数据无限增长
