# 事务消息方案

## 一、RocketMQ 事务消息

```java
rocketMQTemplate.sendMessageInTransaction("Topic", msg, arg);

@RocketMQTransactionListener
public class TransactionListener implements RocketMQLocalTransactionListener {
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            orderMapper.insert((Order) arg);
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }

    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        // 回查逻辑
        return RocketMQLocalTransactionState.COMMIT;
    }
}
```

## 二、工作原理

RocketMQ 事务消息基于半消息（Half Message）机制实现。第一步：Producer 发送半消息到 Broker，Broker 存储但不对消费者可见。第二步：Producer 执行本地事务。第三步：根据本地事务结果，Producer 向 Broker 发送 Commit 或 Rollback。Commit 后消息对消费者可见，Rollback 后消息被删除。若 Producer 未及时响应，Broker 定期回查本地事务状态（checkLocalTransaction）。回查接口需要根据业务数据判断事务是否成功。

## 三、优缺点

**优点：**
- MQ 原生支持，不需额外开发本地消息表
- 半消息机制保证业务操作和消息发送的最终一致
- 回查机制处理了本地事务结果丢失的场景

**缺点：**
- 仅 RocketMQ 原生支持，Kafka/RabbitMQ 需要额外实现
- 事务消息不支持延迟消息和批量发送
- 回查接口需要额外开发

## 四、最佳实践

1. 回查接口根据业务表状态判断事务结果，必须幂等
2. 事务超时时间设置合理，避免半消息长时间阻塞
3. 消费方实现幂等消费，防止消息重复
4. 生产环境 Broker 部署集群模式

## 五、常见陷阱

1. **回查接口不幂等**，重复回查导致状态判断错误
2. **本地事务异常但未返回 ROLLBACK**，半消息被回查后提交
3. **消费方处理失败无重试**，消息丢失
4. **事务消息与普通消息混用**，消息顺序被打乱
