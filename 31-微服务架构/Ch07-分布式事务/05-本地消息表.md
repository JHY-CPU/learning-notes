# 本地消息表

## 一、实现方案

```java
@Transactional
public void deductInventory(Order order) {
    // 本地事务
    inventoryMapper.deduct(order);
    // 保存消息记录
    messageMapper.insert(new Message("inventory-deducted", order));
}

// 定时投递
@Scheduled(fixedDelay = 1000)
public void deliverMessages() {
    List<Message> pending = messageMapper.selectPending(100);
    for (Message msg : pending) {
        kafkaTemplate.send(msg.getTopic(), msg.getContent());
        messageMapper.updateStatus(msg.getId(), "SENT");
    }
}
```

## 二、工作原理

本地消息表的核心思想是将业务操作和消息记录放在同一个本地事务中，保证二者的原子性。业务操作执行时，同时在本地消息表中插入一条待发送记录。定时任务轮询消息表，将待发送消息投递到 MQ，消费方处理后确认。消息表中的状态流转：PENDING -> SENT -> CONSUMED。由于业务操作和消息记录在同一个数据库事务中，要么都成功要么都失败，从而保证消息一定被发送。

## 三、优缺点

**优点：**
- 实现简单，不依赖外部框架
- 基于数据库事务保证消息可靠投递
- 适合大多数业务场景，通用性强

**缺点：**
- 定时轮询有一定延迟（通常秒级到分钟级）
- 消息表增加数据库存储和查询压力
- 消费方需要实现幂等

## 四、最佳实践

1. 消息表与业务表在同一个数据库中，保证事务一致性
2. 定时任务使用分布式锁，避免多实例重复投递
3. 消息投递失败使用指数退避重试
4. 消息表定期清理已消费记录，防止无限膨胀

## 五、完整实现示例

```java
// 消息表实体
@Entity
@Table(name = "outbox_message")
public class OutboxMessage {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String aggregateType;   // 聚合类型 (Order, Inventory)
    private String aggregateId;     // 聚合 ID
    private String eventType;       // 事件类型
    private String payload;         // JSON 数据
    private String status;          // PENDING, SENT, FAILED
    private LocalDateTime createdAt;
    private LocalDateTime sentAt;
    private int retryCount;
}

// 业务操作 + 写消息表 (同一个事务)
@Service
public class InventoryService {
    @Transactional
    public void deductInventory(Long productId, int quantity) {
        // 业务操作
        inventoryMapper.deduct(productId, quantity);

        // 写消息表
        OutboxMessage message = new OutboxMessage();
        message.setAggregateType("Inventory");
        message.setAggregateId(productId.toString());
        message.setEventType("INVENTORY_DEDUCTED");
        message.setPayload(JSON.toJSONString(Map.of("productId", productId, "quantity", quantity)));
        message.setStatus("PENDING");
        message.setCreatedAt(LocalDateTime.now());
        outboxMessageMapper.insert(message);
    }
}

// 定时投递 + 分布式锁
@Scheduled(fixedDelay = 1000)
public void deliverMessages() {
    // 使用 Redis 分布式锁防止多实例重复投递
    String lockKey = "lock:outbox:deliver";
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", Duration.ofSeconds(30));

    if (Boolean.TRUE.equals(locked)) {
        try {
            List<OutboxMessage> pending = outboxMessageMapper
                .selectByStatus("PENDING", 100);
            for (OutboxMessage msg : pending) {
                try {
                    kafkaTemplate.send(msg.getAggregateType() + "-events", msg.getPayload());
                    msg.setStatus("SENT");
                    msg.setSentAt(LocalDateTime.now());
                    outboxMessageMapper.update(msg);
                } catch (Exception e) {
                    msg.setRetryCount(msg.getRetryCount() + 1);
                    if (msg.getRetryCount() > 5) {
                        msg.setStatus("FAILED");
                        // 发送到死信队列或告警
                    }
                    outboxMessageMapper.update(msg);
                }
            }
        } finally {
            redisTemplate.delete(lockKey);
        }
    }
}

// 消费方幂等处理
@Component
public class OrderEventListener {
    @KafkaListener(topics = "Inventory-events")
    public void onInventoryDeducted(String payload,
            @Header(value = "idempotency-key", required = false) String idempotencyKey) {
        // 幂等检查
        if (processedMessageRepository.existsById(idempotencyKey)) {
            log.info("Message already processed: {}", idempotencyKey);
            return;
        }
        // 处理消息
        processInventoryEvent(payload);
        // 记录已处理
        processedMessageRepository.save(new ProcessedMessage(idempotencyKey));
    }
}
```

## 六、消息表清理策略

```sql
-- 定期清理已消费的消息
DELETE FROM outbox_message
WHERE status = 'SENT'
  AND sent_at < DATE_SUB(NOW(), INTERVAL 7 DAY);

-- 创建索引优化查询性能
CREATE INDEX idx_outbox_status ON outbox_message(status, created_at);
```

## 七、常见陷阱

1. **消息表与业务表不在同一数据库** - 无法保证原子性
2. **定时任务多实例并行执行** - 导致消息重复投递，必须用分布式锁
3. **消费方不幂等** - 重复消费导致数据错误
4. **消息表不清理** - 查询性能随时间下降
5. **消息体过大** - JSON 数据太大影响性能，只存必要的事件信息
6. **没有死信队列** - 失败消息没有兜底机制
