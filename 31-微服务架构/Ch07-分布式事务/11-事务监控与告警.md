# 分布式事务监控与告警

## 一、核心概念

分布式事务涉及多个服务和数据源，故障排查和性能监控比单体事务复杂得多。有效的监控告警体系能帮助团队及时发现事务异常、定位问题根因、评估系统健康度。

### 1.1 监控指标体系

```
指标层级            指标项                     说明
──────────────────────────────────────────────────────
全局层         事务成功率                    整体健康度
              事务平均耗时                   性能基线
              并发事务数                     容量规划
事务类型层     TCC/Saga/2PC 各类成功率        类型对比
              补偿事务触发次数                补偿频率
资源层         数据库锁等待时间               锁竞争
              TC Server 负载                 协调器压力
业务层         各业务线事务成功率              业务健康
              异常事务 TopN                  重点排查
```

## 四、告警规则

### 4.1 Prometheus 告警规则

```yaml
groups:
  - name: seata-alerts
    rules:
      # 事务成功率下降
      - alert: TransactionSuccessRateLow
        expr: >
          sum(rate(seata_transaction_committed_total[5m]))
          / sum(rate(seata_transaction_total[5m])) < 0.95
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "分布式事务成功率下降"
          description: >
            当前事务成功率 {{ $value | humanizePercentage }}
            低于 95%，请立即排查

      # 事务超时增多
      - alert: TransactionTimeoutHigh
        expr: >
          rate(seata_transaction_timeout_total[5m]) > 1
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "分布式事务超时增多"

      # 补偿事务频繁触发
      - alert: CompensationFrequent
        expr: >
          rate(seata_branch_rollback_total[5m]) > 5
        for: 3m
        labels:
          severity: warning
        annotations:
          summary: "补偿事务频繁触发"
          description: >
            5 分钟内回滚 {{ $value }} 次/秒，
            请检查业务逻辑或下游服务

      # 持锁时间过长
      - alert: LockHoldTimeTooLong
        expr: >
          histogram_quantile(0.99,
            rate(seata_lock_hold_seconds_bucket[5m])) > 10
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "持锁时间过长"
          description: >
            P99 持锁时间 {{ $value }}s，
            可能存在锁竞争

      # TC Server 负载过高
      - alert: TCServerOverload
        expr: >
          seata_transaction_active > 1000
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "TC Server 负载过高"
```

## 五、事务排查工具

### 5.1 事务详情查询

```java
@Service
public class TransactionDebugService {

    @Autowired
    private TransactionStoreManager storeManager;

    /**
     * 查询事务详情
     */
    public TransactionDetail getTransactionDetail(
            String xid) {
        GlobalTransactionDO global =
            storeManager.queryGlobalTransaction(xid);
        List<BranchTransactionDO> branches =
            storeManager.queryBranchTransactions(xid);

        return TransactionDetail.builder()
            .xid(xid)
            .status(global.getStatus())
            .beginTime(global.getBeginTime())
            .timeout(global.getTimeout())
            .branches(branches)
            .build();
    }

    /**
     * 查询异常事务列表
     */
    public List<GlobalTransactionDO> queryAbnormal(
            long startTime, long endTime) {
        return storeManager.queryByStatus(
            Arrays.asList(
                GlobalStatus.TimeoutRollbacking,
                GlobalStatus.RollbackRetrying,
                GlobalStatus.CommitRetrying
            ),
            startTime, endTime
        );
    }
}
```

## 六、最佳实践

1. **Seata Server 提供监控界面**，部署并定期查看
2. **事务失败要告警**，通过钉钉/邮件/短信通知
3. **超时事务要重点关注**，分析超时原因
4. **补偿事务频繁触发要分析原因**，可能是下游服务不稳定
5. **定期分析事务性能**，发现趋势性问题
6. **建立事务健康基线**，偏离基线时告警
7. **关联事务和链路追踪**，实现一站式排障
8. **保留事务日志**，支持事后审计和分析
