# 微服务部署最佳实践

## 一、实践清单

```yaml
容器化:
  □ 多阶段构建 Dockerfile
  □ 非 root 用户运行
  □ 健康检查配置
  □ 资源限制设置
  □ 镜像安全扫描

Kubernetes:
  □ Deployment + Service + HPA
  □ ConfigMap/Secret 管理配置
  □ Pod 反亲和性分散部署
  □ PDB 保证可用性
  □ NetworkPolicy 网络隔离

CI/CD:
  □ 代码提交自动触发流水线
  □ 测试自动化集成
  □ 镜像版本化管理
  □ 灰度发布策略
  □ 回滚方案就绪

运维:
  □ 监控告警完善
  □ 日志收集集中
  □ 链路追踪覆盖
  □ 应急预案有文档
```

## 二、部署检查清单

| 检查项 | 说明 | 状态 |
|--------|------|------|
| 健康检查 | liveness + readiness | 必须 |
| 资源限制 | CPU + Memory limits | 必须 |
| 副本数 | >= 2 | 必须 |
| PDB | 最小可用数 | 建议 |
| 反亲和性 | 分散到不同节点 | 建议 |
| 配置外化 | ConfigMap/Secret | 必须 |
| 日志格式 | JSON 结构化 | 必须 |
| 监控指标 | Prometheus 暴露 | 必须 |

## 三、故障应急预案

```yaml
应急流程:
  1. 发现问题:
     - 监控告警触发
     - 用户反馈

  2. 定位问题:
     - 查看 Grafana Dashboard
     - 查看 Kibana 日志
     - 查看 SkyWalking 链路

  3. 止血操作:
     - 回滚到上一版本
     - 扩容副本数
     - 熔断问题服务

  4. 根因分析:
     - 复盘会议
     - 更新应急预案
```

## 四、PDB 与反亲和性

```yaml
# PodDisruptionBudget - 保证最小可用数
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: order-service-pdb
spec:
  minAvailable: 2             # 至少 2 个 Pod 可用
  selector:
    matchLabels:
      app: order-service
---
# Pod 反亲和性 - 分散到不同节点
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
          - weight: 100
            podAffinityTerm:
              labelSelector:
                matchExpressions:
                - key: app
                  operator: In
                  values:
                  - order-service
              topologyKey: kubernetes.io/hostname  # 不同节点
#              topologyKey: topology.kubernetes.io/zone  # 不同可用区
```

## 五、HPA 与 VPA 自动伸缩

```yaml
# HPA - 水平自动伸缩
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  - type: Pods
    pods:
      metric:
        name: http_requests_per_second
      target:
        type: AverageValue
        averageValue: "1000"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Percent
        value: 50
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
```

## 六、部署后验证

```yaml
部署后检查:
  健康检查:
    - 所有 Pod 处于 Running 状态
    - Readiness 探针通过
    - Liveness 探针通过

  功能验证:
    - 冒烟测试通过
    - 核心 API 响应正常
    - 日志无错误

  性能验证:
    - 延迟在基线范围内
    - 错误率 < 0.1%
    - 资源使用在预期范围内

  监控告警:
    - 监控图表正常
    - 告警规则生效
    - 通知渠道畅通
```

## 七、注意事项

1. **不可变基础设施** - 不修改运行中的容器，通过替换实现变更
2. **渐进式发布** - 降低发布风险，金丝雀/蓝绿部署
3. **自动化一切** - 减少人为失误，CI/CD 全自动化
4. **监控先行** - 没有监控不上线
5. **定期演练** - 确保应急预案有效
6. **部署文档化** - 每个服务的部署流程有文档记录
