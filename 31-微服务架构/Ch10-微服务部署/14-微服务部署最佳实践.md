# 微服务部署最佳实践

## 一、实践清单

```yaml
容器化:
  □ 多阶段构建 Dockerfile
  □ 非 root 用户运行
  □ 健康检查配置
  □ 资源限制设置
  □ 镜像安全扫描

Kubernetes:
  □ Deployment + Service + HPA
  □ ConfigMap/Secret 管理配置
  □ Pod 反亲和性分散部署
  □ PDB 保证可用性
  □ NetworkPolicy 网络隔离

CI/CD:
  □ 代码提交自动触发流水线
  □ 测试自动化集成
  □ 镜像版本化管理
  □ 灰度发布策略
  □ 回滚方案就绪

运维:
  □ 监控告警完善
  □ 日志收集集中
  □ 链路追踪覆盖
  □ 应急预案有文档
```

## 二、部署检查清单

| 检查项 | 说明 | 状态 |
|--------|------|------|
| 健康检查 | liveness + readiness | 必须 |
| 资源限制 | CPU + Memory limits | 必须 |
| 副本数 | >= 2 | 必须 |
| PDB | 最小可用数 | 建议 |
| 反亲和性 | 分散到不同节点 | 建议 |
| 配置外化 | ConfigMap/Secret | 必须 |
| 日志格式 | JSON 结构化 | 必须 |
| 监控指标 | Prometheus 暴露 | 必须 |

## 三、故障应急预案

```yaml
应急流程:
  1. 发现问题:
     - 监控告警触发
     - 用户反馈

  2. 定位问题:
     - 查看 Grafana Dashboard
     - 查看 Kibana 日志
     - 查看 SkyWalking 链路

  3. 止血操作:
     - 回滚到上一版本
     - 扩容副本数
     - 熔断问题服务

  4. 根因分析:
     - 复盘会议
     - 更新应急预案
```

## 四、注意事项

1. **不可变基础设施**，不修改运行中的容器
2. **渐进式发布**，降低发布风险
3. **自动化一切**，减少人为失误
4. **监控先行**，没有监控不上线
5. **定期演练**，确保应急预案有效
