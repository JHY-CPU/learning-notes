# 多集群部署

## 一、多集群架构

```
多集群方案:
├── 主备模式 (Active-Standby)
│   └── 主集群处理流量，备集群待命
├── 双活模式 (Active-Active)
│   └── 两个集群同时处理流量
└── 多区域模式 (Multi-Region)
    └── 不同区域独立部署
```

## 二、集群联邦 (KubeFed)

```yaml
apiVersion: types.kubefed.io/v1beta1
kind: FederatedDeployment
metadata:
  name: order-service
  namespace: microservices
spec:
  template:
    metadata:
      labels:
        app: order-service
    spec:
      replicas: 3
      selector:
        matchLabels:
          app: order-service
      template:
        spec:
          containers:
            - name: order-service
              image: order-service:v1.2.3
  placement:
    clusters:
      - name: cluster-cn-east
      - name: cluster-cn-north
  overrides:
    - clusterName: cluster-cn-east
      clusterOverrides:
        - path: "/spec/replicas"
          value: 5
```

## 三、Service Mesh 多集群

```yaml
# Istio 多集群 - Gateway 方式
# 集群 A
apiVersion: networking.istio.io/v1beta1
kind: Gateway
metadata:
  name: cross-cluster-gateway
spec:
  selector:
    istio: eastwestgateway
  servers:
    - port:
        number: 15443
        name: tls
        protocol: TLS
      tls:
        mode: AUTO_PASSTHROUGH
      hosts:
        - "*.local"

# 通过 DNS 实现跨集群服务发现
apiVersion: networking.istio.io/v1beta1
kind: ServiceEntry
metadata:
  name: remote-order-service
spec:
  hosts:
    - order-service.microservices.svc.cluster.local
  location: MESH_INTERNAL
  ports:
    - number: 80
      name: http
      protocol: HTTP
  resolution: DNS
  endpoints:
    - address: cluster-b.example.com
      ports:
        http: 15443
```

## 四、Istio 多集群配置

```yaml
# Istio 多集群 - Primary-Remote 模式
# 集群 A (Primary)
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  values:
    global:
      meshID: mesh1
      multiCluster:
        clusterName: cluster-east
      network: network-east

# 集群 B (Remote) - 连接到集群 A 的控制面
apiVersion: install.istio.io/v1alpha1
kind: IstioOperator
spec:
  profile: remote
  values:
    global:
      remotePilotAddress: istiod.istio-system.svc.cluster.local
    global:
      multiCluster:
        clusterName: cluster-west
      network: network-west
```

## 五、故障切换策略

```
故障切换方案:
├── DNS 故障切换
│   ├── 健康检查探测集群状态
│   ├── 主集群故障时切换 DNS 记录
│   └── 切换延迟: 30s - 5min (取决于 TTL)
├── 负载均衡器故障切换
│   ├── 全局负载均衡器 (如 AWS Global Accelerator)
│   ├── 实时探测各集群健康状态
│   └── 自动将流量切换到健康集群
└── Service Mesh 故障切换
    ├── Istio 故障注入检测
    ├── 自动隔离故障集群
    └── 流量自动切换到健康集群
```

## 六、多集群数据同步

```yaml
数据同步方案:
  数据库:
    - 方案: MySQL 主从复制 / TiDB 跨区域
    - 延迟: 通常 < 1s
    - 注意: 写入只在主集群

  缓存:
    - 方案: Redis Cluster 跨区域
    - 延迟: 取决于网络延迟
    - 注意: 可能短暂不一致

  消息队列:
    - 方案: Kafka MirrorMaker 2
    - 延迟: 取决于网络带宽
    - 注意: 消息可能重复消费

  配置:
    - 方案: Nacos 集群同步
    - 延迟: 通常 < 5s
    - 注意: 各集群配置保持一致
```

## 七、注意事项

1. **数据同步是多集群最大挑战** - 数据库/缓存/消息队列的跨集群同步
2. **网络延迟要考虑** - 同城 < 5ms，跨区域 50-200ms
3. **故障切换策略要明确** - 自动切换 + 手动确认
4. **配置管理要统一** - 所有集群配置保持一致
5. **成本控制要做好** - 多集群资源消耗翻倍
6. **监控要全局视图** - 需要全局聚合的监控仪表板
