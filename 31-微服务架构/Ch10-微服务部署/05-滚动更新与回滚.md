# 滚动更新与回滚

## 一、滚动更新策略

```yaml
# Deployment 滚动更新配置
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多出1个Pod
      maxUnavailable: 0  # 不允许不可用Pod
  minReadySeconds: 30    # Pod就绪后等待30s
  revisionHistoryLimit: 10  # 保留10个历史版本
```

## 二、更新命令

```bash
# 更新镜像
kubectl set image deployment/order-service \
  order-service=registry.example.com/order-service:v1.3.0

# 查看更新状态
kubectl rollout status deployment/order-service

# 暂停更新（发现问题时）
kubectl rollout pause deployment/order-service

# 恢复更新
kubectl rollout resume deployment/order-service

# 回滚到上一版本
kubectl rollout undo deployment/order-service

# 回滚到指定版本
kubectl rollout undo deployment/order-service --to-revision=3

# 查看历史版本
kubectl rollout history deployment/order-service
```

## 三、蓝绿部署

```yaml
# Blue 环境
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  selector:
    app: order-service
    version: blue  # 切换标签实现蓝绿切换
  ports:
    - port: 80

---
# Green 环境 - 先部署，验证后切换流量
# 更新 Service selector: version: green
```

## 四、金丝雀发布

```yaml
# 稳定版本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service-stable
spec:
  replicas: 9
  template:
    metadata:
      labels:
        app: order-service
        track: stable

---
# 金丝雀版本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service-canary
spec:
  replicas: 1
  template:
    metadata:
      labels:
        app: order-service
        track: canary

# Service 同时选择 stable 和 canary，按比例路由
```

## 五、注意事项

1. **滚动更新是最常用策略**
2. **蓝绿部署资源消耗大但风险低**
3. **金丝雀适合新功能灰度验证**
4. **回滚方案要提前测试**
5. **更新前要确认健康检查通过**
