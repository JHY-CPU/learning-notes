# 告警规则

## 一、告警配置

```yaml
# SkyWalking 告警规则
rules:
  - name: response_time_rule
    expression: endpoint_avg > 1000
    period: 10
    count: 3
    message: "接口 {name} 平均响应时间超过 1 秒"

  - name: sla_rule
    expression: endpoint_sla < 95
    period: 10
    count: 2
    message: "接口 {name} SLA 低于 95%"
```

## 二、工作原理

告警规则基于链路追踪数据定义触发条件和通知方式。SkyWalking 的 OAP 服务实时聚合指标数据（如接口平均响应时间、成功率），当指标在连续 N 个周期内超过阈值时触发告警。告警规则支持多条件组合（AND/OR），可按服务、端点、实例粒度配置。告警触发后通过 Webhook、邮件、钉钉、飞书等渠道通知责任人。告警风暴通过静默期和聚合策略抑制：同一规则在静默期内只触发一次，相同告警自动聚合。

## 三、优缺点

**优点：**
- 基于真实链路数据告警，准确度高
- 支持多维度、多条件组合，灵活度强
- 静默期和聚合策略有效抑制告警风暴

**缺点：**
- 告警规则调优需要根据业务特点持续迭代
- 告警延迟受聚合周期影响（通常 1-3 分钟）
- 过多告警规则增加 OAP 服务压力

## 四、最佳实践

1. 告警分级：P0（电话通知）、P1（IM 通知）、P2（邮件通知）
2. 关键指标告警（SLA < 95%、P99 延迟 > 2 秒）设为 P0
3. 设置合理的静默期（5-15 分钟），避免重复告警
4. 定期审查告警规则，清理无效规则

## 五、多渠道通知集成

```java
// SkyWalking Webhook 接收告警
@RestController
@RequestMapping("/api/alert")
public class AlertController {
    @PostMapping("/skywalking")
    public void handleSkyWalkingAlert(@RequestBody AlertMessage alert) {
        for (AlertMessage.Scope scope : alert.getScopes()) {
            String message = String.format("[SkyWalking告警] %s - %s: %s",
                scope.getName(), alert.getRuleName(), alert.getAlarmMessage());

            // 钉钉通知
            dingTalkService.send(message, getOnDutyUsers());

            // 企业微信通知
            weChatService.send(message);

            // 记录告警日志
            alertLogRepository.save(new AlertLog(alert));
        }
    }
}
```

```yaml
# Prometheus + AlertManager 告警
# prometheus-rules.yml
groups:
  - name: tracing-alerts
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(traces_spanmetrics_calls_total{status_code="STATUS_CODE_ERROR"}[5m])) by (service_name)
          / sum(rate(traces_spanmetrics_calls_total[5m])) by (service_name) > 0.01
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "{{ $labels.service_name }} 错误率超过 1%"

      - alert: HighLatencyP99
        expr: |
          histogram_quantile(0.99, sum(rate(traces_spanmetrics_latency_bucket[5m])) by (service_name, le)) > 2000
        for: 5m
        labels:
          severity: warning
```

## 六、告警分级策略

| 级别 | 触发条件 | 通知方式 | 响应时间 |
|------|---------|---------|---------|
| P0 - 致命 | 全站不可用、数据丢失 | 电话 + 短信 | 5 分钟 |
| P1 - 严重 | SLA < 95%、核心接口超时 | 钉钉/企微 @责任人 | 15 分钟 |
| P2 - 警告 | P99 > 2s、错误率 > 0.5% | 钉钉/企微群消息 | 1 小时 |
| P3 - 提示 | 性能下降趋势 | 邮件 | 下个工作日 |

## 七、常见陷阱

1. **阈值设置过于敏感** - 正常波动也触发告警，形成告警风暴
2. **告警无分级** - 所有告警同等对待，重要告警被淹没
3. **告警无人响应** - 通知渠道不合理或责任人不明确
4. **告警规则与业务指标脱节** - 触发了但业务无影响
5. **静默期设置不合理** - 太短导致重复告警，太长导致漏告警
6. **告警没有自动恢复通知** - 问题已恢复但责任人不知道
