# 日志关联

## 一、MDC 存储 TraceID

```java
// MDC 存储
@Component
public class TraceFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) {
        String traceId = UUID.randomUUID().toString();
        MDC.put("traceId", traceId);
        MDC.put("spanId", generateSpanId());
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}

// 日志格式包含 TraceID
logging:
  pattern:
    console: "%d{yyyy-MM-dd HH:mm:ss} [%traceId] [%thread] %-5level %logger{36} - %msg%n"
```

## 二、工作原理

MDC（Mapped Diagnostic Context）是 SLF4J/Logback 提供的线程级上下文存储。Filter 在请求入口将 TraceID 和 SpanID 放入 MDC，日志框架在格式化日志时自动从 MDC 中取出并输出。异步场景下（如 @Async、线程池、CompletableFuture），MDC 不会自动传递，需要使用 MDC.getCopyOfContextMap() 在新线程中恢复。日志聚合系统（如 ELK、Loki）按 TraceID 字段建立索引，运维人员可通过 TraceID 快速检索全链路日志。

## 三、优缺点

**优点：**
- 日志中自动携带 TraceID，无需手动添加
- 与链路追踪系统关联，日志和链路互相跳转
- 对业务代码无侵入，仅需配置日志格式

**缺点：**
- 异步场景需要手动传递 MDC
- MDC 基于 ThreadLocal，协程等框架可能不兼容
- 日志格式变更需要重新部署

## 四、最佳实践

1. 日志格式统一包含 TraceID 和 SpanID
2. 异步线程池封装工具类，自动传递 MDC
3. MQ 消费者在处理消息前从 Header 恢复 MDC
4. 日志聚合系统按 TraceID 建立索引

## 五、常见陷阱

1. **异步场景 MDC 丢失**，TraceID 为空导致链路断裂
2. **线程池复用未清理 MDC**，TraceID 串到其他请求
3. **日志格式未包含 TraceID**，排障时无法关联链路
4. **MDC 内存泄漏**，put 后未在 finally 中 clear
