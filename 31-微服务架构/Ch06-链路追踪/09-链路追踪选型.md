# 链路追踪选型

## 一、方案对比

| 方案 | 侵入性 | 功能 | 性能 | 语言支持 |
|------|--------|------|------|---------|
| SkyWalking | 无侵入 | 全面 | 高 | 多语言 |
| Zipkin | 需埋点 | 基础 | 高 | 多语言 |
| Jaeger | 需埋点 | 中等 | 高 | 多语言 |
| Pinpoint | 无侵入 | 全面 | 中 | Java |

## 二、选型建议

```yaml
国内项目: SkyWalking（首选）
轻量级: Zipkin
云原生: Jaeger
Java 专用: Pinpoint
```

## 三、工作原理

链路追踪方案的核心差异在于数据采集方式。无侵入方案（SkyWalking、Pinpoint）通过 Java Agent 字节码增强，在类加载时自动注入追踪代码，无需修改业务代码。需埋点方案（Zipkin、Jaeger）通过 SDK 手动或借助框架自动传播 TraceID。无侵入方案适合存量系统改造，但字节码增强可能与某些框架不兼容。需埋点方案更可控，但开发成本更高。

## 四、优缺点

**SkyWalking 优点：** 无侵入、功能全面（追踪+指标+告警）、国内社区活跃
**SkyWalking 缺点：** Agent 可能与部分字节码框架冲突

**Jaeger 优点：** CNCF 项目、云原生生态好、支持自适应采样
**Jaeger 缺点：** 功能不如 SkyWalking 全面

**Pinpoint 优点：** 无侵入、Java 专精、UI 丰富
**Pinpoint 缺点：** 仅支持 Java、性能开销较大

## 五、选型决策树

```
需要无侵入接入？
├── 是 → 只有 Java 服务？
│   ├── 是 → SkyWalking (功能最全面)
│   └── 否 → SkyWalking (多语言支持)
└── 否 → 云原生环境 (K8s)?
    ├── 是 → Jaeger (CNCF 标准) 或 OpenTelemetry
    └── 否 → 已有 Spring Cloud?
        ├── 是 → Zipkin + Sleuth (已过时，建议迁移 OTel)
        └── 否 → Jaeger 或 Zipkin

新项目推荐: OpenTelemetry + SkyWalking/Jaeger
存量项目改造: SkyWalking Agent (无侵入)
```

## 六、方案详细对比

| 维度 | SkyWalking | Zipkin | Jaeger | Pinpoint |
|------|-----------|--------|--------|----------|
| 采集方式 | Java Agent | SDK/框架 | SDK | Java Agent |
| 侵入性 | 无 | 有 | 有 | 无 |
| 功能完整度 | 追踪+指标+告警+拓扑 | 追踪 | 追踪+指标 | 追踪+指标+调用栈 |
| 存储 | ES/H2/MySQL | ES/Cassandra/MySQL | ES/Cassandra/Kafka | HBase |
| 性能开销 | 低 (<3%) | 极低 (<1%) | 极低 (<1%) | 中 (3-5%) |
| K8s 集成 | 中等 | 好 | 最好 | 差 |
| 社区活跃度 | 高 (Apache) | 中 | 高 (CNCF) | 中 |

## 七、常见陷阱

1. **选择冷门方案** - 后期问题无人支持，社区已停止维护
2. **无侵入方案与字节码框架冲突** - 如 CGLIB、ASM、部分 AOP 框架
3. **忽略存储容量规划** - 数据量增长后存储不可用，需要提前规划
4. **只看功能不看社区活跃度** - 选型后维护困难
5. **OpenTelemetry 是未来方向** - 新项目应优先考虑 OTel 兼容方案
