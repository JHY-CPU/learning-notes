# 混沌工程

## 一、ChaosBlade

```yaml
# 注入 CPU 负载
blade create cpu load --cpu-percent 80 --timeout 60

# 注入网络延迟
blade create network delay --interface eth0 --time 3000 --timeout 120

# 杀死指定进程
blade create process kill --process-cmd java --signal 9

# 模拟磁盘填充
blade create disk fill --percent 90 --timeout 60
```

```java
// 代码级故障注入
@RestController
public class ChaosController {

    @GetMapping("/api/chaos/delay")
    public ResponseEntity<?> chaosDelay(@RequestParam int ms) {
        try {
            Thread.sleep(ms);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
        return ResponseEntity.ok("delayed");
    }

    @GetMapping("/api/chaos/error")
    public ResponseEntity<?> chaosError(@RequestParam double rate) {
        if (Math.random() < rate) {
            throw new RuntimeException("Chaos injected error");
        }
        return ResponseEntity.ok("ok");
    }
}
```

## 二、故障场景设计

```yaml
故障实验:
  网络故障:
    - 服务间网络延迟 3s
    - 网络分区（部分服务不可达）
    - DNS 解析失败

  资源故障:
    - CPU 使用率 100%
    - 内存不足
    - 磁盘空间不足
    - 文件描述符耗尽

  应用故障:
    - 某个服务实例宕机
    - 数据库连接池耗尽
    - 消息队列堆积
    - 缓存穿透
```

## 三、混沌工程实施流程

```
混沌工程流程:
1. 稳态假设 - 定义系统的正常行为指标
   ├── SLA: 可用性 > 99.9%
   ├── P99 延迟 < 2s
   └── 错误率 < 0.1%

2. 设计实验 - 选择要注入的故障类型
   ├── 故障类型: 网络延迟、CPU 打满、进程 Kill
   ├── 爆炸半径: 1% → 5% → 10% → 50%
   └── 持续时间: 30s → 60s → 5min

3. 执行实验 - 注入故障，观察系统行为
   ├── 监控指标是否异常
   ├── 熔断降级是否生效
   └── 告警是否及时触发

4. 分析结果 - 对比稳态假设
   ├── 指标偏差程度
   ├── 自动恢复时间
   └── 用户影响范围

5. 改进措施 - 修复发现的问题
   ├── 优化熔断规则
   ├── 增加降级逻辑
   └── 改进告警配置
```

## 四、混沌实验场景

```yaml
实验场景库:
  网络层:
    - 延迟注入: 给服务间通信增加 3s 延迟
    - 丢包注入: 10% 的数据包丢失
    - DNS 故障: 模拟 DNS 解析失败

  应用层:
    - JVM OOM: 模拟内存溢出
    - 线程池耗尽: 模拟线程池满
    - 连接池耗尽: 模拟数据库连接池满

  中间件:
    - Redis 不可达: 模拟缓存故障
    - MQ 堆积: 模拟消息积压
    - 数据库慢查询: 模拟查询超时

  基础设施:
    - Pod Kill: 杀死 K8s Pod
    - 节点宕机: 模拟服务器宕机
    - 磁盘写满: 模拟磁盘空间不足
```

## 五、注意事项

1. **先在测试环境实验** - 成熟后再上生产
2. **控制爆炸半径** - 从小范围（1%）开始
3. **有监控和回滚方案** - 快速止血
4. **定期做混沌实验** - 建立团队信心
5. **Chaos Monkey / Litmus / ChaosBlade** - 都是好工具
6. **混沌工程是文化** - 团队要接受"故意搞坏系统"的理念
