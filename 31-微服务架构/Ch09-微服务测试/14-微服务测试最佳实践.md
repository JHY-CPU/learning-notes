# 微服务测试最佳实践

## 一、实践清单

```yaml
单元测试:
  □ 核心业务逻辑 100% 覆盖
  □ Given-When-Then 结构
  □ 不依赖外部资源
  □ 测试运行 < 10s

集成测试:
  □ 使用 Testcontainers
  □ 数据库/中间件隔离
  □ 核心 API 覆盖
  □ CI 流水线集成

契约测试:
  □ 服务间接口契约化
  □ 消费者驱动模式
  □ Pact Broker 管理
  □ CI 自动验证

E2E 测试:
  □ 核心业务流程覆盖
  □ 测试数据独立
  □ 独立环境执行
```

## 二、测试策略矩阵

| 测试类型 | 频率 | 执行时间 | 环境 | 负责人 |
|---------|------|---------|------|--------|
| 单元测试 | 每次提交 | < 10s | 本地/CI | 开发 |
| 集成测试 | 每次 MR | < 5min | CI | 开发 |
| 契约测试 | 每次 MR | < 2min | CI | 双方 |
| E2E 测试 | 合并后 | < 15min | 预发布 | QA |
| 性能测试 | 每周 | < 30min | 沙箱 | 性能团队 |

## 三、常见问题

```yaml
测试失败排查:
  间歇性失败:
    - 原因: 时序问题、资源竞争
    - 方案: 增加重试、使用 Awaitility

  环境不一致:
    - 原因: 本地/CI 环境差异
    - 方案: Testcontainers、Docker Compose

  测试数据问题:
    - 原因: 测试间数据污染
    - 方案: @Transactional 回滚、独立数据库

  Mock 过度:
    - 原因: 测试了实现而非行为
    - 方案: 减少 Mock、增加集成测试
```

## 四、CI/CD 测试集成

```yaml
# GitLab CI 测试流水线
stages:
  - lint
  - unit-test
  - build
  - integration-test
  - contract-test
  - deploy-staging
  - e2e-test
  - deploy-prod

unit-test:
  stage: unit-test
  script:
    - mvn test -Dtest=*Test
  coverage: '/Total.*?([0-9]{1,3})%/'
  artifacts:
    reports:
      junit: target/surefire-reports/TEST-*.xml

integration-test:
  stage: integration-test
  services:
    - mysql:8.0
    - redis:7-alpine
    - rabbitmq:3-management
  script:
    - mvn verify -Dtest=*IntegrationTest
  only:
    - merge_requests

contract-test:
  stage: contract-test
  script:
    - mvn test -Dtest=*ContractTest
  only:
    - merge_requests

e2e-test:
  stage: e2e-test
  script:
    - docker-compose -f docker-compose.e2e.yml up -d
    - mvn verify -Dtest=*E2ETest
    - docker-compose -f docker-compose.e2e.yml down
  environment:
    name: staging
  only:
    - main
```

## 五、测试覆盖率门禁

```yaml
# 覆盖率门禁配置 (JaCoCo)
覆盖率要求:
  整体覆盖率: >= 80%
  新增代码覆盖率: >= 90%
  核心业务逻辑: >= 95%

# pom.xml
<plugin>
    <groupId>org.jacoco</groupId>
    <artifactId>jacoco-maven-plugin</artifactId>
    <configuration>
        <rules>
            <rule>
                <element>PACKAGE</element>
                <limits>
                    <limit>
                        <counter>LINE</counter>
                        <value>COVEREDRATIO</value>
                        <minimum>0.80</minimum>
                    </limit>
                </limits>
            </rule>
        </rules>
    </configuration>
</plugin>
```

## 六、注意事项

1. **测试是投资不是成本** - 测试减少线上故障，节省运维时间
2. **测试左移，尽早发现问题** - 在编码阶段就写测试
3. **自动化一切可自动化的测试** - 手动测试不可持续
4. **定期评审测试用例质量** - 淘汰无效测试
5. **团队共建测试文化** - 每个人都写测试
6. **测试数据独立** - 每个测试独立准备和清理数据
