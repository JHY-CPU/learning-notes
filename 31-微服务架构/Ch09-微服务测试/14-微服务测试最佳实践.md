# 微服务测试最佳实践

## 一、实践清单

```yaml
单元测试:
  □ 核心业务逻辑 100% 覆盖
  □ Given-When-Then 结构
  □ 不依赖外部资源
  □ 测试运行 < 10s

集成测试:
  □ 使用 Testcontainers
  □ 数据库/中间件隔离
  □ 核心 API 覆盖
  □ CI 流水线集成

契约测试:
  □ 服务间接口契约化
  □ 消费者驱动模式
  □ Pact Broker 管理
  □ CI 自动验证

E2E 测试:
  □ 核心业务流程覆盖
  □ 测试数据独立
  □ 独立环境执行
```

## 二、测试策略矩阵

| 测试类型 | 频率 | 执行时间 | 环境 | 负责人 |
|---------|------|---------|------|--------|
| 单元测试 | 每次提交 | < 10s | 本地/CI | 开发 |
| 集成测试 | 每次 MR | < 5min | CI | 开发 |
| 契约测试 | 每次 MR | < 2min | CI | 双方 |
| E2E 测试 | 合并后 | < 15min | 预发布 | QA |
| 性能测试 | 每周 | < 30min | 沙箱 | 性能团队 |

## 三、常见问题

```yaml
测试失败排查:
  间歇性失败:
    - 原因: 时序问题、资源竞争
    - 方案: 增加重试、使用 Awaitility

  环境不一致:
    - 原因: 本地/CI 环境差异
    - 方案: Testcontainers、Docker Compose

  测试数据问题:
    - 原因: 测试间数据污染
    - 方案: @Transactional 回滚、独立数据库

  Mock 过度:
    - 原因: 测试了实现而非行为
    - 方案: 减少 Mock、增加集成测试
```

## 四、注意事项

1. **测试是投资不是成本**
2. **测试左移，尽早发现问题**
3. **自动化一切可自动化的测试**
4. **定期评审测试用例质量**
5. **团队共建测试文化**
