# 测试最佳实践

## 一、FIRST 原则

```yaml
FIRST:
  Fast:        # 测试要快
    - 单元测试毫秒级
    - 并行执行
  Independent: # 测试独立
    - 不依赖执行顺序
    - 数据隔离
  Repeatable:  # 可重复
    - 任何环境结果一致
    - 不依赖外部状态
  Self-Validating: # 自验证
    - 有明确断言
    - 不需人工判断
  Timely:      # 及时
    - 与代码同步编写
    - TDD 模式
```

## 二、测试命名规范

```java
// 推荐: should_期望行为_when_前置条件
@Test
void shouldReturnOrder_whenOrderIdExists() { }

@Test
void shouldThrowException_whenOrderNotFound() { }

// 推荐: 方法_场景_预期
@Test
void createOrder_withValidRequest_shouldSucceed() { }

// 不推荐: test1, testMethod
@Test
void test1() { } // 不好
```

## 三、测试结构 AAA

```java
@Test
void shouldDeductInventoryWhenOrderPaid() {
    // Arrange (准备)
    Order order = anOrder().withStatus(OrderStatus.CREATED).build();
    inventoryService.init("P001", 10);

    // Act (执行)
    order.pay();
    orderService.handlePaid(order);

    // Assert (断言)
    assertThat(inventoryService.getStock("P001")).isEqualTo(8);
    verify(eventPublisher).publish(any(OrderPaidEvent.class));
}
```

## 四、Builder 模式构造测试数据

```java
// 测试数据 Builder
public class OrderBuilder {
    private Long id = 1L;
    private Long userId = 100L;
    private String status = "CREATED";
    private BigDecimal amount = new BigDecimal("99.99");

    public static OrderBuilder anOrder() { return new OrderBuilder(); }

    public OrderBuilder withId(Long id) { this.id = id; return this; }
    public OrderBuilder withUserId(Long userId) { this.userId = userId; return this; }
    public OrderBuilder withStatus(String status) { this.status = status; return this; }
    public OrderBuilder withAmount(BigDecimal amount) { this.amount = amount; return this; }

    public Order build() {
        return Order.builder().id(id).userId(userId).status(status).amount(amount).build();
    }
}

// 使用
Order order = anOrder().withStatus("PAID").withAmount(new BigDecimal("199.99")).build();
```

## 五、测试双刃剑 - 何时 Mock 何时不 Mock

```yaml
应该 Mock:
  - 外部 HTTP 调用 (Feign Client)
  - 第三方服务 (支付、短信)
  - 消息队列发送
  - 价格计算等非核心逻辑

不应该 Mock:
  - 自己的核心业务逻辑
  - 数据库操作 (用 Testcontainers)
  - 缓存操作 (用 EmbeddedRedis)
  - 简单的工具类方法

原则:
  - Mock 边界，不 Mock 实现
  - 测试行为，不测试实现细节
  - 集成测试 > 过度 Mock 的单元测试
```

## 六、注意事项

1. **测试代码也是代码** - 需要维护和重构
2. **避免测试中的逻辑分支** - 循环、条件判断说明测试不够细分
3. **每个测试只断言一个关注点** - 多个断言的测试难以定位失败原因
4. **使用 @Disabled 而非删除过时测试** - 保留历史信息
5. **定期清理无用测试和过时 Mock** - 避免测试债务
6. **测试覆盖率 > 80%** - 核心业务逻辑 100%
