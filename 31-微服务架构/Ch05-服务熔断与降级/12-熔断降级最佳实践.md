# 熔断降级最佳实践

## 一、实践清单

```yaml
配置:
  □ 网关层全局限流
  □ 服务间调用熔断
  □ 核心方法降级
  □ 超时控制合理
  □ 重试配合幂等

监控:
  □ 熔断触发告警
  □ 降级触发告警
  □ 规则动态调整
  □ 定期调优规则
```

## 二、选型建议

| 场景 | 推荐 |
|------|------|
| Spring Cloud | Sentinel |
| 轻量级 | Resilience4j |
| 全功能 | Sentinel |

## 三、工作原理

熔断降级最佳实践的核心是构建多层次的保护体系。网关层做全局限流（QPS 限制），服务间调用做熔断（失败率触发），核心方法做降级（返回缓存或默认值）。三层联动形成纵深防御：网关挡住恶意流量，熔断隔离故障依赖，降级保障核心功能。Sentinel 提供可视化控制台动态调整规则，Resilience4j 通过配置中心下发规则。所有保护机制都需要监控告警支撑，确保规则生效且效果可量化。

## 四、优缺点

**优点：**
- 多层保护显著提高系统可用性
- 动态规则调整适应业务变化
- 监控数据驱动持续调优

**缺点：**
- 多层保护增加架构复杂度
- 规则调优需要持续投入
- 过度保护可能隐藏真正的系统问题

## 五、规则调优方法

```yaml
调优流程:
1. 收集基线数据
   - 各接口 P50/P95/P99 延迟
   - 各接口正常 QPS 和峰值 QPS
   - 各依赖的历史可用性

2. 设定初始规则
   - 超时 = P99 × 1.5
   - 限流 = 峰值 QPS × 1.2
   - 熔断阈值 = 失败率 50%

3. 灰度发布规则
   - 先在测试环境验证
   - 预发环境灰度验证
   - 生产环境灰度发布

4. 监控调优
   - 观察规则触发频率
   - 调整阈值减少误触发
   - 定期复盘调整
```

## 六、多级降级策略

```
降级级别:
Level 0 - 正常: 正常调用，返回真实数据
Level 1 - 轻微降级: 返回缓存数据，数据可能延迟
Level 2 - 中度降级: 返回部分字段，非核心字段缺失
Level 3 - 重度降级: 返回默认值/静态数据
Level 4 - 全站降级: 返回维护页面

触发条件:
├── 下游响应时间 > 超时阈值 → Level 1
├── 下游错误率 > 10% → Level 2
├── 下游熔断开启 → Level 3
└── 整体负载 > 90% → Level 4
```

```java
// 动态降级级别控制
@Component
public class DegradationManager {
    @Autowired
    private NacosConfigService configService;

    public int getDegradationLevel(String serviceName) {
        String level = configService.getConfig("degradation.level." + serviceName);
        return level != null ? Integer.parseInt(level) : 0;
    }

    @SentinelResource(value = "getUser", fallback = "getUserFallback")
    public User getUser(Long id) {
        int level = getDegradationLevel("user-service");
        if (level >= 3) throw new DegradeException("Service degraded");
        return userService.getUser(id);
    }
}
```

## 七、常见陷阱

1. **只在网关层限流不做服务间熔断** - 内部调用故障仍会扩散
2. **降级逻辑过于简单** - 全部返回空，用户体验差
3. **规则一成不变** - 业务变化后阈值不再适用，需要定期调优
4. **无监控告警** - 熔断触发后运维人员不知情
5. **降级级别没有分级** - 要么正常要么全降级，没有中间状态
6. **熔断恢复后无通知** - 熔断恢复后应通知运维确认
