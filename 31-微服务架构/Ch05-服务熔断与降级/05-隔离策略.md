# 隔离策略

## 一、线程池隔离

```java
// 每个服务独立线程池
@HystrixCommand(
    threadPoolKey = "userPool",
    threadPoolProperties = {
        @HystrixProperty(name = "coreSize", value = "10"),
        @HystrixProperty(name = "maxQueueSize", value = "100")
    }
)
public User getUser(Long id) {
    return userService.getUser(id);
}
```

## 二、信号量隔离

```java
// 限制并发请求数
@SentinelResource(value = "getUser")
public User getUser(Long id) {
    return userService.getUser(id);
}
// 配置并发数限制
```

## 三、工作原理

隔离策略的目的是防止一个依赖的故障蔓延到整个系统。线程池隔离为每个依赖分配独立线程池，调用在独立线程中执行，线程池满了就拒绝请求。这种方式隔离最彻底，但线程切换有开销。信号量隔离通过原子计数器限制并发请求数，不创建新线程，在调用线程中同步执行，开销更低但隔离效果弱于线程池。Resilience4j 的 Bulkhead 基于信号量实现，Hystrix 支持两种模式。

## 四、优缺点

**线程池隔离优点：** 隔离彻底，故障不传播，支持异步和超时
**线程池隔离缺点：** 线程切换开销大（约 1-3ms），线程池管理复杂

**信号量隔离优点：** 零线程切换开销，实现简单，适合高频调用
**信号量隔离缺点：** 无法隔离慢调用，调用线程被阻塞

## 五、Resilience4j Bulkhead 隔离

```java
// 信号量隔离 - Bulkhead
@Bulkhead(name = "userService", type = Bulkhead.Type.SEMAPHORE, fallbackMethod = "bulkheadFallback")
public User getUser(Long id) {
    return userService.getUser(id);
}

public User bulkheadFallback(Long id, Throwable t) {
    return User.defaultUser(id);
}

// 线程池隔离 - Bulkhead
@Bulkhead(name = "orderService", type = Bulkhead.Type.THREADPOOL, fallbackMethod = "bulkheadFallback")
public CompletableFuture<Order> getOrder(Long id) {
    return CompletableFuture.supplyAsync(() -> orderService.getOrder(id));
}
```

```yaml
# Resilience4j Bulkhead 配置
resilience4j:
  bulkhead:
    instances:
      userService:
        max-concurrent-calls: 25       # 最大并发数
        max-wait-duration: 0           # 不等待，直接拒绝
        writable-stack-trace-enabled: false
      orderService:
        max-thread-pool-size: 10       # 线程池大小
        core-thread-pool-size: 5       # 核心线程数
        queue-capacity: 20             # 队列容量
        keep-alive-duration: 20ms      # 空闲线程存活时间
```

## 六、隔离策略选择决策

```
选择隔离策略:
├── 调用延迟 > 100ms？
│   ├── 是 → 信号量隔离 (开销小)
│   └── 否 → 线程池隔离 (隔离彻底)
├── 需要超时控制？
│   ├── 是 → 线程池隔离 (可以中断)
│   └── 否 → 信号量隔离
├── 高频调用 (QPS > 1000)？
│   ├── 是 → 信号量隔离 (零开销)
│   └── 否 → 线程池隔离
└── ThreadLocal 依赖？
    ├── 是 → 信号量隔离 (同线程执行)
    └── 否 → 线程池隔离
```

## 七、常见陷阱

1. **线程池数量过多导致线程争用** - 整体性能下降，建议每依赖 10-20 个线程
2. **信号量阈值设置过小** - 正常流量也被拒绝，基于 P99 并发数设定
3. **线程池隔离下 ThreadLocal 丢失** - 上下文传递需要手动处理
4. **隔离粒度太粗** - 一个接口慢导致同服务所有接口被阻塞，应该按接口隔离
5. **隔离 + 超时未配合使用** - 线程池满了只是拒绝，没有超时保护
6. **监控隔离指标缺失** - 线程池使用率、拒绝次数需要监控
