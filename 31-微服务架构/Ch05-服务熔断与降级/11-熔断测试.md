# 熔断测试

## 一、测试策略

```java
// 使用 Chaos Engineering 测试熔断
@Test
public void testCircuitBreaker() {
    // 1. 模拟服务故障
    mockServer.when(request("/api/users"))
        .respond(response().withDelay(TimeUnit.SECONDS, 10));

    // 2. 触发熔断
    for (int i = 0; i < 100; i++) {
        try {
            userService.getUser(1L);
        } catch (Exception e) {
            // 预期异常
        }
    }

    // 3. 验证熔断生效
    assertThat(circuitBreaker.getState()).isEqualTo(State.OPEN);
}
```

## 二、工作原理

熔断测试通过模拟各种故障场景，验证熔断器、降级、隔离等保护机制是否按预期工作。Mock Server 模拟下游服务的延迟、错误响应、连接拒绝等异常。混沌工程工具（如 ChaosBlade、Chaos Monkey）在生产或预发环境中注入真实故障，如 CPU 打满、网络延迟、进程 Kill 等。测试分为单元测试（验证单个熔断逻辑）、集成测试（验证服务间联动）、压力测试（验证阈值设定合理性）。

## 三、优缺点

**优点：**
- 提前发现熔断配置不合理的问题
- 验证降级逻辑在真实故障下的表现
- 压力测试确定最优阈值参数

**缺点：**
- 混沌工程测试可能影响生产环境
- 测试场景难以覆盖所有故障组合
- 测试结果受环境因素影响，不一定可复现

## 四、最佳实践

1. 单元测试覆盖所有 fallback 方法
2. 集成测试模拟下游超时、异常、拒绝连接等场景
3. 压力测试确定熔断阈值和舱壁容量
4. 生产环境定期进行混沌演练，验证保护机制有效性

## 五、混沌工程测试工具

```java
// ChaosBlade - Java 故障注入
// 模拟慢调用
blade create dubbo delay --time 3000 --service com.example.UserService --method getUser

// 模拟异常
blade create dubbo throwCustomException --exception java.io.IOException \
    --service com.example.UserService --method getUser

// 模拟 CPU 打满
blade create cpu load --cpu-percent 80 --timeout 60

// 模拟网络延迟
blade create network delay --time 3000 --interface eth0 --local-port 8080
```

```java
// Resilience4j 测试
@SpringBootTest
public class CircuitBreakerTest {
    @Autowired
    private UserService userService;

    @Test
    public void testCircuitBreakerOpens() {
        // 模拟连续失败触发熔断
        for (int i = 0; i < 20; i++) {
            try {
                simulateServiceFailure();
                userService.getUser(1L);
            } catch (Exception e) {
                // 预期异常
            }
        }

        // 验证熔断器状态为 OPEN
        CircuitBreaker cb = circuitBreakerRegistry.circuitBreaker("userService");
        assertThat(cb.getState()).isEqualTo(CircuitBreaker.State.OPEN);

        // 验证后续请求快速失败
        long start = System.currentTimeMillis();
        try {
            userService.getUser(1L);
        } catch (Exception e) {
            long duration = System.currentTimeMillis() - start;
            assertThat(duration).isLessThan(100);  // 快速失败，< 100ms
        }
    }

    @Test
    public void testCircuitBreakerHalfOpen() throws InterruptedException {
        // 触发熔断
        triggerCircuitBreaker();

        // 等待进入半开状态
        Thread.sleep(6000);

        // 恢复下游服务
        restoreService();

        // 验证请求成功，熔断器回到 CLOSED
        User user = userService.getUser(1L);
        assertThat(user).isNotNull();
    }
}
```

## 六、测试检查清单

```yaml
单元测试:
  □ 所有 fallback 方法都有测试用例
  □ 熔断器状态转换逻辑验证
  □ 异常分类正确（可重试 vs 不可重试）

集成测试:
  □ 下游超时场景
  □ 下游返回 500 场景
  □ 下游连接拒绝场景
  □ 下游部分成功场景

压力测试:
  □ 验证 QPS 阈值设定合理性
  □ 验证熔断器在高并发下的行为
  □ 验证线程池/信号量隔离容量

混沌测试:
  □ 生产环境定期演练
  □ 验证自动化回滚能力
  □ 验证监控告警及时性
```

## 七、常见陷阱

1. **只测正常流程不测故障流程** - 上线后熔断不生效
2. **混沌测试在生产环境无回滚方案** - 导致真实故障
3. **测试用例与实际故障场景不符** - 测试通过但线上仍出问题
4. **熔断阈值基于经验值设定** - 未经过压力测试验证
5. **测试环境与生产环境差异大** - 线程池大小、超时配置不同导致测试失效
