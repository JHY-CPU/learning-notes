# 降级策略

## 一、降级方案

```java
// 1. 返回默认值
public User getUserFallback(Long id) {
    return User.defaultUser(id);
}

// 2. 缓存兜底
public User getUserFromCache(Long id) {
    return redisTemplate.opsForValue().get("user:" + id);
}

// 3. 兜底服务
public User getUserFromFallbackService(Long id) {
    return fallbackClient.getUser(id);
}

// 4. 静态数据
public User getStaticUser() {
    return new User(0L, "游客", "");
}
```

## 二、降级策略选择

| 场景 | 策略 |
|------|------|
| 用户信息 | 返回缓存或默认值 |
| 商品信息 | 返回缓存 |
| 订单信息 | 返回空列表 |
| 支付服务 | 提示稍后重试 |

## 三、工作原理

降级策略的核心是在依赖服务不可用时，用预设的替代方案返回响应，保证核心业务不中断。降级触发条件包括：熔断器开启、调用超时、异常抛出、限流被拒。降级逻辑通常在 fallback 方法中实现，返回缓存数据、默认值或静态数据。降级粒度可以从粗到细：全服务降级（直接返回维护页面）、接口级降级（某个接口返回默认值）、数据级降级（返回部分字段或缓存数据）。

## 四、优缺点

**优点：**
- 保障核心功能在依赖故障时仍可用
- 用户体验比直接报错好得多
- 降级逻辑与业务逻辑分离，不侵入主流程

**缺点：**
- 降级数据可能与真实数据不一致
- 降级逻辑需要额外开发和维护
- 过度降级可能掩盖真正的问题

## 五、降级实现完整示例

```java
// Sentinel 降级实现
@Service
public class ProductService {
    @SentinelResource(
        value = "getProduct",
        blockHandler = "getProductBlock",    // 限流降级
        fallback = "getProductFallback"      // 异常降级
    )
    public Product getProduct(Long id) {
        return productClient.getProduct(id);
    }

    // 限流降级 - 返回缓存
    public Product getProductBlock(Long id, BlockException ex) {
        Product cached = redisTemplate.opsForValue().get("product:" + id);
        if (cached != null) return cached;
        return Product.defaultProduct(id);
    }

    // 异常降级 - 返回默认值
    public Product getProductFallback(Long id, Throwable t) {
        log.warn("Product fallback for id={}, reason={}", id, t.getMessage());
        return Product.builder()
            .id(id)
            .name("商品信息暂不可用")
            .price(BigDecimal.ZERO)
            .available(false)
            .build();
    }
}

// Resilience4j 降级实现
@Service
public class OrderService {
    @CircuitBreaker(name = "orderService", fallbackMethod = "createOrderFallback")
    @Retry(name = "orderService", fallbackMethod = "createOrderFallback")
    @TimeLimiter(name = "orderService", fallbackMethod = "createOrderFallback")
    public CompletableFuture<Order> createOrder(CreateOrderRequest request) {
        return CompletableFuture.supplyAsync(() -> {
            return orderClient.createOrder(request);
        });
    }

    public CompletableFuture<Order> createOrderFallback(CreateOrderRequest request, Throwable t) {
        log.warn("Order fallback, reason={}", t.getMessage());
        Order fallbackOrder = Order.builder()
            .status(OrderStatus.PENDING)
            .message("订单处理中，请稍后查看")
            .build();
        return CompletableFuture.completedFuture(fallbackOrder);
    }
}
```

## 六、降级策略矩阵

| 依赖服务 | 降级策略 | 降级数据 | 用户体验 |
|---------|---------|---------|---------|
| 用户服务 | 缓存兜底 | Redis 缓存的用户信息 | 基本无感 |
| 商品服务 | 缓存+默认值 | 缓存数据或"暂无信息" | 部分功能受限 |
| 搜索服务 | 默认推荐 | 热门商品列表 | 可接受 |
| 支付服务 | 提示重试 | "系统繁忙，请稍后重试" | 明确告知 |
| 推荐服务 | 静态数据 | 空列表 | 无感 |
| 短信服务 | 队列延迟 | 消息入队，延迟发送 | 无感 |

## 七、常见陷阱

1. **降级逻辑抛异常** - 导致降级也不生效，请求直接失败
2. **降级逻辑做了复杂计算** - 本身成为性能瓶颈，降级逻辑要尽可能简单
3. **所有接口都降级返回空** - 用户体验极差，核心功能要优先保障
4. **降级触发后无监控** - 长时间不知道服务出了问题
5. **降级数据与真实数据差距太大** - 影响用户体验
6. **降级逻辑没有覆盖所有异常** - 未预期的异常导致降级失败
