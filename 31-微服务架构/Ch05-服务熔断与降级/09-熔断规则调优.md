# 熔断规则调优

## 一、核心概念

熔断规则的参数设置直接影响熔断器的灵敏度。过于灵敏会导致正常波动也触发熔断，过于迟钝则起不到保护作用。本节介绍慢调用阈值、错误率阈值、熔断时长等核心参数的调优方法。

### 1.1 核心参数说明

```
参数                    说明                    默认值
──────────────────────────────────────────────────────
slowCallDurationThreshold  慢调用 RT 阈值      1000ms
slowCallRateThreshold      慢调用比例阈值       100%
failureRateThreshold       错误率阈值           50%
slidingWindowSize          滑动窗口大小         100
slidingWindowType          滑动窗口类型         COUNT
minimumNumberOfCalls       最小调用数           100
waitDurationInOpenState    熔断持续时长         60s
permittedNumberOfCallsInHalfOpenState  半开状态允许调用数  10
```

### 1.2 参数关系

```
                      ┌──────────────────────┐
                      │   minimumNumberOfCalls│
                      │   (触发统计的最小调用数)│
                      └──────────┬───────────┘
                                 │
                      ┌──────────▼───────────┐
                      │   滑动窗口统计         │
                      │   (slowCall + error)  │
                      └──────────┬───────────┘
                                 │
              ┌──────────────────┼──────────────────┐
              ▼                                     ▼
   ┌─────────────────┐                    ┌─────────────────┐
   │ slowCallRate    │                    │ failureRate     │
   │ >= threshold?   │                    │ >= threshold?   │
   └────────┬────────┘                    └────────┬────────┘
            │                                      │
            └────────────────┬─────────────────────┘
                             │ 任一触发
                             ▼
                    ┌─────────────────┐
                    │   熔断器打开     │
                    │ (Open State)    │
                    └────────┬────────┘
                             │
                    ┌────────▼─────────┐
                    │ waitDuration后   │
                    │ 进入半开状态      │
                    └──────────────────┘
```

## 二、Resilience4j 熔断规则配置

### 2.1 代码配置

```java
@Configuration
public class CircuitBreakerConfig {

    @Bean
    public CircuitBreakerRegistry circuitBreakerRegistry() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
            // 滑动窗口：基于计数
            .slidingWindowType(COUNT_BASED)
            .slidingWindowSize(100)

            // 慢调用阈值
            .slowCallDurationThreshold(Duration.ofMillis(500))
            .slowCallRateThreshold(50.0f)

            // 错误率阈值
            .failureRateThreshold(50.0f)

            // 最小调用数（窗口内至少有这么多调用才开始统计）
            .minimumNumberOfCalls(50)

            // 熔断持续时长
            .waitDurationInOpenState(Duration.ofSeconds(30))

            // 半开状态允许的调用数
            .permittedNumberOfCallsInHalfOpenState(10)

            // 忽略的异常类型
            .ignoreExceptions(
                BusinessWarningException.class)
            // 记录的异常类型
            .recordExceptions(
                RuntimeException.class,
                TimeoutException.class)

            .build();

        return CircuitBreakerRegistry.of(config);
    }
}
```

### 2.2 YAML 配置

```yaml
resilience4j:
  circuitbreaker:
    instances:
      user-service:
        sliding-window-type: COUNT_BASED
        sliding-window-size: 100
        slow-call-duration-threshold: 500ms
        slow-call-rate-threshold: 50
        failure-rate-threshold: 50

        slow-call-rate-threshold: 80
        failure-rate-threshold: 30
        minimum-number-of-calls: 20
        wait-duration-in-open-state: 60s
```

## 三、Sentinel 熔断规则配置

### 3.1 慢调用熔断

```java
@Configuration
public class SentinelCircuitBreakerConfig {

    @PostConstruct
    public void init() {
        // 慢调用比例熔断
        DegradeRule slowCallRule = new DegradeRule();
        slowCallRule.setResource("user-service");
        slowCallRule.setGrade(
            CircuitBreakerStrategy.SLOW_REQUEST_RATIO);
        slowCallRule.setCount(500);    // RT 阈值 500ms
        slowCallRule.setSlowRatioThreshold(0.5); // 慢调用比例 50%
        slowCallRule.setMinRequestAmount(10);    // 最小请求数
        slowCallRule.setStatIntervalMs(1000);    // 统计窗口 1s
        slowCallRule.setTimeWindow(10);          // 熔断时长 10s

        // 异常比例熔断
        DegradeRule errorRule = new DegradeRule();
        errorRule.setResource("payment-service");
        errorRule.setGrade(
            CircuitBreakerStrategy.ERROR_RATIO);
        errorRule.setCount(0.3);     // 错误率 30%
        errorRule.setMinRequestAmount(20);
        errorRule.setStatIntervalMs(1000);
        errorRule.setTimeWindow(10);

        // 异常数熔断
        DegradeRule errorCountRule = new DegradeRule();
        errorCountRule.setResource("inventory-service");
        errorCountRule.setGrade(
            CircuitBreakerStrategy.ERROR_COUNT);
        errorCountRule.setCount(10);  // 错误数 10 次
        errorCountRule.setMinRequestAmount(10);
        errorCountRule.setStatIntervalMs(1000);
        errorCountRule.setTimeWindow(10);

        DegradeRuleManager.loadRules(Arrays.asList(
            slowCallRule, errorRule, errorCountRule));
    }
}
```

## 四、不同场景的参数调优

### 4.1 高 QPS 场景

```
特点：请求量大，统计样本充足
建议：
  - minimumNumberOfCalls: 100~500（需要足够样本）
  - slidingWindowSize: 100~1000
  - failureRateThreshold: 30~50%
  - waitDurationInOpenState: 10~30s（快速恢复）
```

### 4.2 低 QPS 场景

```
特点：请求量小，统计样本不足
建议：
  - minimumNumberOfCalls: 10~20（降低门槛）
  - slidingWindowType: TIME_BASED（基于时间窗口）
  - slidingWindowSize: 60（60 秒窗口）
  - failureRateThreshold: 50~70%（适当放宽）
```

### 4.3 核心链路

```
特点：影响面大，需要保守策略
建议：
  - failureRateThreshold: 20~30%（更敏感）
  - slowCallRateThreshold: 30~50%
  - waitDurationInOpenState: 30~60s（更长保护期）
  - permittedNumberOfCallsInHalfOpenState: 5~10
```

### 4.4 非核心链路

```
特点：影响面小，可以适当放宽
建议：
  - failureRateThreshold: 50~70%
  - slowCallRateThreshold: 70~90%
  - waitDurationInOpenState: 10~20s
  - permittedNumberOfCallsInHalfOpenState: 10~20
```

## 五、规则调优方法论

### 5.1 调优流程

```
1. 基线设定     → 收集正常情况下的 RT、错误率数据
2. 初始规则     → 根据基线设定初始阈值（略高于基线）
3. 灰度验证     → 在非核心环境验证规则效果
4. 观察调整     → 监控熔断触发频率，逐步调整
5. 生效推广     → 验证通过后推广到生产环境
6. 持续优化     → 根据实际运行情况持续调优
```

### 5.2 调优经验

```
场景               初始配置              调优方向
────────────────────────────────────────────────────
频繁误熔断         阈值太低              提高阈值
响应延迟波动       RT 基线不准确          重新评估 RT 基线
偶发故障未熔断     阈值太高              降低阈值
恢复过慢           熔断时间太长           缩短 waitDuration
恢复后又熔断       半开状态放行太少        增加半开请求数
```

## 六、最佳实践

1. **规则要根据实际业务调优**，不要照搬默认配置
2. **太灵敏会影响正常请求**，误熔断伤害用户体验
3. **太迟钝起不到保护作用**，故障扩散影响全局
4. **生产环境要逐步调整**，每次只改一个参数
5. **监控规则效果持续优化**，建立调优闭环
6. **不同服务使用不同规则**，按核心程度分级配置
7. **定期复盘熔断事件**，优化规则配置
8. **使用 A/B 测试验证新规则**，降低变更风险
