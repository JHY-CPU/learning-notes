# 超时控制

## 一、超时配置

```yaml
# Feign 超时
feign:
  client:
    config:
      default:
        connect-timeout: 5000
        read-timeout: 10000

# Resilience4j 超时
resilience4j:
  timelimiter:
    instances:
      userService:
        timeout-duration: 5s
```

## 二、工作原理

超时控制通过设置调用的最大等待时间，超过后立即返回失败并触发降级逻辑。连接超时（connect timeout）控制建立 TCP 连接的最大等待时间，通常设置 1-5 秒。读取超时（read timeout）控制等待响应数据的最大时间，应根据下游服务的 P99 延迟设定，通常是 P99 的 1.5-2 倍。Resilience4j 的 TimeLimiter 在独立线程中执行调用，超时后中断线程并抛出 TimeoutException。Feign 的超时在 HTTP 客户端层面控制。

## 三、优缺点

**优点：**
- 防止请求无限等待，快速释放资源
- 超时后可触发降级，保障用户体验
- 配合熔断器快速发现下游异常

**缺点：**
- 超时设置过短可能导致正常请求被截断
- 超时后请求可能仍在下游执行（非幂等问题）
- 不同接口延迟差异大，统一超时不合理

## 四、最佳实践

1. 连接超时设 3 秒，读取超时根据接口 P99 延迟设定
2. 支付等核心接口超时可适当放宽，但必须有兜底
3. 在调用链路中逐级缩短超时，避免上游超时比下游还长
4. 超时后记录日志和监控指标，便于调优

## 五、完整超时配置

```yaml
# Spring Boot + Feign + HikariCP 超时配置
server:
  connection-timeout: 5000        # Tomcat 连接超时
  tomcat:
    connection-timeout: 5000
    keep-alive-timeout: 60000

spring:
  datasource:
    hikari:
      connection-timeout: 3000    # 获取连接超时
      idle-timeout: 600000        # 空闲连接超时
      max-lifetime: 1800000       # 连接最大生命周期
      validation-timeout: 3000    # 连接验证超时

feign:
  client:
    config:
      default:
        connect-timeout: 3000     # Feign 连接超时
        read-timeout: 5000        # Feign 读取超时
      slow-service:               # 特定服务更长超时
        connect-timeout: 3000
        read-timeout: 30000

resilience4j:
  timelimiter:
    instances:
      userService:
        timeout-duration: 3s      # Resilience4j 超时
        cancel-running-future: true
```

## 六、调用链路超时递减

```
正确: 超时逐级递减
客户端 10s
  └── 网关 8s
      └── 服务A 6s
          ├── 服务B 3s
          └── 服务C 3s

错误: 超时逐级递增
客户端 3s
  └── 网关 5s
      └── 服务A 8s       ← 客户端早就超时了

原则: 上游超时 < 下游超时之和 + 网络开销
```

```java
// 动态超时 - 根据接口设置不同超时
@Component
public class DynamicTimeoutInterceptor implements RequestInterceptor {
    @Override
    public void apply(RequestTemplate template) {
        String url = template.url();
        if (url.contains("/payment")) {
            // 支付接口给更长超时
            template.header("X-Timeout", "30000");
        } else if (url.contains("/search")) {
            // 搜索接口给中等超时
            template.header("X-Timeout", "5000");
        }
    }
}
```

## 七、常见陷阱

1. **超时设太长** - 故障时请求堆积，服务资源被耗尽
2. **超时设太短** - P99 以上的正常请求频繁被截断
3. **只设读取超时不设连接超时** - 连接阶段无限等待
4. **调用链路中超时未递减** - 上游 10 秒超时但下游总共只能 5 秒
5. **超时没有区分接口** - 支付和查询用同一个超时不合理
6. **数据库连接池超时未配置** - 获取连接无限制等待
