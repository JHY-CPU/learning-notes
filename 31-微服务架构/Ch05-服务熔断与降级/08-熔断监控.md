# 熔断监控

## 一、核心概念

熔断监控是对熔断器状态和触发情况的实时监控，帮助运维人员了解系统的自我保护状态，及时发现和处理异常。有效的熔断监控能帮助团队快速定位问题、优化熔断规则。

### 1.1 熔断器状态机

```
        失败率超过阈值          半开状态放行请求
  ┌──────────┐  ──────────►  ┌──────────┐
  │  关闭     │               │  半开     │
  │ (Closed) │               │ (Half-Open)│
  └──────────┘  ◄──────────  └──────────┘
       ▲          成功           │
       │                        │ 失败
       │                        ▼
       │                  ┌──────────┐
       └──────────────────│  打开     │
         重置时间后恢复     │ (Open)   │
                          └──────────┘
                             熔断期间拒绝请求
```

### 1.2 关键监控指标

```
指标类别              指标名称                  说明
────────────────────────────────────────────────
状态指标             circuit_state              熔断器状态
触发指标             circuit_break_count        熔断触发次数
统计指标             slow_call_count            慢调用次数
                     error_call_count           错误调用次数
                     total_call_count           总调用次数
性能指标             slow_call_ratio            慢调用比例
                     error_ratio                错误率
恢复指标             half_open_success_count    半开状态成功次数
                     recovery_time              恢复耗时
```

## 二、Sentinel Dashboard 监控

### 2.1 接入 Sentinel Dashboard

```yaml
spring:
  cloud:
    sentinel:
      transport:
        dashboard: sentinel-dashboard:8080
        port: 8719
      log:
        dir: /tmp/sentinel/logs
      # 数据聚合窗口
      metric:
        file-single-size: 52428800
        file-total-count: 6
```

### 2.2 自定义 Dashboard 数据源

```java
@Configuration
public class SentinelDashboardConfig {

    /**
     * 将熔断规则推送到 Dashboard
     */
    @Bean
    public DataSourceWriter circuitBreakerDataSource() {
        return new NacosDataSource<>(
            "nacos-server:8848",
            "sentinel-rules",
            "circuit-breaker-rules",
            source -> JSON.parseArray(
                source, CircuitBreakerRule.class)
        );
    }
}
```

### 2.3 Sentinel 实时监控数据

```java
@Component
public class SentinelMetricsCollector {

    /**
     * 定期收集 Sentinel 指标
     */
    @Scheduled(fixedRate = 10000)
    public void collectMetrics() {
        // 获取所有资源的秒级监控数据
        Map<String, MetricNode> metrics =
            MetricWriter.readMetrics();

        metrics.forEach((resource, node) -> {
            Metrics.gauge("sentinel_pass_qps",
                Tags.of("resource", resource),
                node.getPassQps());
            Metrics.gauge("sentinel_block_qps",
                Tags.of("resource", resource),
                node.getBlockQps());
            Metrics.gauge("sentinel_error_qps",
                Tags.of("resource", resource),
                node.getErrorQps());
            Metrics.gauge("sentinel_rt",
                Tags.of("resource", resource),
                node.getRt());
        });
    }
}
```

## 三、Prometheus + Grafana 监控

### 3.1 Prometheus 指标暴露

```yaml
management:
  endpoints:
    web:
      exposure:
        include: prometheus,health,metrics
  metrics:
    export:
      prometheus:
        enabled: true
    tags:
      application: ${spring.application.name}
```

### 3.2 熔断器指标注册

```java
@Component
public class CircuitBreakerMetrics {

    private final MeterRegistry registry;

    public CircuitBreakerMetrics(MeterRegistry registry,
            CircuitBreakerRegistry cbRegistry) {
        this.registry = registry;

        // 注册熔断器状态指标
        cbRegistry.circuitBreaker("user-service")
            .getEventPublisher()
            .onStateTransition(event -> {
                registry.gauge("circuit_breaker_state",
                    Tags.of("service", "user-service"),
                    stateToValue(event.getStateTransition()));
            });

        // 注册调用指标
        cbRegistry.circuitBreaker("user-service")
            .getEventPublisher()
            .onSuccess(event -> {
                registry.counter("circuit_breaker_calls",
                    "service", "user-service",
                    "result", "success").increment();
            })
            .onError(event -> {
                registry.counter("circuit_breaker_calls",
                    "service", "user-service",
                    "result", "error").increment();
            })
            .onCallNotPermitted(event -> {
                registry.counter("circuit_breaker_calls",
                    "service", "user-service",
                    "result", "not_permitted").increment();
            });
    }
}
```

### 3.3 Grafana 告警规则

```yaml
# Prometheus 告警规则
groups:
  - name: circuit-breaker-alerts
    rules:
      - alert: CircuitBreakerOpened
        expr: circuit_breaker_state > 0.5
        for: 1m
        labels:
          severity: warning
        annotations:
          summary: "熔断器打开"
          description: >
            服务 {{ $labels.service }}
            的熔断器已打开，请检查下游服务状态

      - alert: HighErrorRate
        expr: >
          rate(circuit_breaker_calls{result="error"}[5m])
          / rate(circuit_breaker_calls[5m]) > 0.1
        for: 2m
        labels:
          severity: critical
        annotations:
          summary: "高错误率"
          description: >
            服务 {{ $labels.service }}
            的错误率超过 10%
```

## 四、熔断告警通知

### 4.1 多渠道告警

```java
@Component
public class CircuitBreakerAlertService {

    @EventListener
    public void onCircuitBreakerStateChange(
            CircuitBreakerOnStateTransitionEvent event) {
        if (event.getStateTransition()
                .getToState() == State.OPEN) {
            // 发送钉钉告警
            sendDingTalkAlert(event);
            // 发送邮件告警
            sendEmailAlert(event);
            // 发送短信告警（严重情况）
            if (isCritical(event)) {
                sendSmsAlert(event);
            }
        }
    }

    private void sendDingTalkAlert(
            CircuitBreakerOnStateTransitionEvent event) {
        String message = String.format(
            "⚠️ 熔断告警\n服务: %s\n状态: %s → %s\n时间: %s",
            event.getCircuitBreaker().getName(),
            event.getStateTransition().getFromState(),
            event.getStateTransition().getToState(),
            LocalDateTime.now()
        );
        dingTalkClient.send(message);
    }
}
```

## 五、最佳实践

1. **Sentinel Dashboard 提供可视化监控**，部署并接入
2. **熔断触发要告警**，通过钉钉/邮件/短信通知
3. **监控数据要持久化**，使用 Prometheus 存储历史数据
4. **定期分析熔断数据调优规则**，避免过度熔断或熔断不足
5. **熔断规则要动态调整**，支持运行时修改
6. **建立熔断事件的复盘机制**，每次熔断都要分析原因
7. **区分业务熔断和依赖熔断**，告警策略不同
8. **监控熔断恢复时间**，评估系统自愈能力
