# 熔断降级概述

## 一、为什么需要熔断降级

```yaml
微服务问题:
  - 服务间调用形成依赖链
  - 一个服务故障可能导致雪崩
  - 网络不可靠，超时和失败常见

熔断降级解决:
  - 熔断器: 检测故障，快速失败
  - 降级逻辑: 故障时返回兜底数据
  - 限流: 控制流量防止过载
```

## 二、熔断器状态

```
Closed (正常) → Open (熔断) → Half-Open (半开)
  │                │                │
  │   失败率超阈值   │   定时尝试恢复   │
  └────────────────→└────────────────→
                          ↓
                    成功则 Closed
                    失败则 Open
```

## 三、主流组件

| 组件 | 特点 |
|------|------|
| Sentinel | 阿里开源，功能全面 |
| Resilience4j | 轻量级，函数式 |
| Hystrix | Netflix，已停止维护 |

## 四、熔断器详细状态机

```
CLOSED (正常状态):
├── 所有请求正常通过
├── 统计失败率/慢调用率
├── 当失败率 > 阈值 (如50%) 且调用次数 > 最小值
└── → 切换到 OPEN

OPEN (熔断状态):
├── 所有请求快速失败，不调用下游
├── 返回降级结果（fallback）
├── 等待指定时间（如 5 秒）
└── → 切换到 HALF_OPEN

HALF_OPEN (半开状态):
├── 放行少量请求探测下游
├── 如果成功次数 > 阈值
│   └── → 切换到 CLOSED
├── 如果仍然失败
│   └── → 切换到 OPEN
└── 在此状态下限制并发探测数
```

## 五、雪崩效应示意

```
服务A → 服务B → 服务C (故障!)
          │
          ├── 服务B 线程池被阻塞请求占满
          │
          ├── 服务A 调用服务B 超时
          │
          ├── 服务A 线程池也被占满
          │
          └── 整条链路全部不可用 = 雪崩

解决方案:
服务A ──熔断──→ 服务B ──熔断──→ 服务C
  │                │                │
  ▼                ▼                ▼
降级A            降级B            降级C
(默认值)         (缓存)           (静态页)
```

## 六、多层次防护

```
纵深防御体系:
┌─────────────────────────────────────┐
│ Layer 1: 网关层限流                  │
│ - QPS 限制、IP 限制、用户级限流      │
├─────────────────────────────────────┤
│ Layer 2: 服务间熔断                  │
│ - 失败率熔断、慢调用熔断             │
├─────────────────────────────────────┤
│ Layer 3: 线程池隔离                  │
│ - 每个依赖独立线程池                 │
├─────────────────────────────────────┤
│ Layer 4: 降级兜底                    │
│ - 缓存、默认值、静态数据             │
└─────────────────────────────────────┘
```

## 七、注意事项

1. **熔断是微服务必备能力** - 没有熔断的微服务系统随时可能雪崩
2. **Sentinel 国内最流行** - 功能全面，可视化控制台，动态规则
3. **降级逻辑要保证核心功能** - 支付失败要提示重试，不是返回空
4. **熔断阈值要根据实际调优** - 基于历史 QPS 和延迟数据设定
5. **监控熔断触发情况** - Prometheus 告警，熔断频繁触发说明依赖有问题
6. **熔断不是目的** - 熔断只是保护手段，根本解决依赖问题才是目标
