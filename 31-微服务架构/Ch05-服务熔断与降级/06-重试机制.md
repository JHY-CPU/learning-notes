# 重试机制

## 一、重试策略

```java
// Resilience4j 重试
@Retry(name = "userService", fallbackMethod = "fallback")
public User getUser(Long id) {
    return userService.getUser(id);
}

// 重试配置
resilience4j:
  retry:
    instances:
      userService:
        max-attempts: 3
        wait-duration: 1s
        exponential-backoff-multiplier: 2
        retry-exceptions:
          - java.io.IOException
        ignore-exceptions:
          - com.example.BusinessException
```

## 二、工作原理

重试机制在调用失败后自动重新发起请求，适用于瞬时故障（网络抖动、服务短暂不可用）。重试策略包括：固定间隔（每次等待相同时间）、指数退避（间隔逐次翻倍）、随机抖动（在退避基础上加随机偏移，避免多个客户端同时重试产生惊群效应）。重试前需判断异常类型：IOException 等瞬时异常可重试，BusinessException 等业务异常不可重试。重试必须配合幂等性，确保重复请求不会产生副作用。

## 三、优缺点

**优点：**
- 自动恢复瞬时故障，提高调用成功率
- 指数退避减少对下游的压力
- 对调用方透明，无需手动处理

**缺点：**
- 增加调用延迟（多次重试叠加超时）
- 非幂等接口重试可能导致数据重复
- 下游持续故障时重试加重雪崩

## 四、重试策略详解

```
重试间隔策略:
├── 固定间隔: 每次等待 N 秒
│   例: 1s, 1s, 1s
├── 指数退避: 间隔逐次翻倍
│   例: 1s, 2s, 4s, 8s
├── 随机抖动: 退避 + 随机偏移
│   例: 1s±0.5s, 2s±1s, 4s±2s
└── Fibonacci: 按斐波那契数列递增
    例: 1s, 1s, 2s, 3s, 5s
```

## 五、重试 + 熔断联动

```yaml
# Resilience4j 重试 + 熔断组合
resilience4j:
  retry:
    instances:
      userService:
        max-attempts: 3
        wait-duration: 500ms
        exponential-backoff-multiplier: 2
        retry-exceptions:
          - java.io.IOException
          - java.net.SocketTimeoutException
          - java.util.concurrent.TimeoutException
        ignore-exceptions:
          - com.example.BusinessException
          - com.example.NotFoundException
  circuitbreaker:
    instances:
      userService:
        sliding-window-size: 10
        failure-rate-threshold: 50
        wait-duration-in-open-state: 5s
```

```java
// 组合使用 - 重试 + 熔断 + 降级
@CircuitBreaker(name = "userService", fallbackMethod = "fallback")
@Retry(name = "userService")
@TimeLimiter(name = "userService")
public CompletableFuture<User> getUser(Long id) {
    return CompletableFuture.supplyAsync(() -> userService.getUser(id));
}

// 执行顺序: TimeLimiter → Retry → CircuitBreaker → 业务方法
// 异常传播: 业务异常 → Retry (可重试) → Retry 耗尽 → CircuitBreaker → fallback
```

## 六、幂等性保障方案

```java
// 方案1: 幂等 Key
@PostMapping("/orders")
public Order createOrder(@RequestBody CreateOrderRequest request,
                         @RequestHeader("Idempotency-Key") String idempotencyKey) {
    // 检查幂等 Key 是否已处理
    Order existing = orderRepository.findByIdempotencyKey(idempotencyKey);
    if (existing != null) return existing;

    Order order = new Order(request);
    order.setIdempotencyKey(idempotencyKey);
    return orderRepository.save(order);
}

// 方案2: 状态机 - 只允许合法状态转移
public Order payOrder(Long orderId) {
    Order order = orderRepository.findById(orderId);
    if (order.getStatus() != OrderStatus.CREATED) {
        throw new IllegalStateException("Order already processed");
    }
    order.setStatus(OrderStatus.PAID);
    return orderRepository.save(order);
}
```

## 七、常见陷阱

1. **非幂等操作使用重试** - 如创建订单重试导致重复订单，必须保证幂等性
2. **重试次数过多** - 单次请求耗时过长，建议最多 3 次
3. **无退避策略** - 大量客户端同时重试冲击下游，必须用指数退避+抖动
4. **重试所有异常** - 包括业务异常，浪费资源，只重试瞬时异常
5. **重试与熔断未联动** - 下游持续故障时重试加重雪崩
6. **重试产生惊群效应** - 大量客户端同时恢复时同时重试，加随机抖动
