# 请求聚合与 BFF 层设计

## 一、核心概念

在微服务架构中，一个前端页面可能需要调用多个后端服务获取数据。请求聚合（API Aggregation）通过在网关或 BFF（Backend for Frontend）层合并多个服务调用，减少客户端请求次数，降低网络开销，提升用户体验。

### 1.1 为什么需要请求聚合

```
无聚合（4次请求）:
  客户端 ──→ 用户服务（获取用户信息）
  客户端 ──→ 订单服务（获取订单列表）
  客户端 ──→ 积分服务（获取积分余额）
  客户端 ──→ 推荐服务（获取推荐商品）

有聚合（1次请求）:
  客户端 ──→ BFF/网关 ──┬──→ 用户服务
                        ├──→ 订单服务
                        ├──→ 积分服务
                        └──→ 推荐服务
  客户端 ←── 聚合结果 ←──┘
```

### 1.2 BFF 架构模式

```
┌─────────┐  ┌─────────┐  ┌─────────┐
│  Web端  │  │  App端  │  │ 小程序  │
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     ▼            ▼            ▼
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Web BFF │  │ App BFF │  │ Mini BFF│  ← 不同 BFF 适配不同端
└────┬────┘  └────┬────┘  └────┬────┘
     │            │            │
     └────────────┼────────────┘
                  ▼
     ┌─────────────────────────┐
     │   微服务集群（共享）      │
     │ User│Order│Payment│...  │
     └─────────────────────────┘
```

## 二、BFF 层实现

### 2.1 基于 CompletableFuture 的并行聚合

```java
@RestController
@RequestMapping("/api/web")
public class WebBFFController {

    @Autowired private UserServiceClient userClient;
    @Autowired private OrderServiceClient orderClient;
    @Autowired private PointsServiceClient pointsClient;
    @Autowired private RecommendServiceClient recommendClient;

    @GetMapping("/user-profile/{userId}")
    public UserProfileVO getUserProfile(
            @PathVariable Long userId) {
        // 并行调用多个微服务
        CompletableFuture<UserDTO> userFuture =
            CompletableFuture.supplyAsync(
                () -> userClient.getUser(userId));
        CompletableFuture<List<OrderDTO>> ordersFuture =
            CompletableFuture.supplyAsync(
                () => orderClient.getRecentOrders(userId, 10));
        CompletableFuture<PointsDTO> pointsFuture =
            CompletableFuture.supplyAsync(
                () -> pointsClient.getPoints(userId));
        CompletableFuture<List<ProductDTO>> recommendFuture =
            CompletableFuture.supplyAsync(
                () -> recommendClient.getRecommendations(userId));

        // 等待所有结果
        CompletableFuture.allOf(
            userFuture, ordersFuture,
            pointsFuture, recommendFuture
        ).join();

        // 聚合结果
        return UserProfileVO.builder()
            .user(userFuture.join())
            .recentOrders(ordersFuture.join())
            .points(pointsFuture.join())
            .recommendations(recommendFuture.join())
            .build();
    }
}
```

### 2.2 带超时和降级的聚合

```java
@RestController
public class RobustBFFController {

    private final ExecutorService executor =
        Executors.newFixedThreadPool(8);

    @GetMapping("/dashboard/{userId}")
    public DashboardVO getDashboard(
            @PathVariable Long userId) {
        CompletableFuture<UserDTO> userFuture =
            CompletableFuture.supplyAsync(
                () -> userClient.getUser(userId), executor)
                .orTimeout(3, TimeUnit.SECONDS)
                .exceptionally(e -> getDefaultUser(userId));

        CompletableFuture<List<OrderDTO>> ordersFuture =
            CompletableFuture.supplyAsync(
                () -> orderClient.getOrders(userId), executor)
                .orTimeout(5, TimeUnit.SECONDS)
                .exceptionally(e -> Collections.emptyList());

        CompletableFuture<PointsDTO> pointsFuture =
            CompletableFuture.supplyAsync(
                () -> pointsClient.getPoints(userId), executor)
                .orTimeout(2, TimeUnit.SECONDS)
                .exceptionally(e -> PointsDTO.zero());

        CompletableFuture.allOf(
            userFuture, ordersFuture, pointsFuture
        ).join();

        return DashboardVO.builder()
            .user(userFuture.join())
            .orders(ordersFuture.join())
            .points(pointsFuture.join())
            .build();
    }
}
```

### 2.3 响应式聚合（WebFlux）

```java
@RestController
public class ReactiveBFFController {

    @Autowired private WebClient userService;
    @Autowired private WebClient orderService;
    @Autowired private WebClient pointsService;

    @GetMapping("/profile/{userId}")
    public Mono<ProfileVO> getProfile(
            @PathVariable String userId) {
        Mono<UserDTO> userMono = userService.get()
            .uri("/users/{id}", userId)
            .retrieve()
            .bodyToMono(UserDTO.class);

        Mono<List<OrderDTO>> ordersMono = orderService.get()
            .uri("/orders?userId={id}", userId)
            .retrieve()
            .bodyToMono(new ParameterizedTypeReference<>() {});

        Mono<PointsDTO> pointsMono = pointsService.get()
            .uri("/points/{id}", userId)
            .retrieve()
            .bodyToMono(PointsDTO.class);

        return Mono.zip(userMono, ordersMono, pointsMono)
            .map(tuple -> ProfileVO.builder()
                .user(tuple.getT1())
                .orders(tuple.getT2())
                .points(tuple.getT3())
                .build());
    }
}
```

## 三、GraphQL 聚合方案

### 3.1 Schema 定义

```graphql
type Query {
    userProfile(id: ID!): UserProfile
    dashboard(userId: ID!): Dashboard
}

type UserProfile {
    user: User!
    orders: [Order!]!
    points: Points
    recommendations: [Product!]!
}

type User {
    id: ID!
    name: String!
    avatar: String
    level: Int
}

type Order {
    id: ID!
    productName: String!
    amount: Float!
    status: OrderStatus!
    createdAt: String!
}

type Points {
    total: Int!
    available: Int!
    expiring: Int
}
```

### 3.2 DataFetcher 实现

```java
@Component
public class UserProfileDataFetcher
        implements DataFetcher<CompletableFuture<UserProfile>> {

    @Autowired private UserServiceClient userClient;
    @Autowired private OrderServiceClient orderClient;
    @Autowired private PointsServiceClient pointsClient;

    @Override
    public CompletableFuture<UserProfile> get(
            DataFetchingEnvironment env) {
        String userId = env.getArgument("id");

        CompletableFuture<User> userF =
            CompletableFuture.supplyAsync(
                () -> userClient.getUser(userId));
        CompletableFuture<List<Order>> ordersF =
            CompletableFuture.supplyAsync(
                () -> orderClient.getOrders(userId));
        CompletableFuture<Points> pointsF =
            CompletableFuture.supplyAsync(
                () -> pointsClient.getPoints(userId));

        return CompletableFuture.allOf(userF, ordersF, pointsF)
            .thenApply(v -> new UserProfile(
                userF.join(), ordersF.join(), pointsF.join()));
    }
}
```

## 四、网关层请求聚合

### 4.1 Spring Cloud Gateway 聚合

```java
@Component
public class AggregationFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                              GatewayFilterChain chain) {
        String path = exchange.getRequest().getPath().value();

        if ("/api/aggregated/profile".equals(path)) {
            return aggregateRequests(exchange);
        }
        return chain.filter(exchange);
    }

    private Mono<Void> aggregateRequests(
            ServerWebExchange exchange) {
        // 并行请求多个后端服务
        Mono<Map> userReq = forwardRequest("/user-service/users/1");
        Mono<Map> orderReq = forwardRequest(
            "/order-service/orders?userId=1");
        Mono<Map> pointsReq = forwardRequest(
            "/points-service/points/1");

        return Mono.zip(userReq, orderReq, pointsReq)
            .map(tuple -> {
                Map<String, Object> result = new HashMap<>();
                result.put("user", tuple.getT1());
                result.put("orders", tuple.getT2());
                result.put("points", tuple.getT3());
                return result;
            })
            .flatMap(result -> writeResponse(exchange, result));
    }
}
```

## 五、最佳实践

### 5.1 聚合层设计原则

```
原则                  说明
──────────────────────────────────────
职责单一     BFF 只做聚合，不放业务逻辑
超时控制     每个下游调用都要有超时
降级策略     部分失败不影响核心数据返回
并行调用     使用异步并行减少总耗时
缓存优化     聚合结果可适当缓存
```

### 5.2 性能优化

```java

           unless = "#result == null")

public UserProfileVO getUserProfile(

    // ...聚合逻辑
}
```

### 5.3 注意事项

1. **并行调用减少总耗时**，避免串行调用叠加延迟
2. **超时设置要合理**，每个下游调用独立超时
3. **部分失败要有降级方案**，返回默认值而非整体失败
4. **BFF 层不要放太多业务逻辑**，只做数据适配和聚合
5. **不同客户端可以有不同的 BFF**，各自优化展示逻辑
6. **监控聚合接口的延迟**，识别慢下游服务
