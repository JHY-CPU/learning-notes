# 网关性能优化

## 一、核心概念

API 网关作为流量入口，性能直接决定系统吞吐量。Spring Cloud Gateway 基于 WebFlux（Netty）实现响应式非阻塞模型，但仍需从多个维度进行优化。本节介绍连接池、缓存、过滤器优化、压缩等关键策略。

### 1.1 性能瓶颈分析

```
请求处理流程及瓶颈点:
客户端 → [1.接受连接] → [2.路由匹配] → [3.过滤器链]
           → [4.后端调用] → [5.响应处理] → [6.返回客户端]

瓶颈点:
  1. 连接数限制 → 调整 Netty 参数
  2. 路由匹配   → 缓存路由配置
  3. 过滤器链   → 减少/优化过滤器
  4. 后端调用   → 连接池复用、超时控制
  5. 响应处理   → 压缩、批量写入
```

## 二、Netty 参数优化

### 2.1 Netty 核心参数

```yaml
server:
  port: 8080
  netty:
    connection-timeout: 5000

# Reactor Netty 参数
reactor:
  netty:
    tcp:
      # 事件循环线程数（默认 CPU 核数）
      event-loop-thread-count: 16
    http:
      # 最大连接数
      max-connections: 10000
      # 连接空闲超时
      idle-timeout: 60s
```

```java
@Configuration
public class NettyConfig {

    @Bean
    public NettyReactiveWebServerFactory webServerFactory() {
        NettyReactiveWebServerFactory factory =
            new NettyReactiveWebServerFactory();
        factory.addServerCustomizers(
            httpServer -> httpServer
                .tcpConfiguration(tcp -> tcp
                    .bootstrap(serverBootstrap -> {
                        serverBootstrap.option(
                            ChannelOption.SO_BACKLOG, 1024);
                        serverBootstrap.option(
                            ChannelOption.SO_REUSEADDR, true);
                        serverBootstrap.childOption(
                            ChannelOption.TCP_NODELAY, true);
                        serverBootstrap.childOption(
                            ChannelOption.SO_KEEPALIVE, true);
                        return serverBootstrap;
                    })
                )
        );
        return factory;
    }
}
```

## 三、连接池优化

### 3.1 WebClient 连接池配置

```java
@Configuration
public class WebClientConfig {

    @Bean
    public WebClient webClient() {
        // 连接池配置
        ConnectionProvider provider = ConnectionProvider
            .builder("custom-pool")
            .maxConnections(500)
            .maxIdleTime(Duration.ofSeconds(20))
            .maxLifeTime(Duration.ofSeconds(60))
            .pendingAcquireTimeout(Duration.ofSeconds(45))
            .evictInBackground(Duration.ofSeconds(30))
            .build();

        HttpClient httpClient = HttpClient
            .create(provider)
            .responseTimeout(Duration.ofSeconds(5));

        return WebClient.builder()
            .clientConnector(
                new ReactorClientHttpConnector(httpClient))
            .build();
    }
}
```

### 3.2 连接池监控

```java
@Component
public class ConnectionPoolMonitor {

    @Autowired
    private ConnectionProvider connectionProvider;

    @Scheduled(fixedRate = 10000)
    public void monitorPool() {
        // 监控连接池状态
        Metrics.gauge("connection.pool.active",
            connectionProvider,
            ConnectionProvider::metrics
                .map(m -> m.acquiredSize()));
        Metrics.gauge("connection.pool.idle",
            connectionProvider,
            ConnectionProvider::metrics
                .map(m -> m.idleSize()));
        Metrics.gauge("connection.pool.pending",
            connectionProvider,
            ConnectionProvider::metrics
                .map(m -> m.pendingAcquireSize()));
    }
}
```

## 四、路由缓存优化

### 4.1 路由定义缓存

```java
@Configuration
public class RouteConfig {

    @Bean
    public RouteLocator routeLocator(
            RouteLocatorBuilder builder) {
        return builder.routes()
            .route("user-service", r -> r
                .path("/api/users/**")
                .uri("lb://user-service"))
            .route("order-service", r -> r
                .path("/api/orders/**")
                .uri("lb://order-service"))
            .build();
        // 路由定义在启动时固定，减少运行时解析开销
    }
}
```

## 五、过滤器链优化

### 5.1 过滤器执行顺序

```java
// 减少全局过滤器数量，精简逻辑
@Component
@Order(-1)
public class OptimizedFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                              GatewayFilterChain chain) {
        // 在一个过滤器中完成多个职责
        long startTime = System.currentTimeMillis();
        String traceId = generateTraceId();

        // 设置 TraceID（替代单独的 TraceFilter）
        exchange.getRequest().mutate()
            .header("X-Trace-Id", traceId);

        return chain.filter(exchange)
            .doFinally(signal -> {
                // 记录指标（替代单独的 MetricsFilter）
                long duration =
                    System.currentTimeMillis() - startTime;
                Metrics.timer("gateway.request",
                    "path", exchange.getRequest()
                        .getPath().value())
                    .record(duration, TimeUnit.MILLISECONDS);
            });
    }
}
```

### 5.2 避免在过滤器中做阻塞操作

```java
// 错误：在过滤器中做阻塞调用
public Mono<Void> filter(ServerWebExchange exchange,
                          GatewayFilterChain chain) {
    // 阻塞！会拖慢整个网关
    User user = blockingUserClient.getUser(id);
    return chain.filter(exchange);
}

// 正确：使用异步调用
public Mono<Void> filter(ServerWebExchange exchange,
                          GatewayFilterChain chain) {
    return Mono.fromCallable(
            () -> userClient.getUser(id))
        .subscribeOn(Schedulers.boundedElastic())
        .flatMap(user -> {
            exchange.getAttributes().put("user", user);
            return chain.filter(exchange);
        });
}
```

## 八、最佳实践

1. **Spring Cloud Gateway 是响应式的**，不要在过滤器中写阻塞代码
2. **连接池要合理配置**，根据 QPS 调整最大连接数
3. **缓存路由配置**，避免频繁的路由解析
4. **精简过滤器链**，合并职责相近的过滤器
5. **启用响应压缩**，减少网络传输量
6. **监控网关的 CPU 和内存**，定期分析性能瓶颈
7. **根据 QPS 调整实例数**，使用自动扩缩容
8. **JVM 参数要针对响应式场景调优**
