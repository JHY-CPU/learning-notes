# 灰度发布

## 一、灰度策略

```yaml
# 按 Header 灰度
spring:
  cloud:
    gateway:
      routes:
      - id: user-service-v1
        uri: lb://user-service-v1
        predicates:
        - Header=X-Version, v1
      - id: user-service-v2
        uri: lb://user-service-v2
        predicates:
        - Header=X-Version, v2
```

## 二、工作原理

灰度发布（金丝雀发布）的核心是将部分流量导向新版本，其余流量保持在旧版本。网关层通过路由断言（Predicate）实现流量分割：基于 Header 匹配、Cookie、请求参数或权重比例。权重灰度通过自定义 Filter 实现，在 Filter 中生成随机数并与配置的权重阈值比较，决定路由到哪个版本。灰度规则通过配置中心动态下发，无需重启网关。灰度期间通过 Prometheus 等监控工具对比新旧版本的错误率、延迟等指标，确认无异常后逐步扩大新版本流量比例。

## 三、优缺点

**优点：**
- 降低发布风险，问题版本影响范围可控
- 可在线对比新旧版本性能指标
- 用户无感知，体验平滑

**缺点：**
- 需要维护多版本同时运行，资源成本翻倍
- 数据库 schema 变更时灰度复杂度大增
- 灰度规则配置和管理增加运维负担

## 四、最佳实践

1. 灰度比例从 1% 开始，逐步扩大到 5% -> 20% -> 50% -> 100%
2. 灰度期间密切监控错误率和延迟，设置自动回滚阈值
3. 使用用户维度灰度（如特定用户 ID）而非随机灰度，便于复现问题
4. 灰度版本和正式版本使用独立的日志标签，便于区分

## 五、权重灰度实现

```java
// 自定义权重灰度 Filter
@Component
public class WeightBasedRoutingFilter implements GlobalFilter, Ordered {
    private final Random random = new Random();

    @Value("${canary.weight:10}")  // 灰度权重百分比
    private int canaryWeight;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        // 优先按用户 ID 灰度
        String userId = exchange.getRequest().getHeaders().getFirst("X-User-Id");
        if (userId != null && Math.abs(userId.hashCode()) % 100 < canaryWeight) {
            exchange.getAttributes().put("targetVersion", "canary");
        } else if (random.nextInt(100) < canaryWeight) {
            exchange.getAttributes().put("targetVersion", "canary");
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() { return 0; }
}
```

## 六、K8s + Nginx Ingress 灰度

```yaml
# Ingress-Nginx 金丝雀注解
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: user-service-canary
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"      # 10% 流量
    nginx.ingress.kubernetes.io/canary-by-header: "X-Canary"  # 强制灰度
    nginx.ingress.kubernetes.io/canary-by-cookie: "canary"    # Cookie 灰度
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: user-service-v2
            port:
              number: 80
```

## 七、灰度发布流程

```
灰度发布完整流程:
1. 准备阶段
   ├── 新版本开发完成，通过测试环境验证
   ├── 编写回滚方案
   └── 通知相关团队

2. 灰度阶段
   ├── 1% 流量 → 监控 30 分钟 → 错误率 < 0.1%？
   ├── 5% 流量 → 监控 1 小时 → 延迟增长 < 10%？
   ├── 20% 流量 → 监控 2 小时 → 无告警？
   └── 50% 流量 → 监控 4 小时 → 指标正常？

3. 全量阶段
   ├── 100% 流量切换到新版本
   ├── 保留旧版本 24 小时待回滚
   └── 24 小时后下线旧版本

4. 异常回滚
   ├── 任一阶段出现异常 → 立即回滚到 0%
   ├── 分析根因
   └── 修复后重新灰度
```

## 八、注意事项

1. **灰度比例直接设为 50%** - 问题版本一上线就影响一半用户，应从 1% 开始
2. **灰度回滚需要手动操作** - 应急响应慢，必须实现自动回滚
3. **数据库不兼容导致灰度版本报错** - 需提前做兼容性设计（向后兼容 schema）
4. **灰度期间忘记清理测试数据** - 污染生产环境，灰度用户数据要隔离
5. **使用用户维度灰度** - 便于复现问题，随机灰度难以排查
