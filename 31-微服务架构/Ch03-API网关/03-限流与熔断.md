# 限流与熔断

## 一、限流策略

```java
// Sentinel 限流
@SentinelResource(value = "getUser", blockHandler = "getUserBlockHandler")
public User getUser(Long id) {
    return userService.getUser(id);
}

public User getUserBlockHandler(Long id, BlockException ex) {
    return User.defaultUser(id);
}
```

```yaml
# 限流规则
spring:
  cloud:
    sentinel:
      datasource:
        flow:
          nacos:
            server-addr: localhost:8848
            dataId: flow-rules
            groupId: SENTINEL_GROUP
```

## 二、熔断策略

```java
// Resilience4j 熔断
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
public User getUser(Long id) {
    return userService.getUser(id);
}

public User getUserFallback(Long id, Throwable t) {
    return User.defaultUser(id);
}
```

## 三、限流算法

| 算法 | 说明 | 特点 |
|------|------|------|
| 令牌桶 | 定时添加令牌 | 允许突发 |
| 漏桶 | 固定速率流出 | 平滑流量 |
| 计数器 | 固定窗口计数 | 简单 |
| 滑动窗口 | 滑动窗口计数 | 精确 |

## 四、Sentinel 网关限流完整配置

```java
// Sentinel 网关限流配置
@Configuration
public class SentinelGatewayConfig {
    @PostConstruct
    public void init() {
        // 网关维度限流
        GatewayFlowRule rule = new GatewayFlowRule("user-service")
            .setCount(100)                    // QPS 阈值
            .setIntervalSec(1)                // 统计窗口
            .setControlBehavior(0)            // 快速失败
            .setBurst(20);                    // 突发容忍

        Set<GatewayFlowRule> rules = new HashSet<>();
        rules.add(rule);
        GatewayRuleManager.loadRules(rules);
    }
}

// API 维度限流
GatewayFlowRule apiRule = new GatewayFlowRule("user-service")
    .setResourceMode(1)                   // API 维度
    .setParseStrategy(1)                  // 按 Path 解析
    .setPattern("/api/users/**")
    .setCount(50)                         // 该 API 限流 50 QPS
    .setIntervalSec(1);
```

## 五、熔断降级详细流程

```
熔断器状态机:
CLOSED ──(错误率>50%)──→ OPEN ──(等待5s)──→ HALF_OPEN
  ▲                                           │
  │                                           │
  └──────(成功次数达标)─────────────────────────┘
  │                                           │
  └──────(仍然失败)────────────────→ OPEN ─────┘

CLOSED: 正常状态，所有请求通过
OPEN:   熔断状态，快速失败，不调用下游
HALF_OPEN: 半开状态，放行少量请求探测
```

```yaml
# Resilience4j 完整配置
resilience4j:
  circuitbreaker:
    instances:
      userService:
        sliding-window-size: 10             # 滑动窗口大小
        failure-rate-threshold: 50          # 失败率阈值 50%
        wait-duration-in-open-state: 5s     # OPEN 状态等待时间
        permitted-number-of-calls-in-half-open-state: 3  # HALF_OPEN 允许请求数
        minimum-number-of-calls: 10         # 最少调用次数才开始计算
  timelimiter:
    instances:
      userService:
        timeout-duration: 3s                # 单次调用超时

# Spring Cloud Gateway 熔断配置
spring:
  cloud:
    gateway:
      routes:
      - id: user-service
        uri: lb://user-service
        predicates:
        - Path=/api/users/**
        filters:
        - name: CircuitBreaker
          args:
            name: userService
            fallbackUri: forward:/fallback/user
```

## 六、限流维度

| 维度 | 说明 | 实现 |
|------|------|------|
| 全局限流 | 网关总 QPS | GatewayFlowRule |
| 服务维度 | 每个服务的 QPS | 按 routeId 限流 |
| API 维度 | 每个 API 的 QPS | 按 Path 限流 |
| 用户维度 | 每个用户的 QPS | 按 Header/Cookie |
| IP 维度 | 每个 IP 的 QPS | 按 RemoteAddr |

## 七、注意事项

1. **限流在网关层统一配置** - 避免各服务重复实现，规则统一管理
2. **熔断阈值要根据实际调优** - 不同服务的 SLA 不同，阈值也不同
3. **降级逻辑要保证核心功能可用** - 返回默认数据或缓存数据
4. **限流规则要支持动态调整** - 通过 Nacos/Apollo 动态下发
5. **监控限流和熔断的触发情况** - Prometheus + Grafana 实时监控
6. **避免误杀正常用户** - 限流粒度要合理，考虑突发流量
