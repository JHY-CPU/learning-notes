# 认证与授权

## 一、统一认证

```java
@Component
public class AuthFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (token == null || !token.startsWith("Bearer ")) {
            return unauthorized(exchange);
        }
        // 验证 JWT
        Claims claims = JwtUtil.parse(token.substring(7));
        // 传递用户信息
        exchange.getRequest().mutate()
            .header("X-User-Id", claims.getSubject())
            .header("X-User-Roles", claims.get("roles", String.class));
        return chain.filter(exchange);
    }
}
```

## 二、工作原理

网关作为所有请求的统一入口，在 Filter 链中拦截请求并验证身份。JWT 方案中，客户端登录后获取 Token，后续请求携带在 Authorization Header 中。网关解析 Token 签名和过期时间，验证通过后提取用户 ID 和角色信息，注入到请求 Header 中传递给下游服务。下游服务从 Header 获取用户信息，不再重复验证。白名单接口（如登录、注册、静态资源）通过路由配置跳过认证 Filter。

## 三、优缺点

**优点：**
- 下游服务无需实现认证逻辑，职责清晰
- JWT 无状态，不需要服务端存储 Session
- 网关层统一处理，减少重复代码

**缺点：**
- 网关成为安全单点，一旦绕过则全量暴露
- JWT 无法主动失效，Token 泄露后在过期前一直有效
- 复杂权限（如数据级权限）仍需下游服务自行判断

## 四、最佳实践

1. Token 过期时间不宜过长，建议 30 分钟到 2 小时
2. 使用 Refresh Token 机制实现无感续签
3. 敏感操作增加二次验证（如短信验证码）
4. 网关与下游服务之间使用内部 Header，不信任外部传入

## 五、常见陷阱

1. **Token 密钥泄露**，攻击者可伪造任意用户 Token
2. **未校验 Token 签名**，只解码不验签等于没有认证
3. **白名单配置过于宽松**，暴露了本该保护的接口
4. **用户信息 Header 可被伪造**，下游服务必须只信任网关注入的 Header
