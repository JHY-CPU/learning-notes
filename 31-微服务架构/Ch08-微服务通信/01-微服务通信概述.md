# 微服务通信概述

## 一、通信模式

```yaml
同步通信:
  - REST HTTP - 简单通用
  - gRPC - 高性能二进制
  - GraphQL - 灵活查询

异步通信:
  - 消息队列 - Kafka/RabbitMQ/RocketMQ
  - 事件驱动 - Event Sourcing
```

## 二、模式选择

| 场景 | 推荐 |
|------|------|
| 查询数据 | REST HTTP |
| 内部调用 | gRPC |
| 事件通知 | MQ |
| 实时推送 | WebSocket |

## 三、工作原理

微服务通信分为同步和异步两种模式。同步通信中，调用方发起请求后阻塞等待响应，典型场景是 REST HTTP 和 gRPC。异步通信中，调用方发送消息后立即返回，消费方在后台处理，典型场景是 MQ 和事件驱动。选择依据：强依赖（调用结果直接影响后续逻辑）用同步，弱依赖（通知、日志、异步处理）用异步。同步调用链过长会导致延迟叠加和级联故障，应通过异步解耦或 CQRS 模式缩短调用链。

## 四、优缺点

**同步优点：** 实时性强、实现简单、调试方便
**同步缺点：** 耦合度高、级联故障风险、延迟叠加

**异步优点：** 解耦、削峰填谷、高吞吐
**异步缺点：** 最终一致性、调试复杂、消息丢失风险

## 五、通信协议选择指南

```
选择依据:
├── 调用结果是否影响后续逻辑？
│   ├── 是 → 同步 (REST/gRPC)
│   └── 否 → 异步 (MQ/事件)
├── 是否需要高性能？
│   ├── 是 → gRPC (Protobuf 序列化)
│   └── 否 → REST (JSON 序列化)
├── 是否需要广播通知？
│   ├── 是 → MQ (发布/订阅)
│   └── 否 → 点对点通信
└── 是否需要实时双向通信？
    ├── 是 → WebSocket
    └── 否 → HTTP 请求/响应
```

## 六、序列化格式对比

| 格式 | 体积 | 速度 | 可读性 | 跨语言 | 适用场景 |
|------|------|------|--------|--------|---------|
| JSON | 大 | 慢 | 好 | 极好 | REST API |
| Protobuf | 小 | 快 | 差 | 好 | gRPC |
| Avro | 中 | 中 | 差 | 好 | Kafka |
| Hessian | 中 | 中 | 差 | 中 | Dubbo |
| MessagePack | 小 | 快 | 差 | 好 | 高性能场景 |

## 七、常见陷阱

1. **同步调用链超过 5 层** - 延迟叠加严重，考虑异步解耦
2. **强依赖场景用异步** - 导致业务逻辑不连贯
3. **序列化格式不统一** - 服务间通信失败，统一用 JSON 或 Protobuf
4. **无超时配置** - 一个服务慢导致整个调用链阻塞
5. **忽略网络分区** - 服务间通信必须考虑网络不可靠性
6. **通信没有监控** - 延迟和错误率上升时无法及时发现
