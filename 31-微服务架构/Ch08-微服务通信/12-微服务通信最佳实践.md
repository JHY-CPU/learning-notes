# 微服务通信最佳实践

## 一、实践清单

```yaml
同步通信:
  □ Feign 声明式调用
  □ 超时必须配置
  □ 熔断降级要有
  □ 接口版本管理
  □ 错误码规范化

异步通信:
  □ 消息格式统一
  □ 消费幂等处理
  □ 死信队列兜底
  □ 消费延迟监控
  □ 消息轨迹追踪
```

## 二、选型建议

| 场景 | 推荐 |
|------|------|
| 查询数据 | REST HTTP |
| 内部调用 | gRPC |
| 事件通知 | MQ |
| 实时推送 | WebSocket |

## 三、工作原理

微服务通信最佳实践围绕协议选择、可靠性保障、可观测性三个维度。协议选择：内部调用优先 gRPC（高性能），对外暴露用 REST（通用），事件通知用 MQ（解耦）。可靠性保障：同步调用配置超时 + 熔断 + 降级，异步通信保证消费幂等 + 死信队列兜底。可观测性：所有通信携带 TraceID，接入链路追踪系统，监控调用成功率和延迟。

## 四、优缺点

**优点：**
- 分层通信模式适配不同业务需求
- 可靠性机制保障通信稳定
- 可观测性支撑问题快速定位

**缺点：**
- 多种协议增加技术栈复杂度
- 可靠性配置需要持续调优
- 可观测性组件增加资源开销

## 五、通信安全

```java
// 服务间 mTLS 认证
@Configuration
public class MtlsConfig {
    @Bean
    public RestTemplate restTemplate() throws Exception {
        SSLContext sslContext = SSLContextBuilder.create()
            .loadKeyMaterial(
                new File("client-keystore.p12"),
                "password".toCharArray(),
                "password".toCharArray())
            .loadTrustMaterial(
                new File("client-truststore.p12"),
                "password".toCharArray())
            .build();

        HttpClient client = HttpClients.custom()
            .setSSLContext(sslContext)
            .build();

        return new RestTemplateBuilder()
            .requestFactory(() -> new HttpComponentsClientHttpRequestFactory(client))
            .build();
    }
}
```

## 六、消息队列最佳实践

```yaml
MQ 最佳实践:
  生产者:
    - 异步发送 + 回调确认
    - 消息体不超过 1MB
    - 关键消息使用事务消息

  消费者:
    - 消费幂等处理
    - 消费失败重试 (最多3次)
    - 超过重试次数进死信队列

  死信队列:
    - 独立消费者处理死信
    - 记录日志 + 告警
    - 支持人工重新消费

  监控:
    - 消费延迟 < 10s
    - 消费成功率 > 99.9%
    - 死信队列积压 < 100
```

## 七、常见陷阱

1. **所有调用都用 REST** - 内部调用性能不佳，gRPC 更适合
2. **异步消息无幂等** - 重复消费导致数据错误
3. **无死信队列** - 消息消费失败后丢失
4. **通信无监控** - 延迟和失败率上升时无法及时发现
5. **消息体过大** - 影响 MQ 性能，应只传递事件标识和关键字段
6. **忽略消息顺序性** - 有顺序要求的场景需要使用分区键
