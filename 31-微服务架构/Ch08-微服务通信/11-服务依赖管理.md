# 服务依赖管理

## 一、核心概念

在微服务架构中，服务之间存在复杂的依赖关系。服务依赖管理是指识别、分析和治理这些依赖关系，确保系统的可维护性和稳定性。

### 1.1 依赖类型

```
依赖类型        特征                    处理策略
──────────────────────────────────────────────────
强依赖         调用失败则业务失败        熔断、降级
弱依赖         调用失败不影响核心功能     容错、返回默认值
循环依赖       A→B→C→A                必须打破
冗余依赖       可以合并的调用           合并优化
```

### 1.2 依赖关系矩阵

```
         │ 用户服务 │ 订单服务 │ 支付服务 │ 库存服务 │ 通知服务
─────────┼─────────┼─────────┼─────────┼─────────┼────────
用户服务  │    -    │    -    │    -    │    -    │   弱
订单服务  │   强    │    -    │   强    │   强    │   弱
支付服务  │   弱    │    -    │    -    │    -    │   弱
库存服务  │    -    │    -    │    -    │    -    │    -
通知服务  │   弱    │   弱    │   弱    │    -    │    -
```

## 二、依赖发现

### 2.1 从链路追踪自动发现

```java
@Service
public class DependencyDiscoveryService {

    @Autowired
    private TraceStorage traceStorage;

    /**
     * 从 Span 数据自动发现服务依赖
     */
    public List<ServiceDependency> discover(
            Duration timeRange) {
        long end = System.currentTimeMillis();
        long start = end - timeRange.toMillis();

        List<Span> spans =
            traceStorage.querySpans(start, end);

        // 提取调用关系
        Map<String, ServiceDependency> depMap =
            new HashMap<>();

        for (Span span : spans) {
            for (Span child : span.getChildren()) {
                String key = span.getServiceName()
                    + "->" + child.getServiceName();

                depMap.computeIfAbsent(key,
                    k -> new ServiceDependency(
                        span.getServiceName(),
                        child.getServiceName()))
                    .increment(child.getDuration(),
                        child.hasError());
            }
        }

        return new ArrayList<>(depMap.values());
    }
}
```

## 三、依赖分析

### 3.1 循环依赖检测

```java
@Service
public class CycleDetector {

    /**
     * 使用 DFS 检测循环依赖
     */
    public List<List<String>> detectCycles(
            List<ServiceDependency> dependencies) {
        // 构建邻接表
        Map<String, List<String>> graph = new HashMap<>();
        for (ServiceDependency dep : dependencies) {
            graph.computeIfAbsent(
                dep.getSource(), k -> new ArrayList<>())
                .add(dep.getTarget());
        }

        List<List<String>> cycles = new ArrayList<>();
        Set<String> visited = new HashSet<>();
        Set<String> recursionStack = new HashSet<>();

        for (String node : graph.keySet()) {
            if (!visited.contains(node)) {
                dfs(node, graph, visited,
                    recursionStack, new Stack<>(), cycles);
            }
        }
        return cycles;
    }

    private void dfs(String node,
                      Map<String, List<String>> graph,
                      Set<String> visited,
                      Set<String> recursionStack,
                      Stack<String> path,
                      List<List<String>> cycles) {
        visited.add(node);
        recursionStack.add(node);
        path.push(node);

        for (String neighbor :
                graph.getOrDefault(node,
                    Collections.emptyList())) {
            if (!visited.contains(neighbor)) {
                dfs(neighbor, graph, visited,
                    recursionStack, path, cycles);
            } else if (recursionStack.contains(neighbor)) {
                // 发现循环
                List<String> cycle = new ArrayList<>();
                for (int i = path.size() - 1; i >= 0; i--) {
                    cycle.add(path.get(i));
                    if (path.get(i).equals(neighbor)) break;
                }
                Collections.reverse(cycle);
                cycles.add(cycle);
            }
        }

        path.pop();
        recursionStack.remove(node);
    }
}
```

### 3.2 依赖健康评估

```java
@Service
public class DependencyHealthEvaluator {

    /**
     * 评估依赖健康度
     */
    public HealthScore evaluate(
            ServiceDependency dependency) {
        double score = 100.0;

        // 错误率评分
        if (dependency.getErrorRate() > 0.01) {
            score -= 30;
        } else if (dependency.getErrorRate() > 0.001) {
            score -= 10;
        }

        // 延迟评分
        if (dependency.getAvgLatencyMs() > 1000) {
            score -= 20;
        } else if (dependency.getAvgLatencyMs() > 500) {
            score -= 10;
        }

        // 依赖类型评分（强依赖更危险）
        if (dependency.getType() == DependencyType.STRONG) {
            score -= 5; // 强依赖扣分更多
        }

        return HealthScore.builder()
            .service(dependency.getSource() + "->"
                + dependency.getTarget())
            .score(Math.max(0, score))
            .grade(score >= 80 ? "A"
                : score >= 60 ? "B"
                : score >= 40 ? "C" : "D")
            .build();
    }
}
```

## 四、依赖治理

### 4.1 强依赖熔断保护

```java
@Service
public class DependencyGovernance {

    /**
     * 为强依赖配置熔断器
     */
    public void configureCircuitBreaker(
            ServiceDependency dependency) {
        if (dependency.getType() == DependencyType.STRONG) {
            CircuitBreakerConfig config =
                CircuitBreakerConfig.custom()
                    .failureRateThreshold(30)   // 更敏感
                    .waitDurationInOpenState(
                        Duration.ofSeconds(30))
                    .slidingWindowSize(50)
                    .minimumNumberOfCalls(20)
                    .build();
            circuitBreakerRegistry
                .circuitBreaker(
                    dependency.getTarget(), config);
        }
    }
}
```

### 4.2 弱依赖降级

```java
@Service
public class WeakDependencyCall {

    @CircuitBreaker(name = "recommend-service",
                    fallbackMethod = "getFallback")
    public List<Product> getRecommendations(Long userId) {
        return recommendClient.getRecommendations(userId);
    }

    public List<Product> getFallback(Long userId,
                                      Exception e) {
        // 弱依赖降级：返回空列表，不影响核心功能
        log.warn("推荐服务降级: {}", e.getMessage());
        return Collections.emptyList();
    }
}
```

### 4.3 打破循环依赖

```
方案一：提取公共模块
  A → B → A  变为  A → Common ← B

方案二：引入事件驱动
  A → B → A  变为  A → MQ → B（异步）

方案三：依赖倒置
  A → B → A  变为  A → Interface ← B
```

```java
// 方案二：事件驱动解耦循环依赖
@Service
public class OrderService {

    @Autowired
    private KafkaTemplate<String, OrderEvent> kafka;

    public Order createOrder(OrderRequest request) {
        Order order = orderRepository.save(
            buildOrder(request));

        // 发布事件，异步通知库存服务
        kafka.send("order-created",
            new OrderEvent(order.getId()));

        return order;
        // 不再直接调用库存服务
    }
}

@Component
public class InventoryEventListener {

    @KafkaListener(topics = "order-created")
    public void onOrderCreated(OrderEvent event) {
        // 库存服务异步处理
        inventoryService.deduct(event.getOrderId());
    }
}
```

### 5.2 依赖变更通知

```java
@Component
public class DependencyChangeDetector {

    @Scheduled(fixedRate = 3600000) // 每小时检查
    public void detectChanges() {
        List<ServiceDependency> current =
            discoveryService.discover(Duration.ofHours(1));
        List<ServiceDependency> previous =
            dependencyRepository.findAllActive();

        // 检测新增依赖
        Set<String> currentKeys = current.stream()
            .map(this::toKey)
            .collect(Collectors.toSet());
        Set<String> previousKeys = previous.stream()
            .map(this::toKey)
            .collect(Collectors.toSet());

        Sets.SetView<String> newDeps = Sets.difference(
            currentKeys, previousKeys);
        Sets.SetView<String> removedDeps = Sets.difference(
            previousKeys, currentKeys);

        if (!newDeps.isEmpty()) {
            alertService.send("NEW_DEPENDENCY",
                "发现新依赖: " + newDeps);
        }
        if (!removedDeps.isEmpty()) {
            alertService.send("REMOVED_DEPENDENCY",
                "依赖移除: " + removedDeps);
        }
    }
}
```

## 六、最佳实践

1. **链路追踪自动发现依赖**，不需要手动维护
2. **强依赖要做熔断**，防止级联故障
3. **弱依赖可以降级**，返回默认值不影响核心功能
4. **避免循环依赖**，使用事件驱动或依赖倒置
5. **依赖关系要文档化**，定期更新依赖矩阵
6. **定期审查依赖健康度**，识别不健康的依赖
7. **新增/移除依赖要通知**，确保团队知情
8. **依赖变更要灰度验证**，避免全量变更导致故障
