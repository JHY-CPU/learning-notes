# 消息队列通信

## 一、核心概念

消息队列（Message Queue）是微服务之间异步通信的核心组件。通过消息队列，服务之间可以实现解耦、异步处理、流量削峰。本节对比主流消息队列（Kafka、RabbitMQ、RocketMQ），介绍使用场景和最佳实践。

### 1.1 同步 vs 异步通信

```
同步调用 (HTTP/RPC):
  服务A ──请求──► 服务B ──响应──► 服务A
  特点: 实时性强，耦合度高，一个挂了另一个也挂

异步调用 (MQ):
  服务A ──消息──► 消息队列 ──消息──► 服务B
  特点: 解耦、削峰、异步，但有延迟
```

### 1.2 消息队列对比

| 特性 | Kafka | RabbitMQ | RocketMQ |
|------|-------|----------|----------|
| 开发语言 | Scala/Java | Erlang | Java |
| 吞吐量 | 极高（百万级/秒） | 中等（万级/秒） | 高（十万级/秒） |
| 延迟 | 较高（ms 级） | 低（us 级） | 低（ms 级） |
| 消息持久化 | 磁盘顺序写 | 内存+磁盘 | 磁盘 |
| 消息可靠性 | 高（副本机制） | 高（ACK） | 高（同步刷盘） |
| 消息顺序 | 分区内有序 | 队列有序 | 队列有序 |
| 消息回溯 | 支持（按 offset） | 不支持 | 支持（按时间） |
| 事务消息 | 支持 | 支持 | 原生支持 |
| 适用场景 | 大数据、日志、流处理 | 业务消息、任务队列 | 电商、金融、事务消息 |

### 2.3 生产者实现

```java
@Service
public class OrderEventProducer {

    @Autowired
    private KafkaTemplate<String, OrderEvent> kafkaTemplate;

    /**
     * 发送订单事件
     */
    public void sendOrderCreated(Order order) {
        OrderEvent event = OrderEvent.builder()
            .eventType("ORDER_CREATED")
            .orderId(order.getId())
            .userId(order.getUserId())
            .amount(order.getAmount())
            .timestamp(System.currentTimeMillis())
            .build();

        // 使用 orderId 作为 key，保证同一订单的消息有序
        kafkaTemplate.send(
            "order-events",
            order.getId().toString(),
            event
        ).addCallback(
            result -> log.info("消息发送成功: {}", event),
            ex -> log.error("消息发送失败", ex)
        );
    }

    /**
     * 事务消息：业务操作和消息发送在同一个事务中
     */
    @Transactional
    public void createOrderWithEvent(OrderRequest request) {
        // 1. 保存订单
        Order order = orderRepository.save(
            buildOrder(request));

        // 2. 在事务中发送消息
        kafkaTemplate.executeInTransaction(ops -> {
            ops.send("order-events",
                order.getId().toString(),
                new OrderEvent("ORDER_CREATED", order));
            return null;
        });
    }
}
```

### 2.4 消费者实现

```java
@Component
public class OrderEventConsumer {

    @KafkaListener(
        topics = "order-events",
        groupId = "inventory-consumer-group",
        containerFactory = "kafkaListenerContainerFactory"
    )
    public void onOrderCreated(
            @Payload OrderEvent event,
            @Header(KafkaHeaders.RECEIVED_KEY) String key,
            @Header(KafkaHeaders.RECEIVED_PARTITION) int partition,
            @Header(KafkaHeaders.OFFSET) long offset,
            Acknowledgment ack) {
        try {
            log.info("收到订单事件: partition={}, offset={}, event={}",
                partition, offset, event);

            // 幂等处理
            if (processedEvents.contains(event.getEventId())) {
                log.info("重复事件，跳过: {}", event.getEventId());
                ack.acknowledge();
                return;
            }

            // 业务处理
            inventoryService.reserve(
                event.getOrderId(),
                event.getItems()
            );

            // 手动确认
            processedEvents.add(event.getEventId());
            ack.acknowledge();

        } catch (Exception e) {
            log.error("处理订单事件失败", e);
            // 重试或发送到死信队列
            throw e;
        }
    }
}
```

```yaml

    addresses: rabbitmq1:5672,rabbitmq2:5672
    username: ${RABBITMQ_USER}

      simple:
        acknowledge-mode: manual        # 手动确认
        concurrency: 5
        max-concurrency: 10
        prefetch: 10                    # 预取数量
        retry:
          enabled: true
          max-attempts: 3
          initial-interval: 1000
```

### 3.2 RabbitMQ 交换机和队列

```java
@Configuration
public class RabbitMQConfig {

    // 直连交换机
    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order.exchange");
    }

    // 延迟队列
    @Bean
    public Queue delayQueue() {
        return QueueBuilder.durable("order.delay.queue")
            .withArgument("x-message-ttl", 30000)
            .withArgument("x-dead-letter-exchange",
                "order.exchange")
            .withArgument("x-dead-letter-routing-key",
                "order.timeout")
            .build();
    }

    // 死信队列
    @Bean
    public Queue deadLetterQueue() {
        return QueueBuilder.durable("order.dlq").build();
    }

    @Bean
    public Binding binding(Queue orderQueue,
                            DirectExchange orderExchange) {
        return BindingBuilder
            .bind(orderQueue)
            .to(orderExchange)
            .with("order.created");
    }
}
```

## 四、RocketMQ 事务消息

### 4.1 事务消息生产者

```java
@Service
public class RocketMQTransactionProducer {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    /**
     * 发送事务消息
     */
    public void sendTransactionMessage(Order order) {
        Message<Order> message = MessageBuilder
            .withPayload(order)
            .setHeader("KEYS", order.getId().toString())
            .build();

        rocketMQTemplate.sendMessageInTransaction(
            "order-tx-topic",
            message,
            order  // arg 传递给本地事务回调
        );
    }
}

@RocketMQTransactionListener
public class OrderTransactionListener
        implements RocketMQLocalTransactionListener {

    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(
            Message msg, Object arg) {
        try {
            Order order = (Order) arg;
            // 执行本地事务
            orderRepository.save(order);
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.UNKNOWN;
        }
    }

    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(
            Message msg) {
        // 回查本地事务状态
        String orderId = msg.getHeaders()
            .get("KEYS", String.class);
        Order order = orderRepository.findById(orderId);
        if (order != null) {
            return RocketMQLocalTransactionState.COMMIT;
        }
        return RocketMQLocalTransactionState.ROLLBACK;
    }
}
```

## 六、最佳实践

1. **异步通信解耦服务**，减少服务间直接依赖
2. **消息要保证幂等消费**，消费者可能收到重复消息
3. **消息格式要统一**，使用统一的事件格式
4. **消费延迟要监控**，积压告警
5. **死信队列处理失败消息**，不要让消息丢失
6. **使用事务消息保证一致性**，业务操作和消息发送原子性
7. **设置合理的消费者并发数**，根据处理能力调整
8. **消息保留时间要合理设置**，平衡存储和回溯需求
