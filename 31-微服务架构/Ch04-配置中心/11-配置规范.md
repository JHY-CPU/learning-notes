# 配置规范

## 一、配置命名规范

```yaml
# 命名规范
格式: {服务名}-{环境}.{扩展名}
示例:
  - user-service-dev.yaml
  - order-service-prod.yaml
  - common.yaml (共享配置)
```

## 二、配置分类

| 类型 | 说明 | 示例 |
|------|------|------|
| 业务配置 | 功能开关、业务参数 | feature.enabled |
| 技术配置 | 数据源、缓存 | datasource.url |
| 环境配置 | 环境特定配置 | server.port |
| 敏感配置 | 密钥、密码 | password |

## 三、工作原理

配置规范的核心是通过统一的命名规则、分类标准和管理流程，使配置在团队协作中保持可维护性。命名规范确保配置可按服务和环境自动归属，分类标准将业务配置与技术配置分离，避免互相干扰。共享配置提取为公共 Namespace，减少重复定义。敏感配置通过加密标记和独立存储确保安全性。

## 四、优缺点

**优点：**
- 规范化降低团队协作成本
- 配置分类清晰，排查问题更高效
- 共享配置减少维护负担

**缺点：**
- 规范执行依赖团队纪律
- 初期建立规范有一定学习成本
- 过于严格的规范可能降低开发效率

## 五、最佳实践

1. 使用 `{服务名}-{环境}.{格式}` 命名，如 `user-service-prod.yaml`
2. 业务配置和技术配置分离到不同 Namespace
3. 所有配置必须有默认值，避免配置缺失导致启动失败
4. 配置变更记录变更原因和负责人

## 六、配置分层管理

```
配置分层结构:
├── 应用级配置（所有服务共享）
│   └── application-common.yaml
│       - 日志配置
│       - 监控配置
│       - 通用超时配置
├── 服务级配置（单个服务）
│   └── {service-name}.yaml
│       - 业务开关
│       - 服务特有参数
├── 环境级配置（按环境区分）
│   └── {service-name}-{profile}.yaml
│       - 数据源地址
│       - 中间件地址
└── 敏感配置（加密存储）
    └── {service-name}-secret.yaml
        - 密码
        - 密钥
```

```yaml
# 配置文件示例 - user-service-prod.yaml
app:
  name: user-service
  version: 1.0.0

# 业务配置
business:
  feature:
    sms-enabled: true
    email-verification: true
  cache:
    user-ttl: 3600
    session-ttl: 1800

# 技术配置
spring:
  datasource:
    url: jdbc:mysql://prod-mysql:3306/user_db
    hikari:
      maximum-pool-size: 50
      minimum-idle: 10
  redis:
    cluster:
      nodes:
        - redis-node1:6379
        - redis-node2:6379
        - redis-node3:6379

# 日志配置
logging:
  level:
    root: INFO
    com.example: DEBUG
  file:
    name: /var/log/user-service/app.log
```

## 七、注意事项

1. **配置命名随意** - 无法通过名称判断归属服务和环境
2. **敏感配置未标记** - 被意外暴露在日志或前端
3. **配置无默认值** - 配置中心不可用时服务启动失败
4. **配置变更无记录** - 出问题后无法追溯
5. **配置格式不统一** - 团队内应该统一使用 YAML 格式
6. **配置值没有校验** - Spring Boot 使用 @Validated 校验配置格式
