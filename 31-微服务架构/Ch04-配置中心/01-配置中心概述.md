# 配置中心概述

## 一、为什么需要配置中心

```yaml
传统方式的问题:
  - 配置分散在各服务，难以管理
  - 修改配置需要重启服务
  - 多环境配置维护困难
  - 敏感信息泄露风险

配置中心解决:
  - 统一管理所有配置
  - 动态推送配置变更
  - 多环境隔离
  - 敏感信息加密
```

## 二、主流配置中心

| 组件 | 特点 |
|------|------|
| Nacos | 注册+配置一体，阿里开源 |
| Apollo | 携程开源，功能全面 |
| Consul | HashiCorp，KV 存储 |
| Spring Cloud Config | Git 存储，简单 |

## 三、工作原理

配置中心的核心是集中存储 + 实时推送。服务启动时从配置中心拉取配置并缓存到本地，运行期间通过长轮询或 WebSocket 监听配置变更。配置中心收到变更后推送给所有订阅的服务实例，服务收到通知后重新加载配置。为保证高可用，配置中心通常支持本地缓存降级：即使配置中心不可用，服务也能使用本地缓存的配置继续运行。

## 四、优缺点

**优点：**
- 统一管理避免配置散落在各服务代码库中
- 动态推送实现不重启服务即可更新配置
- 多环境隔离防止配置串环境

**缺点：**
- 引入额外组件，增加架构复杂度和运维成本
- 配置中心本身需要高可用保障
- 配置变更可能引发线上问题，需要审批流程

## 五、配置中心工作流程

```
配置加载流程:
1. 服务启动 → 从配置中心拉取配置
2. 配置合并（本地默认值 + 远程配置）
3. 注入 Spring Context
4. 监听配置变更（长轮询/WebSocket）

配置变更流程:
1. 管理员在控制台修改配置
2. 配置中心保存到数据库
3. 推送变更通知到所有订阅实例
4. 客户端收到通知 → 重新拉取配置
5. 触发 @RefreshScope Bean 刷新
```

## 六、Spring Cloud 配置集成

```java
// @RefreshScope - 配置动态刷新
@RestController
@RefreshScope
public class ConfigController {
    @Value("${app.feature.enabled:false}")
    private boolean featureEnabled;

    @GetMapping("/config/feature")
    public boolean isFeatureEnabled() {
        return featureEnabled;
    }
}

// 编程式获取配置
@Component
public class DynamicConfigService {
    @Autowired
    private ConfigurableApplicationContext context;

    public String getConfig(String key) {
        return context.getEnvironment().getProperty(key);
    }
}
```

## 七、注意事项

1. **配置中心宕机导致服务无法启动** - 需启用本地缓存降级
2. **敏感配置明文存储** - 存在泄露风险，必须加密
3. **配置变更没有审批** - 随意修改引发故障，需要审批流程
4. **多环境 namespace 混用** - 导致测试配置覆盖生产，严格隔离
5. **配置变更有延迟** - 长轮询模式有 30 秒延迟，WebSocket 模式实时
6. **配置格式要统一** - 推荐 YAML 格式，层次清晰
