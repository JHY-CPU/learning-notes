# 配置监控

## 一、监控目标

### 1.1 为什么需要配置监控

配置是微服务运行的核心依赖，配置问题可能导致：
- 服务启动失败
- 业务逻辑异常
- 性能下降
- 安全漏洞

配置监控的目的是**及时发现配置变更带来的问题**，快速定位和修复。

### 1.2 监控分层

```
┌─────────────────────────────────────────┐
│            业务层监控                      │
│  配置变更对业务指标的影响（错误率、延迟）     │
├─────────────────────────────────────────┤
│            应用层监控                      │
│  配置读取延迟、刷新成功率、监听状态          │
├─────────────────────────────────────────┤
│            平台层监控                      │
│  配置中心健康状态、存储容量、API QPS         │
└─────────────────────────────────────────┘
```

## 二、关键监控指标

### 2.1 配置变更指标

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| 配置变更频率 | 单位时间内的配置修改次数 | >10次/分钟 |
| 变更成功率 | 成功推送到所有实例的比例 | <99% |
| 变更传播延迟 | 从发布到客户端生效的时间 | >30秒 |
| 配置回滚次数 | 回滚操作的频率 | >3次/天 |

### 2.2 配置读取指标

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| 配置读取QPS | 配置读取请求量 | 异常突增 |
| 配置读取延迟 | 从请求到返回的耗时 | P99 > 100ms |
| 缓存命中率 | 本地缓存命中比例 | <80% |
| 配置读取失败率 | 读取失败的比例 | >1% |

### 2.3 配置中心健康指标

| 指标 | 说明 | 告警阈值 |
|------|------|---------|
| 节点可用性 | 配置中心节点存活状态 | <2个节点 |
| 存储容量 | 配置存储使用率 | >80% |
| 连接数 | 客户端连接数 | 突增50% |
| CPU/内存 | 配置中心服务器资源 | >80% |

## 三、监控实现方案

### 3.1 配置变更事件采集

```java
// Spring Cloud Config 变更监听
@Component
public class ConfigChangeMonitor implements ApplicationListener<EnvironmentChangeEvent> {

    private static final Logger log = LoggerFactory.getLogger(ConfigChangeMonitor.class);

    @Autowired
    private MeterRegistry meterRegistry;

    @Override
    public void onApplicationEvent(EnvironmentChangeEvent event) {
        Set<String> keys = event.getKeys();
        log.info("配置变更: {}", keys);

        // 记录变更次数
        meterRegistry.counter("config.change.total").increment(keys.size());

        // 记录变更详情
        for (String key : keys) {
            meterRegistry.counter("config.change.key",
                "key", key).increment();

            // 发送变更事件到监控系统
            sendChangeEvent(key, event);
        }
    }
}
```

### 3.2 配置读取延迟监控

```java
// AOP 切面监控配置读取
@Aspect
@Component
public class ConfigReadMonitor {

    @Autowired
    private MeterRegistry meterRegistry;

    @Around("@annotation(configValue)")
    public Object monitor(ProceedingJoinPoint point, ConfigValue configValue) throws Throwable {
        long start = System.nanoTime();
        try {
            Object result = point.proceed();
            long duration = System.nanoTime() - start;

            Timer.builder("config.read.duration")
                .tag("key", configValue.key())
                .tag("status", "success")
                .register(meterRegistry)
                .record(duration, TimeUnit.NANOSECONDS);

            return result;
        } catch (Exception e) {
            long duration = System.nanoTime() - start;

            Timer.builder("config.read.duration")
                .tag("key", configValue.key())
                .tag("status", "error")
                .register(meterRegistry)
                .record(duration, TimeUnit.NANOSECONDS);

            meterRegistry.counter("config.read.error",
                "key", configValue.key()).increment();

            throw e;
        }
    }
}
```

### 3.3 Prometheus + Grafana 监控

**Prometheus 指标定义：**

```yaml
# prometheus-rules.yaml
groups:
  - name: config_monitor
    rules:
      # 配置变更频率过高
      - alert: ConfigChangeFrequent
        expr: rate(config_change_total[5m]) > 0.2
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "配置变更频率过高"

      # 配置读取失败率过高
      - alert: ConfigReadErrorHigh
        expr: rate(config_read_error_total[5m]) / rate(config_read_total[5m]) > 0.01
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "配置读取失败率超过1%"

      # 配置推送延迟过高
      - alert: ConfigPushLatencyHigh
        expr: histogram_quantile(0.99, rate(config_push_duration_seconds_bucket[5m])) > 30
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "配置推送P99延迟超过30秒"
```

**Grafana 仪表盘配置：**

```json
{
```

## 四、告警策略

### 4.1 告警分级

| 级别 | 条件 | 响应时间 | 处理方式 |
|------|------|---------|---------|
| P0-紧急 | 配置中心不可用 | 5分钟 | 电话通知 |
| P1-严重 | 配置推送失败 > 50% | 15分钟 | 短信通知 |
| P2-警告 | 读取延迟 > 阈值 | 30分钟 | 钉钉/飞书 |
| P3-信息 | 变更频率异常 | 4小时 | 邮件 |

### 4.2 告警收敛

```yaml
# Alertmanager 配置

        severity: critical
```

## 五、配置变更追踪

### 5.1 变更审计日志

```sql
CREATE TABLE config_audit_log (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    config_key VARCHAR(255) NOT NULL,
    old_value TEXT,
    new_value TEXT,
    changed_by VARCHAR(100),
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    environment VARCHAR(50),
    service_name VARCHAR(100),
    status ENUM('SUCCESS', 'FAILURE', 'ROLLBACK'),
    INDEX idx_key_time (config_key, changed_at)
);
```

### 5.2 变更影响分析

```python
def analyze_change_impact(config_key, new_value, old_value):
    """分析配置变更的影响范围"""
    # 查找使用该配置的所有服务
    affected_services = find_services_using_config(config_key)

    # 检查变更是否是破坏性的
    is_breaking = is_breaking_change(config_key, new_value, old_value)

    # 获取相关告警
    related_alerts = get_recent_alerts(affected_services, hours=1)

    return {
        "affected_services": affected_services,
        "is_breaking": is_breaking,
        "related_alerts": related_alerts,
        "rollback_available": True,
    }
```

## 六、最佳实践

1. **监控全覆盖** — 变更、读取、推送、健康四个维度
2. **告警分级** — 避免告警疲劳，P0必须立即响应
3. **变更审计** — 记录所有变更，支持回溯
4. **关联分析** — 配置变更与业务指标的关联
5. **自动化响应** — 异常时自动回滚或降级
