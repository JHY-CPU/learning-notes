# 多环境配置管理

## 一、核心概念

在微服务架构中，通常需要维护多套环境（开发 dev、测试 test、预发布 staging、生产 prod），每套环境的配置各不相同。多环境配置管理需要保证环境隔离、配置继承、安全管控。

### 1.1 环境划分

```
环境        用途              配置特点
──────────────────────────────────────────
dev       开发调试           本地数据库、低安全要求
test      集成测试           测试数据、Mock 服务
staging   预发布验证         接近生产配置
prod      生产运行           高安全、高可用
```

### 1.2 配置层次结构

```
公共配置 (application.yml)
  ├── 环境配置 (application-{env}.yml)
  │     ├── dev
  │     ├── test
  │     ├── staging
  │     └── prod
  └── 服务配置 ({service}-{env}.yml)
        ├── user-service-dev.yml
        ├── user-service-prod.yml
        ├── order-service-dev.yml
        └── order-service-prod.yml
```

## 二、基于 Nacos 的多环境管理

### 2.1 命名空间隔离

```yaml
spring:
  profiles:
    active: ${ENV:dev}
  cloud:
    nacos:
      config:
        server-addr: nacos-cluster:8848
        # 使用命名空间隔离环境
        namespace: ${NACOS_NAMESPACE:dev-namespace-id}
        file-extension: yaml
        group: DEFAULT_GROUP
        # 公共配置
        shared-configs:
          - data-id: common.yaml
            group: DEFAULT_GROUP
            refresh: true
        # 扩展配置
        extension-configs:
          - data-id: datasource.yaml
            group: DEFAULT_GROUP
            refresh: true
```

### 2.2 Nacos 命名空间结构

```
Nacos
├── public (保留命名空间)
├── dev-namespace    (开发环境)
│   ├── user-service.yaml
│   ├── order-service.yaml
│   └── common.yaml
├── test-namespace   (测试环境)
│   ├── user-service.yaml
│   ├── order-service.yaml
│   └── common.yaml
├── staging-namespace (预发布)
│   └── ...
└── prod-namespace   (生产环境)
    ├── user-service.yaml
    ├── order-service.yaml
    └── common.yaml
```

### 2.3 多环境配置示例

```yaml
# common.yaml (公共配置)
app:
  name: microservice-platform
  version: 1.0.0

logging:
  level:
    root: INFO
    com.example: DEBUG

# common-dev.yaml (开发环境覆盖)
app:
  name: microservice-platform
logging:
  level:
    root: DEBUG
    com.example: DEBUG

# common-prod.yaml (生产环境覆盖)
app:
  name: microservice-platform
logging:
  level:
    root: WARN
    com.example: INFO
```

## 三、基于 Apollo 的多环境管理

### 3.1 Apollo 环境配置

```yaml
# application.yml
apollo:
  meta: ${APOLLO_META:http://apollo-config:8080}
  bootstrap:
    enabled: true
    namespaces: application,datasource,redis
    eagerLoad:
      enabled: true
```

### 3.2 Apollo 环境切换

```java
@Component
public class ApolloEnvConfig {

    /**
     * 通过环境变量切换 Apollo 环境
     */
    @Bean
    public ApolloEnvInfo apolloEnv() {
        String env = System.getProperty("env",
            "DEV").toUpperCase();
        return ApolloEnvInfo.builder()
            .env(Env.valueOf(env))
            .meta(getMetaUrl(env))
            .cluster("default")
            .build();
    }

    private String getMetaUrl(String env) {
        return switch (env) {
            case "DEV" -> "http://apollo-dev:8080";
            case "TEST" -> "http://apollo-test:8080";
            case "STAGING" -> "http://apollo-staging:8080";
            case "PROD" -> "http://apollo-prod:8080";
            default -> "http://apollo-dev:8080";
        };
    }
}
```

## 四、配置继承与覆盖

### 4.1 配置优先级

```
优先级从高到低:
  1. 服务-specific-prod.yaml  (服务+环境)
  2. service-prod.yaml        (服务+环境)
  3. service.yaml             (服务级别)
  4. application-prod.yaml    (应用+环境)
  5. application.yaml         (应用级别)
```

### 4.2 Spring Profile 机制

```yaml
# application.yaml (基础配置)
spring:
  application:
    name: user-service

---
# application-dev.yaml
spring:
  datasource:
    url: jdbc:mysql://dev-db:3306/user_service
    username: dev_user
    password: dev_pass

---
# application-prod.yaml
spring:
  datasource:
    url: jdbc:mysql://prod-db-cluster:3306/user_service
    username: ${DB_USER}
    password: ${DB_PASSWORD}
```

## 五、环境安全管控

### 5.1 配置分级管理

```
安全等级          管控措施
─────────────────────────────────
L1 公开配置     所有人可读可改
L2 内部配置     开发可读，运维可改
L3 敏感配置     运维可读，审批后可改
L4 核心密钥     加密存储，专人管理
```

### 5.2 生产环境配置管控

```java
@Configuration
public class ProdConfigProtection {

    /**
     * 生产环境配置变更需额外验证
     */
    public void validateProdChange(ConfigChange change) {
        if ("prod".equals(getCurrentEnv())) {
            // 检查变更人权限
            if (!hasPermission(change.getOperator(),
                    "PROD_CONFIG_CHANGE")) {
                throw new AccessDeniedException(
                    "无权修改生产配置");
            }
            // 检查是否走审批流程
            if (!change.isApproved()) {
                throw new IllegalStateException(
                    "生产配置变更需要审批");
            }
            // 记录审计日志
            auditLogService.log("PROD_CONFIG_CHANGE", change);
        }
    }
}
```

## 六、最佳实践

1. **使用命名空间/环境隔离**不同环境的配置
2. **配置文件按环境分组**，公共配置与环境配置分离
3. **生产配置严格管控**，变更需要审批和灰度
4. **环境切换要方便**，通过环境变量控制
5. **配置差异要最小化**，只覆盖必要的差异项
6. **敏感配置使用环境变量或加密**，不要硬编码
7. **开发环境可以宽松**，但生产环境必须严格
8. **定期同步各环境配置**，检查差异和不一致
