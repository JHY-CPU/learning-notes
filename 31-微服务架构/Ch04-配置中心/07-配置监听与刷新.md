# 配置热更新与监听刷新

## 一、核心概念

配置热更新是指在不重启服务的情况下，让配置变更实时生效。这是配置中心的核心能力之一，可以大幅缩短配置变更的生效时间，降低变更风险。

### 1.1 配置热更新机制

```
配置变更流程:
  用户修改配置 → 配置中心存储 → 推送/通知客户端
    → 客户端接收变更 → 刷新配置 → 生效（无需重启）

推送方式对比:
  长轮询: 客户端定期拉取，延迟较高但实现简单
  推送: 服务端主动推送，实时性好但实现复杂
  混合: 长轮询 + 推送结合，兼顾实时性和可靠性
```

### 1.2 架构流程

```
┌──────────┐     ┌──────────────┐     ┌──────────┐
│ 用户操作  │────►│ 配置中心      │────►│ 客户端    │
│ 修改配置  │     │ (Nacos/Apollo)│     │ (服务实例) │
└──────────┘     └──────┬───────┘     └────┬─────┘
                        │                  │
                        │ 1.保存配置        │
                        │ 2.推送通知        │
                        │              ┌───▼────┐
                        │              │监听器   │
                        │              │接收变更 │
                        │              └───┬────┘
                        │              ┌───▼────┐
                        │              │刷新Bean │
                        │              │更新上下文│
                        │              └────────┘
```

## 二、Nacos 配置监听

### 2.1 注解方式

```java
@Component
@RefreshScope
public class DynamicConfig {

    @Value("${app.feature.enabled:false}")
    private boolean featureEnabled;

    @Value("${app.max-connections:100}")
    private int maxConnections;

    @Value("${app.timeout:3000}")
    private long timeout;

    // getter 方法
    public boolean isFeatureEnabled() {
        return featureEnabled;
    }
}
```

### 2.2 监听器方式

```java
@Component
public class ConfigChangeListener {

    @NacosConfigListener(
        dataId = "user-service-config",
        group = "DEFAULT_GROUP"
    )
    public void onConfigChange(String newConfig) {
        log.info("配置变更: {}", newConfig);

        // 解析新配置
        UserConfig config =
            YAML.loadAs(newConfig, UserConfig.class);

        // 重新初始化相关组件
        reinitConnectionPool(config.getMaxConnections());
        reinitThreadPool(config.getThreadPoolSize());
    }

    @NacosConfigListener(
        dataId = "feature-flags",
        group = "DEFAULT_GROUP",
        timeout = 500  // 500ms 长轮询超时
    )
    public void onFeatureChange(String config) {
        Map<String, Boolean> flags =
            JSON.parseObject(config,
                new TypeReference<>() {});
        FeatureFlags.refresh(flags);
        log.info("特性开关更新: {}", flags);
    }
}
```

### 2.3 编程式监听

```java
@Component
public class CustomConfigListener implements Listener {

    @Autowired
    private ConfigService configService;

    @PostConstruct
    public void init() throws NacosException {
        configService.addListener(
            "app-config", "DEFAULT_GROUP", this);
    }

    @PreDestroy
    public void destroy() throws NacosException {
        configService.removeListener(
            "app-config", "DEFAULT_GROUP", this);
    }

    @Override
    public void receiveConfigInfo(String config) {
        log.info("收到配置变更: {}", config);
        processConfig(config);
    }

    @Override
    public Executor getExecutor() {
        return Executors.newSingleThreadExecutor();
    }
}
```

## 三、Apollo 配置监听

### 3.1 Spring 集成

```java
@Component
public class ApolloConfigListener {

    @ApolloConfigChangeListener(
        interestedKeyPrefixes = {"app.", "feature."}
    )
    public void onChange(ConfigChangeEvent changeEvent) {
        for (String key : changeEvent.changedKeys()) {
            ConfigChange change = changeEvent.getChange(key);
            log.info("配置变更: key={}, old={}, new={}",
                key, change.getOldValue(),
                change.getNewValue());
        }
    }
}
```

### 3.2 配置变更处理

```java
@Configuration
public class DynamicDataSourceConfig {

    @ApolloConfigChangeListener("datasource")
    public void onDataSourceChange(
            ConfigChangeEvent event) {
        if (event.isChanged("spring.datasource.url")
            || event.isChanged("spring.datasource.username")) {
            // 重建数据源
            rebuildDataSource();
        }
    }

    private void rebuildDataSource() {
        HikariDataSource old = dataSource;
        dataSource = createDataSource();
        // 优雅切换
        old.close();
    }
}
```

## 四、Spring Cloud RefreshScope

### 4.1 @RefreshScope 原理

```java
// @RefreshScope 的 Bean 会在配置变更时被销毁并重建
@Component
@RefreshScope
public class RefreshableService {

    @Value("${service.retry-count:3}")
    private int retryCount;

    @Value("${service.timeout:5000}")
    private long timeout;

    // 配置变更时，这个 Bean 会被重新创建
    // retryCount 和 timeout 会更新为新值
}
```

### 4.2 手动触发刷新

```java
@RestController
@RequestMapping("/admin/config")
public class ConfigRefreshController {

    @Autowired
    private ContextRefresher contextRefresher;

    @PostMapping("/refresh")
    public Set<String> refreshConfig() {
        // 触发所有 @RefreshScope Bean 刷新
        Set<String> refreshedKeys =
            contextRefresher.refresh();
        log.info("刷新的配置项: {}", refreshedKeys);
        return refreshedKeys;
    }
}
```

## 五、线程安全处理

### 5.1 配置变更的线程安全

```java
@Component
public class ThreadSafeConfig {

    private volatile AppConfig config;

    @NacosConfigListener(dataId = "app-config")
    public void onConfigChange(String newConfig) {
        // 使用 volatile 保证可见性
        this.config = JSON.parseObject(
            newConfig, AppConfig.class);
        log.info("配置更新完成");
    }

    public AppConfig getConfig() {
        return config;
    }
}
```

### 5.2 批量变更合并处理

```java
@Component
public class BatchConfigProcessor {

    private final BlockingQueue<ConfigChange> changeQueue =
        new LinkedBlockingQueue<>();

    @PostConstruct
    public void init() {
        // 启动后台线程批量处理配置变更
        new Thread(() -> {
            while (true) {
                List<ConfigChange> batch = new ArrayList<>();
                changeQueue.drainTo(batch, 100);
                if (!batch.isEmpty()) {
                    processBatch(batch);
                }
                Thread.sleep(1000); // 合并 1 秒内的变更
            }
        }).start();
    }

    public void onConfigChange(ConfigChange change) {
        changeQueue.offer(change);
    }
}
```

## 六、最佳实践

1. **配置变更要实时生效**，使用监听机制而非定时拉取
2. **监听器要做好异常处理**，避免异常导致后续变更丢失
3. **配置刷新要考虑线程安全**，使用 volatile 或原子操作
4. **批量变更要合并处理**，减少频繁刷新的开销
5. **配置变更要有日志记录**，便于追踪和审计
6. **敏感配置变更需要人工确认**，不要自动生效
7. **热更新要做好灰度验证**，先在部分实例生效
8. **监控配置刷新的延迟**，确保变更及时生效
