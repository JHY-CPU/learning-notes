# 配置权限管理

## 一、核心概念

配置权限管理是指对配置的查看、修改、发布、审批等操作进行访问控制，确保配置变更的安全性和合规性。生产环境的配置变更往往影响重大，必须有严格的权限管控。

### 1.1 权限模型

```
RBAC 权限模型:
  用户(User) → 角色(Role) → 权限(Permission) → 资源(Resource)

角色划分:
  ┌──────────┬──────────────────────────────┐
  │ 角色      │ 权限范围                      │
  ├──────────┼──────────────────────────────┤
  │ 开发者    │ 读 dev/test 配置              │
  │ 测试人员  │ 读 test 配置                  │
  │ 运维人员  │ 读写所有环境配置               │
  │ 架构师    │ 审批配置变更                   │
  │ 管理员    │ 全部权限 + 用户管理            │
  └──────────┴──────────────────────────────┘
```

### 1.2 权限粒度

```
资源维度:
  - 命名空间级别（Namespace）
  - 分组级别（Group/DataId）
  - 配置项级别（Key）

操作维度:
  - 读取（Read）
  - 修改（Write）
  - 发布（Publish）
  - 回滚（Rollback）
  - 删除（Delete）
  - 授权（Grant）
```

## 二、权限控制实现

### 2.1 数据库权限表设计

```sql
-- 用户表
CREATE TABLE config_user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(64) NOT NULL UNIQUE,
    password VARCHAR(256) NOT NULL,
    email VARCHAR(128),
    status TINYINT DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 角色表
CREATE TABLE config_role (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    role_name VARCHAR(64) NOT NULL UNIQUE,
    description VARCHAR(256)
);

-- 权限表
CREATE TABLE config_permission (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    resource_type ENUM('NAMESPACE','GROUP','DATA_ID') NOT NULL,
    resource_id VARCHAR(256) NOT NULL,
    action ENUM('READ','WRITE','PUBLISH',
                'ROLLBACK','DELETE') NOT NULL,
    UNIQUE KEY uk_resource (
        resource_type, resource_id, action)
);

-- 角色-权限关联
CREATE TABLE role_permission (
    role_id BIGINT NOT NULL,
    permission_id BIGINT NOT NULL,
    PRIMARY KEY (role_id, permission_id)
);

-- 用户-角色关联
CREATE TABLE user_role (
    user_id BIGINT NOT NULL,
    role_id BIGINT NOT NULL,
    PRIMARY KEY (user_id, role_id)
);
```

### 2.2 权限检查拦截器

```java
@Component
@Aspect
public class ConfigPermissionAspect {

    @Autowired
    private PermissionService permissionService;

    @Around("@annotation(requirePermission)")
    public Object checkPermission(
            ProceedingJoinPoint joinPoint,
            RequirePermission requirePermission) throws Throwable {
        String currentUser = SecurityUtils.getCurrentUser();
        String resourceId = getResourceId(joinPoint);
        String action = requirePermission.action();

        if (!permissionService.hasPermission(
                currentUser, resourceId, action)) {
            throw new AccessDeniedException(
                "用户 " + currentUser
                + " 无权对 " + resourceId
                + " 执行 " + action);
        }

        return joinPoint.proceed();
    }
}

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequirePermission {
    String action();
}
```

### 2.3 权限服务

```java
@Service
public class PermissionService {

    @Autowired
    private PermissionMapper permissionMapper;

    public boolean hasPermission(String username,
                                  String resourceId,
                                  String action) {
        // 获取用户的所有角色
        List<String> roles =
            permissionMapper.getUserRoles(username);

        // 获取角色对应的所有权限
        List<ConfigPermission> permissions =
            permissionMapper.getPermissionsByRoles(roles);

        // 检查是否有匹配的权限
        return permissions.stream().anyMatch(p ->
            matchResource(p.getResourceId(), resourceId)
            && p.getAction().equals(action)
        );
    }

    private boolean matchResource(
            String pattern, String resourceId) {
        // 支持通配符匹配
        // user-service/* 匹配 user-service/db
        // * 匹配所有
        return "*".equals(pattern)
            || pattern.equals(resourceId)
            || matchWildcard(pattern, resourceId);
    }
}
```

## 三、配置变更审批流程

### 3.1 审批流程设计

```
提交变更 → 审批人审核 → 通过/拒绝
  │                      │
  │                 ┌────┴────┐
  │                 ▼         ▼
  │              发布配置   拒绝通知
  │                 │
  │                 ▼
  │            灰度验证
  │                 │
  │                 ▼
  │            全量发布
```

### 3.2 审批服务实现

```java
@Service
public class ConfigApprovalService {

    @Autowired
    private ApprovalMapper approvalMapper;

    /**
     * 提交配置变更审批
     */
    public Long submitApproval(ConfigChangeRequest request) {
        ConfigApproval approval = new ConfigApproval();
        approval.setDataId(request.getDataId());
        approval.setGroup(request.getGroup());
        approval.setNewContent(request.getContent());
        approval.setOldContent(
            getCurrentConfig(request));
        approval.setApplicant(request.getApplicant());
        approval.setReason(request.getReason());
        approval.setStatus(ApprovalStatus.PENDING);
        approvalMapper.insert(approval);

        // 通知审批人
        notifyApprovers(approval);

        return approval.getId();
    }

    /**
     * 审批通过
     */
    public void approve(Long approvalId, String approver) {
        ConfigApproval approval =
            approvalMapper.selectById(approvalId);
        approval.setStatus(ApprovalStatus.APPROVED);
        approval.setApprover(approver);
        approvalMapper.updateById(approval);

        // 发布配置
        publishConfig(approval);
    }

    /**
     * 审批拒绝
     */
    public void reject(Long approvalId, String approver,
                        String reason) {
        ConfigApproval approval =
            approvalMapper.selectById(approvalId);
        approval.setStatus(ApprovalStatus.REJECTED);
        approval.setApprover(approver);
        approval.setRejectReason(reason);
        approvalMapper.updateById(approval);

        // 通知申请人
        notifyApplicant(approval);
    }
}
```

@Aspect
@Component
public class AuditLogAspect {

    @Autowired
    private AuditLogMapper auditLogMapper;

    @AfterReturning("@annotation(auditLog)")
    public void logOperation(JoinPoint joinPoint,
                              AuditLog auditLog) {
        ConfigAuditLog log = new ConfigAuditLog();
        log.setDataId(extractDataId(joinPoint));
        log.setAction(auditLog.action());
        log.setOperator(SecurityUtils.getCurrentUser());
        log.setIpAddress(
            RequestUtils.getClientIp());
        log.setDetail(
            JSON.toJSONString(joinPoint.getArgs()));
        auditLogMapper.insert(log);
    }
}

## 五、敏感配置保护

### 5.1 配置加密

```java
@Component
public class ConfigEncryption {

    private static final String ALGORITHM = "AES/GCM/NoPadding";

    /**
     * 加密敏感配置
     */
    public String encrypt(String plainText) {
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.ENCRYPT_MODE, getSecretKey());
        byte[] encrypted = cipher.doFinal(
            plainText.getBytes(StandardCharsets.UTF_8));
        return Base64.getEncoder().encodeToString(encrypted);
    }

    /**
     * 解密配置
     */
    public String decrypt(String encryptedText) {
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.DECRYPT_MODE, getSecretKey());
        byte[] decrypted = cipher.doFinal(
            Base64.getDecoder().decode(encryptedText));
        return new String(decrypted, StandardCharsets.UTF_8);
    }
}
```

### 5.2 敏感配置脱敏

```java
@Component
public class ConfigMasking {

    private static final Set<String> SENSITIVE_KEYS =
        Set.of("password", "secret", "token",
               "api-key", "private-key");

    public String mask(String key, String value) {
        if (SENSITIVE_KEYS.stream()
                .anyMatch(key::toLowerCase::contains)) {
            return "******";
        }
        return value;
    }
}
```

## 六、最佳实践

1. **生产配置修改需要审批**，走变更流程
2. **权限按环境分级**，dev 宽松，prod 严格
3. **操作要审计记录**，所有变更可追溯
4. **敏感配置单独授权**，加密存储
5. **定期审查权限分配**，回收不需要的权限
6. **使用最小权限原则**，只授予必要的权限
7. **密码和密钥使用加密存储**，不要明文
8. **配置变更通知相关团队**，保持信息透明
