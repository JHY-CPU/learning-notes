# 微服务架构概述

## 一、什么是微服务

微服务是一种**将应用拆分为多个小型、自治服务**的架构风格。每个服务独立开发、部署、运维，通过轻量级通信机制协作。

```
单体架构:                    微服务架构:
┌─────────────────┐         ┌──────────┐ ┌──────────┐ ┌──────────┐
│  用户模块        │         │ 用户服务  │ │ 订单服务  │ │ 支付服务  │
│  订单模块        │   →     │ 用户库   │ │ 订单库   │ │ 支付库   │
│  支付模块        │         └──────────┘ └──────────┘ └──────────┘
│  通知模块        │              ↕            ↕            ↕
│  共享数据库      │         ┌─────────────────────────────────────┐
└─────────────────┘         │         API Gateway / MQ            │
                            └─────────────────────────────────────┘
```

## 二、微服务的优缺点

### 优点

| 优势 | 说明 |
|------|------|
| 独立部署 | 单个服务可独立发布，不影响其他服务 |
| 技术异构 | 不同服务可以用不同技术栈 |
| 弹性伸缩 | 按需扩缩单个服务 |
| 故障隔离 | 一个服务故障不影响整体 |
| 团队自治 | 小团队负责小服务 |

### 缺点

| 挑战 | 说明 |
|------|------|
| 分布式复杂性 | 网络延迟、数据一致性、分布式事务 |
| 运维复杂度 | 需要容器编排、服务发现、配置中心 |
| 测试难度 | 集成测试需要多个服务协同 |
| 调试困难 | 需要链路追踪定位问题 |

## 三、适用场景

```java
// 适合微服务的场景
// 1. 大型团队（多团队并行开发）
// 2. 复杂业务（不同模块演进速度不同）
// 3. 高可用要求（需要独立伸缩和故障隔离）
// 4. 快速迭代（频繁发布新功能）

// 不适合微服务的场景
// 1. 小型应用（简单 CRUD）
// 2. 小团队（1-5 人）
// 3. 业务不明确（还在快速验证阶段）
```

## 四、微服务核心组件

```
微服务生态:
├── API 网关         - 统一入口、路由、限流
├── 服务注册发现     - Nacos/Eureka/Consul
├── 配置中心         - Nacos/Apollo/Consul
├── 服务通信         - HTTP/RPC/MQ
├── 熔断降级         - Sentinel/Hystrix/Resilience4j
├── 链路追踪         - SkyWalking/Zipkin/Jaeger
├── 日志收集         - ELK/Loki
├── 容器编排         - Kubernetes
└── CI/CD           - Jenkins/GitLab CI
```

## 五、注意事项

1. **微服务不是银弹**，不要为了微服务而微服务
2. **先做好模块化**，再考虑拆分为微服务
3. **基础设施先行**，没有完善的运维体系不建议上微服务
4. **从单体演进**，不要一开始就全拆
5. **团队规模是重要考量**，小团队不适合微服务
