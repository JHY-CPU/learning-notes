# 微服务粒度

## 一、粒度评估

```yaml
过细的信号:
  - 服务间调用链过长（>5层）
  - 频繁需要同时修改多个服务
  - 单个服务只有一两个 API
  - 分布式事务过多

过粗的信号:
  - 服务功能过多（>10个模块）
  - 团队协作频繁冲突
  - 部署频率低（数周一次）
  - 故障影响范围大
```

## 二、粒度参考

| 服务类型 | API 数量 | 团队规模 | 代码量 |
|---------|---------|---------|--------|
| 小型服务 | 5-15 | 1-2 人 | <10K |
| 中型服务 | 15-50 | 2-5 人 | 10-50K |
| 大型服务 | 50+ | 5-10 人 | 50-100K |

## 三、演进策略

```
1. 从单体开始
2. 模块化组织代码
3. 识别变化频率高的模块
4. 将高频变化模块拆为独立服务
5. 持续观察，按需调整
```

## 四、注意事项

1. **粒度要与团队能力匹配**
2. **过细比过粗更糟糕**
3. **拆分要有业务意义**，不是技术驱动
4. **先做好模块化**，再考虑微服务
5. **持续演进**，不要一步到位
