# 微服务粒度

## 一、粒度评估

```yaml
过细的信号:
  - 服务间调用链过长（>5层）
  - 频繁需要同时修改多个服务
  - 单个服务只有一两个 API
  - 分布式事务过多

过粗的信号:
  - 服务功能过多（>10个模块）
  - 团队协作频繁冲突
  - 部署频率低（数周一次）
  - 故障影响范围大
```

## 二、粒度参考

| 服务类型 | API 数量 | 团队规模 | 代码量 |
|---------|---------|---------|--------|
| 小型服务 | 5-15 | 1-2 人 | <10K |
| 中型服务 | 15-50 | 2-5 人 | 10-50K |
| 大型服务 | 50+ | 5-10 人 | 50-100K |

## 三、演进策略

```
1. 从单体开始
2. 模块化组织代码
3. 识别变化频率高的模块
4. 将高频变化模块拆为独立服务
5. 持续观察，按需调整
```

## 四、康威定律与粒度

```
康威定律: 系统设计反映了组织的沟通结构

团队结构          →  服务结构
┌──────────┐        ┌──────────┐
│ 用户团队  │  →    │ 用户服务  │
│ 订单团队  │  →    │ 订单服务  │
│ 商品团队  │  →    │ 商品服务  │
└──────────┘        └──────────┘

反模式: 一个团队维护 10+ 个微服务 → 运维负担过重
正模式: 一个团队维护 1-3 个微服务 → 两个披萨原则
```

## 五、粒度调整决策树

```
服务是否需要独立部署？
├── 是 → 该模块变更频率高吗？
│   ├── 高 → 拆为独立服务 ✓
│   └── 低 → 保持为模块
└── 否 → 服务间是否存在循环依赖？
    ├── 是 → 先解耦依赖，再考虑拆分
    └── 否 → 该模块有独立的扩展需求吗？
        ├── 是 → 拆为独立服务 ✓
        └── 否 → 保持为模块
```

## 六、合并服务的信号

```yaml
需要合并的信号:
  - 两个服务总是一起部署和变更
  - 服务间有大量同步调用
  - 一个服务的修改经常需要同时修改另一个
  - 某个服务只有 1-2 个 API
  - 两个服务共享大量代码

合并策略:
  - 数据库层面先合并（最困难的一步）
  - API 层面保持兼容，逐步迁移
  - 合并后统一测试覆盖
```

## 七、注意事项

1. **粒度要与团队能力匹配** - 团队能力不够时，粗粒度比细粒度好
2. **过细比过粗更糟糕** - 分布式系统的复杂度会远超预期
3. **拆分要有业务意义** - 按业务领域拆分，不要按技术层拆分
4. **先做好模块化** - 在单体内做好模块化是微服务的前提
5. **持续演进** - 微服务粒度不是一次定死的，根据实际情况持续调整
