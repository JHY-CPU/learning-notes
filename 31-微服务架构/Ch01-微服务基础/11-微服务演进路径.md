# 微服务演进路径

## 一、演进路线

```
单体应用 → 模块化单体 → 服务化 → 微服务 → 云原生

阶段1: 单体架构（小团队）
  └── 所有功能在一个应用中

阶段2: 模块化单体
  └── 按业务模块组织代码，共享数据库

阶段3: 服务化
  └── 核心模块拆为独立服务，独立数据库

阶段4: 微服务
  └── 全面拆分，API Gateway + 服务网格

阶段5: 云原生
  └── Kubernetes + Service Mesh + Serverless
```

## 二、绞杀者模式

```java
// 逐步替换单体，不一次性重写
// 1. 新功能用新服务实现
// 2. 旧功能逐步迁移到新服务
// 3. 路由层将请求分配到新旧服务
// 4. 迁移完成后下线单体
```

## 三、各阶段详细对比

| 维度 | 单体 | 模块化单体 | 微服务 | 云原生 |
|------|------|-----------|--------|--------|
| 部署 | 整体部署 | 整体部署 | 独立部署 | 自动化部署 |
| 扩展 | 整体扩展 | 整体扩展 | 按需扩展 | 弹性伸缩 |
| 团队 | 单一团队 | 多模块团队 | 服务团队 | DevOps 团队 |
| 复杂度 | 低 | 中 | 高 | 很高 |
| 适用规模 | <10人 | 10-50人 | 50-200人 | 200+人 |

## 四、防腐层模式

```java
// 在新旧系统之间建立防腐层
@Component
public class LegacyAntiCorruptionLayer {
    // 将旧系统数据格式转换为新系统格式
    public OrderDTO toNewModel(LegacyOrder legacyOrder) {
        return OrderDTO.builder()
            .id(legacyOrder.getOrderId())
            .userId(legacyOrder.getCustomerId())
            .amount(new BigDecimal(legacyOrder.getTotalPrice()))
            .status(mapStatus(legacyOrder.getOrderState()))
            .build();
    }

    private OrderStatus mapStatus(String legacyStatus) {
        return switch (legacyStatus) {
            case "0" -> OrderStatus.CREATED;
            case "1" -> OrderStatus.PAID;
            case "2" -> OrderStatus.SHIPPED;
            default -> OrderStatus.UNKNOWN;
        };
    }
}
```

## 五、演进检查清单

```yaml
阶段1 → 阶段2 (模块化):
  - 代码是否按业务模块组织
  - 模块间接口是否清晰
  - 是否有模块间依赖图
  ✓ 通过后才能进入下一阶段

阶段2 → 阶段3 (服务化):
  - 核心模块是否可以独立部署
  - 数据库是否可以按模块拆分
  - 是否有服务间通信方案
  - 是否有基本的监控
  ✓ 通过后才能进入下一阶段

阶段3 → 阶段4 (微服务):
  - 是否有完整的 CI/CD
  - 是否有容器编排平台
  - 是否有服务治理能力
  - 团队是否具备运维能力
  ✓ 通过后才能进入下一阶段
```

## 六、注意事项

1. **不要一步到位** - 渐进式演进，每个阶段稳定后再进入下一阶段
2. **先解决痛点** - 优先拆分变化最频繁、扩展需求最大的模块
3. **保持旧系统可用** - 新旧并行运行，确保业务连续性
4. **数据迁移是最大的挑战** - 采用双写 + 校验 + 切换策略
5. **演进过程中要做好监控** - 每个阶段都要有监控能力
6. **设立明确的阶段检查点** - 不满足条件不进入下一阶段
