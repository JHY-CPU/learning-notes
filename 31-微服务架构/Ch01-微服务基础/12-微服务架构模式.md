# 微服务架构模式

## 一、Saga 模式

```java
// 编排式 Saga - 通过事件驱动
public class OrderSaga {
    // 正向流程
    public void execute(Order order) {
        inventoryService.reserve(order);  // Step 1
        accountService.deduct(order);     // Step 2
        orderService.create(order);       // Step 3
    }

    // 补偿流程
    public void compensate(Order order, int failedStep) {
        switch (failedStep) {
            case 3: accountService.refund(order);
            case 2: inventoryService.release(order);
        }
    }
}
```

```
Command (写) → 写模型 → 事件 → 读模型 → Query (读)
```

## 四、模式选择

| 模式 | 适用场景 |
|------|---------|
| Saga | 分布式事务 |
| CQRS | 读写分离 |
| Event Sourcing | 审计、回溯 |
| API Gateway | 统一入口 |

## 五、Sidecar 模式

```
┌─────────────────────────────┐
│           Pod               │
│  ┌──────────┐ ┌──────────┐ │
│  │ 业务容器  │ │ Sidecar  │ │
│  │          │ │ (Envoy)  │ │
│  └──────────┘ └──────────┘ │
│      ↕            ↕        │
│   Unix Socket / localhost   │
└─────────────────────────────┘

Sidecar 职责:
├── 服务发现与负载均衡
├── 熔断与限流
├── 链路追踪注入
├── mTLS 加密通信
└── 访问日志收集
```

## 六、BFF（Backend For Frontend）模式

```
┌──────────┐  ┌──────────┐
│  Web端   │  │ 移动端   │
└────┬─────┘  └────┬─────┘
     │             │
┌────▼─────┐  ┌────▼─────┐
│ Web BFF  │  │ App BFF  │
│ GraphQL  │  │ 精简API  │
└────┬─────┘  └────┬─────┘
     │             │
     └──────┬──────┘
            │
     ┌──────▼──────┐
     │  微服务集群  │
     └─────────────┘
```

```java
// BFF - 聚合多个微服务的响应
@RestController
@RequestMapping("/api/app")
public class AppBffController {
    @Autowired private UserClient userClient;
    @Autowired private OrderClient orderClient;
    @Autowired private ProductClient productClient;

    @GetMapping("/home")
    public AppHomePage getHomePage(@RequestHeader("userId") Long userId) {
        // 并行调用多个微服务
        CompletableFuture<User> userFuture = CompletableFuture
            .supplyAsync(() -> userClient.getUser(userId));
        CompletableFuture<List<Order>> ordersFuture = CompletableFuture
            .supplyAsync(() -> orderClient.getRecentOrders(userId));
        CompletableFuture<List<Product>> recommendFuture = CompletableFuture
            .supplyAsync(() -> productClient.getRecommendations(userId));

        CompletableFuture.allOf(userFuture, ordersFuture, recommendFuture).join();

        return AppHomePage.builder()
            .user(userFuture.get())
            .recentOrders(ordersFuture.get())
            .recommendations(recommendFuture.get())
            .build();
    }
}
```

## 七、架构模式选择指南

| 模式 | 适用场景 | 复杂度 | 注意事项 |
|------|---------|--------|---------|
| Saga | 跨服务事务 | 高 | 需要补偿逻辑 |
| CQRS | 读写比例差异大 | 中 | 数据最终一致 |
| Event Sourcing | 审计/回溯需求 | 高 | 事件版本管理 |
| Sidecar | 服务网格/非侵入 | 低 | 资源开销 |
| BFF | 多端适配 | 中 | 避免过度聚合 |
| API Gateway | 统一入口 | 低 | 注意性能瓶颈 |

## 八、注意事项

1. **模式不是银弹** - 根据具体业务场景选择合适的模式
2. **Saga 适合长时间事务** - 短事务优先考虑 TCC
3. **CQRS 增加复杂度** - 读写比例 < 10:1 时不建议引入
4. **Event Sourcing 适合需要审计的场景** - 金融、医疗等合规领域
5. **组合使用多种模式** - 复杂系统通常需要多种模式配合
6. **引入新模式要有明确收益** - 不要为了使用模式而使用模式
