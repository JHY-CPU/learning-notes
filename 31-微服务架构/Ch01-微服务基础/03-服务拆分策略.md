# 服务拆分策略

## 一、DDD 领域驱动拆分

```java
// 按领域划分限界上下文
// 电商领域拆分:
// 用户上下文: 用户注册、登录、个人信息
// 商品上下文: 商品管理、库存
// 订单上下文: 订单创建、支付、物流
// 营销上下文: 优惠券、促销活动

// 每个上下文是一个微服务
@Service("orderService")
public class OrderService {
    // 只处理订单领域逻辑
    public Order createOrder(CreateOrderCommand cmd);
}
```

## 二、按业务拆分

```
拆分粒度参考:
├── 大型服务: 5-10 人团队负责
├── 中型服务: 2-5 人团队负责
└── 小型服务: 1-2 人团队负责

过细的问题:
- 服务间通信成本增加
- 分布式事务复杂度增加
- 运维成本增加

过粗的问题:
- 部署频率受限
- 团队协作冲突
- 故障爆炸半径大
```

## 三、拆分步骤

```
1. 识别业务边界（领域建模）
2. 定义服务接口（API 设计）
3. 数据库拆分（数据所有权）
4. 渐进式迁移（绞杀者模式）
```

## 四、按数据拆分

```java
// 以数据实体为中心进行拆分
// 用户表（users, user_profiles, user_addresses）→ 用户服务
// 订单表（orders, order_items, order_logs）→ 订单服务
// 商品表（products, categories, brands）→ 商品服务

// 判断原则：谁拥有这张表，谁就负责这个服务
// 跨服务数据查询通过 API 组合，不直接 JOIN
```

## 五、绞杀者模式（Strangler Fig Pattern）

```
单体应用                         新微服务
┌──────────────┐                 ┌──────────┐
│  旧订单模块   │  ← 渐进迁移 →  │ 订单服务  │
│  旧用户模块   │  ← 渐进迁移 →  │ 用户服务  │
│  旧商品模块   │                 └──────────┘
└──────────────┘
        │
   API Gateway 路由
   /orders/* → 新订单服务
   /users/*  → 旧单体（待迁移）
```

```java
// 路由层实现
@Component
public class StranglerRouter {
    @Value("${migration.order-service.enabled:false}")
    private boolean orderServiceEnabled;

    public String route(String path) {
        if (path.startsWith("/orders") && orderServiceEnabled) {
            return "http://order-service";
        }
        return "http://legacy-monolith";
    }
}
```

## 六、拆分决策矩阵

| 考虑维度 | 权重 | 评估方法 |
|---------|------|---------|
| 业务独立性 | 高 | 是否有独立的业务闭环 |
| 变更频率 | 高 | 该模块是否频繁变更 |
| 团队边界 | 中 | 是否对应独立团队 |
| 数据耦合度 | 高 | 与其他模块的数据依赖程度 |
| 技术异构 | 低 | 是否需要不同的技术栈 |
| 扩展需求 | 中 | 是否需要独立扩展 |

## 七、实际案例：电商平台拆分

```
拆分前（单体）:
  ┌─────────────────────────────┐
  │  E-commerce Monolith        │
  │  用户|商品|订单|支付|营销|物流  │
  │  共享 MySQL 数据库           │
  └─────────────────────────────┘

拆分后（微服务）:
  ┌────────┐  ┌────────┐  ┌────────┐
  │ 用户   │  │ 商品   │  │ 订单   │
  │ MySQL  │  │ Mongo  │  │ MySQL  │
  └────────┘  └────────┘  └────────┘
  ┌────────┐  ┌────────┐  ┌────────┐
  │ 支付   │  │ 营销   │  │ 物流   │
  │ MySQL  │  │ Redis  │  │ ES     │
  └────────┘  └────────┘  └────────┘
```

## 八、注意事项

1. **先做好模块化再拆微服务** - 没有模块化的单体直接拆微服务会导致混乱
2. **数据库拆分是最大的挑战** - 跨库 JOIN、分布式事务都需要重新设计
3. **渐进式迁移，不要大爆炸式重构** - 使用绞杀者模式逐步替换
4. **拆分粒度要与团队能力匹配** - 两个披萨团队原则
5. **拆分后要做好集成测试** - 接口契约测试必须自动化
6. **拆分要以业务为核心** - 避免按技术层拆分（如数据服务、缓存服务）
