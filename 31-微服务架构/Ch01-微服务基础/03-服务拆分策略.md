# 服务拆分策略

## 一、DDD 领域驱动拆分

```java
// 按领域划分限界上下文
// 电商领域拆分:
// 用户上下文: 用户注册、登录、个人信息
// 商品上下文: 商品管理、库存
// 订单上下文: 订单创建、支付、物流
// 营销上下文: 优惠券、促销活动

// 每个上下文是一个微服务
@Service("orderService")
public class OrderService {
    // 只处理订单领域逻辑
    public Order createOrder(CreateOrderCommand cmd);
}
```

## 二、按业务拆分

```
拆分粒度参考:
├── 大型服务: 5-10 人团队负责
├── 中型服务: 2-5 人团队负责
└── 小型服务: 1-2 人团队负责

过细的问题:
- 服务间通信成本增加
- 分布式事务复杂度增加
- 运维成本增加

过粗的问题:
- 部署频率受限
- 团队协作冲突
- 故障爆炸半径大
```

## 三、拆分步骤

```
1. 识别业务边界（领域建模）
2. 定义服务接口（API 设计）
3. 数据库拆分（数据所有权）
4. 渐进式迁移（绞杀者模式）
```

## 四、注意事项

1. **先做好模块化再拆微服务**
2. **数据库拆分是最大的挑战**
3. **渐进式迁移，不要大爆炸式重构**
4. **拆分粒度要与团队能力匹配**
5. **拆分后要做好集成测试**
