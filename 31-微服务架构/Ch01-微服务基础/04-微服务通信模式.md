# 微服务通信模式

## 一、同步通信

```java
// REST API (HTTP)
@FeignClient(name = "user-service")
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable Long id);
}

// gRPC (高性能)
service UserService {
    rpc GetUser(GetUserRequest) returns (User);
}
```

## 二、异步通信

```java
// 消息队列
kafkaTemplate.send("order-events", order);
@KafkaListener(topics = "order-events")
public void onOrderCreated(Order order) { }

// 事件驱动
eventBus.publish(new OrderCreatedEvent(order));
```

## 三、通信模式对比

| 模式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 同步 REST | 简单直观 | 强耦合 | 查询、简单操作 |
| gRPC | 高性能 | 复杂 | 服务间内部调用 |
| 异步 MQ | 解耦 | 最终一致 | 事件通知 |
| 事件驱动 | 完全解耦 | 复杂 | CQRS/EDA |

## 四、gRPC 详细示例

```protobuf
// user.proto
syntax = "proto3";

service UserService {
    rpc GetUser (GetUserRequest) returns (User);
    rpc ListUsers (ListUsersRequest) returns (stream User);
}

message GetUserRequest {
    int64 id = 1;
}

message User {
    int64 id = 1;
    string name = 2;
    string email = 3;
}
```

```java
// gRPC Server 实现
@GrpcService
public class UserServiceImpl extends UserServiceGrpc.UserServiceImplBase {
    @Override
    public void getUser(GetUserRequest request, StreamObserver<User> responseObserver) {
        User user = userRepository.findById(request.getId());
        responseObserver.onNext(user);
        responseObserver.onCompleted();
    }
}

// gRPC Client 调用
ManagedChannel channel = ManagedChannelBuilder
    .forAddress("user-service", 9090)
    .usePlaintext()
    .build();
UserServiceGrpc.UserServiceBlockingStub stub = UserServiceGrpc.newBlockingStub(channel);
User user = stub.getUser(GetUserRequest.newBuilder().setId(1L).build());
```

## 五、异步消息深度示例

```java
// 生产者：发送事件
@Component
public class OrderEventPublisher {
    @Autowired
    private KafkaTemplate<String, OrderEvent> kafkaTemplate;

    @Transactional
    public void publishOrderCreated(Order order) {
        OrderEvent event = OrderEvent.builder()
            .type("ORDER_CREATED")
            .orderId(order.getId())
            .timestamp(System.currentTimeMillis())
            .payload(order)
            .build();
        kafkaTemplate.send("order-events", order.getId().toString(), event);
    }
}

// 消费者：处理事件
@Component
public class InventoryEventListener {
    @KafkaListener(topics = "order-events", groupId = "inventory-service")
    public void handleOrderEvent(OrderEvent event) {
        if ("ORDER_CREATED".equals(event.getType())) {
            inventoryService.reserve(event.getPayload().getItems());
        }
    }
}
```

## 六、服务调用链路设计

```
客户端请求
    │
    ▼
┌─────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ API     │ ──→ │ 订单服务  │ ──→ │ 商品服务  │     │ 支付服务  │
│ Gateway │     │ (HTTP)   │     │ (gRPC)   │     │ (异步MQ)  │
└─────────┘     └──────────┘     └──────────┘     └──────────┘
                     │                                  ▲
                     │          ┌──────────┐            │
                     └────────→ │ 用户服务  │ ──────────┘
                                │ (gRPC)   │  支付回调
                                └──────────┘

注意：
- 核心同步调用链不超过 3 层
- 非关键路径用异步解耦
- 每层设置合理超时
```

## 七、序列化格式对比

| 格式 | 体积 | 速度 | 可读性 | 适用场景 |
|------|------|------|--------|---------|
| JSON | 大 | 慢 | 好 | REST API、配置文件 |
| Protobuf | 小 | 快 | 差 | gRPC、内部高性能通信 |
| Avro | 中 | 中 | 差 | Kafka 消息序列化 |
| Thrift | 小 | 快 | 差 | 跨语言 RPC 调用 |

## 八、注意事项

1. **强依赖用同步，弱依赖用异步** - 查询类用同步，事件通知用异步
2. **同步调用要做好超时和重试** - 每个调用都可能失败
3. **异步消息要保证幂等** - 消息可能重复消费
4. **避免同步调用链过长** - 超过 3 层要考虑异步化
5. **选择合适的序列化格式** - 内部用 Protobuf，外部用 JSON
6. **做好服务降级方案** - 下游服务不可用时要有兜底策略
