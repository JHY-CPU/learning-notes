# 微服务挑战

## 一、分布式复杂性

```
挑战:
├── 网络不可靠 - 超时、重试、幂等
├── 数据一致性 - 最终一致性、Saga
├── 服务发现 - 动态地址管理
├── 负载均衡 - 多实例分配
├── 配置管理 - 多环境配置
└── 故障传播 - 熔断、降级
```

## 二、运维挑战

```yaml
运维复杂度:
  - 需要容器编排（K8s）
  - 需要服务网格（Istio）
  - 需要监控告警（Prometheus）
  - 需要日志聚合（ELK）
  - 需要链路追踪（SkyWalking）
  - 需要 CI/CD 流水线
```

## 三、应对策略

| 挑战 | 应对方案 |
|------|---------|
| 网络不可靠 | 超时、重试、熔断 |
| 数据一致 | 最终一致性、补偿 |
| 服务发现 | Nacos/Eureka |
| 配置管理 | Nacos/Apollo |
| 故障传播 | 熔断降级 |
| 链路追踪 | SkyWalking |

## 四、分布式调试难度

```
单体调试:  问题 → 看日志 → 定位 → 修复（分钟级）

微服务调试: 问题 → 搜索多个服务日志 → 用 TraceID 关联
          → 重现 → 跨服务分析 → 修复（小时级）
```

```java
// 使用 MDC 实现日志链路追踪
@Component
public class TraceFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain) {
        String traceId = request.getHeader("X-Trace-Id");
        if (traceId == null) traceId = UUID.randomUUID().toString().replace("-", "");
        MDC.put("traceId", traceId);
        MDC.put("spanId", generateSpanId());
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}
```

## 五、技术债与团队挑战

| 挑战 | 表现 | 解决方案 |
|------|------|---------|
| 学习曲线陡峭 | 团队生产力短期下降 | 分阶段引入，先核心后外围 |
| 技术栈碎片化 | 每个服务用不同框架 | 建立技术雷达，统一标准 |
| 文档缺失 | 新人无法理解系统 | API 文档自动化（Swagger） |
| 人员流动 | 知识流失 | 代码评审 + 结对编程 |
| 过度工程 | 引入不需要的组件 | YAGNI 原则，按需引入 |

## 六、成本考量

```
微服务额外成本:
├── 基础设施成本
│   ├── 服务器/云资源（多实例）
│   ├── 容器编排平台（K8s 集群）
│   ├── 注册中心/配置中心
│   └── 监控告警平台
├── 运维成本
│   ├── CI/CD 流水线维护
│   ├── 多环境管理
│   └── 故障排查时间增加
└── 开发成本
    ├── 服务间通信开发
    ├── 分布式事务处理
    └── 接口版本管理

建议：日活 < 10万 的系统，模块化单体可能更合适
```

## 七、注意事项

1. **基础设施先行**，没有完善的运维体系不要上微服务
2. **从简单开始**，不要一开始就过度设计，先拆 2-3 个核心服务
3. **自动化是关键**，手动操作在微服务中不可持续
4. **团队能力要匹配**，运维能力不足会成为瓶颈
5. **监控告警要完善**，没有监控就是在裸奔
6. **评估是否真的需要微服务**，模块化单体对大多数团队够用
