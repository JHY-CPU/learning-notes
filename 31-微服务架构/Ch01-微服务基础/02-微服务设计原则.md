# 微服务设计原则

## 一、单一职责原则

每个服务只负责**一个业务能力**，修改一个服务的理由应该只有一个。

```java
// 好的设计 - 单一职责
// 用户服务 - 只负责用户管理
@Service
public class UserService {
    public User createUser(UserDTO dto);
    public User getUser(Long id);
    public void updateProfile(Long id, ProfileDTO dto);
}

// 订单服务 - 只负责订单管理
@Service
public class OrderService {
    public Order createOrder(OrderDTO dto);
    public Order getOrder(Long id);
    public void cancelOrder(Long id);
}

// 坏的设计 - 职责混乱
// 一个服务负责用户、订单、支付、通知
```

## 二、自治原则

```java
// 每个服务自治，拥有独立的数据库
// 用户服务 -> 用户数据库 (MySQL)
// 订单服务 -> 订单数据库 (MySQL)
// 商品服务 -> 商品数据库 (MongoDB)
// 搜索服务 -> 搜索引擎 (Elasticsearch)

// 服务间通过 API 通信，不直接访问其他服务的数据库
@RestController
public class OrderController {
    @Autowired
    private UserClient userClient;  // Feign 客户端调用用户服务

    @PostMapping("/orders")
    public Order createOrder(@RequestBody OrderDTO dto) {
        // 通过 API 获取用户信息，不直接查数据库
        User user = userClient.getUser(dto.getUserId());
        return orderService.create(dto, user);
    }
}
```

## 三、接口隔离

```java
// 每个服务暴露细粒度的 API
// 用户服务 API
@GetMapping("/users/{id}")
public User getUser(@PathVariable Long id);

@GetMapping("/users/{id}/profile")
public Profile getProfile(@PathVariable Long id);

// 避免粗粒度 API
@GetMapping("/users/{id}/everything")  // 不好 - 返回所有数据
```

## 四、容错设计

```java
// 服务调用要处理异常
@FeignClient(name = "user-service", fallbackFactory = UserFallbackFactory.class)
public interface UserClient {
    @GetMapping("/users/{id}")
    User getUser(@PathVariable Long id);
}

@Component
public class UserFallbackFactory implements FallbackFactory<UserClient> {
    @Override
    public UserClient create(Throwable cause) {
        return new UserClient() {
            @Override
            public User getUser(Long id) {
                // 降级逻辑
                return User.defaultUser(id);
            }
        };
    }
}
```

## 五、设计原则总结

| 原则 | 说明 |
|------|------|
| 单一职责 | 一个服务只做一件事 |
| 自治 | 独立部署、独立数据库 |
| 接口隔离 | 细粒度 API |
| 容错设计 | 降级、熔断、重试 |
| 去中心化 | 避免中心化瓶颈 |

## 六、注意事项

1. **不要过度拆分**，微服务太小会增加管理复杂度
2. **数据库隔离是关键**，共享数据库会破坏自治
3. **接口设计要考虑版本演进**
4. **容错设计要前置**，不要等到线上出问题
5. **团队结构要与服务划分匹配**（康威定律）
