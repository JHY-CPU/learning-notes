# 微服务数据管理

## 一、数据库拆分

```java
// 每个服务拥有独立数据库
// 用户服务 -> MySQL (用户库)
// 订单服务 -> MySQL (订单库)
// 商品服务 -> MongoDB (商品库)
// 缓存服务 -> Redis

// 服务间通过 API 查询数据，不跨库查询
```

## 二、数据一致性

```java
// 最终一致性方案
// 本地消息表 + MQ
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);
    messageMapper.insert(new Message("order.created", order));
}

// Saga 模式
// 正向: 扣库存 -> 扣余额 -> 创建订单
// 补偿: 恢复余额 -> 恢复库存
```

## 三、CQRS 模式

```java

// 读模型 - Elasticsearch/Redis

```

## 四、分布式查询方案

```java
// 方案1: API 组合查询
@Service
public class OrderDetailService {
    @Autowired private OrderClient orderClient;
    @Autowired private UserClient userClient;
    @Autowired private ProductClient productClient;

    public OrderDetail getOrderDetail(Long orderId) {
        Order order = orderClient.getOrder(orderId);
        User user = userClient.getUser(order.getUserId());
        List<Product> products = productClient.getProducts(order.getProductIds());
        return new OrderDetail(order, user, products);
    }
}

// 方案2: 数据同步到搜索服务
// 使用 Canal 监听 MySQL binlog，同步到 Elasticsearch
@Component
public class BinlogSyncHandler {
    public void handle(CanalMessage message) {
        if ("orders".equals(message.getTable())) {
            elasticSearchTemplate.index("orders", message.getData());
        }
    }
}
```

## 五、数据分片策略

```
按租户分片:
  租户A → DB-Shard-1
  租户B → DB-Shard-2
  租户C → DB-Shard-3

按范围分片:
  订单号 1-1000000 → Shard-1
  订单号 1000001-2000000 → Shard-2

按哈希分片:
  shard = hash(userId) % shardCount
```

```yaml
# ShardingSphere 配置
spring:
  shardingsphere:
    datasource:
      names: ds0,ds1
    sharding:
      tables:
        orders:
          actual-data-nodes: ds$->{0..1}.orders_$->{0..1}
          database-strategy:
            inline:
              sharding-column: user_id
              algorithm-expression: ds$->{user_id % 2}
```

## 六、缓存架构

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 客户端   │ → │ 本地缓存  │ → │ Redis    │ → │ 数据库   │
│          │    │ Caffeine │    │ 集群     │    │ MySQL    │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
     缓存穿透：布隆过滤器
     缓存雪崩：过期时间加随机值
     缓存击穿：互斥锁 + 逻辑过期
```

```java
// 多级缓存实现
@Service
public class CacheService {
    private LoadingCache<Long, User> localCache = Caffeine.newBuilder()
        .maximumSize(10000)
        .expireAfterWrite(5, TimeUnit.MINUTES)
        .build(id -> redisTemplate.opsForValue().get("user:" + id));

    public User getUser(Long id) {
        return localCache.get(id);  // 本地缓存未命中 → Redis → DB
    }
}
```

## 七、注意事项

1. **数据库拆分是微服务最大的挑战** - 需要仔细规划数据边界
2. **避免分布式事务**，优先用最终一致性 + 补偿机制
3. **数据所有权要明确**，一个数据只有一个服务拥有写权限
4. **跨服务查询用 API 组合或数据同步**，避免跨库 JOIN
5. **做好数据备份和恢复方案** - 定期备份 + 恢复演练
6. **缓存与数据库的一致性** - 采用延迟双删或 Canal 监听 binlog
7. **大数据量时考虑分库分表** - ShardingSphere 或 Vitess
