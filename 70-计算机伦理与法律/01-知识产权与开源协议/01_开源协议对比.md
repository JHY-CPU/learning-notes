# 开源协议对比


## 开源协议对比


伦理开源许可证


理解开源协议是每个开发者的基本素养。


## 协议分类


```
宽松型 (Permissive)：允许闭源使用
├── MIT：最宽松，只需保留版权声明
├── BSD 2/3-Clause：类似MIT，略有差异
├── Apache 2.0：含专利授权，需标注修改

Copyleft型：修改后必须开源
├── GPL v2/v3：衍生作品必须GPL开源
├── AGPL v3：网络服务也算衍生作品
├── LGPL v2.1：链接不算衍生作品
└── MPL 2.0：文件级Copyleft

选择指南：
- 商业闭源项目 → MIT / Apache 2.0
- 保护开源生态 → GPL v3
- 库/框架 → MIT / Apache 2.0 / LGPL
- SaaS服务 → AGPL v3（防止白嫖）
```


## 主要协议详解


```
┌──────────┬────────┬──────────┬──────────┬──────────┐
│          │ MIT    │ Apache2.0│ GPL v3   │ LGPL     │
├──────────┼────────┼──────────┼──────────┼──────────┤
│ 商业使用  │ 允许   │ 允许     │ 允许     │ 允许      │
│ 修改      │ 允许   │ 允许     │ 允许     │ 允许      │
│ 分发      │ 允许   │ 允许     │ 允许     │ 允许      │
│ 闭源使用  │ 允许   │ 允许     │ 禁止     │ 链接允许  │
│ 专利授权  │ 无     │ 有       │ 有       │ 无        │
│ 标注修改  │ 不需要 │ 需要     │ 需要     │ 需要      │
│ 相同协议  │ 不需要 │ 不需要   │ 必须     │ 库修改必须│
└──────────┴────────┴──────────┴──────────┴──────────┘

MIT（最流行）：
- 任何人可以无限制使用、复制、修改
- 唯一要求：保留版权声明
- Node.js, React, jQuery, Rails

Apache 2.0（企业友好）：
- 包含明确的专利授权
- 防止专利诉讼
- Android, Kubernetes, TensorFlow

GPL v3（自由软件）：
- 衍生作品必须开源且使用GPL
- "传染性"：用了GPL代码，整个项目需GPL
- Linux内核(GPL v2), GCC, Git

LGPL v2.1（库友好）：
- 修改库需要开源
- 仅链接使用不需要开源
- glibc, Qt(LGPL选项)
```


## 专利与合规


```
专利授权：
- MIT/BSD：无明确专利授权，有专利风险
- Apache 2.0：明确授予专利使用权
- GPL v3：授予专利权，且不能起诉用户

企业开源合规：
1. 识别项目中使用的开源组件（SBOM）
2. 检查各组件的许可证类型
3. 确保满足许可证要求
4. Copyleft组件需要考虑传染风险
5. 在产品中包含许可证声明

常见合规工具：
- FOSSA：自动许可证扫描
- Black Duck：软件组成分析
- Snyk：安全和许可证检查

Dual Licensing（双重许可）：
- 同时提供开源版(GPL)和商业版
- 不想开源的用户购买商业许可
- 例：MySQL, Qt, MongoDB(SSPL)
```


> **Note:** 选择项目许可证时，MIT是最安全的选择。如果有专利顾虑，选Apache 2.0。
>         想保护开源生态选GPL。企业使用开源组件务必做好合规检查。
