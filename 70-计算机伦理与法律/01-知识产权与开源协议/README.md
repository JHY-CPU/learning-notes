# 01-知识产权与开源协议

## 1. 知识产权概述

知识产权（Intellectual Property, IP）是指人类智力创造成果所享有的专有权利，在计算机软件领域尤为重要。

| 类型 | 保护对象 | 保护期限 | 登记要求 |
|------|----------|----------|----------|
| 著作权 | 代码表达形式 | 作者终身+50年 | 自动产生，登记可增强证据效力 |
| 专利权 | 技术方案/方法 | 20年（需缴费） | 需申请审批 |
| 商标权 | 品牌标识 | 10年（可续展） | 需注册 |
| 商业秘密 | 技术/经营秘密 | 无期限（保密前提下） | 无需登记，需采取保密措施 |

## 2. 软件著作权

### 2.1 软件著作权的保护范围

- **受保护**：源代码、目标代码、软件文档的**表达形式**
- **不受保护**：算法思想、数学概念、编程语言本身、操作方法
- 核心原则：**思想/表达二分法**（Idea/Expression Dichotomy）
- 软件界面、菜单布局等UI元素的保护存在争议

### 2.2 登记流程与法律效力

1. **准备材料**：源代码（前30页+后30页）、申请表、身份证明
2. **提交申请**：向中国版权保护中心提交
3. **审查**：形式审查（约60个工作日）
4. **颁发证书**：《计算机软件著作权登记证书》
5. **法律效力**：登记证书是权利的初步证明，有助于侵权诉讼中的举证

### 2.3 开源软件的著作权

- 开源不等于放弃著作权，作者依然享有著作权
- 开源协议是著作权人授予他人的**附条件使用许可**
- 违反开源协议 = 侵犯著作权（可追究侵权责任）
- 典型案例：Artifex v. Hancom（GPL协议的法律效力）

## 3. 软件专利

### 3.1 软件相关发明的可专利性

各国标准不同：

| 地区 | 可专利性 | 要求 |
|------|----------|------|
| 中国 | 需与硬件结合或构成技术方案 | 纯算法/商业方法不可专利 |
| 美国 | 较为宽松（Alice案后收紧） | 需超越"抽象概念" |
| 欧洲 | 需有"技术贡献" | 纯软件不可专利 |
| 日本 | 需利用自然规律 | 软件相关发明可专利 |

### 3.2 专利申请流程

1. **专利检索**：确认新颖性
2. **撰写申请文件**：权利要求书、说明书、摘要
3. **提交申请**：向国家知识产权局提交
4. **初步审查**：形式审查
5. **实质审查**：申请人需在3年内请求实审
6. **授权/驳回**：审查意见答复
7. **缴费授权**：缴纳年费维持专利权

### 3.3 方法专利 vs 产品专利

- **方法专利**：保护实现某功能的方法/流程，如"一种图像识别方法"
- **产品专利**：保护实现某功能的装置/系统，如"一种图像识别装置"
- 软件领域常见做法：方法权利要求 + 装置权利要求同时撰写
- 侵权判定：方法专利需证明对方使用了该方法；产品专利需证明对方制造/销售了该产品

## 4. 开源协议详解

### 4.1 GPL（v2/v3）

- **传染性**：强Copyleft，使用GPL代码的衍生作品必须同样以GPL发布
- **GPLv2 vs v3**：v3增加了反DRM条款、专利授权要求、兼容性改进
- **适用场景**：希望确保衍生作品永远开源
- **代表项目**：Linux内核（v2）、GCC、GNU工具链

### 4.2 LGPL

- **弱传染性**：允许以动态链接方式在闭源软件中使用
- 静态链接或修改LGPL代码则衍生部分需以LGPL发布
- **适用场景**：开源库希望被闭源项目使用
- **代表项目**：glibc、Qt（早期版本）

### 4.3 MIT协议

- **最宽松**：几乎无限制，只需保留版权声明
- 允许闭源、商用、修改、再分发
- 不提供专利授权和免责担保
- **适用场景**：希望最大程度推广使用
- **代表项目**：React、jQuery、Rails、Node.js

### 4.4 Apache 2.0

- 宽松许可，但提供**明确的专利授权**
- 包含专利报复条款：对使用者提起专利诉讼则自动终止许可
- 需保留版权和NOTICE文件
- **适用场景**：企业开源项目，需要专利保护
- **代表项目**：Android、Kubernetes、TensorFlow

### 4.5 BSD协议

- **2-Clause**：最简形式，仅需保留版权声明
- **3-Clause**：额外禁止使用项目名称做背书推广
- 宽松许可，允许闭源
- **适用场景**：学术项目、希望简单授权
- **代表项目**：FreeBSD、Go（3-Clause变体）

### 4.6 MPL 2.0

- **文件级Copyleft**：修改的文件需以MPL开源，但新文件可使用其他协议
- 介于GPL和MIT之间的中间地带
- 兼容GPL（可选择GPL二次许可）
- **适用场景**：希望保护核心代码但允许扩展闭源
- **代表项目**：Firefox、Thunderbird

### 4.7 Creative Commons协议族

- 主要用于**非软件作品**（文档、图片、数据集等）
- 组合元素：BY（署名）、SA（相同方式共享）、NC（非商业）、ND（禁止演绎）
- CC BY-SA：类似GPL的Copyleft
- CC0：完全放弃权利进入公有领域
- **注意**：CC协议不推荐用于软件

## 5. 开源协议的兼容性

协议兼容性指在同一个项目中组合使用不同协议代码的合法性：

| 组合 | 兼容性 | 说明 |
|------|--------|------|
| MIT + Apache 2.0 | 兼容 | 宽松协议间通常兼容 |
| MIT + GPLv2 | 兼容 | 宽松协议代码可被GPL项目使用 |
| Apache 2.0 + GPLv2 | **不兼容** | Apache 2.0的专利条款与GPLv2冲突 |
| Apache 2.0 + GPLv3 | 兼容 | GPLv3解决了专利条款问题 |
| GPLv2 + GPLv3 | 不直接兼容 | 需v2声明"or later version" |
| MPL 2.0 + GPLv2+ | 兼容 | MPL 2.0包含GPL兼容条款 |

**原则**：从宽松协议到严格协议是单向兼容的，严格协议代码不能用于宽松协议项目。

## 6. 开源合规管理

### 6.1 企业开源治理

- 建立**开源治理委员会**，制定开源使用政策
- 建立开源组件引入审批流程
- 定期审计项目依赖中的开源组件
- 制定开源贡献指南（对外贡献和内部开源）

### 6.2 SBOM（软件物料清单）

- SBOM是软件组件的完整清单，记录所有依赖及其版本和协议
- **格式标准**：SPDX（Linux基金会）、CycloneDX（OWASP）、SWID
- 美国行政令（2021）要求供应商提供SBOM
- 作用：漏洞追踪、合规审计、供应链安全

### 6.3 License扫描工具

| 工具 | 类型 | 说明 |
|------|------|------|
| FOSSology | 开源 | 全面的许可证扫描和合规分析 |
| ScanCode Toolkit | 开源 | 检测许可证、版权声明、依赖关系 |
| Black Duck | 商业 | 企业级SCA和合规管理 |
| FOSSA | 商业/免费 | 自动化许可证合规和漏洞检测 |
| LicenseFinder | 开源 | 检测项目依赖的许可证 |
| ClearlyDefined | 开源 | 收集开源组件的许可证元数据 |

**合规最佳实践**：
1. 在CI/CD流水线中集成许可证扫描
2. 设置许可证允许/禁止列表
3. 对copyleft协议的使用进行重点审查
4. 维护内部开源组件清单和协议映射表
5. 定期更新SBOM并进行安全审计
