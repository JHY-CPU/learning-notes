## 1. React DevTools Profiler

### 1.1 安装与开启

```bash
# Chrome/Firefox 扩展
# Chrome: https://chrome.google.com/webstore/detail/react-developer-tools
# Firefox: https://addons.mozilla.org/firefox/addon/react-devtools/

# 开发环境自动启用
# 生产环境需要额外配置：
# - 设置 __REACT_DEVTOOLS_GLOBAL_HOOK__
# - 使用 profiling build of React
```

### 1.2 Profiler 的使用

```
步骤：
1. 打开浏览器开发者工具
2. 切换到 "Profiler" 标签
3. 点击录制按钮（蓝色圆点）
4. 在页面上执行操作（点击、输入、滚动等）
5. 停止录制
6. 分析结果
```

### 1.3 Profiler 数据解读

```
Profiler 提供的信息：
  - 每次渲染的耗时
  - 哪些组件触发了重新渲染
  - 渲染的原因（props 变化、hooks 变化等）
  - 为什么某个组件的渲染花了这么长时间
  - 渲染提交到 DOM 的时间
```

## 2. Flamegraph 与 Ranked Chart

### 2.1 Flamegraph（火焰图）

```
Flamegraph 展示组件树的渲染时间层级结构：

横轴：渲染时间（宽度越大，渲染越慢）
纵轴：组件层级（从上到下，父组件到子组件）

颜色含义：
  - 灰色：没有重新渲染
  - 黄色到红色：渲染时间从短到长

解读方法：
  - 找到最宽的"条" → 这是最耗时的组件
  - 查看是否整棵子树都重新渲染了 → 如果是，说明父组件变化导致
  - 对比多次录制结果 → 确认优化效果
```

### 2.2 Ranked Chart（排名图）

```
Ranked Chart 按渲染时间排序展示组件：

  ComponentA   ████████████████  45ms
  ComponentB   ████████████      32ms
  ComponentC   ████████          22ms
  ComponentD   ████              12ms

用途：
  - 快速找到最耗时的组件
  - 从高到低逐步优化
  - 确认优化效果
```

### 2.3 高亮更新的组件

```
开启方式：
1. 打开 React DevTools
2. 勾选 "Highlight updates when components render"
3. 操作页面，观察哪些组件出现高亮边框

高亮说明：
  - 绿色边框闪烁 = 组件重新渲染了
  - 频繁闪烁 = 可能存在不必要的渲染
  - 大范围闪烁 = 状态提升过高或缺少 memo
```

## 3. why-did-you-render

### 3.1 安装

```bash
npm install @welldone-software/why-did-you-render
```

### 3.2 配置

```js
// src/wdyr.js - 必须在 React 导入之前加载
import React from "react";

if (process.env.NODE_ENV === "development") {
  const whyDidYouRender = await import(
    "@welldone-software/why-did-you-render"
  );
  whyDidYouRender.default(React, {
    trackAllPureComponents: true, // 跟踪所有 memo/PureComponent
    trackHooks: true, // 跟踪 Hook 变化
    logOwnerReasons: true, // 记录父组件导致重新渲染的原因
  });
}
```

```jsx
// src/index.js
import "./wdyr"; // 必须在最前面
import React from "react";
import App from "./App";
```

### 3.3 控制台输出解读

```js
// 控制台会输出类似信息：
// [WhyDidYouRender] MyComponent
// Props changes:
//   onClick: function () {} !== function () {}  ← 函数引用变化
//   style: { color: "red" } !== { color: "red" }  ← 对象引用变化
//   count: 1 === 1  ← 没有变化

// 根据输出修复：
// - onClick 用 useCallback 包裹
// - style 用 useMemo 包裹
// - count 没变，但因为其他 props 变了导致重新渲染
```

### 3.4 指定跟踪的组件

```jsx
// 只跟踪特定组件
function MyComponent() {
  return <div>Hello</div>;
}
MyComponent.whyDidYouRender = true;

// 类组件
class MyClassComponent extends React.Component {
  static whyDidYouRender = true;
  render() {
    return <div>Hello</div>;
  }
}
```

## 4. React.memo 陷阱与过早优化

### 4.1 过早优化的问题

```jsx
// ❌ 过早优化 - 每个组件都用 memo
const Header = memo(function Header({ title }) {
  return <h1>{title}</h1>; // 渲染成本极低，memo 的比较成本反而更高
});

const Footer = memo(function Footer({ text }) {
  return <footer>{text}</footer>; // 同上
});

const Button = memo(function Button({ onClick, children }) {
  return <button onClick={onClick}>{children}</button>;
});
// 问题：需要额外管理 onClick 的 useCallback，增加代码复杂度
```

### 4.2 何时应该使用 memo

```
使用 memo 的条件：
  1. 组件渲染频率高（父组件频繁重新渲染）
  2. 组件渲染成本高（复杂计算、大量 DOM 节点）
  3. Props 引用稳定（或能用 useMemo/useCallback 稳定）
  4. 实际有性能问题（通过 Profiler 验证）

不使用 memo 的情况：
  1. 组件渲染成本很低
  2. Props 经常变化
  3. 没有实际的性能问题
  4. 组件很少重新渲染
```

### 4.3 性能优化决策流程

```
发现性能问题
    ↓
使用 Profiler 定位瓶颈
    ↓
瓶颈在重新渲染？
    ↓ 是
分析原因（props 变化？状态位置不对？）
    ↓
选择优化策略：
  - 状态就近原则（state colocation）
  - 组件拆分
  - React.memo + useMemo/useCallback
  - 虚拟化
    ↓
再次使用 Profiler 验证效果
    ↓
问题解决？
  - 是 → 完成
  - 否 → 继续分析其他瓶颈
```

## 5. Chrome Performance Tab

### 5.1 录制性能

```
步骤：
1. 打开 Chrome DevTools → Performance 标签
2. 点击录制按钮或按 Ctrl+Shift+E
3. 执行要分析的操作
4. 停止录制
5. 分析火焰图

关键指标：
  - FPS (Frames Per Second)：帧率，60fps 为流畅
  - CPU 使用情况：不同颜色代表不同类型的活动
  - Main Thread：主线程活动
  - Network：网络请求
```

### 5.2 CPU 颜色含义

```
蓝色  → 加载 (Loading)：网络、HTML 解析
黄色  → 脚本 (Scripting)：JavaScript 执行
紫色  → 渲染 (Rendering)：样式计算、布局
绿色  → 绘制 (Painting)：绘制操作
灰色  → 空闲 (Idle)
```

### 5.3 定位 React 相关性能问题

```
在 Performance 中寻找：
1. 长任务（Long Tasks，>50ms）
   - 红色三角形标记
   - 展开查看调用栈

2. 频繁的 Layout
   - 紫色块
   - 可能由同步 DOM 读写导致

3. Forced Synchronous Layout
   - 在 JS 中读取布局属性（offsetHeight 等）后立即修改样式
   - 导致浏览器被迫同步计算布局

4. 大量的 GC（垃圾回收）
   - 橙色块
   - 说明有大量临时对象被创建和回收
```

## 6. Lighthouse

### 6.1 使用方式

```
方式一：Chrome DevTools
1. 打开 DevTools → Lighthouse 标签
2. 选择要分析的类别（Performance、Accessibility 等）
3. 选择设备（Mobile / Desktop）
4. 点击 "Analyze page load"

方式二：命令行
npx lighthouse https://example.com --output json --output-path report.json

方式三：CI/CD 集成
npx lighthouse-ci autorun
```

### 6.2 Lighthouse 评分指标

```
Performance（性能）：
  - FCP (First Contentful Paint)
  - LCP (Largest Contentful Paint)
  - TBT (Total Blocking Time)
  - CLS (Cumulative Layout Shift)
  - SI (Speed Index)

Accessibility（可访问性）：
  - 颜色对比度
  - ARIA 属性
  - 键盘可访问性

Best Practices（最佳实践）：
  - HTTPS
  - 图片尺寸
  - 控制台错误

SEO：
  - meta 标签
  - 可索引性
```

### 6.3 Lighthouse 常见优化建议

```
React 应用常见建议：
  - "Reduce unused JavaScript" → 代码分割、Tree Shaking
  - "Eliminate render-blocking resources" → 懒加载、async/defer
  - "Serve images in next-gen formats" → WebP/AVIF
  - "Reduce unused CSS" → CSS Modules、PurgeCSS
  - "Minimize main-thread work" → 优化 React 渲染
  - "Avoid enormous network payloads" → 压缩、CDN
```

## 7. Web Vitals 核心指标

### 7.1 Core Web Vitals

```
三大核心指标：

LCP (Largest Contentful Paint) - 最大内容绘制
  - 衡量加载性能
  - 目标：< 2.5 秒
  - 测量视口内最大元素的渲染时间

INP (Interaction to Next Paint) - 交互到下次绘制
  - 衡量交互响应性（2024年起替代 FID）
  - 目标：< 200 毫秒
  - 测量用户交互后页面更新的延迟

CLS (Cumulative Layout Shift) - 累积布局偏移
  - 衡量视觉稳定性
  - 目标：< 0.1
  - 测量页面元素意外移动的程度
```

### 7.2 在 React 中测量 Web Vitals

```bash
npm install web-vitals
```

```js
// src/reportWebVitals.js
import { onCLS, onINP, onLCP } from "web-vitals";

function sendToAnalytics(metric) {
  const body = JSON.stringify({
    name: metric.name,
    value: metric.value,
    rating: metric.rating, // "good" | "needs-improvement" | "poor"
    delta: metric.delta,
    id: metric.id,
  });

  // 发送到分析服务
  if (navigator.sendBeacon) {
    navigator.sendBeacon("/api/vitals", body);
  } else {
    fetch("/api/vitals", { body, method: "POST", keepalive: true });
  }
}

onLCP(sendToAnalytics);
onINP(sendToAnalytics);
onCLS(sendToAnalytics);
```

### 7.3 React 中改善 Core Web Vitals

```
改善 LCP：
  - 使用 Suspense 流式渲染
  - 代码分割，减少首屏 JS 体积
  - 服务端渲染关键内容
  - 预加载关键资源 <link rel="preload">

改善 INP：
  - 使用 startTransition 标记低优先级更新
  - 避免在事件处理器中做大量同步计算
  - 使用 Web Worker 处理复杂计算
  - 使用虚拟化减少 DOM 更新量

改善 CLS：
  - 图片和视频设置固定尺寸
  - 动态内容预留空间
  - 避免在已有内容上方插入新内容
  - 使用 CSS contain 属性
```

## 8. Bundle 分析工具

### 8.1 webpack-bundle-analyzer

```bash
npm install --save-dev webpack-bundle-analyzer
```

```js
// vite.config.js
import { visualizer } from "rollup-plugin-visualizer";

export default {
  plugins: [
    visualizer({
      filename: "dist/stats.html",
      open: true,
      gzipSize: true,
    }),
  ],
};
```

```js
// webpack.config.js
const BundleAnalyzerPlugin =
  require("webpack-bundle-analyzer").BundleAnalyzerPlugin;

module.exports = {
  plugins: [
    new BundleAnalyzerPlugin({
      analyzerMode: "static",
      reportFilename: "bundle-report.html",
    }),
  ],
};
```

### 8.2 source-map-explorer

```bash
npm install --save-dev source-map-explorer
```

```bash
# 分析构建产物
npx source-map-explorer dist/assets/*.js

# 生成可视化报告
npx source-map-explorer dist/assets/index-*.js --html report.html
```

### 8.3 Bundle 分析常见发现

```
常见大依赖：
  - moment.js → 替换为 dayjs（体积减少 97%）
  - lodash → 使用 lodash-es 或按需导入
  - @mui/material → 按需导入组件
  - chart.js → 考虑 lighter 的替代品

优化策略：
  - 代码分割 (Dynamic Import)
  - Tree Shaking
  - 替换大型依赖
  - 使用 ESM 版本的库
  - 压缩和 Gzip/Brotli
```

## 小结

- React DevTools Profiler 是分析 React 渲染性能的首选工具
- Flamegraph 帮助理解组件渲染的层级关系和耗时
- why-did-you-render 自动检测不必要的重新渲染
- Chrome Performance Tab 提供底层的浏览器性能数据
- Lighthouse 综合评估应用的性能、可访问性和最佳实践
- Web Vitals 是衡量用户体验的核心指标（LCP、INP、CLS）
- Bundle 分析工具帮助发现和减少包体积
- 优化前先用工具定位瓶颈，避免过早优化
