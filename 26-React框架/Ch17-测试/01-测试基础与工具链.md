# 测试基础与工具链

## 一、为什么要测试

### 1.1 测试的价值

```
没有测试的代码 = 没有保险的房子
修改任何代码都可能引发不可预期的连锁反应
```

**测试带来的核心收益：**

| 收益 | 说明 |
|------|------|
| **信心** | 重构和添加功能时不怕破坏现有逻辑 |
| **文档** | 测试用例即代码的使用文档 |
| **设计** | 可测试的代码通常是更好的设计 |
| **回归防护** | 防止已修复的 Bug 再次出现 |
| **协作** | 新成员通过测试理解系统行为 |

### 1.2 测试金字塔

```
        /  E2E  \          <- 少量：端到端测试
       /  集成测试  \        <- 适量：组件交互
      /   单元测试    \       <- 大量：函数/工具
```

```typescript
// 单元测试 - 测试纯函数，快速独立
function formatCurrency(amount: number): string {
  return `¥${amount.toFixed(2)}`
}

// 集成测试 - 测试组件交互
// 测试 "用户填写表单并提交" 的完整流程

// E2E测试 - 测试真实用户流程
// 在真实浏览器中测试 "用户注册->登录->下单" 全流程
```

**各层对比：**

| 维度 | 单元测试 | 集成测试 | E2E测试 |
|------|---------|---------|---------|
| 数量 | 多 (70%) | 中 (20%) | 少 (10%) |
| 速度 | 毫秒级 | 秒级 | 秒~分钟级 |
| 置信度 | 低 | 中 | 高 |
| 维护成本 | 低 | 中 | 高 |
| 调试难度 | 低 | 中 | 高 |

## 三、React Testing Library 哲学

### 3.1 核心理念：测试行为而非实现

```typescript
// ❌ 测试实现细节 - 脆弱、难维护
test('点击按钮计数增加', () => {
  const wrapper = shallow(<Counter />);
  expect(wrapper.state('count')).toBe(0);       // 测试内部 state
  wrapper.find('button').simulate('click');
  expect(wrapper.state('count')).toBe(1);       // 紧耦合实现
});

// ✅ 测试用户行为 - 稳定、可重构
test('点击按钮计数增加', () => {
  render(<Counter />);
  const button = screen.getByRole('button', { name: /increment/i });

  expect(screen.getByText('Count: 0')).toBeInTheDocument();
  userEvent.click(button);
  expect(screen.getByText('Count: 1')).toBeInTheDocument();
});
```

**Kent C. Dodds 的测试原则：**

> "The more your tests resemble the way your software is used, the more confidence they can give you."
>
> （测试越接近软件的实际使用方式，它能给你带来的信心就越大。）

### 3.2 不该测什么

```typescript
// ❌ 不要测试实现细节

// 不要直接测试内部 state
expect(component.state.loading).toBe(false);

// 不要直接测试内部方法
component.instance().handleClick();

// 不要测试 props 传递（这是框架的责任）
expect(component.props().onClick).toBeDefined();

// ✅ 应该测试的内容

// 测试用户看到的内容
expect(screen.getByText('加载完成')).toBeInTheDocument();

// 测试用户可执行的操作
await userEvent.click(screen.getByRole('button', { name: '提交' }));

// 测试可观测的副作用
expect(mockFetch).toHaveBeenCalledWith('/api/users');
```

## 四、jsdom 浏览器环境模拟

### 4.1 jsdom 的作用

```
真实浏览器：完整的 DOM + CSS + JS 引擎
jsdom：纯 JS 实现的 DOM 模拟（无布局、无渲染）
```

```typescript
// jsdom 提供了这些浏览器 API：
document.querySelector('#app')       // ✅ 可用
document.createElement('div')        // ✅ 可用
window.location.href                 // ✅ 可用
localStorage.getItem('token')        // ✅ 可用

// jsdom 不支持：
// ❌ CSS 布局计算（getBoundingClientRect 返回 0）
// ❌ 真实网络请求（需配合 MSW）
// ❌ Canvas 渲染
// ❌ 真实的浏览器导航
```

### 4.2 jsdom 局限性与应对

```typescript
// 场景1：测试需要真实布局的组件
// 使用 @testing-library/react 的 query 不依赖布局
render(<Modal open>内容</Modal>);
expect(screen.getByRole('dialog')).toBeInTheDocument();

// 场景2：测试 CSS 样式
// jsdom 不计算样式，使用 snapshot 或 visual regression 测试替代

// 场景3：测试图片加载
// 需要手动触发 load 事件
const img = screen.getByAltText('avatar');
fireEvent.load(img);
expect(screen.getByAltText('avatar')).toHaveClass('loaded');
```

## 六、describe / it / expect 基础

### 6.1 测试结构

```typescript
import { render, screen } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { Button } from './Button';

describe('Button', () => {
  // ===== 设置阶段 =====
  const defaultProps = {
    onClick: vi.fn(),
    children: '点击我',
  };

  // 每个测试前重置
  beforeEach(() => {
    vi.clearAllMocks();
  });

  // ===== 测试用例 =====
  it('should render children text', () => {
    render(<Button {...defaultProps} />);
    expect(screen.getByText('点击我')).toBeInTheDocument();
  });

  it('should call onClick when clicked', async () => {
    const user = userEvent.setup();
    const onClick = vi.fn();
    render(<Button onClick={onClick}>点击</Button>);

    await user.click(screen.getByRole('button'));
    expect(onClick).toHaveBeenCalledTimes(1);
  });

  it('should be disabled when disabled prop is true', () => {
    render(<Button {...defaultProps} disabled />);
    expect(screen.getByRole('button')).toBeDisabled();
  });

  // ===== 嵌套 describe 分组 =====
  describe('variants', () => {
    it('should apply primary styles', () => {
      render(<Button {...defaultProps} variant="primary" />);
      expect(screen.getByRole('button')).toHaveClass('btn-primary');
    });

    it('should apply danger styles', () => {
      render(<Button {...defaultProps} variant="danger" />);
      expect(screen.getByRole('button')).toHaveClass('btn-danger');
    });
  });
});
```

### 6.3 同步 vs 异步测试

```typescript
// 同步测试
it('should render immediately', () => {
  render(<StaticComponent />);
  expect(screen.getByText('Hello')).toBeInTheDocument();
});

// 异步测试 - 等待元素出现
it('should load data asynchronously', async () => {
  render(<UserProfile userId="123" />);

  // findBy 会等待元素出现（默认 1000ms 超时）
  expect(screen.getByText('加载中...')).toBeInTheDocument();

  const name = await screen.findByText('张三');
  expect(name).toBeInTheDocument();
  expect(screen.queryByText('加载中...')).not.toBeInTheDocument();
});

// 异步测试 - waitFor
it('should update after async operation', async () => {
  render(<Counter />);

  await userEvent.click(screen.getByRole('button'));

  await waitFor(() => {
    expect(screen.getByText('Count: 1')).toBeInTheDocument();
  });
});

// 异步测试 - 等待元素消失
it('should hide loading after data loaded', async () => {
  render(<DataList />);

  expect(screen.getByTestId('spinner')).toBeInTheDocument();

  await waitForElementToBeRemoved(() => screen.getByTestId('spinner'));

  expect(screen.getByText('数据列表')).toBeInTheDocument();
});
```

## 七、测试运行器

### 7.1 Jest vs Vitest 命令对比

```bash
# Jest
npx jest                        # 运行所有测试
npx jest --watch                # 监听模式
npx jest Button.test            # 运行匹配的测试
npx jest --coverage             # 生成覆盖率报告
npx jest --testNamePattern="should render"  # 按名称过滤

# Vitest
npx vitest                      # 运行所有测试（watch 模式）
npx vitest run                  # 单次运行
npx vitest run Button           # 运行匹配的测试
npx vitest run --coverage       # 生成覆盖率报告
npx vitest --ui                 # 可视化界面
```

## 八、实用技巧

### 8.1 自定义 render（带 Provider）

```typescript
// src/test/test-utils.tsx
import { render, RenderOptions } from '@testing-library/react';
import { ReactElement, ReactNode } from 'react';
import { BrowserRouter } from 'react-router-dom';
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';
import { ThemeProvider } from '@/contexts/ThemeContext';
import { AuthProvider } from '@/contexts/AuthContext';

interface WrapperProps {
  children: ReactNode;
}

function AllTheProviders({ children }: WrapperProps) {
  const queryClient = new QueryClient({
    defaultOptions: {
      queries: { retry: false },
      mutations: { retry: false },
    },
  });

  return (
    <BrowserRouter>
      <QueryClientProvider client={queryClient}>
        <ThemeProvider>
          <AuthProvider>
            {children}
          </AuthProvider>
        </ThemeProvider>
      </QueryClientProvider>
    </BrowserRouter>
  );
}

// 自定义 render
function customRender(
  ui: ReactElement,
  options?: Omit<RenderOptions, 'wrapper'>
) {
  return render(ui, { wrapper: AllTheProviders, ...options });
}

// 导出替换原始 render
export * from '@testing-library/react';
export { customRender as render };
```

## 总结

| 概念 | 要点 |
|------|------|
| **测试金字塔** | 单元测试最多，E2E最少，平衡速度和置信度 |
| **RTL 哲学** | 测试用户行为，不测实现细节 |
| **jsdom** | 纯 JS 的 DOM 模拟，不支持真实布局 |
| **文件约定** | `*.test.tsx`，与源文件同目录或 `__tests__/` |
| **Jest vs Vitest** | API 兼容，Vitest 更快且 Vite 原生 |
| **自定义 render** | 包装常用 Provider，避免重复代码 |
