# E2E 测试与测试策略

## 一、E2E 测试概览

### 1.1 什么是 E2E 测试

```
E2E (End-to-End) 测试 = 在真实浏览器中模拟完整用户操作流程

用户视角：打开页面 → 点击按钮 → 填写表单 → 提交 → 看到结果
E2E测试：自动化执行上述每一步，并验证结果
```

### 1.2 测试金字塔中的 E2E

```
        / E2E \          最高置信度，最慢，最少
       / 集成   \        中等置信度，中速
      /  单元     \      最低置信度，最快，最多

E2E 测试适用于：
✅ 关键业务流程（注册、登录、支付）
✅ 跨页面导航
✅ 真实 API 交互
✅ 浏览器兼容性

不适用于：
❌ 纯逻辑验证（用单元测试）
❌ UI 细节（用组件测试）
❌ 边界条件（用单元测试）
```

### 2.2 编写 E2E 测试

```typescript
// tests/login.spec.ts
import { test, expect } from '@playwright/test';

test.describe('用户登录', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/login');
  });

  test('应该显示登录表单', async ({ page }) => {
    await expect(page.getByRole('heading', { name: '登录' })).toBeVisible();
    await expect(page.getByLabel('邮箱')).toBeVisible();
    await expect(page.getByLabel('密码')).toBeVisible();
    await expect(page.getByRole('button', { name: '登录' })).toBeVisible();
  });

  test('成功登录后跳转到首页', async ({ page }) => {
    // 填写表单
    await page.getByLabel('邮箱').fill('zhangsan@example.com');
    await page.getByLabel('密码').fill('password123');

    // 点击登录
    await page.getByRole('button', { name: '登录' }).click();

    // 验证跳转
    await expect(page).toHaveURL('/dashboard');

    // 验证页面内容
    await expect(page.getByText('欢迎回来')).toBeVisible();
    await expect(page.getByRole('button', { name: '退出' })).toBeVisible();
  });

  test('错误凭证显示错误消息', async ({ page }) => {
    await page.getByLabel('邮箱').fill('wrong@example.com');
    await page.getByLabel('密码').fill('wrongpassword');
    await page.getByRole('button', { name: '登录' }).click();

    await expect(page.getByText('邮箱或密码错误')).toBeVisible();
    await expect(page).toHaveURL('/login');
  });

  test('空字段显示验证错误', async ({ page }) => {
    await page.getByRole('button', { name: '登录' }).click();

    await expect(page.getByText('请输入邮箱')).toBeVisible();
    await expect(page.getByText('请输入密码')).toBeVisible();
  });

  test('记住登录状态', async ({ page }) => {
    await page.getByLabel('邮箱').fill('zhangsan@example.com');
    await page.getByLabel('密码').fill('password123');
    await page.getByLabel('记住我').check();
    await page.getByRole('button', { name: '登录' }).click();

    await expect(page).toHaveURL('/dashboard');

    // 刷新页面，应该保持登录
    await page.reload();
    await expect(page).toHaveURL('/dashboard');
  });
});
```

## 四、Page Object Model (POM)

### 4.1 为什么需要 POM

```
问题：多个测试都操作同一个页面 → 选择器重复、修改困难
解决：将页面操作封装成类 → 一处修改，全部生效
```

### 4.2 Playwright POM 实现

```typescript
// pages/LoginPage.ts
import { type Page, type Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly submitButton: Locator;
  readonly errorMessage: Locator;
  readonly rememberMeCheckbox: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByLabel('邮箱');
    this.passwordInput = page.getByLabel('密码');
    this.submitButton = page.getByRole('button', { name: '登录' });
    this.errorMessage = page.getByRole('alert');
    this.rememberMeCheckbox = page.getByLabel('记住我');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string, remember = false) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    if (remember) {
      await this.rememberMeCheckbox.check();
    }
    await this.submitButton.click();
  }

  async getErrorMessage() {
    return this.errorMessage.textContent();
  }
}

// pages/DashboardPage.ts
export class DashboardPage {
  readonly page: Page;
  readonly welcomeMessage: Locator;
  readonly logoutButton: Locator;
  readonly userMenu: Locator;

  constructor(page: Page) {
    this.page = page;
    this.welcomeMessage = page.getByText(/欢迎回来/);
    this.logoutButton = page.getByRole('button', { name: '退出' });
    this.userMenu = page.getByRole('button', { name: '用户菜单' });
  }

  async logout() {
    await this.userMenu.click();
    await this.logoutButton.click();
  }

  async getWelcomeText() {
    return this.welcomeMessage.textContent();
  }
}

// tests/login-with-pom.spec.ts
import { test, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';
import { DashboardPage } from '../pages/DashboardPage';

test.describe('登录流程', () => {
  test('成功登录并跳转', async ({ page }) => {
    const loginPage = new LoginPage(page);
    const dashboardPage = new DashboardPage(page);

    await loginPage.goto();
    await loginPage.login('zhangsan@example.com', 'password123');

    await expect(page).toHaveURL('/dashboard');
    await expect(dashboardPage.welcomeMessage).toBeVisible();
  });

  test('登录后可以退出', async ({ page }) => {
    const loginPage = new LoginPage(page);
    const dashboardPage = new DashboardPage(page);

    await loginPage.goto();
    await loginPage.login('zhangsan@example.com', 'password123');
    await dashboardPage.logout();

    await expect(page).toHaveURL('/login');
  });

  test('显示错误消息', async ({ page }) => {
    const loginPage = new LoginPage(page);

    await loginPage.goto();
    await loginPage.login('wrong@example.com', 'wrong');

    expect(await loginPage.getErrorMessage()).toBe('邮箱或密码错误');
  });
});
```

## 五、API Mocking

### 5.1 Playwright API Mocking

// 使用 route 拦截网络请求
test('显示用户列表', async ({ page }) => {
  // Mock API 响应
  await page.route('**/api/users', async (route) => {
    await route.fulfill({
      status: 200,

      body: JSON.stringify([
        { id: 1, name: '张三' },
        { id: 2, name: '李四' },
      ]),

test('处理 API 错误', async ({ page }) => {
  await page.route('**/api/users', (route) =>
    route.fulfill({
      status: 500,
      body: JSON.stringify({ error: '服务器错误' }),
    })
  );

  await page.goto('/users');
  await expect(page.getByText('加载失败')).toBeVisible();

// 修改现有请求
test('添加认证头', async ({ page }) => {
  await page.route('**/api/**', async (route) => {
    const headers = {
      ...route.request().headers(),
      Authorization: 'Bearer test-token',
    };
    await route.continue({ headers });
  });
});

// 延迟响应
test('显示加载状态', async ({ page }) => {
  await page.route('**/api/users', async (route) => {
    await new Promise((resolve) => setTimeout(resolve, 2000));
    await route.fulfill({
      body: JSON.stringify([]),
    });
  });

  await expect(page.getByTestId('skeleton')).toBeVisible();
});

## 七、测试策略

### 7.1 什么该测，什么不该测

```
✅ 应该测试：
├── 关键业务流程（注册、登录、支付、下单）
├── 用户核心路径（浏览、搜索、筛选、排序）
├── 权限和安全性（未授权访问、角色权限）
├── 数据完整性（表单提交、数据验证）
├── 错误处理（网络错误、API 错误、404）
├── 跨浏览器兼容性
└── 响应式布局（移动端、平板、桌面端）

❌ 不需要测试：
├── 第三方库的内部行为
├── CSS 样式的精确值（用视觉回归测试）
├── 框架本身的路由功能
├── 简单的 getter/setter
└── 临时/原型代码
```

### 7.2 Testing Library 指导原则

```typescript
/*
1. 测试应该与软件的使用方式保持一致
   → 测试用户能看到和交互的内容

2. 优先测试行为，而非实现
   → 不要测试 state，测试 UI 输出

3. 测试应该有弹性，能承受重构
   → 使用语义化查询（getByRole），而非 DOM 选择器

4. 测试应该让人有信心
   → 如果测试通过，代码就一定是对的

5. 测试应该快速提供反馈
   → 单元测试毫秒级，E2E 测试分钟级
*/
```

### 7.3 各层测试的定位

```
┌─────────────────────────────────────────────────────────┐
│  单元测试                                                │
│  ├─ 纯函数（format、validate、transform）                 │
│  ├─ 自定义 hooks（useToggle、useDebounce）               │
│  ├─ 工具函数（sort、filter、计算逻辑）                    │
│  └─ 数据转换（API 响应 → UI 模型）                       │
├─────────────────────────────────────────────────────────┤
│  组件测试                                                │
│  ├─ 组件渲染（props → DOM 输出）                         │
│  ├─ 用户交互（点击、输入、选择）                          │
│  ├─ 状态变化（loading → data → error）                   │
│  └─ 组件协作（父子通信、Context）                        │
├─────────────────────────────────────────────────────────┤
│  E2E 测试                                                │
│  ├─ 完整用户流程（注册→登录→使用→退出）                  │
│  ├─ 关键业务路径（浏览商品→加购→下单→支付）              │
│  ├─ 跨页面流程（导航、表单向导）                         │
│  └─ 真实环境集成（真实 API、真实浏览器）                 │
└─────────────────────────────────────────────────────────┘
```

### 8.3 覆盖率不是唯一指标

```
高覆盖率 ≠ 好测试

例：100% 覆盖率但无断言
function add(a, b) { return a + b; }

test('add', () => {
  add(1, 2);  // 执行了，但没验证结果！
});

正确做法：
test('add', () => {
  expect(add(1, 2)).toBe(3);
  expect(add(-1, 1)).toBe(0);
  expect(add(0, 0)).toBe(0);
});

关注点：
1. 覆盖率是底线，不是目标
2. 关注分支覆盖率（更能发现遗漏的逻辑路径）
3. 优先覆盖核心业务逻辑
4. E2E 覆盖率不需要很高（10-20% 足够覆盖关键路径）
```

## 九、CI/CD 集成

### 9.1 GitHub Actions 配置

```yaml
# .github/workflows/test.yml
name: Tests

on: [push, pull_request]

jobs:
  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npm run test:run -- --coverage
      - uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}

  e2e-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npm run build
      - run: npx playwright test
      - uses: actions/upload-artifact@v4
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 30
```

### 9.2 最佳实践

```
CI 流水线建议：
1. lint + type check → 快速反馈
2. unit tests → 并行执行，快速
3. build → 验证构建成功
4. e2e tests → 并行分片执行
5. coverage report → 上传到 Codecov

本地开发：
1. vitest --watch → 持续运行受影响的测试
2. 定期运行完整 E2E 测试
3. 提交前运行 lint + test
```

## 总结

| 概念 | 要点 |
|------|------|
| **Playwright vs Cypress** | Playwright 支持多浏览器、并行更好；Cypress 开发体验好 |
| **POM** | 将页面操作封装成类，提高复用性和可维护性 |
| **API Mock** | 使用 route/intercept 拦截网络请求，隔离外部依赖 |
| **视觉回归** | 截图对比，自动发现 UI 回归 |
| **测试策略** | 单元多、E2E少，关注关键业务路径 |
| **覆盖率** | 80% 是合理目标，关注分支覆盖率 |
