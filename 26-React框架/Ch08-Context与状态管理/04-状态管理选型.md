# 状态管理选型

React 生态中有众多状态管理方案。本文从简单到复杂梳理各种方案，并提供决策矩阵帮助你做出正确的选择。

---

## 一、状态分类

在选择方案之前，先明确状态的类型：

```
状态
├── 本地状态 (Local State)
│   ├── UI 状态：开关、选项卡、表单输入
│   ├── 瞬时状态：加载中、错误提示
│   └── 聚合状态：派生计算
│
├── 共享状态 (Shared State)
│   ├── 用户认证：登录状态、用户信息
│   ├── 主题/语言：全局 UI 偏好
│   └── 购物车：跨页面数据
│
├── 服务端状态 (Server State)
│   ├── API 数据缓存
│   ├── 乐观更新
│   └── 同步与失效策略
│
└── URL 状态 (URL State)
    ├── 查询参数 (query params)
    ├── 路径参数 (path params)
    └── Hash
```

## 二、方案对比

### 2.1 简单对比表

| 方案 | 包大小 | 学习成本 | TypeScript | 适合规模 | 核心思想 |
|---|---|---|---|---|---|
| useState | 0 (内建) | 低 | 优秀 | 任意 | 本地状态 |
| Context | 0 (内建) | 低 | 良好 | 小型 | 共享注入 |
| useReducer | 0 (内建) | 中 | 良好 | 中型 | 状态机 |
| Redux Toolkit | ~12KB | 中 | 优秀 | 中大型 | 集中 store |
| Zustand | ~1KB | 低 | 优秀 | 中型 | 外部 store |
| Jotai | ~2.4KB | 低 | 优秀 | 中型 | 原子化 |
| Recoil | ~18KB | 中 | 良好 | 中型 | 原子化 |
| Valtio | ~4KB | 低 | 良好 | 中型 | 代理响应式 |
| XState | ~40KB+ | 高 | 优秀 | 复杂 | 状态机 |
| TanStack Query | ~12KB | 中 | 优秀 | 任意 | 服务端状态 |

### 2.2 特性对比

## 三、各方案详解

### 3.1 useState — 组件内部

```jsx
// 最基础的状态管理，适用于组件内部
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
```

**适用**：组件私有状态，不需共享。

### 3.2 Lifting State Up — 父子共享

```jsx
// 通过提升状态到共同的父组件实现共享
function App() {
  const [filter, setFilter] = useState('all');
  const [todos, setTodos] = useState([]);

  return (
    <>
      <FilterBar filter={filter} onChange={setFilter} />
      <TodoList todos={todos} filter={filter} onToggle={toggleTodo} />
    </>
  );
}
```

**适用**：状态只需在少量兄弟组件间共享。

### 3.3 Context — 跨层级共享

```jsx
// 跨越多层组件共享，避免 prop drilling
const AuthContext = createContext();

function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const value = useMemo(() => ({ user, setUser }), [user]);
  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}
```

**适用**：低频变化的全局数据（主题、认证、配置）。

### 3.4 Context + useReducer — 复杂共享

```jsx
// 适合有复杂更新逻辑的共享状态
const TodoContext = createContext();

function todoReducer(state, action) {
  switch (action.type) {
    case 'ADD':    return [...state, action.payload];
    case 'TOGGLE': return state.map(t => t.id === action.payload ? { ...t, done: !t.done } : t);
    case 'DELETE': return state.filter(t => t.id !== action.payload);
    default:       return state;
  }
}

function TodoProvider({ children }) {
  const [todos, dispatch] = useReducer(todoReducer, []);
  return (
    <TodoContext.Provider value={{ todos, dispatch }}>
      {children}
    </TodoContext.Provider>
  );
}
```

**适用**：需要集中管理更新逻辑的共享状态。

### 3.5 Redux Toolkit — 大型应用

```jsx
import { createSlice, configureStore } from '@reduxjs/toolkit';

const todoSlice = createSlice({
  name: 'todos',
  initialState: [],
  reducers: {
    addTodo: (state, action) => {
      state.push({ id: Date.now(), text: action.payload, done: false });
    },
    toggleTodo: (state, action) => {
      const todo = state.find(t => t.id === action.payload);
      if (todo) todo.done = !todo.done;
    },
  },
});

const store = configureStore({ reducer: { todos: todoSlice.reducer } });
```

**适用**：团队协作、需要完善 DevTools、有复杂异步逻辑的中大型应用。

### 3.6 Zustand — 轻量级替代

```jsx
import { create } from 'zustand';

const useStore = create((set) => ({
  todos: [],
  addTodo: (text) => set((s) => ({
    todos: [...s.todos, { id: Date.now(), text, done: false }]
  })),
  toggleTodo: (id) => set((s) => ({
    todos: s.todos.map(t => t.id === id ? { ...t, done: !t.done } : t)
  })),
}));
```

**适用**：需要全局状态但不想引入 Redux 的复杂度。

### 3.7 Jotai — 原子化状态

```jsx
import { atom, useAtom } from 'jotai';

const countAtom = atom(0);
const doubledAtom = atom((get) => get(countAtom) * 2);

function Counter() {
  const [count, setCount] = useAtom(countAtom);
  const doubled = useAtom(doubledAtom);
  return <button onClick={() => setCount(c => c + 1)}>{count} (x2={doubled})</button>;
}
```

**适用**：状态粒度细、派生关系复杂的场景。

### 3.8 Valtio — 代理响应式

```jsx
import { proxy, useSnapshot } from 'valtio';

const store = proxy({
  count: 0,
  user: null,
  increment: () => { store.count++; },
});

function Counter() {
  const snap = useSnapshot(store);
  return <button onClick={store.increment}>{snap.count}</button>;
}
```

**适用**：喜欢 Vue/MobX 风格的响应式编程。

### 3.9 XState — 状态机

```jsx
import { createMachine, interpret } from 'xstate';

const toggleMachine = createMachine({
  id: 'toggle',
  initial: 'inactive',
  states: {
    inactive: { on: { TOGGLE: 'active' } },
    active:   { on: { TOGGLE: 'inactive' } },
  },
});
```

**适用**：有明确状态转换逻辑的复杂 UI（表单流程、支付流程、游戏状态）。

## 四、服务端状态 vs 客户端状态

这是选型中最容易混淆的区别。

```
客户端状态 ──── UI 交互产生的状态
├── 表单输入
├── 按钮状态
├── 弹窗开关
└── 本地偏好

服务端状态 ──── 来自 API 的数据
├── 用户列表
├── 商品数据
├── 文章内容
└── 搜索结果
```

### 服务端状态应该用专用工具

```jsx
// ❌ 错误：用 Redux/Zustand 管理服务端数据
const useStore = create((set) => ({
  users: [],
  loading: false,
  error: null,
  fetchUsers: async () => {
    set({ loading: true });
    try {
      const res = await fetch('/api/users');
      const users = await res.json();
      set({ users, loading: false });
    } catch (e) {
      set({ error: e.message, loading: false });
    }
  },
}));

// ✅ 正确：用 TanStack Query 管理服务端数据
import { useQuery } from '@tanstack/react-query';

function UserList() {
  const { data: users, isLoading, error } = useQuery({
    queryKey: ['users'],
    queryFn: () => fetch('/api/users').then(r => r.json()),
    staleTime: 5 * 60 * 1000,  // 5 分钟缓存
  });

  if (isLoading) return <Spinner />;
  if (error) return <Error error={error} />;

  return users.map(u => <div key={u.id}>{u.name}</div>);
}
```

**服务端状态管理工具**：TanStack Query（React Query）、SWR、Apollo Client（GraphQL）。

## 五、决策矩阵

### 5.1 快速决策流程图

```
1. 状态只在组件内使用？
   └── Yes → useState / useReducer

2. 状态需要在父子组件间共享？
   └── Yes → 提升状态 (Lifting State) / Props

3. 状态需要跨多层组件共享？
   ├── 变化频率低（主题、认证）→ Context
   ├── 变化频率中等 → Context + useReducer
   └── 变化频率高 → 外部状态库

4. 状态来自服务端 API？
   └── Yes → TanStack Query / SWR

5. 状态有复杂的状态转换逻辑？
   └── Yes → XState

6. 团队规模和项目复杂度？
   ├── 小团队/小项目 → Zustand / Jotai
   ├── 中等团队 → Zustand
   └── 大团队/大型项目 → Redux Toolkit
```

### 5.2 按场景推荐

| 场景 | 推荐方案 |
|---|---|
| 按钮点击状态 | useState |
| 表单状态 | useState / react-hook-form |
| 多步表单 | useReducer / XState |
| 主题切换 | Context |
| 用户认证 | Context |
| 购物车 | Context + useReducer / Zustand |
| 实时协作编辑 | Yjs / CRDT |
| API 数据获取 | TanStack Query |
| 全局搜索状态 | Zustand / URL params |
| 复杂工作流 | XState |
| 游戏状态 | XState / Zustand |

### 5.3 按应用规模推荐

```
小型应用（5-10 个组件）
└── useState + Context（认证/主题）+ TanStack Query

中型应用（10-50 个组件）
└── useState + Zustand + TanStack Query

大型应用（50+ 个组件，多个团队）
└── Redux Toolkit + RTK Query + XState（复杂流程）
```

## 六、组合使用

实际项目中通常组合多种方案：

```jsx
function App() {
  return (
    // 认证状态：低频变化 → Context
    <AuthProvider>
      {/* 主题：低频变化 → Context */}
      <ThemeProvider>
        {/* 全局 UI 状态：中频变化 → Zustand */}
        {/* 服务端数据 → TanStack Query (在子组件中使用 hooks) */}
        <Router>
          <Routes>
            <Route element={<Layout />}>
              <Route index element={<HomePage />} />
              <Route path="products" element={<ProductsPage />} />
              <Route path="cart" element={<CartPage />} />
            </Route>
          </Routes>
        </Router>
      </ThemeProvider>
    </AuthProvider>
  );
}
```

**原则**：每种状态管理工具用在它最擅长的地方，不要试图用一个工具解决所有问题。

## 七、URL 也是状态

很多开发者忽略了 URL 作为状态管理工具。

```jsx
// 应该放 URL 的状态
// - 当前页面/路由
// - 筛选条件、排序方式
// - 分页参数
// - 搜索关键词
// - 可分享的视图配置

import { useSearchParams } from 'react-router-dom';

function ProductList() {
  const [searchParams, setSearchParams] = useSearchParams();

  const category = searchParams.get('category') || 'all';
  const page = parseInt(searchParams.get('page') || '1');
  const sort = searchParams.get('sort') || 'name';

  // URL 变化 → 自动重新请求数据
  const { data } = useQuery({
    queryKey: ['products', { category, page, sort }],
    queryFn: () => fetchProducts({ category, page, sort }),
  });

  return (
    <div>
      <select
        value={category}
        onChange={(e) => setSearchParams(prev => {
          prev.set('category', e.target.value);
          prev.set('page', '1');  // 切换分类时重置分页
          return prev;
        })}
      >
        <option value="all">全部</option>
        <option value="electronics">电子产品</option>
      </select>
      {data?.items.map(p => <ProductCard key={p.id} product={p} />)}
    </div>
  );
}
```
