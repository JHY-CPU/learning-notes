# 组件设计原则

## 1. 单一职责原则（Single Responsibility）

每个组件应该**只做一件事**。如果一个组件承担了多个职责，就应该拆分。

### 1.1 违反单一职责

```jsx
// 不好：一个组件做了太多事情
function UserProfile({ userId }) {
  const [user, setUser] = React.useState(null);
  const [loading, setLoading] = React.useState(true);
  const [isEditing, setIsEditing] = React.useState(false);

  // 数据获取
  React.useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(setUser)
      .finally(() => setLoading(false));
  }, [userId]);

  // 保存逻辑
  const handleSave = async (data) => {
    await fetch(`/api/users/${userId}`, { method: 'PUT', body: JSON.stringify(data) });
    setIsEditing(false);
  };

  if (loading) return <Spinner />;

  // 混合了头像、信息、编辑表单
  return (
    <div>
      <img src={user.avatar} alt={user.name} />
      <h1>{user.name}</h1>
      <p>{user.bio}</p>
      {isEditing ? (
        <form onSubmit={handleSave}>
          <input defaultValue={user.name} />
          <textarea defaultValue={user.bio} />
          <button type="submit">Save</button>
        </form>
      ) : (
        <button onClick={() => setIsEditing(true)}>Edit</button>
      )}
    </div>
  );
}
```

### 1.2 遵循单一职责

```jsx
// 数据获取 Hook
function useUser(userId) {
  const [user, setUser] = React.useState(null);
  const [loading, setLoading] = React.useState(true);

  React.useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(setUser)
      .finally(() => setLoading(false));
  }, [userId]);

  return { user, loading };
}

// 头像展示
function UserAvatar({ src, alt }) {
  return <img className="avatar" src={src} alt={alt} />;
}

// 用户信息展示
function UserInfo({ user }) {
  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.bio}</p>
    </div>
  );
}

// 编辑表单
function UserEditForm({ user, onSave, onCancel }) {
  const [name, setName] = React.useState(user.name);
  const [bio, setBio] = React.useState(user.bio);

  return (
    <form onSubmit={() => onSave({ name, bio })}>
      <input value={name} onChange={(e) => setName(e.target.value)} />
      <textarea value={bio} onChange={(e) => setBio(e.target.value)} />
      <button type="submit">Save</button>
      <button type="button" onClick={onCancel}>Cancel</button>
    </form>
  );
}

// 主组件：只负责编排
function UserProfile({ userId }) {
  const { user, loading } = useUser(userId);
  const [isEditing, setIsEditing] = React.useState(false);

  if (loading) return <Spinner />;

  return (
    <div>
      <UserAvatar src={user.avatar} alt={user.name} />
      {isEditing ? (
        <UserEditForm
          user={user}
          onSave={handleSave}
          onCancel={() => setIsEditing(false)}
        />
      ) : (
        <>
          <UserInfo user={user} />
          <button onClick={() => setIsEditing(true)}>Edit</button>
        </>
      )}
    </div>
  );
}
```

## 2. Props 钻透问题（Prop Drilling）

当数据需要穿过多个中间层级才能到达目标组件时，就产生了 **prop drilling** 问题。

### 2.1 什么是 Prop Drilling

```jsx
// 数据需要穿过 App -> Sidebar -> UserMenu -> UserAvatar 才能到达需要它的地方
function App() {
  const user = useUser();
  return <Sidebar user={user} />;
}

function Sidebar({ user }) {
  // Sidebar 本身不需要 user，只是传递给子组件
  return <UserMenu user={user} />;
}

function UserMenu({ user }) {
  // UserMenu 也不直接需要 user
  return <UserAvatar user={user} />;
}

function UserAvatar({ user }) {
  // 终于用到了 user
  return <img src={user.avatar} alt={user.name} />;
}
```

### 2.2 Prop Drilling 的问题

- 中间组件被迫接收和传递不需要的 props
- 重构困难：改变数据结构需要修改所有中间组件
- 组件之间的耦合度增加

### 2.3 解决方案

**方案一：组件组合**（将需要数据的组件提升到拥有数据的位置）

```jsx
function App() {
  const user = useUser();
  return (
    <Sidebar>
      <UserAvatar user={user} />  {/* 直接放置，不需要中间传递 */}
    </Sidebar>
  );
}

function Sidebar({ children }) {
  return (
    <aside>
      <nav>...</nav>
      {children}
    </aside>
  );
}
```

**方案二：React Context**（跨层级共享数据）

```jsx
const UserContext = React.createContext(null);

function App() {
  const user = useUser();
  return (
    <UserContext.Provider value={user}>
      <Sidebar />
    </UserContext.Provider>
  );
}

function UserAvatar() {
  const user = React.useContext(UserContext);  // 直接消费
  return <img src={user.avatar} alt={user.name} />;
}
```

## 3. 状态提升（Lifting State Up）

当多个组件需要共享同一份数据时，将 state 提升到它们最近的共同父组件。

### 3.1 经典示例：温度转换器

```jsx
// 问题：两个输入框各自管理自己的 state，不同步
function TemperatureConverter() {
  return (
    <>
      <CelsiusInput />
      <FahrenheitInput />
    </>
  );
}

// 解决：将 state 提升到父组件
function TemperatureConverter() {
  const [temperature, setTemperature] = React.useState('');
  const [scale, setScale] = React.useState('c');

  const handleCelsiusChange = (val) => {
    setTemperature(val);
    setScale('c');
  };

  const handleFahrenheitChange = (val) => {
    setTemperature(val);
    setScale('f');
  };

  const celsius = scale === 'f' ? tryConvert(temperature, toCelsius) : temperature;
  const fahrenheit = scale === 'c' ? tryConvert(temperature, toFahrenheit) : temperature;

  return (
    <>
      <CelsiusInput value={celsius} onChange={handleCelsiusChange} />
      <FahrenheitInput value={fahrenheit} onChange={handleFahrenheitChange} />
      {parseFloat(celsius) >= 100
        ? <p>The water would boil.</p>
        : <p>The water would not boil.</p>}
    </>
  );
}
```

### 3.2 何时提升状态

- 多个组件需要显示相同的数据
- 多个组件需要修改同一份数据
- 需要在兄弟组件之间同步数据

### 3.3 派生状态（Derived State）

不要把可以计算的值存为 state，应该从已有 state/props 中派生。

```jsx
function ShoppingCart({ items }) {
  // 不要这样——items 变化时 itemCount 可能不同步
  // const [itemCount, setItemCount] = React.useState(0);

  // 推荐：派生计算
  const itemCount = items.length;
  const totalPrice = items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  const hasDiscount = totalPrice > 100;

  return (
    <div>
      <p>Items: {itemCount}</p>
      <p>Total: ${totalPrice.toFixed(2)}</p>
      {hasDiscount && <p>Discount applied!</p>}
    </div>
  );
}
```

## 4. 避免冗余状态

以下情况不需要存储为 state：

### 4.1 可从 props 计算的值

```jsx
// 不好
function UserCard({ firstName, lastName }) {
  const [fullName, setFullName] = React.useState('');

  React.useEffect(() => {
    setFullName(`${firstName} ${lastName}`);  // 冗余！
  }, [firstName, lastName]);

  return <h2>{fullName}</h2>;
}

// 好：直接计算
function UserCard({ firstName, lastName }) {
  const fullName = `${firstName} ${lastName}`;
  return <h2>{fullName}</h2>;
}
```

### 4.2 可从已有 state 推导的值

```jsx
// 不好
function FilteredList({ items }) {
  const [filtered, setFiltered] = React.useState([]);
  const [query, setQuery] = React.useState('');

  React.useEffect(() => {
    setFiltered(items.filter(i => i.name.includes(query)));
  }, [items, query]);
}

// 好：派生
function FilteredList({ items }) {
  const [query, setQuery] = React.useState('');
  const filtered = items.filter(i => i.name.includes(query));
}
```

### 4.3 反转数据流

```jsx
// 不好：存储了可以从其他值推导的值
function Toggle() {
  const [isOn, setIsOn] = React.useState(false);
  const [label, setLabel] = React.useState('OFF');  // 冗余

  const toggle = () => {
    setIsOn(!isOn);
    setLabel(isOn ? 'OFF' : 'ON');  // 需要同步维护
  };
}

// 好：只存储源数据
function Toggle() {
  const [isOn, setIsOn] = React.useState(false);
  const label = isOn ? 'ON' : 'OFF';  // 派生

  return <button onClick={() => setIsOn(!isOn)}>{label}</button>;
}
```

## 5. Smart vs Dumb 组件（智能组件与哑组件）

这是容器/展示组件模式的另一种表述：

| 特性 | Smart（智能） | Dumb（哑） |
|------|-------------|-----------|
| 知道数据来源 | 是（API、Store） | 否 |
| 管理状态 | 是 | 否 |
| 包含业务逻辑 | 是 | 否 |
| 可复用性 | 低 | 高 |
| 测试难度 | 中等 | 容易 |
| 典型位置 | 页面级组件 | 原子/分子级组件 |

```jsx
// Dumb 组件：纯展示，高复用
function Button({ variant, size, disabled, onClick, children }) {
  return (
    <button
      className={`btn btn-${variant} btn-${size}`}
      disabled={disabled}
      onClick={onClick}
    >
      {children}
    </button>
  );
}

// Smart 组件：包含业务逻辑
function LoginPage() {
  const [email, setEmail] = React.useState('');
  const [password, setPassword] = React.useState('');
  const { login, error } = useAuth();

  const handleSubmit = async (e) => {
    e.preventDefault();
    await login(email, password);
  };

  return (
    <form onSubmit={handleSubmit}>
      <Input value={email} onChange={setEmail} placeholder="Email" />
      <Input value={password} onChange={setPassword} type="password" />
      {error && <Alert type="error">{error}</Alert>}
      <Button variant="primary" type="submit">Log in</Button>
    </form>
  );
}
```

## 6. 命名规范

### 6.1 组件命名

```jsx
// 使用 PascalCase
function UserProfile() {}       // 好
function user_profile() {}      // 错误

// 名词或名词短语
function Button() {}            // 好
function Clickable() {}         // 不好（形容词）
function handleClick() {}       // 这是函数，不是组件

// 特定用途加前缀
function UserCard() {}          // 用户卡片
function ProductCard() {}       // 产品卡片
function PostCard() {}          // 文章卡片
```

### 6.2 Props 命名

```jsx
// 事件处理：on + 动词
<Button onClick={handleClick} />
<Form onSubmit={handleSubmit} />
<Input onChange={handleChange} />

// 处理函数：handle + 动词 或 动词 + 名词
const handleClick = () => {};
const handleSubmit = () => {};
const updateUser = () => {};

// 布尔值：is/has/can/should + 形容词/名词
<Button isLoading />
<MenuItem isActive />
<Form hasError />
<Feature canEdit />

// 回调 props（子组件接收时）：on + 动词
function Button({ onClick }) {}

// 回调函数定义（父组件中）：handle + 动词
function Parent() {
  const handleClick = () => {};
  return <Button onClick={handleClick} />;
}
```

### 6.3 文件命名

```
# 方式一：组件文件与组件同名（推荐）
Button.jsx
Button.css
Button.test.jsx
Button.stories.jsx

# 方式二：使用 index.jsx
Button/
  index.jsx
  styles.css
  test.jsx
```

## 7. 文件组织

### 7.1 按功能组织（Feature-based，推荐）

```
src/
  features/
    auth/
      components/
        LoginForm.jsx
        SignupForm.jsx
      hooks/
        useAuth.js
      api/
        authApi.js
    dashboard/
      components/
        DashboardLayout.jsx
        StatsCard.jsx
      hooks/
        useDashboardData.js
  shared/
    components/
      Button.jsx
      Input.jsx
      Modal.jsx
    hooks/
      useDebounce.js
      useLocalStorage.js
    utils/
      formatDate.js
```

### 7.2 按类型组织（Type-based）

```
src/
  components/
    Button.jsx
    Input.jsx
    Modal.jsx
  hooks/
    useAuth.js
    useFetch.js
  pages/
    Home.jsx
    About.jsx
  utils/
    formatDate.js
```

> 小项目用类型组织足够。中大型项目推荐功能组织，便于维护和删除整个功能模块。

### 7.3 Co-location 原则

将相关的文件放在一起，而不是按文件类型分散到各处：

```
# 不好：组件、测试、样式分散
src/
  components/
    Button.jsx
  tests/
    Button.test.jsx
  styles/
    Button.css

# 好：相关文件共处
src/
  components/
    Button/
      Button.jsx
      Button.test.jsx
      Button.css
      Button.stories.jsx
```

## 8. 组件粒度

### 8.1 粒度过细

```jsx
// 不要过度拆分——每个元素一个组件没有意义
function FirstNameLabel() {
  return <span>First Name</span>;
}

function FirstNameInput() {
  return <input />;
}

function LastNameLabel() {
  return <span>Last Name</span>;
}

function LastNameInput() {
  return <input />;
}
```

### 8.2 粒度过粗

```jsx
// 不要一个组件包含整个页面的所有内容
function EntirePage() {
  return (
    <div>
      {/* 500 行 JSX */}
      {/* Header, Nav, Sidebar, Main, Footer, Modals, Forms... */}
    </div>
  );
}
```

### 8.3 合理的粒度

- 当一段 JSX 在多个地方使用时，提取为组件
- 当组件超过 ~150-200 行时，考虑拆分
- 当组件逻辑和展示混杂时，分离为 Hook + 展示组件
- 当组件有独立的职责边界时，独立出来

## 9. 设计 Checklist

在创建组件时，问自己：

- [ ] 这个组件只做一件事吗？
- [ ] Props 是否最小化、有合理的默认值？
- [ ] 命名是否清晰、遵循团队规范？
- [ ] 是否可以从已有数据派生而非存储冗余状态？
- [ ] 组件粒度是否合理？（不过细也不过粗）
- [ ] 是否有适当的类型定义（TypeScript）？
- [ ] 文件组织是否清晰、相关文件是否共处？
- [ ] 是否考虑了复用性？

## 10. 常见面试问题

### Q1：如何决定组件的拆分粒度？

看三个信号：(1) 一段 JSX 是否在多处复用；(2) 组件是否超过 150-200 行；(3) 是否有独立的职责边界。满足任一条件就应考虑拆分。

### Q2：Prop Drilling 怎么解决？

三种方案：(1) 组件组合，将需要数据的组件提升到拥有数据的位置；(2) React Context，跨层级共享；(3) 状态管理库（Redux、Zustand 等）。

### Q3：什么数据应该存为 State，什么应该派生？

只有**不能从已有数据计算**的、且**会随用户交互变化**的数据才应存为 State。能从 props 或其他 state 计算的值应派生。
