# Prompt Engineering - Prompt工程与微调

*深入掌握提示工程核心技术，涵盖 Zero-shot/Few-shot、Chain-of-Thought、消息角色、Prompt 模板、Guardrails 及评估方法*

Zero-shot 与 Few-shot 对比

| 方式 | 说明 | 适用场景 | token 开销 |
| --- | --- | --- | --- |
| **Zero-shot** | 不提供示例，直接描述任务 | 简单任务、GPT-4 等强模型 | 最低 |
| **One-shot** | 提供 1 个示例 | 格式对齐、简单模式引导 | 低 |
| **Few-shot** | 提供 3-5 个示例 | 复杂格式、特定风格、弱模型 | 较高 |

消息角色详解

| 角色 | 说明 | 用途 |
| --- | --- | --- |
| **system** | 系统指令，设定模型行为 | 角色扮演、格式约束、安全边界 |
| **user** | 用户输入 | 问题、指令、上下文信息 |
| **assistant** | 模型回复 | 对话历史、Few-shot 示例 |

Prompt Engineering 高级技巧

| 技巧 | 说明 | 效果 |
| --- | --- | --- |
| **角色扮演** | 让 LLM 扮演特定专家角色 | 提升回答专业度和格式一致性 |
| **任务分解** | 将复杂任务拆分为多个子任务 | 减少错误，提高完成度 |
| **输出约束** | 明确规定输出格式（JSON/Markdown/列表） | 便于程序解析 |
| **负面约束** | 明确说明"不要做什么" | 减少不需要的输出 |
| **思维链** | 要求逐步推理 | 提高复杂推理准确率 |
| **自洽性** | 多次生成取共识 | 提高答案可靠性 |
| **回溯提示** | 先回答更一般的问题 | 为具体问题提供更丰富的上下文 |
| **情感激励** | "这对我的职业非常重要" | （存在争议）部分场景可能提升质量 |

Prompt 效果评估方法

| 方法 | 适用场景 | 说明 |
| --- | --- | --- |
| **人工评估** | 小规模、高质量要求 | 人工打分，最可靠但不 scalable |
| **LLM-as-Judge** | 中大规模自动化评估 | 用强模型（GPT-4）评估弱模型输出 |
| **精确匹配** | 有标准答案的任务 | 对比输出与标准答案 |
| **模糊匹配** | 开放性任务 | BLEU/ROUGE/语义相似度 |
| **A/B 测试** | Prompt 迭代优化 | 两个版本在同一批输入上对比 |
| **回归测试** | Prompt 修改后验证 | 确保修改没有破坏已有功能 |
