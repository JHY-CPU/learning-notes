# 渲染管线回顾

## 1. 核心理论

渲染管线（Rendering Pipeline）是将三维场景数据转换为屏幕二维像素的一系列有序处理阶段。现代GPU采用**可编程管线**（Programmable Pipeline），允许开发者通过Shader自定义顶点处理和像素着色阶段，而固定功能阶段（如光栅化、深度测试）仍由硬件实现。

### 1.1 CPU与GPU的分工与协作

CPU和GPU之间是一种**生产者-消费者**关系。CPU负责逻辑、剔除、排序和命令录制，GPU负责并行执行渲染命令。理解两者分工是性能优化的前提。

**CPU端工作流程：**
```
场景遍历 → 视锥剔除(Frustum Culling) → 遮挡剔除(Occlusion Culling)
→ 排序(Sorting) → 状态分组(State Sorting) → 构建Draw Call → 提交命令缓冲区
```

**GPU端工作流程：**
```
命令解析 → 顶点获取 → 顶点着色器 → 曲面细分 → 几何着色器
→ 裁剪 → 光栅化 → 片元着色器 → 深度/模板测试 → Alpha混合 → 帧缓冲输出
```

**关键公式：帧时间**
```
帧时间 T_frame = max(T_cpu, T_gpu)
帧率 FPS = 1000 / T_frame (ms)

当 T_cpu > T_gpu 时，瓶颈在CPU（通常是Draw Call过多）
当 T_gpu > T_cpu 时，瓶颈在GPU（通常是着色器过重或过度绘制）
```

### 1.2 Draw Call的代价与原理

Draw Call是CPU向GPU发送的一条渲染命令，包含"用这些顶点、这个Shader、这些纹理渲染一个物体"的含义。每次Draw Call的CPU端开销包括：

1. **状态验证**：检查当前渲染状态（Shader、纹理、混合模式）是否需要切换
2. **命令编码**：将渲染指令编码为GPU可执行的命令格式
3. **驱动开销**：图形驱动对命令的校验和转换

```cpp
// Draw Call开销的量化理解
// 假设每次Draw Call的CPU开销约为 0.01~0.05ms
// 在60fps下，每帧时间预算 = 16.67ms
// 纯CPU可支撑约 300~1500 次Draw Call（不含GPU时间）

// 实际测量Draw Call开销（伪代码）
auto start = HighResolutionTimer::Now();
for (int i = 0; i < 1000; i++) {
    device->SetShader(shader);
    device->SetTexture(0, texture);
    device->SetVertexBuffer(vb);
    device->SetIndexBuffer(ib);
    device->DrawIndexed(indexCount);  // 每次约 10~50μs
}
auto elapsed = HighResolutionTimer::Now() - start;
// 1000次Draw Call ≈ 10~50ms CPU时间 → 严重超标
```

### 1.3 Batching（合批）原理

合批的核心思想是**将多个小Draw Call合并为一个大Draw Call**，减少CPU-GPU通信次数。

**Static Batching（静态合批）：**
- 在加载时将使用相同材质的静态物体的网格合并为一个大网格
- 优点：运行时零开销
- 缺点：增加内存（合并后的网格无法共享顶点），不支持移动物体

**Dynamic Batching（动态合批）：**
- 运行时将小网格（通常<300顶点）动态合并提交
- 优点：支持移动物体，无需额外内存
- 缺点：有CPU合并开销，仅对小网格有效

**GPU Instancing（GPU实例化）：**
- 一次Draw Call渲染同一网格的多个实例，通过实例属性（位置、颜色等）区分
- 优点：最高效的合批方式，支持每个实例不同变换
- 缺点：要求所有实例使用相同的网格和材质

```hlsl
// Unity URP中的GPU Instancing
// Shader端：启用Instancing
#pragma multi_compile_instancing

struct Attributes
{
    float4 positionOS : POSITION;
    UNITY_VERTEX_INPUT_INSTANCE_ID  // 实例ID
};

struct Varyings
{
    float4 positionCS : SV_POSITION;
    UNITY_VERTEX_INPUT_INSTANCE_ID
};

Varyings Vert(Attributes input)
{
    Varyings output;
    UNITY_SETUP_INSTANCE_ID(input);
    UNITY_TRANSFER_INSTANCE_ID(input, output);
    // UNITY_INSTANCING_BUFFER_START/END 宏定义逐实例属性
    output.positionCS = TransformObjectToHClip(input.positionOS.xyz);
    return output;
}
```

**SRP Batcher（可编程渲染管线合批）：**
- URP/HDRP特有的合批机制，不合并网格，而是将常量缓冲区（CBUFFER）按布局分组
- 相同Shader变体的物体可以共享CBUFFER布局，只更新变化的部分
- 大幅减少驱动开销，是现代URP项目的主要合批方式

```hlsl
// SRP Batcher要求：所有uniform必须在CBUFFER中
CBUFFER_START(UnityPerMaterial)
    float4 _BaseColor;
    float _Metallic;
    float _Smoothness;
CBUFFER_END
// 不在CBUFFER中的变量会破坏SRP Batcher兼容性
```

## 2. 管线各阶段详解

### 2.1 应用阶段（CPU）

应用阶段完全在CPU上运行，负责：
- **场景管理**：遍历场景树，收集可见物体列表
- **剔除（Culling）**：视锥剔除（Frustum Culling）丢弃视锥体外的物体，遮挡剔除（Occlusion Culling）丢弃被完全遮挡的物体
- **排序**：不透明物体从前到远（利用Early-Z），半透明物体从远到近
- **命令提交**：将渲染命令写入命令缓冲区（Command Buffer），提交给GPU

```
视锥剔除原理：
        /|\
       / | \  ← 近裁剪面
      /  |  \
     /   |   \
    /    |    \  ← 远裁剪面
   +-----+-----+
  相机位置

物体包围球与视锥体6个面做相交测试，完全在外面的被剔除
O(n) 检查，n = 场景物体数量
```

### 2.2 顶点着色器（Vertex Shader）

对**每个顶点**独立执行，核心任务是坐标变换。将顶点从模型空间变换到裁剪空间，同时可计算逐顶点属性（法线、UV、颜色等）。

```
变换流水线：
模型空间 (Object Space)
    ↓ × Model矩阵
世界空间 (World Space)
    ↓ × View矩阵
观察空间 (View/Camera Space)
    ↓ × Projection矩阵
裁剪空间 (Clip Space)
    ↓ 透视除法 (w分量除法)
NDC空间 (Normalized Device Coordinates) [-1,1]³
    ↓ 视口变换
屏幕空间 (Screen Space) [0, width] × [0, height]
```

### 2.3 曲面细分阶段（Tessellation）

可选阶段，包含三个子阶段：
1. **Hull Shader**：决定每个面片的细分级别（Tessellation Factor）
2. **Tessellator（固定功能）**：根据细分级别在面片内生成新顶点
3. **Domain Shader**：对新生成的顶点做变换

```
细分示意：
原始面片（1个四边形）          细分后（TF=4）
+---+---+---+---+              +---+---+---+---+
|   |   |   |   |              | / | / | / | / |
+---+---+---+---+    →         +---+---+---+---+
|   |   |   |   |              | \ | \ | \ | \ |
+---+---+---+---+              +---+---+---+---+

适合：地形LOD、角色近景细节、置换贴图
```

### 2.4 几何着色器（Geometry Shader）

可选阶段，以**图元**（点/线/三角形）为单位输入，可输出零到多个新图元。常用于粒子扩展、线框渲染等。

```
输入：1个三角形（3个顶点）
输出：可选
  - 0个图元（剔除该三角形）
  - 1个三角形（修改顶点）
  - N个三角形（扩展几何体，如将点扩展为四边形）
```

### 2.5 光栅化（Rasterization）

**固定功能阶段**，将三角形转换为片元（Fragment）。

过程：
1. **图元装配**：将顶点组装为三角形
2. **三角形遍历**：确定哪些像素被三角形覆盖
3. **属性插值**：使用重心坐标对顶点属性（UV、法线、颜色）进行透视校正插值

```
重心坐标插值公式：
P = w0 * P0 + w1 * P1 + w2 * P2
其中 w0 + w1 + w2 = 1
(P0, P1, P2 为三角形三个顶点)

透视校正插值：
实际插值在透视空间进行，不是屏幕空间
属性值 = (w0*v0/z0 + w1*v1/z1 + w2*v2/z2) / (w0/z0 + w1/z1 + w2/z2)
其中 z0, z1, z2 是各顶点在观察空间的深度
```

### 2.6 片元着色器（Fragment/Pixel Shader）

对**每个片元**（潜在像素）独立执行，计算最终颜色。光照、纹理采样、特效等主要在此阶段完成。

### 2.7 输出合并（Output Merger）

**固定功能阶段**，执行：
- **深度测试**（Depth Test）：比较片元深度与深度缓冲区值，决定是否通过
- **模板测试**（Stencil Test）：根据模板缓冲区值进行条件判断
- **Alpha混合**（Blending）：将片元颜色与帧缓冲区已有颜色混合

```
深度测试公式：
if (fragment.depth < depthBuffer[x][y]) {
    // 通过测试，写入新深度和颜色
    depthBuffer[x][y] = fragment.depth;
    frameBuffer[x][y] = fragment.color;
} else {
    // 丢弃该片元
    discard;
}
```

## 3. Unity URP/HDRP中的管线实现

### 3.1 URP渲染管线帧结构

```
URP单帧渲染流程：
1. Shadow Map渲染（每个产生阴影的光源一个Pass）
2. Depth Pre-Pass（可选，写入深度图供后续SSAO等使用）
3. 正向渲染Pass（不透明 → AlphaTest → 半透明）
4. 后处理链（Bloom → 色调映射 → FXAA/TAA）
5. UI渲染
```

### 3.2 SRP Batcher实现示例

```csharp
// C#端：确保材质使用SRP Batcher兼容的Shader
// Shader端必须满足：
// 1. 所有uniform在CBUFFER中声明
// 2. 使用Unity内置的CBUFFER名称（UnityPerMaterial, UnityPerDraw, UnityPerFrame）

// Shader代码（HLSL）
CBUFFER_START(UnityPerDraw)
    float4x4 unity_ObjectToWorld;
    float4x4 unity_WorldToObject;
CBUFFER_END

CBUFFER_START(UnityPerMaterial)
    float4 _BaseColor;
    float _Cutoff;
CBUFFER_END
```

## 4. 性能分析与优化策略

| 瓶颈类型 | 诊断方法 | 优化策略 |
|---------|---------|---------|
| CPU Draw Call | RenderDoc查看Draw Call数 | Batching、Instancing、SRP Batcher |
| GPU顶点处理 | GPU Profile工具查看顶点数 | LOD、背面剔除、减少细分 |
| GPU片元处理 | 查看Overdraw热力图 | Early-Z、排序、减少半透明 |
| 显存带宽 | 查看纹理采样次数 | 压缩纹理、Mipmap、减少采样 |

## 5. 常见问题与调试

**问题1：帧率低但GPU占用不高**
- 原因：CPU瓶颈，Draw Call过多
- 解决：使用Frame Debugger查看Draw Call分布，启用合批

**问题2：画面撕裂（Tearing）**
- 原因：GPU在显示器刷新过程中修改了帧缓冲区
- 解决：启用VSync或使用三重缓冲

**问题3：半透明物体渲染错误**
- 原因：渲染顺序不正确
- 解决：确保半透明物体从远到近排序，或使用OIT技术

## 6. 实际使用案例

- Unity URP使用Forward+渲染，支持多光源高效渲染，SRP Batcher减少驱动开销
- Unreal Engine 5使用Nanite（虚拟几何体）和Lumen（全局光照），重新定义了渲染管线的几何和光照阶段
- 移动端游戏大量使用Tile-Based Deferred Rendering（如Mali/Adreno GPU），减少带宽消耗
- 《原神》使用自定义渲染管线，针对不同机型使用不同的Shader变体和LOD策略
