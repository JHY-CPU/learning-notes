# 云原生架构 - 云服务与DevOps笔记


云原生（Cloud Native）是一种构建和运行应用程序的方法论，充分利用云计算模型的优势。CNCF（云原生计算基金会）将其定义为：容器化、微服务、持续交付和DevOps的结合。


### 1.1 云原生的核心要素


| 要素 | 说明 | 代表技术 |
| --- | --- | --- |
| **容器化** | 应用打包为轻量级容器镜像 | Docker, containerd, CRI-O |
| **容器编排** | 自动化部署、扩展和管理容器 | Kubernetes (K8s) |
| **微服务** | 应用拆分为小而独立的服务 | Spring Cloud, Dubbo, gRPC |
| **不可变基础设施** | 部署后不修改，变更通过重新部署实现 | 容器镜像、AMI |
| **声明式API** | 描述期望状态，系统自动达成 | K8s YAML, Terraform HCL |
| **持续交付** | 自动化构建、测试、部署流水线 | Jenkins, GitLab CI, ArgoCD |


### 1.2 云原生 vs 传统架构


| 维度 | 传统架构 | 云原生架构 |
| --- | --- | --- |
| 部署单位 | 单体应用/虚拟机 | 容器/微服务 |
| 扩展方式 | 垂直扩展（加CPU/内存） | 水平扩展（加实例） |
| 发布频率 | 月度/季度发布 | 每日多次发布 |
| 故障处理 | 避免故障 | 拥抱故障、快速恢复 |
| 基础设施 | 手动管理、可变 | 自动化、不可变 |
| 服务发现 | 硬编码/配置文件 | 服务注册中心自动发现 |


十二要素应用是由Heroku团队提出的构建SaaS应用的方法论，是云原生应用设计的重要参考标准。


| # | 要素 | 核心原则 |
| --- | --- | --- |
| 1 | **基准代码（Codebase）** | 一份代码库，多次部署（dev/staging/prod） |
| 2 | **依赖（Dependencies）** | 显式声明依赖关系，不依赖系统级包 |
| 3 | **配置（Config）** | 配置存储在环境变量中，与代码分离 |
| 4 | **后端服务（Backing Services）** | 把后端服务当作附加资源，通过URL访问 |
| 5 | **构建/发布/运行（Build/Release/Run）** | 严格分离构建和运行阶段 |
| 6 | **进程（Processes）** | 应用作为无状态进程运行 |
| 7 | **端口绑定（Port Binding）** | 通过端口绑定对外提供服务 |
| 8 | **并发（Concurrency）** | 通过进程模型进行扩展 |
| 9 | **易处理（Disposability）** | 快速启动和优雅终止，增强健壮性 |
| 10 | **开发环境与线上等价（Dev/Prod Parity）** | 保持开发、预发、线上环境尽可能一致 |
| 11 | **日志（Logs）** | 把日志当作事件流，输出到stdout |
| 12 | **管理进程（Admin Processes）** | 后台管理任务作为一次性进程运行 |

**实践要点：**

- **配置外部化**
   ：数据库连接串、API密钥等敏感配置通过环境变量或配置中心注入
- **无状态设计**
   ：会话数据存Redis/数据库，不存本地文件或内存
- **日志标准化**
   ：统一JSON格式输出，由日志采集系统（Fluentd/Filebeat）统一收集
- **健康检查**
   ：提供 /health 和 /ready 端点供编排系统探测


### 3.1 微服务核心模式


- **服务拆分**
   ：按业务领域（DDD）拆分，每个服务拥有独立数据库
- **服务注册与发现**
   ：服务启动时注册到注册中心，消费端通过注册中心发现服务地址
- **负载均衡**
   ：客户端负载均衡（Ribbon）或服务端负载均衡（SLB/ALB）
- **熔断降级**
   ：当依赖服务不可用时，快速失败并返回降级结果
- **分布式配置**
   ：配置中心统一管理所有服务配置，支持动态刷新
- **链路追踪**
   ：跟踪一次请求在多个服务间的调用链路


### 3.2 云端微服务技术栈


| 能力 | Spring Cloud | 阿里云方案 | AWS方案 |

| 配置中心 | Nacos / Apollo | ACM / MSE | Parameter Store / AppConfig |

### 4.1 什么是服务网格


服务网格是处理服务间通信的专用基础设施层，通常以一组轻量级网络代理（Sidecar）的形式与应用部署在一起，负责服务间的流量管理、安全和可观测性。


### 4.2 Sidecar模式


```
┌──── Pod ────────────────────────┐
  │  ┌──────────┐  ┌──────────┐   │
  │  │  应用容器  │  │ Sidecar  │   │
  │  │ (业务代码) │←→│ (Envoy)  │   │
  │  └──────────┘  └──────────┘   │
  └────────────────────────────────┘
         │                │
    应用只需关注业务逻辑，所有网络通信由Sidecar接管
```


### 4.3 主流服务网格方案


| 方案 | 控制面 | 数据面 | 特点 |
| --- | --- | --- | --- |
| **Istio** | istiod | Envoy | 功能最全面，社区活跃 |
| **Linkerd** | control-plane | linkerd2-proxy (Rust) | 轻量级，性能好 |
| **Consul Connect** | Consul Server | Envoy | HashiCorp生态集成 |
| **阿里云ASM** | 托管Istio | Envoy | 免运维Istio |


### 4.4 服务网格核心能力


- **流量管理**
   ：灰度发布、金丝雀部署、流量镜像、故障注入
- **安全**
   ：mTLS自动加密、服务间认证授权、RBAC策略
- **可观测性**
   ：自动收集指标（Prometheus）、分布式追踪（Jaeger）、访问日志
- **弹性**
   ：超时、重试、熔断、限流（无需修改业务代码）

**何时引入服务网格？**
当微服务数量较多（通常50+），且需要统一管理流量策略、安全策略和可观测性时，服务网格的价值才充分体现。对于小型项目，SDK方式（如Spring Cloud）更简单直接。

### 5.1 API网关的职责


- **请求路由**
   ：根据路径、Header等将请求转发到对应后端服务
- **协议转换**
   ：HTTP→gRPC、WebSocket代理等
- **认证鉴权**
   ：JWT验证、OAuth2.0、API Key校验
- **限流熔断**
   ：保护后端服务不被过量请求压垮
- **请求/响应转换**
   ：Header改写、参数映射、响应格式化
- **日志与监控**
   ：记录API调用日志、统计QPS和延迟
- **API版本管理**
   ：支持多版本API并存和灰度切换


### 5.2 主流API网关


| 网关 | 类型 | 特点 |
| --- | --- | --- |
| Nginx / OpenResty | 通用反向代理 | 高性能、可编程（Lua） |
| Kong | API网关 | 基于Nginx/Lua，插件丰富 |
| Spring Cloud Gateway | Java网关 | Spring生态集成，响应式 |
| APISIX | API网关 | Apache项目，高性能，插件热加载 |
| Envoy | Sidecar/网关 | C++编写，服务网格数据面 |
| AWS API Gateway | 托管网关 | 全托管，与Lambda深度集成 |
| 阿里云API网关 | 托管网关 | 云原生API网关，基于Envoy |


### 6.1 可观测性三大支柱


| 支柱 | 说明 | 工具 |
| --- | --- | --- |
| **指标（Metrics）** | 量化系统状态的数值数据 | Prometheus, CloudWatch, 云监控 |
| **日志（Logs）** | 记录系统事件的文本数据 | ELK/EFK, Loki, SLS, CloudWatch Logs |
| **追踪（Traces）** | 跟踪请求在分布式系统中的完整路径 | Jaeger, Zipkin, X-Ray, SkyWalking |


### 6.2 云端可观测性架构


```
应用服务 ──→ [Prometheus] ──→ [Grafana] ──→ 可视化看板
      │
      ├──→ [Fluentd/Filebeat] ──→ [Elasticsearch/Loki] ──→ 日志查询
      │
      └──→ [OpenTelemetry SDK] ──→ [Jaeger/Zipkin] ──→ 链路追踪

  告警流程: Prometheus Alertmanager → Webhook/邮件/短信 → 值班人员
```


### 6.3 OpenTelemetry（统一标准）


OpenTelemetry是CNCF旗下的可观测性标准，提供统一的API、SDK和工具，用于收集Metrics、Logs和Traces。


- **目标**
   ：一套SDK同时收集三大支柱数据，与后端解耦
- **支持语言**
   ：Java、Go、Python、Node.js、.NET、Rust等
- **组件**
   ：API（接口定义）、SDK（实现）、Collector（数据收集转发）
- **优势**
   ：避免厂商锁定，一套代码支持多种后端


> **Example:** **云上可观测性最佳实践：**
>
> - 所有服务统一输出JSON格式日志到stdout
> - 使用OpenTelemetry SDK自动注入追踪上下文
> - 定义统一的SLO（服务等级目标），如P99延迟 < 200ms
> - 建立告警分级：P0（电话）/ P1（短信）/ P2（邮件）/ P3（工单）
> - 使用Grafana构建统一监控大盘，涵盖基础设施、中间件、业务指标
