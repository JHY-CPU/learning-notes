并发编程模型

Concurrency Programming Models — 多线程、异步、Actor与CSP

1. 并发基础概念

并发 vs 并行

概念 定义 关注点
并发 (Concurrency) 多个任务在重叠的时间段内推进 任务调度与结构
并行 (Parallelism) 多个任务同时执行 同时执行速度
并发 (单核交替执行): ┌──────┐ ┌──────┐ ┌──────┐ │ 任务A │ │ 任务B │ │ 任务A │ ... └──────┘ └──────┘ └──────┘ 时间 ──────────────────────▶ 并行 (多核同时执行): 核心1: ┌──────┐┌──────┐┌──────┐ │ 任务A ││ 任务A ││ 任务A │ └──────┘└──────┘└──────┘ 核心2: ┌──────┐┌──────┐┌──────┐ │ 任务B ││ 任务B ││ 任务B │ └──────┘└──────┘└──────┘

核心问题：共享可变状态

并发的核心困难在于多个执行单元同时访问共享的可变数据，导致竞态条件 (Race Condition)。

// 经典竞态条件示例
int counter = 0;

// 线程A和线程B同时执行
void increment() {
    // 读取-修改-写入 三步操作非原子
    int temp = counter;   // 1. 读
    temp = temp + 1;      // 2. 改
    counter = temp;       // 3. 写
    // 交错执行导致丢失更新!
}

2. 线程模型 (Threads)

线程是操作系统调度的基本单位，由OS内核管理，是抢占式并发的典型实现。

线程的特点

  • 共享内存：同一进程的线程共享堆内存
  • 私有栈：每个线程有自己的调用栈和寄存器状态
  • OS调度：由操作系统内核进行抢占式调度
  • 重量级：创建/切换开销较大 (通常MB级栈空间)

同步原语

// 互斥锁 (Mutex)
std::mutex mtx;
std::lock_guard<std::mutex> lock(mtx);  // RAII锁
shared_data = new_value;

// 信号量 (Semaphore) — 控制并发数量
sem_t sem;
sem_init(&sem, 0, 3);  // 允许最多3个线程同时访问
sem_wait(&sem);        // P操作, 计数-1
// 临界区
sem_post(&sem);        // V操作, 计数+1

// 条件变量 (Condition Variable) — 等待特定条件
std::condition_variable cv;
std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, [] { return ready; });  // 等待条件满足

死锁 (Deadlock)

四个必要条件：互斥、持有并等待、不可剥夺、循环等待。

// 死锁示例
// 线程A: lock(mtx1) → lock(mtx2)
// 线程B: lock(mtx2) → lock(mtx1)  ← 循环等待!

// 解决方案: 按固定顺序加锁
std::scoped_lock lock(mtx1, mtx2);  // C++17 同时加锁
// 或使用 std::lock(mtx1, mtx2) 避免死锁

3. Async/Await 模型

基于协程 (Coroutine) 和事件循环的协作式并发。任务在await点主动让出控制权，而非被抢占。

核心原理

┌─────────────────────────────────────────────┐ │ 事件循环 (Event Loop) │ │ │ │ ┌──────────┐ ┌──────────┐ ┌────────┐ │ │ │ 任务队列 │ │ 微任务队列 │ │ I/O多路│ │ │ │ (MacroTask)│ │(MicroTask)│ │ 复用器 │ │ │ └──────────┘ └──────────┘ └────────┘ │ │ ↑ ↑ ↑ │ │ setTimeout Promise回调 epoll/kqueue │ └─────────────────────────────────────────────┘ 
// Python async/await
import asyncio

async def fetch_data(url):
    // 遇到await时让出控制权, 不阻塞线程
    response = await aiohttp.get(url)
    data = await response.json()
    return data

// 并发执行多个协程
async def main():
    // gather创建并发任务, 都在一个线程中运行!
    results = await asyncio.gather(
        fetch_data("https://api.a.com"),
        fetch_data("https://api.b.com"),
        fetch_data("https://api.c.com"),
    )

// JavaScript async/await
async function fetchUser(id) {
    const response = await fetch(`/api/users/${id}`);
    return response.json();
    // await期间JS引擎可以处理其他任务
}

协程 vs 线程

特性线程协程/Async
调度方式OS抢占式用户态协作式
切换开销大 (内核态切换)小 (用户态切换)
内存开销MB级栈KB级栈
数据竞争需要同步天然避免 (单线程)
CPU密集适合不适合
IO密集浪费资源非常高效

4. Actor 模型

Actor是独立的计算单元，通过异步消息传递通信，不共享状态。由Carl Hewitt于1973年提出。

Actor的三要素

  1. 状态 (State) — 每个Actor有私有状态，其他Actor不可直接访问
  2. 行为 (Behavior) — 接收到消息时如何响应
  3. 邮箱 (Mailbox) — 消息队列，异步通信
Actor A Actor B Actor C ┌─────────┐ msg1 ┌─────────┐ msg3 ┌─────────┐ │ 状态 s │ ──────▶ │ 状态 s' │ ──────▶ │ 状态 s'' │ │ 邮箱 │ │ 邮箱 │ │ 邮箱 │ │ 行为 f │ ◀────── │ 行为 g │ ◀────── │ 行为 h │ └─────────┘ msg2 └─────────┘ msg4 └─────────┘ 不共享状态! 只通过消息通信 
// Erlang/Elixir Actor模型
%% 定义一个Actor(进程)
counter(Count) ->
    receive
        {increment} ->
            counter(Count + 1);          %% 递归调用, 新状态
        {get, Pid} ->
            Pid ! {count, Count},        %% 发送消息
            counter(Count);
        {stop} ->
            ok
    end.

%% 使用
Pid = spawn(fun() -> counter(0) end),    %% 创建Actor
Pid ! {increment},                       %% 发消息
Pid ! {get, self()},                     %% 请求当前值
receive {count, N} -> N end.             %% 接收回复

// Scala Akka Actor
class CounterActor extends Actor {
    var count = 0
    def receive = {
        case "increment" => count += 1
        case "get" => sender() ! count
    }
}

5. CSP 模型 (Communicating Sequential Processes)

由Tony Hoare于1978年提出。通过Channel通信而非直接发送消息给Actor。

Actor vs CSP

特性Actor模型CSP模型
通信目标发送给特定Actor发送到Channel
耦合度发送方知道接收方发送方不知道谁接收
代表语言Erlang, AkkaGo, Occam
通道所有权每个Actor有自己的邮箱Channel是独立实体
// Go语言的CSP实现
func producer(ch chan<- int) {
    for i := 0; i < 10; i++ {
        ch <- i  // 发送到channel
    }
    close(ch)
}

func consumer(ch <-chan int, done chan<- bool) {
    for v := range ch {
        fmt.Println(v)
    }
    done <- true
}

func main() {
    ch := make(chan int, 5)  // 带缓冲的channel
    done := make(chan bool)

    go producer(ch)          // goroutine = 轻量级协程
    go consumer(ch, done)

    <-done  // 等待完成
}

6. 共享内存 vs 消息传递

并发编程的两种根本范式，代表了不同的哲学。

特性共享内存消息传递
通信方式读写共享变量发送/接收消息
同步机制锁、原子操作Channel/邮箱
典型语言C/C++/JavaErlang/Go/Elixir
调试难度难 (竞态条件)相对简单
性能直接访问, 快序列化开销
可扩展性差 (锁竞争)好 (天然分布式)

对比示例：生产者-消费者问题

// 共享内存方式 (Java)
BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(10);
// 生产者
queue.put(item);  // 满了会阻塞
// 消费者
item = queue.take();  // 空了会阻塞

// 消息传递方式 (Go)
ch := make(chan int, 10)
// 生产者
ch <- item  // 满了会阻塞
// 消费者
item = <-ch  // 空了会阻塞

// 哲学家Hewitt: "不要通过共享内存来通信,
//              而要通过通信来共享内存"
