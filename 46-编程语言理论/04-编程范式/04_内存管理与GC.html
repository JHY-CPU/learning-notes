1. 程序内存布局

高地址 ┌──────────────────┐ │ 命令行参数 │ │ 环境变量 │ ├──────────────────┤ │ 栈 (Stack) │ ← 局部变量、函数参数、返回地址 │ ↓ 增长 │ │ │ │ (空闲区域) │ │ │ │ ↑ 增长 │ │ 堆 (Heap) │ ← 动态分配 (malloc/new) ├──────────────────┤ │ BSS (未初始化数据) │ ← 全局未初始化变量 ├──────────────────┤ │ Data (初始化数据) │ ← 全局已初始化变量 ├──────────────────┤ │ Text (代码段) │ ← 程序指令 (只读) 低地址 └──────────────────┘

栈 (Stack) vs 堆 (Heap)

特性栈堆
分配速度极快 (移动栈指针)较慢 (需要查找空闲块)
管理方式自动 (函数调用管理)手动或GC
生命周期函数作用域程序员/GC控制
大小限制小 (通常1-8MB)大 (受物理内存限制)
碎片问题无有
数据结构LIFO任意
// 栈分配 vs 堆分配 (C++)
void example() {
    int x = 42;              // 栈上分配, 函数返回时自动释放
    int* p = new int(42);    // 堆上分配, 需手动delete
    std::vector<int> v;      // 对象在栈上, 内部数据在堆上
    delete p;                // 手动释放堆内存
}   // x 和 v 自动销毁 (RAII)

2. 手动内存管理

程序员显式控制内存的分配和释放。典型于C/C++。

常见问题

  • 内存泄漏 (Memory Leak) — 分配后忘记释放
  • 悬空指针 (Dangling Pointer) — 释放后继续使用
  • 双重释放 (Double Free) — 同一内存释放两次
  • 缓冲区溢出 (Buffer Overflow) — 写入超出分配范围
// 内存泄漏
void leak() {
    char* p = (char*)malloc(100);
    // 忘记 free(p); → 泄漏!
}

// 悬空指针
int* dangling() {
    int x = 42;
    return &x;  // x在函数返回后被销毁, 返回无效地址!
}

// C++ RAII解决手动管理问题
void safe() {
    std::unique_ptr<int> p = std::make_unique<int>(42);
    // 自动释放, 不可能泄漏
    std::shared_ptr<int> sp = std::make_shared<int>(42);
    // 引用计数, 最后一个shared_ptr销毁时释放
}

3. 引用计数 (Reference Counting)

每个对象维护一个引用计数器，当引用为0时立即回收。

引用计数示例: A.refcount = 1 A.refcount = 2 ┌─────────┐ ┌─────────┐ │ Object A │ ◀─ p1 │ Object A │ ◀─ p1 │ ref: 1 │ │ ref: 2 │ p2 └─────────┘ └─────────┘ ↓ p1 = p2 ↓ p1 = null A.refcount = 2 A.refcount = 1 ┌─────────┐ ┌─────────┐ │ Object A │ ◀─ p2 │ Object A │ ◀─ p2 │ ref: 2 │ │ ref: 1 │ └─────────┘ └─────────┘ ↓ p2 = null A.refcount = 0 → 回收! 
// Python的引用计数
import sys
a = []               # refcount = 1
b = a                # refcount = 2
sys.getrefcount(a)   # 返回 3 (getrefcount本身也引用了一次)
del b                # refcount = 1
del a                # refcount = 0 → 回收

// Swift自动引用计数 (ARC)
class Person {
    let name: String
    var apartment: Apartment?
    init(name: String) { self.name = name }
}
// 循环引用需要用weak/unowned打破
class Apartment {
    weak var tenant: Person?  // weak不增加引用计数
}

优缺点

优点缺点
确定性回收 (引用为0立即释放)循环引用问题
增量式开销 (无需暂停)计数器更新开销
内存局部性好多线程需要原子操作

4. 标记-清除 (Mark and Sweep)

分两阶段：标记 (Mark) 从根可达的对象，清除 (Sweep) 不可达的对象。

标记前: 标记后 (从Root遍历): Root Root │ │ ▼ ▼ [A] ──▶ [B] [A*]──▶[B*] (* = 标记为可达) │ │ ▼ ▼ [C] ──▶ [D] ←孤岛 [C*] [D ] ← D不可达, 待回收 
// 标记-清除算法伪代码
function markSweepGC() {
    // 阶段1: 标记
    markPhase();
    // 阶段2: 清除
    sweepPhase();
}

function markPhase() {
    for root in getRoots() {     // 根: 全局变量、栈变量、寄存器
        mark(root);
    }
}

function mark(obj) {
    if obj == null || obj.marked {
        return;
    }
    obj.marked = true;
    for child in obj.references {
        mark(child);             // 深度优先遍历
    }
}

function sweepPhase() {
    for obj in allObjects {
        if !obj.marked {
            free(obj);           // 回收不可达对象
        } else {
            obj.marked = false;  // 重置标记
        }
    }
}

5. 分代垃圾回收 (Generational GC)

基于弱分代假说：大多数对象很快死亡 (朝生夕灭)。

┌─────────────────────────────────────────────┐ │ 堆内存 │ │ ┌──────────────────┬──────────────────────┐ │ │ │ 年轻代 (Young) │ 老年代 (Old/Tenured) │ │ │ │ │ │ │ │ │ ┌───┬───┬───┐ │ │ │ │ │ │Eden│ S0│ S1│ │ 长期存活的对象 │ │ │ │ │ │(to)│(from)│ │ │ │ │ │ └───┴───┴───┘ │ │ │ │ │ │ │ │ │ │ Minor GC频繁 │ Major GC不频繁 │ │ │ └──────────────────┴──────────────────────┘ │ └─────────────────────────────────────────────┘ 对象生命周期: 新对象 → Eden (Minor GC存活) → S0/S1 (多次GC存活) → 老年代 
// JVM分代GC示例 (参数设置)
// 年轻代
-Xmn256m                    // 年轻代大小256MB
-XX:NewRatio=2              // 老年代:年轻代 = 2:1
-XX:SurvivorRatio=8         // Eden:Survivor = 8:1:1

// GC选择
-XX:+UseG1GC                // G1收集器 (默认)
-XX:+UseZGC                 // ZGC (低延迟)
-XX:+UseShenandoahGC        // Shenandoah (低延迟)

// Minor GC: 只回收年轻代, 速度快 (<10ms)
// Major/Full GC: 回收整个堆, 速度慢
// 大多数对象在Minor GC中就被回收了

6. 内存压缩 (Compaction)

解决内存碎片问题：将存活对象移动到一端，连续排列。

压缩前 (碎片化): 压缩后: [A][ ][B][ ][C][ ] [A][B][C][ 空闲大块 ] 碎片分散, 无法分配大对象 连续空间, 可以分配大对象
算法方式特点
标记-整理标记后向一端移动需要额外时间整理
拷贝算法将存活对象复制到新区域简单, 但需要双倍空间
滑动整理滑动存活对象填补空隙保持相对顺序
// 拷贝算法 (Copying GC)
// 使用两个半空间 (Semi-space)
function copyGC() {
    from = currentSpace;        // 当前空间
    to = otherSpace;            // 目标空间
    top = to.start;

    for root in roots {
        root = copy(root);      // 递归拷贝存活对象
    }

    swap(from, to);             // 交换空间角色
}

function copy(obj) {
    if obj.forwarding != null {
        return obj.forwarding;   // 已拷贝, 返回转发指针
    }
    newObj = allocate(to, obj.size);
    memcpy(newObj, obj, obj.size);
    obj.forwarding = newObj;    // 设置转发指针
    for child in newObj.references {
        child = copy(child);    // 递归拷贝子对象
    }
    return newObj;
}

7. 逃逸分析 (Escape Analysis)

编译器分析对象的作用域，判断对象是否"逃逸"出当前函数/线程，从而优化内存分配策略。

逃逸类型

类型说明优化
无逃逸对象只在函数内使用栈分配
参数逃逸对象传递给其他函数可能栈分配
全局逃逸对象被返回或赋值给全局变量必须堆分配
// Go语言逃逸分析
func noEscape() {
    x := 42       // 不逃逸 → 栈分配
    fmt.Println(x)
}

func escapes() *int {
    x := 42       // 逃逸! → 堆分配 (返回了指针)
    return &x
}
