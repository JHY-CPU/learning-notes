# 面向对象与函数式


## 一、面向对象编程 (OOP) 四大原则


### 1. 封装 (Encapsulation)


将数据和操作数据的方法绑定在一起，对外隐藏实现细节。


> **Example:** ```
> // Java 封装
> public class BankAccount {
>     private double balance;           // 私有数据
>
>     public void deposit(double amount) {
>         if (amount > 0) balance += amount;  // 受控访问
>     }
>     public double getBalance() { return balance; }
> }
> ```


### 2. 继承 (Inheritance)


子类继承父类的属性和方法，实现代码复用。


> **Example:** ```
> class Animal {
>     String name;
>     void speak() { System.out.println("..."); }
> }
> class Dog extends Animal {
>     @Override void speak() { System.out.println("Woof!"); }
>     void fetch() { System.out.println("Fetching..."); }
> }
> ```


### 3. 多态 (Polymorphism)


同一接口的不同实现，运行时根据实际类型调用相应方法。


> **Example:** ```
> Animal[] pets = { new Dog(), new Cat(), new Bird() };
> for (Animal pet : pets) {
>     pet.speak();  // 运行时决定调用哪个speak() — 多态！
> }
> ```


### 4. 抽象 (Abstraction)


定义接口和抽象类，关注"做什么"而非"怎么做"。


> **Example:** ```
> abstract class Shape {
>     abstract double area();  // 抽象方法：子类必须实现
> }
> class Circle extends Shape {
>     double radius;
>     double area() { return Math.PI * radius * radius; }
> }
> ```


## 二、SOLID设计原则


| 原则 | 全称 | 含义 |
| --- | --- | --- |
| **S** | 单一职责 | 一个类只负责一个功能 |
| **O** | 开闭原则 | 对扩展开放，对修改关闭 |
| **L** | 里氏替换 | 子类可替换父类使用 |
| **I** | 接口隔离 | 使用多个专用接口优于一个通用接口 |
| **D** | 依赖倒置 | 依赖抽象而非具体实现 |


## 三、函数式编程核心概念


### 纯函数与副作用


> **Example:** ```
> -- 纯函数（无副作用）
> add :: Int -> Int -> Int
> add x y = x + y
> -- 相同输入永远得到相同输出，不修改任何外部状态
>
> -- 有副作用的函数
> counter :: IORef Int -> IO Int
> counter ref = do
>     modifyIORef ref (+1)  -- 修改了外部状态！
>     readIORef ref
> ```


### 不可变性


```
# Python - 用不可变数据
from collections import namedtuple

Point = namedtuple('Point', ['x', 'y'])
p1 = Point(1, 2)
p2 = p1._replace(x=3)   # 创建新对象，p1不变
# p1 = Point(x=1, y=2), p2 = Point(x=3, y=2)

# 对比：可变数据
p = [1, 2]
p.append(3)   # 直接修改 — 命令式风格
```


### 高阶函数


```
// JavaScript 高阶函数
const twice = f => x => f(f(x));
const addOne = x => x + 1;
twice(addOne)(5);  // 7

// 函数返回函数
const multiplier = factor => x => x * factor;
const double = multiplier(2);
const triple = multiplier(3);
[1,2,3].map(double);  // [2,4,6]
```


## 四、Monad（单子）


Monad是函数式编程中处理**计算上下文**的抽象。


### Monad的定义


```
-- Haskell Monad类型类
class Monad m where
    return :: a -> m a                    -- 包装值
    (>>=)  :: m a -> (a -> m b) -> m b   -- 链接计算（bind）

-- 三个Monad法则（必须满足）：
-- 1. Left identity:  return a >>= f  ≡  f a
-- 2. Right identity: m >>= return    ≡  m
-- 3. Associativity:  (m >>= f) >>= g ≡  m >>= (\x -> f x >>= g)
```


### 常见Monad


| Monad | 上下文 | 用途 |
| --- | --- | --- |
| **Maybe** | 可能失败的计算 | 空值处理 |
| **List** | 非确定性计算 | 多值返回 |
| **IO** | 有副作用的计算 | 输入输出 |
| **State** | 有状态的计算 | 状态传递 |
| **Either** | 可能出错的计算 | 错误处理 |
| **Reader** | 依赖环境的计算 | 配置读取 |


### Maybe Monad示例


> **Example:** ```
> -- 链式安全计算，任何一步失败整个链返回Nothing
> safeDivide :: Double -> Double -> Maybe Double
> safeDivide _ 0 = Nothing
> safeDivide x y = Just (x / y)
>
> -- do记法（Monad语法糖）
> calculation :: Maybe Double
> calculation = do
>     a <- safeDivide 10 2     -- Just 5.0
>     b <- safeDivide a 0      -- Nothing（失败）
>     c <- safeDivide b 3      -- 不会执行
>     return c
> -- 结果：Nothing（因为第二步失败）
> ```


## 五、OOP vs FP 对比


### 面向对象 (OOP)


- 围绕
   **对象**
   和
   **状态**
   组织
- 数据和操作绑定
- 通过
   **继承**
   复用
- 可变状态是核心
- 适合：GUI、游戏、大型系统


### 函数式 (FP)


- 围绕
   **函数**
   和
   **数据变换**
   组织
- 数据和操作分离
- 通过
   **组合**
   复用
- 不可变性是核心
- 适合：数据处理、并发、DSL


### 同一问题的不同解法


> **Example:** **问题：**计算员工总薪资
>
>
> ```
> // OOP风格
> class EmployeeList {
>     private List employees;
>     public double totalSalary() {
>         double total = 0;
>         for (Employee e : employees) {
>             total += e.getSalary();
>         }
>         return total;
>     }
> }
>
> // FP风格
> totalSalary :: [Employee] -> Double
> totalSalary = sum . map salary
> -- 或：totalSalary employees = sum (map salary employees)
> ```


## 六、持久化不可变数据结构


不可变数据结构通过**结构共享**实现高效更新：


```
-- Haskell 持久化列表
xs = [1, 2, 3]
ys = 0 : xs      -- ys = [0, 1, 2, 3]，xs不变
-- ys和xs共享尾部 [1, 2, 3] 的内存

-- Clojure 持久化向量
(def v1 [1 2 3 4 5])
(def v2 (assoc v1 2 99))   -- v2 = [1 2 99 4 5]，v1不变
-- v1和v2共享大部分节点（哈希数组映射树）
```


### 各语言的不可变数据支持


| 语言 | 不可变数据支持 |
| --- | --- |
| Haskell | 所有数据默认不可变 |
| Clojure | 内置持久化数据结构 |
| Rust | 变量默认不可变，`mut`显式声明可变 |
| Java | `final`、不可变集合 (Collections.unmodifiableList) |
| JavaScript | `Object.freeze`、Immutable.js 库 |
| Python | `tuple`、`frozen set`、dataclass(frozen=True) |


## 七、OOP与FP的融合


现代编程趋势是**两种范式的融合**：


| 语言 | 融合方式 |
| --- | --- |
| **Scala** | 统一OOP和FP，类和函数都是一等公民 |
| **Kotlin** | OOP基础 + FP特性（lambda、高阶函数、数据类） |
| **Rust** | 无OOP继承，但有trait（类型类），强FP支持 |
| **Swift** | 协议导向编程（类似FP的类型类）+ 传统OOP |
| **C#** | OOP基础 + LINQ（函数式查询）+ record类型 |
| **Java 8+** | OOP基础 + Stream API + lambda + Optional |


> **Theorem:** **结论：**OOP和FP不是对立的，而是互补的。最好的实践是根据问题选择合适的工具——用OOP组织模块和接口，用FP处理数据变换和并发。关键是理解每种范式的核心理念和适用场景。
