# 02-实时操作系统

实时操作系统（Real-Time Operating System, RTOS）是嵌入式系统软件的核心平台，负责管理硬件资源并为应用程序提供确定性的任务调度与同步机制。本章系统介绍RTOS的核心概念、调度算法、同步机制以及主流RTOS平台。

---

## 1. 实时系统的定义与分类

### 1.1 什么是实时系统

实时系统的核心特征是**确定性**——必须在规定的时间截止期限（Deadline）内完成计算并给出正确结果。实时不等于"快"，而是"可预测"。

关键指标：
- **截止时间（Deadline）**：任务必须完成的最晚时间
- **最坏情况执行时间（WCET）**：任务在最坏条件下的执行时间
- **响应时间**：从事件发生到系统响应的时间

### 1.2 硬实时 vs 软实时

| 特性 | 硬实时系统 | 软实时系统 |
|------|-----------|-----------|
| 截止时间要求 | 绝对不可违反 | 尽量满足 |
| 违反后果 | 系统故障、安全灾难 | 性能降级、用户体验下降 |
| 典型应用 | 飞控系统、ABS制动、医疗设备 | 视频播放、游戏、语音通话 |
| 调度保证 | 需要形式化验证 | 统计性保证 |

### 1.3 实时系统的可调度性

一个实时系统是**可调度的**，当且仅当所有任务在其截止时间前完成。可调度性分析需要：
- 任务集参数（周期、执行时间、截止时间）
- 调度算法
- 系统开销（上下文切换、中断处理）

## 2. RTOS的核心组件

一个完整的RTOS通常包含以下核心组件：

```
┌──────────────────────────────────────────┐
│              应用程序                      │
├──────────────────────────────────────────┤
│  任务管理  │  调度器  │  同步与通信机制     │
├──────────────────────────────────────────┤
│         内存管理      │    中断处理        │
├──────────────────────────────────────────┤
│            硬件抽象层（HAL）               │
├──────────────────────────────────────────┤
│              硬件平台                     │
└──────────────────────────────────────────┘
```

- **任务管理**：任务创建、删除、挂起、恢复
- **调度器**：决定哪个任务在何时运行
- **同步机制**：信号量、互斥锁、事件标志
- **通信机制**：消息队列、邮箱、管道
- **内存管理**：静态分配、动态分配、内存池
- **时间管理**：系统节拍、延时、软件定时器
- **中断管理**：中断服务程序与任务的协调

## 3. 任务管理

### 3.1 任务（线程）的概念

任务是RTOS中最小的可调度执行单元。每个任务拥有：
- 独立的栈空间
- 独立的程序计数器和寄存器上下文
- 任务优先级
- 任务控制块（TCB）描述符

### 3.2 任务状态机

```
          创建任务          调度器选择
  创建 ──────────→ 就绪 ──────────→ 运行
  (Created)      (Ready)        (Running)
                   ↑    ┌──────────┤
                   │    │  时间片到/被抢占
                   │    ▼
                   │   就绪
                   │
                   │  事件发生/延时到
                   │
                 阻塞 ──────────→ 挂起
               (Blocked)        (Suspended)
                   ↑                 │
                   │    恢复          │
                   └─────────────────┘
```

- **就绪态**：等待调度器分配CPU
- **运行态**：正在占用CPU执行
- **阻塞态**：等待某个事件（信号量、消息、延时）
- **挂起态**：被显式挂起，不参与调度
- **删除态**：任务已被终止

### 3.3 任务控制块（TCB）

TCB是描述任务所有信息的数据结构：

```c
typedef struct TCB {
    uint32_t *stackPointer;    // 栈指针（上下文保存位置）
    uint32_t  stackBase;       // 栈底地址
    uint32_t  stackSize;       // 栈大小
    uint8_t   priority;        // 任务优先级
    uint8_t   state;           // 任务状态
    char      name[16];        // 任务名称
    uint32_t  tickCount;       // 延时计数
    // ... 链表指针、事件等待标志等
} TCB_t;
```

## 4. 调度算法

调度算法决定在任何时刻哪个任务获得CPU。实时调度算法分为静态优先级和动态优先级两大类。

### 4.1 速率单调调度（Rate Monotonic, RM）

静态优先级调度的经典算法：

- **规则**：周期越短的任务优先级越高
- **优点**：实现简单，开销小
- **可调度性条件**（Liu & Layland上界）：
  - 对于n个独立周期任务，若总利用率 U = Σ(Ci/Ti) ≤ n(2^(1/n) - 1)
  - n→∞时，上界约为 ln(2) ≈ 0.693
- **适用条件**：任务集满足周期等于截止时间、静态优先级

示例：
| 任务 | 周期(ms) | 执行时间(ms) | 优先级 |
|------|---------|-------------|--------|
| T1 | 10 | 2 | 最高 |
| T2 | 20 | 3 | 中 |
| T3 | 50 | 5 | 最低 |

利用率 U = 2/10 + 3/20 + 5/50 = 0.2 + 0.15 + 0.1 = 0.45 ≤ 0.779（3任务RM上界），可调度。

### 4.2 截止时间单调调度（Deadline Monotonic, DM）

- **规则**：截止时间越短，优先级越高
- **优势**：比RM更精确，适用于相对截止时间不等于周期的场景
- **RM是DM的特例**（当截止时间=周期时，两者等价）

### 4.3 最早截止时间优先（Earliest Deadline First, EDF）

动态优先级调度算法：

- **规则**：截止时间最早的任务获得最高优先级
- **优先级随时间动态变化**
- **可调度性条件**：U = Σ(Ci/Ti) ≤ 1（理论最优，利用率100%）
- **优点**：可调度性最强
- **缺点**：运行时开销大（需动态排序），过载时行为不确定

### 4.4 最小裕度优先（Least Slack Time First, LST）

动态优先级调度算法：

- **规则**：裕度（Slack）= 截止时间 - 剩余执行时间，裕度最小的任务优先级最高
- **优势**：在任务执行时间变化时表现更好
- **缺点**：实现复杂，需要精确估计剩余执行时间

### 4.5 调度算法对比

| 算法 | 类型 | 可调度上界 | 实现复杂度 | 适用场景 |
|------|------|-----------|-----------|---------|
| RM | 静态优先级 | 69.3% | 低 | 周期任务，最常用 |
| DM | 静态优先级 | ≥ RM | 低 | 相对截止时间≠周期 |
| EDF | 动态优先级 | 100% | 中 | 利用率要求高 |
| LST | 动态优先级 | 100% | 高 | 执行时间变化大 |

## 5. 优先级反转与优先级继承

### 5.1 优先级反转问题

当高优先级任务（H）等待低优先级任务（L）持有的资源时，若中优先级任务（M）抢占L，会导致H被无限期阻塞：

```
时间 →
H:  ░░░░░░░░░░░░░░░░░░░░░░ 等待资源...░░░░░░运行░░░
M:                        ░░抢占L░░░░运行░░░░░░░░░
L:  获取资源→运行░░░░░░░░░░被抢占░░░░░░░░释放资源→
```

著名的火星探路者（Mars Pathfinder）就曾遭遇此问题。

### 5.2 优先级继承协议（PIP）

当高优先级任务被阻塞在低优先级任务持有的互斥锁上时，临时提升低优先级任务的优先级至被阻塞任务的优先级：

- 减少中优先级任务的干扰
- 不能完全避免死锁（多个锁时）
- 阻塞时间有界但可能较长

### 5.3 优先级天花板协议（PCP）

每个资源预先设定一个"天花板优先级"（等于所有可能访问该资源的任务中的最高优先级）：

- 任务获取资源时，立即提升至天花板优先级
- 完全避免优先级反转和死锁
- 需要预先知道所有任务的资源使用情况
- 实现开销略大

## 6. 信号量与互斥锁

### 6.1 二值信号量

仅取0和1两个值，用于任务间同步或简单互斥：

- **P操作（take/pend）**：信号量值减1，若为0则阻塞
- **V操作（give/post）**：信号量值加1，唤醒等待任务

典型用途：中断通知任务（ISR中give，任务中take）

### 6.2 计数信号量

取值范围0-N，用于管理多个同类资源：

- 典型应用：缓冲区管理、连接池
- 初始值 = 可用资源数量

### 6.3 互斥锁（Mutex）

专用于互斥访问的信号量，具有**所有权**概念：

- 只有获取锁的任务才能释放锁
- 支持优先级继承
- 不能在ISR中使用

### 6.4 信号量 vs 互斥锁

| 特性 | 信号量 | 互斥锁 |
|------|--------|--------|
| 所有权 | 无（任何任务可释放） | 有（只有获取者可释放） |
| 用途 | 同步、计数 | 互斥访问共享资源 |
| ISR使用 | 可以（二值信号量） | 不可以 |
| 优先级继承 | 不支持 | 支持 |

## 7. 消息队列与邮箱

### 7.1 消息队列

消息队列是任务间传递数据的主要机制：

- **FIFO缓冲区**：存储固定大小的消息
- **阻塞/非阻塞发送与接收**
- **可设置超时时间**
- 支持多个生产者和消费者

```c
// FreeRTOS示例
QueueHandle_t xQueue = xQueueCreate(10, sizeof(uint32_t));
xQueueSend(xQueue, &data, portMAX_DELAY);  // 发送
xQueueReceive(xQueue, &data, portMAX_DELAY); // 接收
```

### 7.2 邮箱

邮箱是消息队列的简化形式，通常只能存储一条消息：

- 每个邮箱只有一个消息槽
- 新消息覆盖旧消息
- 适用于状态通知、参数传递

## 8. 内存管理

### 8.1 嵌入式内存管理的特殊性

- 不能使用标准malloc/free（碎片化、不可预测）
- 优先使用静态分配
- 需要确定性（最坏情况时间可预测）

### 8.2 内存池（Memory Pool）

预先分配固定大小的内存块：

```
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│ 块0 │ 块1 │ 块2 │ 块3 │ 块4 │ 块5 │ 块6 │ 块7 │  ← 固定大小块
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
  空闲   已用  空闲   已用  空闲  空闲  已用  空闲
```

- 分配和释放均为O(1)时间复杂度
- 无外部碎片
- 可能有内部碎片（块大小 > 实际需求）

### 8.3 FreeRTOS内存管理方案

FreeRTOS提供5种heap实现（heap_1到heap_5）：

- **heap_1**：只分配不释放，适合确定性场景
- **heap_2**：最佳适配，不合并相邻空闲块
- **heap_3**：封装标准malloc/free（加互斥锁保护）
- **heap_4**：首次适配，合并相邻空闲块（推荐）
- **heap_5**：支持非连续内存区域

## 9. FreeRTOS基础

### 9.1 任务创建

```c
void vTaskFunction(void *pvParameters) {
    for (;;) {
        // 任务主循环
        vTaskDelay(pdMS_TO_TICKS(100)); // 延时100ms
    }
}

// 创建任务
xTaskCreate(
    vTaskFunction,    // 任务函数
    "MyTask",         // 任务名称
    256,              // 栈深度（字）
    NULL,             // 参数
    2,                // 优先级
    NULL              // 任务句柄
);
```

### 9.2 调度策略

- **抢占式调度**：高优先级就绪任务立即抢占低优先级任务
- **协作式调度**：任务主动放弃CPU（通过yield或阻塞）
- **时间片轮转**：同优先级任务之间轮流执行

### 9.3 常用API

| 功能 | API |
|------|-----|
| 任务延时 | `vTaskDelay()` / `vTaskDelayUntil()` |
| 信号量 | `xSemaphoreCreateBinary()` / `xSemaphoreTake()` / `xSemaphoreGive()` |
| 互斥锁 | `xSemaphoreCreateMutex()` |
| 消息队列 | `xQueueCreate()` / `xQueueSend()` / `xQueueReceive()` |
| 软件定时器 | `xTimerCreate()` / `xTimerStart()` |
| 事件组 | `xEventGroupCreate()` / `xEventGroupSetBits()` |

## 10. RT-Thread简介

RT-Thread是一个来自中国的开源RTOS，具有丰富的组件生态：

- **内核**：完全抢占式调度，支持优先级继承
- **组件**：文件系统（FatFS、LittleFS）、网络协议栈（lwIP）、GUI（LVGL）
- **设备驱动框架**：统一的设备接口（rt_device）
- **包管理器（Env/Packages）**：丰富的第三方软件包
- **RT-Thread Nano**：最小3KB Flash占用，适合资源极受限的MCU
- **开发工具**：RT-Thread Studio IDE

## 11. μC/OS-II/III简介

Micrium公司开发的商业级RTOS：

- **μC/OS-II**：最多256个任务，代码量小（5-24KB ROM）
- **μC/OS-III**：任务数无限制，支持时间片轮转
- 通过了安全认证（DO-178B、IEC 61508）
- 代码规范，注释详尽，适合学习
- 现已开源（Apache 2.0许可）

## 12. 嵌入式Linux基础

对于需要完整操作系统功能的嵌入式场景，嵌入式Linux是重要选择：

### 12.1 构建系统

| 工具 | 特点 |
|------|------|
| **BusyBox** | 将常用Unix工具集成到单一可执行文件 |
| **Buildroot** | 简单的交叉编译构建系统，适合小型系统 |
| **Yocto Project** | 元数据驱动的构建框架，适合定制化大型系统 |
| **OpenWrt** | 专为路由器和嵌入式设备优化的Linux发行版 |

### 12.2 与RTOS的对比

| 方面 | RTOS | 嵌入式Linux |
|------|------|------------|
| 实时性 | 硬实时保证 | 需要PREEMPT_RT补丁 |
| 资源需求 | KB级ROM/RAM | MB级ROM/RAM |
| 生态 | 轻量级 | 完整Linux生态 |
| 启动速度 | 毫秒级 | 秒级 |
| 适用场景 | 实时控制 | 复杂应用、网关 |

## 总结

RTOS是嵌入式系统的灵魂。理解任务调度、同步机制、优先级管理是开发可靠嵌入式软件的关键。从FreeRTOS入手学习，逐步深入调度理论和形式化验证方法，是掌握嵌入式实时系统的有效路径。对于复杂的物联网网关和边缘计算场景，嵌入式Linux提供了更丰富的功能和生态支持。
