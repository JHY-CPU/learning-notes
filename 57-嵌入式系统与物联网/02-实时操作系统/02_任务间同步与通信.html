任务间同步与通信

一、同步与通信机制概述

在RTOS多任务系统中，任务之间需要协调执行顺序（同步）和交换数据（通信）。正确的同步与通信机制是系统稳定运行的基础。

1.1 机制分类

类别机制主要用途
同步信号量资源计数、事件通知
互斥量临界区保护
事件标志组多条件组合等待
通信消息队列任务间传递数据
邮箱传递固定大小消息

二、信号量（Semaphore）

信号量是最基本的同步机制，可以理解为一个计数器，任务通过获取（Take）和释放（Give）操作来协调执行。

2.1 信号量类型

类型计数范围典型用途
二值信号量0 或 1任务同步、事件通知
计数信号量0 ~ N资源池管理（如N个缓冲区）
互斥信号量0 或 1（带优先级继承）临界区保护

2.2 信号量操作

信号量操作语义：

Give（释放/V操作）：
信号量值 + 1。如果值 > 0，等待的任务可以获取信号量。

Take（获取/P操作）：
如果信号量值 > 0，则值 - 1，任务继续执行。
如果信号量值 = 0，任务被阻塞等待。

超时机制：Take操作通常支持超时参数，避免无限等待。

2.3 二值信号量同步示例

// 中断中通知任务处理数据
SemaphoreHandle_t xDataReadySem;

void vTaskDataHandler(void *pvParam) {
    for (;;) {
        // 等待信号量（阻塞等待）
        if (xSemaphoreTake(xDataReadySem, portMAX_DELAY) == pdTRUE) {
            // 处理数据
            processSensorData();
        }
    }
}

// UART接收中断回调
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
    BaseType_t xHigherPriorityTaskWoken = pdFALSE;
    xSemaphoreGiveFromISR(xDataReadySem, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}

三、互斥量（Mutex）

互斥量是一种特殊的二值信号量，具有"所有权"概念：谁获取了互斥量，就必须由谁释放。这是防止优先级反转的关键机制。

3.1 信号量 vs 互斥量

特性信号量互斥量
所有权无（任何任务都可释放）有（获取者才能释放）
优先级继承不支持支持
递归获取不支持支持递归互斥量
用途任务同步、事件通知临界区保护

3.2 优先级反转问题

优先级反转是实时系统中的经典问题：低优先级任务持有资源，高优先级任务等待，而中优先级任务抢占低优先级任务，导致高优先级任务被间接阻塞。

优先级反转场景：
1. 低优先级任务L获取了互斥量
2. 高优先级任务H需要同一个互斥量，被阻塞
3. 中优先级任务M就绪，抢占L执行
4. 结果：H在等待M完成，而M的优先级实际低于H

解决方案 — 优先级继承：
当H等待L持有的互斥量时，L的优先级临时提升到H的级别，防止M抢占L，使L尽快释放互斥量。

3.3 递归互斥量

递归互斥量允许同一个任务多次获取同一个互斥量，每次获取需要对应一次释放。适合需要嵌套调用的场景。

// 递归互斥量使用
SemaphoreHandle_t xRecursiveMutex = xSemaphoreCreateRecursiveMutex();

void vFunctionA(void) {
    xSemaphoreTakeRecursive(xRecursiveMutex, portMAX_DELAY);
    // 临界区A
    vFunctionB(); // FunctionB也获取同一互斥量，不会死锁
    xSemaphoreGiveRecursive(xRecursiveMutex);
}

void vFunctionB(void) {
    xSemaphoreTakeRecursive(xRecursiveMutex, portMAX_DELAY);
    // 临界区B
    xSemaphoreGiveRecursive(xRecursiveMutex);
}

四、消息队列（Message Queue）

消息队列是任务间通信的核心机制，支持先进先出（FIFO）的数据传递，可以传递任意大小的数据。

4.1 队列特性

  • 数据复制：发送时将数据复制到队列缓冲区（非引用传递）
  • 阻塞机制：发送和接收都支持超时阻塞
  • 多生产者/消费者：多个任务可以向同一队列发送或接收
  • 中断安全：提供FromISR版本在中断中使用

4.2 队列操作模式

操作普通覆盖写入（ToFront）
xQueueSend追加到队尾—
xQueueSendToBack追加到队尾—
xQueueSendToFront插入到队头—
xQueueOverwrite—覆盖队列中唯一的元素

4.3 队列使用示例

// 定义消息结构体
typedef struct {
    uint8_t type;
    uint16_t length;
    uint8_t data[64];
} Message_t;

// 创建队列
QueueHandle_t xMsgQueue = xQueueCreate(10, sizeof(Message_t));

// 发送任务
void vSenderTask(void *pvParam) {
    Message_t msg;
    msg.type = 1;
    msg.length = sprintf((char*)msg.data, "Sensor: 25.6C");
    xQueueSend(xMsgQueue, &msg, portMAX_DELAY);
}

// 接收任务
void vReceiverTask(void *pvParam) {
    Message_t rxMsg;
    for (;;) {
        if (xQueueReceive(xMsgQueue, &rxMsg, pdMS_TO_TICKS(1000)) == pdTRUE) {
            // 处理消息
            printf("Type=%d, Data=%s\n", rxMsg.type, rxMsg.data);
        } else {
            // 超时处理
            printf("Queue timeout\n");
        }
    }
}

4.4 队列集（Queue Set）

队列集允许任务同时等待多个队列/信号量中的任意一个就绪，类似于select/poll机制。

使用xQueueCreateSet()创建队列集，用xQueueAddToSet()将队列和信号量加入集合，任务通过xQueueSelectFromSet()等待任意一个就绪。

五、事件标志组（Event Group）

事件标志组允许任务等待多个事件的任意组合，每个事件对应一个位（bit），非常适合多条件触发场景。

5.1 核心操作

操作说明
xEventGroupSetBits设置事件位（触发事件）
xEventGroupClearBits清除事件位
xEventGroupWaitBits等待指定事件位（AND/OR逻辑）
xEventGroupGetBits获取当前事件位状态

5.2 与/或等待

等待模式：
- pdTRUE（wait for all bits）：等待所有指定的事件位都被设置（AND逻辑）
- pdFALSE（wait for any bit）：等待任意一个指定的事件位被设置（OR逻辑）

清除模式：
- xClearOnExit = pdTRUE：退出等待时自动清除匹配的事件位 
// 事件标志组示例：等待WiFi连接和传感器就绪
EventGroupHandle_t xSystemEvents;
#define WIFI_CONNECTED_BIT  (1 << 0)
#define SENSOR_READY_BIT    (1 << 1)
#define ALL_BITS (WIFI_CONNECTED_BIT | SENSOR_READY_BIT)

void vAppTask(void *pvParam) {
    // 等待两个条件同时满足
    EventBits_t bits = xEventGroupWaitBits(
        xSystemEvents,  // 事件组
        ALL_BITS,       // 等待的位
        pdTRUE,         // 退出时清除
        pdTRUE,         // AND模式（等待所有位）
        portMAX_DELAY   // 无限等待
    );
    // 两个条件都已满足，开始工作
}

六、软件定时器

软件定时器在定时器服务任务的上下文中执行回调函数，可以实现定时触发或周期性触发的功能。

6.1 定时器类型

类型说明API
一次性定时器触发一次后自动停止xTimerStart()
自动重载定时器周期性重复触发xTimerStart()，period不为0
注意：软件定时器的精度受定时器服务任务优先级影响。如果服务任务优先级不够高，回调可能延迟执行。对精度要求高的场景应使用硬件定时器。

七、同步通信机制选择指南

需求推荐机制原因
保护共享资源互斥量支持优先级继承，防止优先级反转
中断通知任务二值信号量简单高效，中断Give任务Take
管理资源池计数信号量自动计数可用资源数
任务间传递数据消息队列支持数据复制，类型安全
多条件组合触发事件标志组AND/OR等待多个事件
定时触发软件定时器方便的定时/周期触发
