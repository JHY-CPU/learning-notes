HTTPS与物联网安全

一、物联网安全挑战

物联网设备通常资源受限、部署分散、长期无人值守，面临的安全威胁比传统IT设备更加严峻。安全必须从设备端到云端全链路考虑。

1.1 主要安全威胁

威胁类别具体威胁影响
通信安全窃听、中间人攻击、重放攻击数据泄露、伪造指令
设备安全固件提取、调试接口攻击、侧信道攻击知识产权泄露、密钥提取
网络安全DDoS攻击（如Mirai僵尸网络）服务中断、设备被控
数据安全数据篡改、隐私泄露信任破坏、合规风险

1.2 安全架构层次

物联网安全分层：
- 设备层：安全启动、硬件安全模块（HSM/TPM）、安全存储
- 通信层：TLS/DTLS加密、证书认证
- 云端层：设备身份管理、访问控制、安全监控
- 应用层：数据加密、访问审计、固件签名验证

二、TLS/SSL（传输层安全）

TLS是互联网上最广泛使用的加密协议，在TCP之上提供加密、认证和完整性保护。MQTT、HTTP等协议都依赖TLS保障传输安全。

2.1 TLS握手过程

  1. ClientHello：客户端发送支持的TLS版本、密码套件列表、客户端随机数
  2. ServerHello：服务器选择密码套件，发送服务器随机数
  3. Certificate：服务器发送其X.509证书
  4. ServerKeyExchange：服务器发送密钥交换参数（ECDHE等）
  5. CertificateRequest：（可选）服务器要求客户端证书
  6. CertificateVerify：客户端验证服务器证书，发送客户端证书
  7. 密钥计算：双方使用预主密钥计算会话密钥
  8. Finished：双方发送加密的Finished消息，握手完成

2.2 IoT适用的密码套件

密码套件密钥交换对称加密适用场景
TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256ECDHEAES-128-GCM推荐，高效安全
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256ECDHEAES-128-GCMRSA证书兼容性好
TLS_PSK_WITH_AES_128_CCM预共享密钥AES-128-CCM资源极受限设备
IoT TLS优化：
- 使用ECC（椭圆曲线）而非RSA：密钥更短（256位ECC ≈ 3072位RSA）
- 使用预共享密钥（PSK）：跳过证书验证，减少计算和带宽
- TLS会话恢复：减少握手开销
- 推荐TLS 1.2+，TLS 1.3更优（更少握手往返）

三、DTLS（数据报传输层安全）

DTLS是TLS的UDP版本，为基于UDP的协议（如CoAP）提供加密保护。由于UDP不可靠、无序，DTLS需要额外处理丢包和乱序。

3.1 DTLS vs TLS

特性TLSDTLS
传输层TCPUDP
可靠性TCP保证DTLS自己处理
握手顺序可靠需处理丢包和重传
加密方式流加密块加密（添加sequence number防重放）
适用协议MQTT、HTTPSCoAP、LwM2M

3.2 DTLS防重放攻击

重放攻击防护：
DTLS在每个记录中包含一个sequence number，并使用滑动窗口检测重复记录。
接收方维护一个位图窗口，记录已接收的sequence number范围。收到超出窗口范围的记录时，更新窗口；收到窗口内的重复记录时，丢弃。

四、证书与密钥管理

4.1 X.509证书体系

X.509是公钥证书的标准格式，用于在TLS中验证通信双方的身份。

  • 根CA证书：自签名，预置在设备或操作系统中
  • 中间CA证书：由根CA签发，可以有多级
  • 设备证书：由CA签发，包含设备的公钥和身份信息

4.2 证书配置方式

方式说明优点缺点
工厂预置生产时烧录到设备安全可靠不可更换
引导配置首次上电从服务器获取灵活首次连接不安全
证书更新定期更新证书长期安全需安全通道

4.3 证书验证流程

  1. 检查证书有效期（notBefore ~ notAfter）
  2. 验证证书链（设备证书 → 中间CA → 根CA）
  3. 检查证书吊销状态（CRL或OCSP）
  4. 验证Subject Alternative Name匹配服务器域名
  5. 检查证书扩展（Key Usage等）

五、固件安全与OTA更新

固件OTA（Over-The-Air）更新是IoT设备生命周期管理的关键能力。安全的OTA机制需要保证固件来源可信、传输加密、完整性校验。

5.1 安全OTA流程

  1. 固件签名：开发人员用私钥对固件镜像签名
  2. 安全传输：通过TLS/HTTPS将固件下载到设备
  3. 签名验证：设备用预置的公钥验证固件签名
  4. 完整性校验：计算SHA-256哈希与预期值对比
  5. A/B分区切换：写入非活动分区，验证后切换启动
  6. 回滚保护：版本号单调递增，拒绝旧版本
// 固件验证伪代码
int verify_firmware(uint8_t *fw_data, size_t fw_len,
                    uint8_t *signature, size_t sig_len) {
    // 1. 计算固件SHA-256哈希
    uint8_t hash[32];
    sha256(fw_data, fw_len, hash);

    // 2. 用公钥验证RSA/ECDSA签名
    if (ecdsa_verify(public_key, hash, 32, signature, sig_len) != 0) {
        return ERR_SIGNATURE_INVALID;
    }

    // 3. 检查版本号（防回滚）
    uint32_t new_version = get_firmware_version(fw_data);
    uint32_t current_version = get_running_version();
    if (new_version <= current_version) {
        return ERR_VERSION_ROLLBACK;
    }

    return SUCCESS;
}

5.2 A/B分区方案

A/B分区机制：
Flash中划分为两个固件分区（A和B）。当前运行A分区时，OTA下载到B分区。下载完成后，重启引导到B分区。如果B分区启动失败，引导加载程序自动回退到A分区，确保设备不会变砖。

六、安全启动（Secure Boot）

安全启动确保设备只执行经过验证的固件代码，防止恶意固件的注入。它建立从硬件信任根到操作系统启动的信任链。

6.1 信任链

  1. 硬件信任根（Root of Trust）：芯片内不可修改的ROM代码或熔断密钥
  2. 第一级引导（BL1/ROM）：验证第二级引导加载程序的签名
  3. 第二级引导（BL2/U-Boot）：验证操作系统内核的签名
  4. 操作系统：内核验证应用程序/驱动的签名
安全启动失败处理：
如果任何一级签名验证失败，启动过程终止。设备可能进入恢复模式、锁定调试接口、或完全拒绝启动。具体行为取决于安全策略配置。
