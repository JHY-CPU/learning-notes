# 03-物联网协议

物联网（Internet of Things, IoT）通过各种通信协议将物理世界的传感器、执行器与数字世界连接起来。协议的选择直接影响系统的功耗、可靠性、实时性和覆盖范围。本章系统介绍物联网中常用的通信协议。

---

## 1. 物联网架构

### 1.1 三层架构模型

```
┌─────────────────────────────────────────────┐
│              应用层（Application）             │
│  智能家居 / 工业监控 / 智慧农业 / 车联网       │
├─────────────────────────────────────────────┤
│              网络层（Network）                 │
│  路由 / 交换 / 协议转换 / 网关 / 云连接        │
├─────────────────────────────────────────────┤
│              感知层（Perception）              │
│  传感器 / 执行器 / RFID / 嵌入式设备           │
└─────────────────────────────────────────────┘
```

### 1.2 各层职责

- **感知层**：数据采集（温度、湿度、位置等）和物理控制（开关、电机等）
- **网络层**：数据传输、路由、网关汇聚、协议转换
- **应用层**：数据处理、业务逻辑、用户界面、云平台集成

### 1.3 协议栈全景

| 层次 | 有线协议 | 短距离无线 | 长距离无线 | 互联网协议 |
|------|---------|-----------|-----------|-----------|
| 应用层 | Modbus | — | — | MQTT/CoAP/HTTP |
| 传输层 | — | — | — | TCP/UDP |
| 网络层 | — | ZigBee/Thread | LoRaWAN/NB-IoT | IPv6/6LoWPAN |
| 物理层 | Ethernet/RS485 | BLE/ZigBee | LoRa/NB-IoT | WiFi/5G |

## 2. MQTT协议

MQTT（Message Queuing Telemetry Transport）是物联网中最广泛使用的应用层协议。

### 2.1 发布/订阅模型

```
    Publisher          Broker           Subscriber
    ┌──────┐      ┌──────────┐      ┌──────────┐
    │ 发布 │ ──→ │  消息代理  │ ──→ │  订阅接收  │
    │ 消息 │     │  (路由分发) │     │  消息     │
    └──────┘      └──────────┘      └──────────┘
                      ↑     ↓
                  ┌──────────┐
                  │  订阅/取消  │
                  └──────────┘
```

- **Broker**：消息代理服务器，负责消息路由
- **Topic**：主题，使用层级结构（如 `home/livingroom/temperature`）
- **通配符**：`+` 匹配单层，`#` 匹配多层
- **轻量级**：最小2字节固定头部，适合带宽受限场景
- **基于TCP**：默认端口1883（加密端口8883）

### 2.2 QoS级别

| QoS | 含义 | 保证机制 | 开销 |
|-----|------|---------|------|
| 0 | 最多一次 | 不保证送达，不重试 | 最低 |
| 1 | 至少一次 | PUBACK确认，可能重复 | 中等 |
| 2 | 恰好一次 | 四次握手（PUBLISH→PUBREC→PUBREL→PUBCOMP） | 最高 |

### 2.3 遗嘱消息（Last Will and Testament, LWT）

客户端连接时设置遗嘱消息，当客户端异常断开（网络故障、进程崩溃）时，Broker自动发布遗嘱消息到指定Topic，通知其他客户端该设备已离线。

### 2.4 保留消息（Retained Message）

发布时设置保留标志，Broker保存最后一条保留消息。新订阅者订阅该Topic时，立即收到最新状态，无需等待。

### 2.5 MQTT 5.0新特性

- 用户属性（User Properties）
- 共享订阅（Shared Subscriptions）
- 主题别名（Topic Alias）减少传输开销
- 请求-响应模式（Request-Response）
- 会话过期间隔

## 3. CoAP协议

CoAP（Constrained Application Protocol）是专为受限设备设计的应用层协议。

### 3.1 RESTful设计

CoAP遵循REST架构风格：

| 方法 | 说明 | 对应HTTP |
|------|------|---------|
| GET | 读取资源 | GET |
| POST | 创建/更新资源 | POST |
| PUT | 更新/替换资源 | PUT |
| DELETE | 删除资源 | DELETE |

### 3.2 特点

- **基于UDP**：开销远小于HTTP/TCP
- **消息模型**：Confirmable (CON) 和 Non-confirmable (NON)
- **请求-响应模式**：通过Token匹配请求与响应
- **观察机制（Observe）**：客户端可订阅资源变化通知
- **块传输**：支持大数据分块传输
- **URI支持**：coap://host/path?query

### 3.3 DTLS安全

由于基于UDP，CoAP使用DTLS（Datagram Transport Layer Security）提供安全性：

- 预共享密钥（PSK）：适合资源极度受限设备
- 原始公钥（RPK）：减少证书开销
- 证书模式（X.509）：最高安全级别
- OSCORE：CoAP专用对象安全层，端到端加密

## 4. HTTP/HTTPS在IoT中的应用

HTTP是最通用的互联网协议，在IoT中有特定用途：

- **适用场景**：网关到云端、固件OTA更新、RESTful API调用
- **优点**：通用性强，防火墙穿透好，工具链成熟
- **缺点**：头部开销大（数百字节），无状态，不适合实时双向通信
- **HTTPS**：TLS加密，增加握手开销（至少2-RTT）
- **HTTP/2**：多路复用、头部压缩，减少连接数

IoT中的轻量替代：使用JSON或CBOR（Concise Binary Object Representation）作为消息格式，减小负载。

---

## 5. WebSocket在IoT中的应用

WebSocket提供全双工双向通信：

- **握手**：HTTP Upgrade机制建立WebSocket连接
- **数据帧**：轻量级帧头部（最小2字节）
- **保持连接**：长连接，避免频繁TCP握手
- **适用场景**：Web控制面板、实时数据流、设备-浏览器直连

在IoT中常用于：
- 浏览器实时查看传感器数据
- 云端向设备下发控制指令
- 替代MQTT over WebSocket（MQTT也可封装在WebSocket中传输）

## 6. 蓝牙（Bluetooth）

### 6.1 经典蓝牙（Bluetooth Classic）

- 用于音频传输（A2DP）、文件传输、串口替代（SPP）
- 数据速率可达3Mbps（EDR）
- 功耗较高，持续连接

### 6.2 低功耗蓝牙（BLE, Bluetooth Low Energy）

BLE是物联网的核心无线技术之一：

- **连接模式**：Central ↔ Peripheral
- **广播模式**：Peripheral广播，Central扫描（适合Beacon）
- **功耗极低**：纽扣电池可运行数月到数年
- **GATT（通用属性配置文件）**：
  - Service → Characteristic → Value/Descriptor
  - 标准服务：心率、电池、环境传感器等
- **速率**：125kbps（长距离）到2Mbps（高速模式）
- **距离**：Class 2约10米，BLE 5.0长距离模式可达数百米

### 6.3 BLE 5.0+新特性

- **2Mbps PHY**：传输速率翻倍
- **长距离模式（Coded PHY）**：编码方案提升灵敏度
- **广播扩展**：广播数据量从31字节扩展到255字节
- **Mesh组网**：支持大规模Mesh网络

## 7. ZigBee（IEEE 802.15.4）

ZigBee是低功耗无线网状网络协议：

### 7.1 网络拓扑

```
    协调器（Coordinator）         协调器
        │                           │
    ┌───┴───┐                    ┌──┴──┐
  路由器  路由器               路由器  路由器
   │  ╲    │  ╲                ╱ │    ╱ │
  终端 终端 终端 终端    终端 终端 终端 终端

   星形拓扑               网状拓扑（Mesh）
```

### 7.2 关键特性

- **物理层**：IEEE 802.15.4，2.4GHz全球通用（16信道），另有868MHz/915MHz
- **数据速率**：250kbps（2.4GHz）
- **网络容量**：单网络最多65535个节点
- **安全**：AES-128加密
- **应用层**：ZigBee Cluster Library (ZCL) 定义标准设备类型

### 7.3 典型应用

- 智能照明（飞利浦Hue）
- 智能家居（ZigBee 3.0）
- 工业传感器网络
- 楼宇自动化

## 8. LoRa与LoRaWAN

### 8.1 LoRa（物理层）

LoRa是Semtech公司的长距离低功耗无线调制技术：

- **调制方式**：CSS（Chirp Spread Spectrum）扩频调制
- **频段**：ISM频段（433MHz、868MHz、915MHz等免授权频段）
- **传输距离**：城市2-5km，郊区15km+，视距可达数十公里
- **数据速率**：0.3kbps - 50kbps（距离与速率权衡）
- **功耗极低**：接收电流约10mA，休眠微安级

### 8.2 LoRaWAN（网络层）

LoRaWAN在LoRa物理层之上定义网络协议：

- **设备分类**：
  - Class A：最低功耗，仅在发送后短暂开接收窗口
  - Class B：定时开接收窗口，平衡功耗和延迟
  - Class C：持续监听，最低延迟，功耗最高
- **自适应数据速率（ADR）**：网络根据链路质量自动调整速率
- **安全**：AES-128端到端加密（网络层 + 应用层双密钥）

### 8.3 典型应用

- 智慧城市（停车、照明、环境监测）
- 智慧农业（土壤、气象监测）
- 资产追踪
- 智能表计（水表、气表远程抄表）

---

## 9. NB-IoT（窄带物联网）

NB-IoT是3GPP标准的蜂窝物联网技术：

- **频段**：利用LTE频谱的窄带资源（200kHz带宽）
- **数据速率**：上行约62.5kbps，下行约250kbps
- **深度覆盖**：比GSM增强20dB，可穿透地下室
- **海量连接**：单基站支持5万+设备
- **低功耗**：PSM（省电模式）+ eDRX（扩展不连续接收），电池寿命可达10年
- **安全性**：继承蜂窝网络安全机制（SIM认证、加密）

### NB-IoT vs LoRaWAN

| 对比项 | NB-IoT | LoRaWAN |
|--------|--------|---------|
| 频谱 | 授权频段 | 免授权频段 |
| 覆盖 | 运营商基站 | 自建网关 |
| 成本 | 模组较贵，流量费 | 模组便宜，无流量费 |
| 部署 | 即插即用 | 需部署网关 |
| 适用 | 需运营商覆盖的场景 | 私有网络、偏远地区 |

## 10. WiFi（IEEE 802.11）

WiFi是物联网中高速数据传输的主要选择：

### 10.1 主要变体

| 标准 | 名称 | 频段 | 最大速率 |
|------|------|------|---------|
| 802.11b | — | 2.4GHz | 11Mbps |
| 802.11g | — | 2.4GHz | 54Mbps |
| 802.11n | WiFi 4 | 2.4/5GHz | 600Mbps |
| 802.11ac | WiFi 5 | 5GHz | 6.9Gbps |
| 802.11ax | WiFi 6 | 2.4/5/6GHz | 9.6Gbps |

### 10.2 IoT中的WiFi应用

- **智能家居网关**：连接互联网
- **IP摄像头**：高清视频流传输
- **智能音箱**：语音交互
- **低功耗WiFi（802.11ah HaLow）**：Sub-1GHz频段，适合IoT

### 10.3 WiFi的IoT挑战

- 功耗较高（相对于BLE、ZigBee）
- 2.4GHz频段拥挤
- 需要路由器/AP基础设施

## 11. Thread与Matter协议

### 11.1 Thread

Thread是基于IEEE 802.15.4和6LoWPAN的Mesh网络协议：

- **IP原生**：每个设备有IPv6地址
- **自愈Mesh**：无单点故障
- **低功耗**：适合电池供电设备
- **安全**：AES-CCM加密，设备认证
- **边界路由器**：连接Thread网络与互联网

### 11.2 Matter（原Project CHIP）

Matter是智能家居互操作性标准：

- **目标**：解决智能家居碎片化，统一不同品牌设备的互操作
- **底层**：支持Thread、WiFi、以太网
- **应用层**：基于IPv6的端到端通信
- **安全**：设备认证、数据加密
- **支持者**：Apple、Google、Amazon、Samsung等联合推动
- **设备类型**：照明、开关、传感器、门锁、窗帘等

## 12. 边缘计算基础

边缘计算将数据处理从云端下沉到靠近数据源的设备或网关：

### 12.1 优势

- **低延迟**：本地处理，毫秒级响应
- **节省带宽**：只上传有价值的数据
- **隐私保护**：敏感数据不上传云端
- **离线可用**：断网时仍可执行关键逻辑

### 12.2 边缘计算架构

```
云端 ←──────── 边缘网关 ←──────── 终端设备
                    │
              ┌─────┴─────┐
              │ 数据预处理   │
              │ 规则引擎    │
              │ 本地推理    │
              │ 缓存/缓冲   │
              └───────────┘
```

### 12.3 边缘计算框架

- **AWS Greengrass**：AWS云服务的边缘扩展
- **Azure IoT Edge**：Azure云的边缘运行时
- **EdgeX Foundry**：Linux基金会开源边缘平台
- **KubeEdge**：Kubernetes边缘计算扩展

## 总结

物联网协议栈层次丰富，从物理层的LoRa、BLE到应用层的MQTT、CoAP，每种协议都有其适用场景。选择协议时需综合考虑功耗、带宽、距离、延迟、安全和成本等因素。Matter等新兴标准正在推动智能家居的互操作性，边缘计算则将智能下沉到设备端，减少对云端的依赖。掌握这些协议的原理和适用场景，是设计可靠物联网系统的基础。
