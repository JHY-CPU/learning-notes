# 缓存一致性协议详解

第一部分：缓存一致性问题

## 一、缓存一致性问题概述


在多处理器系统中，每个处理器都有自己的私有缓存。当多个处理器缓存同一内存地址的数据时，如何保证所有处理器看到的该地址数据值一致，这就是缓存一致性（Cache Coherence）问题。


### 1.1 一致性的三个核心要求


1. **写传播（Write Propagation）**
   ：一个处理器对某位置的写操作最终必须对其他处理器可见
2. **事务串行化（Transaction Serialization）**
   ：所有处理器看到的读/写操作顺序必须一致
3. **写序列化（Write Serialization）**
   ：对同一位置的写操作必须被所有处理器以相同顺序观察到


### 1.2 一致性问题的产生场景


```
初始状态: 地址 X 的值 = 5, 两个处理器 P1 和 P2 都缓存了 X

  ┌──────────┐          ┌──────────┐
  │   P1     │          │   P2     │
  │ 缓存:X=5 │          │ 缓存:X=5 │
  └────┬─────┘          └────┬─────┘
       │                     │
       │ P1 写 X = 10        │ 读取 X
       │                     │
  ┌────┴─────┐          ┌────┴─────┐
  │ 缓存:X=10│          │ 缓存:X=5 │ ← 不一致！
  └──────────┘          └──────────┘

P2 读到旧值 X=5，产生了不一致！
```

第二部分：MESI 协议

## 二、MESI 协议（四状态协议）


MESI 协议是 Intel 处理器广泛使用的缓存一致性协议，每个缓存行维护 4 种状态之一：


### 2.1 四种状态定义


| 状态 | 全称 | 含义 | 数据是否有效 | 是否与内存一致 | 是否独占 |
| --- | --- | --- | --- | --- | --- |
| **M** | Modified | 缓存行已被修改（脏数据） | 是 | 否（内存过时） | 是（唯一副本） |
| **E** | Exclusive | 缓存行与内存一致且独占 | 是 | 是 | 是（唯一副本） |
| **S** | Shared | 缓存行与内存一致且可共享 | 是 | 是 | 否（多份副本） |
| **I** | Invalid | 缓存行无效（未命中或已被作废） | 否 | — | — |


### 2.2 MESI 状态转换图


```
本地写命中
                     ┌──────────────────┐
                     │                  ▼
              ┌──────────┐  本地读    ┌──────────┐  远程读
    初始 ───▶│ Invalid  │◀─────────▶│  Shared  │◀───────┐
              │   (I)    │  总线写回  │   (S)    │        │
              └──┬───┬───┘           └──┬───┬───┘        │
       本地读   │   │ 远程读/写       │   │ 本地写命中  │
       (总线读) │   │ (总线使无效)    │   │ (总线使无效)│
              ┌──▼───┴───┐           ┌──▼───────┐       │
              │Exclusive │  本地写   │Modified  │───────┘
              │   (E)    │─────────▶│   (M)    │  被替换时
              └──────────┘          └──────────┘  写回内存
```


### 2.3 MESI 详细状态转换表


| 当前状态 | 事件 | 动作 | 新状态 |
| --- | --- | --- | --- |
| I (Invalid) | 本地读（其他缓存无副本） | 总线读，获取数据 | E |
| 本地读（其他缓存有S/E/M） | 总线读，获取数据 | S |
| 本地写 | 总线读+使无效（RFO），获取独占权 | M |
| S (Shared) | 本地读 | 直接读取（命中） | S |
| 本地写 | 总线使无效（Invalidate），发送给其他缓存 | M |
| 远程使无效 | 标记为无效 | I |
| E (Exclusive) | 本地读 | 直接读取（命中） | E |
| 本地写 | 直接修改（无需总线事务） | M |
| 远程读 | 提供数据，降级为共享 | S |
| M (Modified) | 本地读 | 直接读取（命中） | M |
| 本地写 | 直接修改 | M |
| 远程读 | 写回内存，提供数据 | S |
| 远程写（被替换） | 写回内存，使无效 | I |

第三部分：MOESI 协议

## 三、MOESI 协议（五状态扩展）


MOESI 在 MESI 基础上增加了 Owned（O） 状态，主要用于 AMD 和 ARM 处理器。


### 3.1 Owned 状态的作用


> **Note:** **核心思想：**
> 在 S 状态下，如果有某个缓存修改了数据，该缓存变为 M 状态，其他缓存变为 I 状态。但这会导致其他缓存再次读取时必须访问慢速内存。O 状态允许一个缓存充当"脏数据的共享源"，避免不必要的内存访问。


| 状态 | 含义 | 数据有效性 | 与内存一致性 | 共享 |
| --- | --- | --- | --- | --- |
| M (Modified) | 唯一副本且已修改 | 有效 | 不一致 | 独占 |
| O (Owned) | 拥有者，数据已修改但允许共享 | 有效 | 不一致 | 可共享（提供脏数据） |
| E (Exclusive) | 唯一副本且与内存一致 | 有效 | 一致 | 独占 |
| S (Shared) | 数据与内存一致（来自内存或O状态缓存） | 有效 | 一致 | 可共享 |
| I (Invalid) | 无效 | 无效 | — | — |


### 3.2 MOESI 相对于 MESI 的优势


- 减少内存带宽消耗：O 状态的缓存可以直接响应其他处理器的读请求，不需要写回内存
- 适合多处理器系统：在 4 核以上的系统中效果更显著
- 降低延迟：共享脏数据时跳过内存访问

第四部分：监听协议 vs 目录协议

## 四、Snooping vs Directory 协议


#### 监听协议（Snooping）


- 所有缓存控制器监听共享总线
- 每个缓存观察总线上的所有事务
- 适用于小规模系统（2-8 个处理器）
- 总线带宽是瓶颈
- 实现简单，延迟低
- 典型应用：Intel x86、桌面 CPU


#### 目录协议（Directory）


- 集中式目录记录每个缓存行的状态
- 点对点消息通信，不使用广播
- 可扩展到大规模系统（数百处理器）
- 目录存储开销大
- 延迟稍高，但可扩展性好
- 典型应用：AMD Opteron、NUMA 系统


### 4.1 目录协议的工作流程


```
处理器 P1 请求读取地址 X:

P1 ──读请求──▶ 目录（Directory）
                │
                ├─ 查找 X 的目录项
                │  ┌────────────────────────────┐
                │  │ 地址X: 状态=Modified       │
                │  │ 拥有者: {P3}               │
                │  │ 共享者: {}                 │
                │  └────────────────────────────┘
                │
                ▼
          向 P3 发送"写回+降级为共享"消息
                │
                ▼
          P3 写回数据，通知目录
                │
                ▼
          目录更新，转发数据给 P1
                │
                ▼
          P1 获得数据，目录记录 P1 为共享者
```


### 4.2 两种协议的对比表


| 特性 | Snooping | Directory |
| --- | --- | --- |
| 通信方式 | 广播/总线 | 点对点消息 |
| 可扩展性 | 差（受限于总线带宽） | 好（数百处理器） |
| 存储开销 | 低 | 高（目录存储） |
| 延迟 | 低 | 中等 |
| 互连网络 | 共享总线 | 任意互连（Crossbar、Mesh 等） |
| 典型规模 | 2-8 核 | 16-1024+ 核 |

第五部分：False Sharing

## 五、伪共享（False Sharing）


伪共享是缓存一致性协议导致的性能问题：两个处理器访问同一缓存行中**不同**的变量，但缓存一致性协议以缓存行为单位进行管理，导致不必要的缓存失效。


### 5.1 伪共享示例


```
// C 语言示例：伪共享问题
#include <pthread.h>

// 共享数据（在同一缓存行中）
struct {
    volatile int counter_a;  // 处理器 0 修改此变量
    volatile int counter_b;  // 处理器 1 修改此变量
    // 两者在同一个 64 字节缓存行中 → 伪共享！
} shared_data;

void* thread_func_a(void* arg) {
    for (int i = 0; i < 10000000; i++) {
        shared_data.counter_a++;  // 每次写使 P1 的缓存行失效
    }
    return NULL;
}

void* thread_func_b(void* arg) {
    for (int i = 0; i < 10000000; i++) {
        shared_data.counter_b++;  // 每次写使 P0 的缓存行失效
    }
    return NULL;
}

// 解决方案：缓存行填充（Padding）
struct aligned_data {
    volatile int counter_a;
    char padding[60];  // 填充到 64 字节（一个缓存行）
    volatile int counter_b;
} __attribute__((aligned(64)));
```


### 5.2 伪共享的影响


```
无伪共享时的缓存行为：
  P0 缓存行: [counter_a][padding...]  ← 独立缓存行
  P1 缓存行: [counter_b][padding...]  ← 独立缓存行

有伪共享时的缓存行为：
  P0 缓存行: [counter_a | counter_b]  ← 共享缓存行
  P1 缓存行: [counter_a | counter_b]  ← 每次对方写都导致本方失效！

  P0 写 counter_a → P1 缓存行失效 → P1 重新加载
  P1 写 counter_b → P0 缓存行失效 → P0 重新加载
  → 大量缓存一致性流量，性能严重下降
```

第六部分：预取策略

## 六、缓存预取（Prefetching）策略


### 6.1 预取类型


| 类型 | 触发方式 | 适用场景 | 优点 | 缺点 |
| --- | --- | --- | --- | --- |
| 硬件自动预取 | 检测顺序访问模式 | 数组遍历、顺序读取 | 无需软件干预 | 可能预取无用数据 |
| 软件预取指令 | 编译器或程序员插入 PREFETCH 指令 | 链表遍历、稀疏访问 | 精确控制 | 增加代码复杂度 |
| Stride 预取 | 检测固定步长的访问模式 | 二维数组、结构体数组 | 适应多种步长 | 步长过大时效果差 |
| Stream 预取 | 检测多个独立的顺序流 | 矩阵运算、多数组并行处理 | 支持多流 | 硬件开销大 |


### 6.2 预取与一致性的交互


> **Note:** **关键问题：**
> 预取的数据进入缓存后，如果该缓存行在其他处理器上被修改，预取的副本会通过一致性协议自动失效。这意味着预取操作本质上是在赌"预取的数据在使用前不会被修改"。


```
// x86 软件预取示例
#include <xmmintrin.h>

void process_linked_list(Node* head) {
    Node* current = head;
    while (current != NULL) {
        // 预取下一个节点（提前 2 个节点）
        if (current->next != NULL && current->next->next != NULL) {
            _mm_prefetch((char*)current->next->next, _MM_HINT_T0);
        }

        // 处理当前节点
        process(current->data);

        current = current->next;
    }
}
```


### 6.3 预取对缓存一致性的影响


- **积极影响**
   ：减少缓存未命中延迟，隐藏内存访问延迟
- **消极影响**
   ：

   - 预取可能增加一致性流量（预取的数据被其他处理器修改）
   - 预取占用缓存空间，可能驱逐有用数据
   - 过度预取浪费内存带宽
- **最佳实践**
   ：在数据被修改前完成预取，或使用独占预取（Exclusive Prefetch）减少后续写操作的一致性开销


## 七、实际处理器中的一致性协议


| 处理器 | 协议 | 监听/目录 | 缓存行大小 |
| --- | --- | --- | --- |
| Intel Core 系列 | MESIF | Snooping + Snoop Filter | 64 字节 |
| AMD Zen 系列 | MOESI | Snooping + Probe Filter | 64 字节 |
| ARM Cortex-A | MOESI / MESI | Snooping 或 Directory | 64 字节 |
| IBM POWER | MOESI | Directory | 128 字节 |


> **Note:** **Intel MESIF：**
> Intel 在 MESI 基础上添加了 Forward（F）状态，用于在多个 S 状态缓存中指定一个作为数据转发者，减少内存访问。F 状态与 S 状态类似，但当有远程读请求时，由 F 状态的缓存响应而非内存。

总结

## 总结


缓存一致性协议是多处理器系统正确运行的基石。MESI/MOESI 协议通过精心设计的状态机，平衡了性能与正确性。选择 Snooping 还是 Directory 取决于系统规模。理解和避免伪共享是多线程性能优化的关键。合理的预取策略能在不破坏一致性的前提下显著提升性能。
