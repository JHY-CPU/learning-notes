# 内存一致性模型 - 计算机体系结构


# 内存一致性模型（Memory Consistency Models）

第一部分：内存一致性 vs 缓存一致性

## 一、内存一致性与缓存一致性的区别


这是两个经常混淆但完全不同的概念：


#### 缓存一致性（Cache Coherence）


- 解决：
   **同一地址**
   的多个副本值是否一致
- 回答："所有处理器看到的 X 的值是否相同？"
- 粒度：单个内存地址
- 实现：MESI/MOESI 等协议


#### 内存一致性（Memory Consistency）


- 解决：
   **不同地址**
   的读写操作的全局可见顺序
- 回答："处理器看到的多地址操作顺序是否正确？"
- 粒度：操作序列（Program Order → Global Order）
- 实现：由 ISA 定义的内存模型


### 1.1 一致性问题示例


```
经典的 MP（多处理器）测试：

初始状态：A = 0, B = 0

  处理器 P0              处理器 P1
  ─────────              ─────────
  STORE A = 1            STORE B = 1
  LOAD  B                 LOAD  A

问题：P0 读到 B=0 且 P1 读到 A=0 是否可能？

在不同内存模型下，答案不同！
```

第二部分：顺序一致性

## 二、顺序一致性（Sequential Consistency, SC）


由 Lamport（1979）提出的最强内存一致性模型。


### 2.1 定义


> **Note:** **顺序一致性的两个条件：**
>
> 1. 每个处理器的内存操作按照程序顺序（Program Order）执行
> 2. 所有处理器看到的内存操作顺序相同（全局单一顺序）
>
> 等价于：所有处理器的操作以某种方式交错执行，且每个处理器的操作在其交错位置保持程序顺序。


### 2.2 顺序一致性下的 MP 测试


```
在 SC 下，所有操作必须有一个全局的交错顺序：

可能的交错（共 6 种）：

顺序 1: St(A,1) → St(B,1) → Ld(B) → Ld(A)  → P0: B=1, P1: A=1 ✓
顺序 2: St(A,1) → Ld(B) → St(B,1) → Ld(A)  → P0: B=0, P1: A=1 ✓
顺序 3: St(A,1) → Ld(B) → Ld(A) → St(B,1)  → P0: B=0, P1: A=1 ✓
顺序 4: Ld(B) → St(A,1) → St(B,1) → Ld(A)  → P0: B=0, P1: A=1 ✓
顺序 5: Ld(B) → St(A,1) → Ld(A) → St(B,1)  → P0: B=0, P1: A=0 ✓
顺序 6: Ld(B) → Ld(A) → St(A,1) → St(B,1)  → P0: B=0, P1: A=0 ✓

结论：在 SC 下，P0 读到 B=0 且 P1 读到 A=0 是可能的！
（通过交错 5 或 6 实现）
```


### 2.3 SC 的硬件实现


```
// SC 的朴素实现：所有内存访问都经过一个全局有序的内存系统
// 问题：性能极差，无法利用任何重排优化

// 优化：使用写缓冲区（Store Buffer）
// Store 操作先进入 FIFO 写缓冲区，再写入缓存
// 但写缓冲区会破坏 SC！（后面会详细讨论）

// 实际实现：SC 通常需要在 Load 操作前添加屏障（Fence）
// 以确保写缓冲区中的数据对 Load 可见
```

第三部分：x86 TSO 模型

## 三、x86 TSO 模型（Total Store Order）


x86/x64 架构使用的内存一致性模型，比 SC 略弱但性能更好。


### 3.1 TSO 的核心规则


| 规则 | 说明 |
| --- | --- |
| Load-Load 有序 | 程序中先出现的 Load 一定先执行（不可重排） |
| Store-Store 有序 | 程序中先出现的 Store 一定先执行（不可重排） |
| Load-Store 有序 | Load 不会推迟到后续 Store 之后 |
| **Store-Load 无序** | Store 可能延迟到写缓冲区，后续 Load 可能先完成！ |


### 3.2 TSO 下的 MP 测试（Store-Load 重排）


```
初始状态：A = 0, B = 0

  处理器 P0              处理器 P1
  ─────────              ─────────
  STORE A = 1            STORE B = 1
  LOAD  B                 LOAD  A

TSO 下可能发生：
  P0 的 Store A=1 → 写入 P0 的写缓冲区（尚未到达缓存）
  P0 的 Load B → 直接从缓存读取 → B = 0 ✓
  P1 的 Store B=1 → 写入 P1 的写缓冲区
  P1 的 Load A → 直接从缓存读取 → A = 0 ✓

结果：P0: B=0 且 P1: A=0
这在 SC 下也可以发生（通过交错 5/6）
所以 TSO 与 SC 的区别在于其他场景
```


### 3.3 TSO 独有的"不可预测"行为


```
经典 TSO 测试（Store Buffer 导致的"自读取旧值"）：

初始状态：A = 0

  处理器 P0
  ─────────
  STORE A = 1
  LOAD  A

在 TSO 下可能读到 A = 0！（读到自己写之前旧值）
原因：
  1. Store A=1 进入写缓冲区
  2. Load A 发出时，写缓冲区还未写入缓存
  3. Load A 从缓存读取旧值 A = 0

在 SC 下这是不可能的！（自己必须看到自己的写）

x86 解决方案：写缓冲区转发（Store-to-Load Forwarding）
  → 同一地址的 Load 可以从写缓冲区读取最新值
  → 避免了"自读取旧值"的问题
```


### 3.4 x86 内存屏障指令


```
// x86 内存屏障指令（MFENCE, SFENCE, LFENCE）

// MFENCE: 完全内存屏障
// 确保 MFENCE 前的所有内存操作完成后，才执行 MFENCE 后的操作
// 等价于清空写缓冲区
asm volatile("mfence" ::: "memory");

// SFENCE: Store 屏障
// 确保 SFENCE 前的所有 Store 完成后，才执行 SFENCE 后的 Store
asm volatile("sfence" ::: "memory");

// LFENCE: Load 屏障
// 确保 LFENCE 前的所有 Load 完成后，才执行 LFENCE 后的 Load
// 也用于序列化指令流（防止推测执行穿越）
asm volatile("lfence" ::: "memory");

// LOCK 前缀也隐含 MFENCE
asm volatile("lock addl $0, 0(%%rsp)" ::: "memory");
```

第四部分：PSO 模型

## 四、PSO 模型（Partial Store Order）


SPARC 架构使用的中间模型，比 TSO 更弱。


### 4.1 PSO vs TSO


| 规则 | TSO | PSO |
| --- | --- | --- |
| Load-Load 有序 | 是 | 是 |
| Store-Store 有序 | 是 | **否（可重排）** |
| Load-Store 有序 | 是 | 是 |
| Store-Load 有序 | 否 | 否 |


### 4.2 PSO 下的问题示例


```
初始状态：A = 0, B = 0, flag = 0

  处理器 P0              处理器 P1
  ─────────              ─────────
  STORE A = 1            LOAD flag
  STORE B = 1            if (flag == 1)
  STORE flag = 1             LOAD A   ← 可能读到 A = 0！

在 PSO 下：
  Store A=1 和 Store B=1 可能被重排
  即使 flag=1 已经对 P1 可见，A=1 可能还没到达

解决方案：使用 STBAR（SPARC 的 Store 屏障）
  STORE A = 1
  STORE B = 1
  STBAR                    ← 确保前面的 Store 都完成
  STORE flag = 1
```

第五部分：弱一致性模型

## 五、弱一致性模型（Weak Ordering）


弱一致性模型是 ARM、POWER 等 RISC 架构使用的最激进的内存模型，追求最高性能。


### 5.1 弱一致性的核心思想


> **Note:** **核心理念：**
> 将内存操作分为两类：
>
> - **数据操作（Data Operations）**
>    ：普通的 Load/Store
> - **同步操作（Synchronization Operations）**
>    ：带屏障（Fence）的 Load/Store
>
> 只有同步操作之间需要保证顺序，数据操作之间可以任意重排。


### 5.2 弱一致性下的允许行为


```
在弱一致性下，以下所有重排都是允许的：

Load-Load 重排:
  LOAD A        LOAD B        LOAD B 可能先执行
  LOAD B   →    LOAD A

Store-Store 重排:
  STORE A=1     STORE B=1     STORE B=1 可能先到达
  STORE B=1  →  STORE A=1

Load-Store 重排:
  LOAD A        STORE B=1     STORE B=1 可能先执行
  STORE B=1  →  LOAD A

Store-Load 重排:
  STORE A=1     LOAD B        LOAD B 可能先执行
  LOAD B     →  STORE A=1

所有四种重排都是允许的！
```


### 5.3 弱一致性下的同步


```
// ARM 弱一致性模型的内存屏障

// DMB（Data Memory Barrier）
// 确保 DMB 前的内存操作在 DMB 后的操作之前完成
asm volatile("dmb sy" ::: "memory");  // 完全屏障
asm volatile("dmb st" ::: "memory");  // Store 屏障
asm volatile("dmb ld" ::: "memory");  // Load 屏障

// DSB（Data Synchronization Barrier）
// 比 DMB 更强：等待所有内存操作真正完成（包括中断处理）
asm volatile("dsb sy" ::: "memory");

// ISB（Instruction Synchronization Barrier）
// 刷新流水线，确保后续指令使用新的上下文
asm volatile("isb" ::: "memory");

// ARM 的 acquire/release 语义（ARMv8 引入）
// LDAR：带有 acquire 语义的 Load
// 确保此 Load 之后的内存操作不会被重排到此 Load 之前
asm volatile("ldar %0, [%1]" : "=r"(val) : "r"(addr));

// STLR：带有 release 语义的 Store
// 确保此 Store 之前的内存操作不会被重排到此 Store 之后
asm volatile("stlr %0, [%1]" :: "r"(val), "r"(addr));
```

第六部分：Acquire/Release 语义

## 六、Acquire/Release 语义


Acquire/Release 是弱一致性模型中实现同步的常用模式，比全屏障更高效。


### 6.1 定义


| 语义 | 作用 | 方向 | 类比 |
| --- | --- | --- | --- |
| **Acquire** | 后续操作不会被重排到此操作之前 | 向后保护 | 锁的获取（lock） |
| **Release** | 之前的操作不会被重排到此操作之后 | 向前保护 | 锁的释放（unlock） |


### 6.2 使用场景：自旋锁


```
// 使用 Acquire/Release 实现自旋锁

typedef struct {
    int lock;  // 0 = 未锁定, 1 = 锁定
} spinlock_t;

// 获取锁（Acquire 语义）
void spin_lock(spinlock_t* s) {
    // CAS（Compare-And-Swap）带 acquire 语义
    while (__atomic_exchange_n(&s->lock, 1, __ATOMIC_ACQUIRE) == 1) {
        // 自旋等待
        while (__atomic_load_n(&s->lock, __ATOMIC_RELAXED) == 1)
            ;  // 暂停指令或 NOP
    }
    // acquire 确保：锁之后的临界区操作不会被重排到锁获取之前
}

// 释放锁（Release 语义）
void spin_unlock(spinlock_t* s) {
    // store 带 release 语义
    __atomic_store_n(&s->lock, 0, __ATOMIC_RELEASE);
    // release 确保：临界区的所有操作在锁释放之前完成
}

// 对比：如果使用全屏障
void spin_lock_full_fence(spinlock_t* s) {
    while (__atomic_exchange_n(&s->lock, 1, __ATOMIC_SEQ_CST) == 1)
        while (__atomic_load_n(&s->lock, __ATOMIC_RELAXED) == 1)
            ;
    // SEQ_CST 比 ACQUIRE 更强，但这里不需要
}

// 使用 C++11 原子操作（跨平台）
#include <atomic>
std::atomic<int> lock{0};

void lock_acquire() {
    int expected = 0;
    while (!lock.compare_exchange_weak(
        expected, 1,
        std::memory_order_acquire,  // 成功时的顺序
        std::memory_order_relaxed   // 失败时的顺序
    )) {
        expected = 0;
    }
}

void lock_release() {
    lock.store(0, std::memory_order_release);
}
```

第七部分：C++ 内存序

## 七、C++11 内存序（Memory Order）


C++11 引入的标准化内存模型，提供了可移植的多线程编程接口。


| 内存序 | 强度 | 保证 | 典型用法 |
| --- | --- | --- | --- |
| memory_order_relaxed | 最弱 | 仅保证原子性，无顺序保证 | 引用计数器 |
| memory_order_consume | 弱 | 数据依赖有序 | RCU（几乎不使用） |
| memory_order_acquire | 中 | 后续操作有序 | 锁的获取 |
| memory_order_release | 中 | 之前操作有序 | 锁的释放 |
| memory_order_acq_rel | 强 | 同时具有 acquire + release | 读-改-写操作 |
| memory_order_seq_cst | 最强 | 全局单一顺序（SC） | 默认值、需要强保证时 |


### 7.1 强度对比


```
内存序强度排序（从弱到强）：

relaxed < consume < acquire/release < acq_rel < seq_cst
   │                                                   │
   │  性能最好              ←─────→              最易理解  │
   │  最难用正确                                最不容易出错 │
   ▼                                                   ▼
   ARM/POWER 优化                  x86 默认（几乎所有都是 seq_cst）

x86 特点：x86 本身就是 TSO 模型，比弱一致性强
  - relaxed/acquire/release 在 x86 上几乎无额外开销
  - seq_cst 在 x86 上只需要一个 MFENCE（如果需要的话）

ARM/POWER 特点：弱一致性模型，需要显式屏障
  - 每个 acquire 需要一个 DMB/LWSYNC
  - 每个 release 需要一个 DMB/LWSYNC
  - 选择正确的内存序对性能影响很大
```

总结

## 总结


内存一致性模型定义了多处理器系统中内存操作的全局可见顺序，与缓存一致性（解决单地址多副本问题）互补。从最强的 SC 到最弱的弱一致性，不同的模型在编程简易性和性能之间做出不同的权衡。现代处理器普遍使用 TSO（x86）或弱一致性（ARM/POWER），通过 Acquire/Release 语义和内存屏障实现高效同步。C++11 内存序提供了可移植的编程抽象。
