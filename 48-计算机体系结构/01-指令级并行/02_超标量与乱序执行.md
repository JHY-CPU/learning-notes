# 超标量与乱序执行 - 计算机体系结构笔记


超标量处理器在每个时钟周期内可以**发射多条指令**到不同的功能单元执行，通过硬件并行度提高性能。


### 1.1 度（Degree of Superscalar）


超标量的"度"指每个周期最多发射的指令数。


| 处理器 | 发射宽度 | 功能单元数 |
| --- | --- | --- |
| ARM Cortex-A53 | 2路 | ALU×2, MAC, Load/Store, Branch |
| Intel Skylake | 6路 | ALU×4, Load, Store |
| Apple M1 (Firestorm) | 8路 | ALU×6, Load×2, Store×2, FP×4 |


### 1.2 超标量流水线


```
周期:   1    2    3    4    5    6
  I1:    IF   ID   EX  MEM  WB
  I2:    IF   ID   EX  MEM  WB        ← 同一周期取/发射两条指令
  I3:         IF   ID   EX  MEM  WB
  I4:         IF   ID   EX  MEM  WB
  I5:              IF   ID   EX  MEM  WB
  I6:              IF   ID   EX  MEM  WB
```


### 1.3 顺序超标量 vs 乱序超标量


| 类型 | 指令完成顺序 | 硬件复杂度 | 性能 |
| --- | --- | --- | --- |
| 顺序发射+顺序完成 | 程序顺序 | 低 | 较低 |
| 顺序发射+乱序完成 | 乱序 | 中 | 中 |
| 乱序发射+乱序完成 | 乱序 | 高 | 最高 |


乱序执行允许指令在操作数就绪后立即执行，不必等待前面的指令完成，从而突破数据依赖带来的延迟。


### 2.1 乱序执行的流水线阶段


```
取指 → 译码 → 重命名 → 发射/派遣 → 执行 → 写回 → 提交
  IF    ID    Rename   Dispatch   EX     WB    Commit

  ┌─────────────────────────────────────────────────┐
  │                  前端(Frontend)                  │
  │  IF → ID → Rename → Dispatch                   │
  ├─────────────────────────────────────────────────┤
  │                  后端(Backend)                   │
  │  Issue Queue → Execute → WB → Reorder Buffer   │
  └─────────────────────────────────────────────────┘
```


### 2.2 关键硬件结构


- **重命名寄存器**
   （Register Renaming）：消除WAR和WAW冒险
- **保留站**
   （Reservation Station）：存放等待操作数的指令
- **重排序缓冲区**
   （Reorder Buffer, ROB）：维持程序顺序，保证精确异常
- **指令队列**
   （Issue Queue）：选择就绪指令发射


寄存器重命名通过将**架构寄存器**映射到更大的**物理寄存器**池，消除假依赖（WAR/WAW）。


### 3.1 示例


> **Example:** **原始代码（存在WAR冒险）：**
>
>
> ```
> ADD R1, R2, R3    // 写R1
> SUB R4, R1, R5    // 读R1 (RAW - 真依赖)
> MUL R1, R6, R7    // 写R1 (WAR - 反依赖，WAW - 输出依赖)
> ```
>
>
> **重命名后：**
>
>
> ```
> ADD P1, P2, P3    // R1→P1, R2→P2, R3→P3
> SUB P4, P1, P5    // R4→P4, R1→P1(读), R5→P5
> MUL P6, P7, P8    // R1→P6(新映射!), R6→P7, R7→P8
> ```
>
>
> ADD 和 MUL 之间不再有依赖，可以并行执行。


### 3.2 重命名方式


| 方式 | 描述 | 典型使用 |
| --- | --- | --- |
| 重命名表（RAT） | 架构寄存器→物理寄存器映射表 | Intel/AMD/x86 |
| ROB直接重命名 | ROB条目直接作为重命名寄存器 | 早期MIPS R10000 |


Tomasulo算法是Robert Tomasulo于1967年为IBM 360/91设计的**动态调度算法**，是现代乱序执行的基础。


### 4.1 核心组件


| 组件 | 功能 |
| --- | --- |
| **保留站（RS）** | 存放等待执行的指令及操作数 |
| **公共数据总线（CDB）** | 广播计算结果，同时更新所有等待该值的保留站 |
| **寄存器状态表** | 记录每个架构寄存器由哪个保留站写入 |


### 4.2 三阶段


1. **发射（Issue）**
   ：从指令队列取指令，若有空闲保留站则发射，记录操作数来源
2. **执行（Execute）**
   ：当所有操作数就绪，在功能单元执行
3. **写回（Write Back）**
   ：通过CDB广播结果，更新所有等待的保留站


### 4.3 执行过程示例


```
指令序列:
  LD  F6, 34(R2)
  LD  F2, 45(R3)
  MULTD F0, F2, F4
  SUBD  F8, F6, F2
  DIVD  F10, F0, F6
  ADDD  F6, F8, F2

  ┌─────────────────────────────────────────────────────┐
  │ 保留站内容（MULTD完成后的状态）：                     │
  │                                                     │
  │ Busy  Op     Vj       Vk      Qj      Qk           │
  │ ───   ────   ────     ────    ────    ────         │
  │ Yes   MULTD  [F2值]   [F4值]  -       -   ←就绪    │
  │ Yes   SUBD   [F6值]   -       RS(MULTD) -  ←等F2   │
  │ Yes   DIVD   -        [F6值]  RS(MULTD) -  ←等F0   │
  │ Yes   ADDD   -        -       RS(SUBD)  RS(MULTD)  │
  └─────────────────────────────────────────────────────┘
```


### 4.4 Tomasulo 的优势


- 通过CDB广播实现
   **隐含的数据流调度**
- 寄存器重命名效果 — 消除WAR和WAW冒险
- 分布式控制 — 每个保留站独立判断何时执行


### 5.1 ROB（Reorder Buffer）


ROB 是一个**环形 FIFO 缓冲区**，按程序顺序记录所有未提交的指令，确保指令按序**提交（Commit/Retire）**。


### 5.2 精确异常（Precise Exception）


精确异常要求：当第 i 条指令发生异常时，i 之前的指令全部完成，i 及之后的指令全部作废。


```
ROB状态:
  ┌─────┬──────┬────────┬────────┬────────┐
  │ 序号│ 指令 │ 目标   │ 值     │ 状态   │
  ├─────┼──────┼────────┼────────┼────────┤
  │  1  │ ADD  │ R1     │ 42     │ 完成 ✓ │ ← 可提交
  │  2  │ SUB  │ R2     │ 17     │ 完成 ✓ │ ← 可提交
  │  3  │ MUL  │ R3     │        │ 执行中  │ ← 等待
  │  4  │ DIV  │ R4     │        │ 发射   │ ← 等待
  └─────┴──────┴────────┴────────┴────────┘
  头指针→1(待提交)  尾指针→5(待分配)
```


分支预测是乱序执行的关键配套技术 — 在分支结果确定之前就需要取指和发射后续指令。


### 6.1 静态预测


- **预测不跳转**
   （BTFNT）：简单但准确率低
- **预测后向跳转**
   ：循环分支通常跳转（for/while）


### 6.2 动态预测


| 方法 | 原理 | 准确率 |
| --- | --- | --- |
| 1-bit 饱和计数器 | 上次跳则预测跳 | ~60% |
| 2-bit 饱和计数器 | 需连续两次变化才切换预测 | ~85% |
| 两级自适应预测器 | 用分支历史模式查表 | ~95% |
| GShare / GSelect | 全局历史+PC异或/拼接索引 | ~96% |
| Tournament预测器 | 多个预测器竞争选择 | ~97% |


### 6.3 分支目标缓冲（BTB）


BTB 缓存分支指令的 PC 和上次的跳转目标地址，实现**零周期分支**（若预测正确）。

**推测执行的代价**
：分支预测错误时，需要冲刷（flush）所有推测执行的指令。现代处理器的分支预测准确率约95-97%，剩余3-5%的误预测是性能损失的主要来源之一。
