# 06-系统安全与访问控制

## 一、操作系统安全

### 1.1 Linux安全加固

#### 用户与权限管理

```bash
# 用户管理
useradd -m -s /bin/bash username    # 创建用户
passwd username                      # 设置密码
usermod -L username                  # 锁定账户
userdel -r username                  # 删除用户

# 密码策略
# /etc/login.defs
PASS_MAX_DAYS 90       # 密码最长有效期
PASS_MIN_DAYS 7        # 密码最短修改间隔
PASS_MIN_LEN 12        # 密码最短长度
PASS_WARN_AGE 14       # 密码过期提醒天数

# PAM配置加强
# /etc/pam.d/common-password
password requisite pam_pwquality.so minlen=12 dcredit=-1 ucredit=-1 ocredit=-1 lcredit=-1
password requisite pam_pwhistory.so remember=5
```

#### 文件系统安全

```bash
# 文件权限
chmod 750 /sensitive_dir     # 所有者rwx，组rx，其他无权限
chown root:admin /sensitive  # 设置所有者和组
chattr +i /etc/passwd        # 设置不可修改属性

# SUID/SGID审计
find / -perm -4000 -type f 2>/dev/null  # 查找SUID文件
find / -perm -2000 -type f 2>/dev/null  # 查找SGID文件

# 关键文件权限
chmod 600 /etc/shadow
chmod 644 /etc/passwd
chmod 600 /etc/ssh/sshd_config
```

#### SSH安全

```bash
# /etc/ssh/sshd_config
Port 2222                    # 修改默认端口
PermitRootLogin no           # 禁止root登录
PasswordAuthentication no    # 禁止密码登录，使用密钥
MaxAuthTries 3               # 最大认证尝试
ClientAliveInterval 300      # 客户端存活检测
ClientAliveCountMax 2        # 最大存活检测次数
AllowUsers user1 user2       # 白名单用户
Protocol 2                   # 仅使用SSHv2
X11Forwarding no             # 禁止X11转发
PermitEmptyPasswords no      # 禁止空密码
```

#### 防火墙（iptables/nftables/ufw）

```bash
# UFW（推荐简单场景）
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable

# iptables
iptables -A INPUT -p tcp --dport 22 -j ACCEPT
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -j DROP
```

#### SELinux/AppArmor

| 特性 | SELinux | AppArmor |
|------|---------|----------|
| 类型 | 强制访问控制（MAC） | 强制访问控制（MAC） |
| 粒度 | 基于标签 | 基于路径 |
| 复杂度 | 较高 | 较低 |
| 发行版 | RHEL/CentOS/Fedora | Ubuntu/SUSE |

```bash
# SELinux
getenforce                    # 查看状态
setenforce 1                  # 设置为强制模式
semanage port -a -t http_port_t -p tcp 8080  # 添加端口

# AppArmor
aa-status                     # 查看状态
aa-enforce /etc/apparmor.d/usr.bin.firefox  # 强制模式
```

#### 系统审计

```bash
# auditd审计
# /etc/audit/rules.d/audit.rules
-w /etc/passwd -p wa -k passwd_changes
-w /etc/shadow -p wa -k shadow_changes
-w /var/log/ -p wa -k log_changes
-a always,exit -F arch=b64 -S execve -k exec_log

# 查看审计日志

```

## 二、身份认证安全

### 2.1 认证因素

```
认证因素三要素：
├── 你知道什么（Knowledge）- 密码、PIN、密保问题
├── 你拥有什么（Possession）- 手机、令牌、证书
└── 你是什么（Inherence）- 指纹、人脸、虹膜

多因素认证（MFA）= 至少两种不同类型的认证因素
```

### 2.2 密码安全

#### 密码策略最佳实践

| 旧建议（已过时） | 新建议（NIST SP 800-63B） |
|------------------|---------------------------|
| 定期更换密码 | 仅在泄露时更换 |
| 要求复杂字符组合 | 仅要求足够长度 |
| 最短8位 | 最少8位，建议15+ |
| 强制记住多个密码 | 允许密码管理器 |

#### 密码存储

```python
# 使用argon2id
from argon2 import PasswordHasher
ph = PasswordHasher()
hash = ph.hash("user_password")
ph.verify(hash, "user_password")

# 使用bcrypt
import bcrypt
hash = bcrypt.hashpw(b"password", bcrypt.gensalt(rounds=12))
bcrypt.checkpw(b"password", hash)
```

### 2.3 多因素认证（MFA）

| 方式 | 安全性 | 用户体验 |
|------|--------|----------|
| SMS验证码 | 中（可被SIM卡劫持） | 好 |
| TOTP（Google Authenticator） | 高 | 中 |
| 硬件令牌（YubiKey） | 最高 | 中 |
| 生物识别 | 高 | 最好 |
| 推送通知 | 高 | 好 |
| FIDO2/WebAuthn | 最高 | 好 |

### 2.4 单点登录（SSO）

```
SSO协议对比：
├── SAML 2.0 - 企业级，XML格式，适合Web SSO
├── OAuth 2.0 - 授权框架，适合API授权
├── OpenID Connect - 基于OAuth 2.0的身份层
├── Kerberos - 企业内部，票据机制
└── CAS - 开源SSO协议
```

### 2.5 JWT（JSON Web Token）

```
结构：Header.Payload.Signature

Header:
{
  "alg": "RS256",
  "typ": "JWT"
}

Payload:
{
  "sub": "1234567890",
  "name": "John",
  "iat": 1516239022,
  "exp": 1516242622
}

# 安全注意：
✅ 使用RS256/ES256，不要使用HS256（密钥混淆攻击）
✅ 设置合理的过期时间
✅ 验证签名和声明
✅ 安全存储（HttpOnly Cookie优于localStorage）
❌ 不要在JWT中存储敏感信息
```

## 三、访问控制模型

### 3.1 DAC（自主访问控制）

资源所有者自主决定谁能访问。
- 例：Unix文件权限
- 缺点：权限传递不受控

### 3.2 MAC（强制访问控制）

系统强制实施访问策略，用户无法更改。
- 例：SELinux、Bell-LaPadula模型
- 安全等级：机密、秘密、绝密

### 3.3 RBAC（基于角色的访问控制）

```
用户 → 角色 → 权限

用户1 ──→ 管理员 ──→ 读写删
用户2 ──→ 编辑者 ──→ 读写
用户3 ──→ 访客   ──→ 只读
```

**RBAC核心概念：**
- 用户-角色分配（UA）
- 角色-权限分配（PA）
- 角色继承
- 角色互斥约束
- 最小权限原则

### 3.4 ABAC（基于属性的访问控制）

基于用户、资源、环境等多维属性动态决策。

```
Policy: IF user.department == "HR"
        AND resource.classification <= "internal"
        AND environment.time IN "work_hours"
        THEN ALLOW
```

### 3.5 零信任访问控制

```
零信任决策因素：
├── 用户身份与认证强度
├── 设备安全状态
├── 网络位置和环境
├── 资源敏感度
├── 行为模式异常
└── 实时风险评分
```

### 4.1 审计范围

| 类别 | 审计内容 |
|------|----------|
| 登录审计 | 成功/失败登录、来源IP、时间 |
| 权限变更 | 用户增删改、权限修改 |
| 文件访问 | 敏感文件的读写操作 |
| 系统配置 | 配置变更、服务启停 |
| 网络活动 | 异常连接、端口扫描 |
| 数据操作 | 数据库增删改查 |

### 4.2 日志管理

```
日志管理架构：
├── 日志生成（应用/系统/安全设备）
├── 日志收集（Filebeat/Fluentd/Logstash）
├── 日志传输（TLS加密通道）
├── 日志存储（Elasticsearch/日志服务器）
├── 日志分析（SIEM平台）
└── 告警与响应

日志保留要求：
- 等保三级：至少6个月
- 金融行业：至少3年
- 医疗行业：至少15年
```

## 六、虚拟化与容器安全

### 6.1 虚拟化安全

| 威胁 | 说明 |
|------|------|
| 虚拟机逃逸 | 从虚拟机突破到宿主机 |
| 侧信道攻击 | 通过共享资源攻击同宿主VM |
| 虚拟机镜像安全 | 镜像中包含漏洞或敏感信息 |
| 虚拟网络攻击 | 虚拟网络隔离不足 |
| 管理接口暴露 | 虚拟化管理平台暴露在公网 |

### 6.2 Docker安全

```bash
# 安全最佳实践
# 1. 不要以root运行
USER nonroot

# 2. 使用最小基础镜像
FROM alpine:latest

# 3. 不要在镜像中存储敏感信息
# 使用Docker Secrets或环境变量

# 4. 扫描镜像漏洞
docker scan myimage:latest

# 5. 只读文件系统
docker run --read-only

# 6. 限制资源
docker run --memory=512m --cpus=1

# 7. 限制能力
docker run --cap-drop=ALL --cap-add=NET_BIND_SERVICE

# 8. 使用Docker Bench Security审计
docker run --net host --pid host docker/docker-bench-security
```
