# 03-网络协议安全与攻击防御

## 一、OSI模型与安全威胁

### 1.1 各层安全威胁

| OSI层 | 典型协议 | 安全威胁 | 防护措施 |
|-------|----------|----------|----------|
| 物理层 | 电缆、光纤 | 窃听、物理破坏、电磁泄漏 | 物理隔离、电磁屏蔽 |
| 数据链路层 | Ethernet、ARP | MAC欺骗、ARP欺骗、VLAN跳跃 | 802.1X、端口安全、DAI |

## 二、数据链路层安全

### 2.1 ARP欺骗（ARP Spoofing）

**原理：** 攻击者发送伪造的ARP响应，将自己的MAC地址与目标IP绑定。

```
正常ARP：  Who has 192.168.1.1? → 192.168.1.1 is at AA:BB:CC:DD:EE:FF

攻击ARP：  192.168.1.1 is at 11:22:33:44:55:66（攻击者MAC）
```

**攻击效果：** 中间人攻击、流量劫持、DoS

**防御措施：**
- **静态ARP绑定**：手动配置ARP表
- **动态ARP检测（DAI）**：交换机验证ARP包
- **ARP防火墙**：检测异常ARP流量
- **802.1X认证**：端口级认证

### 2.2 MAC地址泛洪

**原理：** 发送大量伪造源MAC地址的帧，填满交换机的MAC地址表。

**效果：** 交换机退化为集线器模式，所有流量广播

**防御：** 端口安全（限制每个端口的MAC数量）

### 2.3 VLAN跳跃攻击

**原理：** 利用802.1Q标签绕过VLAN隔离。

**两种方式：**
- **Switch Spoofing**：伪造DTP协商为Trunk
- **Double Tagging**：双802.1Q标签

**防御：**
- 禁用未使用的端口
- 将未使用的端口分配到非生产VLAN
- 关闭DTP
- 手动配置Trunk端口

## 三、网络层安全

### 3.1 IP欺骗（IP Spoofing）

**原理：** 伪造IP数据包的源地址。

**用途：**
- 绕过基于IP的访问控制
- 隐藏攻击来源
- Smurf/Fraggle攻击

**防御：**
- **入口过滤（Ingress Filtering）/ BCP38**：ISP过滤源地址不匹配的包
- **uRPF（单播逆向路径转发）**：验证源地址的路由可达性
- **TCP序列号随机化**：使猜测ISN变得困难

### 3.2 Smurf攻击

**原理：** 向广播地址发送大量ICMP Echo Request，源地址伪造为受害者IP。

```
攻击者 → 广播地址（ICMP Ping，源IP=受害者）
所有主机 → 受害者（ICMP Reply）
```

**防御：**
- 禁用IP定向广播
- 配置路由器不转发广播包
- 入口过滤

### 3.3 ICMP攻击

| 攻击类型 | 描述 |
|----------|------|
| Ping of Death | 发送超大ICMP包导致缓冲区溢出 |
| ICMP重定向 | 伪造ICMP重定向消息修改路由表 |
| ICMP隧道 | 在ICMP载荷中隐藏通信数据 |

### 3.4 路由安全

#### BGP劫持
**原理：** 恶意宣告不属于自己的IP前缀
**防御：** RPKI（资源公钥基础设施）、ROA（路由起源授权）

#### 路由黑洞
**原理：** 通过BGP宣告更具体的前缀，将流量吸引并丢弃
**防御：** 前缀过滤、BGP社区属性验证

## 四、IPSec协议

### 4.1 IPSec概述

IPSec（Internet Protocol Security）是网络层安全协议族，提供IP层的加密和认证。

**两大协议：**

| 协议 | 功能 | IP协议号 |
|------|------|----------|
| **AH**（认证头） | 数据完整性、认证、防重放 | 51 |
| **ESP**（封装安全载荷） | 加密、完整性、认证 | 50 |

### 4.2 工作模式

#### 传输模式（Transport Mode）
- 只保护IP载荷，IP头不变
- 用于端到端通信

```
┌──────┬──────┬────────────────┐     ┌──────┬──────┬──────┬────────────────┐
│ IP头 │ AH/  │     载荷       │ →  │ IP头 │ AH/  │ ESP  │     载荷       │
│      │ ESP  │                │     │      │ ESP  │ 头   │     (加密)     │
└──────┴──────┴────────────────┘     └──────┴──────┴──────┴────────────────┘
```

#### 隧道模式（Tunnel Mode）
- 保护整个原始IP包，添加新的IP头
- 用于站点到站点VPN、远程访问

```
┌──────┬──────┬────────────────┐     ┌──────┬──────┬──────┬──────┬──────────┐
│ IP头 │      │     载荷       │ →  │新IP头│ AH/  │ ESP  │旧IP头│  载荷    │
│      │      │                │     │      │ ESP  │ 头   │      │  (加密)  │
└──────┴──────┴────────────────┘     └──────┴──────┴──────┴──────┴──────────┘
```

### 4.3 IKE（Internet Key Exchange）

**IKEv1 两阶段：**
- 阶段1：建立ISAKMP SA（安全关联），协商加密/认证算法
- 阶段2：建立IPSec SA，协商数据加密参数

**IKEv2 改进：**
- 简化为4条消息
- 内置NAT穿透
- 支持EAP认证
- MOBIKE移动性支持

### 4.4 SA（安全关联）

SA是IPSec的基础，定义了安全通信的参数：
- SPI（安全参数索引）
- 目标IP地址
- 安全协议（AH/ESP）
- 加密/认证算法
- 密钥
- 生存时间

## 五、传输层安全

### 5.1 TCP安全威胁

#### SYN Flood攻击
**原理：** 发送大量SYN包但不完成三次握手，耗尽服务器半连接队列。

```
攻击者 ──SYN──→ 服务器
攻击者 ──SYN──→ 服务器
攻击者 ──SYN──→ 服务器（大量）
...
服务器半连接队列满了，无法接受新的合法连接
```

**防御：**
- **SYN Cookie**：不保存半连接状态，通过Cookie验证
- **SYN Proxy**：代理完成握手后再转发
- **增大半连接队列**
- **缩短SYN Timeout**

#### TCP序列号预测
**原理：** 预测TCP序列号，伪造RST或数据包进行会话劫持

**防御：** 使用随机化的初始序列号（ISN）

#### TCP RST攻击
**原理：** 发送伪造的RST包中断TCP连接

**防御：** 验证TCP序列号、加密通信

### 5.2 TLS/SSL协议

#### TLS协议版本演进

| 版本 | 年份 | 状态 |
|------|------|------|
| SSL 2.0 | 1995 | 已废弃，存在多个安全漏洞 |
| SSL 3.0 | 1996 | 已废弃，POODLE攻击 |
| TLS 1.0 | 1999 | 已废弃 |
| TLS 1.1 | 2006 | 已废弃 |
| TLS 1.2 | 2008 | 广泛使用 |
| TLS 1.3 | 2018 | 推荐使用 |

#### TLS 1.3改进

```
TLS 1.2握手（2-RTT）：
ClientHello →
          ← ServerHello
          ← Certificate
          ← ServerKeyExchange
          ← ServerHelloDone
ClientKeyExchange →
ChangeCipherSpec →
Finished →
          ← ChangeCipherSpec
          ← Finished

TLS 1.3握手（1-RTT）：
ClientHello + KeyShare →
          ← ServerHello + KeyShare
          ← {EncryptedExtensions}
          ← {CertificateRequest}
          ← {Certificate}
          ← {CertificateVerify}
          ← {Finished}
{Certificate} →
{CertificateVerify} →
{Finished} →
```

**TLS 1.3关键改进：**
- 1-RTT握手（0-RTT可选）
- 移除不安全算法（RC4、DES、3DES、MD5、SHA-1）
- 移除不安全特性（压缩、重协商）
- 仅支持前向保密密钥交换（ECDHE、DHE）
- 加密更多握手消息

#### TLS安全威胁

| 攻击 | 描述 | 影响版本 |
|------|------|----------|
| BEAST | CBC IV预测 | TLS 1.0 |
| CRIME | 压缩信息泄露 | TLS压缩 |
| POODLE | 填充Oracle | SSL 3.0 |
| Heartbleed | OpenSSL心跳扩展漏洞 | 部分OpenSSL |
| DROWN | 跨协议攻击SSLv2 | SSLv2共存时 |
| ROBOT | RSA加密的Bleichenbacher攻击 | RSA密钥交换 |

#### TLS最佳实践

- 使用TLS 1.3或TLS 1.2
- 禁用SSL 2.0/3.0和TLS 1.0/1.1
- 使用强密码套件（优先AES-GCM、ChaCha20-Poly1305）
- 启用HSTS（HTTP Strict Transport Security）
- 使用证书透明（Certificate Transparency）
- 配置OCSP Stapling
- 使用前向保密（ECDHE）

### 5.3 QUIC协议

- 基于UDP的传输协议，集成TLS 1.3
- 0-RTT连接建立
- 解决TCP队头阻塞
- 连接迁移（IP变化不断连）
- HTTP/3的传输层

## 六、DNS安全

### 6.1 DNS攻击类型

| 攻击 | 描述 |
|------|------|
| DNS缓存投毒 | 向DNS缓存注入虚假记录 |
| DNS劫持 | 修改DNS解析结果（运营商/恶意软件） |
| DNS隧道 | 在DNS查询中隐藏数据通道 |
| DNS放大攻击 | 利用DNS服务器放大DDoS流量 |
| DNS域传送漏洞 | 未授权获取完整DNS区域信息 |
| DNS重绑定 | 绕过同源策略 |

### 6.2 DNS缓存投毒

**Kaminsky攻击（2008）：**
1. 攻击者查询随机子域名（如xyz123.target.com）
2. 发送大量伪造应答（包含NS记录）
3. 如果匹配成功，整个域的解析被劫持

**防御：**
- **DNSSEC**：DNS安全扩展，使用数字签名验证DNS数据
- **0x20编码**：随机化查询大小写
- **源端口随机化**

### 6.3 DNSSEC

**核心机制：**
- **RRSIG**：资源记录的数字签名
- **DNSKEY**：公钥
- **DS**：委托签名，链接上下级信任链
- **NSEC/NSEC3**：认证不存在的记录

**信任链：**
```
根DNS密钥签名 → .com DS记录验证 → example.com DNSKEY → www.example.com RRSIG验证
```

### 6.4 DoH和DoT

| 协议 | 端口 | 传输方式 | 说明 |
|------|------|----------|------|
| 传统DNS | 53 | 明文UDP/TCP | 不加密 |
| DoT | 853 | TLS over TCP | DNS over TLS |
| DoH | 443 | HTTPS | DNS over HTTPS，难以检测/阻断 |

## 七、HTTP/HTTPS安全

### 7.1 HTTP安全头

| 头部 | 作用 |
|------|------|
| `Strict-Transport-Security` | 强制HTTPS |
| `Content-Security-Policy` | 限制资源加载来源 |
| `X-Content-Type-Options` | 防止MIME类型嗅探 |
| `X-Frame-Options` | 防止页面被iframe嵌入 |
| `X-XSS-Protection` | 启用浏览器XSS过滤 |
| `Referrer-Policy` | 控制Referer头信息 |
| `Permissions-Policy` | 控制浏览器功能权限 |

### 7.2 Cookie安全

```
Set-Cookie: session=abc123; Secure; HttpOnly; SameSite=Strict; Path=/; Max-Age=3600
```

| 属性 | 作用 |
|------|------|
| Secure | 仅通过HTTPS传输 |
| HttpOnly | 禁止JavaScript访问 |
| SameSite | 防止CSRF（Strict/Lax/None） |
| Domain | 指定可接收Cookie的域 |
| Path | 指定可接收Cookie的路径 |

### 7.3 CORS（跨源资源共享）

```
# 安全的CORS配置
Access-Control-Allow-Origin: https://trusted-domain.com
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Content-Type
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 3600
```

**危险配置：**
- `Access-Control-Allow-Origin: *` （允许所有来源）
- 配合 `Allow-Credentials: true` 会泄露凭据

## 八、无线网络安全

### 8.1 Wi-Fi安全协议演进

| 协议 | 年份 | 加密 | 安全性 |
|------|------|------|--------|
| WEP | 1999 | RC4 | 极不安全，数分钟可破解 |
| WPA | 2003 | TKIP(RC4) | 较弱 |
| WPA2 | 2004 | AES-CCMP | 安全（KRACK除外） |
| WPA3 | 2018 | AES-GCMP + SAE | 最安全 |

### 8.2 Wi-Fi攻击

| 攻击 | 描述 |
|------|------|
| 无线嗅探 | 捕获无线数据包 |
| Evil Twin | 建立同名钓鱼热点 |
| KRACK | WPA2四次握手密钥重装攻击 |
| Deauth攻击 | 发送去认证帧断开客户端连接 |
| Pixie Dust | WPS PIN离线暴力破解 |

### 8.3 WPA3改进

- **SAE（同步认证等值）**：替代PSK，防止离线字典攻击
- **前向保密**：即使密码泄露，历史流量不可解密
- **192位安全套件**：企业级高安全
- **开放网络加密**：OWE（Opportunistic Wireless Encryption）

## 九、VPN安全

### 9.1 VPN类型

| 类型 | 协议 | 特点 |
|------|------|------|
| IPSec VPN | IPSec | 站点到站点，安全性高 |
| SSL VPN | TLS | 远程访问，基于浏览器 |
| WireGuard | 自定义 | 简洁高效，代码量小 |
| OpenVPN | TLS | 开源，跨平台 |
| L2TP/IPSec | L2TP+IPSec | 结合两者优势 |
| PPTP | GRE | 已不安全 |

### 9.2 WireGuard

- 代码量仅约4000行
- 使用Curve25519、ChaCha20、Poly1305、BLAKE2s
- 基于UDP
- 内核态实现，性能优异
- 无状态设计，移动设备友好

### 9.3 VPN安全考虑

- 密钥管理
- 拆分隧道 vs 全隧道
- DNS泄漏防护
- WebRTC泄漏防护
- Kill Switch（VPN断开时阻断网络）
