# 07-安全运营与应急响应

## 一、安全运营中心（SOC）

### 1.1 SOC架构

```
┌─────────────────────────────────────────────────────┐
│                    SOC架构                           │
├─────────────────────────────────────────────────────┤
│  数据源层：                                           │
│  防火墙日志 │ IDS/IPS │ EDR │ WAF │ 邮件网关        │
│  蜜罐 │ 身份认证日志 │ 应用日志 │ 云平台日志          │
├─────────────────────────────────────────────────────┤
│  数据采集层：                                         │
│  Syslog │ API │ Agent │ SNMP │ NetFlow              │
├─────────────────────────────────────────────────────┤
│  分析平台层：                                         │
│  SIEM │ SOAR │ 威胁情报平台 │ UEBA                    │
├─────────────────────────────────────────────────────┤
│  运营层：                                             │
│  L1监控 │ L2分析 │ L3调查 │ 红蓝对抗                  │
└─────────────────────────────────────────────────────┘
```

### 1.2 SOC人员层级

| 层级 | 职责 | 技能要求 |
|------|------|----------|
| L1 | 告警监控、初步分类、工单处理 | 安全基础、SIEM操作 |
| L2 | 告警分析、事件调查、初步响应 | 威胁分析、取证基础 |
| L3 | 深度调查、威胁猎捕、高级响应 | 恶意软件分析、逆向工程 |
| 红队 | 攻击模拟、安全评估 | 渗透测试、漏洞利用 |

### 1.3 SOC关键指标（KPI）

```
- MTTD（平均检测时间）
- MTTR（平均响应时间）
- 误报率
- 告警处理量/人
- 安全事件数量趋势
- 覆盖的MITRE ATT&CK技术数
```

## 二、SIEM（安全信息和事件管理）

### 2.1 SIEM核心功能

```
SIEM = SEM（安全事件管理）+ SIM（安全信息管理）

核心功能：
├── 日志收集与归一化
├── 实时关联分析
├── 告警与通知
├── 仪表盘与报告
├── 合规性报告
├── 威胁情报集成
└── 调查与取证
```

### 2.2 常见SIEM产品

| 产品 | 特点 |
|------|------|
| Splunk | 商业标杆，查询语言强大（SPL） |
| ELK Stack | 开源方案（Elasticsearch+Logstash+Kibana） |
| QRadar | IBM产品，内置UEBA |
| Microsoft Sentinel | 云原生SIEM，Azure集成 |
| Graylog | 开源，轻量级 |
| 奇安信NGSOC | 国内主流 |
| 绿盟SIP | 国内主流 |

### 2.3 关联规则示例

```sql
-- 检测暴力破解
index=auth sourcetype=linux_secure "Failed password"
| stats count by src_ip, user
| where count > 5

-- 检测横向移动
(index=auth EventCode=4625) OR (index=process EventCode=4688)
| stats dc(dest) as unique_hosts by user
| where unique_hosts > 3

-- 检测数据渗出
index=firewall bytes_out > 100000000
| stats sum(bytes_out) as total_bytes by src_ip
| where total_bytes > 500000000
```

## 三、威胁情报

### 3.1 威胁情报分类

| 类别 | 受众 | 内容 |
|------|------|------|
| 战略情报 | 高管 | 威胁趋势、业务风险 |
| 运营情报 | 安全运营 | 攻击者TTP、活动模式 |
| 战术情报 | SOC分析员 | IOC、恶意IP/域名/哈希 |
| 技术情报 | 自动化系统 | 机器可读的IOC数据 |

### 3.2 IOC（失陷指标）

| 类型 | 示例 |
|------|------|
| IP地址 | 恶意C2服务器IP |
| 域名 | 钓鱼域名 |
| 文件哈希 | 恶意软件MD5/SHA256 |
| URL | 恶意链接 |
| 邮件 | 钓鱼邮件特征 |
| YARA规则 | 恶意软件检测规则 |
| MITRE ATT&CK ID | 攻击技术标识 |

### 3.3 威胁情报来源

```
开源情报（OSINT）：
├── MITRE ATT&CK - 攻击知识库
├── AlienVault OTX - 开放威胁交换
├── VirusTotal - 文件/URL扫描
├── Abuse.ch - 恶意软件追踪
├── PhishTank - 钓鱼URL
├── Exploit-DB - 漏洞利用
├── NVD/CVE - 漏洞数据库
└── GitHub安全公告

商业情报：
├── Recorded Future
├── Mandiant
├── CrowdStrike
├── 微步在线
└── 奇安信威胁情报
```

### 3.4 STIX/TAXII

```
STIX（结构化威胁信息表达）：
- 定义威胁信息的数据模型
- 支持攻击者、攻击活动、IOC、TTP等对象

TAXII（受信任的威胁信息自动交换）：
- 定义威胁情报交换的传输协议
- 支持发布/订阅模式
```

## 四、安全事件响应

### 4.1 事件响应流程（NIST SP 800-61）

```
┌──────────────────────────────────────────────────┐
│              安全事件响应流程                       │
├──────────┬───────────────────────────────────────┤
│ 阶段1    │ 准备（Preparation）                    │
│          │ 制定预案、组建团队、部署工具             │
├──────────┼───────────────────────────────────────┤
│ 阶段2    │ 检测与分析（Detection & Analysis）     │
│          │ 确认事件、分析范围和影响                 │
├──────────┼───────────────────────────────────────┤
│ 阶段3    │ 遏制（Containment）                    │
│          │ 短期遏制、长期遏制策略                   │
├──────────┼───────────────────────────────────────┤
│ 阶段4    │ 根除（Eradication）                    │
│          │ 清除恶意代码、修补漏洞                   │
├──────────┼───────────────────────────────────────┤
│ 阶段5    │ 恢复（Recovery）                       │
│          │ 恢复系统、验证功能                       │
├──────────┼───────────────────────────────────────┤
│ 阶段6    │ 总结改进（Post-Incident）              │
│          │ 事后分析、改进措施、知识沉淀              │
└──────────┴───────────────────────────────────────┘
```

### 4.2 事件分级

| 级别 | 描述 | 响应时间 |
|------|------|----------|
| P1-紧急 | 核心系统被入侵、数据大规模泄露 | 15分钟内 |
| P2-严重 | 重要系统被入侵、小范围数据泄露 | 1小时内 |
| P3-一般 | 普通系统被攻击、未造成实质损失 | 4小时内 |
| P4-低危 | 可疑活动、需要进一步确认 | 24小时内 |

### 4.3 常见安全事件处理

#### 勒索软件事件

```
响应步骤：
1. 隔离：断开受感染主机网络
2. 确认：识别勒索软件家族
3. 遏制：关闭共享文件夹、修改密码
4. 调查：确定感染范围和入口点
5. 恢复：从备份恢复数据
6. 加固：修补漏洞、加强防护
7. 报告：必要时报告监管机构

注意事项：
- 不要立即关机（内存中有价值证据）
- 不要支付赎金（不保证恢复，助长犯罪）
- 保留勒索信和加密样本
```

#### 数据泄露事件

```
响应步骤：
1. 确认泄露范围和数据类型
2. 阻断泄露通道
3. 保全证据
4. 评估影响
5. 通知相关方（用户、监管）
6. 修补漏洞
7. 事后复盘
```

#### DDoS攻击事件

```
响应步骤：
1. 确认攻击类型和规模
2. 启用CDN/云清洗
3. 配置黑洞路由（必要时）
4. 启用速率限制
5. 与ISP/云服务商协作
6. 监控恢复情况
```

### 4.4 事件响应预案模板

```yaml
事件响应预案:
  名称: "勒索软件应急响应预案"
  等级: P1
  触发条件:
    - 文件被加密
    - 出现勒索信
    - EDR告警

  响应团队:
    负责人: "安全经理"
    成员: ["SOC分析员", "系统管理员", "网络管理员"]

  响应步骤:
    第一阶段（0-30分钟）:
      - 确认事件真实性
      - 隔离受感染主机
      - 通知响应团队

    第二阶段（30分钟-4小时）:
      - 确定勒索软件类型
      - 评估感染范围
      - 遏制横向扩散

    第三阶段（4-24小时）:
      - 根除恶意代码
      - 恢复业务系统
      - 加固安全措施

  外部联系:
    - 公安网安部门
    - CERT团队
    - 安全厂商应急响应
```

## 五、漏洞管理

### 5.1 漏洞管理流程

```
发现 → 评估 → 修复 → 验证 → 报告

发现来源：
├── 自动化扫描（Nessus, Qualys）
├── 渗透测试
├── 漏洞公告（CVE, CNVD, CNNVD）
├── 威胁情报
├── 安全研究
└── 漏洞赏金平台
```

### 5.2 漏洞修复优先级

```
修复优先级 = 漏洞CVSS评分 × 资产价值 × 暴露面 × 可利用性

优先修复：
1. 互联网暴露的高危漏洞
2. 有公开EXP的漏洞
3. 核心业务系统漏洞
4. 被APT利用的漏洞

修复时间要求（参考）：
- 严重：24小时内
- 高危：7天内
- 中危：30天内
- 低危：90天内
```

### 5.3 补丁管理

```
补丁管理流程：
1. 补丁监控 → 跟踪厂商安全更新
2. 补丁评估 → 评估影响和兼容性
3. 测试 → 测试环境验证
4. 部署 → 生产环境分阶段部署
5. 验证 → 确认补丁安装成功
6. 回滚预案 → 万一出问题的恢复方案
```

## 六、渗透测试红蓝对抗

### 6.1 红队（Red Team）

```
红队职责：
- 模拟真实攻击者行为
- 测试防御团队检测和响应能力
- 使用TTP而非仅漏洞扫描
- 绕过安全控制
- 测试人员安全意识
```

### 6.2 蓝队（Blue Team）

```
蓝队职责：
- 监控和检测攻击
- 事件分析和响应
- 威胁猎捕
- 安全控制优化
```

### 6.3 紫队（Purple Team）

```
紫队 = 红队 + 蓝队协同
- 攻防双方共享知识
- 加速安全能力提升
- 验证安全控制有效性
```

### 6.4 ATT&CK矩阵应用

```
利用ATT&CK框架：
1. 红队：规划攻击路径（选择ATT&CK技术）
2. 蓝队：建立检测覆盖矩阵
3. 对比：发现检测盲区
4. 改进：补充检测规则
```

## 七、蜜罐与欺骗防御

### 7.1 蜜罐类型

| 类型 | 说明 |
|------|------|
| 低交互蜜罐 | 模拟服务，无真实系统 | Honeyd |
| 高交互蜜罐 | 真实系统，深度诱捕 | honeynet |
| Web蜜罐 | 模拟Web应用 | HFish |
| 数据库蜜罐 | 模拟数据库 | Delilah |
| 工控蜜罐 | 模拟SCADA/PLC | Conpot |
| 蜜标 | 带标记的诱饵数据 | Canarytokens |

### 7.2 蜜罐部署

```
部署位置：
├── 互联网边界 - 检测外部扫描
├── DMZ区域 - 检测DMZ入侵
├── 内网 - 检测横向移动
└── 核心区 - 检测针对核心资产的攻击

注意事项：
- 蜜罐IP不应出现在DNS或任何正常流量中
- 隔离蜜罐网络
- 记录所有交互
- 不要让蜜罐成为攻击跳板
```

## 八、威胁猎捕（Threat Hunting）

### 8.1 猎捕方法

```
1. 假设驱动猎捕
   - 基于ATT&CK框架提出假设
   - 设计数据查询验证假设
   - 例："假设存在横向移动，检查异常认证"

2. IOC驱动猎捕
   - 基于威胁情报的IOC
   - 搜索历史数据匹配IOC
   - 例："检查是否连接过已知C2 IP"

3. 异常驱动猎捕
   - 建立行为基线
   - 检测偏离基线的异常
   - 例："检测异常的数据外传量"
```

### 8.2 猎捕假设示例

```
假设1：攻击者可能通过PowerShell在内网横向移动
查询：搜索Base64编码的PowerShell命令
数据源：Windows事件日志（Event ID 4104）

假设2：攻击者可能通过DNS隧道渗出数据
查询：分析DNS查询频率和子域名长度异常
数据源：DNS日志

假设3：攻击者可能在内网进行凭证窃取
查询：检测Mimikatz特征和LSASS访问
数据源：EDR告警、Sysmon日志
```

## 九、SOAR（安全编排自动化响应）

### 9.1 SOAR功能

```
SOAR = 安全编排 + 自动化 + 响应

核心能力：
├── 工作流自动化 - 自动执行标准操作
├── 案例管理 - 事件跟踪和协作
├── 威胁情报管理 - IOC聚合和去重
└── 剧本（Playbook）- 标准化响应流程
```

### 9.2 自动化剧本示例

```
场景：检测到恶意IP连接

自动化流程：
1. SIEM告警触发SOAR
2. SOAR自动查询威胁情报（判断IP是否恶意）
3. 自动查询CMDB（确定受影响资产）
4. 自动在防火墙封禁IP
5. 自动隔离受感染终端
6. 自动创建工单
7. 通知安全分析员
8. 分析员确认并处置
```

### 9.3 常见SOAR平台

| 平台 | 特点 |
|------|------|
| Splunk SOAR | 与Splunk SIEM深度集成 |
| Cortex XSOAR | Palo Alto产品线 |
| IBM Resilient | IBM QRadar集成 |
| Shuffle | 开源SOAR平台 |
