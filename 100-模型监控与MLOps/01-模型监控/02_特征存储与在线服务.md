# 特征存储与在线服务 - 模型监控与MLOps


## 1. 特征工程在生产中的痛点


在机器学习系统从实验到生产的过程中，特征工程面临诸多挑战，统称为"特征工程的地狱"。


### 核心痛点


- **训练-推理偏差（Training-Serving Skew）：**
   训练时使用批量计算的特征与推理时实时计算的特征存在不一致
- **特征复用困难：**
   不同团队、不同模型之间难以共享和复用特征
- **特征计算重复：**
   相同特征被多个团队重复计算，浪费资源
- **版本管理缺失：**
   特征定义变更缺乏版本追踪，导致不可复现
- **实时特征延迟：**
   实时特征计算需要低延迟，对系统架构提出高要求
- **特征监控薄弱：**
   难以发现特征数据质量异常或分布变化


> **Important:** 训练-推理偏差是生产ML系统中最隐蔽的bug来源。据统计，约60%的线上模型性能问题可追溯到特征不一致。


## 2. Feature Store概念


Feature Store（特征存储）是专门为机器学习系统设计的特征管理平台，统一管理特征的计算、存储、服务和版本控制。


| 组件 | 功能 | 技术选型 |
| --- | --- | --- |
| 离线存储 | 批量特征计算、历史特征存储 | 数据仓库（Hive、BigQuery）、Parquet文件 |
| 在线存储 | 低延迟特征查询服务 | Redis、DynamoDB、Cassandra |
| 特征转换引擎 | 统一的特征定义和转换逻辑 | Spark、Flink、Pandas |
| 特征注册中心 | 特征元数据管理、版本控制 | 内置元数据存储 |
| 特征服务API | REST/gRPC接口获取特征 | FastAPI、gRPC服务 |


### 离线 vs 在线特征


- **离线特征：**
   基于历史数据批量计算，如"过去30天平均消费金额"，延迟要求低（分钟级），存储在数据仓库
- **在线特征：**
   基于实时事件流计算，如"最近一次浏览的品类"，延迟要求高（毫秒级），存储在内存数据库
- **关键原则：**
   离线和在线必须使用相同的特征定义和转换逻辑，保证一致性


## 3. Feast特征存储框架


Feast（Feature Store）是Google开源的特征存储框架，提供统一的特征管理接口，支持多种离线和在线存储后端。


```
from feast import FeatureStore, Entity, Feature, FeatureView, ValueType
from feast.data_source import FileSource
from datetime import timedelta

# 定义实体（Entity）：特征的主键
user = Entity(
    name="user_id",
    value_type=ValueType.INT64,
    description="用户ID"
)

# 定义数据源
user_stats_source = FileSource(
    path="data/user_stats.parquet",
    event_timestamp_column="event_timestamp",
    created_timestamp_column="created_timestamp",
)

# 定义特征视图（Feature View）
user_stats_fv = FeatureView(
    name="user_statistics",
    entities=["user_id"],
    ttl=timedelta(days=1),  # 特征有效期
    features=[
        Feature(name="avg_spend_30d", dtype=ValueType.FLOAT),
        Feature(name="purchase_count_7d", dtype=ValueType.INT64),
        Feature(name="last_login_hours", dtype=ValueType.INT64),
        Feature(name="preferred_category", dtype=ValueType.STRING),
    ],
    online=True,       # 启用在线服务
    source=user_stats_source,
    tags={"team": "recommendation"},
)

# ========== 获取特征 ==========

# 初始化Feature Store
store = FeatureStore(repo_path="./feature_repo/")

# 1. 离线特征获取（用于训练）
from feast import RepoConfig
training_df = store.get_historical_features(
    entity_df=entity_df,  # 包含user_id和event_timestamp的DataFrame
    features=[
        "user_statistics:avg_spend_30d",
        "user_statistics:purchase_count_7d",
        "user_statistics:last_login_hours",
    ],
).to_df()

# 2. 在线特征获取（用于推理）
online_features = store.get_online_features(
    features=[
        "user_statistics:avg_spend_30d",
        "user_statistics:purchase_count_7d",
    ],
    entity_rows=[{"user_id": 12345}, {"user_id": 67890}],
).to_dict()
print(online_features)
```


## 4. 在线特征服务的延迟要求


| 应用场景 | 延迟要求 | 存储方案 | 计算方式 |
| --- | --- | --- | --- |
| 搜索排序 | < 10ms | Redis Cluster | 预计算+缓存 |
| 推荐系统 | < 50ms | Redis / DynamoDB | 预计算+实时拼接 |
| 广告CTR预估 | < 30ms | Redis + 本地缓存 | 预计算为主 |
| 风控实时决策 | < 100ms | Redis + 特征流计算 | Flink实时计算 |
| 离线报表 | 分钟级 | Hive / ClickHouse | Spark批量计算 |


> **Note:** **优化技巧：**
> (1) 使用本地缓存（如LRU Cache）减少Redis访问；
>                 (2) 批量获取特征而非逐条查询；
>                 (3) 特征预计算，避免推理时做复杂计算；
>                 (4) 使用连接池减少TCP握手开销。


## 5. 特征版本管理与一致性


### 特征版本管理


- **特征注册：**
   每个特征有唯一的名称和版本号，支持变更追踪
- **血缘追踪：**
   记录特征的数据来源、转换逻辑、被哪些模型使用
- **回滚能力：**
   特征逻辑变更导致问题时，可快速回滚到上一个版本
- **A/B测试：**
   同时维护多版本特征，支持在线对比实验


### 训练-推理特征一致性


> **Example:** **确保一致性的实践：**
>
>
> 1. 使用同一套特征定义代码（如Feast的FeatureView）
>
>
> 2. 离线和在线使用相同的转换逻辑和参数
>
>
> 3. 定期对比离线特征表和在线特征的数值差异
>
>
> 4. 特征变更必须经过CI/CD流水线验证
>
>
> 5. 监控在线特征的缺失率和分布变化


| 不一致来源 | 表现 | 解决方案 |
| --- | --- | --- |
| 转换逻辑不同 | 同一特征在训练和推理时值不同 | 统一特征定义（Single Source of Truth） |
| 数据时效性差异 | 训练用T-1数据，推理用实时数据 | Point-in-Time Join |
| 缺失值处理不同 | 训练时填充方式与推理时不同 | 统一缺失值处理策略 |
| 窗口计算偏差 | "过去7天"的边界处理不一致 | 使用Feature Store统一窗口计算 |


## 总结


- 特征工程在生产中的核心挑战是训练-推理偏差，约60%的线上问题源于特征不一致
- Feature Store统一管理离线和在线特征，解决特征复用和一致性问题
- Feast是最流行的开源特征存储框架，支持多种存储后端和统一API
- 在线特征服务的延迟要求从10ms到100ms不等，需根据场景选择存储和计算方案
- 特征版本管理和血缘追踪是可复现性的关键
- 确保训练-推理特征一致是Feature Store最核心的价值
