# 27 - 结构体的性能考量

## 一、成员排序与内存对齐

结构体成员的声明顺序直接影响大小和性能。合理的排序可以减少填充字节，提高缓存效率。

### 1.1 降序排列原则

```c
// 差：大小混杂，填充多
struct Bad {
    char  a;    // 1B
    // 3B padding
    int   b;    // 4B
    char  c;    // 1B
    // 3B padding
    double d;   // 8B
    char  e;    // 1B
    // 7B padding
};  // sizeof = 32

// 好：从大到小排列，填充最少
struct Good {
    double d;   // 8B
    int   b;    // 4B
    char  a;    // 1B
    char  c;    // 1B
    char  e;    // 1B
    // 1B padding
};  // sizeof = 16
```

```
Bad 布局 (32B):                      Good 布局 (16B):
+----+----+----+----+               +----+----+----+----+
| a  |  pad3  |  b  |               |        d         |
+----+----+----+----+               +----+----+----+----+
| c  |  pad3  |  d  |               |  b  | a | c |e|p |
+----+----+----+----+               +----+----+----+----+
|  d (续)      | e  |
+----+----+----+----+               Good 节省 50% 内存！
|  pad7         |
+----+----+----+----+
```

## 二、缓存友好性

### 2.1 数据局部性

```c
// 不好：数据分散，缓存命中率低
struct Particle {
    double *x;       // 指针，数据在别处
    double *y;
    double *z;
    int    *color;
};

// 好：数据连续，缓存友好
struct Particle {
    double x;
    double y;
    double z;
    int    color;
};
```

### 2.2 AoS vs SoA

```c
// AoS (Array of Structures) —— 对象数组
struct Point {
    float x, y, z;
};
struct Point points[1000];

// SoA (Structure of Arrays) —— 结构的数组
struct PointCloud {
    float x[1000];
    float y[1000];
    float z[1000];
};
```

**对比：**

| 特性 | AoS | SoA |
|------|-----|-----|
| 缓存效率 | 单个对象好 | 批量处理好 |
| SIMD 友好 | 差 | 好 |
| 数据组织 | 自然（面向对象） | 扁平（数据并行） |
| 适用场景 | 随机访问单个对象 | 逐属性批量计算 |

```c
// AoS 适合：同时读取一个粒子的所有属性
for (int i = 0; i < n; i++) {
    process_particle(points[i]);  // 每次访问 x, y, z
}

// SoA 适合：对一个属性做批量计算
for (int i = 0; i < n; i++) {
    cloud.x[i] += velocity_x[i] * dt;  // 连续内存，缓存友好
}
```

## 三、packed 与性能权衡

### 3.1 `#pragma pack(1)` 的性能代价

```c
#pragma pack(push, 1)
struct Packed {
    char  a;
    int   b;
    short c;
};  // sizeof = 7
#pragma pack(pop)

struct Natural {
    char  a;
    short c;  // 排在 int 之前
    int   b;
};  // sizeof = 8（自然对齐）
```

**性能影响：**
- 未对齐访问：CPU 可能需要多次内存访问
- x86：性能下降 ~2x
- ARM：可能直接硬件异常

### 3.2 何时使用 packed

```
使用 pack(1) 的场景：
  - 网络协议头部
  - 文件格式定义
  - 硬件寄存器映射
  - 需要精确二进制布局

不要使用 pack(1) 的场景：
  - 普通内存中使用的结构体
  - 性能敏感的代码
  - 大量随机访问的数据结构
```

## 四、避免频繁的堆分配

```c
// 不好：每次调用都 malloc
struct Node *create_bad(void) {
    struct Node *n = malloc(sizeof(struct Node));  // 系统调用，慢
    return n;
}

// 好：使用内存池 / 对象池
#define POOL_SIZE 1024
static struct Node pool[POOL_SIZE];
static int pool_idx = 0;

struct Node *alloc_from_pool(void) {
    if (pool_idx >= POOL_SIZE) return NULL;
    return &pool[pool_idx++];
}
```

## 五、结构体大小与指针传递

```c
// 小结构体（< 16 字节）：值传递可能更快
struct Small {
    int x, y;
};
void process_small(struct Small s);  // 直接寄存器传递

// 大结构体（> 16 字节）：指针传递更高效
struct Large {
    double data[1024];
};
void process_large(const struct Large *s);  // 只传 8 字节指针
```

**经验法则：**
- 小于等于 16 字节：值传递（寄存器传递）
- 大于 16 字节：指针传递
- 需要修改：始终用指针

## 六、关键要点

> **重要提示：**
>
> 1. 成员按大小**降序排列**，可以最小化填充字节。
> 2. 频繁一起访问的成员应该放在一起，提高缓存命中率。
> 3. AoS 适合随机访问单个对象，SoA 适合批量处理单个属性。
> 4. `pack(1)` 节省内存但牺牲访问性能，只在必要时使用。
> 5. 避免频繁的 malloc/free，使用内存池可以大幅提高性能。
> 6. 大结构体传参使用指针，小结构体可以使用值传递。
