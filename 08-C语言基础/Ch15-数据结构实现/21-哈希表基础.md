# 哈希表基础

## 1. 概述

哈希表（Hash Table），又称散列表，是根据**键（Key）直接访问值（Value）**的数据结构。它通过哈希函数将键映射到数组下标，实现 O(1) 的平均访问速度。

### 核心思想
```
Key --[哈希函数]--> Index --[数组]--> Value
```

## 2. 哈希函数

哈希函数将任意大小的键映射为固定范围的整数（数组下标）。

### 2.1 好的哈希函数的要求
- **确定性**：相同输入总是产生相同输出
- **均匀性**：输出尽可能均匀分布
- **高效性**：计算速度快

### 2.2 常见哈希函数

```c
// 1. 除留余数法（最简单常用）
int hash_mod(int key, int tableSize) {
    return key % tableSize;
}

// 2. 乘法散列法
int hash_multiply(int key, int tableSize, double A) {
    double frac = key * A - (int)(key * A);
    return (int)(tableSize * frac);
}

// 3. 字符串哈希（DJB2 算法）
unsigned int hash_string(const char *str, int tableSize) {
    unsigned int hash = 5381;
    int c;
    while ((c = *str++)) {
        hash = ((hash << 5) + hash) + c;  // hash * 33 + c
    }
    return hash % tableSize;
}

// 4. FNV-1a 哈希
unsigned int fnv1a(const char *str, int tableSize) {
    unsigned int hash = 2166136261u;
    while (*str) {
        hash ^= (unsigned char)(*str++);
        hash *= 16777619u;
    }
    return hash % tableSize;
}
```

### 2.3 选择哈希函数的考虑因素
- 数据类型（整数、字符串、浮点数）
- 数据分布特性
- 性能要求

## 3. 哈希冲突

不同的键经过哈希函数计算后得到相同的下标，称为**冲突（Collision）**。

### 冲突不可避免
根据鸽巢原理，当键的数量超过表的大小时，必然存在冲突。

## 4. 冲突处理方法

### 4.1 链地址法（Separate Chaining）

每个数组位置维护一个链表，冲突的元素挂在同一个链表中。

```
Index 0: -> NULL
Index 1: -> [K1,V1] -> [K4,V4] -> NULL
Index 2: -> [K2,V2] -> NULL
Index 3: -> [K3,V3] -> [K5,V5] -> NULL
Index 4: -> NULL
```

**优点**：实现简单，不会溢出
**缺点**：需要额外指针空间，链表过长时性能下降

### 4.2 开放地址法（Open Addressing）

冲突时，在表中寻找下一个空闲位置。

#### 线性探测（Linear Probing）
```
h(key, i) = (h(key) + i) % tableSize   (i = 0, 1, 2, ...)
```
探测序列：h, h+1, h+2, h+3, ...

**问题**：容易产生**聚集（Clustering）**，连续的占用位置形成区块。

#### 二次探测（Quadratic Probing）
```
h(key, i) = (h(key) + i²) % tableSize   (i = 0, 1, 2, ...)
```
探测序列：h, h+1, h+4, h+9, ...

**优点**：减少聚集
**缺点**：可能无法探测到所有位置

#### 双重散列（Double Hashing）
```
h(key, i) = (h1(key) + i * h2(key)) % tableSize
```
使用第二个哈希函数来确定探测步长。

**优点**：探测序列更均匀
**缺点**：需要两个哈希函数

### 4.3 方法对比

| 方法 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 链地址法 | 实现简单，不溢出 | 额外指针开销 | 通用场景 |
| 线性探测 | 缓存友好 | 容易聚集 | 装载因子低 |
| 二次探测 | 减少聚集 | 可能无法全覆盖 | 中等装载因子 |
| 双重散列 | 分布最均匀 | 计算开销大 | 高装载因子 |

## 5. 装载因子

**装载因子（Load Factor）** = 已存储元素数量 / 哈希表大小

```
装载因子 α = n / m
n = 元素个数
m = 表大小
```

- **链地址法**：α 可以大于 1（链表可以无限增长）
- **开放地址法**：α 必须小于 1（通常控制在 0.7 以下）
- α 越大，冲突越多，性能越差

### 平均查找长度

| 方法 | 查找成功 | 查找失败 |
|------|---------|---------|
| 链地址法 | 1 + α/2 | α |
| 线性探测 | (1+1/(1-α))/2 | (1+1/(1-α)²)/2 |

## 7. 重点总结

| 要点 | 说明 |
|------|------|
| 哈希函数是关键 | 决定数据分布和冲突率 |
| 冲突不可避免 | 需要选择合适的处理方法 |
| 链地址法最常用 | 实现简单，不溢出 |
| 装载因子影响性能 | 一般控制在 0.75 以下 |
| 平均 O(1) | 最坏情况 O(n)（所有键冲突） |
| 动态扩容保证性能 | 扩容时重新哈希所有元素 |

> **核心思想**：哈希表的本质是用空间换时间。通过预先分配足够大的数组，将查找从 O(n) 降低到 O(1)。
