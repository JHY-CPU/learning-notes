# 35 - 数据结构选择指南

## 概述

选择合适的数据结构是程序设计的关键。本节系统对比各种数据结构的特点和适用场景，帮助在实际开发中做出正确选择。

---

## 2. 按操作需求选择

### 需要快速查找

| 场景 | 推荐数据结构 | 查找时间 |
|------|-------------|----------|
| 精确查找（无序） | 哈希表 | O(1) 平均 |
| 精确查找（有序） | 平衡BST/跳表 | O(log n) |
| 范围查询 | 平衡BST/B+树 | O(log n + k) |
| 前缀匹配 | Trie树 | O(m) |
| 近似存在性 | 布隆过滤器 | O(k) |

### 需要频繁插入/删除

| 场景 | 推荐数据结构 | 操作时间 |
|------|-------------|----------|
| 头部操作 | 双向链表 | O(1) |
| 尾部操作 | 动态数组/链表 | O(1) 平均 |
| 任意位置 | 链表 | O(1) 找到后 |
| 有序插入 | 平衡BST/跳表 | O(log n) |

### 需要最大/最小值

| 场景 | 推荐数据结构 | 操作时间 |
|------|-------------|----------|
| 取最大/最小 | 堆 | O(1) |
| 删除最大/最小 | 堆 | O(log n) |
| 有序遍历所有元素 | 平衡BST | O(n) |

```c
    // ...
} LRUCache;
```

**选择理由**：需要O(1)查找和O(1)更新访问顺序

### 场景2：排行榜

```c
// 方案1: 有序数组 - 适合小规模、更新不频繁
// 方案2: 跳表 - 适合大规模、频繁更新
// 方案3: 堆 - 只需要Top-K
```

**选择理由**：
- Top-K：用堆（O(n log k)）
- 完整排序：用跳表或平衡BST
- 小规模简单场景：有序数组

### 场景3：字典/单词本

```c
// 推荐：Trie树
// 支持前缀匹配、自动补全
```

**选择理由**：需要高效的前缀匹配功能

### 场景4：图的最短路径

```c
// 推荐：邻接表 + 堆优化Dijkstra
// 邻接表：稀疏图空间效率高
// 堆：高效取出当前最短距离的节点
```

**选择理由**：稀疏图用邻接表节省空间，堆加速Dijkstra

### 场景5：去重

```c
// 小数据：哈希表
// 大数据、允许假阳性：布隆过滤器
// 需要有序：平衡BST/跳表
```

## 5. 常见错误选择

| 错误选择 | 问题 | 正确选择 |
|----------|------|----------|
| 用链表做频繁查找 | 查找O(n) | 哈希表 |
| 用数组做频繁头部插入 | 移动O(n) | 链表或双端队列 |
| 用哈希表做范围查询 | 不支持 | 平衡BST或跳表 |
| 用普通BST处理最坏情况 | 可能退化O(n) | 平衡BST |
| 用数组模拟大量稀疏索引 | 浪费空间 | 哈希表 |

## 6. 决策流程图

```
需要存储数据？
├── 需要键值映射？
│   ├── 需要有序 → 红黑树/跳表
│   └── 不需要有序 → 哈希表
├── 需要优先级？
│   ├── 需要排序 → 堆
│   └── 只取极值 → 堆
├── 需要字符串操作？
│   ├── 前缀匹配 → Trie
│   └── 精确匹配 → 哈希表
├── 需要判断集合关系？
│   └── 连通性 → 并查集
├── 数据有序？
│   ├── 需要频繁查找 → 平衡BST
│   └── 主要遍历 → 有序数组
└── 数据无序？
    ├── 需要随机访问 → 数组
    └── 频繁增删 → 链表
```

## 7. C语言标准库中的数据结构

C标准库提供的数据结构有限，但有一些基础工具：

| 头文件 | 功能 |
|--------|------|
| `<stdlib.h>` | `qsort()` 排序、`bsearch()` 二分查找 |
| `<string.h>` | `memcmp()`, `memcpy()` 等内存操作 |

其他数据结构需要自行实现或使用第三方库（如GLib、uthash）。

---

## 要点总结

1. 没有万能的数据结构，选择取决于操作需求
2. 查找频繁用哈希表，有序需求用平衡树
3. 注意时间复杂度中的隐藏常数和最坏情况
4. 实际选择要考虑数据规模、内存限制和操作频率
5. 组合数据结构（如LRU = 哈希表 + 双向链表）往往能解决复杂问题
