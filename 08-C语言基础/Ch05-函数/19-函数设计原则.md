# 19 - 函数设计原则

## 一、单一职责原则

### 1.1 一个函数只做一件事

```c
// 不好的设计：一个函数做多件事
void process_student(const char *name, int age, double score) {
    // 验证
    if (!name || age < 0 || score < 0) return;

    // 格式化输出
    printf("学生: %s\n", name);
    printf("年龄: %d\n", age);
    printf("成绩: %.1f\n", score);

    // 写入文件
    FILE *f = fopen("students.txt", "a");
    if (f) {
        fprintf(f, "%s,%d,%.1f\n", name, age, score);
        fclose(f);
    }

    // 更新全局统计
    g_total_students++;
    g_total_score += score;
}
```

```c
// 好的设计：每个函数只做一件事
int validate_student(const char *name, int age, double score) {
    return name && age > 0 && score >= 0;
}

void print_student(const char *name, int age, double score) {
    printf("学生: %s, 年龄: %d, 成绩: %.1f\n", name, age, score);
}

int save_student(const char *name, int age, double score) {
    FILE *f = fopen("students.txt", "a");
    if (!f) return -1;
    fprintf(f, "%s,%d,%.1f\n", name, age, score);
    fclose(f);
    return 0;
}

void update_stats(int age, double score) {
    g_total_students++;
    g_total_score += score;
}
```

## 二、高内聚低耦合

### 2.1 高内聚

函数内部的代码应该紧密相关，完成一个完整的功能：

```c
// 高内聚：所有操作都与"排序"紧密相关
void sort_and_print(int arr[], int n) {
    // 排序逻辑
    for (int i = 0; i < n - 1; i++)
        for (int j = 0; j < n - 1 - i; j++)
            if (arr[j] > arr[j+1]) {
                int t = arr[j]; arr[j] = arr[j+1]; arr[j+1] = t;
            }

    // 打印也跟排序相关，可以接受
    for (int i = 0; i < n; i++)
        printf("%d ", arr[i]);
    printf("\n");
}
```

### 2.2 低耦合

函数之间的依赖应该尽可能少：

```c
// 低耦合：函数不依赖全局状态
int count_positive(const int arr[], int n) {
    int count = 0;
    for (int i = 0; i < n; i++)
        if (arr[i] > 0) count++;
    return count;
}

// 高耦合：依赖全局变量和文件系统
int count_positive_bad(const int arr[], int n) {
    // 依赖全局配置、读文件、写日志……
    if (g_config.debug) log_to_file("counting...");
    // ...
}
```

## 三、函数长度

### 3.1 推荐长度

| 建议         | 说明                                          |
|-------------|---------------------------------------------|
| 20-30 行    | 一个屏幕能看完                                  |
| 最多 50 行   | 超过应考虑拆分                                  |
| 超过 100 行  | 必须重构                                      |

### 3.2 何时拆分函数

```c
// 过长的函数——应该拆分
void process_data(const char *filename) {
    // 50 行读取逻辑...
    // 30 行验证逻辑...
    // 40 行转换逻辑...
    // 20 行输出逻辑...
}
// 总共 140 行——应该拆成 4 个函数
```

## 四、参数数量

### 4.1 参数数量建议

| 参数个数 | 建议                     |
|---------|-------------------------|
| 0-3     | 最佳                    |
| 4-5     | 可接受                   |
| 6+      | 考虑用结构体封装或重构    |

```c
// 参数太多——不好
int create_user(const char *name, int age, const char *email,
                const char *phone, const char *address,
                int role, int status, const char *password);

// 好的设计——用结构体封装
typedef struct {
    const char *name;
    int age;
    const char *email;
    const char *phone;
    const char *address;
    int role;
    int status;
    const char *password;
} UserInfo;

int create_user(const UserInfo *info);
```

## 五、命名规范

### 5.1 函数命名原则

```c
// 动词 + 名词：清晰表达做什么
int get_count(void);            // 获取
void set_name(const char *s);   // 设置
int is_valid(void);             // 判断
void print_report(void);        // 打印
int find_index(int target);     // 查找
void remove_item(int id);       // 删除
void create_node(void);         // 创建
void destroy_list(void);        // 销毁
```

### 5.2 一致性

```c
// 保持风格一致——要么全用动词开头，要么统一风格
// 好：
list_create()
list_destroy()
list_push()
list_pop()

// 不好（风格混乱）：
create_list()
list_destroy()
push()
pop_from_list()
```

## 六、错误处理

### 6.1 返回错误码

```c
// 返回 0 表示成功，负数表示错误
int open_connection(const char *host, int port, Connection **out) {
    if (!host || !out) return -1;  // 参数错误

    Connection *conn = malloc(sizeof(Connection));
    if (!conn) return -2;          // 内存不足

    // ... 连接逻辑
    *out = conn;
    return 0;                      // 成功
}
```

### 6.2 检查前置条件

```c
int safe_divide(int a, int b, int *result) {
    if (b == 0) return -1;   // 前置检查
    if (!result) return -2;  // 空指针检查
    *result = a / b;
    return 0;
}
```

## 七、函数的可测试性

```c
// 好的设计：无副作用，易于测试
int is_leap_year(int year) {
    return (year % 4 == 0 && year % 100 != 0) || (year % 400 == 0);
}
// 可以用简单的断言测试：
// assert(is_leap_year(2000) == 1);
// assert(is_leap_year(1900) == 0);

// 差的设计：依赖全局状态和时间
int is_today_holiday(void) {
    time_t now = time(NULL);         // 依赖系统时间
    struct tm *t = localtime(&now);  // 依赖系统时区
    return g_holidays[t->tm_yday];   // 依赖全局数据
    // 难以测试！
}
```

## 八、要点总结

> **关键点：**
> 1. **单一职责**：一个函数只做一件事，做好一件事。
> 2. **高内聚低耦合**：函数内部紧密相关，函数之间依赖尽量少。
> 3. **函数长度**：保持在 20-30 行以内，一个屏幕能看完。
> 4. **参数数量**：最好不超过 3-5 个，太多时用结构体封装。
> 5. **命名清晰**：使用动词+名词的命名方式，保持风格一致。
> 6. **错误处理**：返回错误码，检查前置条件，避免空指针。
> 7. **可测试性**：减少副作用，输入决定输出，避免依赖全局状态。
> 8. **文档化**：复杂的函数应该有注释说明用途、参数和返回值。
