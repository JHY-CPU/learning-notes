# 26 - 静态库创建与使用

## 一、什么是静态库

**静态库（Static Library）** 是将多个目标文件（`.o`）打包成一个归档文件，在链接阶段被完整地复制到可执行文件中。程序运行时不再需要库文件。

```
源文件 (.c) ──编译──> 目标文件 (.o) ──打包──> 静态库 (.a)
                                                  |
可执行文件 <──────────────────────链接────────────┘
（库代码被完整复制到可执行文件中）
```

| 平台     | 静态库扩展名 | 工具        |
|---------|-------------|------------|
| Linux   | `.a`        | `ar`       |
| macOS   | `.a`        | `ar`       |
| Windows | `.lib`      | `lib`      |

## 二、创建静态库

### 2.1 示例项目结构

```
mylib/
  include/
    math_utils.h
  src/
    math_utils.c
    string_utils.c
  lib/              -- 输出目录
```

### 2.2 第一步：编译为目标文件

```bash
# 将每个 .c 文件编译为 .o 文件
gcc -c -I include -o math_utils.o src/math_utils.c
gcc -c -I include -o string_utils.o src/string_utils.c
```

### 2.3 第二步：用 ar 打包

```bash
# ar = archiver（归档器）
# r = replace（替换已有的）
# c = create（创建库）
# s = create index（创建索引，等价于 ranlib）

ar rcs libmyutils.a math_utils.o string_utils.o
```

### 2.4 ar 命令参数

| 参数 | 说明                              |
|-----|----------------------------------|
| `r` | 将文件插入库中，已存在则替换         |
| `c` | 创建库（如果不存在）                |
| `s` | 创建目标文件索引（等同于 `ranlib`）  |
| `t` | 列出库中的文件                     |
| `x` | 从库中提取文件                     |
| `d` | 从库中删除文件                     |

### 2.5 查看库内容

```bash
# 列出库中的目标文件
ar t libmyutils.a
# 输出:
# math_utils.o
# string_utils.o

# 查看库中的符号
nm libmyutils.a
```

## 三、使用静态库

### 3.1 编译链接

```bash
# 方式一：直接在命令行指定
gcc -o myapp main.c -I include -L lib -lmyutils

# 方式二：指定完整路径
gcc -o myapp main.c -I include lib/libmyutils.a
```

### 3.2 参数说明

| 参数        | 说明                                     |
|------------|-----------------------------------------|
| `-I dir`   | 添加头文件搜索路径                         |
| `-L dir`   | 添加库文件搜索路径                         |
| `-lname`   | 链接名为 `libname.a` 或 `libname.so` 的库 |
| `-static`  | 强制使用静态链接                           |

### 3.3 链接顺序很重要

```bash
# 错误：库在依赖它的文件之前
gcc -o myapp -lmyutils main.c  # 可能链接失败！

# 正确：依赖关系从左到右
gcc -o myapp main.c -lmyutils  # 先源文件，后库
```

> **规则：** 链接器按从左到右的顺序解析符号。被依赖的库应该放在依赖它的文件/库**之后**。

## 四、完整的 Makefile 示例

```makefile
CC = gcc
CFLAGS = -Wall -Wextra -I include
AR = ar
ARFLAGS = rcs

# 目录
SRC_DIR = src
INC_DIR = include
LIB_DIR = lib
OBJ_DIR = obj

# 源文件和目标文件
SRCS = $(wildcard $(SRC_DIR)/*.c)
OBJS = $(patsubst $(SRC_DIR)/%.c, $(OBJ_DIR)/%.o, $(SRCS))

# 静态库
LIB = $(LIB_DIR)/libmyutils.a

# 目标
.PHONY: all clean

all: $(LIB)

$(OBJ_DIR)/%.o: $(SRC_DIR)/%.c | $(OBJ_DIR)
	$(CC) $(CFLAGS) -c $< -o $@

$(LIB): $(OBJS) | $(LIB_DIR)
	$(AR) $(ARFLAGS) $@ $^

$(OBJ_DIR) $(LIB_DIR):
	mkdir -p $@

clean:
	rm -rf $(OBJ_DIR) $(LIB_DIR)
```

## 五、静态库的优缺点

### 5.1 优点

| 优点           | 说明                                       |
|---------------|-------------------------------------------|
| 部署简单        | 可执行文件自包含，无需附带库文件              |
| 运行时无依赖    | 不依赖外部库文件的存在                       |
| 启动快          | 无需动态加载                                 |
| 版本确定        | 链接时确定了库的版本，不会有版本冲突          |
| 优化好          | 链接器可以移除未使用的函数                    |

### 5.2 缺点

| 缺点           | 说明                                       |
|---------------|-------------------------------------------|
| 文件体积大      | 库代码被复制到每个可执行文件中                |
| 内存浪费        | 多个程序运行时各自有一份库代码的副本           |
| 更新困难        | 库更新需要重新编译链接所有使用它的程序          |
| 重复存储        | 同一个库在多个程序中各有一份                   |

## 六、实际应用场景

### 6.1 适合使用静态库的场景

```
1. 小型工具程序，希望自包含
2. 嵌入式系统，没有动态链接器
3. 分发闭源库，不想暴露 .so 文件
4. 对启动速度要求极高
5. 库的 API 不稳定，避免版本兼容问题
```

### 6.2 常见的静态库

```bash
# 标准 C 库的静态版本
/usr/lib/x86_64-linux-gnu/libc.a

# 数学库
/usr/lib/x86_64-linux-gnu/libm.a

# pthread 库
/usr/lib/x86_64-linux-gnu/libpthread.a
```

## 七、要点总结

> **关键点：**
> 1. 静态库是 `.o` 文件的打包，使用 `ar` 命令创建。
> 2. 创建步骤：`.c` -> `.o` -> `.a`。
> 3. 链接时使用 `-L` 指定库路径，`-l` 指定库名（去掉 `lib` 和 `.a`）。
> 4. **链接顺序重要**：被依赖的库放在后面。
> 5. 静态库代码被复制到可执行文件中，运行时不再需要库文件。
> 6. 优点是部署简单、无运行时依赖；缺点是体积大、更新困难。
