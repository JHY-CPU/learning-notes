# 12 - 内联函数

## 一、什么是内联函数

**内联函数（Inline Function）** 是使用 `inline` 关键字声明的函数，建议编译器将函数调用直接替换为函数体代码，从而消除函数调用的开销。

```c
// 普通函数调用
int add(int a, int b) { return a + b; }
int r = add(3, 5);  // 有函数调用开销（压栈、跳转、返回）

// 内联函数
inline int add(int a, int b) { return a + b; }
int r = add(3, 5);  // 编译器可能将其展开为：int r = 3 + 5;
```

## 二、inline 关键字

### 2.1 基本语法

```c
inline 返回值类型 函数名(参数列表) {
    // 函数体
}
```

### 2.2 示例

```c
#include <stdio.h>

inline int square(int x) {
    return x * x;
}

inline double max_double(double a, double b) {
    return a > b ? a : b;
}

int main() {
    printf("square(5) = %d\n", square(5));
    printf("max(3.14, 2.71) = %.2f\n", max_double(3.14, 2.71));
    return 0;
}
```

### 2.3 内联的展开过程

```c
// 源代码
inline int square(int x) { return x * x; }
int main() {
    int a = 5;
    int b = square(a);        // 编译器尝试展开
    int c = square(a + 1);    // 编译器尝试展开
    return 0;
}

// 编译器可能将其优化为：
int main() {
    int a = 5;
    int b = a * a;            // 替换为函数体
    int c = (a + 1) * (a + 1); // 替换为函数体
    return 0;
}
```

## 三、inline 只是"建议"

### 3.1 编译器可以忽略 inline

`inline` 关键字只是对编译器的**建议**，不是强制要求。编译器可能因为以下原因选择不内联：

```c
// 编译器可能不内联的情况
inline int complex_func(int x) {
    // 函数体太长（通常超过 10-20 行）
    int sum = 0;
    for (int i = 0; i < 1000; i++) sum += i;
    // ... 很多代码
    return sum;
}

inline void recursive_inline(int n) {
    if (n <= 0) return;
    recursive_inline(n - 1);  // 递归函数无法完全内联
}
```

### 3.2 强制内联（GCC/Clang）

```c
// GCC/Clang 扩展
__attribute__((always_inline)) inline int add(int a, int b) {
    return a + b;
}

// MSVC
__forceinline int add(int a, int b) {
    return a + b;
}
```

## 四、内联函数与宏的对比

### 4.1 宏的陷阱

```c
// 宏的问题
#define SQUARE(x) ((x) * (x))

int a = 5;
int b = SQUARE(a++);    // 展开为: ((a++) * (a++))
                        // a 被自增了两次！未定义行为

int c = SQUARE(a + 1);  // 展开为: ((a + 1) * (a + 1)) OK

// 宏没有类型检查
#define ADD(a, b) ((a) + (b))
int r = ADD("hello", 5);  // 编译不报错，运行出错
```

### 4.2 内联函数的安全性

```c
// 内联函数：类型安全
inline int square(int x) { return x * x; }

int a = 5;
int b = square(a++);    // a++ 只执行一次，传入 5
                        // b = 25, a = 6  （符合预期）

// int r = square("hello");  // 编译错误！类型不匹配
```

### 4.3 对比总结

| 特性           | 宏（#define）              | 内联函数（inline）          |
|---------------|--------------------------|--------------------------|
| 处理阶段       | 预处理阶段（文本替换）     | 编译阶段（语法分析）        |
| 类型检查       | **无**                    | **有**                    |
| 作用域         | 无作用域概念              | 遵循 C 作用域规则          |
| 调试           | 难以调试                  | 可以调试（在未展开时）      |
| 副作用         | 可能导致多次求值           | 参数只求值一次              |
| 括号陷阱       | 必须大量使用括号          | 无此问题                   |
| 返回值         | 没有返回值概念            | 有明确的返回类型            |

## 五、适用场景

### 5.1 适合内联的情况

```c
// 1. 非常短小的函数（1-5 行）
inline int max(int a, int b) { return a > b ? a : b; }
inline int min(int a, int b) { return a < b ? a : b; }
inline int abs(int x) { return x < 0 ? -x : x; }

// 2. 频繁调用的简单函数
inline void set_pixel(Image *img, int x, int y, Color c) {
    img->data[y * img->width + x] = c;
}

// 3. 类型安全的通用操作（替代宏）
inline int clamp(int val, int lo, int hi) {
    if (val < lo) return lo;
    if (val > hi) return hi;
    return val;
}
```

### 5.2 不适合内联的情况

```c
// 1. 函数体较长
void process_data(const char *data, int len) {
    // 50+ 行代码...
    // 内联会导致代码膨胀
}

// 2. 递归函数
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 递归调用
}

// 3. 函数地址被取用
inline int add(int a, int b) { return a + b; }
int (*fp)(int, int) = add;  // 需要函数有地址，可能无法内联
```

## 六、static inline

### 6.1 头文件中的内联函数

在头文件中定义内联函数时，通常使用 `static inline`：

```c
// math_utils.h
#ifndef MATH_UTILS_H
#define MATH_UTILS_H

static inline int max(int a, int b) {
    return a > b ? a : b;
}

static inline int min(int a, int b) {
    return a < b ? a : b;
}

#endif
```

`static inline` 的含义：
- `static`：每个编译单元有自己的副本，避免链接冲突
- `inline`：建议编译器内联展开

### 6.2 为什么需要 static

```c
// 在头文件中仅用 inline：
inline int add(int a, int b) { return a + b; }

// 如果多个 .c 文件都包含了这个头文件：
// main.c 和 utils.c 都有 add 的定义
// 链接器需要知道如何处理重复的定义
// 使用 static 可以让每个 .c 文件有自己的副本，避免冲突
```

## 七、内联与性能

### 7.1 优化效果

```c
// 未内联的函数调用开销
call function        ; 函数调用指令
push rbp             ; 保存帧指针
mov rbp, rsp         ; 设置新帧指针
; ... 函数体
pop rbp              ; 恢复帧指针
ret                  ; 返回

// 内联后（假设）
mov eax, 3           ; 直接计算
add eax, 5           ; 无函数调用开销
```

### 7.2 代码膨胀的代价

```c
// 一个 100 行的函数被内联到 100 个调用点
// = 10000 行代码，可能导致：
// - 指令缓存未命中增加
// - 可执行文件体积增大
// - 实际性能可能反而下降
```

## 八、要点总结

> **关键点：**
> 1. `inline` 建议编译器将函数调用替换为函数体，消除调用开销。
> 2. `inline` 是建议不是命令，编译器可以忽略。
> 3. 内联函数比宏更安全：有类型检查、无副作用、可调试。
> 4. 头文件中使用 `static inline`，避免多文件链接冲突。
> 5. 适合内联：短小（1-5 行）、频繁调用的函数。
> 6. 不适合内联：长函数、递归函数、需要取地址的函数。
> 7. 过度内联会导致代码膨胀，反而影响性能。
> 8. 现代编译器在 `-O2` 优化下会自动内联合适的函数，即使没有 `inline` 关键字。
