# 22 - 动态链接详解

## 动态链接概述

动态链接（Dynamic Linking）将库代码的链接推迟到**程序运行时**。与静态链接不同，动态库代码不复制到可执行文件中，而是在运行时由动态链接器加载。

## 共享库

### 创建共享库

```bash
# 编译为位置无关代码（PIC）
gcc -c -fPIC shared.c -o shared.o

# 创建共享库
gcc -shared -o libshared.so shared.o

# 一步完成
gcc -shared -fPIC -o libshared.so shared1.c shared2.c
```

### 使用共享库

```bash
# 编译时链接
gcc main.c -L./lib -lshared -o program

# 运行时需要设置库路径
export LD_LIBRARY_PATH=./lib:$LD_LIBRARY_PATH
./program

# 或者安装到系统路径
sudo cp libshared.so /usr/local/lib/
sudo ldconfig
```

## 位置无关代码（PIC）

### 为什么需要 PIC

共享库在不同进程中可能被加载到不同的内存地址。PIC（Position Independent Code）使得代码可以在任意地址运行。

```bash
# 生成 PIC 代码
gcc -fPIC -c lib.c -o lib.o

# 查看是否使用了 GOT
objdump -d lib.o | grep -i got
```

### GOT（全局偏移表）

GOT 存储全局变量和函数的实际地址。访问全局数据通过 GOT 间接进行：

```asm
; PIC 代码访问全局变量
mov rax, [rip + offset_to_GOT]  ; 从 GOT 读取地址
mov rax, [rax]                   ; 读取变量值
```

### PLT（过程链接表）

PLT 用于函数调用的延迟绑定：

```
调用 printf → jmp *GOT[printf] → PLT[printf] → 解析 printf 地址 → 更新 GOT[printf]
                ↓
            后续调用直接跳转到 printf
```

## 延迟绑定（Lazy Binding）

### 工作原理

函数地址在**第一次调用时**才解析，而非程序启动时：

```
首次调用 printf:
1. 调用 printf@plt
2. PLT 跳转到 GOT（初始指向 PLT[0]）
3. PLT[0] 调用动态链接器
4. 链接器解析 printf 的真实地址
5. 更新 GOT[printf] = 真实地址
6. 跳转到 printf

后续调用 printf:
1. 调用 printf@plt
2. PLT 跳转到 GOT（现在指向真实地址）
3. 直接执行 printf
```

### 禁用延迟绑定

```bash
# 程序启动时立即绑定所有符号
gcc -Wl,-z,now main.c -o main

# 或设置环境变量
LD_BIND_NOW=1 ./program
```

## 动态加载（dlopen）

### 运行时加载库

```c
#include <dlfcn.h>
#include <stdio.h>

int main(void) {
    // 打开共享库
    void *handle = dlopen("./libplugin.so", RTLD_LAZY);
    if (!handle) {
        fprintf(stderr, "加载失败: %s\n", dlerror());
        return 1;
    }
    
    // 查找符号
    typedef int (*func_t)(int);
    func_t func = (func_t)dlsym(handle, "process");
    
    char *error = dlerror();
    if (error) {
        fprintf(stderr, "符号查找失败: %s\n", error);
        dlclose(handle);
        return 1;
    }
    
    // 调用函数
    int result = func(42);
    printf("结果: %d\n", result);
    
    // 关闭库
    dlclose(handle);
    return 0;
}
```

```bash
# 编译时需要链接 dl 库
gcc main.c -ldl -o main
```

### dlopen 标志

```c
// RTLD_LAZY   — 延迟绑定（默认）
// RTLD_NOW    — 立即绑定所有符号
// RTLD_GLOBAL — 导出符号供后续加载的库使用
// RTLD_LOCAL  — 符号不导出（默认）
// RTLD_DEEPBIND — 优先使用库自身的符号
```

## 查看动态库信息

```bash
# 查看程序依赖的动态库
ldd program

# 查看动态符号表
nm -D libshared.so

# 查看动态库信息
readelf -d libshared.so

# 查看共享库的 SONAME
objdump -p libshared.so | grep SONAME
```

## 版本管理

### SONAME

```bash
# 创建带版本号的库
gcc -shared -fPIC -Wl,-soname,libfoo.so.1 -o libfoo.so.1.0.0 foo.c

# 创建符号链接
ln -s libfoo.so.1.0.0 libfoo.so.1
ln -s libfoo.so.1 libfoo.so

# 编译程序时使用
gcc main.c -L. -lfoo -o program
```

## 动态链接的优缺点

### 优点

- 多个程序**共享同一份库代码**，节省内存
- 库更新时**不需要重新编译**程序
- 减小可执行文件体积
- 支持**插件架构**（运行时加载）

### 缺点

- 运行时需要找到库文件
- **启动速度稍慢**（需要加载和解析库）
- 版本兼容性问题（DLL Hell）
- 略微的性能开销（间接调用）

## 重要注意事项

> **关键点总结**：
> 1. 动态库使用 `-shared -fPIC` 编译，生成位置无关代码
> 2. **GOT** 和 **PLT** 是动态链接的核心数据结构
> 3. **延迟绑定**将函数地址解析推迟到第一次调用
> 4. `dlopen`/`dlsym`/`dlclose` 提供运行时加载能力
> 5. `ldd` 查看程序的动态库依赖
> 6. 使用 `LD_LIBRARY_PATH` 或 `ldconfig` 管理库搜索路径
