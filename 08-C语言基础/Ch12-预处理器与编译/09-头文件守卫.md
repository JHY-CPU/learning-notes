# 09 - 头文件守卫

## 重复包含问题

当一个头文件被多次包含时，会导致**重定义错误**。头文件守卫（Header Guard）是最常用的解决方案。

### 问题示例

```c
// types.h（没有守卫）
typedef struct {
    int x;
    int y;
} Point;

// main.c
#include "types.h"
#include "utils.h"  // utils.h 内部也 #include "types.h"

// 编译错误: 重定义 typedef 'Point'
// 因为 types.h 被包含了两次
```

## 传统头文件守卫：#ifndef/#define/#endif

### 基本结构

```c
// myheader.h
#ifndef MYHEADER_H    // 第1步：检查宏是否已定义
#define MYHEADER_H    // 第2步：定义宏（标记"已包含"）

// 头文件的实际内容
typedef struct {
    int value;
    char name[32];
} Item;

void process_item(Item *item);

#endif /* MYHEADER_H */  // 第3步：结束条件编译
```

### 工作原理

```
第一次包含 myheader.h:
1. MYHEADER_H 未定义 → #ifndef 条件为真
2. #define MYHEADER_H → 宏被定义
3. 头文件内容被处理
4. #endif 结束

第二次包含 myheader.h:
1. MYHEADER_H 已定义 → #ifndef 条件为假
2. 跳过整个头文件内容
3. #endif 结束
```

### 命名规范

宏名通常由头文件路径或名称派生，避免冲突：

```c
// 文件: include/utils/string_utils.h
#ifndef UTILS_STRING_UTILS_H
#define UTILS_STRING_UTILS_H

// 内容...

#endif /* UTILS_STRING_UTILS_H */

// 文件: src/network/connection.h
#ifndef NETWORK_CONNECTION_H
#define NETWORK_CONNECTION_H

// 内容...

#endif /* NETWORK_CONNECTION_H */
```

常见的命名模式：
- `PROJECT_MODULE_FILENAME_H`
- `PROJECT_FILENAME_H`
- `_FILENAME_H_`（以下划线开头，但不推荐——以下划线开头的标识符保留给实现）

## #pragma once

### 基本用法

```c
// myheader.h
#pragma once

// 头文件内容
typedef struct {
    int value;
} Data;
```

### 优势

1. **更简洁**：只需一行，不需要命名宏
2. **无命名冲突**：不存在宏名冲突的可能
3. **编译器优化**：某些编译器可以更高效地处理 `#pragma once`

### 劣势

1. **非标准**：`#pragma once` 不是 C 标准的一部分（但几乎所有现代编译器都支持）
2. **符号链接问题**：如果文件通过符号链接被多次访问，某些编译器可能无法正确识别
3. **文件系统依赖**：依赖编译器对文件路径的解析能力

### 编译器支持

| 编译器 | 支持情况 |
|--------|----------|
| GCC | 支持（4.2+） |
| Clang | 支持 |
| MSVC | 支持 |
| Intel C++ | 支持 |
| 旧版编译器 | 可能不支持 |

## 两种方式的对比

```c
// 方式1: 传统守卫（可移植）
#ifndef CONFIG_H
#define CONFIG_H

#define VERSION "1.0"
#define MAX_CONN 100

#endif /* CONFIG_H */

// 方式2: pragma once（更简洁但非标准）
#pragma once

#define VERSION "1.0"
#define MAX_CONN 100
```

## 双重保护策略

某些项目同时使用两种方式以确保兼容性：

```c
// 兼容性最好的写法
#pragma once
#ifndef MYHEADER_H
#define MYHEADER_H

// 头文件内容...

#endif /* MYHEADER_H */
```

## 常见错误

### 错误1：宏名拼写不一致

```c
// myheader.h — 错误示例
#ifndef MYHEADER_H
#define MYHEADEAR_H  // 拼写错误！H 和 E 颠倒

// 内容...

#endif /* MYHEADER_H */
// 问题：宏从未被正确定义，导致每次包含都会重新处理
```

### 错误2：忘记 #define

```c
// 错误
#ifndef MYHEADER_H
// 忘记 #define MYHEADER_H

// 内容...

#endif
// 问题：#ifndef 永远为真，守卫形同虚设
```

### 错误3：#endif 位置错误

```c
// 错误
#ifndef MYHEADER_H
#define MYHEADER_H

void function1(void);
void function2(void);

// 一些其他代码（应该被守卫保护的）

#endif  // #endif 放在了太早的位置

// 这些代码不受守卫保护
typedef struct { int x; } Data;
```

### 错误4：宏名与其他文件冲突

```c
// file1.h
#ifndef UTILS_H  // 太通用的名称
#define UTILS_H
// ...
#endif

// file2.h
#ifndef UTILS_H  // 同样的宏名！
#define UTILS_H
// ...
#endif
// 问题：如果先包含 file1.h，file2.h 的内容会被跳过
```

## 项目级别的命名规范

### 推荐的命名方案

```c
// 对于项目 "MyApp", 模块 "network", 文件 "socket.h"
#ifndef MYAPP_NETWORK_SOCKET_H
#define MYAPP_NETWORK_SOCKET_H

// 或使用项目缩写
#ifndef MA_NET_SOCKET_H
#define MA_NET_SOCKET_H

#endif /* MA_NET_SOCKET_H */
```

## 检测头文件守卫问题

```bash
# 查找可能缺少守卫的头文件
find . -name "*.h" -exec sh -c '
    file="$1"
    if ! grep -q "#pragma once" "$file" && \
       ! grep -q "#ifndef" "$file"; then
        echo "可能缺少守卫: $file"
    fi
' _ {} \;

# 检查守卫名是否一致
for f in $(find . -name "*.h"); do
    guard=$(grep "#ifndef" "$f" | head -1 | awk "{print \$2}")
    define=$(grep "#define" "$f" | head -1 | awk "{print \$2}")
    if [ "$guard" != "$define" ]; then
        echo "守卫不一致: $f ($guard vs $define)"
    fi
done
```

## 重要注意事项

> **关键点总结**：
> 1. 每个头文件**必须**有头文件守卫（`#ifndef` 或 `#pragma once`）
> 2. 宏名必须**全局唯一**，推荐使用 `项目_模块_文件_H` 格式
> 3. `#ifndef` 和 `#define` 中的宏名必须**完全一致**
> 4. `#pragma once` 更简洁但非标准，现代编译器通常支持
> 5. 头文件守卫防止的是**同一翻译单元**内的重复包含
> 6. 守卫不能防止不同源文件分别包含同一头文件（这是正常的，由链接器处理）
