# 20 - 链接深入

## 链接器的作用

链接器（Linker）是编译流程的最后一个阶段，它将多个目标文件和库文件组合成一个可执行文件或共享库。链接器的核心任务是**符号解析**和**重定位**。

## 符号与符号表

### 什么是符号

符号是程序中定义或引用的函数和变量的名称。每个目标文件都有一个**符号表**，记录：

- **已定义的符号**：在本文件中定义的全局函数和变量
- **引用的符号**：在本文件中使用但未定义的符号（外部符号）
- **局部符号**：static 函数和变量（不参与链接）

```c
// main.c
extern int shared_var;        // 引用符号（未定义）
int global_var = 42;          // 定义符号
static int local_var = 10;    // 局部符号（不导出）

int main(void) {              // 定义符号
    return shared_var + global_var;
}
```

### 查看符号表

```bash
# 查看目标文件的符号表
nm main.o

# 输出示例（简化）：
#                  U shared_var    → 未定义（Undefined）
# 0000000000000000 D global_var    → 已初始化数据（Data）
# 0000000000000000 d local_var     → 局部数据（小写 d）
# 0000000000000000 T main          → 代码段（Text）
```

### 符号类型

| nm 输出 | 含义 |
|---------|------|
| `T` / `t` | 代码段（Text），大写全局，小写局部 |
| `D` / `d` | 已初始化数据（Data） |
| `B` / `b` | 未初始化数据（BSS） |
| `U` | 未定义（Undefined）— 需要其他文件提供 |
| `W` / `w` | 弱符号（Weak） |
| `C` | 公共符号（Common） |

## 符号解析

### 强符号与弱符号

```c
// 强符号：函数和已初始化的全局变量
int strong_var = 10;           // 强符号
void strong_func(void) { }     // 强符号

// 弱符号：未初始化的全局变量（某些编译器）
int weak_var;                  // 可能是弱符号（取决于编译器设置）
```

**链接规则**：
1. 不允许有多个**强符号**同名
2. 一个强符号 + 多个弱符号 → 选择强符号
3. 多个弱符号 → 选择任意一个

### 符号冲突

```c
// file1.c
int x = 10;  // 强符号 x

// file2.c
int x = 20;  // 强符号 x → 链接错误：多重定义！
```

```c
// file1.c
int x = 10;  // 强符号

// file2.c
int x;       // 弱符号 → 链接器选择 file1.c 的强符号，x = 10
```

### static 修饰符

```c
// file1.c
static int x = 10;  // 局部符号，不参与链接

// file2.c
static int x = 20;  // 不同的符号（各自文件内可见），无冲突
```

## 重定位

### 重定位过程

编译器在编译单个 `.c` 文件时，不知道外部符号的最终地址，使用**占位地址**。链接器在链接阶段修正这些地址。

```c
// main.c
extern int data;
int main(void) {
    return data;  // 编译时：data 的地址未知
}
```

```asm
; main.o 中的汇编（重定位前）
main:
    mov eax, [rip + 0x00000000]  ; data 的地址占位符
    ret
```

```asm
; 链接后的汇编（重定位后）
main:
    mov eax, [rip + 0x00001028]  ; data 的实际地址偏移
    ret
```

### 重定位表

目标文件中的重定位表记录所有需要修正的位置：

```bash
# 查看重定位信息
objdump -r main.o

# RELA 段
OFFSET   TYPE            VALUE
0x0009   R_X86_64_PC32   data-0x00000004
```

## 链接过程示例

```c
// math.h
int add(int a, int b);
int sub(int a, int b);

// math.c
#include "math.h"
int add(int a, int b) { return a + b; }
int sub(int a, int b) { return a - b; }

// main.c
#include <stdio.h>
#include "math.h"
int main(void) {
    printf("3+2=%d\n", add(3, 2));
    return 0;
}
```

```bash
# 编译
gcc -c math.c -o math.o     # math.o 定义 add, sub
gcc -c main.c -o main.o     # main.o 引用 add, printf

# 链接
gcc main.o math.o -o program
# 链接器：
# 1. 在 math.o 中找到 add 定义
# 2. 在 libc 中找到 printf 定义
# 3. 重定位所有引用的地址
```

## 链接脚本基础

链接脚本控制链接器如何组织输出文件的各个段：

```ld
/* 简单的链接脚本示例 */
SECTIONS
{
    . = 0x400000;        /* 设置基地址 */
    
    .text : {
        *(.text)         /* 所有目标文件的 .text 段 */
    }
    
    .rodata : {
        *(.rodata)
    }
    
    .data : {
        *(.data)
    }
    
    .bss : {
        *(.bss)
    }
}
```

```bash
# 使用自定义链接脚本
gcc -T custom.ld main.o -o program
```

## 链接顺序

```bash
# 链接顺序很重要
gcc main.o -lm -lpthread   # 正确：被依赖的库放在后面
gcc -lm -lpthread main.o   # 可能出错：库在依赖它的文件之前
```

规则：被引用的符号应该在引用之后出现。GCC 从左到右处理文件。

## 重要注意事项

> **关键点总结**：
> 1. 链接器的核心工作是**符号解析**（匹配引用和定义）和**重定位**（修正地址）
> 2. 不允许**强符号多重定义**，这会导致链接错误
> 3. 使用 `static` 限制符号的作用域，避免命名冲突
> 4. 链接顺序重要——被依赖的库放在命令行后面
> 5. `nm` 和 `objdump` 是分析目标文件的有用工具
> 6. 理解链接过程有助于诊断"未定义引用"等链接错误
