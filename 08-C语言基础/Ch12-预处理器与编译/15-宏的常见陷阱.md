# 15 - 宏的常见陷阱

## 陷阱概述

宏的强大来自于它的简单——纯文本替换。但正是这种简单性，使得宏容易引入隐蔽的错误。本章总结宏使用中最常见的陷阱及规避方法。

## 陷阱一：运算符优先级问题

### 问题

```c
#define DOUBLE(x) x * 2

int result = DOUBLE(3 + 1);
// 展开为: 3 + 1 * 2 = 5，而不是期望的 8
```

### 更危险的例子

```c
#define IS_EVEN(n) n % 2 == 0

if (IS_EVEN(5)) { }        // 展开为: if (5 % 2 == 0) → 正确
if (IS_EVEN(5) && x > 0) { }  // 展开为: if (5 % 2 == 0 && x > 0) → OK
if (!IS_EVEN(5)) { }       // 展开为: if (!5 % 2 == 0) → 错误！
                            // !5 = 0, 0 % 2 = 0, 0 == 0 = true
```

### 解决方案：始终加括号

```c
#define DOUBLE(x) ((x) * 2)
#define IS_EVEN(n) ((n) % 2 == 0)
#define MAX(a, b) ((a) > (b) ? (a) : (b))

int result = DOUBLE(3 + 1);      // ((3 + 1) * 2) = 8 ✓
if (!IS_EVEN(5)) { }            // (!((5) % 2 == 0)) ✓
```

**规则**：每个参数用括号包裹，整个表达式也用括号包裹。

## 陷阱二：参数多次求值

### 问题

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))

int x = 5, y = 3;
int result = MAX(x++, y++);
// 展开为: ((x++) > (y++) ? (x++) : (y++))
// x++ 执行了 2 次，y++ 执行了 2 次！
// x = 7, y = 5, result = 6（取决于求值顺序）
```

### 更隐蔽的例子

```c
#define SQUARE(x) ((x) * (x))

int i = 5;
int result = SQUARE(++i);
// 展开为: ((++i) * (++i))
// i 被自增两次，结果未定义行为！
```

```c
#define ABS(x) ((x) < 0 ? -(x) : (x))

int val = get_value();  // 假设 get_value 有副作用（每次调用返回不同值）
int result = ABS(get_value());
// get_value() 被调用了两次！返回值可能不同
```

### 解决方案

```c
// 方案1: 使用内联函数
static inline int max_int(int a, int b) { return a > b ? a : b; }

// 方案2: GCC 语句表达式
#define MAX_SAFE(a, b) ({ \
    typeof(a) _a = (a); \
    typeof(b) _b = (b); \
    _a > _b ? _a : _b; \
})

// 方案3: 调用者保证无副作用
int result = MAX(x, y);  // OK：x 和 y 没有副作用
```

## 陷阱三：意外的文本替换

### 问题

```c
#define begin {
#define end   }

// 看起来正常
if (x > 0) begin
    printf("positive\n");
end

// 但是...
struct Data begin  // 灾难！begin 被替换为 {
    int x;
end;               // end 被替换为 }，语法混乱
```

### 更实际的例子

```c
#define ERROR -1

int get_status(void) {
    return ERROR;  // 展开为: return -1; → 正确
}

// 但是...
printf("Error code: %d\n", ERROR);  // 展开为: printf("Error code: %d\n", -1); → 正确
char *msg = "ERROR";  // "ERROR" 不会被替换（字符串内的文本不受宏影响）

// 危险情况
#define ERROR goto error_handler
// 任何 ERROR 出现的地方都会被替换！
```

### 解决方案

```c
// 使用更具描述性的宏名
#define STATUS_ERROR (-1)
#define ERROR_CODE_INVALID -1

// 避免用宏定义关键字替代
// 不要: #define begin {
```

## 陷阱四：宏与分号

### 问题

```c
#define LOG(msg) printf("[LOG] %s\n", msg); fflush(stdout)

if (error)
    LOG("Error occurred");
else
    process();
// 展开为:
// if (error)
//     printf("[LOG] Error occurred\n"); fflush(stdout);
// else  // ← 编译错误！else 前面有多余的语句
//     process();
```

### 解决方案：do-while(0)

```c
#define LOG(msg) do { \
    printf("[LOG] %s\n", msg); \
    fflush(stdout); \
} while(0)

if (error)
    LOG("Error occurred");  // do { ... } while(0); 是一个完整语句
else
    process();  // 正确！
```

## 陷阱五：缺少括号包裹

### 问题

```c
#define ADD(a, b) (a + b)

int result = ADD(2, 3) * 4;  // 展开为 (2 + 3) * 4 = 20 ✓
// 看起来正确，但是...

#define MULTIPLY(a, b) (a * b)
int result2 = MULTIPLY(2+3, 4+5);  // 展开为 (2+3 * 4+5) = 2+12+5 = 19
// 期望: (2+3) * (4+5) = 5 * 9 = 45
```

### 正确写法

```c
#define ADD(a, b) ((a) + (b))
#define MULTIPLY(a, b) ((a) * (b))

int result2 = MULTIPLY(2+3, 4+5);  // ((2+3) * (4+5)) = 45 ✓
```

## 陷阱六：宏中的逗号

### 问题

```c
#define CALL(f, a, b) f(a, b)

CALL(printf, "x=%d, y=%d\n", x, y);
// 预处理器将逗号视为参数分隔符：
// 参数1: printf
// 参数2: "x=%d
// 参数3: y=%d\n"
// 参数4: x, y  → 参数个数不对！
```

### 解决方案

```c
// 使用可变参数宏
#define PRINTF(fmt, ...) printf(fmt, __VA_ARGS__)
PRINTF("x=%d, y=%d\n", x, y);  // ✓

// 或者将含逗号的表达式用括号包裹
#define CALL3(f, a, b, c) f(a, b, c)
CALL3(printf, "x=%d, y=%d\n", x, y);  // ✓
```

## 陷阱七：调试困难

### 问题

```c
#define ADD(a, b) ((a) + (b))
int x = ADD(1, 2);

// 在调试器中：
// 你看到的是: int x = ((1) + (2));
// 而不是: int x = ADD(1, 2);
// 调试器不知道原始的宏调用
```

### 解决方案

```bash
# 使用 GCC 的 -g3 选项，包含宏调试信息
gcc -g3 -O0 program.c -o program

# 在 GDB 中查看宏定义
(gdb) macro expand ADD(1, 2)
# 展开为: ((1) + (2))

# 或使用 -E 查看预处理结果
gcc -E program.c | less
```

## 陷阱八：宏名冲突

### 问题

```c
// mylib.h
#define MAX(a, b) ((a) > (b) ? (a) : (b))

// otherlib.h（另一个库）
#define MAX(x, y) ((x) > (y) ? (x) : (y))

// 同时包含两个头文件 → 重定义错误（或静默覆盖）
```

### 解决方案

```c
// 使用命名前缀
#define MYLIB_MAX(a, b) ((a) > (b) ? (a) : (b))
#define OTHERLIB_MAX(x, y) ((x) > (y) ? (x) : (y))

// 或使用 inline 函数代替宏
static inline int mylib_max(int a, int b) { return a > b ? a : b; }
```

## 陷阱九：宏与 sizeof

### 问题

```c
#define ARRAY_SIZE(arr) (sizeof(arr) / sizeof(arr[0]))

void process(int *arr) {
    size_t size = ARRAY_SIZE(arr);  // 错误！arr 在这里是指针
    // 展开为: sizeof(arr) / sizeof(arr[0]) = sizeof(int*) / sizeof(int)
    // 在 64 位系统上可能是 8/4 = 2，而不是数组实际大小
}
```

### 解决方案

```c
// ARRAY_SIZE 只能用于真正的数组
#define ARRAY_SIZE(arr) (sizeof(arr) / sizeof((arr)[0]))

int main(void) {
    int arr[100];
    size_t size = ARRAY_SIZE(arr);  // ✓ = 100
    process(arr, ARRAY_SIZE(arr));  // 传递大小作为参数
}

void process(int *arr, size_t size) {
    // 使用传入的 size，而非试图计算
}
```

## 陷阱总结

| 陷阱 | 症状 | 解决方案 |
|------|------|----------|
| 优先级 | 计算结果错误 | 每个参数和整个表达式加括号 |
| 多次求值 | 副作用执行多次 | 使用内联函数或语句表达式 |
| 意外替换 | 代码被意外替换 | 使用有前缀的宏名 |
| 分号问题 | 编译错误 | do-while(0) 模式 |
| 缺少括号 | 运算顺序错误 | 严格加括号 |
| 逗号问题 | 参数个数错误 | 可变参数宏 |
| 调试困难 | 无法查看宏调用 | gcc -g3 |
| 名称冲突 | 重定义 | 命名前缀或 inline 函数 |
| sizeof 误用 | 得到错误大小 | 只用于真正的数组 |

## 重要注意事项

> **关键点总结**：
> 1. **始终加括号**：每个参数和整个宏体
> 2. **避免副作用**：不要向宏传递有副作用的参数
> 3. **使用 do-while(0)**：包裹多语句宏
> 4. **优先使用 inline 函数**：除非必须使用宏的特性
> 5. **使用 gcc -E**：检查宏展开结果
> 6. **使用 gcc -g3**：调试时保留宏信息
