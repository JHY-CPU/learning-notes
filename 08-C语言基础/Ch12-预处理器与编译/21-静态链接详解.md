# 21 - 静态链接详解

## 静态链接概述

静态链接是将程序所需的所有库代码**在编译时**复制到最终可执行文件中的方式。生成的可执行文件是自包含的，不依赖外部库文件。

## 静态库

### 静态库的创建

静态库是多个目标文件的归档文件（`.a` 在 Unix/Linux，`.lib` 在 Windows）。

```bash
# 编译源文件为目标文件
gcc -c file1.c -o file1.o
gcc -c file2.c -o file2.o
gcc -c file3.c -o file3.o

# 创建静态库
ar rcs libmylib.a file1.o file2.o file3.o

# ar 工具说明：
# r — 替换/插入文件到归档
# c — 创建归档（如果不存在）
# s — 创建符号索引（等同于 ranlib）
```

### 查看静态库内容

```bash
# 列出静态库中的目标文件
ar -t libmylib.a

# 查看符号表
nm libmylib.a

# 详细信息
ar -v -t libmylib.a
```

### 使用静态库

```bash
# 方法1: 直接指定
gcc main.c libmylib.a -o program

# 方法2: 使用 -L 和 -l
gcc main.c -L./lib -lmylib -o program
# -L./lib 指定库搜索路径
# -lmylib 指定库名（libmylib.a 中的 mylib）

# 方法3: 在系统路径中安装
sudo cp libmylib.a /usr/local/lib/
gcc main.c -lmylib -o program
```

## 符号解析顺序

链接器按以下顺序解析符号：

```
1. 目标文件中的符号（命令行中直接指定的 .o 文件）
2. 静态库中的符号（从左到右处理命令行中的库）
3. 动态库中的符号
```

### 库的处理规则

链接器对静态库的处理是**按需提取**：

1. 扫描所有目标文件，建立未解析符号集合
2. 从左到右处理每个静态库
3. 如果库中的某个目标文件定义了**当前未解析的符号**，提取该文件
4. 重复直到没有新的未解析符号或处理完所有库

```c
// libmath.a 包含:
//   add.o: 定义 add
//   sub.o: 定义 sub
//   mul.o: 定义 mul
//   div.o: 定义 div

// main.c 只使用了 add
gcc main.c -lmath -o program
// 链接器只从 libmath.a 中提取 add.o，其他文件不会被包含
```

### 链接顺序问题

```c
// mylib.a 中使用了 libm 的 cos 函数
// main.c 使用了 mylib.a 的 func 函数

// 错误顺序
gcc main.c -lmylib -lm -o program  // 可能OK
gcc main.c -lm -lmylib -o program  // 可能失败！
// 原因：处理 -lm 时，还没有未解析的 cos 符号
//       处理 -lmylib 时，cos 未被解析，最终报错
```

**规则**：依赖者在前，被依赖者在后

```bash
# 正确顺序
gcc main.o -lhigh_level -lmiddle_level -llow_level -o program
```

## 链接器选项

```bash
# 查看链接过程
gcc -Wl,--verbose main.c -o main

# 保留未使用的符号（默认会删除）
gcc -Wl,--whole-archive libmylib.a -Wl,--no-whole-archive main.c -o main

# 设置入口点
gcc -Wl,-e,my_entry main.c -o main

# 生成 map 文件
gcc -Wl,-Map=output.map main.c -o main

# 删除未使用的段（减小体积）
gcc -Wl,--gc-sections main.c -o main
```

## 常见静态链接问题

### 1. 未定义引用

```bash
/usr/bin/ld: main.o: in function `main':
main.c:(.text+0xa): undefined reference to `my_function'
collect2: error: ld returned 1 exit status
```

原因：找不到符号定义。解决：
- 确保包含定义该符号的 `.o` 文件或库
- 检查链接顺序
- 检查函数声明和定义是否一致

### 2. 多重定义

```bash
/usr/bin/ld: file1.o:(.data+0x0): multiple definition of `global_var'
/usr/bin/ld: file2.o:(.data+0x0): first defined here
```

原因：同一个全局变量在多个文件中定义。解决：
- 使用 `extern` 声明，只在一个文件中定义
- 使用 `static` 限制作用域

### 3. 静态库顺序

```bash
# 如果库之间有依赖关系，需要正确排序
gcc main.o -la -lb -lc -o program
# 如果 a 依赖 b，b 依赖 c
# 顺序应为: -la -lb -lc
```

## 静态链接 vs 动态链接

| 特性 | 静态链接 | 动态链接 |
|------|----------|----------|
| 文件大小 | 较大（包含库代码） | 较小（仅引用） |
| 可移植性 | 好（自包含） | 需要库文件 |
| 内存使用 | 每个进程独占 | 多进程共享 |
| 更新 | 需要重新编译 | 替换库文件即可 |
| 启动速度 | 较快 | 较慢（需要加载库） |
| 运行速度 | 略快 | 略慢（间接调用） |

## 重要注意事项

> **关键点总结**：
> 1. 静态库是 `.o` 文件的归档，使用 `ar` 创建
> 2. 链接器**按需提取**——只提取包含所需符号的目标文件
> 3. 链接顺序很重要——依赖者在前，被依赖者在后
> 4. 使用 `-Wl,--whole-archive` 强制包含库中所有目标文件
> 5. "未定义引用"通常意味着库未被包含或顺序错误
> 6. `nm` 和 `objdump` 用于分析库和目标文件的内容
