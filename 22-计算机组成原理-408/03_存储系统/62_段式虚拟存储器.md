# 62_段式虚拟存储器

## 核心概念
- **段(Segment)**: 按程序逻辑功能划分的连续地址空间
- **段式虚拟存储**: 以段为单位进行虚实地址映射
- **段表(Segment Table)**: 记录每个段的物理基址和段长
- **段表基址寄存器(STBR)**: 存放段表在主存中的起始地址

### 常见的段划分
| 段号 | 名称 | 内容 |
|------|------|------|
| 0 | 代码段(.text) | 程序指令 |
| 1 | 数据段(.data) | 已初始化全局变量 |
| 2 | BSS段(.bss) | 未初始化全局变量 |
| 3 | 堆(heap) | 动态分配内存 |
| 4 | 栈(stack) | 函数调用、局部变量 |

### 逻辑地址结构
$$\text{逻辑地址} = \text{段号}(S) + \text{段内偏移}(W)$$

- 段号: 确定访问哪个段
- 段内偏移: 段内的相对位置

### 段表项结构
每个段表项包含:
- **段基址(Base)**: 该段在主存中的起始物理地址
- **段长(Limit/Length)**: 该段的最大长度
- **存取权限**: 读/写/执行权限
- **存在位**: 该段是否已调入主存

## 原理分析

### 地址转换过程

```
逻辑地址: | 段号S | 段内偏移W |
              |          |
              v          |
        段表基址+        |
        S×段表项大小     |
              |          |
              v          |
        查段表第S项      |
         /      \        |
      越界检查  W < 段长? |
         |      /  \     |
         |   是     否   |
         |   |      |    |
         v   v      v    |
       段基址B + W       |
              |          |
              v          |
        物理地址 <--------+
```

**步骤详解**:
1. CPU发出逻辑地址 $(S, W)$
2. 用段号 $S$ 作为索引, 查段表
3. 检查: 若 $W \geq \text{段长}$, 则产生越界中断(段异常)
4. 物理地址 = 段基址 $B$ + 段内偏移 $W$

### 段表大小计算
$$\text{段表大小} = \text{最大段数} \times \text{段表项大小}$$

**例**: 最大段数 $2^{10}$ = 1024, 段表项大小8B(基址4B+段长2B+其他2B)
$$\text{段表大小} = 1024 \times 8B = 8KB$$

### 越界检查
每条访存指令都需要检查:
$$W < \text{段长Limit}$$
若不满足则产生**段越界异常**, 由操作系统处理。

### 存取权限检查
- 代码段: 可读可执行, 不可写
- 数据段: 可读可写, 一般不可执行
- 常量段: 只读
- 违反权限产生**保护异常**

## 直观理解

**图书馆类比**:
- 段 = 不同主题的书架区(文学区、科技区、历史区...)
- 段号 = 你告诉管理员要哪个区的书
- 段基址 = 该区在图书馆的起始位置
- 段长 = 该区有多大的空间
- 段内偏移 = 在该区内从起点走几步

### 与页式的本质区别
| 对比项 | 页式 | 段式 |
|--------|------|------|
| 划分依据 | 物理等分(固定大小) | 逻辑功能(大小不等) |
| 对程序员 | 透明(不可见) | 可见(编译时确定) |
| 单位 | 页(Page) | 段(Segment) |
| 地址含义 | VPN+Offset无语义 | 段号有语义(代码段/数据段) |

### 记忆要点
> **"段按逻辑分, 大小不固定, 段表记基址, 越界要保护"**

## 知识关联

### 跨章节联系
- **操作系统(进程管理)**: 段式与进程地址空间划分
- **编译原理**: 编译器生成代码段、数据段等
- **计组第4章**: 段越界异常属于程序性中断

### 段式的优缺点

**优点**:
- 符合程序逻辑, 便于共享和保护
- 不同段可设不同权限
- 段作为逻辑单位, 便于编译器管理

**缺点**:
- **外部碎片**: 段长不等, 反复分配回收产生不连续小空闲区
- 段表较大时查找效率低
- 地址转换需先查段表再相加, 开销较大

### 外部碎片详解
由于各段长度不同, 内存中会出现大小不等的空闲块。当需要分配一个较大的段时, 即使总的空闲空间足够, 但没有任何一个空闲块够大, 就产生了**外部碎片**。

解决方法: **紧凑(Compaction)**——将已分配的段移动到一起, 合并空闲块。但开销很大。

### 常考组合
- 段式 + 页式 = 段页式(结合两者优点)
- 段式虚拟存储 + Cache = 现代处理器架构
