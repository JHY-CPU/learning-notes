# 93_中断优先级

## 核心概念

- **中断优先级**：多个中断同时请求时，决定先响应哪个的规则
- 优先级由**硬件排队器**（菊花链/链式排队）或**软件查询**实现
- NMI > 异常 > INTR

## 原理分析

### 一、优先级排序

$$\text{NMI} > \text{终止异常} > \text{故障异常} > \text{陷阱异常} > \text{INTR}_0 > \text{INTR}_1 > \cdots > \text{INTR}_n$$

具体优先级（x86 示例）：

| 优先级 | 中断源 |
|-------|--------|
| 最高 | 机器检查（终止） |
| ↓ | NMI |
| ↓ | 故障（缺页等） |
| ↓ | INTR（通过 8259A） |
| 最低 | 单步陷阱 |

### 二、硬件排队器

#### 链式排队器（菊花链）

```
INTR1 ──→ [排队逻辑] ──→ INTR2 ──→ [排队逻辑] ──→ INTR3 ──→ ...
              │                      │                      │
              ↓                      ↓                      ↓
           若请求则              若请求则              若请求则
           响应，阻断            响应，阻断            响应
           后面的                后面的
```

- 离 CPU 最近的优先级最高
- 一旦某个中断被响应，后面的被阻断

#### 编码器+比较器

- 对所有中断请求进行编码
- 比较器选出最高优先级

### 三、软件查询方式

```c
// 按优先级顺序查询
if (INTR1_request) { service_INTR1(); }
else if (INTR2_request) { service_INTR2(); }
else if (INTR3_request) { service_INTR3(); }
```

- 优点：灵活，可软件调整优先级
- 缺点：速度慢

### 四、中断优先级管理

#### 固定优先级

- 优先级固定不变
- 简单但不灵活

#### 动态优先级

- **循环优先级**：每次响应后，被响应的中断降为最低优先级
- 防止高优先级中断**饿死**低优先级中断

### 五、中断嵌套与优先级

- 高优先级中断可以**打断**低优先级中断的处理
- 低优先级中断**不能打断**高优先级中断的处理
- 需要**开中断**（$IF = 1$）才允许嵌套

## 直观理解

**中断优先级**像**医院急诊分诊**：

- 心脏骤停（NMI）→ 立即抢救
- 骨折（故障）→ 先处理
- 普通发烧（INTR 高优先级）→ 等一等
- 感冒（INTR 低优先级）→ 最后处理

**链式排队器**像**银行叫号**：号小的先办，号大的排后面。

## 知识关联

- 与**中断响应**（第 90 节）的关系：响应前需要判优
- 与**中断屏蔽**（第 94 节）的关系：可以屏蔽低优先级中断
- 与**中断嵌套**（第 95 节）的关系：高优先级可嵌套低优先级
- 与**IO 接口**（第 7 章）的关系：8259A 中断控制器管理优先级
