# 73_数据旁路

## 核心概念

- **数据旁路（Forwarding / Bypassing）**：将计算结果直接从一个阶段送到另一个阶段的输入端
- 避免不必要的暂停，提高流水线效率
- 是解决 RAW 冲突的**核心硬件技术**

## 原理分析

### 一、旁路的基本原理

无旁路时：结果必须经过 WB 写入寄存器堆后，后续指令才能读到。

有旁路时：ALU 计算结果出来后，**直接旁路**到下一条指令的 ALU 输入端。

```
                  ┌──── EX 结果 ────┐
                  │                 ↓
  寄存器堆 ──→ [MUX] ──→ ALU ──→ 结果
       ↑                        │
       └──── MEM/WB 结果 ───────┘（备选旁路源）
```

### 二、旁路的来源和去向

#### 旁路来源（3 个）

1. **EX/MEM 寄存器**：上一条指令 EX 阶段的输出
2. **MEM/WB 寄存器**：上上条指令的 EX 输出（经过 MEM 阶段）
3. **寄存器堆**：更早的指令已写回（正常读取）

#### 旁路去向

- ALU 的两个输入端（A 和 B）
- 通过**多路选择器 MUX** 选择数据来源

### 三、旁路能解决的情况

| 间隔 | 无旁路 | 有旁路 | 旁路源 |
|------|--------|--------|--------|
| 0 条（相邻） | 暂停 2 周期 | **不需暂停** | EX→EX 直接旁路 |
| 1 条（隔 1 条） | 暂停 1 周期 | **不需暂停** | MEM→EX 旁路 |
| 2 条及以上 | 不需暂停 | 不需暂停 | WB 写回已可用 |

### 四、旁路仍需暂停的情况

**Load-Use 冲突**：

```
I1: LD   R1, 0(R2)    ; R1 ← Mem[R2]，结果在 MEM 阶段末尾才可用
I2: ADD  R3, R1, R4   ; 需要 R1 的值
```

- I1 的结果在 **MEM 阶段结束**时才产生
- I2 在 **EX 阶段**需要 R1
- 即使有旁路，也无法在 I2 的 EX 阶段之前拿到数据
- **必须暂停 1 个周期**

```
时间→    1    2    3    4    5    6    7
I1:     [IF] [ID] [EX][MEM][WB]
I2:          [IF] [ID] [--] [EX][MEM][WB]
                              ↑
                        旁路 MEM→EX，但需等 MEM 完成
                        所以暂停 1 周期
```

### 五、旁路控制逻辑

需要判断何时启用旁路：

```
if (EX/MEM.Rd == ID/EX.Rs1) then
    ALU_A ← EX/MEM.ALUOut    // EX→EX 旁路
if (MEM/WB.Rd == ID/EX.Rs1) then
    ALU_A ← MEM/WB.ALUOut    // MEM→EX 旁路
```

还需避免对 `nop` 或不写寄存器的指令进行旁路。

### 六、旁路的局限性

1. **Load 指令**：结果在 MEM 阶段末尾才可用，Load-Use 需暂停
2. **分支指令**：分支结果在 EX 才知道，已取的指令可能作废
3. **长延迟操作**：乘除法等多周期操作，旁路点不确定
4. **硬件复杂度**：需要大量 MUX 和比较器

## 直观理解

**旁路**就像**快递直送**：

- 传统方式：商品（数据）先入库（寄存器堆），再出库发给客户
- 旁路方式：商品从工厂（ALU）直接送给客户（下一条指令的 ALU 输入）
- **Load-Use**：商品还在运输途中（MEM 阶段），客户已经要用，只能暂停等

## 知识关联

- 与**RAW 冲突**（第 71 节）的关系：旁路是解决 RAW 的最佳方案
- 与**暂停机制**（第 77 节）的关系：旁路不能解决时，仍需暂停
- 与**控制冲突**（第 74 节）的关系：分支结果也需要旁路到 PC
- 408 高频考点：判断指令序列需要暂停几个周期（考虑旁路后）
