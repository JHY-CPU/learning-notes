# 95_中断嵌套

## 核心概念

- **中断嵌套**：在处理一个中断的过程中，被更高优先级的中断打断
- 嵌套的前提：**开中断**（$IF = 1$）
- 嵌套的规则：高优先级可打断低优先级，反之不可

## 原理分析

### 一、中断嵌套的条件

1. 当前正在处理中断（已进入中断服务程序）
2. 新中断的优先级**高于**当前中断
3. CPU 处于**开中断**状态（$IF = 1$）

$$\text{嵌套发生} = (\text{正在处理中断}) \land (\text{新中断优先级更高}) \land (IF = 1)$$

### 二、嵌套执行过程

```
主程序执行
    ↓
中断1请求（低优先级）→ 响应 → 关中断 → 保存现场
    ↓
执行中断1服务程序（开中断）
    ↓
中断2请求（高优先级）→ 响应 → 关中断 → 保存现场
    ↓
执行中断2服务程序
    ↓
中断2完成 → 恢复现场 → 开中断 → 返回中断1
    ↓
继续中断1服务程序
    ↓
中断1完成 → 恢复现场 → 开中断 → 返回主程序
    ↓
继续主程序
```

### 三、栈的变化（嵌套时）

```
栈顶 ↑
┌────────────────┐
│ 中断2 的 PSW   │  ← 最后压入，最先弹出
├────────────────┤
│ 中断2 的 PC    │
├────────────────┤
│ 中断1 的 PSW   │
├────────────────┤
│ 中断1 的 PC    │
├────────────────┤
│ 主程序 PSW     │
├────────────────┤
│ 主程序 PC      │  ← 最先压入，最后弹出
└────────────────┘
```

### 四、嵌套的限制

#### 不允许嵌套的情况

1. **关中断期间**：$IF = 0$，不能响应新的 INTR
2. **同级或低优先级中断**：不能打断高优先级中断的处理
3. **相同中断源**：通常屏蔽自身，防止重入

#### 允许嵌套的情况

1. **更高优先级中断**：在开中断后可打断
2. **NMI**：始终可打断 INTR 的处理

### 五、中断嵌套的示例

假设优先级：NMI > INTR3 > INTR2 > INTR1

```
时间→
处理主程序    ▓▓▓▓▓
处理 INTR1        ░░░░░░░░░░░░░░
处理 INTR3            ▓▓▓▓▓▓▓
处理 NMI                  ░░░░
```

- INTR1 服务程序中开了中断
- INTR3（更高）打断 INTR1
- NMI（最高）打断 INTR3
- 处理完后逐级返回

## 直观理解

**中断嵌套**像**开会接电话**：

- 你正在开普通会议（低优先级中断处理）
- 老板来电（高优先级中断）→ 你暂停会议接老板电话
- 更紧急：董事长来电（NMI）→ 老板电话也得暂停
- 接完后逐级恢复

## 知识关联

- 与**中断优先级**（第 93 节）的关系：嵌套由优先级决定
- 与**中断屏蔽**（第 94 节）的关系：通过屏蔽字控制嵌套
- 与**中断处理过程**（第 96 节）的关系：嵌套是处理过程的扩展
- 与**栈**（第 3 章）的关系：嵌套需要足够的栈空间保存多层现场
