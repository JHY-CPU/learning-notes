# 14_取指周期

## 核心概念

- **取指周期（Fetch Cycle）**：指令周期的第一个阶段，从主存中取出当前需要执行的指令。
- **取指是所有指令共有的阶段**：无论什么指令，都必须先从主存取到CPU中。
- **取指的核心操作**：PC → MAR → 主存 → MDR → IR，同时 PC + 1。

### 取指周期的微操作序列

| 节拍 | 微操作 | 说明 |
|------|--------|------|
| T0 | PC → MAR | 将PC中存放的指令地址送MAR |
| T1 | 1 → R（读命令） | 向主存发出读命令 |
| T1 | MAR → 地址总线 → 主存 | 地址送主存，选中存储单元 |
| T2 | M(MAR) → MDR | 主存将指令数据送MDR |
| T3 | MDR → IR | 指令从MDR送IR |
| T3 | PC + 1 → PC | PC指向下一条指令 |

> 注：不同教材的节拍划分略有不同，以上是一种典型的3节拍模型。

## 原理分析

### 完整数据流

$$PC \xrightarrow{送地址} MAR \xrightarrow{地址总线} \text{主存} \xrightarrow{读命令} MDR \xrightarrow{传送} IR$$

同时：$PC \leftarrow PC + 1$

### 微操作的硬件控制信号

| 微操作 | 涉及的控制信号 |
|--------|---------------|
| PC → MAR | PC_out, MAR_in |
| 1 → R | Read |
| M(MAR) → MDR | Memory_data_out, MDR_in |
| MDR → IR | MDR_out, IR_in |
| PC + 1 → PC | PC_in, +1（或专门的PC自增信号） |

**注意**：
- `PC_out` 使PC的内容送上内部总线
- `MAR_in` 使MAR从总线接收数据
- 单总线结构中，`PC_out, MAR_in` 必须在同一时钟周期内完成（PC的内容从总线送MAR）

### 取指后的两个关键动作

1. **IR获得指令**：IR的内容被后续的译码和执行阶段使用。
   - OP字段 → 指令译码器 → CU
   - Ad字段 → （可能）后续的间址或执行阶段使用

2. **PC递增**：$PC \leftarrow PC + \Delta$
   - 定长指令：$\Delta$ = 指令字长（如4字节）
   - 变长指令：$\Delta$ = 当前指令的实际长度

### 取指周期与主存的关系

$$T_{取指} \geq T_{主存访问}$$

因为取指的核心操作就是一次主存读取，所以取指时间不可能短于主存访问时间。

## 直观理解

### 生活类比

取指就像**从书架上取一本书**：
- **PC**= 你记在便签上的"书架编号"
- **MAR**= 你把便签交给图书管理员（送地址）
- **读命令**= 你说"请帮我拿这本书"（发出读请求）
- **MDR**= 管理员把书递给你（数据到手）
- **IR**= 你把书翻开放在桌上准备阅读（当前任务）
- **PC+1**= 你在便签上写下"下一本要拿的书的编号"

### 记忆技巧

- **取指数据流口诀**：「**P-M-M-M-I加**」（PC→MAR→主存→MDR→IR，PC+1）
- **本质**：取指就是"PC告诉主存在哪里，主存把指令送回来"

### 易错陷阱

1. **陷阱**：PC+1中的"1"不是数字1，而是"一条指令的长度"。如果指令4字节，PC+4。
2. **陷阱**：取指阶段读的是**指令**（从PC所指地址），不是数据。
3. **陷阱**：单总线结构中PC_out和MAR_in虽然在同一节拍控制，但需要总线仲裁确保PC是唯一的数据源。
