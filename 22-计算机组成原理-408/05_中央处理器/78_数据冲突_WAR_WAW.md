# 72_数据冲突_WAR_WAW

## 核心概念

- **WAR（Write After Read）**：先读后写，写入操作不能在读取操作之前完成
- **WAW（Write After Write）**：先写后写，两条写指令的完成顺序必须正确
- WAR 和 WAW 属于**名字依赖（Name Dependence）**，不是真正的数据依赖
- 在**按序单发射流水线**中不会发生，在**乱序执行**中才出现

## 原理分析

### 一、WAR（读后写）详解

#### 定义

指令 $I_j$ 要**写入**的寄存器，是 $I_i$（$i < j$）要**读取**的寄存器。

$$I_i: \text{读 } R_k \quad I_j: \text{写 } R_k \quad (i < j)$$

#### 示例

```
I1: MUL  R1, R2, R3    ; 读 R2, R3 → R1
I2: ADD  R2, R4, R5    ; 写 R2
```

如果 I2 先写入 R2，而 I1 还没读完 R2，则 I1 会读到错误值。

#### 何时发生

- **按序流水线**：不会发生。I1 先经过 ID（读寄存器），I2 后经过 ID，自然顺序正确
- **乱序执行**：可能 I2 先执行完并写回，I1 还没读——发生 WAR
- **提前写回**：某些设计中写回可能提前

#### 解决方法

- **寄存器重命名**：将 I2 的目标寄存器改为 R2'（物理寄存器），避免名字冲突
- **保持按序写回**：即使乱序执行，也按序写回

### 二、WAW（写后写）详解

#### 定义

两条指令都要**写入同一个寄存器**。

$$I_i: \text{写 } R_k \quad I_j: \text{写 } R_k \quad (i < j)$$

#### 示例

```
I1: LD   R1, 0(R2)     ; R1 ← Mem[R2]（可能 Cache 未命中，慢）
I2: ADD  R1, R3, R4    ; R1 ← R3 + R4（快）
```

最终 R1 应该是 I2 的结果。但如果 I1 因为 Cache 未命中延迟写回，I1 可能在 I2 之后才实际写入，导致 R1 错误。

#### 何时发生

- **按序单发射流水线**：不会发生。两条写指令按序进入 WB，按序写回
- **乱序执行 / 多发射**：可能发生

#### 解决方法

- **寄存器重命名**：I1 写 R1，I2 写 R1'（不同物理寄存器）
- **按序写回**：通过重排序缓冲（ROB）保证

### 三、三种冲突综合对比

| 冲突类型 | 读/写关系 | 依赖性质 | 按序流水线 | 乱序执行 |
|---------|----------|---------|-----------|---------|
| RAW | 先写后读 | **真数据依赖** | **会发生** | 会发生 |
| WAR | 先读后写 | 名字依赖（反依赖） | **不会发生** | 可能发生 |
| WAW | 先写后写 | 名字依赖（输出依赖） | **不会发生** | 可能发生 |

### 四、寄存器重命名

核心思想：将**架构寄存器**映射到更多的**物理寄存器**。

```
架构寄存器: R1, R2, R3, ...
物理寄存器: P1, P2, P3, P4, P5, P6, ...（更多）
```

- I1 写 R1 → 分配物理寄存器 P5
- I2 也写 R1 → 分配物理寄存器 P6
- P5 和 P6 是不同的物理寄存器，消除 WAW
- WAR 同理消除

## 直观理解

- **WAR**：图书馆里，A 还没读完一本书（读），B 就要把书借走换掉（写）
- **WAW**：两个人都要在黑板上写最终答案，顺序必须正确
- **寄存器重命名**：给每人发一块自己的小黑板，消除冲突

## 知识关联

- 与**数据冲突 RAW**（第 71 节）对比：RAW 是真依赖，WAR/WAW 是名字依赖
- 与**乱序执行**（第 78 节）的关系：WAR/WAW 在乱序执行中需要解决
- 与**多发射处理器**（第 82 节）的关系：多发射更容易出现 WAR/WAW
