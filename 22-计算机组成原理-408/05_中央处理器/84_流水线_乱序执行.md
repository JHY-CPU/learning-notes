# 78_流水线_乱序执行

## 核心概念

- **乱序执行（Out-of-Order Execution, OoOE）**：指令不按程序顺序执行，谁的数据准备好谁先执行
- 提高流水线利用率，减少暂停损失
- 需要**重排序**保证最终结果与顺序执行一致

## 原理分析

### 一、乱序执行的动机

按序执行的问题：
```
I1: LD   R1, 0(R2)     ; Cache 未命中，需 100 周期
I2: ADD  R3, R4, R5    ; 数据已就绪，但必须等 I1
```

I2 本来可以立即执行，但因按序要求被迫等待 I1。

### 二、乱序执行的硬件结构

#### 1. 保留站（Reservation Station）

- 每个功能单元前设置**保留站**
- 指令的操作数准备好后才发射到功能单元
- 保留站监听**公共数据总线（CDB）**，获取旁路数据

#### 2. 重排序缓冲（ROB, Reorder Buffer）

- **先进先出**队列
- 保存指令的**原始程序顺序**
- 指令执行结果先写入 ROB，按序从 ROB 写入寄存器堆
- 保证**精确异常**

#### 3. 寄存器重命名

- 将架构寄存器映射到更多的物理寄存器
- 消除 WAR 和 WAW 冲突

### 三、乱序执行的流程

```
1. 发射（Issue）：从指令队列取指令，分配 ROB 条目，读操作数
2. 执行（Execute）：操作数就绪后执行
3. 写回（Write Back）：结果写入 ROB，广播到 CDB
4. 提交（Commit / Retire）：按 ROB 顺序，将结果写入架构状态
```

### 四、精确异常

- 乱序执行时，若指令 $I_k$ 产生异常，需保证 $I_k$ 之前的指令已完成，$I_k$ 之后的指令未修改架构状态
- **ROB 的按序提交**保证了精确异常

### 五、乱序执行的优缺点

| 优点 | 缺点 |
|------|------|
| 减少暂停损失 | 硬件复杂度高 |
| 提高指令级并行 | 功耗增大 |
| 隐藏长延迟操作 | 实现精确异常复杂 |

## 直观理解

**乱序执行**像**餐厅后厨**：

- 按序执行：一道菜做完才做下一道，即使下一道材料已备好
- 乱序执行：哪个菜的材料先备好先做，做完后**按菜单顺序上菜**（ROB 按序提交）

## 知识关联

- 与**数据冲突**（第 70-72 节）的关系：乱序执行需要解决 WAR/WAW
- 与**数据旁路**（第 73 节）的关系：CDB 就是一种广播式旁路
- 与**多发射处理器**（第 82 节）的关系：超标量常结合乱序执行
- 与**精确异常**的关系：ROB 保证异常处理的正确性
