# 14_中断响应条件

## 核心概念

CPU不是收到中断请求就立即响应的，需要同时满足多个条件才能响应中断。

### 中断响应的四个条件(缺一不可)

| 条件 | 说明 | 全称 |
|------|------|------|
| ① 中断源有请求 | 至少一个中断源发出中断请求 | Interrupt Request |
| ② 中断允许触发器IE=1 | CPU允许响应中断 | Interrupt Enable |
| ③ 当前指令执行完毕 | 不能打断正在执行的指令 | 指令周期结束 |
| ④ 无更高优先级中断 | 没有更高优先级的中断正在被处理 | 优先级满足 |

### 条件详解

#### 条件①：中断源有请求

- 中断源(如IO设备)通过中断请求线(INTR)向CPU发出请求
- 接口中断请求触发器置1
- 多个中断源的请求通过"或"逻辑汇总到CPU的INTR引脚

#### 条件②：中断允许触发器IE=1

- **IE(Interrupt Enable)**：CPU内部的中断允许触发器
- IE=1：开中断，CPU允许响应可屏蔽中断
- IE=0：关中断，CPU不响应可屏蔽中断
- **设置方式**：
  - `STI` 指令：开中断(IE=1)
  - `CLI` 指令：关中断(IE=0)
  - 中断响应时硬件自动关中断(IE=0)，防止嵌套
- **注意**：不可屏蔽中断(NMI)不受IE影响，有请求就响应

#### 条件③：当前指令执行完毕

- CPU在**指令周期的末尾**检测中断请求
- 不能在指令执行中间响应中断，否则指令状态不完整
- **中断响应时机**：一条指令执行完毕，下一条指令取指之前
- **原子性**：指令是原子操作，不可被中断打断

```
指令周期: 取指 → 译码 → 执行 → (中断检测点) → 取下一条指令
                                          ↑
                                    在此处检测中断
```

#### 条件④：无更高优先级中断

- 若CPU正在处理某个中断(处于中断服务程序中)，只有更高优先级的中断才能打断
- 低优先级的中断请求被挂起，等待当前中断处理完毕
- 这是多重中断(中断嵌套)的前提条件

### 不可屏蔽中断(NMI) vs 可屏蔽中断(INTR)

| 特征 | 不可屏蔽中断NMI | 可屏蔽中断INTR |
|------|----------------|---------------|
| 是否受IE控制 | 否，IE=0也能响应 | 是，需要IE=1 |
| 中断请求线 | 专用NMI引脚 | 共享INTR引脚 |
| 典型来源 | 电源故障、硬件错误 | IO设备、软件中断 |
| 优先级 | 最高 | 较低 |

## 原理分析

### 中断检测逻辑

```
每个指令周期末尾，CPU执行:
  if (INTR = 1) and (IE = 1) then
    响应中断
  else
    继续执行下一条指令
```

### 中断响应的时间窗口

```
指令1执行 → [中断检测] → 指令2执行 → [中断检测] → ...
                  ↑                      ↑
            可能响应中断            可能响应中断
```

- 中断请求可以在任意时刻发出
- 但CPU只在指令周期末尾检测
- 最坏情况：中断请求刚发出时指令刚开始执行，需要等一整条指令执行完毕

## 直观理解

**中断响应条件类比**：开会时要发言
1. 你举手了(有请求)
2. 主持人允许发言(IE=1)
3. 当前发言人讲完了一段话(指令执行完)
4. 没有更高优先级的人要先发言(优先级满足)

四条缺一不可：没举手不会被叫到；举手了但主持人不让说也不行；人家话没说完不能打断；领导要先说你得等着。

### 易错陷阱

1. **"执行完毕"指整条指令完毕**：不是某个机器周期完毕，是一条完整指令执行完
2. **IE=0不是永久关中断**：中断服务程序中会在适当时机开中断(STI)
3. **NMI不受IE限制**：即使IE=0，NMI仍可响应
4. **中断请求可以持续**：如果CPU暂时不响应，中断请求信号可以保持直到被响应
