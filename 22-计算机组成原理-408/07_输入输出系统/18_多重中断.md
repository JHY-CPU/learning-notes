# 18_多重中断

## 核心概念

- **单重中断**：CPU正在处理一个中断时，不允许响应其他中断请求
- **多重中断(中断嵌套)**：CPU正在处理一个中断时，允许更高优先级的中断打断当前处理
- **关键区别**：单重中断在整个中断处理期间关中断；多重中断在保护现场后开中断

### 单重中断 vs 多重中断

| 特征 | 单重中断 | 多重中断 |
|------|----------|----------|
| 中断处理期间 | 全程关中断 | 保护现场后开中断 |
| 中断嵌套 | 不允许 | 允许(高优先级) |
| 实现复杂度 | 简单 | 复杂(需优先级管理) |
| 响应速度 | 较慢(需等当前中断处理完) | 较快(高优先级可立即响应) |
| 现场保护 | 只需一层 | 需多层(堆栈嵌套) |

## 原理分析

### 多重中断的执行流程

```
主程序执行
    │
    ↓ (设备A中断)
中断响应(硬件) → 关中断
    │
    ↓
中断服务程序A:
    保护现场(软件)
    开中断 ←── 关键！允许更高优先级中断嵌套
    执行IO操作A
        │
        ↓ (设备B中断, 优先级更高)
    中断响应(硬件) → 关中断
        │
        ↓
    中断服务程序B:
        保护现场
        开中断
        执行IO操作B
        关中断
        恢复现场
        IRET(返回A)
        │
        ↓
    关中断
    恢复现场
    IRET(返回主程序)
    │
    ↓
主程序继续执行
```

### 堆栈变化(中断嵌套)

```
原始堆栈:          中断A后:           中断B嵌套后:
┌──────────┐    ┌──────────┐    ┌──────────┐
│          │    │  PSW_A   │    │  PSW_B   │
│          │    │  PC_A    │    │  PC_B    │
│          │    │ 现场A    │    │ 现场B    │
│          │    │  PSW_主  │    │  PSW_A   │
│          │    │  PC_主   │    │  PC_A    │
│          │    │ 现场主   │    │ 现场A    │
│ 原数据   │    │  PSW_主  │    │  PSW_主  │
│          │    │  PC_主   │    │  PC_主   │
│          │    │ 现场主   │    │ 现场主   │
└──────────┘    └──────────┘    └──────────┘
```

每嵌套一层，堆栈增加一层保存的现场。

### 中断屏蔽与多重中断

- **中断屏蔽字**：控制哪些中断源可以打断当前处理
- 高优先级中断源的请求不受屏蔽，可以打断低优先级的中断处理
- 低优先级中断源的请求被屏蔽，不能打断高优先级的中断处理

### 多重中断的实现条件

1. **提前开中断**：在保护现场之后、执行IO之前开中断
2. **优先级管理**：通过中断屏蔽字或优先级排队器管理
3. **堆栈空间充足**：每嵌套一层需要额外的堆栈空间
4. **中断嵌套深度有限**：堆栈空间决定最大嵌套层数

## 直观理解

**多重中断类比**：你是经理，正在处理部门A的紧急事务
- **单重中断**：不允许任何人打扰，A的事处理完再接待其他人
- **多重中断**：CEO(B)找你，你必须放下A的事先处理CEO的指令；CEO说完后你继续处理A

**嵌套限制类比**：CEO在和你谈话时，董事长(C)来了 — 如果C优先级更高，CEO也得等着

### 易错陷阱

1. **开中断不是在中断响应时**：中断响应时硬件关中断，开中断是在中断服务程序中(软件STI)
2. **不能嵌套同级中断**：只有更高优先级的中断才能嵌套
3. **堆栈溢出风险**：嵌套太深可能导致堆栈溢出
4. **开中断前必须完成保护现场**：否则嵌套中断会破坏未保存的寄存器
