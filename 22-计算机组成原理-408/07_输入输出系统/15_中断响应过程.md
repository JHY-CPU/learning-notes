# 15_中断响应过程

## 核心概念

**中断响应过程**：从CPU检测到中断请求到转入中断服务程序之间的过程，由**硬件自动完成**。

### 中断响应的完整步骤

| 步骤 | 操作 | 执行者 | 说明 |
|------|------|--------|------|
| 1 | 中断判优 | 硬件 | 多个中断同时请求时，选优先级最高的 |
| 2 | 关中断 | 硬件 | IE←0，防止响应过程被打断 |
| 3 | 保存断点(PC) | 硬件 | 将PC压入堆栈，保存返回地址 |
| 4 | 保存程序状态字(PSW) | 硬件 | 将PSW压入堆栈，保存CPU状态 |
| 5 | 识别中断源 | 硬件 | 通过中断向量找到中断服务程序入口 |
| 6 | 转入中断服务程序 | 硬件 | PC←中断服务程序入口地址 |

### 中断响应与中断处理的区分

```
                    ← 中断响应(硬件自动) →│← 中断处理(软件为主) →
                    │                       │
中断请求 → 判优 → 关中断 → 保存PC/PSW → 找入口 → 转ISR → 保护现场 → ... → 恢复现场 → 开中断 → IRET
```

- **中断响应**：硬件自动，不可编程，快速(几个机器周期)
- **中断处理**：执行中断服务程序，软件编写，可编程

## 原理分析

### 步骤详解

#### 步骤1：中断判优

- 多个中断源同时请求时，需要确定先响应哪个
- 实现方式：
  - **硬件判优(优先级排队器)**：速度快，通过硬件电路实现
  - **软件判优(查询程序)**：按优先级顺序逐个查询中断源
  - **链式判优(菊花链)**：优先级由物理位置决定

#### 步骤2：关中断

- 将IE(中断允许触发器)清零
- 目的：确保中断响应过程不被打断
- 这是响应过程的第一步硬件操作

#### 步骤3：保存断点(PC)

- 将当前PC(程序计数器)压入堆栈
- PC记录了被中断程序的下一条指令地址
- 这是中断返回的依据
- **由硬件自动完成**

#### 步骤4：保存程序状态字(PSW)

- 将当前PSW(程序状态字)压入堆栈
- PSW包含：条件码(进位、零标志等)、中断允许位、当前模式等
- **由硬件自动完成**

#### 步骤5：识别中断源

- 通过**中断向量**找到对应中断服务程序的入口地址
- 实现方式：
  - **向量中断**：每个中断源对应一个中断向量号，通过中断向量表查入口地址
  - **中断向量表**：存放在主存固定区域，每个表项4~8字节(入口地址)

```
中断向量号 → 中断向量表(基地址+向量号×4) → 中断服务程序入口地址
```

#### 步骤6：转入中断服务程序

- 将中断服务程序入口地址装入PC
- CPU从下一条指令开始执行中断服务程序

### 中断响应时间

$$T_{\text{响应}} = T_{\text{指令剩余}} + T_{\text{判优}} + T_{\text{关中断}} + T_{\text{保存PC/PSW}} + T_{\text{查向量表}}$$

- $T_{\text{指令剩余}}$：当前指令剩余执行时间(最坏情况 = 一条完整指令时间)
- 后续几项为固定开销，通常为几个机器周期

## 直观理解

**中断响应过程类比**：消防演习接到警报
1. 判断哪个区域火情最严重(中断判优)
2. 锁门防止无关人员进入(关中断)
3. 记录你当前的工作位置(保存PC)
4. 记录你的工作状态和工具位置(保存PSW)
5. 查看消防预案找到你的灭火任务(识别中断源)
6. 跑向火场(转入中断服务程序)

### 易错陷阱

1. **中断响应由硬件完成**：不是软件，是纯硬件操作
2. **保存PC和PSW是硬件自动**：不是程序员写的指令
3. **关中断是响应过程的第一步**：不是中断服务程序的第一步
4. **中断响应 ≠ 中断处理**：响应是硬件，处理是软件(中断服务程序)
5. **PC保存的是断点，不是当前指令地址**：是下一条指令的地址
