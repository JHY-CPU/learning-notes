# 10_程序查询方式

## 核心概念

- **程序查询方式(Programmed I/O / Polling)**：CPU主动、反复地查询IO设备的状态，直到设备准备好后再进行数据传送
- **核心思想**：CPU"主动询问"设备状态，忙则等待，空闲则传送
- **又称**：轮询方式、忙等方式(Busy-Waiting)
- **特点**：最简单、最原始的IO控制方式

### 程序查询方式的基本特征

| 特征 | 说明 |
|------|------|
| CPU参与度 | CPU全程参与IO过程 |
| 数据传送 | 通过CPU的寄存器中转(设备→CPU→主存) |
| CPU利用率 | 极低，大量时间浪费在等待循环 |
| 硬件复杂度 | 最低，只需简单的状态寄存器 |
| 响应速度 | 慢，取决于查询频率 |
| 适用场景 | 简单低速设备、嵌入式系统 |

## 原理分析

### 工作流程

```
CPU工作流程:
┌──────────────────┐
│ CPU发IO启动命令   │
└────────┬─────────┘
         ↓
┌──────────────────┐
│ CPU读状态寄存器   │←──────────┐
└────────┬─────────┘           │
         ↓                     │
    ┌─────────┐                │
    │ 设备忙？ │──是──→ 循环等待─┘
    └────┬────┘
         │否(设备空闲)
         ↓
┌──────────────────┐
│ CPU传送一个数据   │(读或写)
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 传送完毕？        │──否──→ 回到"读状态"循环
└────────┬─────────┘
         │是
         ↓
┌──────────────────┐
│ 结束IO操作        │
└──────────────────┘
```

### CPU利用率分析

设设备每 $T$ 秒准备一个数据，每次查询需要 $t_q$ 秒，每次数据传送需要 $t_t$ 秒。

**每次IO操作中CPU的时间开销**：
- 查询等待：$t_q$（假设只查询一次就发现设备就绪，最理想情况）
- 数据传送：$t_t$
- 总开销：$t_q + t_t$

**实际中设备可能还没准备好**：
- 设备忙时，CPU需要多次查询
- 等待期间CPU一直执行查询循环
- **CPU利用率极低**：设备越慢，CPU浪费越多

$$\text{CPU利用率} = \frac{t_t}{T} = \frac{\text{数据传送时间}}{\text{设备数据周期}}$$

**例**：某设备每100ms输出一个数据，每次传送需1ms。CPU每次查询需1μs。
- 等待期间CPU需查询约100次
- 每个数据周期CPU有效工作仅1ms
- CPU利用率 = $1\text{ms} / 100\text{ms} = 1\%$，99%的时间在空转

### 多设备查询

当系统有多个IO设备时，CPU按顺序轮询每个设备：

```
循环:
  查询设备1 → 数据传送(如有) →
  查询设备2 → 数据传送(如有) →
  查询设备3 → 数据传送(如有) →
  回到开头
```

- **公平性**：每个设备被查询的机会均等
- **问题**：设备越多，每个设备被查询的间隔越长，响应越慢
- **问题**：高速设备被低速设备拖累

## 直观理解

**程序查询类比**：你去餐厅取外卖
- 你每隔1分钟问服务员："我的餐好了吗？"（CPU读状态寄存器）
- 服务员说"还没好"（BUSY=1），你继续等
- 你不能做其他事情，只能反复问（CPU被占用）
- 终于说"好了"（DONE=1），你取餐走人（数据传送）

**对比类比**：中断方式 = 餐好了服务员叫你（你先去做别的事）

### 易错陷阱

1. **程序查询不等于中断**：查询是CPU主动，中断是设备主动
2. **"忙等"不等于CPU空转**：CPU在执行查询指令(读端口+条件判断)，只是这些指令没有有效产出
3. **查询方式的效率与设备速度成反比**：设备越慢，CPU浪费越严重
4. **查询方式也有数据缓冲**：接口仍然有数据缓冲寄存器，不是直接从设备读取
