# 25_异步定时_全互锁

## 核心概念

### 全互锁方式定义

**全互锁（Fully Interlocked）**：完整的四步握手协议。主设备等从设备的ACK，从设备也等主设备撤销请求，双方完全确认。

### 全互锁的时序

```
主设备: ──Req有效──────────────────Req撤销──────────
从设备: ────────ACK有效──────────────ACK撤销──────
数据:   ────────有效数据──────────────────────────
```

**四步握手过程**：
1. 主设备发出请求（Req↑）
2. 从设备检测到请求，准备数据，发出ACK（ACK↑）
3. 主设备检测到ACK，读取数据，撤销请求（Req↓）
4. 从设备检测到Req撤销，撤销ACK（ACK↓）
5. 一次完整传输结束，总线恢复空闲

### 全互锁的特点

| 特点 | 说明 |
|------|------|
| 速度 | 最慢（需要四步握手） |
| 可靠性 | 最高（双向完全确认） |
| 硬件 | 最复杂 |
| 适用 | 高可靠性要求的系统 |

## 原理分析

### 全互锁的可靠性

每一步都有明确的确认：
- 主设备发请求 → 从设备确认（ACK）
- 从设备发ACK → 主设备确认（Req撤销）
- 主设备撤销Req → 从设备确认（ACK撤销）

任何一步失败，系统都能检测到（超时检测）。

### 全互锁的时间

$$
T_{\text{传输}} = T_{\text{Req传播}} + T_{\text{从设备处理}} + T_{\text{ACK传播}} + T_{\text{主设备读取}} + T_{\text{Req撤销传播}} + T_{\text{ACK撤销传播}}
$$

全互锁的传输时间最长，但最可靠。

## 直观理解

### 生活类比

全互锁就像**完全确认的电话通话**：

```
A: "你听到了吗？"（Req↑）
B: "听到了！数据是XXX"（ACK↑）
A: "好，我收到数据了"（Req↓）
B: "好的，我知道你收到了"（ACK↓）
— 通话结束 —
```

每一步都有确认，不会出错。

### 记忆技巧

全互锁：「四步握手，完全确认」
- Req↑ → ACK↑ → Req↓ → ACK↓
- 每一步都等对方确认
- 最慢但最可靠

## 知识关联

### 三种异步定时方式对比

| 对比 | 不互锁 | 半互锁 | 全互锁 |
|------|--------|--------|--------|
| 握手步骤 | 1步 | 2-3步 | 4步 |
| 主设备等ACK | 否 | 是 | 是 |
| 从设备等Req撤销 | 否 | 否 | 是 |
| 速度 | 最快 | 中等 | 最慢 |
| 可靠性 | 最低 | 中等 | 最高 |

### 易错陷阱

1. **全互锁不是最快的**：是最慢但最可靠的
2. **四步握手的顺序不能颠倒**
3. **全互锁适合速度差异大的设备**：因为不需要预设时间
