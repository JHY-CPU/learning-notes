# 14_集中仲裁_链式查询

## 核心概念

### 链式查询方式

**链式查询（Daisy Chain）**：各设备通过一条BG（总线授权）线串行连接，仲裁器发出的BG信号沿着链逐个设备传递。

### 需要的信号线

| 信号 | 数量 | 说明 |
|------|------|------|
| BR（Bus Request） | 1根 | 所有设备共用一根请求线 |
| BG（Bus Grant） | 1根 | 授权线串行经过所有设备 |
| BS（Bus Busy） | 1根 | 总线忙信号 |

总共只需要**3根信号线**。

### 工作原理

```
                    BR（所有设备共用）
设备1 ──→ 设备2 ──→ 设备3 ──→ ... ──→ 仲裁器
  ↑         ↑         ↑                    │
  └────BG链────────────────────────────────┘
```

**工作过程**：
1. 设备需要总线时，通过BR线向仲裁器发出请求
2. 仲裁器收到请求后，如果总线空闲（BS=0），发出BG信号
3. BG信号沿链从设备1开始逐个传递
4. 如果当前设备没有请求，将BG传给下一个设备
5. 如果当前设备有请求，截获BG信号，不再下传
6. 该设备设置BS=1（占用总线），开始传输数据
7. 传输完成后，设备释放总线（BS=0）

### 优先级特点

**优先级固定**：离仲裁器越近的设备优先级越高。

- 设备1优先级最高
- 设备n优先级最低
- 优先级不可动态改变

## 原理分析

### 链式查询的时序

```
时序：
t0: 设备2和设备3同时请求（BR=1）
t1: 仲裁器发出BG
t2: BG到达设备1（设备1未请求，转发BG）
t3: BG到达设备2（设备2有请求，截获BG）
t4: 设备2设置BS=1，开始使用总线
```

### 链式查询的性能分析

**最好情况**：最近的设备请求 → 1个时钟周期后获得授权
**最坏情况**：最远的设备请求 → n个时钟周期后获得授权
**平均情况**：$(n+1)/2$ 个时钟周期

$$
T_{\text{仲裁}} = k \times T_{\text{传递}}
$$

其中 $k$ 是请求设备的链上位置。

## 直观理解

### 生活类比

**链式查询就像排队领奖品**：
- 队伍从第一个人开始
- BG就像一个"领奖棒"，从头传到尾
- 谁想要奖品，拿到棒就领走，后面的就等下次
- 离发奖处越近的人越容易先拿到

### 记忆技巧

链式查询：「一线三根，近高远低」
- 3根信号线
- 离仲裁器近的设备优先级高

## 知识关联

### 优缺点

| 优点 | 缺点 |
|------|------|
| 只需3根信号线 | 优先级固定，不灵活 |
| 结构简单 | 离仲裁器远的设备可能"饿死" |
| 易于扩展 | BG链的故障会导致后续设备无法获得授权 |
| 仲裁器逻辑简单 | 仲裁时间与设备数成正比 |

2. 优先级排列方式

## 代码/模拟

### Python模拟链式查询总线仲裁

```python
"""链式查询总线仲裁模拟 - 适用于408考研复习"""

class DaisyChainArbitrator:
    """链式查询（Daisy Chain）总线仲裁器"""

    def __init__(self, num_devices):
        self.num_devices = num_devices
        # 设备编号1~n，设备1离仲裁器最近（优先级最高）
        self.requests = [False] * (num_devices + 1)  # 请求信号
        self.bus_busy = False                          # 总线忙信号
        self.current_owner = None

    def request_bus(self, device_id):
        """设备发出总线请求"""
        if 1 <= device_id <= self.num_devices:
            self.requests[device_id] = True
            print(f"  设备{device_id}发出总线请求 (BR=1)")

    def arbitrate(self):
        """仲裁器执行一轮仲裁过程"""
        if self.bus_busy:
            print("  总线忙，等待...")
            return None

        # 检查是否有请求
        has_request = any(self.requests[1:])
        if not has_request:
            print("  无设备请求，总线空闲")
            return None

        print("  仲裁器发出BG（总线授权）信号")
        print("  BG信号沿链传递:")

        # BG信号沿链逐设备传递
        for i in range(1, self.num_devices + 1):
            if self.requests[i]:
                # 设备i有请求，截获BG信号
                self.bus_busy = True
                self.current_owner = i
                self.requests[i] = False
                print(f"    → 设备{i}: 有请求！截获BG，占用总线 (BS=1)")
                print(f"    设备{i}获得总线使用权，仲裁耗时={i}个周期")
                return i
            else:
                # 设备i无请求，转发BG
                print(f"    → 设备{i}: 无请求，转发BG")

        return None

    def release_bus(self, device_id):
        """设备释放总线"""
        if self.current_owner == device_id:
            self.bus_busy = False
            self.current_owner = None
            print(f"  设备{device_id}释放总线 (BS=0)")

# 模拟场景
print("=== 链式查询总线仲裁模拟 ===\n")
arbiter = DaisyChainArbitrator(num_devices=4)

print("场景1: 设备3和设备4同时请求总线")
arbiter.request_bus(3)
arbiter.request_bus(4)
owner = arbiter.arbitrate()  # 设备3优先级更高
print()

print("场景2: 总线被占用时新请求到达")
arbiter.request_bus(2)
arbiter.arbitrate()  # 总线忙
print()

print("场景3: 总线释放后重新仲裁")
arbiter.release_bus(3)
arbiter.request_bus(1)  # 设备1也请求
owner = arbiter.arbitrate()  # 设备1优先级最高
print()

# 仲裁时间分析
print("=== 仲裁时间分析 ===")
print("设备位置  最好情况  最坏情况  平均情况")
for i in range(1, 5):
    best = 1          # 设备1: 1个周期
    worst = i         # 设备i: i个周期
    print(f"  设备{i}      {best}周期    {worst}周期    {(1+i)/2:.1f}周期")
```

**关键结论验证**：
- 设备1（最近）仲裁时间 = 1个周期
- 设备n（最远）仲裁时间 = n个周期
- 一个设备故障 → 后续所有设备无法获得授权
