# 15_集中仲裁_计数器定时

## 核心概念

### 计数器定时查询方式

**计数器定时查询（Counter Polling）**：仲裁器内部维护一个计数器，当有总线请求时，仲裁器从计数器的当前值开始，依次查询每个设备是否需要总线。

### 需要的信号线

| 信号 | 数量 | 说明 |
|------|------|------|
| BR（总线请求） | 1根 | 所有设备共用 |
| BS（总线忙） | 1根 | 总线占用标志 |
| 设备地址线 | $\lceil \log_2 n \rceil$ 根 | 用于发送设备编号 |

设备数量为 $n$ 时，需要 $\lceil \log_2 n \rceil$ 根地址线来编码 $n$ 个设备。

**总信号线数**：$2 + \lceil \log_2 n \rceil$

### 工作原理

```
设备1 ──→ 设备2 ──→ 设备3 ──→ ... ──→ 设备n
  ↑         ↑         ↑                ↑
  └───────────── 地址线 ────────────────┘
                         │
                    仲裁器（含计数器）
```

**工作过程**：
1. 设备通过BR线发出总线请求
2. 仲裁器收到请求后，如果总线空闲
3. 仲裁器将计数器的当前值发送到设备地址线上
4. 地址匹配的设备获得总线使用权
5. 如果该设备正好有请求，它设置BS=1，使用总线
6. 如果该设备没有请求，计数器加1，查询下一个设备
7. 重复3-6直到找到有请求的设备

### 优先级特点

| 模式 | 说明 |
|------|------|
| 固定优先级 | 计数器从0开始，设备0优先级最高 |
| 轮询优先级 | 计数器从上次结束的值开始，各设备轮流优先级最高 |
| 可编程优先级 | 通过软件设置计数器的初始值 |

## 原理分析

### 计数器定时查询的时序

假设3个设备（编号0、1、2），计数器当前值=1：

```
t0: 设备0和设备2同时请求（BR=1）
t1: 仲裁器发送计数器值=1到地址线
t2: 设备1检查：地址匹配，但设备1没有请求
t3: 计数器+1=2，仲裁器发送地址=2
t4: 设备2检查：地址匹配，设备2有请求
t5: 设备2设置BS=1，开始使用总线
```

### 信号线数量计算

| 设备数 | 地址线数量 | 总信号线数 |
|--------|-----------|-----------|
| 4 | $\lceil \log_2 4 \rceil = 2$ | 2 + 2 = 4 |
| 8 | $\lceil \log_2 8 \rceil = 3$ | 2 + 3 = 5 |
| 16 | $\lceil \log_2 16 \rceil = 4$ | 2 + 4 = 6 |
| 64 | $\lceil \log_2 64 \rceil = 6$ | 2 + 6 = 8 |

## 直观理解

### 生活类比

**计数器定时查询就像叫号系统**：
- 仲裁器像叫号机
- 每次叫一个号码（计数器值）
- 对应号码的人如果是本人需要服务，就去办理
- 如果不是，叫下一个号
- 可以设置从哪个号开始叫（可编程）

### 记忆技巧

计数器定时：「叫号查询，可编程」
- 像叫号一样逐个查询
- 起始号码可编程

## 知识关联

### 优缺点

| 优点 | 缺点 |
|------|------|
| 优先级可灵活设置 | 需要设备地址线，信号线较多 |
| 可实现轮询公平 | 仲裁时间不确定（取决于计数器位置） |
| 不会因单个设备故障影响其他设备 | 比链式查询复杂 |

### 与链式查询对比

| 对比维度 | 链式查询 | 计数器定时 |
|---------|---------|-----------|
| 信号线数 | 3 | $2 + \lceil \log_2 n \rceil$ |
| 优先级 | 固定（近高远低） | 可灵活设置 |
| 故障影响 | 一个故障影响后续 | 无影响 |
| 扩展性 | 好（加设备即可） | 需增加地址线 |

### 常考组合

1. 信号线数量的计算
2. 计数器的初始值设置与优先级的关系
3. 轮询公平性的实现

### 易错陷阱

1. **设备地址线不是BR线**：BR是请求信号线
2. **计数器的值不一定是从0开始**：可编程
3. **仲裁时间是不确定的**：取决于计数器位置和请求设备的位置
