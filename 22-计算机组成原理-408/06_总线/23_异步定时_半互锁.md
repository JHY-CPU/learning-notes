# 24_异步定时_半互锁

## 核心概念

### 半互锁方式定义

**半互锁（Semi-interlocked）**：主设备发出请求后，必须等待从设备的应答信号（ACK）才继续操作。但从设备发出ACK后，不需要等待主设备撤销请求。

### 半互锁的时序

```
主设备: ──Req有效──────────────────Req撤销──
从设备: ────────ACK有效────ACK撤销──────
数据:   ────────有效数据──────────────────
```

**工作过程**：
1. 主设备发出请求（Req有效）
2. 从设备检测到请求，准备数据
3. 从设备将数据放到数据线，发出ACK
4. 主设备检测到ACK，读取数据
5. 主设备撤销请求（从设备不等待这一步）
6. 从设备看到合适时机撤销ACK

### 半互锁的特点

| 特点 | 说明 |
|------|------|
| 主设备 | 必须等ACK才能继续 |
| 从设备 | 发出ACK后可立即开始其他工作 |
| 速度 | 中等（比不互锁慢，比全互锁快） |
| 可靠性 | 中等 |

## 原理分析

### 半互锁的改进

相比不互锁：
- 增加了ACK信号，主设备能确认从设备是否响应
- 不再依赖预设的固定等待时间

相比全互锁：
- 从设备不需要等主设备撤销请求
- 减少了等待时间

### 半互锁的风险

- 如果主设备撤销请求的速度太慢，从设备可能已经开始了下一次操作
- 在极少数情况下可能导致信号冲突

## 直观理解

### 生活类比

半互锁就像**打电话中的情况**：
- A说："你听到了吗？"（请求）
- B说："听到了！"（ACK）
- B说完后就去做别的事了，不管A是否听到了自己的回复
- A确认收到后，挂电话（撤销请求）

### 记忆技巧

半互锁：「单向确认」
- 主设备确认从设备的ACK（从→主方向有确认）
- 从设备不确认主设备的请求撤销（主→从方向无确认）

## 知识关联

### 易错陷阱

1. **半互锁的"半"是指确认方向只有一半**
2. **ACK是主动信号**：从设备主动发出
3. **半互锁比不互锁可靠，但比全互锁快**
