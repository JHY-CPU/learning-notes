# 23_异步定时_不互锁

## 核心概念

### 不互锁方式定义

**不互锁（Non-interlocked）**：主设备发出请求信号后，等待一个固定的时间（足够从设备响应），然后就认为从设备已经收到并处理了请求。主设备不确认从设备是否真的收到了请求。

### 不互锁的时序

```
主设备: ──Req有效────────等待固定时间────Req撤销──
从设备: ──────────────数据有效────────────────
```

**工作过程**：
1. 主设备发出请求信号（Req有效）
2. 主设备等待一个预设的固定时间
3. 从设备在这段时间内完成响应（假设一定来得及）
4. 主设备读取数据，撤销请求

### 不互锁的特点

| 特点 | 说明 |
|------|------|
| 速度 | 最快（没有等待ACK的时间） |
| 可靠性 | 最低（无法确认从设备是否响应） |
| 硬件 | 最简单 |
| 适用 | 速度已知且稳定的设备 |

## 原理分析

### 不互锁的时间设计

固定等待时间必须足够长，确保最慢的从设备也能完成响应：

$$
T_{\text{等待}} \geq T_{\text{最慢从设备响应}}
$$

如果等待时间太短，快设备没问题，但慢设备可能来不及响应。

### 不互锁的风险

- 如果从设备故障，主设备不会知道
- 如果从设备比预期的慢，会读到错误数据
- 不适合速度差异大或不确定的设备

## 直观理解

### 生活类比

不互锁就像**寄信**：
- 你把信寄出去（发出请求）
- 等了几天（固定时间）
- 假设对方一定收到了
- 你不知道对方是否真的收到了

### 记忆技巧

不互锁：「发了就等，不确认」
- 发出请求
- 等固定时间
- 不管对方是否收到

## 知识关联

### 与其他互锁方式的对比

| 对比 | 不互锁 | 半互锁 | 全互锁 |
|------|--------|--------|--------|
| 主设备等ACK | 否 | 是 | 是 |
| 从设备等Req撤销 | 否 | 否 | 是 |
| 速度 | 最快 | 中等 | 最慢 |
| 可靠性 | 最低 | 中等 | 最高 |

### 易错陷阱

1. **不互锁不是握手**：没有ACK信号
2. **不互锁的等待时间是预设的**：不是动态的
3. **不互锁适合速度一致的设备**
