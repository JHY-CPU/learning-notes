# 09_字符串存储_大小端

## 核心概念

### 字节序（Byte Order）

**字节序**是指多字节数据在内存中存储时字节的排列顺序。

### 大端模式（Big-Endian）
- **高位字节存储在低地址**
- **低位字节存储在高地址**
- 符合人类阅读习惯：从左到右，从高到低

### 小端模式（Little-Endian）
- **低位字节存储在低地址**
- **高位字节存储在高地址**
- 与阅读顺序相反

### 对比示例

以32位整数 $0x12345678$ 为例（假设起始地址为 $1000$）：

| 地址 | 大端模式 | 小端模式 |
|------|----------|----------|
| 1000 | 12 | 78 |
| 1001 | 34 | 56 |
| 1002 | 56 | 34 |
| 1003 | 78 | 12 |

- **大端**：高位字节 `12` 在最低地址 $1000$
- **小端**：低位字节 `78` 在最低地址 $1000$

### 名称由来
- 来源于《格列佛游记》中关于鸡蛋从大头还是小头打开的争论
- Big-Endian = 大端 = 高位字节在前（大头在前）
- Little-Endian = 小端 = 低位字节在前（小头在前）

## 原理分析

### 字符串的字节序

对于**纯ASCII字符串**，不存在字节序问题，因为每个字符只占1字节。

对于**多字节字符**（如UTF-8编码的汉字、UTF-16编码的Unicode），字节序会影响存储。

**示例**：UTF-16编码下的汉字"中"（Unicode码点 $0x4E2D$）

| 模式 | 内存内容（从低地址到高地址） |
|------|---------------------------|
| 大端（UTF-16BE） | `4E` `2D` |
| 小端（UTF-16LE） | `2D` `4E` |

UTF-16文件通常以 **BOM**（Byte Order Mark）`FE FF`（大端）或 `FF FE`（小端）开头来标识字节序。

### 常见处理器架构的字节序

| 架构 | 字节序 | 代表处理器 |
|------|--------|-----------|
| x86 / x86-64 | **小端** | Intel、AMD |
| ARM | 可配置（通常小端） | 手机、嵌入式 |
| PowerPC | 大端 | 旧款Mac |
| MIPS | 可配置 | 嵌入式、路由器 |
| RISC-V | 小端（推荐） | 新一代架构 |

## 直观理解

### 生活类比

**大端**：就像写日期"2026年5月5日"，高位（年）写在左边，低位（日）写在右边。

**小端**：就像倒着写日期"5月5日2026年"，低位在前。

### 记忆技巧
- 大端 = 网络字节序（Network Byte Order）= 高字节在前 = 符合人类直觉
- 小端 = 主机字节序（Host Byte Order）= 低字节在前 = Intel体系
- 口诀："大端高前，小端低前"

## 知识关联

### 408考试要点

**典型题目**：给定数据和内存布局，判断是大端还是小端。

**解题思路**：
1. 确认数据的高位字节和低位字节
2. 查看低地址存的是哪个字节
3. 低地址存高位字节 → 大端；低地址存低位字节 → 小端

**例题**：32位数据 $0x00000001$ 存储在起始地址 $0x1000$，$0x1000$ 处的内容为 $0x00$，则该系统是（）

分析：$0x00000001$ 的高字节为 $0x00$，低字节为 $0x01$。低地址 $0x1000$ 处存的是 $0x00$（高字节），所以是**大端模式**。

### 易错陷阱
1. **混淆字节序方向**：记忆法——看最低地址存的是"大的"还是"小的"
2. **字符串不需要考虑字节序**：字符串是逐字节存储的，不存在多字节的整体重排问题
3. **网络传输统一用大端**：socket编程中需要 `htons()`、`htonl()` 等函数进行字节序转换
4. **浮点数也有字节序问题**：IEEE 754浮点数在不同字节序的机器上存储不同
