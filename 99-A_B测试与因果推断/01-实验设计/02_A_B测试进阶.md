# A/B测试进阶


## 一、多重检验问题


当同时进行多个假设检验时，整体第一类错误率（Family-Wise Error Rate, FWER）会增大。如果做 m 个独立检验，每个 α=0.05，整体犯错概率约为 1-(1-0.05)^m^。


### 1.1 Bonferroni 校正


最简单的方法：将显著性水平除以检验次数。


$$
校正后 α' = α / m
                例如：20个检验，α' = 0.05/20 = 0.0025
$$


- **优点：**
   简单，严格控制FWER
- **缺点：**
   过于保守，容易漏检真实效应（功效低）


### 1.2 FDR控制（Benjamini-Hochberg）


FDR（False Discovery Rate）控制的是"被拒绝的假设中错误的比例"，比Bonferroni更灵活。


1. 将 m 个p值从小到大排序：p
   ~(1)~
   ≤ p
   ~(2)~
   ≤ ... ≤ p
   ~(m)~
2. 找到最大的 k 使得 p
   ~(k)~
   ≤ (k/m) × α
3. 拒绝前 k 个假设


| 方法 | 控制目标 | 严格程度 | 适用场景 |
| --- | --- | --- | --- |
| Bonferroni | FWER ≤ α | 最严格 | 少数关键检验 |
| Holm | FWER ≤ α | 严格 | Bonferroni的改进版 |
| BH (FDR) | FDR ≤ α | 适中 | 大量检验（推荐） |
| 无校正 | — | 无 | 探索性分析 |


## 二、序贯检验（Sequential Testing）


传统A/B测试需要预先确定样本量，实验期间不能偷看数据（Peeking Problem）。序贯检验允许**随时检查结果**并在达到结论时提前停止。


### 2.1 Peeking Problem


如果每看一次数据就做一次检验，错误率会膨胀。10次偷看的实际错误率可能达到15-20%，远高于设定的5%。


### 2.2 序贯概率比检验（SPRT）


$$
每次新数据到来时计算似然比：
                LRn = L(H1 | data1..n) / L(H0 | data1..n)
                设定两个阈值 A 和 B：
                LRn ≥ A → 拒绝H0（实验组胜）
                LRn ≤ B → 不拒绝H0（无差异）
                B < LRn < A → 继续收集数据
$$


### 2.3 mSPRT（混合序贯概率比检验）


Google 推荐使用 mSPRT，结合了置信序列（Confidence Sequences），可以在任意时刻查看结果而控制错误率。这是许多大公司（Google, Spotify等）在线实验平台的核心方法。


> **Note:** **实践建议：**
> 如果使用序贯检验框架（如Google的CUPED+SPRT），可以设置每周检查一次结果，在达到显著性时提前结束实验，平均可节省30-50%的实验时间。


## 三、分层实验与互斥实验


### 3.1 实验分层


在大型互联网产品中，可能同时运行数十甚至上百个实验。合理的实验架构设计至关重要。


| 类型 | 原理 | 适用场景 |
| --- | --- | --- |
| **互斥实验** | 同一用户只进入一个实验 | 修改相同功能的实验 |
| **正交实验** | 同一用户可同时进入多个不相关实验 | 修改不同模块的实验 |
| **分层实验** | 将流量分成多层，层内互斥，层间正交 | 大规模实验平台 |


### 3.2 流量分层架构


$$
全量流量 → 分桶（如1000桶）
                Layer 1: 推荐算法实验（占用桶1-300）
                Layer 2: UI实验（占用桶301-600）
                Layer 3: 广告策略实验（占用桶601-900）
                公共池：桶901-1000用于保留流量
$$


> **Important:** **关键原则：**
> 修改同一功能的实验必须互斥（如两个推荐算法实验不能同时作用于同一用户）。不同模块的实验可以正交（如推荐算法和UI改版可以同时实验）。


## 四、常见陷阱与注意事项


### 4.1 辛普森悖论（Simpson's Paradox）


在总体层面看到的趋势可能与分组层面的趋势相反。这是因为分组比例不均衡导致的。


> **Example:** **经典案例：**
> 某APP实验显示实验组总体转化率更高。但按新/老用户拆分后发现：两组中新用户转化率都更低，老用户转化率也更低。总体提升的原因是实验组吸引了更多新用户（新用户转化率高）。这个"提升"是虚假的。


### 4.2 新奇效应（Novelty Effect）


新功能刚上线时用户因为好奇而使用更多，但效果会随时间衰减。对策：运行足够长时间（至少1-2周），观察指标是否收敛。


### 4.3 网络效应（Network Effect）


社交产品中，实验组用户的行为会影响控制组用户（如聊天功能改版后实验组用户发消息更多，控制组收到的消息也变多）。对策：使用聚类随机化（按社交圈分组）或地理随机化。


### 4.4 其他陷阱


- **选择偏差：**
   分流不均匀，两组用户特征不同
- **数据污染：**
   cookie清理、设备切换导致用户被重复分配
- **指标游戏：**
   选择性报告最佳指标（cherry-picking）
- **提前终止偏见：**
   看到好结果就停止，导致假阳性


## 五、实验平台设计


### 5.1 核心模块


| 模块 | 功能 |
| --- | --- |
| 分流服务 | 用户分桶、实验分配、哈希一致性 |
| 参数管理 | 实验配置、白名单、灰度发布 |
| 数据采集 | 曝光日志、行为日志、指标计算 |
| 统计分析 | 假设检验、置信区间、功效分析 |
| 实验报告 | 可视化、多维度下钻、异常检测 |


### 5.2 分流一致性


```
import hashlib

def assign_bucket(user_id, experiment_id, num_buckets=1000):
    """基于哈希的一致性分流"""
    hash_input = f"{user_id}_{experiment_id}"
    hash_value = int(hashlib.md5(hash_input.encode()).hexdigest(), 16)
    bucket = hash_value % num_buckets
    return bucket

def assign_group(user_id, experiment_id, control_ratio=0.5):
    """分配实验组"""
    bucket = assign_bucket(user_id, experiment_id)
    threshold = int(1000 * control_ratio)
    return "control" if bucket < threshold else "treatment"

# 示例
print(assign_group("user_12345", "exp_recommendation_v2"))
# 输出: "control" 或 "treatment"
```


## 六、实际案例：APP按钮颜色实验


> **Example:** ### 案例背景
>
>
> 某电商APP想测试将"购买"按钮从蓝色改为红色是否能提高转化率。
>
>
> ### 实验设计
>
>
> - **核心指标：**
>    购买转化率（下单用户/访问用户）
> - **护栏指标：**
>    跳出率、平均订单金额
> - **基线转化率：**
>    8.5%
> - **MDE：**
>    相对提升10%（即从8.5%到9.35%）
> - **α=0.05, Power=0.80**
> - **每组所需样本量：**
>    约16,000
> - **实验周期：**
>    14天（覆盖两个完整周末）
>
>
> ### 结果分析
>
>
> | 指标 | 控制组（蓝色） | 实验组（红色） | 差异 | p值 |
> | --- | --- | --- | --- | --- |
> | 转化率 | 8.52% | 9.41% | +0.89% | 0.003 |
> | 跳出率 | 35.2% | 34.8% | -0.4% | 0.31 |
> | 平均订单金额 | 156元 | 158元 | +2元 | 0.42 |
>
>
> **结论：**红色按钮显著提高了转化率（p=0.003 < 0.05），护栏指标无显著退化，建议上线。


## 总结


- 多重检验需用Bonferroni或FDR校正控制错误率
- 序贯检验允许随时查看结果，避免Peeking Problem
- 分层实验架构让不同模块的实验可以并行运行
- 辛普森悖论提醒我们按关键维度下钻分析
- 新奇效应和网络效应需要在实验设计阶段考虑
- 一致性分流（基于哈希）是实验平台的基础
