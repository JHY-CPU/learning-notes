常见面试题

// ========== SQL 面试高频题 ========== // Q1: INNER JOIN vs LEFT JOIN 区别? // A: INNER JOIN 只返回匹配行, LEFT JOIN 返回左表所有行 // 右表不匹配处为 NULL // Q2: WHERE 和 HAVING 区别? // A: WHERE 在 GROUP BY 前过滤行 // HAVING 在 GROUP BY 后过滤分组 // 聚合条件必须在 HAVING 中 // Q3: CHAR vs VARCHAR 区别? // A: CHAR(N) 固定长度, 不足补空格 // VARCHAR(N) 可变长度, 按实际存储+1-2字节 // 固定长度字段用 CHAR 更快 // Q4: 四种隔离级别? // A: RU(读未提交) → 脏读 // RC(读已提交) → 不可重复读 // RR(可重复读) → 幻读 (MySQL默认) // SERIALIZABLE → 无问题但性能差 // Q5: 索引为什么能加速查询? // A: B-Tree 结构 O(log N) 查找 // 不排序索引: 全表扫描 O(N) // 相当于书的目录 // Q6: 什么是覆盖索引? // A: 索引包含查询所需的所有列 // Extra: Using Index (不回表) // 减少磁盘 IO // Q7: UNION vs UNION ALL? // A: UNION 去重 (额外排序), UNION ALL 保留重复 // 能确定无重复就用 UNION ALL, 更快 // Q8: 如何优化慢查询? // A: 1. EXPLAIN 分析 // 2. 检查 type 是否为 ALL // 3. 加合适索引 // 4. 避免 SELECT * // 5. 拆分大查询 // Q9: DELETE vs TRUNCATE vs DROP? // A: DELETE 逐行删, 可回滚, 不释放空间 // TRUNCATE 清表, 不可回滚, 释放空间 // DROP 删表结构 // Q10: N+1 查询问题? // A: 查询主表 N 行后每条再查关联表 // 解决方案: JOIN 一次查出, 或批量查询

故障排查 Checklist

// ========== SQL 故障排查 ========== // ┌─ 查询慢 ───────────────────────────────── // 1. EXPLAIN 查看 type/rows/Extra // 2. 检查是否有合适索引 (key 列) // 3. 避免 SELECT * (改为具体列) // 4. 避免 WHERE 中函数包裹列 // 5. 检查数据量是否突然增长 // 6. 检查连接数是否耗尽 // └──────────────────────────────────────── // ┌─ 连接失败 ─────────────────────────────── // 1. 检查数据库是否运行 // 2. 检查网络/防火墙/端口 // 3. 验证用户名密码 // 4. 检查 host 权限 ('user'@'%' vs 'user'@'localhost') // 5. 检查 max_connections 上限 // 6. 检查 SSL 配置是否匹配 // └──────────────────────────────────────── // ┌─ 死锁 ─────────────────────────────────── // 1. SHOW ENGINE INNODB STATUS \G // 2. 查看 LATEST DETECTED DEADLOCK 部分 // 3. 确保事务按相同顺序访问资源 // 4. 缩短事务时间 // 5. 应用层捕获 1213 并重试 // 6. 考虑降低隔离级别 // └──────────────────────────────────────── // ┌─ 数据不一致 ───────────────────────────── // 1. 检查事务是否忘记 COMMIT // 2. 检查是否有外键约束 // 3. 检查并发是否导致竞态 // 4. 检查是否有触发器副作用 // 5. 检查是否有存储过程错误 // └──────────────────────────────────────── // ┌─ 备份恢复 ─────────────────────────────── // 1. 验证备份文件完整性 // 2. 在测试环境演练恢复 // 3. 检查 binlog/WAL 是否完整 // 4. 确认时间点精确性 // 5. 定期测试恢复流程 // └──────────────────────────────────────── // ========== 最佳实践 ========== // 1. 开发阶段: 写 SQL 先写 SELECT 再转 DELETE/UPDATE // 2. 上线前: EXPLAIN 所有查询 // 3. 表设计: 每表要有主键, 字段 NOT NULL 优先 // 4. 索引设计: 复合索引最左前缀, 不要过度索引 // 5. 安全: 永远不要拼接 SQL, 参数化查询
