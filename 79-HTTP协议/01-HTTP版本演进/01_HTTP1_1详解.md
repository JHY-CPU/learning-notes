# HTTP/1.1 详解


## HTTP/1.1 详解


HTTP持久连接缓存


HTTP/1.1 是 HTTP 协议的第一个标准化版本（RFC 2616，1999年），引入了持久连接、管线化、分块传输编码等关键特性，至今仍广泛使用。


## 持久连接（Persistent Connection）


```
HTTP/1.0 的问题：
  每个请求都需要建立新的TCP连接
  TCP三次握手 + TLS握手 → 延迟高
  一个页面有几十个资源 → 几十次连接建立

HTTP/1.1 默认持久连接：
  Connection: keep-alive（默认行为）
  多个请求复用同一个TCP连接
  减少了连接建立的开销

客户端请求：
  GET /index.html HTTP/1.1
  Host: www.example.com
  Connection: keep-alive

服务端响应：
  HTTP/1.1 200 OK
  Content-Type: text/html
  Content-Length: 1024
  Connection: keep-alive

  [响应体]

// 同一连接上继续发送下一个请求
  GET /style.css HTTP/1.1
  Host: www.example.com

关键机制：
  - Content-Length 头部：告诉客户端响应体有多大
  - 客户端读取指定字节数后知道响应结束
  - 服务端也可以使用 Transfer-Encoding: chunked

关闭持久连接：
  Connection: close
  服务端发送完响应后关闭TCP连接

超时管理：
  - keepalive_timeout：空闲连接保持时间（如 60s）
  - 超过时间无新请求则关闭连接
  - 防止连接资源浪费
```


## 管线化（Pipelining）


```
管线化原理：
  客户端可以在收到上一个响应之前发送下一个请求
  不需要等待上一个请求的响应

无管线化（串行）：
  Client                    Server
    |--- Request A --------->|
    |<--- Response A --------|
    |--- Request B --------->|
    |<--- Response B --------|
    |--- Request C --------->|
    |<--- Response C --------|
  总时间 = 3 × RTT

管线化（并行发送）：
  Client                    Server
    |--- Request A --------->|
    |--- Request B --------->|
    |--- Request C --------->|
    |<--- Response A --------|
    |<--- Response B --------|
    |<--- Response C --------|
  总时间 ≈ 1 × RTT（理想情况）

队头阻塞问题（Head-of-Line Blocking）：
  ┌─────────────────────────────────────────────┐
  │  响应必须按请求顺序返回                      │
  │  如果 Response A 处理慢                      │
  │  Response B 和 C 即使已处理完也要等待        │
  │                                             │
  │  Request A → [慢]                           │
  │  Request B → [快] → 等待A完成才能返回       │
  │  Request C → [快] → 等待A、B完成才能返回    │
  └─────────────────────────────────────────────┘

实际使用情况：
  - 大多数浏览器默认关闭管线化
  - 队头阻塞导致性能不稳定
  - 通常使用多个TCP连接（Chrome 6个）来并行请求
  - HTTP/2 的多路复用解决了这个问题
```


## 分块传输编码（Chunked Encoding）


```
使用场景：
  服务端事先不知道响应体大小
  动态生成内容（如模板渲染、数据库查询结果）
  流式传输（大文件、实时日志）

响应格式：
  HTTP/1.1 200 OK
  Transfer-Encoding: chunked
  Content-Type: text/html

  5\r\n           ← 块大小（十六进制）
  Hello\r\n       ← 块数据
  6\r\n
  World!\r\n
  0\r\n           ← 最后一个块（大小为0）
  \r\n            ← 结束

实际示例：
  // 服务端动态生成响应
  HTTP/1.1 200 OK
  Transfer-Encoding: chunked

  1a\r\n
  <html><body>Hello\r\n
  10\r\n
  , World!</body>\r\n
  9\r\n
  </html>\r\n
  0\r\n
  \r\n

与 Content-Length 的对比：
  ┌──────────────────┬──────────────────────────┐
  │  Content-Length  │  Transfer-Encoding       │
  ├──────────────────┼──────────────────────────┤
  │  需提前知道大小  │  不需要知道总大小        │
  │  静态文件适用    │  动态内容适用            │
  │  一次发送完毕    │  边生成边发送            │
  │  不支持压缩传输  │  可与 gzip 结合          │
  └──────────────────┴──────────────────────────┘

注意：
  - 不能同时使用 Content-Length 和 Transfer-Encoding
  - 每个 chunk 后面需要 \r\n
  - 最后一个 chunk 大小为 0
```


## 内容协商（Content Negotiation）


```
服务器驱动的内容协商：
  客户端在请求头中说明偏好，服务端选择最合适的内容

Accept 头部：
  Accept: text/html, application/xhtml+xml, application/xml;q=0.9, */*;q=0.8
  // q 值表示优先级（0-1），默认为1

Accept-Language 头部：
  Accept-Language: zh-CN,zh;q=0.9,en;q=0.8
  // 优先中文，其次英文

Accept-Encoding 头部：
  Accept-Encoding: gzip, deflate, br
  // 支持的压缩算法

Accept-Charset 头部：
  Accept-Charset: UTF-8, ISO-8859-1;q=0.5

服务端响应：
  HTTP/1.1 200 OK
  Content-Type: text/html; charset=UTF-8
  Content-Language: zh-CN
  Content-Encoding: gzip
  Vary: Accept-Language, Accept-Encoding

协商类型：
  ┌─────────────────────┬──────────────────────────────┐
  │  服务器驱动协商     │  客户端在请求头中说明偏好    │
  │  (Server-driven)    │  服务端选择最佳表示          │
  ├─────────────────────┼──────────────────────────────┤
  │  代理驱动协商       │  服务端返回300 Multiple      │
  │  (Agent-driven)     │  客户端选择后重新请求        │
  ├─────────────────────┼──────────────────────────────┤
  │  透明协商           │  代理/CDN代替客户端协商      │
  │  (Transparent)      │  减少往返次数                │
  └─────────────────────┴──────────────────────────────┘

Vary 头部：
  Vary: Accept-Language, Accept-Encoding
  // 告诉缓存服务器：不同 Accept-Language 值对应不同缓存
  // 缓存时需要考虑这些头部的值
```


## 缓存机制（ETag / Cache-Control）


```
Cache-Control 头部（HTTP/1.1）：
  Cache-Control: max-age=3600
  // 缓存有效期 3600 秒

  Cache-Control: no-cache
  // 每次都向服务端验证（可以用缓存）

  Cache-Control: no-store
  // 禁止缓存（完全不存储）

  Cache-Control: public
  // 中间代理也可以缓存

  Cache-Control: private
  // 只有浏览器可以缓存（代理不能）

  Cache-Control: must-revalidate
  // 缓存过期后必须向服务端验证

ETag（实体标签）：
  // 服务端为资源生成唯一标识
  HTTP/1.1 200 OK
  ETag: "686897696a7c876b7e"
  Content-Length: 1024

  // 客户端再次请求时带上
  GET /page.html HTTP/1.1
  If-None-Match: "686897696a7c876b7e"

  // 如果资源未变化
  HTTP/1.1 304 Not Modified
  // 如果资源已变化
  HTTP/1.1 200 OK
  ETag: "新的ETag值"

Last-Modified / If-Modified-Since：
  // 第一次响应
  HTTP/1.1 200 OK
  Last-Modified: Wed, 21 Oct 2025 07:28:00 GMT

  // 再次请求
  GET /page.html HTTP/1.1
  If-Modified-Since: Wed, 21 Oct 2025 07:28:00 GMT

  // 未修改则返回 304

缓存决策流程：
  ┌─────────────────────────────────────────────┐
  │  浏览器请求资源                              │
  │     ↓                                       │
  │  本地是否有缓存？                            │
  │  ├─ 否 → 向服务端请求                       │
  │  └─ 是 → 检查 Cache-Control                 │
  │         ├─ no-store → 不使用缓存            │
  │         ├─ max-age 未过期 → 使用缓存        │
  │         └─ 已过期 → 向服务端验证            │
  │                ├─ ETag (If-None-Match)       │
  │                └─ Last-Modified (If-Modified)│
  │                   ↓                         │
  │                304 → 使用缓存               │
  │                200 → 使用新响应             │
  └─────────────────────────────────────────────┘

Expires vs Cache-Control：
  // HTTP/1.0
  Expires: Thu, 01 Dec 2025 16:00:00 GMT
  // 绝对时间，受客户端时钟影响

  // HTTP/1.1（优先）
  Cache-Control: max-age=3600
  // 相对时间，更可靠

实际最佳实践：
  HTML：Cache-Control: no-cache（每次验证）
  CSS/JS：Cache-Control: max-age=31536000 + 文件名哈希
  图片：Cache-Control: max-age=2592000 + ETag
  API：Cache-Control: no-store（不缓存）
```


## HTTP/1.1 其他重要特性


```
Host 头部（必须）：
  GET /index.html HTTP/1.1
  Host: www.example.com
  // 一个IP可以托管多个域名（虚拟主机）
  // HTTP/1.0 不强制要求 Host 头部

状态码扩展：
  100 Continue：客户端先发头部，服务端确认后再发body
  // 大文件上传前先确认服务端是否接受
  Expect: 100-continue

  206 Partial Content：范围请求
  Range: bytes=0-1023
  // 支持断点续传

  301 Moved Permanently：永久重定向
  302 Found：临时重定向
  304 Not Modified：缓存命中

范围请求（Range Request）：
  GET /large-file.zip HTTP/1.1
  Range: bytes=0-1023

  HTTP/1.1 206 Partial Content
  Content-Range: bytes 0-1023/10485760
  Content-Length: 1024

  // 多范围请求
  Range: bytes=0-99, 200-299

  // 用于断点续传、视频拖动播放

Cookie 支持：
  // 服务端设置
  Set-Cookie: session=abc123; Path=/; HttpOnly; Secure

  // 客户端自动携带
  Cookie: session=abc123

请求方法：
  GET     - 获取资源
  POST    - 提交数据
  PUT     - 完整替换资源
  DELETE  - 删除资源
  HEAD    - 只获取头部
  OPTIONS - 获取支持的方法
  PATCH   - 部分更新资源

HTTP/1.1 局限性：
  ┌─────────────────────────────────────────────┐
  │  1. 队头阻塞：TCP层面上的队头阻塞           │
  │     一个TCP包丢失，后续包都要等待重传        │
  │                                             │
  │  2. 头部冗余：每次请求都携带相同头部        │
  │     Cookie、User-Agent 等重复传输           │
  │                                             │
  │  3. 明文传输：安全性不足                    │
  │     需要 TLS（HTTPS）来加密                 │
  │                                             │
  │  4. 并发限制：浏览器限制同域6个连接         │
  │     资源多时需要排队等待                    │
  └─────────────────────────────────────────────┘
```


> **Note:** HTTP/1.1 的核心改进是持久连接和管线化，但队头阻塞问题限制了性能。缓存机制（ETag/Cache-Control）是性能优化的关键手段。实际项目中合理设置缓存策略可以大幅提升用户体验。
