# 03-数据仓库与数据湖

## 1. 数据仓库概念

数据仓库（Data Warehouse）是一个 **面向主题的、集成的、非易失的、随时间变化的** 数据集合，用于支持管理决策。

**Inmon方法论（自顶向下）**：
- 先建设企业级数据仓库（EDW），再构建数据集市
- 强调数据的一致性和规范性
- 采用第三范式（3NF）建模
- 适合大型企业，强调全局视角

**Kimball方法论（自底向上）**：
- 先构建业务主题的数据集市，再集成为数据仓库
- 强调业务驱动和快速交付
- 采用维度建模（星型模型/雪花模型）
- 适合快速迭代和业务敏捷

两种方法论的核心区别在于 **建模方式** 和 **实施路径**，现代实践中常采用两者结合的方式。

---

## 2. ETL（抽取、转换、加载）

ETL 是数据仓库建设中的核心数据集成流程：

- **Extract（抽取）**：
  - 从源系统（数据库、日志、API、文件等）抽取数据
  - 全量抽取 vs 增量抽取（基于时间戳、CDC、触发器）
  - 需要考虑源系统对业务的影响

- **Transform（转换）**：
  - 数据清洗：去重、空值处理、异常值处理
  - 数据标准化：统一编码、格式、单位
  - 数据转换：字段映射、计算衍生字段
  - 数据聚合：汇总计算
  - 业务规则应用：数据校验、数据脱敏

- **Load（加载）**：
  - 全量加载：每次覆盖目标表
  - 增量加载：仅加载新增或变更的数据
  - 需要考虑加载性能和对目标系统的影响

**ELT 与 ETL 的区别**：
- ELT（Extract-Load-Transform）：先加载到目标系统，再利用目标系统（如数据湖、MPP数据库）的计算能力进行转换
- 现代数据平台越来越多采用ELT模式，充分利用云基础设施的弹性计算能力

## 3. 星型模型与雪花模型

### 3.1 星型模型（Star Schema）

- 一个**事实表**居中，多个**维度表**围绕事实表
- 维度表直接与事实表相连，维度表之间不连接
- 维度表通常采用反范式设计，数据冗余较多
- **优点**：查询简单、性能好、易于理解
- **缺点**：数据冗余、可能存在更新异常

### 3.2 雪花模型（Snowflake Schema）

- 维度表进一步规范化，拆分为子维度表
- 维度表之间形成层次关系，整体形状类似雪花
- **优点**：数据冗余少、节省存储空间
- **缺点**：查询需要更多JOIN、性能较差、模型复杂

**实际应用中，星型模型使用更广泛**，因为存储成本已不再是主要瓶颈，而查询性能和易用性更重要。

## 4. 事实表与维度表

### 4.1 事实表（Fact Table）

事实表存储业务过程的 **度量（Measure）** 数据：

- **事务事实表**：记录单次业务事件（如订单、交易），粒度最细
- **周期快照事实表**：定期对某指标进行快照（如每日账户余额）
- **累积快照事实表**：记录业务过程各里程碑的时间（如订单从下单到完成的各阶段）
- **无事实事实表**：仅记录事件的发生（如页面访问日志），度量为1或无

事实表通常包含：
- 外键（关联各维度表）
- 度量值（数值型，如金额、数量）
- 通常只进行INSERT操作

### 4.2 维度表（Dimension Table）

维度表存储描述性属性，用于分析的角度：

- 时间维度：年、季度、月、周、日
- 地理维度：国家、省、城市
- 产品维度：品类、品牌、规格
- 用户维度：年龄段、性别、等级

**缓慢变化维度（SCD）**：
- Type 0：保留原始值，不做任何更新
- Type 1：直接覆盖旧值
- Type 2：添加新行，记录历史（最常用，通过生效日期和失效日期标识）
- Type 3：添加新列记录旧值

## 5. 数据仓库分层

经典的数据仓库分层架构：

```
ADS (Application Data Store)  ←  应用层/数据集市
  ↑
DWS (Data Warehouse Summary)  ←  汇总层/数据服务层
  ↑
DWD (Data Warehouse Detail)   ←  明细层
  ↑
DIM (Dimension)               ←  维度层
  ↑
ODS (Operational Data Store)  ←  操作数据层/贴源层
  ↑
Source Systems                ←  源系统
```

- **ODS层**：原始数据的镜像，基本不做加工，保留原始粒度，支持数据回溯
- **DIM层**：公共维度数据，维护各种维度表和维表映射关系
- **DWD层**：对ODS数据进行清洗、标准化、关联维度，保留最细粒度的明细事实数据
- **DWS层**：按主题对DWD数据进行轻度汇总，提供通用的汇总指标
- **ADS层**：面向具体应用的数据集市，直接支撑报表和业务查询

**设计原则**：
- 各层职责清晰，数据从业务明细逐层汇总到应用层
- 上层数据从下层数据加工而来，不直接跨层引用
- 公共逻辑下沉到DWD/DWS，避免重复计算

## 6. OLAP vs OLTP

| 维度 | OLTP（联机事务处理） | OLAP（联机分析处理） |
|------|---------------------|---------------------|
| 目的 | 支持日常业务操作 | 支持数据分析和决策 |
| 数据来源 | 业务系统实时产生 | 数据仓库历史数据 |
| 操作类型 | 增、删、改、查（CRUD） | 复杂查询、聚合、多维分析 |
| 数据量 | 单次操作少量记录 | 操作大量记录 |
| 响应时间 | 毫秒级 | 秒到分钟级 |
| 用户 | 业务操作人员 | 数据分析师、管理层 |
| 数据模型 | 第三范式（3NF） | 星型/雪花模型 |
| 代表系统 | MySQL、PostgreSQL、Oracle | ClickHouse、Doris、Kylin、Greenplum |

**OLAP的三种实现方式**：
- **MOLAP**（多维OLAP）：预计算多维立方体，查询快但预计算代价高
- **ROLAP**（关系型OLAP）：基于关系数据库，灵活性好但查询性能依赖索引
- **HOLAP**（混合OLAP）：结合两者优点，细节数据用ROLAP，汇总数据用MOLAP

## 7. 数据湖概念与架构

数据湖（Data Lake）是一个集中存储系统，可存储任意规模的 **结构化、半结构化和非结构化** 数据：

- **核心特点**：
  - 存储原始数据，不强制Schema
  - 支持多种数据格式（JSON、CSV、Parquet、图片、视频、日志等）
  - 存储与计算分离
  - 支持多种计算引擎（批处理、流处理、SQL、ML）

- **典型架构**：
  - 存储层：对象存储（如S3、HDFS）
  - 格式层：开放格式（Parquet、ORC、Avro）
  - 计算层：Spark、Flink、Presto、Hive
  - 治理层：元数据管理、数据目录、访问控制

- **常见问题**：
  - 数据沼泽（Data Swamp）：缺乏治理导致数据湖变成混乱的垃圾场
  - 数据质量问题：原始数据未经过充分清洗
  - 缺乏数据发现能力

---

## 8. 数据湖与数据仓库的区别

| 维度 | 数据仓库 | 数据湖 |
|------|----------|--------|
| 数据类型 | 主要是结构化数据 | 结构化、半结构化、非结构化 |
| Schema | 写时Schema（Schema-on-Write） | 读时Schema（Schema-on-Read） |
| 数据质量 | 经过清洗和转换 | 保留原始数据 |
| 用户 | 数据分析师、BI工程师 | 数据科学家、数据工程师 |
| 存储成本 | 较高（专用存储引擎） | 较低（对象存储） |
| 灵活性 | 变更成本高 | 灵活适应变化 |
| 查询性能 | 经过优化，性能好 | 依赖计算引擎和格式 |

现代趋势是将两者融合，即 **Lakehouse架构**。

---

## 9. Lakehouse架构

Lakehouse是在数据湖之上构建数据仓库能力的新一代架构：

- 核心思想：在低成本对象存储之上，通过 **表格式（Table Format）** 实现ACID事务、Schema演化、时间旅行等数据仓库特性

**三大开源表格式**：

- **Delta Lake**（Databricks）：
  - 基于Parquet文件的事务日志层
  - 支持ACID事务、Schema演化、时间旅行
  - 与Spark生态深度集成

- **Apache Iceberg**（Netflix开源）：
  - 面向超大规模分析场景
  - 支持隐藏分区、分区演化、Schema演化
  - 引擎无关，不绑定特定计算引擎

- **Apache Hudi**（Uber开源）：
  - 擅长增量数据处理（Upsert、Incremental Pull）
  - 支持Copy-on-Write和Merge-on-Read两种存储模式
  - 适合近实时数据分析场景

**Lakehouse的优势**：
- 存储成本低（基于对象存储）
- 数据无需搬迁（统一存储）
- 同时支持BI和ML工作负载
- 开放格式避免厂商锁定

---

## 10. 列式存储格式

列式存储是大数据分析场景的主流存储格式：

**Apache Parquet**：
- 开源的列式存储格式
- 同一列的数据物理上存储在一起
- 支持高效的压缩和编码（Dictionary、RLE、Delta）
- 支持谓词下推，只读取需要的列
- 支持嵌套数据结构（Dremel论文中的record shredding）
- 是Spark、Hive、Presto等引擎的首选格式

**Apache ORC**：
- 优化的列式存储格式（Optimized Row Columnar）
- Hive生态中广泛使用
- 内置索引（行组索引、布隆过滤器）
- 支持ACID操作（Hive事务表）

**行式 vs 列式对比**：
- 行式：适合OLTP（需要读取完整行），写入性能好
- 列式：适合OLAP（只需读取部分列），压缩率高，查询性能好

## 11. 数据治理基础

数据治理是确保数据资产可用、可信、安全的管理体系：

### 11.1 元数据管理

- **技术元数据**：表结构、字段类型、分区信息、数据血缘、存储位置
- **业务元数据**：业务含义、指标定义、数据Owner、更新频率
- **操作元数据**：任务运行状态、数据量统计、数据质量指标
- 常用工具：Apache Atlas、DataHub、Amundsen

### 11.2 数据质量

数据质量维度：
- **完整性**：数据是否缺失
- **准确性**：数据是否正确
- **一致性**：不同系统中的同一数据是否一致
- **及时性**：数据是否按时更新
- **唯一性**：是否存在重复

常用手段：数据校验规则、异常检测、数据血缘追踪

### 11.3 数据血缘

- 描述数据从产生到消费的完整流转路径
- 支持影响分析（修改上游对下游的影响）
- 支持溯源分析（数据出问题时定位根因）
- 通常通过SQL解析、日志采集等方式自动构建

**数据治理不是技术问题，而是组织和流程问题**，需要建立明确的数据标准、角色职责和管理流程。
