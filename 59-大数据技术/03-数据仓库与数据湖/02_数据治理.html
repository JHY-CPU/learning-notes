数据治理

大数据数据治理数据管理

数据治理是确保数据质量、安全性和可用性的管理体系，是企业数据资产化的核心。

数据质量管理

数据质量是数据治理的核心目标，直接影响数据分析和决策的可靠性。

数据质量六大维度： ┌──────────────┬──────────────────────────────────────┐ │ 准确性 │ 数据值与真实情况的一致程度 │ │ 完整性 │ 数据记录和字段的缺失程度 │ │ 一致性 │ 不同系统间数据的矛盾情况 │ │ 时效性 │ 数据是否及时更新 │ │ 唯一性 │ 数据是否存在重复 │ │ 有效性 │ 数据是否符合业务规则和格式要求 │ └──────────────┴──────────────────────────────────────┘ 数据质量治理流程： 1. 质量规则定义 - 空值率检查：字段空值比例不超过阈值 - 值域检查：数值在合理范围内 - 格式检查：符合正则表达式模式 - 唯一性检查：主键/业务键无重复 - 关联检查：外键引用完整性 2. 质量检测 - 离线检测：定时批量扫描（每日/每周） - 实时检测：数据流中实时校验 - 采样检测：对大规模数据抽样验证 3. 质量改进 - 数据清洗：修正、去重、补全 - 源头治理：修复数据录入流程 - 持续监控：建立质量评分看板

元数据管理

元数据是"关于数据的数据"，是数据治理的基础设施。

元数据分类： ┌────────────────────────────────────────────────────────┐ │ 技术元数据 │ │ ├── 表名、字段名、字段类型 │ │ ├── 数据量、存储大小、分区信息 │ │ ├── 任务调度信息（ETL 依赖关系） │ │ └── 数据库连接信息、API 接口定义 │ │ │ │ 业务元数据 │ │ ├── 业务指标定义和计算口径 │ │ ├── 数据标准和编码规范 │ │ ├── 业务规则和数据映射关系 │ │ └── 数据 owner 和使用方 │ │ │ │ 管理元数据 │ │ ├── 数据权限和访问控制 │ │ ├── 数据安全等级分类 │ │ ├── 合规要求（GDPR/个人信息保护） │ │ └── 数据保留策略和生命周期 │ └────────────────────────────────────────────────────────┘ 元数据管理工具： - Apache Atlas：开源元数据管理平台 - DataHub（LinkedIn）：现代化数据目录 - Amundsen（Lyft）：数据发现和治理 - Apache Hive Metastore：Hive 表元数据

数据目录（Data Catalog）

数据目录是企业数据资产的索引系统，帮助用户发现、理解和使用数据。

数据目录核心能力： ┌─────────────────────────────────────────────────┐ │ 1. 数据发现（Discovery） │ │ ├── 全文搜索：按名称、描述、标签搜索 │ │ ├── 分类浏览：按业务域、数据源分类 │ │ ├── 智能推荐：基于使用习惯推荐相关数据 │ │ └── 数据预览：快速查看数据样例 │ │ │ │ 2. 数据理解（Understanding） │ │ ├── Schema 信息：字段类型、含义 │ │ ├── 数据画像：分布、空值率、统计指标 │ │ ├── 使用说明：业务含义、使用示例 │ │ └── 质量评分：数据质量评估结果 │ │ │ │ 3. 数据协作（Collaboration） │ │ ├── 用户评价和评论 │ │ ├── 标签和书签 │ │ ├── 数据问题反馈 │ │ └── 使用统计和热度排名 │ └─────────────────────────────────────────────────┘

数据血缘（Data Lineage）

数据血缘追踪数据从源头到消费端的完整流转路径，是影响分析和问题追溯的关键工具。

数据血缘类型： ┌──────────────────────────────────────────────────────┐ │ 表级血缘（Table-level Lineage） │ │ 原始表 → ETL → 中间表 → 聚合 → 报表 │ │ 例：ods_orders → dw_orders → dm_sales_summary │ │ │ │ 字段级血缘（Column-level Lineage） │ │ 表A.field1 + 表B.field2 → 表C.total_amount │ │ 可精确定位字段问题的上游来源 │ └──────────────────────────────────────────────────────┘ 血缘采集方式： 1. 静态解析 - 解析 SQL 语句（CREATE TABLE AS SELECT） - 解析 ETL 配置文件 - 解析数据管道定义（Airflow DAG） 2. 运行时采集 - Hook 机制拦截计算引擎（Spark/Hive Hook） - 数据库触发器和审计日志 - CDC（变更数据捕获）事件追踪 血缘应用场景： - 影响分析：表结构变更前评估下游影响 - 问题追溯：报表数据异常时定位上游故障 - 合规审计：追踪敏感数据的流转路径 - 依赖治理：识别和清理僵尸数据链路
字段级血缘比表级血缘更精确，但采集成本也更高，通常需要结合 SQL 解析和运行时采集。

数据网格（Data Mesh）

数据网格是一种去中心化的数据架构范式，强调数据的领域所有权和自治。

数据网格四大原则： ┌─────────────────────────────────────────────────────┐ │ 1. 域所有权（Domain Ownership） │ │ 业务域团队负责自己的数据产品 │ │ 数据生产者同时也是数据治理者 │ │ │ │ 2. 数据即产品（Data as a Product） │ │ 每个数据集被视为一个可独立消费的产品 │ │ 关注：可用性、文档、SLA、质量保证 │ │ 产品元数据：Owner、消费者、质量指标 │ │ │ │ 3. 自服务平台（Self-serve Platform） │ │ 提供数据基础设施即服务 │ │ 域团队可自主创建、发布、监控数据产品 │ │ 降低数据工程团队的瓶颈 │ │ │ │ 4. 联邦计算治理（Federated Computational Governance）│ │ 全局策略 + 域自治 │ │ 自动化的策略执行（安全、质量、互操作性） │ │ 标准化接口和数据契约 │ └─────────────────────────────────────────────────────┘ 数据网格 vs 传统架构： ┌───────────────┬──────────────────┬──────────────────┐ │ │ 数据湖/数仓 │ 数据网格 │ ├───────────────┼──────────────────┼──────────────────┤ │ 架构模式 │ 集中式 │ 去中心化 │ │ 数据所有权 │ 平台团队 │ 业务域团队 │ │ 扩展性 │ 平台团队瓶颈 │ 各域独立扩展 │ │ 治理模式 │ 自上而下 │ 联邦式自治 │ │ 适用规模 │ 中小规模 │ 大型组织 │ └───────────────┴──────────────────┴──────────────────┘
数据网格不是要取代数据湖或数据仓库，而是一种组织和架构层面的补充。它适合数据团队和业务域都比较大的企业。
