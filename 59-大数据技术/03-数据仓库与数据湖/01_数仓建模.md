# 数仓建模


## 数仓建模


数据仓库建模ETL


数据仓库建模方法论：星型/雪花模式、事实/维度表、ETL与ELT流程。


## 星型模式 (Star Schema)


以事实表为中心，维度表围绕四周，形成星型结构。是数据仓库最常用的建模方式。


```
星型模式结构：
                    ┌──────────┐
                    │ 时间维度  │
                    │ dim_time │
                    └────┬─────┘
                         │
┌──────────┐    ┌────────┴────────┐    ┌──────────┐
│ 产品维度  ├────┤   销售事实表    ├────┤ 地区维度  │
│dim_product│    │  fact_sales    │    │dim_region │
└──────────┘    └────────┬────────┘    └──────────┘
                         │
                    ┌────┴─────┐
                    │ 客户维度  │
                    │dim_customer│
                    └──────────┘

事实表 (Fact Table)：
- 存储业务度量/指标数据（如销售额、数量）
- 包含外键引用维度表
- 数据量大，持续增长
- 粒度(Grain)：每条记录的详细程度（如：每笔订单）

维度表 (Dimension Table)：
- 存储描述性属性（如产品名称、地区、时间）
- 数据量相对较小
- 包含主键供事实表引用
- 支持维度的层次结构（如：年→季→月→日）
```


## 雪花模式 (Snowflake Schema)


```
雪花模式：维度表进一步规范化
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 年表     │    │ 产品表   │    │ 国家表   │
│ dim_year │    │dim_product│    │dim_country│
└────┬─────┘    └────┬─────┘    └────┬─────┘
     │               │               │
┌────┴─────┐    ┌────┴─────┐    ┌────┴─────┐
│ 季度表   │    │ 类别表   │    │ 省份表   │
│dim_quarter│   │dim_category│   │dim_province│
└────┬─────┘    └──────────┘    └────┬─────┘
     │                              │
┌────┴─────┐                    ┌────┴─────┐
│ 月表     │                    │ 城市表   │
│dim_month │                    │dim_city  │
└──────────┘                    └──────────┘

星型 vs 雪花：
┌──────────┬────────────────┬────────────────┐
│ 对比维度  │ 星型模式        │ 雪花模式        │
├──────────┼────────────────┼────────────────┤
│ 冗余度   │ 高（反规范化）   │ 低（规范化）     │
│ 查询性能  │ 快（JOIN少）    │ 慢（JOIN多）    │
│ 存储空间  │ 大              │ 小              │
│ 维护成本  │ 低              │ 高              │
│ 适用场景  │ 数据仓库/BI     │ 数据规范化存储    │
└──────────┴────────────────┴────────────────┘
```


> **Note:** 实际项目中，星型模式更常用。数据仓库追求查询性能而非存储节省。


## ETL vs ELT


```
ETL (Extract-Transform-Load)：
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 数据源   │ →  │ Transform │ →  │ 数据仓库  │
│ Extract  │    │ (在ETL    │    │  Load    │
│ 抽取     │    │  工具中)  │    │ 加载     │
└──────────┘    └──────────┘    └──────────┘
- 传统方式，适合中小数据量
- 转换在中间服务器完成
- 工具：Informatica, DataStage, SSIS

ELT (Extract-Load-Transform)：
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 数据源   │ →  │  数据湖/  │ →  │ 转换处理  │
│ Extract  │    │  Staging  │    │ Transform│
│ 抽取     │    │   Load    │    │ (在目标)  │
└──────────┘    └──────────┘    └──────────┘
- 现代方式，适合大数据量
- 原始数据先加载到数据湖/仓库
- 利用数据仓库的计算能力进行转换
- 工具：dbt, Spark SQL, Hive

数据分层（数仓分层）：
ODS (操作数据层) → DWD (明细数据层) → DWS (汇总数据层) → ADS (应用数据层)
原始数据         → 清洗、标准化      → 聚合汇总        → 面向应用
```


## 数据湖 (Data Lake)


```
数据仓库 vs 数据湖 vs 湖仓一体：

┌──────────┬────────────────┬────────────────┬────────────────┐
│ 特性      │ 数据仓库        │ 数据湖          │ 湖仓一体        │
├──────────┼────────────────┼────────────────┼────────────────┤
│ 数据类型  │ 结构化          │ 结构/半/非结构化 │ 全类型          │
│ 存储格式  │ 列式(Parquet)   │ 原始格式+列式    │ Delta/Iceberg  │
│ Schema   │ 写时Schema      │ 读时Schema      │ 两者的结合       │
│ 质量     │ 高（经过清洗）   │ 原始（可能脏）   │ 可控             │
│ 查询性能  │ 高（优化过）    │ 低（未优化）     │ 高               │
│ 成本     │ 高              │ 低（对象存储）   │ 中               │
│ 典型工具  │ Hive/Redshift  │ S3/HDFS        │ Delta Lake/Iceberg│
└──────────┴────────────────┴────────────────┴────────────────┘

湖仓一体 (Lakehouse)：
- 结合数据湖的灵活性和数据仓库的管理能力
- 核心技术：Delta Lake / Apache Iceberg / Apache Hudi
- 支持ACID事务、Schema演化、时间旅行
- 底层用对象存储（S3/HDFS），上层提供SQL查询

常见数据湖存储格式：
- Parquet：列式存储，压缩率高，适合分析
- ORC：Hive优化的列式格式
- Avro：行式存储，适合写入密集
- JSON/CSV：文本格式，可读性好但性能差
```


> **Note:** 现代数据架构趋势是湖仓一体：用Delta Lake/Iceberg等技术在数据湖之上提供数据仓库级别的管理能力。
