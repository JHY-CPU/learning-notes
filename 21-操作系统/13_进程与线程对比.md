# 13_进程与线程对比

## 核心概念

- **进程**：资源分配的基本单位，拥有独立的地址空间和资源
- **线程（Thread）**：CPU调度的基本单位，是进程中的一个执行流
- **引入线程的目的**：减少进程切换开销，提高并发效率
- **关键区分**：
  - 进程 = 资源分配单位 + 调度单位（传统）
  - 进程 = 资源分配单位（引入线程后）
  - 线程 = CPU调度单位（引入线程后）
- **同一进程中的线程共享**：地址空间、全局变量、打开的文件、信号处理方式
- **每个线程私有**：线程ID、寄存器组、栈（用于函数调用和局部变量）、状态字

## 原理分析

### 进程与线程的对比表

| 对比项 | 进程 | 线程 |
|-------|------|------|
| 调度单位 | 传统OS中是调度单位 | 引入线程后是调度单位 |
| 资源分配 | 拥有独立资源 | 不拥有资源，共享所属进程的资源 |
| 切换开销 | 大（切换地址空间/页表） | 小（同一地址空间内切换） |
| 并发性 | 进程间并发 | 同一进程内多个线程也可并发 |
| 地址空间 | 独立的虚拟地址空间 | 共享所属进程的地址空间 |
| 通信方式 | 需要IPC（管道/消息/共享内存） | 可直接读写共享变量 |
| 健壮性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程崩溃 |
| 创建开销 | 大（需分配地址空间等） | 小（共享进程资源） |

### 线程带来的优势

以Web服务器为例（多线程 vs 多进程）：

**多进程模型**：
- 每个请求创建一个进程 → 创建/销毁开销大
- 进程间通信困难 → 需要IPC机制

**多线程模型**：
- 每个请求创建一个线程 → 创建开销小
- 共享地址空间 → 直接共享缓冲区数据

$$\text{进程切换耗时} \approx 1000 \sim 数千\mu s$$
$$\text{线程切换耗时} \approx 数十\mu s$$

## 直观理解

- **进程** = 一家公司：有自己的办公室（地址空间）、银行账户（资源）、营业执照（PCB）
- **线程** = 公司里的员工：共享办公室和账户，但每人有自己的工位（栈）和工作进度（寄存器）
- **进程间通信** = 公司之间要签合同、发传真（复杂、开销大）
- **线程间通信** = 同一办公室的同事直接喊一声就行（简单、开销小）
- **进程崩溃** = 公司倒闭，不影响其他公司
- **线程崩溃** = 一个员工搞砸了，整个公司可能都受影响

## 知识关联

- **线程的实现方式**详见第14个知识点（用户级/内核级/组合）
- **线程模型**详见第16、17个知识点
- **线程间通信**直接用共享变量（但需要同步机制），关联信号量和PV操作
- **现代OS（Linux、Windows）**：进程是资源分配单位，线程是调度单位
- **Java线程**是映射到内核线程的（一对一模型）
- **408考法**：
  - 选择题：进程和线程哪个是资源分配单位/调度单位
  - 简答题：引入线程的好处
  - 综合题：比较进程切换和线程切换的开销差异
- **常见陷阱**：
  - 线程**没有**独立的地址空间
  - 线程切换**不需要**切换页表（同一进程内）
  - 不是所有OS都有线程（早期UNIX没有用户可见的线程）
