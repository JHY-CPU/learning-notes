# 17_多对一一对一多对多模型对比

## 核心概念

- 本节是对第16个知识点的**专题深化对比**，重点厘清三种多线程模型在408考试中的核心区别
- **核心判断标准**：能否利用多核？阻塞是否牵连？切换开销如何？

## 原理分析

### 决策树：如何判断应使用哪种模型

```
需要利用多核CPU？
  ├── 是 → 一对一 或 多对多
  │        ├── 实现简单 → 一对一
  │        └── 需要大量线程且节省内核资源 → 多对多
  └── 否 → 多对一 或 用户级线程
           └── 切换效率优先 → 多对一
```

### 一对一模型详解

**映射关系**：每个ULT（用户级线程）绑定一个KLT（内核级线程）

- 创建ULT时，系统调用创建对应的KLT
- 线程调度完全由OS内核负责
- **Linux pthread** 默认一对一

**性能分析**：
$$T_{\text{切换}} = T_{\text{用户态→内核态}} + T_{\text{内核调度}} + T_{\text{内核态→用户态}}$$

典型值：数十微秒（比用户级切换慢5-10倍，但比进程切换快很多）

### 多对一模型详解

**映射关系**：N个ULT共享1个KLT

- 线程库在用户态维护TCB，OS只看到1个进程
- 线程切换 = 用户态上下文保存/恢复（不陷入内核）

**致命缺陷**：
```
线程1执行read()系统调用 → 陷入内核 → OS发现只有1个KLT → 整个进程阻塞
→ 线程2、线程3即使就绪也无法运行（因为整个KLT被阻塞了）
```

### 多对多模型详解

**映射关系**：M个ULT映射到N个KLT（M ≥ N）

- 线程库负责ULT到KLT的动态映射
- 同一KLT上的ULT之间切换在用户态完成
- 不同KLT上的ULT切换需要内核参与

**优势**：M个线程只需N个内核线程（N远小于M），节省内核资源，同时不丧失并发能力

### 综合对比矩阵

| 特性 | 多对一 | 一对一 | 多对多 |
|-----|-------|-------|-------|
| ULT阻塞 | 整个进程阻塞 | 仅该线程阻塞 | 仅该线程阻塞 |
| 多核并行 | 不支持 | 支持 | 支持 |
| 内核资源 | 极少（1个KLT） | 大（每个ULT一个KLT） | 中等（M个ULT用N个KLT） |
| 用户线程数 | 无限制 | 受OS限制 | 基本无限制 |
| 切换速度 | 最快 | 最慢 | 中等 |
| 实现难度 | 低 | 低 | 高 |
| 代表系统 | Green Thread | Linux, Windows | Solaris旧版 |

## 直观理解

用**餐厅**比喻：
- **多对一**：一家餐厅只有一个服务员（KLT），多个顾客（ULT）轮流服务。一个顾客要求去厨房催菜（阻塞），其他顾客都得等
- **一对一**：每个顾客配一个专属服务员。灵活但服务员数量有限（内核资源有限）
- **多对多**：餐厅有若干服务员，顾客比服务员多。服务员A可以同时照看3号和5号桌，灵活调度

## 知识关联

- **一对一模型**关联Linux线程实现（NPTL），是408实际考查的主流
- **内核级线程的优点**（第15个知识点）解释了一对一成为主流的原因
- **用户级线程**（第14个知识点）的问题导致了多对一模型的淘汰
- **408考法**：
  - 选择题：三种模型的对比，判断特定场景应使用哪种
  - 常见陷阱：一对一模型并非"完美"（开销大），但现代硬件使其成为最优选择；多对多模型在理论上最优但实现复杂
