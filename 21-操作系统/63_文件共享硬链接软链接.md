# 63_文件共享硬链接软链接

## 核心概念

- **文件共享**：多个用户或进程需要访问同一个文件。
- **硬链接（Hard Link）**：多个目录项指向同一个 inode（索引节点）。
- **软链接（Soft Link / Symbolic Link）**：一个目录项包含指向另一个文件路径的指针。
- **链接计数（link count）**：inode 中记录指向该 inode 的硬链接数。
- 408 考点：硬链接与软链接的对比，删除行为，适用场景。

## 原理分析

### 硬链接

**工作原理**：

```
目录结构:
  /home/张三/link_a  ──→  inode #1234 ──→ [数据块]
  /home/李四/link_b  ──→  inode #1234 ──→ [数据块]

inode #1234:
  link_count = 2
  owner = 张三
  data_blocks = [块1, 块2, ...]
```

两个目录项指向同一个 inode，link_count 从 1 变为 2。

**创建**：`ln source.txt link.txt`（Linux 命令）

**删除行为**：
- 删除一个硬链接：link_count 减 1。
- link_count > 0：文件仍然存在（通过其他链接可访问）。
- link_count = 0：inode 和数据块被回收（文件真正删除）。

**限制**：
- **不能跨文件系统**：inode 号只在本文件系统内唯一。
- **不能链接目录**：防止形成环路导致死循环（`ln -d` 禁止）。

### 软链接（符号链接）

**工作原理**：

```
目录结构:
  /home/张三/soft_link  ──→  新inode #5678 ──→ ["/home/李四/target.txt"]
  /home/李四/target.txt ──→  inode #1234   ──→ [实际数据]

新inode #5678:
  文件类型: 符号链接
  内容: "/home/李四/target.txt"（目标路径字符串）
```

软链接文件本身是一个小文件，内容是目标文件的路径名。

**创建**：`ln -s target.txt soft_link`

**删除行为**：
- 删除目标文件：软链接仍然存在，但变成**悬挂链接**（dangling link），访问时报错。
- 删除软链接：只删除链接文件本身，不影响目标文件。

**特点**：
- 可以跨文件系统（路径名是通用的）。
- 可以链接目录。
- 每次访问需要额外的路径解析开销（解析链接内容中的路径）。

### 硬链接 vs 软链接对比

| 特性 | 硬链接 | 软链接 |
|------|--------|--------|
| 实现方式 | 多目录项指向同一 inode | 独立文件存储目标路径 |
| 跨文件系统 | 不支持 | 支持 |
| 链接目录 | 不支持 | 支持 |
| 删除目标 | link_count--，其他链接仍有效 | 悬挂链接（dangling） |
| 额外空间 | 仅目录项（16B） | 需要额外 inode + 路径字符串 |
| 访问速度 | 与原文件相同 | 需额外路径解析 |
| inode | 共享 inode | 独立 inode |

### Windows 的快捷方式

Windows 的快捷方式类似于软链接，但实现不同：
- 存储目标路径、图标、参数等信息。
- 由资源管理器解析，不是内核级别实现。
- 目标删除后快捷方式失效。

## 直观理解

- **硬链接** = 同一个人有多个名字（身份证号相同），死了才算真死。
- **软链接** = 纸条上写着"去找张三"，张三搬走了纸条就成了废纸。
- 硬链接是"本体"，软链接是"指向"。

## 知识关联

- 与 61 FCB 和 inode：硬链接共享 inode，软链接有独立 inode。
- 与 60 目录结构：硬链接形成无环图（DAG），需避免环路。
- 与 62 文件保护：硬链接共享权限（同一 inode），软链接用目标文件权限。
- 408 考查：给出场景判断应使用硬链接还是软链接；删除文件后链接的行为。
