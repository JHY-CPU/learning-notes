# 42_死锁检测与解除

## 核心概念

- **死锁检测（Detection）**：允许死锁发生，定期检测并处理
- **检测时机**：①定时检测 ②资源利用率低时 ③进程执行异常慢时
- **单实例资源检测**：资源分配图化简法（看是否有环）
- **多实例资源检测**：类似银行家算法的检测版本
- **死锁解除方法**：
  1. **资源剥夺法**：从某些进程剥夺资源给死锁进程
  2. **进程终止法**：终止一个或多个死锁进程
  3. **进程回退法**：让进程回退到之前的某个检查点

**408核心考点**：
- 检测算法的执行过程
- 解除方法的选择策略
- 检测 vs 避免 vs 预防的区别

## 原理分析

**多实例资源的死锁检测算法**：

与安全性算法类似，但不需要Max，只需要Allocation和Request：

```
Work = Available
Finish[i] = (Allocation[i] == 0) ? true : false

循环：
    找一个进程Pi满足：
        Finish[i] == false  AND  Request[i] <= Work
    如果找到：
        Work = Work + Allocation[i]
        Finish[i] = true
        继续循环
    如果找不到：退出

Finish[i] == false 的进程 → 处于死锁状态
```

**死锁解除的代价分析**：

| 方法 | 策略 | 代价 |
|------|------|------|
| 资源剥夺 | 从某些进程抢资源 | 被剥夺进程可能需要回退 |
| 终止所有死锁进程 | 暴力解决 | 损失最大 |
| 逐个终止 | 按优先级/代价最小的终止 | 需要选择策略 |
| 进程回退 | 回到检查点重新执行 | 需要检查点机制 |

**选择终止进程的策略**（考虑因素）：
- 进程优先级
- 已执行时间（已执行时间短的优先终止，损失小）
- 还需要的资源量
- 已占用资源量
- 是交互式还是批处理进程

## 直观理解

**体检的类比**：
- 死锁预防就像"每天注意饮食锻炼"（预防为主）
- 死锁避免就像"每次吃东西前查热量"（实时检查）
- 死锁检测就像"定期体检"（发生后才发现）
- 死锁解除就像"治疗"（发现后处理）

**检测 vs 避免的区别**：
- 避免：每次分配资源前都检查，从源头防止死锁（保守但安全）
- 检测：允许死锁发生，定期检查是否已经死锁（效率高但有延迟）

## 知识关联

**与资源分配图的关系（第38节）**：
- 单实例资源的死锁检测就是找环路
- 化简法中不可化简的进程即死锁进程

**与银行家算法的关系（第40节）**：
- 检测算法结构类似安全性算法
- 区别：检测用Request（请求向量），安全性用Need（需求矩阵）

**三种死锁处理方式对比**：

| 方面 | 预防 | 避免 | 检测与解除 |
|------|------|------|-----------|
| 思路 | 破坏必要条件 | 动态检查安全状态 | 允许死锁再处理 |
| 资源利用率 | 低 | 中 | 高 |
| 系统开销 | 小 | 中 | 检测小，解除大 |
| 实现难度 | 简单 | 中等 | 复杂 |
| 典型方法 | 有序分配 | 银行家算法 | 资源分配图 |
| 适用场景 | 资源少 | 实时系统 | 通用OS |

**408常见考法**：
- 给出系统状态，用检测算法找死锁进程
- 问"哪种解除方法代价最小"
- 比较预防/避免/检测的优缺点（简答题常考）
- 问"检测算法与安全性算法的区别"
