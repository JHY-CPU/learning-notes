# 33_经典同步问题读者写者

## 核心概念

- **读者-写者问题**：多个进程共享一个文件（数据库），读者只读，写者只写
- **读者之间不互斥**：多个读者可以同时读
- **写者与一切互斥**：写者不能与任何其他读者或写者同时操作
- **读写公平性**：读者优先 vs 写者优先 vs 公平策略
- **reader_count**：记录当前正在读的读者数量

**408核心考点**：
- 读者优先方案的PV操作
- 理解为什么第一个读者要P(wmutex)而后续读者不需要
- 写者饥饿问题

## 原理分析

**读者优先方案**：

```
Semaphore rmutex = 1;    // 保护reader_count的互斥
Semaphore wmutex = 1;    // 写操作的互斥
int reader_count = 0;    // 当前读者数

// 读者进程
Reader:
    P(rmutex);                  // 保护reader_count
    if (reader_count == 0)
        P(wmutex);              // 第一个读者阻止写者
    reader_count++;
    V(rmutex);

    读文件;

    P(rmutex);
    reader_count--;
    if (reader_count == 0)
        V(wmutex);              // 最后一个读者允许写者
    V(rmutex);

// 写者进程
Writer:
    P(wmutex);                  // 申请写（读者存在时会阻塞）
    写文件;
    V(wmutex);
```

**执行过程分析**：

$$
\text{第一个读者到来} \Rightarrow reader\_count: 0 \to 1 \Rightarrow P(wmutex) \Rightarrow \text{写者被阻塞}
$$

$$
\text{后续读者到来} \Rightarrow reader\_count: n \to n+1 \Rightarrow \text{直接读，无需P(wmutex)}
$$

$$
\text{最后一个读者离开} \Rightarrow reader\_count: 1 \to 0 \Rightarrow V(wmutex) \Rightarrow \text{写者可以进入}
$$

**rmutex的作用**：保护reader_count变量本身的互斥访问（不是读操作的互斥）。

**读者优先的问题**：
读者不断到来，reader_count始终不为0，写者永远拿不到wmutex——写者饥饿。

**写者优先方案（改进）**：

```
Semaphore rmutex = 1;
Semaphore wmutex = 1;
Semaphore readTry = 1;     // 新增：读者尝试信号量
int reader_count = 0;
int writer_count = 0;
Semaphore wcount_mutex = 1;

Reader:
    P(readTry);                  // 新读者先检查
    P(rmutex);
    if (reader_count == 0)
        P(wmutex);
    reader_count++;
    V(rmutex);
    V(readTry);

    读文件;

    P(rmutex);
    reader_count--;
    if (reader_count == 0)
        V(wmutex);
    V(rmutex);

Writer:
    P(wcount_mutex);
    if (writer_count == 0)
        P(readTry);              // 第一个写者阻止新读者
    writer_count++;
    V(wcount_mutex);

    P(wmutex);
    写文件;
    V(wmutex);

    P(wcount_mutex);
    writer_count--;
    if (writer_count == 0)
        V(readTry);              // 最后一个写者放行新读者
    V(wcount_mutex);
```

## 直观理解

**图书馆的类比**：
- 读者 = 读者在阅览室看书（多人可同时看）
- 写者 = 管理员在整理书架（整理时不能有人看书）
- reader_count = 阅览室里的读者人数
- 第一个读者进入时要"锁门防管理员"，后续读者直接进来
- 最后一个读者离开时才"开锁"让管理员进来

**rmutex就像计数器的保险柜**：
reader_count是共享变量，多个读者进程要同时修改它，所以需要rmutex保护。但rmutex保护的不是"读文件"，而是"改计数器"。

## 知识关联

**与互斥同步分离（第31节）**：
- wmutex是互斥信号量
- rmutex也是互斥信号量（保护计数器）
- 读者-写者问题主要是互斥问题，没有明确的同步先后关系

**与数据库事务的联系**：
- 读者-写者问题是数据库并发控制的基础
- 读写锁（RWLock）就是这个问题的实际实现
- 共享锁(S锁)对应读者，排他锁(X锁)对应写者

**408常见考法**：
- 给出PV操作序列，问哪个信号量是什么作用
- 分析写者饥饿的原因
- 问"读者之间需要互斥吗"（不需要，但修改count需要）
- 给出执行序列，判断是否会产生竞态条件
