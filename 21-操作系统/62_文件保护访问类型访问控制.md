# 62_文件保护访问类型访问控制

## 核心概念

- **文件保护**：防止文件被有意或无意地非法访问或破坏。
- **访问类型**：读（R）、写（W）、执行（X）、追加（A）、删除（D）、列表目录（L）等。
- **访问控制**：基于用户身份决定其对文件的访问权限。
- 三种主要保护机制：访问控制表（ACL）、用户权限表、口令与密码。
- 408 考点：访问控制的实现方式、权限矩阵、Unix 权限模型。

## 原理分析

### 访问控制表（Access Control List, ACL）

**原理**：每个文件维护一个 ACL，列出所有用户（或用户组）对该文件的访问权限。

```
文件: report.txt
ACL:
  Owner(张三):  rwx
  Group(研发部): r-x
  Other:        r--
```

**Unix/Linux 的实现**：每个文件的 inode 中存储 9 位权限 + owner/group。

```
权限位: rwx r-x r--
        所有者 组  其他
数字:   7    5    4

文件权限: 754
```

**特点**：
- 精确控制每个用户/组的权限。
- ACL 长度可能很大（用户多时）。
- 查找效率取决于 ACL 组织方式（链表、排序数组等）。

### 用户权限表（Capability List）

**原理**：每个用户维护一个权限表，记录该用户可以访问的所有文件及权限。

```
用户: 张三
权限表:
  /home/张三/report.txt:  rwx
  /shared/data.db:       r-x
  /tmp/cache.tmp:        rw-
```

**特点**：
- 从用户角度组织，易于撤销用户的所有权限。
- 但难以回答"谁能访问文件 X"这个问题。
- Linux 中不直接使用此方式。

### 口令保护

**原理**：为文件设置口令，用户必须提供正确口令才能访问。

**优点**：实现简单。
**缺点**：
- 口令易泄露（被共享、被猜中）。
- 每个文件的口令可能不同，用户需要记住多个口令。
- 系统无法区分不同用户的权限（知道口令就有完全访问权）。

### 加密保护

**原理**：用密码对文件内容加密，密钥由文件主保管。没有密钥的用户即使获取了文件也无法解读。

**优点**：安全性高。
**缺点**：加解密开销大，每次读写都需要计算。

### 408 常见：访问权限矩阵

| | 文件A | 文件B | 文件C |
|---|---|---|---|
| 用户1 | R,W | R | - |
| 用户2 | R | R,W,X | R |
| 用户3 | - | R | R,W |

**问题**：当用户和文件很多时，矩阵稀疏且庞大，通常用 ACL 或能力表实现。

### Unix 权限模型详解

权限分为三组，每组 3 位（rwx）：

```
文件类型  所有者权限  组权限  其他权限
d        rwx        r-x     r-x
-        rw-        r--     ---
```

**特殊权限**：
- **SUID (Set User ID)**：执行时以文件所有者权限运行（如 `/usr/bin/passwd`）。
- **SGID (Set Group ID)**：执行时以文件所属组权限运行。
- **Sticky Bit**：目录中只有文件所有者才能删除自己的文件（如 `/tmp`）。

## 直观理解

- **ACL** = 每扇门上贴的"谁可以进来"名单。
- **用户权限表** = 每个人口袋里的"我可以进哪些门"清单。
- **口令** = 门上的密码锁，知道密码就能进。
- **加密** = 门里的东西全用箱子锁着，没钥匙拿到也没用。

## 知识关联

- 与 60 目录结构：目录权限（r=列目录、w=创建/删除、x=进入目录）与文件权限不同。
- 与 63 文件共享：共享时需要协调权限——硬链接共享 inode 权限，软链接用目标文件权限。
- 与 67 空闲管理：保护机制的元数据也需要存储在磁盘上，占用空间。
