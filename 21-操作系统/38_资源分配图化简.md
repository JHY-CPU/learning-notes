# 38_资源分配图化简

## 核心概念

- **资源分配图（Resource Allocation Graph, RAG）**：有向图，表示进程和资源的分配关系
- **进程节点**：用圆圈表示，圆圈内写进程名
- **资源节点**：用方框表示，方框内的圆点表示资源实例
- **分配边（Assignment Edge）**：资源 → 进程（Ri → Pj），表示资源已分配给进程
- **请求边（Request Edge）**：进程 → 资源（Pj → Ri），表示进程申请该资源
- **化简法**：按顺序消除可完成的进程，最终判断是否有死锁

**408核心考点**：
- 根据资源分配图判断是否存在死锁
- 化简过程的正确执行
- 同类资源多个实例时，有环不一定有死锁

## 原理分析

**资源分配图的组成**：

```
进程 P1, P2, ... 用 ○ 表示
资源 R1, R2, ... 用 □ 表示（内部有●表示实例数）

分配边：R ● → P  （资源给了进程）
请求边：P → R ●  （进程申请资源）
```

**化简法步骤**：

1. 找到一个进程Pi，它的所有请求边都能被满足（即它申请的每个资源类型都有空闲实例或分配给它的资源已足够）
2. 假设Pi获得所需资源并执行完毕，释放所有资源
3. 删除Pi的所有边（分配边和请求边）
4. 重复步骤1-3
5. 若所有进程都能消除，则无死锁；若剩余进程无法消除，则这些进程死锁

**化简示例**：

```
R1有2个实例，R2有1个实例

P1: 持有R1的1个实例，申请R2
P2: 持有R2的1个实例，申请R1

R1[●] → P1    (P1持有R1)
P1 → R2[●]    (P1请求R2，但R2给了P2)
R2[●] → P2    (P2持有R2)
P2 → R1[●]    (P2请求R1，但R1给了P1)
```

化简：
- P1请求R2，R2无空闲实例，P1无法执行
- P2请求R1，R1还有1个空闲实例（2个实例给了P1一个），P2可执行！
- P2执行完释放R2 → R2→P1可满足 → P1执行
- **无死锁**

$$
\text{有环路} + \text{每类资源只有1个实例} = \text{死锁}
$$

$$
\text{有环路} + \text{某类资源有多个实例} \neq \text{死锁（可能有，可能没有）}
$$

$$
\text{无环路} = \text{一定无死锁}
$$

**不同情况总结**：

| 资源情况 | 有环路 | 无环路 |
|----------|--------|--------|
| 每类资源1个实例 | 有死锁 | 无死锁 |
| 某类资源多个实例 | 可能有死锁（需化简） | 无死锁 |

## 直观理解

**多米诺骨牌的类比（化简）**：
化简就像推多米诺骨牌——找到一个能"推倒"（执行完毕）的进程，它倒了之后释放资源，可能让其他进程也能倒。如果最后全倒了就没死锁，如果有几个互相撑着谁也倒不了，就死锁了。

**环路的类比**：
每个人抓住前面人的衣角，形成一个环——谁也走不了。但如果某个人（资源有多个实例时）手里有富余，他可以松手让别人走。

## 知识关联

**与死锁定义的关系（第37节）**：
- 资源分配图是判断死锁的可视化工具
- 环路对应循环等待条件

**与死锁检测的关系（第42节）**：
- 资源分配图化简法就是死锁检测的一种方法
- 检测后可决定是否需要解除死锁

**与银行家算法的关系（第40节）**：
- 银行家算法是预防性的（分配前检查）
- 资源分配图化简是检测性的（分配后检查）

**408常见考法**：
- 给出资源分配图，判断是否存在死锁
- 要求执行化简过程，写出化简序列
- 问"有环路是否一定有死锁"
- 多个资源实例时的化简（注意检查空闲实例数）
