# 29_硬件实现方法TestAndSet

## 核心概念

- **关中断**：在进入临界区前关闭中断，退出后再打开，防止进程切换
- **TestAndSet（TS/TSL指令）**：原子地读出旧值并写入true
- **Swap（XCHG指令）**：原子地交换两个变量的值
- **原子性**：硬件保证指令执行不可分割，不会被中断
- **自旋锁（Spinlock）**：基于TS指令实现的忙等待锁

**考试重点**：
- 关中断只适用于单CPU，多CPU中关中断无法阻止其他CPU上的进程
- TS和Swap是原子指令，由硬件保证不可分割
- 硬件方法也**不满足让权等待**（都是忙等待）

## 原理分析

**关中断方法**：

```
关中断;
  临界区;
开中断;
```

- 优点：简单直接
- 缺点：①只适用于单CPU ②关中断时间过长影响系统效率 ③将权限交给用户很危险
- 现代OS中，关中断仅用于内核自身的短临界区

**TestAndSet指令（TSL）**：

```c
// 硬件原子操作
bool TestAndSet(bool *lock) {
    bool old = *lock;    // 读取旧值
    *lock = true;        // 设置为true（上锁）
    return old;          // 返回旧值
}

// 使用方式
bool lock = false;       // 初始未上锁
while(TestAndSet(&lock)); // 忙等待：若old=true说明已被锁
  临界区;
lock = false;            // 释放锁
```

$$
\text{TS操作} = \text{读旧值} + \text{写true} \quad \text{（原子操作，不可分割）}
$$

**执行过程**：
1. 进程P0调用TS：lock原为false，TS返回false，while退出，P0进入临界区，lock变为true
2. 进程P1调用TS：lock为true，TS返回true，while继续自旋
3. P0退出临界区：lock=false，P1下次TS返回false，进入

**Swap指令（XCHG）**：

```c
void Swap(bool *a, bool *b) {
    bool temp = *a;
    *a = *b;
    *b = temp;    // 原子交换
}

// 使用方式
bool lock = false;
bool key = true;
do {
    Swap(&key, &lock);   // 交换key和lock
} while(key);             // 若key=true说明lock原为true，被别人占着
  临界区;
lock = false;
```

**三种硬件方法对比**：

| 方法 | 原理 | 多CPU支持 | 让权等待 |
|------|------|-----------|----------|
| 关中断 | 禁止进程切换 | 不支持 | 不满足 |
| TS | 原子读+写 | 支持 | 不满足 |
| Swap | 原子交换 | 支持 | 不满足 |

## 直观理解

**自助存包柜的类比（TS指令）**：
- 柜子有一个状态锁，你伸手去拿锁的同时就把锁拨到"占用"
- 如果拿到的是"空闲"（旧值false），你就用这个柜子
- 如果拿到的是"占用"（旧值true），你就在旁边等着
- 关键：伸手拿锁和拨锁是同时完成的（原子操作）

**自旋锁就像等红灯**：
你在路口不停看红灯变没变（自旋/忙等待），而不是停好车去休息（阻塞/让权等待）。消耗精力（CPU）但响应快。

## 知识关联

**与软件方法的关系（第28节）**：
- 硬件方法比软件方法更可靠，不依赖进程数量限制
- 但都有忙等待问题

**与信号量的关系（第30节）**：
- 信号量的P/V操作内部通常用TS或Swap实现
- 信号量实现了"让权等待"（进程可以阻塞）

**实际OS中的应用**：
- Linux内核大量使用自旋锁（基于TS），用于短临界区
- 单CPU场景下用关中断+自旋锁
- 多CPU场景下用纯自旋锁

**408常见考法**：
- 问TS操作的原子性体现在哪里（读和写不可分割）
- 问关中断为什么不能用于多CPU
- 给出TS或Swap的代码片段，分析互斥效果
- 判断是否满足让权等待
