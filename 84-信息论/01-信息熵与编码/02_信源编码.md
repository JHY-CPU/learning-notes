# 信源编码 - 信息论笔记


### 1.1 问题背景


信源编码的核心问题是：**如何用最少的比特数表示信源输出的符号序列？**这本质上是数据压缩问题。


```
信源X → [信源编码器] → 压缩后的比特流 → [信源解码器] → 恢复的符号

  目标：最小化平均码长 L = Σ P(xᵢ) l(xᵢ)
  约束：可解码（唯一解码性、即时解码性）
```


### 1.2 香农信源编码定理


$$
对于熵为H(X)的离散无记忆信源：
        • 可以设计编码方案使平均码长 L 满足：H(X) ≤ L < H(X) + 1
        • 不可能存在平均码长小于H(X)的无失真编码方案
$$


该定理给出了无损压缩的理论极限：**信息熵H(X)是无损压缩的下界**。


### 1.3 编码效率


$$
编码效率 η = H(X) / L
        其中L = Σ P(xᵢ) lᵢ 为平均码长
        冗余度 = 1 - η
$$


> **Example:** **示例：**
>
> 信源X = {a, b, c, d}，概率分别为 {0.5, 0.25, 0.125, 0.125}
>
>
> H(X) = -(0.5×(-1) + 0.25×(-2) + 0.125×(-3) + 0.125×(-3)) = 1.75 bits
>
>
> 等长编码需要2比特/符号，效率 = 1.75/2 = 87.5%
>
>
> 最优Huffman编码平均码长 = 1.75 bits/符号，效率 = 100%


### 1.4 编码的分类


| 维度 | 类型 | 说明 |
| --- | --- | --- |
| 失真度 | 无损编码 | 可完全恢复原始数据（Huffman、LZ77） |
|  | 有损编码 | 允许一定失真，追求更高压缩率（JPEG、MP3） |
| 编码方式 | 定长编码 | 所有符号用相同长度的码字 |
|  | 变长编码 | 不同符号用不同长度的码字 |
| 符号粒度 | 逐符号编码 | 每次编码一个符号（Huffman） |
|  | 块编码 | 将多个符号作为整体编码 |


### 2.1 编码的性质


- **非奇异码**
   ：不同的符号对应不同的码字
- **唯一可解码码**
   ：任意码字序列只有唯一的解码方式
- **即时码（前缀码）**
   ：没有任何码字是另一个码字的前缀，可以即时解码无需查看后续码字

前缀码（Prefix Code）也叫即时码，是最常用的一类编码。Huffman编码产生的是最优前缀码。

### 2.2 Kraft不等式


$$
对于码元数为D的前缀码，码字长度 {l₁, l₂, ..., lₙ} 存在的充要条件为：
        Σ D^(-lᵢ) ≤ 1
$$


> **Example:** **验证示例：**
>
> 二进制前缀码（D=2），码长为 {1, 2, 3, 3}
>
>
> 2^(-1) + 2^(-2) + 2^(-3) + 2^(-3) = 0.5 + 0.25 + 0.125 + 0.125 = 1 ≤ 1 ✓
>
>
> 这样的前缀码是存在的。例如：{0, 10, 110, 111}


### 2.3 前缀码的树形表示


```
前缀码 {0, 10, 110, 111} 的二叉树表示：

           (根)
          /    \
         0      1
        (a)    / \
              0   1
             (b) / \
                0   1
              (c) (d)

  叶子节点 = 码字，左分支=0，右分支=1
  前缀码 = 二叉树的叶子节点编码
```


### 3.1 算法思想


Huffman编码是1952年由David Huffman提出的最优前缀码构造方法。核心思想：概率越高的符号使用越短的码字。


### 3.2 算法步骤


1. 将所有符号按概率从大到小排序
2. 取概率最小的两个符号合并为一个新节点（概率相加）
3. 将新节点放回列表，重新排序
4. 重复步骤2-3直到只剩一个节点
5. 从根节点出发，左分支标0，右分支标1，得到各符号的码字


### 3.3 构造示例


> **Example:** **信源：{a, b, c, d, e}，概率：{0.3, 0.25, 0.2, 0.15, 0.1}**
>
> **步骤1：**排序：a(0.3), b(0.25), c(0.2), d(0.15), e(0.1)
>
>
> **步骤2：**合并d和e → de(0.25)
>
>
> **步骤3：**排序：a(0.3), b(0.25), de(0.25), c(0.2)
>
>
> **步骤4：**合并c和de → cde(0.45)
>
>
> **步骤5：**排序：cde(0.45), a(0.3), b(0.25)
>
>
> **步骤6：**合并a和b → ab(0.55)
>
>
> **步骤7：**合并ab和cde → root(1.0)


```
Huffman树结构：

                  root(1.0)
                 /         \
              0/             \1
           ab(0.55)        cde(0.45)
           /    \           /    \
         0/      \1       0/      \1
       a(0.3)  b(0.25)  c(0.2)  de(0.25)
                                /    \
                              0/      \1
                            d(0.15) e(0.1)

  编码结果：
  a: 00      (2 bits)
  b: 01      (2 bits)
  c: 10      (2 bits)
  d: 110     (3 bits)
  e: 111     (3 bits)

  平均码长 L = 0.3×2 + 0.25×2 + 0.2×2 + 0.15×3 + 0.1×3 = 2.25 bits
  信息熵 H(X) = 2.227 bits
  编码效率 η = 2.227/2.25 = 98.98%
```


### 3.4 Huffman编码的性质


- **最优性**
   ：在所有前缀码中，Huffman编码的平均码长最小
- **不唯一性**
   ：交换左右子树、合并等概率节点时顺序不同，会得到不同的Huffman编码，但平均码长相同
- **压缩效率**
   ：平均码长满足 H(X) ≤ L < H(X) + 1
- **局限性**
   ：需要预先知道符号概率，且编码表需与数据一起传输


```
# Python实现Huffman编码
import heapq
from collections import Counter

def huffman_encode(data):
    # 统计频率
    freq = Counter(data)
    heap = [[f, [sym, ""]] for sym, f in freq.items()]
    heapq.heapify(heap)

    # 构建Huffman树
    while len(heap) > 1:
        lo = heapq.heappop(heap)
        hi = heapq.heappop(heap)
        for pair in lo[1:]:
            pair[1] = '0' + pair[1]
        for pair in hi[1:]:
            pair[1] = '1' + pair[1]
        heapq.heappush(heap, [lo[0] + hi[0]] + lo[1:] + hi[1:])

    # 返回编码表
    return {sym: code for sym, code in heap[0][1:]}

codes = huffman_encode("aabbbccccddde")
print(codes)
# {'c': '0', 'b': '10', 'a': '110', 'd': '1110', 'e': '1111'}
```


### 4.1 算法思想


算术编码将整个消息编码为[0, 1)区间内的一个小数，而不是为每个符号分配码字。理论上可以更接近熵极限。


### 4.2 编码过程


> **Example:** **编码消息"abc"，概率：P(a)=0.5, P(b)=0.25, P(c)=0.25**
>
> **步骤1：**初始区间 [0, 1)
>
>
> **步骤2：**编码a：区间划分为 [0, 0.5), [0.5, 0.75), [0.75, 1.0)
>
>
> 选择a → 新区间 [0, 0.5)
>
>
> **步骤3：**编码b：将 [0, 0.5) 按比例划分为 [0, 0.25), [0.25, 0.375), [0.375, 0.5)
>
>
> 选择b → 新区间 [0.25, 0.375)
>
>
> **步骤4：**编码c：将 [0.25, 0.375) 按比例划分为 [0.25, 0.3125), [0.3125, 0.34375), [0.34375, 0.375)
>
>
> 选择c → 新区间 [0.34375, 0.375)
>
>
> 最终区间 [0.34375, 0.375)，取其中任意一个值即可，如 0.35


```
算术编码过程可视化：

  [0              0.25    0.5      0.75    1.0)
  |-- c=0.25 --|-- b=0.25 --|-- a=0.5 ---------|
                    编码'a' ↓
  [0                                          0.5)
  |-- c=0.125 --|-- b=0.125 --|-- a=0.25 -----|
                        编码'b' ↓
  [0.25                              0.375)
  [0.25  0.3125  0.34375  0.375)
  |-- c --|--- b ---|--- a --|
                      编码'c' ↓
  [0.34375                   0.375)
  最终编码：0.35 (在这个区间内任选一个值)
```


### 4.3 与Huffman编码的比较


| 维度 | Huffman编码 | 算术编码 |
| --- | --- | --- |
| 编码单位 | 逐符号（整数比特） | 整个消息（分数比特） |
| 压缩效率 | 每个符号接近1比特浪费 | 理论上更接近熵极限 |
| 计算复杂度 | 较低 | 较高（需要浮点运算） |
| 适用场景 | 符号概率差异大 | 符号概率接近时更优 |
| 实现难度 | 简单 | 较复杂（需处理精度） |


### 5.1 LZ77算法


LZ77（1977年）使用滑动窗口机制，通过引用之前出现过的字符串来实现压缩。


```
LZ77 编码过程：

  输入：ABABABABAB

  滑动窗口（已编码）│ 待编码部分（前瞻缓冲区）
  ──────────────────│─────────────────────
                    │ ABABABABAB
  A                 │ BABABABAB     → (0,0,'A')
  AB                │ ABABABAB      → (0,0,'B')
  ABA               │ BABABAB       → (0,0,'A')    // 未找到匹配
  ABAB              │ ABABAB        → (2,2,'A')    // 找到"AB"匹配，距离2长度2
  ABABAB            │ ABAB          → (4,4,'A')    // 找到"ABAB"匹配
  ...

  输出格式：(距离, 长度, 下一个字符)
```


### 5.2 LZ78算法


LZ78（1978年）不使用滑动窗口，而是维护一个字典，将输入字符串分解为字典中已有短语+新字符。


> **Example:** **LZ78编码示例：输入"AABABABABA"**
>
> | 步骤 | 输入 | 匹配字典索引 | 新短语 | 输出 | 字典更新 |
> | --- | --- | --- | --- | --- | --- |
> | 1 | A | 0（空） | A | (0,A) | 1:A |
> | 2 | AB | 1(A) | AB | (1,B) | 2:AB |
> | 3 | ABA | 2(AB) | ABA | (2,A) | 3:ABA |
> | 4 | BAB | 2(AB) | BAB | (2,B) | 4:BAB |
> | 5 | A | 0（空） | A | (0,A) | 5:A（重复） |


### 5.3 LZ系列的应用


| 算法/工具 | 基于 | 应用 |
| --- | --- | --- |
| gzip / DEFLATE | LZ77 + Huffman | 文件压缩、HTTP压缩 |
| zlib | DEFLATE | PNG图片、网络传输 |
| LZW | LZ78改进 | GIF图片、Unix compress |
| LZMA / xz | LZ77改进 | 7z压缩、Linux内核压缩 |
| Snappy / LZ4 | LZ77变体 | 大数据（Hadoop）、实时压缩 |
| Zstandard (zstd) | LZ77 + Huffman + FSE | Facebook开源，综合性能优秀 |
| Brotli | LZ77 + Huffman | Google，Web内容压缩 |


### 6.1 算法思想


Shannon-Fano编码是1949年由Shannon和Fano独立提出的变长编码方法，采用自顶向下的递归划分策略。


### 6.2 算法步骤


1. 将符号按概率从大到小排序
2. 将符号列表划分为两组，使两组的概率和尽可能相等
3. 第一组所有符号的码字加"0"，第二组加"1"
4. 对每组递归执行步骤2-3，直到每组只有一个符号


### 6.3 构造示例


> **Example:** **信源：{a, b, c, d, e}，概率：{0.35, 0.25, 0.2, 0.15, 0.05}**
>
> **第1次划分：**{a(0.35), b(0.25)} | {c(0.2), d(0.15), e(0.05)}
>
>
> 概率和：0.6 | 0.4，分别加前缀"0"和"1"
>
>
> **第2次划分左组：**{a(0.35)} | {b(0.25)}
>
>
> a: 00, b: 01
>
>
> **第2次划分右组：**{c(0.2)} | {d(0.15), e(0.05)}
>
>
> c: 10
>
>
> **第3次划分：**{d(0.15)} | {e(0.05)}
>
>
> d: 110, e: 111


```
Shannon-Fano编码树：

              root
             /    \
           0/      \1
         {a,b}    {c,d,e}
         /   \     /    \
       0/     \1 0/      \1
       a      b  c      {d,e}
                         / \
                       0/   \1
                       d     e

  编码结果：a:00, b:01, c:10, d:110, e:111
  平均码长 = 0.35×2 + 0.25×2 + 0.2×2 + 0.15×3 + 0.05×3 = 2.2 bits
```


### 6.4 Shannon-Fano vs Huffman


| 维度 | Shannon-Fano | Huffman |
| --- | --- | --- |
| 构建方向 | 自顶向下（分裂） | 自底向上（合并） |
| 最优性 | 不一定最优 | 最优前缀码 |
| 平均码长 | ≥ Huffman编码 | 最小 |
| 实现复杂度 | 较简单 | 稍复杂（需堆） |
| 实用性 | 较少使用 | 广泛使用 |


| 编码方法 | 类型 | 核心思想 | 应用场景 |
| --- | --- | --- | --- |
| Huffman | 统计编码 | 概率高的符号码长短 | JPEG、MP3、DEFLATE |
| 算术编码 | 统计编码 | 整个消息映射为一个数 | JPEG2000、H.265 |
| LZ77/LZ78 | 字典编码 | 引用已出现的字符串 | gzip、LZW、Zstd |
| Shannon-Fano | 统计编码 | 递归概率划分 | 教学为主 |
| 游程编码(RLE) | 统计编码 | 连续重复字符压缩 | BMP、PCX |
| 算术+上下文 | 高级编码 | 条件概率压缩 | PPM、PAQ |

**关键结论：**

- 信息熵H(X)是无损压缩的理论下界
- Huffman编码是最优前缀码，但可能达不到熵极限（每个符号需要整数比特）
- 算术编码可以接近熵极限，但计算复杂度更高
- 实际压缩工具通常组合使用多种方法（如DEFLATE = LZ77 + Huffman）
- 数据的随机性越低（可预测性越高），压缩率越高
