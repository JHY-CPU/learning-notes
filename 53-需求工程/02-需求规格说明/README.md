# 需求规格说明

## 一、SRS（软件需求规格说明书）的结构与内容

### 1.1 SRS 的定义

软件需求规格说明书（Software Requirements Specification, SRS）是一份全面描述系统**必须做什么**和**如何约束**的文档。它是开发团队与客户之间的契约基础，也是后续设计、开发和测试的依据。

### 1.2 SRS 的作用

- 为客户和开发团队提供统一的理解
- 作为系统设计和实现的基础
- 作为测试验收的依据
- 作为合同和协议的组成部分
- 支持项目的估算和规划

### 1.3 典型 SRS 目录结构

1. **引言**
   1.1 目的
   1.2 范围
   1.3 定义、缩写和术语
   1.4 参考文献
   1.5 概述

2. **总体描述**
   2.1 产品前景
   2.2 产品功能概述
   2.3 用户特征
   2.4 约束条件
   2.5 假设与依赖

3. **具体需求**
   3.1 功能需求
   3.2 外部接口需求
   3.3 非功能需求
   3.4 数据需求

4. **附录**

## 二、IEEE 830 标准

### 2.1 标准概述

IEEE Std 830-1998 是编写 SRS 的推荐实践标准（已整合入 ISO/IEC/IEEE 29148:2018）。

### 2.2 SRS 的质量特性

| 特性 | 说明 |
|------|------|
| **正确性** | 需求反映真实意图，无错误描述 |
| **无歧义** | 每个需求只有一种解释 |
| **完整性** | 包含所有必要需求，无遗漏 |
| **一致性** | 需求之间无矛盾 |
| **可验证性** | 每个需求可通过测试验证 |
| **可修改性** | 结构清晰，便于更新 |
| **可追踪性** | 每个需求可追溯到来源 |
| **可理解性** | 文档清晰易懂 |

### 2.3 编写原则

- 使用**可测量的**语言（避免"快速"、"友好"等模糊词）
- 每条需求有**唯一标识号**
- 区分**必须**（shall）和**应该**（should）
- 包含**验收标准**
- 避免设计约束（说"做什么"而非"怎么做"）

## 三、需求描述的形式化 vs 非形式化

### 3.1 非形式化描述

使用**自然语言**描述需求，易于理解和沟通。

**示例（自然语言）**：
> 用户输入用户名和密码后，系统验证凭据，如果正确则进入系统主页，否则显示错误信息。

**优点**：
- 易于理解，无需专门培训
- 所有利益相关者都能参与评审
- 灵活，适应各种需求

**缺点**：
- 容易产生歧义
- 难以进行形式化验证
- 描述复杂逻辑时冗长

### 3.2 半形式化描述

使用**结构化语言或图形**来增加精确性。

**示例（伪代码风格）**：
```
IF 用户输入.用户名 IN 数据库 AND 用户输入.密码 = 数据库.密码 THEN
    显示系统主页
ELSE
    显示错误信息("用户名或密码错误")
    登录失败次数 + 1
    IF 登录失败次数 >= 5 THEN
        锁定账户(30分钟)
    END IF
END IF
```

### 3.3 形式化描述

使用**数学符号和形式语言**精确描述需求。

**示例（Z notation 片段）**：
```
LoginSuccessful
    users : USERNAME ↔ PASSWORD
    current? : USERNAME × PASSWORD
    result! : RESULT

    current? ∈ users
    result! = success
```

**优点**：
- 完全无歧义
- 可以进行数学证明和验证
- 支持自动化测试生成

**缺点**：
- 学习成本高
- 利益相关者难以理解
- 描述复杂系统工作量大

### 3.4 对比总结

| 维度 | 非形式化 | 半形式化 | 形式化 |
|------|---------|---------|--------|
| 精确性 | 低 | 中 | 高 |
| 可理解性 | 高 | 中 | 低 |
| 验证难度 | 难 | 中 | 易 |
| 适用场景 | 一般项目 | 大多数项目 | 安全关键系统 |
| 代表方法 | 自然语言 | 伪代码、UML | Z、VDM、B方法 |

## 四、原型在需求说明中的作用

### 4.1 原型的定义

原型是系统的**早期可运行版本或可视化模型**，用于探索和验证需求。

### 4.2 原型的类型

| 类型 | 说明 | 工具 |
|------|------|------|
| 纸面原型 | 手绘界面草图 | 纸笔 |
| 线框图 | 低保真布局图 | Balsamiq、draw.io |
| 可交互原型 | 高保真可点击模型 | Figma、Axure、Sketch |
| 抛弃式原型 | 验证后丢弃 | 任何工具 |
| 演化式原型 | 逐步发展为最终产品 | 开发框架 |

### 4.3 原型在需求说明中的价值

1. **消除沟通障碍**：可视化比文字更直观
2. **尽早发现需求问题**：在编码前发现问题
3. **验证用户界面**：测试可用性和工作流
4. **管理期望**：让客户对最终产品有清晰预期
5. **降低风险**：在低投入阶段验证核心假设

### 4.4 原型使用建议

- 明确告诉用户这是原型，不是最终产品
- 先原型后文档，用原型辅助编写需求
- 核心功能优先，避免过度设计
- 收集反馈后及时迭代
- 选择合适的保真度

## 五、场景描述

### 5.1 场景的定义

场景（Scenario）是对系统**特定使用情况的逐步描述**，包括用户目标、交互步骤和预期结果。

### 5.2 场景的组成

- **触发事件**：什么引发了这个场景
- **前置条件**：执行前必须满足的条件
- **执行步骤**：用户和系统的交互序列
- **后置条件**：执行后系统状态的变化
- **异常路径**：可能出现的异常情况

### 5.3 场景描述示例

**场景名称**：在线支付

**主要参与者**：注册用户

**触发事件**：用户在购物车点击"去结算"

**前置条件**：用户已登录，购物车非空，收货地址已选择

**主成功场景**：
1. 系统显示订单确认页面（商品列表、收货地址、应付金额）
2. 用户选择支付方式
3. 用户确认支付
4. 系统调用支付网关
5. 支付网关返回成功结果
6. 系统更新订单状态为"已支付"
7. 系统显示支付成功页面
8. 系统发送订单确认邮件

**扩展**：
- 3a：用户取消支付 → 返回购物车
- 5a：支付失败 → 显示错误信息，允许重新支付
- 5b：支付超时 → 订单状态变为"待支付"，保留15分钟

### 5.4 场景与用例的关系

- 用例 = 场景的集合
- 一个用例可以包含多个场景（主场景 + 备选场景）
- 场景是用例的具体实例化

## 六、需求跟踪矩阵（RTM）

### 6.1 定义

需求跟踪矩阵（Requirements Traceability Matrix, RTM）是将需求与**设计、实现、测试**等工件关联起来的表格，确保每个需求都被正确实现和验证。

### 6.2 基本结构

| 需求ID | 需求描述 | 设计文档 | 代码模块 | 测试用例 | 状态 |
|--------|---------|---------|---------|---------|------|
| FR-001 | 用户登录 | DD-001 | LoginController | TC-001~TC-005 | 已实现 |
| FR-002 | 密码重置 | DD-002 | ResetController | TC-006~TC-008 | 开发中 |
| NFR-001 | 响应时间<2s | DD-010 | CacheManager | TC-050 | 已验证 |

### 6.3 跟踪方向

- **正向跟踪**：需求 → 设计 → 代码 → 测试（确保需求被覆盖）
- **反向跟踪**：测试 → 代码 → 设计 → 需求（确保无多余实现）

### 6.4 RTM 的作用

- 验证每个需求都有对应的实现和测试
- 评估需求变更的影响范围
- 确保没有遗漏的需求
- 支持回归测试的用例选择
- 为项目验收提供证据

### 6.5 工具支持

- Excel / Google Sheets（最常见）
- Jira + Confluence
- IBM DOORS
- Azure DevOps
- Jama Connect
