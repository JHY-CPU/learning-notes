# SRS文档结构（IEEE 830标准）

一、SRS概述

## 一、SRS概述


软件需求规格说明（Software Requirements Specification，SRS）是描述系统功能和非功能需求的正式文档。IEEE 830标准提供了SRS的推荐结构和编写指南。


> **Note:** **SRS的作用：**
> 作为开发团队与客户之间的合同基础，为设计、编码、测试提供依据，是项目验收的参考标准。


### 1.1 SRS的质量特性


| 特性 | 说明 |
| --- | --- |
| 正确性 | SRS描述的需求应反映用户的真实意图 |
| 无歧义 | 每个需求只有一种解释 |
| 完整性 | 包含所有有意义的需求，包括约束条件 |
| 一致性 | 需求之间不自相矛盾 |
| 可验证性 | 每条需求可通过测试等方式验证 |
| 可修改性 | 结构清晰，便于修改和维护 |
| 可追踪性 | 需求来源和去向清晰可追溯 |

二、IEEE 830 SRS模板

## 二、IEEE 830 SRS模板结构


### 2.1 标准模板章节


1. **引言（Introduction）**

   - 1.1 目的（Purpose）
   - 1.2 范围（Scope）
   - 1.3 定义、缩写和缩略语
   - 1.4 参考文献
   - 1.5 概述
2. **总体描述（Overall Description）**

   - 2.1 产品前景（Product Perspective）
   - 2.2 产品功能（Product Functions）
   - 2.3 用户特征（User Characteristics）
   - 2.4 约束（Constraints）
   - 2.5 假设和依赖
3. **具体需求（Specific Requirements）**

   - 3.1 外部接口需求
   - 3.2 功能需求
   - 3.3 性能需求
   - 3.4 设计约束
   - 3.5 质量属性
   - 3.6 其他需求
4. **附录（Appendix）**

三、功能需求

## 三、功能需求（Functional Requirements）


功能需求描述系统"做什么"，即系统应该提供的功能和服务。


### 3.1 功能需求的特点


- 描述输入与输出之间的关系
- 定义系统对外部事件的响应
- 指定数据的处理和存储方式


### 3.2 功能需求编写示例

**格式模板：**


`FR-[编号]：当 [条件/触发] 时，系统应 [动作/行为]，以 [结果/目的]。`


**示例：**


`FR-001：当用户输入正确的用户名和密码时，系统应验证身份并跳转到主页。`
四、非功能需求

## 四、非功能需求（Non-Functional Requirements）


非功能需求描述系统"做得怎么样"，定义系统的质量属性和约束条件。


### 4.1 非功能需求分类


| 类别 | 说明 | 示例 |
| --- | --- | --- |
| 性能需求 | 响应时间、吞吐量、并发 | 页面加载时间 < 3秒 |
| 安全性需求 | 认证、授权、加密 | 密码采用SHA-256加密 |
| 可用性需求 | 易用性、可访问性 | 新用户10分钟内学会操作 |
| 可靠性需求 | MTBF、容错能力 | 系统可用性 ≥ 99.9% |
| 可维护性需求 | 模块化、文档化 | 修改任意模块不超过2小时 |
| 可移植性需求 | 跨平台兼容性 | 支持Windows/Linux/macOS |
| 可扩展性需求 | 容量扩展能力 | 支持用户数从1万扩展到100万 |


> **Note:** **FURPS+模型：**
> Functionality, Usability, Reliability, Performance, Supportability + 设计约束、实现约束、接口约束、物理约束。

五、需求追踪

## 五、需求可追踪性（Requirements Traceability）


需求可追踪性是指追踪需求从来源到实现整个生命周期的能力。


### 5.1 追踪方向


- **前向追踪：**
   需求 → 设计 → 代码 → 测试用例
- **后向追踪：**
   测试用例 → 代码 → 设计 → 需求
- **需求间追踪：**
   高层需求 → 细化需求


### 5.2 需求追踪矩阵（RTM）


| 需求ID | 需求描述 | 设计文档 | 代码模块 | 测试用例 | 状态 |
| --- | --- | --- | --- | --- | --- |
| FR-001 | 用户登录功能 | DD-003 | LoginModule | TC-001~TC-005 | 已实现 |
| FR-002 | 用户注册功能 | DD-004 | RegisterModule | TC-006~TC-010 | 开发中 |
| NFR-001 | 响应时间 < 3s | DD-005 | CacheLayer | TC-011~TC-012 | 待验证 |


### 5.3 追踪性的作用


- 确保所有需求都被实现和测试
- 评估需求变更的影响范围
- 支持审计和合规检查
- 便于缺陷定位和回归分析

六、SRS编写技巧

## 六、SRS编写最佳实践


1. **使用明确的语言：**
   避免"通常"、"一般"等模糊词汇
2. **量化指标：**
   使用具体数字而非定性描述
3. **编号规范：**
   采用层次化编号体系
4. **版本控制：**
   记录每次变更的内容和原因
5. **评审机制：**
   组织需求评审会议，确保质量
