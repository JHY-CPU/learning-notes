# 需求获取与分析

## 一、需求工程概述

### 1.1 定义

需求工程（Requirements Engineering, RE）是系统化地**发现、分析、文档化和验证**软件系统需求的过程。它是软件开发生命周期的起点，直接影响项目的成败。

### 1.2 需求的定义

IEEE 将需求定义为：
1. 用户解决问题或达到目标所需的条件或能力
2. 系统或系统组件必须满足的合同、标准、规格说明中规定的条件或能力
3. 上述条件或能力的文档化表述

### 1.3 需求的分类

```
需求
├── 功能需求（Functional Requirements）
│   └── 系统应该做什么
├── 非功能需求（Non-Functional Requirements）
│   └── 系统做得怎么样（性能、安全、可用性等）
├── 约束条件（Constraints）
│   └── 限制系统实现的条件
└── 业务需求（Business Requirements）
    └── 为什么要做这个系统
```

## 二、功能需求与非功能需求

### 2.1 功能需求

描述系统**必须执行的功能或服务**，定义了系统的输入、处理和输出行为。

示例：
- 用户能够通过邮箱和密码登录系统
- 系统应在收到订单后24小时内发送确认邮件
- 管理员能够导出用户活动报告（CSV/PDF格式）

### 2.2 非功能需求

描述系统运行的**质量属性和约束**，影响用户体验和系统架构。

| 类别 | 说明 | 示例 |
|------|------|------|
| 性能 | 响应时间、吞吐量 | 页面加载 < 2秒 |
| 可用性 | 系统运行时间 | 99.9% 可用性 |
| 安全性 | 数据保护、访问控制 | 支持 OAuth 2.0 |
| 可扩展性 | 支持用户/数据增长 | 支持10万并发用户 |
| 兼容性 | 与现有系统集成 | 支持 REST API |
| 可维护性 | 代码和架构的可维护程度 | 模块化设计 |
| 可移植性 | 跨平台运行能力 | 支持 Windows/Linux |
| 可靠性 | 故障恢复能力 | RTO < 4小时 |

### 2.3 FURPS+ 模型

一种常见的需求分类框架：
- **F**unctionality（功能性）
- **U**sability（可用性）
- **R**eliability（可靠性）
- **P**erformance（性能）
- **S**upportability（可支持性）
- **+**：设计约束、实现约束、接口约束、物理约束

## 三、需求获取技术

### 3.1 访谈（Interview）

与利益相关者进行一对一或小组对话。

- **结构化访谈**：预设固定问题
- **半结构化访谈**：有大纲但允许灵活追问
- **非结构化访谈**：开放式对话

优点：深入了解，可追问细节
缺点：耗时，受访谈者表达能力影响

### 3.2 问卷调查（Questionnaire）

针对大量用户设计标准化问题。

适用场景：用户基数大、地理位置分散、需要量化数据。

### 3.3 观察法（Observation）

直接观察用户的工作流程和操作习惯。

- **被动观察**：仅记录，不干预
- **主动观察**：在过程中与用户交流

适用场景：用户无法准确描述自己的工作流程时。

### 3.4 原型法（Prototyping）

快速构建系统原型让用户试用和反馈。

- **低保真原型**：纸面原型、线框图
- **高保真原型**：可交互的界面原型（Figma、Axure）

适用场景：需求不明确、需要快速验证用户界面。

### 3.5 头脑风暴（Brainstorming）

团队成员自由发散思维，产生大量创意和想法。

规则：不批评、鼓励狂想、量多为优、组合改进。

### 3.6 联合应用开发（JAD）

将所有利益相关者集中在一起，通过引导式研讨会快速达成共识。

参与者：业务用户、开发人员、分析师、管理层、主持人。

## 四、利益相关者分析

### 4.1 利益相关者定义

所有对系统有利益关系或受系统影响的个人和组织。

### 4.2 常见利益相关者

- 最终用户
- 客户 / 业务负责人
- 项目经理
- 开发团队
- 运维团队
- 监管机构
- 第三方集成商

### 4.3 利益-影响力矩阵

```
影响力
高 │ 重点管理      │ 密切合作
   │ （客户）      │ （业务负责人）
───┼───────────────┼───────────────
低 │ 最低关注      │ 保持知情
   │ （普通用户）  │ （运维团队）
   └───────────────┴───────────────
         低              高
               利益程度
```

### 4.4 分析步骤

1. 识别所有利益相关者
2. 分析各方的利益、期望和影响力
3. 确定沟通和管理策略
4. 持续跟踪利益相关者的变化

## 五、用例与用例图（UML）

### 5.1 用例定义

用例（Use Case）描述系统与外部参与者之间的交互，定义系统为参与者提供的**功能和行为**。

### 5.2 用例图元素

- **参与者（Actor）**：与系统交互的外部实体（人、系统、设备）
- **用例（Use Case）**：系统提供的功能
- **系统边界**：界定系统的范围
- **关系**：关联、包含（include）、扩展（extend）、泛化

### 5.3 示例

```
┌─────────────────────────────────────┐
│           网上银行系统                │
│                                      │
│  ┌──────────┐    ┌──────────────┐   │
│  │ 查询余额  │    │  转账汇款     │   │
│  └──────────┘    └──────────────┘   │
│       │                 │           │
│  ┌──────────┐    ┌──────────────┐   │
│  │ 登录验证  │◄───│  填写转账信息  │   │
│  └──────────┘    └──────────────┘   │
│                                      │
└─────────────────────────────────────┘
     │                │
   银行客户         管理员
```

### 5.4 用例描述模板

| 字段 | 内容 |
|------|------|
| 用例名称 | 转账汇款 |
| 参与者 | 银行客户 |
| 前置条件 | 用户已登录 |
| 基本流程 | 1. 选择转账 2. 输入收款人 3. 输入金额 4. 确认 5. 完成 |
| 备选流程 | 余额不足→提示错误 |
| 后置条件 | 余额更新，生成交易记录 |

## 六、用户故事与 INVEST 原则

### 6.1 用户故事格式

```
作为 <角色>，我想要 <功能>，以便 <价值>
```

示例：作为购物者，我想要将商品加入收藏夹，以便之后快速找到心仪商品。

### 6.2 INVEST 原则

| 字母 | 含义 | 说明 |
|------|------|------|
| I | Independent | 故事之间尽量独立 |
| N | Negotiable | 可协商，不是合同 |
| V | Valuable | 对用户有价值 |
| E | Estimable | 可以估算工作量 |
| S | Small | 足够小，一个迭代内完成 |
| T | Testable | 可以编写验收测试 |

### 6.3 验收标准（Acceptance Criteria）

每个用户故事应附带明确的验收标准，通常使用 **Given-When-Then** 格式：

```
Given 用户已登录且购物车有商品
When 点击结算按钮
Then 跳转到支付页面并显示订单总额
```

### 6.4 用户故事 vs 用例

| 维度 | 用户故事 | 用例 |
|------|---------|------|
| 格式 | 一句话 | 结构化文档 |
| 详细程度 | 概要性 | 详细的流程描述 |
| 驱动 | 敏捷开发 | 传统瀑布模型 |
| 关注点 | 用户价值 | 系统行为 |

## 七、需求分析方法

### 7.1 结构化分析

以**数据流**为中心，通过分层的模型描述系统的功能和数据。

核心工具：
- 数据流图（DFD）
- 数据字典
- 实体关系图（ER）
- 状态转换图

### 7.2 面向对象分析（OOA）

以**对象**为中心，通过类、对象、继承、封装等概念描述系统。

核心工具：
- UML 图（用例图、类图、顺序图、活动图、状态图）
- CRC 卡（类-职责-协作）

### 7.3 方法对比

| 维度 | 结构化分析 | 面向对象分析 |
|------|-----------|-------------|
| 核心概念 | 过程、数据流 | 对象、类 |
| 适用领域 | 数据处理系统 | 复杂交互系统 |
| 代表工具 | DFD、ER | UML |
| 可复用性 | 较低 | 较高 |

## 八、数据流图（DFD）

### 8.1 基本元素

- **外部实体**：系统的数据来源和去向（方框）
- **处理/加工**：数据的转换过程（圆形/圆角矩形）
- **数据存储**：数据的保存位置（开口矩形）
- **数据流**：数据的流动方向（箭头线）

### 8.2 分层结构

```
顶层（上下文图）：整个系统为一个处理
    ↓
0层图：系统的主要子功能
    ↓
1层图：每个子功能的细化
    ↓
... 逐层细化直到基本处理
```

### 8.3 示例（网上书店）

```
[顾客] → 订单 → (处理订单) → [订单存储]
                 → 发货通知 → [仓库]
[顾客] ← 发货单 ← [订单存储]
```

### 8.4 命名规范

- 处理：动词+名词（如"处理订单"、"验证密码"）
- 数据流：名词（如"订单信息"、"用户凭证"）
- 数据存储：D+编号或名词（如"D1 会员表"）

---

## 九、实体关系图（ER图）

### 9.1 基本元素

- **实体（Entity）**：需要存储信息的对象（矩形）
- **属性（Attribute）**：实体的特征（椭圆形）
- **关系（Relationship）**：实体之间的联系（菱形）

### 9.2 关系类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 1:1 | 一对一 | 用户 ↔ 用户配置 |
| 1:N | 一对多 | 部门 → 员工 |
| M:N | 多对多 | 学生 ↔ 课程 |

### 9.3 ER图符号

```
[顾客] —(下单)——<订单>——(包含)——[商品]
  │                                │
  (属性)                          (属性)
  姓名、邮箱                      名称、价格
```

### 9.4 从 ER 到关系数据库

- 实体 → 表
- 属性 → 字段
- 关系：
  - 1:1 → 外键或合并
  - 1:N → 在N方加外键
  - M:N → 创建关联表

## 十、需求优先级排序（MoSCoW 方法）

### 10.1 定义

MoSCoW 是一种需求优先级排序方法，将需求分为四个等级。

### 10.2 优先级分类

| 等级 | 含义 | 比例建议 |
|------|------|---------|
| **Must have** | 必须有，没有就无法交付 | ~60% |
| **Should have** | 应该有，但没有也能临时替代 | ~20% |
| **Could have** | 可以有，锦上添花 | ~20% |
| **Won't have (this time)** | 这次不做，未来再考虑 | 记录但不计入 |

### 10.3 示例（电商平台）

- **Must**：用户注册登录、商品浏览、购物车、订单支付
- **Should**：商品评价、订单追踪、优惠券系统
- **Could**：个性化推荐、商品对比、心愿单
- **Won't**：AR试穿、直播带货

### 10.4 其他优先级方法

- **价值-成本矩阵**：按业务价值和实现成本排序
- **Kano 模型**：基本需求、期望需求、兴奋需求
- **100美元法**：利益相关者分配虚拟预算来投票
- **WSJF（加权最短作业优先）**：考虑延迟成本的优先级排序
