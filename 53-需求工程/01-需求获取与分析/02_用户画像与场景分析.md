# 用户画像与场景分析


## 用户画像与场景分析

一、用户画像

## 一、用户画像（User Persona）


用户画像是基于真实数据和调研构建的虚拟用户代表，帮助团队建立对目标用户的共识。


### 1.1 用户画像模板


| 要素 | 内容 | 示例 |
| --- | --- | --- |
| 基本信息 | 姓名、年龄、职业、地区 | 张伟，28岁，北京，后端开发工程师 |
| 技术背景 | 技术水平、常用工具、经验年限 | 5年Go/Java经验，熟悉K8s |
| 目标/动机 | 使用产品的核心目的 | 快速搭建微服务，减少运维工作量 |
| 痛点/挫败 | 当前解决方案的问题 | 部署流程繁琐，环境配置耗时 |
| 行为模式 | 使用场景、频率、偏好 | 每天部署3-5次，偏好CLI而非GUI |
| 影响因素 | 决策驱动因素 | 团队规范、技术社区口碑 |


### 1.2 用户画像构建方法


- **定量分析：**
   用户行为数据（GA、埋点）、用户属性统计、留存和转化漏斗
- **定性研究：**
   用户访谈（5-8人可发现80%问题）、问卷调查、竞品用户评价
- **聚类分析：**
   基于行为数据进行用户分群，识别典型模式

**用户画像数量建议：**
3-5个典型画像覆盖主要用户群。过多画像会导致焦点分散。优先覆盖核心用户群体和高价值用户群体。
二、用户旅程图

## 二、用户旅程图（User Journey Map）


用户旅程图可视化用户完成目标的全过程，揭示各阶段的体验和痛点。


### 2.1 旅程图组成要素


| 维度 | 说明 |
| --- | --- |
| 阶段 (Stages) | 用户完成目标的各个阶段（认知→考虑→使用→推荐） |
| 行为 (Actions) | 每个阶段用户的具体操作 |
| 想法 (Thoughts) | 用户在每个阶段的心理活动 |
| 情绪 (Emotions) | 用户的情感曲线（满意/沮丧/困惑） |
| 痛点 (Pain Points) | 导致负面体验的关键时刻 |
| 机会点 (Opportunities) | 可优化的环节 |


### 2.2 旅程图示例（开发者API集成）


| 阶段 | 发现 | 评估 | 集成 | 使用 | 扩展 |
| --- | --- | --- | --- | --- | --- |
| 行为 | 搜索技术文章 | 阅读文档和定价 | 安装SDK、写Demo | 接入业务代码 | 高级功能探索 |
| 情绪 | 好奇 | 犹豫 | 挫败→兴奋 | 稳定 | 满足 |
| 痛点 | 信息分散 | 文档不清晰 | SDK版本冲突 | 错误信息不明 | 高级功能缺乏示例 |
| 机会 | SEO优化 | Quick Start指南 | 一键脚手架 | 更好的错误提示 | 用例库和社区 |

三、用例图

## 三、用例图（Use Case Diagram）


用例图从用户视角描述系统功能，是需求分析中最常用的UML图之一。


### 3.1 用例图元素


| 元素 | 符号 | 说明 |
| --- | --- | --- |
| 参与者(Actor) | 小人图标 | 与系统交互的外部实体（人或系统） |
| 用例(Use Case) | 椭圆 | 系统提供的功能单元 |
| 关联 | 实线 | 参与者与用例的关系 |
| 包含(include) | 虚线+<<include>> | 必须执行的子用例 |
| 扩展(extend) | 虚线+<<extend>> | 条件执行的扩展行为 |
| 泛化 | 空心三角箭头 | 参与者或用例之间的继承关系 |


### 3.2 用例图示例（电商系统）


> **Note:** **用例关系：**
>
>
> 普通用户 → [浏览商品] → [加入购物车] → [下单] → <<include>> → [支付]
>
>
> 普通用户 → [下单] → <<extend>> → [使用优惠券]
>
>
> 管理员 → [管理商品] → [审核评论]
>
>
> 外部系统(支付网关) → [处理支付]


### 3.3 用例图绘制要点


- 参与者是系统的外部角色，不是系统内部组件
- 用例命名使用动词短语（"创建订单"而非"订单"）
- include表示"必须做"，extend表示"可以做"
- 避免过多用例（每张图不超过10-15个用例）
- 先识别主要参与者和核心业务流程

四、场景分析

## 四、场景分析（Scenario Analysis）


场景分析通过构建具体的使用情境，帮助发现隐含需求和边界条件。


### 4.1 场景分类


| 场景类型 | 描述 | 示例 |
| --- | --- | --- |
| 正常场景 (Happy Path) | 用户顺利完成目标的流程 | 用户正常下单并支付成功 |
| 异常场景 | 出现问题时的系统行为 | 支付超时、网络中断 |
| 边界场景 | 极端条件下的系统表现 | 库存为0、同时秒杀 |
| 负面场景 | 恶意用户的行为 | 刷单、注入攻击 |
| 兼容场景 | 不同环境下的行为 | 移动端vs桌面端、不同浏览器 |


> **Note:** **场景分析的价值：**
> 用户画像告诉你"为谁设计"，用户旅程图告诉你"用户如何体验"，场景分析告诉你"具体需要什么功能"。三者结合构成完整的需求分析框架。
