需求验证与变更管理

一、需求验证（Requirements Verification）

需求验证是确认需求规格说明是否正确、完整、一致地反映了用户期望的过程。其目的是在进入设计和编码之前尽早发现并纠正需求缺陷。

验证 vs 确认：验证（Verification）检查"是否正确地构建了产品"，确认（Validation）检查"是否构建了正确的产品"。需求验证主要侧重于确认。

1.1 需求评审（Reviews）

需求评审是最常用的需求验证方法，通过组织相关人员审查需求文档来发现缺陷。

评审类型

类型特点参与者
同行评审开发人员之间互相审查开发工程师
审查（Inspection）正式、有角色分工、数据驱动主持人、作者、审查员、记录员
走查（Walkthrough）作者引导，参与者提问作者、相关利益者
技术评审评估技术可行性和质量技术专家

审查流程（Fagan Inspection）

  1. 计划：确定审查范围、选择审查团队
  2. 总体会议：作者介绍背景和材料
  3. 准备：审查员独立审查，标记问题
  4. 审查会议：逐一讨论问题，记录缺陷
  5. 返工：作者修正发现的缺陷
  6. 跟进：确认缺陷已修正

1.2 原型验证

通过构建原型让用户实际体验系统功能，收集反馈来验证需求是否满足用户期望。

  • 用户通过交互原型直观感受系统行为
  • 快速发现需求理解偏差
  • 适合界面和交互密集型系统

1.3 其他验证方法

方法说明
测试用例生成从需求推导测试用例，不可测试的需求往往有缺陷
形式化方法使用数学方法证明需求的正确性
场景分析通过具体场景检验需求的完整性和可行性
检查表法使用标准化检查清单逐项核对

二、需求变更控制（Change Control）

需求变更是不可避免的，有效的变更控制机制能在接受合理变更的同时防止范围蔓延。

2.1 变更控制流程

  1. 变更申请：提交变更请求（Change Request），描述变更内容和原因
  2. 影响分析：评估变更对进度、成本、质量的影响
  3. 变更审批：由变更控制委员会（CCB）审议决定
  4. 变更实施：批准后更新需求文档和相关设计
  5. 变更验证：确认变更正确实施
  6. 变更关闭：记录变更结果，更新追踪矩阵
变更控制委员会（CCB）：由项目经理、技术负责人、客户代表等组成，负责审批需求变更。CCB评估变更的必要性、可行性和影响。

2.2 变更影响分析

影响维度分析内容
工作量影响需要多少额外的人力和时间
进度影响是否导致里程碑延期
成本影响额外的资源和费用
技术影响需要修改哪些设计和代码
风险影响引入哪些新的技术或管理风险
质量影响对系统质量属性的影响

三、需求追踪矩阵（Requirements Traceability Matrix）

需求追踪矩阵（RTM）是记录需求与其相关工件之间关系的表格，是需求管理的核心工具。

3.1 RTM的作用

  • 完整性验证：确保每个需求都有对应的设计、代码和测试
  • 变更影响：快速定位变更影响的范围
  • 覆盖分析：检查测试是否覆盖了所有需求
  • 双向追踪：支持从需求到测试和从测试到需求的双向追踪

3.2 RTM示例

需求ID需求描述设计元素代码模块测试用例缺陷状态
REQ-001用户登录DES-001LoginServiceTC-001~TC-005Bug-003已关闭
REQ-002密码重置DES-002PasswordServiceTC-006~TC-008-已测试
REQ-003数据导出DES-005ExportModuleTC-009~TC-012-开发中

3.3 追踪维度

  • 前向追踪：需求 → 设计 → 实现 → 测试（保证需求被实现）
  • 后向追踪：测试 → 实现 → 设计 → 需求（保证实现有依据）
  • 水平追踪：不同层级需求之间的关联

四、需求基线（Requirements Baseline）

需求基线是在特定时间点经过评审和批准的需求集合，是后续开发工作的基准。

基线管理要点：
  • 基线一旦建立，变更需通过正式的变更控制流程
  • 每个基线版本应有明确的版本号和发布日期
  • 基线文档应纳入配置管理
