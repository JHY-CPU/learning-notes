# 需求验证与管理

## 一、需求验证方法

### 1.1 定义

需求验证（Verification）回答的问题是：**"我们是否正确地构建了系统？"** 即确认需求文档是否符合规范和标准，内容是否正确、完整、一致。

### 1.2 需求评审（Review）

**正式评审（Inspection）**：
- 由受过训练的主持人引导
- 有明确的角色分工：主持人、作者、评审员、记录员
- 使用检查清单逐项审查
- 记录缺陷并跟踪修复
- 适用场景：关键系统、合同型项目

**非正式评审（Walkthrough）**：
- 由作者引导，团队参与
- 重点在理解需求、发现明显问题
- 适用场景：内部项目、迭代开发

**评审检查要点**：
- 需求是否清晰、无歧义
- 需求是否可测试、可验证
- 需求之间是否一致、无矛盾
- 是否存在遗漏的需求
- 非功能需求是否有量化指标

### 1.3 原型验证

通过构建原型来验证需求的理解是否正确：
- 让用户操作原型并收集反馈
- 验证工作流和界面设计
- 发现隐含需求和误解

### 1.4 测试用例设计验证

为每条需求编写测试用例：
- 如果无法编写测试用例，说明需求不可验证
- 测试用例设计过程本身就是对需求的深入理解过程
- 帮助发现需求中的模糊和遗漏

## 二、需求确认 vs 验证

### 2.1 概念区分

| 维度 | 确认（Validation） | 验证（Verification） |
|------|-------------------|---------------------|
| 核心问题 | 是否构建了**正确的系统**？ | 是否**正确地构建**了系统？ |
| 关注点 | 需求是否满足用户的真实需要 | 需求文档是否符合规范 |
| 方法 | 用户评审、原型、用户验收测试 | 同行评审、检查清单、形式化审查 |
| 判断者 | 用户/客户 | 开发团队/评审员 |

### 2.2 常见误区

- **只验证不确认**：需求文档写得完美，但不符合用户实际需求
- **只确认不验证**：方向正确但文档质量差，导致开发偏差

### 2.3 实践建议

- 需求获取和分析阶段侧重**确认**
- 需求文档编写完成后侧重**验证**
- 两个过程贯穿需求工程的始终

## 三、需求变更管理

### 3.1 变更的来源

- 客户业务变化
- 市场环境变化
- 技术约束变化
- 法规政策变化
- 需求理解偏差的纠正
- 新增需求（Scope Creep）

### 3.2 变更管理流程

```
变更请求提交 → 影响分析 → 变更评审 → 批准/拒绝 → 实施变更 → 验证变更 → 更新文档
```

1. **提交变更请求**：记录变更内容、原因、优先级
2. **影响分析**：评估对进度、成本、技术、其他需求的影响
3. **变更评审**：由变更控制委员会（CCB）审议
4. **决策**：批准、拒绝、推迟
5. **实施与验证**：执行变更并验证结果
6. **更新文档**：修订所有受影响的文档

### 3.3 变更控制委员会（CCB）

- 由项目关键利益相关者组成
- 负责审批所有需求变更
- 评估变更的优先级和影响
- 平衡业务价值和开发成本

### 3.4 变更管理原则

- 所有变更必须书面记录
- 影响分析不可省略
- 避免在开发后期做重大变更
- 建立变更基线，便于追踪
- 沟通变更影响给所有相关方

## 四、需求版本控制

### 4.1 版本控制的重要性

- 需求文档随项目演进不断更新
- 需要追踪每次修改的内容和原因
- 支持回溯到历史版本
- 多人协作时避免覆盖冲突

### 4.2 版本号规范

- **主版本号.次版本号.修订号**（如 1.2.3）
- 主版本号：需求重大变更或结构重组
- 次版本号：新增或删除需求
- 修订号：文字修正、格式调整

### 4.3 版本控制实践

- 使用 Git、SVN 或 SharePoint 管理文档版本
- 每次提交附带变更说明
- 使用基线（Baseline）标记关键版本节点
- 定期生成需求差异报告（Diff Report）

### 4.4 基线管理

- **基线**：某一时间点需求的快照，作为后续工作的参照
- 关键基线：需求获取完成基线、需求评审通过基线、每个迭代的需求基线
- 基线变更需要正式审批

## 五、需求跟踪

### 5.1 跟踪的目的

- 确保每个需求都被正确实现
- 评估需求变更的影响范围
- 支持项目状态的可视化
- 为项目验收提供依据

### 5.2 跟踪维度

**正向跟踪**（需求 → 产品）：
```
业务需求 → 用户需求 → 功能需求 → 设计规格 → 代码 → 测试用例
```

**反向跟踪**（产品 → 需求）：
```
测试用例 → 代码 → 设计规格 → 功能需求 → 用户需求 → 业务需求
```

### 5.3 跟踪粒度

| 粒度 | 说明 | 适用场景 |
|------|------|---------|
| 需求-需求 | 上下游需求的关联 | 复杂系统的需求层级 |
| 需求-设计 | 需求到设计文档的映射 | 架构评审 |
| 需求-代码 | 需求到代码模块的映射 | 变更影响分析 |
| 需求-测试 | 需求到测试用例的映射 | 覆盖率分析 |

### 5.4 跟踪实践

- 为每个需求分配唯一标识符
- 维护需求跟踪矩阵（RTM）
- 在 CI/CD 流水线中自动生成覆盖率报告
- 定期审计跟踪链的完整性

## 六、需求工具

### 6.1 IBM DOORS

- **类型**：专业的需求管理工具
- **特点**：
  - 强大的需求跟踪和追溯能力
  - 支持需求基线和版本管理
  - 支持变更影响分析
  - 符合 DO-178C、ISO 26262 等标准
- **适用**：大型项目、安全关键系统（航空、汽车、医疗）

### 6.2 Atlassian Confluence + Jira

- **Confluence**：需求文档的编写和协作
  - 支持模板化需求文档
  - 实时协作编辑
  - 与 Jira 无缝集成
- **Jira**：需求的任务管理和跟踪
  - 用户故事和任务管理
  - 看板和冲刺规划
  - 工作流自定义
- **适用**：敏捷团队、中型项目

### 6.3 其他常用工具

| 工具 | 特点 | 适用场景 |
|------|------|---------|
| Azure DevOps | 微软全家桶，集成度高 | .NET/微软生态团队 |
| Jama Connect | 需求+测试+风险一体化 | 复杂产品开发 |
| Polarion | ALM 工具，支持追溯 | 汽车、嵌入式系统 |
| ReqView | 轻量级需求管理 | 中小团队 |
| Notion | 灵活的知识管理 | 初创团队、轻量需求 |
| Enterprise Architect | UML 建模+需求管理 | 传统软件工程 |

### 6.4 工具选择建议

- **项目规模**：大项目选专业工具（DOORS、Jama），小项目选轻量工具（Notion、Confluence）
- **开发模式**：敏捷团队选 Jira，瀑布模型选 DOORS
- **合规要求**：安全关键系统选通过认证的工具
- **集成需求**：考虑与现有工具链的集成能力
- **预算约束**：开源选项（GitLab Issues、Redmine）也是可行选择
