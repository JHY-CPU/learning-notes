前端测试策略

核心概念：前端测试金字塔分为三层：单元测试（Unit Test，测试独立的函数和组件）、集成测试（Integration Test，测试模块之间的交互）、端到端测试（E2E Test，测试完整的用户流程）。此外还有视觉回归测试和Storybook组件测试。Vitest是当前推荐的单元测试框架，Playwright是E2E测试首选。

一、测试金字塔

1.1 测试层级

/\ E2E测试（少量） / \ 浏览器中完整用户流程 / \ Playwright / Cypress /______\ / \ 集成测试（适量） / \ 组件 + API 交互 / \ Testing Library + MSW /______________\ / \ 单元测试（大量） / \ 纯函数、工具类、组件 / \ Vitest / Jest /______________________\ 原则：单元测试多（快、稳定），E2E测试少（慢、易碎）

1.2 各层测试对比

类型测试对象速度稳定性工具
单元测试函数、类、Hook快（ms级）高Vitest, Jest
组件测试React/Vue组件中等中高Testing Library
集成测试多组件协作中等中等Testing Library + MSW
E2E测试完整用户流程慢（秒级）低（易碎）Playwright, Cypress
视觉回归UI外观中等中等Chromatic, Percy

二、Vitest（单元测试框架）

2.1 Vitest配置

// vitest.config.ts
import { defineConfig } from 'vitest/config';

export default defineConfig({
    test: {
        globals: true,             // 全局API（describe/it/expect）
        environment: 'jsdom',      // 浏览器环境模拟
        include: ['src/**/*.{test,spec}.{ts,tsx}'],
        exclude: ['node_modules', 'dist'],
        coverage: {
            provider: 'v8',        // 或 'istanbul'
            reporter: ['text', 'html', 'lcov'],
            include: ['src/**/*.{ts,tsx}'],
            exclude: ['src/**/*.d.ts', 'src/**/*.test.*'],
            thresholds: {
                branches: 80,
                functions: 80,
                lines: 80,
                statements: 80
            }
        },
        setupFiles: ['./src/test/setup.ts'],  // 测试前置文件
        css: true,                 // 支持CSS模块
        alias: {
            '@': '/src'
        }
    }
});

2.2 基本测试语法

// src/utils/math.test.ts
import { describe, it, expect, vi, beforeEach } from 'vitest';
import { add, divide, fetchData } from './math';

describe('math utils', () => {
    describe('add', () => {
        it('should add two positive numbers', () => {
            expect(add(1, 2)).toBe(3);
        });

        it('should handle negative numbers', () => {
            expect(add(-1, -2)).toBe(-3);
        });

        it('should handle zero', () => {
            expect(add(0, 5)).toBe(5);
        });
    });

    describe('divide', () => {
        it('should divide two numbers', () => {
            expect(divide(6, 2)).toBe(3);
        });

        it('should throw on division by zero', () => {
            expect(() => divide(1, 0)).toThrow('Division by zero');
        });
    });

    describe('fetchData', () => {
        it('should fetch data successfully', async () => {
            // Mock fetch
            const mockData = { id: 1, name: 'test' };
            global.fetch = vi.fn().mockResolvedValue({
                ok: true,
                json: () => Promise.resolve(mockData)
            });

            const result = await fetchData('/api/data');
            expect(result).toEqual(mockData);
            expect(global.fetch).toHaveBeenCalledWith('/api/data');
        });
    });
});

2.3 Mock（模拟）

import { vi, describe, it, expect, beforeEach } from 'vitest';

// 1. Mock函数
const mockFn = vi.fn();
mockFn.mockReturnValue(42);
mockFn.mockResolvedValue(42);  // async
mockFn.mockRejectedValue(new Error('fail'));  // async error

expect(mockFn).toHaveBeenCalled();
expect(mockFn).toHaveBeenCalledWith('arg1', 'arg2');
expect(mockFn).toHaveBeenCalledTimes(2);

// 2. Mock模块
vi.mock('./api', () => ({
    getUser: vi.fn().mockResolvedValue({ id: 1, name: 'test' }),
    updateUser: vi.fn().mockResolvedValue(true)
}));

// 3. Mock部分模块
vi.mock('./api', async () => {
    const actual = await vi.importActual('./api');
    return {
        ...actual,
        getUser: vi.fn().mockResolvedValue({ id: 1, name: 'test' })
    };
});

// 4. Mock定时器
vi.useFakeTimers();
vi.advanceTimersByTime(1000);
vi.runAllTimers();
vi.useRealTimers();

// 5. Mock DOM API
Object.defineProperty(window, 'location', {
    value: { href: 'http://localhost:3000' },
    writable: true
});

// 6. beforeEach / afterEach
beforeEach(() => {
    vi.clearAllMocks();  // 清除所有mock的调用记录
});

// 7. Spy（监视）
const spy = vi.spyOn(console, 'log');
expect(spy).toHaveBeenCalledWith('hello');
spy.mockRestore();  // 恢复原实现

三、Testing Library（组件测试）

3.1 React Testing Library

// src/components/Counter.test.tsx
import { describe, it, expect } from 'vitest';
import { render, screen, fireEvent } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import Counter from './Counter';

describe('Counter', () => {
    it('should render with initial count', () => {
        render(<Counter initialCount={0} />);
        expect(screen.getByText('Count: 0')).toBeInTheDocument();
    });

    it('should increment count on click', async () => {
        const user = userEvent.setup();
        render(<Counter initialCount={0} />);

        const button = screen.getByRole('button', { name: /increment/i });
        await user.click(button);

        expect(screen.getByText('Count: 1')).toBeInTheDocument();
    });

    it('should call onChange when count changes', async () => {
        const user = userEvent.setup();
        const onChange = vi.fn();
        render(<Counter initialCount={0} onChange={onChange} />);

        await user.click(screen.getByRole('button', { name: /increment/i }));

        expect(onChange).toHaveBeenCalledWith(1);
    });

    it('should be accessible', () => {
        render(<Counter initialCount={0} />);

        // 检查ARIA标签
        expect(screen.getByRole('button')).toHaveAccessibleName('Increment');
        // 检查状态
        expect(screen.getByRole('status')).toHaveTextContent('Count: 0');
    });
});

3.2 Testing Library查询优先级

优先级查询方法用途
1（推荐）getByRole通过ARIA角色查询（最接近用户感知）
2getByLabelText通过label文本查询表单元素
3getByPlaceholderText通过placeholder查询
4getByText通过文本内容查询
5getByDisplayValue通过当前值查询
6（不推荐）getByTestId通过data-testid查询（最后手段）
Testing Library哲学：测试应该模拟用户如何与应用交互，而不是测试实现细节。避免：查询内部DOM结构、测试组件状态、模拟内部方法。

4.2 Playwright测试示例

// e2e/login.spec.ts
import { test, expect } from '@playwright/test';

test.describe('Login Flow', () => {
    test.beforeEach(async ({ page }) => {
        await page.goto('/login');
    });

    test('should login with valid credentials', async ({ page }) => {
        await page.getByLabel('Email').fill('user@example.com');
        await page.getByLabel('Password').fill('password123');
        await page.getByRole('button', { name: 'Sign In' }).click();

        // 断言跳转到dashboard
        await expect(page).toHaveURL('/dashboard');
        await expect(page.getByText('Welcome')).toBeVisible();
    });

    test('should show error for invalid credentials', async ({ page }) => {
        await page.getByLabel('Email').fill('wrong@example.com');
        await page.getByLabel('Password').fill('wrong');
        await page.getByRole('button', { name: 'Sign In' }).click();

        await expect(page.getByRole('alert')).toHaveText('Invalid credentials');
    });

    test('should navigate between pages', async ({ page }) => {
        // 登录
        await page.getByLabel('Email').fill('user@example.com');
        await page.getByLabel('Password').fill('password123');
        await page.getByRole('button', { name: 'Sign In' }).click();

        // 导航到设置页面
        await page.getByRole('link', { name: 'Settings' }).click();
        await expect(page).toHaveURL('/settings');

        // 返回dashboard
        await page.getByRole('link', { name: 'Dashboard' }).click();
        await expect(page).toHaveURL('/dashboard');
    });

    test('should be responsive on mobile', async ({ page, isMobile }) => {
        if (!isMobile) test.skip();

        await expect(page.getByRole('navigation')).toBeHidden();
        await page.getByRole('button', { name: 'Menu' }).click();
        await expect(page.getByRole('navigation')).toBeVisible();
    });
});

六、测试最佳实践

6.1 测试原则（FIRST）

  • Fast（快速）：测试应该快速执行，否则开发人员不会频繁运行
  • Isolated（隔离）：每个测试独立运行，不依赖其他测试的状态
  • Repeatable（可重复）：每次运行结果一致，不依赖外部环境
  • Self-Validating（自验证）：测试自动判断通过/失败，不需要人工检查
  • Timely（及时）：测试应该与代码同时编写（TDD）或紧随其后

6.2 测试覆盖率

// vitest.config.ts 覆盖率配置
export default defineConfig({
    test: {
        coverage: {
            provider: 'v8',
            reporter: ['text', 'html', 'json-summary'],
            thresholds: {
                branches: 80,
                functions: 80,
                lines: 80,
                statements: 80
            }
        }
    }
});

// package.json
{
    "scripts": {
        "test": "vitest",
        "test:run": "vitest run",
        "test:coverage": "vitest run --coverage",
        "test:ui": "vitest --ui"  // 可视化UI
    }
}
覆盖率不是万能的：100%覆盖率不等于代码没有bug。应该关注关键业务逻辑的覆盖率，而不是追求100%。测试质量比覆盖率数字更重要。

总结：前端测试策略遵循测试金字塔：大量单元测试（Vitest）提供快速反馈，适量集成测试（Testing Library）验证模块协作，少量E2E测试（Playwright）确保用户流程正确。视觉回归测试（截图对比/Chromatic）补充UI层面的质量保障。测试应该关注用户行为而非实现细节。
