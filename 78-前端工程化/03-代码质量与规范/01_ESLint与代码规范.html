ESLint与代码规范

核心概念：ESLint是JavaScript/TypeScript的静态代码分析工具，通过可配置的规则检测代码中的问题和风格不一致。配合Prettier统一格式化，通过husky + lint-staged在Git提交时自动检查，commitlint规范提交信息，形成完整的代码质量管控体系。

一、ESLint配置

1.1 Flat Config（ESLint 9+）

// eslint.config.js (ESLint 9+ Flat Config)
import js from '@eslint/js';
import tsPlugin from '@typescript-eslint/eslint-plugin';
import tsParser from '@typescript-eslint/parser';
import reactPlugin from 'eslint-plugin-react';
import reactHooksPlugin from 'eslint-plugin-react-hooks';

export default [
    // 全局忽略
    {
        ignores: ['dist/**', 'node_modules/**', 'coverage/**']
    },

    // 基础规则
    js.configs.recommended,

    // TypeScript配置
    {
        files: ['**/*.{ts,tsx}'],
        languageOptions: {
            parser: tsParser,
            parserOptions: {
                project: './tsconfig.json',
                ecmaFeatures: { jsx: true }
            }
        },
        plugins: {
            '@typescript-eslint': tsPlugin
        },
        rules: {
            ...tsPlugin.configs.recommended.rules,
            '@typescript-eslint/no-unused-vars': ['error', {
                argsIgnorePattern: '^_',
                varsIgnorePattern: '^_'
            }],
            '@typescript-eslint/no-explicit-any': 'warn',
            '@typescript-eslint/consistent-type-imports': 'error',
            'no-unused-vars': 'off'  // 交给@typescript-eslint处理
        }
    },

    // React配置
    {
        files: ['**/*.{jsx,tsx}'],
        plugins: {
            react: reactPlugin,
            'react-hooks': reactHooksPlugin
        },
        rules: {
            ...reactPlugin.configs.recommended.rules,
            ...reactHooksPlugin.configs.recommended.rules,
            'react/react-in-jsx-scope': 'off',  // React 17+ 不需要import React
            'react/prop-types': 'off'  // 使用TypeScript代替
        },
        settings: {
            react: { version: 'detect' }
        }
    }
];

二、ESLint常用规则

2.1 最佳实践规则

规则级别说明
no-varerror禁止使用var，使用let/const
prefer-consterror不会重新赋值的变量使用const
eqeqeqerror强制使用===和!==
no-unused-varserror禁止未使用的变量
no-undeferror禁止未声明的变量
no-consolewarn禁止console.log（允许warn/error）
no-debuggererror禁止debugger语句
no-alertwarn禁止使用alert/confirm/prompt
no-evalerror禁止使用eval
no-implied-evalerror禁止隐式eval（如setTimeout("code")）
no-param-reassignwarn禁止修改函数参数
no-return-awaitwarn禁止不必要的return await

2.2 TypeScript专用规则

规则说明
@typescript-eslint/no-explicit-any限制使用any类型
@typescript-eslint/consistent-type-imports强制使用import type
@typescript-eslint/no-non-null-assertion禁止使用!非空断言
@typescript-eslint/ban-ts-comment限制@ts-ignore等注释
@typescript-eslint/no-unnecessary-type-assertion禁止不必要的类型断言
@typescript-eslint/prefer-nullish-coalescing推荐使用??代替||
@typescript-eslint/prefer-optional-chain推荐使用?.链式调用

3.2 ESLint + Prettier 配置

// 安装依赖
// pnpm add -D eslint prettier eslint-config-prettier eslint-plugin-prettier

// eslint.config.js (Flat Config)
import prettierPlugin from 'eslint-plugin-prettier';
import prettierConfig from 'eslint-config-prettier';

export default [
    // ... 其他配置
    {
        plugins: { prettier: prettierPlugin },
        rules: {
            'prettier/prettier': 'error'  // Prettier格式问题报ESLint错误
        }
    },
    prettierConfig  // 关闭与Prettier冲突的ESLint规则
];

// 也可以只用eslint-config-prettier关闭冲突规则
// 不把Prettier作为ESLint规则运行（推荐）
// 格式化交给编辑器和prettier命令

四、Husky + Lint-staged

4.1 Husky（Git Hooks）

# 安装husky
pnpm add -D husky

# 初始化husky（创建.husky目录）
pnpm exec husky init

# 创建pre-commit hook
echo "pnpm exec lint-staged" > .husky/pre-commit

# 创建commit-msg hook（配合commitlint）
echo "pnpm exec commitlint --edit \$1" > .husky/commit-msg

# .husky/pre-commit 文件内容：
#!/usr/bin/env sh
. "$(dirname -- "$0")/_/husky.sh"
pnpm exec lint-staged

4.2 Lint-staged（只检查暂存文件）

// package.json
{
    "lint-staged": {
        "*.{js,jsx,ts,tsx}": [
            "eslint --fix",
            "prettier --write"
        ],
        "*.{json,md,yaml,yml}": [
            "prettier --write"
        ],
        "*.css": [
            "stylelint --fix",
            "prettier --write"
        ]
    }
}

// 或使用单独的配置文件 .lintstagedrc.js
export default {
    '*.{js,jsx,ts,tsx}': [
        'eslint --fix --max-warnings=0',  // 0个警告
        'prettier --write'
    ],
    '*.{json,md,yaml,yml,css,scss}': [
        'prettier --write'
    ]
};
工作流程：
  1. 开发者修改代码并 git add
  2. 执行 git commit
  3. Husky触发 pre-commit hook
  4. lint-staged只对暂存文件运行ESLint和Prettier
  5. 如果有错误，commit被阻止，修复后重新提交
  6. Husky触发 commit-msg hook，commitlint检查提交信息格式

五、Commitlint（提交信息规范）

5.1 Conventional Commits规范

# 提交信息格式：
# <type>(<scope>): <subject>
#
# <body>
#
# <footer>

# type类型：
feat:     新功能
fix:      修复bug
docs:     文档更新
style:    代码格式（不影响逻辑）
refactor: 重构
perf:     性能优化
test:     测试相关
build:    构建相关
ci:       CI配置
chore:    其他杂项
revert:   回退

# 示例：
feat(auth): 添加微信登录功能
fix(cart): 修复商品数量为负数的问题
docs(readme): 更新安装说明
refactor(api): 重构用户接口调用方式
feat!: 支持Node.js 18+（破坏性变更）

BREAKING CHANGE: 不再支持Node.js 16

5.2 Commitlint配置

// commitlint.config.js
export default {
    extends: ['@commitlint/config-conventional'],
    rules: {
        // type必须是以下之一
        'type-enum': [2, 'always', [
            'feat', 'fix', 'docs', 'style', 'refactor',
            'perf', 'test', 'build', 'ci', 'chore', 'revert'
        ]],
        // subject不能为空
        'subject-empty': [2, 'never'],
        // type不能为空
        'type-empty': [2, 'never'],
        // subject最大长度
        'subject-max-length': [2, 'always', 100],
        // scope大小写
        'scope-case': [2, 'always', 'lower-case']
    }
};

总结：ESLint负责代码质量检查，Prettier负责代码格式化，两者配合形成完整的代码规范。通过husky + lint-staged在Git提交时自动检查和格式化，commitlint规范提交信息格式，形成"提交即规范"的工程化体系。
