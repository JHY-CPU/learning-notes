Monorepo架构

核心概念：Monorepo是一种将多个项目/包放在同一个代码仓库中的管理方式。主流工具包括Turborepo（Vercel出品，增量构建+远程缓存）、Nx（功能全面，依赖图分析）、Lerna（历史最久，已逐渐被替代）。配合pnpm workspace的workspace:协议和依赖提升，形成高效的Monorepo开发体验。

一、Monorepo vs Polyrepo

1.1 两种模式对比

Monorepo（单仓库多项目）： Polyrepo（多仓库多项目）： repo/ repo-app/ ├── packages/ ├── src/ │ ├── ui/ └── package.json │ ├── utils/ repo-lib/ │ └── api-client/ ├── src/ ├── apps/ └── package.json │ ├── web/ repo-shared/ │ └── admin/ ├── src/ └── package.json └── package.json
特性MonorepoPolyrepo
代码共享容易（直接引用本地包）困难（需发布npm包）
依赖管理统一管理，版本一致各仓库独立管理
原子性变更支持（一次提交跨多个包）不支持
CI/CD可按需构建（受影响的包）各仓库独立CI
权限控制粒度粗粒度细（按仓库）
仓库大小大小
学习曲线需要工具支持简单直接

1.2 适用场景

  • Monorepo适合：共享组件库、全栈项目（前端+后端+移动端）、设计系统、多产品线共享基础设施
  • Polyrepo适合：独立项目、团队规模大需要细粒度权限、微服务架构
  • 知名Monorepo项目：React、Vue、Next.js、Babel、Vite、Angular

二、Turborepo

2.1 Turborepo核心特性

  • 增量构建：只构建发生变化的包及其依赖
  • 远程缓存：构建结果可以缓存到远程（Vercel/自建），CI/CD和开发者共享
  • 并行构建：自动分析依赖关系，并行执行无依赖的任务
  • 任务管道（Pipeline）：定义任务之间的依赖关系
  • 零配置：与pnpm/yarn/npm workspace无缝集成

2.3 turbo.json配置

// turbo.json
{
    "$schema": "https://turbo.build/schema.json",
    "globalDependencies": [
        ".env.*",           // 环境变量变化时重新构建
        "tsconfig.json"
    ],
    "globalEnv": [
        "NODE_ENV",
        "API_URL"
    ],
    "pipeline": {
        // build任务
        "build": {
            "dependsOn": ["^build"],  // 依赖先构建（上游包先build）
            "outputs": ["dist/**", ".next/**", ".vite/**"],
            "env": ["NODE_ENV", "API_URL"]
        },
        // dev任务（不缓存）
        "dev": {
            "cache": false,
            "persistent": true      // 持续运行的任务
        },
        // test任务
        "test": {
            "dependsOn": ["build"],
            "outputs": ["coverage/**"],
            "inputs": [
                "src/**/*.tsx",
                "src/**/*.ts",
                "test/**/*.ts"
            ]
        },
        // lint任务
        "lint": {
            "dependsOn": ["^build"],  // TypeScript类型检查需要上游构建结果
            "outputs": []
        },
        // typecheck任务
        "typecheck": {
            "dependsOn": ["^build"],
            "outputs": []
        }
    }
}

2.4 Turborepo常用命令

# 构建所有包（增量，只构建变化的）
turbo run build

# 并行运行所有包的dev
turbo run dev

# 只构建web应用及其依赖
turbo run build --filter=web

# 排除某些包
turbo run build --filter='!admin'

# 运行自上次提交以来变化的包的build
turbo run build --filter='[HEAD~1]'

# 强制重新构建（忽略缓存）
turbo run build --force

# 清理缓存
turbo prune --scope=web  # 只保留web所需的依赖

# 远程缓存
turbo login              # 登录Vercel
turbo link               # 关联远程缓存

三、Nx

3.1 Nx核心特性

  • 依赖图可视化：自动生成项目依赖关系图
  • 代码生成器（Generators）：快速生成组件、库、应用模板
  • 受影响分析：精确知道哪些项目受到代码变更影响
  • 分布式任务执行（DTE）：在CI中将任务分布到多台机器
  • 插件生态：React、Vue、Angular、Node.js等官方插件
# Nx常用命令
npx create-nx-workspace@latest my-org  # 创建workspace
nx graph                                # 查看依赖图
nx affected --target=test               # 运行受影响项目的测试
nx run-many --target=build --all        # 构建所有项目
nx generate @nx/react:lib my-lib        # 生成React库
nx g @nx/react:component MyComp --project=my-lib  # 生成组件

3.2 nx.json配置

// nx.json
{
    "targetDefaults": {
        "build": {
            "dependsOn": ["^build"],
            "cache": true
        },
        "test": {
            "dependsOn": ["build"],
            "inputs": ["default", "^default"],
            "cache": true
        }
    },
    "namedInputs": {
        "default": ["{projectRoot}/**/*", "!{projectRoot}/**/*.test.*"],
        "sharedGlobals": []
    }
}

四、Lerna（传统方案）

4.1 Lerna概述

Lerna是最早的前端Monorepo工具之一，现由Nx团队维护。新版Lerna底层使用Nx进行任务运行和缓存：

# 安装
npm install -g lerna

# 初始化
lerna init

# 发布所有变更的包
lerna publish          # 自动检测变更、更新版本、发布npm

# 运行所有包的脚本
lerna run build        # 构建所有包
lerna run test         # 测试所有包

# 在所有包中执行命令
lerna exec -- ls -la   # 在每个包目录执行ls

# 版本管理
lerna version          # 交互式选择版本号
lerna version --conventional-commits  # 基于conventional commits自动确定版本

4.2 lerna.json配置

// lerna.json
{
    "$schema": "node_modules/lerna/schemas/lerna-schema.json",
    "version": "independent",  // 独立版本（每个包独立版本号）
    // "version": "1.0.0",    // 固定版本（所有包统一版本号）
    "npmClient": "pnpm",
    "useWorkspaces": true,     // 使用package.json的workspaces
    "command": {
        "publish": {
            "conventionalCommits": true,
            "message": "chore(release): publish"
        },
        "version": {
            "conventionalCommits": true
        }
    }
}
Lerna现状：Lerna适合需要发布npm包的场景。对于纯应用型Monorepo，推荐使用Turborepo或Nx，它们的增量构建和缓存能力更强。

五、pnpm workspace:协议

5.1 workspace协议

// packages/ui/package.json
{
    "name": "@myorg/ui",
    "version": "1.0.0",
    "dependencies": {
        // workspace:* → 始终使用本地版本
        "@myorg/utils": "workspace:*",

        // workspace:~ → 发布时转换为 ~1.0.0
        "@myorg/core": "workspace:~",

        // workspace:^ → 发布时转换为 ^1.0.0（推荐）
        "@myorg/shared": "workspace:^"
    }
}

// workspace协议在不同阶段的行为：
// 开发时：始终指向本地包
// 发布npm时：workspace:* 会被替换为实际版本号
// workspace:^ → "^1.0.0"
// workspace:~ → "~1.0.0"
// workspace:* → "1.0.0"（精确版本）

5.2 依赖提升配置

// .npmrc
// pnpm配置
shamefully-hoist=false        // 不提升依赖（严格模式）
// shamefully-hoist=true      // 模仿npm/yarn的扁平结构

// 选择性提升
public-hoist-pattern[]=*eslint*
public-hoist-pattern[]=*prettier*
auto-install-peers=true       // 自动安装peerDependencies

// package.json 配置不提升的依赖
{
    "pnpm": {
        "neverBuiltDependencies": ["fsevents"],
        "peerDependencyRules": {
            "ignoreMissing": ["@types/*"]
        }
    }
}

六、任务管道与缓存

6.1 依赖图与执行顺序

依赖关系： @myorg/utils ← @myorg/ui ← web @myorg/utils ← @myorg/api-client ← web @myorg/ui ← admin 构建顺序（拓扑排序）： 1. @myorg/utils （无依赖） 2. @myorg/ui （依赖utils） 3. @myorg/api-client （依赖utils） 4. web （依赖ui, api-client） 5. admin （依赖ui） Turborepo自动按此顺序构建，无依赖的包并行执行

6.2 缓存策略

// Turborepo缓存原理：
// 输入：源代码 + 依赖 + 环境变量 + 配置文件
// → 计算hash
// → 检查本地/远程缓存
// → 命中：跳过构建，恢复输出文件
// → 未命中：执行构建，缓存结果

// 增量构建示例：
// 只修改了 packages/utils/src/format.ts
// 1. @myorg/utils: 未命中缓存 → 重新build
// 2. @myorg/ui: 依赖utils → 重新build
// 3. @myorg/api-client: 依赖utils → 重新build
// 4. web: 依赖ui和api-client → 重新build
// 5. admin: 只依赖ui → 重新build
// 6. packages/tsconfig: 无变化 → 命中缓存，跳过
// 7. packages/eslint-config: 无变化 → 命中缓存，跳过

七、Monorepo工具选型

7.1 工具对比

特性TurborepoNxLerna
学习曲线低中等低
增量构建优秀优秀依赖Nx
远程缓存原生支持原生支持依赖Nx
分布式执行Vercel支持原生支持不支持
依赖图基础可视化优秀基础
代码生成无强大无

生态Vercel生态丰富插件维护模式

7.2 选型建议

  • Turborepo：首选。简单易用，与pnpm workspace配合完美，Vercel部署无缝集成
  • Nx：大型团队、需要代码生成器、分布式任务执行、依赖图可视化
  • Lerna：只需要发布npm包的场景（结合Changesets更现代）
  • pnpm workspace alone：简单Monorepo，不需要高级构建优化

7.3 现代推荐方案

// 最佳实践组合：
// pnpm workspace + Turborepo + Changesets

// 1. pnpm workspace 管理依赖和本地包引用
// 2. Turborepo 管理任务编排和缓存
// 3. Changesets 管理版本号和CHANGELOG

// 使用Changesets管理版本
// 安装
pnpm add -D @changesets/cli

// 初始化
pnpm changeset init

// 添加变更集（交互式）
pnpm changeset
// 选择哪些包有变更
// 选择版本号（patch/minor/major）
// 填写变更描述

// 应用变更集，更新版本号
pnpm changeset version

// 发布到npm
pnpm changeset publish
完整工程化体系：
  • 包管理：pnpm workspace 管理依赖和包引用
  • 构建编排：Turborepo 增量构建、并行执行、远程缓存
  • 代码规范：ESLint + Prettier + husky + commitlint
  • 版本发布：Changesets 管理版本和CHANGELOG
  • CI/CD：GitHub Actions + Turborepo远程缓存
  • 类型安全：共享tsconfig，workspace协议引用类型
总结：Monorepo是管理多包/多项目代码的有效方式。Turborepo以简单易用为核心优势，配合pnpm workspace形成最佳组合。Nx适合需要高级功能的大型团队。Lerna专注于npm包发布。核心理念是通过依赖图分析实现增量构建和智能缓存，大幅提升CI/CD和开发体验。
