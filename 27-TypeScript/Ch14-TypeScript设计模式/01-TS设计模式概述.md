# TS设计模式概述

## 一、概念说明

TypeScript 的类型系统让设计模式更加安全和可维护。类型化的模式能在编译时捕获错误，减少运行时异常。TypeScript 支持的经典模式包括创建型、结构型和行为型模式。

## 二、具体用法

### 2.1 类型安全的核心价值

```typescript
// 没有类型的设计模式 — 运行时才能发现错误
// const handler = strategyMap[action]; // 可能是 undefined

// 有类型的设计模式 — 编译时就发现错误
type Action = 'save' | 'delete' | 'update';
const handler: Record<Action, () => void> = {
  save: () => {},
  delete: () => {},
  update: () => {},
};
// handler['unknown']; // 编译错误：类型 '"unknown"' 的参数不能赋给 '"save" | "delete" | "update"'
```

### 2.2 常用模式分类

| 类型 | 模式 | TypeScript 特色 |
|------|------|----------------|
| 创建型 | 单例、工厂、建造者 | 泛型约束产品类型 |
| 结构型 | 适配器、装饰器、代理 | 类型转换与约束 |
| 行为型 | 策略、观察者、状态机 | Discriminated Union |

### 2.3 TypeScript 特有模式

```typescript
// 1. 品牌类型（Branded Types）
type PositiveNumber = number & { __brand: 'Positive' };

// 2. Result 类型
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

// 3. Option 类型
type Option<T> = { some: true; value: T } | { some: false };

// 4. 类型安全状态机
type State = 'idle' | 'loading' | 'success' | 'error';
type Event = 'FETCH' | 'RESOLVE' | 'REJECT' | 'RETRY';
```

## 三、注意事项与常见陷阱

1. **不要为了用模式而用模式**：简单场景用简单方案
2. **类型是编译时的**：运行时仍需要验证
3. **泛型让模式更通用**：但不要过度抽象
4. **Discriminated Union 是 TS 最强大的模式之一**
5. **优先考虑函数式模式**：比类模式更简洁
