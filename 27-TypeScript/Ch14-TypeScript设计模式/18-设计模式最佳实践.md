# 设计模式最佳实践

## 一、概念说明

选择合适的设计模式需要权衡可读性、可维护性和类型安全。TypeScript 的类型系统可以简化某些模式（如策略、状态机），但也可能引入不必要的复杂度。

## 二、具体用法

### 2.1 模式选择指南

```typescript
// 场景一：条件分支 → Discriminated Union
// 错误：用类实现简单的条件逻辑
// class Circle { ... }
// class Rectangle { ... }

// 正确：用联合类型
type Shape =
  | { type: 'circle'; radius: number }
  | { type: 'rectangle'; width: number; height: number };

// 场景二：数据转换 → 函数组合
// 错误：用建造者模式转换数据
// 正确：用管道函数
const process = (data: string) =>
  data.trim().toLowerCase().split(',').map(Number);

// 场景三：事件处理 → 观察者模式
type Events = { click: [x: number, y: number]; resize: [w: number, h: number] };
```

### 2.2 何时不用模式

```typescript
// 不要：过度设计
class SingletonFactoryBuilderManager {
  // 只是创建一个对象...
}

// 要：简单直接
const config = {
  apiUrl: process.env.API_URL,
  timeout: 5000,
};
```

### 2.3 可读性优先

```typescript
// 模式带来的类型安全很重要
// 但可读性更重要

// 清晰
function getStatusLabel(status: 'active' | 'inactive'): string {
  return status === 'active' ? '启用' : '禁用';
}

// 过度设计
abstract class StatusLabelStrategy { ... }
class ActiveStatusLabel extends StatusLabelStrategy { ... }
class InactiveStatusLabel extends StatusLabelStrategy { ... }
```

### 2.4 TypeScript 特色模式

```typescript
// 1. Discriminated Union — 最常用
type Result<T, E> = { ok: true; value: T } | { ok: false; error: E };

// 2. 品牌类型 — 编译时约束
type PositiveNumber = number & { __brand: 'Positive' };

// 3. 类型守卫 — 运行时收窄
function isError(value: unknown): value is Error {
  return value instanceof Error;
}

// 4. satisfies — 保持字面量类型
const routes = {
  home: '/',
  about: '/about',
} satisfies Record<string, string>;
```

## 三、注意事项与常见陷阱

1. **简单问题用简单方案**：不要为了用模式而用模式
2. **Discriminated Union 是 TS 最强大的模式**：替代复杂的类层次结构
3. **函数优于类**：大多数场景函数就够了
4. **类型安全很重要，但可读性更重要**
5. **团队协作时保持模式一致**：约定比正确更重要
