# 类型兼容性矩阵

## 一、概念说明

TypeScript 版本之间的类型兼容性涉及语言特性、API 类型和编译输出。了解兼容性矩阵有助于规划版本升级和确保代码在不同版本下的可移植性。

## 二、具体用法

### 2.1 版本间兼容性

```typescript
// TypeScript 5.0+ 的新语法在旧版本中不可用
// const 类型参数 — 5.0+
function f<const T>(value: T): T { return value; }

// NoInfer — 5.4+
type X = NoInfer<string>;

// using — 5.2+
using resource = getResource();
```

### 2.2 lib 兼容性

```json
{
  "compilerOptions": {
    // ES2022 包含：Array.at(), Object.hasOwn(), Error.cause 等
    "lib": ["ES2022", "DOM", "DOM.Iterable"],

    // ESNext — 始终包含最新的 ECMAScript 特性
    "lib": ["ESNext", "DOM"]
  }
}
```

### 2.3 类型定义兼容性

```typescript
// @types/node 版本要与 Node.js 版本匹配
// Node.js 18 → @types/node@18
// Node.js 20 → @types/node@20

// @types/react 版本要与 React 版本匹配
// React 18 → @types/react@18
```

### 2.4 输出兼容性

```json
{
  "compilerOptions": {
    // target 决定输出的 JS 版本
    "target": "ES2020",   // 输出 ES2020 语法
    "target": "ES2022",   // 输出 ES2022 语法
    "target": "ESNext"    // 输出最新的 JS 语法
  }
}
```

### 2.5 兼容性检查工具

```bash
# 使用 ts-bridge 检查类型兼容性
npx ts-bridge@latest

# 使用 attw 检查包的类型兼容性
npx @arethetypeswrong/cli --pack .
```

## 三、注意事项与常见陷阱

1. **`target` 和 `lib` 要匹配运行时**：Node.js 18 支持 ES2022
2. **`@types/*` 版本锁定**：避免类型不匹配
3. **新特性需要对应的 TypeScript 版本**：检查版本要求
4. **编译输出的 JS 版本要与目标环境匹配**
5. **使用 `skipLibCheck` 加快编译**：但可能隐藏类型问题
