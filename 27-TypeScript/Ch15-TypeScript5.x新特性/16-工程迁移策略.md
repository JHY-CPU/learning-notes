# 工程迁移策略

## 一、概念说明

TypeScript 版本升级需要系统性的策略，确保类型兼容性和不引入破坏性变更。从 TypeScript 4.x 到 5.x 的迁移需要关注装饰器、枚举、模块解析等变化。

## 二、具体用法

### 2.1 迁移步骤

```bash
# 1. 检查当前版本
npx tsc --version

# 2. 更新 TypeScript
npm install -D typescript@latest

# 3. 运行类型检查
npx tsc --noEmit

# 4. 修复类型错误
# 通常会有 10-50 个新的类型错误需要修复
```

### 2.2 常见的破坏性变更

```typescript
// 1. 装饰器变更
// 旧：experimentalDecorators: true
// 新：移除 experimentalDecorators，使用 Stage 3 装饰器

// 2. 枚举变更
// 旧：非联合枚举
// 新：所有枚举变为联合枚举

// 3. 模块解析
// 旧：moduleResolution: "node"
// 新：moduleResolution: "bundler"
```

### 2.3 渐进式迁移

```json
// 第一阶段：更新 tsconfig
{
  "compilerOptions": {
    "strict": true,
    "skipLibCheck": true,
    "noUncheckedIndexedAccess": false // 逐步开启
  }
}

// 第二阶段：开启更严格的检查
{
  "compilerOptions": {
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true
  }
}
```

### 2.4 兼容性矩阵

| TypeScript | Node.js | 最低 ECMAScript |
|-----------|---------|-----------------|
| 5.0+ | 12.20+ | ES2020 |
| 5.4+ | 14+ | ES2020 |
| 5.6+ | 18+ | ES2022 |

### 2.5 迁移检查清单

```markdown
- [ ] 运行 tsc --noEmit，记录所有新错误
- [ ] 检查装饰器是否需要重写
- [ ] 检查 enum 是否需要改为联合类型
- [ ] 更新 @types/* 包版本
- [ ] 检查第三方库的 TS 兼容性
- [ ] 运行所有测试
- [ ] 检查构建输出是否正确
```

## 三、注意事项与常见陷阱

1. **先在分支上测试新版本**：不要直接在 main 上升级
2. **TypeScript 用 `~` 锁定 minor 版本**：避免意外更新
3. **装饰器迁移成本最高**：需要重写装饰器代码
4. **`skipLibCheck: true`**：加快编译，跳过第三方类型检查
5. **保留旧版本的 tsconfig 备份**
