# 声明文件最佳实践

## 一、概念说明

编写和维护声明文件需要遵循一系列最佳实践，确保类型声明的正确性、可维护性和兼容性。从文件组织到类型设计，每个环节都影响开发体验。

## 二、具体用法

### 2.1 文件组织

```
project/
├── src/
│   ├── types/
│   │   ├── index.ts           # 统一导出
│   │   ├── api.ts             # API 类型
│   │   ├── models.ts          # 数据模型
│   │   ├── env.d.ts           # 环境变量
│   │   └── global.d.ts        # 全局扩展
│   └── ...
├── custom-types/              # 第三方库自定义类型
│   └── my-lib/
│       └── index.d.ts
├── tsconfig.json
└── package.json
```

### 2.2 类型导出策略

```typescript
// types/index.ts — 统一导出
export type { User, CreateUserDto, UpdateUserDto } from './user';
export type { ApiResponse, PaginatedResponse } from './api';
export type { AppConfig, FeatureFlags } from './config';

// 值和类型分开导出
export { AppError, ValidationError } from './errors';
export type { ErrorDetails, ErrorCode } from './errors';
```

### 2.3 编写声明文件的原则

```typescript
// 1. 精确类型，避免 any
// 错误
declare function process(data: any): any;

// 正确
declare function process<T>(data: T): Processed<T>;

// 2. 使用接口描述对象
interface User {
  id: number;
  name: string;
  email: string;
}

// 3. 使用联合类型限制取值
type Status = 'active' | 'inactive' | 'pending';

// 4. 回调函数有完整签名
type EventHandler = (event: Event, data: EventData) => void;

// 5. 泛型增加灵活性
declare function map<T, U>(items: T[], fn: (item: T) => U): U[];
```

### 2.4 维护策略

```typescript
// 1. 使用 JSDoc 注释
/**
 * 创建新用户
 * @param data - 用户数据
 * @returns 创建的用户对象
 * @throws {ValidationError} 数据验证失败时抛出
 */
declare function createUser(data: CreateUserDto): Promise<User>;

// 2. 标注废弃
/**
 * @deprecated 请使用 `getUserById` 替代
 */
declare function findUser(id: number): Promise<User | null>;

// 3. 版本标注
/**
 * @since 2.0.0
 */
declare function newFeature(): void;
```

### 2.5 测试声明文件

```typescript
// 测试文件 — 验证类型是否正确
import { createUser, type CreateUserDto } from 'my-lib';

// 正确用法
const user = await createUser({
  name: '张三',
  email: 'test@example.com',
});

// 错误用法（应报错）
// createUser({ name: 123 }); // 类型错误

// 测试类型导出
import type { ApiResponse, User } from 'my-lib';
const response: ApiResponse<User> = {
  success: true,
  data: user,
};
```

### 2.6 发布检查

```json
// package.json
{
  "types": "dist/index.d.ts",
  "exports": {
    ".": {
      "types": "dist/index.d.ts",
      "import": "dist/index.mjs",
      "require": "dist/index.js"
    }
  },
  "files": ["dist"],
  "scripts": {
    "prepublishOnly": "npm run type-check && npm run test",
    "type-check": "tsc --noEmit"
  }
}
```

## 三、注意事项与常见陷阱

1. **类型声明要与运行时行为一致**：确保类型描述准确
2. **不要在声明文件中泄露内部类型**：只暴露公共 API
3. **使用 `export type` 确保 `isolatedModules` 兼容**
4. **声明文件要随包一起发布**：`files` 字段包含 `dist`
5. **定期检查 @types 包的更新**：确保类型与库版本匹配
