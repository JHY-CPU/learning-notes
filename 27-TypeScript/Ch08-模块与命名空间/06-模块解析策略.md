# 模块解析策略

## 一、概念说明

模块解析策略决定 TypeScript 如何查找 `import` 语句引用的模块。`tsconfig.json` 的 `moduleResolution` 选项控制解析行为，常见值包括 `bundler`（推荐现代前端项目）、`node16`（Node.js ESM）、`node`（Node.js CJS）和 `classic`（旧式）。选择正确的策略确保 TypeScript 的模块解析与构建工具和运行时一致。

## 二、具体用法

### 2.1 策略对比

```json
// tsconfig.json
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "Bundler"
  }
}
```

| 策略 | 适用场景 | 扩展名要求 | exports 支持 | index 解析 |
|------|----------|-----------|-------------|-----------|
| `bundler` | Vite/Webpack | 不要求 | 完整 | 是 |
| `node16` | Node.js ESM | 必须 `.js` | 完整 | 否 |
| `node` | Node.js CJS | 不要求 | 部分 | 是 |
| `classic` | AMD/SystemJS | 不要求 | 否 | 否 |

### 2.2 Bundler 策略（推荐）

```typescript
// bundler 模式下 — 不需要扩展名
import { add } from "./math";           // OK
import { config } from "./config";      // OK
import { User } from "@/types/user";    // OK（如果配置了 paths）

// 支持 package.json 的 exports 字段
import { createApp } from "vue";        // 正确解析 vue 的 exports
```

### 2.3 Node16 策略

```typescript
// node16 模式下 — 必须写扩展名
import { add } from "./math.js";        // OK
// import { add } from "./math";        // 错误

// 支持 package.json exports
import { something } from "my-package"; // 解析 my-package 的 exports 字段
```

### 2.4 解析过程示例

```typescript
// import { helper } from "./utils";
// bundler 解析顺序:
// 1. ./utils.ts
// 2. ./utils.tsx
// 3. ./utils.js
// 4. ./utils.jsx
// 5. ./utils/index.ts
// 6. ./utils/index.tsx
// 7. ./utils/index.js
// 8. ./utils/index.jsx

// node16 解析顺序:
// 1. ./utils.js（必须有扩展名）
// 2. ./utils.js/index.js
```

### 2.5 自定义解析（高级）

```json
// tsconfig.json — 配合 tsconfig-paths
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  }
}
```

```bash
# 运行时解析路径映射
npx ts-node -r tsconfig-paths/register src/main.ts
```

### 2.6 与 JavaScript 的对比

```javascript
// Node.js ESM：必须写扩展名
import { add } from "./math.js";

// Node.js CJS：不需要扩展名
const { add } = require("./math");

// 浏览器 ESM：必须写完整路径
import { add } from "./math.js";

// TypeScript bundler：模拟打包工具行为，最灵活
import { add } from "./math"; // 自动查找
```

## 三、注意事项与常见陷阱

1. **`bundler` 最灵活**：Vite、Webpack、esbuild 项目推荐，不需要扩展名
2. **`node16` 最严格**：Node.js 原生 ESM 项目使用，要求 `.js` 扩展名
3. **`module` 和 `moduleResolution` 必须搭配**：`ESNext` + `Bundler`，`Node16` + `Node16`
4. **路径映射需配套配置**：`paths` 需要 `baseUrl`，构建工具也需要配置别名
5. **`exports` 字段中的 `types` 条件**：必须放在最前面，否则类型解析可能失败
6. **`typeVersions` 降级方案**：旧版 Node.js 不支持 `exports` 时用 `typesVersions`
