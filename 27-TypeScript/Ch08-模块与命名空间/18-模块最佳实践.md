# 模块最佳实践

## 一、概念说明

遵循模块最佳实践可以确保项目结构清晰、类型安全且性能良好。核心原则：**ESM 优先**、**避免 namespace**、**合理组织文件结构**、**善用类型导入**。这些实践不仅影响代码质量，还影响 tree-shaking 效果和编译速度。

## 二、具体用法

### 2.1 ESM 优先配置

```json
// tsconfig.json — 推荐配置
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "target": "ES2020",
    "esModuleInterop": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true
  }
}
```

### 2.2 合理的文件组织

```
src/
  types/              # 共享类型定义
    user.ts
    api.ts
    index.ts          # barrel 导出
  utils/              # 工具函数
    math.ts
    string.ts
    index.ts
  services/           # 业务逻辑
    user-service.ts
    auth-service.ts
  components/         # UI 组件
    Button.tsx
    Modal.tsx
  index.ts            # 入口文件
```

```typescript
// src/types/index.ts — barrel 文件
export type { User, CreateUserDto } from "./user.js";
export type { ApiResponse, PaginatedResponse } from "./api.js";

// 使用方统一导入
import type { User, ApiResponse } from "@/types";
```

### 2.3 类型导入

```typescript
// ✅ 类型用 import type
import type { User, Config } from "./types.js";

// ✅ 混合导入用内联语法
import { UserService, type User } from "./services/user-service.js";

// ❌ 不要混用值和类型
import { User } from "./types.js"; // User 只是类型，应用 import type
```

### 2.4 避免 namespace

```typescript
// ❌ 不要用 namespace
namespace Utils {
  export function add(a: number, b: number) { return a + b; }
}

// ✅ 使用 ES Module
export function add(a: number, b: number): number {
  return a + b;
}
```

### 2.5 避免循环依赖

```typescript
// ❌ 循环依赖：A -> B -> A
// user.ts
import { Post } from "./post.js";
export interface User { posts: Post[]; }

// post.ts
import { User } from "./user.js";
export interface Post { author: User; }

// ✅ 解决方案：提取公共类型
// types.ts
export interface User { id: number; name: string; }
export interface Post { id: number; title: string; authorId: number; }
```

### 2.6 Barrel 文件的取舍

```typescript
// ✅ 小型项目：barrel 文件方便导入
export { UserService } from "./services/user-service.js";
export type { User } from "./types/user.js";

// ❌ 大型项目：barrel 可能影响 tree-shaking
// 逐模块导入可能更高效
import { UserService } from "./services/user-service.js";
```

### 2.7 与 JavaScript 的对比

```javascript
// JavaScript：无类型组织，靠约定
const { helper } = require("./utils");

// TypeScript：类型 + 模块组织
import type { User } from "@/types";
import { UserService } from "@/services/user-service";
// 编译时保证类型一致，IDE 提供补全
```

## 三、注意事项与常见陷阱

1. **ESM 优先**：不要用 namespace，用 ES Module 的 `import`/`export`
2. **`import type`**：只导入类型时使用 `import type`，减小 bundle 体积
3. **避免循环依赖**：A -> B -> A 的导入链会导致运行时问题和类型推断错误
4. **Barrel 文件谨慎使用**：大型项目 barrel 可能导致打包体积增大、影响 tree-shaking
5. **一个模块一个职责**：每个文件只做一件事，保持模块小而聚焦
6. **路径别名适度**：2-3 个别名（如 `@/`、`@types/`）即可，过多增加认知成本
7. **`verbatimModuleSyntax`**：TS 5.0+ 推荐开启，强制严格区分值导入和类型导入
