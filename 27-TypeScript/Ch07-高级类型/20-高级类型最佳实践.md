# 高级类型最佳实践

## 一、概念说明

高级类型（条件类型、映射类型、递归类型、模板字面量类型等）非常强大，但过度使用会导致代码**难以理解和维护**。遵循最佳实践，在类型安全和可读性之间找到平衡，是成为高级 TypeScript 开发者的关键。

## 二、具体用法

### 2.1 何时使用高级类型

```typescript
// ✅ 适合：通用可复用的工具类型
type DeepReadonly<T> = T extends object
  ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
  : T;

// ✅ 适合：框架/库层面的类型设计
type ExtractRouteParams<T extends string> =
  T extends `${string}:${infer Param}/${infer Rest}`
    ? { [K in Param]: string } & ExtractRouteParams<`/${Rest}`>
    : T extends `${string}:${infer Param}` ? { [K in Param]: string } : {};

// ❌ 不适合：一次性使用的简单类型
// 直接用 interface 定义更清晰
// interface User { id: number; name: string; }
// 不需要: type User = { id: number; name: string; }
```

### 2.2 命名和文档

```typescript
/**
 * 递归地将 T 的所有属性变为只读
 * 函数类型保持不变
 * @template T - 要冻结的对象类型
 * @example
 * type Frozen = DeepReadonly<{ a: { b: string } }>;
 * // { readonly a: { readonly b: string } }
 */
type DeepReadonly<T> = T extends (...args: any[]) => any
  ? T
  : T extends object
    ? { readonly [K in keyof T]: DeepReadonly<T[K]> }
    : T;
```

### 2.3 类型测试

```typescript
// 编译时类型断言
type Expect<T extends true> = T;
type Equal<X, Y> =
  (<T>() => T extends X ? 1 : 2) extends
  (<T>() => T extends Y ? 1 : 2) ? true : false;

// 测试 DeepReadonly
type _test1 = Expect<Equal<
  DeepReadonly<{ a: { b: string } }>,
  { readonly a: { readonly b: string } }
>>;

// 测试边界情况
type _test2 = Expect<Equal<DeepReadonly<() => void>, () => void>>; // 函数不变
type _test3 = Expect<Equal<DeepReadonly<string>, string>>;         // 基本类型不变
```

### 2.4 性能优化

```typescript
// ❌ 避免无限制递归
// type Bad<T> = T extends object ? { [K in keyof T]: Bad<T[K]> } : T;
// 深层嵌套会显著拖慢编译

// ✅ 限制递归深度
type Prev = [never, 0, 1, 2, 3, 4, ...0[]];
type DeepPartial<T, D extends number = 5> = [D] extends [never]
  ? T
  : T extends object
    ? { [K in keyof T]?: DeepPartial<T[K], Prev[D]> }
    : T;

// ✅ 缓存中间类型
type ProcessedUser = DeepReadonly<User>; // 一次计算，多次使用
```

### 2.5 渐进采用策略

```typescript
// 第一步：简单工具类型
type Nullable<T> = T | null;
type Maybe<T> = T | undefined;

// 第二步：条件类型
type ApiResponse<T> = T extends void
  ? { success: boolean }
  : { success: boolean; data: T };

// 第三步：映射类型
type ReadonlyPartial<T> = { readonly [K in keyof T]?: T[K] };

// 第四步：递归 + infer（复杂场景）
type DeepMerge<A, B> = A extends object
  ? B extends object
    ? { [K in keyof A | keyof B]:
        K extends keyof B ? K extends keyof A ? DeepMerge<A[K], B[K]> : B[K]
        : K extends keyof A ? A[K] : never }
    : B
  : B;
```

### 2.6 与 JavaScript 的对比

```javascript
// JavaScript：运行时工具函数
function deepClone(obj) { return JSON.parse(JSON.stringify(obj)); }
function pick(obj, keys) { /* ... */ }

// TypeScript：编译时工具类型
// type DeepClone<T> = ...  // 类型层面的深拷贝
// type Pick<T, K> = { [P in K]: T[P] }
// 编译时保证类型正确，零运行时开销
```

## 三、注意事项与常见陷阱

1. **可读性优先**：代码是给人读的，类型也是。如果团队成员看不懂，类型设计就失败了
2. **添加注释**：复杂类型必须有 JSDoc 注释说明意图、参数含义和使用示例
3. **类型测试**：使用 `Expect<Equal<...>>` 确保类型在各种输入下正确
4. **性能监控**：编译变慢时，检查是否有过度递归或不必要的复杂类型
5. **渐进采用**：从简单类型操作开始，逐步学习和使用高级特性
6. **工具优先**：优先使用内置工具类型（`Partial`、`Pick`、`Omit` 等），只在内置类型不够时自定义
