# 工程化最佳实践

## 一、概念说明

TypeScript 项目的工程化最佳实践涵盖了配置模板、CI/CD 集成、团队协作规范等方面。遵循这些实践能确保项目从创建到维护都有良好的类型安全保障。

## 二、具体用法

### 2.1 配置模板

```json
// 标准 tsconfig.json 模板
{
  "compilerOptions": {
    "strict": true,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "jsx": "react-jsx",
    "noEmit": true,
    "isolatedModules": true,
    "skipLibCheck": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noFallthroughCasesInSwitch": true,
    "forceConsistentCasingInFileNames": true,
    "resolveJsonModule": true,
    "esModuleInterop": true,
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

### 2.2 标准 package.json

```json
{
  "scripts": {
    "dev": "vite",
    "build": "tsc && vite build",
    "type-check": "tsc --noEmit",
    "lint": "eslint src --ext .ts,.tsx",
    "lint:fix": "eslint src --ext .ts,.tsx --fix",
    "format": "prettier --write \"src/**/*.{ts,tsx,json,css}\"",
    "format:check": "prettier --check \"src/**/*.{ts,tsx,json,css}\"",
    "test": "vitest",
    "test:run": "vitest run",
    "test:coverage": "vitest run --coverage",
    "precommit": "lint-staged"
  }
}
```

### 2.3 CI/CD 流程

```yaml
# 完整 CI 流程
jobs:
  check:
    steps:
      - checkout
      - setup-node
      - install
      - type-check    # 类型检查
      - lint          # 代码规范
      - format-check  # 格式检查
      - test          # 单元测试
      - build         # 构建
      - deploy        # 部署
```

### 2.4 团队规范

```markdown
# TypeScript 团队规范
1. 始终使用 strict 模式
2. 不使用 any，用 unknown + 类型守卫
3. Props 用 interface，内部用 type
4. 导出类型用 export type
5. 常量用 as const
6. 函数返回类型通常不标注（自动推断）
7. 复杂类型用工具类型简化
```

### 2.5 版本管理

```json
{
  "devDependencies": {
    "typescript": "~5.4.0",
    "@typescript-eslint/eslint-plugin": "^7.0.0",
    "eslint": "^9.0.0"
  }
}
```

## 三、注意事项与常见陷阱

1. **TypeScript 用 `~` 锁定 minor 版本**：避免类型定义不兼容
2. **CI 中不要跳过类型检查**：这是最重要的质量关卡
3. **新成员入职文档**：包含环境搭建和规范说明
4. **定期更新依赖**：但先在分支上测试
5. **配置文件要版本控制**：`.editorconfig`、`.prettierrc` 等
