# 类型断言

## 一、概念说明

类型断言（Type Assertion）告诉编译器"我知道这个值的类型比你推断的更具体"。有两种语法：`as 类型` 和尖括号 `<类型>`。类型断言不改变运行时值，只影响编译时类型检查。它是 TypeScript 中"手动覆盖类型推断"的手段，应谨慎使用。

## 二、具体用法

### 2.1 as 语法

```typescript
// 从宽泛类型断言为具体类型
const someValue: unknown = "Hello, TypeScript";
const str: string = someValue as string;

console.log(str.toUpperCase());
// 输出: HELLO, TYPESCRIPT
```

### 2.2 DOM 元素断言

```typescript
// getElementById 返回 HTMLElement | null
const input = document.getElementById("username") as HTMLInputElement;

// 现在可以访问 input 特有的属性
input.value = "默认值";
console.log(input.value);
// 输出: 默认值
```

### 2.3 非空断言操作符 `!`

```typescript
// ! 告诉编译器"这个值不是 null 或 undefined"
function processUser(name: string | undefined) {
  // 非空断言：跳过 undefined 检查
  const upper = name!.toUpperCase();
  console.log(upper);
}

// ⚠️ 如果 name 是 undefined，运行时会崩溃
processUser("alice");
// 输出: ALICE
```

### 2.4 双重断言

```typescript
// 当 as 直接断言不兼容时，先断言为 unknown 再断言为目标类型
const num = 42;
const str2 = num as unknown as string; // 一般应避免

console.log(typeof str2); // 运行时仍是 number
// 输出: number
```

### 2.5 const 断言

```typescript
// as const —— 字面量断言，不同于类型断言
const config = {
  port: 3000,
  host: "localhost",
} as const;
// 类型变为 { readonly port: 3000; readonly host: "localhost" }

// 数组 as const 变为只读元组
const colors = ["red", "green", "blue"] as const;
// 类型变为 readonly ["red", "green", "blue"]
```

### 2.6 更安全的替代方案

```typescript
// 使用类型守卫替代非空断言
function processUserSafe(name: string | undefined) {
  if (name === undefined) {
    console.log("名称未提供");
    return;
  }
  // name 被收窄为 string，不需要 !
  console.log(name.toUpperCase());
}

// 使用类型谓词替代 as
function isInputElement(el: HTMLElement): el is HTMLInputElement {
  return el instanceof HTMLInputElement;
}
```

## 三、注意事项与常见陷阱

1. **类型断言不是转换**：`42 as string` 不会将数字变成字符串，只是骗过编译器
2. **避免非空断言**：`!` 会跳过安全检查，尽量用类型守卫替代
3. **双重断言是危险信号**：说明类型设计有问题，应重新审视类型定义
4. **`const 断言` vs `类型断言`**：`as const` 是字面量断言，使值变为只读字面量类型
5. **不能断言为完全不兼容的类型**：`string as number` 会报错，需要先断言为 `unknown`
6. **慎用 `as any`**：这完全关闭了类型检查，等同于放弃 TypeScript 的优势
