# 消息认证码MAC


## 消息认证码 MAC


密码学HMAC认证


消息认证码（MAC）使用共享密钥为消息生成认证标签，提供数据完整性和消息来源认证。


## 消息认证码基础


```
MAC 提供的安全性：
┌─────────────────────────────────────────────────────────┐
│  1. 数据完整性 (Integrity)                                │
│     消息在传输过程中未被篡改                               │
│                                                         │
│  2. 消息来源认证 (Authentication)                         │
│     消息确实来自持有共享密钥的发送方                       │
│                                                         │
│  注意：MAC 不提供不可否认性 (Non-repudiation)             │
│  → 因为收发双方共享密钥，接收方也可以伪造 MAC              │
│  → 需要不可否认性时使用数字签名                           │
└─────────────────────────────────────────────────────────┘

MAC vs 数字签名 vs 哈希：
┌──────────┬──────────────┬──────────────┬──────────────┐
│ 特性      │ MAC          │ 数字签名      │ 纯哈希       │
├──────────┼──────────────┼──────────────┼──────────────┤
│ 密钥      │ 共享对称密钥  │ 公钥/私钥对   │ 无密钥       │
│ 完整性    │ ✅           │ ✅           │ ✅           │
│ 身份认证  │ ✅           │ ✅           │ ❌           │
│ 不可否认  │ ❌           │ ✅           │ ❌           │
│ 速度      │ 快           │ 慢           │ 最快         │
│ 密钥管理  │ 需安全分发    │ 公钥可公开    │ 无需         │
│ 适用场景  │ 通信认证      │ 法律签名      │ 完整性校验   │
└──────────┴──────────────┴──────────────┴──────────────┘

MAC 应用场景：
- TLS/SSH 中的消息认证
- API 请求签名（HMAC-SHA256）
- 文件完整性验证（HMAC）
- 磁盘加密中的数据认证（XTS 中的 tweakable cipher）
- 无线通信认证（WPA2 的 CCMP 使用 CMAC）
```


## HMAC (Hash-based MAC)


```
HMAC 定义 (RFC 2104)：
┌─────────────────────────────────────────────────────────┐
│  HMAC(K, M) = H((K' ⊕ opad) || H((K' ⊕ ipad) || M))  │
│                                                         │
│  其中：                                                  │
│  K  = 密钥                                              │
│  M  = 消息                                              │
│  H  = 哈希函数（如 SHA-256）                             │
│  K' = K 填充到块大小（如 64 字节）                       │
│  ipad = 0x36 重复（与块大小相同）                        │
│  opad = 0x5C 重复（与块大小相同）                        │
│  || = 拼接                                              │
└─────────────────────────────────────────────────────────┘

HMAC 计算过程：
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  消息 M ─────────────────────────────────┐              │
│                                          │              │
│  密钥 K ─→ K' ⊕ ipad ─→ 内部哈希 ─→ Hi ─┤              │
│                        (K'⊕ipad||M)      │              │
│                                          ▼              │
│  密钥 K ─→ K' ⊕ opad ─→ 外部哈希 ─→ HMAC(K,M)          │
│                        (K'⊕opad||Hi)                    │
│                                                         │
│  示例（HMAC-SHA256）：                                    │
│  - 块大小：64 字节                                       │
│  - 输出：32 字节（256 位）                                │
│  - 密钥推荐长度：≥ 32 字节                               │
│                                                         │
│  HMAC-SHA256("my secret key", "Hello World")            │
│  = 734cc62f32841568f45715aeb9f1d7d9f933f0c3f2bce53e     │
│    2e07f8b5e6a1f0a7                                     │
└─────────────────────────────────────────────────────────┘

HMAC 的安全性证明：
┌─────────────────────────────────────────────────────────┐
│  Bellare 等人证明：                                       │
│  HMAC 的安全性可归约为底层压缩函数的安全性                 │
│                                                         │
│  要求底层哈希函数具有：                                    │
│  - 抗碰撞（Collision Resistance）                         │
│  - 弱抗原像性（Weak Preimage Resistance）                 │
│                                                         │
│  即使底层哈希函数被发现弱点：                              │
│  - HMAC 仍可能安全（安全性证明不依赖完整哈希的安全性）     │
│  - 这是 HMAC 优于 NMAC（嵌套 MAC）的地方                 │
│                                                         │
│  安全强度：                                               │
│  - HMAC-SHA256：128 位（生日攻击）                        │
│  - HMAC-SHA512：256 位                                    │
│  - HMAC-MD5：不安全（MD5 已被破解）                       │
└─────────────────────────────────────────────────────────┘

HMAC 密钥管理：
┌─────────────────────────────────────────────────────────┐
│  密钥长度：                                               │
│  - 最小：哈希函数输出长度（如 SHA-256 需 32 字节）        │
│  - 最大：块大小（64 字节），超过会被哈希到块大小           │
│  - 推荐：≥ 32 字节                                       │
│                                                         │
│  密钥生成：                                               │
│  - 使用 CSPRNG 生成随机密钥                              │
│  - 不应使用密码直接作为 HMAC 密钥                        │
│  - HKDF 是推荐的密钥派生函数                             │
│                                                         │
│  密钥轮换：                                               │
│  - 定期更换密钥（如每 90 天）                             │
│  - 使用密钥版本号标识当前密钥                             │
│  - 旧密钥保留一段时间用于验证旧消息                       │
└─────────────────────────────────────────────────────────┘
```


## CMAC (Cipher-based MAC)


```
CMAC (NIST SP 800-38B)：
┌─────────────────────────────────────────────────────────┐
│  基于分组密码（如 AES）的 MAC 算法                        │
│  前身 OMAC1/OMAC2 的标准化版本                            │
│                                                         │
│  优点：                                                   │
│  - 只需要分组密码（AES），无需哈希函数                    │
│  - 硬件实现效率高（AES-NI 指令）                          │
│  - 可证明安全性                                          │
└─────────────────────────────────────────────────────────┘

CMAC 密钥派生：
K → AES_K → L
从 L 派生两个子密钥：
K1 = L << 1              （左移 1 位）
    若 L 最高位为 1，则 XOR 常量 Rb
K2 = K1 << 1             （左移 1 位）
    若 K1 最高位为 1，则 XOR 常量 Rb

AES-128 的 Rb = 0x00000000000000000000000000000087

CMAC 计算过程：
┌─────────────────────────────────────────────────────────┐
│  消息 M 分为 n 个块：M₁, M₂, ..., Mₙ                    │
│                                                         │
│  C₀ = 0                                                 │
│  C₁ = AES_K(C₀ ⊕ M₁)                                   │
│  C₂ = AES_K(C₁ ⊕ M₂)                                   │
│  ...                                                    │
│  Cₙ = AES_K(Cₙ₋₁ ⊕ (Mₙ ⊕ K₁))   ← 如果 Mₙ 是完整块    │
│  Cₙ = AES_K(Cₙ₋₁ ⊕ (pad(Mₙ) ⊕ K₂)) ← 如果 Mₙ 不完整   │
│                                                         │
│  MAC = Cₙ（截断到所需长度，如 64 或 128 位）             │
└─────────────────────────────────────────────────────────┘

CMAC vs HMAC 对比：
┌──────────────┬──────────────┬──────────────────────────┐
│ 特性          │ CMAC         │ HMAC                     │
├──────────────┼──────────────┼──────────────────────────┤
│ 底层原语      │ 分组密码(AES)│ 哈希函数(SHA-256)         │
│ 密钥长度      │ 128/192/256位│ 任意（推荐≥哈希输出长度）  │
│ 输出长度      │ ≤ 块大小     │ 哈希函数输出长度           │
│ 硬件效率      │ 高(AES-NI)   │ 中等                     │
│ 软件效率      │ 中等         │ 高                       │
│ 标准化        │ NIST         │ RFC 2104                 │
│ 用途          │ 无线/IPSec   │ TLS/API/Web              │
└──────────────┴──────────────┴──────────────────────────┘
```


## Poly1305 一次性 MAC


```
Poly1305 (Daniel Bernstein, 2005)：
┌─────────────────────────────────────────────────────────┐
│  类型：一次性 MAC (One-Time MAC / Wegman-Carter MAC)     │
│  密钥：256 位（128 位 r + 128 位 s）                     │
│  输出：128 位标签                                        │
│  安全性：信息论安全（前提：r 不重用）                     │
└─────────────────────────────────────────────────────────┘

Poly1305 计算过程：
┌─────────────────────────────────────────────────────────┐
│  输入：消息 M，密钥 (r, s)，其中 r 128 位，s 128 位      │
│                                                         │
│  1. 将消息分成 16 字节块：m₁, m₂, ..., mₙ               │
│     每块末尾附加 0x01 字节（作为标记）                    │
│                                                         │
│  2. 限制 r 的低 4 位为 0（防止侧信道攻击）               │
│     r &= 0x0ffffffc0ffffffc0ffffffc0fffffff             │
│                                                         │
│  3. 在模 2¹³⁰ - 5 下累加多项式：                        │
│     acc = 0                                             │
│     acc = (acc + m₁) × r mod (2¹³⁰ - 5)                │
│     acc = (acc + m₂) × r mod (2¹³⁰ - 5)                │
│     ...                                                 │
│     acc = (acc + mₙ) × r mod (2¹³⁰ - 5)                │
│                                                         │
│  4. 最终标签：tag = (acc + s) mod 2¹²⁸                  │
└─────────────────────────────────────────────────────────┘

ChaCha20-Poly1305 AEAD：
┌─────────────────────────────────────────────────────────┐
│  结合 ChaCha20 流密码和 Poly1305 MAC 的认证加密           │
│                                                         │
│  加密过程：                                              │
│  1. 用 ChaCha20 生成密钥流，加密明文                     │
│  2. 用 Poly1305 计算认证标签                             │
│     - 认证数据 = AAD + 密文 + 长度                       │
│     - Poly1305 密钥 = ChaCha20 的前 256 位输出           │
│                                                         │
│  TLS 1.3 支持的套件：                                    │
│  TLS_CHACHA20_POLY1305_SHA256                           │
│                                                         │
│  优势：                                                   │
│  - 在没有 AES-NI 硬件加速的设备上更快                     │
│  - 常数时间实现（避免时序攻击）                           │
│  - Google 在 Android 上优先使用（比 AES-GCM 更快）       │
│  - WireGuard 默认使用                                    │
└─────────────────────────────────────────────────────────┘

Poly1305 安全注意事项：
┌─────────────────────────────────────────────────────────┐
│  核心约束：r 绝对不能重用！                               │
│                                                         │
│  如果 r 被重用：                                          │
│  tag₁ = (m₁ × r + s) mod p                             │
│  tag₂ = (m₂ × r + s) mod p                             │
│  tag₁ - tag₂ = (m₁ - m₂) × r mod p                    │
│  → r = (tag₁ - tag₂) / (m₁ - m₂) mod p                │
│  → 攻击者可以伪造任意消息的 MAC                          │
│                                                         │
│  解决方案：                                               │
│  - 每次使用新的 r（ChaCha20 生成每条消息不同的 r）       │
│  - AES-CMAC 中无需此限制（密钥可重用）                   │
│  - HMAC 中也无需此限制（密钥可重用）                     │
└─────────────────────────────────────────────────────────┘
```


## GMAC (Galois MAC)


```
GMAC（AES-GCM 中的认证部分）：
┌─────────────────────────────────────────────────────────┐
│  GMAC 是 GCM 模式中的认证算法                             │
│  基于有限域 GF(2¹²⁸) 上的多项式乘法                      │
│  常与 AES-CTR 加密组合使用（AES-GCM）                    │
└─────────────────────────────────────────────────────────┘

GHASH 计算过程：
┌─────────────────────────────────────────────────────────┐
│  1. 生成认证密钥 H：                                      │
│     H = AES_K(0¹²⁸)（用密钥 K 加密全零块）              │
│                                                         │
│  2. 准备认证数据 X：                                      │
│     X = AAD || C || len(AAD) || len(C)                  │
│     其中 C = 密文，AAD = 附加认证数据                    │
│                                                         │
│  3. GHASH 迭代：                                         │
│     Y₀ = 0                                              │
│     Y₁ = (Y₀ ⊕ X₁) × H                                 │
│     Y₂ = (Y₁ ⊕ X₂) × H                                 │
│     ...                                                 │
│     Yₙ = (Yₙ₋₁ ⊕ Xₙ) × H                               │
│     其中 × 是 GF(2¹²⁸) 上的乘法                         │
│                                                         │
│  4. GMAC 标签 = Yₙ（通常截断到 128/120/112/96 位）       │
└─────────────────────────────────────────────────────────┘

GCM (Galois/Counter Mode) 完整流程：
┌─────────────────────────────────────────────────────────┐
│  输入：K (密钥), IV (初始化向量), P (明文), AAD (认证数据)│
│  输出：C (密文), T (认证标签)                             │
│                                                         │
│  1. 生成认证密钥 H = AES_K(0)                            │
│  2. 生成计数器序列：J₀, J₁, J₂, ...                     │
│     J₀ = IV || 0³¹ || 1（特殊情况 IV 处理）              │
│  3. 加密：C_i = P_i ⊕ AES_K(J_i)                        │
│  4. 认证：T = GHASH_H(AAD || C || len) ⊕ AES_K(J₀)     │
│  5. 返回 (C, T)                                          │
│                                                         │
│  GCM 安全性约束：                                         │
│  - IV 绝对不能重用（同一密钥下）                          │
│  - IV 重用 → 认证密钥 H 泄露 → 可伪造任意消息            │
│  - 推荐：96 位随机 IV 或计数器 IV                        │
└─────────────────────────────────────────────────────────┘

AEAD 模式对比：
┌──────────────┬──────────────┬──────────────┬───────────┐
│ 模式          │ 底层 MAC     │ 并行性       │ 推荐       │
├──────────────┼──────────────┼──────────────┼───────────┤
│ AES-GCM      │ GMAC         │ 完全并行     │ ✅ 首选   │
│ ChaCha20-Poly│ Poly1305     │ 完全并行     │ ✅ 首选   │
│ AES-CCM      │ CBC-MAC      │ 不可并行     │ ✅ 嵌入式  │
│ AES-SIV      │ CMAC×2       │ 部分并行     │ ✅ 安全   │
│ AES-OCB      │ OCB          │ 完全并行     │ ⚠️ 专利   │
└──────────────┴──────────────┴──────────────┴───────────┘
```


> **Note:** 在 TLS 1.3 中，AES-GCM 和 ChaCha20-Poly1305 是仅有的两种 AEAD 密码套件。GMAC 在有 AES-NI 的服务器上更快，Poly1305 在移动设备上更高效。


## MAC 算法综合对比与选择指南


```
MAC 算法安全性对比：
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│ 算法      │ 类型      │ 安全性    │ 速度      │ 推荐场景  │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ HMAC-SHA256│ 哈希MAC  │ 128位   │ 快       │ 通用/API  │
│ HMAC-SHA512│ 哈希MAC  │ 256位   │ 快(64位) │ 高安全级  │
│ CMAC-AES  │ 分组MAC   │ 128位   │ 很快     │ 硬件环境  │
│ Poly1305  │ 一次性MAC │ 128位   │ 最快     │ AEAD     │
│ GMAC      │ 有限域MAC │ 128位   │ 很快     │ AEAD     │
│ SipHash   │ 非加密MAC │ 非加密  │ 最快     │ 哈希表   │
└──────────┴──────────┴──────────┴──────────┴──────────┘

选择指南：
┌─────────────────────────────────────────────────────────┐
│  通用 Web/API 签名：                                      │
│  → HMAC-SHA256（最广泛支持，安全可靠）                     │
│                                                         │
│  TLS/网络通信加密+认证：                                  │
│  → AES-GCM 或 ChaCha20-Poly1305（AEAD）                 │
│                                                         │
│  嵌入式/IoT 设备：                                        │
│  → AES-CCM（低资源消耗）或 AES-GCM（有 AES-NI 时）       │
│                                                         │
│  哈希表 DoS 防护：                                        │
│  → SipHash（非密码学安全，但速度极快）                    │
│                                                         │
│  金融/银行系统：                                          │
│  → CMAC-AES（符合金融标准）                               │
│                                                         │
│  密钥派生：                                               │
│  → HKDF（基于 HMAC 的密钥派生函数）                       │
└─────────────────────────────────────────────────────────┘

常见错误：
┌─────────────────────────────────────────────────────────┐
│  ❌ 用 MD5 做 HMAC（MD5 已不安全）                        │
│  ❌ 用 SHA-1 做 HMAC（SHA-1 碰撞已公开）                  │
│  ❌ 自己实现 MAC 算法（应使用标准库）                     │
│  ❌ MAC-then-Encrypt（应使用 AEAD 或 Encrypt-then-MAC）  │
│  ❌ 密钥太短（HMAC 密钥至少等于哈希输出长度）             │
│  ❌ 不验证 MAC 直接解密（可能遭受 padding oracle 攻击）   │
└─────────────────────────────────────────────────────────┘
```
