# 03-哈希与数字签名

## 1. 密码学哈希函数的性质

密码学哈希函数 H 将任意长度的输入映射为固定长度的输出（摘要/哈希值）。必须满足以下性质：

| 性质 | 定义 | 数学表述 |
|------|------|----------|
| **确定性** | 相同输入始终产生相同输出 | H(m₁) = H(m₂) 当 m₁ = m₂ |
| **快速计算** | 对任意输入能高效计算哈希值 | 计算复杂度 O(n) |
| **抗原像性**（Preimage Resistance） | 给定哈希值 h，难以找到消息 m 使得 H(m) = h | 单向性 |
| **抗第二原像性**（Second Preimage Resistance） | 给定消息 m₁，难以找到 m₂ ≠ m₁ 使得 H(m₁) = H(m₂) | 弱抗碰撞性 |
| **抗碰撞性**（Collision Resistance） | 难以找到任意两个不同消息 m₁ ≠ m₂ 使得 H(m₁) = H(m₂) | 强抗碰撞性 |

输出长度为 n 位的哈希函数：
- 原像攻击复杂度：O(2ⁿ)
- 碰撞攻击复杂度：O(2^(n/2))（生日攻击）

## 2. 常用哈希算法

### 2.1 MD5

- 输出长度：128位
- 设计者：Ron Rivest，1991年
- 结构：Merkle-Damgard，64轮操作
- **状态：已不安全**
  - 2004年王小云等展示碰撞攻击
  - 2008年研究者构造了伪造的SSL证书
  - 不应在任何安全场景中使用

### 2.2 SHA-1

- 输出长度：160位
- 设计者：NSA，1995年
- **状态：已不安全**
  - 2017年Google公开了SHA-1碰撞实例（SHAttered攻击）
  - 利用了两个不同PDF文件产生相同SHA-1值
  - 仅在遗留系统中使用

### 2.3 SHA-2系列

SHA-2是NSA于2001年设计的哈希函数族：

| 算法 | 输出长度 | 分组长度 | 轮数 |
|------|----------|----------|------|
| SHA-224 | 224位 | 512位 | 64 |
| SHA-256 | 256位 | 512位 | 64 |
| SHA-384 | 384位 | 1024位 | 80 |
| SHA-512 | 512位 | 1024位 | 80 |

**SHA-256工作流程**（简化）：
1. 消息预处理：填充至512位的整数倍
2. 将消息分为512位块
3. 初始化8个32位哈希值（取自前8个素数的平方根小数部分）
4. 对每个块执行64轮压缩函数（涉及逻辑运算、模加法、常量）
5. 输出最终的256位哈希值

SHA-2至今未发现严重漏洞，是当前主流选择。

### 2.4 SHA-3（Keccak）

- 2012年NIST通过公开竞赛选出（设计者：Guido Bertoni等）
- 结构：**海绵结构**（Sponge Construction），与SHA-2完全不同
  - 吸收阶段（Absorb）：将消息块与内部状态混合
  - 挤出阶段（Squeeze）：从状态中提取输出
- 输出长度：224/256/384/512位可选
- 优势：不受SHA-2可能存在的结构攻击影响，提供多样性
- 应用：以太坊使用Keccak-256（SHA-3变体）

## 3. 哈希函数的攻击方法

### 3.1 生日攻击（Birthday Attack）

基于生日悖论：在 N 个元素中随机选取约√N 个，就有约50%的概率出现重复。

对 n 位哈希函数：需要约 2^(n/2) 次尝试找到碰撞。

| 哈希函数 | 输出长度 | 生日攻击复杂度 |
|----------|----------|----------------|
| MD5 | 128位 | 2⁶⁴ |
| SHA-1 | 160位 | 2⁸⁰ |
| SHA-256 | 256位 | 2¹²⁸ |

这就是为什么SHA-1（160位输出）被认为不够安全，而SHA-256仍然安全。

### 3.2 长度扩展攻击

针对采用Merkle-Damgard结构的哈希函数（MD5、SHA-1、SHA-2系列）：

给定 H(m) 和 len(m)，攻击者可以计算 H(m || padding || m') 而不需要知道 m 的内容。

**原因**：Merkle-Damgard结构的最终哈希值就是内部状态，可作为新的初始状态继续计算。

**防御**：
- 使用HMAC而非直接哈希
- 使用SHA-3（海绵结构不受此攻击影响）
- 对输出做截断

## 4. 消息认证码（MAC）

MAC提供**消息完整性**和**消息认证**，确保消息未被篡改且来自合法发送方。

### 4.1 HMAC

HMAC（Hash-based Message Authentication Code）基于哈希函数构造MAC：

```
HMAC_K(m) = H((K' XOR opad) || H((K' XOR ipad) || m))
```

- K'：密钥（若短于块大小则填充，若长则先哈希）
- ipad：内部填充（0x36重复）
- opad：外部填充（0x5C重复）
- H：底层哈希函数（如SHA-256）

**安全性**：HMAC的安全性可归约到底层哈希函数的抗原像性和抗碰撞性。

**应用**：TLS的记录层认证、API签名（AWS Signature）、JWT（JSON Web Token）。

### 4.2 CMAC

CMAC（Cipher-based MAC）基于分组密码（如AES）构造MAC：

- 使用CBC模式，仅保留最后一个分组的输出作为MAC
- 由两个子密钥（由主密钥派生）处理不同长度的消息
- 标准：NIST SP 800-38B

## 5. 数字签名

### 5.1 数字签名的性质与作用

数字签名提供：
- **认证性**：确认签名者的身份
- **完整性**：验证消息未被篡改
- **不可否认性**：签名者不能否认其签名

**基本流程**：
```
签名：S = Sign(私钥, Hash(消息))
验证：Verify(公钥, 消息, S) → 接受/拒绝
```

### 5.2 RSA签名

```
签名：σ = H(m)^d mod n
验证：计算 σ^e mod n，检查是否等于 H(m)
```

实践中使用**RSA-PSS**（Probabilistic Signature Scheme）填充，引入随机化以提高安全性。

### 5.3 DSA（数字签名算法）

DSA由NIST于1991年提出，基于离散对数问题。

**参数**：素数 p、素数 q（q 整除 p-1）、生成元 g

**签名**：
1. 选择随机数 k（1 < k < q）
2. r = (g^k mod p) mod q
3. s = k⁻¹(H(m) + x·r) mod q（x 为私钥）
4. 签名为 (r, s)

**验证**：通过公钥 (p, q, g, y = g^x mod p) 验证 (r, s)。

### 5.4 ECDSA

ECDSA是DSA的椭圆曲线版本（详见第02章）。相比RSA签名：
- 签名尺寸更小（如256位曲线的签名为512位，而2048位RSA签名为2048位）
- 计算更快
- 被广泛用于比特币、以太坊、TLS 1.3等

## 6. 公钥基础设施（PKI）

### 6.1 数字证书（X.509）

X.509证书将公钥与身份绑定，核心字段：

```
证书内容：
  版本号
  序列号
  签名算法
  颁发者（Issuer）：CA的名称
  有效期：Not Before, Not After
  主体（Subject）：证书持有者名称
  主体公钥信息
  扩展字段（SAN、Key Usage等）
签名值：CA对以上内容的签名
```

### 6.2 证书颁发机构（CA）

CA是受信任的第三方，负责：
- 验证申请者身份（域名所有权、组织信息等）
- 签发证书
- 管理证书吊销

**信任模型**：浏览器和操作系统预置根CA证书（信任锚）。

### 6.3 证书链与信任链

```
根CA（自签名证书，预置于操作系统/浏览器）
  └── 中间CA（由根CA签名）
       └── 终端实体证书（由中间CA签名）
```

验证证书链时，从终端证书逐级向上验证签名，直到找到受信任的根CA。

### 6.4 证书吊销

证书在过期前可能需要吊销（私钥泄露、域名转让等）：

- **CRL（证书吊销列表）**：CA定期发布已吊销证书的列表。缺点是文件大、更新延迟。
- **OCSP（在线证书状态协议）**：客户端向OCSP Responder查询特定证书状态。实时性好，但存在隐私和可用性问题。
- **OCSP Stapling**：服务器定期获取OCSP响应并附在TLS握手中发送，兼顾隐私和效率。

## 7. 身份认证协议

### 7.1 挑战-响应（Challenge-Response）

基本流程：
1. 验证方发送随机挑战值（Nonce）
2. 被验证方用密钥对挑战值计算MAC或签名
3. 验证方验证响应

防止重放攻击：每次挑战值不同。

### 7.2 Kerberos
Kerberos是MIT开发的网络认证协议，基于对称加密和可信第三方（KDC）。

**认证流程（简化）**：
1. **AS交换**：用户用密码派生密钥向AS（认证服务器）请求TGT
2. **TGS交换**：用户用TGT向TGS（票据授予服务器）请求服务票据
3. **AP交换**：用户向目标服务器出示服务票据，完成双向认证

**特点**：时间戳防重放；前向安全性；仅依赖对称加密；Windows域环境核心认证协议。
