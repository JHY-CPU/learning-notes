# TLS协议详解


## TLS 协议详解


密码学TLS/SSL网络安全


TLS（Transport Layer Security）是互联网通信安全的基础协议，为 HTTP、邮件、VPN 等提供加密保护。


## TLS 1.3 握手流程


TLS 1.3 相比 1.2 大幅简化了握手过程，仅需 1-RTT 即可完成。


```
TLS 1.3 完整握手（1-RTT）：
┌────────────┐                              ┌────────────┐
│   Client   │                              │   Server   │
└─────┬──────┘                              └──────┬─────┘
      │                                           │
      │ ─── ClientHello ──────────────────────→   │
      │     - 支持的密码套件列表                    │
      │     - 支持的 TLS 版本                      │
      │     - 客户端随机数 (Client Random)          │
      │     - key_share (客户端公钥)               │
      │     - supported_versions                  │
      │                                           │
      │ ←── ServerHello ──────────────────────    │
      │     - 选择的密码套件                        │
      │     - 服务端随机数 (Server Random)          │
      │     - key_share (服务端公钥)               │
      │                                           │
      │ ←── {EncryptedExtensions} ──────────      │
      │ ←── {Certificate} ──────────────────      │
      │ ←── {CertificateVerify} ────────────      │
      │     - 用服务端私钥签名握手摘要              │
      │ ←── {Finished} ─────────────────────      │
      │                                           │
      │ ─── {Finished} ──────────────────────→    │
      │                                           │
      │ ═══ 应用数据传输 (加密) ═══                │

握手密钥推导：
Client Random + Server Random + key_share
    → ECDHE 共享密钥 → HKDF 扩展 → 密钥材料
    → 客户端密钥 / 服务端密钥 / MAC 密钥
```


## 证书链验证


TLS 使用 X.509 证书和 PKI（公钥基础设施）来验证服务器身份。


```
证书链结构：
┌─────────────────────────────────────────────────┐
│  Root CA (根证书)                                │
│  └── 自签名，预置在操作系统/浏览器中               │
│  └── 例：DigiCert Root CA                        │
│      │                                          │
│      ▼                                          │
│  Intermediate CA (中间证书)                      │
│  └── 由 Root CA 签发                             │
│  └── 例：DigiCert SHA2 Extended Validation       │
│      │                                          │
│      ▼                                          │
│  Server Certificate (服务器证书)                  │
│  └── 由 Intermediate CA 签发                     │
│  └── 包含：域名、公钥、有效期、SAN                │
└─────────────────────────────────────────────────┘

证书内容（X.509 v3）：
- 版本号、序列号
- 签名算法（SHA256withRSA 等）
- 颁发者（Issuer）：CA 信息
- 有效期（Not Before / Not After）
- 主体（Subject）：域名信息
- 公钥信息
- 扩展字段：SAN、Key Usage、OCSP URL
- 签名值

验证过程：
1. 检查证书域名是否匹配
2. 检查证书是否在有效期内
3. 验证证书签名（逐级向上验证）
4. 检查证书是否被吊销（CRL / OCSP）
5. 检查是否在信任存储中
```


> **Note:** 现代浏览器要求使用 SAN（Subject Alternative Name）来标识域名，不再使用 CN 字段。


## 密钥交换（ECDHE）


TLS 1.3 强制使用前向安全（Forward Secrecy）的密钥交换方式。


```
ECDHE（Elliptic Curve Diffie-Hellman Ephemeral）：
┌─────────────────────────────────────────────────┐
│  客户端                          服务端           │
│  生成临时密钥对：                 生成临时密钥对：  │
│  私钥 a，公钥 A = a·G            私钥 b，公钥 B = b·G │
│                                                 │
│  ──── A 发送给服务端 ────→                       │
│  ←─── B 发送给客户端 ────                        │
│                                                 │
│  共享密钥 = a·B = a·b·G                          │
│  共享密钥 = b·A = b·a·G  （相同）                 │
│                                                 │
│  "临时"意味着每次握手生成新的密钥对                │
│  即使长期私钥泄露，之前的会话仍安全（前向安全）    │
└─────────────────────────────────────────────────┘

TLS 1.3 支持的密钥交换曲线：
- x25519（Curve25519）：最常用，性能优异
- secp256r1 (P-256)：NIST 标准曲线
- secp384r1 (P-384)：更高安全级别

TLS 1.3 不支持：
- 静态 DH（无前向安全）
- RSA 密钥交换（无前向安全）
- 仅支持 ECDHE 和 DHE
```


## 密码套件（Cipher Suites）


密码套件定义了 TLS 连接中使用的加密算法组合。


```
TLS 1.2 密码套件格式：
TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
│    │      │    │     │   │    │
│    │      │    │     │   │    └── PRF 哈希算法
│    │      │    │     │   └─────── 认证加密模式
│    │      │    │     └─────────── 对称加密算法
│    │      │    └───────────────── 对称加密
│    │      └────────────────────── 身份验证算法
│    └───────────────────────────── 密钥交换算法
└────────────────────────────────────── 协议

TLS 1.3 密码套件格式（简化）：
TLS_AES_256_GCM_SHA384
│    │     │   │    │
│    │     │   │    └── 哈希算法（HKDF）
│    │     │   └─────── 认证加密模式
│    │     └─────────── 密钥长度
│    └───────────────── 对称加密算法
└────────────────────── 协议
（密钥交换和认证算法在握手协商中确定）

TLS 1.3 只有 5 个密码套件：
- TLS_AES_128_GCM_SHA256
- TLS_AES_256_GCM_SHA384
- TLS_CHACHA20_POLY1305_SHA256
- TLS_AES_128_CCM_SHA256
- TLS_AES_128_CCM_8_SHA256
```


## HSTS（HTTP Strict Transport Security）


HSTS 强制浏览器使用 HTTPS 连接，防止降级攻击和 Cookie 劫持。


```
HSTS 工作原理：
1. 浏览器首次访问 https://example.com
2. 服务器返回响应头：
   Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
3. 浏览器记录该域名，在 max-age 时间内：
   - 自动将 http:// 转换为 https://
   - 拒绝不安全的证书（不显示"继续访问"选项）

HSTS 头参数：
┌───────────────────┬────────────────────────────────┐
│ max-age           │ 策略有效期（秒）                 │
│ includeSubDomains │ 是否包含所有子域名               │
│ preload           │ 是否加入浏览器预加载列表          │
└───────────────────┴────────────────────────────────┘

攻击防护：
┌─────────────────────────────────────────────────┐
│ 降级攻击（SSL Stripping）：                       │
│  攻击者将 HTTPS 降级为 HTTP                       │
│  HSTS 强制浏览器只使用 HTTPS，阻止降级            │
│                                                 │
│ SSL 剥离攻击流程（无 HSTS）：                     │
│  用户 → 攻击者 → 服务器（HTTP）                   │
│  用户以为在安全连接，实际数据被窃听                │
│                                                 │
│ HSTS Preload List：                              │
│  浏览器内置的域名列表，首次访问即使用 HTTPS         │
│  解决首次访问的信任问题                            │
│  提交地址：hstspreload.org                        │
└─────────────────────────────────────────────────┘
```


> **Note:** 启用 HSTS 前必须确保 HTTPS 配置正确且证书不会过期，否则用户将无法访问站点。建议先用较短的 max-age 测试。
