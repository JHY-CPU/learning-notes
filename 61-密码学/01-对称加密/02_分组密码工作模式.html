分组密码工作模式

密码学对称加密分组密码

分组密码工作模式定义了如何将加密算法应用于长于单个分组的数据，是实际加密应用的基础。

ECB 模式（电子密码本）

最简单的分组模式，每个分组独立加密。但由于确定性，存在严重的安全问题。

ECB (Electronic Codebook) 工作原理： 明文: P1 P2 P3 P4 ↓ ↓ ↓ ↓ E(K) E(K) E(K) E(K) ↓ ↓ ↓ ↓ 密文: C1 C2 C3 C4 特点： - 相同的明文分组 → 相同的密文分组 - 不需要 IV（初始化向量） - 支持并行加密和解密 - 分组间无关联，支持随机访问 安全问题： ┌──────────────────────────────────────────┐ │ ECB 模式下加密的企鹅图片： │ │ 原图可清晰辨认轮廓！ │ │ 因为相同颜色块加密后仍然相同 │ │ │ │ ❌ 确定性：相同明文 → 相同密文 │ │ ❌ 重放攻击：可替换已知密文块 │ │ ❌ 模式泄露：保留数据模式 │ └──────────────────────────────────────────┘ 适用场景：仅适用于加密随机数据（如密钥），不应用于加密消息或文件。
ECB 是唯一不需要 IV 的模式，但正因如此它也是最不安全的模式。实际应用中应避免使用 ECB。

CBC 模式（密码分组链接）

通过将前一个密文块与当前明文块异或，引入分组间的依赖关系，是最常用的模式之一。

CBC (Cipher Block Chaining) 工作原理： 明文: P1 P2 P3 ↓ ↓ ↓ IV→⊕ C1→⊕ C2→⊕ C3→⊕ ↓ ↓ ↓ E(K) E(K) E(K) ↓ ↓ ↓ 密文: C1 C2 C3 加密：C_i = E(K, P_i ⊕ C_{i-1})，其中 C_0 = IV 解密：P_i = D(K, C_i) ⊕ C_{i-1} 特点： - 相同明文 + 不同 IV → 不同密文（概率性加密） - 加密不可并行，解密可并行（解密时 C_{i-1} 已知） - 需要 IV 且 IV 应随机生成 - IV 不需要保密，可与密文一起传输 安全考虑： - IV 必须随机且不可重复使用 - 明文长度需为分组大小的倍数（需要填充） - 填充预言攻击（Padding Oracle Attack） → 使用 PKCS#7 填充时需注意 → 推荐使用密文窃取（CTS）或换用认证加密

CTR 模式（计数器模式）

将分组密码转换为流密码，通过加密计数器值生成密钥流。

CTR (Counter) 工作原理： Nonce+Counter: (N,1) (N,2) (N,3) ↓ ↓ ↓ E(K) E(K) E(K) ↓ ↓ ↓ Keystream: KS1 KS2 KS3 ↓ ↓ ↓ Plaintext: P1⊕KS1 P2⊕KS2 P3⊕KS3 ↓ ↓ ↓ Ciphertext: C1 C2 C3 加密/解密使用相同的操作（异或）： C_i = P_i ⊕ E(K, Nonce || Counter_i) P_i = C_i ⊕ E(K, Nonce || Counter_i) 特点： - 加密和解密完全可并行 - 不需要填充（支持任意长度明文） - 支持随机访问（可直接计算第 i 个密钥块） - 计数器值必须唯一（Nonce + Counter 组合）

GCM 与 CCM（认证加密模式）

认证加密（AEAD）同时提供机密性和完整性/认证性，是现代加密的标准要求。

GCM (Galois/Counter Mode)： ┌────────────────────────────────────────────┐ │ 底层使用 CTR 模式加密 │ │ + GHASH 函数计算认证标签（GMAC） │ │ │ │ C, T = AES-GCM(K, IV, P, AAD) │ │ 其中：C = 密文，T = 认证标签 │ │ P = 明文，AAD = 附加认证数据 │ │ │ │ 特点： │ │ - 可并行，高性能 │ │ - 提供机密性 + 完整性 + 认证 │ │ - 广泛用于 TLS 1.3、IPSec、SSH │ │ - 标签长度常见 128/120/112/96 bit │ └────────────────────────────────────────────┘ CCM (Counter with CBC-MAC)： ┌────────────────────────────────────────────┐ │ CBC-MAC 计算认证标签 │ │ CTR 模式进行加密 │ │ │ │ 特点： │ │ - 不可并行（CBC-MAC 部分） │ │ - 用于 WPA2 (WiFi)、蓝牙、ZigBee │ │ - 实现相对简单 │ └────────────────────────────────────────────┘ 模式对比总结： ┌──────┬──────┬──────┬──────────┬──────────┬──────┐ │ 模式 │ 并行 │ 填充 │ 认证加密 │ 安全性 │ 推荐 │ ├──────┼──────┼──────┼──────────┼──────────┼──────┤ │ ECB │ 加解密│ 需要 │ ❌ │ 差 │ ❌ │ │ CBC │ 解密 │ 需要 │ ❌ │ 中 │ ⚠️ │ │ CTR │ 加解密│ 不需要│ ❌ │ 好 │ ⚠️ │ │ GCM │ 加解密│ 不需要│ ✅ │ 优秀 │ ✅ │ │ CCM │ 加密 │ 不需要│ ✅ │ 好 │ ✅ │ └──────┴──────┴──────┴──────────┴──────────┴──────┘
现代应用应优先选择 GCM 或其他 AEAD 模式（如 ChaCha20-Poly1305），避免使用 ECB 和单独的 CBC/CTR。

IV/Nonce 与填充

IV（Initialization Vector，初始化向量）： - 确保相同明文在不同加密操作中产生不同密文 - CBC 模式：IV 必须随机且不可预测 - CTR/GCM 模式：Nonce 必须唯一（不需要随机） - IV/Nonce 不需要保密，通常与密文一起传输 Nonce 重用的灾难性后果： - CTR 模式：Nonce 重用 → 密钥流重用 → 可恢复明文 C1 = P1 ⊕ KS, C2 = P2 ⊕ KS C1 ⊕ C2 = P1 ⊕ P2（异或两个密文即可消除密钥流） - GCM 模式：Nonce 重用 → 认证密钥泄露 → 可伪造消息 填充方案： ┌────────────┬──────────────────────────────────────┐ │ PKCS#7 │ 填充 N 个字节，每个字节值为 N │ │ │ 例：需填 4 字节 → 04 04 04 04 │ ├────────────┼──────────────────────────────────────┤ │ Zero │ 用 0x00 填充，需要额外记录填充长度 │ ├────────────┼──────────────────────────────────────┤ │ ANSI X9.23 │ 最后一字节记录填充长度，其余填 0x00 │ ├────────────┼──────────────────────────────────────┤ │ 无填充 │ CTR/GCM 等流密码模式不需要填充 │ └────────────┴──────────────────────────────────────┘
