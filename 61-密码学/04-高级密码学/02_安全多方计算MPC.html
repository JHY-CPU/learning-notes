安全多方计算 MPC

密码学MPC隐私计算

安全多方计算（MPC）使多方能在不暴露各自私有输入的情况下联合计算一个函数，是隐私保护计算的核心技术。

Shamir 秘密分享 (SSS)

Shamir 秘密分享 (Adi Shamir, 1979)： ┌─────────────────────────────────────────────────────────┐ │ 目标：将秘密 S 拆分为 n 个份额，任意 k 个可恢复 S │ │ 少于 k 个份额得不到 S 的任何信息 │ │ 安全基础：拉格朗日插值多项式 │ │ 性质：信息论安全（无条件安全） │ └─────────────────────────────────────────────────────────┘ 方案构造（(k, n)-阈值方案）： ┌─────────────────────────────────────────────────────────┐ │ 1. 选择大素数 p > S 且 p > n │ │ │ │ 2. 构造 k-1 次随机多项式： │ │ f(x) = S + a₁x + a₂x² + ... + aₖ₋₁x^(k-1) mod p │ │ 其中 a₁, a₂, ..., aₖ₋₁ 随机选取 │ │ 注意：f(0) = S │ │ │ │ 3. 生成 n 个份额： │ │ s₁ = f(1), s₂ = f(2), ..., sₙ = f(n) │ │ 分发给 n 个参与者 │ │ │ │ 4. 恢复秘密（任意 k 个份额）： │ │ 使用拉格朗日插值： │ │ S = f(0) = Σ sᵢ × Lᵢ(0) mod p │ │ 其中 Lᵢ(0) = Π(0-j)/(i-j) mod p (j≠i) │ │ │ │ 示例（(3,5)-阈值方案）： │ │ 秘密 S = 1234，p = 1613 │ │ f(x) = 1234 + 166x + 94x² mod 1613 │ │ 份额： │ │ s₁ = f(1) = 1494 │ │ s₂ = f(2) = 1942 │ │ s₃ = f(3) = 2578 mod 1613 = 965 │ │ s₄ = f(4) = ... │ │ s₅ = f(5) = ... │ │ 任意 3 个份额可恢复 S = 1234 │ │ 任意 2 个份额得不到 S 的任何信息 │ └─────────────────────────────────────────────────────────┘ Shamir 秘密分享的性质： ┌─────────────────────────────────────────────────────────┐ │ 1. 信息论安全： │ │ k-1 个份额对 S 的条件概率 = 无条件概率 │ │ 攻击者拥有 k-1 个份额时，S 可以是任意值 │ │ 不依赖计算困难假设 │ │ │ │ 2. 完美阈值： │ │ k 个份额可 100% 恢复 S │ │ k-1 个份额得到 0 信息 │ │ │ │ 3. 线性性质： │ │ 份额的线性组合 = 对应秘密的线性组合 │ │ s₁ + s₂ 对应 S₁ + S₂（同态加法） │ │ c × s₁ 对应 c × S₁（同态标量乘法） │ │ 这使得在秘密分享上直接计算成为可能 │ │ │ │ 4. 动态更新： │ │ 可以在不恢复秘密的情况下更换份额 │ │ 新份额 = f'(i)，但 f'(0) = f(0) = S │ └─────────────────────────────────────────────────────────┘

混淆电路 (Garbled Circuits)

混淆电路 (Andrew Yao, 1986)： ┌─────────────────────────────────────────────────────────┐ │ 核心思想：将计算函数表示为布尔电路，然后对电路"加密" │ │ 使一方可以评估电路得到输出，但双方都不会泄露自己的输入 │ │ │ │ 角色： │ │ - Garbler（混淆方）：构建混淆电路 │ │ - Evaluator（评估方）：在电路上计算 │ └─────────────────────────────────────────────────────────┘ 构建过程（以 AND 门为例）： ┌─────────────────────────────────────────────────────────┐ │ 1. 为每条线路分配两个标签（密钥）： │ │ 输入线 A：w_A^0 (代表 0)，w_A^1 (代表 1) │ │ 输入线 B：w_B^0 (代表 0)，w_B^1 (代表 1) │ │ 输出线 C：w_C^0 (代表 0)，w_C^1 (代表 1) │ │ │ │ 2. 构建混淆表（AND 真值表的加密形式）： │ │ Enc(w_A^0, w_B^0) → w_C^0 (0 AND 0 = 0) │ │ Enc(w_A^0, w_B^1) → w_C^0 (0 AND 1 = 0) │ │ Enc(w_A^1, w_B^0) → w_C^0 (1 AND 0 = 0) │ │ Enc(w_A^1, w_B^1) → w_C^1 (1 AND 1 = 1) │ │ │ │ 3. 打乱混淆表（行顺序随机化） │ │ → Evaluator 无法区分哪行对应哪个输入组合 │ │ │ │ 4. Evaluator 拥有 w_A 和 w_B（代表实际输入） │ │ 在混淆表中解密得到 w_C │ │ 重复所有门，最终得到输出线的标签 │ │ 通过标签映射表得到实际输出值 │ └─────────────────────────────────────────────────────────┘ 输入传递（Obligated Transfer）： ┌─────────────────────────────────────────────────────────┐ │ 问题：Evaluator 如何获得输入线的标签？ │ │ - Garbler 知道自己的输入标签（自己生成的） │ │ - Evaluator 需要获取自己的输入标签 │ │ - 但不能让 Garbler 知道 Evaluator 的输入 │ │ │ │ 解决：不经意传输 (OT, Oblivious Transfer) │ │ - 2-out-of-1 OT： │ │ Evaluator 输入选择位 b ∈ {0,1} │ │ Garbler 提供两个标签 w^0, w^1 │ │ Evaluator 得到 w^b │ │ Garbler 不知道 b 的值 │ │ │ │ 实际使用 1-out-of-2 OT： │ │ Evaluator 选择获取 w^0 或 w^1 │ │ Garbler 不知道选择了哪个 │ │ │ │ OT 扩展 (OT Extension)： │ │ - 少量 OT → 大量 OT（效率提升） │ │ - IKNP 协议：n 次 OT + 对称加密 → n 次 OT │ └─────────────────────────────────────────────────────────┘ 混淆电路优化： ┌─────────────────────────────────────────────────────────┐ │ Free XOR (Kolesnikov & Schneider, 2008)： │ │ - XOR 门不需要混淆表！ │ │ - 令 w^1 = w^0 ⊕ Δ（全局偏移量） │ │ - XOR 门输出 = 输入标签直接 XOR │ │ - 节省大量计算和通信 │ │ │ │ Half Gates (Zahur et al., 2015)： │ │ - AND 门只需 2 行混淆表（而非 4 行） │ │ - 进一步优化通信开销 │ │ │ │ Point-and-Permute： │ │ - 利用标签的随机位来排列混淆表 │ │ - 无需解密所有 4 行 │ │ - 评估效率提升 4 倍 │ └─────────────────────────────────────────────────────────┘

不经意传输 OT (Oblivious Transfer)

1-out-of-2 OT 协议： ┌─────────────────────────────────────────────────────────┐ │ 输入： │ │ Sender：两条消息 m₀, m₁ │ │ Receiver：选择位 σ ∈ {0, 1} │ │ │ │ 输出： │ │ Receiver：得到 m_σ │ │ Sender：不知道 σ 的值（不知道接收了哪条消息） │ │ │ │ OT 是 MPC 的基础原语： │ │ - 混淆电路中的输入传递 │ │ - 秘密分享协议中的输入阶段 │ │ - 任何 MPC 协议都可以用 OT 构造 │ └─────────────────────────────────────────────────────────┘ Naor-Pinkas OT 协议（基于 Diffie-Hellman）： ┌─────────────────────────────────────────────────────────┐ │ Setup：群 G，生成元 g │ │ │ │ 1. Receiver 选择随机数 k，发送 C = g^k │ │ │ │ 2. Sender 选择随机数 a, b，计算： │ │ pk₀ = g^a │ │ pk₁ = C / g^a = g^(k-a) （如果 σ=1） │ │ 或 pk₁ = g^b （随机） │ │ 计算密钥：K₀ = pk₀^b, K₁ = pk₁^b │ │ 加密：e₀ = m₀ ⊕ KDF(K₀) │ │ e₁ = m₁ ⊕ KDF(K₁) │ │ 发送：pk₀, e₀, e₁ │ │ │ │ 3. Receiver 计算： │ │ 如果 σ=0：K = pk₀^k = g^(ak)，解密 e₀ │ │ 如果 σ=1：K = pk₁^k = g^((k-a)k)，但... │ │ 实际上需要更复杂的协议 │ │ │ │ 简化理解： │ │ - Receiver 的选择位隐含在公钥中 │ │ - Sender 无法判断 Receiver 选了哪个 │ │ - Receiver 只能解密选中的那条消息 │ └─────────────────────────────────────────────────────────┘ OT 扩展 (OT Extension)： ┌─────────────────────────────────────────────────────────┐ │ 基本 OT 计算代价大（需要公钥操作） │ │ OT 扩展用少量 OT + 大量对称加密实现大量 OT │ │ │ │ IKNP 协议 (Ishai et al., 2003)： │ │ 基础 OT：k 次（k 为安全参数，如 128） │ │ 扩展后：n 次 OT（n 可达百万级） │ │ │ │ 原理： │ │ 1. 基础 OT：Receiver 获取 k 个选择位的密钥对 │ │ 2. Receiver 生成 n × k 矩阵 T，选择列由选择位决定 │ │ 3. Sender 生成 n × k 矩阵 T' │ │ 4. 每行的哈希值用于加密 OT 消息 │ │ │ │ 代价：n 次 OT ≈ n 次哈希 + k 次 OT │ │ 效率：比直接 OT 快 ~1000 倍 │ └─────────────────────────────────────────────────────────┘

SPDZ 协议

SPDZ (Speedz) 协议： ┌─────────────────────────────────────────────────────────┐ │ 设计者：Damgard et al. (2012) │ │ 类型：基于秘密分享和同态加密的 MPC 协议 │ │ 特点：支持恶意安全模型，可容忍 t < n/2 个腐败方 │ │ 名称来源：作者姓氏首字母的"speedz"发音 │ └─────────────────────────────────────────────────────────┘ SPDZ 协议框架： ┌─────────────────────────────────────────────────────────┐ │ 阶段 1：离线预处理 (Preprocessing) │ │ ───────────────────────────── │ │ 生成乘法三元组 (Multiplicative Triples)： │ │ - [a], [b], [c] 其中 c = a × b │ │ - a, b 随机，c = a × b │ │ - 使用同态加密（Paillier 或 BFV） │ │ - 这个阶段与具体计算无关，可预先完成 │ │ │ │ 还需要生成： │ │ - 随机掩码 [r] │ │ - 比较用的比特分解值 │ │ │ │ 阶段 2：在线计算 (Online Phase) │ │ ───────────────────────────── │ │ 输入秘密化： │ │ - 每个参与方将输入 x 拆分为 [x] = x + r - r │ │ - 公开 r 给其他方 │ │ - 实际值被随机化，无法从单个份额恢复 │ │ │ │ 加法计算： │ │ [x] + [y] = [x + y] │ │ 直接本地计算，无需通信 │ │ │ │ 乘法计算： │ │ [x] × [y] 需要使用乘法三元组 [a], [b], [c]： │ │ 1. 公开 e = [x] - [a]，f = [y] - [b] │ │ 2. [xy] = [c] + f×[a] + e×[b] + e×f │ │ 因为 xy = (a+e)(b+f) = ab + af + be + ef │ │ 公开 e, f 不泄露 x, y（因为 a, b 是随机的） │ │ │ │ 输出揭秘： │ │ - 公开份额求和得到最终结果 │ │ - MAC 验证确保无恶意篡改 │ └─────────────────────────────────────────────────────────┘ SPDZ 的恶意安全： ┌─────────────────────────────────────────────────────────┐ │ 信息论 MAC： │ │ - 每个共享值 [x] 关联一个 MAC 共享 [MAC(x)] │ │ - MAC(x) = α × x (全局密钥 α) │ │ - 揭秘时验证 MAC 标签 │ │ - 恶意方无法伪造有效 MAC │ │ │ │ 恶意安全保证： │ │ - 恶意参与方的行为等同于某个理想世界中的行为 │ │ - 通过零知识证明确保计算正确性 │ │ - 信息论 MAC 防止份额篡改 │ │ │ │ 通信轮数： │ │ - 离线阶段：取决于 HE 方案（通常较多轮） │ │ - 在线阶段：每层乘法 1 轮通信 │ │ - 总在深度 d 的电路：O(d) 轮 │ └─────────────────────────────────────────────────────────┘ SPDZ 变体与改进： ┌─────────────────────────────────────────────────────────┐ │ SPDZ-2 (2013)： │ │ - 改进的在线阶段，减少通信量 │ │ - 支持定点数运算 │ │ │ │ Overdrive (2018)： │ │ - 使用全同态加密加速离线阶段 │ │ - 离线吞吐量提升 10-100 倍 │ │ │ │ MP-SPDZ (开源实现)： │ │ - 支持多种后端：SPDZ, SPDZ2Y, ABY3, Replicated 等 │ │ - 编译器将高级语言编译为 MPC 协议 │ │ - GitHub: data61/MP-SPDZ │ └─────────────────────────────────────────────────────────┘

MPC 实际应用

隐私集合求交 (PSI - Private Set Intersection)： ┌─────────────────────────────────────────────────────────┐ │ 场景：两个公司找出共同的恶意 IP 地址 │ │ - 公司 A 有自己的 IP 黑名单 │ │ - 公司 B 有自己的 IP 黑名单 │ │ - 双方想找出交集，但不想暴露非交集部分 │ │ │ │ 协议：基于 OT 的 PSI / 基于哈希的 PSI │ │ 复杂度：O(n log n) 或 O(n) │ └─────────────────────────────────────────────────────────┘ 隐私机器学习 (Privacy-Preserving ML)： ┌─────────────────────────────────────────────────────────┐ │ 场景：多家医院联合训练疾病诊断模型 │ │ - 每家医院有患者数据（隐私敏感） │ │ - 希望联合训练更好的模型 │ │ - 不暴露任何一家医院的原始数据 │ │ │ │ 方案： │ │ 1. 秘密分享训练数据 │ │ 2. 在秘密分享值上计算神经网络前向传播 │ │ 3. 安全计算梯度（反向传播） │ │ 4. 安全更新模型参数 │ │ │ │ 挑战： │ - 非线性函数（ReLU, Sigmoid）需要特殊处理（分段线性/多项式近似） │ - 通信开销巨大 │ │ - 训练时间比明文慢 100-1000 倍 │ └─────────────────────────────────────────────────────────┘ 其他应用场景： ┌─────────────────────────────────────────────────────────┐ │ 联邦数据分析： │ │ - 跨公司统计分析（薪资比较、用户画像） │ │ - 电子投票（保护选票隐私） │ │ - 联邦风控（银行间黑名单比对） │ │ │ │ 区块链隐私： │ │ - 门限签名（多方共同签名交易） │ │ - 隐私交易（隐藏交易金额和参与方） │ │ - DEX 清算（暗池交易） │ │ │ │ 供应链与合规： │ │ - 多方联合 KYC（银行间共享客户验证） │ │ - 反洗钱（跨机构交易模式检测） │ │ - 税务审计（跨公司数据验证） │ └─────────────────────────────────────────────────────────┘
MPC 在工业界已有实际部署：Google 在密码学产品中使用 MPC，Signal 使用密封竞价拍卖，多个区块链项目使用门限签名。MP-SPDZ 是最活跃的开源 MPC 框架。
