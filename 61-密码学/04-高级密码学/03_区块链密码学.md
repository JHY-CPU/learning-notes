# 区块链密码学


## 区块链密码学


密码学区块链Merkle Tree


区块链的核心安全建立在密码学之上：Merkle 树保证数据完整性，数字签名保证交易不可伪造，哈希保证链式不可篡改。


## Merkle 树 (Merkle Tree)


```
Merkle 树结构：
┌─────────────────────────────────────────────────────────┐
│  Merkle 树是一种哈希树，用于高效验证大规模数据的完整性      │
│                                                         │
│                   Root Hash                              │
│                  /           \                           │
│           Hash(AB)          Hash(CD)                     │
│           /     \           /     \                      │
│      Hash(A)  Hash(B)  Hash(C)  Hash(D)                 │
│        |        |        |        |                      │
│       Data_A  Data_B  Data_C  Data_D                    │
│                                                         │
│  节点计算：                                               │
│  - 叶子节点：Hash(数据块)                                │
│  - 内部节点：Hash(左子节点 || 右子节点)                   │
│  - 根节点：整个数据集的指纹                               │
└─────────────────────────────────────────────────────────┘

Merkle 证明 (Merkle Proof)：
┌─────────────────────────────────────────────────────────┐
│  目标：证明数据块 D 在 Merkle 树中，无需下载全部数据      │
│                                                         │
│  示例：验证 Data_B                                        │
│  需要的路径（Merkle Proof）：                             │
│  1. Hash(A)  ← 兄弟节点                                  │
│  2. Hash(CD) ← 叔父节点                                  │
│                                                         │
│  验证过程：                                              │
│  Step 1: h₁ = Hash(Data_B)                              │
│  Step 2: h₂ = Hash(Hash(A) || h₁)                       │
│  Step 3: h₃ = Hash(h₂ || Hash(CD))                      │
│  Step 4: 比较 h₃ == Root Hash                           │
│  → 一致则证明 Data_B 确实在树中                          │
│                                                         │
│  效率：                                                  │
│  - n 个数据块的树，证明大小为 O(log n) 个哈希值          │
│  - 百万级数据仅需 ~20 个哈希值                           │
│  - 验证时间 O(log n)                                     │
└─────────────────────────────────────────────────────────┘

比特币中的 Merkle 树：
┌─────────────────────────────────────────────────────────┐
│  用途：验证交易是否包含在区块中                            │
│                                                         │
│  轻节点 (SPV) 验证：                                      │
│  1. 轻节点只知道区块头（80 字节）                         │
│  2. 向全节点请求某交易的 Merkle Proof                     │
│  3. 用 Merkle Proof 验证交易在区块中                     │
│  4. 无需下载完整区块（可能数 MB）                        │
│                                                         │
│  注意事项：                                               │
│  - 叶子节点数为奇数时，最后一个节点复制一份                │
│  - 比特币使用 SHA-256(SHA-256(x)) 双重哈希               │
│  - 以太坊使用改进的 Merkle Patricia Trie                 │
└─────────────────────────────────────────────────────────┘

稀疏 Merkle 树 (Sparse Merkle Tree, SMT)：
┌─────────────────────────────────────────────────────────┐
│  特点：2^256 个叶子节点（覆盖所有可能的地址）              │
│  - 大多数叶子是空值（默认哈希）                           │
│  - 可以高效证明"某值不存在"                               │
│  - 以太坊状态树的变体                                     │
│                                                         │
│  优势：                                                  │
│  - 统一存在性和不存在性证明                               │
│  - 简化轻客户端实现                                      │
│  - 适用于跨链桥验证                                      │
└─────────────────────────────────────────────────────────┘
```


## 区块链中的数字签名


```
ECDSA (Elliptic Curve Digital Signature Algorithm)：
┌─────────────────────────────────────────────────────────┐
│  比特币和以太坊使用 secp256k1 曲线的 ECDSA               │
│                                                         │
│  曲线参数 secp256k1：                                    │
│  y² = x³ + 7  over F_p                                  │
│  p = 2²⁵⁶ - 2³² - 977                                   │
│  选择原因：特殊参数使得运算效率极高                       │
│                                                         │
│  密钥生成：                                              │
│  私钥 k ∈ [1, n-1]（256 位随机数）                      │
│  公钥 K = k × G（椭圆曲线点）                            │
│  地址 = RIPEMD160(SHA256(K)) 的 Base58Check 编码        │
│                                                         │
│  签名过程：                                              │
│  1. 计算 e = SHA256(交易数据)                            │
│  2. 选择随机数 k（nonce），计算 R = k × G               │
│  3. r = R.x mod n                                       │
│  4. s = k^(-1) × (e + r × privKey) mod n               │
│  5. 签名 = (r, s) + recovery_id                         │
│                                                         │
│  关键安全要求：                                           │
│  - nonce k 必须随机且唯一                                │
│  - k 重用 → 私钥泄露！                                   │
│    k = (e₁ - e₂) / (s₁ - s₂) mod n                     │
│    privKey = (s₁ × k - e₁) / r mod n                   │
│  - 索尼 PS3 签名事件：k 固定导致私钥泄露                 │
│  - RFC 6979：确定性 ECDSA，用 HMAC-DRBG 生成 k          │
└─────────────────────────────────────────────────────────┘

EdDSA (Edwards-curve Digital Signature Algorithm)：
┌─────────────────────────────────────────────────────────┐
│  Ed25519 (基于 Curve25519)                               │
│  用于：Solana, Cardano, Polkadot 等新公链               │
│                                                         │
│  密钥生成：                                              │
│  私钥 seed（32 字节随机数）                               │
│  扩展密钥 = SHA-512(seed)                                │
│  私钥标量 a = 扩展密钥的前 32 字节（经过位处理）         │
│  前缀 = 扩展密钥的后 32 字节                             │
│  公钥 A = a × B（B 为基点）                              │
│                                                         │
│  签名过程：                                              │
│  1. r = SHA-512(prefix || message) mod L（确定性nonce）  │
│  2. R = r × B                                           │
│  3. S = (r + SHA-512(R || A || message) × a) mod L     │
│  4. 签名 = (R, S)（64 字节）                             │
│                                                         │
│  Ed25519 vs ECDSA (secp256k1) 对比：                     │
│  ┌──────────────┬──────────────┬──────────────────┐     │
│  │ 特性          │ Ed25519      │ ECDSA (secp256k1)│     │
│  ├──────────────┼──────────────┼──────────────────┤     │
│  │ 签名速度      │ 更快         │ 快               │     │
│  │ 验证速度      │ 更快         │ 中等             │     │
│  │ 签名大小      │ 64字节       │ ~71字节(可变)    │     │
│  │ 确定性        │ 内置         │ 需RFC 6979       │     │
│  │ 安全证明      │ 可证明安全   │ 较弱             │     │
│  │ 常数时间      │ 天然支持     │ 需要仔细实现     │     │
│  │ 批量验证      │ 支持         │ 不支持           │     │
│  └──────────────┴──────────────┴──────────────────┘     │
└─────────────────────────────────────────────────────────┘

Schnorr 签名（比特币 Taproot 升级）：
┌─────────────────────────────────────────────────────────┐
│  比特币 Taproot (2021) 引入 Schnorr 签名                 │
│                                                         │
│  签名过程：                                              │
│  1. 选择随机数 k，计算 R = k × G                        │
│  2. e = SHA256(R || PubKey || message)                  │
│  3. s = k + e × privKey                                 │
│  4. 签名 = (R, s)（64 字节）                             │
│                                                         │
│  核心优势：签名聚合 (Signature Aggregation)              │
│  - 多个签名可合并为一个签名                               │
│  - 多签交易外观与单签相同（隐私提升）                     │
│  - 减少区块空间（N 个签名 → 1 个签名）                   │
│                                                         │
│  MuSig2（多方 Schnorr 签名）：                            │
│  - N 方共同生成一个签名                                   │
│  - 无需暴露参与方数量和公钥                               │
│  - 两轮交互即可完成                                      │
└─────────────────────────────────────────────────────────┘
```


## 承诺方案 (Commitment Schemes)


```
承诺方案的性质：
┌─────────────────────────────────────────────────────────┐
│  1. 隐藏性 (Hiding)：                                    │
│     承诺不泄露被承诺的值                                  │
│                                                         │
│  2. 约束性 (Binding)：                                   │
│     一旦做出承诺，不能改变被承诺的值                      │
│                                                         │
│  应用：零知识证明、拍卖协议、电子投票、区块链隐私         │
└─────────────────────────────────────────────────────────┘

Pedersen 承诺：
┌─────────────────────────────────────────────────────────┐
│  公开参数：群 G，生成元 g, h（log_g(h) 未知）            │
│                                                         │
│  承诺：C = g^v × h^r                                     │
│  其中 v = 被承诺的值，r = 随机盲化因子                   │
│                                                         │
│  打开：公开 (v, r)，验证 C == g^v × h^r                  │
│                                                         │
│  性质：                                                  │
│  - 隐藏性：随机 r 使得 C 不泄露 v                        │
│  - 约束性：无法找到 (v', r') ≠ (v, r) 使得              │
│           g^v × h^r = g^v' × h^r'                       │
│           （等价于离散对数问题）                           │
│  - 加法同态：                                            │
│    C₁ × C₂ = g^(v₁+v₂) × h^(r₁+r₂)                    │
│    承诺的乘积 = 值之和的承诺                              │
│                                                         │
│  区块链应用：                                             │
│  - Monero：隐藏交易金额                                  │
│  - Bulletproofs：范围证明，证明 v ∈ [0, 2^64)           │
└─────────────────────────────────────────────────────────┘

哈希承诺（Hash-based Commitment）：
┌─────────────────────────────────────────────────────────┐
│  承诺：C = Hash(v || r)                                  │
│  打开：公开 (v, r)，验证 C == Hash(v || r)               │
│                                                         │
│  特点：                                                  │
│  - 实现简单，计算快速                                    │
│  - 无同态性质                                            │
│  - 适用于简单场景（猜谜、锁仓证明）                       │
│                                                         │
│  Verkle 树中的向量承诺：                                  │
│  - 以太坊计划迁移到 Verkle 树                            │
│  - 基于多项式承诺（KZG 承诺）                            │
│  - 证明大小 O(1)（而非 Merkle 树的 O(log n)）            │
└─────────────────────────────────────────────────────────┘

Kate 承诺 (KZG Commitment)：
┌─────────────────────────────────────────────────────────┐
│  基于配对的多项式承诺方案                                 │
│  用于：以太坊 EIP-4844 (Proto-Danksharding)              │
│                                                         │
│  可信设置：                                               │
│  生成公开参数 [1]_1, [s]_1, [s²]_1, ..., [s^n]_1        │
│  其中 s 为秘密值（设置完成后删除）                        │
│                                                         │
│  承诺多项式 P(x)：                                       │
│  C = [P(s)]_1 = P(s) × G₁                               │
│                                                         │
│  证明 P(z) = y：                                         │
│  商多项式 Q(x) = (P(x) - y) / (x - z)                   │
│  证明 π = [Q(s)]_1                                       │
│  验证：e(C - [y]_1, G₂) = e(π, [s]_2 - [z]_2)         │
│                                                         │
│  优势：                                                  │
│  - 承诺和证明大小恒定（1 个群元素）                      │
│  - 验证时间恒定                                          │
│  - 适用于数据可用性采样 (DAS)                            │
└─────────────────────────────────────────────────────────┘
```


## 门限签名 (Threshold Signatures)


```
门限签名：
┌─────────────────────────────────────────────────────────┐
│  目标：t-out-of-n 签名方案                                │
│  - n 个参与方中任意 t 个可以合作生成签名                  │
│  - 少于 t 个方无法生成签名                               │
│  - 最终签名与单方签名完全相同（验证者不区分）             │
│                                                         │
│  与多重签名的区别：                                       │
│  ┌──────────────┬──────────────┬──────────────────┐     │
│  │ 特性          │ 门限签名      │ 多重签名          │     │
│  ├──────────────┼──────────────┼──────────────────┤     │
│  │ 签名大小      │ 单签大小      │ N × 单签大小     │     │
│  │ 链上可见性    │ 不可见参与方  │ 可见所有参与方   │     │
│  │ 隐私性        │ 好           │ 差               │     │
│  │ 验证成本      │ 恒定         │ 线性增长         │     │
│  │ 实现复杂度    │ 高           │ 低               │     │
│  └──────────────┴──────────────┴──────────────────┘     │
└─────────────────────────────────────────────────────────┘

FROST (Flexible Round-Optimized Schnorr Threshold)：
┌─────────────────────────────────────────────────────────┐
│  基于 Schnorr 签名的门限签名方案                          │
│  设计者：Chelsea Komlo, Ian Goldberg (2020)              │
│                                                         │
│  阶段 1：分布式密钥生成 (DKG)                             │
│  ──────────────────────────                             │
│  1. 每个参与方 P_i 选择多项式 f_i(x)（t-1 次）           │
│     f_i(x) = a₀ + a₁x + ... + aₜ₋₁x^(t-1)             │
│     私钥份额 s_i = f_i(0)                                │
│                                                         │
│  2. P_i 将 f_i(j) 安全发送给 P_j                        │
│     （使用加密通道）                                      │
│                                                         │
│  3. 每方计算自己的私钥份额：                              │
│     sk_i = Σ f_j(i)  (j=1..n)                           │
│     公钥 PK = Σ f_j(0) × G = a₀_total × G              │
│                                                         │
│  阶段 2：签名（两轮交互）                                 │
│  ──────────────────────                                 │
│  Round 1 - 承诺阶段：                                    │
│  - 每方选择随机数 d_i, e_i                               │
│  - 计算承诺 D_i = d_i × G, E_i = e_i × G               │
│  - 广播承诺 (D_i, E_i)                                  │
│                                                         │
│  Round 2 - 响应阶段：                                    │
│  - 计算挑战 c = H(R, PK, msg)                            │
│  - 每方计算响应：                                        │
│    z_i = d_i + e_i × ρ_i + λ_i × sk_i × c              │
│    其中 ρ_i = H(i, D集合, E集合)                         │
│    λ_i = 拉格朗日系数                                    │
│  - 广播 z_i                                              │
│                                                         │
│  组合签名：                                              │
│  R = Σ D_i + Σ ρ_i × E_i                                │
│  z = Σ z_i                                               │
│  签名 = (R, z)（与单方 Schnorr 签名格式相同）            │
└─────────────────────────────────────────────────────────┘

门限签名应用场景：
┌─────────────────────────────────────────────────────────┐
│  加密货币钱包安全：                                       │
│  - 2-of-3 签名：手机 + 硬件钱包 + 备份                   │
│  - 单个设备丢失不影响资产安全                            │
│  - 比传统多签更节省手续费                                │
│                                                         │
│  跨链桥：                                                │
│  - 多个验证者共同签名跨链交易                            │
│  - 防止单点故障和串谋攻击                                │
│                                                         │
│  机构级托管：                                             │
│  - 多部门共同授权大额交易                                │
│  - 防止内部人员单方面操作                                │
│                                                         │
│  区块链共识：                                             │
│  - 随机数生成 (DRAND)：门限签名生成随机数                │
│  - 防止任何一方操纵随机数                                │
└─────────────────────────────────────────────────────────┘
```


> **Note:** Taproot 升级让比特币支持 Schnorr 签名和 MuSig2 聚合，使得多签交易和门限签名在链上不可区分，极大提升了隐私性。
