# 静态分析工具


## 1. 抽象解释 (Abstract Interpretation)

**核心思想（Patrick & Radhia Cousot, 1977）：**
用一个"抽象域"来近似地执行程序，获得程序行为的保守估计。如果在抽象域中没有发现错误，则真实程序也不会犯该类错误。

#### 具体域 vs 抽象域


- **具体域**
   ：所有可能的值（如所有整数）
- **抽象域**
   ：简化的值集合（如符号、区间）
- 示例：x = 3 + 4 → 具体值 7
- 抽象：x = 正 + 正 → 区间 [6, +∞)


#### 常见抽象域


- **符号域**
   ：正(+)/负(-)/零(0)/未知(?#)
- **区间域**
   ：[a, b] 表示值的范围
- **八角形域**
   ：±x ± y ≤ c
- **多面体域**
   ：线性不等式组
- **同余域**
   ：x ≡ a (mod n)


```
// 抽象解释示例：符号分析
int x = input();      // x: ? (未知)
if (x > 0) {
    // x: + (正)
    int y = x + 1;     // y: + (正 + 正 = 正)
    int z = x - 10;    // z: ? (正 - 正 = 未知，可能是负)
    if (z < 0) {
        // z: - (负)
        // 此时 x < 10 且 x > 0，所以 x ∈ [1, 9]
    }
}
int w = 1 / x;
// 抽象分析检查：x 是否可能为零？在本分支中 x > 0，所以安全。
```


### Galois 连接

**形式化基础：**
抽象解释通过 Galois 连接 (α, γ) 建立具体域 C 和抽象域 A 之间的映射关系。α 是抽象函数（具体→抽象），γ 是具体化函数（抽象→具体）。保证：如果抽象域中无错误，具体域中也无错误（sound but incomplete）。

## 2. 数据流分析 (Data Flow Analysis)

**核心思想：**
通过分析程序中变量的定义和使用路径，发现潜在的编程错误。是编译器优化和静态检查的基础。

### 分析类型


| 分析类型 | 信息流动方向 | 问题 | 传播规则 |
| --- | --- | --- | --- |
| 到达定义 (Reaching Defs) | 前向 (Forward) | 变量在哪定义？ | Gen ∪ (In - Kill) |
| 活跃变量 (Live Variables) | 后向 (Backward) | 变量还会被使用吗？ | Use ∪ (Out - Def) |
| 可用表达式 (Available Expr) | 前向 | 表达式已计算过吗？ | Gen ∩ (In - Kill) |
| 非常量表达式 (Very Busy) | 后向 | 表达式一定会被计算吗？ | Use ∩ (Out - Def) |


```
// 数据流分析示例：未初始化变量检测
void foo(int flag) {
    int x;              // x: 未初始化
    if (flag > 0) {
        x = 10;          // x: 已初始化（分支1）
    }
    // 控制流汇合点：x 可能已初始化（flag>0）或未初始化（flag≤0）
    print(x);            // ⚠️ 警告：x 可能未初始化
}

// 数据流分析：构建控制流图 (CFG)，传播"已初始化"集合
// 在汇合点取交集（必须在所有路径上都初始化才算安全）
```


### 流敏感 vs 流不敏感分析


#### 流敏感 (Flow-Sensitive)


- 考虑语句执行顺序
- 精度高，开销大
- 示例：活跃变量分析


#### 流不敏感 (Flow-Insensitive)


- 忽略执行顺序，分析整个函数
- 速度快，精度低
- 示例：指针别名分析


## 3. 符号执行 (Symbolic Execution)

**核心思想：**
用符号值（而非具体值）执行程序，收集路径约束条件，然后用约束求解器 (SMT) 求解出触发各路径的具体输入。

```
// 符号执行示例
int mystery(int x, int y) {
    int z = x + y;        // 符号: z = x + y
    if (z == 42) {         // 路径约束: x + y = 42
        if (x > y) {       // 路径约束: x + y = 42 ∧ x > y
            return 1;
        } else {           // 路径约束: x + y = 42 ∧ x ≤ y
            return 2;
        }
    } else {               // 路径约束: x + y ≠ 42
        return 0;
    }
}

// SMT 求解器求解：
// 路径1: x=30, y=12 (30+12=42 ∧ 30>12) → return 1
// 路径2: x=21, y=21 (21+21=42 ∧ 21≤21) → return 2
// 路径3: x=0,  y=0  (0+0≠42)           → return 0
```

## 4. 主流静态分析工具对比


| 工具 | 开发商 | 语言 | 技术 | 特点 |
| --- | --- | --- | --- | --- |
| Coverity | Synopsys | C/C++/Java/C#... | 路径敏感分析 | 工业级，精确率高，支持大规模项目 |
| Infer | Meta (Facebook) | C/ObjC/Java/C++ | 分离逻辑 + 抽象解释 | 增量分析，CI 友好，开源 |
| SonarQube | SonarSource | 30+ 种语言 | 模式匹配 + 数据流 | 规则丰富，与 CI 集成方便 |
| CodeQL | GitHub (Semmle) | 多语言 | 语义查询 | 可编写自定义查询，安全分析 |
| PVS-Studio | POuVS | C/C++/C# | 模式 + 数据流 | C++ 领域分析能力强 |
| Clang Static Analyzer | LLVM | C/C++/ObjC | 路径敏感符号执行 | 开源，集成在 Clang 中 |

## 5. 工具选择策略


#### 按项目需求选


- 安全关键系统 → Coverity / PVS-Studio
- CI/CD 集成 → Infer / SonarQube
- 开源安全审计 → CodeQL
- 代码规范检查 → ESLint / Checkstyle


#### 最佳实践


- 组合使用：Lint + 深度分析 + 安全扫描
- CI 中集成快速规则，定期运行深度分析
- 管理误报：逐步消除，记录抑制原因
- 关注 True Positive 率而非工具发现数量

**Soundness vs Completeness：**
静态分析工具是 sound 的（不会漏报）但不是 complete 的（会有误报）。这是根本性的取舍——要保证不漏报，就必须接受一定误报。实践中通过调优规则和抑制策略来平衡。
