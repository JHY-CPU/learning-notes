合约式设计与断言

一、合约式设计（Design by Contract）

合约式设计由Bertrand Meyer提出，通过明确的前置条件、后置条件和类不变式来定义软件组件之间的"契约"，从而提高软件可靠性。

核心思想：将软件组件之间的交互类比为商业合同。调用者（客户）满足前置条件，被调用者（供应商）保证后置条件。任何一方违约都会导致契约违反。

1.1 契约的三个组成部分

契约类型定义责任方检查时机
前置条件（Precondition）调用方法前必须满足的条件调用者（客户）方法入口
后置条件（Postcondition）方法返回时保证满足的条件被调用者（供应商）方法出口
类不变式（Class Invariant）类的每个实例在稳定状态下必须满足的条件双方构造后、每个公开方法前后

1.2 契约的作用

  • 文档化：契约比注释更精确地说明了方法的行为
  • 调试辅助：运行时检查契约，快速定位违反点
  • 测试支持：契约可自动生成测试用例
  • 设计指导：契约帮助明确职责分配

二、Eiffel语言

Eiffel是由Bertrand Meyer设计的面向对象编程语言，是合约式设计的原生支持语言。

2.1 Eiffel语法示例

银行账户示例：
class ACCOUNT
feature
    balance: INTEGER

    deposit (amount: INTEGER)
        require          -- 前置条件
            positive: amount > 0
        do
            balance := balance + amount
        ensure           -- 后置条件
            updated: balance = old balance + amount
        end

    withdraw (amount: INTEGER)
        require
            positive: amount > 0
            sufficient: balance >= amount
        do
            balance := balance - amount
        ensure
            updated: balance = old balance - amount
        end

invariant                 -- 类不变式
    non_negative: balance >= 0
end

2.2 Eiffel的继承与契约

  • 前置条件弱化：子类可以削弱（OR）前置条件
  • 后置条件强化：子类可以加强（AND）后置条件
  • 不变式继承：子类继承父类的不变式并可添加新条件
Liskov替换原则的体现：前置条件弱化和后置条件强化确保了子类可以安全替换父类，这是LSP的形式化表达。

三、Java断言（Assertions）

Java从1.4版本开始支持断言机制，通过assert语句检查运行时条件。

3.1 assert语法

assert condition;
assert condition : expression;

如果condition为false，抛出AssertionError。expression的值作为详细消息。

3.2 使用示例

// 前置条件检查
public void withdraw(double amount) {
    assert amount > 0 : "Amount must be positive";
    assert amount <= balance : "Insufficient funds";
    // ... 方法实现
    // 后置条件检查
    assert balance >= 0 : "Balance cannot be negative";
}

// 类不变式检查
private void checkInvariant() {
    assert balance >= 0 : "Balance must be non-negative";
    assert accountNumber != null : "Account number required";
}

3.3 启用与禁用

方式命令
启用断言java -ea MyClass
禁用断言（默认）java -da MyClass
启用特定类的断言java -ea:com.example.MyClass
启用包中所有类java -ea:com.example...
最佳实践：
  • 断言用于检查"不应该发生"的情况，而非正常的参数验证
  • 断言不应有副作用，因为可能被禁用
  • 公开方法的参数验证应使用IllegalArgumentException
  • 内部方法可使用断言进行不变式检查

五、契约违反处理

违反类型含义处理方式
前置条件违反调用者未满足输入要求调用者的bug，抛出异常
后置条件违反方法未达到承诺的结果被调用者的bug，抛出异常
不变式违反对象处于不一致状态类实现的bug，抛出异常
契约 vs 异常：
- 契约检查编程错误（bugs），异常处理运行时错误
- 契约违反通常不可恢复，异常通常可恢复
- 契约可在发布时禁用，异常始终需要处理
