# 大型项目通信策略

## 一、概念说明
大型项目中组件数量众多，通信关系复杂。需要建立清晰的**分层通信策略**，避免通信方式混乱导致的维护困难。

## 二、具体用法

### 2.1 分层通信策略
```
层级1: 父子直接通信 → Props + Emit
层级2: 祖先后代通信 → Provide/Inject
层级3: 全局共享状态 → Pinia Store
层级4: 跨应用通信   → postMessage / 事件
```

### 2.2 按模块划分 Store
```js
// stores/user.js
export const useUserStore = defineStore('user', () => {
  const profile = ref(null)
  const isLoggedIn = computed(() => !!profile.value)
  return { profile, isLoggedIn }
})

// stores/order.js
export const useOrderStore = defineStore('order', () => {
  const orders = ref([])
  const addOrder = (order) => orders.value.push(order)
  return { orders, addOrder }
})
```

### 2.3 Feature-based 架构
```
src/
├── features/
│   ├── user/
│   │   ├── components/
│   │   ├── composables/
│   │   └── stores/
│   └── order/
│       ├── components/
│       ├── composables/
│       └── stores/
├── shared/
│   ├── components/
│   ├── composables/
│   └── utils/
```

### 2.4 Store 间通信
```js
// stores/cart.js
import { useUserStore } from './user'

export const useCartStore = defineStore('cart', () => {
  const items = ref([])

  const checkout = async () => {
    const userStore = useUserStore()
    if (!userStore.isLoggedIn) {
      throw new Error('请先登录')
    }
    // ...
  }

  return { items, checkout }
})
```

### 2.5 通信类型决策树
```
1. 是否父子关系？→ Props/Emit
2. 是否跨 2-3 层？→ Provide/Inject
3. 是否全局共享？→ Pinia
4. 是否跨应用？→ postMessage
```

## 三、常见用例

| 项目规模 | 推荐策略 |
|---------|---------|
| 小型（< 5 页面） | Props/Emit + provide/inject |
| 中型（5-20 页面） | Props/Emit + Pinia |
| 大型（> 20 页面） | Feature-based + Pinia 模块化 |

## 四、注意事项与常见陷阱

- 不要在大项目中使用事件总线，会导致调试困难
- Pinia store 不宜过多，按业务模块划分
- 避免"上帝组件"（God Component），合理拆分职责
- 建立团队通信规范文档，保持一致性
- Store 之间可以互相引用，但避免循环依赖
- 使用 TypeScript 的 InjectionKey 统一管理依赖注入的 key
