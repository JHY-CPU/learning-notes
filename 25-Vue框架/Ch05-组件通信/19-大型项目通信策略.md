# 大型项目通信策略

## 一、概念说明
大型项目中组件数量众多，通信关系复杂。需要建立清晰的**分层通信策略**，避免通信方式混乱导致的维护困难。

## 二、具体用法

### 2.1 分层通信策略

```
层级1: 父子直接通信 → Props + Emit
层级2: 祖先后代通信 → Provide/Inject
层级3: 全局共享状态 → Pinia Store
层级4: 跨应用通信   → postMessage / 事件
```

### 2.2 按模块划分 Store
```js
// stores/user.js
export const useUserStore = defineStore('user', () => {
  const profile = ref(null)
  const isLoggedIn = computed(() => !!profile.value)
  return { profile, isLoggedIn }
})

// stores/order.js
export const useOrderStore = defineStore('order', () => {
  const orders = ref([])
  return { orders }
})
```

### 2.3 Feature-based 架构
```
src/
├── features/
│   ├── user/
│   │   ├── components/
│   │   ├── composables/
│   │   └── stores/
│   └── order/
│       ├── components/
│       ├── composables/
│       └── stores/
```

### 2.4 通信类型决策树
```vue
<script setup>
// 决策流程:
// 1. 是否父子关系? → Props/Emit
// 2. 是否跨 2-3 层? → Provide/Inject
// 3. 是否全局共享? → Pinia
// 4. 是否跨应用? → postMessage
</script>
```

## 三、注意事项与常见陷阱
- 不要在大项目中使用事件总线，会导致调试困难
- Pinia store 不宜过多，按业务模块划分
- 避免"上帝组件"（God Component），合理拆分职责
- 建立团队通信规范文档，保持一致性
