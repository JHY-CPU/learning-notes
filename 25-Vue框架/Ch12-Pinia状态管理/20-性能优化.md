# 性能优化

## 一、概念说明

Pinia本身性能良好，但在大型应用中需要注意：避免过度订阅、合理使用computed缓存、批量修改状态等。

```js
// ❌ 多次单独修改（多次触发更新）
store.name = '新名字'
store.age = 25
store.email = 'new@example.com'

// ✅ $patch批量修改（一次更新）
store.$patch({
  name: '新名字',
  age: 25,
  email: 'new@example.com'
})
```

## 二、具体用法

### 使用computed缓存

```js
// ❌ 每次渲染都计算
const totalPrice = () => {
  return store.items.reduce((s, i) => s + i.price * i.quantity, 0)
}

// ✅ computed有缓存
const totalPrice = computed(() =>
  store.items.reduce((s, i) => s + i.price * i.quantity, 0)
)
```

### 使用shallowRef存储大对象

```js
export const useDataStore = defineStore('data', () => {
  // 大数据使用shallowRef，不深度代理
  const largeList = shallowRef([])

  const fetchData = async () => {
    largeList.value = await api.getLargeList()
  }

  return { largeList, fetchData }
})
```

### 按需订阅

```vue
<script setup>
// ❌ 订阅整个store
const store = useBigStore()

// ✅ 只解构需要的字段
const { name, age } = storeToRefs(useBigStore())
</script>
```

## 四、避免不必要的响应式

```js
import { markRaw } from 'vue'

export const useConfigStore = defineStore('config', () => {
  // 配置对象不需要响应式
  const settings = markRaw({
    apiUrl: 'https://api.example.com',
    timeout: 10000,
    retries: 3
  })

  // 工具函数不需要响应式
  const utils = markRaw({
    format: (val) => val.toFixed(2),
    parse: (str) => JSON.parse(str)
  })

  return { settings, utils }
})
```

## 五、列表数据优化

```js
export const useListStore = defineStore('list', () => {
  const items = shallowRef([])  // 浅层响应

  // 整体替换，避免深度代理
  const setItems = (newItems) => {
    items.value = newItems
  }

  // 单个修改时触发整体更新
  const updateItem = (id, data) => {
    items.value = items.value.map(item =>
      item.id === id ? { ...item, ...data } : item
    )
  }

  // 删除项
  const removeItem = (id) => {
    items.value = items.value.filter(item => item.id !== id)
  }

  return { items, setItems, updateItem, removeItem }
})
```

## 六、性能优化检查清单

```
状态修改：
  ✓ 批量修改用 $patch
  ✓ 大对象用 shallowRef
  ✓ 不变数据用 markRaw
  ✓ 避免循环中频繁修改

计算缓存：
  ✓ 复杂计算用 getter/computed
  ✓ 避免在模板中直接调用方法计算

订阅：
  ✓ 只解构需要的字段
  ✓ 不需要的组件不订阅 store
  ✓ $subscribe 加 detached 仅在需要时

内存：
  ✓ 及时清理不需要的数据
  ✓ 避免存储大量缓存数据
  ✓ 大列表考虑虚拟滚动
```

## 三、注意事项与常见陷阱

1. `$patch`比逐个修改性能更好（合并为一次更新）
2. 大数组/对象使用`shallowRef`避免深度响应式
3. 避免在循环中频繁读取store
4. getter有缓存，复杂计算放在getter中
5. 不需要的组件不要导入不必要的store
6. `markRaw`标记不需要响应式的配置对象
7. shallowRef + 整体替换是处理大列表的最佳实践
