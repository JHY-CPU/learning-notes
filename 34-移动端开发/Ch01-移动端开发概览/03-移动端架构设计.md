# 移动端架构设计

## 一、概念说明

移动端架构设计是指在开发移动应用时，对代码结构、模块划分、数据流和依赖关系进行合理规划的过程。良好的架构设计能提高代码的可维护性、可测试性和可扩展性。

常见的移动端架构模式包括 MVC、MVP、MVVM 和 Clean Architecture 等。

```javascript
// React Native MVVM 架构示例目录结构
/*
src/

│   └── UserRepository.js
└── utils/           // 工具类
    └── helpers.js
*/
```

## 二、主流架构模式

### 2.1 MVC 模式

MVC（Model-View-Controller）是最经典的架构模式，将应用分为模型、视图和控制器三部分。

```swift
// iOS MVC 示例
// Model
struct User {
    let id: Int
    let name: String
    let email: String
}

// View
class UserView: UIView {
    let nameLabel = UILabel()
    let emailLabel = UILabel()

    func configure(with user: User) {
        nameLabel.text = user.name
        emailLabel.text = user.email
    }
}

// Controller
class UserViewController: UIViewController {
    var userView: UserView!
    var user: User?

    override func viewDidLoad() {
        super.viewDidLoad()
        loadUser()
    }

    func loadUser() {
        // Controller 负责协调 Model 和 View
        user = User(id: 1, name: "张三", email: "zhangsan@example.com")
        if let user = user {
            userView.configure(with: user)
        }
    }
}
```

### 2.2 MVVM 模式

MVVM（Model-View-ViewModel）通过 ViewModel 将视图逻辑与业务逻辑分离，支持数据绑定。

```javascript
// React Native MVVM 示例
// ViewModel - 使用自定义 Hook 实现
import { useState, useEffect } from 'react';
import { fetchUserData } from '../services/ApiService';

const useUserViewModel = (userId) => {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    loadUser();
  }, [userId]);

  const loadUser = async () => {
    try {
      setLoading(true);
      const data = await fetchUserData(userId);
      setUser(data);
      setError(null);
    } catch (err) {
      setError(err.message);
    } finally {
      setLoading(false);
    }
  };

  const updateUser = async (newData) => {
    try {
      const updated = await updateUser(userId, newData);
      setUser(updated);
    } catch (err) {
      setError(err.message);
    }
  };

  return { user, loading, error, updateUser, refresh: loadUser };
};

// View 层
const UserScreen = ({ userId }) => {
  const { user, loading, error, updateUser } = useUserViewModel(userId);

  if (loading) return <LoadingSpinner />;
  if (error) return <ErrorMessage message={error} />;

  return (
    <View>
      <Text>{user.name}</Text>
      <Text>{user.email}</Text>
      <Button title="编辑" onPress={() => updateUser(user)} />
    </View>
  );
};
```

### 2.3 Clean Architecture

Clean Architecture 强调关注点分离，将代码分为多个层级，依赖关系由外向内。

```dart
// Flutter Clean Architecture 示例
// Domain Layer - 业务实体
abstract class UserRepository {
  Future<User> getUser(int id);
  Future<void> saveUser(User user);
}

// Use Case - 用例
class GetUserUseCase {
  final UserRepository repository;

  GetUserUseCase(this.repository);

  Future<User> execute(int id) async {
    return await repository.getUser(id);
  }
}

// Data Layer - 数据实现
class UserRepositoryImpl implements UserRepository {
  final ApiClient apiClient;
  final LocalStorage localStorage;

  UserRepositoryImpl(this.apiClient, this.localStorage);

  @override
  Future<User> getUser(int id) async {
    // 先尝试从本地缓存获取
    final cached = await localStorage.getCachedUser(id);
    if (cached != null) return cached;

    // 从网络获取并缓存
    final user = await apiClient.fetchUser(id);
    await localStorage.cacheUser(user);
    return user;
  }

  @override
  Future<void> saveUser(User user) async {
    await apiClient.updateUser(user);
    await localStorage.cacheUser(user);
  }
}
```

## 三、状态管理与数据流

### 3.1 单向数据流

```javascript
// Redux 风格的单向数据流
// Action -> Reducer -> Store -> View -> Action
const userReducer = (state = initialState, action) => {
  switch (action.type) {
    case 'FETCH_USER_REQUEST':
      return { ...state, loading: true };
    case 'FETCH_USER_SUCCESS':
      return { ...state, loading: false, user: action.payload };
    case 'FETCH_USER_FAILURE':
      return { ...state, loading: false, error: action.error };
    default:
      return state;
  }
};
```

## 四、注意事项与常见陷阱

1. **避免过度架构**：小项目不需要复杂的分层架构，选择适合项目规模的模式
2. **保持单向依赖**：上层模块不应依赖下层实现细节，使用接口/抽象类解耦
3. **模块化拆分**：大型项目应按功能模块拆分，每个模块独立开发和测试
4. **避免循环依赖**：模块间依赖应是有向无环图（DAG），不能形成环路
5. **关注可测试性**：架构设计时应考虑单元测试的可行性，避免在 View 层编写业务逻辑
