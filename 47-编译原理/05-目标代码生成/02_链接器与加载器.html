链接器与加载器

Linker & Loader — 静态/动态链接、符号解析、重定位、ELF结构

1. 从源代码到执行

源代码 (.c) ──▶ 预处理 ──▶ 编译 ──▶ 汇编 ──▶ 链接 ──▶ 加载执行 main.c main.i main.s main.o a.out 内存中运行 (目标文件) (可执行文件) 编译器只处理单个翻译单元 链接器负责将多个翻译单元合并为一个可执行文件

2. 目标文件与ELF结构

ELF (Executable and Linkable Format) 是Unix/Linux下的标准二进制格式。

ELF文件结构: ┌─────────────────────────┐ │ ELF Header │ 魔数、类型、入口地址、段表偏移 ├─────────────────────────┤ │ Program Header Table │ 描述段 (加载时用) ├─────────────────────────┤ │ .text (代码段) │ 机器指令 ├─────────────────────────┤ │ .rodata (只读数据) │ 字符串常量等 ├─────────────────────────┤ │ .data (已初始化数据) │ 全局变量 ├─────────────────────────┤ │ .bss (未初始化数据) │ 不占文件空间 ├─────────────────────────┤ │ .symtab (符号表) │ 函数/变量名与地址映射 ├─────────────────────────┤ │ .strtab (字符串表) │ 符号名等字符串 ├─────────────────────────┤ │ .rel.text (重定位表) │ 代码段中需要修正的位置 ├─────────────────────────┤ │ Section Header Table │ 描述各节 (链接时用) └─────────────────────────┘ 
# 查看ELF文件信息
readelf -h main.o          # ELF头信息
readelf -S main.o          # 节头表
readelf -s main.o          # 符号表
readelf -r main.o          # 重定位表
objdump -d main.o          # 反汇编
nm main.o                  # 查看符号
file a.out                 # 文件类型

3. 符号解析 (Symbol Resolution)

每个目标文件都有符号表，记录定义和引用的符号。链接器的任务之一是将每个符号引用与唯一的符号定义关联。

符号类型

类型含义示例
Global Defined已定义的全局符号函数体、全局变量
Global Undefined引用但未定义extern声明
Local只在本文件可见static函数/变量
// main.c
extern int sum(int*, int);  // sum — 全局未定义符号
int array[2] = {1, 2};      // array — 全局已定义符号
static int count = 0;       // count — 本地符号

int main() {
    int s = sum(array, 2);  // 引用sum, 需要链接器解析
    return 0;
}

// sum.c
int sum(int* arr, int n) {  // sum — 全局已定义符号
    int total = 0;
    for (int i = 0; i < n; i++) total += arr[i];
    return total;
}

符号解析规则

  • 强符号：函数和已初始化的全局变量
  • 弱符号：未初始化的全局变量
  • 规则1：不允许有多个同名强符号
  • 规则2：若一个强符号和多个弱符号同名，选强符号
  • 规则3：若都是弱符号，任选一个

4. 重定位 (Relocation)

编译器生成目标文件时，不知道外部符号的最终地址。链接器在合并目标文件后，需要修正这些地址引用。

重定位过程: main.o (编译时): a.out (链接后): ┌──────────────┐ ┌──────────────┐ │ call 0x0 │ ← 占位地址 │ call 0x4010 │ ← 真实地址 │ ... │ │ ... │ └──────────────┘ └──────────────┘ .rel.text 记录了需要修正的位置: offset: 0x12, type: R_X86_64_PC32, symbol: sum, addend: -4 
// 重定位类型 (x86-64)
// R_X86_64_PC32 — PC相对寻址 (call/jmp指令)
//   修正值 = S + A - P
//   S = 符号的实际地址
//   A = addend (偏移量)
//   P = 需要修正的位置地址

// R_X86_64_32 — 绝对寻址 (mov指令)
//   修正值 = S + A

// 查看重定位信息
// $ objdump -r main.o
// OFFSET   TYPE              VALUE
// 00000012 R_X86_64_PC32     sum-0x00000004

5. 静态链接 (Static Linking)

在编译/链接时将所有库代码完整复制到可执行文件中。

静态链接过程: main.o ──┐ ├──▶ 链接器 (ld) ──▶ 完整的可执行文件 a.out sum.o ──┤ ↕ │ libc.a (静态库) utils.o ──┘ (只复制用到的目标文件) 静态库 (.a) = ar归档的多个 .o 文件 
# 创建和使用静态库
gcc -c utils.c -o utils.o        # 编译为目标文件
ar rcs libutils.a utils.o        # 创建静态库
gcc main.c -L. -lutils -o main   # 链接静态库

# 查看静态库内容
ar -t libutils.a                 # 列出库中目标文件

# 优点: 独立部署, 不依赖外部库
# 缺点: 文件大, 库更新需重新链接, 内存浪费(多进程各一份)

6. 动态链接 (Dynamic Linking)

库代码在运行时才被加载和链接。共享库 (`.so`/`.dll`/`.dylib`) 可被多个进程共享。

动态链接过程: 编译时: main.c ──▶ main.o ──▶ a.out (包含对 libmath.so 的引用, 不含库代码) 运行时: a.out 启动 │ ▼ 动态链接器 (ld-linux.so) │ ├── 加载 libmath.so 到内存 ├── 解析 PLT/GOT 中的符号 ├── 修正地址引用 ▼ 程序正常执行 多个进程共享同一份库的物理内存页!

PLT与GOT

PLT (Procedure Linkage Table) 和 GOT (Global Offset Table) 是实现动态链接的核心机制。

// 调用动态库函数的过程:
// 1. 代码调用PLT表项 (call printf@PLT)
// 2. PLT跳转到GOT中存储的地址
// 3. 首次调用: GOT指向PLT的下一条(解析代码), 解析后更新GOT
// 4. 后续调用: GOT直接指向真实函数, 跳转开销极小

// 懒绑定 (Lazy Binding) 伪代码:
PLT[n]:
    jmp *GOT[n]           // 跳到GOT记录的地址
    push n                // 压入符号索引 (首次才执行)
    jmp PLT[0]            // 调用动态链接器解析

// 查看动态链接信息
ldd a.out                 # 列出依赖的动态库
readelf -d a.out          # 查看.dynamic段
objdump -T libfoo.so     # 查看动态符号表

7. 加载器 (Loader)

加载器负责将可执行文件从磁盘读入内存并开始执行。

加载过程

  1. 读取ELF头，验证文件格式
  2. 解析Program Header Table
  3. 为各段创建虚拟内存映射 (mmap)
  4. 将文件内容映射到对应虚拟地址
  5. BSS段：映射匿名页 (零填充)
  6. 设置栈、堆、环境变量
  7. 跳转到入口点 (_start)
# 虚拟地址空间布局 (Linux x86-64)
0xffffffffffffffff ┌──────────────────┐ 内核空间
0x0000800000000000 ├──────────────────┤
                   │     栈 (↓增长)     │
                   │        ...        │
                   │    共享库映射区     │ ← mmap区域
                   │        ...        │
                   │     堆 (↑增长)     │
0x0000000000400000 ├──────────────────┤
                   │  可执行文件映射     │
                   │ .text .data .bss  │
0x0000000000000000 └──────────────────┘

8. 位置无关代码 (PIC, Position-Independent Code)

共享库需要支持加载到任意地址。PIC通过相对寻址和GOT实现。

// 编译位置无关代码
gcc -fPIC -shared -o libfoo.so foo.c

// PIC关键技巧:
// 1. 数据引用: 通过GOT间接访问 (GOT地址可由PC相对得到)
// 2. 函数调用: 通过PLT间接调用
// 3. 代码本身不包含绝对地址

// 获取GOT地址 (x86-64)
call __x86.get_pc_thunk.ax  // 将下一条指令地址放入eax
add eax, _GLOBAL_OFFSET_TABLE_  // 加上GOT的偏移
