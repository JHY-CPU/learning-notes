01 - 指令选择与寄存器分配

一、代码生成阶段概述

目标代码生成是编译器的最后一个阶段，将优化后的中间代码转换为目标机器的汇编代码或机器码。

1.1 代码生成的三个核心问题

中间代码(IR) │ ▼ ┌─────────────┐ ┌──────────────────┐ │ 指令选择 │ │ 选择哪些机器指令 │ │ Instruction │ │ 实现IR操作 │ │ Selection │ │ (树模式匹配) │ └──────┬──────┘ └──────────────────┘ │ ▼ ┌─────────────┐ ┌──────────────────┐ │ 寄存器分配 │ │ 变量分配到哪些 │ │ Register │ │ 寄存器/溢出到内存 │ │ Allocation │ │ (图着色/线性扫描) │ └──────┬──────┘ └──────────────────┘ │ ▼ ┌─────────────┐ ┌──────────────────┐ │ 指令调度 │ │ 重排指令以利用 │ │ Instruction │ │ 流水线/多发射 │ │ Scheduling │ │ (列表调度) │ └──────┬──────┘ └──────────────────┘ │ ▼ 目标机器代码

二、指令选择（Instruction Selection）

指令选择的任务是为目标机器的每条IR指令选择最优的机器指令序列。

2.1 基于树模式匹配的方法

将IR表示为树，将机器指令表示为树模式（pattern），通过模式匹配选择指令。

IR树： t1 = a + b * c

(+) / \ a (*) / \ b c 可选的指令模式: ┌────────────────────────────────────────┐ │ 模式1: (+) ──→ ADD reg, reg │ │ 模式2: (*) ──→ MUL reg, reg │ │ 模式3: (+(*))──→ LEA reg, [reg+reg*scale]│ └────────────────────────────────────────┘

2.2 树覆盖（Tree Covering）

用最少的指令模式覆盖整棵IR树，是NP完全问题，实际用贪心或动态规划近似。

2.3 BURG / IBURG 工具

BURG（Bottom-Up Rewrite Generator）根据机器描述自动生成树模式匹配器：

/* BURG 规则示例 */ stmt: REG /* 寄存器直接可用 */ stmt: ADD(reg, reg) = { emit("add", $1, $2); } stmt: MUL(reg, reg) = { emit("mul", $1, $2); } reg: LOAD(mem) = { emit("mov", $$, $1); } mem: ADD(reg, ICON) = { emit("lea", $$, "[$1+$2]"); }
DAG覆盖 vs 树覆盖：IR可以是DAG（共享公共子表达式），DAG覆盖比树覆盖更高效，但更复杂。LLVM使用DAG模式匹配。

三、寄存器分配（Register Allocation）

寄存器分配决定程序中的每个值（变量/临时变量）存储在哪个物理寄存器中。目标是最小化寄存器溢出（spilling）到内存的开销。

3.1 干涉图（Interference Graph）

若两个变量在同一时刻同时活跃（live），则它们之间存在干涉边 — 它们不能分配到同一个寄存器。

基本块：

t1 = a + b // t1 定义 t2 = t1 * c // t1 使用, t2 定义 d = t2 + 1 // t2 使用, d 定义 // t1 和 t2 在指令2处同时活跃 → 干涉 // t2 和 d 不同时活跃（t2在d定义后不再使用）

干涉图：

a ─── t1 b ─── t1 c ─── t2 t1 ── t2 ← 干涉边 t2 ── d

3.2 图着色寄存器分配

寄存器分配问题等价于干涉图的k-着色问题（k为可用寄存器数）。若图可k-着色，每个颜色对应一个物理寄存器。

Chaitin 算法步骤：

  1. 构建干涉图：根据活跃变量分析构建
  2. 简化：若存在度 < k 的节点，移除并压栈
  3. 溢出：若所有节点度 ≥ k，选择一个溢出候选节点压栈
  4. 选择：从栈中弹出节点，分配颜色（寄存器）
  5. 实际溢出处理：若有节点无法着色，插入 spill 代码，重新构建干涉图
4个寄存器可用 (k=4) 简化: 选择: 移除度<4的节点 d 弹栈并着色 移除度<4的节点 t2 a=R0, b=R1, c=R2 ... t1=R3, t2=R0, d=R1 压栈: [d, t2, t1, ...]

四、寄存器溢出（Register Spilling）

当寄存器不够用时，需要将某些变量溢出到内存（栈帧）。

4.1 溢出策略

  • 积极溢出：每次使用/定义都插入 load/store 指令
  • 惰性溢出：仅在必要时才插入 spill 代码，可能利用值仍在缓存中的情况

4.2 溢出代码的插入

溢出前：
t1 = a + b
t2 = t1 * c
（t1需要溢出）
溢出后：
t1 = a + b
store t1, [spill_addr]
t1 = load [spill_addr]
t2 = t1 * c

4.3 溢出代价模型

选择溢出哪个变量时，考虑以下因素：

  • 变量的使用频率（use-def chain）
  • 溢出导致的额外 load/store 指令数
  • 变量是否在循环中被频繁访问

五、线性扫描寄存器分配

线性扫描（Linear Scan）是图着色的简化替代，效率更高，常用于 JIT 编译器。

5.1 算法思想

  1. 计算每个变量的活跃区间（live interval）
  2. 按起始位置排序
  3. 依次为每个变量分配寄存器，优先重用已释放的寄存器
  4. 溢出代价最高的活跃区间优先溢出

5.2 活跃区间示例

指令序号: 1 2 3 4 5 6 7 8 a: [===) b: [=======) c: [===) t1: [===) t2: [=======) d: [===) 可用寄存器 k=3: 区间1: a → R0 区间2: b → R1, c → R2 区间3: t1 → R0 (a已结束) 区间4: t2 → R1 (b已结束) 区间5: d → R0 (t1已结束) → 无需溢出！

六、指令调度（Instruction Scheduling）

6.1 指令级并行

现代处理器支持超标量和乱序执行，指令调度的目标是重排指令顺序以最大化指令级并行。

6.2 数据依赖

依赖类型定义能否重排
真依赖 (RAW)指令j读取指令i写入的数据不能
反依赖 (WAR)指令j写入指令i读取的数据寄存器重命名后可消除
输出依赖 (WAW)指令i和j写入同一位置寄存器重命名后可消除

6.3 列表调度算法

  1. 构建数据依赖图（DAG）
  2. 计算每个节点的优先级（如最长路径长度）
  3. 从就绪队列中选择最高优先级的指令发射
  4. 更新就绪队列，重复直到调度完成

七、代码生成完整流程

优化后的IR (SSA形式) │ ├─ 1. 指令选择 (树模式匹配 / DAG覆盖) │ 将IR操作映射为机器指令 │ ├─ 2. 构建干涉图 │ 根据活跃区间确定变量间的干涉关系 │ ├─ 3. 寄存器分配 (图着色 / 线性扫描) │ 将虚拟寄存器映射为物理寄存器 │ 不够时插入 spill 代码 │ ├─ 4. 指令调度 (列表调度) │ 重排指令以利用流水线 │ ├─ 5. 代码布局 │ 基本块排序、分支优化 │ ▼ 目标汇编代码
