# 05-目标代码生成（Code Generation）

> 目标代码生成是编译器的最后一个阶段，其任务是将经过优化的中间代码转换为目标机器的汇编代码或机器码。这一阶段需要考虑目标机器的指令集、寄存器组织、寻址方式等硬件特性。

---

## 1. 目标代码生成器的输入与输出

### 1.1 输入

- 经过优化的**中间代码**（通常是三地址码或SSA形式）
- **符号表**信息（变量类型、存储位置等）
- **寄存器信息**（可用寄存器集合）

### 1.2 输出

- **绝对机器代码**：可直接执行的机器码，内存位置固定
- **可重定位机器代码**：需要链接器处理，可加载到任意内存位置（最常用）
- **汇编代码**：需要汇编器处理

### 1.3 设计目标

- **正确性**：生成的代码必须与源程序语义一致
- **高效性**：充分利用目标机器资源（寄存器、指令集特性）
- **快速生成**：代码生成本身不应耗费过多时间

---

## 2. 目标机器模型

### 2.1 寄存器组织

假设目标机器有 `n` 个通用寄存器 `R0, R1, ..., Rn-1`。

典型架构示例（简化）：
- **通用寄存器**：用于算术运算和存储临时值
- **专用寄存器**：如PC（程序计数器）、SP（栈指针）、FP（帧指针）
- **特殊寄存器**：如乘法结果寄存器

### 2.2 指令格式

典型的指令集支持以下指令类型：

| 指令类型 | 格式 | 含义 |
|----------|------|------|
| **加载指令** | `LD R, x` | 将x的内容加载到寄存器R |
| **存数指令** | `ST x, R` | 将R的内容存储到x |
| **算术指令** | `OP R1, R2, R3` | R1 = R2 OP R3 |
| **立即数运算** | `OP R1, R2, c` | R1 = R2 OP c |
| **跳转指令** | `BR L` | 无条件跳转到L |
| **条件跳转** | `Bcond R, L` | 条件满足时跳转到L |

### 2.3 寻址方式

| 寻址方式 | 格式 | 示例 |
|----------|------|------|
| 寄存器寻址 | `R` | `LD R1, R2` |
| 直接寻址 | `M` | `LD R1, x`（x为内存地址） |
| 间接寻址 | `(R)` | `LD R1, (R2)` |
| 立即数寻址 | `#c` | `ADD R1, R2, #5` |
| 变址寻址 | `c(R)` | `LD R1, 100(R2)`（基址+偏移） |

## 3. 指令选择

### 3.1 树模式匹配

将中间代码表示为树结构，用机器指令的模式去匹配树的子树。

```
赋值语句: a = b + c

树结构:
    +
   / \
  b   c

匹配模式:
ADD R1, R2, R3  对应  R1 ← R2 + R3 的子树
```

### 3.2 动态规划方法（树覆盖）

将指令选择问题转化为**最优树覆盖**问题：
- 用指令模式覆盖表达式树的所有节点
- 目标：使总代价最小

**标记算法（标记法）**：
```
1. 自底向上遍历树
2. 对每个节点 n，找到代价最小的模式覆盖以 n 为根的子树
3. 记录每个节点的最优覆盖代价和使用的模式
```

### 3.3 代价模型

每条指令有一个**代价**：
- 代价为1：简单寄存器操作
- 代价为1+内存访问代价：涉及内存的操作
- 代价为特殊值：根据指令执行时间设定

目标：生成总代价最小的代码序列。

## 4. 寄存器分配

### 4.1 寄存器描述符与地址描述符

**寄存器描述符（Register Descriptor）**：
- 对每个寄存器，记录当前存储了哪些变量的值
- `value(R1) = {a, c}` 表示R1中保存了变量a和c的值

**地址描述符（Address Descriptor）**：
- 对每个变量（名字），记录其当前值存储在哪些位置
- `location(a) = {R1, M[addr]}` 表示a的值同时在R1和内存中

### 4.2 简单寄存器分配（基本块内）

对基本块内的每条指令，选择寄存器的原则：

1. **getReg(I)** 函数：为指令 `I: x = y op z` 分配寄存器
   - 如果 y 已在某寄存器中，优先使用该寄存器
   - 如果没有空闲寄存器，需要选择一个寄存器溢出（spill）到内存
   - 溢出策略：选择将来最晚使用的寄存器

2. **生成加载指令**：如果操作数不在寄存器中，生成 `LD` 指令
3. **生成计算指令**：执行算术运算
4. **更新描述符**：更新寄存器和地址描述符
5. **生成存数指令**：如果结果需要存到内存

### 4.3 图着色寄存器分配

**基本思想**：
1. 构建**冲突图（Interference Graph）**：每个变量为一个节点，如果两个变量在同一时间活跃，则它们之间有边相连
2. 对冲突图进行**图着色**：相邻节点颜色不同，每种颜色对应一个寄存器
3. 如果颜色数（寄存器数）不足，需要溢出某些变量到内存

**简化与选择**：
- **简化**：如果节点的度数 < K（寄存器数），则可以移除该节点
- **溢出**：如果所有节点度数 >= K，选择一个节点标记为溢出
- **选择**：按简化时移除的逆序为节点分配寄存器

### 4.4 寄存器分配的优先级

决定哪些变量优先分配寄存器的准则：
1. **引用次数**：引用越多的变量越应该驻留在寄存器中
2. **活跃范围**：活跃范围短的变量优先（更容易释放寄存器）
3. **循环内变量**：循环内的变量引用比循环外的更值得放在寄存器中
4. **定义-使用距离**：定义后很快使用的变量适合放在寄存器中

## 5. 基本块内的代码生成

### 5.1 代码生成算法

```
对基本块中的每条指令 I: x = y op z:
  1. 调用 getReg(I) 为 x 分配寄存器 Rx
  2. 若 y 不在 Rx 中：
     生成 LD Ry, y
  3. 若 z 不在寄存器中：
     生成 LD Rz, z
  4. 生成 op Rx, Ry, Rz
  5. 更新描述符
```

### 5.2 示例

基本块：
```
t1 = a + b
t2 = t1 * c
t3 = t2 + d
```

假设可用寄存器为 R0、R1、R2：

```
LD  R0, a        // R0 = a, location(a)={R0}
LD  R1, b        // R1 = b, location(b)={R1}
ADD R0, R0, R1   // R0 = a + b, location(t1)={R0}
LD  R1, c        // R1 = c
MUL R0, R0, R1   // R0 = t1 * c, location(t2)={R0}
LD  R1, d        // R1 = d
ADD R0, R0, R1   // R0 = t2 + d, location(t3)={R0}
```

## 6. 基于DAG的代码生成

### 6.1 基本块的DAG

将基本块表示为DAG可以发现：
- 公共子表达式：同一节点被多个变量引用
- 常量：叶节点为常量
- 变量重定义：一个节点附加多个变量名

### 6.2 从DAG生成代码

**拓扑排序**：按DAG的拓扑序（自底向上）生成代码。

**节点顺序选择**：
1. 优先选择引用计数高的节点（更频繁使用的子表达式）
2. 尽量将结果直接放入目标变量的寄存器中
3. 考虑指令级并行的机会

### 6.3 示例

```
基本块:
t1 = a + b
t2 = a + b    // 与t1相同
t3 = t1 * c
t4 = t2 * c    // 与t3相同

DAG:
      *        ← t3, t4
     / \
   +    c
  / \
 a   b         ← t1, t2

优化后代码:
t1 = a + b
t3 = t1 * c
t2 = t1        // 复写
t4 = t3        // 复写
```

## 7. 窥孔优化（Peephole Optimization）

### 7.1 基本思想

在生成的目标代码上使用一个固定大小的"窗口"（窥孔），扫描并替换局部的低效指令序列。

### 7.2 常见优化模式

**冗余加载/存储消除**：
```
LD R0, a        →    LD R0, a
ST a, R0              （删除冗存）
```

**不可达代码消除**：
```
BR L1           →    BR L1
... (L1之前的代码)    （删除不可达代码）
L1:
```

**控制流简化**：
```
BR L1           →    Bcond R, L2
...                  （删除跳转到跳转）
L1: BR L2
```

**强度削弱**：
```
MUL R0, R1, #2  →    SHL R0, R1, #1
```

**代数化简**：
```
ADD R0, R1, #0  →    （删除，直接用R1）
MUL R0, R1, #1  →    （删除，直接用R1）
```

**消除比较指令**：
```
CMP R0, #0      →    （如果后续是条件跳转，可用TEST指令）
BNE L1
```

### 7.3 窥孔优化的特点

- 实现简单，效果明显
- 通常在代码生成的最后阶段执行
- 可以多次迭代扫描直到没有改进

## 8. 垃圾回收算法

### 8.1 基本概念

垃圾回收（Garbage Collection, GC）是自动内存管理的核心机制，用于回收程序不再使用的堆内存。

**可达性分析**：从根集合（全局变量、栈上变量、寄存器）出发，通过指针追踪能到达的所有对象，不可到达的对象即为垃圾。

### 8.2 标记-清除算法（Mark and Sweep）

**步骤**：
1. **标记阶段**：从根集合出发，深度优先遍历所有可达对象并标记
2. **清除阶段**：扫描整个堆，回收所有未标记的对象

**特点**：
- 实现简单
- 不需要移动对象，指针不会失效
- 产生**内存碎片**
- 暂停时间与堆大小成正比

### 8.3 复制算法（Copying / Scavenging）

**步骤**：
1. 将堆分为两个相等的半区：From空间和To空间
2. 从根集合出发，将所有可达对象从From空间复制到To空间
3. 清空From空间
4. 交换From和To的角色

**特点**：
- 没有内存碎片
- 只需扫描存活对象，与存活对象数量成正比
- 内存利用率只有50%（需两个半区）
- 适用于存活对象少的场景（年轻代）

### 8.4 标记-整理算法（Mark and Compact）

**步骤**：
1. **标记阶段**：同标记-清除
2. **整理阶段**：将所有存活对象移动到堆的一端，紧凑排列
3. **清除阶段**：回收整理后剩余的空闲空间

**特点**：
- 消除内存碎片
- 指针需要更新（对象移动后）
- 移动对象的开销较大

### 8.5 分代回收（Generational GC）

**基本思想**：根据对象的存活时间将堆划分为多个代（Generation）：
- **年轻代（Young Generation）**：新创建的对象，死亡率高，使用复制算法
- **老年代（Old Generation）**：多次GC后存活的对象，使用标记-清除或标记-整理
- **持久代/元空间**：存储类元数据

**晋升策略**：对象在年轻代经过一定次数的GC后仍存活，则晋升到老年代。

**特点**：
- 大多数对象朝生夕死，年轻代GC频繁但代价小
- 老年代GC频率低，减少全堆扫描
- 是现代JVM、V8等运行时的主流GC策略

### 8.6 各算法比较

| 算法 | 时间复杂度 | 空间开销 | 碎片问题 | 指针更新 | 适用场景 |
|------|-----------|---------|---------|---------|---------|
| 标记-清除 | O(存活+堆) | 低 | 有碎片 | 不需要 | 通用 |
| 复制 | O(存活) | 50% | 无碎片 | 需要 | 年轻代 |
| 标记-整理 | O(存活+堆) | 低 | 无碎片 | 需要 | 老年代 |
| 分代 | 综合 | 中等 | 较少 | 需要 | 大型应用 |

## 9. 即时编译（JIT）基础

### 9.1 基本概念

即时编译（Just-In-Time Compilation）是在程序**运行时**将中间代码（字节码）编译为本地机器码的技术。

```
源代码 → 编译器 → 字节码 → JIT编译器 → 本地机器码 → 执行
                              ↑
                         运行时触发
```

### 9.2 与AOT编译的对比

| 特性 | AOT（提前编译） | JIT（即时编译） |
|------|----------------|----------------|
| 编译时机 | 程序运行前 | 程序运行时 |
| 启动速度 | 快 | 慢（首次执行需编译） |
| 运行时优化 | 无运行时信息 | 可利用运行时信息 |
| 峰值性能 | 取决于静态分析 | 可能更高（自适应优化） |
| 内存占用 | 较低 | 较高（需存储编译结果） |

### 9.3 热点检测

JIT编译器通常只编译"热点代码"（频繁执行的代码）：
- **方法调用计数**：统计方法被调用的次数
- **回边计数**：统计循环体执行的次数
- **阈值触发**：当计数超过阈值时触发编译

### 9.4 分层编译

现代JIT通常采用多层编译策略：

```
第0层：解释执行（无编译）
第1层：简单的JIT编译（C1编译器），快速但优化少
第2层：带有完整优化的JIT编译（C2编译器），慢但优化充分
```

### 9.5 运行时优化技术

JIT可以利用AOT无法获得的运行时信息进行优化：
- **内联缓存（Inline Caching）**：缓存虚函数调用的实际目标
- **基于Profile的优化**：根据分支频率优化热点路径
- **去虚拟化**：当某个接口类型只有一个实现时，直接调用
- **逃逸分析**：对象不逃逸出方法时可在栈上分配

### 9.6 去优化（Deoptimization）

当运行时假设不再成立时，JIT需要回退到解释执行或更保守的编译代码：
- 类加载导致类型信息变化
- 分支预测失败
- 内联缓存失效

去优化是JIT区别于AOT的重要特性，允许激进优化同时保证正确性。

---

## 参考与总结

目标代码生成是编译器将优化后的中间代码转化为机器可执行代码的关键阶段。指令选择、寄存器分配和调度是代码生成的三大核心问题。408考试中，简单寄存器分配策略、窥孔优化的常见模式是重点考查内容。垃圾回收算法（标记-清除、复制、标记-整理、分代回收）虽然不属于传统编译器的范畴，但在现代语言实现中极为重要，也是408常考知识点。
