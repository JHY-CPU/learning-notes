# 02-语法分析（Syntax Analysis / Parsing）

> 语法分析是编译过程的第二个阶段，其任务是根据语言的语法规则（文法），将Token流组织成语法结构（通常是语法树），并检测语法错误。语法分析器是编译器的核心组件。

---

## 1. 自顶向下分析（Top-Down Parsing）

自顶向下分析从文法的开始符号出发，试图推导出与输入Token串匹配的句型。

### 1.1 基本思想

- 从根节点（开始符号）开始，自上而下构造语法树
- 每一步选择一个非终结符进行扩展（替换为它的一个产生式右部）
- 目标：产生与输入串匹配的最左推导

### 1.2 面临的问题

- **回溯**：当选择错误的产生式时，需要撤销并尝试其他产生式，效率低
- **左递归**：存在形如 `A → Aα` 的产生式会导致无限递归

---

## 2. 递归下降分析（Recursive Descent Parsing）

### 2.1 基本原理

为文法中的每个非终结符编写一个递归的子程序（函数）。当分析过程中需要扩展某个非终结符时，调用对应的子程序。

### 2.2 实现示例

对于文法：
```
E  → T E'
E' → + T E' | ε
T  → F T'
T' → * F T' | ε
F  → ( E ) | id
```

对应的递归下降分析器（伪代码）：
```c
void E()  { T(); E_prime(); }
void E_prime() {
    if (lookahead == '+') { match('+'); T(); E_prime(); }
    // else: ε产生式，什么都不做
}
void T()  { F(); T_prime(); }
void T_prime() {
    if (lookahead == '*') { match('*'); F(); T_prime(); }
}
void F() {
    if (lookahead == 'id') match('id');
    else if (lookahead == '(') { match('('); E(); match(')'); }
    else error();
}
```

### 2.3 优缺点

- **优点**：实现简单，易于理解
- **缺点**：需要消除左递归和提取左公因子；回溯导致效率低

## 3. LL(1)文法

### 3.1 LL(1)的含义

- 第一个 **L**：从左到右扫描输入（Left-to-right scan）
- 第二个 **L**：产生最左推导（Leftmost derivation）
- **(1)**：每步只需向前看1个符号（1 symbol lookahead）

### 3.2 三个关键集合

#### FIRST集

`FIRST(α)` = 从 `α` 能推导出的所有串的首终结符集合。

**计算规则：**
1. 若 `α = aβ`（a为终结符），则 `FIRST(α)` 包含 `a`
2. 若 `α = Xβ`（X为非终结符），则 `FIRST(α)` 包含 `FIRST(X) - {ε}`
3. 若 `X →* ε`，则 `FIRST(α)` 还包含 `FIRST(β)`
4. 若 `α →* ε`，则 `FIRST(α)` 包含 `ε`

#### FOLLOW集

`FOLLOW(A)` = 在某个句型中紧跟在非终结符 `A` 右边的终结符集合。

**计算规则：**
1. 将 `$`（输入结束符）放入 `FOLLOW(S)`（S为开始符号）
2. 若有产生式 `B → αAβ`，则将 `FIRST(β) - {ε}` 加入 `FOLLOW(A)`
3. 若有产生式 `B → αA` 或 `B → αAβ` 且 `β →* ε`，则将 `FOLLOW(B)` 加入 `FOLLOW(A)`

#### SELECT集

`SELECT(A → α)` 为使用产生式 `A → α` 的条件集合。

**计算规则：**
- 若 `ε ∉ FIRST(α)`：`SELECT(A → α) = FIRST(α)`
- 若 `ε ∈ FIRST(α)`：`SELECT(A → α) = (FIRST(α) - {ε}) ∪ FOLLOW(A)`

### 3.3 LL(1)文法的判定条件

文法G是LL(1)文法，当且仅当对于每个非终结符A的任意两个不同产生式 `A → α` 和 `A → β`：

```
SELECT(A → α) ∩ SELECT(A → β) = ∅
```

即同一非终结符的不同产生式的SELECT集互不相交。

## 4. LL(1)分析表的构造

### 4.1 分析表结构

LL(1)分析表是一个二维表 `M[A, a]`：
- **行**：非终结符
- **列**：终结符 + `$`
- **表项**：产生式编号（或产生式本身）

### 4.2 构造算法

```
对于每个产生式 A → α：
  计算 SELECT(A → α)
  对于 SELECT 中的每个终结符 a：
    将 A → α 填入 M[A, a]
```

### 4.3 预测分析过程

使用一个**分析栈**和**输入缓冲区**：

```
1. 将 $ 和开始符号 S 压入栈（S在栈顶）
2. 设 a 为当前输入符号
3. 令 X 为栈顶符号：
   a) X 是终结符且 X == a：弹出X，读入下一个符号
   b) X 是终结符且 X != a：报告错误
   c) X 是非终结符：查表 M[X, a]
      - 若 M[X, a] = X → Y1Y2...Yk：弹出X，将Yk...Y2Y1压栈
      - 若 M[X, a] = 空：报告错误
4. 重复直到栈为空
```

## 5. 消除左递归

### 5.1 直接左递归

形如 `A → Aα | β`（β不以A开头）

**消除方法**：改写为右递归
```
A  → βA'
A' → αA' | ε
```

### 5.2 间接左递归

```
S → Aa | b
A → Ac | Sd | ε
```

**消除步骤（通用算法）**：

```
1. 将非终结符排序为 A1, A2, ..., An
2. for i = 1 to n:
     for j = 1 to i-1:
       将 Aj → δ 替换为 Aj → γ1γ2...γk（其中 δ = Aiγ1...γk）
     消除 Ai 的直接左递归
```

### 5.3 注意事项

- 消除左递归不改变语言本身，但可能改变文法的分析方式
- 消除后需检查是否满足LL(1)条件

---

## 6. 提取左公因子

### 6.1 问题描述

当一个非终结符的多个产生式有共同前缀时：
```
A → αβ1 | αβ2 | γ
```
分析器无法确定使用哪个产生式。

### 6.2 提取方法

```
A  → αA' | γ
A' → β1 | β2
```

### 6.3 迭代提取

左公因子可能需要多次提取才能完全消除。提取左公因子是使文法满足LL(1)条件的必要步骤之一。

## 7. 自底向上分析（Bottom-Up Parsing）

### 7.1 基本思想

从输入串（叶子节点）出发，不断将句柄（handle）归约为非终结符，最终归约到开始符号（根节点）。

### 7.2 句柄（Handle）

- 句柄是当前句型中与某个产生式右部匹配的子串
- 对句柄进行归约是最右推导的逆过程
- 句柄是"最左直接短语"

### 7.3 移进-归约分析

使用一个栈来存储已分析的符号：

| 操作 | 描述 |
|------|------|
| **移进（Shift）** | 将下一个输入符号移入栈顶 |
| **归约（Reduce）** | 将栈顶的句柄替换为对应的非终结符 |
| **接受（Accept）** | 分析成功 |
| **报错（Error）** | 发现语法错误 |

## 8. LR(0)项目集与自动机

### 8.1 LR(0)项目

在产生式右部的某个位置加一个点 `.`，表示当前分析进度。

例如产生式 `A → XYZ` 对应的项目：
```
A → .XYZ    (尚未分析任何符号)
A → X.YZ    (已分析X)
A → XY.Z    (已分析XY)
A → XYZ.    (已完整分析，可归约)
```

### 8.2 增广文法

引入新的开始符号 `S'`，添加产生式 `S' → S`，确保接受项目唯一。

### 8.3 项目集规范族（CLOSURE与GOTO）

**CLOSURE(I)**：计算项目集I的闭包
```
CLOSURE(I) = I ∪ {所有能从I通过非终结符推导出的项目}
具体：
  若 A → α.Bβ ∈ CLOSURE(I)，B为非终结符，
  则 B → .γ 也加入 CLOSURE(I)（对B的所有产生式）
```

**GOTO(I, X)**：从项目集I经符号X转移
```
GOTO(I, X) = CLOSURE({ A → αX.β | A → α.Xβ ∈ I })
```

### 8.4 LR(0)自动机

- 状态 = 项目集规范族中的每个项目集
- 转移 = GOTO函数
- 初始状态 = CLOSURE({S' → .S})

### 8.5 LR(0)分析表

根据项目集构造分析表：
- 若项目 `A → α.aβ ∈ Ii` 且 `GOTO(Ii, a) = Ij`：`ACTION[i, a] = Shift j`
- 若项目 `A → α. ∈ Ii`（A ≠ S'）：对所有终结符a，`ACTION[i, a] = Reduce A → α`
- 若项目 `S' → S. ∈ Ii`：`ACTION[i, $] = Accept`
- 若 `GOTO(Ii, A) = Ij`：`GOTO[i, A] = j`

### 8.6 LR(0)的局限

LR(0)不使用向前看符号，当项目集中同时存在移进项目和归约项目时产生**移进-归约冲突**；存在多个归约项目时产生**归约-归约冲突**。SLR通过FOLLOW集来解决此问题。

## 9. SLR(1)分析

### 9.1 基本思想

SLR（Simple LR）在LR(0)的基础上，利用FOLLOW集来限制归约动作。

### 9.2 分析表构造规则

- **移进**：同LR(0)
- **归约**：若项目 `A → α. ∈ Ii`，仅对 `a ∈ FOLLOW(A)` 设 `ACTION[i, a] = Reduce A → α`

### 9.3 SLR(1)的局限

SLR虽然能解决部分LR(0)的冲突，但有时FOLLOW集过于宽松，仍无法消除所有冲突。例如当 `a ∈ FOLLOW(A)` 但 `a` 实际上不会出现在 `A` 的当前位置之后时。

---

## 10. LR(1)分析（规范LR分析）

### 10.1 LR(1)项目

在LR(0)项目的基础上增加一个**向前看符号**：

```
[A → α.β, a]
```

其中 `a` 是一个终结符（或 `$`），表示在归约为A之后，期望的下一个输入符号。

### 10.2 CLOSURE与GOTO的修改

**CLOSURE(I)**：
```
若 [A → α.Bβ, a] ∈ CLOSURE(I)
对 B → γ 的每个产生式，对每个 b ∈ FIRST(βa)：
  将 [B → .γ, b] 加入 CLOSURE(I)
```

**GOTO(I, X)**：
```
GOTO(I, X) = CLOSURE({ [A → αX.β, a] | [A → α.Xβ, a] ∈ I })
```

### 10.3 LR(1)分析表构造

- 若 `[A → α.aβ, b] ∈ Ii` 且 `GOTO(Ii, a) = Ij`：`ACTION[i, a] = Shift j`
- 若 `[A → α., a] ∈ Ii`：`ACTION[i, a] = Reduce A → α`

### 10.4 LR(1)的特点

- 能分析的文法范围最广（比SLR和LALR都广）
- 状态数非常多（通常是LR(0)的数十倍），实际开销大

---

## 11. LALR(1)分析

### 11.1 基本思想

LALR（Look-Ahead LR）是LR(1)和SLR的折中方案。将LR(1)中**核心相同**（LR(0)项目部分相同）的项目集合并。

### 11.2 合并方法

1. 构造完整的LR(1)项目集族
2. 找出核心相同的项目集（向前看符号可能不同）
3. 将它们的向前看符号取并集，合并为一个状态

### 11.3 LALR(1)的特点

- 状态数与LR(0)相同（比LR(1)少得多）
- 分析能力介于SLR和LR(1)之间
- 不会产生新的移进-归约冲突，但可能产生新的归约-归约冲突
- **实践中最常用**（YACC/Bison默认使用LALR(1)）

---

## 12. LR分析表的构造总结

### 各方法的分析表比较

| 方法 | 项目形式 | 向前看 | 状态数 | 分析能力 |
|------|---------|--------|--------|---------|
| LR(0) | `[A → α.β]` | 无 | 少 | 最弱 |
| SLR(1) | `[A → α.β]` + FOLLOW | FOLLOW集 | 少 | 较弱 |
| LALR(1) | `[A → α.β, a]` 合并 | 精确 | 少 | 较强 |
| LR(1) | `[A → α.β, a]` | 精确 | 多 | 最强 |

## 13. 分析过程中的冲突处理

### 13.1 移进-归约冲突（Shift-Reduce Conflict）

- 同一状态中既有移进项目又有归约项目
- 解决方法：优先移进（常用策略），或使用优先级和结合性规则

### 13.2 归约-归约冲突（Reduce-Reduce Conflict）

- 同一状态中有多个归约项目
- 解决方法：选择文法中先定义的产生式，或修改文法消除歧义

### 13.3 文法改写消除冲突

- 消除二义性（如悬空else问题）
- 引入新非终结符
- 调整产生式结构

---

## 14. 各方法的比较与适用范围

| 方法 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 递归下降 | 简单文法、手工实现 | 直观、灵活 | 需消除左递归和左公因子 |
| LL(1) | 预测分析、编译器前端 | 自顶向下、实现简单 | 文法限制严格 |
| LR(0) | 理论分析 | 最简单 | 冲突多，实用性差 |
| SLR(1) | 入门教学 | 比LR(0)强 | 仍不够精确 |
| LALR(1) | 实际编译器 | 平衡效率与能力 | **YACC/Bison使用** |
| LR(1) | 理论最优 | 分析能力最强 | 状态数爆炸 |

## 15. 错误恢复策略

### 15.1 紧急方式恢复（Panic Mode）

- 发现错误后，跳过输入符号直到找到某个合法的同步符号（如`;`、`}`、`begin`等）
- 简单有效，但可能跳过较多输入

### 15.2 短语级恢复（Phrase-Level Recovery）

- 发现错误后，对栈顶内容或输入进行局部修改（插入/删除/替换符号）
- 需要预先设计好恢复动作

### 15.3 出错产生式（Error Productions）

- 在文法中加入含有特殊终结符 `error` 的产生式
- `A → error α` 表示当出现某些错误时用此产生式

### 15.4 全局纠正

- 寻找对输入串做出最少修改使其合法的方案
- 理论上最优但实现复杂，实际中很少使用

---

## 参考与总结

语法分析是编译器的核心阶段。自顶向下的LL(1)分析和自底向上的LR分析是两大主流方法。408考试中，FIRST/FOLLOW集计算、LL(1)分析表构造、LR(0)/SLR(1)/LALR(1)项目集构造与分析表填写是必考重点。务必熟练掌握手工构造过程和冲突检测方法。
