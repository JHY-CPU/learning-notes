运行时存储管理

Runtime Storage Management — 活动记录、栈帧、Display、堆管理

1. 活动记录 (Activation Record)

每当调用一个函数时，编译器为其在栈上分配一块内存区域，称为活动记录或栈帧 (Stack Frame)。

活动记录结构 (从高地址到低地址): ┌──────────────────────┐ 高地址 │ 实际参数 (Arguments) │ 调用者传入的参数 ├──────────────────────┤ │ 返回地址 (Return Addr)│ 调用后继续执行的地址 ├──────────────────────┤ │ 控制链 (Control Link) │ 调用者的帧指针 (旧FP) ├──────────────────────┤ │ 访问链 (Access Link) │ 指向外围过程的活动记录 ├──────────────────────┤ ← 帧指针 (FP) │ 保存的寄存器 │ 被调用者需要保存的寄存器 ├──────────────────────┤ │ 局部变量 (Locals) │ 函数内部的局部变量 ├──────────────────────┤ │ 临时变量 (Temporaries)│ 表达式求值的中间结果 ├──────────────────────┤ ← 栈指针 (SP) │ │ │ (下一块活动记录) │ └──────────────────────┘ 低地址

各字段说明

字段用途设置者
实际参数传递给被调用函数的参数值调用者
返回地址函数返回后继续执行的指令地址call指令
控制链指向调用者的活动记录 (旧FP)被调用者序言
访问链访问非局部变量的静态链取决于语言
局部变量函数内声明的变量编译时确定偏移
临时变量中间表达式的值编译器生成

2. 栈帧操作 (Stack Frame Operations)

函数调用序列

// x86-64 典型的函数调用序列

// === 调用者 (Caller) ===
    // 1. 将参数放入寄存器/栈
    mov rdi, 1          // 第1个参数
    mov rsi, 2          // 第2个参数
    push arg3           // 第3个参数压栈 (超出寄存器数量)
    // 2. 执行call
    call foo            // 压入返回地址, 跳转到foo
    // 3. 清理栈 (cdecl约定: 调用者清理)
    add rsp, 8          // 弹出第3个参数

// === 被调用者 (Callee) 序言 ===
foo:
    push rbp            // 保存旧的帧指针
    mov rbp, rsp        // 建立新的帧指针
    sub rsp, 32         // 为局部变量分配空间
    mov [rbp-8], rdi    // 保存参数到局部区域

// === 被调用者 (Callee) 尾声 ===
    mov rsp, rbp        // 释放局部变量空间
    pop rbp             // 恢复旧的帧指针
    ret                 // 弹出返回地址, 跳回调用者
函数调用过程中栈的变化: 调用main前: 调用foo时: foo执行中: ┌──────────┐ ┌──────────┐ │ 返回main │ │ 返回main │ ├──────────┤ ├──────────┤ │ arg = 1 │ │ 旧FP │ ├──────────┤ ├──────────┤ │ │ │ locals │ │ │ ├──────────┤ ← SP ── SP │ │ │ │

3. 访问链与Display

当函数可以嵌套定义时 (如Pascal)，需要机制来访问非局部变量。

静态链 (Static Link / Access Link)

每个活动记录包含一个指向其词法外围函数活动记录的指针链。

procedure A() { var x; procedure B() { var y; procedure C() { var z; x = 1; // C → B的access link → A的access link → 找到x } } } 调用栈 (C正在执行): ┌─────────────────┐ │ C的活动记录 │ │ access_link ────│──┐ ├─────────────────┤ │ │ B的活动记录 │ │ │ access_link ────│──┤ ├─────────────────┤ │ │ A的活动记录 │◀─┘ ← x在这里! │ x = ... │ ├─────────────────┤ │ main的活动记录 │ └─────────────────┘ 查找x: C的access_link → B的活动记录 B的access_link → A的活动记录 → 找到x

Display

Display是全局数组，display[i] 指向当前处于嵌套深度i的函数的活动记录。查找非局部变量时直接查表，无需沿链遍历。

// Display数组
// display[0] → 全局作用域的活动记录
// display[1] → 嵌套深度1的函数的活动记录
// display[2] → 嵌套深度2的函数的活动记录
// ...

// 函数调用时更新Display:
// 1. 保存旧的 display[depth] 到栈帧中
// 2. 设置 display[depth] = 新的活动记录

// 函数返回时恢复Display:
// 1. 从栈帧中恢复旧的 display[depth]

// 优点: 查找非局部变量只需 display[depth].offset
// 代价: 需要额外的Display数组, 调用/返回时更新

4. 堆管理 (Heap Management)

堆用于存储生命周期不确定的数据 (如动态分配的对象、闭包捕获的变量)。

空闲链表 (Free List)

空闲链表组织: free ──▶ [8|next] ──▶ [16|next] ──▶ [32|next] ──▶ null 8字节 16字节 32字节 分配10字节: 找到16字节的块 → 分割: [10|used] + [6|next] ┌──────┬────────────────┐ │ size │ 数据区域 │ 每个块有头部记录大小和状态 └──────┴────────────────┘

分配策略

策略描述特点
首次适配 (First Fit)找到第一个足够大的块快, 可能碎片化
最佳适配 (Best Fit)最小的足够大的块碎片小, 搜索慢
最差适配 (Worst Fit)最大的块剩余块大, 适合小分配
伙伴系统 (Buddy)大小为2的幂, 对半分割合并快, 内碎片
// 伙伴系统 (Buddy System)
// 所有块大小为2^n
// 分配时: 找最小的 >= 请求大小的2的幂
// 如果没有: 分割更大的块
// 释放时: 检查伙伴是否空闲, 合并

// 伙伴地址计算:
// buddy(addr, size) = addr XOR size

// 例: 32字节块分割
// [          32          ]
// [    16    ][    16    ] ← 分割
// [  8 ][  8 ][    16    ] ← 再分割左边
// 释放[  8 ]时, 与它的伙伴[  8 ]合并为[    16    ]

6. 寄存器分配与溢出

寄存器比内存快得多。编译器需要将活跃变量尽可能放入寄存器，放不下时溢出 (Spill)到栈上。

// 寄存器溢出 (Register Spilling)
// 没有足够的寄存器时, 将变量存到栈上

// 溢出前:
mov rax, [x]     // x在寄存器中
add rax, [y]     // y在寄存器中

// 溢出后 (假设寄存器不够):
mov [rbp-8], rax // 将x溢出到栈
mov rax, [y]     // y占用rax
...              // 使用y
add rax, [rbp-8] // 从栈恢复x并计算
