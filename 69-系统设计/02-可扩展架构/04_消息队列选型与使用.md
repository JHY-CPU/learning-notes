# 消息队列选型与使用


## 消息队列选型与使用


系统设计消息队列解耦


消息队列是可扩展架构的核心组件，实现服务间异步解耦、流量削峰和事件驱动。


## 消息队列的核心价值


```
消息队列解决的三大问题：

1. 异步解耦
   订单服务 → [MQ] → 库存服务、积分服务、通知服务
   订单服务只需发一条消息，下游服务各自消费
   新增下游服务无需修改订单服务代码

2. 流量削峰
   秒杀场景：
   瞬时QPS 10万 → MQ缓冲 → 消费者按能力消费（如1000 QPS）
   避免数据库被瞬时流量打垮

3. 事件驱动
   用户注册 → [MQ] → 发送欢迎邮件、初始化配置、创建默认数据
   生产者和消费者完全解耦，各司其职

消息模型对比：
┌──────────────┬──────────────┬──────────────────┐
│ 模型          │ 特点          │ 代表             │
├──────────────┼──────────────┼──────────────────┤
│ 队列 (Queue) │ 一条消息只被  │ RabbitMQ         │
│              │ 一个消费者消费│ ActiveMQ         │
│              │ 点对点通信    │                  │
├──────────────┼──────────────┼──────────────────┤
│ 发布/订阅    │ 一条消息被所有│ Kafka            │
│ (Pub/Sub)   │ 订阅者消费    │ RocketMQ         │
│              │ 广播通信      │ Pulsar           │
└──────────────┴──────────────┴──────────────────┘
```


## 主流消息队列对比


```
┌──────────────┬──────────┬──────────┬──────────┬──────────┐
│ 特性          │ Kafka    │ RabbitMQ │ RocketMQ │ Pulsar   │
├──────────────┼──────────┼──────────┼──────────┼──────────┤
│ 吞吐量        │ 百万/s   │ 万/s     │ 十万/s   │ 百万/s   │
│ 延迟          │ ms级     │ μs级     │ ms级     │ ms级     │
│ 消息持久化    │ 磁盘     │ 内存+磁盘│ 磁盘     │ 磁盘     │
│ 消息顺序      │ 分区有序 │ 不保证   │ 队列有序 │ 分区有序 │
│ 消息回溯      │ 支持     │ 不支持   │ 支持     │ 支持     │
│ 消费模式      │ Pull     │ Push     │ Pull/Push│ Pull     │
│ 事务消息      │ 支持     │ 支持     │ 支持     │ 支持     │
│ 延迟消息      │ 不支持   │ 插件     │ 支持     │ 支持     │
│ 死信队列      │ 手动实现 │ 支持     │ 支持     │ 支持     │
│ 消息过滤      │ 无       │ routing  │ Tag/SQL  │ Topic    │
│ 多租户        │ 无       │ vhost    │ 无       │ 支持     │
│ 运维复杂度    │ 高(ZK)   │ 低       │ 中       │ 高       │
│ 语言          │ Scala/Java│ Erlang  │ Java     │ Java     │
│ 社区/商业     │ Confluent│ VMware   │ 阿里     │ StreamNative│
│ 适合场景      │ 大数据/  │ 企业级/  │ 电商/金融│ 云原生/  │
│              │ 日志     │ 复杂路由 │          │ 多租户   │
└──────────────┴──────────┴──────────┴──────────┴──────────┘

Kafka详解：
核心概念：
  Topic（主题）→ 多个Partition（分区）→ 每个分区有序
  Producer → Topic → Consumer Group → 多个Consumer

  Partition作用：
  - 并行消费：一个Partition只能被Consumer Group中一个Consumer消费
  - 顺序保证：同一Partition内消息有序
  - 水平扩展：增加Partition数可提高消费并行度

  示例：
  Topic "orders" 有6个Partition
  Consumer Group "order-service" 有3个Consumer
  分配：Consumer1→P0,P1  Consumer2→P2,P3  Consumer3→P4,P5

RabbitMQ详解：
核心概念：
  Exchange（交换机）→ Binding → Queue（队列）→ Consumer

  Exchange类型：
  - Direct：精确匹配routing key
  - Fanout：广播到所有绑定的队列
  - Topic：通配符匹配routing key（*.order.#）
  - Headers：根据消息头属性匹配

  示例：
  Producer → Exchange("order.direct")
    → Binding key="created" → Queue("order.created")
    → Binding key="paid"    → Queue("order.paid")
    → Binding key="shipped" → Queue("order.shipped")
```


## 分区与消费者协调


```
Kafka分区策略：
Producer端分区：
┌─────────────────────────────────────────────┐
│  1. 指定Partition                          │
│     producer.send(record, partition=2)     │
│                                             │
│  2. Key Hash                               │
│     partition = hash(key) % num_partitions │
│     同一key的消息进入同一分区（保序）        │
│     例：order_id=123 → 始终进入Partition 2  │
│                                             │
│  3. 轮询（无Key时）                         │
│     依次发送到各分区                        │
│                                             │
│  4. 自定义Partitioner                      │
│     实现Partitioner接口，自定义路由逻辑     │
└─────────────────────────────────────────────┘

Consumer Rebalancing（消费者重平衡）：
触发条件：
- Consumer加入/离开Group
- Topic的Partition数量变化
- Consumer心跳超时

重平衡策略：
┌─────────────────────────────────────────────┐
│  Range（默认）：                             │
│  按Topic分配，可能导致不均衡                 │
│  Topic1(P0-P5) → C1分到P0-P2, C2分到P3-P5  │
│                                             │
│  RoundRobin：                                │
│  所有Topic的Partition统一轮询分配            │
│  更均衡，但不保证同一Topic的分区连续         │
│                                             │
│  Sticky：                                    │
│  尽量保持原有的分配关系                      │
│  减少重平衡时的分区迁移                      │
│  Kafka 0.11+ 支持                           │
│                                             │
│  CooperativeSticky：                         │
│  增量式重平衡，不是一次性全部重分配          │
│  减少重平衡期间的消费中断                    │
│  Kafka 2.4+ 推荐使用                        │
└─────────────────────────────────────────────┘

重平衡的影响：
- 重平衡期间所有Consumer暂停消费
- 可能导致消息重复消费
- Consumer越多，重平衡越频繁
- 解决方案：
  - 调大 session.timeout.ms（默认45s）
  - 调大 heartbeat.interval.ms（默认3s）
  - 使用Sticky/CooperativeSticky分配策略
  - 使用静态成员（group.instance.id）
```


## 死信队列与重试机制


```
死信队列 (Dead Letter Queue, DLQ)：
存放消费失败消息的特殊队列

消息变为死信的条件：
1. 消息被消费者拒绝（reject/nack）且不重新入队
2. 消息TTL过期
3. 队列达到最大长度，最早的消息被丢弃
4. 消费重试次数超过上限

流程设计：
┌─────────────────────────────────────────────┐
│  业务队列                                     │
│    ↓ 消费                                     │
│  成功 → ACK                                   │
│  失败 → 重试（最多3次）                       │
│    ↓ 超过重试次数                             │
│  死信队列 (DLQ)                               │
│    ↓ 监控                                     │
│  告警 → 人工介入处理                          │
└─────────────────────────────────────────────┘

RabbitMQ死信配置：
// 声明业务队列（绑定死信交换机）
Map<String, Object> args = new HashMap<>();
args.put("x-dead-letter-exchange", "dlx.exchange");
args.put("x-dead-letter-routing-key", "dlq.routing");
args.put("x-message-ttl", 60000);  // TTL=60s
channel.queueDeclare("business.queue", true, false, false, args);

// 声明死信交换机和队列
channel.queueDeclare("dlq.queue", true, false, false, null);
channel.queueBind("dlq.queue", "dlx.exchange", "dlq.routing");

Kafka死信处理：
// Kafka没有内置DLQ，需要手动实现
try {
    processMessage(record);
    consumer.commitSync();
} catch (Exception e) {
    if (retryCount < MAX_RETRY) {
        // 重试：发送到重试Topic（带延迟）
        retryProducer.send(retryTopic, record);
    } else {
        // 超过重试次数：发送到死信Topic
        dlqProducer.send(dlqTopic, record);
    }
    consumer.commitSync();
}

重试Topic设计：
  retry-1-topic (延迟1分钟)
  retry-2-topic (延迟5分钟)
  retry-3-topic (延迟15分钟)
  → dlq-topic (不再重试)
```


## 消息队列最佳实践


```
消息可靠性保证：
┌─────────────────────────────────────────────┐
│  生产者端：                                   │
│  - 开启acks=all（等待所有ISR确认）           │
│  - 开启重试 retries=3                        │
│  - 使用事务消息保证精确一次                   │
│  - 同步发送或带回调的异步发送                 │
│                                             │
│  Broker端：                                   │
│  - 副本因子 replication.factor=3             │
│  - min.insync.replicas=2                    │
│  - 持久化到磁盘                              │
│                                             │
│  消费者端：                                   │
│  - 手动提交offset（关闭自动提交）            │
│  - 处理完成后再提交                          │
│  - 幂等消费（处理重复消息）                  │
│  - 失败消息进死信队列                        │
└─────────────────────────────────────────────┘

消费幂等性：
问题：消息可能重复消费（至少一次语义）
解决：消费逻辑设计为幂等

方案1：数据库唯一约束
  INSERT INTO orders (id, ...) VALUES (?, ...)
  重复插入会报唯一约束错误，捕获后忽略

方案2：版本号/状态机
  UPDATE orders SET status='paid'
  WHERE id=? AND status='created'
  只有状态匹配才更新

方案3：去重表
  INSERT INTO message_log (msg_id) VALUES (?)
  先检查msg_id是否已处理过

方案4：Redis原子操作
  SET msg_id 1 NX EX 3600
  设置成功才处理消息

消息积压处理：
┌─────────────────────────────────────────────┐
│  临时方案：                                   │
│  1. 增加消费者数量                           │
│  2. 增加消费线程数                           │
│  3. 扩容Partition（需重建Topic）             │
│                                             │
│  紧急方案：                                   │
│  1. 新建消费者组消费积压Topic                │
│  2. 消费者跳过非关键消息（丢弃）             │
│  3. 临时降低消息保留时间                     │
│                                             │
│  预防方案：                                   │
│  1. 监控消费延迟（Consumer Lag）             │
│  2. 设置告警阈值                             │
│  3. 预估峰值流量，提前扩容                   │
└─────────────────────────────────────────────┘

选型决策树：
需要大数据/日志处理？→ Kafka
需要复杂路由/低延迟？→ RabbitMQ
需要事务消息/金融级？→ RocketMQ
需要云原生/多租户？   → Pulsar
小团队简单场景？      → RabbitMQ 或 Redis Stream
```


> **Note:** 消息队列选型没有银弹，Kafka适合大数据场景，RabbitMQ适合企业集成，RocketMQ适合电商金融。关键是保证消息可靠性和消费幂等性。
