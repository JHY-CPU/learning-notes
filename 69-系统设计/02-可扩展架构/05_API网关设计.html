API网关设计

系统设计网关微服务

API网关是微服务架构的统一入口，处理认证、限流、路由、协议转换等横切关注点。

API网关核心功能

架构位置： ┌─────────────────────────────────────────────┐ │ 客户端 (Web/App/小程序/第三方) │ │ ↓ HTTPS │ │ ┌──────────────────────────────┐ │ │ │ API Gateway │ │ │ │ ┌────┐ ┌────┐ ┌────┐ ┌──┐ │ │ │ │ │认证│ │限流│ │路由│ │..│ │ │ │ │ └────┘ └────┘ └────┘ └──┘ │ │ │ └──────────┬───────────────────┘ │ │ ↓ │ │ ┌────┐ ┌────┐ ┌────┐ ┌────┐ │ │ │用户│ │订单│ │商品│ │支付│ 微服务集群 │ │ └────┘ └────┘ └────┘ └────┘ │ └─────────────────────────────────────────────┘ 核心功能清单： 1. 请求路由 (Routing) /api/users/** → user-service /api/orders/** → order-service /api/products/** → product-service 2. 认证鉴权 (Authentication & Authorization) JWT Token验证 OAuth2.0授权 API Key校验 请求签名验证 3. 限流熔断 (Rate Limiting & Circuit Breaking) 按IP/用户/API限流 熔断降级保护 4. 协议转换 (Protocol Translation) HTTP ↔ gRPC WebSocket ↔ HTTP REST ↔ GraphQL 5. 负载均衡 (Load Balancing) 轮询/加权/最少连接 6. 日志监控 (Logging & Monitoring) 请求日志、链路追踪、指标采集 7. 响应缓存 (Response Caching) 缓存GET请求结果 8. 请求/响应转换 Header添加/删除 请求体改写 响应格式统一

限流算法详解

1. 固定窗口计数器 (Fixed Window) ┌─────────────────────────────────────────────┐ │ |----1分钟----|----1分钟----|----1分钟----│ │ │ | 100次 ✓ | 100次 ✓ | 100次 ✓ │ │ │ │ │ 问题：边界突发 │ │ |----1分钟----|----1分钟----| │ │ | 99次|99次 | │ │ | ← 1秒内198次 → | │ │ 实际1秒内可能超过限流阈值 │ └─────────────────────────────────────────────┘ 2. 滑动窗口计数器 (Sliding Window) ┌─────────────────────────────────────────────┐ │ 将固定窗口分为多个小窗口 │ │ 统计当前时间往前1分钟内的请求数 │ │ │ │ |---10s---|---10s---|---10s---|---10s---| │ │ | 10 | 15 | 20 | 18 | │ │ | 过期 | | | 当前 | │ │ │ │ 当前计数 = 15 + 20 + 18 = 53 │ │ 与阈值100比较 │ │ │ │ 解决了边界突发问题，但实现更复杂 │ └─────────────────────────────────────────────┘ 3. 漏桶算法 (Leaky Bucket) ┌─────────────────────────────────────────────┐ │ 请求 → ┌─────────┐ → 处理 │ │ │ 🌊 漏桶 │ │ │ │ 桶容量N │ │ │ │ 固定速率│ │ │ └────┬────┘ │ │ ↓ 固定速率流出 │ │ │ │ 特点： │ │ - 请求以固定速率被处理 │ │ - 桶满则拒绝新请求 │ │ - 输出速率恒定，平滑流量 │ │ - 适合限制流出速率 │ └─────────────────────────────────────────────┘ 4. 令牌桶算法 (Token Bucket) ★ 最常用 ┌─────────────────────────────────────────────┐ │ 令牌以固定速率放入桶中 │ │ 每个请求需要获取一个令牌才能通过 │ │ 桶满则丢弃新令牌 │ │ 桶空则拒绝请求 │ │ │ │ ┌──────────────────┐ │ │ │ 令牌生成器 │ │ │ │ (每秒100个令牌) │ │ │ └────────┬─────────┘ │ │ ↓ │ │ ┌──────────────────┐ 有令牌 → 通过 │ │ │ 令牌桶(容量1000)│ ──────→ │ │ └──────────────────┘ 无令牌 → 拒绝/等待 │ │ │ │ 特点： │ │ - 允许突发流量（桶里有积累的令牌） │ │ - 长期平均速率受控 │ │ - Google Guava RateLimiter实现 │ └─────────────────────────────────────────────┘ 5. 滑动日志 (Sliding Log) - 记录每个请求的时间戳 - 统计窗口内的请求数 - 最精确但最消耗内存 Redis实现令牌桶： -- KEYS[1]=限流key, ARGV[1]=桶容量, ARGV[2]=令牌生成速率, ARGV[3]=当前时间, ARGV[4]=请求数 local key = KEYS[1] local capacity = tonumber(ARGV[1]) local rate = tonumber(ARGV[2]) local now = tonumber(ARGV[3]) local requested = tonumber(ARGV[4]) local data = redis.call('HMGET', key, 'tokens', 'last_time') local tokens = tonumber(data[1]) or capacity local last_time = tonumber(data[2]) or now -- 计算新生成的令牌 local delta = math.max(0, now - last_time) tokens = math.min(capacity, tokens + delta * rate) if tokens >= requested then tokens = tokens - requested redis.call('HMSET', key, 'tokens', tokens, 'last_time', now) redis.call('EXPIRE', key, 60) return 1 -- 允许 else redis.call('HMSET', key, 'tokens', tokens, 'last_time', now) redis.call('EXPIRE', key, 60) return 0 -- 拒绝 end

网关设计模式

模式1：单一网关 (Monolithic Gateway) 所有API请求通过一个网关 优点：简单 缺点：单点故障，所有团队共享一个网关 模式2：微网关 (Micro Gateway) 每个团队维护自己的网关 ┌─────────────────────────────────────────────┐ │ 客户端 │ │ ↓ │ │ API Gateway (统一入口，公共逻辑) │ │ ↓ │ │ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ │ │ 用户网关 │ │ 订单网关 │ │ 商品网关 │ │ │ │ (团队A) │ │ (团队B) │ │ (团队C) │ │ │ └──────────┘ └──────────┘ └──────────┘ │ └─────────────────────────────────────────────┘ 模式3：Sidecar网关 (Service Mesh) 每个服务实例旁边部署一个Envoy Sidecar ┌─────────────────────────────────────────────┐ │ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ │ │ Service │ │ Service │ │ Service │ │ │ │ A │ │ B │ │ C │ │ │ ├─────────┤ ├─────────┤ ├─────────┤ │ │ │ Envoy │ │ Envoy │ │ Envoy │ │ │ │ (Sidecar)│ │ (Sidecar)│ │ (Sidecar)│ │ │ └─────────┘ └─────────┘ └─────────┘ │ │ ↕ ↕ ↕ │ │ ┌─────────────────────────────────────┐ │ │ │ Control Plane (Istio) │ │ │ └─────────────────────────────────────┘ │ └─────────────────────────────────────────────┘ BFF (Backend for Frontend) 模式： ┌─────────────────────────────────────────────┐ │ Web App ──→ BFF (Web) ──→ 微服务集群 │ │ App ──→ BFF (Mobile) ──→ 微服务集群 │ │ 小程序 ──→ BFF (Mini) ──→ 微服务集群 │ │ │ │ 每种客户端有独立的BFF │ │ BFF负责数据聚合、格式转换、裁剪 │ │ 适合多端场景（Web/App/小程序返回不同字段） │ └─────────────────────────────────────────────┘ 网关安全最佳实践： ┌─────────────────────────────────────────────┐ │ 1. API网关是微服务架构的关键组件。Kong和Envoy适合生产环境，Spring Cloud Gateway适合Spring生态。令牌桶是最常用的限流算法，支持突发流量的同时控制平均速率。
