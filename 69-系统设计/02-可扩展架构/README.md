# 02 - 可扩展架构

可扩展架构关注系统在业务增长时，如何通过合理的架构设计来承载更大的数据量、更高的并发和更复杂的业务逻辑。

---

## 1. 可扩展性的维度

### 1.1 垂直扩展（Scale Up）

通过提升单机硬件配置（CPU、内存、磁盘、网络）来提高系统处理能力。

**优点**：实施简单，不改变架构，无需修改应用代码。
**缺点**：存在物理上限，成本非线性增长，单点故障风险集中。

### 1.2 水平扩展（Scale Out）

通过增加更多机器来分摊负载，用数量弥补单机性能不足。

**优点**：理论上无上限，成本线性增长，消除单点故障。
**缺点**：架构复杂度高，需要解决数据分布、一致性、运维管理等问题。

**原则**：优先设计水平扩展能力，垂直扩展作为临时优化手段。

---

## 2. 数据库扩展

### 2.1 读写分离

主库处理写操作，从库处理读操作。通过数据复制（主从复制）保持数据同步。

**适用场景**：读多写少的业务（如新闻网站、社交平台）。

**实现方式**：
- **应用层分离**：代码中显式指定读库和写库
- **中间件代理**：使用MyCat、ShardingSphere等中间件自动路由
- **驱动层实现**：通过数据库驱动的读写分离插件

**注意事项**：
- 主从复制存在延迟，可能导致读到旧数据
- 同一事务中的读操作应路由到主库
- 写后的立即读取需强制走主库

### 2.2 分库分表

当单库单表数据量过大时，需要进行拆分。

**水平分片**：将同一张表的数据按规则分散到多个库/表中（如按用户ID取模）。
**垂直分片**：按业务领域将不同表拆分到不同库中（如用户库、订单库分离）。

**分片策略**：
- **Range分片**：按范围划分（如按时间、ID区间）
- **Hash分片**：对分片键取模或哈希
- **一致性哈希分片**：节点增减时仅影响部分数据

### 2.3 分片键选择

分片键的选择直接影响分片效果：

- **均匀分布**：分片键值分布应尽量均匀，避免数据倾斜
- **查询友好**：尽量让常见查询落在单一分片上，避免跨分片查询
- **避免热点**：避免某个分片承载过多流量
- **扩展性**：选择支持未来扩容的分片方案

**常见分片键**：用户ID、地理位置、时间戳、业务ID。

### 2.4 跨分片查询与分布式JOIN

跨分片查询是分库分表后的主要挑战：

- **广播查询**：将查询发送到所有分片，汇总结果
- **二次查询**：先查询一个分片获取关联键，再查询另一个分片
- **冗余存储**：在每个分片中冗余存储常用维度表
- **异步汇总**：通过异步任务预计算跨分片结果

### 2.5 数据迁移与在线扩容

**双写方案**：新旧数据库同时写入，逐步迁移读流量，最终切换写流量。
**停机迁移**：业务低峰期停机进行数据迁移，简单但影响业务。
**在线迁移步骤**：
1. 启动数据同步工具，将旧数据全量同步到新库
2. 增量数据通过binlog实时同步
3. 数据追平后，灰度切换读流量
4. 确认无误后，切换写流量
5. 停止旧库写入，完成迁移

## 3. 缓存策略

缓存是提升系统性能最有效的手段之一，通过在高速存储层保留热点数据来减少对后端存储的访问。

### 3.1 多级缓存

**本地缓存**：JVM堆内（Guava/Caffeine）或堆外缓存。延迟最低（纳秒级），容量有限，无法共享。

**分布式缓存**：Redis/Memcached集群。延迟较低（毫秒级），容量较大，多实例共享。

**CDN缓存**：静态资源缓存到边缘节点。适用于图片、视频、JS/CSS等静态内容。

**查询流程**：请求 → 本地缓存 → 分布式缓存 → 数据库，逐级穿透。

### 3.2 缓存穿透

查询一个**不存在的数据**，每次请求都穿透缓存直达数据库。

**解决方案**：
- **缓存空值**：即使查询结果为空，也将空值写入缓存（设置较短过期时间）
- **布隆过滤器**：在缓存前加一层布隆过滤器，快速判断key是否存在，不存在则直接返回

**布隆过滤器**：使用多个哈希函数将元素映射到一个位数组中。可能存在误判（假阳性），但不会有假阴性。

### 3.3 缓存击穿

某个**热点key过期**的瞬间，大量并发请求同时穿透到数据库。

**解决方案**：
- **互斥锁**：只有一个请求能获取锁去查数据库，其他请求等待或返回旧缓存
- **永不过期**：热点key不设置过期时间，通过后台任务定期更新
- **提前刷新**：在key过期前，异步刷新缓存

### 3.4 缓存雪崩

**大量key同时过期**或**缓存服务整体宕机**，导致所有请求直接打到数据库。

**解决方案**：
- **随机过期时间**：在基础过期时间上增加随机偏移量
- **熔断降级**：数据库压力过大时触发熔断，返回降级数据
- **多级缓存**：即使分布式缓存失效，本地缓存仍可提供服务
- **缓存预热**：系统启动时提前加载热点数据

### 3.5 缓存一致性

- **Cache Aside（旁路缓存）**：读时先查缓存，未命中再查数据库并回填缓存；写时先更新数据库再删除缓存。
- **Read/Write Through（读写穿透）**：缓存层负责与数据库同步，应用只与缓存交互。
- **Write Behind（异步写回）**：写入只更新缓存，异步批量写入数据库。写入性能最高，但有数据丢失风险。

**Cache Aside是实践中最常用的模式**，注意写操作应删除缓存而非更新缓存（避免并发写导致的脏数据）。

## 4. 消息队列

消息队列（MQ）是实现系统解耦、异步处理、流量削峰的关键组件。

### 4.1 异步解耦

将同步调用改为异步消息，上游服务发送消息后即可返回，下游服务异步消费处理。

**解耦效果**：生产者和消费者独立演进，一方变更不影响另一方。

### 4.2 削峰填谷

在流量高峰期，消息队列将请求暂存为消息，消费者以恒定速率消费，避免瞬时流量压垮后端系统。

**适用场景**：秒杀、抢购、定时任务触发等有明显流量峰值的场景。

### 4.3 顺序保证

- **全局有序**：整个队列中的消息严格按发送顺序消费（性能代价高）
- **分区有序**：同一分片（partition）内的消息有序，不同分片之间无序（常见方案）
- **业务有序**：同一业务实体的消息有序，不同业务实体无序

**实现方式**：使用相同的分片键（如订单ID）将相关消息路由到同一分区。

### 4.4 消息可靠性

- **持久化**：消息写入磁盘后再返回确认，防止宕机丢失
- **ACK机制**：消费者处理完成后发送ACK确认，未确认的消息会被重新投递
- **重试策略**：消费失败后进行有限次重试，重试失败进入死信队列（DLQ）
- **事务消息**：确保本地事务与消息发送的原子性（如RocketMQ的事务消息）

### 4.5 消息幂等性

同一条消息被消费多次，结果与消费一次相同。

**幂等方案**：
- **唯一ID+去重表**：每条消息携带唯一ID，消费前检查去重表
- **状态机**：利用业务状态的有限状态机，重复操作自然幂等
- **乐观锁**：使用版本号或CAS操作保证更新幂等

## 5. CDN与边缘计算

**CDN（Content Delivery Network）** 将内容缓存到离用户最近的边缘节点，减少网络延迟和源站压力。

**工作原理**：用户请求被DNS解析到最近的CDN节点，节点上有缓存则直接返回，没有则回源站获取并缓存。

**边缘计算**：在CDN节点上运行计算逻辑，将部分计算从中心下沉到边缘，进一步降低延迟。适用场景包括：A/B测试、个性化内容、安全防护（WAF）、图像处理。

---

## 6. 微服务拆分原则

- **单一职责**：每个服务专注于一个业务领域
- **高内聚低耦合**：同一领域的功能聚合在一起，跨领域通过接口交互
- **按业务域拆分**：而非按技术层拆分（如拆分用户服务、订单服务，而非拆分CRUD服务）
- **限界上下文（Bounded Context）**：参照DDD的限界上下文进行服务划分
- **数据库自治**：每个微服务拥有独立数据库，避免跨服务直接访问数据库

## 7. 数据最终一致性方案

### 7.1 事件驱动架构

通过事件通知来实现跨服务的数据同步。服务A完成操作后发布事件，服务B订阅事件并更新本地数据。

### 7.2 事件溯源（Event Sourcing）

不存储实体的当前状态，而是存储导致状态变化的所有事件。通过回放事件可以重建任意时刻的状态。

**优点**：完整的审计日志、支持事件回放和时间旅行查询。
**缺点**：事件不可修改、查询当前状态需要聚合所有事件、需要快照优化。

### 7.3 CDC（Change Data Capture）

通过监听数据库的变更日志（如MySQL的binlog），实时捕获数据变更事件并发布到消息队列，下游服务消费这些事件进行数据同步。

**常用工具**：Debezium、Canal、Maxwell、Flink CDC。

**优势**：对业务代码无侵入，基于数据库原生机制，数据变更捕获可靠。
