# 多级缓存架构


## 多级缓存架构


系统设计缓存性能


多级缓存通过本地缓存和分布式缓存的组合，在不同层级拦截请求，最大化降低数据库压力。


## 缓存层级体系


```
请求处理路径（由近到远）：
┌─────────────────────────────────────────────┐
│  用户请求                                     │
│    ↓                                        │
│  L1: 浏览器缓存 (HTTP Cache)                │
│    Cache-Control / ETag / Last-Modified     │
│    延迟：0ms，命中率：30-50%                 │
│    ↓ 未命中                                  │
│  L2: CDN缓存                                 │
│    静态资源、页面片段                         │
│    延迟：5-30ms，命中率：80-95%              │
│    ↓ 未命中                                  │
│  L3: 反向代理缓存 (Nginx)                   │
│    proxy_cache，缓存API响应                  │
│    延迟：1-5ms，命中率：40-60%               │
│    ↓ 未命中                                  │
│  L4: 应用本地缓存 (Caffeine/Guava)          │
│    JVM内存中，访问最快                       │
│    延迟：<0.01ms，命中率：60-80%             │
│    ↓ 未命中                                  │
│  L5: 分布式缓存 (Redis/Memcached)           │
│    独立缓存集群，多实例共享                   │
│    延迟：0.5-2ms，命中率：90-99%             │
│    ↓ 未命中                                  │
│  L6: 数据库 (MySQL/PostgreSQL)              │
│    持久化存储                                 │
│    延迟：5-50ms                              │
└─────────────────────────────────────────────┘

各级缓存对比：
┌────────────┬──────────┬────────┬──────────┬──────────┐
│ 缓存层级    │ 延迟     │ 容量   │ 共享性    │ 一致性   │
├────────────┼──────────┼────────┼──────────┼──────────┤
│ 浏览器      │ 0ms      │ 小     │ 单用户   │ 弱       │
│ CDN        │ 5-30ms   │ 大     │ 全网     │ 最终     │
│ Nginx      │ 1-5ms    │ 中     │ 单节点   │ 节点级   │
│ 本地缓存    │ <0.01ms  │ 小     │ 单实例   │ 实例级   │
│ Redis      │ 0.5-2ms  │ 大     │ 集群共享 │ 最终     │
│ 数据库      │ 5-50ms   │ 无限   │ 全局     │ 强       │
└────────────┴──────────┴────────┴──────────┴──────────┘
```


## 本地缓存：Caffeine


```
Caffeine是Java中性能最高的本地缓存库（Google Guava Cache的替代品）

核心特性：
- W-TinyLFU淘汰策略（接近最优的命中率）
- 异步加载和刷新
- 自动过期和大小限制
- 统计监控

基本使用：
// 1. 手动管理
Cache<String, User> cache = Caffeine.newBuilder()
    .maximumSize(10_000)           // 最大10000条
    .expireAfterWrite(10, TimeUnit.MINUTES)  // 写入后10分钟过期
    .expireAfterAccess(5, TimeUnit.MINUTES)  // 访问后5分钟过期
    .recordStats()                 // 开启统计
    .build();

cache.put("user:1", user);
User user = cache.getIfPresent("user:1");
User user = cache.get("user:1", k -> loadFromDB(k));

// 2. Spring集成
@Configuration
@EnableCaching
public class CacheConfig {
    @Bean
    public CacheManager cacheManager() {
        CaffeineCacheManager manager = new CaffeineCacheManager();
        manager.setCaffeine(Caffeine.newBuilder()
            .maximumSize(10_000)
            .expireAfterWrite(10, TimeUnit.MINUTES));
        return manager;
    }
}

// 使用
@Cacheable(value = "users", key = "#id")
public User getUser(Long id) {
    return userRepository.findById(id);
}

淘汰策略对比：
┌─────────────────────────────────────────────┐
│  LRU (Least Recently Used)：                │
│  淘汰最久未访问的                            │
│  问题：扫描式访问会冲掉热点数据              │
│                                             │
│  LFU (Least Frequently Used)：              │
│  淘汰访问频率最低的                          │
│  问题：历史热点可能长期占用空间              │
│                                             │
│  W-TinyLFU (Caffeine使用)：                 │
│  结合频率和新近度                            │
│  - 主窗口：大部分空间，LRU淘汰               │
│  - 候选窗口：小空间，新数据先进入这里        │
│  - 候选数据与主窗口数据比较频率              │
│  - 频率高的替换频率低的                      │
│  效果：接近最优命中率                        │
└─────────────────────────────────────────────┘

本地缓存的问题：
- 只在单个JVM实例内有效
- 多实例间数据不一致
- 重启后缓存丢失
- 内存受限
```


## 分布式缓存：Redis


```
Redis缓存策略：

1. Cache Aside (旁路缓存) ★ 最常用
┌─────────────────────────────────────────────┐
│  读流程：                                     │
│  读Redis → 命中 → 返回                       │
│  读Redis → 未命中 → 读DB → 写Redis → 返回   │
│                                             │
│  写流程：                                     │
│  写DB → 删除Redis → 返回                     │
│  （不更新Redis，而是删除，下次读再加载）      │
│                                             │
│  为什么删除而非更新？                         │
│  更新缓存：可能被其他写覆盖导致不一致         │
│  删除缓存：下次读时加载，天然保证一致性       │
└─────────────────────────────────────────────┘

2. Read/Write Through (读写穿透)
缓存代理数据库的读写，应用只和缓存交互
┌─────────────────────────────────────────────┐
│  应用 → 缓存 → 数据库                        │
│  读：缓存有→返回，缓存无→读DB→写缓存→返回    │
│  写：写缓存→缓存同步写DB                     │
│  实现复杂，需要缓存系统支持                   │
└─────────────────────────────────────────────┘

3. Write Behind (异步写回)
┌─────────────────────────────────────────────┐
│  应用 → 写缓存 → 返回（异步批量写DB）        │
│  写性能最好，但有数据丢失风险                 │
│  适合允许短暂不一致的场景                     │
└─────────────────────────────────────────────┘

Spring Redis配置：
@Configuration
public class RedisConfig {
    @Bean
    public RedisTemplate<String, Object> redisTemplate(
            RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);
        // JSON序列化
        Jackson2JsonRedisSerializer<Object> serializer =
            new Jackson2JsonRedisSerializer<>(Object.class);
        template.setValueSerializer(serializer);
        template.setKeySerializer(new StringRedisSerializer());
        return template;
    }
}

// 使用
public User getUser(Long id) {
    String key = "user:" + id;
    // 1. 查缓存
    User user = redisTemplate.opsForValue().get(key);
    if (user != null) return user;
    // 2. 查数据库
    user = userRepository.findById(id);
    // 3. 写缓存（随机TTL防止雪崩）
    int ttl = 3600 + ThreadLocalRandom.current().nextInt(300);
    redisTemplate.opsForValue().set(key, user, ttl, TimeUnit.SECONDS);
    return user;
}

// 更新时删除缓存
@Transactional
public void updateUser(Long id, UserDTO dto) {
    // 1. 更新数据库
    userRepository.update(id, dto);
    // 2. 删除缓存
    redisTemplate.delete("user:" + id);
}
```


## 热点Key问题与解决方案


```
热点Key问题：
某个Key被大量请求访问，打到同一台Redis节点
导致该节点负载过高甚至宕机

场景示例：
- 微博热搜第一名的微博
- 秒杀商品的库存
- 爆款商品详情

检测方法：
1. Redis命令：redis-cli --hotkeys（Redis 4.0+）
2. 客户端统计：统计Key访问频率
3. Proxy层统计：如Twemproxy/Redis Cluster Proxy
4. 慢查询日志：监控慢查询热点

解决方案：

方案1：本地缓存 + Redis
┌─────────────────────────────────────────────┐
│  热点Key本地缓存，不过期或长过期             │
│  Caffeine缓存热点数据，拦截大部分请求        │
│  Redis只处理本地缓存未命中的请求             │
│                                             │
│  User → 本地缓存 → Redis → DB               │
│        (命中90%)  (命中9%) (命中1%)          │
└─────────────────────────────────────────────┘

方案2：Key打散 (Key拆分)
将热点Key拆分为多个子Key，分散到不同Redis节点
┌─────────────────────────────────────────────┐
│  原始Key：product:123                       │
│  拆分为：                                    │
│    product:123:0  → Node1                   │
│    product:123:1  → Node2                   │
│    product:123:2  → Node3                   │
│    product:123:3  → Node4                   │
│                                             │
│  读取：随机选一个子Key读取                   │
│  写入：更新所有子Key                         │
│  问题：写入开销变大，一致性变弱              │
└─────────────────────────────────────────────┘

方案3：Redis代理层限流
在Redis Proxy层对热点Key做限流
超过阈值的请求直接返回默认值或降级

方案4：多级缓存架构
┌─────────────────────────────────────────────┐
│  浏览器缓存 → CDN → Nginx缓存 → 本地缓存 → Redis → DB │
│  每一层都拦截一部分请求                      │
│  热点Key被最上层的缓存处理                   │
└─────────────────────────────────────────────┘

方案5：Redis Cluster + 读写分离
热点Key通过读写分离分散读压力
Master处理写，多个Slave处理读
```


## 缓存预热与一致性


```
缓存预热 (Cache Warm-up)：
系统启动或缓存清空后，主动加载热点数据

预热策略：
┌─────────────────────────────────────────────┐
│  1. 离线统计热点                             │
│     通过历史访问日志统计Top-N热点Key         │
│     定时任务每天更新热点Key列表              │
│                                             │
│  2. 启动时预热                               │
│     @PostConstruct                          │
│     public void warmup() {                  │
│       List<String> hotKeys = getHotKeys();  │
│       for (String key : hotKeys) {          │
│         Object value = db.query(key);       │
│         redis.set(key, value, ttl);         │
│         localCache.put(key, value);         │
│       }                                      │
│     }                                        │
│                                             │
│  3. 渐进式预热                               │
│     不一次性加载所有数据                     │
│     分批加载，避免预热期间DB压力过大         │
│                                             │
│  4. 实时预热                                 │
│     在缓存即将过期前异步刷新                 │
│     Caffeine的refreshAfterWrite             │
└─────────────────────────────────────────────┘

缓存一致性保证：
┌─────────────────────────────────────────────┐
│  问题1：缓存穿透                             │
│  查询不存在的数据，缓存和DB都未命中           │
│  解决：布隆过滤器 + 空值缓存                 │
│                                             │
│  问题2：缓存击穿                             │
│  热点Key过期瞬间大量请求打到DB               │
│  解决：互斥锁（SETNX）+ 逻辑过期             │
│                                             │
│  问题3：缓存雪崩                             │
│  大量Key同时过期，请求全部打到DB             │
│  解决：随机TTL + 多级缓存 + 限流降级         │
│                                             │
│  问题4：数据不一致                           │
│  DB和缓存数据不同步                          │
│  解决：延迟双删 + 消息队列重试               │
│     1. 删除缓存                              │
│     2. 更新DB                                │
│     3. 延迟500ms再删一次缓存                 │
│     （防止其他线程读到旧数据写入缓存）        │
└─────────────────────────────────────────────┘

一致性方案选择：
┌────────────────┬──────────────┬──────────────┐
│ 场景            │ 方案          │ 一致性级别   │
├────────────────┼──────────────┼──────────────┤
│ 强一致          │ 分布式锁      │ 强一致       │
│ 准实时          │ 延迟双删      │ 最终一致     │
│ 允许短暂不一致  │ Cache Aside   │ 最终一致     │
│ 只读数据        │ 长缓存+预热   │ 始终一致     │
└────────────────┴──────────────┴──────────────┘
```


> **Note:** 多级缓存的核心是在不同层级拦截请求。Caffeine是Java本地缓存的最佳选择，Redis是分布式缓存的标准方案。热点Key通过本地缓存+Key打散解决，缓存穿透/击穿/雪崩各有对应的防御策略。
