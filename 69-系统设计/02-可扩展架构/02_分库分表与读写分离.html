分库分表与读写分离

一、拆分策略

1. 垂直拆分（Vertical）

原始单库：users + orders + products + payments
         │
         ▼ 垂直分库
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户库    │  │ 订单库    │  │ 商品库    │  │ 支付库    │
│users     │  │orders    │  │products  │  │payments  │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
  • 按业务维度将不同表拆分到不同数据库
  • 解决：单库连接数瓶颈、业务耦合
  • 不解决：单表数据量过大

2. 水平拆分（Horizontal）

订单表（5000万行）→ 水平分表
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ order_0  │  │ order_1  │  │ order_2  │  │ order_3  │
│(user%4=0)│  │(user%4=1)│  │(user%4=2)│  │(user%4=3)│
└──────────┘  └──────────┘  └──────────┘  └──────────┘
  • 将同一张表的数据按规则分散到多个表/库
  • 解决：单表数据量过大（千万级以上）
  • 核心难点：分片键选择、跨库JOIN、分布式事务

二、分片策略

1. Hash 取模

shard = hash(sharding_key) % num_shards
// 例：user_id = 10016，4 个分片
shard = 10016 % 4 = 0  → 路由到 shard_0
  • 数据分布均匀
  • 扩容困难（需要重新迁移数据）
  • 解决扩容：一致性哈希

2. 一致性哈希（Consistent Hashing）

         ┌──────── Node A (0°-120°)
         │
 哈希环   ├──────── Node B (120°-240°)
(0~2^32) │
         └──────── Node C (240°-360°)

数据 key 哈希后顺时针找到第一个节点
虚拟节点：每个物理节点映射多个虚拟节点，解决数据倾斜
  • 扩容/缩容只影响相邻节点的数据迁移
  • 虚拟节点解决数据倾斜问题
  • 适合需要动态扩容的场景

3. 范围分片

// 按时间范围
2024-Q1 → shard_0
2024-Q2 → shard_1
2024-Q3 → shard_2

// 按 ID 范围
1 ~ 1000万    → shard_0
1000万+1 ~ 2000万 → shard_1
  • 扩容简单，直接加新分片
  • 可能导致热点（新数据集中在最新分片）
  • 范围查询友好

三、分库分表带来的问题

1. 跨库 JOIN

-- 原始 SQL（无法直接跨库执行）
SELECT o.*, u.name FROM orders o JOIN users u ON o.user_id = u.id

-- 解决方案
-- 1. 应用层组装：先查订单，再批量查用户，代码层关联
-- 2. 冗余字段：订单表冗余用户名，避免JOIN
-- 3. 全局表/广播表：小表（如字典表）在每个库都保留完整副本

2. 跨库排序与分页

// 每个分片各自排序取 Top N
分片0: [1,3,5,7,9]
分片1: [2,4,6,8,10]
// 应用层归并排序（Merge Sort）取全局 Top N
// 成本：N × shard_count 的数据量

3. 分布式 ID

自增主键在分表后会冲突，需要全局唯一ID（参见分布式ID生成笔记）。

4. 分布式事务

见分布式事务章节（TCC、Saga、消息最终一致性）。

四、读写分离

1. 架构

        ┌──────────┐
        │  应用层   │
        └────┬─────┘
             │ 写操作 ──────────→ ┌──────────┐
             │                   │  主库     │ Master
             │ 读操作             │ (写)      │
             │              ┌────│          │────┐
             ▼              │    └──────────┘    │
        ┌──────────┐       │       主从复制       │
        │ 代理/中间件│       ▼                    ▼
        │(Sharding │  ┌──────────┐        ┌──────────┐
        │ Sphere)  │  │ 从库-1   │        │ 从库-2   │
        └──────────┘  │ (读)     │        │ (读)     │
                      └──────────┘        └──────────┘

2. 主从复制延迟问题

核心问题：主从复制是异步的，写入主库后，从库可能有毫秒到秒级的延迟。用户写入后立即读取可能读不到最新数据。
  • 强制走主库：关键读（如写后立即读）强制路由到主库
  • 延迟判断：判断主从延迟时间，延迟期间读主库
  • GTID追踪：记录写入时的 GTID，读从库时检查是否已同步到该 GTID
  • 缓存标记：写入时在缓存中设置标记，读取时检查标记决定走主还是从

五、ShardingSphere 分片配置示例

# application.yml - ShardingSphere-JDBC
spring:
  shardingsphere:
    datasource:
      ds0:
        jdbc-url: jdbc:mysql://host0:3306/db0
      ds1:
        jdbc-url: jdbc:mysql://host1:3306/db1
    rules:
      sharding:
        tables:
          order:
            actual-data-nodes: ds$->{0..1}.order_$->{0..3}
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: db-hash
            table-strategy:
              standard:
                sharding-column: order_id
                sharding-algorithm-name: table-mod
        sharding-algorithms:
          db-hash:
            type: HASH_MOD
            props:
              sharding-count: 2
          table-mod:
            type: MOD
            props:
              sharding-count: 4

六、扩容方案

双写迁移方案（不停机扩容）：
1. 新库上线，配置双写（同时写新旧库）
2. 历史数据迁移（job 同步旧库到新库）
3. 数据校验（对比新旧库数据一致性）
4. 切换读流量到新库
5. 验证无误后，停止旧库写入
6. 下线旧库
面试要点：分库分表的拆分维度（垂直按业务、水平按数据量）；分片键选择原则（高频查询、分布均匀、不可变）；跨库JOIN的三种解法；主从延迟的处理策略；扩容双写迁移流程。
