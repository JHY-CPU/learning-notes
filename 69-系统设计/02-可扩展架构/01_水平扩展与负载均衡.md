# 水平扩展与负载均衡


## 水平扩展与负载均衡


系统设计扩展性负载均衡


通过增加服务器数量实现系统扩展，负载均衡分发请求到各节点。


## 扩展方式对比


```
垂直扩展 (Scale Up)：
- 增加单台服务器的CPU/内存/存储
- 简单，无架构改动
- 有上限，成本非线性增长
- 单点故障风险

水平扩展 (Scale Out)：
- 增加服务器数量
- 理论上无上限
- 需要无状态设计
- 需要负载均衡
- 现代架构的首选

无状态设计原则：
- 应用服务器不存储会话状态
- 状态存储在Redis/数据库中
- 任意请求可被任意服务器处理
- 便于水平扩展和故障转移
```


## 负载均衡算法


```
┌─────────────────────────────────────────────┐
│  轮询 (Round Robin)                          │
│  依次分配请求到各服务器                       │
│  简单，假设各服务器性能相同                    │
│                                             │
│  加权轮询 (Weighted Round Robin)             │
│  根据服务器权重分配更多/更少请求               │
│  适合异构服务器环境                           │
│                                             │
│  最少连接 (Least Connections)                │
│  将请求分配给当前连接数最少的服务器            │
│  适合长连接场景                               │
│                                             │
│  一致性哈希                                  │
│  同一用户的请求总是到同一服务器                │
│  适合有状态服务（需配合session）               │
│                                             │
│  最短响应时间                                │
│  分配给响应时间最短的服务器                   │
│  自适应服务器性能                             │
│                                             │
│  随机 (Random)                               │
│  随机选择服务器                               │
│  大样本下近似均匀                             │
└─────────────────────────────────────────────┘

负载均衡层级：
- L4 (传输层)：基于IP/端口，性能高
  LVS, HAProxy (TCP模式)
- L7 (应用层)：基于HTTP内容，灵活性高
  Nginx, HAProxy (HTTP模式), Envoy

CDN (内容分发网络)：
- 将静态资源缓存到边缘节点
- 用户就近访问，降低延迟
- 减少源站压力
- CloudFlare, AWS CloudFront, Akamai
```


## 自动扩缩容


```
Auto Scaling：
基于指标自动调整实例数量

触发条件：
- CPU利用率 > 70% → 扩容
- CPU利用率 < 30% → 缩容
- 请求队列长度 > 阈值 → 扩容
- 自定义业务指标

扩缩容策略：
- 目标追踪：维持指标在目标值附近
- 步进缩放：根据指标范围调整不同数量
- 预测扩缩：基于历史模式提前扩容

注意事项：
- 冷却时间：避免频繁扩缩
- 最小/最大实例数限制
- 优雅缩容：先摘流量再终止
- 预热：新实例需要初始化时间
```


> **Note:** 水平扩展的前提是无状态设计。状态需要外部化到Redis/数据库等共享存储。
