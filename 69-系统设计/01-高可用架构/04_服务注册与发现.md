# 服务注册与发现


## 服务注册与发现


系统设计高可用微服务


服务注册与发现是微服务架构的基础设施，解决服务实例动态变化时的寻址问题。


## 为什么需要服务发现


```
单体架构：服务间调用通过硬编码地址
  ServiceA → ServiceB (192.168.1.10:8080)

问题：
- ServiceB扩容到3台机器，地址变了怎么办？
- ServiceB某台机器挂了，怎么自动切换？
- ServiceB迁移到新机器，所有调用方都要改配置？

微服务架构中的挑战：
┌─────────────────────────────────────────────┐
│  服务实例动态变化：                           │
│  - 自动扩缩容：实例数量随时变化               │
│  - 滚动发布：新旧实例交替                     │
│  - 故障恢复：故障实例被替换                   │
│  - 弹性调度：K8s调度到不同节点               │
│                                             │
│  手动配置不可行：                             │
│  - 数百个服务 × 数十个实例 = 数千个地址       │
│  - 每次变更都要修改配置并重启                 │
│  - 无法做到实时感知故障                       │
└─────────────────────────────────────────────┘

服务发现解决的问题：
1. 服务注册：服务启动时注册自己的地址
2. 服务发现：调用方查询可用实例列表
3. 健康检查：自动剔除不可用实例
4. 负载均衡：在多个实例间分配请求
```


## 注册中心方案对比


```
┌────────────┬────────────┬──────────┬──────────────┐
│ 特性        │ Consul     │ Etcd     │ ZooKeeper    │
├────────────┼────────────┼──────────┼──────────────┤
│ 一致性协议  │ Raft       │ Raft     │ ZAB          │
│ 健康检查    │ 多种(TCP/  │ 租约     │ Session      │
│            │ HTTP/gRPC/ │ (Lease)  │ + 临时节点   │
│            │ TTL/脚本)  │          │              │
│ KV存储      │ 支持       │ 支持     │ 支持         │
│ 多数据中心  │ 原生支持   │ 不支持   │ 不支持       │
│ DNS接口     │ 支持       │ 不支持   │ 不支持       │
│ Web UI      │ 支持       │ 不支持   │ 第三方       │
│ 语言        │ Go         │ Go       │ Java         │
│ 性能        │ 高         │ 很高     │ 中等         │
│ 使用场景    │ 通用服务   │ K8s/配置 │ 分布式协调   │
│            │ 发现       │ 管理     │              │
└────────────┴────────────┴──────────┴──────────────┘

Consul详解：
┌─────────────────────────────────────────────┐
│  服务注册（HTTP API）：                       │
│  PUT /v1/agent/service/register             │
│  {                                          │
│    "ID": "web-1",                          │
│    "Name": "web-service",                  │
│    "Address": "192.168.1.10",              │
│    "Port": 8080,                           │
│    "Check": {                              │
│      "HTTP": "http://192.168.1.10:8080/health",│
│      "Interval": "10s",                    │
│      "Timeout": "5s"                       │
│    }                                        │
│  }                                          │
│                                             │
│  服务发现：                                   │
│  GET /v1/health/service/web-service?passing │
│  → 返回健康实例列表                          │
│                                             │
│  DNS接口：                                   │
│  dig @127.0.0.1 -p 8600 web-service.service.consul│
│  → 返回健康实例的IP地址                      │
└─────────────────────────────────────────────┘

Etcd详解：
┌─────────────────────────────────────────────┐
│  核心概念：租约(Lease) + 键值对              │
│                                             │
│  服务注册：                                   │
│  // 创建租约（TTL=10秒）                     │
│  lease, _ := client.Grant(ctx, 10)          │
│  // 写入服务信息（绑定租约）                  │
│  client.Put(ctx, "/services/web/192.168.1.10",│
│    "8080", etcd.WithLease(lease.ID))        │
│  // 定期续租（KeepAlive）                    │
│  client.KeepAlive(ctx, lease.ID)            │
│  // 服务宕机 → 不续租 → 租约过期 → 自动删除  │
│                                             │
│  服务发现：                                   │
│  resp, _ := client.Get(ctx, "/services/web/",│
│    etcd.WithPrefix())                       │
│  // 返回所有 /services/web/ 前缀的键值对     │
│                                             │
│  Watch机制：                                  │
│  watchChan := client.Watch(ctx, "/services/",│
│    etcd.WithPrefix())                       │
│  for resp := range watchChan {              │
│    // 实时收到服务变更通知                    │
│  }                                           │
└─────────────────────────────────────────────┘
```


## 客户端发现 vs 服务端发现


```
模式1：客户端发现 (Client-Side Discovery)
┌─────────────────────────────────────────────┐
│                                             │
│  ┌──────────┐      ┌──────────────┐        │
│  │ Service  │←────→│  Registry    │        │
│  │ Consumer │      │  (Consul/    │        │
│  │          │      │   Etcd)      │        │
│  └────┬─────┘      └──────┬───────┘        │
│       │                   │                 │
│       │  ② 获取实例列表   │ ① 服务注册      │
│       ↓                   ↓                 │
│  ┌──────────┐      ┌──────────────┐        │
│  │ 负载均衡 │      │ Service-A    │        │
│  │ (客户端) │      │ 实例1/2/3    │        │
│  └────┬─────┘      └──────────────┘        │
│       │ ③ 直接请求                           │
│       ↓                                     │
│  Service-A 实例                              │
│                                             │
│  优点：简单，客户端控制负载均衡策略           │
│  缺点：客户端与注册中心耦合，每种语言都要实现 │
│  代表：Eureka Client, Nacos Client          │
└─────────────────────────────────────────────┘

模式2：服务端发现 (Server-Side Discovery)
┌─────────────────────────────────────────────┐
│                                             │
│  ┌──────────┐      ┌──────────────┐        │
│  │ Service  │─────→│ Load Balancer│        │
│  │ Consumer │      │ / API Gateway│        │
│  └──────────┘      └──────┬───────┘        │
│                           │                 │
│                    ② 转发请求               │
│                           ↓                 │
│                    ┌──────────────┐         │
│                    │ Service-A    │←─① 注册  │
│                    │ 实例1/2/3   │          │
│                    └──────┬───────┘         │
│                           │                 │
│                    ┌──────┴───────┐         │
│                    │  Registry    │         │
│                    └──────────────┘         │
│                                             │
│  优点：客户端无感知，与注册中心解耦           │
│  缺点：负载均衡器是单点/需高可用              │
│  代表：K8s Service + Kube-Proxy             │
└─────────────────────────────────────────────┘

DNS服务发现：
┌─────────────────────────────────────────────┐
│  最简单的服务发现方式                         │
│                                             │
│  注册：服务注册时添加DNS A/SRV记录           │
│  发现：客户端通过DNS查询获取IP地址            │
│                                             │
│  示例：                                      │
│  A记录：                                     │
│    web-service.example.com → 192.168.1.10   │
│    web-service.example.com → 192.168.1.11   │
│    web-service.example.com → 192.168.1.12   │
│  SRV记录（含端口）：                          │
│    _http._tcp.web.example.com → 8080 192.168.1.10│
│                                             │
│  问题：                                      │
│  - DNS缓存导致变更不及时（TTL过期前）        │
│  - 不支持健康检查（需配合其他机制）           │
│  - SRV记录支持度有限                         │
│                                             │
│  K8s中的DNS：                                │
│  web-service.default.svc.cluster.local      │
│  → 自动解析到Service的ClusterIP             │
│  Kube-Proxy + iptables/IPVS做负载均衡       │
└─────────────────────────────────────────────┘
```


## 健康检查机制


```
健康检查类型：

1. 服务端主动心跳 (Push模式)
   服务定期向注册中心发送心跳
   ┌──────────┐  心跳(每5s)  ┌──────────┐
   │ Service  │────────────→│ Registry │
   └──────────┘             └──────────┘
   超过TTL未收到心跳 → 移除服务
   适用：Etcd租约、Eureka心跳

2. 注册中心主动探测 (Pull模式)
   注册中心定期探测服务的健康接口
   ┌──────────┐  HTTP GET   ┌──────────┐
   │ Service  │←───────────│ Registry │
   └──────────┘  /health   └──────────┘
   连续N次探测失败 → 标记不健康
   适用：Consul健康检查

3. 混合模式
   服务心跳 + 注册中心探测
   两者结合，更可靠

健康检查接口设计：
GET /health

{
  "status": "UP",
  "components": {
    "db": { "status": "UP", "details": { "database": "MySQL" } },
    "redis": { "status": "UP" },
    "diskSpace": { "status": "UP", "details": { "free": "10GB" } }
  }
}

状态值：
- UP：健康，可接收流量
- DOWN：不健康，应摘除流量
- OUT_OF_SERVICE：手动下线
- UNKNOWN：状态未知

检查要点：
1. 连通性检查：TCP能否连接
2. 依赖检查：数据库/Redis/MQ是否可达
3. 业务检查：关键逻辑是否正常
4. 资源检查：CPU/内存/磁盘是否充足

注意事项：
- 检查间隔不宜太短（避免误判）：5-30秒
- 超时时间要合理：2-5秒
- 连续失败阈值：2-3次
- 自我保护机制：避免网络分区时误删所有服务
```


> **Note:** 服务发现是微服务架构的核心基础设施。Consul适合通用场景，Etcd是K8s的标配，ZooKeeper适合需要分布式协调的场景。客户端发现模式简单直接，服务端发现模式对客户端更友好。
