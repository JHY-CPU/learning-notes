故障转移与冗余

系统设计高可用故障转移

高可用架构通过冗余和故障转移确保系统持续可用。

冗余模式

Active-Passive (主备模式)： - 主节点处理所有请求，备节点待命 - 主节点故障时切换到备节点 - 简单但浪费50%资源 - 适合：数据库主从复制 Active-Active (双活模式)： - 多个节点同时处理请求 - 共享受理负载 - 资源利用率高 - 需要处理数据一致性 - 适合：无状态服务 多活部署： - 同城双活：两个机房，低延迟同步 - 异地多活：多地部署，数据最终一致 - 单元化架构：按用户分片，各单元独立

故障转移机制

健康检查 (Health Check)： - 心跳检测：定期发送心跳包 - HTTP健康检查：GET /health → 200 - TCP端口检查：能否建立连接 - 超时判定：连续N次失败判定为故障 故障转移流程： 1. 检测到主节点故障 2. 选举新的主节点（Raft/ZooKeeper） 3. 更新DNS/VIP指向新主 4. 客户端重连到新主 5. 数据同步（如有延迟） 熔断器 (Circuit Breaker)： ┌──────────────────────────────────────┐ │ CLOSED → 失败率超阈值 → OPEN │ │ OPEN → 超时 → HALF_OPEN │ │ HALF_OPEN → 成功 → CLOSED │ │ HALF_OPEN → 失败 → OPEN │ └──────────────────────────────────────┘ 关键指标： - 可用性 = MTBF / (MTBF + MTTR) - 99.9% → 年停机8.76小时 - 99.99% → 年停机52.6分钟 - 99.999% → 年停机5.26分钟
