限流与熔断

一、限流算法

1. 固定窗口计数器（Fixed Window）

将时间划分为固定大小的窗口（如每分钟），在窗口内统计请求数，超过阈值则拒绝。

// 伪代码
if (current_window.count < limit) {
    current_window.count++;
    allow();
} else {
    reject();
}
缺陷：窗口边界问题——两个窗口交界处可能在极短时间内通过 2 倍限额的请求（突发流量问题）。

2. 滑动窗口计数器（Sliding Window）

将固定窗口细分为更小的子窗口，根据当前时间计算加权计数，平滑边界问题。

// 计算公式
请求数 = 上一窗口请求数 × (1 - 当前偏移比例) + 当前窗口请求数
// 例：窗口 60s，子窗口 10s，当前过了 25s
权重 = 1 - 25/60 = 0.583
有效请求数 = 上窗口数 × 0.583 + 当前窗口数
  • 优点：更精确的流量控制，消除边界突发
  • 缺点：需要存储子窗口计数，内存开销略增

3. 漏桶算法（Leaky Bucket）

请求进入队列（桶），以固定速率流出。桶满则拒绝新请求。

┌──────────┐
│ 请求流入  │
└────┬─────┘
     ▼
 ┌────────┐
 │  桶队列 │ ← 桶容量有限
 │ ▓▓▓▓▓▓ │
 └────┬───┘
      ▼  固定速率流出
   处理请求
  • 输出速率恒定，适合需要平滑流量的场景
  • 无法应对突发流量（即使系统有余力）
  • 典型应用：网络带宽控制、API 网关限流

4. 令牌桶算法（Token Bucket）

以固定速率向桶中放入令牌，请求需要获取令牌才能通过。桶有最大容量，允许一定程度的突发。

┌──────────────┐
│ 令牌生成器    │ 固定速率产生令牌
└────┬─────────┘
     ▼
 ┌────────┐
 │ 令牌桶  │ ← 最大容量 = burst
 │ ○○○○○  │
 └────┬───┘
      ▼  请求消耗令牌
   处理请求（无令牌则拒绝）
// Go 语言示例 - golang.org/x/time/rate
limiter := rate.NewLimiter(rate.Limit(10), 50) // 10/s，burst=50
if limiter.Allow() {
    // 处理请求
} else {
    // 拒绝
}
  • 优点：允许突发流量（桶内有令牌即可放行），同时限制平均速率
  • 令牌桶 vs 漏桶：令牌桶允许突发，漏桶强制恒定输出

5. 算法对比

算法突发处理精度实现复杂度典型场景
固定窗口差（边界问题）低简单粗粒度限流
滑动窗口较好高中等API 限流
漏桶不允许突发高简单流量整形
令牌桶允许突发高中等API 网关

二、分布式限流

1. Redis 实现令牌桶

-- Lua 脚本保证原子性
local key = KEYS[1]
local rate = tonumber(ARGV[1])     -- 每秒生成令牌数
local capacity = tonumber(ARGV[2]) -- 桶容量
local now = tonumber(ARGV[3])      -- 当前时间戳

local data = redis.call('HMGET', key, 'tokens', 'last_time')
local tokens = tonumber(data[1]) or capacity
local last_time = tonumber(data[2]) or now

-- 计算新生成的令牌
local delta = math.max(0, (now - last_time) * rate)
tokens = math.min(capacity, tokens + delta)

if tokens >= 1 then
    tokens = tokens - 1
    redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
    redis.call('EXPIRE', key, 60)
    return 1  -- 允许
else
    redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
    return 0  -- 拒绝
end

2. 多级限流策略

请求进入
  │
  ├── L1: 应用层限流（本地令牌桶，纳秒级）
  │
  ├── L2: 网关层限流（Redis 集中限流，毫秒级）
  │
  └── L3: 负载均衡层（Nginx limit_req，连接数限制）

三、熔断器（Circuit Breaker）

1. 三种状态

        请求成功
  ┌────────────────────┐
  │                    │
  ▼        超过阈值     │  超时后
┌──────┐ ─────────→ ┌──────┐ ─────────→ ┌──────┐
│ 关闭  │            │ 打开  │            │ 半开  │
│CLOSED │ ←──────── │ OPEN  │            │HALF  │
└──────┘  探测成功   └──────┘            │ OPEN │
                                        └──────┘
  • 关闭（Closed）：正常放行请求，统计失败率。当失败率超过阈值（如 50%），切换到打开状态
  • 打开（Open）：直接拒绝所有请求（快速失败），等待超时时间后进入半开状态
  • 半开（Half-Open）：放行少量探测请求。成功则关闭熔断器，失败则重新打开

2. 熔断触发条件

指标阈值示例说明
失败率50%（10s 内）请求失败比例超过阈值
慢调用率80%（RT > 500ms）慢请求比例超过阈值
异常数20 次（1min 内）异常次数超过阈值

3. Sentinel 熔断示例

// 配置慢调用熔断
DegradeRule rule = new DegradeRule("serviceA")
    .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
    .setCount(0.5)          // 慢调用比例阈值 50%
    .setSlowRatioThreshold(500) // RT 超过 500ms 视为慢调用
    .setMinRequestAmount(10)    // 最小请求数
    .setStatIntervalMs(10000)   // 统计窗口 10s
    .setTimeWindow(30);         // 熔断持续 30s

4. Resilience4j 熔断示例

// 配置
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)
    .slowCallRateThreshold(80)
    .slowCallDurationThreshold(Duration.ofMillis(500))
    .waitDurationInOpenState(Duration.ofSeconds(30))
    .permittedNumberOfCallsInHalfOpenState(5)
    .slidingWindowSize(100)
    .build();

CircuitBreaker breaker = CircuitBreaker.of("myService", config);

// 使用
Supplier decorated = CircuitBreaker
    .decorateSupplier(breaker, () -> remoteService.call());
Try result = Try.ofSupplier(decorated)
    .recover(CallNotPermittedException.class, e -> "降级响应");

四、限流与熔断的协同

客户端请求
    │
    ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  限流层   │ ──→│  熔断层   │ ──→│  服务处理  │
│Rate Limit│    │ Circuit  │    │ Service  │
└──────────┘    │ Breaker  │    └──────────┘
                └──────────┘
                     │
              熔断时触发降级
                     ▼
              ┌──────────┐
              │  降级策略  │
              │Fallback  │
              └──────────┘
  • 限流：保护自身不被过量请求压垮（入口保护）
  • 熔断：保护自身不被下游故障拖垮（出口保护）
  • 两者结合实现完整的弹性架构
面试要点：令牌桶是最常用的限流算法，因为它兼顾了速率控制和突发容忍。熔断器三状态模型是 Hystrix/Resilience4j/Sentinel 的核心设计。分布式限流的难点在于 Redis 节点间同步和时钟漂移问题。
