# 优雅停机与启动


## 优雅停机与启动


系统设计高可用滚动发布


优雅停机确保服务在停止前处理完已有请求，优雅启动确保新实例准备好后再接收流量，两者共同保障零宕机发布。


## 优雅停机 (Graceful Shutdown)


```
问题场景：
直接kill进程会导致什么？
- 正在处理的请求被中断，返回502错误
- 数据库事务未提交，数据不一致
- 消息消费未ACK，消息可能重复
- 缓存更新写到一半，缓存不一致

优雅停机的流程：
┌─────────────────────────────────────────────┐
│  1. 收到停止信号（SIGTERM/SIGINT）           │
│  2. 标记服务状态为"停止中"                   │
│  3. 从注册中心/负载均衡摘除流量              │
│  4. 等待正在处理的请求完成（设置超时）        │
│  5. 停止接受新请求                           │
│  6. 清理资源：关闭连接池、刷新缓冲区等       │
│  7. 正常退出                                 │
└─────────────────────────────────────────────┘

信号处理：
SIGTERM (kill默认)：优雅停机，可被处理
SIGKILL (kill -9)：强制终止，无法处理
SIGINT (Ctrl+C)：中断，可被处理

Java (Spring Boot) 实现：
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication app = new SpringApplication(Application.class);
        // 注册关闭钩子
        app.registerShutdownHook();
        app.run(args);
    }
}

// application.yml
server:
  shutdown: graceful

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s

// 自定义优雅停机
@PreDestroy
public void shutdown() {
    // 1. 停止接受新请求
    // 2. 等待线程池中的任务完成
    executor.shutdown();
    executor.awaitTermination(30, TimeUnit.SECONDS);
    // 3. 关闭数据库连接池
    dataSource.close();
    // 4. 刷新日志缓冲区
    logbackContext.stop();
}

Go实现：
srv := &http.Server{Addr: ":8080"}

// 在goroutine中启动服务
go func() {
    if err := srv.ListenAndServe(); err != http.ErrServerClosed {
        log.Fatalf("Server error: %v", err)
    }
}()

// 等待中断信号
quit := make(chan os.Signal, 1)
signal.Notify(quit, syscall.SIGTERM, syscall.SIGINT)
<-quit

// 收到信号，优雅停机
ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
defer cancel()
if err := srv.Shutdown(ctx); err != nil {
    log.Fatalf("Server forced to shutdown: %v", err)
}

Kubernetes中的优雅停机：
# Pod收到SIGTERM信号
# 1. Pod标记为Terminating
# 2. 从Service Endpoints中移除（停止接收新流量）
# 3. 等待preStop hook执行
# 4. 发送SIGTERM给容器
# 5. 等待terminationGracePeriodSeconds（默认30s）
# 6. 超时后发送SIGKILL强制终止

spec:
  terminationGracePeriodSeconds: 60
  containers:
  - name: app
    lifecycle:
      preStop:
        exec:
          command: ["/bin/sh", "-c", "sleep 5"]
        # sleep 5确保kube-proxy规则更新生效
```


## 连接排空 (Connection Draining)


```
连接排空：确保已建立的连接在停机前处理完

HTTP长连接场景：
┌─────────────────────────────────────────────┐
│  Client ←──HTTP Keep-Alive──→ Server        │
│                                             │
│  问题：多个请求复用同一个TCP连接             │
│  如果直接关闭连接，正在传输的请求会中断      │
│                                             │
│  排空策略：                                   │
│  1. 停止在连接上发送新的响应                 │
│  2. 等待已有请求的响应发送完成               │
│  3. 发送Connection: close头                  │
│  4. 客户端收到后不再复用该连接               │
│  5. 关闭TCP连接                              │
└─────────────────────────────────────────────┘

gRPC连接排空：
// Go gRPC
server := grpc.NewServer()
// ... 注册服务 ...

// 开启连接排空
grpc.GracefulStop(server)
// 等待所有RPC完成后再关闭

// 或者设置超时
ctx, _ := context.WithTimeout(context.Background(), 30*time.Second)
go func() { server.GracefulStop() }()
<-ctx.Done()
server.Stop() // 超时后强制关闭

消息队列消费排空：
┌─────────────────────────────────────────────┐
│  停机前：                                     │
│  1. 停止从MQ拉取新消息                       │
│  2. 等待当前正在处理的消息完成                │
│  3. 提交消费位移（ACK）                      │
│  4. 关闭消费者连接                           │
│                                             │
│  Spring Kafka配置：                          │
│  spring:                                     │
│    kafka:                                    │
│      listener:                              │
│        ack-mode: MANUAL_IMMEDIATE           │
│        # 优雅停机时自动提交offset            │
└─────────────────────────────────────────────┘

数据库连接排空：
// 停止从连接池获取新连接
// 等待正在使用的连接归还
// 关闭空闲连接
// 超时后强制关闭未归还的连接

HikariCP配置：
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      # 连接最大生命周期
      max-lifetime: 1800000
      # 优雅关闭超时
      # Spring Boot自动处理
```


## 优雅启动与预热 (Graceful Startup & Warm-up)


```
优雅启动的目标：
确保新实例完全准备好后再接收流量

启动阶段：
┌─────────────────────────────────────────────┐
│  1. 应用初始化                               │
│     - 加载配置                               │
│     - 初始化Bean/组件                        │
│     - 建立连接池                             │
│                                             │
│  2. 预热 (Warm-up)                           │
│     - JIT编译热点代码                        │
│     - 填充本地缓存                           │
│     - 预热模型/计算资源                      │
│                                             │
│  3. 健康检查通过                             │
│     - Readiness Probe返回OK                  │
│     - 注册到负载均衡                         │
│     - 开始接收流量                           │
│                                             │
│  4. 流量渐增                                 │
│     - 不要一上来就接全量流量                 │
│     - 逐步增加权重                           │
└─────────────────────────────────────────────┘

预热的重要性：
不预热的后果：
- JVM刚启动时大量代码未JIT编译 → 性能差
- 缓存为空 → 大量穿透到数据库
- 连接池刚建立 → 连接数不足
- 模型未加载 → 推理延迟高

Java (JVM) 预热：
1. JIT预热
   // 启动时执行一些模拟请求
   for (int i = 0; i < 10000; i++) {
       // 执行热点代码路径
       service.process(dummyRequest);
   }

2. Spring Boot预热
   @Component
   public class WarmupRunner implements ApplicationRunner {
       @Override
       public void run(ApplicationArguments args) {
           // 预热缓存
           cacheService.warmup();
           // 预热连接池
           warmupConnectionPool();
       }
   }

3. 分级启动
   @Component
   public class ReadinessProbe {
       private volatile boolean ready = false;

       @PostConstruct
       public void init() {
           // 异步初始化
           CompletableFuture.runAsync(() -> {
               warmupCaches();
               warmupModels();
               ready = true;
           });
       }

       public boolean isReady() { return ready; }
   }

健康检查分级：
┌─────────────────────────────────────────────┐
│  Liveness Probe（存活检查）：                │
│  - 应用是否还活着                            │
│  - 失败 → K8s重启Pod                         │
│  - 检查：进程是否响应、是否死锁              │
│                                             │
│  Readiness Probe（就绪检查）：               │
│  - 应用是否准备好接收流量                    │
│  - 失败 → 从Service Endpoints中移除          │
│  - 检查：依赖是否就绪、缓存是否预热          │
│                                             │
│  Startup Probe（启动检查）：                 │
│  - 应用是否完成启动                          │
│  - 启动期间暂时禁用liveness                  │
│  - 避免启动慢被K8s误杀                       │
└─────────────────────────────────────────────┘

K8s配置：
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5

startupProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  failureThreshold: 30
  periodSeconds: 10
  # 最多等30×10=300秒完成启动
```


## 滚动发布与流量渐增


```
滚动发布 (Rolling Update)：
逐步替换旧实例为新实例

┌─────────────────────────────────────────────┐
│  初始状态：V1 V1 V1 V1 V1                  │
│  步骤1：  V2 V1 V1 V1 V1  (替换1个)        │
│  步骤2：  V2 V2 V1 V1 V1  (替换1个)        │
│  步骤3：  V2 V2 V2 V1 V1  (替换1个)        │
│  ...                                        │
│  最终：   V2 V2 V2 V2 V2  (全部替换)        │
│                                             │
│  每一步都验证新实例健康后才继续              │
└─────────────────────────────────────────────┘

K8s滚动更新配置：
spec:
  replicas: 5
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多1个Pod
      maxUnavailable: 0  # 最少保留5个可用
  # maxSurge=1, maxUnavailable=0
  # → 每次先启动1个新Pod，等Ready后再终止1个旧Pod

流量渐增策略：
┌─────────────────────────────────────────────┐
│  金丝雀发布 (Canary)：                       │
│  - 新版本先接收1-5%的流量                    │
│  - 观察指标（错误率、延迟）                  │
│  - 正常则逐步增加流量比例                    │
│  - 异常则快速回滚                            │
│                                             │
│  实现方式：                                   │
│  1. Nginx权重分配                            │
│     upstream backend {                      │
│       server v1:8080 weight=95;            │
│       server v2:8080 weight=5;             │
│     }                                        │
│                                             │
│  2. Istio VirtualService                    │
│     route:                                   │
│     - destination:                           │
│         host: v1                            │
│       weight: 95                            │
│     - destination:                           │
│         host: v2                            │
│       weight: 5                             │
│                                             │
│  3. K8s + Argo Rollouts                     │
│     自动化金丝雀发布，配合Prometheus指标     │
└─────────────────────────────────────────────┘

蓝绿部署 (Blue-Green)：
┌─────────────────────────────────────────────┐
│  阶段1：V1(蓝)接收100%流量，V2(绿)空闲      │
│  ┌──────┐    ┌──────┐                      │
│  │ LB   │───→│ V1 蓝│ ← 100% 流量         │
│  └──────┘    └──────┘                      │
│              ┌──────┐                      │
│              │ V2 绿│ ← 0% 流量            │
│              └──────┘                      │
│                                             │
│  阶段2：切换流量到V2                         │
│  ┌──────┐    ┌──────┐                      │
│  │ LB   │───→│ V2 绿│ ← 100% 流量         │
│  └──────┘    └──────┘                      │
│              ┌──────┐                      │
│              │ V1 蓝│ ← 回滚备用           │
│              └──────┘                      │
│                                             │
│  优点：瞬间切换，回滚快                      │
│  缺点：需要双倍资源                          │
└─────────────────────────────────────────────┘

关键检查清单：
□ 优雅停机：SIGTERM信号处理、请求排空
□ 连接排空：HTTP/gRPC/MQ连接处理
□ 健康检查：Liveness + Readiness分级
□ 预热：JIT/缓存/模型预热完成
□ 流量渐增：金丝雀/滚动发布
□ 监控告警：新版本指标异常自动回滚
□ 回滚方案：保留旧版本镜像，快速切换
```


> **Note:** 优雅停机和启动是零宕机发布的基石。停机时要排空连接和请求，启动时要预热就绪后再接流量。K8s配合健康检查和滚动更新可以实现自动化的高可用发布。
