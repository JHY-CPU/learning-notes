# 01 - 高可用架构

高可用架构是大规模分布式系统设计的基石，目标是最大程度减少系统停机时间，保证服务的持续可用性。

---

## 1. 高可用的定义与量化

### 1.1 什么是高可用

高可用（High Availability, HA）指系统在面对硬件故障、软件异常、人为操作失误、自然灾害等意外场景时，仍能持续对外提供服务的能力。核心指标是**系统正常运行时间占总时间的比例**。

### 1.2 SLA（Service Level Agreement）

SLA 是服务提供商与用户之间的协议，定义了服务可用性的目标承诺。常见的可用性等级：

| 可用性等级 | 每年最大宕机时间 | 典型应用场景 |
|-----------|----------------|-------------|
| 99% | 3.65 天 | 内部系统、开发测试环境 |
| 99.9% | 8.76 小时 | 一般互联网服务 |
| 99.99% | 52.56 分钟 | 电商核心交易、支付 |
| 99.999% | 5.26 分钟 | 电信核心网、金融清算 |

### 1.3 可用性计算

可用性 = (总时间 - 故障时间) / 总时间。通常用**N个9**来衡量：

- **3个9（99.9%）**：每季度宕机约2小时
- **4个9（99.99%）**：每月宕机约4分钟
- **5个9（99.999%）**：每年宕机约5分钟

### 1.4 影响可用性的因素

- **硬件故障**：服务器宕机、磁盘损坏、网络设备异常
- **软件缺陷**：代码Bug、内存泄漏、死锁
- **依赖故障**：上游服务不可用、第三方接口异常
- **流量冲击**：突发流量超出容量设计
- **运维事故**：误操作、配置变更失败
- **自然灾害**：地震、洪水、断电

## 2. 冗余设计

冗余是实现高可用的根本手段，通过部署多余副本消除单点故障。

### 2.1 主从复制（一主多从）

主节点负责写操作，从节点负责读操作，数据从主节点异步或同步复制到从节点。

**同步复制**：写入必须等待从节点确认才返回，数据一致性强，但延迟高。
**异步复制**：写入在主节点确认后即返回，延迟低，但存在数据丢失风险。
**半同步复制**：至少一个从节点确认写入，兼顾一致性和延迟。

**故障处理**：主节点故障时，从节点通过选举机制（如Raft）提升为新主节点。

### 2.2 主主复制

两个节点互为主从，各自接受写入并同步到对方。适用于双机房部署。

**挑战**：
- 自增ID冲突：使用不同步长（奇偶）
- 数据冲突：两个节点同时修改同一数据
- 写入风暴：复制延迟导致不一致

### 2.3 双活与多活

**双活**：两个数据中心同时对外提供服务，互为备份。用户流量可以动态分配到任一中心。

**多活**：三个或更多数据中心同时承担业务流量，具备更强的容灾能力。

**多活架构模式**：
- **同城多活**：同城市多个机房，网络延迟低
- **异地多活**：不同城市数据中心，应对区域性灾难
- **全球化多活**：全球分布的数据中心，就近接入

## 3. 故障转移（Failover）

故障转移指当主节点失效时，自动将流量切换到备用节点的过程。

### 3.1 自动故障检测

**心跳检测**：主节点定期向监控组件发送心跳包，连续N次未收到心跳则判定故障。

**健康检查**：
- **TCP检查**：能否建立TCP连接
- **HTTP检查**：HTTP响应状态码是否为200
- **业务检查**：返回内容是否符合预期（如数据库查询是否正常）

**检测机制**：
- 主动探测（Pull模式）：监控系统定期发起健康检查
- 被动上报（Push模式）：服务节点定期上报健康状态
- 多维度投票：多个探测节点投票决定，避免单点误判

### 3.2 VIP漂移

VIP（Virtual IP）是一个虚拟IP地址，绑定在当前主节点上。当主节点故障时，VIP自动漂移到备用节点，客户端无感知。

**ARP广播方式**：备用节点接管VIP后，发送免费ARP广播，通知网络设备更新MAC地址映射。

### 3.3 脑裂问题及解决方案

**脑裂**：由于网络分区，两个节点都认为对方已故障，同时以主节点身份对外提供服务，导致数据不一致。

**解决方案**：
- **仲裁机制**：引入第三方仲裁节点（如ZooKeeper），由仲裁者决定谁是主
- **隔离Fence**：强制将故障节点从集群中隔离（如关闭电源、断开网络）
- **投票仲裁**：奇数节点投票，超过半数成为主
- **共享存储锁**：通过共享存储（如SAN）上的锁机制保证只有一个主
- **数据中心优先级**：指定优先级，脑裂时优先级高的中心当选

## 4. 负载均衡

负载均衡将流量合理分配到多个服务实例，提升系统吞吐量和可用性。

### 4.1 四层负载均衡（LVS）

工作在传输层（TCP/UDP），基于IP和端口进行转发，性能极高。

**LVS三种模式**：

- **NAT模式**：负载均衡器作为网关，客户端请求经过NAT转换后转发到后端服务器，响应数据也经过负载均衡器返回。配置简单，但负载均衡器成为瓶颈。
- **DR模式（直接路由）**：负载均衡器只转发请求，响应数据直接由后端服务器返回给客户端（修改MAC地址）。性能最优，但要求负载均衡器与后端服务器在同一局域网。
- **TUN模式（IP隧道）**：通过IP隧道封装数据包，负载均衡器可以与后端服务器在不同网络。适用于跨机房场景。

### 4.2 七层负载均衡

工作在应用层（HTTP/HTTPS），可以基于URL、Header、Cookie等应用层信息进行路由。

**Nginx**：高性能反向代理和负载均衡器，支持丰富的路由规则和缓存。
**HAProxy**：专业级负载均衡器，健康检查功能强大，支持精细的流量控制。
**云服务**：AWS ALB/ELB、阿里云SLB等云原生负载均衡服务。

### 4.3 负载均衡算法

- **轮询（Round Robin）**：依次分配请求到每个服务器，适用于服务器配置相同的场景。
- **加权轮询（Weighted Round Robin）**：根据服务器权重分配流量，权重高的获得更多请求，适用于异构服务器。
- **最少连接（Least Connections）**：将请求分配到当前连接数最少的服务器，适用于长连接场景。
- **一致性哈希（Consistent Hashing）**：根据请求特征（如用户ID）进行哈希，同一请求始终路由到同一服务器，适用于有状态服务。
- **最短响应时间（Least Response Time）**：选择响应时间最短的服务器，综合考虑连接数和响应速度。

### 4.4 会话保持

确保同一用户的请求始终路由到同一后端服务器。

- **Cookie会话保持**：负载均衡器植入Cookie记录服务器标识
- **IP哈希**：根据客户端IP进行哈希
- **URL参数哈希**：根据URL中的特定参数（如userId）进行哈希
- **会话外部化**：将Session存储到Redis等外部存储，从根本上避免会话依赖

## 5. 容错设计

容错设计旨在当部分组件出现故障时，系统仍能优雅降级而非完全不可用。

### 5.1 重试策略

**指数退避**：重试间隔按指数增长（如1s、2s、4s、8s），避免瞬时大量重试加剧故障。

**抖动（Jitter）**：在指数退避基础上增加随机偏移量，避免多个客户端同时重试产生同步效应。

**退避+抖动公式**：`sleep = random(0, min(cap, base * 2 ^ attempt))`

**重试注意事项**：
- 设置最大重试次数（通常3-5次）
- 仅对可重试错误进行重试（超时、5xx错误）
- 避免对非幂等操作盲目重试

### 5.2 熔断器模式（Circuit Breaker）

熔断器有三种状态：

- **关闭（Closed）**：正常状态，请求正常通过。持续统计失败次数。
- **打开（Open）**：故障状态，所有请求立即返回失败，不再调用下游服务。经过冷却时间后进入半开状态。
- **半开（Half-Open）**：尝试恢复，允许少量试探请求通过。如果成功则关闭熔断器；如果失败则重新打开。

**关键参数**：失败阈值、冷却时间、半开状态允许的请求数。

### 5.3 降级策略

- **功能降级**：关闭非核心功能（如推荐、评论），保证核心链路可用
- **数据降级**：返回缓存数据或默认数据，而非实时查询数据库
- **读降级**：降级为读取本地缓存或静态数据
- **写降级**：将写操作改为异步处理（如先写入本地队列再批量同步）

### 5.4 限流

限制请求速率，防止流量突增压垮系统。

**令牌桶算法**：以固定速率向桶中放入令牌，请求需要获取令牌才能通过。支持突发流量（桶中积累的令牌可以一次性消耗）。

**漏桶算法**：请求先进入桶中，以固定速率流出。严格控制输出速率，平滑突发流量。

**滑动窗口计数器**：将时间窗口划分为更小的子窗口，统计当前滑动窗口内的请求数量，比固定窗口更精确。

### 5.5 超时控制

- **连接超时**：建立连接的最大等待时间（通常几百毫秒）
- **读取超时**：等待响应数据的最大时间
- **全局超时**：整个请求链路的最大时间预算
- **上下文传递**：上游超时应传递到下游，避免下游做无用功

## 6. 数据备份与恢复

### 6.1 RPO与RTO

**RPO（Recovery Point Objective）**：灾难发生后，允许丢失的最大数据量。RPO越小，数据保护越完善。
**RTO（Recovery Time Objective）**：灾难发生后，系统恢复服务所需的最长时间。RTO越小，恢复速度越快。

| 场景 | RPO | RTO | 实现方式 |
|------|-----|-----|---------|
| 核心数据库 | 0 | 分钟级 | 同步复制+自动故障转移 |
| 业务系统 | 小时级 | 小时级 | 异步复制+手动切换 |
| 归档数据 | 天级 | 天级 | 定期备份+手动恢复 |

### 6.2 备份类型

- **热备份（Hot Standby）**：备份系统实时同步主系统数据，随时可接管。成本高，恢复最快。
- **温备份（Warm Standby）**：备份系统已部署但未运行，需要一定时间启动和恢复数据。
- **冷备份（Cold Backup）**：备份数据存储在离线介质上，恢复需要较长时间。成本最低。

### 6.3 灾难恢复计划

**灾难恢复计划（DRP）**的核心要素：
- **风险评估**：识别可能的灾难场景及其影响
- **备份策略**：确定备份频率、存储位置、保留周期
- **恢复流程**：定义详细的恢复步骤和负责人
- **定期演练**：定期进行灾难恢复演练，验证方案有效性
- **通信计划**：灾难发生时的沟通渠道和升级流程
