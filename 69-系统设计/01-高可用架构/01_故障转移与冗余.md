# 故障转移与冗余


## 故障转移与冗余


系统设计高可用故障转移


高可用架构通过冗余和故障转移确保系统持续可用。


## 冗余模式


```
Active-Passive (主备模式)：
- 主节点处理所有请求，备节点待命
- 主节点故障时切换到备节点
- 简单但浪费50%资源
- 适合：数据库主从复制

Active-Active (双活模式)：
- 多个节点同时处理请求
- 共享受理负载
- 资源利用率高
- 需要处理数据一致性
- 适合：无状态服务

多活部署：
- 同城双活：两个机房，低延迟同步
- 异地多活：多地部署，数据最终一致
- 单元化架构：按用户分片，各单元独立
```


## 故障转移机制


```
健康检查 (Health Check)：
- 心跳检测：定期发送心跳包
- HTTP健康检查：GET /health → 200
- TCP端口检查：能否建立连接
- 超时判定：连续N次失败判定为故障

故障转移流程：
1. 检测到主节点故障
2. 选举新的主节点（Raft/ZooKeeper）
3. 更新DNS/VIP指向新主
4. 客户端重连到新主
5. 数据同步（如有延迟）

熔断器 (Circuit Breaker)：
┌──────────────────────────────────────┐
│  CLOSED → 失败率超阈值 → OPEN        │
│  OPEN → 超时 → HALF_OPEN             │
│  HALF_OPEN → 成功 → CLOSED           │
│  HALF_OPEN → 失败 → OPEN             │
└──────────────────────────────────────┘

关键指标：
- 可用性 = MTBF / (MTBF + MTTR)
- 99.9% → 年停机8.76小时
- 99.99% → 年停机52.6分钟
- 99.999% → 年停机5.26分钟
```


## 负载均衡与故障检测

### 负载均衡策略

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| 轮询(Round Robin) | 依次分配请求 | 节点性能相近 |
| 加权轮询 | 按权重分配 | 节点性能差异大 |
| 最少连接 | 分配给连接数最少的节点 | 长连接场景 |
| 一致性哈希 | 按请求特征哈希到固定节点 | 有状态服务 |
| 最短响应时间 | 分配给响应最快的节点 | 对延迟敏感 |

### 分布式选举算法

**Raft 选举流程**：
```
1. 节点启动时为 Follower 状态
2. 若超时未收到 Leader 心跳，变为 Candidate
3. Candidate 向其他节点发送投票请求
4. 获得多数票的 Candidate 成为 Leader
5. Leader 定期向 Follower 发送心跳维持权威
```

**脑裂问题(Split Brain)**：
- 场景：网络分区导致集群分裂为两个子集群
- 后果：两个子集群各选出一个 Leader，数据不一致
- 解决方案：
  - 使用奇数节点（3/5/7）+ 多数投票
  - 引入仲裁节点(Arbiter)或见证者(Witness)
  - Fencing 机制：旧 Leader 检测到失联后自动降级

## 容灾与多活架构

### 容灾等级

| 等级 | RTO | RPO | 方案 | 成本 |
|------|-----|-----|------|------|
| 冷备 | 小时级 | 小时级 | 定期备份+手动恢复 | 低 |
| 温备 | 分钟级 | 分钟级 | 异步复制+自动切换 | 中 |
| 热备 | 秒级 | 秒级 | 同步复制+自动故障转移 | 高 |
| 双活 | 亚秒级 | 近零 | 多数据中心同时服务 | 最高 |

### 数据同步策略

```
同步复制：
  主 → [写入] → 从（等待确认）→ 返回客户端
  优点：数据零丢失
  缺点：延迟高，受网络影响大

异步复制：
  主 → [写入] → 返回客户端
  主 → [异步] → 从
  优点：延迟低
  缺点：主故障时可能丢失最近的数据

半同步复制：
  主 → [写入] → 至少一个从确认 → 返回客户端
  折中方案，兼顾数据安全和性能
```

## 实际案例

### Redis Sentinel 高可用

```
Sentinel 集群 (3+ 节点)
    │
    ├── 监控 Redis Master
    ├── 检测 Master 故障
    ├── 选举新 Master
    └── 通知客户端新地址

故障转移步骤：
1. Sentinel 检测到 Master 主观下线
2. 多个 Sentinel 确认客观下线
3. Sentinel 领导者选举
4. 选择最优 Slave 提升为 Master
5. 通知其他 Slave 复制新 Master
6. 通过 Pub/Sub 通知客户端
```

### Kubernetes 高可用

```
Master 高可用：
┌──────────────────────────────────┐
│  Load Balancer (HAProxy/云LB)    │
├──────┬──────┬───────────────────┤
│ API-1│ API-2│ API-3 (奇数个)     │
├──────┴──────┴───────────────────┤
│  etcd 集群 (Raft共识, 3/5节点)  │
└──────────────────────────────────┘

Pod 自愈：
- ReplicaSet 确保 Pod 副本数
- Node 故障时 Pod 自动调度到其他 Node
- 健康检查失败自动重启容器
```
