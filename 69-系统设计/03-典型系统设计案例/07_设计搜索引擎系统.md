# 设计搜索引擎系统


## 设计搜索引擎系统


系统设计搜索引擎端到端


从网页爬取到用户搜索结果展示，端到端设计一个搜索引擎系统。

└─────────────────────────────────────────────────┘

## 爬虫子系统


```
爬虫架构：
┌─────────────────────────────────────────────┐
│  种子URL                                     │
│    ↓                                        │
│  URL Frontier（优先级+礼貌队列）             │
│    ├── 优先级队列：重要页面优先              │
│    ├── 每站点独立队列：控制抓取频率          │
│    └── Host Scheduler：轮询调度             │
│    ↓                                        │
│  DNS Resolver（本地DNS缓存）                │
│    ↓                                        │
│  Fetcher集群（数百个爬虫进程）               │
│    ├── 多线程并发抓取                        │
│    ├── 遵守robots.txt                       │
│    └── If-Modified-Since增量抓取            │
│    ↓                                        │
│  内容解析                                    │
│    ├── 文本提取 → 送入索引管线              │
│    ├── 新链接提取 → 回到URL Frontier        │
│    └── 元数据提取（标题/描述）               │
└─────────────────────────────────────────────┘

URL去重（布隆过滤器）：
- 数十亿URL，用布隆过滤器去重
- 1%误判率 → 每URL约10bit
- 100亿URL × 10bit ≈ 12GB内存
- 可以放入单机内存

调度策略：
- 深度优先 vs 广度优先 vs 优先级
- Google的三阶段爬虫：
  1. Fresh crawler：高频更新的页面（新闻）
  2. Deep crawler：深层页面
  3. 补充抓取：低优先级页面

规模参考：
- Google：每天抓取数百亿页面
- 需要百万级爬虫节点
- 每秒处理数十万URL
```


## 索引子系统


```
索引构建流程：
┌─────────────────────────────────────────────┐
│  文档 → 分词 → 构建倒排表 → 压缩 → 分片存储 │
│                                             │
│  分词处理（中文）：                           │
│  - jieba / IK Analyzer / HanLP              │
│  - 同义词扩展                                │
│  - 停用词过滤                                │
│                                             │
│  倒排索引结构：                               │
│  Term → [(DocID, TF, Positions), ...]       │
│                                             │
│  索引分片：                                   │
│  - 按DocID范围分片                           │
│  - 每个分片有多个副本（容灾）                │
│  - 分片数固定（如64个分片）                  │
└─────────────────────────────────────────────┘

索引更新策略：
┌─────────────────────────────────────────────┐
│  全量索引：                                   │
│  - 定期（每天/每周）从头构建完整索引         │
│  - 构建期间旧索引继续服务                    │
│  - 构建完成后原子切换                        │
│                                             │
│  增量索引：                                   │
│  - 新文档构建小型倒排索引                    │
│  - 定期合并到主索引中                        │
│  - 实时性高，但查询需要合并多个索引          │
│                                             │
│  Elasticsearch的方案：                       │
│  - Segment（段）：不可变的索引文件           │
│  - 新文档写入内存Buffer                      │
│  - Buffer定期Flush为新Segment               │
│  - 后台Merge小Segment为大Segment            │
│  - 搜索时查询所有Segment，合并结果          │
└─────────────────────────────────────────────┘

规模估算（100亿网页）：
- 平均每页500个词条 → 5万亿词条
- 倒排表压缩后约原始大小的1/4
- 假设每词条倒排表平均1KB → 5TB索引
- 分64个分片 → 每分片约80GB
- 每分片3副本 → 总共约15TB
```


## 查询处理与排序


```
查询处理流程：
┌─────────────────────────────────────────────┐
│  用户输入："机器学习 入门教程"                │
│    ↓                                        │
│  1. 查询分析                                 │
│     - 分词：["机器学习", "入门", "教程"]     │
│     - 意图识别：教程类查询                   │
│     - 语言检测：中文                         │
│    ↓                                        │
│  2. 查询改写                                 │
│     - 拼写纠正                               │
│     - 同义词扩展                             │
│     - 查询简化（去停用词）                   │
│    ↓                                        │
│  3. 检索（多路召回）                         │
│     - 倒排索引：BM25匹配                     │
│     - 向量检索：语义相似                     │
│     - 融合：RRF                              │
│    ↓                                        │
│  4. 排序                                     │
│     - 第一阶段：BM25粗排（Top-1000）         │
│     - 第二阶段：Learning to Rank精排（Top-100）│
│     - 第三阶段：业务规则调整（Top-10）       │
│    ↓                                        │
│  5. 结果处理                                 │
│     - 去重（相似URL合并）                    │
│     - 摘要生成（高亮关键词）                 │
│     - 分页                                   │
│    ↓                                        │
│  返回给前端                                  │
└─────────────────────────────────────────────┘

排序信号（数百个特征）：
┌─────────────────────────────────────────────┐
│  文本相关性：                                 │
│  - BM25得分                                  │
│  - 标题匹配度                                │
│  - URL中的关键词                             │
│  - 锚文本（其他网站链接的描述文字）          │
│                                             │
│  链接分析：                                   │
│  - PageRank                                  │
│  - 入链数量和质量                            │
│                                             │
│  内容质量：                                   │
│  - 页面新鲜度                                │
│  - 内容长度和丰富度                          │
│  - 是否原创                                  │
│  - 用户体验信号（Core Web Vitals）           │
│                                             │
│  用户行为：                                   │
│  - 点击率（CTR）                             │
│  - 停留时间                                  │
│  - 跳出率                                    │
│                                             │
│  个性化：                                     │
│  - 用户搜索历史                              │
│  - 用户地理位置                              │
│  - 用户语言偏好                              │
└─────────────────────────────────────────────┘
```


## 扩展性与容灾


```
水平扩展：
┌─────────────────────────────────────────────┐
│  爬虫扩展：                                   │
│  - 增加爬虫节点即可                          │
│  - 无状态，共享URL Frontier（分布式队列）    │
│                                             │
│  索引扩展：                                   │
│  - 增加索引分片                              │
│  - 每个分片独立构建和更新                    │
│  - 搜索时并行查询所有分片                    │
│                                             │
│  查询扩展：                                   │
│  - 增加查询节点（无状态）                    │
│  - 负载均衡分发请求                          │
│  - 缓存热门查询结果                          │
│                                             │
│  分片示意：                                   │
│  ┌────────┐ ┌────────┐ ┌────────┐          │
│  │ Shard 0│ │ Shard 1│ │ Shard 2│ ...      │
│  │ Doc 0- │ │ Doc N- │ │ Doc 2N-│          │
│  │ Doc N-1│ │ Doc 2N │ │ Doc 3N │          │
│  │ +副本  │ │ +副本  │ │ +副本  │          │
│  └────────┘ └────────┘ └────────┘          │
│       ↓          ↓          ↓               │
│  ┌─────────────────────────────────┐        │
│  │  查询聚合（合并各分片结果）      │        │
│  │  排序 → 取Top-K → 返回          │        │
│  └─────────────────────────────────┘        │
└─────────────────────────────────────────────┘

容灾设计：
- 索引分片多副本（至少3副本）
- 跨机房/跨区域部署
- 爬虫节点故障自动重分配URL
- 查询节点无状态，故障自动摘除
- 数据中心级别的容灾（多活）

性能指标参考：
┌─────────────────────────────────────────────┐
│  爬虫规模：                                  │
│  - Google：数百万爬虫节点                    │
│  - 每秒抓取数十万网页                        │
│  - 总索引数千亿网页                          │
│                                             │
│  查询规模：                                  │
│  - Google：每秒处理数万查询                  │
│  - 查询延迟：平均200-500ms                  │
│  - 其中网络延迟占大部分                      │
│                                             │
│  索引规模：                                  │
│  - Google：数百PB的索引数据                  │
│  - 每天更新数十亿网页的索引                  │
└─────────────────────────────────────────────┘
```


> **Note:** 搜索引擎是分布式系统的经典案例，涵盖了爬虫、索引、检索、排序等多个子系统。核心挑战在于处理海量数据（数千亿网页）的同时保证查询延迟（<500ms）。
