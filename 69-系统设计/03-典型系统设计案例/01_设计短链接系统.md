# 设计短链接系统


## 设计短链接系统


系统设计短链接URL


将长URL转换为短链接，重定向到原始URL。


## 需求分析


```
功能需求：
- 给定长URL，生成短链接
- 访问短链接，重定向到原始URL
- 短链接可自定义（可选）
- 短链接有过期时间（可选）
- 点击统计（可选）

非功能需求：
- 高可用：99.99%
- 低延迟：重定向 < 100ms
- 短链接不可预测

容量估算（每日1亿次创建）：
- 写QPS：100M / 86400 ≈ 1160/s
- 读QPS：读写比100:1 → 116000/s
- 存储5年：100M × 365 × 5 × 500B ≈ 9TB
- 缓存：20%的热点URL → 内存约2TB
```


## 短码生成方案


```
Base62编码：
字符集：[a-z, A-Z, 0-9] 共62个字符
7位Base62 = 62^7 ≈ 3.5万亿种组合

方案1：哈希截断
short_code = md5(long_url)[:7]
优点：相同URL得到相同短码
缺点：哈希冲突，需要检查数据库

方案2：自增ID + Base62
ID: 10000 → Base62: "2bI"
优点：简单无冲突
缺点：可预测（递增），暴露URL数量

方案3：预生成 + 分发
预先生成随机短码池
各应用服务器从池中获取
优点：高性能，不可预测
缺点：需要管理短码池

方案4：Snowflake ID
分布式ID生成器，保证唯一
再Base62编码
```


## 架构设计


```
┌─────────────────────────────────────────────┐
│  客户端 → CDN → 负载均衡 → 应用服务器         │
│                           ↓                  │
│                     缓存(Redis)               │
│                           ↓                  │
│                     数据库(MySQL分库分表)      │
│                           ↓                  │
│                     分析(Kafka → Spark)       │
└─────────────────────────────────────────────┘

写入流程：
1. 检查URL是否已存在（缓存/数据库）
2. 生成短码
3. 存入数据库（短码 → 长URL）
4. 写入缓存
5. 返回短链接

读取流程：
1. 从缓存查找短码 → 长URL
2. 缓存未命中 → 查数据库 → 回写缓存
3. 301/302重定向到长URL
4. 异步记录访问日志

数据库分片：
- 按短码哈希分片
- 64个分片，每分片约150GB(5年)

缓存策略：
- Redis集群
- LRU淘汰
- 热点URL自动缓存

301 vs 302重定向：
- 301永久：浏览器缓存，减少服务器压力
- 302临时：每次经过服务器，利于统计
```
