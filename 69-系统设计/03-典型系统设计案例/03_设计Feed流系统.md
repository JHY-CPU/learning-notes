# 设计 Feed 流系统


## 一、需求分析


- 用户发布内容（微博/朋友圈/推文）
- 用户查看关注者的内容（Timeline）
- Feed 按时间/算法排序
- 规模：千万用户，亿级内容，每秒数万发布


## 二、Fanout 模式


### 1. 推模式（Push / Write Fanout）


```
用户A 发布内容
    │
    ▼
查询A的所有粉丝 [B, C, D, E, ...]
    │
    ▼
将内容ID写入每个粉丝的 Timeline（收件箱）
    │
    ▼
B 的 Timeline: [..., post_A_1001]
C 的 Timeline: [..., post_A_1001]

读取时：直接查自己的 Timeline，零计算
```


- **优点：**
   读取极快（预计算好了），适合读多写少场景
- **缺点：**
   大V发布时写放大严重（1亿粉丝 → 1亿次写入），冷启动问题
- **适用：**
   普通用户（粉丝数较少）


### 2. 拉模式（Pull / Read Fanout）


```
用户B 查看 Feed
    │
    ▼
查询B关注的所有人 [A, C, D, ...]
    │
    ▼
拉取每个人最新的内容
    │
    ▼
合并排序，返回 Feed
```


- **优点：**
   写入零开销，没有大V写放大问题
- **缺点：**
   读取慢（需要聚合N个用户的最新内容），关注数多时性能差
- **适用：**
   被关注的大V


### 3. 推拉结合（Hybrid）—— 实际首选


```
发布者粉丝数 < 阈值（如5000）→ 推模式：写入粉丝收件箱
发布者粉丝数 >= 阈值（大V）   → 拉模式：粉丝读取时实时拉取

读取时：
1. 从自己的收件箱（推模式）拉取内容
2. 从关注的大V（拉模式）拉取最新内容
3. 合并排序
4. 可加缓存减少重复聚合
```


## 三、核心存储设计


### 1. Feed 流存储（Timeline / 收件箱）


```
// Redis Sorted Set —— 最常用方案
// key: feed:{user_id}
// member: post_id
// score: timestamp

ZADD feed:1001 1700000001 "post:2001"
ZADD feed:1001 1700000002 "post:2002"

// 读取最新 20 条
ZREVRANGE feed:1001 0 19

// 按 score 范围翻页
ZREVRANGEBYSCORE feed:1001 +inf 1700000000 LIMIT 0 20
```


### 2. 内容存储


```
// 内容表
posts: {
    post_id: "post:2001",
    user_id: 1001,
    content: "今天天气不错",
    media_urls: [...],
    created_at: "2024-01-15 10:00:00",
    like_count: 0,
    comment_count: 0
}

// 存储选型
热数据（最近7天） → Redis Hash / MongoDB
温数据（7-30天）  → MySQL + 缓存
冷数据（30天+）  → 归档到 HBase / ES
```


### 3. 关注关系存储


```
// Redis Set 存储关注/粉丝关系
SADD following:1001 2001 2002 2003  // 1001 关注了谁
SADD followers:2001 1001 1002       // 2001 的粉丝

// 查粉丝列表
SMEMBERS followers:2001

// 判断是否关注
SISMEMBER following:1001 2001
```


## 四、Feed 排序


### 1. 时间线排序（Chronological）


按发布时间倒序，简单直接。Twitter 默认模式。


### 2. 算法排序（Ranking）


```
score = w1 * 时间衰减 + w2 * 互动率 + w3 * 相关性 + w4 * 内容质量

// 影响因子
- 时间衰减：越新分越高（指数衰减）
- 互动率：点赞/评论/转发比例
- 相关性：用户兴趣标签匹配度
- 内容质量：作者权重、内容类型偏好
- 社交关系：共同好友互动
```


- 离线特征计算 + 在线排序服务
- 需要 A/B 测试持续优化


## 五、整体架构


```
发帖请求
    │
    ▼
┌──────────┐     ┌──────────┐     ┌──────────┐
│ API 服务  │ ──→│ 发帖服务   │ ──→│ 内容存储  │ (MySQL/MongoDB)
└──────────┘     └────┬─────┘     └──────────┘
                      │
                      ▼
                ┌──────────┐
                │ Fanout   │
                │ Worker   │ ← 异步消息队列(Kafka)
                └────┬─────┘
                     │
          ┌──────────┼──────────┐
          ▼          ▼          ▼
     ┌────────┐ ┌────────┐ ┌────────┐
     │Redis   │ │Redis   │ │Redis   │
     │粉丝A的  │ │粉丝B的  │ │粉丝C的  │
     │收件箱   │ │收件箱   │ │收件箱   │
     └────────┘ └────────┘ └────────┘

读取 Feed：
用户请求 → API → Feed聚合服务 → 合并收件箱+拉取大V → 排序 → 返回
```


## 六、关键优化


- **Fanout 异步化：**
   通过 Kafka 异步 fanout，发布请求立即返回
- **Fanout Cache：**
   热门用户的收件箱预计算 + 缓存
- **分级存储：**
   活跃用户 Feed 放 Redis，不活跃用户放 DB
- **分页优化：**
   使用游标（cursor）而非 offset，避免深翻页性能问题
- **已读标记：**
   记录用户上次阅读位置，避免重复推送


> **Note:** **面试要点：**
> 推拉结合是 Feed 系统的标准答案。推模式写放大大V问题、拉模式读放大问题、如何结合（阈值分界）是核心考点。Redis Sorted Set 作为 Timeline 存储是最常见方案。Fanout 通过消息队列异步化是性能关键。

总结：Feed流三种fanout模式——推（写放大）、拉（读放大）、推拉结合（实际首选，大V用拉普通用户用推）；Redis Sorted Set 是Timeline存储首选；Fanout通过Kafka异步解耦；排序分时间线和算法两种；整体架构是发帖→异步Fanout→写入收件箱→读取聚合排序
