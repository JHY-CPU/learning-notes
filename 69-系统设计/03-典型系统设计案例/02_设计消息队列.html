设计消息队列

系统设计消息队列Kafka

消息队列实现生产者和消费者的异步解耦。

核心概念

┌─────────────────────────────────────────────┐ │ Producer → Topic → Partition → Consumer │ │ │ │ Topic：消息分类（如订单事件、日志） │ │ Partition：Topic的分片，实现并行 │ │ Offset：消息在Partition中的序号 │ │ Consumer Group：消费者组，组内竞争消费 │ │ Broker：消息队列服务器节点 │ └─────────────────────────────────────────────┘ 消息语义： - At-most-once：可能丢失，不重复 - At-least-once：不丢失，可能重复（最常用） - Exactly-once：不丢失不重复（最难实现） 顺序保证： - Partition内有序 - Topic整体无序（除非单Partition） - 按Key分区保证同一Key的消息有序

架构设计

存储设计（Kafka方式）： ┌─────────────────────────────────────────────┐ │ Partition = 有序的、不可变的消息序列 │ │ 物理存储：追加写入的日志文件 │ │ index/00000000000000000000.log │ │ index/00000000000000000000.index │ │ index/00000000000000000000.timeindex │ │ │ │ - 顺序写磁盘，利用OS Page Cache │ │ - 零拷贝(zero-copy)发送给消费者 │ │ - Segment按大小或时间滚动 │ │ - 过期数据自动删除 │ └─────────────────────────────────────────────┘ 高可用： - 每个Partition有多个副本（ISR集合） - Leader处理读写，Follower同步 - Leader故障时从ISR中选举新Leader - min.insync.replicas控制最少同步副本 消费者模型： - Push模型：Broker推送给Consumer - Pull模型：Consumer主动拉取（Kafka方式） - Consumer控制消费速度 - 空Topic时无额外开销 - 批量拉取提高吞吐 Exactly-once实现： - 幂等生产者：ProducerID + SequenceNumber - 事务：原子写入多个Partition - 消费者：提交offset和处理结果在同一事务中

关键设计决策

┌─────────────────────────────────────────────┐ │ 消息持久化： │ │ - 内存：快但丢失风险 │ │ - 磁盘：安全但较慢 │ │ - Kafka：磁盘+Page Cache ≈ 内存速度 │ │ │ │ 消息顺序： │ │ - 单Partition保证顺序 │ │ - 多Partition需要业务层处理 │ │ │ │ 死信队列(DLQ)： │ │ - 处理失败的消息进入DLQ │ │ - 后续人工处理或重试 │ │ │ │ 延迟队列： │ │ - 延迟消息的投递 │ │ - RocketMQ原生支持 │ │ - Kafka需要多层Topic模拟 │ │ │ │ 消息回溯： │ │ - 按时间或offset重新消费 │ │ - Kafka支持任意offset消费 │ └─────────────────────────────────────────────┘ 主流MQ对比： - Kafka：高吞吐，大数据/流处理 - RabbitMQ：功能丰富，中小规模 - RocketMQ：事务消息，电商场景 - Pulsar：存算分离，多租户
Kafka的高性能源于顺序写磁盘+零拷贝+批量处理。设计MQ需要平衡一致性、可用性和性能。
