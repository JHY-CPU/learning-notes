设计配置中心

系统设计配置中心微服务

配置中心统一管理微服务的配置，支持动态更新、版本管理和灰度发布，避免修改配置重启服务。

为什么需要配置中心

传统配置管理的问题： ┌─────────────────────────────────────────────┐ │ 1. 配置散落在各服务的配置文件中 │ │ service-a/application.yml │ │ service-b/application.yml │ │ 修改同一个配置要改多个文件 │ │ │ │ 2. 修改配置需要重启服务 │ │ 改完配置 → 重新部署 → 服务中断 │ │ 对于需要频繁调整的配置不方便 │ │ │ │ 3. 缺乏版本管理和审计 │ │ 谁改了什么配置？什么时候改的？ │ │ 改错了怎么回滚？ │ │ │ │ 4. 多环境管理困难 │ │ 开发/测试/预发/生产环境配置不同 │ │ 容易搞混 │ │ │ │ 5. 无法灰度发布配置 │ │ 配置变更直接全量生效 │ │ 出问题影响所有实例 │ └─────────────────────────────────────────────┘ 配置中心解决的问题： - 配置统一存储和管理 - 配置动态推送，无需重启 - 配置版本管理和回滚 - 多环境、多集群配置隔离 - 配置灰度发布 - 配置变更审计

核心架构

┌─────────────────────────────────────────────┐ │ ┌──────────────────────────────┐ │ │ │ 管理界面 (Admin Portal) │ │ │ │ ├── 配置编辑 │ │ │ │ ├── 版本管理 │ │ │ │ ├── 灰度发布 │ │ │ │ └── 审计日志 │ │ │ └──────────┬───────────────────┘ │ │ ↓ │ │ ┌──────────────────────────────┐ │ │ │ 配置服务 (Config Server) │ │ │ │ ├── 配置读写API │ │ │ │ ├── 配置推送（长轮询/推送） │ │ │ │ ├── 权限控制 │ │ │ │ └── 灰度路由 │ │ │ └──────────┬───────────────────┘ │ │ ↓ │ │ ┌──────────────────────────────┐ │ │ │ 配置存储 (Storage) │ │ │ │ ├── MySQL：配置元数据 │ │ │ │ ├── 配置文件备份 │ │ │ │ └── 版本历史 │ │ │ └──────────────────────────────┘ │ │ ↑ │ │ ┌──────────────────────────────┐ │ │ │ 客户端SDK │ │ │ │ ├── 本地缓存（磁盘） │ │ │ ├── 长轮询/订阅 │ │ │ │ ├── 配置变更回调 │ │ │ │ └── 降级：读本地缓存 │ │ │ └──────────────────────────────┘ │ └─────────────────────────────────────────────┘ 配置数据模型： ┌─────────────────────────────────────────────┐ │ Namespace（命名空间）：最高层级隔离 │ │ ├── application（默认） │ │ ├── payment-service │ │ └── common-config │ │ │ │ 集群/环境： │ │ ├── dev │ │ ├── test │ │ └── production │ │ │ │ 配置项（Key-Value）： │ │ db.url = jdbc:mysql://... │ │ db.maxPoolSize = 50 │ │ feature.newUI.enabled = true │ │ │ │ 唯一标识：namespace + cluster + key │ └─────────────────────────────────────────────┘

配置变更推送机制

方案1：长轮询 (Long Polling) ★ Apollo使用 ┌─────────────────────────────────────────────┐ │ 客户端发起HTTP请求，服务端hold住 │ │ 如果配置有变更 → 立即返回变更通知 │ │ 如果超时（如60秒）无变更 → 返回空，客户端重试│ │ │ │ 客户端： │ │ while (true) { │ │ response = http.get( │ │ "/notifications?appId=xxx&keys=..."); │ │ if (response.hasChanges) { │ │ fetchLatestConfig(); │ │ notifyListeners(); │ │ } │ │ } │ │ │ │ 服务端： │ │ def long_poll(request): │ │ changes = wait_for_changes(request.keys, timeout=60s)│ │ return changes │ │ │ │ 优点：简单，不需要特殊协议 │ │ 缺点：有一定延迟（取决于轮询间隔） │ └─────────────────────────────────────────────┘ 方案2：WebSocket推送 ┌─────────────────────────────────────────────┐ │ 客户端与服务端建立WebSocket连接 │ │ 配置变更时，服务端主动推送 │ │ │ │ 优点：实时性高 │ │ 缺点：需要维护长连接，实现复杂 │ └─────────────────────────────────────────────┘ 方案3：基于消息队列 ┌─────────────────────────────────────────────┐ │ 配置变更 → 发送MQ消息 → 客户端消费 │ │ 优点：解耦，支持大量客户端 │ │ 缺点：引入MQ依赖 │ └─────────────────────────────────────────────┘ 客户端本地缓存： ┌─────────────────────────────────────────────┐ │ 配置中心不可用时，客户端读取本地缓存 │ │ │ │ // 本地缓存文件 │ │ /opt/config-cache/app-namespace.properties │ │ │ │ 启动流程： │ │ 1. 尝试从配置中心拉取最新配置 │ │ 2. 成功 → 更新本地缓存 → 使用最新配置 │ │ 3. 失败 → 使用本地缓存 → 后台重试拉取 │ │ │ │ 这保证了即使配置中心宕机，服务也能正常启动 │ └─────────────────────────────────────────────┘

配置灰度发布与版本管理

灰度发布 (Gray Release)： 配置变更不直接全量生效，而是先在部分实例上生效 灰度策略： ┌─────────────────────────────────────────────┐ │ 1. 按IP灰度 │ │ 只对指定IP的实例生效 │ │ 灰度IP：192.168.1.10, 192.168.1.11 │ │ 其他实例保持旧配置 │ │ │ │ 2. 按集群灰度 │ │ 只对指定集群生效 │ │ 灰度集群：pre-production │ │ 其他集群保持旧配置 │ │ │ │ 3. 按百分比灰度 │ │ 随机选择10%的实例应用新配置 │ │ 观察无异常后扩大比例 │ │ │ │ 灰度发布流程： │ │ 1. 创建灰度规则（选择灰度对象） │ │ 2. 发布灰度版本（灰度实例应用新配置） │ │ 3. 观察灰度效果（监控指标是否正常） │ │ 4. 全量发布 or 回滚 │ └─────────────────────────────────────────────┘ 客户端灰度逻辑： public String getConfig(String key) { String clientIp = getLocalIp(); // 检查是否有灰度配置 GrayRule rule = getGrayRule(namespace, key); if (rule != null && rule.matches(clientIp)) { return rule.getValue(); // 使用灰度值 } return getNormalValue(namespace, key); // 使用正式值 } 版本管理与回滚： ┌─────────────────────────────────────────────┐ │ 每次配置变更都创建一个新版本 │ │ │ │ 版本历史： │ │ v5 (当前) 2024-01-15 10:30 admin 修改超时 │ │ v4 2024-01-14 15:00 admin 修改连接池│ │ v3 2024-01-10 09:00 admin 初始配置 │ │ │ │ 回滚操作： │ │ 1. 选择要回滚到的版本 │ │ 2. 系统将该版本的配置设为当前生效版本 │ │ 3. 推送变更通知到所有客户端 │ │ 4. 记录回滚操作到审计日志 │ │ │ │ 重要：配置版本不可物理删除 │ │ 只标记为历史版本，保留完整历史 │ └─────────────────────────────────────────────┘ 配置加密： ┌─────────────────────────────────────────────┐ │ 敏感配置（数据库密码、API Key等）需要加密 │ │ │ │ 方案1：服务端加密存储 │ │ - 配置值用AES加密后存储 │ │ - 客户端SDK解密后使用 │ │ - 密钥通过独立渠道分发 │ │ │ │ 方案2：集成密钥管理服务(KMS) │ │ - 配置值存储密文，实际值在KMS中 │ │ - 客户端通过KMS API获取明文 │ │ - 更安全，但增加依赖 │ │ │ │ 展示时脱敏： │ │ db.password = ****** │ │ 只有授权用户才能查看明文 │ └─────────────────────────────────────────────┘

配置中心是微服务基础设施的重要组成。Apollo适合Java微服务生态，Nacos集配置和服务发现于一体。核心能力是配置动态推送、版本管理和灰度发布，客户端本地缓存保证高可用。
