# 设计支付系统


## 设计支付系统


系统设计支付金融


支付系统处理资金流转，对一致性和可靠性要求极高，核心是资金安全和精确一次处理。


## 支付流程总览


```
用户下单支付全流程：
┌─────────────────────────────────────────────┐
│  1. 用户在商户下单                           │
│  2. 商户系统 → 支付系统：创建支付单         │
│  3. 支付系统 → 收银台：展示支付方式         │
│  4. 用户选择支付方式（微信/支付宝/银行卡）   │
│  5. 收银台 → 支付渠道：发起扣款             │
│  6. 支付渠道 → 用户：验证（密码/指纹/人脸） │
│  7. 支付渠道 → 支付系统：异步通知扣款结果   │
│  8. 支付系统 → 商户：通知支付结果           │
│  9. 商户系统 → 用户：展示订单状态           │
│  10. 支付系统 → 清算系统：日终清算对账      │
└─────────────────────────────────────────────┘

系统组件：
┌─────────────────────────────────────────────┐
│  ┌─────────┐    ┌──────────────┐            │

│                        ↓                    │
│  ┌──────────────────────────────────────┐   │
│  │           支付核心                    │   │
│  │  ┌────┐ ┌────┐ ┌────┐ ┌────┐        │   │
│  │  │订单│ │路由│ │风控│ │账务│        │   │
│  │  └────┘ └────┘ └────┘ └────┘        │   │
│  └──────────┬───────────────────────────┘   │
│             ↓                               │
│  ┌──────────────────────────────────────┐   │
│  │           渠道网关                    │   │
│  │  ┌──────┐ ┌──────┐ ┌──────┐         │   │
```

## 支付幂等性保证


```
幂等性的必要性：
网络超时、用户重复点击、系统重试都可能导致重复支付
支付系统必须保证：同一笔支付只扣一次钱

幂等方案：

方案1：支付流水号（推荐）
┌─────────────────────────────────────────────┐
│  商户生成唯一的支付流水号（payment_no）      │
│  支付系统用 payment_no 做幂等判断           │
│                                             │
│  // 数据库唯一索引                           │
│  CREATE TABLE payment_order (               │
│    id BIGINT PRIMARY KEY,                   │
│    payment_no VARCHAR(64) UNIQUE,  -- 幂等键│
│    merchant_no VARCHAR(32),                 │
│    amount DECIMAL(10,2),                    │
│    status TINYINT,                          │
│    ...                                       │
│  );                                          │
│                                             │
│  // 处理逻辑                                 │
│  PaymentOrder existing = findByPaymentNo(paymentNo);│
│  if (existing != null) {                    │
│    return existing;  // 直接返回已有结果    │
│  }                                            │
│  // 新请求，创建支付单                       │
│  createPaymentOrder(request);               │
└─────────────────────────────────────────────┘

方案2：状态机 + CAS
支付单有明确的状态流转：
待支付 → 支付中 → 支付成功/支付失败/已关闭

只有在特定状态下才能执行特定操作：
UPDATE payment_order
SET status = '支付成功'
WHERE payment_no = ? AND status = '支付中'
-- 影响行数=1：状态更新成功
-- 影响行数=0：状态不匹配（已处理或非法状态）

方案3：分布式锁
// 支付前先获取锁
Boolean locked = redisTemplate.opsForValue()
    .setIfAbsent("lock:payment:" + paymentNo, "1", 5, TimeUnit.MINUTES);
if (!locked) {
    // 可能正在处理，轮询结果
    return queryPaymentResult(paymentNo);
}
try {
    return processPayment(request);
} finally {
    redisTemplate.delete("lock:payment:" + paymentNo);
}
```


## 记账与资金安全


```
复式记账 (Double-Entry Bookkeeping)：
每笔交易至少涉及两个账户：一个借方(Debit)，一个贷方(Credit)
借贷金额必须相等，保证资金守恒

┌─────────────────────────────────────────────┐
│  用户A支付100元给商户B：                      │
│                                             │
│  借方                    贷方                │
│  用户A的账户 -100       商户B的账户 +100    │
│                                             │
│  记账分录：                                   │
│  ┌─────┬──────────┬────────┬────────┐      │
│  │序号 │ 账户      │ 借方   │ 贷方   │      │
│  ├─────┼──────────┼────────┼────────┤      │
│  │ 1   │ 用户A余额 │ 100.00 │        │      │
│  │ 2   │ 商户B余额 │        │ 100.00 │      │
│  └─────┴──────────┴────────┴────────┘      │
│  合计：               100.00  100.00        │
│  验证：借方合计 == 贷方合计                  │
│                                             │
│  如果有手续费：                               │
│  │ 1   │ 用户A余额 │ 100.00 │        │      │
│  │ 2   │ 商户B余额 │        │  99.50 │      │
│  │ 3   │ 平台收入  │        │   0.50 │      │
│  合计：               100.00  100.00        │
└─────────────────────────────────────────────┘

账务表设计：
CREATE TABLE account_ledger (
    id BIGINT PRIMARY KEY,
    transaction_no VARCHAR(64),    -- 交易流水号
    account_id BIGINT,             -- 账户ID
    direction TINYINT,             -- 1=借方(扣减), 2=贷方(增加)
    amount DECIMAL(10,2),          -- 金额
    balance_before DECIMAL(10,2),  -- 交易前余额
    balance_after DECIMAL(10,2),   -- 交易后余额
    created_at BIGINT,
    INDEX idx_txn (transaction_no),
    INDEX idx_account (account_id, created_at)
);

记账原子性：
-- 在一个事务中完成记账
BEGIN;
-- 检查余额充足
SELECT balance FROM account WHERE id = ? FOR UPDATE;
-- 扣减用户余额
UPDATE account SET balance = balance - 100 WHERE id = ? AND balance >= 100;
-- 增加商户余额
UPDATE account SET balance = balance + 99.50 WHERE merchant_id = ?;
-- 插入记账记录
INSERT INTO account_ledger (...) VALUES (...);
INSERT INTO account_ledger (...) VALUES (...);
COMMIT;
-- 验证：每笔交易的借方金额之和 == 贷方金额之和

对账 (Reconciliation)：
┌─────────────────────────────────────────────┐
│  日终对账流程：                               │
│  1. 下载渠道对账文件（微信/支付宝/银联）     │
│  2. 与本地交易记录逐笔比对                   │
│  3. 处理差异：                                │
│     - 长款：渠道有，本地无 → 补记账          │
│     - 短款：本地有，渠道无 → 人工核实        │
│     - 金额不一致 → 人工介入                  │
│  4. 生成对账报告                              │
└─────────────────────────────────────────────┘
```


## 风控与异常处理


```
风控规则引擎：
┌─────────────────────────────────────────────┐
│  事前风控（支付前）：                         │
│  - 交易限额：单笔/日累计/月累计             │
│  - 黑名单：商户/用户/IP/设备                 │
│  - 异常行为：短时间高频支付                  │
│  - 地理位置：异地登录+大额支付               │
│                                             │
│  事中风控（支付中）：                         │
│  - 实时规则匹配                              │
│  - 机器学习模型评分                          │
│  - 人工审核队列                              │
│                                             │
│  事后风控（支付后）：                         │
│  - 批量分析异常交易                          │
│  - 商户巡检                                  │
│  - 投诉处理                                  │
│                                             │
│  规则引擎示例：                               │
│  规则1：IF 单笔金额 > 50000 THEN 需要人工审核│
│  规则2：IF 1小时内交易次数 > 20 THEN 冻结账户│
│  规则3：IF IP属地 != 注册地 AND 金额 > 10000│
│         THEN 需要额外验证                    │
└─────────────────────────────────────────────┘

退款处理：
┌─────────────────────────────────────────────┐
│  退款流程：                                   │
│  1. 商户发起退款请求                         │
│  2. 校验：退款金额 ≤ 原订单金额             │
│  3. 冻结商户资金（防止退款时余额不足）       │
│  4. 调用渠道退款接口                         │
│  5. 等待渠道退款结果通知                     │
│  6. 更新订单和账务状态                       │
│                                             │
│  退款幂等性：                                │
│  - 同一订单多次退款请求，只退一次           │
│  - 用退款流水号做幂等键                     │
│                                             │
│  退款超时处理：                              │
│  - 渠道退款可能有延迟（几小时到几天）       │
│  - 超时后查询渠道退款状态                   │
│  - 状态同步到本地                           │
└─────────────────────────────────────────────┘

Exactly-Once保证：
┌─────────────────────────────────────────────┐
│  支付系统的Exactly-Once通过以下组合实现：     │
│                                             │
│  1. 幂等性：同一请求处理多次结果相同         │
│  2. 状态机：明确的状态流转，防止非法状态变更 │
│  3. 对账：发现不一致并修正                   │
│  4. 重试 + 超时处理：网络异常时安全重试     │
│  5. 人工介入：自动化无法处理的情况人工兜底   │
│                                             │
│  核心原则：                                   │
│  - 宁可不付，不可多付                       │
│  - 所有操作可追溯、可审计                   │
│  - 资金流向必须闭环                         │
└─────────────────────────────────────────────┘
```


> **Note:** 支付系统的灵魂是资金安全。复式记账保证借贷平衡，幂等性防止重复扣款，日终对账发现差异。宁可不付不可多付是核心原则。
