设计分布式锁

一、分布式锁基本要求

特性说明
互斥性任意时刻只有一个客户端持有锁
防死锁持有锁的客户端崩溃后，锁能被自动释放（超时机制）
可重入同一客户端可多次获取同一把锁
高可用锁服务不能成为单点故障
加锁/解锁原子性操作不能被中断

二、Redis 分布式锁

1. 基础实现（单节点）

-- 加锁：SET key value NX EX seconds
SET lock:order:1001 "uuid:thread-1" NX EX 30
-- NX: 不存在才设置（互斥）
-- EX 30: 30秒过期（防死锁）

-- 解锁：Lua 脚本保证原子性（先检查再删除）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
为什么解锁要用 Lua？ 如果用 GET + DEL 两步操作，在 GET 之后锁可能已过期被其他客户端获取，此时 DEL 会误删别人的锁。Lua 脚本保证检查和删除的原子性。

2. 锁续期（Watch Dog）

// 问题：业务执行时间超过锁的 TTL，锁自动过期，其他客户端获取到锁
// 解决：后台线程定期续期

// Redisson 实现
RLock lock = redisson.getLock("orderLock");
// 默认 30s 过期，watchdog 每 10s 续期到 30s
lock.lock();  // 不指定 leaseTime 时自动续期
try {
    // 业务逻辑...
} finally {
    lock.unlock();
}

3. Redlock 算法

解决单节点 Redis 的单点故障问题。

前提：N 个独立的 Redis Master 节点（N=5 推荐）

加锁流程：
1. 记录当前时间 T1
2. 依次向 N 个节点请求加锁（使用相同的 key 和 value，设置超时）
3. 当 >= (N/2 + 1) 个节点加锁成功（如 5 个中 3 个），且总耗时 < 锁的有效时间
   → 加锁成功
4. 锁的有效时间 = TTL - 加锁耗时
5. 如果加锁失败，向所有节点发起解锁

┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐   ┌──────┐
│Redis1│   │Redis2│   │Redis3│   │Redis4│   │Redis5│
│  ✓   │   │  ✓   │   │  ✓   │   │  ✗   │   │  ✗   │
└──────┘   └──────┘   └──────┘   └──────┘   └──────┘
3/5 成功 → 加锁成功（多数派）
Redlock 争议：Martin Kleppmann（DDIA 作者）质疑 Redlock 在时钟漂移和 GC暂停场景下仍有安全问题。建议使用 Fencing Token 进一步保证安全性。

三、Fencing Token（栅栏令牌）

问题场景（锁过期后的写冲突）：
1. 客户端A 获取锁（TTL=10s）
2. A 发生长时间 GC 停顿（15s）
3. 锁过期，客户端B 获取到锁
4. A 恢复，仍然认为自己持有锁，执行写操作
5. B 也在执行写操作 → 数据不一致！

Fencing Token 解决方案：
加锁成功时返回一个单调递增的 token
写入存储时携带 token，存储拒绝旧 token 的写入

时间线：
A 获取锁 token=33 → A GC 停顿 → B 获取锁 token=34
→ A 恢复写入 token=33 → 存储拒绝（33 < 34）
→ B 写入 token=34 → 存储接受
// 存储层校验
if (request.token > last_seen_token) {
    last_seen_token = request.token;
    execute_write();
} else {
    reject("stale token");
}

四、ZooKeeper 分布式锁

1. 临时顺序节点方案

/locks/
  ├── order-1001-lock-0000000001  ← 客户端A
  ├── order-1001-lock-0000000002  ← 客户端B
  └── order-1001-lock-0000000003  ← 客户端C

加锁流程：
1. 创建临时顺序节点 /locks/order-1001-lock-0000000002
2. 获取 /locks 下所有子节点，排序
3. 如果自己是最小节点 → 获取锁成功
4. 否则 watch 前一个节点（-0000000001）
5. 前一个节点被删除时，收到通知，检查自己是否成为最小节点

解锁：删除自己的临时节点，或客户端断开自动删除

2. ZK 锁 vs Redis 锁

特性Redis 锁ZooKeeper 锁
实现复杂度简单中等
性能极高中等
可靠性单节点较低，Redlock 中等高（ZAB 协议保证一致性）
锁等待通知需要轮询Watch 机制，事件驱动
可重入需要额外实现天然支持（同 session）
适用场景高性能、短暂锁高可靠、长时间锁

五、Redisson 分布式锁实战

// Redisson 配置
Config config = new Config();
config.useClusterServers()
    .addNodeAddress("redis://127.0.0.1:7000")
    .addNodeAddress("redis://127.0.0.1:7001");
RedissonClient redisson = Redisson.create(config);

// 可重入锁
RLock lock = redisson.getLock("myLock");
lock.lock(30, TimeUnit.SECONDS);  // 指定 TTL
try {
    // 业务逻辑
} finally {
    lock.unlock();
}

// 读写锁（ReadWriteLock）
RReadWriteLock rwLock = redisson.getReadWriteLock("myRWLock");
rwLock.readLock().lock();    // 读锁共享
rwLock.writeLock().lock();   // 写锁排他

// 红锁（Redlock）
RedissonRedLock redLock = new RedissonRedLock(lock1, lock2, lock3);
redLock.lock();

六、方案选型建议

性能优先（允许极小概率不一致） → Redis 单节点 + Lua 解锁
高可靠（金融交易等）           → ZooKeeper 或 etcd
平衡方案                       → Redis + Redlock + Fencing Token
已有 Redis 基础设施            → Redisson（功能完善）
面试要点：Redis 分布式锁的三大问题：1）锁过期但业务未完成（用 watchdog 续期）；2）误删别人的锁（Lua 脚本原子检查+删除）；3）单点故障（用 Redlock 多数派）。Fencing Token 是解决 GC 暂停导致锁过期后并发写入的终极方案。ZK 天然可靠但性能不如 Redis。
