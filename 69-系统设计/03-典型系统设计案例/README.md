# 03 - 典型系统设计案例

通过具体的设计案例来训练系统设计思维，每个案例遵循统一的设计方法论，覆盖需求分析、架构设计、关键技术选型和权衡决策。

---

## 1. 系统设计方法论

系统设计没有标准答案，但有一个通用的思考框架：

### 1.1 四阶段法

1. **需求澄清**：明确功能需求和非功能需求（QPS、延迟、一致性要求）
2. **高层设计**：画出系统架构图，定义核心组件和数据流向
3. **详细设计**：深入每个组件的技术选型和实现细节
4. **权衡分析**：讨论每种方案的优缺点和取舍理由

### 1.2 需求分析模板

- **功能需求**：系统需要做什么？列出核心功能和次要功能
- **非功能需求**：性能（QPS/延迟）、可用性（SLA）、一致性、可扩展性
- **容量估算**：日活用户、峰值QPS、存储量、带宽需求
- **约束条件**：预算、团队技术栈、已有基础设施

### 1.3 估算技巧

- 1天 ≈ 10万秒
- 1个字符 ≈ 1字节
- 每个用户每天产生N次操作
- 峰值QPS ≈ 平均QPS × 2~5倍

## 2. 案例1：设计短链接服务（TinyURL）

### 2.1 需求分析

**功能需求**：输入长URL生成短链接，访问短链接跳转到原始URL。
**非功能需求**：读写比100:1（读远大于写），延迟<100ms，SLA 99.99%。
**容量估算**：每天新增100万条短链接，1亿次访问。

### 2.2 哈希方案

**Base62编码**：使用a-z、A-Z、0-9共62个字符生成短码。7位Base62可表示约3.5万亿种组合。

**生成策略**：
- **哈希截取**：对长URL进行MD5/SHA256哈希，取前7位Base62编码。需处理冲突（数据库唯一索引+重试）。
- **自增ID**：全局自增ID转Base62编码。简单但暴露业务规模，可用预分配ID段+随机化解决。
- **预生成Key服务**：预先生成一批短码放入队列，使用时直接取用，避免实时生成的性能开销。

### 2.3 存储设计

**数据模型**：`short_code -> (original_url, created_at, expire_at, user_id)`

**存储选型**：
- **关系型数据库（MySQL）**：支持复杂查询，适合中小规模
- **NoSQL（DynamoDB/Cassandra）**：键值对读写性能极高，适合大规模
- **缓存（Redis）**：热点数据缓存，降低数据库压力

### 2.4 缓存策略

- 使用Redis缓存热门短链接，LRU淘汰策略
- 缓存命中率可达到80%以上（热门链接访问集中）
- 数据库承载20%的访问量

### 2.5 高可用

- 数据库主从复制+自动故障转移
- 应用层无状态，多实例部署+负载均衡
- Redis集群模式，避免缓存单点故障
- 301重定向（永久跳转，浏览器缓存）vs 302重定向（临时跳转，便于统计）

## 3. 案例2：设计消息推送系统

### 3.1 需求分析

**功能需求**：一对一聊天、群聊、消息送达确认、离线消息。
**非功能需求**：消息延迟<200ms，支持亿级用户在线。

### 3.2 长连接管理

**WebSocket**：全双工通信，适合实时消息推送。服务器维护每个客户端的长连接。

**连接层架构**：
- 客户端通过负载均衡连接到一台接入服务器
- 每台接入服务器维护数十万条长连接
- 用户ID与接入服务器的映射关系存储在Redis中

**连接保活**：心跳机制（每30秒发送Ping），超时未收到Pong则断开并清理连接映射。

### 3.3 消息路由

**一对一消息**：查询接收方所在的接入服务器 → 将消息投递到该服务器 → 推送给接收方。

**群消息路由**：
- **推模式**：每发一条群消息，立即推送给所有群成员。适合小群（<500人）
- **拉模式**：群成员主动拉取新消息。适合大群（>500人）
- **推拉结合**：小群推送，大群拉取

### 3.4 离线消息

用户不在线时，消息持久化到离线消息存储（如HBase）。用户上线后拉取离线消息，按时间戳排序后逐条推送给客户端。

**漫游消息**：用户在新设备登录时，拉取最近N天的消息，实现多端消息同步。

### 3.5 群发扇出

大群消息的扇出优化：
- **写扩散**：每条消息写入每个成员的收件箱。简单但写放大严重。
- **读扩散**：消息只写一份到群消息流中，成员拉取时按需读取。写入简单但读取复杂。
- **混合模式**：小群写扩散，大群读扩散。

## 4. 案例3：设计分布式限流器

### 4.1 需求分析

**功能需求**：按用户/API/服务维度限流，支持多种限流规则。
**非功能需求**：限流判断延迟<1ms，支持分布式环境，高可用。

### 4.2 计数器方案

**固定窗口计数器**：每个时间窗口维护一个计数器，请求到达时计数器+1，超过阈值则拒绝。实现简单但存在窗口边界突刺问题。

**滑动窗口**：将固定窗口划分为更小的子窗口，滑动计算当前窗口内的总请求数。比固定窗口更精确。

**滑动窗口日志**：记录每个请求的时间戳，统计窗口内的请求数。精确但内存开销大，适合限流阈值较小的场景。

### 4.3 分布式一致性

**Redis实现**：使用Redis的INCR命令实现原子计数，EXPIRE设置窗口过期时间。

**关键问题**：
- **原子性**：Redis单命令天然原子，多命令需要Lua脚本保证原子性
- **时间同步**：各节点时间可能存在偏差，使用Redis服务器时间
- **故障恢复**：Redis宕机时的降级策略（本地限流或放行）
- **集群同步**：Redis集群中限流数据的分布和一致性

**算法选择**：
- **令牌桶**：适合需要平滑流量且允许一定突发的场景
- **漏桶**：适合需要严格控制输出速率的场景
- **滑动窗口计数器**：适合精确控制时间窗口内请求数的场景

## 5. 案例4：设计新闻Feed流系统

### 5.1 需求分析

**功能需求**：用户发布动态，关注其他用户后在Feed流中看到其动态。
**非功能需求**：Feed流生成延迟<1秒，支持亿级用户。

### 5.2 推模式 vs 拉模式

**推模式（写扩散）**：用户发布动态时，立即将动态写入所有粉丝的收件箱（Timeline）。粉丝读取Feed时直接查询收件箱即可。

- **优点**：读取Feed速度极快
- **缺点**：大V发布动态时需要写入海量收件箱（假设有1000万粉丝），延迟高

**拉模式（读扩散）**：用户发布动态时只写入自己的发布列表。粉丝读取Feed时，实时拉取所有关注用户的最新动态并按时间排序合并。

- **优点**：写入简单高效
- **缺点**：读取Feed时需要合并大量数据，延迟高

**推拉结合**：普通用户用推模式，大V（粉丝数超过阈值）用拉模式。读取Feed时，从收件箱获取推模式的数据，实时拉取大V的最新动态，合并后排序返回。

### 5.3 时间线生成

Feed流排序策略：
- **时间线排序**：按发布时间倒序排列，简单直观
- **兴趣排序**：基于用户画像和内容特征进行个性化排序
- **混合排序**：时间线+推荐，先按时间筛选，再按兴趣重排

### 5.4 冷启动

新用户没有关注任何人时的Feed内容：
- 基于注册时选择的兴趣标签推荐内容
- 推荐热门内容（全局热度排序）
- 基于社交图谱推荐（"你可能认识的人"）

## 6. 案例5：设计分布式文件存储

### 6.1 需求分析

**功能需求**：文件上传、下载、删除，支持大文件分片上传。
**非功能需求**：数据持久性99.999999999%（11个9），支持PB级存储。

### 6.2 元数据管理

元数据服务管理文件属性（名称、大小、创建时间）与物理存储位置的映射关系。

**存储方案**：
- 关系型数据库：适合中小规模
- 分布式KV存储（etcd/ZooKeeper）：适合大规模元数据

**元数据与数据分离**：元数据服务独立于数据存储层，各自独立扩展。

### 6.3 数据分片

**分片策略**：
- 将大文件切分为固定大小的数据块（如64MB/块）
- 每个数据块独立存储和复制
- 数据块使用哈希值作为唯一标识（内容寻址）

### 6.4 副本放置

每个数据块存储3个副本，放置策略：
- **跨机架**：同一数据块的副本分布在不同机架，避免机架级故障
- **跨数据中心**：重要数据跨数据中心存储，抵御数据中心级故障
- **放置策略**：综合考虑磁盘空间利用率、网络带宽、故障域

**副本一致性**：强一致性（所有副本写入成功才返回）或最终一致性（主副本写入成功即返回，异步同步从副本）。

**数据修复**：后台持续检查副本数量和完整性，发现丢失或损坏时自动从其他副本恢复，通过CRC校验保证数据完整性。

## 7. 案例6：设计搜索引擎

### 7.1 系统架构

搜索引擎由三大子系统组成：爬虫系统、索引系统、查询系统。

### 7.2 爬虫系统

从互联网自动抓取网页内容，核心组件包括：URL Frontier（待抓取URL优先级队列）、布隆过滤器URL去重、DNS缓存、Politeness策略（对同一域名限速避免被封禁）、网页解析（提取文本和链接）。

### 7.3 索引构建

**倒排索引（Inverted Index）**：核心数据结构，记录每个词出现在哪些文档中。

**构建过程**：文本预处理（分词、去停用词、词干化）→ 建立词项到文档的映射 `term → [(doc_id, position, frequency), ...]` → 索引压缩 → 索引分片。

### 7.4 查询处理与排序

1. 查询解析：分词、纠错、同义词扩展
2. 索引查询：查找包含查询词的文档集合
3. 评分排序：使用BM25/TF-IDF或Learning to Rank模型计算相关性得分
4. 结果聚合：合并各分片的查询结果，排序后返回

## 8. 系统设计面试技巧

### 8.1 面试流程

1. **开场（2-3分钟）**：澄清需求，确认功能范围和非功能指标
2. **高层设计（5-10分钟）**：画出核心架构图，定义主要组件和接口
3. **深入讨论（15-20分钟）**：选择1-2个关键组件深入讨论技术细节
4. **收尾（2-3分钟）**：总结方案，讨论扩展性和潜在风险

### 8.2 常见错误与得分关键

**常见错误**：过早深入细节忽略整体架构、只讨论一种方案不进行对比分析、忽略非功能需求、低估容量估算、没有主动引导讨论方向。

**得分关键**：结构化思维（按步骤推进逻辑清晰）、权衡意识（每个决策都有利弊分析）、沟通能力（主动确认假设和需求）、深度与广度平衡、用数据说话（QPS、延迟、存储量）。
