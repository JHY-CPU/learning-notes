# 设计秒杀系统


## 设计秒杀系统


系统设计秒杀高并发


秒杀系统需要在极短时间内处理海量并发请求，核心是库存扣减和流量控制。


## 需求分析


```
业务场景：
- 特定时间点（如10:00:00）开放限量商品抢购
- 商品数量有限（如100件），价格极低
- 预估有10万+用户同时参与抢购
- 要保证公平性：先到先得，不超卖

核心指标：
- 瞬时QPS：10万+/s
- 库存：100件
- 持续时间：几秒内售罄
- 体验要求：不能超卖，不能卡死

核心挑战：
┌─────────────────────────────────────────────┐
│  1. 瞬时流量峰值极高（是平时的100-1000倍）    │
│  2. 库存扣减需要精确（不能超卖也不能少卖）    │
│  3. 重复抢购需要防重                         │
│  4. 系统不能因为秒杀影响其他正常业务          │
└─────────────────────────────────────────────┘
```


## 整体架构设计


```
架构分层（从前到后逐层削减流量）：
┌─────────────────────────────────────────────┐
│  浏览器/客户端                               │
│  ├── 静态化：秒杀页面部署到CDN               │
│  ├── 按钮置灰：未到时间不能点击              │
│  ├── 前端限流：点击后N秒内不能重复点击       │
│  └── 验证码：图形/滑块验证码分散请求         │
│      ↓                                       │
│  CDN / Nginx                                 │
│  ├── 静态资源CDN缓存                         │
│  ├── Nginx限流（按IP/用户）                  │
│  └── 请求过滤（User-Agent、Referer校验）     │
│      ↓                                       │
│  API Gateway                                 │
│  ├── 用户身份校验（提前登录）                │
│  ├── 活动状态检查（未开始/已结束）           │
│  └── 全局限流                               │
│      ↓                                       │
│  秒杀服务（无状态，可水平扩展）               │
│  ├── 本地标记判断（售罄标记）                │
│  ├── Redis预扣库存                           │
│  └── 请求入队（MQ）                          │
│      ↓                                       │
│  消息队列 (RocketMQ/Kafka)                   │
│  ├── 流量削峰                                │
│  └── 异步下单                                │
│      ↓                                       │
│  订单服务                                     │
│  ├── 数据库扣库存（乐观锁防超卖）            │
│  ├── 创建订单                                │
│  └── 超时未支付自动取消释放库存              │
└─────────────────────────────────────────────┘

流量削减效果：
100万请求 → CDN/浏览器拦截80% → 20万
20万 → Nginx限流 → 5万
5万 → Gateway校验 → 3万
3万 → Redis预扣（只放100个）→ 100个真正下单
```


## 库存预扣减方案


```
方案1：Redis原子扣减（推荐）
┌─────────────────────────────────────────────┐
│  // Lua脚本保证原子性                        │
│  local stock = tonumber(redis.call('GET', key))│
│  if stock <= 0 then                          │
│    return -1  // 库存不足                    │
│  end                                         │
│  redis.call('DECR', key)                     │
│  return 1  // 扣减成功                       │
│                                             │
│  // Java代码                                 │
│  Long result = redisTemplate.execute(        │
│    new DefaultRedisScript<>(lua, Long.class),│
│    List.of("stock:seckill:1"),               │
│    userId                                     │
│  );                                           │
│  if (result == -1) {                         │
│    return "已售罄";                          │
│  }                                            │
│  // 入队，异步创建订单                        │
│  mqProducer.send("seckill-order", orderMsg);  │
└─────────────────────────────────────────────┘

方案2：Redis Decr + 判断
// 问题：先DECR再判断可能导致库存变为负数
Long stock = redisTemplate.opsForValue().decrement("stock:1");
if (stock < 0) {
    redisTemplate.opsForValue().increment("stock:1"); // 回补
    return "已售罄";
}
// 改进：使用Lua脚本先判断再扣减

方案3：分段库存
// 将100件库存分为10段，每段10件
// 10台服务器各持有10件库存
// 每台服务器本地扣减（无网络开销）
// 本地售罄则直接返回
int localStock = localDecrement(); // 本地AtomicInteger
if (localStock < 0) return "售罄";
// 本地还有库存，再走Redis扣减

方案4：预扣 + 回补
// 预扣：先在Redis中标记占用
// 支付成功：确认扣减
// 支付超时：回补库存（延迟队列30分钟）
SET stock:reserved:123 userId EX 1800
// 30分钟后过期，自动释放
```


## 防超卖与防重


```
防超卖（数据库层）：
// 乐观锁扣库存
UPDATE products
SET stock = stock - 1
WHERE id = ? AND stock > 0
// stock > 0 条件保证不会扣成负数
// 返回影响行数=1表示成功，=0表示库存不足

// 或使用版本号
UPDATE products
SET stock = stock - 1, version = version + 1
WHERE id = ? AND version = ?

// CAS + 重试
for (int i = 0; i < 3; i++) {
    Product p = selectProduct(id);
    if (p.stock <= 0) return false;
    int rows = updateStock(id, p.stock - 1, p.version);
    if (rows == 1) return true; // 成功
}
return false; // 重试失败

防重复抢购：
// 方案1：数据库唯一索引
CREATE TABLE seckill_order (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    UNIQUE KEY uk_user_product (user_id, product_id)
);
// 同一用户同一商品只能有一条记录

// 方案2：Redis Set
Boolean added = redisTemplate.opsForValue()
    .setIfAbsent("seckill:user:" + userId + ":product:" + productId, "1", 1, TimeUnit.HOURS);
if (!added) return "您已经抢购过了";

// 方案3：先查Redis再写
Boolean member = redisTemplate.opsForSet()
    .isMember("seckill:users:1", userId.toString());
if (member) return "已抢购";
redisTemplate.opsForSet().add("seckill:users:1", userId.toString());

售罄标记：
// Redis售罄后设置标记
redisTemplate.opsForValue().set("seckill:soldout:1", "1", 24, TimeUnit.HOURS);

// 本地售罄标记（更快）
private static final Map<Long, AtomicBoolean> SOLD_OUT = new ConcurrentHashMap<>();

public String seckill(Long productId, Long userId) {

        return "已售罄";
```

## 异步下单与排队


```
异步下单流程：
┌─────────────────────────────────────────────┐
│  用户点击秒杀                                │
│    ↓                                        │
│  Redis预扣库存成功                           │
│    ↓                                        │
│  发送消息到MQ（包含userId, productId）       │
│    ↓                                        │
│  返回"排队中"给用户                          │
│                                             │
│  MQ消费者异步处理：                          │
│    ↓                                        │
│  检查是否重复下单                            │
│    ↓                                        │
│  数据库扣库存（最终确认）                    │
│    ↓                                        │
│  创建订单                                    │
│    ↓                                        │
│  通知前端"抢购成功"（WebSocket/轮询）        │
│                                             │
│  30分钟未支付：                              │
│    ↓                                        │
│  取消订单 → 回补库存 → 清除用户抢购记录      │
└─────────────────────────────────────────────┘

前端轮询查询结果：
// 前端提交秒杀请求后，得到一个秒杀订单号
// 然后轮询查询结果
function pollResult(orderNo) {
    fetch('/api/seckill/result?orderNo=' + orderNo)
        .then(res => res.json())
        .then(data => {
            if (data.status === 'SUCCESS') {
                showSuccess(data.order);
            } else if (data.status === 'PENDING') {
                setTimeout(() => pollResult(orderNo), 500);
            } else {
                showFail(data.message);
            }
        });
}

// 或使用WebSocket实时推送
@OnMessage
public void onMessage(String message) {
    // 服务端推送秒杀结果
    if ("SECKILL_SUCCESS".equals(message.type)) {
        showOrder(message.order);
    }
}

库存回补（延迟队列）：
// 订单创建后，30分钟未支付则取消
// 使用RocketMQ延迟消息
Message msg = new Message("order-timeout", JSON.toJSONBytes(order));
msg.setDelayTimeLevel(16); // 30分钟
producer.send(msg);

// 消费者处理超时订单
@RocketMQMessageListener(topic = "order-timeout")
public void onTimeout(Order order) {
    if (order.getStatus() == OrderStatus.UNPAID) {
        // 1. 取消订单
        orderService.cancel(order.getId());
        // 2. 回补库存
        redisTemplate.opsForValue().increment("stock:seckill:" + order.getProductId());
        // 3. 清除用户抢购记录
        redisTemplate.opsForSet().remove("seckill:users:" + order.getProductId(), order.getUserId());
    }
}
```


> **Note:** 秒杀系统的核心是分层削减流量（CDN→Nginx→网关→Redis→MQ→DB），Redis原子扣减预库存，MQ异步下单削峰，数据库乐观锁防超卖。售罄标记快速拒绝后续请求。
