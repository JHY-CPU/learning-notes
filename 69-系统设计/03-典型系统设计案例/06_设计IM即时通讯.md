# 设计IM即时通讯


## 设计IM即时通讯


系统设计IMWebSocket


即时通讯系统需要支持实时消息收发、群聊、离线推送等功能，核心是长连接管理和消息投递。


## 需求分析与容量估算


```
功能需求：
- 单聊：一对一消息
- 群聊：多人群组消息
- 消息类型：文本、图片、文件、语音、视频
- 消息状态：已发送、已送达、已读
- 离线消息：用户不在线时暂存，上线后推送
- 消息漫游：多端同步历史消息

非功能需求：
- 消息延迟：<200ms（端到端）
- 在线用户：1000万
- 日活用户：5000万
- 消息量：日均100亿条
- 消息可靠性：不丢失、不重复

容量估算：
- 消息QPS：100亿 / 86400 ≈ 12万/s（平均），峰值50万/s
- 消息存储：每条100B，100亿 × 100B ≈ 1TB/天
- 长连接数：1000万在线用户
- 每台服务器维持5万长连接 → 需要200台连接服务器
```

## 消息投递模型


```
单聊消息投递：
┌─────────────────────────────────────────────┐
│  A发送消息给B：                              │
│  A → Gateway1 → 消息服务                     │
│       1. 生成消息ID（Snowflake）             │
│       2. 存储消息到DB                        │
│       3. 查询B的网关映射（Redis）            │
│       4. 如果B在线 → Gateway2 → B           │
│       5. 如果B离线 → 存离线消息 → APNs推送  │
│       6. 返回ACK给A（已存储）                │
│                                             │
│  消息ID的作用：                               │
│  - 全局唯一，用于去重                        │
│  - 递增，用于排序                            │
│  - 客户端用ID确认哪些消息已同步              │
└─────────────────────────────────────────────┘

写扩散 vs 读扩散：

写扩散（推模式）：
┌─────────────────────────────────────────────┐
│  群消息发送时，写入每个群成员的收件箱        │
│                                             │
│  A在群G发送消息                              │
│    → 写入群成员各自的收件箱：                │
│      inbox:B → [msg1]                       │
│      inbox:C → [msg1]                       │
│      inbox:D → [msg1]                       │
│      inbox:E → [msg1]                       │
│                                             │
│  优点：读取简单，直接从收件箱拉取            │
│  缺点：大群写放大严重（500人群×500次写入）   │
│  适合：小群（<100人）                        │
└─────────────────────────────────────────────┘

读扩散（拉模式）：
┌─────────────────────────────────────────────┐
│  群消息只写一份，读取时去各群拉取            │
│                                             │
│  A在群G发送消息                              │
│    → 只写入群的消息列表：                    │
│      group:G → [msg1, msg2, ...]            │
│                                             │
│  B上线后：                                   │
│    → 遍历所有群，拉取未读消息                │
│    → group:G拉取最新消息                     │
│    → group:H拉取最新消息                     │
│                                             │
│  优点：写入量小                              │
│  缺点：读取复杂，需要遍历所有群              │
│  适合：大群，用户加入的群多但大部分不活跃    │
└─────────────────────────────────────────────┘

混合模式（推荐）：
┌─────────────────────────────────────────────┐
│  小群（<100人）：写扩散                      │
│  大群（≥100人）：读扩散                      │
│  普通用户加入的群数量有限（通常<100个）      │
│  遍历100个群拉取消息是可以接受的             │
└─────────────────────────────────────────────┘

消息投递确认：
┌─────────────────────────────────────────────┐
│  发送方A        服务端          接收方B      │
│    │──msg──→     │              │            │
│    │             │──存储成功──→ │            │
│    │←──ACK──     │              │            │
│    │             │    │──推送──→│            │
│    │             │    │         │──ACK──→    │
│    │←──已送达──  │←───┘         │            │
│    │             │              │            │
│    │             │   B打开对话   │            │
│    │             │←──已读回执──→│            │
│    │←──已读──    │              │            │
└─────────────────────────────────────────────┘
```


## 消息存储与检索


```
消息表设计：
┌─────────────────────────────────────────────┐
│  单聊消息表（按会话分库分表）：               │
│  CREATE TABLE im_message_{shard} (          │
│    msg_id BIGINT PRIMARY KEY,               │
│    conversation_id VARCHAR(64),  // 会话ID  │
│    sender_id BIGINT,                        │
│    receiver_id BIGINT,                      │
│    msg_type TINYINT,  // 文本/图片/文件     │
│    content TEXT,                             │
│    status TINYINT,  // 发送中/已送达/已读    │
│    created_at BIGINT,                       │
│    INDEX idx_conv_time (conversation_id, created_at)│
│  );                                          │
│                                             │
│  分库分表策略：                               │
│  - 按 conversation_id 哈希分片              │
│  - 或按时间分表（每月一张表）                │
│  - 冷数据归档到HBase/TiDB                   │
└─────────────────────────────────────────────┘

消息漫游（多端同步）：
┌─────────────────────────────────────────────┐
│  用户B在手机端聊了一段时间后，打开电脑端      │
│  电脑端需要同步手机端的消息                   │
│                                             │
│  同步方案：                                   │
│  1. 客户端记录最后同步的消息ID               │
│  2. 上线时请求服务端：                        │
│     GET /sync?conversation_id=xxx&last_msg_id=12345│
│  3. 服务端返回 last_msg_id 之后的新消息      │
│  4. 客户端追加到本地DB                       │
│                                             │
│  优化：                                      │
│  - 只同步有更新的会话（会话列表对比版本号）  │
│  - 增量同步，不是全量拉取                    │
│  - 图片/文件只传URL，按需下载               │
└─────────────────────────────────────────────┘

消息搜索：
- 消息量大，不适合关系型数据库全文搜索
- 方案1：Elasticsearch索引消息内容
  - 只索引文本消息，不索引图片/文件
  - 按会话/时间范围过滤
- 方案2：客户端本地搜索
  - 消息已同步到本地SQLite
  - 本地搜索速度更快
```


## 长连接与离线推送


```
WebSocket连接管理：
┌─────────────────────────────────────────────┐
│  连接建立：                                   │
│  1. 客户端发起WebSocket握手                  │
│  2. 网关验证Token                            │
│  3. 记录映射：userId → (gatewayId, connId)   │
│     存入Redis：HSET user_conn userId gw1:conn1│
│                                             │
│  心跳保活：                                   │
│  客户端每30秒发送PING                        │
│  服务端回复PONG                              │
│  超过60秒未收到PING → 关闭连接               │
│                                             │
│  断线重连：                                   │
│  1. 客户端检测到连接断开                     │
│  2. 指数退避重连（1s, 2s, 4s, 8s...）       │
│  3. 重连成功后同步离线消息                   │
│                                             │
│  多端登录：                                   │
│  - 同一用户多设备同时在线                    │
│  - 消息投递到所有在线设备                    │
│  - 已读状态同步到所有设备                    │
└─────────────────────────────────────────────┘

网关路由查找：
// 发送消息时查找接收方所在网关
String receiverConn = redisTemplate.opsForHash()
    .get("user_conn", receiverId);
// 返回 "gw3:conn42"
// 消息服务 → gw3（通过内部RPC） → 推送给用户

离线消息处理：
┌─────────────────────────────────────────────┐
│  方案1：离线消息表                            │
│  用户离线时，消息写入offline_message表       │
│  用户上线时，拉取离线表中的消息              │
│  拉取后删除                                  │
│                                             │
│  方案2：基于消息ID差值                       │
│  记录用户最后收到的消息ID                    │
│  上线时拉取 > last_msg_id 的所有消息         │
│  不需要单独的离线表                          │
│                                             │
│  方案3：推拉结合                             │
│  在线消息：实时推送                          │
│  离线消息：APNs/FCM推送通知                  │
│  用户打开App后拉取完整消息                   │
└─────────────────────────────────────────────┘

消息幂等性：
┌─────────────────────────────────────────────┐
│  客户端生成一个唯一的消息ID（UUID）          │
│  服务端用消息ID去重                          │
│                                             │
│  Redis去重：                                 │
│  SETNX msg:idempotent:{msg_id} 1 EX 86400  │
│  设置成功 → 新消息 → 处理                   │
│  设置失败 → 重复消息 → 返回已有结果          │
│                                             │
│  数据库去重：                                │
│  msg_id设置为主键或唯一索引                  │
│  INSERT失败 → 重复消息                      │
└─────────────────────────────────────────────┘
```


> **Note:** IM系统的核心是长连接管理和消息投递。小群用写扩散、大群用读扩散。消息用Snowflake ID全局排序和去重，离线消息用推送+拉取结合保证送达。
