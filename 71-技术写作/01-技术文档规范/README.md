# 01-技术文档规范

## 1. 技术文档的类型

| 类型 | 目的 | 特点 | 示例 |
|------|------|------|------|
| 概念文档 | 解释"是什么"和"为什么" | 概念性、无代码要求 | 架构概述、设计原理 |
| 教程（Tutorial） | 引导用户完成任务 | 手把手、有明确结果 | 快速入门、构建第一个应用 |
| 参考文档 | 查阅具体信息 | 精确、完整、结构化 | API参考、配置参数 |
| 故障排除指南 | 解决问题 | 问题-原因-方案结构 | 常见错误、Debug指南 |

## 2. Diátaxis文档框架

Diátaxis由Daniele Procida提出，将文档按用户需求分为四个象限：

```
                学习                         做事
          ┌─────────────────┬─────────────────┐
   实践   │  教程            │  操作指南        │
          │  (Tutorials)     │  (How-to Guides) │
          ├─────────────────┼─────────────────┤
   理论   │  解释            │  参考            │
          │  (Explanation)   │  (Reference)     │
          └─────────────────┴─────────────────┘
```

- **教程（Tutorials）**：面向初学者的学习导向，提供安全的学习体验
- **操作指南（How-to Guides）**：面向有经验用户的目标导向
- **参考（Reference）**：面向查阅者的面向信息，描述系统是什么
- **解释（Explanation）**：面向学习者的理解导向，提供背景和讨论

**原则**：每种文档服务于不同的用户需求，不应混合不同类型。

## 3. 技术写作原则

### 3.1 清晰性与简洁性

- 使用简单直接的语言，避免不必要的术语
- 一个段落一个主题
- 使用主动语态："系统发送请求"而非"请求被系统发送"
- 避免冗余：删除"基本上"、"实际上"等填充词
- 使用短句（平均20词以内）

### 3.2 受众导向

- **初学者**：提供背景知识、使用类比、避免术语堆砌
- **中级用户**：提供具体步骤、关注常见陷阱
- **专家用户**：提供精确参考、支持快速查阅
- 在文档开头明确标注目标读者和前置知识

### 3.3 一致性

- 术语一致：同一概念使用同一术语
- 格式一致：标题层级、列表样式、代码格式统一
- 结构一致：同类文档遵循相同模板
- 建立术语表（Glossary）和写作风格指南（Style Guide）

### 3.4 可操作性

- 提供可复制执行的代码示例
- 步骤编号清晰，每个步骤只做一件事
- 标注预期输出和可能的错误
- 提供验证成功的方法

## 4. README编写规范

一个好的README应包含以下结构：

```markdown
# 项目名称
一句话描述

## 功能特性
- 核心功能1
- 核心功能2

## 快速开始
### 环境要求
### 安装步骤
### 基本用法

## 文档链接
## 贡献指南
## 许可证
## 联系方式
```

**最佳实践**：
- 第一段即说明项目是什么和解决什么问题
- 使用徽章（Badges）展示构建状态、版本、覆盖率
- 代码示例可直接复制运行
- 提供完整和最小化的使用示例
- 定期更新以反映项目最新状态

## 5. 变更日志（Changelog）规范

遵循 [Keep a Changelog](https://keepachangelog.com/) 规范：

```markdown
# Changelog

## [1.2.0] - 2024-03-15
### Added
- 新增批量导出功能

### Changed
- 优化查询性能，响应时间减少40%

### Deprecated
- 旧版API将于2.0版本移除

### Removed
- 移除已弃用的v1认证接口

### Fixed
- 修复并发写入导致的数据不一致问题

### Security
- 升级依赖修复CVE-2024-XXXX漏洞
```

**原则**：
- 为人类编写，非机器
- 每个版本一个条目
- 相同类型的变更分组
- 包含版本号和日期
- 链接到相关Issue或PR

## 6. 贡献指南（CONTRIBUTING.md）

标准CONTRIBUTING.md应包含：

1. **行为准则**（Code of Conduct）
2. **如何报告Bug**：Bug报告模板
3. **如何提交功能请求**
4. **开发环境搭建**：依赖安装、IDE配置
5. **代码规范**：Lint规则、格式化工具
6. **提交流程**：
   - Fork → Branch → Commit → PR
   - Commit Message规范（Conventional Commits）
   - PR模板和检查清单
7. **代码审查流程**
8. **版本发布流程**

## 7. 代码注释规范

### 通用原则

- 注释说明**为什么**这么做，而非**做什么**（代码本身应说明做什么）
- 避免无意义的注释：`i++ // i自增1`
- 保持注释与代码同步更新

### 常见规范

| 语言 | 文档注释格式 | 工具 |
|------|-------------|------|
| Java | Javadoc (`/** */`) | Javadoc |
| Python | Docstring (`"""`) | Sphinx、pydoc |
| JavaScript/TS | JSDoc (`/** */`) | JSDoc、TypeDoc |
| C/C++ | Doxygen (`/** */`) | Doxygen |
| Go | GoDoc (`//`) | godoc |
| Rust | Rustdoc (`///`) | rustdoc |

### 文档注释内容

- 函数/方法：功能描述、参数说明、返回值、异常、示例
- 类/模块：职责描述、使用场景、示例
- 复杂逻辑：算法说明、时间/空间复杂度、参考链接

## 8. 技术博客写作技巧

### 结构模板

1. **标题**：具体、包含关键词（"如何使用X实现Y"）
2. **引言**：问题背景、文章价值、阅读收益
3. **正文**：逻辑递进、代码与文字交替
4. **总结**：要点回顾、下一步行动
5. **参考**：相关链接和延伸阅读

### 写作技巧

- 以问题或场景开头，而非概念定义
- 代码示例应最小化且可运行
- 使用图表辅助解释复杂概念
- 控制文章长度（1500-3000字为宜）
- 技术准确性优先，发布前同行审核

## 9. 文档版本管理

- 文档与代码同仓库管理（Docs as Code）
- 使用Git分支管理不同版本文档
- 版本化策略：
  - 文档随代码版本发布（推荐）
  - 独立文档版本管理
- 工具支持：
  - Docusaurus：内置版本管理
  - GitBook：版本化文档空间
  - ReadTheDocs：多版本构建

## 10. 文档工具

| 工具 | 特点 | 适用场景 | 语言/框架 |
|------|------|----------|-----------|
| MkDocs | 简洁、Markdown驱动 | 中小型项目文档 | Python（Material主题流行） |
| Docusaurus | React、版本化、i18n | 大型开源项目文档 | React/MDX |
| VuePress | Vue驱动、简洁 | Vue生态项目文档 | Vue |
| GitBook | 托管服务、协作友好 | 商业产品文档 | 云端SaaS |
| Sphinx | 强大、多格式输出 | Python项目、学术文档 | reStructuredText/Markdown |
| Antora | 多仓库、AsciiDoc | 企业级文档平台 | AsciiDoc |
| Starlight | Astro框架、现代化 | 新项目文档 | Astro |

**选型建议**：
- 团队使用React → Docusaurus
- 团队使用Vue → VuePress / VitePress
- 需要简洁快速 → MkDocs Material
- 需要托管和协作 → GitBook
- Python项目 → Sphinx
