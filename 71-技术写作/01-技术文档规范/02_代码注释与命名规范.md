# 代码注释与命名规范


## 一、命名规范


### 1. 通用命名原则


- **见名知意：**
   名称应能表达意图，不需要注释辅助
- **避免缩写：**
   除非是广泛认知的缩写（如 URL、HTML、ID）
- **有意义的区分：**
   避免
   `a1`
   、
   `a2`
   这类无意义命名
- **可发音：**
   名称应该能被读出来，方便口头沟通
- **可搜索：**
   避免单字母变量名（循环计数器除外）


### 2. 常见命名风格


| 风格 | 格式 | 适用场景 | 语言示例 |
| --- | --- | --- | --- |
| camelCase | firstName | 变量、方法 | Java, JavaScript, TypeScript |
| PascalCase | FirstName | 类名、接口 | Java, C#, TypeScript |
| snake_case | first_name | 变量、函数 | Python, Ruby, Rust |
| kebab-case | first-name | URL, CSS, 文件名 | HTML, CSS, Shell |
| UPPER_SNAKE | MAX_RETRY | 常量 | 所有语言通用 |
| SCREAMING_SNAKE | DB_HOST | 环境变量 | 配置文件 |


### 3. 命名示例对比


```
// 反面示例
int d;              // d 是什么？
List<User> data;    // data 太泛了
void process();     // 处理什么？
boolean temp;       // 临时什么？

// 正面示例
int elapsedDays;
List<User> activeUsers;
void calculateMonthlyRevenue();
boolean isEligibleForDiscount;
```


### 4. 函数/方法命名


```
// 动词 + 名词模式
getUserById(id)
calculateTotalPrice(items)
validateEmailAddress(email)
isAuthorized(user, resource)
hasPermission(user, action)
convertCelsiusToFahrenheit(temp)

// 避免
getData()        // 太模糊
handleStuff()    // 无信息量
doWork()         // 什么工作？
```


### 5. 布尔变量命名


```
// 使用 is/has/can/should 等前缀
isActive
hasError
canEdit
shouldRetry
isEmpty
wasProcessed
```


## 二、代码注释规范


### 1. 注释的黄金法则


> **Note:** **最好的注释是不需要注释。**
> 通过良好的命名和清晰的代码结构，让代码自解释。注释应该解释 WHY（为什么这么做），而不是 WHAT（做了什么）。


### 2. 什么时候写注释


| 场景 | 示例 |
| --- | --- |
| 业务逻辑原因 | // 商家费率低于5%时免手续费（政策要求，2024-Q1起） |
| 非显而易见的权衡 | // 使用二分而非哈希，因数据已排序且内存受限 |
| 警告/副作用 | // WARNING: 此方法会修改输入参数 |
| 待办事项 | // TODO: 重构为策略模式，当前硬编码不满足新增渠道 |
| 正则表达式含义 | // 匹配中国大陆手机号：1开头，第二位3-9，后接9位数字 |


### 3. 什么时候不写注释


```
// 反面：注释重复代码（噪音）
int count = 0;  // 初始化计数器为0

// 正面：代码已自解释
int retryCount = 0;

// 反面：过时注释（比没有注释更危险）
// 连接数据库（实际已改为连接缓存）
cacheClient.connect();

// 正面：如果需要说明，写在文档里
/** 使用缓存客户端替代数据库连接，详见迁移文档 ADR-015 */
```


### 4. 注释类型


```
// ===== 文件级注释 =====
/**
 * UserService - 用户账户管理服务
 *
 * 职责：用户注册、登录、权限校验、资料管理
 * 依赖：UserRepository, AuthService, EmailService
 *
 * @author 张三
 * @since 2024-01-15
 */

// ===== 类级注释 =====
/**
 * 实现指数退避重试策略。
 *
 * 每次重试等待时间 = min(baseDelay * 2^attempt + jitter, maxDelay)
 * 支持可配置的最大重试次数和可重试异常类型。
 */

// ===== 方法级 JSDoc/Javadoc =====
/**
 * 计算购物车折扣后的总价。
 *
 * 折扣规则：
 * - 满100减10，满200减30（不叠加）
 * - VIP 用户额外 95 折（在满减基础上）
 * - 优惠券优先级最高（覆盖满减）
 *
 * @param {CartItem[]} items - 购物车商品列表
 * @param {User} user - 当前用户（含 VIP 状态）
 * @param {Coupon} [coupon] - 可选的优惠券
 * @returns {number} 折扣后的总价（保留两位小数）
 * @throws {InvalidCouponError} 优惠券已过期或不适用
 *
 * @example
 * const total = calculateDiscountedTotal(cartItems, currentUser, coupon);
 */
```


## 三、文档注释格式


### 1. JSDoc（JavaScript/TypeScript）


```
/**
 * 向指定用户发送验证码。
 * @param {string} phone - 手机号（11位中国大陆号码）
 * @param {'sms'|'voice'} channel - 发送渠道
 * @param {number} [expireMinutes=5] - 验证码有效期（分钟）
 * @returns {Promise<SendResult>} 发送结果
 * @throws {RateLimitError} 发送频率超限
 * @see {@link verifyCode} 用于验证验证码
 */
async function sendVerificationCode(phone, channel, expireMinutes = 5) {
    // ...
}
```


### 2. Javadoc（Java）


```
/**
 * 使用 RSA 算法对数据进行签名。
 *
 * <p>采用 SHA256withRSA 签名算法，私钥长度至少 2048 位。
 *
 * @param data 待签名的原始数据（UTF-8 编码）
 * @param privateKey PEM 格式的私钥
 * @return Base64 编码的签名结果
 * @throws InvalidKeyException 私钥格式无效
 * @throws NoSuchAlgorithmException 不支持 SHA256withRSA
 * @since 1.2.0
 */
public String sign(byte[] data, PrivateKey privateKey) { ... }
```


### 3. Docstring（Python）


```
def calculate_compound_interest(
    principal: float,
    rate: float,
    times: int,
    years: int
) -> float:
    """计算复利终值。

    使用公式 A = P(1 + r/n)^(nt)

    Args:
        principal: 本金（元）
        rate: 年利率（如 0.05 表示 5%）
        times: 每年计息次数
        years: 投资年数

    Returns:
        复利终值，保留两位小数

    Raises:
        ValueError: 当本金或利率为负数时

    Examples:
        >>> calculate_compound_interest(10000, 0.05, 12, 10)
        16470.09
    """
    if principal < 0 or rate < 0:
        raise ValueError("本金和利率不能为负数")
    return round(principal * (1 + rate / times) ** (times * years), 2)
```


## 四、自文档代码实践


```
// 反面：过度注释，代码混乱
// 检查用户是否满足条件
if (u.age > 18 && u.t == 1 && u.s > 0) {  // 年龄>18且类型为1且分数>0
    // ...
}

// 正面：自文档代码
boolean isEligibleAdult = user.getAge() > LEGAL_AGE
    && user.getAccountType() == AccountType.PREMIUM
    && user.getCreditScore() > MIN_CREDIT_SCORE;

if (isEligibleAdult) {
    // 只需注释业务原因，不需要注释逻辑
    // 符合2024年Q1风控政策：仅限优质成年用户参与
    grantAccess(user);
}
```


> **Note:** **核心理念：**
> 代码是写给人看的，附带能被机器执行。好的命名让代码即文档，好的注释解释设计意图和业务背景。过时注释比没有注释更有害——如果代码和注释不一致，两个都可能是错的。

总结：命名遵循见名知意、动词+名词、布尔用is/has/can前缀等原则；注释解释WHY而非WHAT，包括业务原因、权衡取舍、警告副作用；文档注释格式有JSDoc/Javadoc/Docstring三种；自文档代码通过有意义的命名和提取变量/方法减少注释需求；过时注释比没有注释更有害
