架构图与技术图表工具

技术写作图表架构文档

好的图表胜过千言万语。选择合适的工具和图类型是技术文档的关键。

PlantUML - 文本驱动图表

PlantUML 优势： - 用文本描述图表，可版本控制 - 支持多种图表类型 - IDE集成丰富（VS Code, IntelliJ） - 可自动化生成 时序图示例： @startuml title 用户登录流程 actor 用户 as User participant "前端" as FE participant "API网关" as GW participant "认证服务" as Auth database "用户数据库" as DB User -> FE: 输入用户名密码 FE -> GW: POST /api/login GW -> Auth: 验证凭证 Auth -> DB: 查询用户 DB --> Auth: 用户记录 Auth --> GW: JWT Token GW --> FE: 200 OK + Token FE --> User: 登录成功，跳转首页 note right of Auth: 密码使用bcrypt哈希 @enduml 类图示例： @startuml class Order { - id: UUID - status: OrderStatus - items: List<OrderItem> + addItem(item: OrderItem): void + calculateTotal(): Money + cancel(): void } class OrderItem { - productId: UUID - quantity: int - unitPrice: Money } Order "1" *-- "*" OrderItem @enduml 部署图示例： @startuml cloud "AWS" { node "ECS Cluster" { container "API Server" as api container "Worker" as worker } database "RDS PostgreSQL" as rds queue "SQS" as sqs } api --> rds : 读写 api --> sqs : 发消息 worker --> sqs : 消费消息 @enduml

Mermaid - Markdown原生图表

Mermaid 优势： - GitHub/GitLab原生支持 - Markdown内嵌，无需额外工具 - 轻量级，学习成本低 - 支持实时预览 流程图示例： ```mermaid flowchart TD A[用户请求] --> B{缓存命中?} B -->|是| C[返回缓存] B -->|否| D[查询数据库] D --> E[写入缓存] E --> F[返回结果] C --> G[记录日志] F --> G ``` ER图示例： ```mermaid erDiagram USER ||--o{ ORDER : places ORDER ||--|{ ORDER_ITEM : contains PRODUCT ||--o{ ORDER_ITEM : "referenced in" USER { string id PK string name string email } ORDER { string id PK datetime created_at enum status } ``` 甘特图示例： ```mermaid gantt title 项目迭代计划 dateFormat YYYY-MM-DD section 需求阶段 需求评审 :a1, 2024-01-01, 5d 技术方案设计 :a2, after a1, 3d section 开发阶段 后端开发 :b1, after a2, 10d 前端开发 :b2, after a2, 8d section 测试阶段 集成测试 :c1, after b1, 5d UAT测试 :c2, after c1, 3d ``` 状态图示例： ```mermaid stateDiagram-v2 [*] --> 待支付 待支付 --> 已支付 : 支付成功 待支付 --> 已取消 : 超时/取消 已支付 --> 已发货 : 发货 已发货 --> 已签收 : 确认收货 已签收 --> [*] ```

C4 模型 - 架构文档标准

C4模型四级抽象（Simon Brown）： ┌──────────────────────────────────────────────┐ │ Level 1: 系统上下文图 (System Context) │ │ 展示系统与外部用户、系统的关系 │ │ 受众：所有人（包括非技术人员） │ ├──────────────────────────────────────────────┤ │ Level 2: 容器图 (Container) │ │ 展示系统的技术构建块（应用、数据库等） │ │ 受众：技术人员、架构师 │ ├──────────────────────────────────────────────┤ │ Level 3: 组件图 (Component) │ │ 展示容器内部的模块/服务/组件 │ │ 受众：开发人员 │ ├──────────────────────────────────────────────┤ │ Level 4: 代码图 (Code) │ │ 展示组件的类图/模块图 │ │ 受众：开发人员（可选，可用UML替代） │ └──────────────────────────────────────────────┘ C4图使用Structurizr（DSL）示例： workspace { model { user = person "用户" admin = person "管理员" system = softwareSystem "电商平台" { web = container "Web应用" "React" "前端SPA" api = container "API服务" "Spring Boot" "REST API" db = container "数据库" "PostgreSQL" "数据存储" cache = container "缓存" "Redis" "会话和热数据" } payment = softwareSystem "支付网关" "外部" user -> web "浏览商品、下单" web -> api "API调用" api -> db "读写数据" api -> cache "缓存查询" api -> payment "发起支付" admin -> api "管理操作" } } 绘制工具： - Structurizr：C4官方工具，DSL驱动 - draw.io：通用图表工具，免费 - Excalidraw：手绘风格，适合白板 - Miro/Mural：在线协作白板 - Figma：设计导向，适合UI图

图表选择指南与最佳实践

图表类型选择： ┌──────────────┬──────────────────────────────────┐ │ 想表达的内容 │ 推荐图表类型 │ ├──────────────┼──────────────────────────────────┤ │ 系统间交互 │ 时序图、系统上下文图 │ │ 数据流/控制流 │ 流程图、数据流图 │ │ 系统架构 │ C4图、部署图 │ │ 数据模型 │ ER图、类图 │ │ 业务流程 │ 活动图、BPMN │ │ 状态变化 │ 状态图 │ │ 时间安排 │ 甘特图、时间线 │ │ 组织结构 │ 组织架构图 │ │ 项目计划 │ 甘特图、看板 │ └──────────────┴──────────────────────────────────┘ 图表最佳实践： 1. 每张图有明确标题 2. 图例说明所有符号和颜色 3. 保持简洁（每图不超过7±2个元素） 4. 标注方向（从左到右/从上到下） 5. 文本与代码仓库一起维护（PlantUML/Mermaid） 6. 定期审查和更新过时图表 7. 针对受众选择抽象级别 工具选型建议： - 纯文本、可版本控制 → PlantUML / Mermaid - 专业美观 → draw.io / Figma - 快速白板 → Excalidraw - 团队协作 → Miro / Mural - 架构文档 → Structurizr (C4)
PlantUML和Mermaid让图表可像代码一样版本控制。C4模型提供了从高层到细节的四级视图。选择图表类型时，先明确受众和想传达的信息，再选择最合适的表达方式。
