# 架构决策记录 (ADR - Architecture Decision Record)


## 一、什么是 ADR


ADR 是记录重要架构决策的轻量级文档。它捕捉每个决策的上下文、选项、选择和后果，形成团队的架构知识库。


- **起源：**
   Michael Nygard 在 2011 年提出，被 ThoughtWorks 技术雷达推荐
- **核心理念：**
   决策比设计图更重要——记录"为什么"比"是什么"更有价值
- **形式：**
   Markdown 文件，按编号存放在版本控制仓库中


## 二、什么时候写 ADR


| 应该写 ADR 的场景 | 不需要写 ADR 的场景 |
| --- | --- |
| 选择编程语言/框架 | 命名规范等已有明确标准的 |
| 选择数据库类型 | 纯实现层面的细节 |
| 微服务拆分策略 | 临时方案/快速修复 |
| API 设计风格（REST vs gRPC） | 已被团队标准覆盖的决策 |
| 消息队列选型 | 明显的唯一选项 |
| 安全/认证方案 |  |
| 部署/基础设施方案 |  |


## 三、ADR 模板


```
# ADR-{编号}: {标题}

## 状态
{提议 | 已接受 | 已弃用 | 被取代（附指向新ADR的链接）}

## 日期
{YYYY-MM-DD}

## 背景（Context）
{描述驱动这个决策的技术/业务背景，包括：
- 面临的问题或约束
- 相关的业务需求
- 技术环境限制}

## 决策（Decision）
{清晰地陈述做出的决策，一句话总结}

## 备选方案（Alternatives Considered）
### 方案A: {名称}
- 优点: ...
- 缺点: ...

### 方案B: {名称}
- 优点: ...
- 缺点: ...

## 后果（Consequences）
{描述采纳此决策后的正面和负面影响：
- 获得了什么能力
- 放弃了什么可能性
- 团队需要做什么调整
- 长期影响}

## 参考资料
{相关链接、文档、会议纪要}
```


## 四、ADR 示例


### ADR-007: 选择 PostgreSQL 作为主数据库


**状态：**已接受


**日期：**2024-03-15


**背景：**

        订单系统需要关系型数据库存储订单、支付和库存数据。团队预计日订单量将达到 100 万级，
        需要强一致性保证（ACID）。同时需要支持 JSON 字段存储灵活的商品属性。团队已有 MySQL 运维经验。


**决策：**

        选择 PostgreSQL 15 作为主数据库。


**备选方案：**


| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| MySQL 8.0 | 团队熟悉；运维经验丰富 | JSON 支持弱；缺少原生部分索引 |
| PostgreSQL 15 | JSONB 强大；部分索引；CTE 优化好 | 团队需学习；复制方案不同 |
| TiDB | 水平扩展；兼容 MySQL 协议 | 运维复杂；成本高；团队无经验 |


**后果：**


- 正面：利用 JSONB 存储灵活的商品规格，减少表结构变更；部分索引提升查询性能
- 负面：团队需要 2 周 PostgreSQL 培训；需建立新的备份/监控方案
- 后续：需要制定 PostgreSQL 运维手册（见 ADR-008）


## 五、ADR 与 RFC 的关系


```
时间线：
RFC（征求意见）          ADR（决策记录）
    │                        │
    ▼                        ▼
┌──────────┐            ┌──────────┐
│ 提出方案  │ ──讨论──→ │ 记录结果  │
│ 征求意见  │            │ 已成定论  │
│ 尚未决定  │            │ 团队共识  │
└──────────┘            └──────────┘

关系：
- RFC 是决策前的讨论过程
- ADR 是决策后的记录结果
- RFC 达成共识后 → 产出 ADR
- ADR 是版本化的、不可变的历史记录
```


- **RFC (Request For Comments)：**
   用于提出新方案，征求团队意见，是一个"提议"
- **ADR：**
   记录已做出的决策，是一个"结论"
- 可以将 RFC 转化为 ADR，也可以直接写 ADR（对于无需广泛讨论的小决策）


## 六、ADR 管理实践


### 1. 目录结构


```
docs/
  adr/
    0001-use-microservices-architecture.md
    0002-adopt-rest-api-convention.md
    0003-use-kafka-for-async-messaging.md
    0004-choose-postgresql-over-mysql.md
    0005-implement-jwt-authentication.md
    README.md  ← ADR 索引目录
```


### 2. 状态管理


- **Proposed（提议）：**
   刚提出，等待讨论
- **Accepted（已接受）：**
   团队已同意并采纳
- **Deprecated（已弃用）：**
   不再推荐使用，但可能仍在生产中
- **Superseded（被取代）：**
   被新 ADR 替代，附指向新 ADR 的链接


### 3. 最佳实践


- **不可变性：**
   一旦 Accepted，不要修改内容。如果决策变化，创建新的 ADR 来取代
- **简洁：**
   理想长度 1-2 页，最多不超过 5 页
- **及时：**
   决策做出后尽快记录，趁记忆清晰
- **版本控制：**
   ADR 文件提交到 Git，参与 Code Review
- **定期回顾：**
   季度回顾 ADR，标记已过时的决策


> **Note:** **核心价值：**
> ADR 解决了"我们当初为什么这么设计"这个问题。新人入职可以通过 ADR 理解架构演进的历史；老成员也不会因为时间久远而忘记决策原因。它是团队的架构记忆。

总结：ADR是记录重要架构决策的轻量级文档，核心是记录WHY；标准模板包含状态、背景、决策、备选方案和后果；RFC是决策前的讨论过程，ADR是决策后的记录；ADR应保持不可变，决策变化创建新ADR；管理要点：版本控制、及时记录、定期回顾、保持简洁
