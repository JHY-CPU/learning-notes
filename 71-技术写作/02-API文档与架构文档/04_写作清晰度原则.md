# 写作清晰度原则


## 写作清晰度原则


技术写作清晰度信息架构


技术文档的核心价值是让读者快速理解并采取行动。


## Plain Language（简明语言）


```
简明语言核心原则：

1. 使用常见词汇
   ✗ "utilize" → ✓ "use"
   ✗ "initiate" → ✓ "start"
   ✗ "facilitate" → ✓ "help"
   ✗ "aforementioned" → ✓ "this" / "the above"

2. 避免行话或加以解释
   ✗ "这个API是幂等的"
   ✓ "调用多次此API，结果与调用一次相同（幂等）"

3. 句子简短（平均15-20字）
   ✗ "当用户在界面上点击提交按钮之后，系统会首先对表单数据
      进行客户端验证，然后再通过AJAX异步请求将数据发送到服务器。"
   ✓ "用户点击提交后，系统先验证表单数据，再发送到服务器。"

4. 一个句子一个意思
   ✗ "配置数据库连接并启动服务，然后检查日志确认是否成功。"
   ✓ "配置数据库连接。启动服务。检查日志确认启动成功。"

5. 使用列表代替长段落
   处理流程包括三个步骤：首先...其次...最后...
   →
   处理流程：
   1. 步骤一
   2. 步骤二
   3. 步骤三
```


## 主动语态与任务导向写作


```
主动语态 vs 被动语态：

✗ "错误将被系统记录到日志文件中"
✓ "系统将错误记录到日志文件中"

✗ "数据库连接可以通过配置文件进行配置"
✓ "在配置文件中设置数据库连接参数"

✗ "当请求被处理完成后，响应会被返回"
✓ "服务器处理请求后，返回响应"

被动语态适用场景（例外）：
- 强调动作承受者："该漏洞已被修复"
- 避免指责："配置已被更改"（而不是"你改了配置"）
- 未知执行者："数据已加密存储"

任务导向写作 (Task-Oriented Writing)：
原则：文档应按用户要完成的任务组织，而非按功能描述。

✗ 功能导向：
"配置管理模块支持以下功能：新增配置、修改配置、删除配置。"

✓ 任务导向：
"## 如何添加新配置
1. 进入管理后台 > 配置管理
2. 点击「新增配置」按钮
3. 填写配置名称和值
4. 点击「保存」"

任务导向结构模板：
## 如何 [动词] [对象]
[简短说明目的和适用场景]
### 前提条件
- 需要XX权限
- 需要XX已配置
### 操作步骤
1. 第一步
2. 第二步
3. 第三步
### 验证结果
- 成功标志
### 常见问题
- 问题A → 解决方案
```


## 信息架构 (Information Architecture)


```
信息组织模式：

1. 层次结构 (Hierarchy)
   - 从概到细，逐层展开
   - 适用：文档导航、目录结构

2. 顺序结构 (Sequence)
   - 按时间或逻辑顺序
   - 适用：教程、安装指南、故障排查

3. 矩阵结构 (Matrix)
   - 多维度交叉组织
   - 适用：API参考（按资源×操作）

4. 有机结构 (Organic)
   - 自由链接，非线性
   - 适用：Wiki、知识库

文档类型及其最佳结构：
┌──────────────┬──────────────┬────────────────────┐
│ 文档类型       │ 信息结构      │ 读者目标            │
├──────────────┼──────────────┼────────────────────┤
│ 入门教程       │ 顺序结构      │ 学会基本操作         │
│ 操作指南       │ 顺序结构      │ 完成特定任务         │
│ API参考       │ 矩阵结构      │ 快速查找用法         │
│ 概念说明       │ 层次结构      │ 理解原理和背景       │
│ 故障排查       │ 决策树结构    │ 解决具体问题         │
│ 架构文档       │ C4分层       │ 理解系统设计         │
└──────────────┴──────────────┴────────────────────┘

Diátaxis文档框架（推荐）：
将所有文档分为四类：
1. Tutorial (教程)：学习导向，手把手教
2. How-to Guide (操作指南)：任务导向，解决具体问题
3. Reference (参考)：信息导向，完整准确
4. Explanation (说明)：理解导向，阐述原理
```


## 渐进式披露 (Progressive Disclosure)


```
原则：先给用户最需要的信息，细节按需展示。

层级设计：
Level 1 - 快速概览（30秒可读完）
- 一句话描述功能
- 最简用法示例
- 链接到详细文档

Level 2 - 常见用法（5分钟可读完）
- 核心参数说明
- 常见使用场景
- 代码示例

Level 3 - 完整参考（按需查阅）
- 所有参数列表
- 高级配置选项
- 边界条件和限制

示例：API文档的渐进式披露
Level 1：
## 获取用户信息
GET /api/users/{id}
返回指定用户的详细信息。

Level 2：
### 请求参数
| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | 是 | 用户ID |

### 响应示例
```json
{"id": "u001", "name": "张三", "email": "zhang@example.com"}
```

Level 3（折叠/链接）：

完整参数列表和错误码
支持字段筛选：GET /api/users/{id}?fields=name,email
错误码：404用户不存在, 403无权限...


写作检查清单：
□ 标题是否清楚说明内容？
□ 第一段是否概述了要点？
□ 是否使用主动语态？
□ 句子是否简短（<20字）？
□ 是否使用列表代替长段落？
□ 术语是否首次出现时解释？
□ 示例是否完整可运行？
□ 按任务而非功能组织？
□ 删除了所有不必要的词？
```


> **Note:** 清晰写作的黄金法则：为读者而非自己写作。使用简明语言、主动语态、任务导向结构。采用Diátaxis框架将文档分为教程、指南、参考和说明四类。渐进式披露让不同需求的读者各取所需。
