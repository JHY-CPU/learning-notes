# 微服务与事件驱动架构


## 微服务与事件驱动架构

一、微服务架构

## 一、微服务架构（Microservices）


微服务架构将应用构建为一组小型、自治的服务，每个服务围绕业务能力构建，独立部署、独立扩展。


### 1.1 核心特征


- **单一职责：**
   每个服务聚焦于一个业务能力
- **独立部署：**
   服务可独立发布，不影响其他服务
- **去中心化治理：**
   各服务可选择不同的技术栈
- **去中心化数据管理：**
   每个服务管理自己的数据库
- **容错设计：**
   单个服务故障不应导致系统整体不可用
- **演进式设计：**
   服务可以独立演进和替换


### 1.2 服务间通信


| 方式 | 协议/技术 | 特点 |
| --- | --- | --- |
| 同步通信 | REST/gRPC/GraphQL | 实时响应，简单直接 |
| 异步通信 | 消息队列（Kafka/RabbitMQ） | 解耦、削峰、可靠 |
| 事件驱动 | Event Bus/Event Streaming | 松耦合、可扩展 |


### 1.3 微服务基础设施


- **API网关：**
   统一入口，路由、认证、限流（如Kong、Zuul）
- **服务注册与发现：**
   服务自动注册和发现（如Consul、Eureka）
- **配置中心：**
   集中管理配置（如Spring Cloud Config、Nacos）
- **熔断器：**
   故障隔离和降级（如Hystrix、Resilience4j）
- **链路追踪：**
   分布式调用链追踪（如Zipkin、Jaeger）


### 1.4 微服务 vs 单体架构


| 维度 | 单体架构 | 微服务架构 |
| --- | --- | --- |
| 部署 | 整体部署 | 独立部署 |
| 扩展 | 整体扩展 | 按需扩展单个服务 |
| 技术栈 | 统一 | 可异构 |
| 数据管理 | 共享数据库 | 每个服务独立数据库 |
| 团队组织 | 按技术层分工 | 按业务能力组建全功能团队 |
| 复杂度 | 开发简单，运维复杂 | 开发复杂，需完善的基础设施 |

二、事件驱动架构

## 二、事件驱动架构（Event-Driven Architecture）


事件驱动架构以事件的产生、检测和消费为核心，组件之间通过事件进行异步通信。


### 2.1 核心概念


- **事件（Event）：**
   系统状态变化的通知，包含事件类型、数据和元信息
- **事件生产者（Producer）：**
   检测状态变化并发布事件
- **事件消费者（Consumer）：**
   订阅并处理感兴趣的事件
- **事件通道（Channel）：**
   事件传输的中间件（消息队列/事件总线）


### 2.2 拓扑模式


| 模式 | 说明 | 适用场景 |
| --- | --- | --- |
| Mediator模式 | 事件经过中央协调器处理后分发 | 需要编排的复杂事件处理 |
| Broker模式 | 事件直接分发到订阅者，无中央协调 | 高度解耦的响应式系统 |

三、Serverless架构

## 三、Serverless架构


Serverless架构将计算资源管理完全交给云平台，开发者只关注业务逻辑代码。


### 3.1 核心组件


- **FaaS（Function as a Service）：**
   按需运行函数（如AWS Lambda、Azure Functions）
- **BaaS（Backend as a Service）：**
   使用托管服务替代自建后端（如数据库、认证、存储）


### 3.2 特点

**优势：**
零运维、自动扩缩容、按使用付费、快速开发


**限制：**
冷启动延迟、执行时间限制、厂商锁定、调试困难
四、Service Mesh

## 四、服务网格（Service Mesh）


服务网格是处理服务间通信的专用基础设施层，通过Sidecar代理实现流量管理、安全和可观测性。


### 4.1 架构组件


- **数据平面（Data Plane）：**
   Sidecar代理（如Envoy），拦截和处理网络流量
- **控制平面（Control Plane）：**
   管理和配置代理（如Istio、Linkerd）


### 4.2 核心功能


| 功能 | 说明 |
| --- | --- |
| 流量管理 | 负载均衡、熔断、重试、灰度发布 |
| 安全 | mTLS、认证授权、加密通信 |
| 可观测性 | 指标收集、分布式追踪、访问日志 |
