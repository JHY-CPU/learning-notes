# 01-架构风格

## 1. 软件架构的定义与作用

软件架构（Software Architecture）是指系统的基本组织方式，体现在其组件、组件之间的关系、组件与环境的关系，以及指导其设计和演进的原则中。

架构的核心作用：
- **沟通媒介**：为开发团队、利益相关者提供共同的理解框架
- **早期设计决策**：架构是最难更改的决策集合，决定了系统的质量属性
- **约束与指导**：为详细设计和编码提供约束框架
- **风险管理**：在早期识别技术风险和复杂度来源

> 架构不仅仅是技术选型，更是对系统结构、行为和约束的系统性描述。

## 2. 架构风格 vs 架构模式

| 维度 | 架构风格（Architecture Style） | 架构模式（Architecture Pattern） |
|------|-------------------------------|--------------------------------|
| 抽象层级 | 更抽象，关注组织原则 | 更具体，关注实现方式 |
| 示例 | 分层、管道-过滤器、事件驱动 | MVC、MVP、MVVM |
| 约束程度 | 定义一组约束和规则 | 提供具体的结构模板 |
| 组合性 | 可组合使用 | 通常单独使用 |

实际项目中，架构风格和模式常混合使用。例如一个系统可以同时采用分层风格和MVC模式。

## 3. 分层架构（Layered Architecture）

分层架构是最经典的架构风格，将系统划分为若干水平层，每层只与相邻层交互。

### 典型分层结构

```
┌─────────────────────┐
│   表示层（Presentation） │  ← 用户界面、API入口
├─────────────────────┤
│   业务层（Business）     │  ← 业务逻辑、领域规则
├─────────────────────┤
│   持久层（Persistence）  │  ← 数据访问、ORM映射
├─────────────────────┤
│   数据层（Database）     │  ← 数据库、文件系统
└─────────────────────┘
```

### 核心原则

- **单向依赖**：上层依赖下层，下层不依赖上层
- **层间隔离**：每层通过接口暴露服务，隐藏实现细节
- **可替换性**：任意层可被同接口的实现替换

### 优缺点

**优点**：关注点分离清晰、易于理解、支持团队按层分工
**缺点**：性能开销（层层传递）、可能导致"下沉"反模式（业务逻辑泄漏到下层）、对变化的传播不友好

## 4. 管道-过滤器架构

管道-过滤器（Pipes and Filters）将系统分解为一系列处理步骤（过滤器），数据通过管道在过滤器之间流动。

```
[数据源] → [过滤器A] → [管道] → [过滤器B] → [管道] → [过滤器C] → [输出]
```

### 特点

- 每个过滤器是独立的处理单元，有输入和输出
- 管道负责数据传输，不做处理
- 过滤器可并行执行（数据流并行）
- 支持过滤器的动态组合和重用

### 适用场景

- 数据处理流水线（ETL、编译器）
- 信号处理、图像处理
- Unix Shell 管道命令

### 局限性

- 不适合交互式应用
- 数据格式需要统一或可转换
- 错误处理和回滚较困难

## 5. 事件驱动架构（Event-Driven）

事件驱动架构以事件的产生、检测和消费为核心，组件之间通过事件进行松耦合通信。

### 核心概念

- **事件（Event）**：系统状态的显著变化，如"订单已创建"
- **事件生产者**：检测到状态变化并发出事件
- **事件消费者**：订阅并响应事件
- **事件通道**：事件传输的中介（消息队列、事件总线）

### 拓扑模型

1. **Mediator 拓扑**：中心事件中介负责编排事件流
2. **Broker 拓扑**：事件直接在 broker 中流转，无中心编排

### 适用场景

- 异步处理（订单支付后触发库存扣减、通知发送）
- 实时数据流处理
- 用户界面交互（DOM事件）

## 6. 发布-订阅架构

发布-订阅（Pub/Sub）是事件驱动架构的一种具体实现模式。

### 核心角色

- **发布者（Publisher）**：发布消息到特定主题
- **订阅者（Subscriber）**：订阅感兴趣的主题
- **消息代理（Broker）**：负责消息路由和分发

### 与观察者模式的区别

| 维度 | 观察者模式 | 发布-订阅模式 |
|------|-----------|-------------|
| 耦合度 | 观察者与被观察者直接关联 | 通过消息中介解耦 |
| 跨进程 | 通常在同一进程内 | 可跨进程、跨服务 |
| 灵活性 | 添加观察者需修改被观察者 | 动态订阅/取消 |

### 典型实现

- **消息队列**：RabbitMQ、Apache Kafka、RocketMQ
- **云服务**：AWS SNS/SQS、Google Pub/Sub
- **前端**：EventEmitter、Redux Store

## 7. 微服务架构的核心原则

微服务架构将应用拆分为一组小型、自治的服务，每个服务围绕业务能力构建。

### 核心原则

1. **单一职责**：每个服务只做一件事
2. **独立部署**：服务可独立构建、测试、部署
3. **去中心化治理**：各服务可选择不同技术栈
4. **去中心化数据管理**：每个服务拥有自己的数据存储
5. **为失败而设计**：容错、降级、熔断
6. **演进式设计**：服务可独立演进和替换

### 服务拆分策略

- **按业务能力拆分**：用户服务、订单服务、支付服务
- **按子域拆分**（DDD）：核心域、支撑域、通用域
- **按数据拆分**：每种数据类型对应一个服务

### 通信方式

- **同步**：REST API、gRPC
- **异步**：消息队列、事件驱动

## 8. 服务网格（Service Mesh）

服务网格是微服务间通信的基础设施层，将通信逻辑从业务代码中剥离。

### 核心架构

```
[服务A] ←→ [Sidecar Proxy A] ←→ [网络] ←→ [Sidecar Proxy B] ←→ [服务B]
```

### 核心功能

- **流量管理**：负载均衡、重试、超时、熔断
- **可观测性**：分布式追踪、指标收集、日志
- **安全**：mTLS、认证、授权
- **策略控制**：限流、访问控制

### 主流实现

- **Istio**：基于Envoy Proxy，功能最全面
- **Linkerd**：轻量级，CNCF毕业项目
- **Consul Connect**：HashiCorp生态

## 9. Serverless架构

Serverless 不意味着没有服务器，而是开发者无需管理服务器基础设施。

### 两种核心服务

1. **FaaS（Function as a Service）**：按需执行函数，如 AWS Lambda、Azure Functions
2. **BaaS（Backend as a Service）**：使用第三方服务，如 Firebase、Supabase

### 核心特点

- **事件驱动**：由HTTP请求、消息、定时器等事件触发
- **自动扩缩**：从零扩展到任意规模
- **按用量计费**：只为实际执行时间付费
- **无状态**：函数执行完即销毁

### 适用场景

- API网关后面的业务逻辑
- 数据处理管道
- 定时任务
- 事件响应（文件上传、数据库变更）

### 局限性

- 冷启动延迟
- 执行时间限制（通常15分钟）
- 调试和监控复杂
- 供应商锁定风险

## 10. 边车模式（Sidecar）

边车模式将辅助功能部署为与主应用同生命周期的独立进程。

### 架构示意

```
┌──────────────────────────┐
│  Pod / 容器组              │
│  ┌──────────┐ ┌─────────┐│
│  │ 主应用    │ │ Sidecar ││
│  │          │ │ Proxy   ││
│  └──────────┘ └─────────┘│
└──────────────────────────┘
```

### 典型用途

- 日志收集代理
- 代理/负载均衡（如Envoy）
- 配置同步
- 安全代理（mTLS终止）
- 监控数据采集

### 优势

- 与主应用独立演进和部署
- 语言无关——同一Sidecar可服务不同语言的主应用
- 关注点分离

## 11. API网关模式

API网关是所有客户端请求的统一入口，负责请求路由、协议转换和横切关注点。

### 核心功能

- **请求路由**：将请求转发到对应的后端服务
- **协议转换**：HTTP ↔ gRPC、WebSocket等
- **认证授权**：统一鉴权
- **限流熔断**：保护后端服务
- **请求聚合**：将多个微服务响应合并
- **缓存**：减少后端压力

### 主流实现

- **Kong**：基于Nginx + Lua
- **APISIX**：Apache顶级项目
- **Spring Cloud Gateway**：Java生态
- **AWS API Gateway**：云原生

## 12. BFF模式（Backend for Frontend）

BFF为每种前端客户端提供专属的后端服务。

### 架构示意

```
[Web App]    → [BFF-Web]    → [微服务集群]
[Mobile App] → [BFF-Mobile] → [微服务集群]
[小程序]      → [BFF-Mini]   → [微服务集群]
```

### 设计动机

- 不同客户端的数据需求和交互模式不同
- 避免一个"万能"API导致的过度设计
- 各BFF可独立优化和演进

### 适用场景

- 多端应用（Web、iOS、Android、小程序）
- 前后端团队有明确边界
- 需要为不同客户端优化性能

## 13. 各架构风格的适用场景对比

| 架构风格 | 适用场景 | 团队规模 | 复杂度 | 性能要求 |
|---------|---------|---------|--------|---------|
| 分层架构 | 单体应用、CRUD系统 | 小-中 | 低-中 | 中 |
| 管道-过滤器 | 数据处理、ETL | 小 | 中 | 高（并行） |
| 事件驱动 | 异步处理、实时系统 | 中-大 | 中-高 | 高 |
| 微服务 | 大型分布式系统 | 大 | 高 | 高 |
| Serverless | 事件驱动、轻量API | 小 | 低-中 | 中 |
| 服务网格 | 微服务通信治理 | 大 | 高 | 高 |
| BFF | 多端应用 | 中 | 中 | 中 |

### 选型建议

1. **从简单开始**：优先选择分层架构，避免过度设计
2. **按需演进**：当单体成为瓶颈时，考虑微服务拆分
3. **关注团队能力**：架构复杂度要与团队能力匹配
4. **混合使用**：一个系统可同时采用多种架构风格
