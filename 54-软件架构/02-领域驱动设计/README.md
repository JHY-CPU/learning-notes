# 02-领域驱动设计

## 1. DDD的核心理念

领域驱动设计（Domain-Driven Design, DDD）由Eric Evans在2003年提出，核心思想是：**以业务领域为中心驱动软件设计**。

### 核心主张

- 软件的核心价值在于解决业务问题，而非技术炫技
- 领域专家与开发人员应使用同一套语言沟通
- 复杂业务逻辑应集中在领域层，而非散落在各处
- 模型是设计的核心，而非数据库表结构

### DDD解决的问题

- 传统"贫血模型"导致业务逻辑泄漏到Service层
- 技术驱动的设计忽视了业务本质
- 大型系统缺乏清晰的边界和组织方式

## 2. 通用语言（Ubiquitous Language）

通用语言是DDD的基石，是开发团队和领域专家共同使用的统一语言。

### 建立原则

- **领域术语一致**：代码中的类名、方法名、变量名使用领域术语
- **消除歧义**：同一概念在不同上下文中含义不同时，需要明确区分
- **持续维护**：随着领域理解的深入，通用语言也在演进

### 示例

```java
// 不好的命名（技术视角）
public class DataProcessor {
    public void processItem(ItemDTO dto) { ... }
}

// 好的命名（领域视角）
public class 订单处理器 {
    public void 确认订单(订单 订单) { ... }
}
```

### 产出物

- 领域术语表（Glossary）
- 领域模型图
- 用户故事中的领域语言

## 3. 限界上下文（Bounded Context）

限界上下文是DDD中最重要的战略模式，定义了模型适用的边界。

### 核心概念

- 在一个大型系统中，同一概念在不同上下文中有不同含义
- 例如"用户"在"身份认证上下文"中是账号，在"订单上下文"中是买家
- 每个限界上下文内维护自己的模型和通用语言

### 边界划分依据

- **业务能力**：按业务功能划分
- **团队组织**：遵循康威定律
- **数据所有权**：谁创建谁拥有
- **变更频率**：经常一起变化的放在一起

### 示例

```
┌─────────────────────────────────────────────┐
│                电商系统                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐   │
│  │ 商品上下文 │  │ 订单上下文 │  │ 支付上下文 │   │
│  │          │  │          │  │          │   │
│  │ 商品-Product│ │ 订单-Order│  │ 支付-Payment│ │
│  │ 库存-Stock │  │ 买家-Buyer│  │ 账户-Account│ │
│  └──────────┘  └──────────┘  └──────────┘   │
└─────────────────────────────────────────────┘
```

## 4. 实体与值对象

### 实体（Entity）

实体是有唯一标识的对象，其身份在其生命周期中保持不变。

特征：
- 有唯一标识（ID）
- 可变（状态可变更）
- 通过标识判断相等性

```java
public class 订单 {
    private 订单号 id;        // 唯一标识
    private 订单状态 状态;     // 可变属性
    private List<订单项> 明细;
}
```

### 值对象（Value Object）

值对象是没有唯一标识的对象，通过属性值判断相等性。

特征：
- 无唯一标识
- 不可变（创建后不可修改）
- 通过属性值判断相等性
- 可自由替换和共享

```java
public class 金额 {
    private final BigDecimal 数值;
    private final String 币种;

    public 金额(BigDecimal 数值, String 币种) {
        this.数值 = 数值;
        this.币种 = 币种;
    }

    public 金额 加(金额 另一个) {
        // 返回新的值对象，而非修改自身
        return new 金额(this.数值.add(另一个.数值), this.币种);
    }
}
```

### 选择原则

- 需要跟踪变化和生命周期的 → 实体
- 只描述某种特征的 → 值对象
- 优先使用值对象，减少复杂度

## 5. 聚合与聚合根

### 聚合（Aggregate）

聚合是一组紧密关联的对象的集合，作为数据修改的事务边界。

### 聚合根（Aggregate Root）

聚合根是聚合的入口对象，外部只能通过聚合根访问聚合内部对象。

### 设计原则

1. **聚合边界内保证一致性**：聚合内的不变量必须在同一事务中满足
2. **外部引用只指向聚合根**：不直接引用聚合内部对象
3. **聚合间通过ID引用**：避免跨聚合的直接对象引用
4. **一个事务只修改一个聚合**：避免分布式事务

```
┌─────── 订单聚合 ───────┐
│                         │
│  订单（聚合根）          │
│    ├── 订单项A（值对象） │
│    ├── 订单项B（值对象） │
│    └── 收货地址（值对象） │
│                         │
└─────────────────────────┘
  ↑ 外部只能通过 订单 引用
```

## 6. 领域服务

当业务逻辑不属于任何实体或值对象时，使用领域服务封装。

### 特征

- 操作涉及多个领域对象
- 操作本身不属于任何单一对象的职责
- 通常是无状态的

### 示例

```java
public class 转账服务 {
    public void 执行转账(账户 源账户, 账户 目标账户, 金额 金额) {
        源账户.扣减(金额);
        目标账户.增加(金额);
        // 记录领域事件
        领域事件.publish(new 转账已完成事件(源账户, 目标账户, 金额));
    }
}
```

### 与应用服务的区别

| 维度 | 领域服务 | 应用服务 |
|------|---------|---------|
| 职责 | 业务逻辑 | 流程编排 |
| 位置 | 领域层 | 应用层 |
| 复杂度 | 包含业务规则 | 协调调用，不含业务规则 |

## 7. 领域事件

领域事件是领域中发生的有意义的事情，用于表达领域中的事实。

### 特征

- 不可变（已经发生的事实不可更改）
- 使用过去时态命名
- 包含事件发生的关键信息

### 示例

```java
public class 订单已创建事件 {
    private final 订单号 订单编号;
    private final LocalDateTime 发生时间;
    private final 买家信息 买家;
    private final List<订单项> 明细;
}
```

### 应用场景

- 跨聚合通信：订单创建后触发库存扣减
- 跨上下文通信：支付完成通知订单上下文
- 事件溯源（Event Sourcing）：通过事件流重建状态
- 审计日志：自动记录业务操作

## 8. 仓储模式（Repository）

仓储封装了领域对象的持久化逻辑，使领域层不依赖具体的数据访问技术。

### 接口定义在领域层

```java
// 领域层 - 接口
public interface 订单仓储 {
    订单 根据ID查询(订单号 id);
    void 保存(订单 订单);
    List<订单> 查询待发货订单();
}
```

### 实现放在基础设施层

```java
// 基础设施层 - JPA实现
@Repository
public class Jpa订单仓储 implements 订单仓储 {
    @Autowired
    private 订单JpaRepository jpaRepo;

    @Override
    public 订单 根据ID查询(订单号 id) {
        return jpaRepo.findById(id).orElseThrow();
    }
}
```

### 设计原则

- 每个聚合根对应一个仓储
- 仓储接口属于领域层，实现在基础设施层
- 仓储返回领域对象，而非DTO

## 9. 上下文映射（Context Mapping）

上下文映射描述不同限界上下文之间的关系和集成方式。

### 常见模式

| 模式 | 说明 | 耦合度 |
|------|------|--------|
| 共享内核（Shared Kernel） | 两个上下文共享部分模型 | 高 |
| 客户-供应商 | 下游依赖上游的模型 | 中 |
| 遵奉者（Conformist） | 下游完全遵从上游模型 | 高 |
| 防腐层（ACL） | 下游通过适配层翻译上游模型 | 低 |
| 开放主机服务（OHS） | 上游提供标准化API | 低 |
| 发布语言（PL） | 定义通用的数据交换格式 | 低 |

### 防腐层示例

```
[外部系统API] → [防腐层(适配器)] → [本上下文领域模型]
```

防腐层负责将外部模型转换为本上下文的领域模型，保护核心领域不受外部变化的影响。

## 10. 战略设计 vs 战术设计

### 战略设计（Strategic Design）

关注宏观的系统组织和边界划分：
- 限界上下文划分
- 上下文映射
- 通用语言建立
- 核心域识别

### 战术设计（Tactical Design）

关注微观的领域模型实现：
- 实体、值对象
- 聚合、聚合根
- 领域服务、领域事件
- 仓储

### 关系

```
战略设计（画大图）        战术设计（画细节）
┌─────────────┐      ┌─────────────┐
│ 限界上下文    │  →   │ 聚合设计      │
│ 上下文映射    │  →   │ 实体/值对象   │
│ 核心域识别    │  →   │ 领域服务      │
└─────────────┘      └─────────────┘
```

**实践建议**：先做战略设计，再做战术设计。战略设计的错误代价远大于战术设计。

## 11. 六边形架构与DDD的关系

六边形架构（Hexagonal Architecture，又称端口-适配器架构）与DDD天然契合。

### 核心思想

- 领域模型位于中心
- 外部依赖（数据库、消息队列、UI）通过适配器接入
- 依赖方向从外向内

```
        ┌─── 适配器 ───┐
        │               │
   [HTTP Controller]   [REST Client]
        │               │
   [应用服务 Port]      [外部服务 Port]
        │               │
   ┌────┴───────────────┴────┐
   │       领域模型            │
   │  (实体、值对象、聚合)     │
   └──────────────────────────┘
        │               │
   [Repository Port]   [消息 Port]
        │               │
   [JPA Adapter]     [Kafka Adapter]
```

### 与DDD的对应

- **端口** = DDD中的接口（Repository、领域服务接口）
- **适配器** = DDD中的基础设施实现
- **内部领域** = DDD的领域层

## 12. CQRS + Event Sourcing

### CQRS（命令查询职责分离）

将读操作和写操作分离到不同的模型中。

```
         ┌──────────────┐
写操作 → │ Command Model │ → 写数据库（关系型）
         └──────────────┘

         ┌──────────────┐
读操作 → │ Query Model   │ → 读数据库（搜索/缓存/物化视图）
         └──────────────┘
```

### Event Sourcing（事件溯源）

不存储当前状态，而是存储导致状态变化的事件序列。

```
事件流：
  订单已创建 → 商品已添加 → 订单已提交 → 支付已完成 → 订单已发货

当前状态 = 重放所有事件后的最终状态
```

### 优势

- **完整审计日志**：每个状态变化都有记录
- **时间旅行**：可回溯到任意历史时刻的状态
- **与CQRS天然配合**：通过重放事件更新读模型
- **解耦**：事件可被多个消费者独立处理

### 适用与不适用

**适合**：金融系统、审计需求强烈的系统、复杂事件驱动系统
**不适合**：简单CRUD、查询远多于写入的系统、团队不熟悉时慎用

### 组合架构示意

```
[Command] → [聚合根] → [事件存储]
                         │
                    [事件总线] ──→ [投影到读模型]
                                     │
[Query]  ←─────────────────── [读模型/物化视图]
```
