# DDD 战术设计模式


## 1. 仓储模式 (Repository)

**核心思想：**
将数据持久化逻辑封装在类似集合的接口后面，领域层通过 Repository 接口访问数据，不知道底层存储技术。

```
// 仓储接口 — 属于领域层
public interface OrderRepository {
    Order findById(OrderId id);
    List<Order> findByCustomer(CustomerId customerId);
    void save(Order order);
    void delete(OrderId id);
}

// JPA 实现 — 属于基础设施层
public class JpaOrderRepository implements OrderRepository {
    @PersistenceContext private EntityManager em;

    public Order findById(OrderId id) {
        var entity = em.find(OrderJpaEntity.class, id.getValue());
        return OrderMapper.toDomain(entity);
    }

    public void save(Order order) {
        var entity = OrderMapper.toJpaEntity(order);
        em.merge(entity);
    }
}
```


#### Repository 设计要点


- 只对聚合根提供（非所有实体）
- 接口在领域层，实现在基础设施层
- 返回领域对象，不返回 DTO/Entity
- 仓储内不包含业务逻辑


#### Repository vs DAO


- Repository 粒度粗（聚合根级别）
- DAO/Mapper 粒度细（表级别）
- Repository 封装查询逻辑
- DAO 仅做 CRUD 映射


## 2. 聚合根 (Aggregate Root) 规则

**聚合根**
是聚合中唯一对外可见的实体，外部只能通过聚合根引用内部对象。聚合根负责维护聚合内部的一致性。

```
// 聚合根：Order
public class Order {
    private OrderId id;
    private OrderStatus status;
    private List<OrderItem> items = new ArrayList<>(); // 内部实体
    private Money totalAmount;                         // 值对象

    // 只有聚合根可以操作内部实体
    public void addItem(Product product, int quantity) {
        if (status != OrderStatus.DRAFT)
            throw new IllegalStateException("已确认订单不能修改");
        items.add(new OrderItem(product.getId(), product.getPrice(), quantity));
        recalculateTotal();
    }

    public OrderConfirmed confirm() {
        if (items.isEmpty()) throw new BusinessException("空订单");
        status = OrderStatus.CONFIRMED;
        return new OrderConfirmed(id, totalAmount);
    }
}
```


### 聚合根设计五原则


| 原则 | 说明 | 反例 |
| --- | --- | --- |
| 1. 聚合边界内一致性 | 一个事务只修改一个聚合 | 一个事务修改两个聚合 |
| 2. 聚合间引用用 ID | Order 引用 customerId 而非整个 Customer 对象 | 直接持有其他聚合根对象 |
| 3. 一个事务一个聚合 | 跨聚合用最终一致性 | 分布式事务 |
| 4. 聚合尽量小 | 只包含必须保持一致性的元素 | 把所有相关实体都放进一个聚合 |
| 5. 通过聚合根访问 | 外部不能直接访问内部实体/值对象 | 直接 new OrderItem 并添加 |


## 3. 领域事件 (Domain Events)

**核心思想：**
领域中发生的有意义的事情。聚合在状态变更时发布事件，其他聚合或服务通过订阅事件做出响应。

```
// 领域事件定义
public class OrderConfirmed {
    public final OrderId orderId;
    public final Money totalAmount;
    public final LocalDateTime occurredAt;

    public OrderConfirmed(OrderId id, Money amount) {
        this.orderId = id;
        this.totalAmount = amount;
        this.occurredAt = LocalDateTime.now();
    }
}

// 事件发布
public class Order {
    private List<Object> domainEvents = new ArrayList<>();

    public void confirm() {
        status = OrderStatus.CONFIRMED;
        domainEvents.add(new OrderConfirmed(id, total)); // 记录事件
    }

    public List<Object> getDomainEvents() { return domainEvents; }
    public void clearEvents() { domainEvents.clear(); }
}

// 事件处理器
public class InventoryEventHandler {
    @EventHandler
    public void handle(OrderConfirmed event) {
        // 扣减库存
        inventoryService.reserve(event.orderId);
    }
}
```


## 4. 防腐层 (Anti-Corruption Layer)

**核心思想：**
在两个限界上下文（或遗留系统）之间放置一个翻译层，防止外部模型"污染"内部模型。ACL 负责将外部概念翻译为内部概念。
遗留系统模型
→
ACL 翻译层
→
内部领域模型

```
// 防腐层接口（定义翻译契约）
public interface LegacySystemAdapter {
    Product translateProduct(LegacyProductDTO legacy);
    Customer translateCustomer(LegacyClientRecord legacy);
}

// 防腐层实现
public class LegacyAntiCorruptionLayer implements LegacySystemAdapter {
    public Product translateProduct(LegacyProductDTO legacy) {
        // 将遗留系统的概念翻译为内部领域模型
        return new Product(
            new ProductId(legacy.SKU_CODE),         // SKU → ProductId
            legacy.ITEM_DESC,                        // ITEM_DESC → name
            Money.of(legacy.UNIT_PRICE, "CNY")    // 数字 + 硬编码 → 值对象
        );
    }

    public Customer translateCustomer(LegacyClientRecord legacy) {
        return new Customer(
            new CustomerId(legacy.CLIENT_NO),
            legacy.NAME_CN + " " + legacy.NAME_EN,
            Email.of(legacy.E_MAIL)                 // 格式验证
        );
    }
}
```


## 5. 限界上下文集成方式


| 方式 | 同步/异步 | 耦合度 | 适用场景 |
| --- | --- | --- | --- |
| 共享内核 (Shared Kernel) | 编译时 | 高 | 紧密协作的团队 |
| 客户-供应商 | 同步调用 | 中 | 上游提供 API |
| 防腐层 (ACL) | 同步/异步 | 低 | 对接遗留系统或第三方 |
| 开放主机服务 (OHS) | API/事件 | 低 | 多团队多系统 |
| 事件驱动集成 | 异步 | 低 | 微服务间通信 |

**设计原则：**
每个限界上下文有自己独立的领域模型。避免在多个上下文中复用同一个实体类。上下文间通过防腐层或事件集成，而非共享数据库。
