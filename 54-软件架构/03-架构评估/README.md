# 03-架构评估

## 1. 架构评估的必要性

软件架构一旦确定，修改成本极高。架构评估的目的是在实现之前发现架构设计中的问题。

### 评估的价值

- **降低风险**：早期发现架构缺陷，避免后期返工
- **优化决策**：在多种架构方案中做出有依据的选择
- **统一认知**：帮助团队和利益相关者达成共识
- **质量保障**：验证架构是否满足质量属性需求

### 常见的评估维度

| 维度 | 关注点 |
|------|--------|
| 性能 | 响应时间、吞吐量、并发能力 |
| 可用性 | 系统无故障运行时间、故障恢复 |
| 可扩展性 | 水平/垂直扩展能力 |
| 可维护性 | 代码可读性、模块化程度 |
| 安全性 | 认证、授权、数据保护 |
| 可测试性 | 单元测试、集成测试的便利性 |

## 2. ATAM（架构权衡分析方法）的步骤

ATAM（Architecture Tradeoff Analysis Method）是SEI提出的最成熟的架构评估方法，关注质量属性之间的权衡。

### 九个步骤

1. **介绍ATAM**：向参与者说明评估方法和流程
2. **介绍业务驱动因素**：阐述业务目标、上下文和约束
3. **介绍架构**：架构师展示架构设计
4. **识别架构方法**：识别架构中采用的关键设计决策
5. **生成质量属性效用树**：将质量属性需求具体化
6. **分析架构方法**：针对高优先级场景分析架构决策
7. **集体讨论和场景优先级排序**：参与者提出新场景并排序
8. **分析架构方法（再一轮）**：深入分析排名最高的场景
9. **展示结果**：汇总发现的风险、敏感点、权衡点

### 参与角色

- **评估小组**：独立第三方，负责组织评估
- **项目决策者**：项目经理、架构师
- **利益相关者**：开发人员、测试人员、运维人员、客户代表

## 3. SAAM（软件架构分析方法）

SAAM是最早的架构评估方法，侧重于可修改性评估。

### 核心流程

1. **描述架构**：使用场景描述系统行为
2. **分类场景**：
   - **直接场景**：当前架构直接支持的场景
   - **间接场景**：需要修改架构才能支持的场景
3. **评估间接场景的影响**：分析每个间接场景需要修改哪些模块
4. **建立总体评估**：统计修改的代价和影响范围

### 与ATAM的对比

| 维度 | SAAM | ATAM |
|------|------|------|
| 关注质量属性 | 主要关注可修改性 | 多种质量属性及权衡 |
| 复杂度 | 较简单 | 较复杂 |
| 输出 | 修改代价评估 | 风险、敏感点、权衡点 |
| 适用阶段 | 早期架构评审 | 中后期详细评审 |

## 4. 质量属性效用树

质量属性效用树（Utility Tree）是ATAM的核心工具，将抽象的质量属性分解为具体的、可评估的场景。

### 结构

```
效用（Utility）
├── 性能
│   ├── 场景1：在峰值负载下，搜索响应时间 < 200ms
│   └── 场景2：批量导入100万条记录在30分钟内完成
├── 可用性
│   ├── 场景3：数据库故障后，系统在30秒内自动切换
│   └── 场景4：计划内停机每月不超过10分钟
├── 可扩展性
│   └── 场景5：用户量从1万增长到100万时，无需重构架构
└── 安全性
    └── 场景6：用户密码以bcrypt加密存储，密钥不落盘明文
```

### 场景描述模板

每个场景包含六个要素：
- **刺激源（Source）**：谁/什么触发
- **刺激（Stimulus）**：发生了什么
- **环境（Environment）**：在什么条件下
- **制品（Artifact）**：系统的哪个部分受影响
- **响应（Response）**：系统如何响应
- **响应度量（Response Measure）**：如何衡量响应

## 5. 敏感点与权衡点

### 敏感点（Sensitivity Point）

某个设计决策或组件对特定质量属性有显著影响。

> 例：缓存策略对性能和一致性都是敏感点

### 权衡点（Tradeoff Point）

某个设计决策同时影响多个质量属性，且方向不同。

> 例：
> - 增加缓存 → 性能提升，一致性下降
> - 增加冗余 → 可用性提升，成本增加
> - 使用加密 → 安全性提升，性能下降

### 发现权衡点的意义

权衡点是架构决策最需要谨慎对待的地方。ATAM的核心目标之一就是找出这些权衡点。

## 6. 架构决策记录（ADR）

架构决策记录（Architecture Decision Record）是一种轻量级文档格式，用于记录重要的架构决策及其背景。

### ADR模板

```markdown
# ADR-001: [决策标题]

## 状态
[提议 | 已接受 | 已弃用 | 已取代]

## 背景
描述驱动这一决策的背景和约束。

## 决策
明确记录做出的决策内容。

## 后果
记录这一决策带来的正面和负面影响。

## 替代方案
列出考虑过的其他方案及被否决的原因。
```

### ADR的优点

- **可追溯**：知道"为什么"这样做，而非只是"做了什么"
- **轻量级**：Markdown文件，可放入版本控制
- **增量式**：随项目演进逐步积累
- **促进沟通**：为团队提供讨论架构决策的框架

### ADR管理实践

- 每个ADR独立一个文件，按编号排列
- 已弃用的ADR标记状态但不删除（保留历史）
- 重要的跨团队决策需评审后接受

## 7. 架构评审检查清单

架构评审检查清单用于系统性地检查架构设计的完整性。

### 设计完整性

- [ ] 是否覆盖了所有关键需求（功能和非功能）？
- [ ] 架构文档是否足够支撑开发团队理解系统？
- [ ] 是否有明确的模块/服务边界和接口定义？

### 质量属性

- [ ] 性能瓶颈是否已识别并有应对方案？
- [ ] 系统是否有容错和降级机制？
- [ ] 数据存储是否考虑了备份和恢复？
- [ ] 安全措施是否覆盖认证、授权、数据加密？

### 技术选型

- [ ] 技术栈是否与团队能力匹配？
- [ ] 是否评估了关键技术的风险？
- [ ] 依赖的第三方服务是否有替代方案？

### 可演进性

- [ ] 架构是否支持渐进式演进？
- [ ] 是否考虑了未来可能的功能扩展？
- [ ] 模块间的耦合度是否可控？

### 运维视角

- [ ] 是否有监控和告警方案？
- [ ] 部署策略是否支持零停机？
- [ ] 日志和追踪是否覆盖关键路径？
