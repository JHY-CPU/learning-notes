# 容器选择指南

## 一、概念说明

选择正确的容器是C++性能优化的基础。本指南总结了各种场景下的最佳容器选择，基于操作复杂度、缓存友好性、内存开销和实际性能测试。

## 二、决策树

```cpp
/*
选择流程：

1. 元素数量固定且编译时已知？
   → array

2. 需要键值对？
   ├─ 是 → 需要有序？
   │    ├─ 是 → map（读多写少考虑flat_map）
   │    └─ 否 → unordered_map
   └─ 否 → 继续

3. 需要唯一元素？
   ├─ 是 → 需要有序？
   │    ├─ 是 → set（读多写少考虑flat_set）
   │    └─ 否 → unordered_set
   └─ 否 → 继续

4. 需要随机访问？
   ├─ 是 → 需要头部操作？
   │    ├─ 是 → deque
   │    └─ 否 → vector（默认）
   └─ 否 → 需要频繁中间插入删除？
        ├─ 是 → 需要双向遍历？
        │    ├─ 是 → list
        │    └─ 否 → forward_list
        └─ 否 → vector

5. 特殊需求：
   - LIFO → stack
   - FIFO → queue
   - 优先级 → priority_queue
   - 可选值 → optional
   - 多类型值 → variant
   - 非拥有视图 → span, string_view
   - 允许重复有序 → multiset/multimap
   - 允许重复无序 → unordered_multiset/unordered_multimap
*/
```

## 三、场景推荐

```cpp
/*
| 场景                          | 推荐容器                    | 原因                     |
|------------------------------|---------------------------|-------------------------|
| 默认选择                      | vector                    | 缓存友好，大多数最快       |
| 字典/映射                     | unordered_map             | O(1)查找                 |
| 有序字典                      | map                       | 有序遍历                  |
| 去重                          | unordered_set             | O(1)查找                 |
| 有序去重                      | set                       | 有序遍历                  |
| 频繁头部操作                  | deque                     | O(1)两端操作             |
| 频繁任意位置插入删除           | list                      | O(1)给定迭代器           |
| 固定大小数组                  | array                     | 零开销                   |
| 栈操作                        | stack                     | LIFO语义                 |
| 队列操作                      | queue                     | FIFO语义                 |
| 优先级处理                    | priority_queue            | 堆排序                   |
| 函数返回多个值                | pair / tuple              | 多返回值                 |
| 可选值                        | optional                  | 类型安全                 |
| 字符串                        | string                    | SSO优化                  |
| 非拥有视图                    | string_view / span        | 零拷贝                   |
| 读多写少有序                  | flat_map / flat_set       | 缓存友好（C++23）         |
| 类型安全联合                  | variant                   | 编译时类型检查            |
| 任意类型存储                  | any                       | 运行时类型擦除            |
*/
```

## 四、性能优化原则

```cpp
/*
1. vector默认
   - 缓存友好，大多数场景最快
   - 即使理论复杂度不优，实际也常更快

2. 避免过早优化
   - 先用vector，性能分析后再考虑切换
   - 不要仅凭理论复杂度选择容器

3. reserve预分配
   - 已知元素数量时使用reserve
   - 避免多次扩容

4. emplace替代push
   - 避免不必要的临时对象
   - 对非平凡类型有显著性能差异

5. 哈希表 vs 红黑树
   - 查找密集用unordered
   - 有序遍历用ordered
   - 需要确定性性能用ordered

6. 小数据用array
   - 零开销，栈上存储
   - 大小固定时优于vector

7. 稳定引用用list
   - 引用/指针不随插入删除失效
   - 但缓存不友好
*/
```

## 五、内存开销对比

```cpp
/*
| 容器              | 每元素额外开销      | 说明                   |
|-------------------|-------------------|-----------------------|
| array             | 0                 | 栈上，固定大小          |
| vector            | 3个指针（整体）     | start, finish, end    |
| deque             | 中等              | 中控表+分段             |
| list              | 2个指针（每元素）   | prev + next           |
| forward_list      | 1个指针（每元素）   | next                  |
| set/map           | 3个指针+颜色（每元素）| 红黑树节点             |
| unordered_*       | 1个指针+桶数组      | 哈希表                 |
*/
```

## 六、常见反模式

```cpp
// 反模式1：用list存储小对象并频繁遍历
// 正确：用vector（缓存友好）
std::list<int> bad1;     // 每元素额外16字节
std::vector<int> good1;  // 连续内存

// 反模式2：频繁在vector头部插入
// 正确：用deque
std::vector<int> bad2;
bad2.insert(bad2.begin(), 1);  // O(n)每次
std::deque<int> good2;
good2.push_front(1);           // O(1)

// 反模式3：用map做纯查找（不需要有序）
// 正确：用unordered_map
std::map<std::string, int> bad3;  // O(log n)
std::unordered_map<std::string, int> good3;  // O(1)

// 反模式4：vector存储多态对象
// 正确：用智能指针
std::vector<Base> bad4;  // 切片问题！
std::vector<std::unique_ptr<Base>> good4;

// 反模式5：不用reserve（已知大小时）
std::vector<int> bad5;
for (int i = 0; i < 10000; ++i) bad5.push_back(i);  // 多次扩容

std::vector<int> good5;
good5.reserve(10000);  // 一次分配
for (int i = 0; i < 10000; ++i) good5.push_back(i);
```

## 七、总结

1. **没有万能容器**：选择取决于具体场景
2. **先用默认容器（vector）**：性能分析后再优化
3. **考虑异常安全、迭代器失效、内存布局**
4. **自定义类型作为键需要hash或比较器**
5. **不要过度工程化**：简单场景用简单容器
