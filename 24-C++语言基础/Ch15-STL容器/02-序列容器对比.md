# 序列容器对比

## 一、概念说明

序列容器（vector、deque、list、forward_list、array）按元素插入顺序存储，但底层数据结构不同（C++标准 §23.2.1-§23.3），导致操作复杂度差异巨大。正确选择容器对性能至关重要。现代CPU的缓存架构使得连续内存容器（vector）通常在实际运行中远快于理论复杂度更好的链表容器。

### 1.1 底层结构对比

| 容器 | 底层结构 | 内存布局 | 每元素额外开销 |
|------|---------|---------|--------------|
| vector | 动态数组 | 完全连续 | 无 |
| deque | 分段数组 | 分段连续 | 中等 |
| list | 双向链表 | 非连续 | 2个指针 |
| forward_list | 单向链表 | 非连续 | 1个指针 |
| array | 固定数组 | 完全连续 | 无 |

## 二、性能对比

### 2.1 操作复杂度对比

```cpp
/*
| 操作              | vector      | deque       | list        | forward_list |
|-------------------|-------------|-------------|-------------|-------------|
| 随机访问          | O(1)        | O(1)        | O(n)        | O(n)        |
| 尾部插入          | 均摊O(1)    | O(1)        | O(1)        | N/A         |
| 头部插入          | O(n)        | O(1)        | O(1)        | O(1)        |
| 中间插入          | O(n)        | O(n)        | O(1)*       | O(1)*       |
| 尾部删除          | O(1)        | O(1)        | O(1)        | N/A         |
| 头部删除          | O(n)        | O(1)        | O(1)        | O(1)        |
| 中间删除          | O(n)        | O(n)        | O(1)*       | O(1)*       |
| 缓存友好度        | 高          | 中          | 低          | 低          |
| 查找              | O(n)        | O(n)        | O(n)        | O(n)        |

* 需要已定位到该位置的迭代器
*/
```

### 2.2 实际性能测试

```cpp
#include <vector>
#include <deque>
#include <list>
#include <forward_list>
#include <iostream>
#include <chrono>
#include <numeric>

template<typename Container, typename Name>
void benchmark(Name name, int N) {
    Container c;
    auto start = std::chrono::high_resolution_clock::now();

    // 顺序插入
    for (int i = 0; i < N; ++i) {
        if constexpr (requires { c.push_back(i); })
            c.push_back(i);
        else
            c.push_front(i);
    }

    auto end = std::chrono::high_resolution_clock::now();
    auto ms = std::chrono::duration_cast<std::chrono::milliseconds>(end - start);
    std::cout << name << " 插入" << N << "个元素: " << ms.count() << "ms" << std::endl;
}

int main() {
    const int N = 1000000;
    benchmark<std::vector<int>>("vector", N);
    benchmark<std::deque<int>>("deque", N);
    benchmark<std::list<int>>("list", N);
    // vector通常最快（缓存友好）
}
```

### 2.3 选择策略详解

```cpp
// 场景1：默认选择vector
std::vector<int> default_choice;

// 场景2：需要频繁头部操作（如LRU缓存）
std::deque<int> needs_front_ops;
needs_front_ops.push_front(1);
needs_front_ops.pop_front();

// 场景3：需要频繁任意位置插入删除（如编辑器缓冲区）
std::list<int> needs_middle_ops;
auto it = needs_middle_ops.begin();
needs_middle_ops.insert(it, 42);  // O(1)

// 场景4：需要元素引用/指针稳定
std::list<int> needs_stable_refs;
int& ref = *needs_stable_refs.begin();
needs_stable_refs.push_back(99);  // ref仍然有效

// 场景5：内存受限，只需单向遍历
std::forward_list<int> memory_constrained;
```

## 三、选择决策树

```cpp
/*
选择流程：

1. 元素数量固定且编译时已知？
   → array

2. 需要随机访问？
   ├─ 是 → 需要头部操作？
   │    ├─ 是 → deque
   │    └─ 否 → vector（默认）
   └─ 否 → 继续

3. 频繁中间位置插入删除？
   ├─ 是 → 需要双向遍历？
   │    ├─ 是 → list
   │    └─ 否 → forward_list
   └─ 否 → vector

4. 需要元素引用/指针稳定？
   → list
*/
```

## 四、注意事项与常见陷阱

1. **缓存友好性胜过理论复杂度**：vector的缓存友好性通常使其即使在理论复杂度不占优时也更快
2. **list的指针开销**：每个元素额外两个指针（16字节在64位系统），存储小对象时开销显著
3. **deque不是完全连续**：不能直接传给需要连续内存的C接口（如`data()`不可用）
4. **不要过早优化**：先用vector，通过性能分析后再考虑切换
5. **forward_list没有`size()`**：为了O(1)操作故意省略
6. **vector<bool>特化**：位压缩实现，行为不同于普通vector（返回代理对象而非引用）
