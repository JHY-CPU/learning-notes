# 类设计最佳实践

## 一、概念说明

良好的类设计遵循SOLID原则和其他设计准则，确保代码可维护、可扩展、可测试。核心理念：**最小暴露**、**单一职责**、**面向接口编程**。

## 二、SOLID原则

### 2.1 单一职责（SRP）

```cpp
// 不好：一个类做太多事情
class UserManager {
public:
    void createUser() { }
    void sendEmail() { }  // 职责混入
    void logToFile() { }
};

// 好：职责分离
class UserCreator { public: void create(); };
class EmailSender { public: void send(); };
class Logger { public: void log(); };
```

### 2.2 开闭原则（OCP）

```cpp
// 对扩展开放，对修改关闭
class Shape {
public:
    virtual double area() const = 0;
    virtual ~Shape() = default;
};

// 添加新形状无需修改Shape
class Triangle : public Shape {
    double base, height;
public:
    Triangle(double b, double h) : base(b), height(h) {}
    double area() const override { return 0.5 * base * height; }
};
```

### 2.3 接口设计

```cpp
// 最小接口原则
class Reader {
public:
    virtual std::string read() = 0;
    virtual ~Reader() = default;
};

class Writer {
public:
    virtual void write(const std::string&) = 0;
    virtual ~Writer() = default;
};

// 需要读写时组合接口
class ReadWriter : public Reader, public Writer {};
```

### 2.4 封装与最小暴露

```cpp
class Connection {
public:
    // 暴露必要接口
    void connect();
    void disconnect();
    bool isConnected() const;

    // 隐藏实现细节
private:
    struct Impl;
    std::unique_ptr<Impl> pImpl;
};
```

## 三、实践指南

```cpp
// 1. 优先值语义
// 2. 用const成员函数表达不修改
// 3. 显式禁止不需要的操作 (=delete)
// 4. RAII管理资源
// 5. 考虑Rule of Zero（不管理资源就不定义特殊成员）
// 6. explicit单参数构造函数
// 7. 虚析构函数（如果可能被继承）
// 8. 使用智能指针而非裸指针
```

## 三、注意事项与常见陷阱

- 不要过度设计——YAGNI（You Aren't Gonna Need It）
- 接口应简洁、一致、可预测
- 虚函数默认应是private（模板方法模式）
- const正确性从设计之初就应该考虑
- 类的public接口是类的契约，一旦发布很难更改
