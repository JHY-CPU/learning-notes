# 异常设计原则

## 一、概念说明

良好的异常设计遵循一系列原则：不用于控制流、提供上下文信息、合理选择异常层次、确保异常安全。这些原则使异常处理更可预测和维护。

## 二、设计原则详解

### 2.1 不要用于控制流

```cpp
// 错误：用异常做控制流（性能差且语义不清）
int bad_loop() {
    int i = 0;
    try {
        while (true) {
            if (i > 10) throw std::runtime_error("结束");
            ++i;
        }
    } catch (...) {
        return i;
    }
}

// 正常：用正常控制流
int good_loop() {
    for (int i = 0; i <= 10; ++i) {}
    return 11;
}
```

### 2.2 提供上下文信息

```cpp
// 差：信息太少
void bad_example() {
    throw std::runtime_error("错误");
}

// 好：提供详细上下文
class ConfigError : public std::runtime_error {
    std::string file;
    int line;
public:
    ConfigError(const std::string& f, int l, const std::string& msg)
        : std::runtime_error(msg + " (文件: " + f + ", 行: " + std::to_string(l) + ")")
        , file(f), line(l) {}

    const std::string& get_file() const { return file; }
    int get_line() const { return line; }
};

void good_example() {
    throw ConfigError("config.xml", 42, "缺少必要字段");
}
```

### 2.3 异常层次设计

```cpp
// 好的异常层次：清晰的语义
class AppException : public std::runtime_error {
public:
    using std::runtime_error::runtime_error;
};

class NetworkException : public AppException {
public:
    using AppException::AppException;
};

class DatabaseException : public AppException {
public:
    using AppException::AppException;
};

class ValidationException : public AppException {
public:
    using AppException::AppException;
};

// 使用
void process_request() {
    try {
        // 网络操作
    } catch (const NetworkException& e) {
        // 特定处理
    } catch (const AppException& e) {
        // 通用处理
    }
}
```

### 2.4 异常安全与资源管理

```cpp
// 原则：每个操作要么成功要么不改变状态
class Transaction {
    std::vector<int> data;
    std::vector<int> backup;
public:
    void begin() {
        backup = data;  // 备份
    }

    void commit() {
        backup.clear();  // 提交，丢弃备份
    }

    void rollback() noexcept {
        data = std::move(backup);  // 回滚
    }

    void modify(int val) {
        data.push_back(val);
        if (val < 0) throw std::invalid_argument("负值");
    }
};
```

## 三、注意事项与常见陷阱

- 异常只应用于真正的异常情况，不应做控制流
- 提供足够上下文但不要包含敏感信息
- 异常层次应扁平（通常2-3层足够）
- 文档化每个函数的异常安全级别
- 考虑异常对性能的影响
- 在库的公共接口中，异常类型应是公共API的一部分
