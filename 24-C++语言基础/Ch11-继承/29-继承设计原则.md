# 继承设计原则

## 一、概念说明

良好的继承设计遵循**Liskov替换原则**（LSP，SOLID中的L）：派生类对象必须能够替换基类对象而不影响程序的正确性。即，任何使用基类的地方都应该能使用派生类。这是面向对象设计中最重要也最常被违反的原则。

### 1.1 SOLID与继承

| 原则 | 含义 | 与继承的关系 |
|------|------|-------------|
| S - 单一职责 | 一个类一个职责 | 继承层次中每个类职责清晰 |
| O - 开闭原则 | 对扩展开放，对修改关闭 | 通过继承扩展，不修改基类 |
| L - 里氏替换 | 派生类可替换基类 | 继承的核心约束 |
| I - 接口隔离 | 接口小而精 | 接口继承应最小化 |
| D - 依赖倒置 | 依赖抽象不依赖具体 | 基类指针/引用管理派生类 |

## 二、具体用法

### 2.1 遵守LSP

```cpp
#include <iostream>
#include <vector>
#include <memory>

// 好的设计：所有形状都有面积和周长
class Shape {
public:
    virtual double area() const = 0;
    virtual double perimeter() const = 0;
    virtual void describe() const {
        std::cout << "面积=" << area()
                  << ", 周长=" << perimeter() << std::endl;
    }
    virtual ~Shape() = default;
};

class Circle : public Shape {
    double r;
public:
    Circle(double r) : r(r) {}
    double area() const override { return 3.14159 * r * r; }
    double perimeter() const override { return 2 * 3.14159 * r; }
};

class Square : public Shape {
    double s;
public:
    Square(double s) : s(s) {}
    double area() const override { return s * s; }
    double perimeter() const override { return 4 * s; }
};

// 函数只依赖Shape接口，不关心具体类型
void printInfo(const Shape& shape) {
    std::cout << "面积=" << shape.area()
              << ", 周长=" << shape.perimeter() << std::endl;
}

// 所有派生类都可以替换Shape使用
void processShapes(const std::vector<std::unique_ptr<Shape>>& shapes) {
    for (const auto& s : shapes) {
        s->describe();  // 多态：任何Shape都能正确工作
    }
}

int main() {
    std::vector<std::unique_ptr<Shape>> shapes;
    shapes.push_back(std::make_unique<Circle>(5));
    shapes.push_back(std::make_unique<Square>(3));
    processShapes(shapes);

    Circle c(5);
    printInfo(c);  // Circle可以替换Shape

    return 0;
}
```

### 2.2 违反LSP的经典例子

```cpp
#include <iostream>

// 经典违反LSP：Square继承Rectangle
// 问题：Rectangle有setWidth/setHeight独立设置，
// Square需要保持w==h，破坏了Rectangle的契约

class Rectangle {
protected:
    int width, height;
public:
    Rectangle(int w, int h) : width(w), height(h) {}
    virtual void setWidth(int w) { width = w; }
    virtual void setHeight(int h) { height = h; }
    int getWidth() const { return width; }
    int getHeight() const { return height; }
    int area() const { return width * height; }
};

// 不好的设计
class BadSquare : public Rectangle {
public:
    BadSquare(int s) : Rectangle(s, s) {}
    void setWidth(int w) override { width = height = w; }  // 改变了Rectangle的契约
    void setHeight(int h) override { width = height = h; }
};

// 测试函数：对Rectangle成立，对BadSquare不成立
void testRectangle(Rectangle& r) {
    r.setWidth(5);
    r.setHeight(2);
    // 期望面积 = 5 * 2 = 10
    if (r.area() != 10) {
        std::cout << "LSP违反！期望10，实际" << r.area() << std::endl;
    }
}

// 好的设计：使用单独的类层次
class ShapeV2 {
public:
    virtual int area() const = 0;
    virtual ~ShapeV2() = default;
};

class RectangleV2 : public ShapeV2 {
    int w, h;
public:
    RectangleV2(int w, int h) : w(w), h(h) {}
    void setWidth(int v) { w = v; }
    void setHeight(int v) { h = v; }
    int area() const override { return w * h; }
};

class SquareV2 : public ShapeV2 {
    int side;
public:
    SquareV2(int s) : side(s) {}
    void setSide(int s) { side = s; }
    int area() const override { return side * side; }
};

int main() {
    BadSquare bs(5);
    testRectangle(bs);  // LSP违反！

    return 0;
}
```

### 2.3 继承层次设计最佳实践

```cpp
#include <iostream>
#include <memory>

// 策略1：接口继承（推荐）
class IDrawable {
public:
    virtual void draw() const = 0;
    virtual ~IDrawable() = default;
};

class IClickable {
public:
    virtual void onClick() = 0;
    virtual ~IClickable() = default;
};

// 策略2：使用final限制不合理的扩展
class SecurityToken final {
    std::string token;
public:
    SecurityToken(const std::string& t) : token(t) {}
    // 不能被继承：Token不应该有子类型
};

// 策略3：组合优于继承
class Logger {
public:
    virtual void log(const std::string& msg) {
        std::cout << "[LOG] " << msg << std::endl;
    }
    virtual ~Logger() = default;
};

class Service {
    Logger& logger;  // 组合：Service有一个Logger
public:
    Service(Logger& l) : logger(l) {}
    void doWork() {
        logger.log("开始工作");
        // ...
    }
};

int main() {
    // 设计检查清单：
    // 1. is-a关系？ → 用继承
    // 2. has-a关系？ → 用组合
    // 3. 需要多态？ → 接口继承
    // 4. 不应扩展？ → final类
    // 5. 继承层次 ≤ 3层？

    return 0;
}
```

## 三、注意事项与常见陷阱

1. **派生类不应加强前置条件或减弱后置条件**：违反LSP的核心表现
2. **不要用继承仅仅为了代码复用**：用组合，继承应表达is-a关系
3. **继承层次不宜过深**：建议不超过3层，过深增加理解难度
4. **基类的`const`约束应被派生类遵守**：不能将const方法改为修改状态
5. **考虑使用`final`类/函数限制过度扩展**：防止不合理的继承
6. **接口继承优于实现继承**：纯虚函数定义契约，实现用组合
7. **遵循开闭原则**：通过继承/组合扩展行为，不修改已有代码
8. **依赖注入优于硬编码依赖**：基类引用管理派生类，便于测试和替换
