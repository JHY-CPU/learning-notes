# 函数作用域与链接

## 一、概念说明

函数的**链接性**（linkage，C++11标准 §3.5）决定了不同翻译单元中的同名函数是否代表同一实体。C++中函数有三种链接：

| 链接类型 | 声明方式 | 可见范围 |
|---------|---------|---------|
| 外部链接（external） | 默认，或显式 `extern` | 所有翻译单元 |
| 内部链接（internal） | `static` 或匿名命名空间 | 仅当前翻译单元 |
| 无链接（none） | 局部函数、lambda | 仅当前作用域 |

ODR（One Definition Rule）：每个函数在整个程序中只能有一个定义（除非是inline函数）。

## 二、具体用法

### 2.1 外部链接（默认）

```cpp
// ========== file1.cpp ==========
// 具有外部链接，其他文件可extern声明后使用
int add(int a, int b) {
    return a + b;
}

// ========== file2.cpp ==========
// extern声明（extern可省略，但推荐显式写出）
extern int add(int a, int b);

int main() {
    cout << add(3, 5) << endl;  // 8，链接到file1的定义
    return 0;
}
```

### 2.2 内部链接 — static函数

```cpp
// ========== utils.cpp ==========
// static函数只在当前文件可见
static int helper(int x) {
    return x * x;
}

int compute(int x) {
    return helper(x) + 1;  // OK，同文件内可调用
}

// ========== main.cpp ==========
// extern int helper(int);  // 链接错误！helper在utils.cpp中是static

int main() {
    // helper(5);  // 链接错误
    return 0;
}
```

### 2.3 匿名命名空间（推荐替代static）

```cpp
// ========== utils.cpp ==========
// 现代C++推荐使用匿名命名空间替代static
namespace {
    int internalHelper(int x) {
        return x * 2;
    }

    class InternalConfig {
        int value;
    public:
        InternalConfig(int v) : value(v) {}
        int get() const { return value; }
    };
}

void process(int val) {
    cout << internalHelper(val) << endl;
    InternalConfig cfg(42);
    cout << cfg.get() << endl;
}
```

### 2.4 inline函数的链接

```cpp
// ========== header.h ==========
// inline函数允许在多个翻译单元中重复定义（放宽ODR规则）
inline int square(int x) { return x * x; }

// ========== file1.cpp ==========
#include "header.h"
int result1 = square(5);  // 实例化

// ========== file2.cpp ==========
#include "header.h"
int result2 = square(10);  // 实例化

// 链接器选择其中一个定义（或合并），不会报多重定义错误
```

### 2.5 内联变量（C++17）

```cpp
// ========== header.h ==========
// C++17 inline变量：头文件中定义全局常量
inline constexpr int MAX_BUFFER = 4096;
inline const char* APP_NAME = "MyApp";

// 线程安全的inline变量
#include <atomic>
inline atomic<int> globalCounter{0};

// ========== 任意源文件 ==========
#include "header.h"
// 所有文件共享同一个MAX_BUFFER，不会多重定义
```

### 2.6 extern模板（避免重复实例化）

```cpp
// ========== header.h ==========
template <typename T>
void printValue(T val) { cout << val << endl; }

// ========== file1.cpp ==========
#include "header.h"
template void printValue<int>(int);      // 显式实例化定义

// ========== file2.cpp ==========
#include "header.h"
extern template void printValue<int>(int);  // 抑制隐式实例化
// 编译器不再在file2中实例化int版本，而是链接到file1的定义
```

## 三、static vs 匿名命名空间

| 特性 | static函数 | 匿名命名空间函数 |
|------|-----------|----------------|
| 链接性 | 内部链接 | 内部链接 |
| 适用范围 | 函数和变量 | 函数、变量、类型、模板 |
| 能否包含类定义 | 不能 | 能 |
| C++风格 | C风格 | 现代C++风格 |
| 模板支持 | 不支持 | 支持 |

## 四、注意事项与常见陷阱

1. **static函数和匿名命名空间函数都是内部链接**：其他翻译单元无法访问
2. **在头文件中定义非inline函数会导致多重定义错误**：每个包含该头文件的翻译单元都会生成一个定义
3. **extern声明不需要函数体**：只需签名，告知编译器定义在别处
4. **inline函数允许在多个翻译单元中重复定义**：放宽ODR规则，是头文件中定义函数的前提
5. **C++17的inline变量也具有类似特性**：解决全局常量定义的ODR问题
6. **匿名命名空间的函数也可以声明为inline**：虽然已经是内部链接，但可以避免同一个翻译单元内的多重定义
7. **注意名称修饰（name mangling）**：不同编译器对函数名的修饰方式不同，影响跨编译器链接
