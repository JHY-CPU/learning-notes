# 文本纠错与标准化


## 文本纠错与标准化


NLP文本纠错标准化


文本纠错与标准化是保证文本质量的重要预处理环节，对搜索引擎、语音识别后处理、机器翻译等场景至关重要。


## 拼写纠错基础


```
中文拼写错误类型：
┌─────────────────────────────────────────────────────┐
│  1. 同音错误（最常见，约70%）                         │
│     "在见" → "再见"                                  │
│     "以经" → "已经"                                  │
│     "迫不急待" → "迫不及待"                          │
│                                                     │
│  2. 形近错误                                        │
│     "已" → "己"（已/己/巳 容易混淆）                 │
│     "戎" → "戒"                                     │
│                                                     │
│  3. 语法/搭配错误                                    │
│     "提高了学习的方法" → "改进了学习的方法"           │
│     "做了一个决定" vs "做了一个判决"                  │
│                                                     │
│  4. 多字/少字                                        │
│     "我觉得的" → "我觉得"                            │
│     "我高兴" → "我很高兴"                            │
└─────────────────────────────────────────────────────┘

英文拼写错误类型：
┌─────────────────────────────────────────────────────┐
│  Non-word Error (NWE)：拼出的不是合法单词              │
│  "accomodate" → "accommodate"                       │
│  "recieve" → "receive"                              │
│                                                     │
│  Real-word Error (RWE)：拼出的是合法单词但语境错误     │
│  "Their going home" → "They're going home"          │
│  "peace of mind" → "piece of mind" (取决于语境)      │
│                                                     │
│  错误来源：                                          │
│  - 键盘相邻键误触（hte → the）                       │
│  - 发音相似混淆（their/there/they're）               │
│  - 规则应用错误（i before e）                        │
└─────────────────────────────────────────────────────┘
```


## 编辑距离 (Edit Distance)


```
Levenshtein 距离定义：
将字符串A变换为字符串B所需的最少单字符编辑操作次数
操作：插入(Insert)、删除(Delete)、替换(Replace)

动态规划算法：
┌─────────────────────────────────────────────────────┐
│  D[i][j] = 将 A[0..i-1] 变换为 B[0..j-1] 的最小距离  │
│                                                     │
│  递推公式：                                          │
│  if A[i-1] == B[j-1]:                               │
│      D[i][j] = D[i-1][j-1]           # 无需操作      │
│  else:                                              │
│      D[i][j] = 1 + min(                             │
│          D[i-1][j],          # 删除 A[i-1]          │
│          D[i][j-1],          # 插入 B[j-1]          │
│          D[i-1][j-1]         # 替换 A[i-1] → B[j-1] │
│      )                                              │
│                                                     │
│  初始条件：                                          │
│  D[i][0] = i  （删除i个字符）                        │
│  D[0][j] = j  （插入j个字符）                        │
└─────────────────────────────────────────────────────┘

示例：计算 "kitten" → "sitting" 的编辑距离
┌───────┬───┬───┬───┬───┬───┬───┬───┬───┐
│       │   │ s │ i │ t │ t │ i │ n │ g │
├───────┼───┼───┼───┼───┼───┼───┼───┼───┤
│       │ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │
│ k     │ 1 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │
│ i     │ 2 │ 2 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │
│ t     │ 3 │ 3 │ 2 │ 1 │ 2 │ 3 │ 4 │ 5 │
│ t     │ 4 │ 4 │ 3 │ 2 │ 1 │ 2 │ 3 │ 4 │
│ e     │ 5 │ 5 │ 4 │ 3 │ 2 │ 2 │ 3 │ 4 │
│ n     │ 6 │ 6 │ 5 │ 4 │ 3 │ 3 │ 2 │ 3 │
└───────┴───┴───┴───┴───┴───┴───┴───┴───┘
编辑距离 = 3（替换k→s, 替换e→i, 插入g）

扩展编辑距离：
- Damerau-Levenshtein：增加转置操作（相邻字符交换）
  "ab" → "ba" 只需1次转置，而非2次替换
- Jaro-Winkler距离：适用于短字符串匹配
- 汉明距离：仅适用于等长字符串
```


## 噪声信道模型 (Noisy Channel)


```
核心思想：
┌─────────────────────────────────────────────────────┐
│  作者想写正确词 w → 经过噪声信道 → 观测到错误词 x     │
│                                                     │
│  P(w|x) = P(x|w) × P(w) / P(x)                    │
│                                                     │
│  P(x|w)：错误模型（Error/Channel Model）             │
│  "作者想写w但打成了x"的概率                          │
│  反映拼写错误的规律（同音、键盘距离等）               │
│                                                     │
│  P(w)：语言模型（Language Model）                    │
│  词w在语境中出现的先验概率                            │
│  反映语言的流畅性和合理性                             │
│                                                     │
│  求解：argmax_w P(x|w) × P(w)                       │
│  在候选词集合中找后验概率最大的词                      │
└─────────────────────────────────────────────────────┘

错误模型设计：
┌─────────────────────────────────────────────────────┐
│  1. 编辑距离作为错误模型                              │
│     P(x|w) ∝ α^edit_distance(x,w)                  │
│     α 通常取 0.8-0.95                               │
│     距离越小，概率越高                                │
│                                                     │
│  2. 基于混淆矩阵（Confusion Matrix）                │
│     统计真实语料中的错误替换模式                       │
│     P(substitute e→a) = count(e→a) / count(e)       │
│     如：e→a 高频，说明 "e" 和 "a" 容易混淆          │
│                                                     │
│  3. 拼音/发音模型（中文特有）                         │
│     P(x|w) 基于拼音相似度                            │
│     "在见"(zài jiàn) → "再见"(zài jiàn)             │
│     同音字替换是最常见的中文错误                      │
└─────────────────────────────────────────────────────┘

候选词生成：
┌─────────────────────────────────────────────────────┐
│  1. 编辑距离 ≤ 1 或 2 的词                           │
│     从词典中筛选相近的词                              │
│                                                     │
│  2. 同音/近音候选                                    │
│     通过拼音索引查找同音字                            │
│     "以经" → 拼音 yijing → 候选：已经、意境、意境    │
│                                                     │
│  3. 形近候选                                        │
│     部首/笔画相似的字                                 │
│     通过字形编码（五笔、笔画码）计算相似度             │
│                                                     │
│  4. BERT掩码预测                                     │
│     将疑似错误词替换为[MASK]，用BERT预测候选          │
│     利用上下文语义，不限于局部特征                     │
└─────────────────────────────────────────────────────┘
```


## 深度学习文本纠错


```
Seq2Seq 方法（端到端纠错）：
┌─────────────────────────────────────────────────────┐
│  输入：错误句子 "我觉的这个方案很好"                   │
│  输出：正确句子 "我觉得这个方案很好"                   │
│                                                     │
│  模型架构：                                          │
│  Encoder-Decoder (Transformer / BART / T5)          │
│                                                     │
│  优点：                                              │
│  - 端到端学习，不需要显式的错误检测                    │
│  - 可以处理多字、少字、替换等复杂错误                  │
│  - 利用预训练模型的语言知识                            │
│                                                     │
│  缺点：                                              │
│  - 可能过度纠正（将正确的词改错）                      │
│  - 需要大量平行纠错语料                                │
│  - 推理速度较慢                                       │
│                                                     │
│  代表模型：                                          │
│  - Soft-Masked BERT：先检测再纠正                    │
│  - BART-Corrector：BART微调                         │
│  - T5-Corrector：多任务纠错                          │
│  - MuCGEC：多任务中文纠错                            │
│  - ReaLiSe：基于拼音和字形的中文纠错                  │
└─────────────────────────────────────────────────────┘

检测+纠正 两阶段方法：
┌─────────────────────────────────────────────────────┐
│  阶段1：错误检测（序列标注）                           │
│  "我 觉 的 这 个 方 案 很 好"                         │
│   O  O  E  O  O  O  O  O  O  (E=错误)               │
│  使用BERT做token级二分类                              │
│                                                     │
│  阶段2：错误纠正（分类/生成）                          │
│  对检测到的错误位置，预测正确字符                      │
│  "的" → 从候选集{"得","地","的"}中选择                │
│  使用BERT的MLM能力或分类器                            │
│                                                     │
│  代表：BERT-MDC (Masked Detection and Correction)   │
└─────────────────────────────────────────────────────┘

数据增强与训练技巧：
┌─────────────────────────────────────────────────────┐
│  训练数据获取：                                       │
│  1. SIGHAN数据集：中文拼写检测评测数据               │
│  2. 自构造数据：对正确文本随机引入错误                │
│     - 同音字替换（使用拼音表）                        │
│     - 形近字替换（使用字形相似度）                    │
│     - 随机插入/删除                                  │
│  3. 真实用户数据：搜索引擎query纠错                   │
│                                                     │
│  训练策略：                                           │
│  - Focal Loss：解决正负样本不平衡                     │
│  - 对抗训练：FGM/PGD 提升鲁棒性                      │
│  - 多任务学习：同时训练检测和纠正                     │
│  - 前向/后向一致性：正向纠错和反向纠错结果一致        │
└─────────────────────────────────────────────────────┘
```


## 文本归一化 (Text Normalization)


```
文本归一化目标：将不同形式的文本统一为标准形式
┌─────────────────────────────────────────────────────┐
│  1. 大小写归一化                                      │
│     "NLP" → "nlp" 或 "Nlp"                          │
│     注意：缩写词（IBM）、人名（McDonald）需特殊处理    │
│                                                     │
│  2. 全角/半角转换                                     │
│     ＡＢＣ → ABC                                     │
│     １２３ → 123                                     │
│     ，。！ → ,.!                                     │
│                                                     │
│  3. 数字归一化                                        │
│     "100" vs "一百" vs "1百"                         │
│     "1st" → "第一"                                   │
│     "3.14" → 统一格式                                │
│                                                     │
│  4. 标点归一化                                        │
│     中文标点 → 英文标点（编程场景）                   │
│     统一引号风格："..." vs "..."                     │
│     去除多余标点！！！→ ！                            │
│                                                     │
│  5. 空白字符归一化                                    │
│     多个空格 → 单个空格                              │
│     去除首尾空白                                      │
│     Tab、换行符统一处理                               │
│                                                     │
│  6. Unicode归一化                                     │
│     NFC/NFKC/NFD/NFKD 形式统一                      │
│     "é" → "e" + "́" (NFD) vs "é" (NFC)              │
│     NFKC：兼容性归一化，最常用                       │
└─────────────────────────────────────────────────────┘

URL/邮箱/电话号码归一化：
┌─────────────────────────────────────────────────────┐
│  URL标准化：                                          │
│  "HTTP://WWW.Example.COM/" → "http://example.com/"  │
│  去除末尾斜杠、统一大小写                              │
│                                                     │
│  邮箱标准化：                                         │
│  "User@Gmail.COM" → "user@gmail.com"                │
│  （注意Gmail中点号和大小写可能有含义）                 │
│                                                     │
│  电话号码标准化：                                     │
│  "+86 138-0000-0000" → "13800000000"                │
│  "(010)12345678" → "01012345678"                    │
└─────────────────────────────────────────────────────┘
```


## 繁简转换


```
繁简转换的挑战：
┌─────────────────────────────────────────────────────┐
│  一对一映射（简单情况）：                              │
│  国 → 國    学 → 學    书 → 書                       │
│                                                     │
│  一对多映射（需要上下文消歧）：                        │
│  "发" → 發(fā, 发展) vs 髮(fà, 头发)               │
│  "干" → 幹(gàn, 干活) vs 乾(gān, 干燥)             │
│  "后" → 後(hòu, 后面) vs 后(hòu, 皇后)             │
│  "里" → 裡(lǐ, 里面) vs 里(lǐ, 公里)               │
│                                                     │
│  一对多约有1000+组，需要根据上下文判断                 │
└─────────────────────────────────────────────────────┘

转换方法：
┌─────────────────────────────────────────────────────┐
│  1. 词典映射 + 最大匹配                               │
│     构建繁简词典，使用正向/逆向最大匹配                 │
│     简单快速，但对歧义字处理不好                       │
│                                                     │
│  2. 基于语言模型的消歧                                │
│     对一对多情况，用n-gram或神经语言模型选择最佳转换   │
│     P(乾燥) > P(干燥) 在繁体语境中                    │
│                                                     │
│  3. 开源工具                                          │
│     OpenCC：最流行的中文转换库                        │
│     支持：简体↔繁体↔香港↔台湾                         │
│     考虑地区用字差异                                   │
│                                                     │
│     用法示例：                                        │
│     from opencc import OpenCC                        │
│     cc = OpenCC('s2t')  # 简体→繁体                  │
│     cc = OpenCC('t2s')  # 繁体→简体                  │
│     cc = OpenCC('s2tw') # 简体→台湾繁体              │
└─────────────────────────────────────────────────────┘

地区用字差异：
┌────────────┬──────────────┬──────────────┬──────────────┐
│ 简体        │ 台湾繁体      │ 香港繁体      │ 说明         │
├────────────┼──────────────┼──────────────┼──────────────┤
│ 信息        │ 訊息          │ 資訊          │ 用词不同     │
│ 软件        │ 軟體          │ 軟件          │ 用词不同     │
│ 网络        │ 網路          │ 網絡          │ 用词不同     │
│ 质量        │ 品質          │ 質素          │ 用词不同     │
│ 里（里面）  │ 裡            │ 裏            │ 字形不同     │
└────────────┴──────────────┴──────────────┴──────────────┘
```


## 领域术语标准化


```
问题定义：将同一概念的不同表达统一为标准术语
┌─────────────────────────────────────────────────────┐
│  医学领域：                                           │
│  "乙肝" = "乙型肝炎" = "Hepatitis B" = "HBV感染"    │
│  统一为标准ICD-10编码：B18.1                         │
│                                                     │
│  "心梗" = "心肌梗塞" = "心肌梗死" = "MI"            │
│  "新冠肺炎" = "COVID-19" = "2019-nCoV"             │
│                                                     │
│  法律领域：                                           │
│  "合同法" → 已并入《民法典》合同编                    │
│  需要更新为最新的法律条文编号                          │
│                                                     │
│  金融领域：                                           │
│  "沪深300" = "沪深三百" = "CSI 300"                 │
│  "市盈率" = "PE ratio" = "P/E"                      │
└─────────────────────────────────────────────────────┘

标准化方法：
┌─────────────────────────────────────────────────────┐
│  1. 术语词典 + 精确/模糊匹配                          │
│     构建领域术语表，包含同义词和标准形式               │
│     使用编辑距离/拼音相似度进行模糊匹配                │
│                                                     │
│  2. 术语链接 (Term Linking)                          │
│     类似实体链接，将文本中术语链接到标准概念           │
│     使用UMLS（医学）、MeSH等标准术语体系              │
│                                                     │
│  3. 基于嵌入的语义匹配                                │
│     将候选术语和标准术语编码为向量                     │
│     计算余弦相似度进行匹配                             │
│     可处理术语表述的多样性                             │
│                                                     │
│  4. LLM辅助标准化                                    │
│     利用大语言模型的医学/法律知识                      │
│     "将以下医学术语标准化为ICD-10编码"                │
│     优势：处理长尾和新术语                             │
│     劣势：可能产生幻觉，需要验证                       │
└─────────────────────────────────────────────────────┘

语音识别后处理标准化：
┌─────────────────────────────────────────────────────┐
│  ASR输出 → 文本标准化 → 最终文本                     │
│                                                     │
│  典型处理：                                          │
│  "一百二十三" → "123"（数字标准化）                   │
│  "百分之五十" → "50%"                                │
│  "三月十五号" → "3月15日"                            │
│  "两点三十分" → "2:30"                               │
│  "千克" → "kg"（度量衡）                             │
│                                                     │
│  SACL/ITN (Inverse Text Normalization)：             │
│  将口语化的文本形式转换为书面化形式                    │
│  半监督方法：利用规则+统计模型+深度学习               │
│                                                     │
│  开源工具：                                          │
│  - NeMo ITN (NVIDIA)                                │
│  - Sparrowhawk (Google)                             │
│  - WeTextProcessing                                  │
└─────────────────────────────────────────────────────┘

实践建议：
┌─────────────────────────────────────────────────────┐
│  1. 建立领域术语库并持续维护                           │
│  2. 多层级处理：规则兜底 + 模型优化                    │
│  3. 保留原文映射关系，便于溯源                         │
│  4. 人工审核：高风险领域（医疗/法律）必须有人工审核    │
│  5. 版本管理：术语标准随时间更新                       │
└─────────────────────────────────────────────────────┘
```


> **Note:** 文本纠错与标准化看似简单，实则涉及语言学、统计学和深度学习的综合应用。在实际系统中，通常采用规则兜底+模型优化的混合方案，并需要针对领域特点进行定制化处理。
