# 信息抽取深入


## 信息抽取深入


NLP关系抽取事件抽取


信息抽取从非结构化文本中提取结构化知识，包括实体关系、事件、开放信息等，是知识图谱构建的核心环节。


## 关系抽取概述


```
关系抽取目标：识别文本中实体对之间的语义关系
┌─────────────────────────────────────────────────────┐
│  "爱因斯坦出生于德国乌尔姆"                           │
│  → (爱因斯坦, 出生于, 德国乌尔姆)                     │
│                                                     │
│  "清华大学位于北京海淀区"                             │
│  → (清华大学, 位于, 北京海淀区)                       │
│                                                     │
│  关系类型示例（FewRel数据集部分关系）：                │
│  - 出生于 (place of birth)                          │
│  - 位于 (located in)                                │
│  - 创始人 (founded by)                              │
│  - 首都 (capital of)                                │
│  - 属于 (member of)                                 │
│  - 发现/发明 (discoverer/inventor)                  │
└─────────────────────────────────────────────────────┘

关系抽取范式：
┌──────────┬──────────────────────────────────────────┐
│ Pipeline │ 先NER，再对实体对做关系分类                 │
│          │ 步骤1：识别所有实体                         │
│          │ 步骤2：对每对实体判断关系类型               │
│          │ 优点：简单，模块化                         │
│          │ 缺点：NER错误传播，忽略了实体间的交互       │
├──────────┼──────────────────────────────────────────┤
│ Joint    │ 同时抽取实体和关系                          │
│          │ 共享编码器，联合解码                        │
│          │ 实体信息和关系信息相互增强                   │
│          │ 效果通常优于Pipeline                        │
├──────────┼──────────────────────────────────────────┤
│ End2End  │ 端到端生成式抽取                            │
│          │ 直接生成(头实体,关系,尾实体)三元组          │
│          │ UIE等统一框架                              │
└──────────┴──────────────────────────────────────────┘
```


## Pipeline 关系抽取


```
经典Pipeline流程：
┌─────────────────────────────────────────────────────┐
│  文本 → NER → 实体对枚举 → 关系分类 → 三元组         │
│                                                     │
│  步骤1：命名实体识别（NER）                           │
│  识别所有PER/LOC/ORG等实体                           │
│                                                     │
│  步骤2：实体对构建                                    │
│  对所有实体两两配对                                   │
│  n个实体 → n×(n-1)个实体对                           │
│  大量负样本（无关系的实体对）                          │
│                                                     │
│  步骤3：关系分类                                      │
│  对每个实体对判断关系类型                              │
│  输入：[CLS] 句子（实体标记）[SEP]                   │
│  例如：[E1]爱因斯坦[/E1]出生于[E2]德国[/E2]          │
│  输出：born_in 关系概率                               │
└─────────────────────────────────────────────────────┘

实体标记策略：
┌─────────────────────────────────────────────────────┐
│  1. 特殊标记（Entity Markers）：                      │
│     [E1] 爱因斯坦 [/E1] 出生于 [E2] 德国 [/E2]       │
│     取[E1]和[E2]的表示做分类                         │
│                                                     │
│  2. 实体类型嵌入（Typed Entity Marker）：             │
│     在标记中加入实体类型信息                           │
│     [E1:PER] 爱因斯坦 [/E1:PER]                      │
│     [E2:LOC] 德国 [/E2:LOC]                          │
│                                                     │
│  3. 高亮实体（Highlight）：                           │
│     不加特殊标记，通过位置信息识别                      │
│     用实体起始位置的token表示做分类                    │
│                                                     │
│  4. Span表示：                                       │
│     实体的开始和结束位置拼接                           │
│     entity_repr = [h_start; h_end]                  │
│     保留更完整的实体信息                               │
└─────────────────────────────────────────────────────┘

负采样策略：
- 随机负采样：随机选取无关系的实体对
- 全负样本：使用所有无关系实体对（数据不平衡）
- 对抗负采样：选取容易混淆的负样本
- 部分负采样：限制负样本比例（如1:5）
```


## Joint 联合抽取


```
方法一：参数共享（Shared Representation）：
┌─────────────────────────────────────────────────────┐
│  共享BERT编码器                                      │
│         ↓                                           │
│  ┌──────┴──────┐                                    │
│  NER解码器    RE解码器                               │
│  (序列标注)   (关系分类)                              │
│                                                     │
│  训练：联合损失 L = L_NER + λ × L_RE                │
│  推理：先NER，再RE（仍然有误差传播）                  │
│                                                     │
│  改进：添加交互机制                                   │
│  NER结果作为RE的输入特征                              │
│  RE结果反馈增强NER的实体表示                          │
└─────────────────────────────────────────────────────┘

方法二：序列标注框架 (TPLINKER等)：
┌─────────────────────────────────────────────────────┐
│  将联合抽取建模为token-pair标注                       │
│                                                     │
│  二维表格：(token_i, token_j) 的关系标签             │
│  - head_rel：token_i和token_j是否属于同一头实体      │
│  - tail_rel：token_i和token_j是否属于同一尾实体      │
│  - head_tail_rel：头实体到尾实体的关系类型            │
│                                                     │
│  "爱因斯坦出生于德国"                                │
│  (爱因, 德国) → born_in (头尾关系)                   │
│  (爱因, 斯坦) → 同一头实体                           │
│  (德, 国) → 同一尾实体                               │
│                                                     │
│  优点：避免了NER错误传播，端到端                      │
│  缺点：表格规模为O(n²)，长文本需要分段                │
└─────────────────────────────────────────────────────┘

方法三：Seq2Seq生成 (CopyRE, PURE等)：
┌─────────────────────────────────────────────────────┐
│  将三元组抽取转化为序列生成任务                        │
│                                                     │
│  输入：爱因斯坦出生于德国乌尔姆                        │
│  输出：(爱因斯坦, born_in, 德国乌尔姆)               │
│                                                     │
│  或者使用Copy机制：                                   │
│  - 从源文本中复制实体                                 │
│  - 生成关系类型                                       │
│  - 适合处理重叠三元组问题                              │
│                                                     │
│  CasRel (2020)：                                     │
│  1. 识别所有主体（subject）                           │
│  2. 对每个主体，识别其所有(关系,客体)对               │
│  基于BERT的跨度预测                                   │
│  解决了重叠三元组问题（同一实体参与多个三元组）        │
└─────────────────────────────────────────────────────┘
```


## 事件抽取 (Event Extraction)


```
事件定义：由触发词标识的特定类型事情的发生
┌─────────────────────────────────────────────────────┐
│  事件结构：                                          │
│  - 事件类型 (Event Type)：如"并购"、"地震"、"发射"   │
│  - 触发词 (Trigger)：标识事件发生的词                 │
│  - 论元 (Argument)：事件的参与者和属性               │
│                                                     │
│  示例：                                              │
│  "2023年3月，微软以687亿美元收购了动视暴雪"           │
│                                                     │
│  事件类型：Transaction/Business-Merger               │
│  触发词：收购                                        │
│  论元：                                              │
│    Buyer: 微软                                      │
│    Seller: 动视暴雪                                  │
│    Money: 687亿美元                                  │
│    Time: 2023年3月                                   │
└─────────────────────────────────────────────────────┘

事件抽取子任务：
┌─────────────────────────────────────────────────────┐
│  1. 事件检测 (Event Detection)                       │
│     识别触发词并判断事件类型                          │
│     "收购" → Business-Merger 事件                    │
│     本质：句子级+token级分类                          │
│                                                     │
│  2. 事件论元抽取 (Argument Extraction)               │
│     识别事件的各个论元及其角色                         │
│     对给定事件，找出"谁做了什么"                      │
│     本质：类似于SRL的角色标注                         │
│                                                     │
│  3. 事件关系检测                                     │
│     识别事件间的时序、因果等关系                      │
│     事件A先于事件B发生                                │
│     事件A导致事件B                                    │
└─────────────────────────────────────────────────────┘

事件抽取方法：
┌─────────────────────────────────────────────────────┐
│  1. 基于分类的方法                                    │
│     触发词识别：token级二分类                         │
│     事件类型分类：对触发词进行多分类                   │
│     论元角色分类：对每个token判断其角色               │
│     代表：DYGIE++, OneIE                             │
│                                                     │
│  2. 基于阅读理解的方法                                │
│     将论元抽取转化为问答任务                          │
│     "谁是收购方？" → 微软                            │
│     "收购金额是多少？" → 687亿美元                   │
│     代表：BERT-QA, BERT-MRC                         │
│                                                     │
│  3. 基于生成的方法                                    │
│     直接生成事件结构                                  │
│     输入：句子                                       │
│     输出：{type:Merger, trigger:收购,                │
│           args:{buyer:微软,...}}                     │
│     代表：DEGREE, DEGREE++, PaIE                    │
│                                                     │
│  4. 基于提示的方法                                    │
│     使用模板化的提示                                  │
│     "在这句话中，[事件类型]事件由[触发词]触发，       │
│      [角色1]是[实体1]，[角色2]是[实体2]"             │
│     代表：InstructUIE, Text2Event                   │
└─────────────────────────────────────────────────────┘
```


## 开放信息抽取 (Open IE)


```
Open IE vs 传统IE：
┌─────────────────────────────────────────────────────┐
│  传统IE：                                           │
│  - 预定义关系类型（如"位于"、"创始人"）               │
│  - 关闭的类别集合                                    │
│  - 精确但不灵活                                      │
│                                                     │
│  Open IE：                                          │
│  - 不预定义关系类型                                   │
│  - 从文本中直接抽取出自然语言关系                     │
│  - 灵活但可能不规范                                   │
│                                                     │
│  "爱因斯坦1879年出生于德国乌尔姆"                     │
│  传统IE：(爱因斯坦, born_in, 乌尔姆)                 │
│  Open IE：(爱因斯坦, 出生于, 德国乌尔姆)             │
│          (爱因斯坦, 出生于, 1879年)                  │
└─────────────────────────────────────────────────────┘

Open IE 系统演进：
┌─────────────────────────────────────────────────────┐
│  1. 基于规则/句法分析（早期）                         │
│     ReVerb：基于动词的抽取                           │
│     使用句法模式识别 (主语, 动词短语, 宾语)           │
│     限制：只关注动词，覆盖有限                        │
│                                                     │
│  2. 基于序列标注                                     │
│     按词标注：O-O-B1-I1-I1-O-B2-I2                  │
│     标出三元组的主语、谓语、宾语                       │
│     代表：OpenIE4, OpenIE6                           │
│                                                     │
│  3. 基于Seq2Seq生成                                  │
│     输入：句子                                       │
│     输出：多个三元组                                  │
│     代表：IMoJIE, DeepExtr                         │
│     使用BART/T5生成三元组文本                        │
│                                                     │
│  4. 基于Span提取                                     │
│     提取所有可能的文本span                           │
│     判断span之间的语义关系                            │
│     代表：SpanOIE                                   │
└─────────────────────────────────────────────────────┘

中文Open IE挑战：
┌─────────────────────────────────────────────────────┐
│  - 没有空格分词，需要先分词或使用字符级方法            │
│  - 动词短语边界模糊                                   │
│  - 主语/宾语经常省略                                  │
│  - 长距离依赖和复杂句式                               │
│                                                     │
│  工具推荐：                                          │
│  - PKUSeg + 自定义规则                               │
│  - BERT中文微调 + 三元组生成                         │
│  - GPT-4直接抽取（适合探索性分析）                   │
└─────────────────────────────────────────────────────┘
```


## 多跳推理信息抽取


```
多跳推理定义：需要跨越多个事实/文档才能推断出答案
┌─────────────────────────────────────────────────────┐
│  单跳问题：                                          │
│  "爱因斯坦出生于哪个国家？"                           │
│  直接从一个三元组回答：(爱因斯坦, 出生于, 德国)        │
│                                                     │
│  多跳问题：                                          │
│  "爱因斯坦出生城市的市长叫什么？"                     │
│  第1跳：(爱因斯坦, 出生于, 乌尔姆)                   │
│  第2跳：(乌尔姆, 市长是, Martin Ansbacher)           │
│  需要连接两个事实才能回答                             │
└─────────────────────────────────────────────────────┘

多跳推理方法：
┌─────────────────────────────────────────────────────┐
│  1. 基于路径的方法                                    │
│     在知识图谱中找到实体间的路径                       │
│     头实体 → 关系1 → 中间实体 → 关系2 → 尾实体      │
│     评分函数：score(h,r,t) = f(path(h,t))           │
│                                                     │
│  2. 基于图推理                                       │
│     构建问题相关的推理子图                            │
│     使用GNN在子图上传播和聚合信息                     │
│     代表：GraftNet, PullNet, GreaseLM              │
│                                                     │
│  3. 基于迭代推理                                     │
│     多轮推理，每轮更新实体表示和注意力分布            │
│     第1轮：找到第一跳相关实体                         │
│     第2轮：利用第一跳信息找到第二跳答案               │
│     代表：HOTPOTQA上的分解式推理                      │
│                                                     │
│  4. 基于LLM的推理链                                  │
│     Chain-of-Thought：让LLM分步推理                  │
│     "首先，爱因斯坦出生于乌尔姆。然后，乌尔姆的       │
│      市长是Martin Ansbacher。因此答案是..."          │
│     结合检索：每步检索相关知识                         │
└─────────────────────────────────────────────────────┘

多跳推理数据集：
┌─────────────────────────────────────────────────────┐
│  HotpotQA：基于维基百科的多跳问答                     │
│  - 支持推理：需要组合多个段落信息                     │
│  - 对比推理：比较两个实体的属性                       │
│  - 桥接推理：通过中间实体连接                         │
│                                                     │
│  MetaQA：基于电影知识图谱的多跳问答                  │
│  - 1-hop, 2-hop, 3-hop 问题                         │
│  - 测试不同跳数的推理能力                              │
│                                                     │
│  Complex WebQuestions：复杂开放域问答                │
│  - 真实用户问题                                       │
│  - 需要多跳推理和组合操作                              │
└─────────────────────────────────────────────────────┘

统一信息抽取框架 UIE (Universal IE)：
┌─────────────────────────────────────────────────────┐
│  统一各种IE任务为结构化生成                           │
│                                                     │
│  统一的Schema表示：                                   │
│  [Spot] 实体类型名称 [Asp] 属性名称                  │
│                                                     │
│  示例：                                              │
│  输入：微软以687亿美元收购动视暴雪                    │
│  Schema：[Spot] 事件 [Asp] 收购                      │
│  输出：{事件: {收购方:微软, 被收购方:动视暴雪,       │
│              金额:687亿美元}}                        │
│                                                     │
│  优点：一个模型处理NER/RE/EE/ASPA所有任务            │
│  代表：UIE, InstructUIE, USM                        │
└─────────────────────────────────────────────────────┘
```


> **Note:** 信息抽取从Pipeline方法发展到Joint端到端抽取，再到统一的生成式框架(如UIE)。多跳推理和开放信息抽取是解决复杂现实问题的关键能力。大语言模型为信息抽取带来了零样本和少样本的新范式。
