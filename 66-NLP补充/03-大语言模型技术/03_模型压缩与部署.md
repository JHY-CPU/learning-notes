# 模型压缩与部署


## 模型压缩与部署


NLP模型压缩部署


大语言模型的压缩与部署是将研究成果转化为实际产品应用的关键环节。


## 量化（Quantization）


量化将模型权重和/或激活从高精度（FP32/FP16）降低到低精度（INT8/INT4），减少存储和计算量。


```
量化类型对比：
┌────────────┬──────────┬──────────┬──────────────────────┐
│ 精度        │ 每参数位数 │ 模型大小  │ 特点                  │
├────────────┼──────────┼──────────┼──────────────────────┤
│ FP32       │ 32 bit   │ 100%     │ 原始精度              │
│ FP16/BF16  │ 16 bit   │ 50%      │ 训练/推理标准         │
│ INT8       │ 8 bit    │ 25%      │ 主流推理量化          │
│ INT4       │ 4 bit    │ 12.5%    │ 极致压缩，质量损失    │
│ GPTQ-4bit │ 4 bit    │ ~12.5%   │ 高质量 4-bit 量化     │
│ AWQ-4bit  │ 4 bit    │ ~12.5%   │ 激活感知 4-bit 量化   │
└────────────┴──────────┴──────────┴──────────────────────┘

训练后量化（PTQ）：
┌─────────────────────────────────────────────────────┐
│  不需要重新训练，直接对已有模型量化                    │
│                                                     │
│  1. 确定量化参数（scale, zero_point）                │
│     scale = (max - min) / (2^n - 1)                │
│     zero_point = round(-min / scale)               │
│                                                     │
│  2. 量化公式：                                      │
│     q = round(x / scale) + zero_point              │
│     x̂ = (q - zero_point) × scale                   │
│                                                     │
│  3. 校准（Calibration）：                           │
│     用少量数据确定最优的量化范围                       │
│     方法：Min-Max, MSE, Entropy, Percentile         │
└─────────────────────────────────────────────────────┘

GPTQ（量化感知训练后量化）：
- 逐层量化，利用 Hessian 信息补偿量化误差
- 支持 4-bit / 3-bit / 2-bit
- GPU 推理高效（结合 Triton kernel）

AWQ（激活感知权重量化）：
- 通过分析激活分布来保护重要的权重通道
- 对显著权重（salient weights）保留高精度
- 比 GPTQ 在 4-bit 下质量更好

GGUF/GGML：
- llama.cpp 项目的量化格式
- CPU 友好，支持多种量化级别
- Q4_K_M, Q5_K_M, Q8_0 等级别
```


## 剪枝（Pruning）


剪枝移除模型中不重要的权重或结构，减少计算量和存储需求。


```
剪枝分类：
┌─────────────────────────────────────────────────────┐
│ 非结构化剪枝（Unstructured Pruning）                 │
│ 移除单个权重（将小权重置零）                          │
│ 稀疏矩阵：大部分元素为 0                              │
│ 压缩率高，但需要稀疏计算硬件支持                      │
│                                                     │
│ 结构化剪枝（Structured Pruning）                     │
│ 移除整个神经元、注意力头、或层                        │
│ 模型结构保持规则，不需要特殊硬件                      │
│ 压缩率相对较低，但实用性更强                          │
│                                                     │
│ LLM 结构化剪枝方法：                                 │
│ - 移除不重要的注意力头                               │
│ - 缩减 FFN 中间维度                                  │
│ - 跳过某些 Transformer 层                            │
│ - 合并相似的层（LayerMerge）                         │
└─────────────────────────────────────────────────────┘

SparseGPT（非结构化剪枝）：
- 基于 Hessian 信息的逐层剪枝
- 支持 50-70% 的稀疏度而质量损失很小
- 结合量化可进一步压缩

LLM-Pruner（结构化剪枝）：
- 任务无关的结构化剪枝
- 基于梯度和参数重要性度量
- 剪枝后可微调恢复性能

Wanda（剪枝方法）：
- 权重 x 激活幅度作为重要性指标
- 不需要反向传播，速度快
- 非结构化剪枝，稀疏度 50%
```


## 知识蒸馏（Knowledge Distillation）


知识蒸馏用大模型（教师）的输出来训练小模型（学生），将大模型的知识迁移给小模型。


```
蒸馏框架：
┌─────────────────────────────────────────────────────┐
│  教师模型（Teacher）：大模型，如 GPT-4                │
│  学生模型（Student）：小模型，如 7B 参数模型           │
│                                                     │
│  1. 响应蒸馏（Response Distillation）                │
│     Teacher: prompt → response                      │
│     Student: prompt → 模仿 Teacher 的 response      │
│     方法：用 Teacher 生成的数据训练 Student            │
│     例：Alpaca, Vicuna（用 GPT-4 数据微调开源模型）  │
│                                                     │
│  2. 特征蒸馏（Feature Distillation）                 │
│     Student 模仿 Teacher 的中间层表示                 │
│     L_distill = Σ ||f_student - f_teacher||²        │
│     保留更多语义信息，但实现复杂                      │
│                                                     │
│  3. Logit 蒸馏（Logit Distillation）                │
│     Student 模仿 Teacher 的 logits 分布              │
│     L = α * CE(y, p_s) + (1-α) * KL(p_t, p_s)     │
│     温度参数 T 控制软标签的平滑度                     │
└─────────────────────────────────────────────────────┘

蒸馏实践：
- Phi 系列（微软）：用高质量数据训练小模型
- DistilBERT：6 层 BERT 蒸馏自 12 层
- TinyLlama：在大量数据上预训练小模型
- MiniCPM：端侧部署的小模型
```


## 推理引擎与部署框架


```
ONNX Runtime：
┌─────────────────────────────────────────────────────┐
│  ONNX（Open Neural Network Exchange）：              │
│  - 跨框架模型格式，支持 PyTorch/TensorFlow 等       │
│  - ONNX Runtime：高性能推理引擎                     │
│  - 支持 CPU/GPU/移动端                               │
│  - 图优化：算子融合、常量折叠                         │
│  - 量化支持：动态/静态量化                            │
│                                                     │
│  部署流程：                                          │
│  PyTorch Model → torch.onnx.export → .onnx         │
│  → ONNX Runtime 加载 → 推理                         │
└─────────────────────────────────────────────────────┘

TensorRT（NVIDIA）：
┌─────────────────────────────────────────────────────┐
│  NVIDIA GPU 专用推理优化引擎                         │
│                                                     │
│  优化技术：                                          │
│  - 层融合（Layer Fusion）：合并连续操作              │
│  - 精度校准（INT8/FP16）：自动选择最优精度           │
│  - 内核自动调优：选择最优 CUDA 内核                  │
│  - 动态 Tensor：支持动态 batch size                 │
│                                                     │
│  流程：                                              │
│  ONNX/PyTorch → TensorRT Builder → .engine         │
│  → TensorRT Runtime 加载 → 推理                    │
│                                                     │
│  性能：通常比原生 PyTorch 快 2-5 倍                 │
└─────────────────────────────────────────────────────┘

vLLM（高吞吐 LLM 推理）：
┌─────────────────────────────────────────────────────┐
│  专为大语言模型推理设计的高性能引擎                   │
│                                                     │
│  核心技术：                                          │
│  - PagedAttention：类似操作系统虚拟内存              │
│    按需分配 KV Cache，减少显存浪费                   │
│    显存利用率从 ~50% 提升到 ~95%                     │
│  - 连续批处理（Continuous Batching）：               │
│    新请求可随时加入批次，GPU 利用率高                │
│  - 投机解码（Speculative Decoding）：               │
│    小模型生成候选，大模型验证，加速 2-3 倍           │
│  - 量化支持：GPTQ, AWQ, FP8                        │
│                                                     │
│  API 兼容：OpenAI API 格式                          │
│  部署方式：python -m vllm.entrypoints.openai.api_server│
└─────────────────────────────────────────────────────┘

其他推理框架：
┌────────────────┬────────────────────────────────────┐
│ llama.cpp      │ C++ 实现，CPU/GPU 混合推理          │
│                │ GGUF 格式，支持 Metal/CUDA/Vulkan   │
├────────────────┼────────────────────────────────────┤
│ TGI (HuggingFace) │ 生产级推理服务，支持流式输出     │
├────────────────┼────────────────────────────────────┤
│ SGLang         │ 结构化生成加速，RadixAttention      │
├────────────────┼────────────────────────────────────┤
│ TensorRT-LLM   │ NVIDIA 官方 LLM 推理优化            │
├────────────────┼────────────────────────────────────┤
│ MLX (Apple)    │ Apple Silicon 专用推理框架          │
└────────────────┴────────────────────────────────────┘
```


> **Note:** vLLM 的 PagedAttention 是 LLM 推理的重大创新，通过虚拟内存思想大幅提高了 GPU 显存利用率和吞吐量。


## 部署策略与优化


```
多级部署策略：
┌─────────────────────────────────────────────────────┐
│  云端部署（Cloud）                                   │
│  - GPU 集群：A100/H100                              │
│  - 使用 vLLM / TensorRT-LLM                        │
│  - 模型并行（Tensor/Pipeline Parallelism）          │
│  - 适合：高并发、低延迟要求                          │
│                                                     │
│  边缘部署（Edge）                                    │
│  - NVIDIA Jetson, Apple M 系列                      │
│  - 4-bit 量化 + 剪枝                                 │
│  - llama.cpp / MLX                                  │
│  - 适合：隐私敏感、离线场景                          │
│                                                     │
│  移动端部署（Mobile）                                │
│  - INT4 量化，模型 < 4GB                            │
│  - ONNX Runtime Mobile / TensorFlow Lite           │
│  - 适合：端侧推理、隐私保护                          │
└─────────────────────────────────────────────────────┘

推理优化技巧：
┌─────────────────────────────────────────────────────┐
│  KV Cache 优化                                      │
│  - 分页存储（PagedAttention）                       │
│  - 量化 KV Cache（FP8/INT8）                       │
│  - 多查询注意力（MQA/GQA）减少 KV Cache             │
│                                                     │
│  批处理优化                                          │
│  - 连续批处理（Continuous Batching）                 │
│  - 动态批处理大小                                    │
│  - 请求调度策略（FCFS, SJF）                        │
│                                                     │
│  解码加速                                          │
│  - 投机解码（Speculative Decoding）                 │
│  - 并行解码（Medusa, Lookahead）                    │
│  - 流式输出（Streaming）                            │
│                                                     │
│  混合精度                                            │
│  - 注意力计算 FP16                                  │
│  - FFN 计算 INT8                                    │
│  - Embedding FP32                                   │
└─────────────────────────────────────────────────────┘
```
