# 长上下文技术


## 长上下文技术


LLM长上下文位置编码


扩展LLM的上下文窗口是从根本上提升模型能力的关键方向，涉及位置编码外推、注意力优化、推理加速等多个技术挑战。


## 位置编码外推问题


```
核心挑战：训练时的上下文长度有限，推理时需要更长的序列
┌─────────────────────────────────────────────────────┐
│  训练：4K tokens                                     │
│  推理：需要处理 128K / 1M+ tokens                    │
│                                                     │
│  问题：位置编码在训练范围外表现退化                    │
│  模型从未见过位置 5000, 10000 的编码                  │
│  注意力模式在超出训练长度时崩溃                        │
│                                                     │
│  位置编码类型：                                      │
│  - 绝对位置编码（sinusoidal）：不可外推               │
│  - 学习的位置嵌入：不可外推                           │
│  - 相对位置编码（RoPE/ALiBi）：有外推潜力            │
└─────────────────────────────────────────────────────┘

RoPE (Rotary Position Embedding)：
┌─────────────────────────────────────────────────────┐
│  Su et al., 2021 - 当前最流行的相对位置编码           │
│                                                     │
│  核心思想：将位置信息编码为旋转操作                    │
│  在复数空间中，将Q/K向量按位置进行旋转                 │
│                                                     │
│  公式：                                              │
│  f(x, m) = R(m,θ) × x                               │
│  R(m,θ)_i = rotation by m×θ_i                       │
│  其中 θ_i = 10000^(-2i/d)                           │
│                                                     │
│  直觉理解：                                          │
│  - 每个维度使用不同的旋转频率                          │
│  - 低频维度：编码全局位置（远距离依赖）                │
│  - 高频维度：编码局部位置（近距离依赖）                │
│  - 两个token的相对位置通过Q·K^T自然体现              │
│                                                     │
│  优势：                                              │
│  - 天然的相对位置编码                                 │
│  - 计算高效（只需旋转操作）                            │
│  - 支持长度外推（在一定范围内）                        │
│                                                     │
│  问题：                                              │
│  - 超出训练长度后，高频维度振荡加剧                    │
│  - 注意力分布变得不稳定                                │
└─────────────────────────────────────────────────────┘

RoPE 外推方法：
┌─────────────────────────────────────────────────────┐
│  1. 线性插值 (Position Interpolation, PI)：          │
│     缩放位置索引，使其落在训练范围内                    │
│     pos' = pos × (L_train / L_target)               │
│     例：训练4K，目标128K → pos' = pos / 32           │
│     优点：简单有效                                    │
│     缺点：位置分辨率降低，短文本质量可能下降            │
│                                                     │
│  2. NTK-aware 插值：                                 │
│     基于NTK (Neural Tangent Kernel) 理论             │
│     不均匀缩放：高频维度缩放少，低频维度缩放多         │
│     保留局部精度，牺牲全局精度                         │
│     效果通常优于线性插值                               │
│                                                     │
│  3. YaRN (Yet another RoPE extensioN)：              │
│     结合NTK插值和注意力温度调节                       │
│     分段处理不同频率的维度                             │
│     高频：不缩放（保留局部精度）                       │
│     中频：NTK插值                                     │
│     低频：线性插值（保证全局稳定性）                   │
│     额外加入注意力logits温度缩放                       │
│     从4K扩展到128K，性能损失最小                      │
│                                                     │
│  4. Dynamic NTK：                                    │
│     推理时动态调整缩放因子                             │
│     根据当前序列长度自动选择最优缩放                   │
│     不需要微调，即插即用                               │
│                                                     │
│  5. LongRoPE：                                       │
│     渐进式扩展策略                                    │
│     8K → 256K → 2048K，每步微调                      │
│     通过进化搜索找到最优的缩放因子                     │
│     实现2M token的上下文窗口                          │
└─────────────────────────────────────────────────────┘

ALiBi (Attention with Linear Biases)：
┌─────────────────────────────────────────────────────┐
│  Press et al., 2022                                  │
│  不使用位置嵌入，而是直接在注意力分数上加线性偏置       │
│                                                     │
│  Attention(Q,K,V) = softmax(QK^T + m × bias) × V   │
│  bias(i,j) = -|i-j|                                  │
│  m：每个头不同的斜率（等比序列）                       │
│                                                     │
│  优势：                                              │
│  - 天然支持外推（线性偏置不受序列长度限制）            │
│  - 近距离token注意力更强（距离越远偏置越负）           │
│  - 不需要额外的位置嵌入参数                            │
│                                                     │
│  局限：                                              │
│  - 只编码距离衰减模式，缺乏更复杂的位置关系            │
│  - 在某些任务上不如RoPE                               │
│  - BLOOM等模型使用，但主流模型多采用RoPE              │
└─────────────────────────────────────────────────────┘
```


## 长序列注意力优化


```
问题：Self-Attention的O(n²)复杂度是长序列的核心瓶颈
┌─────────────────────────────────────────────────────┐
│  n=4K    → 注意力矩阵 16M 元素                       │
│  n=128K  → 注意力矩阵 16B 元素                       │
│  n=1M    → 注意力矩阵 1T 元素                        │
│                                                     │
│  KV Cache大小：                                      │
│  KV Cache = 2 × n_layers × n_heads × head_dim × n  │
│  n=128K, 70B模型：约320GB KV Cache                  │
│  远超单卡显存！                                       │
└─────────────────────────────────────────────────────┘

Ring Attention：
┌─────────────────────────────────────────────────────┐
│  Blockwise Parallel Transformer的分布式扩展          │
│  核心思想：将序列分块分布在多个设备上                   │
│                                                     │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐                │
│  │ GPU1 │  │ GPU2 │  │ GPU3 │  │ GPU4 │              │
│  │块1   │  │块2   │  │块3   │  │块4   │              │
│  └──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘                │
│     │        │        │        │                     │
│     ▼        ▼        ▼        ▼                     │
│  KV块1 →  KV块2 →  KV块3 →  KV块4 (环形传递)        │
│                                                     │
│  每个GPU：                                           │
│  1. 计算本地Q与本地KV的注意力                         │
│  2. 接收邻居GPU的KV块，计算跨块注意力                 │
│  3. 环形通信，依次处理所有KV块                        │
│                                                     │
│  优点：                                              │
│  - 上下文长度 = 单设备长度 × 设备数                   │
│  - 计算和通信高度重叠                                 │
│  - 理论上支持无限长上下文（加设备即可）                │
│                                                     │
│  实现：                                              │
│  - Ring Attention for JAX/PyTorch                    │
│  - 与Flash Attention结合使用                          │
└─────────────────────────────────────────────────────┘

StreamingLLM：
┌─────────────────────────────────────────────────────┐
│  Xiao et al., 2023 - 支持无限长输入流                 │
│                                                     │
│  核心发现：                                          │
│  - 注意力汇聚(Attention Sink)：初始几个token获得大量注意力 │
│  这些token作为"锚点"，不需要实际语义信息              │
│                                                     │
│  方法：保留两种token                                 │
│  ┌──────────────────────────────────────────────┐   │
│  │ Attention Sink tokens：开头的几个token（固定） │   │
│  │ Sliding Window tokens：最近的N个token         │   │
│  │ 中间的token：丢弃                              │   │
│  │                                                │   │
│  │ [CLS][T1][T2] ... [T_{n-N}] ... [T_{n-1}][Tn]│   │
│  │  └────保留────┘   └─丢弃─┘   └───保留─────┘  │   │
│  └──────────────────────────────────────────────┘   │
│                                                     │
│  优点：                                              │
│  - KV Cache大小恒定（不随输入长度增长）               │
│  - 支持无限长输入                                     │
│  - 不需要重新训练                                     │
│  - 端到端推理速度稳定                                 │
│                                                     │
│  局限：                                              │
│  - 无法回忆窗口外的早期信息                            │
│  - 适合流式场景，不适合需要全量理解的任务              │
└─────────────────────────────────────────────────────┘

其他注意力优化：
┌─────────────────────────────────────────────────────┐
│  Flash Attention 2：                                 │
│  IO-aware的精确注意力计算                             │
│  减少HBM读写，利用SRAM缓存                           │
│  不是近似，是精确注意力的高效实现                      │
│  速度提升2-4x，内存节省5-20x                         │
│                                                     │
│  Paged Attention (vLLM)：                            │
│  KV Cache按页管理，类似操作系统虚拟内存               │
│  减少显存碎片，支持更大batch                          │
│                                                     │
│  Multi-Query/Grouped-Query Attention：               │
│  MQA：所有头共享一组KV                                │
│  GQA：分组共享KV                                     │
│  大幅减少KV Cache大小                                 │
│  LLaMA-2 70B使用GQA                                  │
│                                                     │
│  Sliding Window Attention：                          │
│  每个token只关注窗口内的token                         │
│  O(n×w)复杂度，w为窗口大小                            │
│  Mistral使用滑动窗口注意力                            │
│                                                     │
│  Sparse Attention / Strided Attention：              │
│  只计算部分注意力对                                   │
│  Longformer使用：局部窗口+全局token                   │
└─────────────────────────────────────────────────────┘
```


## 长上下文训练策略


```
渐进式长度扩展 (Progressive Training)：
┌─────────────────────────────────────────────────────┐
│  策略：从短序列开始，逐步增加训练长度                  │
│                                                     │
│  示例（LLaMA Long风格）：                            │
│  Phase 1: 4K tokens   (基础预训练)                   │
│  Phase 2: 16K tokens  (继续预训练)                   │
│  Phase 3: 64K tokens  (继续预训练)                   │
│  Phase 4: 128K tokens (指令微调)                     │
│                                                     │
│  每阶段配合位置编码缩放                               │
│  使用高质量长文本数据（书籍、论文、代码）              │
│                                                     │
│  关键要点：                                          │
│  - 数据质量比数量更重要                               │
│  - 需要足够的长文本数据                                │
│  - RoPE基频需要相应调整                               │
│  - 学习率需要精心调度                                  │
└─────────────────────────────────────────────────────┘

数据构造：
┌─────────────────────────────────────────────────────┐
│  长上下文训练数据来源：                               │
│                                                     │
│  1. 高质量长文档                                     │
│     - 书籍（Project Gutenberg等）                    │
│     - 学术论文（ArXiv）                               │
│     - 法律文档                                        │
│     - 代码仓库（完整文件）                             │
│                                                     │
│  2. 拼接短文本                                       │
│     将多个相关短文档拼接为长序列                       │
│     中间添加分隔符                                    │
│     需要保持上下文连贯性                               │
│                                                     │
│  3. 合成长上下文数据                                  │
│     "大海捞针"(Needle in a Haystack)类型：           │
│     在长文本中插入关键信息，训练模型找回               │
│     位置感知：关键信息在不同位置                       │
│                                                     │
│  4. 多文档推理数据                                   │
│     需要跨文档推理的问答数据                           │
│     HotpotQA风格的多跳推理                            │
└─────────────────────────────────────────────────────┘

稀疏注意力预训练：
┌─────────────────────────────────────────────────────┐
│  使用稀疏注意力模式预训练                              │
│                                                     │
│  BigBird (2020)：                                    │
│  随机注意力 + 滑动窗口 + 全局注意力                   │
│  复杂度从O(n²)降到O(n)                               │
│                                                     │
│  Longformer (2020)：                                 │
│  滑动窗口局部注意力 + 少量全局attention token         │
│  支持4K→16K的扩展                                     │
│                                                     │
│  注意：这些方法需要从头预训练                          │
│  对已有模型不适用（无法改变注意力模式）                │
└─────────────────────────────────────────────────────┘
```


## 长上下文评测 (RULER)


```
"大海捞针"测试 (Needle in a Haystack, NIAH)：
┌─────────────────────────────────────────────────────┐
│  测试设置：                                          │
│  在长文档的随机位置插入一句"针"(关键信息)             │
│  提问需要该信息才能回答的问题                          │
│  测试模型能否在不同位置找到该信息                      │
│                                                     │
│  示例：                                              │
│  长文档（100K tokens）中包含：                        │
│  "...[大量无关文本]...                               │
│   最好的颜色是紫色。                                   │
│   ...[大量无关文本]..."                               │
│                                                     │
│  问题："文中提到的最好的颜色是什么？"                  │
│  正确答案：紫色                                       │
│                                                     │
│  评估：在不同上下文长度 × 不同针位置的矩阵上测试       │
│                                                     │
│  GPT-4 128K测试结果：                                │
│  在某些位置和长度组合下出现失败                        │
│  "Lost in the Middle"现象：中间信息容易被忽略         │
└─────────────────────────────────────────────────────┘

RULER (Realistic long-context Understanding Evaluation)：
┌─────────────────────────────────────────────────────┐
│  Hsieh et al., 2024 - 更全面的长上下文评测基准        │
│                                                     │
│  NIAH的局限：                                        │
│  - 只测试简单的检索能力                               │
│  - 不需要真正的长距离推理                              │
│  - 不能反映真实应用场景                                │
│                                                     │
│  RULER的改进任务：                                   │
│  ┌─────────────────────────────────────────────┐    │
│  │ 1. 多针检索 (Multi-value NIAH)               │    │
│  │    在长文中插入多个关键信息，需全部找到        │    │
│  │                                              │    │
│  │ 2. 查询多针 (Query-aware NIAH)               │    │
│  │    只需找到与查询相关的部分信息                │    │
│  │                                              │    │
│  │ 3. 领域内问答 (In-domain QA)                  │    │
│  │    基于长文档的复杂问答                        │    │
│  │                                              │    │
│  │ 4. 领域外泛化 (OOD)                           │    │
│  │    测试未见过的文档类型和长度                  │    │
│  │                                              │    │
│  │ 5. 多跳推理 (Multi-hop)                       │    │
│  │    需要跨多个段落进行推理                      │    │
│  │                                              │    │
│  │ 6. 汇总任务 (Aggregation)                     │    │
│  │    从长文中汇总多条信息                        │    │
│  └─────────────────────────────────────────────┘    │
│                                                     │
│  主要发现：                                          │
│  - 大多数模型在128K长度上都有显著性能下降             │
│  - 声称支持长上下文的模型，实际有效上下文更短          │
│  - 推理型任务的性能下降比检索型更严重                  │
│  - Gemini-1.5-Pro和GPT-4表现相对较好                 │
└─────────────────────────────────────────────────────┘

其他长上下文基准：
┌─────────────────────────────────────────────────────┐
│  InfiniteBench：                                      │
│  支持100K+上下文的评测                                │
│  包含12个任务：检索、代码、数学、摘要等               │
│                                                     │
│  L-Eval：                                             │
│  长文档理解评测                                       │
│  包含考试、摘要、QA等任务                             │
│                                                     │
│  LongBench：                                         │
│  中英双语长上下文评测                                  │
│  多任务覆盖：单文档QA、多文档QA、摘要、代码等         │
│                                                     │
│  LV-Eval：                                            │
│  多粒度评测                                          │
│  从5K到256K的渐进式评估                               │
│                                                     │
│  实际有效上下文长度：                                 │
│  模型声称的上下文窗口 ≠ 实际有效长度                   │
│  需要在目标长度上进行实际测试                          │
└─────────────────────────────────────────────────────┘

长上下文应用：
┌─────────────────────────────────────────────────────┐
│  1. 整本书理解                                        │
│     一次性输入整本书进行问答和分析                      │
│                                                     │
│  2. 代码仓库分析                                      │
│     理解整个代码库的结构和逻辑                          │
│                                                     │
│  3. 长对话历史                                        │
│     保留完整的对话上下文                                │
│                                                     │
│  4. 多文档分析                                        │
│     同时分析多篇论文或报告                              │
│                                                     │
│  5. 法律文档审查                                      │
│     审阅长合同、法规等复杂文档                          │
│                                                     │
│  6. 聊天机器人记忆                                    │
│     长期记忆用户的偏好和历史                            │
└─────────────────────────────────────────────────────┘
```


> **Note:** 长上下文技术从位置编码外推(RoPE/YaRN/NTK)到注意力优化(Ring Attention/StreamingLLM)再到评测基准(RULER)，形成了完整的技术体系。实际应用中需要关注模型的有效上下文长度，而非仅看声称的窗口大小。
