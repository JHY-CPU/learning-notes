# 28_RFC与互联网标准

## 核心概念

### RFC（Request For Comments）

- **定义**：一系列以编号排定的备忘录，包含互联网相关的协议、过程、事件和概念
- **发布者**：IETF（互联网工程任务组）
- **特点**：开放、透明、可自由获取
- **RFC编号**：从RFC 1（1969年）开始递增，截至2024年已超过9500个

### RFC文档的分类

| 类别 | 说明 | 举例 |
|------|------|------|
| **标准类（Standards Track）** | 描述互联网标准协议 | RFC 793 (TCP) |
| **最佳当前实践类（BCP）** | 推荐的操作指南 | BCP 9 (标准制定过程) |
| **实验类（Experimental）** | 实验性技术 | RFC 9000 (QUIC早期) |
| **信息类（Informational）** | 提供一般信息 | RFC 1180 (TCP/IP教程) |
| **历史类（Historic）** | 已被淘汰的协议 | RFC 826 (ARP，部分) |

### 互联网标准的成熟度（当前简化版，RFC 6410后）

```
建议标准（Proposed Standard）
      |
      v
互联网标准（Internet Standard）
```

**历史变化**：早期有三级（提议标准→草案标准→互联网标准），2011年后取消草案标准级别。

**重要**：并非所有RFC都是标准。大部分RFC只是技术文档或实验性提案。只有标注"Standards Track"的RFC才可能成为互联网标准。

## 原理分析

### 著名的RFC

| RFC编号 | 内容 | 意义 |
|---------|------|------|
| RFC 791 | IP协议 | 互联网的核心 |
| RFC 793 | TCP协议 | 可靠传输的基础 |
| RFC 768 | UDP协议 | 不可靠但高效的传输 |
| RFC 792 | ICMP协议 | 差错报告和诊断 |
| RFC 2460 | IPv6 | 下一代互联网协议 |
| RFC 2616 | HTTP/1.1 | Web的基础（已被RFC 7230系列替代） |
| RFC 5246 | TLS 1.2 | HTTPS的安全基础 |
| RFC 8446 | TLS 1.3 | 最新安全传输协议 |

### "rough consensus and running code"

IETF的标准制定原则：
- **粗略共识（Rough Consensus）**：不需要所有人一致同意，但需要大多数人的支持。使用"哼声（Hum）"而非正式投票来判断共识
- **运行代码（Running Code）**：标准必须有实际的实现来证明可行性。"我们相信粗糙的共识和运行的代码"
- **开放过程**：任何人都可以参与IETF的邮件列表讨论，无需会员资格

### RFC的发布流程

```
个人或团体提出想法
      |
      v
互联网草案（Internet Draft）——有效期6个月，可多次更新
      |
      v
建议标准（Proposed Standard）——经过IETF审查
      |
      v
互联网标准（Internet Standard）——广泛实施并验证
      （拥有STD编号，如STD 7 = IP协议）
```

### 408常考要点

- RFC是互联网标准的文档形式，由IETF发布
- 并非所有RFC都是标准，大部分只是技术文档
- IETF的口号："rough consensus and running code"
- ISO标准与RFC标准的区别：ISO更正式严谨，RFC更灵活快速
- 互联网标准需要两个独立的、可互操作的实现

## 直观理解

- **RFC**像维基百科的技术版：任何人都可以查看和参与编辑
- **标准制定**像民主投票：不需要100%同意，大多数同意就行
- **running code**像"实践是检验真理的唯一标准"：纸上谈兵不算数，要能跑起来
- **RFC vs ISO标准**：RFC像开源社区的讨论帖，ISO像政府发布的红头文件

## 协议关联

- RFC 791、793、768等是学习IP、TCP、UDP时的参考原始文档
- RFC与ISO标准的区别：RFC更灵活、更快速，但权威性稍弱
- 408考试中对RFC的考查通常是概念性的选择题
- IETF制定TCP/IP标准，ISO制定OSI标准，IEEE制定802局域网标准
- RFC文档是理解协议细节的最权威来源，考试中偶尔会引用RFC编号
