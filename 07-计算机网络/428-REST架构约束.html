🏗️ REST 架构约束

REST 六大约束、无状态通信、统一接口、HATEOAS、REST 成熟度模型。

REST 是什么

REST (Representational State Transfer) 是 Roy Fielding 在 2000 年博士论文中提出的架构风格，定义了构建 Web API 的六大约束。

// ========== REST 六大约束 ========== // // 1. 客户端-服务器 (Client-Server) // 关注点分离,组件独立演化 // // 2. 无状态 (Stateless) // 每个请求包含所有必要信息 // 服务器不存储客户端状态 // // 3. 可缓存 (Cacheable) // 响应隐式或显式标记为可缓存/不可缓存 // 提高性能 // // 4. 统一接口 (Uniform Interface) // 标准化的资源操作方式 // // 5. 分层系统 (Layered System) // 客户端不知道是直接连服务器还是通过代理 // // 6. 按需代码 (Code on Demand) — 可选 // 服务器可传输可执行代码 (如 JavaScript) // ========== 统一接口的四个子约束 ========== // 1. 资源标识 (Resource Identification) // 每个资源有唯一 URI // // 2. 通过表示操作资源 (Representation) // 客户端通过资源的表示 (JSON/XML) 操作 // // 3. 自描述消息 (Self-descriptive Messages) // 每个消息包含足够的信息描述如何处理 // Content-Type 告诉客户端如何解析 // // 4. HATEOAS // 响应中包含相关资源的链接

无状态通信详解

// ========== 无状态的含义 ========== // 服务器不存储客户端上下文 // 每个请求都是独立的 // 所有必要信息都在请求中 // // 无状态的设计: // GET /api/orders?page=2 // Authorization: Bearer // 服务器从 token 解析用户身份 // 不需要 Session // // ❌ 有状态的设计: // POST /api/login → 服务端创建 Session // GET /api/orders → 服务端从 Session 获取用户 // 需要 Session 同步 (如果多台服务器) // ========== 无状态的优势 ========== // 1. 水平扩展简单 // 任何服务器都可以处理任何请求 // 无需共享 Session 存储 // // 2. 可见性 // 每个请求都包含完整信息 // 监控/调试更容易 // // 3. 可靠性 // 服务器故障不影响其他服务器 // 请求可被任何健康服务器处理 // // 4. 缓存友好 // 响应可被独立缓存 // 不依赖上下文 // ========== 无状态的代价 ========== // 1. 每次请求携带更多数据 (如每次带 token) // 2. 无法像有状态那样保持"连接" (需用其他机制)

REST 成熟度模型 (Richardson Maturity Model)

// ========== 成熟度模型 ========== // // Level 0: 沼泽 (The Swamp of POX) // 使用单一 URL, 所有操作用 POST // 类似 RPC // POST /api { "method": "getUser", "id": 1 } // // Level 1: 资源 (Resources) // 引入资源概念,每个资源有独立 URL // POST /api/users/1 { "method": "get" } // POST /api/users/1 { "method": "delete" } // 但操作仍然在请求体 // // Level 2: HTTP 动词 (HTTP Verbs) // 使用 HTTP 方法表达操作 // GET /api/users/1 (读取) // DELETE /api/users/1 (删除) // PUT /api/users/1 (更新) // 使用状态码表达结果 // 大多数"REST API"停在这一级 // // Level 3: 超媒体 (HATEOAS) // 响应中包含链接,引导客户端发现下一步操作 // GET /api/orders/123 // { // "id": 123, // "status": "pending", // "total": 299, // "links": { // "pay": { "href": "/api/orders/123/pay", "method": "POST" }, // "cancel": { "href": "/api/orders/123", "method": "DELETE" } // } // } // ========== 真正的 REST ========== // 大多数自称"RESTful"的 API 实际只到 Level 2 // 真正的 REST = Level 3 (HATEOAS) // 实际中 Level 3 很少实现,Level 2 已足够
🎯 大多数 API 自称"RESTful",实际只到 Richardson 成熟度 Level 2——正确使用 HTTP 方法和状态码就已经比很多 API 好了。真正的 REST (Level 3 HATEOAS) 在实际中很少完全实现。

REST 设计原则

// ========== REST 资源设计 ========== // 资源 = 名词,不是动词 // // ✅ 好的设计: // GET /users 获取用户列表 // POST /users 创建用户 // GET /users/123 获取单个用户 // PUT /users/123 更新用户 // DELETE /users/123 删除用户 // GET /users/123/orders 用户订单 // // ❌ 坏的设计: // GET /getUsers 动词在 URL // POST /createUser 动词化 // POST /deleteUser 动词化 // POST /userUpdate 动词化 // GET /getUserOrders 动词化 + 嵌套不稳定 // ========== 查询参数 ========== // 过滤: GET /users?role=admin&status=active // 排序: GET /users?sort=created_at&order=desc // 分页: GET /users?page=2&per_page=20 // 字段: GET /users?fields=id,name,email // ========== REST 与 RPC 对比 ========== // REST: 面向资源,CRUD 语义 // RPC: 面向操作,远程过程调用 // GraphQL: 灵活查询,客户端定义需要的数据 // // REST 适合: CRUD 为主,资源模型清晰的 API // RPC 适合: 操作复杂,非 CRUD 场景 // GraphQL 适合: 客户端需求差异大,减少 over-fetching
