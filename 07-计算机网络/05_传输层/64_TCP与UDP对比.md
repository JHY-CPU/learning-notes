# 65_TCP与UDP对比

## 核心概念

- **TCP**：面向连接、可靠、字节流、点对点
- **UDP**：无连接、不可靠、报文、支持组播/广播
- **两者都在IP之上，都使用端口号进行复用/分用**
- **选择TCP还是UDP取决于应用需求**

### 全面对比

| 特征 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠（确认、重传） | 不可靠 |
| 数据形式 | 字节流 | 报文（保留边界） |
| 通信方式 | 点对点 | 一对一/一对多/多对多 |
| 首部长度 | 最少20字节 | 固定8字节 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有（AIMD） | 无 |
| 有序性 | 保证按序到达 | 不保证 |
| 速度 | 较慢 | 快 |
| 开销 | 大 | 小 |
| 头部校验 | 必须（含伪首部） | IPv4可选，IPv6必须 |

## 原理分析

### 使用TCP的应用

| 应用 | 原因 |
|------|------|
| HTTP/HTTPS | 需要可靠传输网页内容 |
| FTP | 需要可靠传输文件 |
| SMTP/POP3/IMAP | 需要可靠传输邮件 |
| Telnet/SSH | 需要可靠的远程终端 |
| TLS | 需要可靠的加密通信 |

### 使用UDP的应用

| 应用 | 原因 |
|------|------|
| DNS | 简单查询，需要快速响应 |
| DHCP | 初始无IP，无法建立TCP连接 |
| SNMP | 简单网络管理，偶尔丢包可接受 |
| TFTP | 简单文件传输，应用层实现可靠性 |
| RTP/RTCP | 实时音视频，延迟敏感 |
| RIP | 路由信息协议 |

### UDP首部格式（8字节）
```
| 源端口(2B) | 目的端口(2B) |
| 长度(2B)   | 校验和(2B)   |
| 数据（可变）               |
```

### TCP首部格式（最少20字节）
```
| 源端口(2B) | 目的端口(2B) |
| 序号(4B)                      |
| 确认号(4B)                    |
| 首部长度(4b)|保留(6b)|标志(6b)|窗口(2B) |
| 校验和(2B) | 紧急指针(2B)    |
| 选项（可变）                  |
```

### TCP标志位
| 标志 | 全称 | 作用 |
|------|------|------|
| URG | Urgent | 紧急指针有效 |
| ACK | Acknowledgment | 确认号有效 |
| PSH | Push | 推送数据 |
| RST | Reset | 连接重置 |
| SYN | Synchronize | 同步序号（建立连接） |
| FIN | Finish | 发送完毕（关闭连接） |

### TCP vs UDP选择决策
```
需要可靠传输？
├── 是 → TCP
│   ├── 大文件传输
│   ├── 网页浏览
│   ├── 邮件
│   └── 远程登录
└── 否 → UDP
    ├── 实时音视频
    ├── DNS查询
    ├── 广播/组播
    └── 简单请求-响应
```

### 408常考要点
- TCP首部最少20字节，UDP固定8字节
- TCP使用IP协议号6，UDP使用17
- TCP面向连接、可靠、字节流、点对点
- UDP无连接、不可靠、报文、支持组播
- DNS使用UDP（大响应可能用TCP）
- DHCP使用UDP（初始无IP，无法建立TCP）
- HTTP/1.1使用TCP，HTTP/3使用QUIC(UDP)

## 直观理解

- **TCP像顺丰快递**：要签收、能追踪、丢了补发、但贵（慢）
- **UDP像平信**：直接投递、不管收没收到、但便宜（快）
- **TCP首部像"复杂包裹单"**——有各种控制信息
- **UDP首部像"简单明信片"**——只有基本地址信息

## 协议关联

- **与应用层**：应用选择TCP还是UDP影响整个应用架构
- **与网络层**：两者都封装在IP数据报中，IP协议字段6=TCP，17=UDP
- **408常见考法**：判断某应用使用TCP还是UDP及原因
- **QUIC协议**：在UDP上实现TCP的可靠传输功能
