# 49_TCP窗口管理

## 核心概念

- **发送窗口 = min(rwnd, cwnd)**，单位是字节
- **发送窗口内包含**：已发送未确认 + 允许发送但未发送
- **接收窗口**：接收方缓冲区剩余空间
- **窗口管理是TCP流量控制和拥塞控制的统一实现**

### 发送窗口的组成

```
|←──── 已确认 ────→|←── 已发送未确认 ──→|←── 可发送 ──→|←── 不可发送 ──→|
|                  |                    |              |               |
    已ACK的字节       发送窗口内已发      发送窗口内未发    窗口外
```

- **已确认**：已收到ACK的数据，可被滑动覆盖
- **已发送未确认**：已发送但还在等ACK
- **可发送（可用窗口）**：窗口内尚未发送的部分
- **不可发送**：超出窗口，需要等ACK后窗口右移

## 原理分析

### 窗口大小的动态调整

**rwnd的变化**：
- 接收方每发送一个ACK，都可能更新rwnd
- rwnd反映接收方缓冲区的实时状态
- rwnd=0时发送方暂停（有持续计时器防死锁）

**cwnd的变化**：
- 慢开始：指数增长
- 拥塞避免：线性增长
- 丢包：减半或归一

### 窗口与缓冲区

- 发送方缓冲区：存放应用层待发送的数据
- 接收方缓冲区：存放已接收但应用层未读取的数据
- 窗口大小反映缓冲区的可用空间

### 窗口滑动过程详解
**初始状态**：SND.UNA=1, SND.NXT=1, SND.WND=5000
- 可发送字节1~5000

**发送2000字节后**：SND.UNA=1, SND.NXT=2001, SND.WND=5000
- 已发未确认：1~2000
- 可发送：2001~5000

**收到ACK=2001后**：SND.UNA=2001, SND.NXT=2001, SND.WND=5000
- 窗口右移2000字节
- 可发送：2001~7000

### 接收窗口的管理
**接收窗口（RCV.WND）**：
- RCV.NXT：期望接收的下一个序号
- RCV.WND：接收窗口大小
- 可用窗口 = RCV.BUFF - (RCV.NXT - LastRead)

**延迟确认**：
- 接收方收到数据后不立即ACK
- 等待最多200ms或收到第二个段
- 减少ACK报文数量

### Nagle算法
- 发送方的小数据先缓冲
- 收到ACK或数据积累到MSS时才发送
- 减少网络中的小报文段
- 与延迟ACK配合可能产生延迟问题

### 糊涂窗口综合症（Silly Window Syndrome）
- 接收方每次通告很小的rwnd
- 发送方每次发送很小的数据
- 效率极低
- **解决方案**：
  - 接收方：Clark方法（rwnd < MSS时不更新）
  - 发送方：Nagle算法

### 408常考要点
- 发送窗口 = min(rwnd, cwnd)
- SND.UNA是窗口左边界，SND.WND是窗口大小
- 可用窗口 = SND.WND - (SND.NXT - SND.UNA)
- ACK推动窗口右移
- rwnd=0时发送零窗口探测报文
- 延迟确认和Nagle算法优化TCP效率

## 直观理解

- **窗口像一个"允许通行证"**：窗口内的数据可以通行，窗口外的数据需要等待
- **窗口大小像限流**：在高速公路上，只允许同时有N辆车在某个路段上
- **糊涂窗口像"挤牙膏"**——每次只挤一点点，效率很低

## 协议关联

- **与流量控制**：rwnd限制窗口大小
- **与拥塞控制**：cwnd限制窗口大小
- **与确认机制**：ACK推动窗口滑动
- **408常见考法**：计算窗口大小、分析窗口滑动过程
- **Nagle算法和延迟ACK**是优化TCP性能的常用技术
