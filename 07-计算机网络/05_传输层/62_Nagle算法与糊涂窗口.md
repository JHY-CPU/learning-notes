# 63_Nagle算法与糊涂窗口

## 核心概念

- **Nagle算法**：解决发送方"小数据大开销"问题，减少网络中的小报文段
- **糊涂窗口综合症（Silly Window Syndrome, SWS）**：接收方每次只腾出很小的空间，导致发送方发送很小的报文段
- **两者都是TCP性能优化机制**

### Nagle算法

**问题**：如果应用层每次写1个字节，TCP就会发送只有1字节数据的报文段，效率极低（40字节首部+1字节数据）。

**解决方案**：
1. 发送方收到第一个字节后立即发送
2. 之后的数据缓存在缓冲区
3. 等到以下任一条件满足时才发送：
   - 收到所有已发送数据的ACK
   - 缓冲区数据达到MSS大小
4. 结果：减少了网络中小报文段的数量

### 糊涂窗口综合症

**问题**：接收方每次只读取1字节，通告1字节的rwnd，导致发送方发送1字节的报文段。

**解决方案（Clark方案）**：
- 接收方：rwnd < min(MSS, 缓冲区/2) 时通告 rwnd = 0
- 发送方：限制自己每次至少发送MSS大小的数据

## 原理分析

### Nagle算法的实现

```python
if 有数据要发送:
    if 数据长度 >= MSS:
        立即发送
    elif 没有未确认的数据:
        立即发送
    else:
        缓存数据，等待ACK
```

### 糊涂窗口综合症的解决

**接收方Clark方案**：
- 只有当 rwnd 可以容纳一个完整的MSS，或者缓冲区空出一半时，才通告非零的rwnd

**发送方延迟确认**：
- 收到数据后延迟最多200ms再发送ACK
- 有机会在ACK中捎带rwnd更新

## 直观理解

- **Nagle像攒快递**：不收到一件就送一件，而是攒够了一起送，减少运输次数
- **糊涂窗口像小气的仓库管理员**：仓库刚腾出1平方米就通知发货1平方米的货，效率极低。应该等腾出足够空间再通知

## 协议关联

- **与TCP交互式应用**：Telnet、SSH等交互式应用特别需要Nagle算法
- **与延迟确认**：延迟确认配合Nagle算法使用效果更好
- **与窗口管理**：糊涂窗口综合症的解决涉及rwnd的通告策略
- **408常见考法**：Nagle算法的作用、糊涂窗口综合症的原因和解决方案
