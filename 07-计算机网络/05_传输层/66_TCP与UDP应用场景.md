# 67_TCP与UDP应用场景

## 核心概念

- **选择依据**：可靠性需求、实时性需求、数据量大小
- **TCP适合**：需要可靠传输、大数据量、对延迟不敏感的应用
- **UDP适合**：实时性要求高、小数据量、支持广播/组播的应用
- **也可以在UDP上构建可靠传输**（如QUIC、TFTP）

### 场景分类

| 场景 | 选择 | 原因 |
|------|------|------|
| 网页浏览 | TCP | 需要完整可靠传输 |
| 文件传输 | TCP | 大文件不能丢失 |
| 电子邮件 | TCP | 邮件内容不能丢失 |
| 远程登录 | TCP | 交互数据需要可靠 |
| 视频会议 | UDP | 实时性优先，丢包可容忍 |
| 在线游戏 | UDP | 低延迟优先 |
| DNS查询 | UDP | 小数据，快速响应 |
| 广播/组播 | UDP | TCP不支持 |
| IoT传感器 | UDP | 简单、低功耗 |

## 原理分析

### TCP适用场景的共同特点

1. **数据完整性关键**：文件、邮件不能丢失
2. **数据量大**：需要流量控制和拥塞控制
3. **交互式**：需要按序、无差错
4. **点对点通信**

### UDP适用场景的共同特点

1. **实时性关键**：延迟比丢包更不可接受
2. **数据量小**：单个报文就能完成交互
3. **简单查询**：问一句答一句
4. **需要广播/组播**

### 在UDP上构建可靠传输

- **QUIC协议**（HTTP/3）：在UDP上实现可靠传输、加密、多路复用
- **TFTP**：在UDP上实现简单的可靠文件传输
- **自定义协议**：游戏、视频等在UDP上实现部分可靠性

### 应用层协议使用的传输层
| 应用层协议 | 传输层 | 原因 |
|-----------|--------|------|
| HTTP/1.x | TCP | 网页内容必须完整 |
| HTTPS | TCP | 加密通信需要可靠 |
| FTP | TCP | 文件传输不能丢失 |
| SMTP | TCP | 邮件必须可靠 |
| POP3/IMAP | TCP | 邮件检索需要可靠 |
| DNS | UDP（大用TCP） | 查询小且要求快 |
| DHCP | UDP | 初始无IP地址 |
| SNMP | UDP | 简单管理，偶丢可接受 |
| TFTP | UDP | 简单文件传输 |
| RIP | UDP | 路由信息 |
| BGP | TCP | 路由信息不能丢失 |
| OSPF | IP直接 | 路由信息 |
| VoIP | UDP | 实时性优先 |
| 视频流 | UDP | 实时性优先 |
| 游戏 | UDP | 低延迟优先 |
| HTTP/3 | QUIC(UDP) | 减少连接建立延迟 |

### 为什么DNS主要用UDP？
1. DNS查询通常很小（几十字节）
2. 单个UDP报文即可完成查询-响应
3. UDP无连接建立开销，响应更快
4. DNS超时重传由应用层实现
5. 大响应（如区域传输）使用TCP

### 为什么HTTP/3使用UDP？
1. HTTP/2的TCP队头阻塞问题
2. QUIC在UDP上实现可靠传输+加密
3. 0-RTT连接建立
4. 连接迁移（IP变化不断连）

### 408常考要点
- HTTP、FTP、SMTP、Telnet使用TCP
- DNS、DHCP、SNMP、RIP使用UDP
- BGP使用TCP，OSPF直接用IP
- 实时应用（VoIP、视频）用UDP
- DNS主要用UDP，大响应用TCP
- HTTP/3使用QUIC（基于UDP）

## 直观理解

- **选择原则**：宁可慢也不能错的选TCP，宁可错也不能慢的选UDP
- **DNS像"查字典"**——翻一下就得到答案，不需要复杂的通信
- **HTTP像"订书"**——需要把所有页按顺序装订好

## 协议关联

- **与应用层协议**：每个应用层协议在设计时就决定了使用TCP还是UDP
- **与传输层**：TCP和UDP是传输层仅有的两个协议
- **408常见考法**：给定应用场景选择合适的传输层协议
- **QUIC是传输层的新趋势**：在UDP上实现TCP的功能
