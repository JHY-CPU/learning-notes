# 29_IntServ与DiffServ

## 核心概念

- **IntServ（综合服务）**：基于每流（per-flow）的QoS保障
  - 使用RSVP（资源预留协议）沿路径预留资源
  - 提供确定性的QoS保障
  - 每个流需要在路由器上维护状态
- **DiffServ（区分服务）**：基于每类（per-class）的QoS保障
  - 数据包标记DSCP（6位），路由器按类别转发
  - 不为单个流预留资源
  - 可扩展性好
- **IntServ vs DiffServ对比**：

| 特性 | IntServ | DiffServ |
|------|---------|---------|
| 粒度 | 每流（per-flow） | 每类（per-class） |
| 信令协议 | RSVP | 无（静态配置） |
| 路由器状态 | 每流状态 | 每类状态（少量） |
| 可扩展性 | 差（流数爆炸） | 好（类数固定） |
| QoS保障 | 确定性 | 统计性 |
| 适用范围 | 小型网络/边缘 | 大型网络/核心 |

- **DSCP字段**：IPv4 ToS字段的前6位，定义了多种PHB（逐跳行为）
  - EF（Expedited Forwarding）：加速转发，低延迟低抖动（语音）
  - AF（Assured Forwarding）：确保转发，4类3丢弃优先级（12种组合）
  - BE（Best Effort）：尽力而为（默认）
- **RSVP工作流程**：
  1. 接收方向发送方发送Resv消息（逆向预留）
  2. 沿途路由器检查是否有足够资源
  3. 有则预留，无则拒绝
  4. 发送方收到确认后开始发送
- **408重点**：IntServ和DiffServ的区别、DSCP概念

## 原理分析

### IntServ/RSVP

**特点**：
- 每个应用流在路由器上建立软状态（需要定期刷新）
- 提供两种服务：
  - **保证服务（Guaranteed Service）**：确定的带宽和延迟上界
  - **受控负载服务（Controlled Load）**：轻负载网络的性能

**局限性**：
- 核心路由器需要维护数百万个流的状态，开销巨大
- RSVP信令增加网络负担
- 不适用于大型互联网

### DiffServ

**工作方式**：
- 边缘路由器：分类、标记、整形、监管
- 核心路由器：只根据DSCP转发，不做复杂处理

```
边缘路由器（复杂）          核心路由器（简单）
分类 + 标记DSCP  →  根据DSCP快速转发  →  边缘路由器
整形 + 监管                    ↓
                         EF: 优先队列
                         AF: 加权队列
                         BE: 默认队列
```

### AF的丢弃优先级

AF类有4个队列（AF1-AF4），每个队列3个丢弃优先级：

| 类别 | 低丢弃 | 中丢弃 | 高丢弃 |
|------|-------|-------|-------|
| AF1 | AF11 | AF12 | AF13 |
| AF2 | AF21 | AF22 | AF23 |
| AF3 | AF31 | AF32 | AF33 |
| AF4 | AF41 | AF42 | AF43 |

拥塞时优先丢弃高丢弃优先级的包。

## 直观理解

- **IntServ**：像坐飞机——每个人有预定座位（每流预留），服务质量有保证
- **DiffServ**：像坐火车——分商务座/一等座/二等座（分类），但不为每个人单独预留
- **IntServ缺点记忆**："流太多，状态爆炸，核心路由器扛不住"
- **DiffServ优点记忆**："边缘复杂核心简单，可扩展性好"
- **DSCP标记**：像邮戳，EF=特快专递，AF=挂号信，BE=平信

## 协议关联

- **IntServ与RSVP**：RSVP是IntServ的信令协议
- **DiffServ与IP**：DSCP在IP头ToS字段中
- **与传输层**：TCP有拥塞控制，可响应QoS降级；UDP不响应
- **与流量整形**：DiffServ边缘路由器使用漏桶/令牌桶进行流量整形
- **408常见考法**：IntServ vs DiffServ对比、DSCP/PHB概念、RSVP作用
