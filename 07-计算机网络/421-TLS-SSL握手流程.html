🔐 TLS/SSL 握手流程

TLS/SSL 协议、握手过程、密码套件、数字证书、HTTPS 安全原理。

TLS 是什么

TLS (传输层安全) 是 HTTPS 的安全基础，在 TCP 之上提供加密、认证和完整性保护。

// ========== TLS 的三个目标 ========== // 1. 加密 (Encryption): 第三方无法窃听 // 2. 认证 (Authentication): 确认对方身份真实 // 3. 完整性 (Integrity): 数据未被篡改 // // ========== 术语澄清 ========== // SSL (Secure Sockets Layer): 最初由 Netscape 开发 // SSL 1.0 (1994) - 从未公开发布 // SSL 2.0 (1995) - 有安全缺陷 // SSL 3.0 (1996) - 已被弃用 (POODLE 攻击) // // TLS (Transport Layer Security): SSL 的继任者 // TLS 1.0 (1999) = SSL 3.1 // TLS 1.1 (2006) // TLS 1.2 (2008) - 目前最广泛使用 // TLS 1.3 (2018) - 最新版本,大幅改进性能和安全 // // 注意: 今天我们说的"SSL"实际都指 TLS // ========== 密码学基础 ========== // 对称加密: 同一个密钥加密解密 // 算法: AES, ChaCha20 // 优点: 速度快 // 缺点: 密钥分发问题 // // 非对称加密: 公钥加密,私钥解密 // 算法: RSA, ECDSA, Ed25519 // 优点: 无需共享密钥 // 缺点: 速度慢 (比对称慢 1000 倍) // // TLS 实际: 用非对称加密安全交换对称密钥 // 然后用对称密钥加密实际数据

TLS 1.2 握手流程

// ========== TLS 1.2 完整握手 ========== // // 客户端 服务器 // │ │ // │ 1. ClientHello │ // │ TLS 版本, 密码套件列表 │ // │ 随机数 (Client Random) │ // │ ────────────────────────────→ │ // │ │ // │ 2. ServerHello │ // │ 选择的版本, 选择的套件 │ // │ 随机数 (Server Random) │ // │ 服务器证书 (含公钥) │ // │ (可选: 请求客户端证书) │ // │ ←──────────────────────────── │ // │ │ // │ 3. 客户端验证证书 │ // │ 生成预主密钥 (Pre-master) │ // │ 用服务器公钥加密预主密钥 │ // │ ────────────────────────────→ │ // │ │ // │ 4. 服务器用私钥解密预主密钥 │ // │ 双方用 Client Random + │ // │ Server Random + Pre-master │ // │ 生成会话密钥 │ // │ │ // │ 5. 客户端: Finished (加密) │ // │ ────────────────────────────→ │ // │ │ // │ 6. 服务器: Finished (加密) │ // │ ←──────────────────────────── │ // │ │ // │ 开始加密数据传输 │ // │ 使用对称加密 (AES) + MAC │ // // 完整握手: 2 RTT (两个往返)
🔑 TLS 握手的灵魂在于——用慢速的非对称加密安全交换一个临时密钥,然后用快速的对称加密传输实际数据。既安全又高效。

TLS 1.3 改进

// ========== TLS 1.3 vs 1.2 ========== // // TLS 1.2 完整握手: 2 RTT // TLS 1.3 完整握手: 1 RTT // TLS 1.3 恢复握手: 0 RTT // // ========== TLS 1.3 握手 (1 RTT) ========== // 客户端 服务器 // │ │ // │ ClientHello + 密钥共享 │ // │ (客户端直接发送自己的密钥分享) │ // │ ───────────────────────────────→ │ // │ │ // │ ServerHello + 密钥共享 + 证书 │ // │ + Finished │ // │ (服务器同时发送密钥和完成消息) │ // │ ←─────────────────────────────── │ // │ │ // │ 客户端: Finished │ // │ ───────────────────────────────→ │ // │ │ // │ 开始加密数据传输 │ // // ========== TLS 1.3 主要改进 ========== // 1. 更快的握手: 1-RTT (完整), 0-RTT (恢复) // 2. 移除不安全算法: RSA 密钥交换, RC4, 3DES, CBC // 3. 前向安全性 (Forward Secrecy): 所有密钥交换使用 ECDHE // 即使私钥泄露,也无法解密已记录的历史数据 // 4. 简化密码套件: 只有 5 个,无法协商不安全选项 // 5. Encrypted Extensions: 更多握手信息被加密 // // ⚠️ 0-RTT 注意事项: // - 0-RTT 允许客户端在第一次往返时就发送数据 // - 但存在重放攻击风险 (Replay Attack) // - 只应用于幂等操作 (GET, 不要用于 POST/支付等) // - 服务器必须实现重放保护机制

密码套件与安全性

// ========== 密码套件 (Cipher Suite) ========== // 一组算法的组合,决定握手各阶段使用的算法 // // TLS 1.2 示例: // TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 // │ │ │ │ │ // │ │ │ │ └─ MAC/PRF 算法 // │ │ │ └───────── AEAD 对称加密 // │ │ └───────────────── 加密算法 // │ └───────────────────────── 证书签名算法 // └───────────────────────────────── 密钥交换算法 // // TLS 1.3 示例 (简化): // TLS_AES_128_GCM_SHA256 // TLS_AES_256_GCM_SHA384 // TLS_CHACHA20_POLY1305_SHA256 // // ========== 安全建议 ========== // TLS 版本: 最低 TLS 1.2, 推荐 TLS 1.3 // 证书签名: ECDSA > RSA (更高效) // 密钥交换: ECDHE (前向安全) // 对称加密: AES-GCM 或 ChaCha20-Poly1305 // 哈希: SHA-256 或 SHA-384 // ========== 前向安全性 (Forward Secrecy) ========== // ECDHE: 每次握手生成临时密钥对 // 私钥泄露不影响之前的会话 // 长期私钥只能用于签名,不能解密流量 // 所有现代 TLS 实现都应使用 ECDHE // ========== 证书验证步骤 ========== // 客户端收到服务器证书后,会进行以下验证: // // 1. 检查证书链 (Certificate Chain) // 服务器证书 → 中间 CA → 根 CA (信任锚) // // 2. 验证签名 // 用 CA 的公钥验证证书签名的有效性 // // 3. 检查有效期 // notBefore <= 当前时间 <=notAfter // // 4. 检查域名匹配 // 证书中的 Subject Alternative Name (SAN) // 必须包含访问的域名 // // 5. 检查吊销状态 // CRL (Certificate Revocation List) 或 // OCSP (Online Certificate Status Protocol) // // 6. 检查扩展密钥用途 // 确保证书用于服务器认证 (TLS Web Server Authentication) //==========HSTS (HTTP Strict Transport Security)==========// 防止 SSL Stripping 攻击 // 服务器响应头: // Strict-Transport-Security: max-age=31536000; includeSubDomains; preload // // 作用: // - 浏览器记住该站点只能用 HTTPS // - 自动将 HTTP 请求升级为 HTTPS // - 防止中间人降级攻击 // - preload: 可加入浏览器预加载列表
