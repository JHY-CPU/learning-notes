# HTTP3与QUIC

## 🚀 HTTP/3 与 QUIC

QUIC 协议设计、HTTP/3 优势、0-RTT 连接、连接迁移、TCP 队头阻塞的终结。

## HTTP/3 概述

HTTP/3 是下一代 HTTP 协议，基于 Google 设计的 QUIC 协议 (原为 HTTP/2 的后续)，将传输层从 TCP 改为 UDP。
```
// ========== 为什么 HTTP/3? ==========
//
// HTTP/2 的问题:
//   多路复用解决 HTTP 级别的队头阻塞
//   但 TCP 级别的队头阻塞仍然存在!
//
// TCP 队头阻塞示例:
//   一个连接上有流1, 流2, 流3 的数据交错
//   如果流1 的一个包丢失:
//   TCP 必须等待该包重传成功
//   流2 和流3 的数据虽然已收到,但被 TCP 阻塞
//   无法交给上层处理
//
// HTTP/3 的解决:
//   基于 UDP, 在用户空间实现可靠传输
//   每个流独立,一个流丢包不影响其他流
//
// ========== QUIC 协议栈 ==========
// HTTP/1.1/2:     HTTP/3:
// ┌─────────┐     ┌─────────┐
// │  HTTP/2  │     │  HTTP/3  │
// ├─────────┤     ├─────────┤
// │   TLS   │     │   QUIC  │
// ├─────────┤     │  (含TLS) │
// │   TCP   │     ├─────────┤
// ├─────────┤     │   UDP   │
// │   IP    │     │   IP    │
// └─────────┘     └─────────┘
```
## QUIC 核心特性
```
// ========== 1. 0-RTT 连接建立 ==========
// 首次连接: 1 RTT
//   QUIC 握手 + TLS 1.3 同时进行
//   比 TCP+TLS 1.3 (2 RTT) 快
//
// 再次连接: 0 RTT
//   使用之前连接的缓存凭据
//   直接发送 HTTP 请求!
//
// 对比:
//   HTTP/1.1 + TLS 1.3: 3 RTT (握手)
//   HTTP/2 + TLS 1.3:   2 RTT (握手)
//   HTTP/3 (首次):      1 RTT (握手)
//   HTTP/3 (恢复):      0 RTT (直接发数据)

// ========== 2. 无队头阻塞 ==========
// QUIC 在单个连接内支持多个独立的流
// 每个流独立可靠传输
//
// 包丢失场景:
//   TCP: 所有流都被阻塞,等待重传
//   QUIC: 只有丢包的流受影响,其他流继续
//
// 例: 流1一个包丢失,流2和流3照常传输

// ========== 3. 连接迁移 ==========
// TCP 连接由 (源IP, 源端口, 目的IP, 目的端口) 标识
// 网络切换 (WiFi→5G) → IP 变化 → TCP 连接断开
//
// QUIC 连接由 Connection ID 标识
// 网络切换 → IP 变化 → 通过 Connection ID 恢复
// 连接不间断!
//
// 实际应用: 手机从 WiFi 走到 5G 覆盖
//   TCP:  视频卡顿,重新连接
//   QUIC: 无缝切换,继续播放

// ========== 其他特性 ==========
// 内置 TLS 1.3: 默认加密 (不像 TCP 可选择明文)
// 用户空间实现: 无需内核更新,可快速迭代
// 前向纠错 (FEC): 通过冗余数据减少重传 (已移除)
// 多路流: 类似 HTTP/2 但无 TCP 队头阻塞
```
> **Note**: 📱 QUIC 的连接迁移特性对移动端体验至关重要——从 WiFi 切换到 5G 时不会断连。YouTube 使用 QUIC 后,视频重新缓冲率降低了 20% 以上。

## QUIC 传输细节
```
// ========== QUIC 包结构 ==========
// ┌─────────────────────────────┐
// │  公共头部 (Public Header)    │
// │  - Connection ID            │
// │  - 包号 (Packet Number)      │
// │  - 版本                      │
// ├─────────────────────────────┤
// │  帧 (Frame)                  │
// │  ├─ STREAM (流数据)          │
// │  ├─ ACK (确认)               │
// │  ├─ CRYPTO (加密握手)        │
// │  ├─ MAX_DATA (流量控制)      │
// │  └─ PING / CONNECTION_CLOSE │
// └─────────────────────────────┘

// ========== QUIC 流量控制 ==========
// 两个级别:
//   连接级别: 限制整个连接的总数据量
//   流级别:   限制单个流的数据量
//
// 自动调整: 基于 RTT 和丢包率
// 类似 TCP 但更精细

// ========== QUIC 丢包恢复 ==========
// 包号严格递增 (不重传重用)
// 更精确的 RTT 测量
// 更快的丢包检测 (基于时间,非 3 DUP ACK)
// 单调递增的包号避免重传歧义

// ========== 部署现状 ==========
// Google 服务: 自 2014 年起使用 QUIC (YouTube, Search, Gmail)
// Facebook/Meta: Instagram, Facebook App
// 苹果: iOS/macOS 原生支持
// CDN: Cloudflare (所有客户默认开启 HTTP/3)
//       Fastly, Akamai 支持
// 浏览器: Chrome, Firefox, Safari, Edge 均支持
// 截至 2024: 约 30% 网站支持 HTTP/3
```
## HTTP/3 部署与实践
```
// ========== 服务器配置 ==========
// Nginx (需编译 quic 模块):
//   server {
//       listen 443 quic reuseport;
//       http3 on;
//       add_header Alt-Svc 'h3=":443"; ma=86400';
//   }
//
// Caddy (默认支持 HTTP/3):
//   example.com {
//       root * /var/www
//       file_server
//   }
//
// Cloudflare: 自动为所有客户启用 HTTP/3
// 无需额外配置

// ========== 客户端检测 ==========
// 浏览器 DevTools:
//   协议列显示 "h3" 表示 HTTP/3
//   "h2" 表示 HTTP/2
//
// curl (需支持 quic):
//   $ curl --http3 https://example.com

// ========== 性能对比 ==========
// 场景: 从洛杉矶请求东京的页面 (RTT ≈ 100ms)
//
// HTTP/1.1 (6 连接): 首次 ~800ms, 后续 ~400ms
// HTTP/2 (1 连接):   首次 ~700ms, 后续 ~300ms
// HTTP/3 (首次):    首次 ~500ms (1 RTT 握手)
// HTTP/3 (恢复):    后续 ~100ms (0 RTT)
//
// 弱网环境 (10% 丢包率):
// HTTP/2: 吞吐量下降 50%+ (TCP 队头阻塞)
// HTTP/3: 吞吐量下降 < 20% (独立流)
```
