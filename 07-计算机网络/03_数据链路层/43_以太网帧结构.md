# 44_以太网帧结构

## 核心概念

- **以太网帧（Ethernet Frame）**：也叫MAC帧，是以太网的数据传输单位
- **IEEE 802.3帧格式**：与DIX Ethernet V2帧略有不同（考试一般考V2格式）

### Ethernet V2帧格式

```
| 前导码(7B) | SFD(1B) | 目的地址(6B) | 源地址(6B) | 类型(2B) | 数据(46-1500B) | FCS(4B) |
```

### 各字段详解

| 字段 | 长度 | 说明 |
|------|------|------|
| 前导码 | 7字节 | 10101010重复7次，用于时钟同步 |
| SFD | 1字节 | 10101011，帧起始定界符 |
| 目的MAC地址 | 6字节 | 接收方的MAC地址 |
| 源MAC地址 | 6字节 | 发送方的MAC地址 |
| 类型/长度 | 2字节 | 上层协议类型（如0x0800=IP） |
| 数据 | 46-1500字节 | 上层数据（IP数据报等） |
| FCS | 4字节 | CRC-32校验码 |

### 关键约束

| 参数 | 值 | 说明 |
|------|-----|------|
| 最大帧长 | 1518字节 | 不含前导码和SFD |
| 最小帧长 | 64字节 | 不含前导码和SFD |
| 最大数据载荷 | 1500字节 | MTU |
| 最小数据载荷 | 46字节 | 不足需填充 |
| 前导码+SFD | 8字节 | 物理层添加，不属于帧 |

### MTU与填充

- 如果数据不足46字节，需要**填充（Padding）**到46字节
- 填充的内容不参与校验
- 接收方通过上层协议的长度字段确定实际数据长度

### 考试重点

- 各字段长度必须记牢
- 最小/最大帧长的原因（CSMA/CD）
- 前导码和SFD**不属于帧**（物理层添加）

## 原理分析

### 帧长计算

$$\text{帧长} = 6 + 6 + 2 + \text{数据长度} + 4 = 18 + \text{数据长度}$$

- 数据长度 $L$ 的范围：$46 \leq L \leq 1500$
- 帧长范围：$64 \leq \text{帧长} \leq 1518$

### CRC-32校验范围

FCS校验范围：目的地址 + 源地址 + 类型 + 数据（不含前导码和FCS本身）

### MAC地址格式

```
| 组织唯一标识OUI(3字节) | 网卡标识(3字节) |
|<-------- 6字节 -------->|
```

- 48位（6字节）
- 表示为12个十六进制数字，如 `00:1A:2B:3C:4D:5E`

### 地址类型

| 类型 | 特征 | 示例 |
|------|------|------|
| 单播 | 第1字节最低位=0 | 00:1A:2B:3C:4D:5E |
| 多播 | 第1字节最低位=1 | 01:00:5E:xx:xx:xx |
| 广播 | 全F | FF:FF:FF:FF:FF:FF |

## 直观理解

- **以太网帧**就像一个**快递包裹**：有收件人地址（目的MAC）、寄件人地址（源MAC）、包裹内容说明（类型）、实际货物（数据）、验货标签（FCS）
- **前导码**就像"预备铃"：告诉接收方"数据要来了，准备好时钟同步"

## 协议关联

- 以太网帧封装的是IP数据报（类型字段=0x0800）
- 也可能封装ARP（类型=0x0806）
- 帧结构与PPP帧、HDLC帧的对比是常见考点
- 交换机根据目的MAC地址进行转发
