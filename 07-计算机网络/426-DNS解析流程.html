DNS 是什么

DNS (域名系统) 将人类可读的域名转换为机器可读的 IP 地址——互联网的"电话簿"。

// ========== DNS 的核心功能 ========== // 域名 → IP 地址映射 // 反向解析: IP 地址 → 域名 // 负载均衡: 一个域名对应多个 IP // 服务发现: SRV 记录查找服务位置 // // ========== DNS 层级结构 ========== // 根域名服务器 (Root): // 全球 13 组根服务器 (字母 A-M) // 由 ICANN 管理 // 存储顶级域名的 NS 记录 // 任何解析的起点 // // 顶级域名服务器 (TLD): // 管理 .com .org .cn 等顶级域名 // 存储二级域名的 NS 记录 // Verisign 管理 .com 和 .net // // 权威域名服务器 (Authoritative): // 存储具体域名的 DNS 记录 // 如 ns1.example.com // 由域名所有者配置 // ========== DNS 记录类型 ========== // A: 域名 → IPv4 地址 // AAAA: 域名 → IPv6 地址 // CNAME: 域名 → 另一个域名 (别名) // MX: 邮件服务器 // NS: 域名服务器 // TXT: 任意文本 (如 SPF, DKIM) // SOA: 区域权威信息 // SRV: 服务位置 // PTR: IP → 域名 (反向)

递归查询 vs 迭代查询

// ========== 完整 DNS 查询流程 ========== // // 用户输入 www.example.com // │ // ▼ // 1. 浏览器缓存 → 有则直接使用 // │ 未命中 // ▼ // 2. 操作系统缓存 (hosts 文件也在此) // │ 未命中 // ▼ // 3. 递归 DNS 服务器 (通常由 ISP 提供) // │ // ▼ // 4. 根域名服务器 (递归查询起点) // ← 返回 .com TLD 服务器地址 // │ // ▼ // 5. .com TLD 服务器 // ← 返回 example.com 权威服务器地址 // │ // ▼ // 6. example.com 权威服务器 // ← 返回 www.example.com 的 IP 地址 // │ // ▼ // 7. 递归服务器缓存并返回结果给客户端 // │ // ▼ // 8. 客户端收到 IP 地址,建立连接 // ========== 迭代 vs 递归 ========== // 递归查询 (客户端→递归服务器): // 客户端问一个服务器,服务器负责查完并返回 // 客户端只需发一次请求 // // 迭代查询 (递归服务器→其他服务器): // 递归服务器依次向根/TLD/权威查询 // 每个服务器返回下一级服务器的地址 // 客户端不参与这个过程 // ========== 查询耗时 ========== // 无缓存: 10-50 ms (递归服务器已缓存) 或 // 100-500 ms (逐级查询) // 有缓存: < 1 ms (直接返回)

DNS 缓存

// ========== 缓存层级 ========== // 浏览器 DNS 缓存: // Chrome: chrome://net-internals/#dns // 默认缓存 60 秒 (可通过 DNS 记录的 TTL 覆盖) // // 操作系统 DNS 缓存: // Windows: ipconfig /displaydns // macOS/Linux: 取决于 nscd/systemd-resolved 配置 // // hosts 文件 (优先级最高): // Windows: C:\Windows\System32\drivers\etc\hosts // Linux/macOS: /etc/hosts // 格式: 127.0.0.1 localhost // // 递归 DNS 服务器缓存: // 共享给所有使用该服务器的用户 // 如 8.8.8.8 (Google), 1.1.1.1 (Cloudflare) // ========== TTL (Time To Live) ========== // DNS 记录缓存时间 (秒) // // 不同用途的 TTL: // A 记录 (生产): 300-3600 (5-60 分钟) // A 记录 (实验): 60-300 (1-5 分钟) // CNAME: 300-3600 // MX: 3600-86400 // TXT: 3600-86400 // NS: 86400-172800 (1-2 天) // // 低 TTL = 变更传播快,但查询多 // 高 TTL = 查询少,但变更慢 // 变更前: 降低 TTL,等待生效,再改记录
⏱️ 修改 DNS 记录后需要等待 TTL 过期才能全球生效。最佳实践: 计划变更前 48 小时降低 TTL 到 60 秒,变更完成后恢复。

DNS 调试命令

// ========== dig 命令 (最强大) ========== // $ dig example.com // ;; ANSWER SECTION: // example.com. 300 IN A 93.184.216.34 // // 查询特定记录类型: // $ dig example.com A // $ dig example.com MX // $ dig example.com CNAME // // 不使用缓存 (直接查权威): // $ dig example.com +trace // // 指定 DNS 服务器: // $ dig @8.8.8.8 example.com // $ dig @1.1.1.1 example.com // ========== nslookup (更简单) ========== // $ nslookup example.com // $ nslookup example.com 8.8.8.8 # 指定服务器 // $ nslookup -type=mx example.com // ========== 其他 ========== // 清除 DNS 缓存: // Windows: ipconfig /flushdns // macOS: sudo dscacheutil -flushcache // Linux: sudo systemd-resolve --flush-caches // // 检查 hosts 文件: // cat /etc/hosts // // 在线 DNS 查询: // https://dns.google (Google) // https://1.1.1.1 (Cloudflare) // ========== DNS 传播检查 ========== // 从全球不同位置查询某个域名的解析结果: // https://www.whatsmydns.net // https://dnschecker.org
