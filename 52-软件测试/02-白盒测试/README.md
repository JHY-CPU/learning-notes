# 白盒测试

## 一、白盒测试原理

### 1.1 定义

白盒测试（White Box Testing / Glass Box Testing / Structural Testing）是一种基于程序内部结构的测试方法。测试人员需要了解代码的实现细节，根据程序的控制流和数据流设计测试用例。

### 1.2 核心思想

- 检查程序的**内部逻辑结构**，而非仅关注输入输出
- 目标是确保代码的每一条路径、每一个分支都被覆盖
- 通常由开发人员在单元测试阶段执行

### 1.3 基本前提

- 可以访问源代码
- 了解程序的内部逻辑
- 能够分析控制流和数据流

---

## 二、语句覆盖（Statement Coverage）

### 2.1 定义

语句覆盖要求程序中的**每条可执行语句至少被执行一次**。

### 2.2 计算公式

$$语句覆盖 = \frac{已执行的语句数}{可执行语句总数} \times 100\%$$

### 2.3 示例

```c
if (a > 0 && b > 0) {
    c = a + b;      // 语句1
}
c = c / a;          // 语句2
```

测试用例：`a=1, b=1` 即可覆盖语句1和语句2。

### 2.4 局限性

- 无法发现逻辑运算中的错误
- 不能保证所有分支都被测试
- 是最弱的覆盖标准之一

---

## 三、判定覆盖（Branch/Decision Coverage）

### 3.1 定义

判定覆盖要求程序中的**每个判定（if、while、for等）的真分支和假分支至少各执行一次**。

### 3.2 计算公式

$$判定覆盖 = \frac{已覆盖的判定结果数}{判定结果总数} \times 100\%$$

### 3.3 示例

```c
if (a > 0 && b > 0) {
    c = a + b;
}
c = c / a;
```

需要两个测试用例：
- `a=1, b=1` → 判定为真
- `a=-1, b=1` → 判定为假

### 3.4 特点

- 比语句覆盖更强
- 满足判定覆盖一定满足语句覆盖
- 但仍无法覆盖复合条件中各子条件的所有取值

---

## 四、条件覆盖（Condition Coverage）

### 4.1 定义

条件覆盖要求每个判定中的**每个简单条件（子条件）的取值至少各出现一次真和假**。

### 4.2 示例

```c
if (a > 0 && b > 0) {
    c = a + b;
}
```

子条件：
- `C1: a > 0`
- `C2: b > 0`

需要满足 C1=true、C1=false、C2=true、C2=false。

测试用例：`a=1,b=1`（C1=T, C2=T）和 `a=-1,b=-1`（C1=F, C2=F）。

### 4.3 局限性

- 条件覆盖**不一定**满足判定覆盖
- 例如：`a=1,b=-1`（T,F）和 `a=-1,b=1`（F,T）满足条件覆盖但两个判定结果都为假

---

## 五、判定-条件覆盖（Decision-Condition Coverage）

### 5.1 定义

同时满足**判定覆盖和条件覆盖**：每个子条件的取值至少各出现一次真和假，同时每个判定的结果也至少出现一次真和假。

### 5.2 示例

```c
if (a > 0 && b > 0) {
    c = a + b;
}
```

测试用例：
- `a=1, b=1` → C1=T, C2=T, 判定=T
- `a=-1, b=-1` → C1=F, C2=F, 判定=F

### 5.3 特点

- 仍不能覆盖所有条件组合
- 实践中常用的标准

---

## 六、条件组合覆盖

### 6.1 定义

要求覆盖**所有简单条件取值的组合**。对于 n 个子条件的判定，需要 2^n 个测试用例。

### 6.2 示例

```c
if (a > 0 && b > 0) {  // 2个子条件
    c = a + b;
}
```

需要 2^2 = 4 个测试用例：

| a>0 | b>0 | 判定结果 |
|-----|-----|----------|
| T   | T   | T        |
| T   | F   | F        |
| F   | T   | F        |
| F   | F   | F        |

### 6.3 局限性

- 当子条件较多时，测试用例数量呈指数增长（组合爆炸）
- 实际应用中可能不切实际

---

## 七、路径覆盖

### 7.1 定义

要求程序中的**每条独立的执行路径至少被执行一次**。

### 7.2 路径分析

```c
if (a > 0) {          // P1
    if (b > 0) {      // P2
        c = a + b;    // 路径1: P1真→P2真
    }
}
```

独立路径：
1. P1为假
2. P1为真，P2为假
3. P1为真，P2为真

### 7.3 特点

- 是最强的覆盖标准之一
- 路径数量可能非常大（循环导致路径爆炸）
- 实际中通常结合基路径测试使用

---

## 八、MC/DC（修正条件判定覆盖）

### 8.1 定义

Modified Condition/Decision Coverage，是 DO-178B/C（航空软件标准）要求的覆盖标准。要求每个条件都能**独立地影响判定结果**。

### 8.2 核心要求

对于判定中的每个子条件，存在一组测试用例使得：
- 仅该子条件的值发生变化
- 其他所有子条件的值保持不变
- 判定的结果随之改变

### 8.3 示例

```c
if (A || (B && C)) {
    // ...
}
```

| 测试 | A | B | C | 结果 | 说明 |
|------|---|---|---|------|------|
| 1    | F | T | T | T    | 基础用例 |
| 2    | T | T | T | T    | A独立影响（1→2） |
| 3    | F | F | T | F    | B独立影响（1→3） |
| 4    | F | T | F | F    | C独立影响（1→4） |

n 个条件只需要 **n+1** 个测试用例，远少于条件组合覆盖的 2^n。

## 九、基路径测试（McCabe 圈复杂度）

### 9.1 控制流图

将程序转换为有向图：
- **节点**：基本块（顺序执行的语句序列）
- **边**：控制转移

### 9.2 圈复杂度（Cyclomatic Complexity）

$$V(G) = E - N + 2P$$

其中：
- E = 边数
- N = 节点数
- P = 连通分量数（通常为1）

简化公式：$V(G) = 判定节点数 + 1$

### 9.3 应用

- 圈复杂度 = 独立路径数 = 最少测试用例数
- 圈复杂度 ≤ 10：结构简单
- 圈复杂度 10-20：中等复杂度
- 圈复杂度 > 20：结构复杂，应考虑重构

### 9.4 基路径集

选取一条基准路径，每次只改变一个判定的分支，得到一组独立的路径。

## 十、数据流测试

### 10.1 定义

基于变量的**定义（define）和使用（use）**关系设计测试用例。

### 10.2 核心概念

- **d-对**：变量的定义点到使用点的路径
- **定义**：变量被赋值
- **使用**：
  - **计算使用（c-use）**：出现在赋值语句的右值
  - **判定使用（p-use）**：出现在判定条件中

### 10.3 覆盖准则

- **全定义覆盖**：每个变量定义至少到达一个使用点
- **全c-use覆盖**：每个变量定义到达所有c-use
- **全p-use覆盖**：每个变量定义到达所有p-use
- **全使用覆盖**：每个定义到达所有使用点

---

## 十一、控制流图

### 11.1 基本块划分

- 基本块只有一个入口和一个出口
- 除最后一条指令外没有跳转
- 除第一条指令外没有跳入

### 11.2 常见控制结构的流图

```
顺序结构:  A → B
分支结构:  A → B
       ↘ C
循环结构:  A → B → (回到A或退出)
```

### 11.3 工具支持

- Graphviz、CodeQL、Understand 等工具可以自动生成控制流图

---

## 十二、白盒测试的优缺点

### 优点

- 可以发现代码中的隐藏路径和逻辑错误
- 覆盖率可量化，便于度量测试完整性
- 有助于发现死代码和不可达代码
- 能够进行最优化的测试用例设计

### 缺点

- 需要了解源代码，对测试人员技术要求高
- 无法检测遗漏的功能需求
- 路径数量可能巨大（循环、递归），难以完全覆盖
- 不验证程序与需求的一致性
- 维护成本随代码变更而增加
