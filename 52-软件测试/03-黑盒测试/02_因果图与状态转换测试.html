因果图与状态转换测试

一、因果图（Cause-Effect Graph）

因果图是一种利用图解法分析输入条件（因）的各种组合以及每种组合对应的输出结果（果）的黑盒测试方法。

1.1 基本符号

符号名称含义表达式
∧与（AND）所有原因都成立，结果才成立E = C1 ∧ C2
∨或（OR）任一原因成立，结果就成立E = C1 ∨ C2
¬非（NOT）原因取反E = ¬C1
⊕异或（XOR）有且仅有一个原因成立E = C1 ⊕ C2

1.2 约束符号

约束符号含义
E约束（互斥）E原因之间最多一个成立
I约束（包含）I原因之间至少一个成立
O约束（唯一）O原因之间有且仅有一个成立
R约束（要求）R原因a成立时原因b也必须成立
M约束（强制）M结果a成立时结果b必须为假

1.3 因果图方法步骤

  1. 分析需求，识别原因（输入条件）和结果（输出）
  2. 画出因果图，标注因果关系和约束
  3. 将因果图转换为判定表
  4. 根据判定表设计测试用例

1.4 示例

需求：文件修改后，如果点击"保存"且文件可写，则保存文件并显示"保存成功"； 如果文件只读，则显示"文件只读"；如果未修改，提示"无需保存"。

原因：C1=文件已修改，C2=点击保存，C3=文件可写
结果：E1=保存文件，E2=显示成功，E3=显示只读，E4=提示无需保存

因果图：
(C1 ∧ C2 ∧ C3) → E1 ∧ E2
(C1 ∧ C2 ∧ ¬C3) → E3
(¬C1 ∧ C2) → E4
规则R1R2R3R4R5
C1（文件已修改）11100
C2（点击保存）11010
C3（文件可写）10---
E1（保存文件）X
E2（显示成功）X
E3（显示只读）X
E4（提示无需保存）X

二、状态转换测试（State Transition Testing）

基于系统的状态和状态之间的转换来设计测试用例，适合测试有状态的系统。

2.1 核心概念

概念说明
状态（State）系统在某一时刻的条件或状况
转换（Transition）从一个状态到另一个状态的变化
事件（Event）触发状态转换的条件
动作（Action）状态转换时执行的操作
初始状态系统的起始状态

2.2 状态图示例：登录系统

未登录 ──输入正确──→ 已登录 ──点击退出──→ 未登录

未登录 ──输入错误──→ 锁定 ──30分钟后──→ 未登录

2.3 状态转换表

当前状态事件/输入下一状态动作
未登录输入正确密码已登录进入主页
未登录输入错误密码未登录/锁定提示密码错误
已登录点击退出未登录清除会话
已登录会话超时未登录提示超时
锁定30分钟后未登录解锁

2.4 测试覆盖策略

  • 所有状态覆盖：每个状态至少被访问一次
  • 所有转换覆盖：每个转换至少被触发一次（推荐的最低覆盖）
  • 所有事件覆盖：每个事件至少触发一次
  • N转换覆盖：覆盖所有长度为N的转换序列

2.5 测试用例设计

用例步骤序列覆盖的转换
TC1未登录→正确密码→已登录→退出→未登录正常登录+退出
TC2未登录→错误密码→锁定→等待超时→未登录错误登录+锁定+解锁
TC3未登录→正确密码→已登录→超时→未登录正常登录+会话超时
状态测试重点：
1. 所有状态都被访问
2. 所有转换都被触发
3. 不可达状态的检测
4. 死状态（无法离开的状态）的检测
5. 非法转换的处理

三、知识要点总结

  1. 因果图：分析输入条件组合与输出的对应关系
  2. 因果图四步骤：识别因果→画图→转判定表→设计用例
  3. 五种约束：E互斥、I包含、O唯一、R要求、M强制
  4. 状态转换测试：基于状态和转换设计用例
  5. 最低覆盖要求：所有状态被访问，所有转换被触发
  6. 状态测试要关注不可达状态和死状态
