# 黑盒测试

## 一、黑盒测试原理

### 1.1 定义

黑盒测试（Black Box Testing / Functional Testing / Behavioral Testing）是一种不考虑程序内部结构的测试方法。测试人员仅根据**需求规格说明**来设计测试用例，关注输入与输出的关系。

### 1.2 核心思想

- 把程序看作一个"黑盒"，只关心外部行为
- 基于功能需求而非代码实现
- 适用于任何测试阶段（单元、集成、系统、验收）

### 1.3 适用场景

- 需求文档完整时
- 测试人员不具备代码阅读能力时
- 系统测试和验收测试阶段
- 接口测试和UI测试

---

## 二、等价类划分

### 2.1 定义

将输入域划分为若干等价类，从每个等价类中选取**代表性数据**作为测试用例。假设同一等价类中的数据对发现缺陷的效果相同。

### 2.2 有效等价类

符合需求规格说明的合理输入集合。

### 2.3 无效等价类

不符合需求规格说明的不合理输入集合。

### 2.4 示例

输入范围：1-100 的整数

| 等价类类型 | 划分 | 测试数据 |
|-----------|------|---------|
| 有效 | 1-100 | 50 |
| 无效（下界） | < 1 | 0 |
| 无效（上界） | > 100 | 101 |
| 无效（类型） | 非整数 | "abc" |
| 无效（空） | 空值 | "" |

### 2.5 划分原则

- 若输入条件指定了范围，划分为一个有效和两个无效
- 若输入条件指定了集合，划分为一个有效和一个无效
- 若输入条件是布尔值，划分为真和假
- 若输入条件是一组值，对每个值分别划分

---

## 三、边界值分析

### 3.1 定义

基于**缺陷往往发生在边界**的经验，对输入域的边界值及其邻近值进行测试。

### 3.2 边界值选取规则

对于范围 [min, max]：
- min - 1（刚好越下界）
- min（下界）
- min + 1（刚进下界）
- 正常值
- max - 1（刚进上界）
- max（上界）
- max + 1（刚好越上界）

### 3.3 示例

输入范围：1-100

测试数据：**0, 1, 2, 50, 99, 100, 101**

### 3.4 健壮性测试

在边界值分析的基础上，增加对极端值的测试（如 -max, 0, max_of_type）。

### 3.5 与等价类的关系

边界值分析通常与等价类划分**配合使用**，等价类确定划分，边界值分析细化边界测试用例。

---

## 四、决策表测试

### 4.1 定义

用表格形式描述**多个输入条件的组合**与对应**输出/动作**之间的关系。

### 4.2 决策表结构

| 条件/动作 | 规则1 | 规则2 | 规则3 | 规则4 |
|----------|-------|-------|-------|-------|
| 条件1    | Y     | Y     | N     | N     |
| 条件2    | Y     | N     | Y     | N     |
| 动作A    | X     | X     |       |       |
| 动作B    |       |       | X     |       |
| 动作C    |       |       |       | X     |

### 4.3 示例

折扣规则：
- 会员且消费>100：8折
- 会员且消费<=100：9折
- 非会员且消费>100：95折
- 非会员且消费<=100：无折扣

| 会员 | 消费>100 | 折扣 |
|------|---------|------|
| Y    | Y       | 8折  |
| Y    | N       | 9折  |
| N    | Y       | 95折 |
| N    | N       | 无折扣 |

### 4.4 适用场景

- 业务规则复杂，条件组合多
- 条件之间相互独立
- 输入条件与输出有明确的逻辑对应关系

---

## 五、因果图法

### 5.1 定义

用图形化方式描述输入条件（因）和输出结果（果）之间的逻辑关系，再转换为决策表。

### 5.2 基本符号

- **恒等（=）**：原因成立则结果成立
- **非（NOT）**：原因成立则结果不成立
- **与（AND）**：所有原因都成立，结果才成立
- **或（OR）**：任一原因成立，结果即成立

### 5.3 约束符号

- **E约束（互斥）**：a和b最多一个为真
- **I约束（包含）**：a和b至少一个为真
- **O约束（唯一）**：a和b恰好一个为真
- **R约束（要求）**：a为真时b也必须为真
- **M约束（屏蔽）**：特定条件下某结果被抑制

### 5.4 使用步骤

1. 分析需求，识别原因和结果
2. 画因果图
3. 添加约束
4. 将因果图转换为决策表
5. 根据决策表设计测试用例

---

## 六、正交试验法

### 6.1 定义

基于**正交表**（Orthogonal Array）从大量因素组合中选取代表性组合进行测试。用最少的测试用例实现最大的覆盖。

### 6.2 正交表表示

$L_n(m^k)$

- n：试验次数（行数）
- k：因素数（列数）
- m：水平数（每个因素的取值个数）

### 6.3 示例

3个因素（A、B、C），每个2个水平：

使用 $L_4(2^3)$ 正交表：

| 试验 | A | B | C |
|------|---|---|---|
| 1    | 0 | 0 | 0 |
| 2    | 0 | 1 | 1 |
| 3    | 1 | 0 | 1 |
| 4    | 1 | 1 | 0 |

### 6.4 优点

- 大幅减少测试用例数量
- 保证任意两个因素的所有水平组合都被覆盖
- 适用于多因素多水平的场景

## 七、场景法（基本流与备选流）

### 7.1 定义

通过描述用户使用系统的**场景**（Scenario）来设计测试用例，包括基本流和各种备选流。

### 7.2 基本流

从系统初始状态到正常结束的**最主要、最常用的业务流程**。

### 7.3 备选流

偏离基本流的其他执行路径：
- 异常处理路径
- 用户操作失误的恢复路径
- 可选的业务流程

### 7.4 示例：网上购物流程

```
基本流：登录→浏览商品→加入购物车→结算→支付→订单完成
备选流1：登录失败→重新输入→继续基本流
备选流2：购物车为空→提示添加商品
备选流3：库存不足→提示更换或取消
备选流4：支付失败→重新支付→继续基本流
```

### 7.5 测试用例设计

每个场景 = 基本流 + 若干备选流的组合，用例数 = 备选流数 + 1（基本流）。

---

## 八、错误推测法

### 8.1 定义

基于测试人员的**经验和直觉**，推测程序中可能存在的错误，有针对性地设计测试用例。

### 8.2 常见的推测方向

- 输入特殊字符、超长字符串
- 空值、null、零值处理
- 重复提交、并发操作
- 日期边界（闰年、跨年）
- 文件大小限制、内存溢出
- 数组越界、除以零
- 编码问题（中文、特殊符号）

### 8.3 特点

- 非系统化的测试方法
- 不能单独作为测试策略
- 作为其他方法的有效补充
- 对经验丰富测试人员尤其有效

---

## 九、随机测试与Fuzz测试

### 9.1 随机测试

在输入域中**随机选取**输入数据进行测试。适用于：
- 探索性测试
- 压力测试
- 验证系统在异常输入下的稳定性

### 9.2 Fuzz测试

向程序输入**大量随机、畸形或异常数据**，观察程序是否崩溃或产生异常。

**类型**：
- **变异型 Fuzz**：基于合法数据进行随机修改
- **生成型 Fuzz**：根据协议或格式规范生成测试数据
- **智能 Fuzz**：结合代码覆盖率反馈进行定向测试

**工具**：AFL、LibFuzzer、Peach Fuzzer、Boofuzz

### 9.3 应用场景

- 安全测试中发现漏洞
- 文件解析器测试
- 协议实现测试
- API鲁棒性测试

---

## 十、组合测试

### 10.1 定义

当系统有多个输入参数时，穷举所有组合不可行。组合测试通过数学方法选取**参数间交互**的代表性组合。

### 10.2 常见组合策略

- **单因素覆盖**：每个参数的每个值至少出现一次
- **两两组合（Pairwise）**：任意两个参数的所有取值组合至少出现一次
- **N-way组合**：任意N个参数的所有取值组合至少出现一次

### 10.3 工具

- PICT（微软）
- AllPairs
- ACTS（NIST）

### 10.4 两两组合的优势

研究表明，约 **70% 以上的缺陷由单个因素或两个因素的交互引起**。两两组合通常能以少量用例发现大部分缺陷。

---

## 十一、黑盒测试用例设计技巧

### 11.1 方法选择策略

| 场景 | 推荐方法 |
|------|---------|
| 有明确输入范围 | 等价类 + 边界值 |
| 多条件组合判断 | 决策表 / 因果图 |
| 多因素多水平 | 正交试验法 |
| 业务流程测试 | 场景法 |
| 经验丰富的测试员 | 错误推测法 |
| 安全/鲁棒性测试 | Fuzz测试 |

### 11.2 综合使用

实际项目中通常**组合使用多种方法**：
1. 使用等价类划分确定有效和无效输入
2. 使用边界值分析细化测试数据
3. 使用决策表处理条件组合
4. 使用场景法覆盖业务流程
5. 使用错误推测法补充遗漏

### 11.3 注意事项

- 始终关注需求规格说明
- 关注异常和边界情况
- 考虑用户的实际使用场景
- 测试用例应具有可重复性
- 保持测试用例的独立性
