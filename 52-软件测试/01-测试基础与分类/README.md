# 测试基础与分类

## 一、软件测试的定义与目的

### 1.1 定义

软件测试（Software Testing）是在规定的条件下对程序进行操作，以发现程序错误、衡量软件质量并评估其是否满足设计要求的过程。

IEEE 对测试的定义：**使用人工或自动手段来运行或测定某个系统的过程，其目的在于检验它是否满足规定的需求或是弄清预期结果与实际结果之间的差别。**

### 1.2 目的

- **发现缺陷**：尽可能早地发现软件中的缺陷（Bug）
- **验证需求**：确认软件是否满足需求规格说明
- **建立信心**：为利益相关者提供软件质量的信心
- **预防缺陷**：通过测试活动发现过程中的薄弱环节
- **质量评估**：为软件发布决策提供数据支撑

> 注意：测试的目的不是证明程序没有缺陷，而是以合理的成本发现尽可能多的缺陷。

---

## 二、测试原则

### 2.1 测试显示缺陷的存在

测试可以显示缺陷的存在，但不能证明软件没有缺陷。即使测试未发现缺陷，也不能证明软件完全正确。

### 2.2 穷尽测试不可能

除了小型项目外，完全测试（输入的所有组合）是不可能的。应采用**风险分析**和**优先级排序**来指导测试。

### 2.3 测试尽早介入

"越早测试越好"。根据缺陷放大理论，在需求阶段发现缺陷的修复成本最低，而发布后修复成本可能高达100倍以上。

### 2.4 缺陷聚集性（Pareto 原则 / 二八定律）

约 **80% 的缺陷集中在 20% 的模块中**。测试时应重点关注风险最高的模块。

### 2.5 杀虫剂悖论

反复使用相同的测试用例进行测试，最终将无法发现新的缺陷。必须**定期审查和更新测试用例**，引入新的测试方法和数据。

### 2.6 测试依赖于上下文

不同类型的软件（如航天系统 vs 博客网站）需要不同的测试策略和测试深度。

### 2.7 无缺陷谬论

即使软件功能完全正确，但如果不能满足用户的实际需求和期望，仍然是一次失败。

## 三、测试分类：按阶段划分

### 3.1 单元测试（Unit Testing）

- **对象**：最小可测试单元（函数、方法、类）
- **执行者**：开发人员
- **特点**：粒度最小、执行速度快、成本低
- **工具**：JUnit、pytest、unittest

### 3.2 集成测试（Integration Testing）

- **对象**：模块之间的接口与协作
- **策略**：
  - **自顶向下集成**：从主控模块开始，逐层向下
  - **自底向上集成**：从底层模块开始，逐层向上
  - **三明治集成**：上下同时进行
  - **大爆炸集成**：所有模块一次性集成

### 3.3 系统测试（System Testing）

- **对象**：完整的集成系统
- **内容**：功能测试、性能测试、安全测试、兼容性测试等
- **环境**：尽可能接近真实生产环境

### 3.4 验收测试（Acceptance Testing）

- **对象**：由用户或客户执行
- **类型**：
  - Alpha 测试：开发环境下由用户进行
  - Beta 测试：实际使用环境下由真实用户进行
  - UAT（用户验收测试）

---

## 四、测试分类：按方法划分

### 4.1 静态测试（Static Testing）

不运行程序代码的测试方法：
- **代码审查（Code Review）**：人工检查代码
- **静态分析（Static Analysis）**：使用工具自动分析（如 SonarQube、FindBugs）
- **走查（Walkthrough）**：由作者引导团队审阅
- **审查（Inspection）**：正式的结构化评审流程

### 4.2 动态测试（Dynamic Testing）

运行程序代码并输入测试数据的测试方法：
- 通过执行程序来观察实际行为与预期行为的差异
- 包括所有需要运行代码的测试活动

---

## 五、测试分类：按目的划分

### 5.1 功能测试

验证软件的功能是否按需求规格说明正确实现。

### 5.2 性能测试

- **负载测试**：验证系统在预期负载下的表现
- **压力测试**：找到系统的性能瓶颈和极限
- **容量测试**：确定系统能处理的最大数据量
- **稳定性测试**：长时间运行下的可靠性

### 5.3 安全测试

验证系统对非法入侵和恶意攻击的防护能力，包括漏洞扫描、渗透测试等。

### 5.4 兼容性测试

验证软件在不同硬件、操作系统、浏览器、网络环境下的兼容性。

### 5.5 回归测试

修改代码后重新执行已有测试，确保修改没有引入新的缺陷。

## 六、测试用例设计要素

一个完整的测试用例应包含以下要素：

| 要素 | 说明 |
|------|------|
| 用例编号 | 唯一标识符，便于追踪 |
| 测试项目 | 所属模块或功能 |
| 测试标题 | 简明描述测试目的 |
| 预置条件 | 执行前必须满足的条件 |
| 测试数据 | 输入的具体数据 |
| 操作步骤 | 详细的执行步骤 |
| 预期结果 | 应该出现的正确结果 |
| 优先级 | 重要程度（P0/P1/P2/P3） |
| 前置依赖 | 需要先执行的用例 |

## 七、测试金字塔模型

```
        /  UI测试  \          少量、慢、脆弱
       /  API测试    \
      /  集成测试      \
     /    单元测试        \    大量、快、稳定
```

- **底层**：大量的单元测试（70%以上），执行快、成本低
- **中层**：适量的集成测试，验证模块间协作
- **顶层**：少量的 UI / 端到端测试，覆盖关键用户流程

遵循测试金字塔可以在控制成本的同时保证测试覆盖率。

## 八、缺陷的生命周期

```
新建(New) → 分配(Assigned) → 打开(Open) → 修复(Fixed) → 验证(Verified) → 关闭(Closed)
                                          ↘ 拒绝(Rejected)
                                          ↘ 延期(Deferred)
                                          ↘ 重新打开(Reopened)
```

- **New**：测试人员发现并提交缺陷
- **Assigned**：项目经理分配给开发人员
- **Open**：开发人员确认并开始修复
- **Fixed**：开发人员完成修复
- **Verified**：测试人员验证修复有效
- **Closed**：缺陷正式关闭
- **Rejected**：不是真正的缺陷（如设计如此、环境问题等）
- **Deferred**：确认但推迟到后续版本修复
- **Reopened**：验证后发现未修复成功，重新打开

## 九、测试文档

### 9.1 测试计划（Test Plan）

描述测试范围、资源、进度、风险和策略的整体规划文档。依据 **IEEE 829** 标准编写。

### 9.2 测试用例（Test Case）

具体的测试输入、执行步骤和预期结果的文档集合。

### 9.3 缺陷报告（Bug Report）

记录缺陷的详细信息，包括：标题、复现步骤、实际结果、期望结果、严重程度、优先级、环境信息、截图/日志。

### 9.4 测试报告（Test Report）

测试执行完成后的总结报告，包含：测试范围、执行情况、缺陷统计、覆盖率数据、风险评估和发布建议。
