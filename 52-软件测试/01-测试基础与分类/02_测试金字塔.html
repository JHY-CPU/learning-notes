测试金字塔 (Test Pyramid)

分层测试策略 — 单元测试、集成测试、端到端测试的比例与实践

1. 测试金字塔模型

E2E / UI 测试 ~10%
集成测试 ~20%
单元测试 ~70%
Mike Cohn 原则：底层测试数量多、速度快、成本低；顶层测试数量少、速度慢、成本高。越往下，测试越稳定、越可靠。
层级测试对象速度成本比例
单元测试函数、类、模块毫秒级低70%
集成测试模块间交互、API秒级中20%
E2E 测试完整用户流程分钟级高10%

2. 单元测试 (Unit Test)

特征

  • 测试最小可测单元（方法/函数）
  • 隔离执行，不依赖外部资源
  • 使用 Mock/Stub 模拟依赖
  • 执行速度快（毫秒级）

原则 (F.I.R.S.T)

  • Fast — 快速执行
  • Isolated — 相互隔离
  • Repeatable — 可重复运行
  • Self-Validating — 自动验证
  • Timely — 及时编写
// JUnit 5 单元测试示例
class OrderServiceTest {
    @Mock private OrderRepository repo;
    @InjectMocks private OrderService service;

    @Test
    void calculateTotal_shouldSumItemPrices() {
        // Arrange (准备)
        var order = new Order(List.of(
            new Item("书", 50), new Item("笔", 10)));
        // Act (执行)
        var total = service.CalculateTotal(order);
        // Assert (断言)
        assertEquals(60, total);
    }

    @Test
    void placeOrder_shouldThrowWhenEmpty() {
        var emptyOrder = new Order(Collections.emptyList());
        assertThrows(EmptyOrderException.class,
            () => service.PlaceOrder(emptyOrder));
    }
}

3. 集成测试 (Integration Test)

测试内容

  • 模块间接口调用
  • 数据库 CRUD 操作
  • 消息队列收发
  • 外部服务调用（可选真实/模拟）

常用工具

  • Testcontainers — 真实数据库容器
  • WireMock — HTTP 服务模拟
  • @SpringBootTest — Spring 集成测试
  • 嵌入式数据库 (H2, SQLite)
// Spring Boot 集成测试
@SpringBootTest
@Testcontainers
class OrderRepositoryTest {
    @Container
    static MySQLContainer<?> mysql =
        new MySQLContainer<>("mysql:8.0");

    @Test
    void saveAndFind_shouldPersistOrder() {
        var order = new Order("ORD-001", BigDecimal.TEN);
        repo.save(order);
        var found = repo.findById("ORD-001");
        assertTrue(found.isPresent());
        assertEquals(BigDecimal.TEN, found.get().getAmount());
    }
}

4. 端到端测试 (E2E / UI Test)

模拟真实用户操作，验证完整业务流程。成本最高但最接近用户体验。

常用工具

  • Selenium / Playwright — 浏览器自动化
  • Cypress — 现代前端 E2E 框架
  • Appium — 移动端自动化
  • Postman/Newman — API E2E 测试

挑战

  • 执行缓慢（分钟级）
  • 不稳定（Flaky Tests）
  • 维护成本高（UI 变更导致失败）
  • 调试困难
// Playwright E2E 测试示例
test('用户可以完成下单流程', async ({ page }) => {
    // 1. 打开首页
    await page.goto('https://shop.example.com');
    // 2. 搜索商品
    await page.fill('#search', '设计模式');
    await page.click('#searchBtn');
    // 3. 添加购物车
    await page.click('.product-card:first-child .add-cart');
    // 4. 进入结算
    await page.click('#checkout');
    // 5. 验证订单金额
    const total = await page.textContent('#order-total');
    expect(total).toBe('¥59.00');
});

5. 补充测试类型

5.1 契约测试 (Contract Testing)

验证服务间 API 契约的一致性，确保提供者不破坏消费者的期望。代表工具：Pact。

// Pact 契约测试 — 消费者端
var pact = Pact.V3("OrderService", "ProductService");
pact.UponReceiving("获取商品详情")
    .Given("商品ID=100存在")
    .WithRequest(HttpMethod.Get, "/api/products/100")
    .WillRespond()
    .WithStatus(200)
    .WithJsonBody(new { Id = 100, Name = "设计模式" });

5.2 组件测试 (Component Test)

测试单个微服务或模块的完整行为（介于集成测试和 E2E 之间）。使用真实代码但模拟外部依赖。

现代演进：测试金字塔并非一成不变。Trophy 模型（Kent C. Dodds）强调集成测试更有价值；蜂巢模型（Spotify）根据服务间依赖调整比例。选择适合团队的策略最重要。
