变异测试与属性测试

Mutation Testing & Property-Based Testing — 深度评估测试质量，发现边界缺陷

1. 变异测试 (Mutation Testing)

核心思想：通过对源代码进行微小改动（"变异"）来评估测试用例的质量。如果测试能发现变异（即变异被"杀死"），说明测试有效；否则说明测试存在盲区。
原始代码→ 生成变异体→ 运行测试→ 检测存活变异→ 补充测试用例

变异算子 (Mutation Operators)

算术变异

  • a + b → a - b
  • a * b → a / b
  • a % b → a * b
  • ++i → --i

条件变异

  • > → >=, <, <=
  • == → !=
  • && → ||
  • true → false

语句变异

  • 删除整条语句
  • 删除方法返回语句
  • 删除 void 方法调用
  • 替换常量值

变异测试指标

// 变异得分 (Mutation Score)
// Mutation Score = 被杀死的变异体数 / 总变异体数 × 100%

// 示例：
总变异体:    100 个
被杀死:      85 个  // 测试能检测到这些变异
存活:        12 个  // 测试未能检测到（测试不足）
等价变异:     3 个  // 变异后行为不变（可忽略）

// Mutation Score = 85 / (100 - 3) = 87.6%
// 目标：通常要求 > 80%

PIT (Pitest) — Java 变异测试工具

<!-- Maven 配置 -->
<plugin>
    <groupId>org.pitest</groupId>
    <artifactId>pitest-maven</artifactId>
    <version>1.15.0</version>
    <configuration>
        <targetClasses>
            <param>com.example.service.*</param>
        </targetClasses>
        <targetTests>
            <param>com.example.service.*Test</param>
        </targetTests>
        <mutationThreshold>80</mutationThreshold> <!-- 最低阈值 -->
        <outputFormats>
            <param>HTML</param>
            <param>XML</param>
        </outputFormats>
    </configuration>
</plugin>

// 运行: mvn org.pitest:pitest-maven:mutationCoverage

存活变异体分析示例

// 原始代码
public int discount(int price, int quantity) {
    if (quantity >= 10) return price * 80 / 100;
    return price;
}

// PIT 发现存活变异体：
// 1. quantity >= 10 → quantity > 10  （存活！）
//    说明：没有测试 quantity == 10 的情况
// 2. price * 80 / 100 → price * 80 * 100 （被杀死）

// 补充测试用例
@Test void discount_boundaryQuantity10() {
    assertEquals(80, discount(100, 10)); // 边界值测试
}

2. 属性测试 (Property-Based Testing)

核心思想：不编写具体的输入-输出用例，而是描述代码应该满足的"属性"（通用规律），让框架自动生成大量随机输入进行验证。

与示例测试的区别

  • 示例测试：给定具体输入，期望具体输出
  • 属性测试：描述通用规律，框架生成输入
  • 属性测试覆盖面更广，能发现意外边界

常用属性类型

  • 幂等性：f(f(x)) == f(x)
  • 交换律：f(a,b) == f(b,a)
  • 逆操作：decode(encode(x)) == x
  • 不变量：排序后长度不变

Hypothesis (Python)

from hypothesis import given, strategies as st

# 属性：排序后的列表长度不变
@given(st.lists(st.integers()))
def test_sort_preserves_length(lst):
    assert len(sorted(lst)) == len(lst)

# 属性：排序后的列表是有序的
@given(st.lists(st.integers()))
def test_sort_is_ordered(lst):
    result = sorted(lst)
    assert all(result[i] <= result[i+1] for i in range(len(result)-1))

# 属性：encode/decode 互逆
@given(st.text())
def test_encode_decode_roundtrip(s):
    assert decode(encode(s)) == s

# Hypothesis 会自动生成边界值：空列表、超大列表、包含负数等
# 发现最小失败案例 (Shrinking)：[3, 1, 2] → [1, 0]

QuickCheck (Haskell)

-- 属性：反转两次得到原列表
prop_reverse_twice :: [Int] -> Bool
prop_reverse_twice xs = reverse (reverse xs) == xs

-- 属性：排序后的列表首元素是最小值
prop_sort_min :: [Int] -> Property
prop_sort_min xs =
    not (null xs) ==>   -- 前置条件（非空）
    head (sort xs) == minimum xs

-- 运行: quickCheck prop_reverse_twice
-- +++ OK, passed 100 tests.

3. 模糊测试 (Fuzzing)

模糊测试是属性测试的一种特殊形式，通过向程序输入随机/畸形数据来发现崩溃、内存泄漏等安全漏洞。

模糊测试类型

  • 生成型 (Generation)：从零生成输入
  • 变异型 (Mutation)：基于已有输入变异
  • 反馈驱动 (Coverage-guided)：利用代码覆盖引导
  • 语法感知 (Grammar-based)：遵循输入格式

常用工具

  • AFL++ / libFuzzer — C/C++ 覆盖引导
  • Jazzer — JVM 模糊测试
  • Atheris — Python 模糊测试
  • go-fuzz — Go 语言
  • oss-fuzz — Google 开源项目持续模糊测试

libFuzzer 示例 (C++)

// 被测函数
bool parse_url(const char* data, size_t size) {
    if (size < 5) return false;
    if (memcmp(data, "http://", 7) != 0) return false;
    // ... 解析逻辑
}

// 模糊测试入口
extern "C" int LLVMFuzzerTestOneInput(const uint8_t* data, size_t size) {
    parse_url((const char*)data, size);
    return 0;
}
// 编译: clang++ -fsanitize=fuzzer,address fuzzer.cpp
// 运行: ./a.out  (自动生成输入、检测崩溃和内存错误)
Shrinking（缩减）：当模糊测试发现崩溃时，框架会自动将输入缩减到最小可复现版本，极大简化调试。这是 Hypothesis 和 AFL++ 都具备的关键能力。
