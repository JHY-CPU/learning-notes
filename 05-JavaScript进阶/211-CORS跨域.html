CORS 跨域

跨域资源共享 — 浏览器安全策略与跨域解决方案。

同源策略

// 同源 = 协议 + 域名 + 端口 完全相同 // https://example.com:443 // 协议: https:// // 域名: example.com // 端口: 443 // 同源示例: // https://example.com/page1 → 同源 // https://example.com/page2 → 同源 // 跨域示例: // https://api.example.com → ❌ 不同域名 // http://example.com → ❌ 不同协议 // https://example.com:8080 → ❌ 不同端口 // https://www.example.com → ❌ 不同子域名 // 浏览器限制: // ❌ 跨域请求的响应数据默认不可读 // ✅ 允许跨域嵌入: <script src>, <img src>, <link href>, <iframe> // ✅ 允许跨域提交: <form action>

CORS 机制

// 服务器通过响应头告知浏览器允许跨域 // 简单请求（GET/POST + 简单 Content-Type） // 浏览器自动添加 Origin 头 // 服务器返回 Access-Control-Allow-Origin // 预检请求（OPTIONS） // 触发条件: 非简单方法(PUT/DELETE) / 自定义头 / 非简单 Content-Type // 浏览器先发 OPTIONS 请求（预检） // 服务器返回允许的方法和头 // 浏览器再发实际请求 // 关键响应头 Access-Control-Allow-Origin: https://example.com // 允许的源 Access-Control-Allow-Origin: * // 允许所有（不能带 credentials） Access-Control-Allow-Methods: GET, POST, PUT // 允许的方法 Access-Control-Allow-Headers: Content-Type, Auth // 允许的请求头 Access-Control-Allow-Credentials: true // 允许携带 Cookie Access-Control-Max-Age: 86400 // 预检结果缓存时间 // 请求头 Origin: https://example.com // 请求来源 Access-Control-Request-Method: PUT // 预检时声明的方法 Access-Control-Request-Headers: Authorization // 预检时声明的头

跨域解决方案

// 1. CORS（推荐） fetch('https://api.example.com/data', { mode: 'cors', // 默认 credentials: 'include', // 携带 Cookie（需要服务器返回 specific origin） }); // 2. 代理服务器（开发常用） // 前端 → 代理(同源) → 目标API // Vite proxy: server.proxy // Webpack proxy: devServer.proxy // 3. JSONP（只能 GET，已不推荐） // <script src="https://api.example.com/data?callback=handleData"></script> // 4. postMessage（跨窗口通信） otherWindow.postMessage(data, targetOrigin); window.addEventListener('message', (e) => { if (e.origin !== 'https://trusted.com') return; console.log(e.data); }); // 5. WebSocket（不受同源策略限制） const ws = new WebSocket('wss://api.example.com');
💡 credentials: 'include' 时，服务器必须指定具体的 origin（不能是 *）。
⚠️ 跨域请求不会发送 Cookie（除非 credentials: 'include'）。
💡 开发中常用代理解决跨域，生产环境配置 CORS 头。
