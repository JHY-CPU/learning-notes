查询处理优化

搜索引擎查询优化WAND

高效的查询处理需要巧妙的倒排表遍历策略，WAND和MaxScore等算法通过动态剪枝大幅减少计算量。

文档遍历 vs 词条遍历

查询处理的两种基本策略： 1. Document-at-a-Time (DAAT) 文档优先 对每个候选文档，计算所有词条的综合得分 查询："搜索引擎 优化" Posting("搜索引擎"): [3, 12, 25, 42] Posting("优化"): [7, 12, 31, 42] 遍历过程： ┌──────────────────────────────────────┐ │ 使用Min-Heap管理Top-K文档 │ │ │ │ 所有倒排表的指针同时前进 │ │ 每次取最小DocID的文档 │ │ 如果文档在所有倒排表中都有贡献 │ │ → 计算完整得分 │ │ → 与Heap中最低分比较 │ │ → 决定是否加入结果集 │ │ │ │ Doc3: 只在"搜索引擎"中出现 │ │ 部分得分，跳过或记分 │ │ Doc7: 只在"优化"中出现 │ │ 部分得分，跳过或记分 │ │ Doc12: 两个词条都有 → 计算完整得分 │ │ Doc42: 两个词条都有 → 计算完整得分 │ └──────────────────────────────────────┘ 优点：只需维护Top-K的内存 缺点：需要同步遍历所有倒排表 2. Term-at-a-Time (TAAT) 词条优先 先处理一个词条的所有文档，累积得分到数组 查询："搜索引擎 优化" 遍历过程： ┌──────────────────────────────────────┐ │ 步骤1：遍历"搜索引擎"倒排表 │ │ scores[3] += BM25("搜索引擎", Doc3) │ │ scores[12] += BM25("搜索引擎", Doc12)│ │ scores[25] += BM25("搜索引擎", Doc25)│ │ scores[42] += BM25("搜索引擎", Doc42)│ │ │ │ 步骤2：遍历"优化"倒排表 │ │ scores[7] += BM25("优化", Doc7) │ │ scores[12] += BM25("优化", Doc12) │ │ scores[31] += BM25("优化", Doc31) │ │ scores[42] += BM25("优化", Doc42) │ │ │ │ 步骤3：从scores中取Top-K │ └──────────────────────────────────────┘ 优点：每个倒排表独立遍历，可按IDF排序从高到低处理 缺点：需要scores数组（可能很大），需要阈值剪枝 实际系统选择： - Lucene/Elasticsearch 使用 DAAT - TAAT配合阈值剪枝也可高效工作 - DAAT天然支持Top-K，更适合生产环境

WAND 算法 (Weak AND)

WAND (Weighted AND)：基于词条权重的动态剪枝算法 2003年由Broder等人提出，被Google和多数搜索引擎采用。 核心思想： - 每个词条有一个上界权重（Upper Bound），表示它对任何文档的最大贡献 - 利用上界快速排除不可能进入Top-K的文档 - 大量文档无需完整评分即可跳过 关键概念： ┌─────────────────────────────────────────────┐ │ 词条上界 UB(t)：词条t在任何文档中的最大BM25得分│ │ 通常用 max(TF) × IDF 来近似 │ │ │ │ 文档上界 UB(d)：文档d中所有词条上界之和 │ │ 如果 UB(d) < 当前Top-K的最低分 → 跳过d │ └─────────────────────────────────────────────┘ WAND算法流程： 查询："搜索引擎 优化 算法" 词条信息（假设已排序）： "搜索引擎" UB=5.2, 当前指针→Doc3 "优化" UB=3.8, 当当前指针→Doc7 "算法" UB=4.5, 当前指针→Doc12 步骤： 1. 设定阈值θ（Top-K中最低分），初始θ=0 2. 将词条按当前DocID排序 ┌─────────────────────────────────────────┐ │ 排序结果： │ │ "搜索引擎" UB=5.2 → Doc3 │ │ "优化" UB=3.8 → Doc7 │ │ "算法" UB=4.5 → Doc12 │ │ │ │ 找分界点p：前p个词条的UB之和 ≥ θ │ │ p=1: UB=5.2 ≥ θ(0) → 分界在"优化" │ │ 候选文档=Doc7（分界点词条的DocID） │ └─────────────────────────────────────────┘ 3. 检查候选文档Doc7： - Doc7之前的所有词条是否都指向 ≤ Doc7？ - "搜索引擎"指向Doc3 ≤ Doc7 ✓ - "优化"指向Doc7 = Doc7 ✓ - "算法"指向Doc12 > Doc7 ✗ - 不满足条件 → 不处理Doc7 4. 推进"搜索引擎"指针到下一个DocID（如12） 重新排序： "优化" UB=3.8 → Doc7 "算法" UB=4.5 → Doc12 "搜索引擎" UB=5.2 → Doc12 ... 5. 重复直到遍历完所有倒排表 WAND的优势： - 大量文档可以跳过完整评分 - 实测可跳过50-90%的文档 - 与DAAT天然配合 - 对长查询效果更显著

MaxScore 算法

MaxScore：更简单但同样高效的剪枝算法 核心思想： - 根据词条的上界权重，将词条分为"必要"和"非必要"两组 - 先用必要词条计算部分得分 - 如果部分得分已经超过阈值，再加入非必要词条 算法流程： 查询："搜索引擎 优化 算法" 词条上界排序： "搜索引擎" UB=5.2 "算法" UB=4.5 "优化" UB=3.8 设定阈值θ（Top-K最低分） 步骤： 1. 设定必要词条的累加上界 非必要集合从空开始逐步加入 轮次1：非必要 = {}，必要 = {搜索引擎, 算法, 优化} 所有词条都是必要词条，无剪枝 2. 逐步将最小权重的词条移入非必要集合 如果剩余必要词条的UB之和 < θ → 那些只有非必要词条贡献的文档可以跳过 实际流程： ┌─────────────────────────────────────────────┐ │ For each document d in 遍历顺序: │ │ score = 0 │ │ For each 必要词条 t: │ │ if d 在 t 的倒排表中: │ │ score += BM25(t, d) │ │ if score < θ: continue # 跳过 │ │ For each 非必要词条 t: │ │ if d 在 t 的倒排表中: │ │ score += BM25(t, d) │ │ if score >= θ: 更新Top-K │ └─────────────────────────────────────────────┘ MaxScore vs WAND： ┌──────────────┬──────────────┬──────────────┐ │ 特性 │ MaxScore │ WAND │ ├──────────────┼──────────────┼──────────────┤ │ 剪枝粒度 │ 词条级别 │ 文档级别 │ │ 实现复杂度 │ 简单 │ 中等 │ │ 剪枝效果 │ 好 │ 更好 │ │ 阈值更新 │ 简单 │ 动态更新 │ │ 适用场景 │ 通用 │ 长查询更佳 │ │ Lucene使用 │ 否 │ 是 │ └──────────────┴──────────────┴──────────────┘

动态阈值与剪枝策略

动态阈值更新： WAND和MaxScore都需要动态更新阈值θ 初始θ = 0（或一个小的负值） 每当Top-K堆中发现更好的结果时： θ = max(θ, heap.min_score) 阈值越高 → 剪枝越激进 → 跳过越多文档 阶段式处理： ┌─────────────────────────────────────────────┐ │ 阶段1：快速粗排（Tiers） │ │ - 只处理每个词条的前N个文档（高频文档） │ │ - 用简单评分公式（如TF） │ │ - 快速建立初始Top-K和阈值 │ │ │ │ 阶段2：精确精排 │ │ - 利用阶段1的阈值进行WAND/MaxScore剪枝 │ │ - 完整BM25评分 │ │ - 最终Top-K结果 │ └─────────────────────────────────────────────┘ 查询扩展与优化： 1. 查询改写 - 同义词扩展："电脑" → "电脑 OR 计算机 OR PC" - 拼写纠正："srearch" → "search" 2. 查询缓存 - 热门查询结果缓存（Redis/内存） - 倒排表交集缓存 - 缓存命中率可达30-50% 3. 结果缓存 - 缓存 (query, page) → results - 对相同查询直接返回缓存结果 - 配合TTL和失效策略 4. 提前终止 (Early Termination) - 如果倒排表按PageRank排序 - 前N个文档很可能就是最好的 - 在质量可接受时提前返回 多词条查询优化： 查询："分布式 系统 设计 架构" 优化顺序： 1. 按IDF降序排列词条（稀有词优先） 2. 从最稀有的词条开始遍历 3. 稀有词的倒排表更短，更快建立阈值 4. 阈值越高 → 后续词条的剪枝越有效

实际系统中的查询处理

Lucene/Elasticsearch 查询处理流程： ┌─────────────────────────────────────────────┐ │ 用户查询 → QueryParser → 查询分析 │ │ ↓ │ │ Term Query / Boolean Query / Phrase Query │ │ ↓ │ │ Weight对象（封装评分逻辑） │ │ ↓ │ │ Scorer（执行DAAT遍历 + WAND剪枝） │ │ ↓ │ │ TopDocs（Top-K结果 + 总命中数） │ │ ↓ │ │ 排序、高亮、返回 │ └─────────────────────────────────────────────┘ 关键类（Java）： - IndexSearcher: 入口，管理查询执行 - Weight: 查询的权重/评分逻辑 - Scorer: 实际的文档遍历和评分 - BulkScorer: 批量评分优化 - MaxScoreCache: WAND/MaxScore的优化 查询类型与处理策略： ┌─────────────────┬──────────────────────────┐ │ 查询类型 │ 处理策略 │ ├─────────────────┼──────────────────────────┤ │ TermQuery │ 单倒排表遍历 │ │ BooleanQuery │ DAAT + WAND │ │ PhraseQuery │ 位置信息交集 │ │ PrefixQuery │ FST/Trie前缀查找 │ │ FuzzyQuery │ 编辑距离 + 倒排表 │ │ RangeQuery │ FST范围查找 │ │ KnnVectorQuery │ HNSW/IVF向量检索 │ └─────────────────┴──────────────────────────┘ 性能基准（参考值）： - 单词条查询：<1ms（百万级文档索引） - 3词条布尔查询：1-5ms - 10词条复杂查询：5-20ms - 带过滤条件：取决于过滤效率 - 向量检索(top-10)：1-10ms（百万级向量）
WAND算法是现代搜索引擎的核心优化技术，通过动态剪枝跳过大量无关文档，在保持结果质量的同时将查询延迟降低一个数量级。
