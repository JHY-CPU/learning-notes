# 索引压缩技术


## 索引压缩技术


搜索引擎压缩倒排索引


索引压缩大幅减少磁盘和内存占用，同时通过减少I/O提升查询性能。核心技术包括倒排表压缩和词典压缩。


## 为什么需要索引压缩


```
未压缩倒排索引的规模：
- Google索引超过千亿网页
- 每个倒排表项至少需要 DocID(4B) + TF(2B) + Positions(变长)
- 假设100亿文档，平均每个词条出现在100万篇文档中
- 一个词条的倒排表：100万 × 8B ≈ 8MB
- 常用词条有数百万个 → 数TB级别的倒排索引

压缩带来的收益：
┌─────────────────────────────────────────────┐
│  1. 减少磁盘存储空间                         │
│  2. 减少内存占用（更多数据可放入内存）        │
│  3. 减少磁盘I/O（读取更少的数据）             │
│  4. 减少网络传输（分布式环境下）              │
│  5. 提高CPU缓存命中率                        │
└─────────────────────────────────────────────┘

压缩的核心观察：
- DocID在倒排表中是递增有序的
- 相邻DocID的差值（Gap）通常很小
- 对小数值进行编码比大数值更高效
- 这就是差值编码（Delta Encoding）的思路
```


## 变长编码 (Variable Byte Encoding)


```
Variable Byte Encoding (VByte)：
用1个字节或多个字节表示一个整数，小数值用更少字节。

编码规则：
- 每个字节的最高位（MSB）是延续位
- MSB = 1：后面还有字节
- MSB = 0：这是最后一个字节
- 低7位是有效数据位

示例：编码数值 356
356 的二进制：1 01100100
分成7位一组：  00000010  1100100
加上延续位：    10000010  01100100
结果：0x82 0x64（2字节，而非4字节）

编码示例表：
┌──────────┬──────────────┬──────────┐
│ 数值      │ 二进制        │ VByte编码 │
├──────────┼──────────────┼──────────┤
│ 5        │ 101          │ 0x05     │
│ 127      │ 1111111      │ 0x7F     │
│ 128      │ 10000000     │ 0x81 0x00│
│ 16383    │ 111111111111 │ 0x81 0xFF│
│          │ 11           │ 0x7F     │
│ 16384    │              │ 0x80 0x80│
│          │              │ 0x00     │
└──────────┴──────────────┴──────────┘

编码过程（伪代码）：
def vbyte_encode(n):
    result = []
    while n >= 128:
        result.append((n & 0x7F) | 0x80)  # 设置延续位
        n >>= 7
    result.append(n & 0x7F)  # 最后一个字节，延续位为0
    return bytes(result)

解码过程（伪代码）：
def vbyte_decode(data):
    n = 0
    shift = 0
    for byte in data:
        n |= (byte & 0x7F) << shift
        if (byte & 0x80) == 0:
            break  # 最后一个字节
        shift += 7
    return n

实际应用中的优化：
- 对倒排表先做差值编码（Delta Encoding）
- 再对差值做VByte编码
- DocID序列 [10, 15, 23, 28, 35]
- 差值序列 [10, 5, 8, 5, 7]
- 差值都在127以内，每个只需1字节
- 原始 5×4B = 20B → 压缩后 5×1B = 5B
```


## PForDelta 压缩算法


```
PForDelta (Packed Frame of Delta)：
块级压缩算法，对一个块内的差值统一编码，利用SIMD指令并行解压。

核心思想：
- 将倒排表分为固定大小的块（如128或256个元素）
- 对块内所有差值，找一个阈值b
- 90%的值可以用b位表示（高效压缩）
- 剩余10%的异常值单独处理（溢出区域）

编码过程：
1. 对块内差值排序，取90%分位数作为b
2. 每个值用b位存储（紧凑排列）
3. 大于 2^b-1 的异常值存入溢出区

示例（块大小=8，差值序列）：
差值序列：[3, 5, 2, 7, 4, 100, 3, 6]
排序：[2, 3, 3, 4, 5, 6, 7, 100]
90%分位数 ≈ 7 → b = 3位 (可表示0-7)

编码结果：
主区域（每值3位）：
  [011, 101, 010, 111, 100, 000, 011, 110]
  （第6个位置填0，表示异常）
溢出区：[(index=5, value=100)]

位打包示意：
┌──────────────────────────────────────────┐
│  3bit │ 3bit │ 3bit │ 3bit │ 3bit │ ... │  → 主区域
│  3    │ 5    │ 2    │ 7    │ 4    │ ... │
│       │      │      │      │      │     │
│  溢出区: [(5, 100)]                        │
└──────────────────────────────────────────┘

PForDelta vs VByte：
┌───────────────┬──────────┬───────────┐
│ 特性           │ VByte    │ PForDelta │
├───────────────┼──────────┼───────────┤
│ 压缩率         │ 中等     │ 更高      │
│ 解压速度       │ 顺序解压 │ SIMD并行  │
│ 适合数据       │ 随机分布 │ 偏小值    │
│ 实现复杂度     │ 低       │ 中等      │
│ 块级随机访问   │ 不支持   │ 支持      │
└───────────────┴──────────┴───────────┘

为什么PForDelta快？
- 定长位打包，可用SIMD（SSE/AVX）一次解压多个值
- 现代CPU一个指令解压128/256位
- 实测解压吞吐可达数GB/s
```


## 词典压缩 (Dictionary Compression)


```
词典存储所有不重复的词条，压缩词典可大幅减少内存使用。

方案1：Front Coding（前端编码）
利用词条之间的公共前缀进行压缩

词条排序后：
automobile
automatic
automation
autonomous

Front Coding编码：
automobile
 8+atic    （公共前缀"automa"长度8，剩余"tic"）
 9+ion     （公共前缀"automat"长度9，剩余"ion"）
 6+nous    （公共前缀"autono"长度6，剩余"mous"）

原始：10+8+10+10 = 38字符
压缩：10+3+3+4 = 20字符（节省47%）

方案2：FST (Finite State Transducer)
Lucene/Elasticsearch使用的核心数据结构

FST特点：
- 有限状态转换器，类似DFA
- 有序词条可共享前缀和后缀
- 支持O(length)时间的精确查询
- 支持前缀查询和范围查询
- 内存占用极小

FST构建示意（键值对：cat→3, cats→8, dog→5）：
        ┌─→ [a]→[t]→[s]→(#,8)
(start)─┤
        └─→ [d]→[o]→[g]→(#,5)
              ↑
        [c]→[a]→[t]→(#,3)

实际构建过程（按排序顺序插入）：
1. 插入 cat→3：  构建 c→a→t→(#,3)
2. 插入 cats→8： 共享前缀cat，追加 s→(#,8)
3. 插入 dog→5：  构建独立路径 d→o→g→(#,5)
4. 冻结（minimize）：合并等价状态

方案3：哈希词典 + 碰撞链
- 桶数组 + 链表处理碰撞
- 空间效率一般，查询O(1)期望
- 适合不需要前缀查询的场景

方案4：Trie (前缀树)
- 共享公共前缀
- 支持前缀查询和自动补全
- 空间开销较大（每个节点有指针数组）
- 可用Patricia Trie / Radix Tree压缩
```


## Blocked Sort-Based Indexing (BSBI)


```
BSBI：大规模索引构建算法，处理超出内存的语料

核心流程：
┌─────────────────────────────────────────────┐
│  1. 将文档集分为多个块（每个块能放入内存）     │
│  2. 对每个块：                               │
│     a. 解析文档，提取 (词条, DocID) 对        │
│     b. 按词条排序                            │
│     c. 将排序后的块写入临时文件               │
│  3. 归并所有临时文件（外部归并排序）           │
│  4. 同时写出倒排表                           │
└─────────────────────────────────────────────┘

内存管理：
┌──────────────────────────────────┐
│           内存                    │
│  ┌──────────┐  ┌──────────────┐ │
│  │ 词条解析  │  │ 排序缓冲区   │ │
│  │ 缓冲区    │  │ (Term,DocID) │ │
│  │ (50%)    │  │ (50%)       │ │
│  └──────────┘  └──────────────┘ │
└──────────────────────────────────┘

外部归并排序：
临时文件1：[apple,1] [banana,3] [cat,2]
临时文件2：[apple,5] [bird,1] [cat,4]
临时文件3：[banana,1] [cat,7] [dog,2]

归并后输出：
apple → [Doc1, Doc5]
banana → [Doc1, Doc3]
bird → [Doc1]
cat → [Doc2, Doc4, Doc7]
dog → [Doc2]

性能分析：
- 假设内存1GB，文档集100GB
- 需要100个块 → 100个临时文件
- 归并阶段：100路归并，顺序I/O
- 总I/O量：2 × 数据大小（读+写）
- 使用顺序I/O最大化磁盘吞吐

SPIMI (Single-Pass In-Memory Indexing)：
- BSBI的改进版
- 不需要在内存中排序
- 每个块直接写出为独立的倒排索引片段
- 最后合并所有片段
- 优势：不需要维护DocID到词条的映射表
```


> **Note:** 实际搜索引擎使用BSBI/SPIMI的变种，结合压缩和并行处理，在商用硬件上构建TB级索引。


## 跳跃表与倒排表加速


```
跳跃表 (Skip List) 在倒排索引中的应用：
在长倒排表中插入跳跃指针，加速交集运算和随机访问。

结构示意：
倒排表：[3, 7, 12, 18, 25, 31, 42, 55, 68, 79, 88, 95]
跳跃指针（每4个元素一个）：
Level 1:  3 ---→ 25 ---→ 55 ---→ 88
Level 0:  3→7→12→18→25→31→42→55→68→79→88→95

查找DocID=42：
- 从头开始，3 < 42，跳到25
- 25 < 42，跳到55
- 55 > 42，回退到25
- 顺序查找：25→31→42 ✓
- 比较次数：3次 vs 7次（无跳跃表）

跳跃表在查询中的作用：
查询："搜索引擎 AND 索引 AND 压缩"
倒排表交集运算：
Posting("搜索引擎"): [3, 12, 25, 42, 68, 95]
Posting("索引"):      [7, 12, 31, 42, 55, 79]
Posting("压缩"):      [12, 25, 42, 68, 88]

无跳跃表：逐步推进指针，O(n+m+k)
有跳跃表：跳过不可能匹配的区间，减少比较次数

优化技巧：
1. 从最短的倒排表开始处理
2. 使用WAND/MaxScore算法跳过低分文档
3. 对倒排表分块压缩 + 跳跃表组合
4. 跳跃步长选择：sqrt(n)或每128个元素

倒排表存储格式（实际引擎）：
┌─────────────────────────────────────────────┐
│  Block 1 (128个DocID)                       │
│  ├── 压缩的DocID差值 (PForDelta/VByte)       │
│  ├── 压缩的TF值                              │
│  ├── 跳跃指针（指向下一个Block起始位置）      │
│  └── Block元数据（最小DocID, 最大DocID）      │
│                                             │
│  Block 2 (128个DocID)                       │
│  ├── ...                                    │
└─────────────────────────────────────────────┘
```
