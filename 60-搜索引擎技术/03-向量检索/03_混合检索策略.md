# 混合检索策略


## 混合检索策略


向量检索混合检索ColBERT


混合检索结合稀疏检索（关键词匹配）和稠密检索（语义匹配）的优势，通过重排序进一步提升结果质量。


## 稀疏检索 vs 稠密检索


```
稀疏检索 (Sparse Retrieval)：
基于词袋模型，用稀疏向量表示文档

代表方法：
- TF-IDF：词频 × 逆文档频率
- BM25：TF-IDF改进版，加入词频饱和和长度归一化
- SPLADE：学习稀疏表示，用BERT产生稀疏向量

特点：
┌─────────────────────────────────────────────┐
│  优势：                                      │
│  - 精确关键词匹配：搜"Python"不会返回Java结果│
│  - 可解释性强：可以看匹配了哪些词            │
│  - 高效：倒排索引支持O(1)级查询              │
│  - 零样本：不需要训练数据                    │
│                                             │
│  劣势：                                      │
│  - 词汇不匹配问题：搜"电脑"找不到"计算机"    │
│  - 同义词问题：搜"hotel"找不到"inn"          │
│  - 语义理解弱：无法理解查询意图              │
└─────────────────────────────────────────────┘

稠密检索 (Dense Retrieval)：
基于深度学习，用密集向量（embedding）表示文档

代表方法：
- DPR (Dense Passage Retrieval)：双编码器架构
- BGE/BGE-M3：中文向量模型
- E5：微软的嵌入模型
- Cohere Embed：商业嵌入模型

特点：
┌─────────────────────────────────────────────┐
│  优势：                                      │
│  - 语义理解：能理解同义词和近义表达          │
│  - 跨语言：多语言模型可跨语言检索            │
│  - 泛化性：可处理未见过的表达方式            │
│                                             │
│  劣势：                                      │
│  - 精确匹配弱：搜"Python"可能返回JavaScript  │
│  - 实体名称匹配不精确                        │
│  - 需要训练数据和计算资源                    │
│  - 向量索引需要大量内存                      │
└─────────────────────────────────────────────┘

为什么需要混合？
示例查询："Python机器学习教程"

稀疏检索能找到：
- 包含"Python"、"机器学习"、"教程"的精确文档
- 但找不到标题是"ML with Python Guide"的文档

稠密检索能找到：
- 语义相似的文档，包括用"人工智能入门"表述的文档
- 但可能返回JavaScript机器学习的文档（精确匹配弱）

混合检索 = 两者取长补短
```


## 混合检索融合策略


```
融合方法1：分数归一化 + 线性加权
┌─────────────────────────────────────────────┐
│  1. 分别执行稀疏检索和稠密检索               │
│  2. 将两组分数归一化到[0, 1]范围             │
│  3. 加权融合：                               │
│     final_score = α × sparse_norm + (1-α) × dense_norm│
│  4. 按final_score排序                        │
│                                             │
│  归一化方法：                                │
│  - Min-Max: (score - min) / (max - min)     │
│  - Z-Score: (score - mean) / std            │
│  - RRF: 基于排名而非分数                    │
└─────────────────────────────────────────────┘

融合方法2：Reciprocal Rank Fusion (RRF)
RRF是工业界最常用的融合方法，简单且效果好

RRF公式：
RRF_score(d) = Σ 1 / (k + rank_i(d))

其中：
- d：文档
- rank_i(d)：文档d在第i个检索器中的排名
- k：平滑常数（通常k=60）

示例：
查询："Python机器学习"

BM25结果：              向量检索结果：
1. DocA (精确匹配)      1. DocC (语义相似)
2. DocB                 2. DocA
3. DocC                 3. DocD
4. DocD                 4. DocB

RRF融合（k=60）：
DocA: 1/(60+1) + 1/(60+2) = 0.0164 + 0.0161 = 0.0325
DocB: 1/(60+2) + 1/(60+4) = 0.0161 + 0.0156 = 0.0317
DocC: 1/(60+3) + 1/(60+1) = 0.0159 + 0.0164 = 0.0323
DocD: 1/(60+4) + 1/(60+3) = 0.0156 + 0.0159 = 0.0315

融合排名：A > C > B > D

RRF的优势：
- 不需要分数归一化（基于排名而非分数）
- 不同检索器的分数尺度可以不同
- k=60是经验值，对大多数场景效果好
- 实现简单，效果稳健

融合方法3：轮询式 (Round-Robin)
- 稀疏结果和稠密结果交替取
- 最简单，效果一般
- 适合快速原型

融合方法4：学习式融合
- 训练一个小模型，输入是两个检索器的分数/排名
- 输出是融合后的分数
- 需要标注数据
```


## ColBERT 晚期交互模型


```
ColBERT (Contextualized Late Interaction)：
2020年由Stanford提出，结合了双编码器的效率和交叉编码器的效果

核心思想：
┌─────────────────────────────────────────────┐
│  双编码器（DPR等）：                         │
│  query → 单个向量                            │
│  doc → 单个向量                              │
│  相似度 = dot(query_vec, doc_vec)           │
│  问题：信息压缩严重，细粒度匹配丢失          │
│                                             │
│  交叉编码器（BERT re-ranker）：              │
│  [CLS] query [SEP] doc [SEP] → score       │
│  效果好，但无法预先计算doc表示              │
│  问题：在线计算开销大，无法用于检索阶段     │
│                                             │
│  ColBERT（晚期交互）：                       │
│  query → 多个向量（每个token一个）           │
│  doc → 多个向量（每个token一个）            │
│  相似度 = Σ max_sim(q_token, d_tokens)     │
│  折中：可以预先计算doc向量，保留细粒度信息  │
└─────────────────────────────────────────────┘

ColBERT架构：
┌─────────────────────────────────────────────┐
│  Query编码器（BERT）：                       │
│  输入："Python 机器学习"                     │
│  输出：[q1, q2, q3] 每个token的embedding    │
│  + 线性投影到低维（128维）                  │
│  + 可选的query扩展（[Q]特殊token）          │
│                                             │
│  Document编码器（共享BERT权重）：            │
│  输入："本文介绍Python的机器学习库..."       │
│  输出：[d1, d2, d3, ..., dn] 每个token的embedding│
│  + 线性投影到低维（128维）                  │
│  + 截断到固定长度（如128个token）           │
└─────────────────────────────────────────────┘

MaxSim 交互计算：
Score(q, d) = Σ max(dj · qi)
              qi∈q  dj∈d

示例：
Query tokens: [q1, q2]  (2个向量)
Doc tokens:   [d1, d2, d3, d4]  (4个向量)

交互矩阵：
       d1    d2    d3    d4
q1    0.8   0.3   0.9   0.2    → max = 0.9
q2    0.1   0.7   0.4   0.85   → max = 0.85

Score = 0.9 + 0.85 = 1.75

ColBERT的优势：
- Doc向量可以预先计算并索引
- 查询时只需编码query + MaxSim交互
- 保留了token级别的细粒度匹配
- 支持高效的ANN检索（先粗筛再精排）

ColBERT v2 改进：
- 残差压缩：doc向量用量化+残差存储
- 离线索引更小（每个向量只需几字节）
- 支持端到端训练
- 效果接近交叉编码器

使用ColBERT（Python）：
from colbert import Indexer, Searcher
from colbert.infra import Run, RunConfig, ColBERTConfig

# 索引
with Run().context(RunConfig(nranks=1)):
    config = ColBERTConfig(doc_maxlen=256, nbits=2)
    indexer = Indexer(checkpoint="colbertv2", config=config)
    indexer.index(name="my_index", collection="docs.tsv")

# 搜索
with Run().context(RunConfig(nranks=1)):
    searcher = Searcher(index="my_index")
    results = searcher.search("Python机器学习", k=10)
```


## 重排序 (Re-ranking) 流水线


```
典型检索-重排序流水线：
┌─────────────────────────────────────────────┐
│                                             │
│  查询                                       │
│   ↓                                        │
│  ┌─────────────┐  ┌─────────────┐          │
│  │ BM25检索    │  │ 向量检索    │  ← 第一阶段│
│  │ Top-1000   │  │ Top-1000   │  (召回)    │
│  └──────┬──────┘  └──────┬──────┘          │
│         └───────┬────────┘                  │
│                 ↓                            │
│         混合融合 (RRF)                       │
│         候选集 Top-200                       │
│                 ↓                            │
│     ┌────────────────────┐                  │
│     │ Cross-Encoder重排序 │  ← 第二阶段     │
│     │ Top-200 → Top-20   │  (精排)         │
│     └──────────┬─────────┘                  │
│                ↓                             │
│     ┌────────────────────┐                  │
│     │ 业务逻辑过滤        │  ← 第三阶段     │
│     │ 权限/去重/多样性    │  (后处理)       │
│     └──────────┬─────────┘                  │
│                ↓                             │
│           Top-10 结果                       │
└─────────────────────────────────────────────┘

Cross-Encoder 重排序器：
┌─────────────────────────────────────────────┐
│  输入：[CLS] query [SEP] document [SEP]     │
│  输出：相关性分数                             │
│                                             │
│  特点：                                      │
│  - query和document同时输入BERT              │
│  - 注意力机制可以捕获token级交互             │
│  - 效果最好，但计算最慢                      │
│  - 适合精排（候选数少）                      │
│  - 不适合检索（候选数多）                    │
└─────────────────────────────────────────────┘

Cross-Encoder vs Bi-Encoder vs ColBERT：
┌────────────────┬───────────┬───────────┬──────────┐
│ 特性            │ Bi-Encoder│ ColBERT   │Cross-Enc │
├────────────────┼───────────┼───────────┼──────────┤
│ Doc预计算      │ 可以      │ 可以      │ 不可以   │
│ 交互粒度       │ 向量级    │ Token级   │ Token级  │
│ 检索速度       │ 最快      │ 快        │ 最慢     │
│ 重排效果       │ 一般      │ 好        │ 最好     │
│ 适合阶段       │ 召回      │ 召回+精排 │ 精排     │
│ 候选数         │ 100万+    │ 1万+      │ 100-500  │
│ 推理延迟/doc   │ <1ms      │ ~5ms      │ ~20ms    │
└────────────────┴───────────┴───────────┴──────────┘

常用Cross-Encoder模型：
- ms-marco-MiniLM：英文，速度快
- bge-reranker-v2-m3：多语言，效果好
- cohere-rerank：商业API
- bce-reranker-base_v1：中文优化

Python实现重排序流水线：
from sentence_transformers import CrossEncoder

# 第一阶段：BM25 + 向量检索混合
bm25_results = bm25_search(query, top_k=1000)
dense_results = vector_search(query, top_k=1000)
candidates = rrf_fusion(bm25_results, dense_results)[:200]

# 第二阶段：Cross-Encoder重排序
reranker = CrossEncoder('BAAI/bge-reranker-v2-m3')
pairs = [(query, doc.text) for doc in candidates]
scores = reranker.predict(pairs)

# 按分数排序
ranked = sorted(zip(candidates, scores), key=lambda x: -x[1])
top_results = ranked[:10]
```


## 实际系统的混合检索架构


```
Elasticsearch 8.x + kNN混合检索：
┌─────────────────────────────────────────────┐
│  PUT my-index                               │
│  {                                          │
│    "mappings": {                            │
│      "properties": {                       │
│        "title": { "type": "text" },        │
│        "content": { "type": "text" },      │
│        "embedding": {                      │
│          "type": "dense_vector",           │
│          "dims": 768,                      │
│          "index": true,                    │
│          "similarity": "cosine"            │
│        }                                   │
│      }                                     │
│    }                                       │
│  }                                         │
│                                             │
│  // 混合查询                                │
│  POST my-index/_search                     │
│  {                                          │
│    "query": {                              │
│      "bool": {                            │
│        "must": [                          │
│          {"match": {"content": "机器学习"}}│
│        ]                                   │
│      }                                     │
│    },                                      │
│    "knn": {                                │
│      "field": "embedding",                │
│      "query_vector": [...],               │
│      "k": 10,                             │
│      "num_candidates": 100                │
│    },                                      │
│    "rank": {                               │
│      "rrf": {}                            │
│    }                                       │
│  }                                         │
└─────────────────────────────────────────────┘

Weaviate 混合搜索：
results = client.query.get("Article", ["title", "content"]) \
    .with_hybrid(
        query="Python机器学习",
        alpha=0.5  # 0=纯BM25, 1=纯向量
    ) \
    .with_limit(10) \
    .do()

Qdrant 混合搜索（使用Query API）：
results = client.query_points(
    collection_name="articles",
    query=models.FusionQuery(
        fusion=models.Fusion.RRF
    ),
    prefetch=[
        models.Prefetch(
            query=query_vector,
            using="dense",
            limit=20
        ),
        models.Prefetch(
            query=models.SparseVector(...),
            using="sparse",
            limit=20
        )
    ],
    limit=10
)

架构选择建议：
┌─────────────────────────────────────────────┐
│  小规模（<100万文档）：                       │
│  Elasticsearch + kNN向量字段 + RRF融合      │
│  全部在ES中完成，部署简单                    │
│                                             │
│  中等规模（100万-1000万）：                  │
│  Elasticsearch(BM25) + Qdrant(向量)         │
│  应用层做RRF融合 + Cross-Encoder重排序      │
│                                             │
│  大规模（>1000万）：                        │
│  自建检索服务                                │
│  BM25：Lucene/Elasticsearch集群             │
│  向量：Milvus/Faiss集群                     │
│  融合+重排：专用服务（GPU推理）              │
│  ColBERT索引：磁盘向量 + ANN检索            │
└─────────────────────────────────────────────┘
```


> **Note:** 混合检索是现代搜索/推荐系统的标配方案。RRF是最简单有效的融合方法，Cross-Encoder重排序进一步提升结果质量。ColBERT的晚期交互模型在效率和效果之间取得了很好的平衡。
