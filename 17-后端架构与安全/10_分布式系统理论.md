## CAP 定理与 PACELC


```
// ========== CAP 定理 ==========
// 分布式系统最多同时满足两个:
//
// C - Consistency (一致性): 所有节点看到相同数据
// A - Availability (可用性): 每个请求都能得到响应
// P - Partition Tolerance (分区容忍): 网络中断仍能运行
//
//          CP                     AP
//   ┌──────────────┐   ┌──────────────┐
//   │ 分区发生时    │   │ 分区发生时    │
//   │ 停止服务保一致 │   │ 继续服务但可能│
//   │              │   │ 数据不一致   │
//   │  ZooKeeper   │   │  DNS         │
//   │  etcd        │   │  CDN         │
//   │  MongoDB(默认)│   │  Cassandra   │
//   │  HBase       │   │  DynamoDB    │
//   └──────────────┘   └──────────────┘
//
//   CA (实际不存在, 因为网络总会分区)
//   单机系统 = CA (无分区)

// ========== PACELC 扩展 ==========
// CAP 只考虑分区时, PACELC 考虑正常时:
//
// 分区时 (Partition):
//   P → A 或 C
// 正常时 (Else):
//   E → L (低延迟) 或 C (强一致)
//
// 举例:
//   Cassandra: PA/EL (可用 + 低延迟)
//   MongoDB:   PC/EC (一致 + 强一致)
//   DynamoDB:  PA/EL (可用 + 低延迟)

// ========== 一致性模型 (由强到弱) ==========
// 1. 强一致性 (Linearizability)
//    写完成后, 任何读都看到最新值
//    性能差, 分布式难实现

// 2. 顺序一致性 (Sequential)
//    每个进程的操作顺序一致
//    不要求实时性

// 3. 因果一致性 (Causal)
//    有因果关系的事件有序
//    无关事件可并发的

// 4. 读己之写 (Read Your Writes)
//    写后总能读到自己的写入
//    其他用户可能读不到

// 5. 最终一致性 (Eventual)
//    不保证何时, 但最终一致
//    DNS: TTL 后最终一致
//    最简单的模型
```


## Raft 共识算法


```
// ========== Raft 核心 ==========
// 管理复制日志, 保证多节点数据一致
// 比 Paxos 更易理解

// 三种角色:
// ┌─────────┐    选举     ┌──────────┐
// │ Leader  │←──────────→│ Follower  │
// │ (主)    │             │ (从)      │
// └────┬────┘   ┌────────┴──────┐
//      │        │  Candidate    │
//      │        │  (候选者)     │
//      └────────┴───────────────┘

// ========== Leader 选举 ==========
// 1. Follower 超时未收到心跳 → 变成 Candidate
// 2. Candidate 发起投票请求
// 3. 获得多数票 → 成为 Leader
// 4. Leader 定期发送心跳 (AppendEntries)
//
// 任期 (Term): 每次选举递增
// 随机超时 (150-300ms): 防止选票分裂

// ========== 日志复制 ==========
// 1. 客户端请求 → Leader
// 2. Leader 追加日志到本地
// 3. Leader 并行发送 AppendEntries 到 Followers
// 4. 多数写入成功 → 日志已提交 (committed)
// 5. Leader 应用日志到状态机
// 6. Leader 响应客户端
// 7. Followers 在下次心跳确认提交

// 流程:
// Client → Leader → Follower 1 (ack)
//                  → Follower 2 (ack)  ← 多数提交
//                  → Follower 3 (慢)
// Leader 提交 → 响应 Client
//            → 下次心跳通知 Followers 提交

// ========== 安全性保证 ==========
// 1. 选举安全: 每个任期最多一个 Leader
// 2. 日志匹配: 相同的 (term, index) 内容相同
// 3. 完整性: 被选上的 Leader 包含所有已提交日志
// 4. 状态机安全: 所有节点按相同顺序执行相同命令

// ========== Go etcd/Raft 示例 ==========
// import "go.etcd.io/raft/v3"

// Node 接口:
// node.Start(data)     // 提议数据
// node.Ready()         // 获取就绪的日志
// node.Advance()       // 处理完成
// node.Step(msg)       // 处理接收的消息
```


## 分布式事务


```
// ========== 2PC (两阶段提交) ==========
// 协调者 (Coordinator) + 参与者 (Participants)
//
// 阶段 1: 准备 (Prepare)
//   协调者 → 所有参与者: "准备提交?"
//   参与者: 执行事务, 写 Undo/Redo 日志
//   参与者 → 协调者: Yes (准备完成) / No (失败)
//
// 阶段 2: 提交 (Commit)
//   全部 Yes → 协调者 → 所有参与者: "提交"
//   有 No    → 协调者 → 所有参与者: "回滚"
//
// 问题:
//   同步阻塞: 参与者持有锁等待
//   单点故障: 协调者宕机
//   脑裂: 部分参与者收不到提交
//   适用: 短事务, 小规模

// ========== Saga 分布式事务 ==========
// 每个操作都有对应的补偿操作 (Compensation)
// 不持有锁, 适合长事务

// 编排 Saga (Choreography):
// 每个服务发布事件触发下一步
//
// Order 服务 → OrderCreated
//   → Payment 服务 → PaymentProcessed
//     → Inventory 服务 → InventoryUpdated
//       → Notification 服务 → Notified
//
// 补偿 (失败时):
// Payment 失败 → Order 取消订单 → Inventory 恢复库存

// 编排器 Saga (Orchestrator):
// Saga Manager 协调所有步骤
//
// Saga Manager:
//   1. 调用 Order 服务创建订单
//   2. 调用 Payment 服务处理支付
//   3. 调用 Inventory 服务扣库存
//   4. 完成
```
