# Kafka 消费者

## 一、消费者配置

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "order-consumer-group");
props.put("key.deserializer", StringDeserializer.class.getName());
props.put("value.deserializer", StringDeserializer.class.getName());

// 消费配置
props.put("auto.offset.reset", "earliest");     // 无 Offset 时从最早开始
props.put("enable.auto.commit", false);          // 手动提交
props.put("max.poll.records", 500);              // 每次最多拉取 500 条
props.put("max.poll.interval.ms", 300000);       // 处理超时 5 分钟
props.put("session.timeout.ms", 10000);          // 会话超时

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(List.of("order-events"));
```

## 二、消费模式

```java
// 1. 自动提交（简单但不可靠）
props.put("enable.auto.commit", true);
props.put("auto.commit.interval.ms", 5000);
// 每 5 秒自动提交 Offset，可能重复消费

// 2. 手动同步提交（推荐）
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        process(record);
    }
    consumer.commitSync();  // 处理完一批后同步提交
}

// 3. 手动异步提交（高性能）
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        process(record);
    }
    consumer.commitAsync((offsets, exception) -> {
        if (exception != null) {
            log.error("提交失败: {}", exception.getMessage());
        }
    });
}

// 4. 指定 Offset 提交
Map<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();
for (ConsumerRecord<String, String> record : records) {
    TopicPartition tp = new TopicPartition(record.topic(), record.partition());
    offsets.put(tp, new OffsetAndMetadata(record.offset() + 1));
}
consumer.commitSync(offsets);
```

## 三、消费分配策略

```java
// 1. RangeAssignor（默认）- 按范围分配
// Topic 有 6 分区，3 消费者 -> 每人 2 分区
// Topic 有 7 分区，3 消费者 -> 2,2,3

// 2. RoundRobinAssignor - 轮询分配
// 所有 Topic 的分区打散后轮询

// 3. StickyAssignor - 粘性分配
// 尽量保持之前的分配，减少 Rebalance 迁移

// 4. CooperativeStickyAssignor - 协作式粘性分配（推荐）
// 增量 Rebalance，不会 Stop-the-World
props.put("partition.assignment.strategy",
    "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");
```

## 四、Rebalance（重平衡）

```java
// Rebalance 监听器
consumer.subscribe(List.of("order-events"), new ConsumerRebalanceListener() {
    @Override
    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {
        // 分区被回收前 - 提交 Offset
        consumer.commitSync(currentOffsets);
        log.info("分区被回收: {}", partitions);
    }

    @Override
    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {
        // 分区被分配后
        log.info("分区被分配: {}", partitions);
    }

    @Override
    public void onPartitionsLost(Collection<TopicPartition> partitions) {
        // 分区丢失（无需提交 Offset）
    }
});
```

## 五、Spring Kafka 消费者

```java
@Component
public class OrderConsumer {

    // 简单消费
    @KafkaListener(topics = "order-events", groupId = "order-group")
    public void consume(ConsumerRecord<String, String> record) {
        log.info("收到消息: partition={}, offset={}, value={}",
            record.partition(), record.offset(), record.value());
    }

    // 手动 ACK
    @KafkaListener(topics = "order-events")
    public void consumeWithAck(ConsumerRecord<String, String> record,
                               Acknowledgment ack) {
        process(record);
        ack.acknowledge();
    }

    // 批量消费
    @KafkaListener(topics = "order-events", containerFactory = "batchFactory")
    public void batchConsume(List<ConsumerRecord<String, String>> records) {
        log.info("批量消费 {} 条", records.size());
    }

    // 指定分区消费
    @KafkaListener(
        topicPartitions = @TopicPartition(
            topic = "order-events",
            partitions = {"0", "1", "2"}
        )
    )
    public void consumePartition(ConsumerRecord<String, String> record) {
        process(record);
    }
}
```

## 六、注意事项

1. **max.poll.records 要与处理能力匹配**，过大会导致 Rebalance
2. **max.poll.interval.ms 要大于处理时间**，否则触发 Rebalance
3. **手动提交比自动提交更可靠**，建议 always 手动提交
4. **CooperativeStickyAssignor 是最平滑的分配策略**，推荐使用
5. **消费端要做好幂等处理**，Rebalance 可能导致重复消费
