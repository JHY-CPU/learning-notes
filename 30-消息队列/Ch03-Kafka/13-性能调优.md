# Kafka 性能调优

## 一、生产者调优

```java
Properties props = new Properties();
// 批量大小 - 增大可提升吞吐量
props.put("batch.size", 65536);           // 64KB（默认 16KB）

// 等待时间 - 增大可攒更多消息
props.put("linger.ms", 20);              // 20ms（默认 0）

// 缓冲区
props.put("buffer.memory", 67108864);    // 64MB（默认 32MB）

// 压缩
props.put("compression.type", "lz4");    // lz4 最快（推荐）

// 请求大小
props.put("max.request.size", 1048576);  // 1MB（默认 1MB）
```

```yaml
# 吞吐量 vs 延迟 权衡
高吞吐配置:
  batch.size: 65536
  linger.ms: 20
  compression.type: lz4
  acks: 1
  结果: 吞吐量最高，延迟较高

低延迟配置:
  batch.size: 0
  linger.ms: 0
  compression.type: none
  acks: 1
  结果: 延迟最低，吞吐量较低
```

## 四、分区数优化

```java
// 分区数规划
// 公式: 目标吞吐量 / 单分区吞吐量
// 单分区写入: 约 10-50 MB/s
// 单分区读取: 约 20-100 MB/s

// 示例:
// 目标: 500 MB/s 写入
// 单分区: 25 MB/s
// 所需分区: 500 / 25 = 20 个分区

// 注意事项:
// - 分区数过多会增加 Leader 选举时间
// - 每个分区需要文件句柄
// - 建议单 Topic 不超过 100 个分区
```

## 五、磁盘与 OS 调优

```bash
# 文件系统
# 使用 XFS（推荐）或 ext4
mkfs.xfs /dev/sdb

# 挂载选项
# /etc/fstab
/dev/sdb /data/kafka xfs noatime,nodiratime,nobarrier 0 0

# 文件描述符
ulimit -n 100000
# /etc/security/limits.conf
* soft nofile 100000
* hard nofile 100000

# 虚拟内存
vm.swappiness=1
vm.dirty_ratio=60
vm.dirty_background_ratio=5

# 网络
net.core.wmem_max=16777216
net.core.rmem_max=16777216
net.ipv4.tcp_wmem=4096 87380 16777216
net.ipv4.tcp_rmem=4096 87380 16777216
```

## 六、性能测试

```bash
# 生产者性能测试
kafka-producer-perf-test.sh \
  --topic perf-test \
  --num-records 10000000 \
  --record-size 1024 \
  --throughput -1 \
  --producer-props bootstrap.servers=localhost:9092 \
    acks=1 batch.size=65536 linger.ms=20 compression.type=lz4

# 消费者性能测试
kafka-consumer-perf-test.sh \
  --bootstrap-server localhost:9092 \
  --topic perf-test \
  --messages 10000000 \
  --threads 3
```

## 七、注意事项

1. **压缩类型选择 lz4 或 zstd**，在压缩率和 CPU 之间平衡
2. **linger.ms 不要设置太大**，超过 100ms 会影响延迟
3. **分区数不是越多越好**，会增加元数据和 Leader 选举开销
4. **磁盘性能是 Kafka 最大瓶颈**，SSD 比 HDD 快 5-10 倍
5. **Linux 系统参数调优效果显著**，生产环境务必优化
