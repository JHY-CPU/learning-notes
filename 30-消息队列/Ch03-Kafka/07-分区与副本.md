# 分区与副本

## 一、Partition 分区

Partition 是 Kafka 的**最小并行单元**，每个 Partition 只能被同一 Consumer Group 中的一个消费者消费。

```java
// 分区数规划
// 公式: 分区数 = max(目标吞吐量 / 单分区吞吐量, 消费者数)
// 示例: 目标 100MB/s, 单分区 10MB/s -> 10 个分区

// 创建 Topic 指定分区数
kafka-topics --bootstrap-server localhost:9092 --create \
  --topic high-throughput \
  --partitions 12 \
  --replication-factor 3

// 增加分区数（只能增加不能减少）
kafka-topics --bootstrap-server localhost:9092 --alter \
  --topic high-throughput \
  --partitions 18
```

## 二、副本机制

Kafka 通过**多副本**保证数据可靠性，每个 Partition 有一个 Leader 和多个 Follower。

```
Partition 0 副本分布:
  Broker 1: Leader    <- 处理读写请求
  Broker 2: Follower  <- 同步 Leader 数据
  Broker 3: Follower  <- 同步 Leader 数据

ISR (In-Sync Replicas): [1, 2, 3]  # 同步中的副本
OSR (Out-of-Sync Replicas): []     # 落后的副本
```

```bash
# 查看副本详情
kafka-topics --bootstrap-server localhost:9092 \
  --describe --topic order-events

# Topic: order-events  PartitionCount: 6  ReplicationFactor: 3
# Partition: 0  Leader: 1  Replicas: 1,2,3  Isr: 1,2,3
# Partition: 1  Leader: 2  Replicas: 2,3,1  Isr: 2,3,1
# Partition: 2  Leader: 3  Replicas: 3,1,2  Isr: 3,1,2
```

## 三、Leader 选举

```java
// Leader 故障时，从 ISR 中选举新 Leader
// 配置参数
Properties props = new Properties();
props.put("acks", "all");                      // 所有 ISR 确认
props.put("min.insync.replicas", 2);           // 最少 2 个同步副本

// 如果 ISR 数量 < min.insync.replicas，生产者抛出 NotEnoughReplicasException
```

```bash
# 手动触发 Leader 选举
kafka-leader-election --bootstrap-server localhost:9092 \
  --election-type preferred \
  --all-topic-partitions

# 指定 Partition 选举
kafka-leader-election --bootstrap-server localhost:9092 \
  --topic order-events --partition 0 \
  --election-type preferred
```

## 四、unclean.leader.election

```properties
# 是否允许非 ISR 副本成为 Leader
# false (默认): 不允许，保证数据一致性，但可能不可用
# true: 允许，可能丢失数据，但保证可用性

unclean.leader.election.enable=false
```

## 五、副本同步策略

```java
// 消息写入流程
// 1. Producer 发送消息到 Leader
// 2. Leader 写入本地日志
// 3. Follower 从 Leader 拉取数据
// 4. Follower 写入本地日志
// 5. Leader 收到所有 ISR 的确认
// 6. Producer 收到确认

// 可靠性保证:
// acks=0: 不等待确认，可能丢失
// acks=1: Leader 确认即可，Leader 挂了可能丢失
// acks=all: 所有 ISR 确认，最可靠
```

## 六、分区再平衡

```bash
# 生成分区分配计划
kafka-reassign-partitions --bootstrap-server localhost:9092 \
  --generate --topics-to-move-json-file topics.json \
  --broker-list "1,2,3"

# 执行分区再平衡
kafka-reassign-partitions --bootstrap-server localhost:9092 \
  --execute --reassignment-json-file reassignment.json

# 验证再平衡进度
kafka-reassign-partitions --bootstrap-server localhost:9092 \
  --verify --reassignment-json-file reassignment.json

# topics.json 示例
{
  "topics": [{"topic": "order-events"}],
  "version": 1
}
```

## 七、注意事项

1. **分区数只能增加不能减少**，减少需要重建 Topic
2. **分区数过多会增加 Leader 选举时间**，建议单 Topic 不超过 100 个分区
3. **副本数不能超过 Broker 数量**
4. **unclean.leader.election=false 是最安全的选择**
5. **跨机架部署副本**可以提高容灾能力
