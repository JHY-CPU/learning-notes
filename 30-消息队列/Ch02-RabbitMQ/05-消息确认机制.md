# 消息确认机制

## 一、Producer Confirm（生产者确认）

RabbitMQ 提供 Publisher Confirm 机制，确保消息被 Broker 正确接收。

### 同步确认

```java
// 同步确认 - 每条消息等待确认
public void sendSyncConfirm(Message message) throws Exception {
    Connection connection = factory.newConnection();
    Channel channel = connection.createChannel();

    // 开启 Confirm 模式
    channel.confirmSelect();

    // 发送消息
    channel.basicPublish("exchange", "routing-key", null, message.getBody());

    // 等待确认（超时 5 秒）
    if (channel.waitForConfirms(5000)) {
        System.out.println("消息发送成功");
    } else {
        System.out.println("消息发送失败，需要重试");
    }
}
```

### 异步确认（推荐）

```java
// 异步确认 - 高性能方案
public void sendAsyncConfirm() throws Exception {
    Channel channel = connection.createChannel();
    channel.confirmSelect();

    // 存储待确认的消息
    ConcurrentNavigableMap<Long, Message> outstandingMessages = new ConcurrentSkipListMap<>();

    // Confirm 回调
    channel.addConfirmListener(new ConfirmListener() {
        @Override
        public void handleAck(long deliveryTag, boolean multiple) {
            if (multiple) {
                // 批量确认
                ConcurrentNavigableMap<Long, Message> confirmed =
                    outstandingMessages.headMap(deliveryTag, true);
                confirmed.clear();
            } else {
                outstandingMessages.remove(deliveryTag);
            }
        }

        @Override
        public void handleNack(long deliveryTag, boolean multiple) {
            // 消息未被确认，需要重试
            Message msg = outstandingMessages.get(deliveryTag);
            retryService.retry(msg);
        }
    });

    // 发送消息
    for (Message msg : messages) {
        long seqNo = channel.getNextPublishSeqNo();
        outstandingMessages.put(seqNo, msg);
        channel.basicPublish("exchange", "key", null, msg.getBody());
    }
}
```

## 二、Consumer ACK（消费者确认）

消费者处理完消息后需要发送 ACK 确认，RabbitMQ 才会将消息从队列中移除。

### 手动 ACK

```java
@Component
public class ManualAckConsumer {

    @RabbitListener(queues = "order-queue")
    public void processOrder(Message message, Channel channel) throws IOException {
        long deliveryTag = message.getMessageProperties().getDeliveryTag();

        try {
            Order order = JSON.parseObject(message.getBody(), Order.class);
            orderService.process(order);

            // 处理成功，确认消息
            // 参数1: deliveryTag - 消息标签
            // 参数2: multiple - 是否批量确认
            channel.basicAck(deliveryTag, false);

        } catch (BusinessException e) {
            // 业务异常 - 拒绝消息，不重新入队（进入死信队列）
            channel.basicNack(deliveryTag, false, false);

        } catch (Exception e) {
            // 系统异常 - 拒绝消息，重新入队重试
            channel.basicNack(deliveryTag, false, true);
        }
    }
}
```

### ACK 方法对比

```java
// 1. basicAck - 确认消息
channel.basicAck(deliveryTag, multiple);
// 消息从队列中移除

// 2. basicNack - 拒绝消息（支持批量）
channel.basicNack(deliveryTag, multiple, requeue);
// requeue=true: 消息重新入队
// requeue=false: 消息进入死信队列或被丢弃

// 3. basicReject - 拒绝单条消息
channel.basicReject(deliveryTag, requeue);
// 不支持批量，其他同 basicNack
```

## 三、Spring Boot ACK 配置

```yaml
spring:
  rabbitmq:
    listener:
      simple:
        acknowledge-mode: manual    # 手动确认
        prefetch: 100               # 预取数量
        retry:
          enabled: true             # 开启重试
          max-attempts: 3           # 最大重试次数
          initial-interval: 1000    # 重试间隔
          multiplier: 2.0           # 间隔倍增
```

```java
// Spring Boot 自动重试配置下，不需要手动处理 NACK
@RabbitListener(queues = "order-queue")
public void processOrder(Order order) {
    // Spring 自动处理 ACK/NACK
    orderService.process(order);
    // 成功 -> 自动 ACK
    // 失败 -> 自动重试，重试耗尽 -> 进入死信
}
```

## 四、预取机制（Prefetch）

```java
// Prefetch 控制消费者未确认消息的数量
// 防止消费者被大量消息压垮
@Configuration
public class PrefetchConfig {

    @Bean
    public SimpleRabbitListenerContainerFactory containerFactory() {
        SimpleRabbitListenerContainerFactory factory = new SimpleRabbitListenerContainerFactory();
        factory.setConnectionFactory(connectionFactory());
        factory.setPrefetchCount(50);      // 每次预取 50 条
        factory.setAcknowledgeMode(AcknowledgeMode.MANUAL);
        return factory;
    }
}
```

## 五、注意事项

1. **生产环境务必使用手动 ACK**，自动 ACK 在消费者崩溃时会丢失消息
2. **ACK 要在业务处理完成后发送**，过早 ACK 可能导致消息丢失
3. **Prefetch 设置过大会导致消息堆积在消费者内存**，过小会降低吞吐
4. **NACK 重新入队要谨慎**，持续失败的消息会导致队列阻塞
5. **异步 Confirm 比同步 Confirm 性能高很多**，生产环境推荐异步
