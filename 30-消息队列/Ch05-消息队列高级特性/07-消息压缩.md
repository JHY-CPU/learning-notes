# 消息压缩

## 一、压缩算法对比

| 算法 | 压缩率 | CPU 开销 | 速度 | 适用场景 |
|------|--------|---------|------|---------|
| none | 无 | 无 | 最快 | 小消息 |
| gzip | 最高 | 最高 | 慢 | 存储优化 |
| snappy | 中等 | 中等 | 快 | 通用场景 |
| lz4 | 低 | 最低 | 最快 | 高吞吐 |
| zstd | 高 | 中等 | 快 | 推荐 |

## 二、Kafka 压缩配置

```java
// 生产者压缩
props.put("compression.type", "snappy");  // gzip / lz4 / zstd

// Broker 端压缩（对已有消息重新压缩）
// log.compression.type=snappy
```

## 三、应用层压缩

```java
// 大消息应用层压缩
public void sendLargeMessage(Object data) {
    String json = JSON.toJSONString(data);
    byte[] compressed = Snappy.compress(json.getBytes());
    kafkaTemplate.send("topic", compressed);
}

// 消费端解压
@KafkaListener(topics = "topic")
public void consume(byte[] compressed) {
    byte[] decompressed = Snappy.uncompress(compressed);
    String json = new String(decompressed);
    Order order = JSON.parseObject(json, Order.class);
}
```

## 四、注意事项

1. **压缩在 Producer 端执行**，Broker 存储压缩后的数据
2. **小消息（<1KB）压缩意义不大**，甚至可能变大
3. **zstd 是 Kafka 2.1+ 推荐的算法**，压缩率和速度平衡最好
4. **压缩会增加 CPU 开销**，在吞吐量和资源之间权衡
5. **批量发送 + 压缩效果最好**，因为可以跨消息压缩
