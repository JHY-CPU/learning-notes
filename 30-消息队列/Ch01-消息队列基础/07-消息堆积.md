# 消息堆积

## 一、什么是消息堆积

消息堆积是指**生产者产生消息的速度远大于消费者消费的速度**，导致消息在 Broker 中不断积累，可能撑爆磁盘或内存。

```
生产速率: 10000 msg/s
消费速率:  1000 msg/s
堆积速度:  9000 msg/s  ->  1小时堆积 3240 万条
```

## 二、堆积的原因分析

### 1. 消费能力不足

```java
// 消费逻辑太慢 - 每条消息处理 100ms，单线程只能 10 msg/s
@KafkaListener(topics = "order-topic", concurrency = "1")
public void consume(ConsumerRecord<String, String> record) {
    // 问题1：复杂业务逻辑
    Order order = parseOrder(record.value());
    enrichOrder(order);           // 调用外部接口
    validateInventory(order);     // 查询库存
    calculateDiscount(order);     // 计算优惠
    saveToDatabase(order);        // 写数据库
    // 问题2：同步调用外部服务
    emailService.send(order);     // 发送邮件
}
```

### 2. 生产者突发流量

```java
// 秒杀活动瞬间产生大量消息
@PostMapping("/seckill")
public Result seckill(Long productId) {
    // 瞬间 10 万请求，产生 10 万条消息
    mq.send("seckill-topic", new SeckillRequest(productId, userId));
    return Result.success("排队中");
}
```

### 3. 消费端异常

```java
// 消费者频繁抛异常，触发重试
@KafkaListener(topics = "order-topic")
public void consume(String message) {
    try {
        process(message);
    } catch (Exception e) {
        // 重试导致消费速率大幅下降
        throw e;  // 触发重试机制
    }
}
```

## 三、堆积处理方案

### 1. 增加消费者数量

```yaml
# Kafka 增加消费实例
spring:
  kafka:
    listener:
      concurrency: 6  # 每个实例启动 6 个消费者线程
```

```java
// 动态调整消费者并发数
@KafkaListener(
    topics = "order-topic",
    groupId = "order-group",
    concurrency = "10"  // 增加并发
)
public void consume(ConsumerRecord<String, String> record) {
    processAsync(record);  // 异步处理
}
```

### 2. 批量消费

```java
// 批量消费 - 一次拉取多条消息
@KafkaListener(
    topics = "order-topic",
    containerFactory = "batchFactory"
)
public void batchConsume(List<ConsumerRecord<String, String>> records) {
    List<Order> orders = records.stream()
        .map(r -> JSON.parseObject(r.value(), Order.class))
        .collect(Collectors.toList());

    // 批量处理，减少 IO
    orderService.batchInsert(orders);
    // 批量提交 Offset
}

@Configuration
public class KafkaConfig {
    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> batchFactory() {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory());
        factory.setBatchListener(true);  // 开启批量模式
        return factory;
    }
}
```

### 3. 降级处理

```java
// 积压时降级为批量处理
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record) {
    long lag = getConsumerLag();  // 获取积压量

    if (lag > 100000) {
        // 大量积压 - 降级处理，只做核心逻辑
        orderService.quickProcess(record.value());
    } else {
        // 正常处理完整逻辑
        orderService.fullProcess(record.value());
    }
}
```

### 4. 临时扩容

```java
// 监控积压量，自动扩容消费者
@Scheduled(fixedRate = 60000)
public void autoScale() {
    long lag = kafkaAdmin.getConsumerLag("order-topic", "order-group");
    int currentInstances = getCurrentConsumerInstances();

    if (lag > 100000 && currentInstances < MAX_INSTANCES) {
        // 积压过多，扩容消费者实例
        k8sService.scaleDeployment("order-consumer", currentInstances + 2);
    } else if (lag < 1000 && currentInstances > MIN_INSTANCES) {
        // 积压消除，缩容
        k8sService.scaleDeployment("order-consumer", currentInstances - 1);
    }
}
```

## 四、监控与告警

```yaml
# Prometheus 监控配置
kafka:
  consumer:
    metrics:
      - name: kafka_consumer_lag
        threshold: 50000
        alert: "消息积压超过 5 万"
      - name: kafka_consumer_rate
        threshold: 100
        alert: "消费速率低于 100 msg/s"
```

## 五、注意事项

1. **增加消费者不能超过分区数**，超过的部分会空闲
2. **批量消费要注意单次处理时间**，避免超时触发 Rebalance
3. **降级方案要提前设计**，不要等到积压了再临时改造
4. **监控消费延迟是最有效的预警手段**，及时发现积压趋势
5. **根本解决是提升消费能力**，而不是无限增加消费者数量
