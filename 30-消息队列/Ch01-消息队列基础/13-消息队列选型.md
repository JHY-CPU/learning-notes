# 消息队列选型

## 一、选型考虑维度

选择消息队列时需要从以下几个维度评估：

| 维度 | 关键问题 |
|------|---------|
| 功能需求 | 是否需要事务消息、延迟消息、消息回溯 |
| 性能需求 | 吞吐量要求、延迟容忍度 |
| 可靠性 | 消息不丢失的要求等级 |
| 运维成本 | 团队技术栈、运维能力 |
| 生态集成 | 与现有框架的集成难度 |
| 社区活跃度 | 文档质量、社区支持 |

## 二、场景化选型指南

### 场景1：订单系统 / 电商交易

```java
// 需求：事务消息、延迟消息（超时取消）、消息可靠性
// 推荐：RocketMQ

// 事务消息 - 保证本地事务和消息发送的原子性
rocketMQTemplate.sendMessageInTransaction("order-topic", msg, order);

// 延迟消息 - 30分钟超时取消
rocketMQTemplate.syncSend("order-delay-topic", msg, 3000, 16); // 延迟级别16=30m
```

### 场景2：日志收集 / 大数据管道

```java
// 需求：超高吞吐、数据持久化、可回溯
// 推荐：Kafka

// 日志写入 Kafka
kafkaTemplate.send("app-logs", JSON.toJSONString(logEntry));

// 配合 Flink/Spark 做实时计算
StreamsBuilder builder = new StreamsBuilder();
builder.stream("app-logs")
    .filter((key, value) -> parseLevel(value).equals("ERROR"))
    .to("error-logs");
```

### 场景3：微服务异步通信

```java
// 需求：可靠投递、灵活路由、低延迟
// 推荐：RabbitMQ

// 复杂路由 - 根据 routing-key 路由到不同队列
rabbitTemplate.convertAndSend("service-exchange", "order.payment.success", event);
rabbitTemplate.convertAndSend("service-exchange", "order.ship.created", event);
```

### 场景4：IoT / 实时监控

```java
// 需求：高并发连接、轻量级消息
// 推荐：NATS / EMQX

// NATS - 极低延迟
Nats.connect("nats://localhost:4222").publish("sensor.temperature", payload);
```

## 三、选型决策流程图

```
                    ┌─ 需要事务消息？ ── 是 ──→ RocketMQ
                    │
用户请求 ──→ 分析需求 ─┼─ 超高吞吐/日志？ ── 是 ──→ Kafka
                    │
                    ├─ 复杂路由/低延迟？ ── 是 ──→ RabbitMQ
                    │
                    └─ 简单队列/小团队？ ── 是 ──→ RabbitMQ 或 Redis Stream
```

## 四、混合使用方案

大型系统通常混合使用多种 MQ：

```yaml
# 架构设计
业务消息层:
  - 订单/支付: RocketMQ (事务消息)
  - 通知/邮件: RabbitMQ (灵活路由)

数据管道层:
  - 日志收集: Kafka (高吞吐)
  - 用户行为: Kafka (流处理)

缓存/轻量层:
  - 实时通知: Redis Stream
  - 任务队列: RabbitMQ
```

## 五、云服务选项

```yaml
# 云消息队列对比
阿里云:
  - 消息队列 RocketMQ 版 (推荐)
  - 消息队列 Kafka 版
  - 微消息队列 MQTT 版

AWS:
  - SQS (标准/ FIFO 队列)
  - SNS (发布订阅)
  - Kinesis (流处理)

Azure:
  - Service Bus (企业级消息)
  - Event Hubs (大数据流入)
```

## 六、注意事项

1. **不要过度选型**，小团队直接用 RabbitMQ 最简单
2. **考虑 3-5 年的演进**，避免后期大规模迁移
3. **优先选择团队熟悉的技术栈**，降低学习和运维成本
4. **云服务适合不想自建运维的团队**，但有供应商锁定风险
5. **性能测试要基于真实业务场景**，不要只看 benchmark 数据
