# 消息队列与微服务

## 一、微服务中的 MQ 角色

在微服务架构中，MQ 扮演着**服务间异步通信的桥梁**，实现服务解耦、事件驱动、削峰填谷等核心能力。

```
┌─────────┐    HTTP/gRPC    ┌─────────┐
│ 服务A    │ ─────────────→ │ 服务B    │  同步调用
└─────────┘                 └─────────┘

┌─────────┐    Message      ┌─────────┐
│ 服务A    │ ──→ MQ ──→    │ 服务B    │  异步消息
└─────────┘                 └─────────┘
                  │
                  └──→ ┌─────────┐
                       │ 服务C    │  多个消费者
                       └─────────┘
```

## 二、异步解耦

```java
// 传统同步调用 - 服务强耦合
@RestController
public class OrderController {
    public Result createOrder(Order order) {
        orderService.create(order);          // 订单服务
        inventoryService.deduct(order);      // 库存服务（同步）
        pointService.add(order);             // 积分服务（同步）
        notificationService.send(order);     // 通知服务（同步）
        // 任一服务失败，整个流程失败
    }
}

// MQ 异步解耦 - 服务独立
@RestController
public class OrderController {
    public Result createOrder(Order order) {
        orderService.create(order);
        // 发布事件，下游服务各自订阅
        eventBus.publish(new OrderCreatedEvent(order));
        return Result.success();  // 立即返回
    }
}

// 库存服务 - 订阅订单创建事件
@EventListener("order.created")
public void onOrderCreated(OrderCreatedEvent event) {
    inventoryService.deduct(event.getOrder());
}

// 积分服务 - 订阅订单创建事件
@EventListener("order.created")
public void onOrderCreated(OrderCreatedEvent event) {
    pointService.add(event.getOrder());
}
```

## 三、事件驱动架构（EDA）

```java
// 事件定义
public class OrderEvent {
    private String eventType;  // CREATED, PAID, SHIPPED, COMPLETED
    private Long orderId;
    private Long timestamp;
    private Map<String, Object> payload;
}

// 事件生产者
@Service
public class OrderEventProducer {
    public void publishEvent(String type, Order order) {
        OrderEvent event = new OrderEvent();
        event.setEventType(type);
        event.setOrderId(order.getId());
        event.setTimestamp(System.currentTimeMillis());
        event.setPayload(Map.of("amount", order.getAmount()));

        kafkaTemplate.send("order-events", order.getId().toString(), JSON.toJSONString(event));
    }
}

// 各服务订阅事件
@KafkaListener(topics = "order-events", groupId = "inventory-service")
public void handleOrderEvent(String message) {
    OrderEvent event = JSON.parseObject(message, OrderEvent.class);
    switch (event.getEventType()) {
        case "PAID":
            inventoryService.confirmDeduct(event.getOrderId());
            break;
        case "CANCELLED":
            inventoryService.restore(event.getOrderId());
            break;
    }
}
```

## 四、Saga 分布式事务

```java
// 基于 MQ 的 Saga 编排模式
@Service
public class OrderSagaOrchestrator {

    // 正向流程
    public void createOrderSaga(Order order) {
        sagaStep1_reserveInventory(order);
        // 等待库存服务回调或超时
    }

    @MQListener("inventory.reserved")
    public void onInventoryReserved(InventoryEvent event) {
        sagaStep2_deductPayment(event.getOrder());
    }

    @MQListener("payment.deducted")
    public void onPaymentDeducted(PaymentEvent event) {
        sagaStep3_confirmOrder(event.getOrder());
    }

    // 补偿流程（任一步骤失败）
    @MQListener("payment.failed")
    public void onPaymentFailed(PaymentEvent event) {
        // 补偿：释放库存
        inventoryService.release(event.getOrderId());
    }
}
```

## 五、服务间通信模式对比

```java
// 同步调用 - 适用于强依赖场景
// 优点：简单直接，实时性好
// 缺点：服务耦合，一个挂全挂
userService.getUser(userId);

// 异步消息 - 适用于弱依赖场景
// 优点：解耦，削峰，可重试
// 缺点：最终一致，调试复杂
mq.send("user.updated", userEvent);

// 事件驱动 - 适用于多消费者场景
// 优点：一对多，易扩展
// 缺点：事件流管理复杂
eventBus.publish(new UserRegisteredEvent(user));
```

## 六、注意事项

1. **不是所有服务间通信都需要 MQ**，强依赖、需要即时响应的场景用同步调用
2. **事件设计要保证幂等**，因为事件可能被重复消费
3. **事件版本管理很重要**，使用事件信封或 Schema Registry
4. **监控事件链路**，确保事件被所有消费者正确处理
5. **避免循环依赖**，A 发事件给 B，B 又发事件给 A 会导致死循环
