# 消息顺序性

## 一、消息顺序的三个级别

消息顺序性分为三个层次，实现难度依次递增：

### 1. 分区有序（Partition Order）

保证同一个分区内的消息按发送顺序消费。这是 Kafka 和 RocketMQ 的默认保证。

```java
// Kafka 保证分区有序 - 使用相同的 Key
@Component
public class OrderProducer {

    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;

    public void sendOrderEvents(Order order) {
        String orderId = order.getId().toString();
        // 同一订单的事件使用相同的 Key，保证落入同一分区
        kafkaTemplate.send("order-topic", orderId, "CREATED: " + JSON.toJSONString(order));
        kafkaTemplate.send("order-topic", orderId, "PAID: " + JSON.toJSONString(order));
        kafkaTemplate.send("order-topic", orderId, "SHIPPED: " + JSON.toJSONString(order));
    }
}

// 自定义分区器 - 按业务 Key 分区
public class OrderPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                        Object value, byte[] valueBytes, Cluster cluster) {
        String orderId = (String) key;
        int numPartitions = cluster.partitionCountForTopic(topic);
        // 同一订单始终进入同一分区
        return Math.abs(orderId.hashCode()) % numPartitions;
    }
}
```

### 2. 全局有序（Global Order）

所有消息严格按照发送顺序消费，吞吐量极低，一般不推荐。

```java
// Kafka 全局有序 - 只用一个分区（性能极差）
@Configuration
public class GlobalOrderConfig {
    @Bean
    public NewTopic globalTopic() {
        // Topic 只有 1 个分区，保证全局有序
        return new NewTopic("global-order-topic", 1, (short) 1);
    }
}

// RabbitMQ 全局有序 - 只用一个队列
@RabbitListener(queues = "single-queue")
public void process(String message) {
    // 单队列天然保证 FIFO
}
```

### 3. 业务有序（Business Order）

保证同一业务实体的消息有序，不同业务实体之间无序。这是最常用的方案。

```java
// 订单状态变更保证有序
public class OrderEventProducer {

    public void sendOrderStatusChange(Long orderId, String status) {
        // 同一订单的所有状态变更使用相同 Key
        String key = "order:" + orderId;
        OrderEvent event = new OrderEvent(orderId, status, System.currentTimeMillis());
        kafkaTemplate.send("order-events", key, JSON.toJSONString(event));
    }
}

// 消费端 - 单线程处理同一订单的消息
@KafkaListener(topics = "order-events", concurrency = "6")
public void handleOrderEvent(ConsumerRecord<String, String> record) {
    OrderEvent event = JSON.parseObject(record.value(), OrderEvent.class);
    // 同一订单的消息在同一分区，由同一线程处理，保证顺序
    orderService.processEvent(event);
}
```

## 二、RabbitMQ 顺序性保证

```java
// RabbitMQ 单队列 + 单消费者 = 严格有序
@Configuration
public class RabbitMQOrderConfig {

    @Bean
    public Queue orderQueue() {
        return QueueBuilder.durable("order-queue")
            .build();
    }
}

// 单消费者处理
@RabbitListener(queues = "order-queue", concurrency = "1")
public void processOrderMessage(Message message) {
    // 单线程消费，保证顺序
}
```

## 三、顺序性与性能的权衡

```java
// 分区有序方案 - 推荐
// 同一业务实体的消息保证有序，不同业务实体并行处理
public class BalancedOrderProducer {

    public void sendEvent(Long orderId, String event) {
        // 使用 orderId 作为 Key
        // 优点：同一订单有序，不同订单并行
        // 缺点：热点订单可能造成分区数据倾斜
        kafkaTemplate.send("order-topic", String.valueOf(orderId), event);
    }
}

// 解决热点 Key 的方案 - 二级分区
public void sendEventWithSubPartition(Long orderId, String event) {
    // 添加随机后缀分散到多个分区，但仍保证同一订单在子分区内有序
    int subPartition = orderId.hashCode() % 10;
    String key = orderId + "-" + subPartition;
    kafkaTemplate.send("order-topic", key, event);
}
```

## 四、注意事项

1. **全局有序性能极差**，生产环境应避免使用
2. **分区有序是最佳实践**，用业务 Key 保证相关消息落入同一分区
3. **顺序性与并行消费是矛盾的**，需要在业务需求和性能之间权衡
4. **消费端处理失败重试可能破坏顺序**，需要特别设计重试机制
5. **多消费者场景**，确保同一业务实体的消息由同一消费者处理
