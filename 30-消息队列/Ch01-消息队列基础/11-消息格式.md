# 消息格式

## 一、常见消息格式对比

消息格式决定了消息的**可读性、体积、跨语言支持**和**演进能力**。

| 格式 | 可读性 | 体积 | 性能 | Schema演进 | 跨语言 |
|------|--------|------|------|-----------|--------|
| JSON | 最好 | 大 | 慢 | 容易 | 最好 |
| XML | 好 | 最大 | 最慢 | 容易 | 好 |
| Protobuf | 不可读 | 最小 | 最快 | 中等 | 好 |
| Avro | 不可读 | 小 | 快 | 最好 | 好 |
| MessagePack | 不可读 | 小 | 快 | 中等 | 好 |

## 二、JSON 格式

```java
// JSON - 最常用的消息格式
public class OrderMessage {
    private Long orderId;
    private String userId;
    private BigDecimal amount;
    private String status;
    private Long timestamp;
}

// 序列化
String json = JSON.toJSONString(orderMessage);
kafkaTemplate.send("order-topic", json);

// 反序列化
OrderMessage msg = JSON.parseObject(record.value(), OrderMessage.class);
```

```json
{
    "orderId": 10001,
    "userId": "U12345",
    "amount": 99.99,
    "status": "PAID",
    "timestamp": 1700000000000
}
```

### JSON 优缺点

- **优点**：人类可读、调试方便、无需预定义 Schema、语言无关
- **缺点**：体积大、无类型约束、解析性能较低

## 三、Protobuf 格式

```protobuf
// order.proto - 定义消息结构
syntax = "proto3";

message OrderMessage {
    int64 order_id = 1;
    string user_id = 2;
    double amount = 3;
    string status = 4;
    int64 timestamp = 5;
}

// 新版本增加字段（向前兼容）
message OrderMessageV2 {
    int64 order_id = 1;
    string user_id = 2;
    double amount = 3;
    string status = 4;
    int64 timestamp = 5;
    string payment_method = 6;   // 新增字段
    repeated string tags = 7;    // 新增字段
}
```

```java
// Java 中使用 Protobuf
OrderMessage order = OrderMessage.newBuilder()
    .setOrderId(10001)
    .setUserId("U12345")
    .setAmount(99.99)
    .setStatus("PAID")
    .setTimestamp(System.currentTimeMillis())
    .build();

// 序列化 - 字节数组
byte[] bytes = order.toByteArray();

// 反序列化
OrderMessage parsed = OrderMessage.parseFrom(bytes);

// 发送到 Kafka
kafkaTemplate.send("order-topic", null, bytes);
```

## 四、Avro 格式

```json
// order.avsc - Avro Schema
{
    "type": "record",
    "name": "OrderMessage",
    "namespace": "com.example",
    "fields": [
        {"name": "orderId", "type": "long"},
        {"name": "userId", "type": "string"},
        {"name": "amount", "type": "double"},
        {"name": "status", "type": "string"},
        {"name": "timestamp", "type": "long"}
    ]
}
```

```java
// Avro 与 Schema Registry 配合使用
@Configuration
public class AvroConfig {

    @Bean
    public ProducerFactory<String, Order> avroProducerFactory() {
        Map<String, Object> props = new HashMap<>();
        props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "localhost:9092");
        props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class);
        props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, KafkaAvroSerializer.class);
        props.put("schema.registry.url", "http://localhost:8081");
        return new DefaultKafkaProducerFactory<>(props);
    }
}
```

## 五、统一消息信封

```java
// 通用消息信封 - 包装各种业务消息
public class MessageEnvelope<T> {
    private String messageId;       // 消息唯一标识
    private String topic;           // 主题
    private String type;            // 消息类型（业务标识）
    private Long timestamp;         // 发送时间
    private String traceId;         // 链路追踪 ID
    private T payload;              // 业务数据
    private Map<String, String> headers;  // 扩展头
}

// 使用信封
MessageEnvelope<OrderMessage> envelope = new MessageEnvelope<>();
envelope.setMessageId(UUID.randomUUID().toString());
envelope.setType("ORDER_CREATED");
envelope.setTimestamp(System.currentTimeMillis());
envelope.setTraceId(TraceContext.getTraceId());
envelope.setPayload(orderMessage);

String json = JSON.toJSONString(envelope);
kafkaTemplate.send("order-topic", envelope.getMessageId(), json);
```

## 六、注意事项

1. **优先选择 JSON**，除非对性能或体积有严格要求
2. **使用消息信封**统一管理消息元数据，便于监控和追踪
3. **Schema 演进要保持向前兼容**，只增加新字段，不删除或修改已有字段
4. **Protobuf/Avro 需要提前定义 Schema**，增加开发复杂度但提供类型安全
5. **生产环境建议配合 Schema Registry** 管理消息格式版本
