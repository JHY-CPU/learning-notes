# 消息队列对比概览

## 一、三大主流 MQ 对比

### 架构对比

```
RabbitMQ:  Producer -> Exchange -> Queue -> Consumer
           (基于 AMQP 协议，Erlang 实现)

Kafka:     Producer -> Topic -> Partition -> Consumer Group
           (分布式流平台，日志存储)

RocketMQ:  Producer -> Topic -> MessageQueue -> Consumer Group
           (阿里开源，Java 实现)
```

### 核心指标对比

| 指标 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| 开发语言 | Erlang | Java/Scala | Java |
| 吞吐量 | 万级/秒 | 百万级/秒 | 十万级/秒 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 消息持久化 | 内存+磁盘 | 磁盘顺序写 | 磁盘 |
| 消息回溯 | 不支持 | 按 Offset/时间 | 按时间/Offset |
| 事务消息 | 不支持 | 支持(0.11+) | 原生支持 |
| 延迟消息 | 插件支持 | 不支持 | 原生18级 |
| 消息过滤 | routing-key | 不支持 | Tag/SQL92 |
| 消息顺序 | 单队列有序 | 分区有序 | 分区有序 |
| 集群模式 | 镜像队列 | ISR副本 | Master-Slave |

## 二、适用场景分析

### RabbitMQ 适用场景

```java
// 1. 业务消息 - 可靠性优先
// 订单通知、支付回调、状态同步
rabbitTemplate.convertAndSend("order-exchange", "order.created", order);

// 2. 复杂路由 - 灵活的消息分发
// Exchange 类型丰富，支持复杂路由规则
```

### Kafka 适用场景

```java
// 1. 日志收集 - 高吞吐量
// 应用日志、监控指标、用户行为日志
kafkaTemplate.send("app-logs", logEntry);

// 2. 流处理 - 实时数据管道
// 点击流分析、实时推荐、风控检测
StreamsBuilder builder = new StreamsBuilder();
builder.<String, String>stream("clicks")
    .groupBy((key, value) -> extractUserId(value))
    .count()
    .toStream()
    .to("click-counts");
```

### RocketMQ 适用场景

```java
// 1. 电商交易 - 事务消息
// 订单创建、库存扣减、积分计算
rocketMQTemplate.sendMessageInTransaction("order-topic", msg, arg);

// 2. 定时任务 - 延迟消息
// 订单超时取消、定时提醒
rocketMQTemplate.syncSend("delay-topic", msg, 3000, delayLevel);
```

## 三、功能特性深度对比

### 消息可靠性

```java
// RabbitMQ - Publisher Confirm + 持久化
rabbitTemplate.setConfirmCallback((data, ack, cause) -> {
    if (!ack) log.error("发送失败: {}", cause);
});

// Kafka - acks=all + 幂等 + 事务
props.put("acks", "all");
props.put("enable.idempotence", true);

// RocketMQ - 同步刷盘 + 同步复制
brokerConfig.setFlushDiskType(FlushDiskType.SYNC_FLUSH);
brokerConfig.setBrokerRole(BrokerRole.SYNC_MASTER);
```

### 高可用架构

```
RabbitMQ: 镜像队列 (3.13+ 推荐 Quorum Queue)
  Node1 (Master) <-> Node2 (Mirror) <-> Node3 (Mirror)

Kafka: ISR 副本机制
  Partition0: Leader(Broker1) + Follower(Broker2, Broker3)

RocketMQ: Dledger (自动主从切换)
  Master(Broker-a) -> Slave(Broker-b) -> Slave(Broker-c)
```

## 四、选型建议速查

```yaml
# 选型决策树
场景判断:
  - 需要复杂路由: RabbitMQ
  - 日志/大数据/流处理: Kafka
  - 电商/金融/事务消息: RocketMQ
  - 小规模/简单队列: RabbitMQ
  - 超高吞吐: Kafka
  - 延迟消息: RocketMQ
```

## 五、注意事项

1. **不要只看吞吐量**，RabbitMQ 的低延迟在某些场景更重要
2. **Kafka 不适合做业务消息队列**，缺少延迟消息、事务消息等特性
3. **RocketMQ 中文文档和社区支持更好**，国内公司使用广泛
4. **选择前考虑团队技术栈**，Erlang 运维对 Java 团队是额外负担
5. **云服务也是一种选择**，AWS SQS、阿里云消息队列等省去运维成本
