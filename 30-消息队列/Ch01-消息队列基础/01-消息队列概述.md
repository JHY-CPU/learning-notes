# 消息队列概述

## 一、什么是消息队列

消息队列（Message Queue，简称 MQ）是一种异步通信机制，用于在分布式系统中传递消息。发送方将消息放入队列后即可返回，接收方从队列中取出消息进行处理，双方无需同时在线。

消息队列本质上是一个**先进先出（FIFO）**的数据结构，生产者将消息写入队列尾部，消费者从队列头部读取消息。

```java
// 最简化的消息队列概念模型
public class SimpleMQ {
    private Queue<Message> queue = new LinkedList<>();

    // 生产者发送消息
    public void send(Message msg) {
        queue.offer(msg);
    }

    // 消费者接收消息
    public Message receive() {
        return queue.poll();
    }
}
```

## 二、为什么需要消息队列

### 1. 系统解耦

传统系统之间通过直接调用耦合在一起，一旦某个服务宕机，调用链路全部中断。使用 MQ 后，服务之间通过消息通信，即使下游服务暂时不可用，消息也不会丢失。

```java
// 直接调用 - 强耦合
orderService.createOrder();
inventoryService.deductStock();  // 库存服务挂了，订单也失败
pointsService.addPoints();

// MQ 解耦 - 异步调用
orderService.createOrder();
mq.send("order_created", order);  // 发送消息后立即返回
// 库存服务和积分服务各自订阅消息，独立处理
```

### 2. 异步处理

某些操作不需要立即返回结果，可以将任务放入 MQ 异步执行，提升用户体验和系统吞吐量。

```java
// 同步处理 - 用户等待 3 秒
@PostMapping("/register")
public Result register(User user) {
    userService.save(user);          // 500ms
    emailService.sendWelcome(user);  // 1000ms
    smsService.sendNotification();   // 800ms
    logService.recordLog();          // 700ms
    return Result.success();
}

// 异步处理 - 用户只需等待 500ms
@PostMapping("/register")
public Result register(User user) {
    userService.save(user);
    mq.send("user_registered", user); // 其他任务异步执行
    return Result.success();
}
```

### 3. 削峰填谷

在秒杀、抢购等高并发场景下，瞬间流量可能远超系统处理能力。MQ 可以作为缓冲区，将突发流量平滑为系统能承受的消费速率。

```java
// 秒杀场景 - 没有 MQ 削峰
// 瞬间 10 万个请求直接打到数据库，系统崩溃

// 使用 MQ 削峰
@PostMapping("/seckill")
public Result seckill(Long productId) {
    // 校验后直接放入消息队列
    if (stockService.check(productId)) {
        mq.send("seckill_order", order);
        return Result.success("排队中");
    }
    return Result.fail("已售罄");
}

// 消费者以稳定速率处理
@MQListener("seckill_order")
public void handleSeckill(Order order) {
    orderService.process(order);  // 以 1000 QPS 稳定消费
}
```

## 三、主流消息队列对比

| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| 开发语言 | Erlang | Java/Scala | Java |
| 吞吐量 | 万级 | 十万级 | 十万级 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 消息可靠性 | 高 | 中 | 高 |
| 消息回溯 | 不支持 | 支持 | 支持 |
| 事务消息 | 不支持 | 支持 | 支持 |
| 适用场景 | 业务消息 | 日志/流处理 | 电商/金融 |

## 四、注意事项

1. **引入 MQ 会增加系统复杂度**，需要考虑消息丢失、重复消费、顺序性等问题
2. **MQ 本身需要高可用**，单点故障会导致整个消息链路中断
3. **不是所有场景都适合 MQ**，同步强依赖的场景不适合异步消息
4. **消息队列不能替代数据库**，它只负责传递消息，不负责持久化业务数据
