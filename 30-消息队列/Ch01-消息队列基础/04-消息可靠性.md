# 消息可靠性

## 一、消息丢失的三个环节

消息从生产到消费，可能在三个环节丢失：

```
Producer -> Broker -> Consumer
  (发送丢失)  (存储丢失)  (消费丢失)
```

### 1. 生产者发送阶段

生产者发送消息到 Broker 可能因为网络抖动、Broker 宕机等原因丢失。

```java
// RabbitMQ 确认机制
@Configuration
public class RabbitMQConfig {

    @Bean
    public RabbitTemplate rabbitTemplate(ConnectionFactory factory) {
        RabbitTemplate template = new RabbitTemplate(factory);
        // 开启 Publisher Confirm
        template.setConfirmCallback((correlationData, ack, cause) -> {
            if (!ack) {
                log.error("消息发送失败: {}", cause);
                // 重试或记录到失败表
                retryService.saveFailedMessage(correlationData);
            }
        });
        // 开启 Publisher Return
        template.setReturnsCallback(returned -> {
            log.error("消息路由失败: {}", returned.getMessage());
        });
        template.setMandatory(true);
        return template;
    }
}

// Kafka 生产者确认配置
Properties props = new Properties();
props.put("acks", "all");           // 所有副本确认
props.put("retries", 3);            // 重试次数
props.put("enable.idempotence", true); // 开启幂等
```

### 2. Broker 存储阶段

Broker 可能因为宕机导致内存中的消息丢失，需要持久化到磁盘。

```yaml
# RabbitMQ 持久化配置
spring:
  rabbitmq:
    template:
      exchange: order-exchange
      routing-key: order.created
    # 队列持久化
    queue:
      durable: true
      auto-delete: false
```

```java
// 消息持久化
MessageProperties props = new MessageProperties();
props.setDeliveryMode(MessageDeliveryMode.PERSISTENT); // 持久化消息
Message message = new Message(body, props);
rabbitTemplate.send("exchange", "routing-key", message);
```

### 3. 消费者处理阶段

消费者处理消息时可能因为程序异常、重启等原因丢失。

```java
// RabbitMQ 手动 ACK
@RabbitListener(queues = "order-queue")
public void processOrder(Message message, Channel channel) {
    try {
        Order order = JSON.parseObject(message.getBody(), Order.class);
        orderService.process(order);
        // 处理成功，手动确认
        channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
    } catch (Exception e) {
        log.error("处理失败", e);
        // 处理失败，重新入队或进入死信队列
        channel.basicNack(message.getMessageProperties().getDeliveryTag(), false, true);
    }
}
```

```java
// Kafka 手动提交 Offset
@KafkaListener(topics = "order-topic")
public void consume(ConsumerRecord<String, String> record, Acknowledgment ack) {
    try {
        processMessage(record);
        ack.acknowledge();  // 处理成功后提交
    } catch (Exception e) {
        log.error("消费失败", e);
        // 不提交 Offset，下次重新消费
    }
}
```

## 二、端到端可靠性方案

```java
// 完整的可靠性保障流程
@Service
public class ReliableMessageService {

    // 1. 本地消息表 - 保证生产端可靠性
    @Transactional
    public void createOrder(Order order) {
        orderMapper.insert(order);
        // 同一事务中保存消息记录
        messageMapper.insert(new MessageRecord(order.getId(), "PENDING"));
    }

    // 2. 定时任务 - 扫描未发送的消息
    @Scheduled(fixedDelay = 5000)
    public void retryPendingMessages() {
        List<MessageRecord> pending = messageMapper.selectByStatus("PENDING");
        for (MessageRecord msg : pending) {
            try {
                mq.send(msg.getTopic(), msg.getContent());
                messageMapper.updateStatus(msg.getId(), "SENT");
            } catch (Exception e) {
                log.error("重试失败", e);
            }
        }
    }
}
```

## 三、注意事项

1. **ack=all 是 Kafka 最高可靠性配置**，但会增加延迟
2. **手动 ACK 比自动 ACK 更可靠**，但需要确保处理完成后再确认
3. **持久化会有性能损耗**，需要在可靠性和性能之间权衡
4. **本地消息表是分布式事务的常用方案**，简单可靠但实现较复杂
5. **重试机制要设置上限**，避免无限重试导致系统雪崩
