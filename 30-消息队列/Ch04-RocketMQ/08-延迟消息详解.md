# RocketMQ 延迟消息详解

## 一、延迟级别

RocketMQ 内置 18 个延迟级别，不能自定义（5.0 之前）。

```java
// 延迟级别映射
// level 1 -> 1s     level 10 -> 6m
// level 2 -> 5s     level 11 -> 7m
// level 3 -> 10s    level 12 -> 8m
// level 4 -> 30s    level 13 -> 9m
// level 5 -> 1m     level 14 -> 10m
// level 6 -> 2m     level 15 -> 20m
// level 7 -> 3m     level 16 -> 30m
// level 8 -> 4m     level 17 -> 1h
// level 9 -> 5m     level 18 -> 2h
```

## 二、发送延迟消息

```java
// 方式1: 设置延迟级别
Message<Order> msg = MessageBuilder.withPayload(order).build();
rocketMQTemplate.syncSend("DelayTopic", msg, 3000, 16); // level 16 = 30分钟

// 方式2: 设置消息属性
DefaultMQProducer producer = new DefaultMQProducer("ProducerGroup");
Message message = new Message("DelayTopic", orderJson.getBytes());
message.setDelayTimeLevel(5);  // 1分钟
producer.send(message);

// RocketMQ 5.0+ 自定义时间
rocketMQTemplate.syncSendDelayTimeAtMills("DelayTopic", order,
    System.currentTimeMillis() + TimeUnit.MINUTES.toMillis(30));
```

## 三、业务场景

```java
// 订单 30 分钟超时取消
@Service
public class OrderTimeoutService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void createOrder(Order order) {
        orderMapper.insert(order);

        // 发送延迟消息，30 分钟后检查
        Message<Order> msg = MessageBuilder
            .withPayload(order)
            .setHeader("type", "TIMEOUT_CHECK")
            .build();
        rocketMQTemplate.syncSend("OrderDelayTopic", msg, 3000, 16);
    }
}

// 延迟消费者
@Component
@RocketMQMessageListener(topic = "OrderDelayTopic", consumerGroup = "TimeoutGroup")
public class OrderTimeoutConsumer implements RocketMQListener<Order> {

    @Override
    public void onMessage(Order order) {
        Order db = orderMapper.selectById(order.getId());
        if ("UNPAID".equals(db.getStatus())) {
            orderMapper.cancelOrder(order.getId());
            log.info("订单超时自动取消: {}", order.getId());
        }
    }
}
```

## 四、延迟消息存储

```
RocketMQ 延迟消息实现:
1. 消息先存储到延迟 Topic (SCHEDULE_TOPIC_XXXX)
2. 定时任务扫描到期的消息
3. 到期后投递到真实 Topic
4. 消费者从真实 Topic 消费
```

## 五、注意事项

1. **延迟级别固定，不支持任意时间**
2. **5.0+ 版本支持自定义延迟时间**
3. **延迟消息不支持顺序消费**
4. **大量延迟消息会增加 Broker 负载**，因为需要定时扫描
5. **消费端要做好幂等处理**，延迟消息可能重复投递
