# 消息重试策略

## 一、重试策略设计

```java
// 指数退避重试
public class ExponentialBackoffRetry {
    private final int maxRetries = 5;
    private final long[] delays = {1000, 5000, 30000, 120000, 600000}; // ms

    public void retry(Message message, int retryCount) {
        if (retryCount >= maxRetries) {
            sendToDLQ(message);  // 进入死信队列
            return;
        }
        long delay = delays[Math.min(retryCount, delays.length - 1)];
        scheduleRetry(message, delay);
    }
}

// RabbitMQ 重试配置
@RabbitListener(queues = "order-queue")
public void process(Message message, Channel channel) throws IOException {
    try {
        doProcess(message);
        channel.basicAck(message.getMessageProperties().getDeliveryTag(), false);
    } catch (Exception e) {
        channel.basicNack(message.getMessageProperties().getDeliveryTag(), false, false);
        // 进入死信队列 -> 延迟重试队列 -> 重新消费
    }
}
```

## 二、重试策略分类

```java
// 固定间隔重试
public class FixedDelayRetry {
    private final int maxRetries = 3;
    private final long delay = 5000;  // 固定 5 秒

    public void retry(Message msg, int count) {
        if (count >= maxRetries) {
            sendToDLQ(msg);
            return;
        }
        scheduleRetry(msg, delay);
    }
}

// 指数退避重试（推荐）
public class ExponentialBackoffRetry {
    private final int maxRetries = 5;
    // 1s -> 5s -> 30s -> 2min -> 10min
    private final long[] delays = {1000, 5000, 30000, 120000, 600000};

    public void retry(Message msg, int count) {
        if (count >= maxRetries) {
            sendToDLQ(msg);
            return;
        }
        // 添加随机抖动，避免惊群效应
        long delay = delays[count] + ThreadLocalRandom.current().nextLong(1000);
        scheduleRetry(msg, delay);
    }
}

// 按异常类型区分重试策略
public class SmartRetry {
    public RetryPolicy getPolicy(Exception e) {
        if (e instanceof BusinessException) {
            // 业务异常：不重试，直接死信
            return RetryPolicy.noRetry();
        }
        if (e instanceof TimeoutException) {
            // 超时异常：少量重试
            return RetryPolicy.fixed(3, 5000);
        }
        if (e instanceof TransientException) {
            // 临时异常：多次退避重试
            return RetryPolicy.exponential(5);
        }
        // 未知异常：保守重试
        return RetryPolicy.fixed(2, 10000);
    }
}
```

## 三、死信队列与重试队列架构

```
          重试与死信架构
    ┌───────────────────────────────────────┐
    │  业务队列                                │
    │    ↓ 消费失败                            │
    │  重试队列 (延迟队列)                       │
    │    ↓ 延迟后重新投递                        │
    │  业务队列 (重新消费)                       │
    │    ↓ 多次重试失败                          │
    │  死信队列 (DLQ)                           │
    │    ↓                                     │
    │  人工处理 / 告警 / 定时重试                  │
    └───────────────────────────────────────┘
```

## 五、注意事项

1. **重试次数要有限制**，避免无限重试
2. **指数退避减少系统压力**
3. **业务异常和系统异常要区分处理**
4. **重试记录要持久化**，便于排查
5. **死信队列是重试的最后兜底**
6. **添加随机抖动**，避免多个消费者同时重试
7. **重试风暴要监控**，异常增长说明系统故障
