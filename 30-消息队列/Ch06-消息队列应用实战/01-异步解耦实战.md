# 异步解耦实战

## 一、订单支付异步流程

```java
// 传统同步方式 - 强耦合
public Result payOrder(Long orderId) {
    orderService.updateStatus(orderId, "PAID");      // 50ms
    inventoryService.deduct(orderId);                // 200ms
    pointService.addPoints(orderId);                 // 150ms
    notificationService.sendPaidNotify(orderId);     // 300ms
    // 总耗时 700ms，任一服务失败整个流程失败
}

// MQ 异步方式 - 解耦
public Result payOrder(Long orderId) {
    orderService.updateStatus(orderId, "PAID");
    // 发布事件，下游服务各自订阅
    eventBus.publish(new OrderPaidEvent(orderId));
    return Result.success();  // 50ms 返回
}

// 库存服务订阅
@EventListener("order.paid")
public void onOrderPaid(OrderPaidEvent event) {
    inventoryService.deduct(event.getOrderId());
}

// 积分服务订阅
@EventListener("order.paid")
public void onOrderPaid(OrderPaidEvent event) {
    pointService.addPoints(event.getOrderId());
}
```

## 二、异步补偿机制

```java
@Service
public class AsyncCompensationService {

    @Transactional
    public void payOrder(Long orderId) {
        orderService.updateStatus(orderId, "PAID");
        // 保存事件记录
        eventLogMapper.insert(new EventLog(orderId, "ORDER_PAID", "PENDING"));
    }

    // 定时投递未发送的事件
    @Scheduled(fixedDelay = 1000)
    public void deliverPendingEvents() {
        List<EventLog> pending = eventLogMapper.selectByStatus("PENDING", 100);
        for (EventLog event : pending) {
            try {
                kafkaTemplate.send(event.getTopic(), event.getContent());
                eventLogMapper.updateStatus(event.getId(), "DELIVERED");
            } catch (Exception e) {
                log.error("事件投递失败: {}", event.getId());
            }
        }
    }
}
```

## 三、注意事项

1. **异步不代表不关心结果**，要做好监控和补偿
2. **事件日志是异步流程的保障**
3. **下游服务失败要有重试和告警机制**
4. **异步流程调试比同步困难**，需要完善的日志和追踪
5. **不是所有场景都适合异步**，需要即时响应的用同步
