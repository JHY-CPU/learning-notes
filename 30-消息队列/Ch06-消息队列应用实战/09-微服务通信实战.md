# 微服务通信实战

## 一、异步事件通信

```java
// 事件定义
public class OrderCreatedEvent {
    private Long orderId;
    private Long userId;
    private BigDecimal amount;
    private Long timestamp;
}

// 发布事件
@Service
public class OrderService {
    public void createOrder(Order order) {
        orderMapper.insert(order);
        kafkaTemplate.send("order-events", order.getId().toString(),
            JSON.toJSONString(new OrderCreatedEvent(order)));
    }
}

// 订阅事件
@Component
@KafkaListener(topics = "order-events", groupId = "inventory-service")
public class InventoryEventConsumer {
    public void onOrderCreated(ConsumerRecord<String, String> record) {
        OrderCreatedEvent event = JSON.parseObject(record.value(), OrderCreatedEvent.class);
        inventoryService.reserve(event.getOrderId());
    }
}
```

## 四、请求-响应模式

```java
// 通过临时队列实现同步调用
public class RpcOverMQ {
    // 发送请求并等待响应
    public OrderResponse callSync(OrderRequest request) {
        String correlationId = UUID.randomUUID().toString();
        String replyQueue = "reply-" + correlationId;

        // 发送请求，附带回调队列
        MessageProperties props = new MessageProperties();
        props.setCorrelationId(correlationId);
        props.setReplyTo(replyQueue);

        Message message = new Message(
            JSON.toJSONBytes(request), props);
        rabbitTemplate.send("order-exchange", "order.create", message);

        // 等待响应（设置超时）
        Message response = rabbitTemplate.receive(replyQueue, 5000);
        if (response == null) {
            throw new TimeoutException("RPC 调用超时");
        }
        return JSON.parseObject(response.getBody(), OrderResponse.class);
    }
}
```

## 五、事件驱动 vs 同步调用

```
方式          优点                    缺点                适用场景
同步调用       实时性好、简单            耦合度高、链路长      简单查询、实时要求高
事件驱动       解耦、可扩展、可审计       最终一致性、复杂      业务流程、跨服务通知
RPC over MQ   解耦 + 同步响应          延迟较高、资源占用     需要解耦的同步场景
```

## 六、注意事项

1. **事件命名要规范化**：{实体}.{动作}
2. **事件要包含足够的上下文**，减少消费者额外查询
3. **事件版本要管理**，支持向后兼容
4. **事件消费者要做好幂等**
5. **监控事件消费延迟和失败率**
6. **Schema Registry 保证生产消费两端 Schema 兼容**
7. **避免用 MQ 做实时性要求高的同步调用**
