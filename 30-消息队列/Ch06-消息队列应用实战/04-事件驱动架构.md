# 事件驱动架构

## 一、Event Sourcing

```java
// 事件定义
public class OrderEvent {
    private String eventId;
    private String eventType;  // CREATED, PAID, SHIPPED, COMPLETED
    private Long orderId;
    private Map<String, Object> payload;
    private Long timestamp;
}

// 事件存储
@Service
public class OrderEventStore {
    public void append(OrderEvent event) {
        kafkaTemplate.send("order-events", event.getOrderId().toString(), JSON.toJSONString(event));
    }

    public List<OrderEvent> getEvents(Long orderId) {
        // 从 Kafka 回溯读取事件流
        return eventReader.readEvents("order-events", orderId);
    }
}

// 从事件重建状态
public Order rebuild(Long orderId) {
    List<OrderEvent> events = eventStore.getEvents(orderId);
    Order order = new Order();
    for (OrderEvent event : events) {
        order.apply(event);  // 事件回放
    }
    return order;
}
```

## 二、Event Sourcing 事件存储架构

```
                    事件溯源架构
    ┌───────────────────────────────────────────┐
    │  写操作                                     │
    │    ↓                                       │
    │  命令处理器 (校验 + 业务逻辑)                  │
    │    ↓                                       │
    │  事件存储 (Kafka Topic / EventStoreDB)       │
    │    │                                        │
    │    ├──→ 事件投影 -> 查询数据库 (读模型)         │
    │    ├──→ 事件投影 -> 搜索索引 (ES)             │
    │    └──→ 事件投影 -> 缓存 (Redis)              │
    │                                             │
    │  读操作 → 查询数据库 / 缓存                    │
    └───────────────────────────────────────────┘
```

## 三、CQRS + Event Sourcing 组合

```java
// 命令端：处理写操作并产生事件
@Service
public class OrderCommandHandler {
    @Autowired
    private EventStore eventStore;

    public void handle(CreateOrderCommand cmd) {
        // 1. 加载当前聚合的事件流
        List<OrderEvent> events = eventStore.loadEvents(cmd.getOrderId());

        // 2. 重建聚合根状态
        OrderAggregate aggregate = new OrderAggregate();
        aggregate.replay(events);

        // 3. 执行业务逻辑，产生新事件
        OrderCreatedEvent event = aggregate.createOrder(cmd);

        // 4. 持久化事件
        eventStore.append(cmd.getOrderId(), event);
    }
}

// 查询端：从事件流读取数据
@Service
public class OrderQueryHandler {
    public OrderView getOrder(Long orderId) {
        // 从读模型查询（ES / MySQL）
        return orderViewRepository.findById(orderId);
    }
}
```

## 四、快照机制

```java
// 快照减少回放时间
@Service
public class SnapshotService {
    private static final int SNAPSHOT_INTERVAL = 100;  // 每 100 个事件做一次快照

    public OrderAggregate loadAggregate(Long orderId) {
        // 加载最近的快照
        Snapshot snapshot = snapshotStore.getLatest(orderId);
        OrderAggregate aggregate;
        int fromVersion;

        if (snapshot != null) {
            aggregate = snapshot.getAggregate();
            fromVersion = snapshot.getVersion();
        } else {
            aggregate = new OrderAggregate();
            fromVersion = 0;
        }

        // 只回放缓照之后的事件
        List<OrderEvent> events = eventStore.loadEvents(orderId, fromVersion);
        aggregate.replay(events);

        // 定期创建新快照
        if (aggregate.getVersion() % SNAPSHOT_INTERVAL == 0) {
            snapshotStore.save(orderId, aggregate);
        }

        return aggregate;
    }
}
```

## 五、注意事项

1. **事件一旦存储不可修改**，只能追加新事件
2. **事件版本要管理好**，支持向后兼容
3. **快照机制减少回放时间**
4. **事件驱动增加了系统复杂度**
5. **适合业务逻辑复杂、需要审计的场景**
6. **事件 Schema 演进要提前规划**，使用 Avro/Protobuf
7. **读写延迟要监控**，确保读模型及时更新
