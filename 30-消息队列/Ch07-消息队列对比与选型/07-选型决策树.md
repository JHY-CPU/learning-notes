# 选型决策树

## 一、决策流程

```
                    ┌─ 需要事务消息？ ── 是 ──→ RocketMQ
                    │
                    ├─ 超高吞吐（百万级）？ ── 是 ──→ Kafka
                    │
分析需求 ──────────┼─ 复杂路由？ ── 是 ──→ RabbitMQ
                    │
                    ├─ IoT/极低延迟？ ── 是 ──→ NATS
                    │
                    ├─ 多租户隔离？ ── 是 ──→ Pulsar
                    │
                    └─ 简单队列/小团队？ ── 是 ──→ RabbitMQ / Redis Stream
```

## 二、场景速查

| 场景 | 推荐 MQ |
|------|---------|
| 订单/支付 | RocketMQ |
| 日志收集 | Kafka |
| 实时推荐 | Kafka + Flink |
| 微服务通知 | RabbitMQ |
| 秒杀削峰 | RocketMQ/Kafka |
| IoT 设备 | NATS / EMQX |
| 简单任务队列 | RabbitMQ / Redis |
| 跨集群同步 | Kafka MirrorMaker |

## 三、详细选型决策流程

```
选型第一步：评估团队能力
├── 小团队（< 5人）→ RabbitMQ / Redis Stream
├── 中等团队（5-20人）→ Kafka / RocketMQ
└── 大团队（> 20人）→ Pulsar / 混合方案

选型第二步：评估业务需求
├── 需要事务消息？→ RocketMQ
├── 需要复杂路由？→ RabbitMQ
├── 需要超高吞吐？→ Kafka
├── 需要延迟消息？→ RocketMQ / RabbitMQ (DLX)
├── 需要多租户？→ Pulsar
└── 简单任务队列？→ Redis / RabbitMQ

选型第三步：评估运维能力
├── 无运维能力 → 云托管服务
├── 有基础运维 → Kafka / RabbitMQ
└── 有专业运维 → Pulsar / 自建集群

选型第四步：评估成本预算
├── 低成本 → Redis Stream / 云服务按量
├── 中等预算 → Kafka / RocketMQ
└── 充足预算 → Pulsar / 全托管服务
```

## 四、典型场景选型参考

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 电商秒杀 | RocketMQ | 削峰、事务消息、延迟消息 |
| 日志收集 | Kafka | 高吞吐、可回溯、流处理 |
| 订单通知 | RabbitMQ | 复杂路由、消息确认、低延迟 |
| IoT 设备 | NATS / EMQX | 轻量、低延迟、MQTT 协议 |
| 实时推荐 | Kafka + Flink | 流处理、高吞吐 |
| 数据同步 | Kafka Connect | CDC、多连接器 |
| 简单任务 | Redis Stream | 轻量、无需额外组件 |
| 跨集群 | Kafka MirrorMaker | 成熟、稳定 |

## 五、注意事项

1. **不要过度选型**，小团队直接用 RabbitMQ
2. **考虑 3-5 年演进**，避免大规模迁移
3. **混合使用是常见做法**
4. **云服务省运维但有锁定风险**
5. **性能不是唯一标准**，运维成本和生态同样重要
6. **先做 POC 验证**，不要仅凭文档选型
7. **考虑团队现有技术栈**，减少学习成本
