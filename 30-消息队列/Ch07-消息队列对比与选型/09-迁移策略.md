# MQ 迁移策略

## 一、迁移步骤

```
1. 双写 - 新旧 MQ 同时写入
2. 双读 - 消费者同时消费新旧 MQ
3. 切换读 - 消费者只消费新 MQ
4. 切换写 - 生产者只写新 MQ
5. 下线旧 MQ
```

```java
// 双写阶段
public void sendOrder(Order order) {
    // 写入旧 MQ
    rabbitTemplate.convertAndSend("old-exchange", "old-key", order);
    // 写入新 MQ
    kafkaTemplate.send("new-topic", JSON.toJSONString(order));
}

// 数据校验
@Scheduled(fixedDelay = 60000)
public void verifyConsistency() {
    long oldCount = rabbitAdmin.getQueueMessageCount("old-queue");
    long newCount = getKafkaLag("new-topic");
    if (Math.abs(oldCount - newCount) > 100) {
        alertService.alert("数据不一致");
    }
}
```

## 二、详细迁移步骤

```
迁移时间线 (建议 4-8 周):

第 1-2 周：准备阶段
  ├── 搭建新 MQ 集群
  ├── 搭建监控告警
  ├── 编写双写代码
  └── 编写数据校验脚本

第 3-4 周：双写阶段
  ├── 生产者双写新旧 MQ
  ├── 消费者只消费旧 MQ
  ├── 数据校验脚本定时运行
  └── 监控新 MQ 指标

第 5-6 周：双读切换
  ├── 消费者同时消费新旧 MQ（选一个做主）
  ├── 对比消费结果
  └── 逐步将消费者切到新 MQ

第 7-8 周：完成迁移
  ├── 生产者只写新 MQ
  ├── 下线旧 MQ 生产代码
  ├── 保留旧 MQ 1-2 周用于对账
  └── 下线旧 MQ 集群
```

## 三、数据校验方案

```java
// 双写阶段的数据校验
@Component
public class MigrationValidator {
    @Scheduled(fixedDelay = 60000)
    public void validate() {
        // 对比新旧 MQ 的消息量
        long oldCount = rabbitAdmin.getQueueMessageCount("old-queue");
        long newLag = kafkaAdmin.getEndOffset("new-topic") - kafkaAdmin.getCurrentOffset("new-topic");

        // 对比业务数据
        long oldDbCount = orderMapper.countByTimeRange(startTime, endTime);
        long newDbCount = orderMapperNew.countByTimeRange(startTime, endTime);

        double diffPercent = Math.abs(oldDbCount - newDbCount) * 100.0 / oldDbCount;

        if (diffPercent > 0.1) {
            alertService.send("迁移数据不一致",
                String.format("旧: %d, 新: %d, 差异: %.2f%%", oldDbCount, newDbCount, diffPercent));
        }
    }

    // 消息内容抽样对比
    public void deepValidate(String oldQueue, String newTopic, int sampleSize) {
        List<String> oldMessages = sampleFromRabbit(oldQueue, sampleSize);
        List<String> newMessages = sampleFromKafka(newTopic, sampleSize);
        // 对比消息内容
        Set<String> diff = new HashSet<>(oldMessages);
        diff.removeAll(newMessages);
        if (!diff.isEmpty()) {
            log.warn("消息内容不一致: {} 条", diff.size());
        }
    }
}
```

## 四、回滚方案

```java
// 回滚策略：通过配置开关控制
@Configuration
public class MigrationConfig {
    @Value("${mq.migration.phase:old}")
    private String migrationPhase;  // old, dual-write, new

    public void sendOrder(Order order) {
        switch (migrationPhase) {
            case "old":
                rabbitTemplate.convertAndSend("old-exchange", "old-key", order);
                break;
            case "dual-write":
                rabbitTemplate.convertAndSend("old-exchange", "old-key", order);
                kafkaTemplate.send("new-topic", JSON.toJSONString(order));
                break;
            case "new":
                kafkaTemplate.send("new-topic", JSON.toJSONString(order));
                break;
        }
    }
}
```

## 五、注意事项

1. **双写期间要做好数据校验**
2. **消费端要兼容新旧两种格式**
3. **切换过程要有回滚方案**
4. **分批迁移，不要一次性全切**
5. **保留旧 MQ 数据一段时间用于对账**
6. **灰度期间持续监控**，异常时立即回滚
7. **迁移要选在业务低峰期**，减少影响
