# 召回与排序架构


## 1. 工业推荐的漏斗架构


### 1.1 整体架构


工业级推荐系统采用**漏斗架构**，从海量候选集中逐层筛选，每层的模型复杂度和候选集规模形成反比关系：


```
┌──────────────────────────────────────────────┐
│              全量候选集（亿级）                 │
├──────────────────────────────────────────────┤
│   召回层     │  多路召回，快速粗筛    → 数千条  │  延迟 < 5ms
├──────────────────────────────────────────────┤
│   粗排层     │  轻量模型初步排序     → 数百条  │  延迟 < 10ms
├──────────────────────────────────────────────┤
│   精排层     │  深度模型精确排序     → 数十条  │  延迟 < 100ms
├──────────────────────────────────────────────┤
│   重排层     │  业务规则与多样性调整  → 最终展示 │  延迟 < 10ms
└──────────────────────────────────────────────┘
```


### 1.2 各层的权衡


| 层级 | 候选规模 | 模型复杂度 | 延迟要求 | 核心目标 |
| --- | --- | --- | --- | --- |
| 召回 | 亿→千 | 极低 | < 5ms | 覆盖度、多样性 |
| 粗排 | 千→百 | 低 | < 10ms | 快速筛选 |
| 精排 | 百→十 | 高 | < 100ms | 精准排序 |
| 重排 | 十→最终 | 业务规则 | < 10ms | 体验与业务目标 |


> **Note:** **为什么要漏斗结构？**
> 如果直接对亿级候选集用深度模型排序，单次推理需要数秒，无法满足在线服务的延迟要求。漏斗结构通过"粗筛→精排"的级联方式，在保证推荐质量的同时控制计算开销。


## 2. 召回层：多路召回


### 2.1 常见召回路


| 召回路 | 方法 | 特点 |
| --- | --- | --- |
| 协同过滤召回 | ItemCF / UserCF | 经典有效，利用群体行为 |
| 热门召回 | 按热度/时效排序 | 解决冷启动，兜底策略 |
| 内容召回 | TF-IDF / 标签匹配 | 基于物品属性，不需行为数据 |
| 向量召回 | 双塔模型（DSSM） | 端到端学习，语义匹配 |
| 图召回 | GraphSAGE / PinSage | 高阶关系挖掘 |
| 实时召回 | 实时行为触发 | 捕捉实时兴趣 |
| 地理召回 | 基于LBS | 本地生活、O2O场景 |


### 2.2 多路召回的融合


各路召回的结果合并后去重，送入粗排层：


- 每路召回独立工作，各取Top-K
- 各路配额可不同（如向量召回取1000，热门取200）
- 融合策略：并集取回、按来源分桶


## 3. 向量召回与ANN检索


### 3.1 双塔模型（DSSM）


双塔模型是向量召回的核心方法，分别用用户塔和物品塔将用户和物品编码为Embedding向量，通过内积计算相似度。


$$
User Tower: u = fuser(用户特征)
                Item Tower: v = fitem(物品特征)
                相似度: score = u · v
$$


- **训练**
   ：正样本为用户交互过的物品，负样本随机采样或热度采样
- **推理**
   ：物品塔离线计算所有物品Embedding存入向量索引；用户Embedding在线计算后查询最近邻
- **优势**
   ：用户塔和物品塔解耦，物品Embedding可离线预计算


### 3.2 ANN（Approximate Nearest Neighbor）近似最近邻


从亿级物品Embedding中找到最相似的Top-K个，精确搜索不可行，需要ANN近似搜索：


| 工具 | 方法 | 特点 |
| --- | --- | --- |
| Faiss（Facebook） | IVF + PQ | 通用性强，支持GPU加速 |
| ScaNN（Google） | 各向异性量化 | 精度-效率平衡好 |
| HNSWlib | 可导航小世界图 | 高召回率，内存占用大 |
| Milvus | 集成多种算法 | 分布式向量数据库 |
| Annoy（Spotify） | 随机投影树 | 内存映射，适合只读场景 |


> **Example:** **Faiss使用示例：**
>
> ```
> import faiss
> import numpy as np
>
> # 构建索引
> d = 64        # 向量维度
> n_items = 1000000  # 物品数量
> item_vectors = np.random.rand(n_items, d).astype('float32')
>
> # IVF索引（倒排文件）
> nlist = 1000  # 聚类中心数
> quantizer = faiss.IndexFlatIP(d)  # 内积相似度
> index = faiss.IndexIVFFlat(quantizer, d, nlist)
> index.train(item_vectors[:100000])
> index.add(item_vectors)
>
> # 搜索
> query = np.random.rand(1, d).astype('float32')
> index.nprobe = 20  # 搜索的聚类数
> distances, indices = index.search(query, k=10)
> print(f"最近邻索引: {indices}")
> ```


## 4. 粗排层与精排层


### 4.1 粗排层（Pre-ranking）


粗排层在召回和精排之间，用轻量级模型对数千个候选进行快速排序，缩减到数百个。


- **模型选择**
   ：浅层网络、双线性模型、COLD（轻量交叉网络）
- **特征设计**
   ：以统计特征和低阶交叉为主，避免复杂特征
- **延迟要求**
   ：通常 < 10ms
- **优化方向**
   ：知识蒸馏（用精排模型蒸馏粗排模型）


### 4.2 精排层（Ranking）


精排层是推荐系统的核心，使用深度模型对候选集进行精确排序。


- **模型选择**
   ：DeepFM、DIN、MMOE、DCN-V2等
- **多目标预测**
   ：同时预估CTR、CVR、停留时长等
- **特征工程**
   ：丰富的用户、物品、交叉、上下文特征
- **最终打分**
   ：多目标加权融合得分


$$
final_score = w1 · pCTR + w2 · pCVR · price + w3 · pStayTime + ...
                （电商场景常用 pCVR × price 作为GMV预估值）
$$


### 4.3 知识蒸馏在粗排中的应用


> **Example:** **蒸馏流程：**
>
> 1. 训练精排模型（Teacher）：使用复杂的DIN+MMOE等
> 2. 用Teacher模型对训练数据打分，生成soft label
> 3. 训练粗排模型（Student）：用soft label + 真实label联合训练
> 4. Student模型更简单更轻量，但能近似Teacher的排序能力


## 5. 重排层（Re-ranking）


### 5.1 重排的目标


精排层只关注单个物品的得分，重排层需要从**列表级**（List-level）视角优化整体推荐效果。


### 5.2 重排的常见策略


| 策略 | 描述 | 方法 |
| --- | --- | --- |
| 多样性保障 | 避免推荐结果过于同质化 | DPP、MMR、类目打散 |
| 去重与频控 | 去除已曝光/已购买物品，控制推荐频率 | 规则过滤 |
| 业务规则 | 广告插入、置顶推荐、新品扶持 | 加权/插槽 |
| 上下文感知 | 考虑位置偏差、时间因素 | 位置消偏、时间调权 |
| 用户体验 | 穿插新鲜内容，避免信息茧房 | 探索策略、随机插入 |


### 5.3 MMR（Maximal Marginal Relevance）


$$
MMR = arg maxDi∈R\S [λ · sim(Di, Q) - (1-λ) · maxDj∈S sim(Di, Dj)]
$$


在相关性和多样性之间做权衡，λ越大越注重相关性，越小越注重多样性。


### 5.4 DPP（Determinantal Point Process）


DPP通过行列式计算物品子集的多样性得分，天然具有"越大越好、越分散越好"的性质。


- 构建物品间的核矩阵L（L
   ~ij~
   = q
   ~i~
   · s
   ~ij~
   · q
   ~j~
   ）
- 最优子集 = arg max det(L
   ~S~
   )
- 实际中使用贪心或Fast Greedy MAP近似求解


## 6. 特征工程


### 6.1 特征分类


| 特征类别 | 示例 | 说明 |
| --- | --- | --- |
| 用户特征 | 用户ID、年龄、性别、城市、注册天数、历史CTR | 描述用户画像 |
| 物品特征 | 物品ID、类目、品牌、价格、标签、创建时间 | 描述物品属性 |
| 交叉特征 | 用户-类目偏好、用户-品牌偏好 | 用户与物品的关联 |
| 上下文特征 | 时间、星期、设备、网络、地理位置 | 当前请求环境 |
| 统计特征 | 物品曝光数、点击数、CTR、7日CTR趋势 | 离线/实时统计 |
| 序列特征 | 用户最近N次行为序列 | 建模兴趣演化 |
| 图特征 | 用户/物品的Graph Embedding | 社交/知识图谱 |


### 6.2 实时特征 vs 离线特征


- **离线特征**
   ：通过Spark/Hive定时计算（天/小时级），如用户历史统计、物品画像
- **近实时特征**
   ：通过Flink流计算（分钟级），如最近30分钟点击数
- **实时特征**
   ：请求到达时在线计算，如当前session内的行为序列


### 6.3 特征系统的架构


```
数据源 → [特征抽取] → [特征存储] → [特征服务] → 推荐模型
                      ├ Redis(实时)   ├ 特征拼接
                      ├ HBase(离线)   ├ 特征验证
                      └ 向量索引      └ 特征监控
```


> **Important:** **特征工程的经验法则：**
>
> - 特征工程对模型效果的影响远大于模型结构的改进
> - 实时特征（最近行为、session特征）通常带来最大的线上提升
> - 特征版本管理和AB实验框架同样重要
> - 需要建立特征监控体系，防止特征异常（如缺失率突增）影响线上


## 7. 端到端推荐系统架构总结


### 7.1 数据流


```
用户请求 → [特征服务] → [召回(并行多路)] → [合并去重]
    → [粗排(轻量模型)] → [精排(深度模型)] → [重排(多样性+规则)]
    → [返回推荐列表] → [曝光日志] → [训练数据] → [模型更新]
```


### 7.2 技术栈概览


| 模块 | 常用技术 |
| --- | --- |
| 特征存储 | Redis、HBase、Feature Store |
| 向量索引 | Faiss、ScaNN、Milvus |
| 模型训练 | TensorFlow、PyTorch、DeepSpeed |
| 模型Serving | TensorFlow Serving、Triton、ONNX Runtime |
| 实时计算 | Flink、Kafka、Spark Streaming |
| AB实验 | 自研平台、Statsig |
| 监控报警 | Grafana、Prometheus、ELK |


### 7.3 关键设计原则


- **解耦**
   ：召回、排序、重排各层独立迭代，降低耦合
- **可观测**
   ：每一层都有监控指标，方便定位问题
- **可回退**
   ：模型出问题时可以快速回退到上一版本
- **增量更新**
   ：支持模型的在线/增量训练，快速响应数据变化
- **资源隔离**
   ：核心链路和非核心链路资源隔离，保证稳定性


> **Note:** **推荐系统工程师的核心能力：**
> 不仅需要掌握深度学习模型，还需要理解系统架构、特征工程、AB实验和业务逻辑。工业推荐系统是一个复杂的系统工程，模型只是其中一环。
