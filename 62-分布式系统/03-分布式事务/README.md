# 03-分布式事务

> 分布式事务是分布式系统中最复杂的难题之一。在微服务架构盛行的今天，如何在保证数据一致性的同时维持系统的可用性和性能，是每位架构师必须面对的挑战。

---

## 1. 分布式事务的挑战

### 问题根源

在单体应用中，事务由数据库的 ACID 特性保证。但在分布式环境下：

- **数据分散**：数据存储在多个独立的数据库或服务中
- **网络不可靠**：消息可能丢失、延迟、重复
- **节点可能故障**：协调者或参与者随时可能宕机
- **缺乏全局时钟**：无法精确排序跨节点的事件

### 一致性与可用性的权衡

根据 CAP 定理，分布式事务面临根本性的取舍：

- 强一致性方案（如 2PC）通常带来较高的延迟和较低的可用性
- 最终一致性方案（如 Saga）牺牲了实时一致性换取更好的可用性和性能

---

## 2. 两阶段提交（2PC）

两阶段提交（Two-Phase Commit）是最经典的强一致性分布式事务协议，由 Jim Gray 于 1978 年提出。

### 基本流程

#### Prepare 阶段（投票阶段）

1. 协调者（Coordinator）向所有参与者（Participant）发送 `Prepare` 请求。
2. 每个参与者执行事务操作（写 redo/undo 日志，但不提交），检查是否可以提交。
3. 参与者回复 `Vote-Yes`（同意提交）或 `Vote-No`（拒绝提交）。
4. 如果参与者在超时时间内未回复，协调者视为拒绝。

#### Commit 阶段（执行阶段）

- **全部同意**：协调者向所有参与者发送 `Commit` 请求，参与者提交事务并释放锁，回复 `ACK`。
- **存在拒绝**：协调者向所有参与者发送 `Rollback` 请求，参与者回滚事务并释放锁。

### 协调者故障处理

| 故障场景 | 处理方式 |
|----------|----------|
| 协调者在 Prepare 前故障 | 参与者超时后自动回滚 |
| 协调者在 Prepare 后、Commit 前故障 | 参与者阻塞等待，需要协调者恢复后决定 |
| 协调者在 Commit 后、收到所有 ACK 前故障 | 协调者恢复后重发 Commit |
| 参与者在 Prepare 前故障 | 协调者超时视为拒绝，触发全局回滚 |
| 参与者在回复 Yes 后故障 | 恢复后询问协调者决定 |

### 2PC的阻塞问题

2PC 最大的缺陷是**阻塞**：

- 在 Prepare 阶段之后，参与者已经持有了锁并写入了日志，但必须等待协调者的最终决定
- 如果协调者故障，所有参与者将无限期阻塞，持有的锁无法释放
- 这会导致严重的资源浪费和可用性下降
- 协调者是单点，一旦故障整个系统不可用

### 3PC 概述

三阶段提交（3PC）在 2PC 基础上增加了超时机制和预提交阶段来缓解阻塞问题（详见下一节）。

---

## 3. 三阶段提交（3PC）

三阶段提交（Three-Phase Commit）在 2PC 基础上引入了预提交阶段和超时机制，旨在减少阻塞。

### 三个阶段

#### CanCommit 阶段

1. 协调者向所有参与者发送 `CanCommit` 请求。
2. 参与者评估自身状态，回复 `Yes` 或 `No`。
3. 此阶段不执行任何实际操作，不锁定资源，只是探询。

#### PreCommit 阶段

- **全部同意**：协调者发送 `PreCommit` 请求，参与者执行事务操作（写日志），回复 `ACK`。
- **存在拒绝或超时**：协调者发送 `Abort`，所有节点中止。

#### DoCommit 阶段

- 协调者发送 `DoCommit` 请求，参与者提交事务。
- 参与者若超时未收到 `DoCommit`，则自动提交（基于预提交阶段已达成的共识）。

### 超时机制

3PC 的关键改进是在 PreCommit 阶段之后引入了超时自动提交机制：

- 如果参与者在 PreCommit 阶段成功回复了 `ACK`，但在 DoCommit 阶段超时未收到指令，参与者可以自主决定提交。
- 这避免了 2PC 中协调者故障导致的无限阻塞。

### 3PC不能完全解决网络分区问题

尽管 3PC 改善了阻塞问题，但存在以下局限：

- **网络分区下的数据不一致**：当网络分区将部分参与者与协调者隔离时，被隔离的参与者可能超时自动提交，而其他参与者收到 Abort 指令回滚，导致数据不一致。
- **实际应用较少**：由于 3PC 对网络分区无能为力且增加了通信轮次，实际系统中很少直接使用。通常结合 Paxos/Raft 等共识算法来解决协调者单点问题。

## 4. TCC（Try-Confirm-Cancel）

TCC 是一种业务层面的两阶段补偿方案，由支付宝等公司推广。

### 三个阶段

#### Try 阶段

- 对业务资源做检查和预留
- 例如：检查库存并冻结库存、检查账户余额并冻结金额
- 不真正执行业务操作，只是"准备"

#### Confirm 阶段

- 确认执行业务操作
- 使用 Try 阶段预留的资源
- 必须保证幂等性（因为可能被重试）

#### Cancel 阶段

- 取消 Try 阶段的资源预留
- 释放冻结的库存或金额
- 同样需要幂等性

### 特点与适用场景

- **侵入性强**：需要业务代码配合改造，每个服务都需要实现 Try/Confirm/Cancel 三个接口
- **强一致性**：在理想情况下可以保证强一致性
- **适用场景**：金融交易、支付等对一致性要求极高的场景
- **缺点**：开发成本高，业务耦合度大，Cancel 逻辑可能非常复杂

## 5. Saga模式

Saga 模式由 Hector Garcia-Molina 和 Kenneth Salem 于 1987 年提出，适用于长事务场景。

### 核心思想

将一个长事务拆分为一系列**本地事务**，每个本地事务有对应的**补偿事务**。当某个本地事务失败时，按逆序执行已成功的补偿事务来回滚。

### 向前恢复与向后恢复

#### 向前恢复（Forward Recovery）

- 当某个步骤失败时，不回滚，而是重试直到成功
- 适用于临时性故障（如网络抖动）
- 需要保证步骤的幂等性

#### 向后恢复（Backward Recovery）

- 当某个步骤失败时，按逆序执行补偿操作
- 补偿事务是正向事务的"逆操作"
- 例如：下单成功 → 扣库存成功 → 支付失败 → 释放库存 → 取消订单

### Saga协调器

#### 协调器模式（Orchestration）

- 引入一个中央协调器（Orchestrator），负责编排整个 Saga 流程
- 协调器调用各个参与者的正向操作或补偿操作
- **优点**：流程清晰、易于理解和调试、参与方解耦
- **缺点**：协调器可能成为单点和性能瓶颈

#### 协同模式（Choreography）

- 没有中央协调器，各参与者通过事件驱动协作
- 每个参与者完成后发布事件，触发下一个参与者执行
- **优点**：去中心化、松耦合、易扩展
- **缺点**：流程分散、难以追踪和调试、可能出现循环依赖

### Saga 的局限性

- **隔离性缺失**：Saga 的中间状态对外可见，可能出现脏读
- **补偿事务复杂**：某些操作难以补偿（如发送邮件、扣减积分）
- **不是所有业务都能接受最终一致性**

## 6. 本地消息表

本地消息表是一种基于可靠消息的最终一致性方案，由 eBay 提出。

### 实现原理

1. 在业务数据库中创建一张"消息表"，与业务操作在同一个本地事务中写入。
2. 本地事务提交后，后台任务读取消息表，将消息发送到消息队列。
3. 下游服务消费消息并执行操作，完成后通知上游标记消息已处理。
4. 后台任务定期重试未成功发送或未被确认的消息。

### 关键设计

- **本地事务保证**：业务操作和消息记录在同一个数据库事务中，要么都成功要么都失败。
- **幂等消费**：下游服务必须保证消息的幂等性处理。
- **异步重试**：通过定时任务扫描未处理的消息进行重试。
- **人工兜底**：对于始终无法成功的消息，最终需要人工介入处理。

### 适用场景

- 对一致性要求不是特别高、可以接受短暂不一致的场景
- 跨服务的异步操作，如订单创建后通知库存服务

## 7. 事务性发件箱（Transactional Outbox）

事务性发件箱模式是本地消息表的改进版本，更加通用。

### 工作原理

1. 业务操作和"发件箱记录"在同一个事务中写入。
2. 使用 **CDC（Change Data Capture）** 工具（如 Debezium）监听发件箱表的变更。
3. CDC 工具将变更事件发布到消息队列（如 Kafka）。
4. 下游服务消费消息并处理。

### 与本地消息表的区别

| 特性 | 本地消息表 | 事务性发件箱 |
|------|-----------|-------------|
| 消息投递方式 | 应用主动轮询发送 | CDC 监听自动投递 |
| 实时性 | 取决于轮询间隔 | 近实时（CDC 延迟极低） |
| 对业务代码侵入 | 需要写消息表 | 发件箱写入与业务同事务 |
| 基础设施依赖 | 仅需消息队列 | 需要 CDC 工具 |

### Debezium + Outbox

这是目前最流行的实现方案：

1. 应用写入 outbox 表
2. Debezium 通过 Kafka Connect 监听 binlog 变更
3. 将变更事件推送到 Kafka Topic
4. 下游消费者处理事件

## 8. 最大努力通知

最大努力通知是一种最弱的一致性保障方案，适用于系统间跨组织或跨平台的通信。

### 实现方式

1. 主动方完成操作后，向被动方发送通知消息（通常通过 MQ）。
2. 被动方收到通知后处理业务逻辑。
3. 如果通知失败，主动方按照一定的策略（间隔递增）反复重试。
4. 经过一定次数后仍不成功，则不再重试，需要人工介入。

### 典型场景

- 支付结果通知（支付系统通知商户系统）
- 对账系统（定期核对双方数据的一致性）
- 跨公司的业务协作

## 9. 分布式事务框架（Seata）

Seata 是阿里开源的分布式事务解决方案，提供 AT、TCC、Saga、XA 等多种模式。

### 架构组件

- **TC（Transaction Coordinator）**：事务协调者，维护全局事务和分支事务的状态。
- **TM（Transaction Manager）**：事务管理器，定义全局事务的边界（开始、提交、回滚）。
- **RM（Resource Manager）**：资源管理器，管理分支事务的资源。

### AT模式（Automatic Transaction）

- **一阶段**：执行业务 SQL，拦截 SQL 生成前后镜像，写入 undo_log 表，本地提交。
- **二阶段提交**：删除 undo_log，异步清理。
- **二阶段回滚**：根据 undo_log 反向补偿，恢复数据到事务前状态。
- **优点**：对业务无侵入，自动回滚，类似本地事务的使用体验。
- **缺点**：需要全局锁保证隔离性，性能有一定损耗。

### 其他模式

| 模式 | 一致性 | 性能 | 侵入性 | 适用场景 |
|------|--------|------|--------|----------|
| AT | 强一致 | 中 | 低 | 常规业务 |
| TCC | 强一致 | 高 | 高 | 高性能、金融 |
| Saga | 最终一致 | 高 | 中 | 长事务 |
| XA | 强一致 | 低 | 低 | 需要数据库XA支持 |

## 10. 最终一致性 vs 强一致性的选型

### 选择强一致性方案的场景

- 金融交易：转账、扣款操作必须实时一致
- 库存管理：超卖问题不能接受
- 账户安全：余额不能出现负值
- 核心业务数据：如订单状态的一致性

**推荐方案**：2PC（Seata AT/TCC）、基于共识算法的强一致存储

### 选择最终一致性方案的场景

- 通知类业务：短信、邮件、推送通知
- 日志和统计：分析数据的短暂不一致可接受
- 跨系统同步：不同组织之间的数据同步
- 长流程业务：审批流、工单流转

**推荐方案**：Saga、本地消息表、事务性发件箱

### 决策矩阵

| 维度 | 强一致性 | 最终一致性 |
|------|----------|-----------|
| 数据正确性 | 实时正确 | 最终正确 |
| 可用性 | 可能降低 | 较高 |
| 性能 | 较低 | 较高 |
| 实现复杂度 | 高 | 中 |
| 运维成本 | 高 | 中 |
| 用户体验 | 即时反馈 | 可能短暂不一致 |

### 混合策略

实际系统中通常采用混合策略：

- 核心路径（支付、扣库存）使用强一致性（TCC/AT）
- 非核心路径（通知、积分）使用最终一致性（Saga/MQ）
- 通过领域驱动设计（DDD）识别聚合根，明确事务边界
