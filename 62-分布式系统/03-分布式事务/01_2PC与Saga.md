# 2PC与Saga


## 2PC与Saga分布式事务


分布式事务Saga


分布式事务确保跨服务/跨数据库操作的原子性。


## 两阶段提交 (2PC)


```
2PC (Two-Phase Commit)：
协调者(Coordinator)控制所有参与者(Participants)的提交/回滚

Phase 1: Prepare（准备阶段）
┌─────────────────────────────────────────────┐
│  Coordinator                                 │
│     │                                        │
│     ├─→ Participant1: PREPARE                │
│     ├─→ Participant2: PREPARE                │
│     └─→ Participant3: PREPARE                │
│                                              │
│  参与者执行事务但不提交，写入redo/undo日志    │
│  返回 YES（可以提交）或 NO（无法提交）        │
└─────────────────────────────────────────────┘

Phase 2: Commit/Rollback（提交/回滚阶段）
┌─────────────────────────────────────────────┐
│  如果所有参与者都返回YES：                    │
│  Coordinator → COMMIT → 所有参与者            │
│  参与者执行COMMIT，释放资源                   │
│                                             │
│  如果任一参与者返回NO或超时：                  │
│  Coordinator → ROLLBACK → 所有参与者          │
│  参与者执行ROLLBACK，回滚事务                 │
└─────────────────────────────────────────────┘

2PC的问题：
- 同步阻塞：准备阶段参与者锁定资源
- 单点故障：协调者宕机，参与者一直阻塞
- 数据不一致：协调者在Phase2宕机，部分提交部分未提交
- 性能差：两轮网络往返，延迟高
```


## 三阶段提交 (3PC)


```
3PC：在2PC基础上增加Pre-Commit阶段

Phase 1: CanCommit（询问阶段）
- 协调者询问参与者是否可以提交
- 参与者仅检查可行性，不锁定资源

Phase 2: PreCommit（预提交阶段）
- 如果所有参与者都YES，发送PreCommit
- 参与者执行事务，写入undo/redo日志

Phase 3: DoCommit（提交阶段）
- 协调者发送DoCommit
- 参与者提交事务

3PC改进点：
- 增加超时机制：参与者在PreCommit后超时可自行提交
- 减少了阻塞：协调者宕机时参与者有应对策略
- 但仍不能完全解决网络分区下的数据不一致问题

实际中3PC很少使用：
- 复杂度增加但问题未完全解决
- 工业界更倾向TCC或Saga
```


## Saga模式


```
Saga（1987年提出，微服务时代复兴）：
长事务拆分为多个本地事务，每个有对应的补偿操作

编排式Saga (Orchestration)：
┌─────────────────────────────────────────────┐
│  Saga协调器                                   │
│  T1 → T2 → T3（全部成功，提交）               │
│                                             │
│  T1成功 → T2失败 → C1（补偿T1）               │
│  T1成功 → T2成功 → T3失败 → C2 → C1         │
│                                             │
│  T = 事务, C = 补偿(Compensation)             │
└─────────────────────────────────────────────┘

示例：订单下单
┌─────────────────────────────────────────────┐
│  T1: 创建订单 (Order Service)                │
│  T2: 扣减库存 (Inventory Service)            │
│  T3: 扣款 (Payment Service)                  │
│  T4: 发货 (Shipping Service)                 │
│                                             │
│  如果T3失败：                                 │
│  C2: 恢复库存                                │
│  C1: 取消订单                                │
│                                             │
│  补偿操作必须是幂等的！                       │
└─────────────────────────────────────────────┘

协同式Saga (Choreography)：
- 没有中心协调器
- 每个服务监听事件，触发下一步
- 通过消息队列传递事件
- 简单但难以追踪和调试

Saga vs 2PC：
┌──────────┬──────────────┬──────────────────┐
│ 对比      │ 2PC          │ Saga             │
├──────────┼──────────────┼──────────────────┤
│ 锁定      │ 长时间锁定    │ 不锁定            │
│ 一致性    │ 强一致        │ 最终一致          │
│ 性能      │ 低（阻塞）    │ 高                │
│ 复杂度    │ 低            │ 高（补偿逻辑）    │
│ 适用场景  │ 短事务        │ 长事务/跨服务     │
└──────────┴──────────────┴──────────────────┘
```


## TCC (Try-Confirm-Cancel)


```
TCC：业务层面的两阶段提交

三个阶段：
1. Try：资源预留（检查+冻结）
   - 检查业务规则
   - 预留业务资源（如冻结库存、冻结余额）
   - 不执行真正的业务操作

2. Confirm：确认提交
   - 使用Try阶段预留的资源
   - 不再做业务检查
   - 必须成功（幂等）

3. Cancel：取消释放
   - 释放Try阶段预留的资源
   - 必须成功（幂等）

示例：订单扣款
Try:   检查余额≥100，冻结100元
Confirm: 实际扣减100元
Cancel: 解冻100元

TCC vs Saga：
- TCC：业务侵入性强，需改造业务代码（Try/Confirm/Cancel）
- Saga：业务侵入性低，只需定义补偿操作
- TCC一致性更好（资源已预留）
- Saga更灵活，适合已有系统
```


> **Note:** Seata框架提供了AT、TCC、Saga、XA等多种分布式事务模式，是Java微服务的首选。
