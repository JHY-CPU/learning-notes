拜占庭容错解决节点可能恶意行为时的共识问题。

拜占庭将军问题

拜占庭将军问题（Lamport 1982年）： - 多个将军围困拜占庭城，需要达成进攻/撤退的共识 - 将军之间通过信使传递消息 - 部分将军可能是叛徒，会发送错误信息 - 问题是：忠诚将军如何达成一致？ 故障模型分类： ┌─────────────────────────────────────────────┐ │ 非拜占庭故障 (Crash Fault) │ │ - 节点只能崩溃或不响应 │ │ - 不会发送错误消息 │ │ - Raft/Paxos解决这类问题 │ │ - 需要 n ≥ 2f+1 个节点容忍f个故障 │ │ │ │ 拜占庭故障 (Byzantine Fault) │ │ - 节点可能任意行为：崩溃、篡改消息、合谋 │ │ - 恶意节点可以发送矛盾的信息 │ │ - 需要 n ≥ 3f+1 个节点容忍f个故障 │ │ - 区块链场景必须考虑 │ └─────────────────────────────────────────────┘ 结论：n ≥ 3f+1 是BFT的下界 - 3f+1个节点中，f个恶意节点 - 剩余2f+1个诚实节点可以形成多数派

PBFT (实用拜占庭容错)

PBFT（Castro & Liskov 1999年）： 第一个实用的BFT算法，适用于许可链/联盟链 三阶段协议： ┌─────────────────────────────────────────────┐ │ Request → Pre-Prepare → Prepare → Commit │ │ │ │ 1. Request：客户端发送请求给主节点 │ │ │ │ 2. Pre-Prepare：主节点分配序号，广播 │ │ <PRE-PREPARE, v, n, d> │ │ v=view, n=序号, d=消息摘要 │ │ │ │ 3. Prepare：副本验证后广播Prepare │ │ 收到2f个Prepare → 进入prepared状态 │ │ │ │ 4. Commit：广播Commit消息 │ │ 收到2f+1个Commit → 执行请求 │ │ │ │ View Change：主节点故障时切换 │ │ 副本超时后发起View Change，选举新主节点 │ └─────────────────────────────────────────────┘ PBFT复杂度： - 消息复杂度：O(n²)（每轮所有节点互相通信） - 因此只适合小规模集群（通常n≤100） - 不适合大规模公有链 Hyperledger Fabric使用PBFT变体： - 支持动态节点增删 - 优化了消息复杂度

区块链共识机制

PoW (Proof of Work) 工作量证明： - 矿工计算哈希值，找到满足难度目标的nonce - 成本：电力和算力 - 安全性：51%攻击需要超过全网一半算力 - 缺点：能耗巨大，确认慢（~10分钟） - 代表：Bitcoin, (旧)Ethereum PoS (Proof of Stake) 权益证明： - 验证者质押代币，按比例获得出块权 - 恶意行为会被罚没(slashing)质押 - 成本：质押代币的机会成本 - 优势：能效高，确认快 - 代表：Ethereum 2.0, Cardano, Polkadot DPoS (Delegated PoS) 委托权益证明： - 代币持有者投票选举少数超级节点 - 超级节点轮流出块 - 优势：高吞吐量（毫秒级确认） - 缺点：中心化程度较高 - 代表：EOS, Tron 其他共识： - PBFT：许可链（Hyperledger） - Raft/Paxos：非拜占庭场景 - DAG共识：IOTA (Tangle)
PoW到PoS是以太坊的重大转型（The Merge, 2022年9月），能耗降低了99.95%。
