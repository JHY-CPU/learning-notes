一致性哈希

分布式数据分区负载均衡

一致性哈希解决分布式系统中节点增删时数据迁移量最小化的问题。

数据分区的问题

普通哈希取模：node = hash(key) % N 问题：当N变化时（增删节点），几乎所有数据都要迁移！ 示例： hash("key1") % 4 = 1 → 节点1 hash("key1") % 5 = 1 → 节点1（不变） hash("key2") % 4 = 2 → 节点2 hash("key2") % 5 = 2 → 节点2（不变） hash("key3") % 4 = 0 → 节点0 hash("key3") % 5 = 3 → 节点3（变了！） 3节点→4节点：大约75%的数据需要迁移 4节点→5节点：大约80%的数据需要迁移 需求：增删节点时只迁移少量数据

一致性哈希原理

一致性哈希：将哈希空间组织为环（0 ~ 2^32-1） ┌─────────────────────────────────────────────┐ │ 0 │ │ / \ │ │ node2 ● ● node0 │ │ / \ │ │ 2^31 ●───────● 2^0 │ │ \ / │ │ node3 ● ● node1 │ │ \ / │ │ 2^32-1 │ │ │ │ 数据映射： │ │ key1 → hash → 落在node0和node1之间 │ │ → 顺时针找到第一个节点 → node1 │ │ key2 → hash → 落在node1和node3之间 │ │ → 顺时针找到第一个节点 → node3 │ └─────────────────────────────────────────────┘ 增删节点的影响： - 删除node1：只有node1上的数据迁移到node3 - 增加node4：只有node3→node4区间的数据从node3迁到node4 - 平均只迁移 N/K 的数据（N=数据量，K=节点数） 问题：数据倾斜 - 节点在环上分布不均匀 - 导致某些节点负载过高 - 解决：虚拟节点

虚拟节点 (Virtual Nodes)

虚拟节点：每个物理节点映射多个虚拟节点到哈希环上 node0 → vnode0-0, vnode0-1, ..., vnode0-k node1 → vnode1-0, vnode1-1, ..., vnode1-k node2 → vnode2-0, vnode2-1, ..., vnode2-k ┌─────────────────────────────────────────────┐ │ ●v0-0 ●v2-1 │ │ ●v1-2 ●v0-2 │ │ ●v2-0 ●v1-0 │ │ ●v0-1 ●v2-2 │ │ ●v1-1 │ └─────────────────────────────────────────────┘ 虚拟节点数量通常100-200个/物理节点 优势： - 数据分布更均匀 - 异构节点：高性能节点分配更多虚拟节点 - 平滑增删：增删节点影响更小 Redis Cluster使用一致性哈希的变体（16384个slot）： - 将哈希空间分为16384个槽 - 每个节点负责一部分槽 - 迁移以槽为单位，粒度可控

数据分区策略对比

┌──────────────┬────────────────┬────────────────┐ │ 策略 │ 优点 │ 缺点 │ ├──────────────┼────────────────┼────────────────┤ │ 哈希取模 │ 简单 │ 扩缩容迁移量大 │ │ 一致性哈希 │ 扩缩容影响小 │ 可能数据倾斜 │ │ 虚拟一致性哈希│ 分布均匀 │ 实现复杂 │ │ 范围分区 │ 支持范围查询 │ 可能热点 │ │ 有序分区 │ 范围查询高效 │ 写入热点 │ └──────────────┴────────────────┴────────────────┘ 实际系统中的应用： - Memcached：客户端一致性哈希 - Cassandra：一致性哈希 + 虚拟节点 - DynamoDB：一致性哈希 - Redis Cluster：固定16384个slot - MongoDB：范围分区（Shard Key） - TiDB：范围分区（Region分裂） - CockroachDB：范围分区（Range）
一致性哈希适合key-value场景，范围分区适合需要范围查询的场景。选择取决于业务访问模式。
