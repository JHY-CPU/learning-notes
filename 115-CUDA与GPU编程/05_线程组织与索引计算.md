# 线程组织与索引计算

## 线程层次结构硬件细节

CUDA的线程按层次组织，理解每一层的硬件含义至关重要：

```
Grid（网格）= 一次kernel启动
├── Block(0,0)          Block(1,0)          ...
│   ├── Warp 0 (T0-T31)   ├── Warp 0 (T0-T31)
│   ├── Warp 1 (T32-T63)  ├── Warp 1 (T32-T63)
│   └── ...               └── ...
└── Block(0,1)          Block(1,1)          ...
    └── ...               └── ...
```

### 每层的硬件映射

| 层次 | 硬件映射 | 关键约束 |
|------|----------|----------|
| **Thread** | 独立的寄存器(32bit)和PC(程序计数器) | 每个线程有独立的执行上下文 |
| **Warp** | 32个线程锁步执行同一指令 | 调度最小粒度，不可分割 |
| **Block** | 分配到**一个SM**上执行 | 同Block的线程在同一SM，可共享内存/同步 |
| **Grid** | 分配到**所有SM**上执行 | 不同Block之间不能直接同步（默认） |

### 线性编号 vs 多维编号

硬件在线性化线程时只用一维编号。多维只是为了方便编程。

```
3D Block中 thread(x,y,z) 的线性编号:
  linear_tid = x + y * blockDim.x + z * blockDim.x * blockDim.y

Block中线程到warp的映射:
  warp_id = linear_tid / 32
  lane_id = linear_tid % 32

  注意：warp按线性编号分组，不是按多维坐标！
  例如 blockDim = (4, 4, 4) 时:
  Thread(0,0,0)=0 → Warp 0
  Thread(1,0,0)=1 → Warp 0
  Thread(2,0,0)=2 → Warp 0
  Thread(3,0,0)=3 → Warp 0
  Thread(0,1,0)=4 → Warp 0
  ...
  Thread(3,3,3)=63 → Warp 1
```

## 一维线程组织

一维配置是最简单也最常用的方式，适合处理一维数组。

```cpp
// 1D Grid, 1D Block
// blockDim.x = 256, gridDim.x = (N + 255) / 256
//
// 线性索引公式:
// globalIdx = threadIdx.x + blockIdx.x * blockDim.x
//
// warp内线程:
// warp内的32个线程: [blockIdx.x*blockDim.x + warp_id*32 + 0, ..., blockIdx.x*blockDim.x + warp_id*32 + 31]

__global__ void kernel_1d(float* data, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        data[idx] = data[idx] * 2.0f;
    }
}

// 调用示例
void launch_1d(float* d_data, int N) {
    int threads = 256;
    int blocks = (N + threads - 1) / threads;  // 向上取整
    kernel_1d<<<blocks, threads>>>(d_data, N);
}
```

索引可视化：

```
Grid: [Block0 | Block1 | Block2 | ... | BlockN-1]
       0..255  256..511  512..767       共 (N+255)/256 个Block

Block内: [Thread0 | Thread1 | ... | Thread255]
           └── Warp 0 ──┘└── Warp 1 ──┘...
           线程 0-31      线程 32-63

全局索引: blockIdx.x * blockDim.x + threadIdx.x

对于 N=1000, threads=256:
  Block 0: idx 0~255   (全部有效)
  Block 1: idx 256~511  (全部有效)
  Block 2: idx 512~767  (全部有效)
  Block 3: idx 768~999  (232有效, 24个线程空闲)
  Block 4: 不需要
```

### Block大小选择指南

| Block大小 | 占用率影响 | 寄存器压力 | 适合场景 |
|-----------|-----------|-----------|----------|
| 64 (2 warps) | 需要更多Block才能满占用 | 低 | 简单kernel，寄存器密集 |
| 128 (4 warps) | 良好 | 中等 | 通用 |
| 256 (8 warps) | 最佳平衡 | 中等 | 大多数场景的默认选择 |
| 512 (16 warps) | 可能减少驻留Block数 | 较高 | 需要大量共享内存 |
| 1024 (32 warps) | 最大Block，限制驻留Block数 | 高 | 特殊优化场景 |

**经验法则**: Block大小必须是32的倍数，256通常是安全的起点。

## 二维线程组织

处理图像、矩阵等二维数据时，2D组织更直观，且有利于合并内存访问。

```cpp
// 2D Grid, 2D Block
// 线性化索引公式:
// row = threadIdx.y + blockIdx.y * blockDim.y
// col = threadIdx.x + blockIdx.x * blockDim.x
// idx = row * width + col  (行优先存储)
//
// 关键: threadIdx.x 对应列方向，threadIdx.y 对应行方向
// 原因: 相邻的 threadIdx.x 在内存中相邻，有利于合并访问

__global__ void matrix_add_2d(const float* A, const float* B, float* C,
                               int rows, int cols) {
    int row = threadIdx.y + blockIdx.y * blockDim.y;
    int col = threadIdx.x + blockIdx.x * blockDim.x;

    if (row < rows && col < cols) {
        int idx = row * cols + col;
        C[idx] = A[idx] + B[idx];
    }
}

// 调用
void launch_matrix_add(float* d_A, float* d_B, float* d_C, int rows, int cols) {
    dim3 threadsPerBlock(16, 16);  // 256 threads/block
    dim3 blocksPerGrid(
        (cols + threadsPerBlock.x - 1) / threadsPerBlock.x,  // ceil(cols/16)
        (rows + threadsPerBlock.y - 1) / threadsPerBlock.y   // ceil(rows/16)
    );
    // 例如 1024x1024 矩阵:
    // blocksPerGrid = (64, 64) = 4096 blocks
    // 总线程 = 4096 * 256 = 1,048,576 = 1024*1024 ✓
    matrix_add_2d<<<blocksPerGrid, threadsPerBlock>>>(d_A, d_B, d_C, rows, cols);
}
```

二维索引可视化（以4x4矩阵，blockDim=(2,2)为例）：

```
4x4矩阵的线性存储（行优先）:
物理内存: [r0c0][r0c1][r0c2][r0c3][r1c0][r1c1]...
偏移:       0     1     2     3     4     5

                      col 0    col 1    col 2    col 3
              ┌────────┬────────┬────────┬────────┐
      row 0   │ (0,0)  │ (0,1)  │ (0,2)  │ (0,3)  │
              ├────────┼────────┼────────┼────────┤
      row 1   │ (1,0)  │ (1,1)  │ (1,2)  │ (1,3)  │
              ├────────┼────────┼────────┼────────┤
      row 2   │ (2,0)  │ (2,1)  │ (2,2)  │ (2,3)  │
              ├────────┼────────┼────────┼────────┤
      row 3   │ (3,0)  │ (3,1)  │ (3,2)  │ (3,3)  │
              └────────┴────────┴────────┴────────┘

blockDim = (2, 2):
  Block(0,0)处理: (0,0)(0,1)(1,0)(1,1) → 索引 0,1,4,5
  Block(1,0)处理: (0,2)(0,3)(1,2)(1,3) → 索引 2,3,6,7
  Block(0,1)处理: (2,0)(2,1)(3,0)(3,1) → 索引 8,9,12,13
  Block(1,1)处理: (2,2)(2,3)(3,2)(3,3) → 索引 10,11,14,15

Thread映射到warp（在Block(0,0)中）:
  Thread(0,0)=线性0 → global: 0
  Thread(1,0)=线性1 → global: 1
  Thread(0,1)=线性2 → global: 4
  Thread(1,1)=线性3 → global: 5
  注意: 这4个线程在同一warp内（如果block内只有4个线程）
```

### 为什么threadIdx.x对应列方向？

这是一个容易出错的细节。原因是**合并内存访问（Coalesced Memory Access）**：

```
正确: threadIdx.x → col → 相邻线程访问相邻内存
  Thread(0,0) 访问 col=blockIdx.x*16+0
  Thread(1,0) 访问 col=blockIdx.x*16+1  ← 相邻！
  → 合并为一次128B的内存事务（高效）

错误: threadIdx.x → row → 相邻线程访问跨行内存
  Thread(0,0) 访问 row=blockIdx.y*16+0, 所有列
  Thread(1,0) 访问 row=blockIdx.y*16+1, 所有列
  → 相邻线程之间间隔一整行（低效，不能合并）
```

## 三维线程组织

处理体积数据（如医学CT、流体模拟）时使用3D。注意3D的block大小限制更严格。

```cpp
// 3D Grid, 3D Block
// 线性化公式:
//   global_x = threadIdx.x + blockIdx.x * blockDim.x
//   global_y = threadIdx.y + blockIdx.y * blockDim.y
//   global_z = threadIdx.z + blockIdx.z * blockDim.z
//   3D线性索引(行优先): idx = z * dimX * dimY + y * dimX + x
//   注意: z是最外层，y是中间层，x是最内层（与内存布局一致）

__global__ void volume_process_3d(float* volume, int dimX, int dimY, int dimZ) {
    int x = threadIdx.x + blockIdx.x * blockDim.x;
    int y = threadIdx.y + blockIdx.y * blockDim.y;
    int z = threadIdx.z + blockIdx.z * blockDim.z;

    if (x < dimX && y < dimY && z < dimZ) {
        // 3D线性索引：x是连续内存方向
        int idx = z * dimX * dimY + y * dimX + x;
        volume[idx] = volume[idx] * 1.1f;
    }
}

void launch_3d(float* d_volume, int dimX, int dimY, int dimZ) {
    // 注意: blockDim.x * blockDim.y * blockDim.z <= 1024 (每Block最大线程数)
    dim3 threads(8, 8, 8);  // 512 threads/block (8*8*8=512 ≤ 1024)
    // 也可以用 (32, 8, 4) = 1024, 或 (16, 8, 8) = 1024

    dim3 blocks(
        (dimX + threads.x - 1) / threads.x,
        (dimY + threads.y - 1) / threads.y,
        (dimZ + threads.z - 1) / threads.z
    );
    volume_process_3d<<<blocks, threads>>>(d_volume, dimX, dimY, dimZ);
}
```

### 3D Block大小的限制

```
约束: blockDim.x * blockDim.y * blockDim.z ≤ 1024 (A100)

常见选择:
  (32, 32, 1)  = 1024  ← 2D图像处理
  (32, 8, 4)   = 1024  ← 3D体积
  (16, 8, 8)   = 1024  ← 3D体积，均衡
  (8, 8, 8)    = 512   ← 3D体积，安全选择
  (64, 4, 4)   = 1024  ← 3D体积

额外限制:
  blockDim.x ≤ 1024
  blockDim.y ≤ 1024
  blockDim.z ≤ 64
```

## 通用索引计算函数

将常用的索引计算封装为设备函数，提高代码复用性。

```cpp
// ========== 基础索引函数 ==========
__device__ int get_global_1d() {
    return threadIdx.x + blockIdx.x * blockDim.x;
}

__device__ int2 get_global_2d() {
    int2 idx;
    idx.x = threadIdx.x + blockIdx.x * blockDim.x;  // col
    idx.y = threadIdx.y + blockIdx.y * blockDim.y;  // row
    return idx;
}

__device__ int3 get_global_3d() {
    int3 idx;
    idx.x = threadIdx.x + blockIdx.x * blockDim.x;
    idx.y = threadIdx.y + blockIdx.y * blockDim.y;
    idx.z = threadIdx.z + blockIdx.z * blockDim.z;
    return idx;
}

// ========== 线程信息函数 ==========
__device__ int get_linear_tid() {
    return threadIdx.x + threadIdx.y * blockDim.x
         + threadIdx.z * blockDim.x * blockDim.y;
}

__device__ int get_warp_id() {
    return get_linear_tid() / 32;
}

__device__ int get_lane_id() {
    return get_linear_tid() % 32;
}

__device__ int get_global_linear_id() {
    // 全局线程的线性编号（用于一维化处理）
    int block_linear_id = blockIdx.x + blockIdx.y * gridDim.x
                        + blockIdx.z * gridDim.x * gridDim.y;
    int threads_per_block = blockDim.x * blockDim.y * blockDim.z;
    return block_linear_id * threads_per_block + get_linear_tid();
}

// ========== 展平/反展平 ==========
__device__ int flatten_2d(int row, int col, int width) {
    return row * width + col;
}

__device__ int2 unflatten_2d(int idx, int width) {
    int2 coord;
    coord.x = idx % width;  // col
    coord.y = idx / width;  // row
    return coord;
}

__device__ int flatten_3d(int x, int y, int z, int dimX, int dimY) {
    return z * dimX * dimY + y * dimX + x;
}

__device__ int3 unflatten_3d(int idx, int dimX, int dimY) {
    int3 coord;
    coord.x = idx % dimX;
    coord.y = (idx / dimX) % dimY;
    coord.z = idx / (dimX * dimY);
    return coord;
}
```

## 边界检查模式

处理数组边界有多种模式，各有适用场景。

### 模式1：简单if检查（最常用）

```cpp
// 简单直接，当N能被blockDim整除时没有开销
// 但当N不能被整除时，最后一个Block的部分线程会空闲
__global__ void pattern_if(float* data, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {  // 边界检查
        data[idx] *= 2.0f;
    }
}

// 性能影响: 最后一个Block中的空闲线程会浪费warp槽位
// 例如 N=1000, blockDim=256:
//   Block 0: 256线程全部有效
//   Block 1: 256线程全部有效
//   Block 2: 256线程全部有效
//   Block 3: 只有232线程有效，24个线程空闲（浪费一个warp中的24个槽位）
```

### 模式2：Grid-Stride Loop（推荐用于灵活性）

```cpp
// 核心思想: 每个线程处理多个元素，通过stride跳跃
// 优势: grid大小可以远小于数据量，灵活控制活跃线程数
// 适合: 大数组、需要精确控制占用率的场景
__global__ void pattern_stride(float* data, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    int stride = blockDim.x * gridDim.x;  // 所有线程一次能处理的元素总数

    // 每次循环跳跃stride个元素
    for (int i = idx; i < N; i += stride) {
        data[i] *= 2.0f;
    }
}

// 实际使用: grid可以固定为较小值
void launch_stride(float* d_data, int N) {
    // 不管N多大，grid大小固定
    // 例如N=1B, grid=4096, block=256
    // 每线程处理 1B/(4096*256) ≈ 954 个元素
    pattern_stride<<<4096, 256>>>(d_data, N);
}
```

### 模式3：完整的Grid-Stride + Boundary

```cpp
// 结合grid-stride和边界检查，提供双重安全
__global__ void pattern_full(float* data, int N) {
    int totalThreads = blockDim.x * gridDim.x;
    int idx = threadIdx.x + blockIdx.x * blockDim.x;

    // 第一个循环: 无检查的快速路径（完整stride区间）
    // 如果 idx + stride <= N，说明整个区间都是有效的
    for (; idx + totalThreads <= N; idx += totalThreads) {
        data[idx] *= 2.0f;
    }

    // 第二个循环: 有边界检查的尾部处理
    for (; idx < N; idx += totalThreads) {
        if (idx < N) {
            data[idx] *= 2.0f;
        }
    }
}
```

### 模式对比

| 模式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 简单if | N接近2的幂次 | 简单直观 | 最后一个Block有浪费 |
| Grid-Stride | 大数组/灵活控制 | grid大小与N无关 | 循环有轻微开销 |
| 完整Grid-Stride | 性能关键路径 | 消除重复边界检查 | 代码稍复杂 |
| CUB库 | 归约/排序/扫描 | 极致优化 | 引入外部依赖 |

## 共享内存中的索引计算

共享内存的索引计算直接影响bank conflict和性能。

```cpp
// 共享内存索引的基本规则:
// 共享内存被分为32个bank（4字节宽），连续地址映射到连续bank
// bank冲突: 多个线程在同一周期访问不同地址但映射到同一bank → 串行化

__global__ void shared_memory_indexing(float* data, int N) {
    // 共享内存: 一维数组，大小为blockDim.x
    __shared__ float s_data[256];

    int tid = threadIdx.x;
    int gid = threadIdx.x + blockIdx.x * blockDim.x;

    // 加载: 线程tid加载data[gid]到s_data[tid]
    // 这里没有bank conflict: 相邻线程访问相邻bank
    // Thread 0 → bank 0, Thread 1 → bank 1, ...
    if (gid < N) {
        s_data[tid] = data[gid];
    }
    __syncthreads();

    // 读取: 如果按stride访问，可能有bank conflict
    // 例如: s_data[tid * 2] → Thread 0访问bank 0, Thread 1访问bank 2, ... → 无冲突
    // 例如: s_data[tid * 8] → Thread 0访问bank 0, Thread 1访问bank 8, ... → 无冲突
    // 例如: s_data[tid * 3] → Thread 0访问bank 0, Thread 1访问bank 3,
    //        Thread 11访问bank 33%32=bank 1 → 无冲突
    // 但 s_data[tid * 32] → 所有线程都访问bank 0! → 完全串行化!

    // 优化模式: 交错访问避免bank conflict
    float val = s_data[(tid * 7) % 256];  // stride=7与32互质，无冲突
    if (gid < N) {
        data[gid] = val;
    }
}
```

### Bank Conflict详解

```
共享内存: 32个bank，每个bank宽4字节
地址 0-3   → Bank 0
地址 4-7   → Bank 1
地址 8-11  → Bank 2
...
地址 124-127 → Bank 31
地址 128-131 → Bank 0 (wrap around)

无冲突 (No Conflict):
  Thread 0 → s_data[0]  (Bank 0)
  Thread 1 → s_data[1]  (Bank 1)
  Thread 2 → s_data[2]  (Bank 2)
  ...每个线程访问不同bank → 并行

2路冲突 (2-way Conflict):
  Thread 0 → s_data[0]  (Bank 0)
  Thread 1 → s_data[32] (Bank 0)  ← 冲突!
  Thread 2 → s_data[1]  (Bank 1)
  Thread 3 → s_data[33] (Bank 1)  ← 冲突!
  → 每2个访问串行，吞吐量减半

32路冲突 (Broadcast):
  所有线程访问同一地址 → 硬件广播（不串行化，反而很快）

解决方法:
  1. Padding: __shared__ float s_data[32][33]; (每行多加1列)
  2. 改变访问模式: 使用互质的stride
  3. XOR技巧: s_data[tid ^ (tid >> 5)] 打乱访问顺序
```

## 二维矩阵转置的索引计算

矩阵转置是理解索引计算和bank conflict的经典例子。

### 朴素实现

```cpp
// 矩阵转置 - 直接实现
// 读取: 合并访问 (row方向连续) → 高效
// 写入: 跨行写入 → 非合并访问 → 低效
__global__ void matrix_transpose_naive(const float* input, float* output,
                                        int rows, int cols) {
    int in_row = threadIdx.y + blockIdx.y * blockDim.y;
    int in_col = threadIdx.x + blockIdx.x * blockDim.x;

    if (in_row < rows && in_col < cols) {
        int in_idx = in_row * cols + in_col;    // 读: 合并
        int out_idx = in_col * rows + in_row;    // 写: 非合并!
        output[out_idx] = input[in_idx];
    }
}
```

```
朴素转置的内存访问模式:
读取 input (合并访问):
  Thread(0,0) → input[0]    Thread(1,0) → input[1]    ← 相邻地址，合并
  Thread(0,1) → input[cols]  Thread(1,1) → input[cols+1]

写入 output (非合并访问):
  Thread(0,0) → output[0]        Thread(1,0) → output[rows]    ← 跨rows间隔!
  Thread(0,1) → output[1]        Thread(1,1) → output[rows+1]

这会导致: 写入时内存事务效率降低约rows/32倍
```

### 共享内存优化版本

```cpp
// 使用共享内存优化的矩阵转置
// 核心思想: 先合并读到共享内存，再转置后合并写回
// +1 padding用于避免bank conflict
__global__ void matrix_transpose_shared(const float* input, float* output,
                                         int rows, int cols) {
    // +1 避免bank conflict的关键！
    // 不加+1时: tile[threadIdx.y][threadIdx.x]
    //   转置读取 tile[threadIdx.x][threadIdx.y] 时
    //   Thread 0读 tile[0][0] → Bank 0
    //   Thread 1读 tile[1][0] → Bank 0 (冲突!)
    //   加了+1后: tile[0][0]和tile[1][0]之间间隔了33个元素
    //   tile[0][0] → Bank 0, tile[1][0] → Bank 1 (不冲突!)
    __shared__ float tile[32][32 + 1];

    int x = blockIdx.x * 32 + threadIdx.x;
    int y = blockIdx.y * 32 + threadIdx.y;

    // 第一步: 从全局内存读取到共享内存（合并访问）
    if (x < cols && y < rows) {
        tile[threadIdx.y][threadIdx.x] = input[y * cols + x];
    }
    __syncthreads();

    // 第二步: 转置坐标，写回全局内存（合并访问）
    // 注意blockIdx也交换了！
    x = blockIdx.y * 32 + threadIdx.x;
    y = blockIdx.x * 32 + threadIdx.y;

    if (x < rows && y < cols) {
        output[y * rows + x] = tile[threadIdx.x][threadIdx.y];
    }
}

void launch_transpose(float* d_input, float* d_output, int rows, int cols) {
    dim3 threads(32, 32);
    dim3 blocks(
        (cols + 31) / 32,
        (rows + 31) / 32
    );
    matrix_transpose_shared<<<blocks, threads>>>(d_input, d_output, rows, cols);
}
```

```
优化后的访问模式:
写入input → 共享内存 tile[threadIdx.y][threadIdx.x]:
  Thread(0,0) → tile[0][0], Thread(1,0) → tile[0][1]  ← 同一行，不同列，无冲突
  Thread(0,1) → tile[1][0], Thread(1,1) → tile[1][1]

从 tile[threadIdx.x][threadIdx.y] 读出并写入output:
  Thread(0,0) → tile[0][0] → Bank 0 (无padding)
  Thread(1,0) → tile[1][0] → Bank 0 (冲突!)
  加padding后: tile[1][0] 在数组中偏移 33 → Bank 1 (无冲突!)

  写output[y * rows + x]:
  Thread(0,0) → output[...] 连续写入 (合并!)
  Thread(1,0) → output[...] 相邻写入 (合并!)
```

## 图像处理中的索引计算

```cpp
// 图像处理kernel - 3x3均值滤波（带共享内存优化）
// 使用共享内存的分块(tiling)策略减少全局内存访问

#define BLOCK_SIZE 16
#define RADIUS 1  // 3x3卷积核半径

__global__ void blur_3x3_shared(const unsigned char* input,
                                 unsigned char* output,
                                 int width, int height) {
    // 共享内存块: 包含 halo 区域 (额外边界)
    // BLOCK_SIZE + 2*RADIUS = 18
    __shared__ unsigned char tile[BLOCK_SIZE + 2 * RADIUS][BLOCK_SIZE + 2 * RADIUS];

    // 全局坐标
    int gx = blockIdx.x * BLOCK_SIZE + threadIdx.x;
    int gy = blockIdx.y * BLOCK_SIZE + threadIdx.y;

    // 共享内存坐标（偏移RADIUS为halo区域留空间）
    int sx = threadIdx.x + RADIUS;
    int sy = threadIdx.y + RADIUS;

    // 加载主区域
    if (gx < width && gy < height) {
        tile[sy][sx] = input[gy * width + gx];
    }

    // 加载halo区域（边界）
    // 上边界
    if (threadIdx.y == 0 && gy > 0) {
        tile[0][sx] = input[(gy - 1) * width + gx];
        if (threadIdx.x == 0 && gx > 0)
            tile[0][0] = input[(gy - 1) * width + gx - 1];
        if (threadIdx.x == BLOCK_SIZE - 1 && gx < width - 1)
            tile[0][sx + 1] = input[(gy - 1) * width + gx + 1];
    }
    // 下边界
    if (threadIdx.y == BLOCK_SIZE - 1 && gy < height - 1) {
        tile[sy + 1][sx] = input[(gy + 1) * width + gx];
    }
    // 左边界
    if (threadIdx.x == 0 && gx > 0) {
        tile[sy][0] = input[gy * width + gx - 1];
    }
    // 右边界
    if (threadIdx.x == BLOCK_SIZE - 1 && gx < width - 1) {
        tile[sy][sx + 1] = input[gy * width + gx + 1];
    }

    __syncthreads();

    // 计算3x3均值（使用共享内存，延迟低）
    if (gx < width && gy < height) {
        int sum = 0;
        #pragma unroll
        for (int dy = -1; dy <= 1; dy++) {
            #pragma unroll
            for (int dx = -1; dx <= 1; dx++) {
                sum += tile[sy + dy][sx + dx];
            }
        }
        output[gy * width + gx] = sum / 9;
    }
}

// 启动配置
void launch_blur(const unsigned char* d_input, unsigned char* d_output,
                 int width, int height) {
    dim3 threads(BLOCK_SIZE, BLOCK_SIZE);
    dim3 blocks(
        (width + BLOCK_SIZE - 1) / BLOCK_SIZE,
        (height + BLOCK_SIZE - 1) / BLOCK_SIZE
    );
    blur_3x3_shared<<<blocks, threads>>>(d_input, d_output, width, height);
}
```

// 朴素实现: 简单if边界检查
__global__ void naive_add(float* a, const float* b, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) a[idx] += b[idx];
}

// Grid-stride实现
__global__ void stride_add(float* a, const float* b, int N) {
    for (int idx = threadIdx.x + blockIdx.x * blockDim.x;
         idx < N;
         idx += blockDim.x * gridDim.x) {
        a[idx] += b[idx];
    }
}

// 向量化的4元素一次处理
__global__ void vector_add(float4* a, const float4* b, int N4) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N4) {
        float4 va = a[idx];
        float4 vb = b[idx];
        va.x += vb.x; va.y += vb.y; va.z += vb.z; va.w += vb.w;
        a[idx] = va;
    }
}

int main() {
    const int N = 1 << 26;  // 256M
    const size_t bytes = N * sizeof(float);
    float *d_a, *d_b;
    CUDA_CHECK(cudaMalloc(&d_a, bytes));
    CUDA_CHECK(cudaMalloc(&d_b, bytes));

    cudaEvent_t start, stop;
    cudaEventCreate(&start);
    cudaEventCreate(&stop);

    // 预热
    naive_add<<<(N+255)/256, 256>>>(d_a, d_b, N);
    cudaDeviceSynchronize();

    float ms;

    // 测试1: 朴素实现
    cudaEventRecord(start);
    for (int i = 0; i < 100; i++)
        naive_add<<<(N+255)/256, 256>>>(d_a, d_b, N);
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    cudaEventElapsedTime(&ms, start, stop);
    printf("Naive:    %.3f ms, %.1f GB/s\n", ms/100,
           (2.0f * bytes) / (ms/100 * 1e6));

    // 测试2: Grid-stride (固定较小的grid)
    cudaEventRecord(start);

    cudaEventSynchronize(stop);
    cudaEventElapsedTime(&ms, start, stop);
    printf("Stride:   %.3f ms, %.1f GB/s\n", ms/100,

## 常见索引错误

### 错误1：混淆row和col

```cpp
// 错误! threadIdx.x应该是col方向
int row = threadIdx.x + blockIdx.x * blockDim.x;  // ← 反了
int col = threadIdx.y + blockIdx.y * blockDim.y;
int idx = row * cols + col;
// 后果: 相邻线程访问跨cols间隔的内存 → 非合并访问 → 性能暴跌

// 正确!
int row = threadIdx.y + blockIdx.y * blockDim.y;
int col = threadIdx.x + blockIdx.x * blockDim.x;
```

### 错误2：忘记边界检查

```cpp
// 错误! N=1000, blockDim=256 → Block 3中线程232-255越界
__global__ void dangerous(float* data, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    data[idx] *= 2.0f;  // 越界访问! 可能段错误或静默错误
}

// 正确!
__global__ void safe(float* data, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {  // 边界检查
        data[idx] *= 2.0f;
    }
}
```

### 错误3：3D索引的维度顺序错误

```cpp
// 错误! 维度顺序不匹配内存布局
int idx = x * dimY * dimZ + y * dimZ + z;  // x是外层? 不对!

// 正确! z是外层(变化最慢), x是内层(变化最快, 连续内存)
int idx = z * dimX * dimY + y * dimX + x;
```

### 错误4：Block大小不是32的倍数

```cpp
// 不好: Block 50线程 → 每个Block有2个warp(64线程)
// 有14个线程永远空闲，浪费warp槽位
__global__ void bad_block(float* data) { ... }
bad_kernel<<<100, 50>>>(...);

// 好: Block 256线程 → 8个完整warp，无浪费
good_kernel<<<100, 256>>>(...);
```

### 错误5：共享内存未初始化的halo区域

```cpp
// 错误! 边界Block的halo区域可能越界或未初始化
__shared__ float tile[18][18];
tile[threadIdx.y + 1][threadIdx.x + 1] = input[...];
// 没有加载 halo! tile[0][*] 和 tile[17][*] 未初始化!
// 当卷积访问这些位置时，结果未定义

// 正确: 显式加载所有halo区域（见上方blur_3x3_shared的完整实现）
```

## 小结

1. 一维组织最简单，`globalIdx = threadIdx.x + blockIdx.x * blockDim.x`，用于数组操作
2. 二维组织适合图像/矩阵，`threadIdx.x`应对应列方向（连续内存方向）以确保合并访问
3. 三维组织用于体积数据，注意3D Block大小的乘积不能超过1024
4. Grid-stride loop提供灵活的线程-数据映射，grid大小与数据量脱钩
5. 边界检查是必须的，否则越界访问会导致未定义行为
6. 共享内存中的padding（如`[32][32+1]`）是避免bank conflict的关键技巧
7. 矩阵转置等操作需要仔细设计读写索引，确保两端都能合并访问
8. 向量化（float4）可以显著提升带宽利用率
9. Block大小应始终是32的倍数，256通常是安全的默认值
