# 原子操作与同步

## 线程同步机制

### __syncthreads()

`__syncthreads()`是CUDA中最基本的同步原语，确保同一线程块内所有线程都执行到该点后才继续。它的硬件实现是一个**屏障（barrier）**计数器。

```
__syncthreads() 硬件执行流程:
1. 线程到达同步点 → warp状态标记为"等待barrier"
2. SM的barrier计数器+1（该warp的所有线程到达时）
3. 当所有warp都到达 → barrier计数器归零
4. 所有warp被唤醒，继续执行
5. 如果有warp永远不到达（如在条件分支内）→ 死锁!
```

```cpp
__global__ void sync_example(float* data, int N) {
    __shared__ float s_data[256];
    int tid = threadIdx.x;
    int idx = threadIdx.x + blockIdx.x * blockDim.x;

    // 第一阶段：加载到共享内存
    if (idx < N) {
        s_data[tid] = data[idx];
    }

    // 屏障：确保所有线程完成加载
    __syncthreads();

    // 第二阶段：安全读取其他线程写入的数据
    // 没有__syncthreads()的话，s_data[tid-1]可能还没被写入
    if (tid > 0 && idx < N) {
        data[idx] = s_data[tid] + s_data[tid - 1];
    }
}
```

**重要规则**：
- `__syncthreads()`只能同步同一线程块内的线程
- 不同Block之间的同步需要结束kernel或使用CUDA 9.0+的Cooperative Groups
- `__syncthreads()`放在条件分支中时要小心，**所有线程必须都到达同步点**（至少在同一warp内）
- 从CUDA 9.0起，可以使用`__syncwarp()`只同步warp内的线程

```cpp
// 错误用法：不是所有线程都到达__syncthreads__()
__global__ void wrong_sync(float* data) {
    if (threadIdx.x < 128) {
        // 只有前128个线程到达这里
        // 后128个线程直接跳过，导致barrier计数器永远等不到它们
        // 硬件行为：SM上所有warp都停止，形成死锁
        __syncthreads();  // 危险！死锁！
    }
}

// 正确用法：所有线程都能到达同步点
__global__ void correct_sync(float* data) {
    if (threadIdx.x < 128) {
        // 做一些操作
        data[threadIdx.x] *= 2.0f;
    }
    // 所有线程都能到达这里
    __syncthreads();
    // 之后所有线程继续执行
    data[threadIdx.x] += 1.0f;
}
```

__global__ void advanced_sync(float* data, int N) {

## 原子操作（Atomic Operations）

原子操作保证在多线程环境下对同一地址的读-改-写操作是不可分割的，防止竞态条件。

### 常用原子函数完整列表

```cpp
// ========== 整数原子操作 ==========
__global__ void atomic_int_examples(int* data, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx >= N) return;

    // 基本操作
    atomicAdd(&data[0], 1);         // data[0] += 1 (原子)
    atomicSub(&data[1], 1);         // data[1] -= 1
    atomicExch(&data[2], idx);      // data[2] = idx (交换)
    atomicMin(&data[3], idx);       // data[3] = min(data[3], idx)
    atomicMax(&data[4], idx);       // data[4] = max(data[4], idx)
    atomicInc(&data[5], 100);       // data[5] = (data[5] >= 100) ? 0 : data[5]+1
    atomicDec(&data[6], 100);       // data[6] = (data[6] == 0 || data[6] > 100) ? 100 : data[6]-1
    atomicAnd(&data[7], 0xFF);      // data[7] &= 0xFF
    atomicOr(&data[8], 0x01);       // data[8] |= 0x01
    atomicXor(&data[9], 0xFF);      // data[9] ^= 0xFF

    // CAS: 最通用的原子原语
    // 如果 data[10] == idx, 则 data[10] = idx+1; 返回旧值
    int old = atomicCAS(&data[10], idx, idx + 1);
}

// ========== 浮点原子操作 ==========
__global__ void atomic_float_examples(float* data, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx >= N) return;

    // FP32原子加法（计算能力2.0+原生支持）
    atomicAdd(&data[0], (float)idx);

    // FP64原子加法（计算能力6.0+原生支持，GTX 9xx / Pascal+）
    // atomicAdd((double*)&data[1], (double)idx);  // 需要double指针

    // FP16原子加法（计算能力7.0+，Volta+）
    // __half val = __float2half(1.0f);
    // atomicAdd((__half*)&data[2], val);  // 需要特定硬件支持
}
```

### 原子操作的硬件实现（CAS循环）

GPU的原子操作在硬件层面通过**内存控制器的原子单元**实现。对于不直接支持的原子操作（如FP32 min/max），需要通过CAS循环模拟。

```
atomicAdd(&global_mem, val) 的硬件流程:
┌──────────────────────────────────────────────────┐
│ 1. 线程发出原子请求 → L1缓存                      │
│ 2. L1锁定该地址（或发送到L2）                      │
│ 3. 读取旧值 old = *addr                           │
│ 4. 计算 new = old + val                           │
│ 5. CAS: 比较 *addr == old ?                       │
│    - 是: 写入 new, 返回 old                       │
│    - 否: 重新从步骤3开始（其他线程抢先修改了）      │
│ 6. 释放锁                                        │
└──────────────────────────────────────────────────┘

CAS循环伪代码（硬件自动完成）:
do {
    old = load(addr);
    new = old + val;
} while (CAS(addr, old, new) != old);
// 最终返回 old

关键: 这个过程对其他线程是"原子"的 —— 任何其他线程
要么看到完全修改前的值，要么看到完全修改后的值
```

### 共享内存原子操作的特殊性

共享内存上的原子操作由SM内部的**共享内存原子单元**处理，比全局内存原子快得多（因为它不需要经过L1/L2缓存和全局内存控制器）。

```
共享内存原子 vs 全局内存原子:

共享内存原子:
  线程 → SM共享内存原子单元 → 共享内存 (5-30周期)
  延迟: ~5-30 cycles
  带宽: 每SM 128 bytes/cycle ≈ ~100 TB/s effective

全局内存原子:
  线程 → L1缓存原子单元 → L2缓存原子单元 → HBM控制器 (200-400周期)
  延迟: ~200-400 cycles
  带宽: 受L1/L2竞争影响，通常 << 全局内存峰值带宽

性能差距: 共享内存原子通常比全局内存原子快 10-100 倍!
```

```cpp
// 浮点原子加法的CAS实现（旧GPU不支持硬件浮点原子时）
// 现代GPU(A100/H100)已原生支持，但了解原理很重要
__device__ float atomicAdd_custom(float* address, float val) {
    // 将float指针转为unsigned int指针，以便使用整数CAS
    unsigned int* address_as_ui = (unsigned int*)address;
    unsigned int old = *address_as_ui;
    unsigned int assumed;

    // CAS循环: 如果 *address == assumed, 则 *address = new_val
    // 返回旧值; 如果旧值 != assumed, 说明其他线程修改了值, 重试
    do {
        assumed = old;
        // 计算新值: old_value + val
        // __float_as_int / __int_as_float: 不改变比特位的类型转换
        float old_float = __int_as_float(assumed);
        float new_float = old_float + val;
        unsigned int new_as_ui = __float_as_int(new_float);
        // CAS: compare and swap
        old = atomicCAS(address_as_ui, assumed, new_as_ui);
        // 如果assumed == old, CAS成功, 循环结束
        // 如果assumed != old, 有其他线程抢先修改了, 用新old值重试
    } while (assumed != old);

    return __int_as_float(old);
}
```

### 使用原子操作的直方图计算

```cpp
// ========== 基本版本: 全局内存原子操作 ==========
__global__ void histogram_global(const unsigned char* image, int* hist, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        unsigned char pixel = image[idx];
        atomicAdd(&hist[pixel], 1);  // 对256个bin中的一个进行原子加
    }
}

// ========== 优化版本: 共享内存缓冲 + 最终合并 ==========
__global__ void histogram_shared(const unsigned char* image, int* hist, int N) {
    // 共享内存中的局部直方图（每个bin一个int）
    __shared__ int s_hist[256];

    int tid = threadIdx.x;

    // 初始化共享内存直方图为0
    if (tid < 256) {
        s_hist[tid] = 0;
    }
    __syncthreads();

    // 每个线程更新共享内存直方图（快10-100倍）
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    int stride = blockDim.x * gridDim.x;
    for (int i = idx; i < N; i += stride) {
        atomicAdd(&s_hist[image[i]], 1);  // 共享内存原子
    }
    __syncthreads();

    // 将共享内存直方图合并到全局内存
    // 每个bin只做一次全局内存原子操作
    if (tid < 256) {
        atomicAdd(&hist[tid], s_hist[tid]);  // 全局内存原子（只有256次）
    }
}

void compute_histogram(const unsigned char* d_image, int* d_hist, int N) {
    cudaMemset(d_hist, 0, 256 * sizeof(int));
    histogram_shared<<<1024, 256>>>(d_image, d_hist, N);
}
```

## 竞态条件（Race Condition）详解

竞态条件发生在多个线程同时读写同一内存位置且至少有一个写操作时。

### 竞态条件的可视化

```
时间线 →
线程A: [读 sum=0]          [计算 0+5=5]      [写 sum=5]
线程B:       [读 sum=0]          [计算 0+3=3]      [写 sum=3]
线程C:             [读 sum=0]          [计算 0+7=7]      [写 sum=7]

期望结果: 0 + 5 + 3 + 7 = 15
实际结果: 取决于哪个线程最后写入，可能是 5, 3, 或 7

这就是"读-改-写"不是原子操作导致的竞态条件！
```

```cpp
// 竞态条件示例 - 错误！
__global__ void race_condition(float* data, float* sum, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        // 危险! *sum += data[idx] 实际上是:
        //   temp = *sum;           // 读
        //   temp += data[idx];     // 改
        //   *sum = temp;           // 写
        // 如果其他线程在读和写之间修改了 *sum，写入的值就错了
        *sum += data[idx];  // BUG: 读-改-写不是原子的！
    }
}

// 解决方案1：使用原子操作
__global__ void fix_atomic(float* data, float* sum, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        atomicAdd(sum, data[idx]);  // 正确：原子操作
    }
}

// 解决方案2：使用共享内存归约（更高效）
// 原理: 每个Block内部先归约，最后只做一次全局内存原子操作
__global__ void fix_reduction(float* data, float* sum, int N) {
    __shared__ float s_data[256];
    int tid = threadIdx.x;
    int idx = threadIdx.x + blockIdx.x * blockDim.x;

    float val = (idx < N) ? data[idx] : 0.0f;
    s_data[tid] = val;
    __syncthreads();

    // 共享内存归约（树形归约）
    // 步骤1: s_data[0] += s_data[128], s_data[1] += s_data[129], ...
    // 步骤2: s_data[0] += s_data[64], ...
    // 步骤3: s_data[0] += s_data[32], ...
    // 最终: s_data[0] 包含整个Block的部分和
    for (int s = blockDim.x / 2; s > 0; s >>= 1) {
        if (tid < s) {
            s_data[tid] += s_data[tid + s];
        }
        __syncthreads();
    }

    // 只用一次原子操作（整个Grid中每个Block只做一次）
    if (tid == 0) {
        atomicAdd(sum, s_data[0]);
    }
}
```

### 归约+原子 vs 纯原子 的性能对比

```
N = 1M 元素, Block=256:

纯原子操作:
  总原子操作次数 = 1,000,000 次 (全部是全局内存原子)
  每次原子操作延迟 = ~200 cycles
  总竞争: 所有线程竞争同一个 *sum 地址 → 串行化!
  实际性能: ~2 GB/s (非常慢)

归约+原子:
  共享内存归约: 每Block 255次共享内存原子
  全局内存原子: 只有 (1M/256) ≈ 3906 次
  总竞争: 3906次全局原子，比100万次快得多
  实际性能: ~150 GB/s (快约75倍!)
```

## 原子操作的性能考虑

### 共享内存 vs 全局内存原子

```cpp
// 性能对比：直接原子操作 vs 共享内存缓冲
// 直接原子操作 - 高竞争，低性能
__global__ void atomic_global_only(const int* keys, int* histogram, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        atomicAdd(&histogram[keys[idx]], 1);  // 全局内存原子操作
    }
    // 问题: N个线程全部竞争256个bin
    // 如果数据分布不均，某些bin会有极高的竞争
}

// 共享内存缓冲 - 低竞争，高性能
__global__ void atomic_shared_buffer(const int* keys, int* histogram, int N) {
    __shared__ int s_hist[256];
    int tid = threadIdx.x;

    if (tid < 256) s_hist[tid] = 0;
    __syncthreads();

    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        atomicAdd(&s_hist[keys[idx]], 1);  // 共享内存原子操作（快）
    }
    __syncthreads();

    if (tid < 256) {
        atomicAdd(&histogram[tid], s_hist[tid]);  // 只做256次全局内存原子操作
    }
}
```

### 原子操作竞争的可视化

```
均匀分布数据 (每个bin竞争程度相同):
  256 bins, 1M 元素 → ~3900 次/bin
  竞争程度: 中等

倾斜分布数据 (少数bin高度竞争):
  bin 0: 500,000 次 ← 极高竞争
  bin 1: 1 次
  bin 2: 1 次
  ...
  实际性能: 取决于最竞争的bin → 可能很慢

共享内存缓冲的解决:
  每个Block有自己的256个bin
  Block内竞争: 256线程竞争256个bin → 几乎无竞争
  最终合并: 4096个Block × 256个bin → 4096次/bin 的全局原子
  相比100万次全局原子: 减少 ~250倍
```

### CAS循环在高竞争下的问题

```cpp
// 自定义原子最大值操作（CAS循环实现）
// 在高竞争下，CAS可能需要多次重试
__device__ float atomicMax_float(float* address, float val) {
    unsigned int* address_as_ui = (unsigned int*)address;
    unsigned int old = *address_as_ui;
    unsigned int assumed;

    // 最大N次重试（避免无限循环）
    int attempts = 0;
    do {
        assumed = old;
        float old_val = __int_as_float(assumed);
        if (old_val >= val) break;  // 当前值更大，无需更新
        old = atomicCAS(address_as_ui, assumed, __float_as_int(val));
        attempts++;
    } while (assumed != old && attempts < 100);

    // 如果attempts很高，说明竞争很严重
    // 考虑使用共享内存缓冲策略
    return __int_as_float(old);
}

// 原子最小值
__device__ float atomicMin_float(float* address, float val) {
    unsigned int* address_as_ui = (unsigned int*)address;
    unsigned int old = *address_as_ui;
    unsigned int assumed;

    do {
        assumed = old;
        float old_val = __int_as_float(assumed);
        if (old_val <= val) break;
        old = atomicCAS(address_as_ui, assumed, __float_as_int(val));
    } while (assumed != old);

    return __int_as_float(old);
}
```

    printf("Warp optimized: %.3f ms\n", ms / 10);

    // 验证结果
    int* h_hist = (int*)malloc(NUM_BINS * sizeof(int));
    CUDA_CHECK(cudaMemcpy(h_hist, d_hist, NUM_BINS * sizeof(int), cudaMemcpyDeviceToHost));
    int total = 0;
    for (int i = 0; i < NUM_BINS; i++) total += h_hist[i];
    printf("Verification: total = %d (expected %d) %s\n",
           total, N, total == N ? "PASS" : "FAIL");

    free(h_keys);
    free(h_hist);
    CUDA_CHECK(cudaFree(d_keys));
    CUDA_CHECK(cudaFree(d_hist));
    cudaEventDestroy(start);
    cudaEventDestroy(stop);
    return 0;
}

## Warp级同步

同一个warp内的线程天然同步执行同一指令，但需要注意内存可见性。

```cpp
// warp内的隐式同步
__global__ void warp_level_example(float* data, int N) {
    int tid = threadIdx.x;
    int warp_id = tid / 32;
    int lane_id = tid % 32;

    __shared__ float s_data[256];
    s_data[tid] = data[tid];

    // __syncthreads() 是必要的，因为共享内存写入
    // 对其他warp不一定可见（不同warp可能在不同时间执行）
    __syncthreads();

    // 同一warp内，线程天然同步执行同一条指令
    // 但不能依赖warp之间的顺序
    float val = s_data[lane_id];  // warp内广播：所有线程同时读同一地址时最快

    // warp shuffle: warp内直接交换数据，不需要共享内存
    // 例如: 获取warp内前一个线程的值
    float from_prev = __shfl_sync(0xFFFFFFFF, val, (lane_id - 1 + 32) % 32);

    // warp投票: 检查warp内所有线程的条件
    bool is_positive = (val > 0.0f);
    bool all_positive = __all_sync(0xFFFFFFFF, is_positive);  // 所有线程都满足?
    bool any_positive = __any_sync(0xFFFFFFFF, is_positive);  // 有线程满足?
    // warp投票结果可以用作后续分支的条件

    data[tid] = from_prev;
}
```

### __shfl_sync 族函数

```cpp
// warp shuffle: 在warp内直接交换寄存器数据
// 比通过共享内存传递数据快得多（不需要内存访问）

__global__ void warp_shuffle_demo(float* data, int N) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    if (tid >= N) return;

    float val = data[tid];
    int lane = threadIdx.x % 32;

    // __shfl_sync(mask, val, srcLane)
    // 从srcLane线程获取val值
    float from_0 = __shfl_sync(0xFFFFFFFF, val, 0);  // 所有线程获取lane 0的值（广播）

    // __shfl_up_sync(mask, val, delta)
    // 获取lane_id - delta 的值，越界的保持自己的值
    float from_prev = __shfl_up_sync(0xFFFFFFFF, val, 1);  // 前一个线程的值

    // __shfl_down_sync(mask, val, delta)
    // 获取lane_id + delta 的值
    float from_next = __shfl_down_sync(0xFFFFFFFF, val, 1);  // 后一个线程的值

    // __shfl_xor_sync(mask, val, laneMask)
    // 获取lane_id ^ laneMask 的值
    float paired = __shfl_xor_sync(0xFFFFFFFF, val, 1);  // 与奇偶对换

    // 实际应用: warp级归约（比共享内存归约快）
    float sum = val;
    for (int offset = 16; offset > 0; offset >>= 1) {
        sum += __shfl_down_sync(0xFFFFFFFF, sum, offset);
    }
    // sum现在包含warp内32个线程的部分和（仅lane 0有效）

    data[tid] = from_0;
}
```

## 共享内存Bank Conflict深入

共享内存被分为32个bank（每个bank 4字节宽），同一线程的同一指令如果访问同一bank的不同地址，会产生bank conflict。

```
共享内存Bank映射:
Bank 0: 地址 0, 128, 256, 384, ...
Bank 1: 地址 4, 132, 260, 388, ...
Bank 2: 地址 8, 136, 264, 392, ...
...
Bank 31: 地址 124, 252, 380, ...

无冲突:
  Thread 0 → s_data[0]  (Bank 0)
  Thread 1 → s_data[1]  (Bank 1)
  ...每个线程不同bank

N路冲突 (N-way conflict):
  Thread 0 → s_data[0]  (Bank 0)
  Thread 32 → s_data[32] (Bank 0)  ← 冲突!
  → 需要串行化成N个周期

广播 (Broadcast):
  所有线程访问同一地址 → 硬件广播，1个周期完成（不串行化）

解决bank conflict的技巧:
1. Padding: s_data[32][33] (每行多加1列)
2. 改变访问模式: 使用与32互质的stride
3. XOR技巧: 打乱访问顺序
```

```cpp
// Bank conflict示例与解决

// 冲突示例: 转置读取
__global__ void bank_conflict_example(float* data) {
    __shared__ float tile[32][32];  // 无padding

    int tid = threadIdx.x;
    // 写入: 无冲突
    tile[threadIdx.y][threadIdx.x] = data[tid];
    __syncthreads();

    // 读取(转置): 有冲突!
    // Thread(0,0)读tile[0][0] → Bank 0
    // Thread(1,0)读tile[0][1] → Bank 1
    // ... 这是正常的按列读取
    // Thread(0,0)读tile[0][0] → Bank 0
    // Thread(0,1)读tile[1][0] → Bank 0  ← 2路冲突!
    float val = tile[threadIdx.x][threadIdx.y];  // 有bank conflict
    data[tid] = val;
}

// 解决: Padding
__global__ void bank_conflict_fixed(float* data) {
    __shared__ float tile[32][32 + 1];  // +1 padding!

    int tid = threadIdx.x;
    tile[threadIdx.y][threadIdx.x] = data[tid];
    __syncthreads();

    // 转置读取: 无冲突!
    // Thread(0,0)读tile[0][0] → Bank 0
    // Thread(0,1)读tile[1][0] → 偏移 33 个float = 132 bytes
    //   132 / 4 = 33, Bank 33 % 32 = Bank 1  ← 不同bank!
    float val = tile[threadIdx.x][threadIdx.y];  // 无bank conflict
    data[tid] = val;
}
```

## Cooperative Groups同步（跨Block同步）

CUDA 9.0引入的Cooperative Groups允许Block之间的同步，但需要特殊的启动方式。

```cpp
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

// 使用Cooperative Groups的多Block归约
__global__ void cooperative_reduce(float* data, float* result, int N) {
    cg::grid_group grid = cg::this_grid();
    cg::thread_block block = cg::this_thread_block();

    __shared__ float s_data[256];
    int tid = block.thread_rank();
    int gid = tid + block.group_index().x * blockDim.x;

    // Block内归约
    float val = (gid < N) ? data[gid] : 0.0f;
    s_data[tid] = val;
    block.sync();

    for (int s = 128; s > 0; s >>= 1) {
        if (tid < s) s_data[tid] += s_data[tid + s];
        block.sync();
    }

    // Block 0收集所有Block的部分和
    // 需要grid级同步
    if (block.group_index().x == 0 && tid == 0) {
        // 从全局内存读取其他Block的结果
        // ... (这里需要巧妙的设计)
    }

    // 等待所有Block完成
    grid.sync();

    // 现在所有Block都看到彼此的结果了
    if (block.group_index().x == 0 && tid == 0) {
        float total = 0.0f;
        for (int i = 0; i < gridDim.x; i++) {
            total += data[i];  // 假设每个Block的部分和存在data[i]中
        }
        *result = total;
    }
}

void launch_cooperative_reduce(float* d_data, float* d_result, int N) {
    cudaDeviceProp prop;
    cudaGetDeviceProperties(&prop, 0);

    int blocks = prop.multiProcessorCount;  // 通常等于SM数量
    int threads = 256;

    // Cooperative launch需要特殊API
    void* args[] = {&d_data, &d_result, &N};
    cudaLaunchCooperativeKernel(
        (void*)cooperative_reduce,
        dim3(blocks), dim3(threads),
        args
    );
}
```

### Cooperative Groups的限制

```
1. 需要计算能力 6.0+ (Pascal+)
2. 需要cudaLaunchCooperativeKernel启动，不能用<<<>>>语法
3. Block数量受限制: 需要GPU支持concurrent kernel execution
4. 性能: grid级同步比kernel间同步更快，但仍有开销
5. 调试: Nsight Compute支持分析cooperative kernel
```

## CUDA vs OpenCL vs ROCm原子操作对比

| 特性 | CUDA | OpenCL | ROCm/HIP |
|------|------|--------|----------|
| 基本原子 | `atomicAdd` | `atomic_add` | `atomicAdd` (CUDA兼容) |
| 共享内存原子 | 直接支持 | 需要`local`地址空间 | 直接支持 |
| CAS | `atomicCAS` | `atomic_cmpxchg` | `atomicCAS` |
| FP64原子 | CC 6.0+ | 扩展支持 | MI系列全支持 |
| warp shuffle | `__shfl_sync` | `sub_group_shuffle` | `__shfl` (CUDA兼容) |
| Cooperative Groups | CUDA 9.0+ | 子组扩展 | 部分支持 |
| 性能 | 最佳 | 因厂商而异 | AMD上优秀 |

## 小结

1. `__syncthreads()`同步Block内线程，确保共享内存操作的可见性；放在条件分支内要格外小心
2. 原子操作（atomicAdd等）防止多线程对同一地址的竞态条件，通过硬件CAS循环实现
3. 共享内存原子操作比全局内存原子快10-100倍，是优化直方图等高竞争场景的关键
4. 归约+一次原子操作比N次原子操作更高效，是减少全局内存竞争的标准模式
5. `atomicCAS`是最通用的原子原语，可以实现任意自定义原子操作（如atomicMax for float）
6. warp shuffle（`__shfl_sync`等）可以在warp内直接交换寄存器数据，比共享内存更快
7. 共享内存bank conflict可通过padding（如`[32][33]`）解决
8. Cooperative Groups允许跨Block同步，但需要特殊启动方式和硬件支持
