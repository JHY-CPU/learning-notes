# 动态并行

## CUDA动态并行概述

CUDA动态并行（Dynamic Parallelism，简称CDP）是CUDA 5.0引入的特性，允许GPU上的kernel直接启动新的kernel，无需CPU干预。这打破了传统模型中"只有CPU能启动kernel"的限制。

### 传统模型 vs 动态并行

```
传统模型（无动态并行）:
CPU: [Launch K1] ──→ [Launch K2] ──→ [Launch K3]
      ↓                ↓                ↓
GPU: [K1 执行中  ]    [K2 执行中  ]    [K3 执行中  ]
     └──cudaDeviceSynchronize──┘
     每次CPU启动kernel都有开销: 驱动API调用 ~10-50μs
     CPU需要知道确切的并行结构

动态并行模型:
CPU: [Launch K0]
      ↓
GPU: [K0 执行中            ]
     ├──→ [K1a] [K1b]       ← K0启动子kernel
     │     └→ [K1a-1][K1a-2] ← K1a启动更深层子kernel
     └──→ [K2]              ← K0启动另一个子kernel
     子kernel可以在父kernel内部决定启动策略
     CPU只做一次启动，所有子kernel在GPU上完成
```

### 动态并行的硬件支持

动态并行需要GPU维护一个**kernel启动缓冲区（Launch Buffer）**来存储子kernel的启动请求。

```
动态并行的硬件架构:
┌──────────────────────────────────────────┐
│  GPU芯片                                  │
│  ┌─────────────┐  ┌──────────────┐       │
│  │ SM 0...N     │  │ 全局启动队列  │       │
│  │ 执行父kernel │──│ (Launch Grid │       │
│  │              │  │  Buffer)     │       │
│  │ 遇到<<<...>> │  │              │       │
│  │ 子kernel启动 │  │ 容量: ~16K   │       │
│  │              │  │ 个子kernel   │       │
│  │ 写入启动队列 │──│              │       │
│  └─────────────┘  └──────┬───────┘       │
│                          │               │
│                    ┌─────▼──────┐        │
│                    │ 子kernel   │        │
│                    │ 调度器     │        │
│                    │ (从队列中  │        │
│                    │  取出并    │        │
│                    │  分配到SM) │        │
│                    └────────────┘        │
└──────────────────────────────────────────┘

关键限制:
- 启动缓冲区大小有限（通常~16K-32K个待处理子kernel）
- 超过缓冲区容量会导致父kernel阻塞等待缓冲区腾出空间
- 子kernel需要在父kernel结束前被取出，否则死锁
```

**硬件要求**：计算能力 3.5+ (Kepler及以后)，即GTX 700系列及以上。

## 基本使用

```cpp
// dynamic_basic.cu - 动态并行入门
#include <cstdio>
#include <cuda_runtime.h>

#define CUDA_CHECK(call) do { \
    cudaError_t err = (call); \
    if (err != cudaSuccess) { \
        fprintf(stderr, "CUDA error: %s\n", cudaGetErrorString(err)); \
        exit(1); \
    } \
} while(0)

// 子kernel：处理一个数据块
__global__ void child_kernel(float* data, int offset, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        // 每个子kernel做不同的计算
        data[offset + idx] = data[offset + idx] * 2.0f + 1.0f;
    }
}

// 父kernel：在GPU上启动4个子kernel
__global__ void parent_kernel(float* data, int N) {
    int chunk_size = N / 4;

    // 在GPU上启动4个子kernel（不需要CPU干预）
    for (int i = 0; i < 4; i++) {
        int blocks = (chunk_size + 255) / 256;
        child_kernel<<<blocks, 256>>>(data, i * chunk_size, chunk_size);
    }

    // 注意：父kernel结束后，子kernel仍然可以继续执行
    // 子kernel的生命周期独立于父kernel
}

// 带同步的父kernel：等待子kernel完成后再继续
__global__ void parent_with_sync(float* data, int N) {
    int chunk_size = N / 4;

    for (int i = 0; i < 4; i++) {
        int blocks = (chunk_size + 255) / 256;
        child_kernel<<<blocks, 256>>>(data, i * chunk_size, chunk_size);
    }

    // 等待所有子kernel完成
    cudaDeviceSynchronize();

    // 以下代码在所有子kernel完成后执行
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < 10) {  // 父kernel只有1个Block 1个线程
        printf("Parent: all children completed, data[0] = %f\n", data[0]);
    }
}

int main() {
    const int N = 4096;
    const size_t bytes = N * sizeof(float);

    float* d_data;
    CUDA_CHECK(cudaMalloc(&d_data, bytes));

    // 初始化数据
    float* h_data = (float*)malloc(bytes);
    for (int i = 0; i < N; i++) h_data[i] = (float)i;
    CUDA_CHECK(cudaMemcpy(d_data, h_data, bytes, cudaMemcpyHostToDevice));

    // 测试1: 无同步
    printf("=== Test 1: No sync ===\n");
    parent_kernel<<<1, 1>>>(d_data, N);
    CUDA_CHECK(cudaDeviceSynchronize());
    CUDA_CHECK(cudaMemcpy(h_data, d_data, bytes, cudaMemcpyDeviceToHost));
    printf("data[0] = %f (expected %f)\n", h_data[0], (0.0f * 2.0f + 1.0f));
    printf("data[1024] = %f (expected %f)\n", h_data[1024], (1024.0f * 2.0f + 1.0f));

    // 测试2: 带同步
    printf("\n=== Test 2: With sync ===\n");
    CUDA_CHECK(cudaMemcpy(d_data, h_data, bytes, cudaMemcpyHostToDevice));
    parent_with_sync<<<1, 1>>>(d_data, N);
    CUDA_CHECK(cudaDeviceSynchronize());

    free(h_data);
    CUDA_CHECK(cudaFree(d_data));
    return 0;
}
```

编译运行：
```bash
nvcc -rdc=true -arch=sm_80 -o dynamic_basic dynamic_basic.cu -lcudadevrt
./dynamic_basic
# 输出:
# === Test 1: No sync ===
# data[0] = 1.000000 (expected 1.000000)
# data[1024] = 2049.000000 (expected 2049.000000)
# === Test 2: With sync ===
# Parent: all children completed, data[0] = 3.000000
```

### `-rdc=true` 和 `-lcudadevrt` 的作用

```
-rdc=true (Relocatable Device Code):
  nvcc生成可重定位的设备代码
  允许kernel内的<<<>>>调用被推迟链接
  没有这个标志，编译器会拒绝kernel内的<<<>>>语法

-lcudadevrt (Device Runtime):
  链接CUDA设备端运行时库
  包含: kernel启动、内存管理、流管理等的设备端实现
  大小: ~500KB，会增加二进制体积
```

## 父子kernel同步详解

父子kernel之间的同步是动态并行中最复杂也最重要的部分。

### 同步模型

```
父子kernel生命周期:
CPU时间线:
  [启动父kernel] ─────────────────────────── [cudaDeviceSynchronize返回]

GPU时间线:
  [父kernel开始        ]
  ├──→ [子A] [子B]       ← 父kernel中的<<<>>>创建
  │    [子A完成] [子B完成]
  │                       ← 如果父kernel先结束:
  │                          子kernel仍然运行
  │                          但不能启动新的子kernel
  [父kernel结束]
  [子A继续  ] [子B继续  ]
  [子A完成  ] [子B完成  ]
         ↑
  cudaDeviceSynchronize() 在这里返回
  (父kernel和所有子kernel都完成)
```

```cpp
// 父子同步的三种模式

// 模式1: 父kernel不等待子kernel
__global__ void parent_no_wait(float* data, int N) {
    child_kernel<<<N/256, 256>>>(data, 0, N);
    // 父kernel立即继续执行
    // 子kernel在后台运行
    // 父kernel结束时，子kernel可能还没完成
}

// 模式2: 父kernel等待子kernel完成
__global__ void parent_with_wait(float* data, int N) {
    child_kernel<<<N/256, 256>>>(data, 0, N);
    cudaDeviceSynchronize();  // 阻塞直到子kernel完成
    // 以下代码安全地使用子kernel的结果
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        data[idx] += 1.0f;  // 子kernel的结果已可用
    }
}

// 模式3: 父kernel在特定流上等待
__global__ void parent_stream_wait(float* data, int N) {
    cudaStream_t s1, s2;
    cudaStreamCreateWithFlags(&s1, cudaStreamNonBlocking);
    cudaStreamCreateWithFlags(&s2, cudaStreamNonBlocking);

    child_kernel<<<N/4/256, 256, 0, s1>>>(data, 0, N/4);
    child_kernel<<<N/4/256, 256, 0, s2>>>(data, N/4, N/4);

    // 只等待特定流上的子kernel
    cudaStreamSynchronize(s1);
    // s2上的子kernel可能还在运行

    cudaStreamDestroy(s1);
    cudaStreamDestroy(s2);
}
```

### 同步的硬件实现

```
cudaDeviceSynchronize() 在kernel内的实现:
1. 父kernel的warp进入"等待barrier"状态
2. SM的调度器检查该kernel的所有子kernel是否完成
3. 如果有未完成的子kernel → 父kernel的warp被挂起
4. SM调度器继续执行其他就绪的warp（可能是子kernel）
5. 当所有子kernel完成 → 父kernel的warp被唤醒
6. 父kernel继续执行

性能影响:
- 同步期间，父kernel的warp不占用执行资源
- 但父kernel的寄存器/共享内存仍然被占用
- 如果子kernel需要同一SM的资源 → 可能死锁!
```

## 递归动态并行

动态并行支持递归调用，适合树形结构的计算，如自适应细分、多分辨率处理。

```cpp
// 递归归约 - 利用动态并行分治
__global__ void recursive_reduce(float* data, float* result,
                                  int N, int depth) {
    // 基准情况：直接用共享内存归约
    if (N <= 256) {
        extern __shared__ float s_data[];
        int tid = threadIdx.x;

        s_data[tid] = (tid < N) ? data[tid] : 0.0f;
        __syncthreads();

        // 树形归约
        for (int s = blockDim.x / 2; s > 0; s >>= 1) {
            if (tid < s) {
                s_data[tid] += s_data[tid + s];
            }
            __syncthreads();
        }

        if (tid == 0) {
            atomicAdd(result, s_data[0]);
        }
        return;
    }

    // 递归情况：分成两半，启动子kernel
    int half = N / 2;

    // 左半部分
    recursive_reduce<<<1, 256, 256 * sizeof(float)>>>(
        data, result, half, depth + 1);

    // 右半部分
    recursive_reduce<<<1, 256, 256 * sizeof(float)>>>(
        data + half, result, half, depth + 1);

    // 等待两个子kernel完成
    cudaDeviceSynchronize();
}

void launch_recursive_reduce(float* d_data, float* d_result, int N) {
    // 启动递归归约
    recursive_reduce<<<1, 256, 256 * sizeof(float)>>>(
        d_data, d_result, N, 0);
    cudaDeviceSynchronize();
}
```

```
递归归约的执行树:
N=1024, 最终depth=2 (因为1024→512→256)
                recursive_reduce(data, 1024)
               /                            \
    recursive_reduce(data, 512)    recursive_reduce(data+512, 512)
         /           \                    /            \
  reduce(data,256) reduce(data+256,256) reduce(data+512,256) reduce(data+768,256)
  共享内存归约     共享内存归约         共享内存归约          共享内存归约
  → atomicAdd    → atomicAdd          → atomicAdd          → atomicAdd

每个叶子节点: 共享内存归约 256 个元素 → atomicAdd到结果
总共: 4 个叶子节点 × 256 元素 = 1024 ✓
```

### 递归深度限制

```
计算能力 vs 最大嵌套深度:
CC 3.5 (Kepler): 2层
CC 3.7 (Kepler): 2层
CC 5.0 (Maxwell): 24层（理论上）
CC 5.2 (Maxwell): 24层
CC 6.0 (Pascal): 24层
CC 7.0 (Volta): 24层
CC 8.0 (Ampere): 24层
CC 9.0 (Hopper): 24层

实际限制因素:
1. 启动缓冲区: ~16K-32K个待处理子kernel
2. 递归2层 × 每层2个子kernel = 4个并发子kernel（安全）
3. 递归24层 × 每层2个子kernel = 2^24 = 16M个（超出缓冲区！→ 死锁）
实际建议: 控制在3-4层以内
```

## 自适应并行

根据数据特性动态决定并行策略，这是动态并行最有价值的应用之一。

```cpp
// 根据数据大小自适应选择并行度
// 子kernel声明（需要在父kernel之前声明，或使用前向声明）
__global__ void small_reduce(const float* data, float* result, int N);
__global__ void large_reduce(const float* data, float* result, int N);

__global__ void adaptive_kernel(const float* input, float* output,
                                 const int* sizes, int num_chunks) {
    int chunk_id = threadIdx.x + blockIdx.x * blockDim.x;
    if (chunk_id >= num_chunks) return;

    // 计算当前chunk的起始位置
    int start = 0;
    for (int i = 0; i < chunk_id; i++) start += sizes[i];
    int size = sizes[chunk_id];

    // 根据数据大小选择不同的kernel
    if (size <= 32) {
        // 小数据：单线程串行处理（启动kernel的开销太大）
        float sum = 0.0f;
        for (int i = 0; i < size; i++) {
            sum += input[start + i];
        }
        output[chunk_id] = sum;
    } else if (size <= 1024) {
        // 中等数据：单Block处理
        small_reduce<<<1, 256>>>(input + start, output + chunk_id, size);
        cudaDeviceSynchronize();  // 等待子kernel完成
    } else {
        // 大数据：多Block处理
        int blocks = (size + 255) / 256;
        large_reduce<<<blocks, 256>>>(input + start, output + chunk_id, size);
        cudaDeviceSynchronize();
    }
}

__global__ void small_reduce(const float* data, float* result, int N) {
    __shared__ float s_data[256];
    int tid = threadIdx.x;
    s_data[tid] = (tid < N) ? data[tid] : 0.0f;
    __syncthreads();

    for (int s = 128; s > 0; s >>= 1) {
        if (tid < s) s_data[tid] += s_data[tid + s];
        __syncthreads();
    }
    if (tid == 0) *result = s_data[0];
}

__global__ void large_reduce(const float* data, float* result, int N) {
    __shared__ float s_data[256];
    int tid = threadIdx.x;
    int idx = threadIdx.x + blockIdx.x * blockDim.x;

    s_data[tid] = (idx < N) ? data[idx] : 0.0f;
    __syncthreads();

    for (int s = 128; s > 0; s >>= 1) {
        if (tid < s) s_data[tid] += s_data[tid + s];
        __syncthreads();
    }
    if (tid == 0) atomicAdd(result, s_data[0]);
}
```

```
自适应并行的决策树:
                   输入数据 chunk
                   /     |      \
            size≤32   32<size≤1K   size>1K
              |          |           |
         单线程串行   1个Block     多个Block
         无kernel    启动kernel    启动kernel
         开销:0       开销:小       开销:小
         适合tiny     适合small     适合large

vs 传统方法（不自适应）:
  所有数据都启动kernel → 小数据的kernel启动开销 >> 计算开销
```

## 动态并行中的内存管理

动态并行允许在kernel内部分配和释放设备内存，但需要谨慎使用。

```cpp
// 在kernel中分配和释放设备内存
__global__ void dynamic_memory_kernel(float** pointers, const int* sizes, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx >= N) return;

    // 在GPU上分配内存
    // 注意: 使用cudaMalloc而非new/malloc
    float* d_mem;
    cudaError_t err = cudaMalloc(&d_mem, sizes[idx] * sizeof(float));
    if (err != cudaSuccess) {
        pointers[idx] = nullptr;
        return;
    }

    // 使用分配的内存
    for (int i = 0; i < sizes[idx]; i++) {
        d_mem[i] = (float)(idx + i);
    }

    // 存储指针（供后续使用或在另一个kernel中释放）
    pointers[idx] = d_mem;
}

__global__ void process_dynamic_memory(float** pointers, const int* sizes, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx >= N || pointers[idx] == nullptr) return;

    // 使用之前分配的内存
    for (int i = 0; i < sizes[idx]; i++) {
        pointers[idx][i] *= 2.0f;
    }
}

__global__ void free_dynamic_memory(float** pointers, int N) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx >= N || pointers[idx] == nullptr) return;

    // 释放之前分配的内存
    cudaFree(pointers[idx]);
}

// 完整使用示例
void use_dynamic_memory() {
    const int N = 100;

    int h_sizes[N];
    float** d_pointers;
    int* d_sizes;

    // 初始化大小
    for (int i = 0; i < N; i++) {
        h_sizes[i] = (i + 1) * 10;  // 10, 20, 30, ..., 1000
    }

    cudaMalloc(&d_pointers, N * sizeof(float*));
    cudaMalloc(&d_sizes, N * sizeof(int));
    cudaMemcpy(d_sizes, h_sizes, N * sizeof(int), cudaMemcpyHostToDevice);

    // 分配内存
    dynamic_memory_kernel<<<4, 256>>>(d_pointers, d_sizes, N);
    cudaDeviceSynchronize();

    // 使用内存
    process_dynamic_memory<<<4, 256>>>(d_pointers, d_sizes, N);
    cudaDeviceSynchronize();

    // 释放内存
    free_dynamic_memory<<<4, 256>>>(d_pointers, N);
    cudaDeviceSynchronize();

    cudaFree(d_pointers);
    cudaFree(d_sizes);
}
```

### 动态内存管理的注意事项

```
1. 内存池大小:
   - GPU上的设备端malloc从固定大小的内存池分配
   - 默认池大小约8MB（可通过cudaDeviceSetLimit调整）
   - 耗尽池会导致分配失败（cudaErrorMemoryAllocation）

2. 内存泄漏:
   - kernel中分配的内存必须在某个kernel中释放
   - 如果父kernel结束但子kernel的分配未释放 → 泄漏
   - 建议: 所有分配都在同一个生命周期内释放

3. 性能:
   - 设备端cudaMalloc比主机端慢（因为要在GPU上管理堆）
   - 频繁分配/释放会导致碎片
   - 推荐: 预分配内存池，kernel内只使用不分配

4. 调整内存池大小:
   cudaDeviceSetLimit(cudaLimitMallocHeapSize, 256 * 1024 * 1024);  // 256MB
```

## 流与动态并行

在子kernel中可以创建和使用流，实现子kernel之间的并发。

```cpp
// 在子kernel中使用流
__global__ void parent_with_streams(float* data, int N) {
    cudaStream_t stream1, stream2;
    // 设备端流创建（必须使用cudaStreamNonBlocking标志）
    cudaStreamCreateWithFlags(&stream1, cudaStreamNonBlocking);
    cudaStreamCreateWithFlags(&stream2, cudaStreamNonBlocking);

    int half = N / 2;

    // 在不同流上启动子kernel（可以并发执行）
    child_kernel<<<half / 256, 256, 0, stream1>>>(data, 0, half);
    child_kernel<<<half / 256, 256, 0, stream2>>>(data, half, half);

    // 等待所有流完成
    cudaStreamSynchronize(stream1);
    cudaStreamSynchronize(stream2);

    // 清理流
    cudaStreamDestroy(stream1);
    cudaStreamDestroy(stream2);
}
```

### 流创建的限制

```
1. 设备端只能使用cudaStreamCreateWithFlags
2. 必须指定cudaStreamNonBlocking标志
   - 原因: 默认流(cudaStreamDefault)会隐式同步所有流
   - 在动态并行中这可能导致死锁
3. 每个kernel中的流数量受限制
4. 流必须在创建它的kernel内销毁
   - 不能将流指针传给子kernel
```

## 完整性能对比程序

```cpp
// dynamic_perf.cu - 动态并行 vs 传统启动的性能对比
#include <cstdio>
#include <cstdlib>
#include <cuda_runtime.h>

#define CUDA_CHECK(call) do { \
    cudaError_t err = (call); \
    if (err != cudaSuccess) { \
        fprintf(stderr, "CUDA error at %s:%d: %s\n", __FILE__, __LINE__, cudaGetErrorString(err)); \
        exit(1); \
    } \
} while(0)

// 计算kernel: 做一些工作
__global__ void compute_work(float* data, int N, int work_amount) {
    int idx = threadIdx.x + blockIdx.x * blockDim.x;
    if (idx < N) {
        float x = data[idx];
        for (int i = 0; i < work_amount; i++) {
            x = fmaf(x, 1.001f, 0.001f);
        }
        data[idx] = x;
    }
}

// 动态并行版本: 父kernel启动多个子kernel
__global__ void dynamic_parent(float* data, int N, int num_children,
                                int work_per_child) {
    int chunk = N / num_children;
    for (int i = 0; i < num_children; i++) {
        int blocks = (chunk + 255) / 256;
        compute_work<<<blocks, 256>>>(data + i * chunk, chunk, work_per_child);
    }
    cudaDeviceSynchronize();
}

int main() {
    const int N = 1 << 22;  // 4M elements
    const size_t bytes = N * sizeof(float);
    const int NUM_CHILDREN = 4;
    const int WORK_AMOUNT = 1000;

    float* d_data;
    CUDA_CHECK(cudaMalloc(&d_data, bytes));
    CUDA_CHECK(cudaMemset(d_data, 0, bytes));

    cudaEvent_t start, stop;
    cudaEventCreate(&start);
    cudaEventCreate(&stop);
    float ms;

    // 预热
    compute_work<<<N/256, 256>>>(d_data, N, WORK_AMOUNT);
    cudaDeviceSynchronize();

    // 方式1: 传统CPU启动多个kernel
    CUDA_CHECK(cudaMemset(d_data, 0, bytes));
    cudaEventRecord(start);
    for (int iter = 0; iter < 10; iter++) {
        int chunk = N / NUM_CHILDREN;
        for (int i = 0; i < NUM_CHILDREN; i++) {
            int blocks = (chunk + 255) / 256;
            compute_work<<<blocks, 256>>>(d_data + i * chunk, chunk, WORK_AMOUNT);
        }
        cudaDeviceSynchronize();
    }
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    cudaEventElapsedTime(&ms, start, stop);
    printf("Traditional (CPU launch %d kernels): %.3f ms/iter\n",
           NUM_CHILDREN, ms / 10);

    // 方式2: 动态并行
    CUDA_CHECK(cudaMemset(d_data, 0, bytes));
    cudaEventRecord(start);
    for (int iter = 0; iter < 10; iter++) {
        dynamic_parent<<<1, 1>>>(d_data, N, NUM_CHILDREN, WORK_AMOUNT);
        cudaDeviceSynchronize();
    }
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    cudaEventElapsedTime(&ms, start, stop);
    printf("Dynamic Parallelism (1 CPU launch): %.3f ms/iter\n", ms / 10);

    // 方式3: 单个大kernel（基准）
    CUDA_CHECK(cudaMemset(d_data, 0, bytes));
    cudaEventRecord(start);
    for (int iter = 0; iter < 10; iter++) {
        compute_work<<<N/256, 256>>>(d_data, N, WORK_AMOUNT);
        cudaDeviceSynchronize();
    }
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    cudaEventElapsedTime(&ms, start, stop);
    printf("Single large kernel (baseline): %.3f ms/iter\n", ms / 10);

    CUDA_CHECK(cudaFree(d_data));
    cudaEventDestroy(start);
    cudaEventDestroy(stop);
    return 0;
}
```

### 性能分析

```
动态并行的开销:
- 子kernel启动开销: ~1-5μs/子kernel (比CPU端启动~50μs快)
- 但仍然有开销: 启动缓冲区写入、调度器查询等
- 如果每个子kernel的工作量很大（如上例），开销可以忽略
- 如果每个子kernel工作量很小，开销可能超过收益

适用场景:
✓ 子kernel工作量大 (>100μs/子kernel)
✓ 并行结构在运行时才能确定
✓ 递归/自适应算法
✗ 子kernel工作量小 (<10μs) - 启动开销主导
✗ 已知的静态并行结构 - 直接CPU启动更简单
```

## 限制与注意事项详解

### 1. 嵌套深度限制

```
CC 3.5-3.7: 最大2层
  父kernel (depth 0) → 子kernel (depth 1) → 孙kernel (depth 2) ✓
  父kernel (depth 0) → 子kernel (depth 1) → 孙kernel (depth 2) → 曾孙 (depth 3) ✗

CC 5.0+: 最大24层（理论值）
  实际上受启动缓冲区大小限制
  每层2个子kernel: 2^24 = 16M个 → 远超缓冲区 → 死锁!
  建议: 实际使用不超过3-4层
```

### 2. 启动缓冲区溢出

```
问题: 如果父kernel快速启动大量子kernel，但子kernel来不及被调度执行
      → 启动缓冲区填满 → 父kernel阻塞等待 → 如果父kernel的warp也阻塞
      → 没有warp可以执行子kernel → 死锁!

解决方案:
1. 控制子kernel的启动频率
2. 使用cudaDeviceSynchronize定期清空缓冲区
3. 限制递归深度
4. 确保父kernel不启动超过缓冲区容量的子kernel
```

### 3. 错误处理

```cpp
// 动态并行的错误处理
__global__ void parent_with_error_handling(float* data, int N) {
    // 设备端不能使用cudaGetLastError
    // 子kernel的错误会在cudaDeviceSynchronize时传播到父kernel

    child_kernel<<<N/256, 256>>>(data, 0, N);
    cudaError_t err = cudaDeviceSynchronize();

    // err如果是cudaSuccess表示子kernel成功
    // 如果子kernel出错，err会包含错误代码
    if (err != cudaSuccess) {
        // 设备端无法使用printf（除非计算能力2.0+且有stdout）
        // 通常设置一个全局错误标志
        return;
    }
}
```

### 4. 与Unified Memory的兼容性

```
动态并行 + Unified Memory:
- 可以工作，但有额外限制
- 子kernel的页面迁移可能与父kernel冲突
- 建议: 在父kernel启动前预取所有需要的数据到GPU
```

## 实际应用案例：自适应八叉树细分

动态并行最适合不规则并行结构，如自适应网格细分。

```cpp
// octree_subdivide.cu - 自适应八叉树细分
// 根据数据密度动态决定是否需要细分

#define MAX_DEPTH 4
#define LEAF_SIZE 64

struct OctreeNode {
    float bounds[6];  // xmin,ymin,zmin, xmax,ymax,zmax
    int data_start;
    int data_count;
    int children[8];  // 8个子节点的索引，-1表示无子节点
    int depth;
};

__device__ bool should_subdivide(const float* data, int start, int count,
                                  float bounds[6]) {
    // 示例判断: 如果数据点密度超过阈值就细分
    if (count > LEAF_SIZE) return true;
    // 也可以检查数据的方差、梯度等
    return false;
}

__global__ void octree_process(OctreeNode* nodes, const float* data,
                                int node_idx, int max_nodes) {
    OctreeNode& node = nodes[node_idx];

    if (node.depth >= MAX_DEPTH || node.data_count <= LEAF_SIZE) {
        // 叶子节点：直接处理
        for (int i = node.data_start; i < node.data_start + node.data_count; i++) {
            // 处理数据...
        }
        return;
    }

    // 需要细分：在GPU上启动8个子kernel
    float* child_bounds[8];
    // 计算8个子节点的边界框...

    for (int c = 0; c < 8; c++) {
        int child_idx = node.children[c];
        if (child_idx >= 0 && child_idx < max_nodes) {
            // 递归处理子节点
            octree_process<<<1, 1>>>(nodes, data, child_idx, max_nodes);
        }
    }

    // 等待所有子节点处理完成
    cudaDeviceSynchronize();
}
```

|------|---------|------------|-------------------|------------|

| 递归支持 | 原生支持 | 需要多次CPU调用 | 不支持 | 预定义深度 |
| 自适应性 | 强（运行时决定） | 需要CPU参与 | 无 | 有限 |
| 嵌套深度 | 2-24层 | 无限制 | N/A | 预定义 |
| 调试难度 | 高 | 低 | 中 | 中 |
| 适用场景 | 不规则并行 | 确定并行 | 跨Block同步 | 固定流水线 |

### CUDA Graph作为替代方案

从CUDA 10.0开始，CUDA Graph提供了更高效的静态kernel启动方式。

```cpp
// CUDA Graph: 预定义kernel执行图
// 比动态并行更高效（因为图在CPU端构建一次，GPU端高效回放）
#include <cuda_runtime.h>

cudaGraph_t graph;
cudaGraphExec_t graphExec;
cudaStream_t stream;
cudaStreamCreate(&stream);

// 构建图
cudaStreamBeginCapture(stream, cudaStreamCaptureModeGlobal);
// 在流上启动kernel（被捕获到图中）
kernel_a<<<100, 256, 0, stream>>>(data, N);
kernel_b<<<200, 256, 0, stream>>>(data, N);
kernel_c<<<50, 256, 0, stream>>>(data, N);
cudaStreamEndCapture(stream, &graph);

// 实例化并执行
cudaGraphInstantiate(&graphExec, graph, nullptr, nullptr, 0);
cudaGraphLaunch(graphExec, stream);  // 比逐个启动kernel快得多
cudaStreamSynchronize(stream);

cudaGraphExecDestroy(graphExec);
cudaGraphDestroy(graph);
cudaStreamDestroy(stream);
```

### 何时选择动态并行

```
选择动态并行的条件:
1. 并行结构在运行时才能确定（数据驱动的递归细分）
2. 需要真正的递归（如八叉树、自适应网格）
3. 子任务粒度很大（>100μs），启动开销可忽略
4. 不需要极致的启动性能

选择CUDA Graph的条件:
1. 并行结构在每次执行时都相同
2. 需要极致的kernel启动性能
3. 流水线式的数据处理

选择传统CPU启动的条件:
1. 简单的数据并行，不需要自适应
2. 不需要递归
3. 调试便利性优先
```

## 小结

1. 动态并行允许GPU kernel启动子kernel，无需CPU干预，适合不规则计算、递归算法和自适应并行
2. 编译时需要`-rdc=true`和`-lcudadevrt`，硬件需要计算能力3.5+
3. 父子kernel之间可以同步（`cudaDeviceSynchronize`），但同步成本较高且有死锁风险
4. 最大嵌套深度通常为2-24层，但受启动缓冲区大小（~16K-32K）实际限制
5. 动态内存管理（设备端cudaMalloc）可用但有性能和碎片问题，建议预分配
6. 设备端流必须使用`cudaStreamNonBlocking`标志
7. 动态并行的子kernel启动开销（~1-5μs）比CPU端启动（~50μs）小，但仍有开销
8. 对于已知静态结构的并行，CUDA Graph是更高效的替代方案
9. 实际建议：控制递归深度在3-4层以内，子kernel工作量应>100μs以抵消启动开销
