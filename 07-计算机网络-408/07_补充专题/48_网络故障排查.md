# 48_网络故障排查

## 核心概念

- **网络故障排查思路**：自底向上（或自顶向下）逐层排查
- **OSI各层常见故障**：

| 层次 | 故障现象 | 排查工具 |
|------|---------|---------|
| 物理层 | 网络不通 | 查看指示灯、测线仪 |
| 数据链路层 | 间歇性不通 | 查看交换机MAC表、检查ARP |
| 网络层 | 无法到达目标 | ping、traceroute |
| 传输层 | 连接被拒绝 | netstat、telnet端口 |
| 应用层 | 应用报错 | 浏览器开发者工具、curl |

- **常用排查工具**：
  - **ping**：测试连通性和RTT
  - **traceroute/tracert**：查看路由路径
  - **nslookup/dig**：DNS解析测试
  - **netstat/ss**：查看网络连接状态
  - **ipconfig/ifconfig**：查看IP配置
  - **arp -a**：查看ARP缓存表
- **分层排查法（五步法）**：
  1. 物理层：网线是否插好？指示灯是否正常？
  2. 链路层：能否ping通网关？ARP是否正常？
  3. 网络层：IP配置是否正确？能否ping通外网？
  4. 传输层：端口是否开放？防火墙规则是否正确？
  5. 应用层：DNS能否解析？应用配置是否正确？
- **408重点**：故障排查的基本思路和工具

## 原理分析

### 典型故障场景分析

**场景1：无法访问网站**
```
排查步骤：
1. ping 网关 → 物理/链路层是否正常
2. ping 8.8.8.8 → 网络层是否正常
3. nslookup www.example.com → DNS是否正常
4. telnet www.example.com 80 → 端口是否开放
5. 浏览器检查 → 应用层是否正常
```

**场景2：能ping通但打不开网页**
- ping用ICMP，HTTP用TCP
- 可能原因：HTTP端口被防火墙封了、DNS解析失败、Web服务未启动

**场景3：间歇性丢包**
- 可能原因：网络拥塞、硬件故障、线缆质量差
- 排查：持续ping观察丢包率、检查交换机端口统计

### ping不通但traceroute能到

- 目的主机禁用了ICMP响应
- 中间路由器禁止了ICMP转发
- 防火墙阻止了ICMP

### DNS故障排查

```
1. nslookup www.example.com → 看能否解析
2. 如果不能：换DNS服务器试试（nslookup www.example.com 8.8.8.8）
3. 如果能：说明本地DNS服务器有问题
4. dig +trace www.example.com → 追踪DNS解析全过程
```

## 直观理解

- **排查口诀**："先通后用，先底后顶"
  - 先确保能通（ping网关→ping外网）
  - 再检查能用（端口→应用）
- **分层排查像医院体检**：
  - 物理层 = 量体温（基本检查）
  - 链路层 = 听心跳
  - 网络层 = 拍X光
  - 传输层 = 血液检查
  - 应用层 = 专科检查
- **ping vs telnet**：ping测试"路通不通"，telnet测试"门开不开"

## 协议关联

- **ping与ICMP**：ping使用ICMP Echo测试连通性
- **traceroute与ICMP/TTL**：用TTL超时定位每一跳
- **nslookup与DNS**：测试DNS解析
- **netstat与TCP**：查看TCP连接状态
- **408考法**：故障排查思路、工具选择、分层排查方法
