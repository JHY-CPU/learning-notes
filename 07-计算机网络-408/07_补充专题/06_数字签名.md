# 06_数字签名

## 核心概念

- **数字签名定义**：用私钥对消息摘要进行加密，证明消息来源和完整性
- **签名过程**：
  1. 对消息 $M$ 计算散列：$H = \text{Hash}(M)$
  2. 用发送方私钥 $SK$ 加密散列值：$S = D_{SK}(H)$（签名）
  3. 发送 $(M, S)$
- **验证过程**：
  1. 接收方用发送方公钥 $PK$ 解密签名：$H' = E_{PK}(S)$
  2. 对消息 $M$ 计算散列：$H'' = \text{Hash}(M)$
  3. 比较 $H'$ 和 $H''$，若相等则验证通过
- **核心性质**：
  - 不可伪造：只有私钥持有者能生成签名
  - 不可否认：签名者不能否认签过名
  - 完整性：消息被篡改则签名验证失败
- **常见算法**：RSA签名、DSA、ECDSA
- **408重点**：签名过程、验证过程、与加密的区别

## 原理分析

### 数字签名的完整流程

```
发送方：
M → Hash() → H → 用SK加密 → S
发送 (M, S)

接收方：
收到 (M, S)
S → 用PK解密 → H'
M → Hash() → H''
比较 H' == H'' ?
```

**注意**：签名时用私钥"解密"（实际上是签名操作），验证时用公钥"加密"（实际上是验签操作）。这里"加密/解密"是从数学运算角度描述的，RSA的签名和加密在数学上是逆运算。

### RSA签名的数学表达

- 签名：$S = M^d \mod n$（用私钥指数 $d$）
- 验证：$M' = S^e \mod n$（用公钥指数 $e$）
- 若 $M' = M$，则签名有效

实际上通常是对散列值签名：
$$S = [H(M)]^d \mod n$$

### 数字签名 vs 消息加密对比

| 特性 | 数字签名 | 消息加密 |
|------|---------|---------|
| 目的 | 身份认证、完整性、不可否认 | 机密性 |
| 密钥使用 | 私钥签名，公钥验签 | 公钥加密，私钥解密 |
| 速度 | 只需对散列值操作 | 需对整个消息操作 |

## 直观理解

- **日常生活类比**：数字签名就像手写签名+指纹+公证
  - 手写签名=私钥签名（只有你能签）
  - 指纹验证=散列验证（证明没被篡改）
  - 公证=CA证书（证明签名者身份）
- **关键区别记忆**：
  - 加密：公钥锁，私钥开（保护内容）
  - 签名：私钥签，公钥验（证明身份）
- **"先散列后签名"**：为什么不对整个消息签名？因为RSA慢，散列值固定长度（如256位），签名快得多

## 协议关联

- **与证书的关联**：数字证书 = CA私钥签名(用户公钥+身份信息)
- **与TLS的关联**：TLS握手时服务器用私钥签名ServerKeyExchange参数
- **与IPsec的关联**：IKE可用RSA签名进行身份认证
- **与PGP的关联**：PGP邮件可用数字签名证明发件人身份
- **408常见陷阱**：
  - "数字签名能保证机密性"——错，签名只保证认证和完整性，不加密内容
  - "签名用公钥"——错，签名用私钥
  - "数字签名和消息认证码（MAC）一样"——错，MAC用对称密钥，不提供不可否认性
- **考试方向**：签名验证过程的判断题、签名与加密的对比
