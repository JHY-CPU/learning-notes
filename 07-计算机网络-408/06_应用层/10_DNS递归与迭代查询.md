# 10_DNS递归与迭代查询

## 核心概念

- **递归查询（Recursive Query）**：被查询的DNS服务器必须返回最终结果或错误
  - 客户端只发送一次查询请求
  - 服务器承担全部查询工作
  - 服务器之间也可能使用递归查询（但不推荐）
- **迭代查询（Iterative Query）**：被查询的DNS服务器返回已知的最佳答案
  - 如果不知道答案，返回下一级服务器的地址
  - 客户端自行继续查询
  - 服务器负担较轻
- **408考试重点**：画出两种查询的流程图，分析查询次数和时间延迟

## 原理分析

### 递归查询详细流程

查询`www.abc.com`（假设所有缓存为空）：

```
主机 → 本地DNS: "www.abc.com的IP是什么？"（递归）
本地DNS → 根DNS: "www.abc.com的IP是什么？"（递归）
根DNS → .com TLD: "www.abc.com的IP是什么？"（递归）
.com TLD → abc.com权威: "www.abc.com的IP是什么？"（递归）
abc.com权威 → .com TLD: "是1.2.3.4"
.com TLD → 根DNS: "是1.2.3.4"
根DNS → 本地DNS: "是1.2.3.4"
本地DNS → 主机: "是1.2.3.4"
```

**查询次数**：本地DNS只需1次查询，但根DNS需要查询其他服务器
**问题**：根DNS和TLD服务器负担过重

### 迭代查询详细流程

```
主机 → 本地DNS: "www.abc.com的IP是什么？"（递归）
本地DNS → 根DNS: "www.abc.com的IP是什么？"（迭代）
根DNS → 本地DNS: "我不知道，但.com TLD知道，地址是xxx"
本地DNS → .com TLD: "www.abc.com的IP是什么？"（迭代）
.com TLD → 本地DNS: "我不知道，但abc.com权威知道，地址是yyy"
本地DNS → abc.com权威: "www.abc.com的IP是什么？"（迭代）
abc.com权威 → 本地DNS: "是1.2.3.4"
本地DNS → 主机: "是1.2.3.4"
```

**查询次数**：本地DNS需要3次查询
**优点**：根DNS和TLD负担轻

### 两种查询对比

| 特性 | 递归查询 | 迭代查询 |
|------|---------|---------|
| 被查询服务器责任 | 必须返回最终结果 | 可以返回其他服务器地址 |
| 查询次数 | 客户端1次，服务器多次 | 客户端多次 |
| 服务器负担 | 重 | 轻 |
| 适用场景 | 主机→本地DNS | 本地DNS→其他DNS |
| 网络流量 | 集中在DNS服务器间 | 分散 |

### 实际DNS查询（混合方式）

- 主机 → 本地DNS：**递归查询**
- 本地DNS → 根/TLD/权威DNS：**迭代查询**
- 这是实际中最常见的方式，结合了两种方式的优点

## 直观理解

**递归查询 = 找秘书**：
- 你把任务交给秘书，秘书帮你跑腿，最后把结果告诉你
- 你只管等，秘书负担大

**迭代查询 = 自己问路**：
- 你问第一个人，他说"去问第二个人"
- 你问第二个人，他说"去问第三个人"
- 你问第三个人，终于得到答案
- 你跑腿多，但每个人回答你的负担小

**记忆**：
- 递 = 传递（任务传递给服务器）
- 迭 = 反复（客户端反复查询）

## 协议关联

- **递归查询与UDP**：一次交互完成，适合UDP
- **迭代查询与多次UDP交互**：每次查询都是独立的UDP交互
- **DNS缓存与迭代查询**：缓存可以减少迭代查询的次数
- **408考点**：问"实际DNS查询中，主机到本地DNS用什么方式？"答：递归查询
- **陷阱题**：根DNS服务器也可以接受递归查询，但通常配置为只接受迭代查询
