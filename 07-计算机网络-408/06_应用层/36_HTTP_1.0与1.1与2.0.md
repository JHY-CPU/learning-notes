# 36_HTTP_1.0与1.1与2.0

## 核心概念

- **HTTP/1.0**：1996年发布，非持久连接，简单请求/响应
- **HTTP/1.1**：1997年发布，持久连接，Host头部，管道化
- **HTTP/2**：2015年发布，多路复用，头部压缩，服务器推送
- **HTTP/3**：基于QUIC（UDP），减少延迟
- **408考试重点**：HTTP/1.0与HTTP/1.1的区别，HTTP/2的新特性

## 原理分析

### HTTP/1.0

**特点**：
- 默认非持久连接
- 每个对象需要单独的TCP连接
- 不支持Host头部
- 不支持管道化

**问题**：
- 每个对象都需要TCP握手，延迟高
- 没有Host头部，不支持虚拟主机
- 每个对象需要2 RTT

### HTTP/1.1

**改进**：
- 默认持久连接（Connection: keep-alive）
- 支持Host头部（必需）：支持虚拟主机
- 支持管道化（Pipelining）：连续发送多个请求
- 支持分块传输编码（Chunked Transfer）
- 支持范围请求（Range Request）
- 增加Cache-Control、ETag等缓存控制

**问题**：
- 管道化存在队头阻塞（Head-of-Line Blocking）
- 头部冗余，每次请求都发送相同头部
- 文本格式，解析效率低

### HTTP/2

**新特性**：
- **多路复用（Multiplexing）**：一个TCP连接上并行传输多个请求/响应
- **头部压缩（HPACK）**：压缩HTTP头部，减少冗余
- **服务器推送（Server Push）**：服务器主动推送资源
- **二进制分帧**：将消息分解为更小的帧
- **优先级和依赖**：请求可以设置优先级

**解决的问题**：
- 解决队头阻塞（在HTTP层面）
- 减少头部冗余
- 支持服务器主动推送

**仍然存在的问题**：
- TCP层仍然存在队头阻塞
- 一个TCP连接的丢包会影响所有流

### HTTP/3

**新特性**：
- 基于QUIC协议（UDP）
- 解决TCP层的队头阻塞
- 连接建立更快（0-RTT或1-RTT）
- 连接迁移（IP地址变化不断开）

### 版本对比

| 特性 | HTTP/1.0 | HTTP/1.1 | HTTP/2 |
|------|----------|----------|--------|
| 连接方式 | 非持久 | 持久 | 持久 |
| Host头部 | 不支持 | 必需 | 必需 |
| 管道化 | 不支持 | 支持 | 多路复用 |
| 头部压缩 | 不支持 | 不支持 | HPACK |
| 服务器推送 | 不支持 | 不支持 | 支持 |
| 队头阻塞 | 有 | 有（管道化） | 解决（HTTP层） |

## 直观理解

**HTTP版本演进就像交通方式**：
- HTTP/1.0 = 每次出门都打车（每次建立新连接）
- HTTP/1.1 = 坐公交车（一个连接送多个请求）
- HTTP/2 = 地铁多线并行（多路复用）
- HTTP/3 = 高铁（基于QUIC，更快更稳）

**记忆技巧**：
- HTTP/1.0 = 非持久，无Host
- HTTP/1.1 = 持久，有Host，管道化
- HTTP/2 = 多路复用，头部压缩，服务器推送
- HTTP/3 = 基于QUIC（UDP）

## 协议关联

- **HTTP/1.1与TCP**：持久连接减少TCP握手次数
- **HTTP/2与TCP**：多路复用但仍受TCP层队头阻塞影响
- **HTTP/3与UDP**：基于QUIC协议，使用UDP
- **408考点**：
  - HTTP/1.0与HTTP/1.1的区别
  - HTTP/2的多路复用
  - HTTP/1.1默认持久连接
- **陷阱**：HTTP/2解决了HTTP层队头阻塞，但TCP层仍存在
