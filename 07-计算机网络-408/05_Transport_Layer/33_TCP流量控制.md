# 33_TCP流量控制

## 核心概念

- **流量控制（Flow Control）**：发送方根据接收方的接收能力控制发送速率，防止接收方缓冲区溢出
- **接收窗口（rwnd, Receive Window）**：接收方告诉发送方自己还能接收多少数据
- **窗口字段**：TCP首部中的16位窗口字段，最大值65535字节
- **流量控制是端到端的**：由接收方控制发送方

### 流量控制的基本原理

```
接收方缓冲区: [已用空间 | 可用空间]
                      ↓
              rwnd = 可用空间大小
                      ↓
              通过TCP首部窗口字段告知发送方
                      ↓
发送方: 发送数据量 ≤ rwnd
```

## 原理分析

### 滑动窗口机制

发送方维护一个**发送窗口**，窗口大小由接收方的rwnd决定：
- 窗口内的数据可以发送（不需要等待确认）
- 窗口外的数据不能发送
- 收到ACK后窗口右移

### 窗口大小约束

发送方实际可发送的数据量：

$$\text{发送窗口} = \min(\text{rwnd}, \text{cwnd})$$

其中：
- **rwnd** = 接收窗口（由接收方通告，流量控制）
- **cwnd** = 拥塞窗口（由发送方维护，拥塞控制）

### 持续计时器（Persistence Timer）

**问题**：如果接收方发送了rwnd=0的报文段（告诉发送方暂停发送），之后接收方有了空间发送rwnd>0的报文段，但该报文段丢失了。双方就会死锁。

**解决**：发送方启动**持续计时器**。当计时器超时时，发送一个**零窗口探测报文段**（1字节），接收方回复当前窗口大小。

## 直观理解

- **流量控制像水管阀门**：接收方告诉发送方"我的水缸还能装多少水"，发送方据此调整出水速度。水缸满了就关阀门（rwnd=0），有空了再开
- **零窗口探测像敲门问**：你告诉我别发了（rwnd=0），过了一会儿我敲门问"现在能发了吗？"

## 协议关联

- **与拥塞控制的区别**：流量控制是端到端的（接收方限制发送方），拥塞控制是全局的（网络限制发送方）
- **与TCP窗口管理的关系**：发送窗口 = min(rwnd, cwnd)
