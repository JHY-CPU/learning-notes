# 16_TCP序号与确认号

## 核心概念

- **序号（Sequence Number）**：32位，对TCP数据部分的每个字节从0开始编号，序号字段的值=本报文段数据部分第一个字节的编号
- **确认号（Acknowledgment Number）**：32位，期望收到对方下一个报文段数据部分第一个字节的序号
- **初始序号（ISN, Initial Sequence Number）**：不是从0开始，而是在连接建立时随机选择，防止旧连接的延迟报文段被误收
- **序号和确认号是TCP可靠传输的核心机制**

### 序号的编号规则

```
数据:   [byte0] [byte1] [byte2] [byte3] [byte4] [byte5] ...
序号:      0       1       2       3       4       5    ...
```

假设ISN=1000，发送100字节数据：
- 第一个报文段（50字节）：SEQ=1000，数据字节序号为1000~1049
- 第二个报文段（50字节）：SEQ=1050，数据字节序号为1050~1099

### 确认号的计算规则

$$\text{确认号} = \text{已正确接收的最后一个字节序号} + 1$$

- 收到SEQ=1000，数据50字节（字节1000~1049）
- 确认号 = 1049 + 1 = **1050**

## 原理分析

### 序号消耗规则

| 报文段类型 | 是否消耗序号 |
|-----------|------------|
| SYN报文段 | 消耗1个序号 |
| FIN报文段 | 消耗1个序号 |
| 数据报文段 | 消耗数据长度个序号 |
| 纯ACK报文段 | 不消耗序号 |

### 确认的含义

- 确认号 = N 表示："我已经正确收到了序号 N 之前的所有字节（即0~N-1），期望你下次从N开始发"
- 这就是TCP的**累积确认（Cumulative Acknowledgment）**

### 序号回绕（Wraparound）

- 序号是32位的，最大值为 $2^{32}-1$
- 当序号达到最大值后，下一个是0（回绕）
- 当数据传输速率很高时，序号可能在数据存活期内回绕，导致混淆
- 解决方案：使用PAWS（Protection Against Wrapped Sequence numbers）机制

## 直观理解

- **序号像书的页码**：每一页（字节）都有编号。如果寄一本撕开的书，收件人按页码排序，缺页就要求补发
- **确认号像回执**："我已经收到第1页到第50页，请从第51页开始发"

## 协议关联

- **与滑动窗口的关系**：序号和确认号构成了滑动窗口的基础
- **与超时重传的关系**：发送方根据确认号判断哪些数据需要重传
- **408常见考法**：给定ISN和数据量，计算SEQ和ACK值
