# 26_TCP可靠传输机制

## 核心概念

- **TCP可靠传输**：保证数据无差错、不丢失、不重复、按序到达
- **可靠传输的四大机制**：序号、确认（ACK）、超时重传、差错检测（校验和）
- **TCP使用累积确认**：确认号表示该序号之前所有数据都已正确收到
- **TCP的可靠传输是通过滑动窗口机制实现的**

### 可靠传输机制总结

| 机制 | 作用 |
|------|------|
| 序号 | 对字节流编号，解决乱序和重复问题 |
| 确认（ACK） | 告知发送方哪些数据已正确收到 |
| 超时重传 | 确认超时后重传丢失的数据 |
| 校验和 | 检测数据是否在传输中出错 |

## 原理分析

### 确认机制详解

**累积确认（Cumulative ACK）**：
- 确认号 = 期望收到的下一个字节的序号
- 表示该序号之前的所有字节都已正确收到
- 即使收到了多个乱序到达的报文段，也只确认连续收到的最后一个

**举例**：
- 发送方发送了字节1~100, 101~200, 201~300
- 接收方收到了1~100和201~300，但101~200丢失
- 接收方发送ACK=**101**（期望收到101开始的数据，即使201~300已到）

### 超时重传机制

- 发送方发送数据后启动**重传计时器（Retransmission Timer）**
- 如果在**RTO（Retransmission Timeout）**内没收到确认，就重传
- RTO基于**RTT（Round-Trip Time，往返时间）**动态计算

### 差错检测

- TCP首部的校验和字段（与UDP类似，使用伪首部）
- 校验和出错的数据段直接丢弃，不发送确认

## 直观理解

- **快递类比**：序号=包裹编号，确认=签收回执，超时重传=包裹丢了补发，校验和=检查包裹是否损坏
- **序号确认号像考试收卷**：老师收卷时按序号检查，发现缺了几张（丢失），就会要求补交（重传）

## 协议关联

- **与数据链路层可靠传输的区别**：数据链路层是点到点（相邻节点），TCP是端到端（源到目的进程）
- **与UDP的对比**：UDP没有这些机制，直接尽最大努力交付
- **与流量控制/拥塞控制的关系**：可靠传输是基础，流量控制和拥塞控制在此基础上限制发送速率
- **408常见考法**：可靠传输的基本原理、累积确认的含义
