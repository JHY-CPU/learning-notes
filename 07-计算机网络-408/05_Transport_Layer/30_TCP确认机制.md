# 30_TCP确认机制

## 核心概念

- **累积确认（Cumulative ACK）**：确认号表示该序号之前所有字节都已正确收到
- **TCP不提供否定确认（NAK/NACK）**：接收方不会告诉发送方"哪些数据没收到"
- **延迟确认（Delayed ACK）**：接收方可以延迟最多500ms再发送ACK，希望能捎带在数据报文段上
- **TCP确认是滑动窗口机制的基础**

### 确认号的含义

$$\text{ACK号} = \text{期望收到的下一个字节的序号}$$

- ACK=N 意味着：序号0~N-1的所有字节都已正确收到
- 发送方收到ACK=N后，知道序号0~N-1的数据已被确认

## 原理分析

### 累积确认的工作方式

```
发送方发送: [SEQ=1, 100字节] [SEQ=101, 100字节] [SEQ=201, 100字节]
接收方收到: [SEQ=1 ✓] [SEQ=201 ✓] [SEQ=101 丢失]
接收方ACK:  ACK=101（只确认连续收到的部分）
```

即使SEQ=201的报文段已到，由于SEQ=101丢失，接收方仍然只发送ACK=101。

### 延迟确认

- 收到数据后，不立即发送ACK，而是等待最多**200ms**（RFC 1122建议）或**500ms**
- 目的：①减少ACK报文段数量 ②有机会将ACK捎带在反向数据报文段上
- 如果等待期间又收到了新数据，可以一并确认

### 捎带确认（Piggybacking）

- 如果接收方恰好有数据要发给发送方，可以将ACK信息放在数据报文段的首部中
- 减少了纯ACK报文段的数量，提高效率

## 直观理解

- **累积确认像收快递**：快递员送了5个包裹，你确认收到1~3号，他不会单独问你4号收到了没——你回复"收到3号"就暗示4号没收到
- **延迟确认像等一下再回复**：收到消息后等一下，看有没有其他消息一起回，省得一个个回复

## 协议关联

- **与快速重传的关系**：收到3个重复ACK意味着某个报文段丢失，触发快速重传
- **与滑动窗口的关系**：确认号推动发送窗口的左边界右移
- **与选择确认（SACK）的关系**：SACK在TCP选项中提供，可以确认非连续的数据块
- **408常见考法**：累积确认的含义、确认号的计算
