# 20_TCP三次握手

## 核心概念

- **三次握手（Three-Way Handshake）**：TCP连接建立的过程，也称为**三报文握手**
- **目的**：同步双方的初始序号，确认双方的收发能力
- **三次握手过程中SYN和ACK标志位的变化**

### 三次握手详细过程

```
        客户端                              服务器
          |                                   |
          |  ① SYN=1, SEQ=x                  |
          |  ───────────────────────────────→  |
          |                                   |  服务器状态: SYN_RCVD
          |  ② SYN=1, ACK=1, SEQ=y, ACK=x+1  |
          |  ←───────────────────────────────  |
          |                                   |
          |  ③ ACK=1, SEQ=x+1, ACK=y+1       |
          |  ───────────────────────────────→  |
          |                                   |  双方状态: ESTABLISHED
```

**第①次握手（客户端→服务器）**：
- SYN=1, ACK=0
- SEQ=客户端ISN（假设为x）
- 客户端状态：SYN_SENT
- 服务器收到后状态：SYN_RCVD

**第②次握手（服务器→客户端）**：
- SYN=1, ACK=1
- SEQ=服务器ISN（假设为y）
- ACK=x+1（确认客户端的SYN）
- 服务器状态：SYN_RCVD

**第③次握手（客户端→服务器）**：
- SYN=0, ACK=1
- SEQ=x+1
- ACK=y+1（确认服务器的SYN）
- 双方状态都变为：ESTABLISHED

## 原理分析

### 三次握手各步骤消耗的序号

| 步骤 | 发送方 | 报文段类型 | SEQ | 消耗序号 |
|------|--------|-----------|-----|---------|
| ① | 客户端 | SYN | x | 消耗1（x→x+1） |
| ② | 服务器 | SYN+ACK | y | 消耗1（y→y+1） |
| ③ | 客户端 | ACK | x+1 | 不消耗（纯ACK） |

### 三次握手能防止什么

**两次握手的隐患**：假设一个旧的SYN报文段延迟到达服务器：
1. 旧SYN到达服务器
2. 服务器回复SYN+ACK，进入SYN_RCVD状态，分配资源
3. 客户端没有发起连接，不会回复ACK
4. 服务器资源被浪费，等待超时才能释放

**三次握手解决**：
1. 旧SYN到达服务器
2. 服务器回复SYN+ACK
3. 客户端不发送ACK（它没有发起连接）
4. 服务器收不到ACK，超时后释放资源，不会建立连接

## 直观理解

- **打电话类比**：
  - 第一次："喂，你在吗？"（确认对方能接收）
  - 第二次："我在，你还在吗？"（确认对方能接收+自己能接收）
  - 第三次："我在"（确认自己能接收）
  - 为什么不能两次？因为第一次和第二次只确认了A能发、B能收和发，还需要第三次确认B的发送A能收到

## 协议关联

- **与SYN Flood攻击**：攻击者发送大量SYN但不完成第三次握手，导致服务器半连接队列溢出
- **与TCP状态转换**：三次握手涉及CLOSED→SYN_SENT→ESTABLISHED（客户端）和CLOSED→LISTEN→SYN_RCVD→ESTABLISHED（服务器）
- **408常见考法**：
  - 三次握手过程中各报文段的标志位和序号
  - 为什么不能两次握手
  - SYN消耗1个序号
