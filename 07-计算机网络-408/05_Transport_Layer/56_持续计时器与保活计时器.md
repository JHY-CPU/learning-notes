# 56_持续计时器与保活计时器

## 核心概念

- **持续计时器（Persistence Timer）**：解决零窗口死锁问题
- **保活计时器（Keepalive Timer）**：检测半开连接，防止资源浪费
- **两个计时器都是为了处理异常情况**

### 持续计时器

**问题场景**：
1. 接收方发送rwnd=0（"别发了"）
2. 接收方有空间了，发送rwnd>0的通知
3. 通知报文段**丢失**
4. 发送方等待（以为rwnd还是0），接收方也等待（以为发送方收到了通知）
5. **死锁**

**解决方案**：
1. 发送方收到rwnd=0时，启动持续计时器
2. 超时后发送**零窗口探测报文段**（1字节数据）
3. 接收方回复当前rwnd
4. 如果rwnd仍为0，重置计时器，继续探测

### 保活计时器

**问题场景**：
- 客户端突然崩溃或网络断开
- 服务器不知道，一直维护连接，浪费资源

**解决方案**：
1. 连接空闲**2小时**后，服务器发送保活探测报文段
2. 如果75秒内没有响应，发送第二个探测
3. 如果再75秒内没有响应，发送第三个探测
4. 如果10个探测都没有响应，关闭连接

## 原理分析

### 持续计时器的细节

- 探测间隔通常为5~60秒
- 探测报文段只有1字节数据
- 接收方必须确认探测报文段
- 探测报文段不消耗序号（有的实现消耗）

### 保活计时器的细节

- 保活功能是**可选的**，不是TCP标准
- 默认关闭，需要通过socket选项开启
- 检测到连接失效后，发送RST报文段关闭连接
- 主要用于服务器端检测客户端是否还在线

## 直观理解

- **持续计时器像定时敲门**：你说"别敲门了"（rwnd=0），我隔段时间就轻轻敲一下问"现在可以敲门了吗？"（零窗口探测）
- **保活计时器像定期体检**：长时间没联系就发个消息确认对方还在，连续多次没回应就认为关系断了

## 协议关联

- **与流量控制**：持续计时器解决rwnd=0的死锁
- **与连接管理**：保活计时器辅助检测连接状态
- **与TIME_WAIT**：TIME_WAIT是时间等待计时器，不是保活计时器
