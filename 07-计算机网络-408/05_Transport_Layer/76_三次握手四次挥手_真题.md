# 76_三次握手四次挥手_真题

## 核心概念

- **408对握手挥手的考查集中在**：序号确认号计算、状态转换、标志位

### 408真题精选

**题目**：设TCP双方的初始序号分别为200和500。三次握手完成后，客户端发送一个数据报文段，其中SEQ=？

**解析**：
- 第①步：SYN=1, SEQ=200（消耗1个序号）
- 第③步：ACK=1, SEQ=201（第三次握手）
- 第一个数据报文段：SEQ=**201**

**题目**：TCP连接建立过程中，第二个报文段（SYN+ACK）消耗几个序号？

**解析**：**1个**。SYN消耗1个序号，ACK不消耗序号。

**题目**：客户端发送FIN后进入FIN_WAIT_1状态。此时：
1. 如果先收到ACK → 进入FIN_WAIT_2
2. 如果先收到FIN → 进入CLOSING
3. 如果同时收到FIN+ACK → 进入TIME_WAIT

请分别说明每种情况的后续状态转换。

**解析**：
1. FIN_WAIT_1 → 收到ACK → **FIN_WAIT_2** → 收到FIN → **TIME_WAIT** → 2MSL → CLOSED
2. FIN_WAIT_1 → 收到FIN → **CLOSING** → 收到ACK → **TIME_WAIT** → 2MSL → CLOSED
3. FIN_WAIT_1 → 收到FIN+ACK → **TIME_WAIT** → 2MSL → CLOSED

**题目**：TCP连接的客户端在TIME_WAIT状态等待2MSL的原因是什么？

**解析**：
1. 确保最后的ACK能到达服务器。如果ACK丢失，服务器会重传FIN，客户端在TIME_WAIT期间可以重新发送ACK
2. 确保本连接的所有报文段从网络中消失，防止被新连接误收

**题目**：如果四次挥手的第4个报文段（最后的ACK）丢失，会发生什么？

**解析**：
- 服务器收不到ACK，停留在LAST_ACK状态
- 服务器超时后重传FIN
- 客户端在TIME_WAIT期间收到重传的FIN，重新发送ACK
- 如果TIME_WAIT结束还没收到重传FIN，服务器最终会超时释放连接

## 直观理解

- **做题技巧**：先确定是客户端还是服务器，再一步步推状态变化

## 协议关联

- **与状态转换图**：握手挥手是状态转换的主体
- **408常见陷阱**：SYN和FIN各消耗1序号、TIME_WAIT是客户端状态
