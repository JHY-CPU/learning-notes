# 18_TCP序号确认号_概念_真题

## 核心概念

- **序号是针对字节流的**：每个字节一个序号，报文段的SEQ=数据部分第一个字节的序号
- **确认号是累积确认**：确认号之前的所有字节都已正确收到
- **序号长度32位**：$2^{32} \approx 4GB$，序号空间用完后回绕到0

### 408真题精选

**题目**：主机甲和乙建立TCP连接，甲的初始序号为100，乙的初始序号为200。甲向乙发送3个报文段，包含数据长度分别为100、200、300。乙正确收到后发送确认，确认号分别为多少？

**解析**：
- 收到第1段（SEQ=101, 100字节）：ACK = 101 + 100 = **201**
- 收到第2段（SEQ=201, 200字节）：ACK = 201 + 200 = **401**
- 收到第3段（SEQ=401, 300字节）：ACK = 401 + 300 = **701**

注意：第一个报文段的SEQ不是100，因为SYN消耗了1个序号，所以三次握手完成后，第一个数据报文段的SEQ=101。

**题目**：TCP报文段的序号为300，确认号为500，数据部分长度为100字节。则接收方期望下一个报文段的序号是多少？

**解析**：当前报文段SEQ=300，数据长度100字节，占用了序号300~399。接收方期望下一个报文段的SEQ = **400**。

**题目**：TCP连接双方的ISN分别为1000和5000。三次握手后，客户端发送一个500字节的数据段，服务器确认。问服务器发送的ACK号是多少？

**解析**：
- 三次握手第3步：客户端发送ACK=1, SEQ=1001, ACK=5001
- 客户端发送数据500字节：SEQ=1001, 数据占用1001~1500
- 服务器ACK：ACK = 1500 + 1 = **1501**

## 直观理解

- **序号确认号公式**：ACK = 已收到的最后一个字节序号 + 1 = 期望下一个字节的序号

## 协议关联

- **与滑动窗口**：序号和确认号定义了窗口的边界
- **与拥塞控制**：拥塞窗口决定了能发送多少字节，序号决定了从哪开始发
- **408常见陷阱**：
  - "SYN不消耗序号"——错误，SYN消耗1个序号
  - "确认号表示已收到的最后一个字节"——错误，是期望下一个
  - "序号从0开始"——不准确，ISN是随机的
