# 31_TCP冗余ACK与快速重传

## 核心概念

- **冗余ACK（Duplicate ACK）**：接收方对同一确认号多次发送ACK
- **快速重传（Fast Retransmit）**：发送方收到**3个连续的冗余ACK**后，不等超时就立即重传丢失的报文段
- **快速重传比超时重传更快**，因为不需要等待RTO超时
- **3个冗余ACK是408标准答案**（RFC 5681规定）

### 快速重传的触发条件

```
发送: [1-100] [101-200] [201-300] [301-400] [401-500]
收到: [1-100 ✓] [101-200 ✗丢失] [201-300 到达] [301-400 到达] [401-500 到达]
ACK:  ACK=101（第1个冗余ACK）
      ACK=101（第2个冗余ACK）
      ACK=101（第3个冗余ACK）→ 触发快速重传！
      发送方立即重传 [101-200]
```

## 原理分析

### 冗余ACK产生的原因

接收方收到乱序到达的报文段时：
1. 发送对最后一个有序字节的确认（累积确认）
2. 每收到一个乱序的报文段，就发送一个冗余ACK
3. 例如：收到201-300（期望101-200），发送ACK=101；再收到301-400，又发送ACK=101

### 快速重传 vs 超时重传

| 特征 | 超时重传 | 快速重传 |
|------|---------|---------|
| 触发条件 | RTO超时 | 3个冗余ACK |
| 速度 | 慢（等待RTO） | 快（不需要等待超时） |
| 拥塞响应 | 慢开始（cwnd=1） | 快恢复（cwnd=ssthresh+3） |
| 信号强度 | 强（可能严重拥塞） | 弱（只是个别丢包） |

### 快速重传后的处理

收到3个冗余ACK后：
1. **立即重传**丢失的报文段
2. 将 **ssthresh = cwnd / 2**
3. 将 **cwnd = ssthresh + 3**（进入快恢复）
4. 收到新的ACK后，将 **cwnd = ssthresh**（进入拥塞避免）

## 直观理解

- **冗余ACK像反复敲门**：你敲了一次门没人开（丢包），再敲（冗余ACK），敲了3次还没开，你决定破门而入（快速重传），而不是傻等（超时重传）

## 协议关联

- **与拥塞控制**：快速重传后进入快恢复阶段，而不是从慢开始
- **与SACK**：选择确认可以让发送方知道具体哪些数据丢失，避免不必要的重传
- **408常见考法**：3个冗余ACK触发快速重传、快速重传后的cwnd变化
