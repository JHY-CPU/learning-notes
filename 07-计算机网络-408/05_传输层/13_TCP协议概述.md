# 13_TCP协议概述

## 核心概念

- **TCP（Transmission Control Protocol，传输控制协议）**：RFC 793定义，提供面向连接的、可靠的、基于字节流的传输层通信
- **TCP连接特点**：点对点（一对一）、全双工、可靠有序、面向字节流
- **TCP数据单元**：称为**报文段（Segment）**
- **TCP关键机制**：三次握手建立连接、四次挥手释放连接、滑动窗口、超时重传、拥塞控制

### TCP与UDP核心对比

| 特征 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 通信方式 | 点对点 | 一对一/一对多/多对多 |
| 可靠性 | 可靠传输 | 不可靠 |
| 数据形式 | 字节流 | 报文 |
| 首部 | 最少20字节 | 固定8字节 |
| 流量控制 | 有 | 无 |
| 拥塞控制 | 有 | 无 |

## 原理分析

### TCP提供的服务特性

1. **面向连接**：通信前必须建立连接（三次握手），通信后必须释放连接（四次挥手）
2. **可靠传输**：通过序号、确认、重传等机制保证数据无差错、不丢失、不重复、按序到达
3. **面向字节流**：TCP把应用层数据看作无结构的字节流，不保留报文边界
4. **全双工**：连接双方可以同时发送和接收数据
5. **点对点**：一个TCP连接只有两个端点（不支持广播/组播）

### TCP的"字节流"含义

```
应用层写入: "Hello" + "World"
TCP发送:   可能拆分为 "Hel" + "loW" + "orld"
接收方:    可能收到 "HelloW" + "orld"
```

- TCP不保证应用层的写入边界
- 应用层需要自己处理消息边界（如HTTP的Content-Length）

## 直观理解

- **TCP像打电话**：先拨号（握手）→ 通话（数据传输）→ 挂断（挥手）。双方有确认机制，听不清会说"请再说一遍"
- **TCP像送挂号信**：建立连接=联系收件人，可靠传输=签收确认，释放连接=确认收货

## 协议关联

- **与网络层IP的关系**：TCP运行在IP之上。IP不可靠，TCP通过自身机制实现了可靠传输
- **与应用层的关系**：HTTP、HTTPS、FTP、SMTP、Telnet等都使用TCP
- **408重点**：TCP的所有机制（握手、挥手、滑动窗口、拥塞控制）都是408必考内容
