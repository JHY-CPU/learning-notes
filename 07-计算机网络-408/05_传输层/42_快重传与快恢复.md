# 42_快重传与快恢复

## 核心概念

- **快重传（Fast Retransmit）**：收到3个冗余ACK后，立即重传丢失的报文段，不等待超时
- **快恢复（Fast Recovery）**：快重传后，不从慢开始开始，而是将cwnd减半后继续拥塞避免
- **快重传和快恢复合称为TCP Reno**（相对于TCP Tahoe的慢开始策略）
- **3个冗余ACK意味着网络还能传输数据**（因为ACK还在到达），所以不需要cwnd=1

### 快重传 + 快恢复的处理流程

```
收到3个冗余ACK（ACK=N）:
  1. ssthresh = cwnd / 2
  2. 重传丢失的报文段（序号N开始的数据）
  3. cwnd = ssthresh + 3 （收到3个冗余ACK，说明有3个报文段已离开网络）
  4. 进入拥塞避免阶段
收到新的ACK（确认了重传的数据）:
  5. cwnd = ssthresh （确认恢复完成，进入正常的拥塞避免）
```

## 原理分析

### 为什么cwnd = ssthresh + 3？

收到3个冗余ACK意味着：
- 接收方已经收到了3个乱序的报文段
- 这3个报文段已经离开了网络
- 网络中减少了3个报文段的负载
- 所以cwnd可以增大3来"补偿"

### 快恢复 vs 超时处理

| 处理方式 | 超时 | 3个冗余ACK |
|----------|------|-----------|
| ssthresh | cwnd/2 | cwnd/2 |
| cwnd | 1 | ssthresh + 3 |
| 阶段 | 慢开始 | 拥塞避免 |
| 严重程度 | 严重拥塞 | 轻微拥塞 |

### TCP Tahoe vs TCP Reno

| 版本 | 超时处理 | 3冗余ACK处理 |
|------|---------|-------------|
| TCP Tahoe | cwnd=1, 慢开始 | cwnd=1, 慢开始 |
| TCP Reno | cwnd=1, 慢开始 | cwnd=ssthresh+3, 拥塞避免 |

Reno对冗余ACK的处理更温和，避免了不必要的慢开始。

## 直观理解

- **快重传像及时补救**：发现丢了1件快递（3个冗余ACK），立刻补发，同时发货量减半（cwnd减半），而不是停下来重新从1件开始发货
- **快恢复的逻辑**：既然ACK还在来，说明网络没完全堵死，不必cwnd=1那么极端

## 协议关联

- **与慢开始/拥塞避免**：快恢复后直接进入拥塞避免，跳过了慢开始
- **与SACK**：SACK可以让发送方精确知道丢失了哪些数据，配合快重传更高效
- **408必考**：超时和冗余ACK的不同处理是高频考点
