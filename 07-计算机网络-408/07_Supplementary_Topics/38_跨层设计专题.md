# 38_跨层设计专题

## 核心概念

- **跨层设计定义**：打破传统分层架构，允许不同层之间直接交互和共享信息
- **传统分层的局限**：每层独立优化，不考虑其他层的状态，可能导致全局性能不佳
- **常见跨层交互**：
  - **物理层→应用层**：信道质量信息（无线网络中选择合适的编码率）
  - **网络层→传输层**：ECN（显式拥塞通知）→TCP调整cwnd
  - **链路层→传输层**：无线链路误码导致丢包→TCP不应误判为拥塞
  - **应用层→传输层**：视频应用根据带宽调整码率
- **经典跨层案例**：
  - TCP拥塞控制：IP层ECN → TCP层降速
  - 无线TCP优化：链路层ARQ → 传输层不触发拥塞控制
  - 移动IP：网络层地址变化 → 传输层连接保持
- **408重点**：理解层间交互关系，特别是TCP与IP的协作

## 原理分析

### TCP与IP的跨层协作

**ICMP通知TCP**：
- ICMP目的不可达（Port Unreachable）→ TCP连接关闭
- ICMP源抑制（Source Quench）→ TCP降低发送速率
- ICMP超时（Time Exceeded）→ TCP重传

**ECN（显式拥塞通知）**：
1. 路由器检测到即将拥塞，在IP头设置ECN标志
2. 接收方在TCP ACK中回传ECN-Echo
3. 发送方收到后减少cwnd（类似收到拥塞信号）

$$cwnd_{new} = \frac{cwnd_{old}}{2}$$

### ARP与IP的协作

```
IP层：发送IP包到下一跳IP地址
  ↓
ARP：查找下一跳IP对应的MAC地址
  ↓
如果找到：封装到以太网帧中发送
如果没找到：发送ARP请求，等待ARP响应后发送
```

### DNS与应用层/传输层的协作

```
应用层（浏览器）：需要访问 www.example.com
  ↓
DNS解析：应用层→UDP→IP→...→DNS服务器→返回IP地址
  ↓
应用层用解析到的IP地址发起TCP连接
```

### 无线网络中的跨层优化

问题：无线链路丢包可能是信道错误（不是拥塞），但TCP会误判为拥塞

解决方案：
- 链路层ARQ重传丢失的帧
- 链路层通知TCP"这是信道错误不是拥塞"
- TCP不做拥塞控制调整

## 直观理解

- **分层 vs 跨层**：
  - 传统分层：像严格的公司层级，员工不能越级汇报
  - 跨层设计：像扁平化管理，部门之间直接沟通
- **ECN类比**：高速公路上的"前方拥堵"标志牌，看到后提前减速
- **无线TCP问题**：像快递在途中丢了（信道错误），不应该认为是目的地爆仓（拥塞）

## 协议关联

- **TCP与IP**：ECN、ICMP通知
- **ARP与IP**：地址解析
- **DNS与应用/传输**：域名解析
- **链路层与传输层**：无线网络中丢包原因区分
- **408考法**：跨层协作场景分析、ECN机制理解
