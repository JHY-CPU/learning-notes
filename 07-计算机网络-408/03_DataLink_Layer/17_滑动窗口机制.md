# 17_滑动窗口机制

## 核心概念

- **滑动窗口（Sliding Window）**：允许发送方在收到ACK之前连续发送多个帧，提高信道利用率
- **核心思想**：维护一个"窗口"，窗口内的帧可以发送或接收，窗口随确认的到达而滑动
- **两个窗口**：发送窗口 $W_s$ 和接收窗口 $W_r$

### 发送窗口

- 窗口内的帧可以连续发送，无需等待确认
- 收到ACK后，窗口向前滑动，新的帧进入窗口
- 窗口大小限制了"已发送但未确认"帧的最大数量

### 接收窗口

- 只有落在窗口内的帧才会被接收
- 窗口外的帧会被丢弃（GBN）或缓存（SR）
- 收到正确的帧后，窗口向前滑动

### 窗口滑动示例（序号空间4位，0-3）

```
时刻1: 发送窗口=[0,1,2,3]
       可以发送帧0,1,2,3

时刻2: 收到ACK0
       发送窗口=[1,2,3,0]  (4号等价于0，但实际是下一个周期的0)
       帧4(即新0)可以发送
```

### 序号空间与窗口约束

- 序号用 $n$ 位二进制表示，序号空间大小 = $2^n$
- 序号空间必须**足够大**，使得新帧的序号不会与未确认的老帧序号混淆

### 考试重点

- 窗口大小与序号位数的关系：**GBN: $W_s \leq 2^n - 1$，SR: $W_s + W_r \leq 2^n$**
- 滑动窗口能同时实现**流量控制**和**可靠传输**
- 理解窗口如何"滑动"

## 原理分析

### 为什么GBN窗口 $\leq 2^n - 1$？

假设有 $n=2$ 位序号（0,1,2,3），如果 $W_s = 4$：

1. 发送方发送帧0,1,2,3
2. ACK全部丢失
3. 超时后重传帧0,1,2,3
4. 但接收方期望的序号是**新周期的0**，还是**重传的0**？无法区分！

如果 $W_s = 3$（$2^2 - 1 = 3$）：
1. 发送帧0,1,2（窗口满，最多3帧）
2. 即使ACK全部丢失，超时重传时接收方还能区分

### 为什么SR窗口 $\leq 2^{n-1}$？

假设有 $n=2$ 位序号（0,1,2,3），如果 $W_s = 3, W_r = 3$：

1. 发送方发送帧0,1,2
2. 接收方期望帧0,1,2（接收窗口=[0,1,2]）
3. 帧2的ACK丢失，发送方超时重传帧2
4. 接收窗口已滑到[3,0,1]，收到帧2会被当作**下一个周期的帧2**（实际上是新帧？）
5. 导致混淆！

如果 $W_s = W_r = 2$（$2^{n-1} = 2$）：
- 不会出现这种混淆

### 窗口大小约束总结

$$\text{GBN}: W_s \leq 2^n - 1, \quad W_r = 1$$

$$\text{SR}: W_s \leq 2^{n-1}, \quad W_r \leq 2^{n-1}, \quad W_s + W_r \leq 2^n$$

### 累积确认 vs 逐一确认

| 类型 | 说明 | 应用 |
|------|------|------|
| 累积确认 | ACKn表示n及之前所有帧都已收到 | GBN |
| 逐一确认 | 每帧单独确认 | SR |

## 直观理解

- **滑动窗口就像生产车间的流水线**：有多个工位（窗口），产品（帧）依次通过。完成一个工位的产品被移走后，新产品才能进入
- **发送窗口**就像"在传送带上的产品数量"，不能超过传送带长度
- **接收窗口**就像"检验台的容量"，超出容量的产品无法检验

## 协议关联

- 停等协议 = 窗口大小为1的特殊滑动窗口
- GBN和SR都是滑动窗口的具体实现
- TCP也使用滑动窗口机制，但面向字节而非帧
- 窗口大小的约束是408考试的**核心难点**

### 与信道利用率的关系

$$U = \frac{W_s \times T_f}{T_f + 2T_p} \quad \text{（当} W_s \times T_f \leq T_f + 2T_p \text{时）}$$

当窗口足够大时，$U = 100\%$。
