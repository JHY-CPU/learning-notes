# 21_选择重传SR协议

## 核心概念

- **SR（Selective Repeat）**：选择重传协议，发送窗口 > 1，接收窗口 > 1
- **核心思想**：只重传**真正出错或丢失**的帧，不对正确帧重复传输
- **缓存失序帧**：接收方缓存乱序到达的帧，等缺失帧到达后一起交付上层

### SR的关键参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 发送窗口 $W_s$ | $\leq 2^{n-1}$ | n为序号位数 |
| 接收窗口 $W_r$ | $\leq 2^{n-1}$ | 可以缓存失序帧 |
| 确认方式 | 逐一确认 | 每帧单独ACK |
| 超时重传 | 只重传1帧 | 只重传超时的帧 |

### SR的工作流程

```
发送方                              接收方
  │                                    │
  │ ──帧0──>                           │  期望帧0 ✓ 缓存
  │ ──帧1──>  丢失！                   │  期望帧1 ✗
  │ ──帧2──>                           │  期望帧2 ✓ 缓存
  │ ──帧3──>                           │  期望帧3 ✓ 缓存
  │                                    │
  │ <──ACK0───                         │
  │ <──ACK2───                         │
  │ <──ACK3───                         │
  │                                    │
  │  帧1超时，只重传帧1               │
  │ ──帧1──>                           │  期望帧1 ✓
  │                                    │
  │ <──ACK1───                         │  交付帧0,1,2,3
```

### SR vs GBN的核心区别

| 特性 | GBN | SR |
|------|-----|-----|
| 接收窗口 | 1 | > 1 |
| 失序帧处理 | 丢弃 | 缓存 |
| 确认方式 | 累积确认 | 逐一确认 |
| 重传范围 | 所有未确认帧 | 仅出错帧 |
| 重传代价 | 高 | 低 |

### SR的确认机制

- 使用**逐一确认**（Individual ACK）
- 每收到一个帧就发送该帧的ACK
- 不使用累积确认

### 考试重点

- SR窗口约束：$W_s \leq 2^{n-1}$（不是 $2^n - 1$）
- SR和GBN的窗口约束不同，**不能混淆**
- $W_s + W_r \leq 2^n$（序号空间约束）

## 原理分析

### 为什么SR窗口 $\leq 2^{n-1}$？

假设有 $n=3$ 位序号（0-7），如果 $W_s = W_r = 5$：

1. 发送方发送帧0,1,2,3,4
2. 全部丢失
3. 发送方超时重传帧0
4. 接收方窗口已经滑到 [5,6,7,0,1]
5. 收到帧0，会被认为是**新的帧0**（序号0出现在接收窗口内）！

如果 $W_s = W_r = 4 = 2^{n-1}$：
- 接收窗口 [0,1,2,3]，发送方重传帧0时，接收窗口不会与新序号冲突

### SR发送方伪代码

```
窗口基序号 base = 0

while(true):
    收到数据:
        if 在窗口内:
            发送帧,设置超时计时器
    
    收到ACK(n):
        标记帧n已确认
        if n == base:
            // 连续确认的帧都推进
            while base已确认:
                base++
    
    超时(n):
        重传帧n
        重启帧n的计时器
```

## 直观理解

- **SR就像精准治疗**：GBN是"全面体检"（重传所有），SR是"靶向治疗"（只治有问题的）
- **缓存失序帧**就像"收集拼图碎片"：不按顺序到的碎片先放好，等缺的那块到了就能拼完
- **逐一确认**就像"每收到一件快递就签收一件"，不像GBN那样"收到第5件就确认前5件"

## 协议关联

- SR适合**误码率高**的信道（出错多时，只重传出错帧，效率更高）
- GBN适合**误码率低**的信道（出错少，累积确认更高效）
- TCP的SACK（选择确认）借鉴了SR的思想
- SR的实现比GBN复杂（需要缓存、逐一计时器）
