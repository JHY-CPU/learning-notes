# 14_停等协议

## 核心概念

- **停等协议（Stop-and-Wait）**：发送方每发送一帧就停下来，等待接收方的确认（ACK），收到ACK后才发送下一帧
- **是最简单的ARQ协议**：发送窗口=1，接收窗口=1
- **序号**：只需1位序号（0和1交替），区分新帧和重传帧

### 停等协议的工作流程

```
发送方                         接收方
  │                              │
  │ ────── 帧0 ────────────────> │
  │                              │
  │ <────── ACK0 ────────────── │
  │                              │
  │ ────── 帧1 ────────────────> │
  │                              │
  │ <────── ACK1 ────────────── │
  │                              │
```

### 三种异常情况

| 异常 | 描述 | 解决方法 |
|------|------|----------|
| 帧丢失 | 发送的帧在路上丢了 | 超时重传 |
| ACK丢失 | 接收方收到帧但ACK丢了 | 超时重传 |
| 帧迟到 | 帧/ACK延迟到达 | 超时重传+序号检查 |

### 停等协议的确认机制

- 接收方收到正确帧后，发送**ACKn**（n为帧的序号）
- 发送方收到ACKn后，知道帧n已正确接收，可以发送帧n+1
- 序号只需1位：0和1交替使用

### 超时计时器

- 发送帧时启动**超时计时器**
- 超时时间内未收到ACK，自动重传
- 超时时间 $T_{\text{out}}$ 应大于**往返时间RTT**

### 考试重点

- 停等协议的**信道利用率**很低（特别是远距离通信）
- 停等协议是GBN和SR的**基础**
- 计算信道利用率是高频考点

## 原理分析

### 停等协议的状态机

**发送方状态机**：
```
[等待上层数据] --收到数据--> [发送帧,启动定时器] 
    ↑                              |
    |                        收到ACK? 
    |                     /          \
    |               超时/收到错误ACK   收到正确ACK
    |                  |                |
    |           [重传帧,重启定时器]  [停止定时器]
    |                  |                |
    +------------------+                +---> [等待上层数据]
```

### 停等协议的信道利用率

$$U = \frac{T_{\text{frame}}}{T_{\text{frame}} + T_{\text{ACK}} + 2T_{\text{prop}}}$$

其中：
- $T_{\text{frame}}$：发送一帧的时间 = $\frac{\text{帧长}}{\text{数据率}}$
- $T_{\text{ACK}}$：发送ACK的时间
- $T_{\text{prop}}$：单向传播时延

当 $T_{\text{ACK}} \ll T_{\text{frame}}$ 时，简化为：

$$U \approx \frac{T_{\text{frame}}}{T_{\text{frame}} + 2T_{\text{prop}}} = \frac{1}{1 + 2a}$$

其中 $a = \frac{T_{\text{prop}}}{T_{\text{frame}}}$

### 例题

**题目**：链路带宽1Mbps，链路长度200km，信号传播速度 $2 \times 10^8$ m/s，帧长1000bit，求停等协议的信道利用率。

**解答**：

- $T_{\text{frame}} = \frac{1000}{10^6} = 1$ ms
- $T_{\text{prop}} = \frac{200 \times 10^3}{2 \times 10^8} = 1$ ms
- $a = \frac{1}{1} = 1$
- $U = \frac{1}{1 + 2 \times 1} = \frac{1}{3} = 33.3\%$

## 直观理解

- 停等协议就像**对讲机**：我说完一句话，按下发送键，等对方回复"收到"，我才说下一句
- **序号交替**就像"这次发0，下次发1"：如果收到重复的序号，说明是重传帧，不是新帧
- **信道利用率低**的原因：大部分时间在等（传播时延），真正发送数据的时间很短

## 协议关联

- 停等协议是**GBN和SR的基础**
- GBN是停等协议的**推广**：允许连续发送多个帧
- 停等协议 = 窗口大小为1的特殊滑动窗口协议
- TCP的可靠传输也借鉴了停等协议的思想（面向字节的滑动窗口）

### 与GBN/SR的关系

```
停等协议：Ws=1, Wr=1 → 简单但效率低
GBN协议：Ws>1, Wr=1 → 提高效率，但出错要重传很多
SR协议： Ws>1, Wr>1 → 效率高，只重传出错的帧
```
