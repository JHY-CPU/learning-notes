# 65_PPP帧格式详解

## 核心概念

- **PPP帧**：面向字节的帧，使用0x7E作为定界符
- **帧长范围**：信息字段最大1500字节（受MTU限制）
- **透明传输**：字节填充法

### PPP帧格式

```
| 标志 | 地址 | 控制 | 协议 | 信息 | FCS | 标志 |
| 1B   | 1B   | 1B   | 2B   | ≤1500B | 2/4B | 1B  |
```

### 各字段详解

| 字段 | 长度 | 值 | 说明 |
|------|------|-----|------|
| Flag | 1字节 | 0x7E | 帧定界符 |
| Address | 1字节 | 0xFF | 广播地址（固定值） |
| Control | 1字节 | 0x03 | 无编号帧 |
| Protocol | 2字节 | 可变 | 标识信息字段类型 |
| Information | 0-1500字节 | 数据 | 上层数据 |
| FCS | 2或4字节 | CRC | 帧校验序列 |

### 常见协议字段值

| 值 | 协议 |
|-----|------|
| 0x0021 | IP数据报 |
| 0xC021 | LCP（链路控制协议） |
| 0x8021 | NCP（网络控制协议） |
| 0xC023 | PAP（密码认证协议） |
| 0xC223 | CHAP（挑战握手认证协议） |

### 字节填充规则

当信息字段中出现特殊字符时：

| 原始字节 | 填充后 |
|---------|--------|
| 0x7E（标志） | 0x7D 0x5E |
| 0x7D（转义） | 0x7D 0x5D |
| ASCII控制字符 (<0x20) | 0x7D {字符+0x20} |

### 零比特填充（另一种透明传输方式）

如果PPP使用同步传输（HDLC-like）：
- 使用零比特填充（和HDLC一样）
- 每5个连续1后插入0

### 考试重点

- PPP帧各字段的长度和含义
- 字节填充的规则
- PPP vs HDLC帧格式的区别

## 原理分析

### 字节填充示例

原始数据：`0x7E 0x03 0x7D`

填充后：`0x7D 0x5E 0x03 0x7D 0x5D`

接收方收到后，遇到0x7D就将下一字节减去0x20（或异或0x20）还原。

### FCS计算

PPP默认使用CRC-16（2字节），也可使用CRC-32（4字节）。

校验范围：地址 + 控制 + 协议 + 信息（不含标志和FCS本身）

### PPP帧与以太网帧对比

| 字段 | PPP | 以太网V2 |
|------|-----|---------|
| 定界符 | 0x7E | 前导码+SFD |
| 地址 | 1字节(0xFF) | 6字节(MAC) |
| 控制 | 1字节 | 无 |
| 协议/类型 | 2字节 | 2字节 |
| FCS | 2/4字节 | 4字节 |
| 最小帧 | 4字节(无信息) | 64字节 |

## 直观理解

- **PPP帧就像一封简单的信**：信封（标志）+ 简单地址（0xFF）+ 信件类型标签（协议）+ 信件内容（信息）+ 验证码（FCS）
- **为什么地址固定为0xFF**：点对点链路只有两端，不需要复杂的寻址

## 协议关联

- PPP的标志字段0x7E与HDLC相同
- PPPoE将PPP帧封装在以太网帧中
- PPP不提供可靠传输——这与HDLC不同（HDLC可提供可靠传输）
- 408考试中常与HDLC对比
